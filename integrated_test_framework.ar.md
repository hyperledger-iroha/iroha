# إطار اختبارات التكامل لـ Hyperledger Iroha (شبكة من 7 عقد)

## مقدمة

يوفر Hyperledger Iroha v2 مجموعة غنية من تعليمات Iroha الخاصة (ISI) لإدارة النطاقات والأصول والصلاحيات والأقران. يحدد هذا المستند إطار اختبارات تكامل لشبكة من 7 أقران للتحقق من الصحة والإجماع والتناسق بين الأقران في الظروف العادية ومع الأعطال (مع تحمل حتى 2 من الأقران المعطّلين).

تعكس هذه الإرشادات الانتقال الأخير لمخطط genesis إلى كتل متعددة المعاملات.

ملاحظة حول واجهة API: في هذه الشيفرة، Torii هي واجهة HTTP/WebSocket (Axum). ينبغي للاختبارات استخدام نقاط النهاية HTTP (غالباً على المنافذ 8080+). لا توجد خدمة RPC إضافية تستمع على 50051.

## الأهداف

- تنسيق تلقائي لشبكة من 7 أقران: تشغيل وإدارة الأقران برمجياً لسرعة CI (الأساسي)، أو عبر Docker Compose (اختياري).
- إعداد genesis: البدء من genesis Norito مشترك مع حسابات/مفاتيح حتمية وصلاحيات مطلوبة.
- تمرين ISI: تغطية تعليمات ISI بشكل منهجي عبر المعاملات والتحقق من الحالة الناتجة.
- تناسق بين الأقران: الاستعلام من عدة أقران بعد كل خطوة لضمان حالة دفتر أستاذ متطابقة.
- فحوص تحمل الأعطال: مع حتى 2 من الأقران خارج الخدمة أو معزولين، تتابع الأقران الباقية العمل؛ وتلحق الأقران العائدة بدون تباين.

## أوضاع التنسيق

- هارنس Rust (موصى به): `crates/iroha_test_network` يشغّل عمليات `irohad` محلياً، ويخصص المنافذ، ويكتب إعدادات لكل تشغيل وgenesis Norito بامتداد `.nrt`، ويراقب الجاهزية وارتفاعات الكتل، ويوفر أدوات للإيقاف/إعادة التشغيل.
- Docker Compose (اختياري): `crates/iroha_swarm` يولّد ملف Compose لعدد N من الأقران. استخدمه عندما تكون الشبكات المعتمدة على الحاويات أو التنسيق الخارجي مطلوبة.

## إعداد شبكة الاختبار

**كتلة genesis:**

- المصدر: `defaults/genesis.json` الذي يجمع التعليمات الآن ضمن مصفوفة `transactions`. يمكن للاختبارات إلحاق تعليمات عبر `NetworkBuilder::with_genesis_instruction` وبدء معاملة جديدة عبر `.next_genesis_transaction()`. تُسلسل الكتلة الناتجة إلى Norito `.nrt`.
- الطوبولوجيا: مخزنة في أول معاملة (`transactions[0].topology`) وتضم كل الأقران السبعة (المفتاح العام + العنوان)، بحيث يعرف كل قرين الشبكة منذ البداية.
- الحسابات/الصلاحيات: يُفضل استخدام الحسابات المعيارية من `crates/iroha_test_samples` (`ALICE_ID`, `BOB_ID`, `SAMPLE_GENESIS_ACCOUNT_KEYPAIR`) مع منح صريحة لسيناريوهات الاختبار، مثل `CanManagePeers`، `CanManageRoles`، `CanMintAssetWithDefinition`.
- استراتيجية الحقن: مع الهارنس، يُقدَّم genesis عادةً لقرين واحد ("genesis submitter")؛ وتلحق الأقران الأخرى عبر مزامنة الكتل. مع Compose، اجعل كل الأقران تشير إلى المسار نفسه لملف genesis.

**الشبكات والمنافذ:**

- واجهة Torii HTTP: `API_ADDRESS` (Axum). مع Compose، اربط المنافذ `8080`–`8086` للأقران السبعة إلى المضيف. الهارنس يخصص منافذ loopback تلقائياً.
- P2P: العنوان الداخلي بين الأقران يُضبط في `trusted_peers` ويُبث عبر gossip. الهارنس يضبط `trusted_peers` تلقائياً لكل تشغيل.

**تخزين البيانات:**

- Kura: تخزين كتل Iroha v2 (لا حاجة لحاويات RocksDB/Postgres). يُضبط عبر `[kura]` (مثل `store_dir`). عطّل اللقطات للاختبارات عبر `[snapshot]`.

**تدفق الهارنس:**

1) بناء الشبكة: `NetworkBuilder::new().with_peers(7)`؛ ويمكن إضافة `.with_pipeline_time(...)` و`.with_config_layer(...)` للتخصيص؛ واختيار وقود IVM عبر `IvmFuelConfig`.
2) تشغيل الأقران: `.start()` أو `.start_blocking()`؛ يكتب طبقات الإعداد، ويضبط `trusted_peers`، ويحقن genesis لقرين واحد، وينتظر الجاهزية.
3) الجاهزية: `Network::ensure_blocks(height)` أو `once_blocks_sync(...)` تضمن وصول ارتفاعات الكتل غير الفارغة للهدف عبر الأقران. أو يمكن الاستعلام عبر `Client::get_status()`.

**تدفق Docker Compose (اختياري):**

1) توليد compose: استخدم `iroha_swarm::Swarm` لإخراج ملف compose بعدد N من الأقران. اربط منافذ API للمضيف واضبط متغيرات البيئة (CHAIN, keys, TRUSTED_PEERS, GENESIS).
2) التشغيل: `docker compose up`.
3) الجاهزية: استقصِ نقاط Torii HTTP حتى يكون الوضع صحياً ويصبح ارتفاع الكتلة >= 1.

## تنفيذ هارنس الاختبارات (Rust)

**العميل والمعاملات:**

- استخدم `iroha::client::Client` (HTTP/WebSocket) لإرسال المعاملات والاستعلامات إلى Torii.
- ابنِ المعاملات من تسلسلات ISI `InstructionBox` أو بايت كود IVM؛ وقّع بمفاتيح `KeyPair` حتمية من `iroha_test_samples`.
- نداءات مفيدة: `submit_blocking`، `submit_all_blocking`، `query(...).execute()/execute_all()`، `get_status()`، وبث الكتل والأحداث عبر WebSocket.

**التناسق بين الأقران:**

- بعد كل عملية، استعلم من كل قرين يعمل (`Network::peers()` -> `peer.client()`) وقارن النتائج (مثل الأرصدة، التعريفات، قائمة الأقران). هذا يضمن فحوص تناسق أقوى من التحقق من قرين واحد.

**حقن الأعطال (بدون حاويات):**

- استخدم أدوات relay/proxy في `integration_tests/tests/extra_functional/unstable_network.rs` لإعادة كتابة `trusted_peers` إلى بروكسيات TCP وتعليق الوصلات بشكل انتقائي. هذا يتيح التقسيمات، والإسقاطات، وإعادة الاتصال المستهدفة.

**متطلبات IVM:**

- بعض الاختبارات تحتاج عينات IVM مبنية مسبقاً؛ تأكد من وجود `crates/ivm/target/prebuilt/build_config.toml`. عند غيابها يمكن تخطي الاختبارات (اختبارات التكامل الحالية تتحقق من ذلك).

## مخطط سيناريو من 7 عقد

1) شغّل شبكة من 7 أقران (هارنس أو Compose) وانتظر اعتماد genesis عبر جميع الأقران.
2) نفّذ مجموعة من ISI:
   - تسجيل نطاقات/حسابات/أصول؛ منح/سحب الصلاحيات؛ سك/حرق/نقل الأصول؛ ضبط/إزالة قيم المفتاح؛ تسجيل المشغلات؛ ترقية الـ executor.
3) بعد كل خطوة منطقية، نفّذ استعلامات بين الأقران للتحقق من تطابق الحالة.
4) أوقف 1–2 من الأقران، وواصل إرسال المعاملات عبر الأقران الخمسة الباقية؛ تأكد من التقدم والتناسق بين الأقران التي تعمل.
5) أعد تشغيل الأقران المتوقفة وتحقق من اللحاق ومن تطابق الحالة بين الأقران.

## الاستخدام في CI

- البناء: `cargo build --workspace`
- بناء عينات IVM مسبقاً (حسب الحاجة)
- الاختبار: `cargo test --workspace`
- تدقيق إضافي (اختياري): `cargo clippy --workspace --all-targets -- -D warnings`

## الخلاصة

يعتمد هذا الإطار على هارنس Rust داخل المستودع (`iroha_test_network`) وعميل HTTP (`iroha::client::Client`) للتحقق من شبكة Iroha v2 المكونة من 7 أقران. يركّز على التناسق بين الأقران، وسيناريوهات أعطال واقعية، وإعداد/تفكيك قابل لإعادة التنفيذ ومناسب لـ CI. يتوفر Docker Compose عبر `iroha_swarm` عندما تكون الحاويات مفضلة.

## المستكشف

- يمكن لأي مستكشف بلوكشين يتحدث Torii HTTP/WebSocket الاتصال بكل قرين بشكل مستقل. كل قرين يوفّر نقطة Torii (host:port) مناسبة للحالة والكتل والاستعلامات والأحداث.
- مع هارنس Rust: بعد بناء الشبكة يمكنك استخراج عناوين Torii لكل الأقران:

  ```rust
  use iroha_test_network::NetworkBuilder;

  let network = NetworkBuilder::new().with_peers(7).build();
  let urls = network.torii_urls();
  // مثال: ["http://127.0.0.1:8080", ..., "http://127.0.0.1:8086"]
  ```

  تتوفر أيضاً أدوات لكل قرين: `peer.api_address()` و`peer.torii_url()`.

- مع Docker Compose (`iroha_swarm`): ولّد compose لسبعة أقران واربط `8080..8086` بالمضيف؛ وجّه المستكشف إلى هذه العناوين. إذا كان المستكشف يدعم عدة نقاط نهاية فاضبط الأقران السبعة؛ وإلا شغّل مثيلًا واحدًا لكل قرين.
