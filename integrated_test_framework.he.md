# מסגרת בדיקות אינטגרציה ל-Hyperledger Iroha (רשת בת 7 צמתים)

<div dir="rtl">

## הקדמה

Hyperledger Iroha v2 מספקת אוסף רחב של Iroha Special Instructions (ISI) לניהול דומיינים, נכסים, הרשאות וצמתים. מסמך זה מגדיר מסגרת בדיקות אינטגרציה עבור רשת בת 7 צמתים, שנועדה לאמת נכונות, קונצנזוס ועקביות בין צמתים בתנאים תקינים ובתרחישי כשל (עד שני צמתים תקולים).

ההנחיות מעודכנות לעיצוב הג׳נסיס החדש שבו בלוקים מכילים מספר טרנזאקציות.

הערה על ה-API: בקוד־בסיס זה Torii הוא API של HTTP/WebSocket (Axum). הבדיקות משתמשות בקצות HTTP (בדרך כלל פורטים ‎8080‎ ומעלה). אין שירות RPC נוסף שמאזין ב-50051.

## מטרות

- תזמור אוטומטי של 7 צמתים: הפעלה וניהול תהליכי `irohad` באופן תכנותי ל-CI מהיר (ברירת מחדל), או באמצעות Docker Compose (אופציונלי).
- קונפיגורציית ג׳נסיס: התחלה מג׳נסיס Norito משותף עם חשבונות/מפתחות דטרמיניסטיים והרשאות נדרשות.
- הפעלת ISI: כיסוי שיטתי של פקודות ISI באמצעות טרנזאקציות ואימות המצב שנוצר.
- עקביות בין צמתים: ביצוע שאילתות מול מספר צמתים אחרי כל שלב כדי לוודא שסמכות הלדג׳ר זהה.
- בדיקות סבילות לתקלות: כאשר עד שני צמתים מנותקים או מופרדים, שאר הצמתים ממשיכים; עם חזרתם, הצמתים מצטרפים מחדש ללא סטייה.

## מצבי תזמור

- הרנס (Rust) מומלץ: `crates/iroha_test_network` יוצר תהליכי `irohad` מקומיים, מקצה פורטים, כותב קונפיגורציות ו־Norito `.nrt` עבור כל ריצה, עוקב אחרי מוכנות וגובה בלוקים ומספק כלים לכיבוי/הפעלה מחדש.
- Docker Compose (אופציונלי): `crates/iroha_swarm` מייצר קובץ Compose עבור N צמתים. מתאים כאשר רוצים רשת ממוכנת או תזמור חיצוני.

## הגדרת רשת הבדיקות

**בלוק ג׳נסיס:**

- מקור: `defaults/genesis.json` שבו ההוראות מאורגנות במערך `transactions`. בדיקות יכולות להוסיף הוראות דרך `NetworkBuilder::with_genesis_instruction` ולפתוח טרנזאקציה חדשה עם `.next_genesis_transaction()`. הבלוק הסופי מסודר לפורמט Norito `.nrt`.
- טופולוגיה: נשמרת בטרנזאקציה הראשונה (`transactions[0].topology`) וכוללת את כל 7 הצמתים (מפתח פומבי + כתובת) כך שכל צומת מכיר את הרשת מרגע האתחול.
- חשבונות/הרשאות: מומלץ להשתמש בחשבונות התקניים מ-`crates/iroha_test_samples` (`ALICE_ID`, ‏`BOB_ID`, ‏`SAMPLE_GENESIS_ACCOUNT_KEYPAIR`) ולהעניק הרשאות מפורשות לתרחישים, למשל `CanManagePeers`, ‏`CanManageRoles`, ‏`CanMintAssetWithDefinition`.
- אסטרטגיית הזרקה: עם ההרנס, הג׳נסיס נמסר בדרך כלל לצומת יחיד (״מגיש הג׳נסיס״); שאר הצמתים מתעדכנים דרך Block Sync. עם Compose כל הצמתים מצביעים לאותו נתיב ג׳נסיס.

**רשת ופורטים:**

- Torii HTTP API: מוגדר כ-`API_ADDRESS` (Axum). ב-Compose ממפים את ‎`8080`–`8086`‎ למכונה המקומית עבור 7 הצמתים. ההרנס מקצה פורטי loopback אוטומטית.
- P2P: כתובת ה-P2P הפנימית מוגדרת ב-`trusted_peers` ומופצת בגוסיפ. ההרנס מייצר `trusted_peers` ייעודיים לכל ריצה.

**אחסון נתונים:**

- Kura: אחסון בלוקים של Iroha v2 (אין צורך ב-RocksDB/Postgres). מגדירים דרך `[kura]` (למשל `store_dir`). משביתים snapshotים בבדיקות עם `[snapshot]`.

**תהליך הרנס:**

1. בניית הרשת: `NetworkBuilder::new().with_peers(7)`; ניתן להוסיף `.with_pipeline_time(...)` ו-`.with_config_layer(...)` להגדרות מותאמות; לבחור דלק IVM דרך `IvmFuelConfig`.
2. הפעלת הצמתים: `.start()` או `.start_blocking()`; מייצר שכבות קונפיגורציה, קובע `trusted_peers`, מזריק ג׳נסיס לצומת אחד וממתין למוכנות.
3. מוכנות: `Network::ensure_blocks(height)` או `once_blocks_sync(...)` לווידוא שגובהי הבלוקים מגיעים לערך הנדרש בכל הצמתים. לחלופין, לקרוא `Client::get_status()`.

**תהליך Docker Compose (רשות):**

1. הפקת Compose: להשתמש ב-`iroha_swarm::Swarm` כדי ליצור קובץ Compose עם N צמתים. למפות פורטי API למארח ולהגדיר משתני סביבה (CHAIN, מפתחות, TRUSTED_PEERS, GENESIS).
2. הפעלה: `docker compose up`.
3. מוכנות: לבצע polling ל-Torii עד שהסטטוס בריא וגובה הבלוק ≥ 1.

## מימוש ההרנס (Rust)

**לקוח וטרנזאקציות:**

- להשתמש ב-`iroha::client::Client` (HTTP/WebSocket) לשליחת טרנזאקציות ושאילתות אל Torii.
- להרכיב טרנזאקציות מרצפי `InstructionBox` של ISI או מקוד בייטקוד של IVM; לחתום באמצעות `KeyPair` דטרמיניסטיים מ-`iroha_test_samples`.
- קריאות שימושיות: `submit_blocking`, ‏`submit_all_blocking`, ‏`query(...).execute()/execute_all()`, ‏`get_status()`, וזרמי בלוקים/אירועים ב-WebSocket.

**עקביות בין צמתים:**

- אחרי כל פעולה, לבצע שאילתה מול כל צומת פעיל (`Network::peers()` → ‎`peer.client()`‎) ולהשוות תוצאות (לדוגמה יתרות, הגדרות, רשימת צמתים). כך מקבלים בדיקה חזקה שמעבר לשאילתה בודדת.

**הזרקת תקלות (ללא קונטיינרים):**

- להשתמש בכלי הריליי/פרוקסי שב-`integration_tests/tests/extra_functional/unstable_network.rs` כדי להפנות את `trusted_peers` דרך פרוקסי TCP ולנתק/לעכב קישורים באופן סלקטיבי. כך ניתן לדמות חלוקות רשת, נפילות וחיבורים מחדש.

**דרישות IVM:**

- חלק מהבדיקות דורשות דוגמאות IVM שנבנו מראש; יש לוודא ש-`crates/ivm/target/prebuilt/build_config.toml` קיים. אם חסר, ניתן לדלג על הבדיקות (כך פועלות בדיקות האינטגרציה כיום).

## תרחיש רשת בת 7 צמתים

1. מפעילים רשת של 7 צמתים (הרנס או Compose) וממתינים שהג׳נסיס יתקבל בכל הצמתים.
2. מבצעים סט של ISI:
   - רישום דומיינים/חשבונות/נכסים; הקצאה/ביטול הרשאות; יצירה/שריפה/העברת נכס; כתיבה/מחיקה של מפתחות; רישום טריגרים; עדכון מנוע הביצוע.
3. אחרי כל שלב לוגי, מריצים שאילתות על פני כל הצמתים כדי לאמת שהמצב זהה.
4. מכבים 1–2 צמתים, ממשיכים לשלוח טרנזאקציות עם 5 הצמתים שנותרו; מוודאים התקדמות ועקביות בין הצמתים הפעילים.
5. מפעילים מחדש את הצמתים העצורים ומאמתים שהן השיגו את שאר הרשת והמצב זהה.

## שימוש ב-CI

- בנייה: `cargo build --workspace`
- בניית דוגמאות IVM מראש (לפי צורך הבדיקות)
- בדיקות: `cargo test --workspace`
- אופציונלי – lint מחמיר: `cargo clippy --workspace --all-targets -- -D warnings`

## סיכום

המסגרת נשענת על ההרנס ב-Rust (`iroha_test_network`) ועל לקוח ה-HTTP (`iroha::client::Client`) כדי לאמת רשת Iroha v2 בת 7 צמתים. היא מדגישה עקביות בין צמתים, תרחישי כשל ריאליסטיים והקמה/פירוק חוזרים המתאימים ל-CI. Docker Compose דרך `iroha_swarm` זמין כאשר מעדיפים סביבה קונטיינרית.

## אקספלורר

- כל אקספלורר בלוקצ׳יין שמדבר Torii HTTP/WebSocket יכול להתחבר לכל צומת בנפרד. כל צומת חושף קצה Torii (host:port) לשימוש בסטטוס, בלוקים, שאילתות ואירועים.
- עם ההרנס ב-Rust: לאחר בניית הרשת ניתן לגזור את כתובות ה-URL של כלל הצמתים:

  ```rust
  use iroha_test_network::NetworkBuilder;

  let network = NetworkBuilder::new().with_peers(7).build();
  let urls = network.torii_urls();
  // לדוגמה ["http://127.0.0.1:8080", ..., "http://127.0.0.1:8086"]
  ```

  קיימים גם הלפרים פרטיים: ‎`peer.api_address()`‎ ו-‎`peer.torii_url()`‎.

- עם Docker Compose (`iroha_swarm`): מייצרים קובץ Compose ל-7 צמתים וממפים את ‎8080..8086‎ למארח; מכוונים את האקספלורר לכל אחת מן הכתובות. אם האקספלורר תומך במספר נקודות, מגדירים את כולן; אחרת מריצים מופע אחד לכל צומת.
