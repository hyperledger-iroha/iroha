# תהליך לוקליזציה של התיעוד

תיקיית `docs/i18n` מרכזת את תצורת הלוקליזציה ואת כלי העבודה. קובצי האנגלית נשארים אמת המידה היחידה; את קובצי התרגום שומרים לצד המקור בתבנית `<name>.<language-code>.<ext>` (לדוגמה, `README.ja.md`).

## תצורה

`manifest.json` מתאר אילו קבצים דורשים תרגום ואילו שפות אנו מתחזקים. המפתחות החשובים:

- `primary_language` – השפה הקנונית (אנגלית).
- `target_languages` – שפות היעד. כל ערך מכיל קוד ISO, שם קריא, שם עצמי (`autonym`, אם קיים) וכיוון כתיבה.
- `include` – תבניות glob יחסיות לשורש הריפו שמוסיפות קבצים לתהליך.
- `exclude` – תבניות glob לקבצים/עץ שאיננו רוצים לשקף.

עדכנו את המניפסט כאשר מוסיפים ספריות תיעוד או שפות חדשות.

שפות היעד הנוכחיות:

- יפנית (`ja`, 日本語, LTR)
- עברית (`he`, עברית, RTL)
- ספרדית (`es`, Español, LTR)
- פורטוגזית (`pt`, Português, LTR)
- צרפתית (`fr`, Français, LTR)
- רוסית (`ru`, Русский, LTR)
- ערבית (`ar`, العربية, RTL)
- אורדו (`ur`, اردو, RTL)

## יצירת קובצי סטב

הפעילו את הסקריפט משורש הריפו כדי ליצור קבצים ראשוניים:

```bash
python3 scripts/sync_docs_i18n.py
```

הסקריפט סורק את קובצי המקור באנגלית שמותאמים על-ידי `include`, מדלג על נתיבים שכבר כוללים תרגומים, ויוצר סטב עם Metadata למתרגמים. השתמשו ב-`--dry-run` כדי להציג את התכנון מראש, או ב-`--lang <code>` כדי להגביל לשפה מסוימת.

## תרגום

- החליפו את תוכן הסטב בתרגום וסמנו את `status` בפרונט-מאטר כמתאים (למשל `in-progress` או `complete`).
- שמרו על מפתחות הפרונט-מאטר כך שהכלים יוכלו לעקוב אחרי ההתקדמות.
- אם מסמך המקור זז או שינה שם, עדכנו בהתאם את שם קובץ התרגום.

## הוספת שפות חדשות

1. הוסיפו ערך ל-`target_languages` ב-`manifest.json`.
2. הריצו `python3 scripts/sync_docs_i18n.py --lang <code>` כדי ליצור סטבים.
3. תרגמו את הקבצים החדשים.
4. עדכנו תהליכי בנייה/פרסום כך שיציגו את השפה החדשה (לדוגמה הוספת פילטר לשפה במחוללי האתר).

## שילוב ב-CI

בעתיד ניתן להוסיף בדיקת CI לוודא שלכל קובץ מקור קיימים סטבים לכל שפה. גישה פשוטה: להריץ את הסקריפט במצב `--dry-run` ולהכשיל את המשימה אם דווח על יצירת קבצים. בריפו הנוכחי בדיקה זו עדיין אינה מאוכפת, אך הפלט של הסקריפט מותאם לעיבוד אוטומטי.
