# RFC по Структуре Аккаунтов

**Статус:** Принят (ADDR‑1)  
**Аудитория:** команды по модели данных, Torii, Nexus, Wallet и Governance  
**Связанные задачи:** будут определены

## Резюме

В этом документе предлагается переработка схемы адресации аккаунтов с целью:

- Ввести человекочитаемый **Iroha Bech32m‑адрес (IH‑B32)** с checksum’ом, который
  привязывает дискриминант цепи к подписанту аккаунта и предоставляет
  детерминированные текстовые формы, удобные для межплатформенной
  совместимости.
- Ввести глобально уникальные идентификаторы доменов, поддерживаемые
  реестром, к которому можно обращаться через Nexus для межцепного
  маршрутизации.
- Сохранить совместимость с alias’ами маршрутизации вида `alias@domain`
  на время миграции кошельков, API и контрактов к новому формату.

## Мотивация

Сегодня кошельки и off‑chain‑инструменты опираются на “сырые” alias’ы
маршрутизации `alias@domain`. Это порождает два существенных недостатка:

1. **Отсутствие привязки к сети.** В строке нет ни checksum’а, ни префикса
   сети, поэтому пользователь может вставить адрес из другой сети и не
   получить немедленного сигнала об ошибке. В итоге транзакция будет
   отклонена (несовпадение `chain_id`) или — хуже — успешно выполнена в
   отношении неожиданного аккаунта, если такой получатель существует локально.
2. **Коллизия доменов.** Домены являются лишь пространством имён и могут
   переиспользоваться на каждой цепи. Федерация сервисов (кастодианы, мосты,
   cross‑chain‑воркфлоу) становится хрупкой, поскольку `finance` в цепи A никак
   не связан с `finance` в цепи B.

Нам нужен человекочитаемый формат адреса, который защищает от ошибок
копирования/вставки, а также детерминированное отображение имени домена в
авторитетную цепь.

## Цели

- Спроектировать Bech32m‑оболочку адреса, вдохновлённую IH58, для аккаунтов
  Iroha и при этом публиковать канонические текстовые alias’ы CAIP‑10.
- Вшить настроенный дискриминант цепи непосредственно в каждый адрес и
  описать процесс его регистрации и governance.
- Описать, как ввести глобальный реестр доменов, не ломая существующие
  развёртывания, и зафиксировать правила нормализации/anti‑spoofing.
- Задокументировать ожидания по совместимости, шаги миграции и открытые
  вопросы.

## Не цели

- Реализация cross‑chain‑переводов активов. Слой маршрутизации лишь возвращает
  целевую цепь.
- Изменение внутренней структуры `AccountId` (по‑прежнему
  `DomainId + PublicKey`).
- Окончательная формализация governance‑модели для глобальной выдачи доменов.
  Данный RFC концентрируется на модели данных и транспортных примитивах.

## Предпосылки

### Текущий alias маршрутизации

```text
AccountId {
    domain: DomainId,   // обёртка над Name (строка, близкая к ASCII)
    signatory: PublicKey // multihash‑строка (например, ed0120...)
}

Отображение / парсинг: "<signatory multihash>@<domain name>"

Эта текстовая форма теперь рассматривается как **alias аккаунта** — удобный
маршрутизатор, указывающий на канонический
[`AccountAddress`](#2-canonical-address-codecs). Она остаётся полезной для
чтения человеком и доменно‑ориентированного управления, но больше не
рассматривается как авторитетный on‑chain‑идентификатор аккаунта.
```

`ChainId` живёт вне `AccountId`. Узлы проверяют `ChainId` транзакции против
конфигурации при приёме (`AcceptTransactionFail::ChainIdMismatch`) и
отклоняют “чужие” транзакции, однако строка аккаунта сама по себе не несёт
сетевого префикса.

### Идентификаторы доменов

`DomainId` оборачивает нормализованную строку `Name` и ограничен локальной
цепью. Каждая цепь может независимо регистрировать домены вроде `wonderland`,
`finance` и т.д.

### Контекст Nexus

Nexus отвечает за координацию между компонентами (lanes/data‑spaces). Сейчас у
него нет понятия межцепной маршрутизации доменов.

## Предлагаемый дизайн

### 1. Детерминированный дискриминант цепи

Структура `iroha_config::parameters::actual::Common` дополняется полем:

```rust
pub struct Common {
    pub chain: ChainId,
    pub chain_discriminant: u16, // новое поле, глобально координируется
    // ... существующие поля
}
```

- **Ограничения:**
  - Уникальность на каждую активную сеть; управляется подписанным публичным
    реестром с явными резервированными диапазонами (например
    `0x0000–0x0FFF` для test/dev, `0x1000–0x7FFF` для community‑выделений,
    `0x8000–0xFFEF` для областей, одобренных governance,
    `0xFFF0–0xFFFF` зарезервированы).
  - Неизменяемость для работающей цепи. Изменение требует hard fork’а и
    обновления реестра.
- **Governance и реестр:** мультисиг‑набор валидаторов поддерживает
  подписанный JSON‑реестр (дополнительно закреплённый в IPFS), сопоставляющий
  дискриминанты с человекочитаемыми alias’ами и CAIP‑2‑идентификаторами.
  Клиенты загружают и кэшируют этот реестр, чтобы валидировать и отображать
  метаданные цепи.
- **Использование:** дискриминант прокидывается через приём состояния,
  Torii, SDK и wallet‑API, так что каждый компонент может встраивать или
  проверять его. Значение экспонируется как часть CAIP‑2 (например
  `iroha:0x1234`).

### 2. Канонические кодеки адресов

Модель данных на Rust экспонирует одну каноническую бинарную
репрезентацию (`AccountAddress`), которая может выводиться в нескольких
человекочитаемых форматах:

- **IH58 (Iroha Base58)** — Base58‑оболочка, встраивающая дискриминант цепи.
  Декодеры проверяют префикс перед тем, как продвигать payload к
  каноническому виду.
- **Sora‑compressed view** — специальный алфавит из **105 символов** для Sora,
  полученный добавлением half‑width‑варианта поэмы イロハ (включая ヰ и ヱ) к
  58‑символьному набору IH58. Строки начинаются с sentinel‑префикса `snx1`,
  содержат checksum на основе Bech32m и опускают префикс сети (Sora Nexus
  подразумевается sentinel’ом).

  ```text
  IH58  : 123456789ABCDEFGHJKMNPQRSTUVWXYZabcdefghijkmnopqrstuvwxyz
  Iroha : ｲﾛﾊﾆﾎﾍﾄﾁﾘﾇﾙｦﾜｶﾖﾀﾚｿﾂﾈﾅﾗﾑｳヰﾉｵｸﾔﾏｹﾌｺｴﾃｱｻｷﾕﾒﾐｼヱﾋﾓｾｽ
  ```
- **Канонический hex** — удобное для отладки `0x…`‑кодирование
  канонического бинарного envelope.

`AccountAddress::parse_any` автоматически распознаёт сжатые строки, IH58 или
канонический hex и возвращает как декодированный payload, так и
обнаруженный `AccountAddressFormat`. Torii вызывает `parse_any` для
дополнительных ISO 20022‑адресов и сохраняет каноническую hex‑форму, чтобы
метаданные оставались детерминированными вне зависимости от исходной
репрезентации.

#### 2.1 Разметка байта заголовка (ADDR‑1a)

Каждый канонический payload имеет вид `header · domain selector · controller`.
Поле `header` — это один байт, задающий правила парсинга для следующих
байтов:

```text
bit index:   7        5 4      3 2      1 0
             ┌─────────┬────────┬────────┬────┐
payload bit: │version  │ class  │  norm  │ext │
             └─────────┴────────┴────────┴────┘
```

Этот первый байт упаковывает метаданные схемы для downstream‑декодеров:

| Биты | Поле          | Допустимые значения | Ошибка при нарушении |
|------|---------------|---------------------|----------------------|
| 7‑5  | `addr_version` | `0` (v1). Значения `1‑7` зарезервированы под будущие ревизии. | Значения вне `0‑7` порождают `AccountAddressError::InvalidHeaderVersion`; реализации ДОЛЖНЫ трактовать ненулевые версии как пока не поддерживаемые. |
| 4‑3  | `addr_class`  | `0` = одиночный ключ, `1` = multisig. | Иные значения дают `AccountAddressError::UnknownAddressClass`. |
| 2‑1  | `norm_version` | `1` (Norm v1). Значения `0`, `2`, `3` зарезервированы. | Значения вне `0‑3` дают `AccountAddressError::InvalidNormVersion`. |
| 0    | `ext_flag`    | ДОЛЖЕН быть `0`.    | Установленный бит → `AccountAddressError::UnexpectedExtensionFlag`. |

Rust‑encoder всегда пишет `0x02` (версия 0, класс одиночного ключа, версия
нормализации 1, флаг расширения сброшен).

#### 2.2 Кодирование селектора домена (ADDR‑1a)

Селектор домена следует сразу за заголовком и представляет собой
tagged‑union:

| Tag   | Значение                   | Payload   | Примечания |
|-------|----------------------------|-----------|------------|
| `0x00` | Неявный домен по умолчанию | нет       | Соответствует настроенному `default_domain_name()`. |
| `0x01` | Локальный digest домена   | 12 байт   | Digest = `blake2s_mac(key = "SORA-LOCAL-K:v1", canonical_label)[0..12]`. |
| `0x02` | Запись глобального реестра | 4 байта  | Big‑endian `registry_id`; зарезервировано до появления глобального реестра. |

Метки доменов канонизируются (UTS‑46 + STD3 + NFC) до вычисления hash’а.
Неизвестные теги приводят к `AccountAddressError::UnknownDomainTag`. При
сопоставлении адреса и домена несогласованный селектор даёт
`AccountAddressError::DomainMismatch`.

```text
domain selector
┌──────────┬──────────────────────────────────────────────┐
│ tag (u8) │ payload (depends on selector kind, see table)│
└──────────┴──────────────────────────────────────────────┘
```

Селектор расположен сразу перед контроллером, поэтому декодер может идти по
on‑wire‑формату последовательно: считать байт тега, затем payload нужной длины
и перейти к байтам контроллера.

**Примеры селектора**

- *Неявный домен по умолчанию* (`tag = 0x00`). Payload отсутствует. Пример
  канонического hex для домена по умолчанию с детерминированным тестовым
  ключом:
  `0x0200000120641297079357229f295938a4b5a333de35069bf47b9d0704e45805713d13c201`.
- *Локальный digest* (`tag = 0x01`). Payload — 12‑байтовый digest. Пример
  (`treasury`, seed `0x01`):
  `0x0201b18fe9c1abbac45b3e38fc5d0001203b77a042f1de02f6d5f418f36a20fd68c8329fe3bbfbecd26a2d72878cd827f8`.
- *Глобальный реестр* (`tag = 0x02`). Payload — big‑endian `registry_id:u32`.
  Последующие байты совпадают с кейсом неявного домена по умолчанию; селектор
  лишь заменяет нормализованную строку домена на указатель в реестр. Пример с
  `registry_id = 0x0000_002A` (42 в десятичном) и детерминированным
  контроллером по умолчанию:
  `0x02020000002a000120641297079357229f295938a4b5a333de35069bf47b9d0704e45805713d13c201`.

#### 2.3 Кодирование payload’а контроллера (ADDR‑1a)

Payload контроллера — ещё один tagged‑union, идущий сразу за селектором
домена:

| Tag   | Контроллер | Layout | Примечания |
|-------|------------|--------|------------|
| `0x00` | Одиночный ключ | `curve_id:u8` · `key_len:u8` · `key_bytes` | `curve_id = 0x01` сегодня соответствует Ed25519. `key_len` ограничен `u8`; большие значения дают `AccountAddressError::KeyPayloadTooLong`. |
| `0x01` | Multisig        | `version:u8` · `threshold:u16` · `member_count:u8` · (`curve_id:u8` · `weight:u16` · `key_len:u16` · `key_bytes`)\* | До 255 участников (`CONTROLLER_MULTISIG_MEMBER_MAX`). Неизвестные кривые → `AccountAddressError::UnknownCurve`; некорректные политики поднимаются как `AccountAddressError::InvalidMultisigPolicy`. |

Multisig‑политики также публикуют CTAP2‑подобную CBOR‑map и канонический
digest, чтобы хосты и SDK могли детерминированно валидировать контроллер.
Схема, процедура hashing’а и золотые фикстуры описаны в
`docs/source/references/address_norm_v1.md` (ADDR‑1c).

Все байты ключей кодируются ровно так, как их возвращает
`PublicKey::to_bytes`; декодеры восстанавливают `PublicKey` и выдают
`AccountAddressError::InvalidPublicKey`, если байты не соответствуют
заявленной кривой.

> **Каноническое требование для Ed25519 (ADDR‑3a).** Ключи с
> `curve_id = 0x01` должны декодироваться в точный байтовый массив,
> полученный от подписанта, и не могут лежать в подгруппе малого порядка.
> Узлы отклоняют некононические кодировки (например, значения, приведённые
> по модулю `2^255‑19`) и слабые точки вроде единичного элемента, поэтому
> SDK должны отражать такие ошибки валидации до отправки адресов.

##### 2.3.1 Реестр идентификаторов кривых (ADDR‑1d)

| ID (`curve_id`) | Алгоритм | Feature‑флаг | Примечания |
|-----------------|----------|-------------|------------|
| `0x00` | Зарезервировано | — | НЕ ДОЛЖЕН эмитироваться; декодеры возвращают `ERR_UNKNOWN_CURVE`. |
| `0x01` | Ed25519       | Всегда включён | Канонический алгоритм v1 (`Algorithm::Ed25519`); единственный id, активный в продакшене сегодня. |
| `0x02` | ML‑DSA (preview) | `ml-dsa` | Зарезервировано под rollout ADDR‑3; по умолчанию выключено до появления ML‑DSA‑подписи. |
| `0x03` | Placeholder GOST | `gost` | Зарезервировано для provisioning/negotiation; контроллеры с этим id ДОЛЖНЫ отклоняться до утверждения конкретного TC26‑профиля governance. |
| `0x0A` | GOST R 34.10‑2012 (256, набор A) | `gost` | Зарезервировано; открывается вместе с крипто‑feature `gost`. |
| `0x0B` | GOST R 34.10‑2012 (256, набор B) | `gost` | Зарезервировано для будущего решения governance. |
| `0x0C` | GOST R 34.10‑2012 (256, набор C) | `gost` | Зарезервировано для будущего решения governance. |
| `0x0D` | GOST R 34.10‑2012 (512, набор A) | `gost` | Зарезервировано для будущего решения governance. |
| `0x0E` | GOST R 34.10‑2012 (512, набор B) | `gost` | Зарезервировано для будущего решения governance. |
| `0x0F` | SM2          | `sm`   | Зарезервировано; станет доступным после стабилизации SM‑криптографии. |

Слоты `0x04–0x09` остаются свободными под дополнительные кривые; добавление
нового алгоритма требует обновления roadmap’а и согласованной поддержки в
SDK/host. Энкодеры ДОЛЖНЫ отклонять неподдерживаемые алгоритмы через
`ERR_UNSUPPORTED_ALGORITHM`, а декодеры ДОЛЖНЫ немедленно падать на
неизвестных id с `ERR_UNKNOWN_CURVE`, сохраняя fail‑closed‑поведение.

Канонический реестр (включая JSON‑экспорт) хранится в
[`docs/source/references/address_curve_registry.md`](source/references/address_curve_registry.md).
Инструменты ДОЛЖНЫ читать эти данные напрямую, чтобы идентификаторы кривых
оставались согласованными между SDK и операционными пайплайнами.

- **SDK‑gating.** SDK по умолчанию поддерживают только Ed25519. В Swift это
  compile‑time‑флаги (`IROHASWIFT_ENABLE_MLDSA`, `IROHASWIFT_ENABLE_GOST`,
  `IROHASWIFT_ENABLE_SM`); Java‑SDK требует вызова
  `AccountAddress.configureCurveSupport(...)`; JS‑SDK использует
  `configureCurveSupport({ allowMlDsa: true, allowGost: true, allowSm2: true })`,
  так что интеграторам нужно явно включать дополнительные кривые перед
  генерацией адресов.
- **Host‑gating.** В `Register<Account>` контроллеры отклоняются, если
  алгоритм подписи отсутствует в `crypto.allowed_signing` **или**
  `curve_id` не входит в `crypto.curves.allowed_curve_ids`; таким образом,
  кластеры обязаны объявить поддержку (конфигурация + genesis) до
  регистрации контроллеров на ML‑DSA/GOST/SM.

#### 2.4 Правила отказа (ADDR‑1a)

- Payload’ы короче требуемых header + selector или с лишними байтами
  приводят к `AccountAddressError::InvalidLength` или
  `AccountAddressError::UnexpectedTrailingBytes`.
- Заголовки, устанавливающие зарезервированный `ext_flag` или объявляющие
  неподдерживаемые версии/классы, ДОЛЖНЫ отклоняться с помощью
  `UnexpectedExtensionFlag`, `InvalidHeaderVersion` или
  `UnknownAddressClass`.
- Неизвестные теги selector/controller поднимают
  `UnknownDomainTag` или `UnknownControllerTag`.
- Слишком длинный или повреждённый payload ключа вызывает
  `KeyPayloadTooLong` или `InvalidPublicKey`.
- Multisig‑контроллеры с числом участников > 255 приводят к
  `MultisigMemberOverflow`.
- IME/NFKC‑конверсии: half‑width‑кана Sora могут нормализоваться в
  full‑width без нарушения декодинга, но ASCII‑sentinel `snx1` и цифры/буквы
  IH58 ДОЛЖНЫ оставаться ASCII. Sentinel’ы в full‑width или с case‑folding
  дают `ERR_MISSING_COMPRESSED_SENTINEL`; ASCII‑payload в full‑width
  приводит к `ERR_INVALID_COMPRESSED_CHAR`, а ошибки checksum’
  всплывают как `ERR_CHECKSUM_MISMATCH`. Property‑тесты в
  `crates/iroha_data_model/src/account/address.rs` покрывают эти ветви, чтобы
  SDK и кошельки могли опираться на детерминированные отказы.
- Парсер alias’ов `address@domain` в Torii и SDK теперь выдаёт те же коды
  `ERR_*`, когда IH58/сжатые входы падают до fallback’а в alias
  (например, при расхождении checksum’а или digest’а домена), так что клиенты
  могут сообщать структурированные причины вместо эвристики по тексту.

#### 2.5 Нормативные бинарные векторы

- **Неявный домен по умолчанию (`default`, байт seed `0x00`)**  
  Канонический hex:
  `0x0200000120641297079357229f295938a4b5a333de35069bf47b9d0704e45805713d13c201`.  
  Разбор: заголовок `0x02`, селектор `0x00` (implicit default), tag
  контроллера `0x00`, id кривой `0x01` (Ed25519), длина `0x20` и далее
  32‑байтовый ключ.
- **Локальный digest домена (`treasury`, байт seed `0x01`)**  
  Канонический hex:
  `0x0201b18fe9c1abbac45b3e38fc5d0001203b77a042f1de02f6d5f418f36a20fd68c8329fe3bbfbecd26a2d72878cd827f8`.  
  Разбор: заголовок `0x02`, tag селектора `0x01` и digest
  `b1 8f e9 c1 ab ba c4 5b 3e 38 fc 5d`, затем payload одиночного ключа
  (`0x00` как tag контроллера, `0x01` как id кривой, `0x20` как длина и
  32‑байтовый ключ Ed25519).

Unit‑тесты (`account::address::tests::parse_any_accepts_all_formats`)
проверяют эти V1‑векторы через `AccountAddress::parse_any`, гарантируя, что
инструменты могут полагаться на один и тот же канонический payload во всех
форматах (hex, IH58, сжатый). Расширенный набор фикстур можно
перегенерировать командой:

```text
cargo xtask address-vectors --out fixtures/account/address_vectors.json
```

### 3. Глобально уникальные домены и нормализация

См. также
[`docs/source/references/address_norm_v1.md`](source/references/address_norm_v1.md)
для канонического pipeline Norm v1, используемого в Torii, модели данных и
SDK.

`DomainId` переопределяется как tagged‑tuple:

```text
DomainId {
    name: Name,
    authority: GlobalDomainAuthority, // новый enum
}

enum GlobalDomainAuthority {
    LocalChain,                  // значение по умолчанию для локальной цепи
    External { chain_discriminant: u16 },
}
```

`LocalChain` оборачивает существующий `Name` для доменов, управляемых текущей
цепью. При регистрации домена в глобальном реестре сохраняется дискриминант
владельца‑цепи. На уровне отображения/парсинга структура пока не меняется, но
расширенная форма позволяет принимать routing‑решения.

#### 3.1 Нормализация и защита от spoofing

Norm v1 задаёт канонический pipeline, который должен применяться при каждом
сохранении имени домена или при его встраивании в `AccountAddress`. Полное
описание находится в
`docs/source/references/address_norm_v1.md`; ниже приведены шаги, которые
должны реализовать кошельки, Torii, SDK и governance‑инструменты.

1. **Валидация входа.** Отклонять пустые строки, пробельные значения и
   зарезервированные разделители `@`, `#`, `$`. Это совпадает с инвариантами
   `Name::validate_str`.
2. **Unicode‑композиция NFC.** Применить нормализацию NFC (через ICU), чтобы
   канонически эквивалентные последовательности сворачивались в одну
   форму (например, `e\u{0301}` → `é`).
3. **Нормализация UTS‑46.** Прогнать результат через UTS‑46 с
   `use_std3_ascii_rules = true`, `transitional_processing = false` и
   включённым контролем DNS‑длин. Выход — A‑label‑строка в нижнем регистре;
   входы, нарушающие STD3, отбрасываются на этом шаге.
4. **Ограничения длины.** Применить DNS‑подобные границы: длина каждого
   label ДОЛЖНА быть в диапазоне 1–63 байта, а полный домен НЕ ДОЛЖЕН
   превышать 255 байт после шага 3.
5. **Опциональная политика по confusable‑символам.** UTS‑39‑проверки
   отслеживаются для Norm v2; операторы могут включать их заранее, но
   несоответствие должно приводить к остановке обработки.

Если все шаги успешны, A‑label‑строка в нижнем регистре кешируется и
используется для кодирования адресов, конфигурации, manifests и запросов к
реестру. Локальные digest‑селекторы вычисляют своё 12‑байтовое значение как
`blake2s_mac(key = "SORA-LOCAL-K:v1", canonical_label)[0..12]` на выходе
шага 3. Любые альтернативные формы ввода (смешанный регистр, “сырой” Unicode
и т.п.) отклоняются с структурированными `ParseError` на границе ввода имени.

Канонические fixtures, демонстрирующие эти правила — включая punycode
round‑trip’ы и строки, нарушающие STD3, — описаны в
`docs/source/references/address_norm_v1.md` и дублируются в CI‑наборах
векторов SDK (ADDR‑2).

### 4. Реестр доменов Nexus и маршрутизация

- **Схема реестра.** Nexus поддерживает подписанную map
  `DomainName -> ChainRecord`, где `ChainRecord` включает дискриминант цепи,
  опциональные метаданные (RPC‑endpoint’ы) и доказательство полномочий
  (например, governance‑multisig).
- **Механизм синхронизации:**
  - Цепи отправляют в Nexus подписанные заявки на домены (на этапе genesis
    или через governance‑инструкции).
  - Nexus публикует периодические manifests (подписанный JSON и, опционально,
    Merkle‑root) по HTTPS и через контент‑адресуемое хранилище (например,
    IPFS). Клиенты закрепляют (pin) последний manifest и проверяют подписи.
- **Поток разрешения:**
  - Torii принимает транзакцию, ссылающуюся на `DomainId`.
  - Если домен неизвестен локально, Torii обращается к кешированному
    manifest’у Nexus.
  - Если из manifest’а следует, что домен принадлежит другой цепи, транзакция
    отклоняется с детерминированной ошибкой `ForeignDomain` и данными о
    удалённой цепи.
  - Если домен отсутствует в Nexus, Torii возвращает `UnknownDomain`.
- **Trust‑anchor’ы и ротация.** Manifest’ы подписываются governance‑ключами;
  ротация/ревокация публикуется как новая версия manifest’а. Клиенты
  применяют TTL (например, 24 ч) и отказываются использовать устаревшие
  данные после истечения окна.
- **Режимы отказа.** Если получение manifest’а не удаётся, Torii использует
  кешированный manifest до истечения TTL; по истечении TTL выдаётся
  `RegistryUnavailable` и кросс‑доменные маршруты блокируются во избежание
  неконсистентных состояний.

#### 4.1 Неизменяемость реестра, alias’ы и tombstone‑записи (ADDR‑7c)

Nexus публикует **append‑only‑manifest**, чтобы каждое назначение домена или
alias’а можно было аудировать и воспроизвести. Операторы должны считать
bundle из
`docs/source/runbooks/address_manifest_ops.md` единственным источником
истины: если manifest отсутствует или не проходит валидацию, Torii обязан
отказаться резолвить соответствующий домен.

Поддержка автоматизации:
`cargo xtask address-manifest verify --bundle <current_dir> --previous <previous_dir>`
переигрывает проверки checksum’а, схемы и `previous_digest`, описанные в
runbook’е. Вывод команды следует прикладывать к change‑tickets, чтобы
подтвердить, что связка `sequence` + `previous_digest` была проверена до
публикации bundle.

##### Заголовок manifest’а и контракт подписи

| Поле              | Требование |
|-------------------|-----------|
| `version`         | Сейчас `1`. Увеличивать только вместе с изменением спецификации. |
| `sequence`        | Увеличивается ровно на **1** при каждом релизе. Torii отклоняет ревизии с пропусками или откатами. |
| `generated_ms` + `ttl_hours` | Определяют свежесть кеша (по умолчанию 24 ч). Если TTL истёк до появления следующего manifest’а, Torii переходит в `RegistryUnavailable`. |
| `previous_digest` | BLAKE3‑digest (hex) тела предыдущего manifest’а. Проверяющие пересчитывают его через `b3sum` для доказательства неизменности. |
| `signatures`      | Manifest’ы подписываются через Sigstore (`cosign sign-blob`). Операционные команды обязаны запускать `cosign verify-blob --bundle manifest.sigstore manifest.json` и проверять идентичность/issuer governance‑ключей до раскатки. |

Release‑автоматизация публикует `manifest.sigstore` и `checksums.sha256`
рядом с JSON‑телом. При зеркалировании на SoraFS или HTTP‑endpoint’ы их
нужно держать вместе, чтобы аудиторы могли дословно воспроизвести процедуру
проверки.

##### Типы записей

| Тип            | Назначение | Обязательные поля |
|----------------|-----------|-------------------|
| `global_domain` | Объявляет, что домен зарегистрирован глобально и должен сопоставляться с дискриминантом цепи и IH58‑префиксом. | `{ "domain": "<label>", "chain": "sora:nexus:global", "ih58_prefix": 753, "selector": "global" }` |
| `tombstone`     | Безвозвратно снимает alias/selector. Обязателен при удалении Local‑8‑digest’ов или полном удалении домена. | `{ "selector": {…}, "reason_code": "LOCAL8_RETIREMENT" \| …, "ticket": "<governance id>", "replaces_sequence": <number> }` |

Записи `global_domain` могут опционально содержать `manifest_url` или
`sorafs_cid` для указания на подписанные метаданные цепи, но каноническое
сопоставление остаётся `{domain, chain, discriminant/ih58_prefix}`. Для
`tombstone`‑записей **обязательно** указывать, какой selector выведен из
обращения, и ссылку на governance‑ticket/артефакт, разрешивший изменение,
чтобы аудиторский след можно было восстановить offline.

##### Workflow alias/tombstone и телеметрия

1. **Обнаружить дрейф.** Используйте
   `torii_address_local8_total{endpoint}` и
   `torii_address_invalid_total{endpoint,reason}`
   (панели в `dashboards/grafana/address_ingest.json`), чтобы убедиться, что
   Local‑8‑строки больше не принимаются в продакшене перед тем, как
   предлагать tombstone.
2. **Вывести канонические digest’ы.** Запустите
   `iroha address convert <address-or-account_id> --format json --expect-prefix 753`
   (или используйте `fixtures/account/address_vectors.json` через
   `scripts/account_fixture_helper.py`), чтобы получить точное значение
   `digest_hex`. CLI принимает ввод вида `snx1...@wonderland`; JSON‑сводка
   содержит домен в поле `input_domain`, а флаг `--append-domain` формирует
   `<ih58>@wonderland` для обновления manifest’а. Для построчных экспортов
   используйте
   чтобы массово перевести Local‑селекторы в каноническую IH58‑форму (или
   compressed/hex/JSON), пропуская нелокальные строки. Для доказательной
   выгрузки под таблицы пригодится
   формирующая CSV (`input,status,format,domain_kind,…`) с пометкой
   Local‑селекторов, канонических кодировок и ошибок парсинга.
3. **Добавить записи в manifest.** Подготовьте запись `tombstone` (и
   последующую `global_domain` при миграции в глобальный реестр) и
   провалидируйте manifest с помощью
   `cargo xtask address-manifest verify` до запроса подписей.
4. **Проверить и опубликовать.** Следуйте чек‑листу из runbook’а (hash’и,
   Sigstore, монотонность `sequence`) до зеркалирования bundle на SoraFS.
   что продакшен‑кластеры сразу начинают требовать канонические IH58/
   compressed‑литералы после публикации bundle.
5. **Мониторить и при необходимости откатить.** Держите панели Local‑8 на
   нуле 30 дней; при регрессиях переиздайте предыдущий manifest‑bundle и,
   только в непроизводственных окружениях, временно выставьте
