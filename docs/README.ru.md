# Документация Iroha

Обзор на японском языке доступен в [`README.ja.md`](./README.ja.md).

В этом репозитории поддерживаются две линии релизов на единой кодовой базе:
**Iroha 2** (самостоятельно развёртываемые сети) и **Iroha 3 / SORA Nexus**
(единый глобальный реестр Nexus). Обе используют одну и ту же виртуальную
машину Iroha (IVM) и компилятор/инструменты Kotodama, поэтому контракты и
байткод остаются переносимыми между разными вариантами развёртывания. Если не
указано иначе, документация относится к обеим линиям.

В [основной документации Iroha](https://docs.iroha.tech/) вы найдёте:

- [Руководство по началу работы](https://docs.iroha.tech/get-started/)
- [Учебные материалы по SDK](https://docs.iroha.tech/guide/tutorials/) для Rust, Python, JavaScript и Java/Kotlin
- [Справочник по API](https://docs.iroha.tech/reference/torii-endpoints.html)

Whitepaper’ы и спецификации по релизам:

- [Whitepaper Iroha 2](./source/iroha_2_whitepaper.md) — спецификация для самостоятельных сетей.
- [Whitepaper Iroha 3 (SORA Nexus)](./source/iroha_3_whitepaper.md) — дизайн многолинейного (multi‑lane) и data‑space‑подхода Nexus.

- [ZK‑конверты (Norito)](./source/zk_envelopes.md) — нативные конверты Norito на основе IPA/STARK и требования к верификатору.

## Norito (кодек сериализации)

Norito — это кодек сериализации для всего workspace. Библиотека
`parity-scale-codec` (SCALE) не используется. Если в документации или
бенчмарках упоминается SCALE, это делается лишь для контекста; все боевые
пути используют Norito. API `norito::codec::{Encode, Decode}` предоставляют
Norito‑payload без заголовка («bare») для удобного хэширования и эффективной
передачи по сети — это Norito, а не SCALE.

Текущее состояние:

- Детерминированное кодирование/декодирование с фиксированным заголовком
  (magic, версия, 16‑байтовая схема, тип компрессии, длина, CRC64, флаги).
- Контрольная сумма CRC64-XZ с выбором ускорения во время выполнения:
  - PCLMULQDQ на x86_64 (умножение без переноса) и Barrett‑редукция поверх блоков по 32 байта.
  - PMULL на aarch64 с тем же алгоритмом свёртки.
  - Реализации slicing‑by‑8 и побитовые варианты для переносимости.
- Подсказки по длине кодируемых структур, реализованные в derives и базовых
  типах, уменьшают количество выделений памяти.
- Увеличенные буферы потокового чтения (64 КиБ) и инкрементальное обновление CRC при декодировании.
- Необязательная компрессия zstd; ускорение на GPU включается через feature flag и остаётся детерминированным.
- Адаптивный выбор пути: `norito::to_bytes_auto(&T)` выбирает между отсутствием
  сжатия, zstd на CPU и zstd на GPU (если поддерживается) в зависимости от
  размера payload’а и кэшированных характеристик железа. Этот выбор влияет
  только на производительность и байт `compression` в заголовке; семантика
  данных не меняется.

См. `crates/norito/README.md` для подробностей о тестах на паритет,
бенчмарках и примерах использования.

Примечание. Документация по некоторым подсистемам (например, ускорение IVM и
ZK‑цепочки) ещё развивается. В таких файлах явно указано, какой объём работ
остаётся и в каком направлении движется дизайн.

Особенности кодирования ответа `/status`

- Ториевый endpoint `/status` по умолчанию использует Norito‑payload без
  заголовка («bare») для компактности. Клиенты должны в первую очередь пытаться
  декодировать Norito.
- Сервер может возвращать JSON по запросу; в этом случае клиенты переходят на
  JSON, если заголовок `content-type` равен `application/json`.
- Каноничный формат на проводе — Norito, а не SCALE. Для bare‑варианта
  используются API `norito::codec::{Encode,Decode}`.
