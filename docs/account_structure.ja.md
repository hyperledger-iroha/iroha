# アカウント構造 RFC

**ステータス:** ドラフト  
**想定読者:** データモデル、Torii、Nexus、ウォレット、ガバナンス各チーム  
**関連 Issue:** TBD

## 概要

このドキュメントは、アカウントアドレス体系を刷新し、以下を実現することを提案します。

- チェックスム付きで人間が扱いやすい **Iroha Bech32m アドレス (IH-B32)** を導入し、チェーン判別子を署名者と結び付けつつ、決定的で相互運用しやすい文字列表現を提供する。
- グローバルに一意なドメイン識別子をレジストリで管理し、Nexus を介したクロスチェーンルーティングに利用できるようにする。
- ウォレット、API、コントラクトを新形式へ移行する間も既存の `alias@domain` エイリアスが機能する互換レイヤーを維持する。
- `AccountAddress` 型で IH58（推奨）／Sora 圧縮（`snx1…`、次善かつ Sora 専用）／正規 Hex の各コーデックを提供し、Torii は `AccountAddress::parse_any` で入力を正規バイト列に揃える。

## 動機

現在、ウォレットやオフチェーンツールは素の `alias@domain` 文字列に依存しています。これには 2 つの重大な欠点があります。

1. **ネットワークが結び付かない。** 文字列にはチェックスムもチェーン接頭辞も無いため、ユーザーは誤ったネットワークのアドレスを貼り付けても即座に気付けません。最終的にはトランザクションが拒否される（チェーン不一致）か、さらに悪ければ目的のチェーンに同じアカウントが存在する場合に意図しない送金が成功してしまいます。
2. **ドメインの衝突。** ドメインは名前空間でしかなく、各チェーンで再利用できます。サービス連携（カストディアン、ブリッジ、クロスチェーンフロー）は `finance` がチェーン A とチェーン B で無関係なため脆くなります。

コピー＆ペーストの誤りを防ぎ、人間にとって扱いやすいアドレス形式と、ドメイン名から権限を持つチェーンへの決定的な対応付けが必要です。

## 目標

- Iroha アカウント向けに IH58 に着想を得た Bech32m アドレス封筒を設計し、CAIP-10 の正規文字列表現を提供する。
- 設定されたチェーン判別子を各アドレスに直接エンコードし、そのガバナンス／レジストリプロセスを定義する。
- 既存デプロイを壊さずにグローバルドメインレジストリを導入する手順を記載し、正規化／なりすまし防止ルールを定める。
- 互換性要件、移行ステップ、および未解決事項を文書化する。

## 非目標

- クロスチェーンでの資産転送を実装すること。本提案のルーティング層はターゲットチェーンを返すのみです。
- 内部の `AccountId` 構造を変更すること（引き続き `DomainId + PublicKey`）。
- グローバルドメイン発行のガバナンスを確定すること。この RFC はデータモデルと伝送プリミティブに焦点を当てます。

## 背景

### 現行のアカウント識別子

```
AccountId {
    domain: DomainId,   // Name（ASCII に近い文字列）をラップ
    signatory: PublicKey // マルチハッシュ文字列（例: ed0120...）
}

表示／パース: "<signatory multihash>@<domain name>"
```

`ChainId` は `AccountId` の外側にあります。ノードはトランザクション受付時に `ChainId` を設定と照合し（`AcceptTransactionFail::ChainIdMismatch`）、外部チェーンのトランザクションを拒否しますが、アカウント文字列自体にはネットワークのヒントがありません。

### ドメイン識別子

`DomainId` は正規化済み文字列である `Name` をラップし、ローカルチェーンに限定されます。各チェーンは `wonderland` や `finance` を独立して登録できます。

### Nexus の文脈

Nexus はコンポーネント間の調整（レーン／データスペース）を担当しますが、現状クロスチェーンのドメインルーティングを扱っていません。

## 提案する設計

### 1. 決定的なチェーン判別子

`iroha_config::parameters::actual::Common` を次のように拡張します。

```rust
pub struct Common {
    pub chain: ChainId,
    pub chain_discriminant: u16, // 新規。グローバルに調整
    // ... 現行フィールド
}
```

- **制約:**
  - 稼働中の各ネットワークで一意。署名付き公開レジストリが管理し、予約レンジを明示します（例: `0x0000–0x0FFF` はテスト／開発、`0x1000–0x7FFF` はコミュニティ配分、`0x8000–0xFFEF` はガバナンス承認、`0xFFF0–0xFFFF` を予約）。
  - 稼働中のチェーンでは不変。変更にはハードフォークとレジストリ更新が必要です。
- **ガバナンスとレジストリ:** マルチシグのガバナンスセットが署名付き JSON（IPFS にピン留め可能）を保持し、判別子を人間向けエイリアスと CAIP-2 識別子に対応付けます。クライアントはレジストリを取得・キャッシュし、チェーンメタデータの検証と表示に利用します。
- **利用:** 状態受付、Torii、SDK、ウォレット API のすべてで判別子を伝播させ、各コンポーネントが埋め込み／検証できるようにします。CAIP-2 の一部として公開（例: `iroha:0x1234`）。

### 2. Iroha Bech32m (IH-B32) アドレスエンコード

IH58 に着想を得つつ、暗号学的 ID に焦点を当ててコンパクトさを維持した Bech32m ベースの新しいアドレス形式を導入します。

```
ヒューマンリーダブルプレフィックス (HRP): "ih"（小文字、固定）
データペイロード (Bech32m ワード、小文字のみ):
    version        : 5 ビット（初期値 0）
    chain_id       : 16 ビット（chain_discriminant）
    signatory      : マルチハッシュバイト列（可変長。アルゴリズムを含む）
チェックサム: Bech32m が 6 文字を自動付与
```

- **バージョニング:** 将来のアカウント表現（例: マルチシグ、ラベル付きアカウント）向けに 0 以外のバージョンを予約。
- **エンコードヘルパー:** `AccountId::to_ih_b32(&Config)` と `AccountId::from_ih_b32(&Config, &str)` を `iroha_data_model` に追加。
- **CLI／SDK の挙動:** 表示と入力の既定を IH-B32 に統一。公開 API は IH-B32 文字列を期待する。
- **エラーハンドリング:** チェーン判別子がローカル設定と一致しない場合は即座にデコード失敗とし、ウォレットがクロスチェーン送信を事前に防止できる。
- **長さ:** ドメインをペイロード外に分離することで Ed25519 アドレスは約 54 文字（チェーン + 34 バイトのマルチハッシュ）。従来案（80 文字超）より可読性が向上。
- **大文字小文字:** IH-B32 文字列は Bech32m に従い小文字のみ。混在大小はデコード時に拒否。

#### 2.1 相互運用向けの文字列表現

- **チェーン別 alias 形式:** ログや人手入力向けに `ih:<chain-alias>:<alias@domain>`。ウォレットは接頭辞を解析し、埋め込まれたチェーンを検証して不一致を拒否する。
- **CAIP-10 形式:** チェーン非依存の統合向けに `iroha:<caip2-chain-id>:<ih-b32-addr>`。ツールは判別子レジストリから CAIP-2 部分を導出できる。
- **マシン支援:** Rust、TypeScript/JavaScript、Python、Kotlin 向けに IH-B32 と CAIP-10 をカバーするコーデックを公開。

#### 2.2 決定的な IH58 エイリアス

- **プレフィックス対応:** `chain_discriminant` を IH58 ネットワークプレフィックスとして再利用。割り当ての正本は [`address_prefix_registry.md`](source/references/address_prefix_registry.md) と対応する JSON レジストリで公開されており、各 SDK は衝突を避けるために必ず同期させる。
- **アカウント素材:** マルチハッシュから生のキー部分を抽出。ペイロードがちょうど 32 バイトでない場合（例: 圧縮 secp256k1 キーや将来の方式）は `AccountId32 = blake2b-256(multihash_bytes)` を導出し、IH58 の幅に揃える。これによりアルゴリズム間で決定的なエイリアスを維持。
- **エンコード:** 対応付けたプレフィックスと `AccountId32` を用い、標準の IH58 チェックスム手順で互換 Base58Check 文字列を生成。コーデックライブラリと共にマッピングロジックも公開し、連携先が再利用できるようにする。

#### 2.3 規範的テストベクター

- **Ed25519、チェーン `0x1234`、ドメイン `wonderland`**  
  入力マルチハッシュ: `ed0120CE7FA46C9DCE7EA4B125E2E36BDB63EA33073E7590AC92816AE1E861B7048B03`  
  IH-B32: `ih1qzg6w6qfqeel6gmyaeel2fvf9ut3khkmragesw0n4jzkf9qt2u85xrdcy3vpsl337y0`  
  CAIP-10: `iroha:0x1234:ih1qzg6w6qfqeel6gmyaeel2fvf9ut3khkmragesw0n4jzkf9qt2u85xrdcy3vpsl337y0`  
  チェーン alias: `ih:iroha-main:ed0120CE7FA46C9DCE7EA4B125E2E36BDB63EA33073E7590AC92816AE1E861B7048B03@wonderland`  
  IH58 エイリアス: プレフィックス `0x1234`, アカウントバイト列 `blake2b-256(multihash)`。
- **secp256k1、チェーン `0x0001`、ドメイン `treasury`**  
  入力マルチハッシュ: `e701210279BE667EF9DCBBAC55A06295CE870B07029BFCDB2DCE28D959F2815B16F81798`  
  IH-B32: `ih1qqqq7wqfpqfumuen7l8wthtz45p3ftn58pvrs9xlumvkuu2xet8egzkcklqtestc32u8`  
  CAIP-10: `iroha:0x0001:ih1qqqq7wqfpqfumuen7l8wthtz45p3ftn58pvrs9xlumvkuu2xet8egzkcklqtestc32u8`  
  チェーン alias: `ih:iroha-dev:e701210279BE667EF9DCBBAC55A06295CE870B07029BFCDB2DCE28D959F2815B16F81798`  
  IH58 エイリアス: プレフィックス `0x0001`, アカウントバイト列 `blake2b-256(multihash)`。
- **失敗例:** 1 つ目のアドレスを `chain_discriminant = 0x0001` のノードでデコードすると、ドメインルーティングを試みる前に必ず `ChainMismatch` を報告する必要があります。文字を改ざんすると Bech32m チェックスムが失敗する。

### 3. グローバル一意ドメインと正規化

Norm v1 の正規化手順については `docs/source/references/address_norm_v1.md`
（英語）を参照してください。

`DomainId` をタグ付きタプルに再定義します。

```
DomainId {
    name: Name,
    authority: GlobalDomainAuthority, // 新 enum
}

enum GlobalDomainAuthority {
    LocalChain,                  // 既定。ローカルチェーンが管理
    External { chain_discriminant: u16 },
}
```

`LocalChain` は従来どおりローカルチェーンで管理されるドメインを表します。グローバルレジストリ経由で登録されたドメインでは、所有チェーンの判別子を保存します。表示／パースは当面変更しませんが、拡張された構造によりルーティング判断が行えます。

#### 3.1 正規化となりすまし防止

- **文字集合:** `Name` は既定で ASCII（英数字、`-`、`_`）を維持。国際化が必要になった場合は **UTS-46** を登録／検索に採用し、**UTS-39** の類似判定と明確なスクリプトポリシーを適用する。
- **大文字小文字:** 小文字での保存を強制し、混在大小入力を拒否。
- **長さ上限:** ラベル長の最大値（例: 63 文字）とドメイン全体の最大値（例: 255 文字）を定義し、DNS ライクな期待値に合わせる。
- **移行:** 既存ドメインは `LocalChain` のまま有効。新しいグローバルドメインは公開前に正規化となりすましチェックを必須とする。

### 4. Nexus ドメインレジストリとルーティング

- **レジストリスキーマ:** Nexus は署名付きマップ `DomainName -> ChainRecord` を保持し、`ChainRecord` はチェーン判別子・任意のメタデータ（RPC エンドポイント）・権限証明（例: ガバナンスマルチシグ）を含む。
- **同期機構:**
  - チェーンは署名付きドメイン要求を Nexus に送信（ジェネシス時またはガバナンス命令経由）。
  - Nexus は署名付き JSON（必要に応じて Merkle root 付き）を定期的に公開し、HTTPS とコンテンツアドレス型ストレージ（例: IPFS）で配布。クライアントは最新マニフェストを取得して署名検証を行う。
- **ルックアップフロー:**
  - Torii が `DomainId` を参照するトランザクションを受信。
  - ドメインがローカルで未知の場合、キャッシュされた Nexus マニフェストを参照。
  - マニフェストが外部チェーンを示す場合、Torii は決定的な `ForeignDomain` エラーとリモートチェーン情報を返して拒否。
  - Nexus にドメインが存在しない場合、Torii は `UnknownDomain` を返す。
- **信頼アンカーとローテーション:** ガバナンス鍵がマニフェストへ署名し、ローテーションや失効は新しいマニフェスト項目として公表。クライアントはマニフェストの TTL（例: 24 時間）を強制し、期間を超過したデータの参照を拒否する。
- **障害モード:** マニフェスト取得に失敗した場合、Torii は TTL 内ならキャッシュを使用し、TTL を超過すると `RegistryUnavailable` を返してクロスドメインルーティングを拒否し、不整合を回避。

自動化サポート: `cargo xtask address-manifest verify --bundle <最新ディレクトリ> --previous <直前ディレクトリ>`
を実行すると、[address manifest runbook](source/runbooks/address_manifest_ops.md) に記載された
チェックサム・エントリスキーマ・`previous_digest` 検証が一括で確認できます。公開前にコマンドの出力を
変更チケットへ添付し、`sequence` 増分と `previous_digest` の連鎖が検証済みであることを示してください。

### 5. ウォレットと API の操作性

- **表示の既定:** ウォレットは IH-B32 アドレス（短く、チェックスム付き）を表示し、レジストリから取得したドメインをラベルとして併記する。ドメインは変更され得る説明的メタデータであると明示し、IH-B32 が安定したアドレスであることを示す。
- **厳格モード:** Torii・SDK・ウォレットはエンドツーエンドで IH-B32 入力のみを求める設定フラグを提供し、カストディアンがコンプライアンスポリシーを予行演習できるようにする。
- **エラー防止:** ウォレットは IH-B32 と CAIP-10 の接頭辞を解析。チェーン不一致は行動可能な診断を伴う致命的エラーとして扱う。
- **コーデックライブラリ:** 公式の Rust、TypeScript/JavaScript、Python、Kotlin ライブラリで IH-B32 のエンコード／デコードと CAIP-10 変換を提供し、実装のばらつきを防ぐ。

#### アクセシビリティと安全な共有ガイドライン

- **安全な共有フロー:** アドレスを表示・コピーする画面では IH-B32 形式を既定とし、同じペイロードから生成した完全な文字列と QR コードを提示する「共有」アクションを隣接させる。画面サイズの制約で文字列を省略する場合は、先頭と末尾を残したまま明確な省略記号を挿入し、クリップボードコピーからは常に完全なアドレスを取得できるようにして誤切りを防ぐ。
- **IME ガード:** IME／かな入力などの変換済み文字を入力に混入させない。ASCII のみを受け付け、全角や仮名を検出した場合はインラインで警告を出す。結合文字を除去してから検証するプレーンテキスト貼り付け欄を用意し、日本語・中国語利用者が IME を一時的に無効化しても入力を失わないようにする。
- **スクリーンリーダー対応:** `aria-label`／`aria-describedby` などの視覚的に隠れたラベルを用意し、HRP を明示しながら 4～8 文字単位に分割して読み上げるよう調整する。コピー／共有成功は polite ライブリージョンで通知し、QR プレビューには「チェーン 0x1234 の <alias> 向け IH-B32 アドレス」など意味のある代替テキストを設定する。
- **Sora 限定の圧縮表示:** `snx1…` 形式は常に「Sora 専用」と明示し、コピー時は確認ダイアログなどの明示的な操作を挟む。チェーン判別子が Sora Nexus 以外の場合は SDK やウォレットが圧縮表示を拒否し、ネットワーク間送金では IH-B32 を利用するよう利用者へ案内することで誤送金を防ぐ。

### 6. 互換性戦略

1. **フェーズ 0 – 準備**
   - エンコード／デコードヘルパーと設定上の判別子を追加しつつ、既定表示は変更しない。SDK をオプトイン更新。
2. **フェーズ 1 – 正式ローンチ**
   - Torii はリクエストで IH-B32 を受け取り、応答も正規形式で返す。
   - API カウンタを計測し、操作者が IH-B32 の採用状況を確認できるようにする。
3. **フェーズ 2 – 強制**
4. **フェーズ 3 – 既定**
   - 移行用エイリアスを廃止し、すべての境界で IH-B32 を必須にする。

移行期間を通じて、チェーンがグローバルレジストリに参加しない限りドメイン登録はローカルのままです。

## 実装チェックリスト

- **IH-B32 封筒:** HRP `ih`、バージョン `0`、ビッグエンディアン 16 ビット `chain_discriminant`、マルチハッシュペイロード、Bech32m の 5 ビットパディングと小文字出力。
- **レジストリ契約:** `{discriminant, ih58_prefix, chain_alias, endpoints}` を含む署名付き JSON（必要に応じて Merkle root）を公開し、TTL 24 時間と鍵ローテーションを付与。
- **ドメインポリシー:** 現時点では ASCII の `Name`。国際化を行う場合は UTS-46 の正規化と UTS-39 の類似判定を適用。ラベル長最大 63、ドメイン全体最大 255 を強制。
- **文字列ヘルパー:** Rust／TypeScript/JavaScript／Python／Kotlin で IH-B32 ↔ CAIP-10 ↔ IH58 のコーデックを提供し、共通テストベクターを共有。
- **テスト:** エンコード／デコードの往復、誤チェーン失敗、マニフェスト参照の単体テストを追加。Torii と SDK に IH-B32 フローのエンドツーエンド統合テストを追加。

## 検討した代替案

- **Base58Check（Bitcoin 形式）。** チェックスムは単純だが Bech32m より誤り検出能力が弱く、将来拡張にも不向き。
- **チェーン名をドメイン文字列に埋め込む（例: `finance@chain`）。** 互換性を壊し、チェックスムも無いため検証が難しい。
- **アドレスを変えず Nexus ルーティングのみに頼る。** ユーザーが曖昧な文字列をコピーする問題が残るため、アドレス自体にコンテキストを持たせたい。
- **IH58 をそのまま採用。** 実績あるレジストリがある一方、Base58Check は QR と相性が悪い。提案する方式はコンパクトな Bech32m を維持しつつ、IH58 や CAIP-10 への決定的マッピングを公開する。

## 未解決事項

- `u16` 判別子と予約レンジで長期需要を満たせるか確認し、不足する場合は可変長エンコード付き `u32` を検討する。
- レジストリ更新のマルチシグガバナンス手順を確定し、失効や回収をどう扱うか決める。
- マニフェスト署名方式（例: Ed25519 マルチシグ）と Nexus 配布時の輸送セキュリティ（HTTPS ピニング、IPFS ハッシュ形式）を定義する。
- 移行時のドメインエイリアス／リダイレクトをサポートするか、および決定性を損なわずに提示する方法を決める。
- Kotodama／IVM コントラクトが IH-B32 ヘルパー（`to_address()`、`parse_address()`）へアクセスする方法、およびオンチェーン保存で IH-B32 と CAIP-10 のどちらを優先すべきかを決める。
- Iroha チェーンを外部レジストリ（IH58 レジストリや CAIP 名前空間ディレクトリなど）へ登録し、エコシステム整合性を高めるか検討する。

## 次のステップ

1. フィーチャーフラグにより `iroha_data_model` で IH-B32 エンコードを試作する。
2. 設定スキーマへ `chain_discriminant` を追加し、既存のテスト／開発環境向けに妥当な既定値を導出する。**（対応済み: `iroha_config` の `[common]` に `chain_discriminant` が追加され、既定値は `0x02F1`。ネットワークごとに上書きできます。）**
3. Nexus レジストリスキーマとマニフェスト公開の PoC を作成する。
4. ウォレット提供者やカストディアンからヒューマンファクター（HRP 名称、表示フォーマット）についてフィードバックを収集する。
5. 実装方針が固まり次第、`docs/source/data_model.md` や Torii API ドキュメントを更新する。
6. 公式コーデックライブラリ（Rust/TS/Python/Kotlin）を出荷し、成功／失敗ケースを網羅する規範的テストベクターを添付する。

### 7. AccountAddress エンベロープのバイナリ構造 (ADDR-1a)

#### 7.1 ヘッダバイト

正規のエンベロープは `header · domain selector · controller` の順に並びます。先頭バイトは後続データの解釈ルールを伝えます。

```
bit index:   7        5 4      3 2      1 0
             ┌─────────┬────────┬────────┬────┐
payload bit: │version  │ class  │  norm  │ext │
             └─────────┴────────┴────────┴────┘
```

| ビット | フィールド | 許可値 | 非対応値の処理 |
|--------|-----------|--------|----------------|
| 7‑5 | `addr_version` | `0`（v1）。`1‑7` は予約済み。 | `AccountAddressError::InvalidHeaderVersion` |
| 4‑3 | `addr_class` | `0` = 単一キー, `1` = マルチシグ | `AccountAddressError::UnknownAddressClass` |
| 2‑1 | `norm_version` | `1`（Norm v1）。その他は予約。 | `AccountAddressError::InvalidNormVersion` |
| 0   | `ext_flag` | 常に `0` | `AccountAddressError::UnexpectedExtensionFlag` |

#### 7.2 ドメインセレクタ

セレクタはヘッダの直後に続き、タグとタグ依存のペイロードを持ちます。

```
domain selector
┌──────────┬──────────────────────────────────────────────┐
│ tag (u8) │ payload (selector kind に応じて可変)        │
└──────────┴──────────────────────────────────────────────┘
```

- `0x00`: 既定ドメイン。ペイロードなし。
- `0x01`: ローカルドメインダイジェスト。`blake2s_mac("SORA-LOCAL-K:v1", label)[0..12]` で得られる 12 バイトを格納。
- `0x02`: グローバルレジストリエントリ。ビッグエンディアンの `registry_id:u32`。

不明なタグは `AccountAddressError::UnknownDomainTag` で失敗し、期待ドメインと一致しない場合は `AccountAddressError::DomainMismatch` を返します。

#### 7.3 正規的な例

- **既定ドメイン（タグ `0x00`）**  
  `0x0200000120641297079357229f295938a4b5a333de35069bf47b9d0704e45805713d13c201`
- **`treasury` のローカルセレクタ（タグ `0x01`）**  
  `0x0201b18fe9c1abbac45b3e38fc5d0001203b77a042f1de02f6d5f418f36a20fd68c8329fe3bbfbecd26a2d72878cd827f8`
- **`registry_id = 42` のグローバルエントリ（タグ `0x02`）**  
  `0x02020000002a000120641297079357229f295938a4b5a333de35069bf47b9d0704e45805713d13c201`

各例とも `0x02` がヘッダ（v0／単一キー／Norm v1／拡張フラグ 0）、続いてセレクタのタグとペイロード、最後にコントローラタグ `0x00`（`curve_id = 0x01`、長さ 32 バイトの Ed25519 公開鍵）という順序です。
