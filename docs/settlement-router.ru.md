# Детерминированный роутер расчётов (NX-3)

**Статус:** Планируется (NX-3)  
**Владельцы:** Economics WG / Core Ledger WG / Treasury / SRE  
**Область:** Реализует унифицированную политику конверсии в XOR и буферизации, описанную в
roadmap‑задаче «Unified lane settlement & XOR conversion (NX-3)».

Этот документ фиксирует алгоритм, телеметрию и операционные контракты унифицированного
роутера расчётов. Он дополняет обзор комиссий в `docs/source/nexus_fee_model.md`,
объясняя, как роутер вычисляет дебеты в XOR, взаимодействует с каноническим DEX
AMM/CLMM (или управляемым RFQ‑fallback’ом) и формирует детерминированные квитанции для
каждого lane.

## Цели

   потоки AMX cross‑DS—используют один и тот же pipeline расчёта, поэтому ночная
   сверка строится на единой модели.
2. **Детерминированная конверсия.** Обязательства в XOR выводятся из on‑chain‑параметров
   (TWAP, надбавка за волатильность, уровень ликвидности) и округляются вверх до
   micro‑XOR, чтобы каждый узел формировал одну и ту же квитанцию.
3. **Защитные буферы.** Каждый dataspace/lane поддерживает XOR‑буфер на уровне P95 за 72 ч.
   Роутер сначала списывает из буфера, по мере падения уровней включает режимы
   throttle/XOR‑only и публикует метрики, позволяющие операторам пополнять буферы
   заранее.
4. **Управляемая ликвидность.** Конверсии по возможности проходят через канонический
   публичный DEX AMM/CLMM; управляемые RFQ/swap‑линии подключаются при падении глубины
   или срабатывании «рубильников» (circuit breakers).

## Архитектура

| Компонент | Расположение | Ответственность |
|----------|--------------|-----------------|
| `crates/settlement_router/` | Crate роутера (новый) | Расчёт shadow price, haircuts, учёт буфера, построение расписания, трекинг P&L‑экспозиции. |
| `crates/oracle_adapter/` | Прослойка для оракла | Поддерживает TWAP‑окно (по умолчанию 60 с) и скользящую волатильность, выдаёт circuit‑price при устаревании фидов. |

## Pipeline конверсии

1. **Агрегация квитанций**
   - Каждая транзакция/нога AMX эмитирует `LaneSettlementReceipt` с локальной суммой в
     micro, обязательством в XOR, tier’ом haircut, реализованным запасом
     (`xor_variance_micro`) и метаданными вызвавшей стороны (как описано в
     `docs/source/nexus_fee_model.md`).
   - Роутер группирует квитанции по `(lane, dataspace)` в ходе исполнения блока.

2. **Shadow price (`shadow_price.rs`)**
   - Берёт TWAP за 60 с (`twap_local_per_xor`) и выборку волатильности за 30 с.
   - Считает epsilon: базовые 25 bp + (bucket волатильности × 25 bp, но не более 75 bp).
   - Сохраняет полученный `volatility_class` (`stable`, `elevated`, `dislocated`) в
     `LaneSwapMetadata`, чтобы аудитор и мониторинг могли сопоставить дополнительный
     запас с режимом оракула, зафиксированным в `pipeline.gas.units_per_gas`.
   - Вычисляет `xor_due = ceil(local_micro / twap × (1 + epsilon))`.
   - Записывает кортеж `(twap, epsilon, tier)` в swap‑метаданные для аудита.

3. **Haircut + вариация (`haircut.rs`)**
   - Применяет haircut по профилю ликвидности (tier1=0 bp, tier2=25 bp, tier3=75 bp).
   - Фиксирует `total_xor_after_haircut` и `total_xor_variance` (разницу между
     `xor_due` и суммой после haircut), чтобы казначейство видело, какой запас
     безопасности потребляется каждым tier’ом.

4. **Списание из буфера (`buffer.rs`)**
   - Для каждого буфера lane/dataspace задаётся Bmin = XOR‑расход P95 за 72 ч.
   - Состояния: **Normal** (≥75 %), **Alert** (<75 %), **Throttle** (<25 %),
     **XOR‑only** (<10 %), **Halt** (<2 % или устаревший оракул).
   - Роутер списывает буфер по каждой квитанции; при превышении порогов публикует
     детерминированную телеметрию и вводит ограничения (например, делит частоту включения
     транзакций на два при уровне <25 %).
   - Метаданные lane должны объявлять казначейский резерв через:
     - `metadata.settlement.buffer_account` – ID аккаунта с резервом
       (например, `buffer::cbdc_treasury`);
     - `metadata.settlement.buffer_asset` – Asset, с которого списывается headroom
       (обычно `xor#sora`);
     - `metadata.settlement.buffer_capacity_micro` – вместимость в micro‑XOR
       (десятичная строка).
   - Телеметрия экспонирует `iroha_settlement_buffer_xor` (оставшийся headroom),
     `iroha_settlement_buffer_capacity_xor` и `iroha_settlement_buffer_state`
     (дискретное состояние).

5. **Планирование свопов (`schedule.rs`)**
   - Группирует квитанции по локальному asset’у и tier’у ликвидности, минимизируя число
     свопов.
   - Соблюдает лимиты размера/количества для каждого AMM/CLMM‑ордера.
   - Строит детерминированный `LaneSwapSchedule`, который описывает:
     - агрегированный локальный asset;
     - целевой объём в XOR;
     - выбранный DEX/tier;
     - применённый haircut;
     - затронутые lane/dataspace.

6. **Расчёт и доказательства блока**
   - Планировщик отправляет ордера на DEX‑слой. Результат каждого свопа кодируется в
     `LaneSwapMetadata` и прикрепляется к `LaneBlockCommitment`.
   - Аудитор может восстановить конверсию XOR‑local из:
     - TWAP и волатильности;
     - tier’а ликвидности;
     - применённого haircut;
     - суммарной XOR‑вариации.

## Телеметрия и алерты

Минимальный набор метрик:

- `iroha_settlement_buffer_xor{dataspace,lane}` – оставшийся headroom в XOR;
- `iroha_settlement_buffer_capacity_xor{dataspace,lane}` – настроенная ёмкость Bmin;
- `iroha_settlement_buffer_state{dataspace,lane}` – дискретное состояние (`normal`,
  `alert`, `throttle`, `xor_only`, `halt`);
- `iroha_settlement_pnl_xor{dataspace,lane}` – накопленный P&L в XOR;
- `iroha_settlement_haircut_bp{dataspace,lane,tier}` – фактически применённый haircut.

Дашборды и оповещения:

- Поднимать алерты при переходе состояния в `alert` (<75 %) и ниже.
- Строить панели, показывающие headroom по lane/dataspace и дневное потребление.
- Подсвечивать lanes, где накопленная вариация (`total_xor_variance`) выходит за ожидаемый
  диапазон.

## Стратегия развёртывания

Рекомендуется включать роутер поэтапно:

1. **Shadow‑режим.** Включить расчёты роутера + телеметрию, но оставить
2. **Включение дебетов.** Включить `settlement_debit=true` для низкорисковых lanes
   (профиль C) и проверить сверки.
3. **Конверсии включены.** Активировать TWAMM‑слайсы + AMM‑свопы, когда телеметрия
   показывает стабильные буферы; держать RFQ‑fallback в standby.
4. **Интеграция с AMX.** Убедиться, что AMX‑lanes пробрасывают квитанции и по ошибке
   роутера выдают `SETTLEMENT_ROUTER_UNAVAILABLE` в SDK для повторов.
   будущие merge’и проходили только при соблюдении целевых метрик по телеметрии расчётов.

Следование этому документу обеспечивает, что задача NX‑3 в roadmap’е даёт
детерминированное поведение, артефакты, пригодные для аудита, и понятные инструкции для
разработчиков и операторов.
