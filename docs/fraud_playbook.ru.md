# Плейбук по управлению мошенничеством

Этот документ описывает базовую инфраструктуру, необходимую для PSP‑стека
борьбы с мошенничеством, пока полнофункциональные микросервисы и SDK находятся
в стадии активной разработки. В нём фиксируются ожидания по аналитике,
рабочим процессам аудиторов и процедурам аварийного режима, чтобы будущие
реализации могли безопасно интегрироваться с реестром.

## Обзор сервисов

1. **API‑шлюз** – принимает синхронные payload’ы `RiskQuery`, пересылает их в
   сервис агрегирования признаков и возвращает ответы `FraudAssessment` обратно
   в потоки реестра. Требуется высокая доступность (active‑active); используйте
   региональные пары с детерминированным хешированием, чтобы избежать перекоса
   нагрузки.
2. **Агрегирование признаков** – формирует вектор признаков для scoring’а.
   Выдаёт только хэши `FeatureInput`; чувствительные payload’ы остаются
   off‑chain. Сервис наблюдаемости должен публиковать гистограммы латентности,
   gauges глубины очередей и счётчики повторных запросов (replay) для каждого
   tenant’а.
3. **Риск‑движок** – оценивает правила/модели и формирует детерминированные
   результаты `FraudAssessment`. Необходимо обеспечить стабильный порядок
   выполнения правил и писать аудиторские логи по каждому ID оценки.

## Аналитика и вывод моделей в прод

- **Обнаружение аномалий**: поддерживать стриминговый job, который отмечает
  отклонения в долях решений по каждому tenant’у. Направлять алерты на
  дашборд по управлению и сохранять сводки для ежеквартальных ревью.
- **Графовый анализ**: выполнять ночные обходы графа по реляционным экспортам
  для выявления кластеров сговоров. Экспортировать результаты в портал
  управления через `GovernanceExport` с ссылками на подтверждающие доказательства.
- **Инжест обратной связи**: агрегировать результаты ручной проверки и
  chargeback‑отчёты. Преобразовывать их в дельты признаков и включать в
  тренировочные выборки. Публиковать метрики статуса ingestion’а, чтобы
  команда риска могла видеть «застрявшие» фиды.
- **Пайплайн вывода моделей**: автоматизировать оценку кандидатов (offline‑
  метрики, canary‑scoring, готовность к откату). При выводе в прод модель
  должна сопровождаться подписанным набором примеров `FraudAssessment` и
  обновлением поля `model_version` в `GovernanceExport`.

## Процесс работы аудитора

1. Снять snapshot последнего `GovernanceExport` и проверить, что
   `policy_digest` совпадает с manifest’ом, предоставленным командой риска.
2. Убедиться, что агрегаты по правилам согласуются с итоговыми числами
   решений на стороне реестра в выбранном окне наблюдения.
3. Просмотреть отчёты по обнаружению аномалий и графовому анализу на предмет
   нерешённых проблем. Задокументировать эскалации и ожидаемых владельцев
   по устранению.
4. Подписать и заархивировать чек‑лист ревью. Сохранить артефакты,
   закодированные в Norito, в портале управления для последующего
   воспроизведения.

## Playbook’и аварийного режима

- **Отказ движка**: если риск‑движок недоступен более 60 секунд, шлюз должен
  перейти в режим только review, возвращая `AssessmentDecision::Review` для
  всех запросов и оповещая операторов.
- **Пробелы в телеметрии**: если метрики или трассы перестают поступать (нет
  данных 5 минут), остановить автоматическое продвижение моделей и уведомить
  инженера дежурной смены.
- **Регрессия модели**: если пост‑деплой feedback показывает рост потерь от
  мошенничества, откатиться к предыдущему подписанному bundle’у модели и
  обновить roadmap с корректирующими действиями.

## Соглашения о обмене данными

- Поддерживать юрисдикционно‑специфичные приложения, описывающие хранение
  данных, шифрование и SLA по уведомлению об инцидентах. Партнёры должны
  подписать соответствующее приложение до начала получения экспортов
  `FraudAssessment`.
- Документировать практики минимизации данных для каждой интеграции
  (например, хеширование идентификаторов аккаунтов, усечение номеров карт).
- Обновлять соглашения ежегодно или при изменении регуляторных требований.

## Учения Red‑team

- Учения проводятся ежеквартально. Следующая сессия запланирована на
  **2026‑01‑15** и включает сценарии отравления признаков (feature poisoning),
  усиления повторных запросов (replay amplification) и попыток подделки
  подписей.
- Результаты нужно зафиксировать в модели угроз по мошенничеству и добавить
  соответствующие задачи в `roadmap.md` в workstream
  «Fraud & Telemetry Governance Loop».

## Схемы API

Шлюз теперь предоставляет конкретные JSON‑envelope’ы, которые находятся в
один‑к‑одному соответствии с Norito‑типами из
`crates/iroha_data_model::fraud`:

- **Risk intake** – `POST /v1/fraud/query` принимает payload по схеме
  `RiskQuery`:
  - `query_id` (`[u8; 32]`, hex‑кодировка)
  - `subject` (`AccountId`, `domainless encoded literal; canonical Katakana i105 only (non-canonical Katakana i105 literals rejected)`)
  - `operation` (tagged‑enum, соответствующий `RiskOperation`; JSON‑поле
    `type` отражает вариант enum’а)
  - `related_asset` (`AssetId`, необязательный)
  - `features` (массив `{ key: String, value_hash: hex32 }`, построенный из
    `FeatureInput`)
  - `issued_at_ms` (`u64`)
  - `context` (`RiskContext`; содержит `tenant_id`, необязательный
    `session_id` и необязательное `reason`)
- **Risk decision** – `POST /v1/fraud/assessment` принимает payload
  `FraudAssessment` (также используемый в экспортируемых артефактах
  управления):
  - `query_id`, `engine_id`, `risk_score_bps`, `confidence_bps`,
    `decision` (enum `AssessmentDecision`), `rule_outcomes`
    (массив `{ rule_id, score_delta_bps, rationale? }`)
  - `generated_at_ms`
  - `signature` (опциональная base64‑обёртка вокруг оценки `FraudAssessment`,
    закодированной в Norito)
- **Governance export** – `GET /v1/fraud/governance/export` возвращает
  структуру `GovernanceExport`, когда включена feature `governance`. В ней
  агрегируются активные параметры, последняя enactment‑запись, версия модели,
  policy‑digest и гистограмма `DecisionAggregate`.

Тесты на round‑trip в `crates/iroha_data_model/src/fraud/types.rs` гарантируют
бинарную совместимость этих схем с Norito‑кодеком, а
`integration_tests/tests/fraud_monitoring_requires_assessment_bands.rs`
проверяет end‑to‑end весь канал intake/decision.

## Ссылки на SDK PSP

Ниже перечислены языковые stubs, отражающие примеры интеграции для PSP:

- **Rust** – `integration_tests/tests/fraud_monitoring_requires_assessment_bands.rs`
  использует клиент `iroha` из workspace для формирования `RiskQuery`‑metadata
  и проверки успешных/неуспешных попыток приёма.
- **TypeScript** – `docs/source/governance_api.md` документирует REST‑поверхность,
  которую использует облегчённый Torii‑gateway в демонстрационной PSP‑панели;
  скриптовый клиент расположен в `scripts/ci/schedule_fraud_scoring.sh` и
  используется для smoke‑дри́ллов.
- **Swift и Kotlin** – существующие SDK (`IrohaSwift` и ссылки в
  `crates/iroha_cli/docs/multisig.md`) предоставляют Torii‑hooks для
  добавления полей `fraud_assessment_*`. PSP‑специфичные helpers отслеживаются
  в milestone «Fraud & Telemetry Governance Loop» в `status.md` и используют
  те же билдеры транзакций.

Эти ссылки будут синхронизироваться с микросервисным gateway’ем, чтобы
реализаторы PSP всегда имели актуальный schema и пример кода для каждого
поддерживаемого языка.
