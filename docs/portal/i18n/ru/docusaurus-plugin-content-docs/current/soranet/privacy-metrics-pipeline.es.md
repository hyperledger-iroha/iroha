# Конвейер показателей конфиденциальности SoraNet

SNNet-8 представляет поверхность телеметрии с учетом конфиденциальности для
время работы реле. Теперь реле добавляет события установления связи и цепи в
одноминутные сегменты и экспортирует только грубые счетчики Prometheus, сохраняя
разделение отдельных цепей, обеспечивая при этом полезную видимость
для операторов.

## Сводка агрегатора

- Реализация среды выполнения находится в `tools/soranet-relay/src/privacy.rs` как
  `PrivacyAggregator`.
- Бакеты индексируются за тактовую минуту (`bucket_secs`, по умолчанию 60 секунд) и
  Они хранятся в ограниченном кольце (`max_completed_buckets`, по умолчанию 120). Акции
  сборщиков поддерживают свой собственный ограниченный журнал невыполненной работы (`max_share_lag_buckets`, по умолчанию 12)
  чтобы устаревшие окна Prio очищались как удаленные сегменты, а не
  утечка памяти или маска застрявших коллекторов.
- `RelayConfig::privacy` напрямую сопоставляется с `PrivacyConfig`, открывая ручки регулировки.
  (`bucket_secs`, `min_handshakes`, `flush_delay_buckets`, `force_flush_buckets`,
  `max_completed_buckets`, `max_share_lag_buckets`, `expected_shares`). Время выполнения
  производство поддерживает настройки по умолчанию, в то время как SNNet-8a вводит пороговые значения
  безопасная агрегация.
— Модули времени выполнения регистрируют события с помощью типизированных помощников:
  И18НИ00000038Х, И18НИ00000039Х, И18НИ00000040Х,
  И18НИ00000041Х, И18НИ00000042Х, И18НИ00000043Х,
  `record_verified_bytes` и `record_gar_category`.

## Администратор конечной точки ретрансляции

Операторы могут проконсультироваться с прослушивателем администратора реле для получения наблюдений.
сырой через `GET /privacy/events`. Конечная точка возвращает JSON, разделенный
новые линии (`application/x-ndjson`) с полезной нагрузкой `SoranetPrivacyEventV1`
отражено от внутреннего `PrivacyEventBuffer`. Буфер хранит события
новее до записей `privacy.event_buffer_capacity` (по умолчанию 4096) и являются
пусто при чтении, поэтому парсеры должны опросить достаточно, чтобы
не оставлять пробелов. События охватывают одно и то же рукопожатие, дроссель,
проверенная полоса пропускания, активная цепь и GAR, которые питают счетчики Prometheus,
позволяя последующим сборщикам архивировать хлебные крошки, безопасные для конфиденциальности
или обеспечить безопасные рабочие процессы агрегирования.

## Конфигурация реле

Операторы регулируют частоту телеметрии конфиденциальности в
настройка реле через раздел `privacy`:

```json
{
  "mode": "Entry",
  "listen": "0.0.0.0:443",
  "privacy": {
    "bucket_secs": 60,
    "min_handshakes": 12,
    "flush_delay_buckets": 1,
    "force_flush_buckets": 6,
    "max_completed_buckets": 120,
    "max_share_lag_buckets": 12,
    "expected_shares": 2
  }
}
```

Значения по умолчанию соответствуют спецификации SNNet-8 и проверяются на
нагрузка:| Поле | Описание | По умолчанию |
|-------|-------------|--------|
| `bucket_secs` | Ширина каждого окна агрегации (в секундах). | `60` |
| `min_handshakes` | Минимальное количество участников перед выпуском корзины. | `12` |
| `flush_delay_buckets` | Заполненные ведра ждут перед промывкой. | `1` |
| `force_flush_buckets` | Максимальный возраст до выдачи удаленного сегмента. | `6` |
| `max_completed_buckets` | Отставание сохраненных сегментов (избегайте неограниченной памяти). | `120` |
| `max_share_lag_buckets` | Срок хранения акций коллекционера перед удалением. | `12` |
| `expected_shares` | Акции Prio необходимы перед объединением. | `2` |
| `event_buffer_capacity` | Журнал невыполненных событий NDJSON для потока администрирования. | `4096` |

Установите `force_flush_buckets` ниже `flush_delay_buckets`, установите
пороговые значения равны нулю или отключение защиты теперь не удается
проверка, чтобы избежать развертываний, фильтрующих ретрансляционную телеметрию.

Предел `event_buffer_capacity` также ограничивает `/admin/privacy/events`,
обеспечение того, чтобы скребки не оставались позади на неопределенный срок.

## Акции от коллекционера Prio

SNNet-8a использует двойные коллекторы, которые выдают сегменты Prio с разделением секретов.
Теперь оркестратор анализирует поток NDJSON `/privacy/events` для обоих
записи `SoranetPrivacyEventV1` как для акций `SoranetPrivacyPrioShareV1`,
пересылая их на `SoranetSecureAggregator::ingest_prio_share`. Ведра
выдается при поступлении взносов `PrivacyBucketConfig::expected_shares`,
отражающее поведение агрегатора в реле. Акции проверены
по выравниванию сегментов и форме гистограммы перед объединением в
`SoranetPrivacyBucketMetricsV1`. Если общее количество рукопожатий падает на
ниже `min_contributors` сегмент экспортируется как `suppressed`, что отражает
поведение агрегатора в реле. Windows удалена сейчас
выдать тег `suppression_reason`, чтобы операторы могли различать
И18НИ00000081Х, И18НИ00000082Х, И18НИ00000083Х
и `forced_flush_window_elapsed` при диагностике разрывов телеметрии. Причина
`collector_window_elapsed` также срабатывает, когда акции Prio остаются дольше
за пределами `max_share_lag_buckets`, что делает засоренные коллекторы видимыми без
храните устаревшие аккумуляторы в памяти.

## Конечные точки приема Torii

Torii теперь предоставляет две конечные точки HTTP со закрытой телеметрией, поэтому ретрансляции и
Коллекционеры передают наблюдения, не встраивая в них специальный транспорт:- `POST /v2/soranet/privacy/event` принимает полезную нагрузку
  `RecordSoranetPrivacyEventDto`. Боди обертывает `SoranetPrivacyEventV1`.
  плюс дополнительная этикетка `source`. Torii проверяет запрос на соответствие
  активный профиль телеметрии, записывает событие и отвечает по HTTP
  `202 Accepted` вместе с конвертом JSON Norito, содержащим окно
  вычисляется на основе корзины (`bucket_start_unix`, `bucket_duration_secs`) и
  режим реле.
- `POST /v2/soranet/privacy/share` принимает полезную нагрузку `RecordSoranetPrivacyShareDto`.
  Тело имеет `SoranetPrivacyPrioShareV1` и дополнительную подсказку `forwarded_by`.
  операторам для проверки потоков коллекторов. Успешные поставки
  вернуть HTTP `202 Accepted` с конвертом JSON Norito, суммирующим
  коллектор, окно ведра и подсказка о подавлении; неудачи
  сопоставление проверки с ответом телеметрии `Conversion` для сохранения
  детерминированная обработка ошибок в сборщиках. Цикл событий оркестратора
  теперь излучает эти доли при опросе реле, поддерживая аккумулятор
  Torii изначально синхронизирован с корзинами на реле.

Обе конечные точки соблюдают профиль телеметрии: они выдают `503 Service
Недоступно`, когда метрики отключены. Клиенты могут отправлять
Тела Norito в двоичном формате (`application/x.norito`) или Norito JSON (`application/x.norito+json`);
Сервер автоматически согласовывает формат с помощью стандартных средств извлечения файлов.
Torii.

## Метрика Prometheus

Каждый экспортированный сегмент имеет теги `mode` (`entry`, `middle`, `exit`) и
`bucket_start`. Выпускаются следующие семейства метрик:

| Метрическая | Описание |
|--------|-------------|
| `soranet_privacy_circuit_events_total{kind}` | Таксономия рукопожатия с `kind={accepted,pow_rejected,downgrade,timeout,other_failure,capacity_reject}`. |
| `soranet_privacy_throttles_total{scope}` | Дроссельные счетчики с `scope={congestion,cooldown,emergency,remote_quota,descriptor_quota,descriptor_replay}`. |
| `soranet_privacy_throttle_cooldown_millis_{sum,count}` | Добавлена ​​длительность перезарядки, обусловленная ограничением рукопожатий. |
| `soranet_privacy_verified_bytes_total` | Подтвержденная пропускная способность слепых измерений. |
| `soranet_privacy_active_circuits_{avg,max}` | Среднее и пиковое количество активных цепей на сегмент. |
| `soranet_privacy_rtt_millis{percentile}` | Оценочные процентили RTT (`p50`, `p90`, `p99`). |
| `soranet_privacy_gar_reports_total{category_hash}` | Счетчики GAR хешируются по дайджестам категорий. |
| `soranet_privacy_bucket_suppressed` | Ведра приостановлены, поскольку порог вкладчика не был достигнут. |
| `soranet_privacy_pending_collectors{mode}` | Доля аккумуляторов, ожидающих объединения, сгруппирована по режиму реле. |
| `soranet_privacy_suppression_total{reason}` | Счетчики сегментов удалены с помощью `reason={insufficient_contributors,collector_suppressed,collector_window_elapsed,forced_flush_window_elapsed}`, чтобы указать на пробелы в конфиденциальности. |
| `soranet_privacy_snapshot_suppression_ratio` | Коэффициент подавления/разряда последнего стока (0-1), полезный для бюджетов оповещений. |
| `soranet_privacy_last_poll_unixtime` | Временная метка UNIX последнего успешного опроса (подает предупреждение о простое коллектора). |
| `soranet_privacy_collector_enabled` | Индикатор, который падает до `0`, когда сборщик конфиденциальности отключен или не запускается (подпитывает предупреждение об отключении сборщика). |
| `soranet_privacy_poll_errors_total{provider}` | Ошибки опроса, сгруппированные по псевдонимам ретрансляции (увеличенные из-за ошибок декодирования, сбоев HTTP или неожиданных кодов). |Ведра без наблюдений хранят молчание, сохраняя дашборды
почистить, не делая окон с нулями.

## Оперативное руководство

1. **Панели мониторинга** – отображают приведенные выше показатели, сгруппированные по `mode` и
   `window_start`. Подчеркивает недостающие окна, чтобы выявить проблемы в коллекторах
   или реле. Используйте `soranet_privacy_suppression_total{reason}`, чтобы отличить
   недостатки подавления участников сортировки сборщиком. Актив
   Grafana теперь включает специальную панель ** «Причины подавления (5 м)»**
   основано на этих счетчиках, а также на показателе **"Suppressed Bucket %"**, который
   вычисляет `sum(soranet_privacy_bucket_suppressed) / count(...)` путем выбора
   поэтому операторы сразу видят дефицит бюджета. Серия
   **Журнал по акциям коллекционеров** (`soranet_privacy_pending_collectors`) и статистика
   **Коэффициент подавления снимков** подчеркивает застрявшие коллекционеры и дрейф бюджета
   во время автоматических запусков.
2. **Оповещение** – активирует сигналы тревоги от счетчиков конфиденциальности: пики отклонений.
   PoW, частота восстановления, дрейф RTT и отказ от емкости. Как
   счетчики монотонны в каждом сегменте, правила основаны на скорости
   они работают хорошо.
3. **Реакция на инцидент** – в первую очередь опирается на совокупные данные. Когда требуется
   глубокая отладка, попросите реле воспроизвести снимки сегментов или
   проверять слепые доказательства измерений вместо сбора журналов трафика
   сырой
4. **Удержание** — очищайте достаточно часто, чтобы не превышать
   `max_completed_buckets`. Экспортерам следует рассматривать выходные данные Prometheus как
   канонический источник и отбрасывать локальные сегменты после пересылки.

## Аналитика удаления и автоматическое выполнение

Принятие SNNet-8 зависит от демонстрации того, что автоматические сборщики
оставаться здоровыми и чтобы подавление оставалось в рамках политики
(<= 10% сегментов на реле в любом 30-минутном окне). Инструментарий
необходимое уже включено в репо; операторы должны интегрировать его в свои
еженедельные ритуалы. Новые заглушки Grafana отражают
Фрагменты PromQL ниже, обеспечивающие прямую видимость дежурных команд перед
прибегнуть к ручным запросам.

### Рецепты PromQL для подавления проверок

Операторам следует иметь под рукой следующие помощники PromQL; оба
ссылка на общую панель мониторинга (`dashboards/grafana/soranet_privacy_metrics.json`)
и в правилах Alertmanager:

```promql
/* Suppression ratio per relay mode (30 minute window) */
(
  increase(soranet_privacy_suppression_total{reason=~"insufficient_contributors|collector_suppressed|collector_window_elapsed|forced_flush_window_elapsed"}[30m])
) /
clamp_min(
  increase(soranet_privacy_circuit_events_total{kind="accepted"}[30m]) +
  increase(soranet_privacy_suppression_total[30m]),
1
)
```

```promql
/* Detect new suppression spikes above the permitted minute budget */
increase(soranet_privacy_suppression_total{reason=~"insufficient_contributors|collector_window_elapsed|collector_suppressed"}[5m])
/
clamp_min(
  sum(increase(soranet_privacy_circuit_events_total{kind="accepted"}[5m])),
1
)
```

Используйте соотношение, чтобы убедиться, что статистика **"Подавленный сегмент %"** сохраняется.
ниже политического бюджета; подключите детектор спайков к
Alertmanager для быстрой обратной связи, когда количество налогоплательщиков падает ниже
неожиданный способ.

### Интерфейс командной строки для автономных отчетов сегмента

Рабочая область предоставляет `cargo xtask soranet-privacy-report` для ловушек NDJSON.
пунктуальный. Наведите его на один или несколько экспортов администратора реле:

```bash
cargo xtask soranet-privacy-report \
  --input artifacts/sorafs_privacy/relay-a.ndjson \
  --input artifacts/sorafs_privacy/relay-b.ndjson \
  --json-out artifacts/sorafs_privacy/relay_summary.json
```Помощник обрабатывает захват с помощью `SoranetSecureAggregator`, печатает сводку
удалить на стандартном выводе и, при необходимости, написать структурированный отчет JSON через
`--json-out <path|->`. Уважайте те же ручки, что и у живого коллектора.
(`--bucket-secs`, `--min-contributors`, `--expected-shares` и т. д.), позволяющие
При расследовании проблемы воспроизводите исторические записи с разными пороговыми значениями.
Прикрепите JSON вместе со скриншотами Grafana, чтобы ворота аналитики
SNNet-8 остается проверяемым.

### Автоматизированный контрольный список при первом запуске

Управление по-прежнему требует доказательств того, что первое автоматизированное исполнение соответствовало требованиям
бюджет подавления. Помощник теперь принимает `--max-suppression-ratio <0-1>`.
чтобы CI или операторы быстро терпели неудачу, когда количество удаленных сегментов превышает
разрешенное окно (по умолчанию 10%) или когда сегментов еще нет. Поток
рекомендуется:

1. Экспортируйте NDJSON из конечной точки администратора ретранслятора и потока.
   `/v2/soranet/privacy/event|share` из оркестратора в
   `artifacts/sorafs_privacy/<relay>.ndjson`.
2. Запустите помощник с бюджетом политики:

   ```bash
   cargo xtask soranet-privacy-report \
     --input artifacts/sorafs_privacy/relay-a.ndjson \
     --input artifacts/sorafs_privacy/relay-b.ndjson \
     --json-out artifacts/sorafs_privacy/relay_summary.json \
     --max-suppression-ratio 0.10
   ```

   Команда печатает наблюдаемое соотношение и завершает работу с кодом, отличным от нуля.
   когда бюджет превышен **или** когда нет готовых сегментов, что указывает
   эта телеметрия еще не была произведена для исполнения. Метрики в
   live должен показать, что `soranet_privacy_pending_collectors` стекает к нулю и
   `soranet_privacy_snapshot_suppression_ratio` остается прежним
   бюджета во время выполнения пробега.
3. Архивируйте выходной JSON и журнал CLI с помощью пакета доказательств SNNet-8.
   перед изменением транспорта по умолчанию, чтобы рецензенты могли
   воспроизводить точные артефакты.
