# Руководство по работе с сервисом головоломок

Демон `soranet-puzzle-service` (`tools/soranet-puzzle-service/`) выдает
входные билеты на основе Argon2, которые отражают политику `pow.puzzle.*`
ретранслятор и, при настройке, оркестрировать токены допуска ML-DSA для
реле края. Он предоставляет пять конечных точек HTTP:

- `GET /healthz` - датчик живучести.
- `GET /v2/puzzle/config` - возвращает результирующие эффективные параметры PoW/головоломки.
  реле JSON (`handshake.descriptor_commit_hex`, `pow.*`).
— `POST /v2/puzzle/mint` — выдает билет Аргон2; необязательное тело JSON
  `{ "ttl_secs": <u64>, "transcript_hash_hex": "<32-byte hex>", "signed": true }`
  запрашивает более короткий TTL (привязка к окну политики), связывает билет с
  хэш транскрипта и возвращает билет, подписанный ретранслятором + отпечаток пальца
  подпись при настройке ключей подписи.
- `GET /v2/token/config` - при `pow.token.enabled = true`, возвращает политику
  активный токен доступа (отпечаток эмитента, пределы TTL/раскоса тактовой частоты, идентификатор реле,
  и набор отзыва объединяется).
- `POST /v2/token/mint` — выдает токен допуска ML-DSA, связанный с хэшем резюме.
  предоставлено; боди принимает `{ "transcript_hash_hex": "...", "ttl_secs": <u64>, "flags": <u8> }`.

Билеты, выпускаемые сервисом, проверяются в интеграционном тесте.
`volumetric_dos_soak_preserves_puzzle_and_latency_slo`, который также осуществляет
дросселирование реле во время объемных сценариев DoS.【tools/soranet-relay/tests/adaptive_and_puzzle.rs:337】

## Настраиваем выдачу токенов

Определите поля JSON реле под `pow.token.*` (см.
`tools/soranet-relay/deploy/config/relay.entry.json` для примера) по порядку
для активации токенов ML-DSA. Как минимум предоставьте открытый ключ эмитента.
и необязательный список отзыва:

```json
"pow": {
  "token": {
    "enabled": true,
    "issuer_public_key_hex": "<ML-DSA-44 public key>",
    "revocation_list_hex": [],
    "revocation_list_path": "/etc/soranet/relay/token_revocations.json"
  }
}
```

Служба головоломок повторно использует эти значения и автоматически перезагружает файл.
Norito Отзыв JSON во время выполнения. Используйте CLI `soranet-admission-token`.
(`cargo run -p soranet-relay --bin soranet_admission_token`) для передачи и
проверить автономные токены, добавить записи `token_id_hex` в файл
отзыв и проверка существующих учетных данных перед отправкой обновлений
в производстве.

Передайте секретный ключ эмитента сервису головоломок через флаги CLI:

```bash
cargo run -p soranet-puzzle-service -- \
  --relay-config /etc/soranet/relay/relay.entry.json \
  --token-secret-path /etc/soranet/relay/token_issuer_secret.hex \
  --token-revocation-file /etc/soranet/relay/token_revocations.json \
  --token-revocation-refresh-secs 60
```

`--token-secret-hex` также доступен, если секрет управляется
внеполосный конвейер. Наблюдатель файлов отзыва сохраняет `/v2/token/config`.
актуальный; координируйте обновления с помощью команды `soranet-admission-token revoke`
во избежание состояния позднего отзыва.Установите `pow.signed_ticket_public_key_hex` в JSON реле, чтобы объявить
открытый ключ ML-DSA-44, используемый для проверки подписанных билетов PoW; `/v2/puzzle/config`
повторите ключ и его отпечаток BLAKE3 (`signed_ticket_public_key_fingerprint_hex`) по порядку
чтобы клиенты могли прикрепить чек. Подписанные билеты действительны.
против идентификатора реле и привязок транскрипта и используют одно и то же хранилище
отзыв; 74-байтовые необработанные билеты PoW остаются действительными после проверки.
подписанный билет настроен. Передайте секрет подписи через `--signed-ticket-secret-hex`.
или `--signed-ticket-secret-path` при запуске сервиса головоломок; начало
отклоняет несовместимые пары ключей, если секрет не проверяется

плюс байты из необработанного билета; ответы включают `signed_ticket_b64` и
`signed_ticket_fingerprint_hex` для отслеживания отпечатков пальцев при воспроизведении.
запросы с `signed = true` отклоняются, если секрет подписи не указан.
настроить.

## Инструкция по ротации ключей

1. **Соберите новый дескриптор фиксации.** Управление публикует реле.
   фиксация дескриптора в каталоге пакета. Скопируйте шестнадцатеричную строку в
   `handshake.descriptor_commit_hex` в общей конфигурации реле JSON
   с сервисом головоломок.
2. **Проверьте границы головоломки политики.** Убедитесь, что значения
   Обновления `pow.puzzle.{memory_kib,time_cost,lanes}` соответствуют плану
   выпуск. Операторы должны сохранять детерминированную конфигурацию Argon2.
   между реле (минимум 4 МБ памяти, 1 <= полос <= 16).
3. **Подготовьтесь к перезагрузке.** Перезагрузите системный модуль или контейнер a.
   как только руководство объявит о прекращении ротации. Сервис не поддерживает
   нет горячей перезагрузки; требуется перезапуск, чтобы принять новый дескриптор
   совершить.
4. **Подтвердить.** Выдайте билет через `POST /v2/puzzle/mint` и подтвердите, что
   `difficulty` и `expires_at` соответствуют новой политике. Отчет
   впитывание (`docs/source/soranet/reports/pow_resilience.md`) захватывает клеммы
   ожидаемая задержка для справки. Когда токены активны, читайте
   `/v2/token/config`, чтобы убедиться, что эмитент отпечатков пальцев объявляет и
   учетная запись отзыва соответствует ожидаемым значениям.

## Процедура аварийной деактивации

1. Установите `pow.puzzle.enabled = false` в конфигурации общего реле.
   Сохраните `pow.required = true`, если резервные билеты hashcash должны остаться.
   обязательный.
2. При необходимости введите записи `pow.emergency`, чтобы отклонить
   дескрипторы устарели, пока ворота Argon2 отключены.
3. Перезапустите реле и службу головоломки, чтобы применить
   изменить.
4. Отслеживайте `soranet_handshake_pow_difficulty`, чтобы убедиться, что проблема
   падает до ожидаемого значения hashcash и подтвердите, что `/v2/puzzle/config`
   сообщает `puzzle = null`.

## Ведение журнала соответствия и аудита

Реле выдают структурированные события `handshake`, в которых указаны причины
время дросселирования и остывания. Убедитесь, что конвейер соответствия
описанный в `docs/source/soranet/relay_audit_pipeline.md`, принимает эти журналы, чтобы
что изменения в политике остаются проверяемыми. Когда дверь-загадка
активен, в архиве отчеканены образцы билетов и снимок
конфигурация Norito с билетом развертывания для будущих аудитов.
жетоны допуска, отчеканенные до периодов обслуживания, должны соблюдаться
со своими значениями `token_id_hex` и вставленными в файл отзыва.
время истекло или аннулировано.
