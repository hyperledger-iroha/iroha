SoraNet — это уровень анонимизации, который поддерживает выборку диапазона в широковещательных RPC-каналах SoraFS и Norito, а также в будущих каналах данных Nexus. Транспортная программа (элементы дорожной карты **SNNet-1**, **SNNet-1a** и **SNNet-1b**) определяют детерминированное рукопожатие, постквантовое (PQ) согласование мощности и планируют чередование солей, чтобы каждое реле, клиент и шлюз соблюдали один и тот же режим безопасности.

## Цели и сетевая модель

- Создание цепей с тремя переходами (вход -> середина -> выход) поверх QUIC v1, чтобы злоумышленники не обращались к Torii напрямую.
- Установите *гибридное* подтверждение Noise XX (Curve25519 + Kyber768) поверх QUIC/TLS, чтобы связать ключи сеанса с текстом TLS.
- Обеспечить возможности TLV, объявляя поддержку PQ KEM/подписи, роли ретранслятора и версии протокола; И использовать GREASE для неизвестных типов, чтобы будущие расширения оставались доступными для публикации.
- Ежедневно чередуйте соли шифрованного контента и устанавливайте защитные реле на 30 дней, чтобы отток каталога не приводил к расшифровке клиентов.
- Сохраняйте ячейки фиксированными на уровне 1024 B, вводите фиктивные поля/ячейки и экспортируйте детерминированную телеметрию, чтобы быстро перехватывать попытки понижения версии.

## путь подтверждения (SNNet-1a)

1. **Конверт QUIC/TLS** — клиенты подключаются к реле через QUIC v1 и завершают подтверждение TLS 1.3 с использованием сертификатов Ed25519, подписанных центром сертификации управления. Экспортер TLS (`tls-exporter("soranet handshake", 64)`) предоставляет слою шума начальные значения, чтобы текст был неотделим.
2. **Гибрид Noise XX** — строка протокола `Noise_XXhybrid_25519+Kyber768_AESGCM_SHA256` с экспортером prologue=TLS. Поток сообщений:

   ```
   -> e, s
   <- e, ee, se, s, pq_ciphertext
   -> ee, se, pq_ciphertext
   ```

   Выходы DH Curve25519 и обеих инкапсуляций Kyber смешиваются в окончательные симметричные переключатели. Неспособность согласовать PQ прерывает все рукопожатие и позволяет использовать только классический запасной вариант.

3. **Билеты и жетоны головоломки**. Реле могут запросить билет Proof of Argon2id до `ClientHello`. Билеты — это кадры с префиксом длины, которые содержат хешированное решение Argon2 и срок действия которых истекает в пределах ограничений политики:

   ```norito
   struct PowTicketV1 {
       version: u8,
       difficulty: u8,
       expires_at: u64,
       client_nonce: [u8; 32],
       solution: [u8; 32],
   }
   ```

   Токены принятия с префиксом `SNTK` обходят головоломки, когда подпись эмитента ML-DSA-44 проверяется на соответствие активной политике и списку отзыва.

4. **Capability Exchange TLV** — конечная полезная нагрузка в Noise TLV несет в себе возможности, описанные ниже. Клиенты разрывают соединение, если какая-либо обязательная возможность (PQ KEM/подпись, роль или версия) отсутствует или не соответствует записи каталога.

5. **Регистрация текста**. Ретранслирует текстовый хэш журнала, отпечаток TLS и содержимое TLV для передачи детекторам понижения версии и отслеживания соответствия.

## TLV возможностей (SNNet-1c)

Возможности повторного использования фиксированной оболочки TLV из `typ/length/value`:

```norito
struct CapabilityTLV {
    typ: u16,
    length: u16,
    value: Vec<u8>,
}
```

Известные на данный момент типы:

- `snnet.pqkem` - Кибер-уровень (`kyber768` для текущего развертывания).
- `snnet.pqsig` - набор подписей PQ (`ml-dsa-44`).
- `snnet.role` - роль реле (`entry`, `middle`, `exit`, `gateway`).
- `snnet.version` - Идентификатор версии протокола.
- `snnet.grease` - Случайные элементы заполнения в пределах зарезервированного диапазона для обеспечения приемлемости будущих TLV.Клиенты поддерживают список разрешений для запрошенных TLV и не выполняют рукопожатия, если они пропущены или понижены. Ретрансляторы передают один и тот же набор в каталог микродескрипторов, чтобы обеспечить детерминированность аутентификации.

## Вращение солей и ослепление CID (SNNet-1b)

- Управление публикует запись `SaltRotationScheduleV1` со значениями `(epoch_id, salt, valid_after, valid_until)`. Реле и шлюзы извлекают подписанную таблицу из каталога издателя.
— Клиенты применяют новую соль по адресу `valid_after`, сохраняют предыдущую соль в течение льготного периода в 12 часов и сохраняют историю за 7 эпох, чтобы допустить поздние обновления.
- Используются стандартные запутанные идентификаторы:

  ```
  cache_key = BLAKE3("soranet.blinding.canonical.v1" ∥ salt ∥ cid)
  ```

  шлюзы принимают запутанный ключ по адресу `Sora-Req-Blinded-CID` и возвращают его по адресу `Sora-Content-CID`. Шифрование цепи/порядка (`CircuitBlindingKey::derive`) находится по адресу `iroha_crypto::soranet::blinding`.
- Если реле пропускает эпоху, оно останавливает новые цепи до тех пор, пока таблица не выйдет из строя и не будет выдан `SaltRecoveryEventV1`, а дежурные панели рассматривают это как пейджинговый сигнал.

## Данные каталога и политика защиты

- Микродескрипторы содержат идентификатор реле (Ed25519 + ML-DSA-65), переключатели PQ, TLV мощности, теги зон, право на защиту и текущую эпоху заявленной соли.
— Клиенты устанавливают комплекты защиты на 30 дней и сохраняют кэш `guard_set` вместе с подписанным снимком каталога. Интерфейсы CLI и SDK предоставляют отпечаток кэша, что позволяет прикреплять доказательства вычитания к журналам изменений.

## Контрольный список телеметрии и развертывания

- Стандарты, которые необходимо экспортировать перед производством:
  - `soranet_handshake_success_total{role}`
  - `soranet_handshake_failure_total{reason}`
  -`soranet_handshake_latency_seconds`
  -`soranet_capability_mismatch_total`
  -`soranet_salt_rotation_lag_seconds`
- Пороговые значения оповещений расположены рядом с матрицей SLO для вращающихся солей SOP (`docs/source/soranet_salt_plan.md#slo--alert-matrix`) и должны быть отменены в Alertmanager перед обновлением сети.
- Оповещения: частота отказов >5% в течение 5 минут, задержка соли >15 минут или мониторинг несоответствий производственных возможностей.
Этапы вычитания:
  1. Запустите тесты совместимости реле/клиента на промежуточном этапе с включенным гибридным подтверждением связи и стеком PQ.
  2. Выполните репетицию ротации СОП (`docs/source/soranet_salt_plan.md`) и добавьте артефакты упражнений в журнал изменений.
  3. Активируйте согласование мощности в каталоге, затем опубликуйте данные для входных, промежуточных и выходных ретрансляторов и, наконец, для клиентов.
  4. Запись отпечатков защитного кэша, солевых таблиц и панелей телеметрии для каждого этапа; И прикрепите пакет доказательств к `status.md`.

Следование этому контрольному списку позволяет операторам, клиентам и командам SDK гармонично реализовать миграцию SoraNet, придерживаясь при этом требований детерминизма и аудита, отраженных в дорожной карте SNNet.
