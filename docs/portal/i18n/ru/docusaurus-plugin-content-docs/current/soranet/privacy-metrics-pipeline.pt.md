# Конвейер показателей конфиденциальности SoraNet

SNNet-8 представляет телеметрическую поверхность с учетом конфиденциальности во время работы реле. Теперь реле объединяет события установления связи и цепей в одноминутные сегменты и экспортирует только грубые счетчики Prometheus, сохраняя отдельные цепи развязанными и обеспечивая при этом полезную видимость для операторов.

## Обзор агрегатора

- Реализация среды выполнения находится в `tools/soranet-relay/src/privacy.rs` как `PrivacyAggregator`.
- Бакеты переключаются каждую минуту (`bucket_secs`, по умолчанию 60 секунд) и сохраняются в ограниченном кольце (`max_completed_buckets`, по умолчанию 120). Общие ресурсы сборщика поддерживают свой собственный ограниченный журнал невыполненной работы (`max_share_lag_buckets`, по умолчанию 12), поэтому старые окна Prio очищаются как подавленные сегменты, а не теряют память или маскируют застрявшие сборщики.
- `RelayConfig::privacy` напрямую сопоставляется с `PrivacyConfig`, открывая ручки регулировки (`bucket_secs`, `min_handshakes`, `flush_delay_buckets`, `force_flush_buckets`, `max_completed_buckets`, `max_share_lag_buckets`, `expected_shares`). Рабочая среда выполнения поддерживает значения по умолчанию, а SNNet-8a вводит пороговые значения безопасного агрегирования.
- Модули времени выполнения регистрируют события через типизированные помощники: `record_circuit_accepted`, `record_circuit_rejected`, `record_throttle`, `record_throttle_cooldown`, `record_capacity_reject`, `record_active_sample`, `record_verified_bytes` и `record_gar_category`.

## Конечная точка администратора ретрансляции

Операторы могут запросить у прослушивателя администратора реле необработанные наблюдения через `GET /privacy/events`. Конечная точка возвращает JSON с разделителями-новой строкой (`application/x-ndjson`), содержащий полезные данные `SoranetPrivacyEventV1`, зеркально отраженные из внутреннего `PrivacyEventBuffer`. Буфер хранит новейшие события до записей `privacy.event_buffer_capacity` (по умолчанию 4096) и опорожняется при чтении, поэтому парсеры должны опрашивать достаточно часто, чтобы избежать пробелов. События охватывают те же сигналы установления связи, регулирования, проверки полосы пропускания, активной цепи и GAR, которые подаются на счетчики Prometheus, что позволяет нижестоящим сборщикам сохранять конфиденциальные хлебные крошки или передавать безопасные рабочие процессы агрегирования.

## Конфигурация реле

Операторы настраивают частоту телеметрии конфиденциальности в файле конфигурации реле через раздел `privacy`:

```json
{
  "mode": "Entry",
  "listen": "0.0.0.0:443",
  "privacy": {
    "bucket_secs": 60,
    "min_handshakes": 12,
    "flush_delay_buckets": 1,
    "force_flush_buckets": 6,
    "max_completed_buckets": 120,
    "max_share_lag_buckets": 12,
    "expected_shares": 2
  }
}
```

Значения поля по умолчанию соответствуют спецификации SNNet-8 и проверяются при загрузке:| Поле | Описание | Стандарт |
|-------|-----------|--------|
| `bucket_secs` | Ширина каждого окна агрегации (в секундах). | `60` |
| `min_handshakes` | Минимальное количество участников, прежде чем сегмент сможет выдать счетчики. | `12` |
| `flush_delay_buckets` | Полные ведра, чтобы подождать перед попыткой промывки. | `1` |
| `force_flush_buckets` | Максимальный возраст до выдачи подавленного сегмента. | `6` |
| `max_completed_buckets` | Отставание сохраненных сегментов (предотвращает неограниченную память). | `120` |
| `max_share_lag_buckets` | Период хранения акций коллекционера перед удалением. | `12` |
| `expected_shares` | Перед сопоставлением требуются акции коллекционера Prio. | `2` |
| `event_buffer_capacity` | Журнал событий NDJSON для потока администрирования. | `4096` |

Установка `force_flush_buckets` ниже, чем `flush_delay_buckets`, сброс пороговых значений или отключение защиты сохранения теперь не проходит проверку, чтобы предотвратить развертывания, которые могут привести к утечке телеметрии ретрансляции.

Ограничение `event_buffer_capacity` также ограничивает `/admin/privacy/events`, гарантируя, что скраперы не могут задерживаться на неопределенный срок.

## Коллекционные акции Prio

SNNet-8a использует двойные коллекторы, которые выдают сегменты Prio с разделением секретов. Теперь оркестратор анализирует поток NDJSON `/privacy/events` на предмет записей `SoranetPrivacyEventV1` и разделяет `SoranetPrivacyPrioShareV1`, перенаправляя их в `SoranetSecureAggregator::ingest_prio_share`. При поступлении вкладов корзины выдают `PrivacyBucketConfig::expected_shares`, что отражает поведение реле. Перед объединением в `SoranetPrivacyBucketMetricsV1` общие ресурсы проверяются на соответствие сегментов и форму гистограммы. Если общее количество подтверждений падает ниже `min_contributors`, сегмент экспортируется как `suppressed`, что отражает поведение агрегатора в реле. Подавленные окна теперь выдают метку `suppression_reason`, чтобы операторы могли различать `insufficient_contributors`, `collector_suppressed`, `collector_window_elapsed` и `forced_flush_window_elapsed` при диагностике пробелов в телеметрии. Мотив `collector_window_elapsed` также срабатывает, когда доли Prio выходят за пределы `max_share_lag_buckets`, делая застрявшие коллекторы видимыми, не оставляя в памяти старые аккумуляторы.

## Torii конечные точки приема

Torii теперь предоставляет две конечные точки HTTP со шлюзом телеметрии, чтобы ретрансляторы и сборщики могли пересылать наблюдения без внедрения специального транспорта:- `POST /v1/soranet/privacy/event` принимает полезную нагрузку `RecordSoranetPrivacyEventDto`. Тело содержит `SoranetPrivacyEventV1` плюс дополнительную метку `source`. Torii проверяет запрос на соответствие активному профилю телеметрии, регистрирует событие и отвечает HTTP `202 Accepted` вместе с конвертом JSON Norito, содержащим вычисляемое окно (`bucket_start_unix`, `bucket_duration_secs`) и режим ретрансляции.
- `POST /v1/soranet/privacy/share` принимает полезную нагрузку `RecordSoranetPrivacyShareDto`. Тело содержит `SoranetPrivacyPrioShareV1` и дополнительную подсказку `forwarded_by`, чтобы операторы могли проверять потоки коллектора. Успешные отправки возвращают HTTP `202 Accepted` с конвертом JSON Norito, в котором кратко описаны сборщик, окно сегмента и подсказка по подавлению; Ошибки проверки сопоставляются с ответом телеметрии `Conversion`, чтобы сохранить детерминированную обработку ошибок между сборщиками. Цикл событий оркестратора теперь генерирует эти общие ресурсы при опросе реле, поддерживая синхронизацию аккумулятора Torii Prio с сегментами в реле.

Обе конечные точки учитывают профиль телеметрии: они выдают `503 Service Unavailable`, когда метрики отключены. Клиенты могут отправлять двоичные тела Norito (`application/x.norito`) или Norito JSON (`application/x.norito+json`); сервер автоматически согласовывает формат с помощью стандартных экстракторов Torii.

## Метрика Prometheus

Каждый экспортированный сегмент имеет метки `mode` (`entry`, `middle`, `exit`) и `bucket_start`. Выводятся следующие семейства метрик:

| Метрическая | Описание |
|--------|-------------|
| `soranet_privacy_circuit_events_total{kind}` | Таксономия рукопожатия с `kind={accepted,pow_rejected,downgrade,timeout,other_failure,capacity_reject}`. |
| `soranet_privacy_throttles_total{scope}` | Счетчики дроссельной заслонки с `scope={congestion,cooldown,emergency,remote_quota,descriptor_quota,descriptor_replay}`. |
| `soranet_privacy_throttle_cooldown_millis_{sum,count}` | Длительность перезарядки суммируется по ограниченным рукопожатиям. |
| `soranet_privacy_verified_bytes_total` | Пропускная способность проверена в результате слепых измерений. |
| `soranet_privacy_active_circuits_{avg,max}` | Среднее и пиковое количество активных цепей на сегмент. |
| `soranet_privacy_rtt_millis{percentile}` | Оценочные процентили RTT (`p50`, `p90`, `p99`). |
| `soranet_privacy_gar_reports_total{category_hash}` | Счетчики отчета о действиях по управлению, хешированные по дайджестам категорий. |
| `soranet_privacy_bucket_suppressed` | Ведра приостановлены, поскольку порог вкладчика не был достигнут. |
| `soranet_privacy_pending_collectors{mode}` | Коллектор разделяет аккумуляторы в ожидании комбинации, сгруппированной по режиму реле. |
| `soranet_privacy_suppression_total{reason}` | Подавлены счетчики сегментов с `reason={insufficient_contributors,collector_suppressed,collector_window_elapsed,forced_flush_window_elapsed}` для информационных панелей для выявления пробелов в конфиденциальности. |
| `soranet_privacy_snapshot_suppression_ratio` | Соотношение подавленного/осушенного последнего стока (0-1), полезно для бюджетов оповещений. |
| `soranet_privacy_last_poll_unixtime` | Временная метка UNIX последнего успешного опроса (подает предупреждение о простое коллектора). |
| `soranet_privacy_collector_enabled` | Индикатор, который становится `0`, когда сборщик конфиденциальности отключен или не запускается (подает предупреждение об отключении сборщика). |
| `soranet_privacy_poll_errors_total{provider}` | Ошибки опроса, сгруппированные по псевдонимам ретрансляции (увеличение количества ошибок декодирования, сбоев HTTP или неожиданных кодов состояния). |Ведра без наблюдений остаются безмолвными, сохраняя чистоту приборных панелей, не создавая при этом нулевых окон.

## Оперативное руководство

1. **Панели мониторинга** — постройте приведенные выше показатели, сгруппированные по `mode` и `window_start`. Выделите недостающие окна, чтобы выявить проблемы с коллектором или реле. Используйте `soranet_privacy_suppression_total{reason}`, чтобы отличить отсутствующих участников от удалений, вызванных сборщиком, при проверке пробелов. Ресурс Grafana теперь отправляет специальную панель **"Причины подавления (5 м)"**, основанную на этих счетчиках, а также статистику **"Подавленный сегмент сегмента"**, которая рассчитывает `sum(soranet_privacy_bucket_suppressed) / count(...)` для каждого выбора, чтобы операторы могли быстро обнаружить нарушения бюджета. Серия **"Журнал доли коллекторов"** (`soranet_privacy_pending_collectors`) и статистика **"Коэффициент подавления моментальных снимков"** выявляют застрявшие коллекторы и отклонения бюджета во время автоматизированных исполнений.
2. **Оповещение** — подача сигналов тревоги от счетчиков, защищенных конфиденциальностью: всплески отклонения PoW, частота восстановления, дрейф RTT и отклонения емкости. Поскольку счетчики в каждом сегменте монотонны, простые правила, основанные на ставках, работают хорошо.
3. **Реакция на инцидент** — в первую очередь доверяйте агрегированным данным. Если необходима более глубокая отладка, попросите ретрансляторы воспроизвести снимки сегмента или проверить данные слепых измерений вместо сбора необработанных журналов трафика.
4. **Удержание** — очищайте нож достаточно часто, чтобы не превышать `max_completed_buckets`. Экспортерам следует рассматривать выходные данные Prometheus как канонический источник и удалять локальные сегменты после пересылки.

## Анализ подавления и автоматическое выполнение

Принятие SNNet-8 зависит от демонстрации того, что автоматические сборщики остаются работоспособными и что подавление остается в пределах политики (<= 10% сегментов на реле в любом 30-минутном окне). Инструменты, необходимые для реализации этого шлюза, теперь поставляются с репозиторием; Операторам следует интегрировать это в свои еженедельные ритуалы. Новые панели подавления Grafana отражают приведенные ниже фрагменты PromQL, предоставляя группам развертывания живую видимость, прежде чем им придется прибегать к ручным запросам.

### Рецепты PromQL для обзора подавления

Операторам следует иметь под рукой следующие помощники PromQL; оба упоминаются в общей информационной панели Grafana (`dashboards/grafana/soranet_privacy_metrics.json`) и в правилах Alertmanager:

```promql
/* Suppression ratio per relay mode (30 minute window) */
(
  increase(soranet_privacy_suppression_total{reason=~"insufficient_contributors|collector_suppressed|collector_window_elapsed|forced_flush_window_elapsed"}[30m])
) /
clamp_min(
  increase(soranet_privacy_circuit_events_total{kind="accepted"}[30m]) +
  increase(soranet_privacy_suppression_total[30m]),
1
)
```

```promql
/* Detect new suppression spikes above the permitted minute budget */
increase(soranet_privacy_suppression_total{reason=~"insufficient_contributors|collector_window_elapsed|collector_suppressed"}[5m])
/
clamp_min(
  sum(increase(soranet_privacy_circuit_events_total{kind="accepted"}[5m])),
1
)
```

Используйте выходные данные соотношения, чтобы подтвердить, что показатель **"% подавленного сегмента"** остается ниже политического бюджета; Подключите детектор всплесков к Alertmanager, чтобы быстро получать обратную связь в случае неожиданного снижения количества участников.

### Интерфейс командной строки для создания автономных отчетов по сегментам

Рабочая область предоставляет `cargo xtask soranet-privacy-report` для захватов NDJSON на определенный момент времени. Укажите один или несколько экспортов администратора ретрансляции:

```bash
cargo xtask soranet-privacy-report \
  --input artifacts/sorafs_privacy/relay-a.ndjson \
  --input artifacts/sorafs_privacy/relay-b.ndjson \
  --json-out artifacts/sorafs_privacy/relay_summary.json
```Помощник передает захват через `SoranetSecureAggregator`, выводит сводную информацию о подавлении на стандартный вывод и (при необходимости) записывает структурированный отчет JSON через `--json-out <path|->`. Он использует те же регуляторы, что и живой коллектор (`--bucket-secs`, `--min-contributors`, `--expected-shares` и т. д.), позволяя операторам воспроизводить исторические записи с различными пороговыми значениями при сортировке инцидента. Прикрепите JSON вместе с записями Grafana, чтобы шлюз анализа подавления SNNet-8 оставался проверяемым.

### Контрольный список для первого автоматического выполнения

Управление по-прежнему требует доказательства того, что первое автоматизированное исполнение соответствовало бюджету на подавление. Помощник теперь принимает `--max-suppression-ratio <0-1>`, чтобы CI или операторы быстро терпели сбой, когда подавленные сегменты превышают разрешенное окно (по умолчанию 10%) или когда сегментов еще нет. Рекомендуемый поток:

1. Экспортируйте NDJSON из конечных точек администратора ретрансляции, а также поток оркестратора `/v1/soranet/privacy/event|share` в `artifacts/sorafs_privacy/<relay>.ndjson`.
2. Запустите помощник с бюджетом политики:

   ```bash
   cargo xtask soranet-privacy-report \
     --input artifacts/sorafs_privacy/relay-a.ndjson \
     --input artifacts/sorafs_privacy/relay-b.ndjson \
     --json-out artifacts/sorafs_privacy/relay_summary.json \
     --max-suppression-ratio 0.10
   ```

   Команда печатает наблюдаемое соотношение и завершает работу с ненулевым кодом, когда бюджет превышен **или**, когда еще нет готовых сегментов, сигнализируя, что телеметрия еще не создана для выполнения. Динамические показатели должны показывать, что `soranet_privacy_pending_collectors` сливается до нуля, а `soranet_privacy_snapshot_suppression_ratio` падает ниже того же бюджета по мере выполнения.
3. Архивируйте выходные данные JSON и журнал CLI с помощью пакета доказательств SNNet-8 перед изменением транспортного значения по умолчанию, чтобы рецензенты могли воспроизвести точные артефакты.

## Следующие шаги (SNNet-8a)

- Интегрируйте два коллектора Prio, подключив прием общих ресурсов к среде выполнения, чтобы реле и коллекторы выдавали согласованные полезные нагрузки `SoranetPrivacyBucketMetricsV1`. *(Завершено — см. `ingest_privacy_payload` в `crates/sorafs_orchestrator/src/lib.rs` и связанные с ним тесты.)*
- Опубликуйте общую панель мониторинга Prometheus и правила оповещений, охватывающие пробелы в подавлении, работоспособность сборщиков и снижение анонимности. *(Завершено — см. `dashboards/grafana/soranet_privacy_metrics.json`, `dashboards/alerts/soranet_privacy_rules.yml`, `dashboards/alerts/soranet_policy_rules.yml` и средства проверки.)*
- Создание артефактов дифференциальной калибровки конфиденциальности, описанных в `privacy_metrics_dp.md`, включая воспроизводимые блокноты и дайджесты управления. *(Завершено — записная книжка и артефакты, созданные `scripts/telemetry/run_privacy_dp.py`; оболочка CI `scripts/telemetry/run_privacy_dp_notebook.sh` запускает записную книжку через рабочий процесс `.github/workflows/release-pipeline.yml`; дайджест управления заархивирован в `docs/source/status/soranet_privacy_dp_digest.md`.)*

Текущая версия представляет собой основу SNNet-8: детерминистическую, защищенную конфиденциальность телеметрию, которая напрямую вписывается в существующие парсеры и информационные панели Prometheus.
