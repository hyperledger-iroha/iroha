# План предпросмотра с контрольной суммой

В этом плане описаны оставшиеся работы, необходимые для того, чтобы каждый документ предпросмотра портала можно было проверить перед публикацией. Цель — добиться, чтобы рецензенты скачивали снимок криптовалюты, собранный в CI, чтобы контрольная сумма манифеста неизменялась, и чтобы предпросмотр доступен через SoraFS с метаданными Norito.

## Цели

- **Детерминированные сборки:** гарантируют, что `npm run build` дает воспроизводимый результат и всегда производит `build/checksums.sha256`.
- **Проверенные предпросмотры:** требует, чтобы каждый документ предпросмотра содержал контрольную сумму манифеста и запрещал публикацию при неудачных проверках.
- **Метаданные, публикуемые через Norito:** сохранение дескрипторов предпросмотра (метаданные коммита, контрольная сумма дайджеста, CID SoraFS) как Norito JSON, чтобы инструменты управления могли аудировать релизы.
- **Инструменты для операторов:** выберите одношаговый скрипт проверки, который пользователь может запускать локально (`./docs/portal/scripts/preview_verify.sh --build-dir build --descriptor <path> --archive <path>`); скрипт теперь оборачивает поток, проверяет контрольную сумму + дескриптора руководителя. Стандартная предкомандапросмотра (`npm run serve`) теперь автоматически вызывает этого помощника перед `docusaurus serve`, чтобы локальные точки были под контролем контрольной суммы (при этом `npm run serve:verified` как сохранен явный псевдоним).

## Фаза 1 — Контроль в CI

1. Обновить `.github/workflows/docs-portal-preview.yml`, чтобы:
   - Запуск `node docs/portal/scripts/write-checksums.mjs` после сборки Docusaurus (уже возникает локально).
   - Выполнять `cd build && sha256sum -c checksums.sha256` и проваливать задание при несовпадении.
   - Упаковывать каталог сборки в `artifacts/preview-site.tar.gz`, копировать контрольную сумму манифеста, питания `scripts/generate-preview-descriptor.mjs` и выполнять `scripts/sorafs-package-preview.sh` с JSON-конфигурацией (см. `docs/examples/sorafs_preview_publish.json`), чтобы рабочий процесс выпускал и метаданные, и детерминированный SoraFS-бандл.
   - Загрузите статический сайт, архивы метаданных (`docs-portal-preview`, `docs-portal-preview-metadata`) и SoraFS-bandl (`docs-portal-preview-sorafs`), чтобы манифестировать сводку CAR и запланировать возможность проверки без повторной сборки.
2. Добавить комментарий с CI-бейджем, резюмирующий результат проверки контрольной суммы в пул-реквесте (реализовано через шаг комментариев GitHub Script в `docs-portal-preview.yml`).
3. Задокументируйте рабочий процесс в `docs/portal/README.md` (раздел CI) и дайте ссылку на этапы проверки в контрольном списке публикации.

## Скрипт проверки

`docs/portal/scripts/preview_verify.sh` впоследствии скачаны архивы предварительного просмотра без ручных вызовов `sha256sum`. Используйте `npm run serve` (или явный псевдоним `npm run serve:verified`), чтобы запустить скрипт и `docusaurus serve` для одного шага при распространении локальных параметров. Логика проверок:

1. Запускает соответствующий SHA-инструмент (`sha256sum` или `shasum -a 256`) против `build/checksums.sha256`.
2. При необходимости указывается дайджест/имя файла дескриптора предварительного просмотра `checksums_manifest` и, если указано, дайджест/имя файла архива предварительного просмотра.
3. Завершается ненулевым кодом при любом несоответствии, чтобы рецензенты могли заблокировать подмененные предпросмотры.

Пример использования (после извлечения документов CI):```bash
./docs/portal/scripts/preview_verify.sh \
  --build-dir build \
  --descriptor artifacts/preview-descriptor.json \
  --archive artifacts/preview-site.tar.gz
```

Инженеры CI и релизы должны включать скрипт каждый раз, когда загружается предпросмотр пакета или прикрепляются документы к релизному билету.

## Фаза 2 — Публикация SoraFS

1. Расширить рабочий процесс предпросмотра партии, которая:
   - Загружает собранный сайт в staging-шлюз SoraFS с помощью `sorafs_cli car pack` и `manifest submit`.
   - Захватывает возвращенный дайджест-манифест и CID SoraFS.
   - Сериализует `{ commit, branch, checksum_manifest, cid }` в Norito JSON (`docs/portal/preview/preview_descriptor.json`).
2. Хранить дескриптор рядом с артефактом сборки и отображать CID в комментариях запроса на включение.
3. Добавить интеграционные тесты, которые гонят `sorafs_cli` в режиме пробного прогона, чтобы будущие изменения сохраняли метаданные схемы совместимости.

## Фаза 3 — Управление и аудит

1. Опубликовать Norito-схему (`PreviewDescriptorV1`), описывающую структуру дескриптора, в `docs/portal/schemas/`.
2. Обновить чеклист по DOCS-SORA, требуя:
   - Запуск `sorafs_cli manifest verify` против загруженного CID.
   - Фиксировать контрольную сумму дайджеста манифеста и CID в описании релизного PR.
3. Включить автоматизацию управления для сверки дескриптора с контрольной суммой манифеста во время голосования по релизам.
