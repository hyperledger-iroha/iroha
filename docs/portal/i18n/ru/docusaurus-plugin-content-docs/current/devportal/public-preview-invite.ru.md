# Плейбук приглашений на общедоступном предварительном просмотре

## Цели программы

Этот плейбук мозг, как анонсировать и проводить публичный предварительный просмотр после того, как
Рабочий процесс онбординга ревьюеров запущен. Он сохраняет честность дорожной карты DOCS-SORA,
гарантируя, что каждое приглашение покидает проверенные документы, искренние.
по безопасности и световому каналу обратной связи.

- **Аудитория:** курированный список членов сообщества, партнеров и мейнтейнеров, которые
  поддерживают политику приемлемого использования для предварительного просмотра.
- **Ограничения:** размер волны по умолчанию <= 25 ревьюеров, доступ к окну 14 дней, на реакцию
  инциденты в течение 24 часов.

## Чеклист гейта перед запуском

Выполните эту задачу перед отправкой приглашения:

1. Последние предпросмотр-артефакты загружены в CI (`docs-portal-preview`,
   манифест контрольной суммы, дескриптор, пакет SoraFS).
2. `npm run --prefix docs/portal serve` (вентиль контрольной суммы) протестирован на том же теге.
3. Билеты онбординга ревьюеров одобрения и поощрения с волной приглашений.
4. Документы по безопасности, наблюдаемости и инцидентам проверены.
   ([`security-hardening`](./security-hardening.md),
   [`observability`](./observability.md),
   [`incident-runbooks`](./incident-runbooks.md)).
5. Подготовлен формуляр обратной связи или шаблон вопроса (поля строгости, шагов шага,
   скриншоты и информация о проведении).
6. Текст анонса проверен Docs/DevRel + Governance.

## Пакет приглашения

Ваше приглашение должно включать:

1. **Проверенные артефакты** — дайте ссылки на манифест/план SoraFS или артефакт GitHub,
   а также манифест и дескриптор контрольной суммы. Явно укажите условия проверки, чтобы
   Рецензенты могли запустить ее перед запуском сайта.
2. **Инструкции служат** — Включите команду предварительного просмотра с контрольной суммой:

   ```bash
   DOCS_RELEASE_TAG=preview-<stamp> npm run --prefix docs/portal serve
   ```

3. **Напоминания по безопасности** — Укажите, что токены закрываются автоматически, ссылки нельзя
   сообщить, об инцидентах необходимо сообщить немедленно.
4. **Канал обратной связи** — Дайте ссылку на шаблон/форму выпуска и обозначьте время ожидания ответа.
5. **Даты программы** — укажите дату начала/окончания, рабочее время или синхронизацию встреч и обновите следующее окно.

Пример письма в
[`docs/examples/docs_preview_invite_template.md`](../../../examples/docs_preview_invite_template.md)
покрывает эти требования. Обновите заполнители (даты, URL-адреса, контакты)
передотправкой.

## Публикация предварительного просмотра хоста

Продвигайте предварительный просмотр хоста только после завершения онбординга и заявления об изменении тикета.
См. [руководство по хосту предварительного просмотра экспозиции](./preview-host-exposure.md) для сквозных шагов
сборка/публикация/проверка, см. в этом разделе.

1. **Сборка и упаковка:** Проставьте тег выпуска и создайте определенные сертификаты.

   ```bash
   cd docs/portal
   export DOCS_RELEASE_TAG="preview-$(date -u +%Y%m%dT%H%M%SZ)"
   npm ci
   npm run build
   ./scripts/sorafs-pin-release.sh \
     --alias docs-preview.sora \
     --alias-namespace docs \
     --alias-name preview \
     --pin-label docs-preview \
     --skip-submit
   node scripts/generate-preview-descriptor.mjs \
     --manifest artifacts/checksums.sha256 \
     --archive artifacts/sorafs/portal.tar.gz \
     --out artifacts/sorafs/preview-descriptor.json
   ```

   Скрипт-пин записывает `portal.car`, `portal.manifest.*`, `portal.pin.proposal.json`,
   и `portal.dns-cutover.json` в `artifacts/sorafs/`. Приложите эти файлы к волне
   приглашение, чтобы каждый рецензент мог проверить те же биты.2. **Псевдоним предварительного просмотра публикации:** Повторите команду без `--skip-submit`
   (укажите `TORII_URL`, `AUTHORITY`, `PRIVATE_KEY[_FILE]`, и псевдоним доказательства от управления).
   Скрипт привяжет манифест к `docs-preview.sora` и выдаст
   `portal.manifest.submit.summary.json` плюс `portal.pin.report.json` для пакета доказательств.

3. **Проба деплоя:** Убедитесь, что псевдоним разрешен и контрольная сумма соответствует тегу.
   перед отправкой приглашения.

   ```bash
   npm run probe:portal -- \
     --base-url=https://docs-preview.sora.link \
     --expect-release="$DOCS_RELEASE_TAG"
   ```

   Держите `npm run serve` (`scripts/serve-verified-preview.mjs`) под рукой как запасной вариант,
   чтобы ревьюеры могли поднять локальную реставрацию, если предварительный просмотр края даст сбой.

## Трекинг доступа и телеметрия

1. Запишите каждому получателю, отметку времени приглашения и отметки отзыва в журнале предварительного просмотра отзывов.
   (см. [`preview-feedback-log`](./preview-feedback-log)), чтобы тест волна разделяла
   один и тот же след доказательств:

   ```bash
   # Добавить новое событие приглашения в artifacts/docs_portal_preview/feedback_log.json
   npm run --prefix docs/portal preview:log -- \
     --wave preview-20250303 \
     --recipient alice@example.com \
     --event invite-sent \
     --notes "wave-01 seed"
   ```

   Поддерживаемые события: `invite-sent`, `acknowledged`, `feedback-submitted`,
   `issue-opened` и `access-revoked`. Лог по умолчанию находится в
   `artifacts/docs_portal_preview/feedback_log.json`; приложите его к тикету волн
   вместе с формами соглашения. Используйте вспомогательное резюме, чтобы подготовить аудируемый
   свернуть перед финальной заметкой:

   ```bash
   npm run --prefix docs/portal preview:summary -- --summary-json \
     > artifacts/docs_portal_preview/preview-20250303-summary.json
   ```

   Сводка JSON пересчитывает приглашения по волнам, открытые получатели, счетчики обратной связи
   и временная метка последних событий. Помощник основан на
   [`scripts/preview-feedback-log.mjs`](../../scripts/preview-feedback-log.mjs),
   поэтому тот же рабочий процесс можно запускать локально или в CI. Используйте дайджест шаблона в
   [`docs/examples/docs_preview_feedback_digest.md`](../../../examples/docs_preview_feedback_digest.md)
   при публикации резюме волн.
2. Тергите телеметрические приборные панели `DOCS_RELEASE_TAG`, использованные для волн, чтобы пики.
   можно было сопоставить приглашения когорты.
3. Запустите `npm run probe:portal -- --expect-release=<tag>` после развертывания, чтобы обеспечить надежность.
   что предварительный просмотр среды объявляет корректные метаданные выпуска.
4. Любые происшествия зафиксируйте в шаблоне Runbook и свяжите с когортой.

## Обратная связь и закрытие1. Собирайте отзывы в общем документе или доске задач. Маркируйте элементы `docs-preview/<wave>`,
   чтобы владельцы дорожной карты могли быстро их найти.
2. Используйте программу предварительного просмотра сводных выводов для получения отчета о волне, а затем запишите кого-то в
   `status.md` (участники, ключевые находки, Рекомендации) и обновите `roadmap.md`,
   если веха DOCS-SORA изменилась.
3. Следуйте шагам выключения из
   [`reviewer-onboarding`](./reviewer-onboarding.md): отзовите доступ, архивируйте заявки и
   Поблагодарите участников.
4. Подготовьте это решение, обновите документы, перезапустив вентили контрольной суммы и
   обновив шаблон приглашения с новыми датами.

Последовательное применение этого плейбука делает предварительный просмотр программы проверяемым и дает
Docs/DevRel повторяющийся способ масштабировать приглашения по мере приближения портала к GA.
