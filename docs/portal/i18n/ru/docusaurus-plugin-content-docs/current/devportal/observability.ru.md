#Наблюдаемость и аналитика портала

Дорожная карта DOCS-SORA требует аналитики, синтетических зондов и автоматизации.
проверяет битовые ссылки для каждой предварительной сборки. В этой заметке описана инфраструктура,
которое используется вместе с порталом, чтобы операторы могли подключить мониторинг
без уничтожения данных посетителей.

## Тегирование релиза

- Установите `DOCS_RELEASE_TAG=<identifier>` (резервный вариант `GIT_COMMIT` или `dev`) при сборке портала.
  Значение внедрено в `<meta name="sora-release">`, чтобы датчики и информационные панели могли различаться.
  развертывания.
- `npm run build` создает `build/release.json` (запись `scripts/write-checksums.mjs`), где описано
  тег, временная метка и опциональный `DOCS_RELEASE_SOURCE`. Этот файл упаковывается в превью-артефакты
  и упоминается в отчете проверки ссылок.

## Аналитика с сохранением приватности

- Настроить `DOCS_ANALYTICS_ENDPOINT=<https://collector.example/ingest>` для включения легкого трекера.
  Пейлоады содержат `{ event, path, locale, release, ts }` без реферера или метаданных IP, и при
  возможность используется `navigator.sendBeacon`, чтобы не блокировать навигацию.
- Управляйте выбором через `DOCS_ANALYTICS_SAMPLE_RATE` (0-1). Трекер хранит последний отправленный путь
  и никогда не отправляет дубликаты событий для одной навигации.
- Реализация находится в `src/components/AnalyticsTracker.jsx` и глобально монтируется через `src/theme/Root.js`.

## Синтетические пробы

- `npm run probe:portal` делает GET-запросы по типичным маршрутам (`/`, `/norito/overview`,
  `/reference/torii-swagger`, и т.д.) и наконец, что метатег `sora-release` соответствует
  `--expect-release` (или `DOCS_RELEASE_TAG`). Пример:

```bash
PORTAL_BASE_URL="https://docs.staging.sora" \
DOCS_RELEASE_TAG="preview-42" \
npm run probe:portal -- --expect-release=preview-42
```

Все появляется на пути, который приводит к успеху компакт-диска.

## Автоматизация битых ссылок

- `npm run check:links` сканирует `build/sitemap.xml`, убеждается, что уникальная запись мапится в локальный файл.
  (проверяя резервную версию `index.html`), и пишет `build/link-report.json`, последовательно метаданные релизы, итоги,
  ошибки и SHA-256 отпечаток `checksums.sha256` (выставлено как `manifest.id`), чтобы каждый отчет можно было
  было связано с манифестом документов.
- Скрипт завершается ненулевым кодом, когда страница отсутствует, поэтому CI может заблокировать релизы.
  при закрытых или сломанных маршрутах. Отчеты содержат кандидатские пути, которые пытались открыть, что
  помогает отслеживать регрессию маршрутизации к документам Tree.

## Дашборд Grafana и оповещения- `dashboards/grafana/docs_portal.json` публикует Grafana доску **Публикация на портале документации**.
  Она включает следующие панели:
  - *Отказы шлюза (5 м)* использует `torii_sorafs_gateway_refusals_total` в разрезе
    `profile`/`reason`, чтобы SRE могли найти плохие политики push или сбои токенов.
  - *Alias Cache Refresh Outcomes* и *Alias Proof Age p90* отслеживают `torii_sorafs_alias_cache_*`,
    Чтобы обеспечить наличие нового доказательства перед отключением DNS.
  - *Количество манифестов реестра контактов* и статистика *Количество активных псевдонимов* отражают невыполненный реестр контактов и общие сведения
    псевдоним числа, чтобы руководство могло аудировать каждый релиз.
  - *Срок действия TLS шлюза (часы)* подсвечивает приближение истечения шлюза публикации сертификата TLS
    (порог оповещения 72 ч).
  - *Результаты SLA репликации* и *Журнал репликации* следят за телеметрией
    `torii_sorafs_replication_*`, чтобы убедиться, что все реплики соответствуют GA после публикации.
- Используйте встроенные переменные шаблоны (`profile`, `reason`), чтобы сфокусироваться на
  публикация профиля `docs.sora` или обсуждение всплески по всем шлюзам.
- Маршрутизация PagerDuty использует панель дашборда в качестве доказательства: оповещения
  `DocsPortal/GatewayRefusals`, `DocsPortal/AliasCache` и `DocsPortal/TLSExpiry` срабатывают,
  когда эти слова звучат за порогом. Привяжите Runbook оповещение к этой странице,
  чтобы дежурные инженеры могли отображать точные запросы Prometheus.

## Сводим вместе

1. Во время `npm run build` установите переменные окружения Release/Analytics и дайте шаг после сборки.
   записать `checksums.sha256`, `release.json` и `link-report.json`.
2. Запустите `npm run probe:portal` против имени хоста предварительного просмотра с `--expect-release`, подключения с тем же тегом.
   Сохраните стандартный вывод для публикации чеклиста.
3. Запустите `npm run check:links`, чтобы быстро упасть на битых записях карты сайта и архивировать.
   сгенерированный отчет JSON вместе с документами предварительного просмотра. CI кладет последний отчет в
   `artifacts/docs_portal/link-report.json`, чтобы управление могло загрузить пакет доказательств прямо из журналов сборки.
4. Прокиньте конечную точку аналитики в свой сборщик данных, сохраняющий конфиденциальность (правдоподобный, самостоятельный прием OTEL и т.д.)
   и убедитесь, что частота дискретизации документирована для каждого релиза, чтобы информационные панели правильно интерпретировались.
   счетчики.
5. CI уже прокладывает эти шаги через рабочие процессы предварительного просмотра/развертывания.
   (`.github/workflows/docs-portal-preview.yml`,
   `.github/workflows/docs-portal-deploy.yml`), поэтому локальные пробные прогоны должны быть закрыты.
   только поведение, связанное с секретами.
