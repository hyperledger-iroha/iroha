# Ранбуки инцидентов и откат тренировок

## Назначение

Пункт дорожной карты **DOCS-9** требует прикладных сценариев и повторений плана, чтобы
операторы портала могли восстановиться после сбоев доставки без догадок. Эта заметка
следует три высокосигнальных инцидента - неудачные деплои, деградация репликации и
сбои аналитики - и документирует квартальные тренировки, доказывающие, что псевдоним отката
и синтетическая валидация продолжают работать от начала до конца.

### Связанные материалы

- [`devportal/deploy-guide`](./deploy-guide) — рабочий процесс доставки, загрузки и псевдоним продвижения.
- [`devportal/observability`](./observability) — теги релизов, аналитики и зондов, упомянутые ниже.
- `docs/source/sorafs_node_client_protocol.md`
  и [`sorafs/pin-registry-ops`](../sorafs/pin-registry-ops)
  — телеметрия реестра и пороги эскалации.
- `docs/portal/scripts/sorafs-pin-release.sh` и помощники `npm run probe:*`
  упомянуто в чеклистах.

### Общие телеметрия и инструменты

| Сигнал / Инструмент | Назначение |
| ------------- | ------- |
| `torii_sorafs_replication_sla_total` (встречен/пропущен/ожидается) | Выявляет остановку репликации и нарушение SLA. |
| `torii_sorafs_replication_backlog_total`, `torii_sorafs_replication_completion_latency_epochs` | Измеряет объем невыполненной работы и задержку для сортировки. |
| `torii_sorafs_gateway_refusals_total`, `torii_sorafs_manifest_submit_total{status="error"}` | Показаны сбои со стороны шлюза, которые часто приводят к ошибкам при развертывании. |
| `npm run probe:portal` / `npm run probe:tryit-proxy` | Синтетические зонды, которые гейт-релизы и проверяют откаты. |
| `npm run check:links` | Ворота битых ссылок; используется после каждого смягчения последствий. |
| `sorafs_cli manifest submit ... --alias-*` (через `scripts/sorafs-pin-release.sh`) | Механизм продвижения/возврата псевдонима. |
| Плата `Docs Portal Publishing` Grafana (`dashboards/grafana/docs_portal.json`) | Агрегирует отказы/псевдоним/TLS/репликацию телеметрии. Алерты PagerDuty ссылаются на эти панели как на доказательства. |

## Ранбук — Неудачный деплой или поврежденный архив

###Условия обеспечения

- Пробы предварительного просмотра/производства падают (`npm run probe:portal -- --expect-release=...`).
- оповещения Grafana на `torii_sorafs_gateway_refusals_total` или
  `torii_sorafs_manifest_submit_total{status="error"}` после развертывания.
- QA вручную замечает сломанные маршруты или свой прокси. Попробуйте сразу после
  псевдоним продвижения.

### Немедленное вооружение

1. **Заморозить деплой:** отметить конвейер CI `DEPLOY_FREEZE=1` (входной рабочий процесс GitHub)
   или приостановить работу Дженкинса, чтобы новые документы не выехали.
2. **Зафиксировать документы:** скачать `build/checksums.sha256`,
   `portal.manifest*.{json,to,bundle,sig}`, и вывести датчики неудачной сборки,
   чтобы откат ссылался на точные дайджесты.
3. **Уведомить стейкхолдеров:** SRE хранилища, ведущего специалиста по документации/разработчикам и ответственного за управление.
   (особенно если касается `docs.sora`).

### Причина отката

1. Определить последний известный удачный манифест (LKG). Рабочий процесс производства хранит их в
   `artifacts/devportal/<release>/sorafs/portal.manifest.to`.
2. Перепривязать псевдоним к этому манифесту с помощью помощника по доставке:

```bash
cd docs/portal
./scripts/sorafs-pin-release.sh \
  --build-dir build \
  --artifact-dir artifacts/revert-$(date +%Y%m%d%H%M) \
  --sorafs-dir artifacts/revert-$(date +%Y%m%d%H%M)/sorafs \
  --pin-min-replicas 5 \
  --alias "docs-prod-revert" \
  --alias-namespace "${PIN_ALIAS_NAMESPACE}" \
  --alias-name "${PIN_ALIAS_NAME}" \
  --alias-proof "${PIN_ALIAS_PROOF_PATH}" \
  --torii-url "${TORII_URL}" \
  --submitted-epoch "$(date +%Y%m%d)" \
  --authority "${AUTHORITY}" \
  --private-key "${PRIVATE_KEY}" \
  --skip-submit

# swap in the LKG artefacts before submission
cp /secure/archive/lkg/portal.manifest.to artifacts/.../sorafs/portal.manifest.to
cp /secure/archive/lkg/portal.manifest.bundle.json artifacts/.../sorafs/

cargo run -p sorafs_orchestrator --bin sorafs_cli -- \
  manifest submit \
  --manifest artifacts/.../sorafs/portal.manifest.to \
  --chunk-plan artifacts/.../sorafs/portal.plan.json \
  --torii-url "${TORII_URL}" \
  --authority "${AUTHORITY}" \
  --private-key "${PRIVATE_KEY}" \
  --alias-namespace "${PIN_ALIAS_NAMESPACE}" \
  --alias-name "${PIN_ALIAS_NAME}" \
  --alias-proof "${PIN_ALIAS_PROOF_PATH}" \
  --metadata rollback_from="${FAILED_RELEASE}" \
  --summary-out artifacts/.../sorafs/rollback.submit.json
```

3. Записать сводный откат в заявку об инциденте вместе с дайджестом LKG и неудачным манифестом.

### Проверка

1. `npm run probe:portal -- --expect-release=${LKG_TAG}`.
2. И18НИ00000040Х.
3. И18НИ00000041Х и И18НИ00000042Х
   (см. развертывание руководства), чтобы убедиться, что повторно продвигаемый манифест соответствует архивному CAR.
4. `npm run probe:tryit-proxy`, чтобы убедиться, что промежуточный прокси-сервер Try-It вернулся.

### После происшествия1. Включайте развертывание конвейера только после выяснения основной причины.
2. Дополнить раздел «Извлеченные уроки» в [`devportal/deploy-guide`](./deploy-guide)
   новые выводы, если есть.
3. Завести дефекты для провалившихся тестов (зонд, проверка ссылок и т.д.).

## Ранбук - Деградация репликации

###Условия обеспечения

- Оповещение: `sum(torii_sorafs_replication_sla_total{outcome="met"}) /
  зажим_мин(сумма(torii_sorafs_replication_sla_total{outcome=~"выполнено|пропущено"}), 1) <
  0,95` в течение 10 минут.
- `torii_sorafs_replication_backlog_total > 10` в течение 10 минут (см.
  `pin-registry-ops.md`).
- Правительство сообщает о псевдониме медленной доступности после освобождения.

### сортировка

1. Проверьте панели мониторинга [`sorafs/pin-registry-ops`](../sorafs/pin-registry-ops), чтобы
   понять, локализован ли отставание в классе хранения или в парке провайдеров.
2. Перепроверить логи Torii на `sorafs_registry::submit_manifest`, чтобы определить,
   не падают ли материалы.
3. Выборочно проверить здоровье реплика через `sorafs_cli manifest status --manifest ...`.
   (показывает исходы репликации провайдерам).

### Смягчение последствий

1. Перевыпустить манифест с более высоким числом реплик (`--pin-min-replicas 7`) через
   `scripts/sorafs-pin-release.sh`, чтобы планировщик распределил нагрузку на больший набор
   провайдеров. Зафиксировать новый дайджест в журнале инцидентов.
2. Если отставание постановлено к одному провайдеру, временно отключите его через планировщик репликации.
   (описано в `pin-registry-ops.md`) и отправьте новый манифест, принуждающий остальные
   провайдеров обновить псевдоним.
3. Когда свежесть псевдонима важнее репликации четности, повторно привяжите псевдоним к теплому манифесту уже в промежуточном состоянии.
   (`docs-preview`), а затем опубликовать последующий манифест после очистки невыполненной работы SRE.

### Восстановление и закрытие

1. Мониторить `torii_sorafs_replication_sla_total{outcome="missed"}` и убедиться, что
   счетчик стабилизировался.
2. Сохраните вывод `sorafs_cli manifest status` как свидетельство того, что нужно повторить реплику снова в норме.
3. Создать или обновить резервную копию репликации посмертно с дальнейшими шагами.
   (масштабирование провайдеров, настройка чанкера и т.д.).

## Ранбук - Отключение аналитики или телеметрии

###Условия обеспечения

- `npm run probe:portal` проходит, но информационные панели перестают принимать события
  `AnalyticsTracker` более чем на 15 минут.
- Проверка конфиденциальности фиксирует неожиданный рост пропущенных событий.
- `npm run probe:tryit-proxy` падает на путях `/probe/analytics`.

### Ответ

1. Проверьте входные данные времени сборки: `DOCS_ANALYTICS_ENDPOINT` и
   `DOCS_ANALYTICS_SAMPLE_RATE` в артефакте выпуска (`build/release.json`).
2. Перезапустить `npm run probe:portal` с `DOCS_ANALYTICS_ENDPOINT`, направленным
   на промежуточном сборщике, чтобы убедиться, что трекер продолжает отправлять полезные данные.
3. Если коллекторы отсутствуют, устанавливаем `DOCS_ANALYTICS_ENDPOINT=""` и пересобираем,
   чтобы трекер закоротил; зафиксировать сбой окна на временной шкале инцидента.
4. Проверьте, что `scripts/check-links.mjs` продолжает отпечаток пальца `checksums.sha256`.
   (аналитические сбои *не* должны блокировать проверку карты сайта).
5. После восстановления коллектора запустить `npm run test:widgets`, чтобы прогнать
   Помощник по аналитике модульных тестов перед повторной публикацией.

### После происшествия1. Обновить [`devportal/observability`](./observability) с новыми ограничениями.
   коллекционер или требует отбор проб.
2. Выпустить уведомление об управлении, если аналитика данных была утеряна или отредактирована.
   внеполитика.

## Квартальные учения по устойчивости

Запускайте оба учения в **первый вторник каждого квартала** (январь/апрель/июль/октябрь)
или сразу после любого крупного изменения. Храните документы в
`artifacts/devportal/drills/<YYYYMMDD>/`.

| Обучение | Шаги | Доказательства |
| ----- | ----- | -------- |
| Повторение отката псевдонима | 1. Повторить откат "Неудачное развертывание" с самого последнего производственного манифеста.2. Повторная привязка к производству после успешных проверок.3. Сохранить `portal.manifest.submit.summary.json` и лог щупов в дрель. | `rollback.submit.json`, вывод щупов и повторение тегов выпуска. |
| Аудит синтетической валидации | 1. Запустить `npm run probe:portal` и `npm run probe:tryit-proxy` против производства и постановки.2. Запустить `npm run check:links` и архивировать `build/link-report.json`.3. Приложить скриншоты/экспорт панелей Grafana, подтверждающих успех зондов. | Логи зондов + `link-report.json` со ссылкой на манифест отпечатков пальцев. |

Эскалируйте пропущенные упражнения с помощью Docs/DevRel и проверки управления SRE,
так как дорожная карта требует определенных квартальных доказательств, что откат псевдонима
и портальные зонды остаются здоровыми.

## PagerDuty и координация по вызову

- Сервис PagerDuty **Публикация портала документов** с оповещениями из
  `dashboards/grafana/docs_portal.json`. Правила `DocsPortal/GatewayRefusals`,
  `DocsPortal/AliasCache` и `DocsPortal/TLSExpiry` первичные документы/DevRel
  с хранилищем SRE как вторичное.
- При пейджинге укажите `DOCS_RELEASE_TAG`, приложите скриншоты приведенных
  панели Grafana и провода вывода датчика/проверки соединения в заметках об аварии до
  начало смягчения последствий.
- После устранения последствий (отката или повторного развертывания) повторно запустите `npm run probe:portal`,
  `npm run check:links` и сохраните свежие снимки Grafana, показывающие красный метрик
  в порогах. Приложите все доказательства происшествия с PagerDuty к закрытию.
- Если два оповещения срабатывают одновременно (например, истечение срока действия TLS и отставание), сначала
  сортировать отказы (остановить публикацию), выполнить откат, затем закрыть TLS/незавершенную работу
  с Склад СРЕ на мосту.
