# Поток приглашений предварительный просмотр

## Назначение

Пункт дорожной карты **DOCS-SORA** указывает на регистрацию рецензентов и приглашение общедоступной предварительной версии программы, как последние блокировщики перед выходом портала из бета-версии. На этой странице описано, как при открытии каждого приглашения, какие документы должны быть отправлены перед рассылкой и как найти, что поток аудируется. Используйте вместе с:

- [`devportal/reviewer-onboarding`](./reviewer-onboarding.md) для работы с каждым ревьюером.
- [`devportal/preview-integrity-plan`](./preview-integrity-plan.md) для контрольной суммы гарантии.
- [`devportal/observability`](./observability.md) для экспорта телеметрии и перехвата уведомлений.

Документируйте активное событие в `status.md` и в трекере запросов предварительного просмотра, чтобы управление видело статус с первого взгляда.

## Предполетный чеклист

Выполните эти действия **перед** планированием приглашений для волн:

1. **Артефакты CI доступны**
   - Последний `docs-portal-preview` + загружен дескриптор `.github/workflows/docs-portal-preview.yml`.
   - Вывод SoraFS отмечен в `docs/portal/docs/devportal/deploy-guide.md` (переключение дескриптора присутствует).
2. **Принудительная контрольная сумма**
   - `docs/portal/scripts/serve-verified-preview.mjs` возникает через `npm run serve`.
   - Инструкции `scripts/preview_verify.sh` протестированы на macOS + Linux.
3. **Базовая телеметрия**
   - `dashboards/grafana/docs_portal.json` показывает здоровый трафик Попробуйте, и алерт `docs.preview.integrity` зеленый.
   - Последнее приложение в `docs/portal/docs/devportal/observability.md` обновлено ссылками Grafana.
4. **Управление артефактами**
   - Трекер приглашений готов (одна проблема для волнений).
   - Шаблон реестра ревьюеров скопирован (см. [`docs/examples/docs_preview_request_template.md`](../../../examples/docs_preview_request_template.md)).
   - Юридические и согласования СРИ приложены к вопросу.

Зафиксируйте завершение предполетной проверки в трекере приглашений перед отправкой писем.

## Шаги потока1. **Выбор кандидатов**
   - Взять из таблицы списка ожидания или партнерской очереди.
   - Убедиться, что у каждого кандидата есть шаблон запроса.
2. **Одобрение доступа**
   - Назначить утверждающего на трекере приглашений на выдачу.
   - Проверить требования (CLA/контракт, приемлемое использование, краткий обзор безопасности).
3. **Отправка приглашений**
   - Заполнить плейсхолдеры в [`docs/examples/docs_preview_invite_template.md`](../../../examples/docs_preview_invite_template.md) (`<preview_tag>`, `<request_ticket>`, контакты).
   - Приложить дескриптор + хэш архива, попробовать промежуточный URL и поддержку поддержки.
   - Сохранить сохраненное письмо (или расшифровку Matrix/Slack) в выпуске.
4. **Отслеживание адаптации**
   - Обновить инвайт-трекер полями `invite_sent_at`, `expected_exit_at`, и статусом (`pending`, `active`, `complete`, `revoked`).
   - Привязать запрос на прием ревьюера для аудитабельности.
5. **Мониторинг телеметрии**
   - Следить за `docs.preview.session_active` и алертами `TryItProxyErrors`.
   - Завести инцидент при отклонении телеметрии от исходного уровня и записать результат рядом с записью приглашения.
6. **Сбор фидбека и выход**
   - Закрыть приглашение после получения фидбека или наступления `expected_exit_at`.
   - Обновить выпуск волн кратким резюме (находки, инциденты, последовательные шаги) перед переходом к такому-когорте.

## Доказательства и отчетность

| Артефакт | Где хранить | Частота обновлений |
| --- | --- | --- |
| Трекер приглашений | Проект GitHub `docs-portal-preview` | Обновить после каждого приглашения. |
| Экспортный список ревьюеров | Реестр, связанный в `docs/portal/docs/devportal/reviewer-onboarding.md` | Еженедельно. |
| Фотографии телеметрии | `docs/source/sdk/android/readiness/dashboards/<date>/` (переиспользовать пакет телеметрии) | На все размышления + после инцидентов. |
| Дайджест отзывов | `docs/portal/docs/devportal/preview-feedback/<wave>/summary.md` (создать документ на волнение) | В течение 5 дней после выхода из волны. |
| Записка о заседании правления | `docs/portal/docs/devportal/preview-invite-notes/<date>.md` | Заполнить до каждого синхронизацию управления DOCS-SORA. |

Запускайте `cargo xtask docs-preview summary --wave <wave_label> --json artifacts/docs_portal_preview/<wave_label>_summary.json`
после каждого батча, чтобы получить машинно-читательный дайджест. Прикрепите рендеренный JSON для создания волн, чтобы ревьюеры могли управлять количеством приглашений без записи всей лога.

Прикрепляйте список доказательств к `status.md` при завершении каждой волны, чтобы дорожную карту можно было быстро обновить.

## Критерии отката и паузы

Приостановите поток приглашений (и уведомите управление), если происходит что-либо из следующего:

- Инцидент Попробуйте прокси, потребовавший откат (`npm run manage:tryit-proxy`).
- Усталость от оповещений: >3 страниц оповещений для конечных точек, доступных только для предварительного просмотра, в течение 7 дней.
- Пробел комплаенса: приглашение отправлено без подписанных условий или без записи шаблона запроса.
- Риск шоколад: несоответствие контрольной суммы, обнаруженный `scripts/preview_verify.sh`.

Возобновляйте только после документирования исправлений в трекере приглашений и подтверждения стабильности телеметрии не менее 48 часов.
