# Онбординг ревьюеров превью

## Обзор

DOCS-SORA отслеживает поэтапный запуск портала разработчика. Сборки с гейтом по контрольной сумме
(`npm run serve`) и усиленные потоки Попробуйте открыть следующий этап: онбординг проверенных
ревьюеров до широкого открытия публичного предварительного просмотра. В этом руководстве описывается, как собирать заявки,
проверить соответствие, предоставить доступ и безопасно завершить участие. См.
[предварительный поток приглашений](./preview-invite-flow.md) для планирования когорт, каденций приглашений
и экспорт телеметрии; Шаги ниже фокусируются на действиях после выбора ревьюера.

- **В рамках:** ревьюеры, которым нужен доступ к предварительной документации (`docs-preview.sora`,
  сборка GitHub Pages или бандлы SoraFS) до GA.
- **Вне рамок:** операторы Torii или SoraFS (покрыться собственными бортовыми наборами) и
  продакшн-развертывания портала (см.
  [`devportal/deploy-guide`](./deploy-guide.md)).

## Роли и требования

| Роль | Типичные цели | Требуемые документы | Примечания |
| --- | --- | --- | --- |
| Основной сопровождающий | Посмотреть новые гайды, запустить дымовые тесты. | Дескриптор GitHub, контакт в Matrix, подписанная CLA. | уже обычно в службе GitHub `docs-preview`; все равно подайте заявку, чтобы доступ был аудируемым. |
| Партнер-рецензент | Просмотрите SDK-фрагменты или контент для управления до публичного релиза. | Корпоративный адрес электронной почты, юридический POC, подписанные условия предварительного просмотра. | Должен оказывать финансовую помощь в области телеметрии и обработки данных. |
| Общественный волонтер | Дать отзывы по удобствам использования гайдов. | Дескриптор GitHub, предпочитаемый контакт, часовой пояс, соглашение с CoC. | держите когорты малых; приоритет ревьюерам, подписавшим соглашение вкладчика. |

Все виды ревьюеров должны:

1. Подтвердите политику допустимого использования предпросмотр-артефактов.
2. Прочитать приложения по безопасности/наблюдаемости
   ([`security-hardening`](./security-hardening.md),
   [`observability`](./observability.md),
   [`incident-runbooks`](./incident-runbooks.md)).
3. Согласиться запустить `docs/portal/scripts/preview_verify.sh` перед темой, как
   обслуживать любой локальный снимок.

## Процесс приема

1. Попросите заявителя предъявить обвинение.
   [`docs/examples/docs_preview_request_template.md`](../../../examples/docs_preview_request_template.md)
   сформировать (или вставить ее в выпуск). Зафиксируйте как минимум: личность, способ связи,
   Дескриптор GitHub, приведенные даты проверки и подтверждение чтения документов безопасности.
2. Зафиксируйте заявку в трекере `docs-preview` (GitHub Issue или тикет управления)
   и назначьте утверждающего.
3. Проверьте требования:
   - CLA/контракт контрибьютора в наличии (или ссылка на партнерский договор).
   - Подтверждение допустимого использования хранилища в заявке.
   - Риск-оценка завершена (например, партнерские ревьюеры одобрения Legal).
4. Утверждающий подписывает заявку и связывает проблему отслеживания с любой записью.
   управление изменениями (пример: `DOCS-SORA-Preview-####`).

## Продовольствие и инструменты

1. **Передать архивы** — Предоставьте последний дескриптор предварительного просмотра + архив из
   Рабочий процесс CI или вывод SoraFS (артефакт `docs-portal-preview`). Напомните ревьюерам запустить:

   ```bash
   ./docs/portal/scripts/preview_verify.sh \
     --build-dir build \
     --descriptor artifacts/preview-descriptor.json \
     --archive artifacts/preview-site.tar.gz
   ```2. **Обслуживать с контрольной суммой** — Укажите команду с воротами по контрольной сумме:

   ```bash
   DOCS_RELEASE_TAG=preview-<stamp> npm run --prefix docs/portal serve
   ```

   Это переиспользует `scripts/serve-verified-preview.mjs`, чтобы неподтвержденный билд
   не запустился случайно.

3. **Дать доступ к GitHub (опционально)** — если нужны неопубликованные ветки, страницы.
   ревьюеров в команде GitHub `docs-preview` на период ревью и зафиксируйте изменения членства
   в заявке.

4. **Коммуникация каналов поддержки** — Поделитесь контактом по вызову (Matrix/Slack) и
   процедурой инцидентов из [`incident-runbooks`](./incident-runbooks.md).

5. **Телеметрия + обратная связь** — Напомните, что собирается анонимизированная аналитика
   (см. [`observability`](./observability.md)). Дайте форму обратной связи или вопрос-шаблон,
   указанный в приглашении, и зафиксируйте событие через помощника
   [`preview-feedback-log`](./preview-feedback-log), чтобы вывести волну продолжительности актуальной.

## Чеклист ревьюера

Перед доступом к предварительному просмотру ревьюеры должны настроить:

1. Скачанные проверенные архивы (`preview_verify.sh`).
2. Запустить портал через `npm run serve` (или `serve:verified`), чтобы контрольная сумма была активирована.
3. Прочитайте упомянутые выше заметки безопасности и наблюдаемости.
4. Проверьте OAuth/попробуйте консоль через вход в систему по коду устройства (если применимо) и не переиспользовать.
   производство токенов.
5. Фиксировать поступления в согласованном трекере (выпуск, общий документ или форму) и тегировать
   Предварительный просмотр их релизным тегом.

## Ответственность мейнтейнеров и оффбординг

| Фаза | Действия |
| --- | --- |
| Начало | Убедиться, что контрольный список приема приложен к заявке, поделиться сертификатами + сильными, добавить запись `invite-sent` через [`preview-feedback-log`](./preview-feedback-log) и запланировать промежуточную синхронизацию, если обзор продлится более недели. |
| Мониторинг | Отследить предварительный просмотр телеметрии (необычный трафик Попробуйте, сбои зонд) и победите инцидент-ранбуку при подозрениях. Логгировать события `feedback-submitted`/`issue-opened` по мере поступления находок, чтобы метрики волн были самыми точными. |
| Оффбординг | Вызовите временный доступ GitHub или SoraFS, запишите `access-revoked`, заархивируйте заявку (включите сводную обратную связь + открытые действия) и обновите список ревьюеров. Попросить ревьюера удалить локальные сборки и приложить дайджест из [`docs/examples/docs_preview_feedback_digest.md`](../../../examples/docs_preview_feedback_digest.md). |

Используйте тот же процесс при ротации ревьюеров между волнами. Сохранение наблюдения в репозиториях
(выпуск + шаблоны) помогает DOCS-SORA стать аудируемым и обеспечивает надежное управление,
что доступ к предварительному просмотру отслеживался документированным контролем.
