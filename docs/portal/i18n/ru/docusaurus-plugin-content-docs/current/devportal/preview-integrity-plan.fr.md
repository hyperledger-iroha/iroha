# План предварительной визуализации по контрольной сумме

Ce plan decrit le travail restant necessaire pour rendre chaque артефакт де предварительной визуализации du portail, поддающийся проверке до публикации. Объект является гарантией того, что отражатели телезарядны для создания моментального снимка в CI, что манифест контрольной суммы является неизменяемым и что предварительная визуализация может быть удалена с помощью SoraFS с метадонами Norito.

## Цели

- **Создает определенные параметры:** Удостоверьтесь, что `npm run build` производит воспроизводимую вылазку и получает возможность использовать `build/checksums.sha256`.
- **Проверенные предварительные визуализации:** Выясните, что артефакт предварительной визуализации является манифестом контрольной суммы и отказывается от публикации или отголоска проверки.
- **Метадоны, публикуемые через Norito:** Сохранять описания предварительной визуализации (метадонны фиксации, дайджест контрольной суммы, CID SoraFS) в JSON Norito до тех пор, пока вы не сможете контролировать выпуски.
- **Оператор Outillage:** Создать сценарий проверки на этапе, который может быть использован локальным исполнителем (`./docs/portal/scripts/preview_verify.sh --build-dir build --descriptor <path> --archive <path>`); Сценарий инкапсулирует десормирующий поток контрольной суммы + дескриптор боя в бою. Стандартная команда предварительной визуализации (`npm run serve`) вызывает команду обслуживания этого помощника по автоматизации до `docusaurus serve`, пока снимки остаются в локальном состоянии под контролем контрольной суммы (Avec `npm run serve:verified` сохраняет явный псевдоним).

## Этап 1. CI приложения

1. Mettre a Jour `.github/workflows/docs-portal-preview.yml` залейте:
   - Исполнитель `node docs/portal/scripts/write-checksums.mjs` после сборки Docusaurus (локальный вызов вызова).
   - Исполнитель `cd build && sha256sum -c checksums.sha256` и повторяет задание в случае расхождения.
   - Вставьте репертуар в `artifacts/preview-site.tar.gz`, скопируйте манифест контрольной суммы, выполните `scripts/generate-preview-descriptor.mjs` и исполнитель `scripts/sorafs-package-preview.sh` с конфигурацией JSON (предварительно `docs/examples/sorafs_preview_publish.json`), чтобы рабочий процесс был выполнен в соответствии с метадонами и др. пакет SoraFS определен.
   - Телеверсировать статический сайт, артефакты метадонов (`docs-portal-preview`, `docs-portal-preview-metadata`) и пакет SoraFS (`docs-portal-preview-sorafs`) для того, чтобы манифестировать, возобновить CAR и план puissent etre проверяет без повтора ле строить.
2. Добавьте комментарий к значку CI, который возобновит результат проверки контрольной суммы в запросах на включение (реализуется через сценарий GitHub с комментариями `docs-portal-preview.yml`).
3. Документируйте рабочий процесс в `docs/portal/README.md` (раздел CI) и этапы проверки в контрольном списке публикации.

## Скрипт проверки

`docs/portal/scripts/preview_verify.sh` действительны для артефактов предварительной визуализации без необходимости ручного вызова `sha256sum`. Используйте `npm run serve` (или явный псевдоним `npm run serve:verified`) для выполнения сценария и lancer `docusaurus serve` на одном этапе, пока вы разделяете локальные снимки. Логика проверки:1. Выполните соответствующий инструмент SHA (`sha256sum` или `shasum -a 256`) против `build/checksums.sha256`.
2. Сравните вариант дайджеста/имя документа описания предварительной визуализации `checksums_manifest` и, затем, иллюстрацию/имя файла архива предварительной визуализации.
3. Сортировка с кодом, не имеющим нулевых отклонений, - это обнаружение того, что отражатели могут блокировать измененные предвизуализации.

Пример использования (после извлечения артефактов CI):

```bash
./docs/portal/scripts/preview_verify.sh \
  --build-dir build \
  --descriptor artifacts/preview-descriptor.json \
  --archive artifacts/preview-site.tar.gz
```

Инженеры CI и релиз doivent doivent appeler le script chaque fois qu'ils telechargen un Bundle de Previsualization или Attachment des Artefacts и билет на выпуск.

## Этап 2 — Публикация SoraFS

1. Создайте рабочий процесс предварительной визуализации с учетом работы, которая:
   - Сайт Televerse построен на основе промежуточного прохода SoraFS с использованием `sorafs_cli car pack` и `manifest submit`.
   - Запишите дайджест отправленного манифеста и CID SoraFS.
   — Сериализировать `{ commit, branch, checksum_manifest, cid }` в JSON Norito (`docs/portal/preview/preview_descriptor.json`).
2. Сохраните описание с артефактом сборки и выставьте CID в комментариях к запросу на вытягивание.
3. Проведите тесты интеграции, которые выполняются `sorafs_cli` в режиме пробного прогона, чтобы гарантировать, что будущие эволюции сохранят согласованность схемы метадоний.

## Этап 3 – Управление и аудит

1. Опубликована схема Norito (`PreviewDescriptorV1`), содержащая структуру описания, соответствующую `docs/portal/schemas/`.
2. Ознакомьтесь с контрольным списком публикации DOCS-SORA для выполнения:
   - Lancer `sorafs_cli manifest verify` за плату CID.
   - Регистратор дайджеста манифеста контрольной суммы и CID в описании PR-релиза.
3. Соединитель автоматизации управления для проверки дескриптора с подвеской манифеста контрольной суммы для голосов за выпуск.
