# Заметки о переходе Nexus

Эта регистрация отслеживает оставшуюся работу **Фаза B — Nexus Transition Foundations** до выполнения контрольного списка многополосного запуска. Он выполняет контрольную запись в `roadmap.md` и содержит доказательства, на которые ссылаются B1-B4, в одном месте, для управления, SRE и лиды SDK, которые вы можете поделиться единым доказательством истины.

## Область и частота кадров

- Описаны аудиты трассировки маршрутов и ограждения телеметрии (B1/B2), набор дельта-конфигураций, одобренное управление (B3) и последующие действия после репетиции многополосного запуска (B4).
- Заменяет временную каденцию, которая была здесь раньше; с аудита за первый квартал 2026 года подробный отчет хранится в `docs/source/nexus_routed_trace_audit_report_2026q1.md`, и эта страница ведет рабочий график и реестр смягчений.
- Обновляйте таблицы после каждого окна трассировки, голосования управления или репетиции запуска. Когда данные переносятся, отразите здесь новый путь, чтобы нисходящие документы (статус, информационные панели, порталы SDK) могли ссылаться на стабильный уровень.

## Интеграция дельта-пакета конфигурации- `docs/source/project_tracker/nexus_config_deltas/2026Q1.md` сохраняет стандартное резюме различий. Когда появятся новые `defaults/nexus/*.toml` или изменения Genesis, сначала обновите трекер, а затем отобразите ключевые пункты здесь.
- Подписанные конфиг-пакеты питают репетиционный пакет телеметрии. Пакет, валидированный `scripts/telemetry/validate_nexus_telemetry_pack.py`, должен быть опубликован вместе с доказательствами дельта-конфигурации, чтобы операторы могли воспроизвести точные артефакты, использованные в B4.
- Bundles Iroha 2 остаются без полос: конфиги с `nexus.enabled = false` теперь отклоняют переопределения полосы/dataspace/маршрутизации, если не включен профиль Nexus (`--sora`), поэтому удалите секцию `nexus.*` из однополосных шаблонов.
- Держите регистрацию голосования управления (GOV-2026-03-19) переключения и повторения с трекером, а также с этой заметкой, чтобы будущие голосовые могли копировать формат без завершения ритуала одобрения.

## Последующие действия после репетиции запуска

- `docs/source/runbooks/nexus_multilane_rehearsal.md` фиксирует канареечный план, список участников и откат шагов; обновляйте runbook при изменении топологии дорожек или экспортера телеметрии.
- `docs/source/project_tracker/nexus_rehearsal_2026q1.md` пересылает все документы, проверенные на повторениях 9 апреля, и теперь включает примечания/повестку дня по подготовке ко второму кварталу. Добавляйте будущие повторы в тот же трекер вместо одноразовых трекеров, чтобы доказательства были монотонными.
- Публикуйте фрагменты сборщика OTLP и экспорт Grafana (см. `docs/source/telemetry.md`) при поддержке экспортера руководства по пакетной обработке; Обновление Q1 подняло размер пакета до 256 образцов, чтобы предотвратить оповещения о запасе.
- Многополосные доказательства CI/tests теперь живут в `integration_tests/tests/nexus/multilane_pipeline.rs` и запускаются через рабочий процесс `Nexus Multilane Pipeline` (`.github/workflows/integration_tests_multilane.yml`), заменяя конфиденциальную ссылку `pytests/nexus/test_multilane_pipeline.py`; держите хеш `defaults/nexus/config.toml` (`nexus.enabled = true`, blake2b `d69eefa2abb8886b0f3e280e88fe307a907cfe88053b5d60a1d459a5cf8549e1`) синхронизированным с трекером при обновлении репетиционных пакетов.

## Жизненный цикл полосы выполнения

- Планы жизненного цикла полосы времени выполнения теперь валидируют привязки к пространству данных и прерываются при собственном согласовании Kura/многоуровневого хранилища, о внесении каталога без изменений. Помощники очищали кэшированные реле полос для удаленных полос, чтобы синтез слияния реестра не переиспользовал конфиденциальные доказательства.
- Применяйте через планы Nexus config/lifecycle helpers (`State::apply_lane_lifecycle`, `Queue::apply_lane_lifecycle`) для добавления/вывода дорожек без рестарта; маршрутизация, снимки TEU и реестры манифестов загружаются автоматически после получения плана.
- Руководство оператора: при сбое плана проверить отсутствие пространств данных или корней хранилищ, которые не могут быть созданы (многоуровневые холодные корневые каталоги/каталоги Kura Lane). Исправьте базовые пути и повторите; Успешные планы повторно эмитируют различия в полосе телеметрии/пространстве данных, чтобы информационные панели отражали новую топологию.

## NPoS телеметрия и доказательства противодавления

Ретро Репетиция запуска Фазы Б запросило определенные телеметрические снимки, подтверждающие, что кардиостимулятор NPoS и слухи продолжают оставаться в пределах противодавления. Интеграционный жгут в `integration_tests/tests/sumeragi_npos_performance.rs` прогоняет эти скрипты и выводит сводки JSON (`sumeragi_baseline_summary::<scenario>::...`) при появлении новых метрик. Запуск локально:```bash
cargo test -p integration_tests sumeragi_npos_performance -- --nocapture
```

Установите `SUMERAGI_NPOS_STRESS_PEERS`, `SUMERAGI_NPOS_STRESS_COLLECTORS_K` или `SUMERAGI_NPOS_STRESS_REDUNDANT_SEND_R`, чтобы учитывать более стрессовые топологии; значение по умолчанию отражают профиль 1 s/`k=3`, использованный в B4.

| Сценарий/тест | Покрытие | Ключевая телеметрия |
| --- | --- | --- |
| `npos_baseline_1s_k3_captures_metrics` | Блокирует 12 раундов с блоком репетиции, чтобы записать конверты задержки EMA, внешние очереди и датчики избыточной отправки перед сериализацией пакета доказательств. | `sumeragi_phase_latency_ema_ms`, `sumeragi_collectors_k`, `sumeragi_redundant_send_r`, `sumeragi_bg_post_queue_depth*`. |
| `npos_queue_backpressure_triggers_metrics` | Перерабатывает очередь транзакций, чтобы обеспечить определенный запуск отсрочки приема и экспортных счетчиков мощности/насыщения. | `sumeragi_tx_queue_depth`, `sumeragi_tx_queue_capacity`, `sumeragi_tx_queue_saturated`, `sumeragi_pacemaker_backpressure_deferrals_total`, `sumeragi_rbc_backpressure_deferrals_total`. |
| `npos_pacemaker_jitter_within_band` | Сэмплирует дрожание кардиостимулятора и просматривает тайм-ауты, пока не будет проверено соблюдение полосы +/- 125 промилле. | `sumeragi_pacemaker_jitter_ms`, `sumeragi_pacemaker_view_timeout_target_ms`, `sumeragi_pacemaker_jitter_frac_permille`. |
| `npos_rbc_store_backpressure_records_metrics` | Проталкивает крупные полезные нагрузки RBC в хранилище мягких/жестких лимитов, чтобы показать рост, откат и стабилизацию счетчиков сеансов/байтов без переполнения хранилища. | `sumeragi_rbc_store_pressure`, `sumeragi_rbc_store_sessions`, `sumeragi_rbc_store_bytes`, `sumeragi_rbc_backpressure_deferrals_total`. |
| `npos_redundant_send_retries_update_metrics` | Форсирует ретрансмиты, чтобы датчики коэффициента избыточных отправок и счетчики целевых сборщиков продвигались, доказывая сквозную связность требуемой телеметрии. | `sumeragi_collectors_targeted_current`, `sumeragi_redundant_sends_total`. |
| `npos_rbc_chunk_loss_fault_reports_backlog` | Детерминированно дропает куски, чтобы мониторы отставания поднимали неисправности вместо тихого дренажа полезной нагрузки. | И18НИ00000105Х, И18НИ00000106Х, И18НИ00000107Х. |

Приложите линию JSON, которая выводит жгут вместе с Prometheus Scrape, захваченным во время запуска, каждый раз, когда управление доказывает доказательства того, что сигналы тревоги противодавления соответствуют репетиционной топологии.

## Обновления Чеклиста

1. Добавьте новые окна с трассировкой и перенесите старые по мере смены кварталов.
2. Обновляйте таблицу митигаций после каждого последующего действия Alertmanager, даже если действие - закрыть тикет.
3. Когда дельты конфигурации изменяются, обновляйте трекер, эта заметка и список собирают пакет телеметрии в один пул-запрос.
4. Ссылайтесь сюда на новые репетиционные/телеметрические документы, чтобы будущие обновления дорожной карты ссылались на один документ, а не разрозненные специальные заметки.
