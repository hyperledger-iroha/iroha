# План рефакторизации реестра Sora Nexus

Этот документ содержит план немедленного рефакторинга Sora Nexus Ledger («Iroha 3»). Он отразит актуальную топологию хранилища и регрессионные наблюдения в совместимом генезисе/WSV, консенсус Sumeragi, триггеры смарт-контрактов, запросы моментальных снимков, привязки указателя хоста-ABI и кодеки Norito. L'objectif - это конвергентная архитектура, согласованная и проверяемая без ограничений, включая все исправления в монолитном патче.

## 0. Принципы режиссеров
- Сохранение детерминированного соответствия гетерогенных материалов; использовать уникальное ускорение с помощью флагов функций, подписавшись на резервные идентификаторы.
- Norito - это диван для сериализации. Все изменения в государстве/схеме включают в себя тесты двустороннего кодирования/декодирования Norito и другие приспособления в течение дня.
- Конфигурация проходит по параметру `iroha_config` (пользователь -> фактические -> настройки по умолчанию). Дополнительная информация о переключении окружающей среды в зависимости от пути производства.
- La politique ABI reste V1 и не подлежит обсуждению. Хосты не могут отклонить определение типов указателей/неправильных системных вызовов.
- `cargo test --workspace` и золотые тесты (`ivm`, `norito`, `integration_tests`) оставляют ворота базы для желтого цвета.

## 1. Снимок топологии склада
- `crates/iroha_core`: акторы Sumeragi, WSV, генезис загрузчика, конвейеры (запрос, наложение, полосы zk), связующий хост смарт-контрактов.
- `crates/iroha_data_model`: авторизованная схема для пользователей и запросов в цепочке.
- `crates/iroha`: API-клиент, использующий CLI, тесты, SDK.
- `crates/iroha_cli`: оператор CLI, отображающий актуальные номера API в `iroha`.
- `crates/ivm`: байт-код Kotodama виртуальной машины, точки входа в указатель интеграции — ABI хоста.
- `crates/norito`: кодек сериализации с адаптерами JSON и серверными модулями AoS/NCB.
- `integration_tests`: утверждения межкомпонентного согласования генезиса/начальной загрузки, Sumeragi, триггеры, нумерация страниц и т. д.
- Документы описывают объекты Sora Nexus Ledger (`nexus.md`, `new_pipeline.md`, `ivm.md`), но реализация является фрагментарной и частично устаревшей по отношению к коду.

## 2. Методы рефакторинга и др.### Фаза А – Фонды и наблюдаемые
1. **Телеметрия WSV + снимки**
   — Добавлена таблица канонического API снимков в `state` (тип `WorldStateSnapshot`), использующая запросы номиналов, Sumeragi и CLI.
   - Утилизатор `scripts/iroha_state_dump.sh` для создания снимков определяется через `iroha state dump --format norito`.
2. **Детерминизм происхождения/бутстрап**
   - Рефакторизация генезиса приема для передачи по уникальной базе конвейера для Norito (`iroha_core::genesis`).
   - Добавлена ​​возможность интеграции/регрессии, которая обновила генезис плюс первый блок и проверила идентификаторы рас WSV в Arm64/x86_64 (suivi dans `integration_tests/tests/genesis_replay_determinism.rs`).
3. **Испытания на фиксацию поперечного ящика**
   - Etendre `integration_tests/tests/genesis_json.rs` для проверки инвариантов WSV, конвейера и ABI в собственном жгуте.
   - Представляем эшафот `cargo xtask check-shape`, который вызывает панику из-за дрейфа схемы (находится в невыполненной работе с инструментами DevEx; отображается элемент действия в `scripts/xtask/README.md`).

### Фаза B – WSV и требуемые поверхности
1. **Транзакции государственного хранилища**
   - Свернуть `state/storage_transactions.rs` в адаптере транзакций, который применяет порядок фиксации и обнаружения конфликтов.
   - Модульные тесты проверяют, что изменения актива/мира/триггеров отката шрифта при проверке.
2. **Рефакторинг запрашиваемых моделей**
   - Заменить логику нумерации страниц/курсор в повторно используемых компонентах под `crates/iroha_core/src/query/`. Выровняйте представления Norito и `iroha_data_model`.
   - Добавление запросов моментальных снимков для триггеров, ресурсов и ролей с определенным порядком (выполняется через `crates/iroha_core/tests/snapshot_iterable.rs` для актуального изменения).
3. **Согласованность снимков**
   - Убедитесь, что CLI `iroha ledger query` использует мем для создания моментального снимка, который Sumeragi/fetchers.
   - Регрессионные тесты CLI для моментальных снимков выполняются в соответствии с `tests/cli/state_snapshot.rs` (функция-зависимая для прогонов).

### Фаза C — Трубопровод Sumeragi
1. **Топология и развитие эпох**
   - Дополнительный `EpochRosterProvider`, содержащий реализации, основанные на моментальных снимках ставки WSV.
   - `WsvEpochRosterAdapter::from_peer_iter` представляет собой простой и удобный конструктор для имитации стендов и тестов.
2. **Упрощение потока консенсуса**
   - Реорганизатор `crates/iroha_core/src/sumeragi/*` в модулях: `pacemaker`, `aggregation`, `availability`, `witness` с типами частей, такими как `consensus`.
   - Заменитель сообщений, передающих специальные типы конвертов Norito, и введение тестов свойств изменения представления (suivi dans le backlog messages Sumeragi).
3. **Интеграция/доказательство**
   - Выравнивание доказательств полосы движения с участием DA и гарантия единой формы RBC.
   - Комплексный тест интеграции `integration_tests/tests/extra_functional/seven_peer_consistency.rs` проверяет обслуживание процесса с активным RBC.### Фаза D — Смарт-контракты и указатель хостов-ABI
1. **Аудит де ла Фронтьер**
   - Объедините проверки типа указателя (`ivm::pointer_abi`) и адаптеры хоста (`iroha_core::smartcontracts::ivm::host`).
   - Внимание к таблице указателей и привязки манифеста хоста соответствуют `crates/iroha_core/tests/ivm_pointer_abi_tlv_types.rs` и `ivm_host_mapping.rs`, которые выполняют сопоставления TLV gold.
2. **Песочница для выполнения триггеров**
   - Рефакторизация триггеров для передачи по `TriggerExecutor`, обеспечивающей применение газа, проверку указателей и ведение журнала событий.
   - Добавлены тесты регрессии для триггеров вызова/времени, связанных с процессами проверки (выполняется через `crates/iroha_core/tests/trigger_failure.rs`).
3. **Согласование интерфейса командной строки и клиента**
   - Убедитесь, что операции CLI (`audit`, `gov`, `sumeragi`, `ivm`) хранятся в функциях клиента, участвующих в `iroha`, чтобы избежать дрейфа.
   - Тесты снимков JSON du CLI vivent в `tests/cli/json_snapshot.rs`; ожидайте, что ядро ​​продолжит соответствовать каноническому стандартному JSON.

### Фаза E — Durcissment du codec Norito
1. **Реестр схем**
   - Создайте реестр схем Norito с помощью `crates/norito/src/schema/` для источника кодирования канонических типов ядра.
   - Документ проверяет кодировку примера полезной нагрузки (`norito::schema::SamplePayload`).
2. **Обновление золотых светильников**
   - В течение дня золотые светильники `crates/norito/tests/*` для соответствия новой схеме WSV для проведения рефакторинга книги.
   - `scripts/norito_regen.sh` регенерирует золотой JSON Norito детерминированного факона с помощью помощника `norito_regen_goldens`.
3. **Интеграция IVM/Norito**
   - Поддерживается сквозная сериализация манифестов Kotodama через Norito, и гарантируется согласованный указатель метаданных ABI.
   - `crates/ivm/tests/manifest_roundtrip.rs` поддерживает парный Norito кодирование/декодирование манифестов.

## 3. Трансверсальные занятия
- **Стратегия тестирования**: модульные тесты поэтапного выполнения -> тесты ящиков -> интеграционные тесты. Тесты и проверка текущих регрессий; «Les Nouveaux» тестирует «Evitent Leur Retour».
- **Документация**: после завершения фазы, в течение дня `status.md` и отчет о пунктах, отображаемых в `roadmap.md`, рекламирует конечные точки.
- **Оценки производительности**: поддержание существующих стендов в `iroha_core`, `ivm` и `norito`; дополнительные базовые меры после рефакторинга для подтверждения отсутствия регрессий.
- **Флаги функций**: Сохранение переключателей или уникальных ящиков для бэкэндов, требующих внешних цепочек инструментов (`cuda`, `zk-verify-batch`). Les chemins SIMD CPU, которые строятся и выбираются для выполнения; Скалярные резервные возможности определяются для неподдерживаемых материалов.## 4. Немедленные действия
- Строительные леса фазы A (характеристика моментального снимка + телеметрия проводов) - voir les taches actionnables dans les mises a Jour du Roadmap.
- L'audit недавних настроек по умолчанию для `sumeragi`, `state` и `ivm` и просмотра следующих точек:
  - `sumeragi`: разрешения защиты мертвого кода для широковещательной передачи с ограничениями на изменение изображения, режим воспроизведения VRF и экспорт телеметрии EMA. Оставшиеся ворота просто являются упрощением потока консенсуса по фазе C и библиотек интеграционного пути/доказательства soient livrees.
  - `state`: сетевое соединение `Cell` и маршрутизация телеметрии по телеметрии трассы WSV фазы A, а также примечания SoA/параллельное применение базового уровня в отложенной оптимизации конвейера фазы C.
  - `ivm`: экспозиция для переключения CUDA, проверка оболочек и покрытия Halo2/Metal отображается на границе хоста фазы D плюс тема, пересекающая ускорение графического процессора; ядра остались в отставании от графического процессора только после того, как они созреют.
- Составление плана межгруппового возобновления RFC для подписания перед внесением изменений в код.
