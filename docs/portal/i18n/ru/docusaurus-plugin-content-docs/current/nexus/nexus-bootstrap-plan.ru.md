# План Bootstrap и Observability Sora Nexus

## Цели
- Развернуть базовую сеть валидаторов/наблюдателей Sora Nexus с ключами управления, Torii API и мониторингом консенсуса.
- Проверьте ключевые сервисы (Torii, консенсус, постоянство) до включения дополнительных деплоев SoraFS/SoraNet.
- Настроить рабочие процессы CI/CD и панели наблюдения/оповещения для состояния сети.

## Предпосылки
- Ключи управления материалами (многоподписный совет, ключи комитета) доступны в HSM или Vault.
- Базовая инфраструктура (кластеры Kubernetes или «голые» узлы) в первичных/вторичных регионах.
- Обновленная конфигурация начальной загрузки (`configs/nexus/bootstrap/*.toml`), отражающая актуальные параметры консенсуса.

## Сетевые окружения
- Эксплуатируйте два окружения Nexus с разными сетевыми префиксами:
- **Sora Nexus (основная сеть)** - префикс продакшн сети `nexus`, Расположение канонического управления и дополнительных сервисов SoraFS/SoraNet (идентификатор цепи `0x02F1` / UUID `00000000-0000-0000-0000-000000000753`).
- **Sora Testus (testnet)** — префикс промежуточной сети `testus`, зеркалирует конфигурацию основной сети для интеграционных тестов и предварительной проверки (цепочка UUID `809574f5-fee7-5e69-bfcf-52451e42d50f`).
- Держите создание исходных файлов, ключей управления и инфраструктурных объектов для каждого окружения. Testus служит полигоном для всех выпусков SoraFS/SoraNet перед продвижением в Nexus.
- Конвейеры CI/CD должны сначала развернуться в Testus, выполнить автоматические дымовые тесты и запросить вручную промоушн в Nexus после прохождения проверок.
- Референсные конфигурационные пакеты отправляются в `configs/soranexus/nexus/` (основная сеть) и `configs/soranexus/testus/` (тестовая сеть), команда содержит файлы `config.toml`, `genesis.json` и каталоги допуска Torii.

## Шаг 1 — Обзор конфигурации
1. Аудировать существующую документацию:
   - `docs/source/nexus/architecture.md` (консенсус, раскладка Torii).
   - `docs/source/nexus/deployment_checklist.md` (инфратребования).
   - `docs/source/nexus/governance_keys.md` (процедуры хранения ключей).
2. Проверьте, что файлы Genesis (`configs/nexus/genesis/*.json`) соответствуют текущему реестру валидаторов и весам ставок.
3. Подтвердите параметры сети:
   - Размер консенсусного комитета и кворум.
   - Интервал блоков/пороги окончательности.
   - Порты сервисов Torii и сертификаты TLS.## Шаг 2. Развертывание загрузочного кластера
1. Подготовить валидаторские узлы:
   - Развернуть `irohad` инстансы (валидаторы) с постоянными томами.
   - Убедиться, что правила межсетевого экрана разрешают консенсус и трафик Torii между узлами.
2. Запустите сервисы Torii (REST/WebSocket) на каждом валидаторе с TLS.
3. Развернуть узлы наблюдателя (только для чтения) для дополнительной устойчивости.
4. Запустите загрузочные скрипты (`scripts/nexus_bootstrap.sh`) для генезиса распространения, запуска консенсуса и регистрации узлов.
5. Выполнить дымовые испытания:
   - Отправить тестовые транзакции через Torii (`iroha_cli tx submit`).
   - Проверьте блоки производства/завершенности через телеметрию.
   - Просмотр реестра репликации между валидаторами/наблюдателем.

## Шаг 3 — Управление и управление ключами
1. Загрузить мультиподписной конфигурационный совет; Уверен, что предложения управления можно внести и ратифицировать.
2. Безопасно хранить ключи консенсуса/комитета; настроить автоматическое резервное копирование с журналированием доступа.
3. Создайте процедуры аварийной ротации ключей (`docs/source/nexus/key_rotation.md`) и проверьте Runbook.

## Шаг 4 — Интеграция CI/CD
1. Проложить трубопроводы:
   - Сборка и публикация валидатора/образов Torii (GitHub Actions или GitLab CI).
   - Автоматическая валидация конфигурации (проверка генезиса, проверка подписей).
   - Конвейеры развертывания (Helm/Kustomize) для Staging и Production кластеров.
2. Внедрить дымовые тесты в CI (поднять эфемерный кластер и запустить стандартную транзакционную систему).
3. Добавьте скрипты отката для неудачных деплоев и задокументируйте Runbook.

## Шаг 5 — Наблюдаемость и оповещения
1. Развернуть мониторинговый стек (Prometheus + Grafana + Alertmanager) по регионам.
2. Собирать ключевые метрики:
  - `nexus_consensus_height`, `nexus_finality_lag`, `torii_request_duration_seconds`, `validator_peer_count`.
   - Логи через Loki/ELK для Torii и сервисов консенсуса.
3. Дашборды:
   - Здоровье консенсуса (высота блоков, окончательность, статус пэров).
   - Латентность/ошибки Torii API.
   - Управление транзакциями и статусы предложений.
4. Предупреждения:
   - Остановка производства блоков (>2 интервала блоков).
   - Падение количества пэров ниже кворума.
   - Спайки ошибок Torii.
   - Отставание в очереди управления предложениями.

## Шаг 6 — Проверка и передача
1. Запустить сквозную валидацию:
   - Отправить предложение по управлению (например, изменение параметра).
   - Пропустить через одобрение совета, чтобы убедиться, что управление трубопроводом работает.
   - Запустить разницу состояний реестра для проверки консистентности.
2. Документировать runbook для дежурства (реагирование на инциденты, отработка отказа, масштабирование).
3. Сообщить о поддержке командым SoraFS/SoraNet; Уверенность в том, что комбинированная деплои может проявиться на узлах Nexus.## Чеклист реализации
- [ ] Аудит генезиса/конфигурации завершен.
- [ ] Валидаторские узлы и узлы-наблюдатели Расширены с здоровым консенсусом.
- [ ] Ключи управления загружены, предложение протестирован.
- [ ] Работают CI/CD-конвейеры (сборка + развертывание + дымовые тесты).
- [ ] Панели мониторинга активны с алертами.
- [ ] Передача документации передается нижестоящим командам.
