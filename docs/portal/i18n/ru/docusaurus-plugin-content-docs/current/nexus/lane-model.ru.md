# Модель дорожек Nexus и партиционирование WSV

> **Статус:** поставка NX-1 - таксономия дорожек, геометрия конфигурации и раскладка хранения готовы к внедрению.  
> **Владельцы:** Nexus Основная рабочая группа, Рабочая группа по управлению  
> **Ссылка на дорожную карту:** NX-1 в `roadmap.md`

На этой странице портал зеркалирует канонический бриф `docs/source/nexus_lanes.md`, чтобы операторы Sora Nexus, владельцы SDK и ревьюеры могли прочитать руководство по полосам без погружения в дерево моно-репозитория. Целевая архитектура сохраняет детерминизм мирового состояния, при этом позволяет отключать пространства данных (линии) запускать общедоступные или частные наборы валидаторов с изолированными рабочими нагрузками.

## Концепции

- **Lane:** логический шард-регистр Nexus со своим набором валидаторов и невыполненной работой. Идентифицируется стабильным `LaneId`.
- **Пространство данных:** сегмент управления, группировка одной или нескольких полос с соблюдением общих политик, маршрутизацией и расчетом.
- **Lane Manifest:** метаданные, контролируемые управлением, описывающая валидаторы, политика DA, газовый токен, правила урегулирования и разрешения на маршрутизацию.
- **Глобальные обязательства:** доказательства, выпускаемая полоса и суммирование новых государственных корней, данные о расчетах и ​​опциональные межполосные трансферты. Глобальное кольцо NPoS упорядочивает обязательства.

##Таксономия переулков

Типы полос регулярно наблюдают за видимостью, поверхностью управления и расчетными крючками. Геометрическая конфигурация (`LaneConfig`) фиксирует эти атрибуты, чтобы узлы, SDK и инструменты могли понимать макет без специальной логики.

| Тип переулок | Вероятность | Членство валидаторов | Экспозиция WSV | Управление по умолчанию | Политика урегулирования | Типичное применение |
|-----------|------------|----------------------|--------------|----------------|----|-------------------|-------------|
| `default_public` | общественный | Без разрешения (глобальная доля) | Полная государственная копия | SORA Парламент | `xor_global` | Базовый публичный реестр |
| `public_custom` | общественный | Без разрешения или с кольями | Полная государственная копия | Модуль взвешивания ставок | `xor_lane_weighted` | Публичные приложения с высокой пропускной способностью |
| `private_permissioned` | ограничено | Фиксированный набор валидаторов (одобренное управление) | Обязательства и доказательства | Совет Федерации | `xor_hosted_custody` | CBDC, консорциумные нагрузки |
| `hybrid_confidential` | ограничено | Смешанная членство; оборачивает ZK доказательства | Обязательства + выборочное раскрытие информации | Программируемый денежный модуль | `xor_dual_fund` | Программируемые деньги, сохраняющие конфиденциальность |

Все виды переулка объявляют:- Псевдоним Dataspace - человекочитаемая группа, связанная с соблюдением политики.
- Дескриптор управления - идентификатор, разрешенный через `Nexus.governance.modules`.
- Расчетный дескриптор - идентификатор, потребляемый расчетным маршрутизатором для написания XOR-буферов.
- Опциональная метаданные телеметрии (описание, контакт, домен деятельности), отображаемая через `/status` и информационные панели.

## Геометрия конфигурации полос (`LaneConfig`)

`LaneConfig` - динамическая геометрия, выводимая из валидированного каталога полос. Она не заменяет манифесты управления; вместо этого предоставляются определенные идентификаторы хранилища и подсказки телеметрии для каждой сконфигурированной полосы.

```text
LaneConfigEntry {
    lane_id: LaneId,           // stable identifier
    alias: String,             // human-readable alias
    slug: String,              // sanitised alias for file/metric keys
    kura_segment: String,      // Kura segment directory: lane_{id:03}_{slug}
    merge_segment: String,     // Merge-ledger segment: lane_{id:03}_merge
    key_prefix: [u8; 4],       // Big-endian LaneId prefix for WSV key spaces
    shard_id: ShardId,         // WSV/Kura shard binding (defaults to lane_id)
    visibility: LaneVisibility,// public vs restricted lanes
    storage_profile: LaneStorageProfile,
    proof_scheme: DaProofScheme,// DA proof policy (merkle_sha256 default)
}
```

- `LaneConfig::from_catalog` пересчитывает геометрию при включении конфигурации (`State::set_nexus`).
- Псевдонимы санитизируются в нижнем пуле регистра; по последовательности неалфавитно-цифровых символов схлопываются в `_`. Если псевдоним дает пустой пул, используйте `lane{id}`.
- `shard_id` выводится из ключа метаданных `da_shard_id` (по умолчанию `lane_id`) и управления курсором в журнале, чтобы DA воспроизводил стабильно определенные между перезапусками/перешардингами.
— Префиксы ключей ограничивают, что WSV сохраняет отдельные диапазоны ключей по полосе, даже если серверная часть общая.
- сегмент Имена Кура определен между хостами; аудиторы могут совместно создавать каталоги сегментов и манифесты без специальных инструментов.
- Сегменты слияния (`lane_{id:03}_merge`) хранят последние корни подсказок о слиянии и глобальные государственные обязательства для этой полосы.

## Разделение мирового государства

- Логическое состояние мира Nexus - это объединение пространств состояний по пер. Переулки общего пользования сохраняют полноценное состояние; Частные/конфиденциальные направления экспорта подтверждают корни Меркла/обязательств в реестре слияний.
- MV Storage префиксирует каждый ключ 4-байтовым префиксом `LaneConfigEntry::key_prefix`, получая ключи вида `[00 00 00 01] ++ PackedKey`.
- Общие таблицы (учетные записи, активы, триггеры, записи управления) хранят записи, сгруппированные по префиксу полосы, сохраняя диапазон сканирования детерминизма.
- Метаданные реестра слияния зеркалируют тот же макет: каждая полоса записывает корни подсказок о слиянии и сокращает глобальные корни состояний в `lane_{id:03}_merge`, что позволяет делать целевое удержание/вытеснение при выводе полосы.
- Межполосные индексы (псевдонимы учетных записей, реестры активов, манифесты управления) хранят явные префиксы полос, чтобы операторы могли быстро выполнять сверку.
- **Политика удержания** - полосы общего пользования сохраняют полные блоки; Полосы только для обязательств могут компактизировать старые тела после контрольно-пропускных пунктов, потому что обязательства значимы. Конфиденциальные полосы хранят журналы зашифрованного текста в выделенных сегментах, чтобы не блокировать другие рабочие нагрузки.
- **Инструменты** - утилиты обслуживания (`kagami`, команды администратора CLI) должны ссылаться на заполненное пространство имен при экспонировании метрик, меток Prometheus или архивировании сегментов Kura.

## Маршрутизация и API- Конечные точки REST/gRPC Torii принимают опциональный `lane_id`; отсутствие означает `lane_default`.
- SDK отображает селекторы полос и мапят удобные псевдонимы в `LaneId` через каталог полос.
- Правила маршрутизации работают по валидированному каталогу и могут выбирать, полосу и пространство данных. `LaneConfig` предоставляет псевдонимы, удобные для телеметрии, для информационных панелей и журналов.

## Расчеты и сборы

- каждая дорожка платит комиссию XOR глобальному набору валидаторов. Lanes могут собирать токены природного газа, но, похоже, депонируют эквиваленты XOR вместе с обязательствами.
- Подтверждения расчетов включают сумму, метаданные конвертации и подтверждение условного депонирования (например, перевод в глобальное хранилище комиссий).
- Единый расчетный маршрутизатор (NX-3) дебетует буферы тех же префиксов дорожек, поэтому расчетная телеметрия совпадает с геометрией хранения.

## Управление

- Лейны объявляют свой модуль управления через каталог. `LaneConfigEntry` содержит оригинальный псевдоним и пул, чтобы телеметрия и журналы аудита были незначительно читаемыми.
- Реестр Nexus распространяет подписанные манифесты полос, включая `LaneId`, привязку пространства данных, дескриптор управления, дескриптор расчета и метаданные.
- Обработчики обновления во время выполнения продолжают применять политики управления (`gov_upgrade_id` по умолчанию) и регистрировать различия через мост телеметрии (события `nexus.config.diff`).

## Телеметрия и статус

- `/status` показывает псевдонимы полос, привязки к пространству данных, дескрипторы управления и профили расчетов, производные из каталога и `LaneConfig`.
- Метрики планировщика (`nexus_scheduler_lane_teu_*`) отображают псевдонимы/слабы, чтобы операторы быстро связывались с отставанием и давлением TEU.
- `nexus_lane_configured_total` считает количество выведенных записей полосы и пересчитывается при конфигурации. Телеметрия испускает подписанные дифференциалы при сохранении геометрии полосы движения.
- Индикаторы невыполненной работы в пространстве данных включают метаданные псевдонимов/описаний, помогающие операторам связывать давление в очереди с бизнес-доменами.

## Конфигурация и типы Norito

- `LaneCatalog`, `LaneConfig` и `DataSpaceCatalog` настраиваются в `iroha_data_model::nexus` и предоставляют Norito-совместимые структуры для манифестов и SDK.
- `LaneConfig` находится в `iroha_config::parameters::actual::Nexus` и автоматически выводится из каталога; Кодировка Norito не требуется, как и внутренний помощник времени выполнения.
- Пользовательская реализация (`iroha_config::parameters::user::Nexus`) продолжает принимать декларативные дескрипторы полосы и пространства данных; синтаксический анализ теперь выводит геометрию и отклоняет невалидные псевдонимы или дублирующиеся идентификаторы дорожек.
