# Sora Nexus План начальной загрузки и наблюдаемости

## اہداف
- گورننس کیز، Torii APIs، اور consensus monitoring کے ساتھ Sora Nexus validator/observer نیٹ ورک کی بنیاد قائم کریں۔
- بنیادی سروسز (Torii، consensus، persistence) کو SoraFS/SoraNet piggyback deployments سے پہلے ویلیڈیٹ کریں۔
- Рабочие процессы CI/CD, панели мониторинга/оповещения наблюдаемости.

## پیشگی شرائط
- Материал ключа управления (мультиподпись совета, ключи комитета) HSM یا Vault میں دستیاب ہو۔
- بنیادی انفراسٹرکچر (кластеры Kubernetes и узлы без ОС)
- В конфигурации начальной загрузки (`configs/nexus/bootstrap/*.toml`) используются параметры консенсуса.

## نیٹ ورک ماحول
- В средах Nexus есть несколько префиксов, которые можно использовать:
- **Sora Nexus (mainnet)** - پروڈکشن نیٹ ورک prefix `nexus` جو canonical governance اور SoraFS/SoraNet piggyback services میزبان بناتا ہے (идентификатор цепи `0x02F1` / UUID `00000000-0000-0000-0000-000000000753`).
- **Sora Testus (testnet)** - промежуточный префикс `testus`, конфигурация основной сети, интеграционное тестирование, предварительная проверка и зеркало (цепочка UUID `809574f5-fee7-5e69-bfcf-52451e42d50f`).
- Среда, файлы Genesis, ключи управления, следы инфраструктуры. Testus تمام SoraFS/SoraNet развертывает испытательный полигон ہے، Nexus میں продвигает کرنے سے پہلے۔
- Конвейеры CI/CD в Testus, развертывание автоматических дымовых тестов, проверка и проверка Nexus, ручное продвижение.
- Reference configuration bundles `configs/soranexus/nexus/` (mainnet) اور `configs/soranexus/testus/` (testnet) کے تحت ہیں، ہر ایک میں نمونہ `config.toml`, `genesis.json` اور Torii каталоги приема شامل ہیں۔

## مرحلہ 1 – Обзор конфигурации
1. Подготовка документации для аудита:
   - `docs/source/nexus/architecture.md` (консенсус, раскладка Torii).
   - `docs/source/nexus/deployment_checklist.md` (инфратребования).
   - `docs/source/nexus/governance_keys.md` (процедуры хранения ключей).
2. Файлы Genesis (`configs/nexus/genesis/*.json`) для проверки списка валидаторов и весов ставок и выравнивания ہیں۔
3. Дополнительные параметры:
   - Размер комитета по согласованию при наличии кворума.
   - Пороги интервала блокировки/финальности.
   - Сервисные порты Torii и сертификаты TLS.## مرحلہ 2 - Bootstrap Cluster Deployment
1. Validator nodes provision کریں:
   - Экземпляры `irohad` (валидаторы) для постоянных томов и возможности развертывания
   - В правилах брандмауэра есть возможность консенсуса в узлах трафика Torii, которые разрешены.
2. Валидатор для служб Torii (REST/WebSocket) и TLS для проверки подлинности.
3. Обеспечение устойчивости и развертывание узлов-наблюдателей (только для чтения).
4. Сценарии начальной загрузки (`scripts/nexus_bootstrap.sh`) Использование генезиса и консенсуса в узлах
5. Smoke tests چلائیں:
   - Torii کے ذریعے тестовые транзакции отправляют کریں (`iroha_cli tx submit`).
   - Телеметрия: создание блока/проверка окончательности.
   - Валидаторы/наблюдатели Возможность репликации реестра

## مرحلہ 3 - Управление и ключевое руководство
1. Конфигурация мультиподписи Совета. Предложения по управлению представляются и ратифицируются.
2. Ключи к консенсусу/комитетам ведение журнала доступа настройка автоматического резервного копирования
3. Процедуры экстренной ротации ключей (`docs/source/nexus/key_rotation.md`)

## مرحلہ 4 - CI/CD Integration
1. Pipelines configure کریں:
   - Образы Validator/Torii собираются и публикуются (GitHub Actions или GitLab CI).
   - Автоматическая проверка конфигурации (линт-генезис, проверка подписей).
   - Конвейеры развертывания (Helm/Kustomize) для создания производственных кластеров.
2. CI-тестирование дымовых тестов (эфемерный кластер, набор канонических транзакций).
3. Развертывание и сценарии отката, а также использование Runbook и других приложений.

## مرحلہ 5 - Observability اور Alerts
1. Стек мониторинга (Prometheus + Grafana + Alertmanager).
2. Core metrics جمع کریں:
  - `nexus_consensus_height`, `nexus_finality_lag`, `torii_request_duration_seconds`, `validator_peer_count`.
   - Torii Службы консенсуса для Loki/ELK и журналы журналов.
3. Панели мониторинга:
   - Состояние консенсуса (высота блока, окончательность, статус узла).
   - Torii API latency/error rates.
   - Транзакции управления и статусы предложений.
4. Оповещения:
   - Block production stall (>2 block intervals).
   - Кворум подсчета одноранговых узлов
   - Torii error rate spikes.
   - Governance proposal queue backlog.

## Шаг 6 — Проверка и передача
1. End-to-end validation چلائیں:
   - Предложение по управлению отправьте کریں (изменение параметра مثلاً).
   - Утверждение Советом процесса процесса управления трубопроводом управления درست ہو۔
   - Разница в состоянии бухгалтерской книги и согласованность данных.
2. Модуль Runbook по вызову (реагирование на инциденты, отработка отказа, масштабирование).
3. SoraFS/SoraNet — проверка готовности. Возможности комбинированного развертывания узлов Nexus и точки подключения.## Контрольный список реализации
