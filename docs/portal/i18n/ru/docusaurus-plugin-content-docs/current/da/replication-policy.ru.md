# Политика репликации Доступность данных (DA-4)

_Статус: В работе — Владельцы: Core Protocol WG / Storage Team / SRE_

Приемный конвейер DA теперь применяет определенные цели хранения для каждого
класса blob, описанного в `roadmap.md` (рабочий поток DA-4). Torii отказывается
сохранение конвертов, предоставленных вызывающим абонентом, если они не совпадают с
настроенной политики, гарантируя, что каждый валидатор/узел удерживает срок хранения
Необходимые эпохи числа и репликации без опоры на намерение отправителя.

## Политика по умолчанию

| Класс блоб | Горячее сохранение | Удержание холода | Требуемые реплики | Класс хранения | Тег управления |
|------------|---------------|----------------|-------------------|----------------|----------------|
| `taikai_segment` | 24 часа | 14 дней | 5 | `hot` | `da.taikai.live` |
| `nexus_lane_sidecar` | 6 часов | 7 дней | 4 | `warm` | `da.sidecar` |
| `governance_artifact` | 12 часов | 180 дней | 3 | `cold` | `da.governance` |
| _По умолчанию (все остальные классы)_ | 6 часов | 30 дней | 3 | `warm` | `da.default` |

Эти значения встроены в `torii.da_ingest.replication_policy` и применяются ко
всем `/v2/da/ingest` отправкам. Torii переписывает манифесты с применением
сохранение профилем и выдает предупреждение, если вызывающие абоненты передают несоответствующие
значения, чтобы операторы могли выявлять конфиденциальность SDK.

### Классы доступности Тайкай

Манифесты маршрутизации Taikai (`taikai.trm`) объявляют `availability_class`
(`hot`, `warm` или `cold`). Torii применяется в соответствии с политикой разделения на фрагменты,
чтобы операторы могли масштабировать число реплик в потоке без редактирования
стабильная таблица. По умолчанию:

| Класс доступности | Горячее сохранение | Удержание холода | Требуемые реплики | Класс хранения | Тег управления |
|-------------------|---------------|----------------|-------------------|----------------|----------------|
| `hot` | 24 часа | 14 дней | 5 | `hot` | `da.taikai.live` |
| `warm` | 6 часов | 30 дней | 4 | `warm` | `da.taikai.warm` |
| `cold` | 1 час | 180 дней | 3 | `cold` | `da.taikai.archive` |

Если подсказок нет, используется `hot`, чтобы в прямом эфире удерживали самых
строгий профиль. Переопределите значения по умолчанию через
`torii.da_ingest.replication_policy.taikai_availability`, если сеть использует
другие цели.

## Конфигурация

Политика находится под `torii.da_ingest.replication_policy` и обеспечивает
*по умолчанию* шаблон плюс переопределение массива для каждого класса. Идентификаторы классов
регистронезависимый и принимает `taikai_segment`, `nexus_lane_sidecar`,
`governance_artifact`, либо `custom:<u16>` для расширений, одобренных органов управления.
Классы хранения принимают `hot`, `warm` или `cold`.

```toml
[torii.da_ingest.replication_policy.default_retention]
hot_retention_secs = 21600          # 6 h
cold_retention_secs = 2592000       # 30 d
required_replicas = 3
storage_class = "warm"
governance_tag = "da.default"

[[torii.da_ingest.replication_policy.overrides]]
class = "taikai_segment"
[torii.da_ingest.replication_policy.overrides.retention]
hot_retention_secs = 86400          # 24 h
cold_retention_secs = 1209600       # 14 d
required_replicas = 5
storage_class = "hot"
governance_tag = "da.taikai.live"
```

Оставьте блок без изменений, чтобы использовать значения по умолчанию выше. Чтобы ужесточить
класс, обновите соответствующее переопределение; чтобы изменить помещения для новых классов,
отредактируйте `default_retention`.Классы доступности Taikai можно переопределять отдельно
`torii.da_ingest.replication_policy.taikai_availability`:

```toml
[[torii.da_ingest.replication_policy.taikai_availability]]
availability_class = "cold"
[torii.da_ingest.replication_policy.taikai_availability.retention]
hot_retention_secs = 3600          # 1 h
cold_retention_secs = 15552000     # 180 d
required_replicas = 3
storage_class = "cold"
governance_tag = "da.taikai.archive"
```

## Применение семантики

- Torii заменяет пользовательский `RetentionPolicy` на принудительный профиль
  перед фрагментированием или выпуском манифеста.
- Предсобранные манифесты, которые декларируют несовпадающий профиль удержания,
  отклоняются с `400 schema mismatch`, чтобы клиенты не могли ослабить
  контракт.
- Каждое событие переопределяется (`blob_class`, политика отправки против
  ожидаемая), чтобы выявлять вызывающих абонентов, не соответствующих требованиям, во время развертывания.

Смотрите [План приема данных о доступности] (ingest-plan.md) (контрольный список проверки) для
обновленного ворот, прикрывающего исполнение удержания.

## Рабочий процесс повторной репликации (продолжение DA-4)

Удержание правоприменения — лишь первый шаг. Операторы также должны подтвердить, что живут
манифесты и приказы тиражирования остаются согласованными с настроенной политикой,
чтобы SoraFS мог автоматически повторно реплицировать несоответствующие большие двоичные объекты.

1. **Следите за дрифтом.** пишет Torii
   `overriding DA retention policy to match configured network baseline` когда
   вызывающий абонент отправляет конфиденциальные значения сохранения. Сопоставьте этот журнал с
   телеметрией `torii_sorafs_replication_*`, чтобы указать недостатки реплики
   или задержанные передислокации.
2. **Сравните намерения и живые копии.** Используйте новый помощник аудита:

   ```bash
   cargo xtask da-replication-audit \
     --config configs/iroha/torii.toml \
     --manifest spool/da/manifests/*.json \
     --replication-order artifacts/da/orders/*.norito \
     --json-out artifacts/da/replication_audit.json
   ```

   Команда загружает `torii.da_ingest.replication_policy` из конфигурации,
   декодирует каждый манифест (JSON или Norito) и опционально дополняет
   полезные нагрузки `ReplicationOrderV1` по дайджест-манифесту. Итог помечает две ситуации:

   - `policy_mismatch` - манифест профиля удержания расходуется с принудительным
     профилем (такого не должно быть, если Torii настроен корректно).
   - `replica_shortfall` - порядок живой репликации запрашивает меньше реплик, чем
     `RetentionPolicy.required_replicas`, или выдает меньше назначений, чем цель.

   Ненулевой кодовый выход означает активный дефицит, для автоматизации CI/по вызову.
   возможность немедленного пейджита. Приложить отчет JSON к пакету
   `docs/examples/da_manifest_review_template.md` для голосования в парламенте.
3. **Запустите повторную репликацию.** Если аудит сообщает о нехватке, выпустите
   новый `ReplicationOrderV1` через инструменты управления, описанные в
   [SoraFS рынок емкости хранения] (../sorafs/storage-capacity-marketplace.md),
   и повторяйте аудит, пока набор реплик не сойдется. Для аварийного отключения
   сопоставьте вывод CLI с `iroha app da prove-availability`, чтобы SRE можно было ссылаться
   на тот же дайджест и доказательства ПРП.

Покрытие регрессии находится в `integration_tests/tests/da/replication_policy.rs`;
пакет отправляет несовпадающую политику сохранения в `/v2/da/ingest` и, наконец,
Полученный манифест показывает принудительный профиль звонившего без намерения.
