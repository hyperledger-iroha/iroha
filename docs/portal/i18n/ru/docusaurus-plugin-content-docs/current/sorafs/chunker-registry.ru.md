## Реестр профиля chunker SoraFS (SF-2a)

Стек SoraFS согласует поведение фрагментации через небольшой namespaced-реестр.
Каждый профиль задает определенные параметры CDC, метаданные семверы и ожидаемый дайджест/мультикодек, прогноз погоды в манифестах и ​​CAR-архивах.

Авторы профилей должны обратиться к
[`docs/source/sorafs/chunker_profile_authoring.md`](./chunker-profile-authoring.md)
для получения требуемых метаданных, чеклиста валидации и шаблонов предложений перед
отправкой новых записей. После одобрения изменений со стороны управления проследите
[чеклист реестра развертывания](./chunker-registry-rollout-checklist.md) и
[манифест playbook в промежуточном состоянии](./staging-manifest-playbook), чтобы продвинуться
приспособления в постановке и производстве.

Код реестра существует как `sorafs_manifest::chunker_registry` (регулируется [`chunker_registry_charter.md`](./chunker-registry-charter.md)). запись
выражается как `ChunkerProfileDescriptor` соблюдая правила:

* `namespace` – логическая группировка имен профилей (например, `sorafs`).
* `name` – читаемый человек ярлык профиля (`sf1`, `sf1-fast`, …).
* `semver` – строка семантической версии для набора параметров.
* `profile` – фактически `ChunkProfile` (мин/цель/макс/маска).
* `multihash_code` – мультихеш, переменная при вычислении дайджеста чанка (`0x1f`
  для SoraFS по умолчанию).

Манифест сериализует профиль через `ChunkingProfileV1`. Структура записей метаданного реестра
(namespace, name, semver) вместе с сырыми CDC-параметрами и списком псевдонимов выше. Потребители
сначала активисты должны выполнить поиск в реестре по `profile_id` и использовать встроенные параметры,
когда встречаются неизвестные идентификаторы; список псевдонимов гарантирует, что HTTP-клиенты могут продолжать работу

### Согласование chunker

Gateways и клиенты объявляют поддерживаемые профили через provider adverts:

```
ProviderAdvertBodyV1 {
    ...
    chunk_profile: Profile_id (неявно через реестр)
    возможности: [...]
}
```

Планирование multi-source чанков объявляется через capability `range`. CLI принимает ее с
`--capability=range[:streams]`, где опциональный числовой суффикс кодирует предпочтительную
параллельность range-fetch у провайдера (например, `--capability=range:64` объявляет бюджет 64 streams).
Когда суффикс отсутствует, потребители возвращаются к общему hint `max_streams`, опубликованному в другом
месте advert.

При запросе CAR-данных клиенты должны отправлять заголовок `Accept-Chunker`, перечисляя кортежи
`(namespace, name, semver)` в порядке предпочтения:

```

Шлюзы выбирают взаимно привязанный профиль (по умолчанию `sorafs.sf1@1.0.0`) и отражают
решение в ответном заголовке `Content-Chunker`. Проявляет встраивает выбранный профиль, чтобы
Нижние узлы могли валидировать раскладку чанков без опор на HTTP-переговоры.

### Соответствие

* Профиль `sorafs.sf1@1.0.0` соответствует публичным светильникам в
  `fixtures/sorafs_chunker` и корпорация, зарегистрированная в
  `fuzz/sorafs_chunker`. Сквозной паритет недавно появился в Rust, Go и Node
  через предоставленные тесты.
* `chunker_registry::lookup_by_profile` подтверждает, что параметры дескриптора
  произошло с `ChunkProfile::DEFAULT`, чтобы защититься от случайной дивергенции.
* Манифесты, исправленные `iroha app sorafs toolkit pack` и `sorafs_manifest_stub`, включают метаданные реестра.
