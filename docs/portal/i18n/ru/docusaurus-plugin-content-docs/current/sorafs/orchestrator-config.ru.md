# Руководство по мульти-источниковому fetch-оркестратору

Мульти-источниковый fetch-оркестратор SoraFS управление определено
параллельными загрузками из наборов провайдеров, опубликованных в рекламных объявлениях под
контроль управления. В этом руководстве указано, как настраивать оркестратор,
какие сигналы сбоев ожидаются при развертывании, и какие потоки телеметрии отображаются
индикаторы здоровья.

## 1. Обзор конфигурации

Оркестратор выбирает три источника конфигурации:

| Источник | Назначение | Примечания |
|----------|------------|------------|
| `OrchestratorConfig.scoreboard` | Нормализует вес провайдеров, тем самым обеспечивает свежесть телеметрии и сохраняет табло JSON для аудитов. | Основан на `crates/sorafs_car::scoreboard::ScoreboardConfig`. |
| `OrchestratorConfig.fetch` | Применяет ограничения времени выполнения (бюджеты ретраев, лимиты параллелизма, переключатели верификации). | Маппится на `FetchOptions` в `crates/sorafs_car::multi_fetch`. |
| Параметры CLI/SDK | Ограничивают число пиров, включают регионы телеметрии и выводят политику отрицания/повышения. | `sorafs_cli fetch` раскрывает эти флаги напрямую; SDK передаёт их через `OrchestratorConfig`. |

JSON-помощники в `crates/sorafs_orchestrator::bindings` сериализуют всё
конфигурация в Norito JSON, обеспечение ее переносимости между SDK и автоматизация.

### 1.1 Пример JSON-конфигурации

```json
{
  "scoreboard": {
    "latency_cap_ms": 6000,
    "weight_scale": 12000,
    "telemetry_grace_secs": 900,
    "persist_path": "/var/lib/sorafs/scoreboards/latest.json"
  },
  "fetch": {
    "verify_lengths": true,
    "verify_digests": true,
    "retry_budget": 4,
    "provider_failure_threshold": 3,
    "global_parallel_limit": 8
  },
  "telemetry_region": "iad-prod",
  "max_providers": 6,
  "transport_policy": "soranet_first"
}
```

Сохраняйте файл через стандартное наслаивание `iroha_config` (`defaults/`, пользователь,
фактически), чтобы определенные деплои наследовали одинаковые лимиты для всех
нодах. Для профиля только прямой, красный рулон SNNet-5a, смотрите
`docs/examples/sorafs_direct_mode_policy.json` и дополнительные рекомендации в
`docs/source/sorafs/direct_mode_pack.md`.

### 1.2 Переопределение соответствия

SNNet-9 обеспечивает соответствие требованиям управления в оркестраторе. Новый объект
`compliance` в конфигурации Norito JSON фиксирует вырезки, которые переводятся
выборка по конвейеру только напрямую:

```json
"compliance": {
  "operator_jurisdictions": ["US", "JP"],
  "jurisdiction_opt_outs": ["US"],
  "blinded_cid_opt_outs": [
    "C6B434E5F23ABD318F01FEDB834B34BD16B46E0CC44CD70536233A632DFA3828"
  ],
  "audit_contacts": ["mailto:compliance@example.org"]
}
```- `operator_jurisdictions` объявляет коды ISO‑3166 альфа‑2, где работает эта
  установка оркестратора. Коды нормализуются в верхнем регистре при разборе.
- `jurisdiction_opt_outs` зеркалирует управление реестром. В любой юрисдикции
  оператор присутствует в списке, оркестратор документов
  `transport_policy=direct-only` и выдает причину отката
  `compliance_jurisdiction_opt_out`.
- `blinded_cid_opt_outs` пересчитывает дайджест манифеста (слепой CID, вверху)
  шестнадцатеричный). Совпадающие полезные нагрузки также форсят только прямое планирование и публикацию
  резервный вариант `compliance_blinded_cid_opt_out` в телеметрии.
- `audit_contacts` записывает URI, который управление ожидает увидеть в GAR
  операторы игровых книг.
- `attestations` фиксирует подписанные комплаенс-пакеты, на которых держится
  политика. каждая запись задаёт опциональную `jurisdiction` (ISO‑3166 альфа‑2),
  `document_uri`, канонический `digest_hex` (64 символа), временная метка выдачи
  `issued_at_ms` и опциональный `expires_at_ms`. Эти материалы сделаны в
  аудит-чеклист оркестратора, чтобы инструменты управления переопределяли
  подписанными документами.

Передавайте блокировку соответствия через стандартное наслаивание конфигурации, чтобы
операторы применяют определенные переопределения. Оркестратор документов
подсказки режима записи _после_: даже если SDK запрашивает `upload-pq-only`,
отказ от участия в соответствии с юрисдикцией или манифестом: всё равно осуществляет транспортировку только напрямую
и быстро устраняет ошибку, если не осталось подключенных провайдеров.

Канонические каталоги
`governance/compliance/soranet_opt_outs.json`; Совет по управлению публикует
обновления через отмеченные релизы. Полный пример конфигурации (включая
аттестации) доступны в `docs/examples/sorafs_compliance_policy.json`, а
операционный процесс описан в
[playbook соответствия GAR](../../../source/soranet/gar_compliance_playbook.md).

| `--telemetry-json` | Вкалывает снимки латентности/сбоев в табло построителя. Устаревшая телеметрия по признаку `telemetry_grace_secs` делает провайдеров неприемлемыми. |

| `--scoreboard-now` | Переопределяет табло временных меток (в секундах Unix), чтобы зафиксировать данные о точно определенных данных. |

Используйте `direct-only` при переходе на более раннюю версию или в соответствии с директивами, а `soranet-strict` оставьте для пилотов, поддерживающих только PQ; соответствие имеет приоритет перед потолком. |

SoraNet-first теперь дефолт, а откаты должны ссылаться на соответствующие
Блокировщик SNNet. После выпуска SNNet-4/5/5a/5b/6a/7/8/12/13 управление ужесточит
требуемую позу (в сторону `soranet-strict`); до этого только переопределить
происшествиям следует приоритизировать `direct-only`, и их нужно фиксировать в журнале.
развертывание.

Все флаги выше принимают синтаксис `--` как в `sorafs_cli fetch`, так и в
разработческий бинаре `sorafs_fetch`. SDK предоставляет те же возможности через типизированный
строители.

### 1.4 Управление кэшем охраны

CLI теперь подключает селекторную защиту SoraNet, чтобы операторы могли
детерминированно обеспечить входные реле до полноценного развертывания SNNet-5.
Рабочий процесс контролирует три новых флага:

| Флаг | Назначение |
|------|-----------|
| `--guard-directory <PATH>` | Указывает JSON-файл с последним консенсусом реле (подмножество ниже). Передача каталога обновляет Guard Cache перед выборкой. |
| `--guard-cache <PATH>` | Сохраняет Norito в кодировке `GuardSet`. Следующие прогоны используют кэш, даже если новый каталог не задан. |
| `--guard-target <COUNT>` / `--guard-retention-days <DAYS>` | Опциональные ограничения для числа охранников на входе (по умолчанию 3) и удержания окон (по умолчанию 30 дней). |
| `--guard-cache-key <HEX>` | Дополнительный 32-байтовый ключ для тега Guard Cache с MAC-адресом Blake3, чтобы файл можно было проверить повторно перед использованием. |

Каталог Payloads Guard использует компактную схему:

Флаг `--guard-directory` теперь предполагает полезную нагрузку, закодированную Norito.
`GuardDirectorySnapshotV2`. Бинарный снимок содержит:- `version` — версия схемы (сейчас `2`).
- И18НИ00000080Х, И18НИ00000081Х, И18НИ00000082Х, И18НИ00000083Х —
  метаданные консенсусы, которые должны соответствовать каждому сертификату.
- `validation_phase` — сертификаты политики ворот (`1` = разрешить одну Ed25519
  Подпись, `2` = предпочтение двойных подключений, `3` = требуют двойные подключения).
- `issuers` — эмитенты управления с `fingerprint`, `ed25519_public` и
  `mldsa65_public`. Отпечаток пальца хранится как
  `BLAKE3("soranet.src.v2.issuer" || ed25519 || u32(len(ml-dsa)) || ml-dsa)`.
- `relays` — список пакетов SRCv2 (выход `RelayCertificateBundleV2::to_cbor()`).
  Каждый пакет содержит реле дескриптора, флаги возможностей, политику ML-KEM и
  двойные расширения Ed25519/ML-DSA-65.

CLI впоследствии каждый пакет против эмитента объявленных ключей перед объединением
снимки.

Вызывайте CLI с `--guard-directory`, чтобы понять актуальный консенсус с
Существующим кэшем. Селектор сохранения фиксированных ограждений, которые все еще находятся в окне
удержания и разрешены в каталоге; Новые реле заменяют просроченные записи.
После получения обновленного кэша для записи по пути `--guard-cache`,
обеспечение детерминированности следующих сессий. SDK воспроизводит поведение,
Вызывающая `GuardSelector::select(&RelayDirectory, existing_guard_set, now_unix_secs)`
и перепропущенный `GuardSet` в `SorafsGatewayFetchOptions`.

`ml_kem_public_hex` Позволяет селектору приоритизировать PQ-защиты во время
внедрение SNNet-5. Переключатели этапов (`anon-guard-pq`, `anon-majority-pq`,
`anon-strict-pq`) теперь автоматически понижают классические реле: когда
имеется защита PQ, селектор сбрасывает лишние классические штифты, чтобы выглядеть
сессии предпочитали гибридные рукопожатия. Сводки CLI/SDK отображают итоговый результат
микс через `anonymity_status`/`anonymity_reason`, `anonymity_effective_policy`,
И18НИ00000107Х, И18НИ00000108Х, И18НИ00000109Х,
`anonymity_classical_ratio` и соответствующие поля кандидатов/дефицита/дельта поставок,
создание отключений и классических аварийных ситуаций явными.

Каталоги Guard теперь могут поддерживать полный пакет SRCv2 через
`certificate_base64`. Оркестратор декодирует каждый пакет, снова и снова
Загрузите Ed25519/ML-DSA и сохраните разобранный сертификат вместе с защитным кэшем.
Когда сертификат присутствует, он становится классическим умным ключом PQ,
настройки рукопожатий и весов; просроченные сертификаты отбрасываются, и селектор
Схема жизненного цикла доступна через `telemetry::sorafs.guard` и
`telemetry::sorafs.circuit`, фиксируя окно валидности, наборы квитирования и
Наличие двойных подписей для каждого охранника.

Используйте помощники CLI, чтобы хранить снимки синхронизированными с
публикаторами:

```bash
sorafs_cli guard-directory fetch \
  --url https://directory.soranet.dev/mainnet_snapshot.norito \
  --output ./state/guard_directory.norito \
  --expected-directory-hash <directory-hash-hex>

sorafs_cli guard-directory verify \
  --path ./state/guard_directory.norito \
  --expected-directory-hash <directory-hash-hex>
```

`fetch` загружает и проверяет снимок SRCv2 перед записью на диск, а `verify`
повторяет конвейер валидации для артефактов из других команд, выдавая JSON
резюме, которое зеркалит селектор защиты вывода CLI/SDK.

### 1.5 Схема управления жизненным цикломКогда доступен каталог ретрансляции и охранный кэш, оркестратор активирует схему.
менеджер жизненного цикла для передовых построений и обновлений схем SoraNet
перед каждой выборкой. Конфигурация находится в `OrchestratorConfig`.
(`crates/sorafs_orchestrator/src/lib.rs:305`) через два новых поля:

- `relay_directory`: сохраняет снимок каталога SNNet-3 для средних/выходных переходов.
  выбирались детерминированно.
- `circuit_manager`: опциональная вставка (включена по умолчанию),
  контролирующая TTL цепей.

Norito JSON теперь принимает блок `circuit_manager`:

```json
"circuit_manager": {
  "enabled": true,
  "circuit_ttl_secs": 900
}
```

SDK передает каталог данных через
`SorafsGatewayFetchOptions::relay_directory`
(`crates/iroha/src/client.rs:320`), CLI подключает их автоматически, когда
передан `--guard-directory` (`crates/iroha_cli/src/commands/sorafs.rs:365`).

Менеджер обновляет схемы, когда меняются метаданные охранники (конечная точка, ключ PQ
или закрепленная временная метка) или подтверждает TTL. Хелпер `refresh_circuits`, вызывающий
перед каждой выборкой (`crates/sorafs_orchestrator/src/lib.rs:1346`), эмитировать логи
`CircuitEvent`, включите оператором средства контроля жизненного цикла. Замочить
тест `circuit_manager_latency_soak_remains_stable_across_rotations`
(`crates/sorafs_orchestrator/src/soranet.rs:1479`) наиболее стабильная
латентность на три ротации охранников; посмотреть отчет в
`docs/source/soranet/reports/circuit_stability.md:1`.

### 1.6 Локальный QUIC-прокси

Оркестратор может опционально запускать локальный QUIC-прокси для браузерных
Расширение и адаптеры SDK не управляют сертификатами или ключами кэша защиты. Прокси
прослушивает петлевой адрес, завершает соединение QUIC и получает манифест Norito,
описывающий сертификат и опциональный ключ защитного кэша. Транспортные события,
эмитируемые прокси, наблюдаются в `sorafs_orchestrator_transport_events_total`.

Включите прокси через новый блок `local_proxy` в JSON оркестратора:

```json
"local_proxy": {
  "bind_addr": "127.0.0.1:9443",
  "telemetry_label": "dev-proxy",
  "guard_cache_key_hex": "0123456789ABCDEF0123456789ABCDEF0123456789ABCDEF0123456789ABCDEF",
  "emit_browser_manifest": true,
  "proxy_mode": "bridge",
  "prewarm_circuits": true,
  "max_streams_per_circuit": 64,
  "circuit_ttl_hint_secs": 300,
  "norito_bridge": {
    "spool_dir": "./storage/streaming/soranet_routes",
    "extension": "norito"
  },
  "kaigi_bridge": {
    "spool_dir": "./storage/streaming/soranet_routes",
    "extension": "norito",
    "room_policy": "public"
  }
}
```- `bind_addr` задает адрес прослушивания (используйте порт `0` для эфемерного
  выезд).
- `telemetry_label` относится к метрикам, чтобы дашборды различались прокси
  и fetch-сессии.
- `guard_cache_key_hex` (опционально) Позволяет прокси отдавать тот же ключ
  Защитный кэш, который использует CLI/SDK, чтобы расширения браузера были исключены.
  синхронизированными.
- `emit_browser_manifest` включает манифест выдачи, который может расширяться.
  сохранять и проверять.
- `proxy_mode` выбирает, будет ли прокси мостить трафик локально (`bridge`) или
  только отдавать метаданные, чтобы SDK открывал схемы SoraNet самостоятельно
  (И18НИ00000140Х). По умолчанию `bridge`; `metadata-only`, если викор
  Рабочая станция должна выдавать манифест без ретрансляции потоков.
- `prewarm_circuits`, `max_streams_per_circuit` и `circuit_ttl_hint_secs`
  передают браузеру дополнительные подсказки, чтобы он мог вести параллельный бюджет
  потоки подразумевают и агрессивность схем повторного использования.
- `car_bridge` (опционально) указывает на локальный кэш CAR-архивов. Поле
  `extension` задает суффикс, добавляющий, когда цель не содержит `*.car`; задайте
  `allow_zst = true` для прямой выдачи `*.car.zst`.
- `kaigi_bridge` (опционально) экспонирует маршруты Kaigi из спула в прокси. Поле
  `room_policy` объявляет режим `public` или `authenticated`, чтобы браузерные
  Клиенты заранее выбирали правильные этикетки ГАР.
- `sorafs_cli fetch` обеспечивает переопределение `--local-proxy-mode=bridge|metadata-only`
  и `--local-proxy-norito-spool=PATH`, вызывающий смену режима выполнения или следования
  альтернативные катушки без изменений JSON-политика.
- `downgrade_remediation` (опционально) настраивает автоматический переход на более раннюю версию.
  Когда включено, оркестратор следит за реле телеметрии для понижения всплесков и,
  после превышения `threshold` в окне `window_secs`, качество переводит прокси
  в `target_mode` (по умолчанию `metadata-only`). Когда понижения прекращаются,
  прокси возвращается к `resume_mode` после `cooldown_secs`. Используйте массив
  `modes`, чтобы проверить триггер происхождения ролей реле (по умолчанию входные реле).

Когда прокси работает в режиме моста, он обслуживает два приложения:

- **`norito`** — целевой поток клиента одобрен разрешено
  `norito_bridge.spool_dir`. Цели санитизируются (без обхода, без
  абсолютных путей), а если файл без расширения, применять суффикс настроения.
  отправьте полезную нагрузку в браузер.
- **`car`** — цели потока разрешаются внутри `car_bridge.cache_dir`, наследуют
  дефолтное расширение и отклонять сжатые полезные нагрузки, если `allow_zst` не включен.
  Успешный мост отвечает `STREAM_ACK_OK` перед передачей байтов архива, чтобы
  услуги клиентов по проверке трубопровода.

В обоих случаях прокси обеспечивает тег кэша HMAC (если ключ защитного кэша был в
время рукопожатия) и записывает коды причин `norito_*` / `car_*`, чтобы дашборды
имеются успехи, отсутствие файлов и ошибки санитизации.`Orchestrator::local_proxy().await` раскрывает ручку для чтения PEM
получение сертификата, манифест браузера или корректное заключение при выводе
приложения.

Когда прокси включен, он теперь выдает записи **manifest v2**. Помимо
существующий сертификат и ключ защитного кэша, v2 добавлены:

- `alpn` (`"sorafs-proxy/1"`) и массив `capabilities`, чтобы клиенты знали
  протокол потока.
- `session_id` для рукопожатия и `cache_tagging` соляной блок для деривации
  привязка защиты сеанса и теги HMAC.
- Подсказки по выбору цепи и защиты (`circuit`, `guard_selection`,
  `route_hints`) для более богатого пользовательского интерфейса для открытия потоков.
- `telemetry_v2` с ручками сэмплинга и приватности для локальной инструментов.
- Каждый `STREAM_ACK_OK` включает `cache_tag_hex`. Клиенты отражают это значение
  в заголовке `x-sorafs-cache-tag` при запросах HTTP/TCP, чтобы кэшировать охрану
  выборки определенным образом зашифрованными на диске.

Эти поля обратно совместимы — старые клиенты могут оставить новые ключи и
продолжать использовать подмножество v1.

## 2. Семантика отказов

Оркестратор устанавливает строгие возможности проверки и бюджетов при передаче.
первая байта. Отказы повышения на три категории:

1. **Отказы по допуску (предполетные).** Провайдеры без возможности дальности,
   просроченные объявления или временная телеметрия фиксируются на табло
   Документы и не учитываются при планировании. Сводные данные CLI по созданию массива
   `ineligible_providers` причины, чтобы операторы могли увидеть дрейф
   управление без парсинга журналов.
2. **Исчерпание времени выполнения.** Каждый провайдер отслеживает последовательные ошибки.
   Когда настраивается `provider_failure_threshold`, провайдер помечается как
   `disabled` до конца сессии. Если все провайдеры стали `disabled`, оркестратор
   возвращает `MultiSourceError::NoHealthyProviders { last_error, chunk_index }`.
3. **Детерминированные прерывания.** Жесткие лимиты поднимаются как структурные.
   ошибки:
   - `MultiSourceError::NoCompatibleProviders` — манифест требует пролета/выравнивания,
     какие основные провайдеры не могут соблюсти.
   - `MultiSourceError::ExhaustedRetries` — исчерпан бюджет ретраев на чанк.
   - `MultiSourceError::ObserverFailed` — нижестоящие наблюдатели (потоковые перехватчики)
     отклонили проверенный чанк.

каждая ошибка содержит индекс проблемного фрагмента и, когда он доступен, финальную
Причина отказа провайдера. Прочтите эти ошибки, блокировщики выпуска — повторные
попытка с тем же вводом воспроизвести вести сбой, пока реклама, телеметрия или здоровье
провайдер не конвертируется.

### 2.1 Табло Сохранения

При настройке `persist_path` оркестратор записывает итоговое табло после
каждый прогона. Документ JSON содержит:

- `eligibility` (`eligible` или `ineligible::<reason>`).
- `weight` (нормализованный вес, назначенный для этого прогона).
- метаданные `provider` (идентификатор, конечные точки, бюджет параллелизма).

Архивируйте табло снимков вместе с релизными документами, чтобы решить проблему.
черный список и распространение случайно проверяемыми.

### 3.2 Цели структурированного журнала

Оркестратор публикует структурированные события в определенных целях:

- `telemetry::sorafs.fetch.lifecycle` — маркеры `start` и `complete` с номером
  куски, ретраев и всю длительность.
- `telemetry::sorafs.fetch.retry` — события ретраев (`provider`, `reason`,
  `attempts`) для ручной сортировки.
- `telemetry::sorafs.fetch.provider_failure` — провайдеры, отключенные из-за
  повторяющихся ошибок.
- `telemetry::sorafs.fetch.error` — окончательные отказы с `reason` и опциональными
  метаданными провайдера.

Направьте эти потоки в существующий конвейер журналов Norito, чтобы предотвратить инцидент.
ответ был единым источником истины. События жизненного цикла отображаются PQ/classical
микс через `anonymity_effective_policy`, `anonymity_pq_ratio`,
`anonymity_classical_ratio` и соответствующие счетчики, что
дашбордов без парсинга метрик. Во время развертывания GA поддерживается уровень журналов
`info` для событий жизненного цикла/повторных попыток и воспользуйтесь `warn` для ошибок терминала.

### 3.3 Сводки в формате JSON

`sorafs_cli fetch` и Rust SDK возвращают структурированное резюме, независимо:- `provider_reports` с числами успехов/сбоев и статусом отключения провайдера.
- `chunk_receipts`, показывающий какой провайдер обслужил каждый чанк.
- массивы `retry_stats` и `ineligible_providers`.

Архивируйте сводку при отладке проблемных провайдеров — поступления напрямую
соотносятся с лог-метаданными выше.

## 4. Операционный контрольный список

1. **Запустите конфигурацию в CI.** Запустите `sorafs_fetch` с выдвижным ящиком.
   конфигурацией, передайте `--scoreboard-out` для просмотра соответствия критериям фиксации и
   сравните с предыдущим выпуском. Любой неожиданный неподходящий провайдер
   блокирует промоут.
2. **Проверьте телеметрию.** Убедитесь, что развертывание экспортирует метрики.
   `sorafs.fetch.*` и структурированные логи перед включением выборки из нескольких источников
   для пользователей. Отсутствие метрики обычно означает, что фасад оркестратора
   не был вызван.
3. **Документируйте переопределения.** В случае чрезвычайной ситуации `--deny-provider` или
   `--boost-provider` зафиксируйте JSON (или вызов CLI) в журнале изменений. Откаты
   необходимо отменить переопределение и снять новый снимок табло.
4. **Повторите дымовые тесты.** После изменения бюджетов ретраев или шапки
   провайдеры полностью выполняют выборку канонического приспособления
   (`fixtures/sorafs_manifest/ci_sample/`) и убедитесь, что поступления по чанкам
   остаются детерминированными.

Следование шагам выше, делая исполнение оркестратора воспроизводимым в постановочном виде.
развертывание и обеспечение телеметрии для реагирования на инциденты.

конфигурация, задав `policy_override.transport_policy` и
