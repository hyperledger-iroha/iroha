# SoraFS Рынок емкости хранения (проект SF-2c)

Пункт дорожной карты SF-2c представляет собой управляемый рынок, на котором поставщики хранилищ заявляют о выделенной мощности, получают заказы на репликацию и получают вознаграждение, пропорциональное предоставленной доступности. Этот документ определяет объем результатов, необходимых для первого выпуска, и делит их на практические направления.

## Цели

- Определение обязательств по пропускной способности провайдеров (общее количество байтов, лимиты каждой полосы, срок действия) в проверяемой форме, которая может использоваться управлением, транспортом SoraNet и Torii.
- Распределение контактов провайдерам в соответствии с заявленной мощностью, долей и политическими ограничениями при сохранении детерминированного поведения.
- Измерение доставки хранилища (успех репликации, время безотказной работы, проверка целостности) и экспорт телеметрии для распределения платы.
- Обеспечение процессов отзыва и оспаривания для наказания или удаления нечестных поставщиков.

## Концепции домена

| Концепция | Описание | Первоначальный результат |
|---------|------------|---------------------|
| `CapacityDeclarationV1` | Полезная нагрузка Norito, которая описывает идентификатор провайдера, поддержку профиля чанкера, выделенные ГиБ, ограничения для конкретных полос, подсказки по ценам, обязательства по стейкингу и срок действия. | схема + валидатор в `sorafs_manifest::capacity`. |
| `ReplicationOrder` | Инструкция, выпущенная руководством, которая назначает манифест CID одному или нескольким поставщикам, включая уровни избыточности и показатели SLA. | Схема Norito + API смарт-контракта, общий с Torii. |
| `CapacityLedger` | Реестр внутри/вне сети, который отслеживает декларации активной мощности, заказы на репликацию, показатели производительности и начисление комиссий. | Модуль смарт-контракта или внесетевая заглушка службы с детерминированным снимком. |
| `MarketplacePolicy` | Политика управления, определяющая минимальную долю, требования к аудиту и кривые штрафов. | config + документ управления в `sorafs_manifest`. |

## Требования к тестированию и CI

— Модульные тесты для всех новых валидаторов схемы (`sorafs_manifest`).
- Интеграционные тесты, которые имитируют: декларацию → порядок репликации → учет → выплату.
— Рабочий процесс CI для регенерации выборочных деклараций/телеметрии емкости и синхронизации подписей (расширьте `ci/check_sorafs_fixtures.sh`).
- Нагрузочные тесты для API реестра (имитация 10 тысяч поставщиков, 100 тысяч заказов).

## Телеметрия и информационные панели

- Панели приборной панели:
  - Заявленная и используемая мощность по поставщикам.
  - Количество невыполненных заказов на репликацию и средняя задержка назначения.
  - Соблюдение SLA (% времени безотказной работы, уровень успешности PoR).
  - Начисление комиссий и штрафов за эпоху.
- Оповещения:
  - Ниже минимальной гарантированной мощности поставщика.
  - Порядок репликации зависал больше, чем SLA.
  - измерения аварий трубопроводов.

## Контрольный список готовности общедоступной информации

Пункт дорожной карты **SF-2c** обеспечивает развертывание производства на основе убедительных доказательств в области бухгалтерского учета, разрешения споров и внедрения. Используйте приведенные ниже артефакты, чтобы синхронизировать критерии приемки с реализацией.

### Ночной учет и сверка XOR
- Извлеките снимок состояния мощности и экспортируйте реестр XOR для одного и того же окна, затем запустите:
  ```bash
  python3 scripts/telemetry/capacity_reconcile.py \
    --snapshot artifacts/sorafs/capacity/state_$(date +%F).json \
    --ledger artifacts/sorafs/capacity/ledger_$(date +%F).ndjson \
    --label nightly-capacity \
    --json-out artifacts/sorafs/capacity/reconcile_$(date +%F).json \
    --prom-out "${SORAFS_CAPACITY_RECONCILE_TEXTFILE:-artifacts/sorafs/capacity/reconcile.prom}"
  ```
  Этот помощник завершает работу с ненулевым значением в случае отсутствия/переплаты или штрафов и выдает сводку текстового файла Prometheus.
- Предупреждение `SoraFSCapacityReconciliationMismatch` (в `dashboards/alerts/sorafs_capacity_rules.yml`)
  Срабатывает при сообщении о пробелах в показателях сверки; Панели мониторинга находятся в `dashboards/grafana/sorafs_capacity_penalties.json`.
- Архивируйте сводку JSON и хеши под `docs/examples/sorafs_capacity_marketplace_validation/`.
  с пакетами управления.

### Споры и сокращение доказательств
- Файловые споры через `sorafs_manifest_stub capacity dispute` (тесты:
  `cargo test -p sorafs_car --test capacity_cli`), чтобы полезные данные оставались каноническими.
- `cargo test -p iroha_core -- capacity_dispute_replay_is_deterministic` и комплекты штрафов
  Запустите (`record_capacity_telemetry_penalises_persistent_under_delivery`), чтобы сделать детерминированный повтор споров и косых черт.
- Следуйте `docs/source/sorafs/dispute_revocation_runbook.md` для сбора и передачи доказательств; Свяжите одобрения с предупреждением обратно с отчетом о проверке.

### Регистрация поставщика и дымовые тесты на выходе
— Восстановить артефакты объявлений/телеметрии из `sorafs_manifest_stub capacity ...` и воспроизвести тесты CLI перед отправкой (`cargo test -p sorafs_car --test capacity_cli -- capacity_declaration`).
- Отправьте через Torii (`/v2/sorafs/capacity/declare`), затем сделайте снимки экрана `/v2/sorafs/capacity/state` и Grafana. Следуйте выходному потоку до `docs/source/sorafs/capacity_onboarding_runbook.md`.

## Зависимости и последовательность

1. Заполните форму SF-2b (политика приема) – полагается на проверенных на рынке поставщиков.
2. Полная схема + уровень реестра (этот документ) перед интеграцией Torii.
3. Завершите конвейер измерения, прежде чем активировать выплаты.
4. Последний шаг: после проверки данных измерений на этапе подготовки включите распределение комиссий, контролируемое управлением.

Отслеживайте прогресс со ссылкой на этот документ в дорожной карте. Обновите дорожную карту, когда каждый основной раздел (схема, плоскость управления, интеграция, измерение, обработка споров) будет завершен.
