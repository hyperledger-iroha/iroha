Этот runbook осуществляет SRE посредством обучения, развёртывания и использования мульти-источникового fetch-оркестратора. Он выполняет руководство разработчика процедурами, рассчитанными на прод-роллауты, включая поэтапное включение и занесение пиров в черный список.

> **См. также:** [Runbook по мульти-источниковому развертыванию](./multi-source-rollout.md) посвящённым волнам развёртывания на уровне флота и аварийному отклонению провайдеров. Используйте его для согласованного управления/постановки, этот документ — для повседневной эксплуатации оркестратора.

## 1. Предстартовый чек-лист

1. **Собрать входные данные от провайдеров**
   - Последние анонсы провайдеров (`ProviderAdvertV1`) и снимки телеметрии для целевого флота.
   - План полезной нагрузки (`plan.json`), полученный из тестируемого манифеста.
2. **Сформировать детерминированное табло**

   ```bash
   sorafs_fetch \
     --plan fixtures/plan.json \
     --telemetry-json fixtures/telemetry.json \
     --provider alpha=fixtures/provider-alpha.bin \
     --provider beta=fixtures/provider-beta.bin \
     --provider gamma=fixtures/provider-gamma.bin \
     --scoreboard-out artifacts/scoreboard.json \
     --json-out artifacts/session.summary.json
   ```

   - Убедитесь, что `artifacts/scoreboard.json` содержит каждого прод-провайдера как `eligible`.
   - Архивируйте JSON-сводку вместе с табло; аудиторы опираются на счётчики ретраев чанков при сертификации запроса на изменение.
3. **Прогон с приборами** — Выполните ту же команду на общедоступных приборах из `docs/examples/sorafs_ci_sample/`, чтобы убедиться, что бинарник оркестратора соответствует ожидаемой версии, прежде чем трогать полезные нагрузки.

## 2. Причина поэтапного развертывания

1. **Канарейка (≤2 провайдера)**
   - Пересоберите табло и запустите `--max-peers=2`, чтобы проверить оркестратор небольшим подмножеством.
   - Мониторьте:
     - `sorafs_orchestrator_active_fetches`
     - `sorafs_orchestrator_fetch_failures_total{reason!="retry"}`
     - `sorafs_orchestrator_retries_total`
   - Продолжайте, когда доля ретраев держится ниже 1% для полной выборки манифеста и ни один провайдер не накапливает ошибок.
2. **Этап разгона (50% провайдеров)**
   - Увеличьте `--max-peers` и перезапустите новый снимок телеметрии.
   - Сохраняйте каждый запуск с `--provider-metrics-out` и `--chunk-receipts-out`. Храните документы в течение ≥7 дней.
3. **Полное внедрение**
   - Выберите `--max-peers` (или установите его в полный набор подходящих).
   - Включите режим оркестратора в клиентских деплоях: распространите сохранённое табло системы и JSON-конфигурацию через конфигурацию управления.
   - Обновите дашборды, чтобы показать `sorafs_orchestrator_fetch_duration_ms` p95/p99 и гистограммы ретраев по регионам.

## 3. Блокировка и способ пиров

Используйте переопределение политики скоринга в CLI, чтобы разбирать проблемные провайдеры без ожидания обновлений управления.

```bash
sorafs_fetch \
  --plan fixtures/plan.json \
  --telemetry-json fixtures/telemetry.json \
  --provider alpha=fixtures/provider-alpha.bin \
  --provider beta=fixtures/provider-beta.bin \
  --provider gamma=fixtures/provider-gamma.bin \
  --deny-provider=beta \
  --boost-provider=gamma=5 \
  --json-out artifacts/override.summary.json
```- `--deny-provider` вызывает указанный псевдоним из просмотра в текущей сессии.
- `--boost-provider=<alias>=<weight>` поднимается вес провайдера в планировщике. Правила применяются к нормализованному весовому табло и применяются только к локальному запуску.
- Зафиксируйте переопределения в инцидент-заявке и предложите JSON-выходы, чтобы ответственная команда могла согласовать состояние после ограничения первопричины.

Для постоянных изменений обновите исходную телеметрию (с учетом нарушения как штрафного) или пересоздайте рекламу с обновлёнными бюджетными потоками до очистки CLI-override.

## 4. Разбор сбоев

Когда выборка падает:

1. Перед повторным запуском выберите следующие принадлежности:
   - `scoreboard.json`
   - `session.summary.json`
   - `chunk_receipts.json`
   - `provider_metrics.json`
2. Проверьте `session.summary.json` на человекочитаемую ошибку:
   - `no providers were supplied` → проверьте путь к провайдерам и объявлению.
   - `retry budget exhausted ...` → увеличьте `--retry-budget` или создайте нестабильные пиры.
   - `no compatible providers available ...` → проверьте метаданные возможности провайдера-нарушителя.
3. Сопоставьте имя провайдера с `sorafs_orchestrator_provider_failures_total` и создайте тикет-наблюдение, если метрика резко растёт.
4. Воспроизведите выборку из Интернета с `--scoreboard-json` и захваченной телеметрией, чтобы детерминированно воспроизвести свой результат.

## 5. Откат

Чтобы откатить выкат оркестратора:

1. Распространите конфигурацию с `--max-peers=1` (фактически отключает мульти-источниковое планирование) или отправьте клиентов на виртуальный одноисточниковый источник.
2. Уберите любые переопределения `--boost-provider`, чтобы табло вернулось к нейтральному весу.
3. Продолжайте собирать метрики оркестратора минимум сутки, чтобы обеспечить отсутствие оставшихся fetch-операций.

Дисциплинированный сбор артефактов и поэтапные развертывания обеспечивают безопасность использования мультиисточникового оркестратора на разнородных флотах провайдеров при сохранении требований по наблюдению и аудиту.
