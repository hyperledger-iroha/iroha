# Обзор безопасности SF-6

**Обзор оценки:** 10 февраля 2026 г. → 18 февраля 2026 г.  
**Главные темы:** Гильдия инженеров безопасности (`@sec-eng`), Рабочая группа по инструментам (`@tooling-wg`)  
**Периметр:** SoraFS CLI/SDK (`sorafs_cli`, `sorafs_car`, `sorafs_manifest`), API-интерфейсы проверки потоковой передачи, управление манифестами в Torii, интеграция. Sigstore/OIDC, крючки освобождения CI.  
**Артефакты :**  
- Исходный CLI и тесты (`crates/sorafs_car/src/bin/sorafs_cli.rs`)  
- Манифест/доказательство обработчиков Torii (`crates/iroha_torii/src/sorafs/api.rs`)  
- Релиз автоматизации (`ci/check_sorafs_cli_release.sh`, `scripts/release_sorafs_cli.sh`)  
- Определенный ремень безопасности (`crates/sorafs_car/tests/sorafs_cli.rs`, [Rapport de parité GA SoraFS Orchestrator](./orchestrator-ga-parity.md))

## Методология

1. **Ателье моделирования угроз** с картографированием возможностей атаки для постов разработчиков, систем CI и узлов Torii.  
2. **Просмотр кода**: обмен поверхностями идентификаторов (обмен токенов OIDC, подпись без ключа), проверка манифестов Norito и противодавление потоковой передачи доказательств.  
3. **Динамические тесты** для проверки манифестов приборов и симуляции режимов панорамирования (воспроизведение токенов, подделка манифестов, проверка потоков) с помощью проводов паритета и нечетких приводов.  
4. **Проверка конфигурации** и проверка значений по умолчанию `iroha_config`, установка флагов CLI и сценариев выпуска для гарантии выполнения определенных действий и объектов аудита.  
5. **Введение процесса** в подтверждение потока исправлений, операций по устранению и сбору доказательств аудита с участием владельцев релиза Tooling WG.

## Проверка элементов управления

- **Идентификаторы:** Шаблоны CI накладываются на дезориентированную аудиторию и явные данные эмитента; CLI и помощник по выпуску быстро выполняются с помощью `--identity-token-audience` и сопровождают `--identity-token-provider`.  
- **Replay deterministe:** Тесты, которые в течение дня совпадают с потоками положительных/отрицательных значений в отправленных манифестах, гарантируют, что дайджесты не совпадают с оставшимися неопределенными результатами и сигналами перед прикосновением к резолюту.  
- **Потоковая передача с защитой от обратного давления:** Torii рассеивает искажения элементов PoR/PoTR через встроенные каналы, и CLI не сохраняет те эффекты, которые происходят в задержках + несколько примеров проверки, évitant la croissance sanslimite tout en gardant des резюме детерминистские.  
- **Наблюдение:** Проверка потоковой передачи данных (`torii_sorafs_proof_stream_*`) и резюме CLI фиксируют причины прерывания, а также хлебные крошки аудита операторов.  
- **Документация:** Руководства для разработчиков ([индекс разработчика](../developer-index.md), [ссылка на CLI](../developer-cli.md)) сигнализируют о разумных флагах и рабочих процессах.

## Дополнительные сведения о контрольном списке выпуска

Менеджеры по выпуску **doivent** присоединяются к превентивным мерам по продвижению кандидата в GA:1. Хэш-память обзора безопасности плюс недавний документ.  
2. Право залога и билет на исправление ситуации (например, `governance/tickets/SF6-SR-2026.md`).  
3. Вывод `scripts/release_sorafs_cli.sh --manifest ... --bundle-out ... --signature-out ...` с указанием явных аргументов аудитории/эмитента.  
4. Записываются журналы паритетной проводки (`cargo test -p sorafs_car -- --nocapture sorafs_cli::proof_stream::bounded_channels`).  
5. Подтверждение того, что примечания к выпуску Torii включают в себя счетчики телеметрии доказательства потоковой передачи.

Не собирайте артефакты, блокирующие подпись GA.
