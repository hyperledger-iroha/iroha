# Руководство по внедрению и настройке оркестратора

Это руководство основано на [справочнике конфигурации](orchestrator-config.md) и
[развертывание Runbook из нескольких источников] (multi-source-rollout.md). Это мысль,
как настраивать оркестратор для каждого этапа развертывания, как интерпретировать
Материалы табло и сигналы телеметрии должны находиться на месте перед
расширению трафика. Применяйте рекомендации последовательно в CLI, SDK и
автоматизация, чтобы каждый узел отслеживал одну и ту же детерминированную
я принесу.

## 1. Базовые наборы параметров

перейдите к общему шаблону схемы и откорректируйте набор небольших ручек по мере
внедрение прогресса. Таблица ниже фиксирует предпочтительные значения для наиболее
распространённые фазы; параметры, не отраженные в таблице, берутся из
`OrchestratorConfig::default()` и `FetchOptions::default()`.

| Фаза | `max_providers` | `fetch.per_chunk_retry_limit` | `fetch.provider_failure_threshold` | `scoreboard.latency_cap_ms` | `scoreboard.telemetry_grace_secs` | Примечания |
|------|-----------------|-------------------------------|------------------------------------|-------------|--------------------------------|------------|
| **Лаборатория / CI** | `3` | `2` | `2` | `2500` | `300` | Жёсткий лимит задержки и короткая грация‑окно быстро выявляют шумовую телеметрию. Держите повторяет попытки, чтобы раньше выявлять некорректированные проявления. |
| **Постановка** | `4` | `3` | `3` | `4000` | `600` | Отражает продакшн-параметры, оставляя запас для экспериментальных аналогов. |
| **Канарейка** | `6` | `3` | `3` | `5000` | `900` | Соответствует значениям по умолчанию; установите `telemetry_region`, чтобы дашборды могли сегментировать canary‑трафик. |
| **Общая доступность** | `None` (использовать всех подходящих) | `4` | `4` | `5000` | `900` | Увеличьте пороги повторных попыток/ошибок, чтобы учитывать переходные сбои, при этом аудит продолжает обеспечивать детерминизм. |

- `scoreboard.weight_scale` остается на дефолтном `10_000`, если только нижестоящая система не требует другой целочисленной точности. Увеличение масштаба не меняет порядка среди провайдеров; оно лишь создает более плотные распределения кредитов.
- При переходе между фазами сохраните JSON‑bundle и воспользуйтесь `--scoreboard-out`, чтобы аудит‑трейл фиксировал набор параметров Google.

## 2. Табло Гигиена

Табло соответствия требованиям манифеста, объявлений провайдеров и телеметрии.
Перед продвижением:1. **Проверьте свежесть телеметрии.** Убедитесь, что снимки сделаны,
   `--telemetry-json`, сняты в пределах заданного благодатного окна. Записи выше
   `telemetry_grace_secs` завершатся неудачей `TelemetryStale { last_updated }`.
   Рассмотрите это как жёсткую блокировку и обновите экспортную телеметрию, прежде чем продолжить.
2. **Проверьте право на участие.** Сохраните документы через
   `--scoreboard-out=/var/lib/sorafs/scoreboards/preflight.json`. запись
   содержит блок `eligibility` с точной причиной отклонения. Не обходите несоответствия
   возможности или предоставление объявления; исправьте исходный полезный груз.
3. **Проверьте дельты веса.** Сравните поле `normalised_weight` с физическим релизом.
   Сдвиги >10% должны соответствовать намеренным изменениям в объявлениях или телеметрии.
   и должны быть отражены в журнале.
4. **Архивируйте документы.** Настройте `scoreboard.persist_path` для каждого запуска.
   опубликованный финальный снимок табло. Приложите документ к релизному сообщению
   вместе с манифестом и телеметрическим комплектом.
5. **Фиксируйте доказательства микса провайдеров.** Метаданные `scoreboard.json` _и_
   соответствующий `summary.json` следует поддерживать `provider_count`,
   `gateway_provider_count` и вычисленный ярлык `provider_mix`, чтобы ревьюеры могли
   Убедительно, что запуск был `direct-only`, `gateway-only` или `mixed`. Gateway‑снимки
   должны показывать `provider_count=0` и `provider_mix="gateway-only"`, смешанные
   прогоны — ненулевые значения для источников. `cargo xtask sorafs-adoption-check`
   валидирует эти поля (и падает при несоответствии счётчиков/лейблов), поэтому запускайте
   его вместе с `ci/check_sorafs_orchestrator_adoption.sh` или своим скриптом захвата,
   чтобы получить комплект `adoption_report.json`. При использовании шлюзов Torii,
   сохраните `gateway_manifest_id`/`gateway_manifest_cid` в табло метаданных, чтобы
   Ворота усыновления могли сопоставить конверт с манифестом с записанным миксом провайдеров.

Подробные определения полей см. в
`crates/sorafs_car/src/scoreboard.rs` и показывает сводку CLI, выдаваемую
`sorafs_cli fetch --json-out`.

## Справочник флагов CLI и SDK

`sorafs_cli fetch` (см. `crates/sorafs_car/src/bin/sorafs_cli.rs`) и обёртка
`iroha_cli app sorafs fetch` (`crates/iroha_cli/src/commands/sorafs.rs`) использовать
одна и ту же поверхность панели оркестратора. Используйте следующие флаги
при сборе доказательств или воспроизведении канонических приспособлений:

Общая справка по флагам с несколькими источниками (содержит справку CLI и синхронную документацию,
правда только этот файл):- `--max-peers=<count>` ограничивает число правомочных провайдеров, табло проходящего фильтра. Оставьте пустым, чтобы стримить всех подходящих провайдеров, и ставьте `1` только при осознанном упражнении резервного варианта с одним источником. Отражает команду `maxPeers` в SDK (`SorafsGatewayFetchOptions.maxPeers`, `SorafsGatewayFetchOptions.max_peers`).
- `--retry-budget=<count>` прокидывается в лимит повторной попытки на чанк, примененный `FetchOptions`. Используйте развертывание таблицы из указаний для рекомендуемых результатов; CLI‑прогоны, собирающие доказательства, должны соответствовать дефолтам SDK для сохранения паритета.
- `--telemetry-region=<label>` помечает серии Prometheus `sorafs_orchestrator_*` (и OTLP‑реле) лейблом региона/окружения, чтобы дашборды разделяли lab, staging, canary и GA‑трафик.
- `--telemetry-json=<path>` инжектит снимок, указанный в табло. Сохраняйте JSON рядом с табло, чтобы аудиторы могли воспроизвести запуск (и чтобы `cargo xtask sorafs-adoption-check --require-telemetry` доказал, какой OTLP-стрим захват капитала).
- `--local-proxy-*` (`--local-proxy-mode`, `--local-proxy-norito-spool`, `--local-proxy-kaigi-spool`, `--local-proxy-kaigi-policy`) включают крюковые мостики‑наблюдателя. При включении оркестратор стримит куски через локальный прокси Norito/Kaigi, чтобы браузерные клиенты, охраняли кэши и комнаты Kaigi‑rooms применяли те же квитанции, что и Rust.
- `--scoreboard-out=<path>` (опционально с `--scoreboard-now=<unix_secs>`) сохранить право на получение снимков для аудиторов. Всегда связывайте сохранённый JSON с реквизитами телеметрии и манифестами, указанными в релизном билете.
- `--deny-provider name=ALIAS` / `--boost-provider name=ALIAS:delta` анализ детерминированных корректировок уровня метаданных бесплатно. Используйте эти флажки только для повторений; Продакшн-понижение должно осуществляться через артефакты управления, чтобы каждый узел применял один и тот же пакет политик.
- `--provider-metrics-out` / `--chunk-receipts-out` сохраняют метрики здоровья по провайдерам и поступления по частям из развертывания‑чек-листа; приложите оба документа при подаче доказательств об усыновлении.

Пример (с использованием опубликованного приспособления):

```bash
sorafs_cli fetch \
  --plan fixtures/sorafs_orchestrator/multi_peer_parity_v1/plan.json \
  --gateway-provider gw-alpha=... \
  --telemetry-source-label otlp::staging \
  --scoreboard-out artifacts/sorafs_orchestrator/latest/scoreboard.json \
  --json-out artifacts/sorafs_orchestrator/latest/summary.json \
  --provider-metrics-out artifacts/sorafs_orchestrator/latest/provider_metrics.json \
  --chunk-receipts-out artifacts/sorafs_orchestrator/latest/chunk_receipts.json

cargo xtask sorafs-adoption-check \
  --scoreboard artifacts/sorafs_orchestrator/latest/scoreboard.json \
  --summary artifacts/sorafs_orchestrator/latest/summary.json
```

SDK использует ту же настройку через `SorafsGatewayFetchOptions` в Rust‑клиенте
(`crates/iroha/src/client.rs`), JS‑привязки
(`javascript/iroha_js/src/sorafs.js`) и Swift SDK
(`IrohaSwift/Sources/IrohaSwift/SorafsOptions.swift`). Держите этих помощников
синхронно с дефолтами CLI, чтобы операторы могли копировать политику в автоматизации.
без специального слоёв съемки.

## 3. Настройка политики выборки

`FetchOptions` повторная попытка управления, параллелизмом и верификацией. При настройке:- **Повторные попытки:** Повышение `per_chunk_retry_limit` выше `4` увеличивает время восстановления,
  но может маскировать проблемы провайдеров. Предпочитайте держать `4` как потолок и
  Опираются на ротацию провайдеров для определения слабых.
- **Порог отказов:** `provider_failure_threshold` определяет, когда провайдер отключается.
  на оставшейся сессии. Согласуйте это значение с повторной политикой: порог ниже бюджета
  retry заставляет оркестратор выгнать сверстник ещё до исчерпания всех повторных попыток.
- **Параллелизм:** предоставляется `global_parallel_limit` unset (`None`), если только конкретная
  Окружающая среда не может насытить широкий диапазон. Если задано, убедитесь, что значение
  ≤ сумма потоков бюджетов провайдеров, чтобы избежать голода.
- **Тогглы верификации:** `verify_lengths` и `verify_digests` должны оставаться включенными.
  в продакшене. Они соблюдают детерминизм при смешанном составе провайдеров; отключайте
  их только в изолированных фаззингах-средах.

## 4. Стадирование транспорта и анонимности

Используйте поля `rollout_phase`, `anonymity_policy` и `transport_policy`, чтобы
опишите приватностную позу:

- Предпочитайте `rollout_phase="snnet-5"` и разрешите дефолтную политику анонимности.
  пройти этапы SNNet-5. Переопределяйте только через `anonymity_policy_override`
  когда руководство издает подписанную директиву.
- Держите `transport_policy="soranet-first"` на базе, пока SNNet-4/5/5a/5b/6a/7/8/12/13 находится в 🈺
  (см. `roadmap.md`). Используйте `transport_policy="direct-only"` только для документированных
  понижение рейтинга/комплаенс‑учений и дождитесь пересмотра PQ‑покрытия перед повышением до
  `transport_policy="soranet-strict"` — этот режим быстро упадёт, если подождать лишь
  классическое реле.
- `write_mode="pq-only"` следует применять только тогда, когда все пишут‑пути (SDK, оркестратор,
  инструменты управления) возможности поддерживать PQ-требования. Во время развертывания держите
  `write_mode="allow-downgrade"`, чтобы аварийные режимы можно было использовать
  маршруты, пока телеметрия отмечает понижение рейтинга.
- Выбор охранников и промежуточных схем зависит от каталога SoraNet. Передавайте подписанный
  снимок `relay_directory` и сохраните кеш `guard_set`, чтобы обеспечить стабильность работы охранников.
  в согласованном удержании‑окне. Кэш отпечаток, зафиксированный `sorafs_cli fetch`, входит
  в развертывании доказательств.

## 5. Хуки деградации и комплаенса

Две подсистемы оркестратора помогают обеспечить политику без ручного контроля:- **Устранение деградаций** (`downgrade_remediation`): отслеживает события.
  `handshake_downgrade_total` и, после превышения `threshold` в пределах `window_secs`,
  переводит локальный прокси в `target_mode` (по умолчанию только метаданные). Сохраняйте
  дефолты (`threshold=3`, `window=300`, `cooldown=900`), если только постмортемы не
  появляется другой узор. Документируйте любые переопределения в журнале развертывания и
  убедитесь, что дашборды отслеживают `sorafs_proxy_downgrade_state`.
- **Политика комплаенса** (`compliance`): исключения в соответствии с юрисдикциями и манифестами.
  передача через организованный отказ, управляемое управление. Никогда не встраивайте
  специальное переопределение в конфигурации пакета; вместо этого запроса отправьте подписанное обновление
  `governance/compliance/soranet_opt_outs.json` и разверните сгенерированный JSON.

Для соответствующей системы сохраните итоговую схему пакета и включите его в доказательства.
релиза, чтобы аудиторы могли отслеживать характер дауншифтов.

##6. Телеметрия и дашборды

Перед выдвижением убедитесь, что в зоне выдвижения активны следующие сигналы:

- `sorafs_orchestrator_fetch_failures_total{reason="no_healthy_providers"}` —
  должно быть нулевым после завершения канарейки.
- `sorafs_orchestrator_retries_total` и
  `sorafs_orchestrator_retry_ratio` — должен стабилизироваться ниже 10% во время канарейки
  и остаюсь ниже 5% после GA.
- `sorafs_orchestrator_policy_events_total` — обеспечение устойчивости вирусной стадии
  выкат (лейбл `stage`) и фиксирует провалы напряжения через `outcome`.
- `sorafs_orchestrator_pq_candidate_ratio` /
  `sorafs_orchestrator_pq_deficit_ratio` — отслеживают доступность PQ‑реле соответственно
  политика ожиданий.
- Лог‑таргеты `telemetry::sorafs.fetch.*` — необходимо перейти в общий лог‑агрегатор с
  сохранёнными запросами для `status=failed`.

Загрузите канонический Grafana‑дашборд из
`dashboards/grafana/sorafs_fetch_observability.json` (экспортируется в портале под
**SoraFS → Получить наблюдаемость**), чтобы выбрать регион/манифест, повторить попытки тепловой карты
по провайдерам, гистограммы задержек чанков и счётчики стагнации имен
тем более, что SRE проверяют во время работы. Подключите правила Alertmanager в
`dashboards/alerts/sorafs_fetch_rules.yml` и проверьте синтаксис Prometheus через
`scripts/telemetry/test_sorafs_fetch_alerts.sh` (помощник автоматически запускает
`promtool test rules` локально или в Docker). Для передачи оповещения требуется то же самое
блок маршрутизации, который печатает скрипт, чтобы операторы могли приложить доказательства
к внедрению‑тикету.

## Процесс прожига телеметрии

Дорожная карта пункта **SF-6e** требует 30-дневной проработки телеметрии перед переключением
множественный источник оркестратора на GA‑дефолты. Используйте скрипты репозитория, чтобы
Ежедневно собирайте воспроизводимый пакет документов по всему окну:

1. Запустите `ci/check_sorafs_orchestrator_adoption.sh` с установленными переменными.
   выгорание. Пример:

   ```bash
   SORAFS_BURN_IN_LABEL=canary-week-1 \
   SORAFS_BURN_IN_REGION=us-east-1 \
   SORAFS_BURN_IN_MANIFEST=manifest-v4 \
   SORAFS_BURN_IN_DAY=7 \
   SORAFS_BURN_IN_WINDOW_DAYS=30 \
   ci/check_sorafs_orchestrator_adoption.sh
   ```Помощник проигрывает `fixtures/sorafs_orchestrator/multi_peer_parity_v1`,
   записывает `scoreboard.json`, `summary.json`, `provider_metrics.json`,
   `chunk_receipts.json` и `adoption_report.json` в
   `artifacts/sorafs_orchestrator/<timestamp>/`, и последнее минимальное число
   правомочных‑провайдеров через `cargo xtask sorafs-adoption-check`.
2. Когда переменные горения отображаются, график также выдает `burn_in_note.json`,
   фиксируя метку, индекс дня, идентификатор манифеста, источник телеметрии и дайджесты документов.
   Приложите этот JSON к журналу развертывания, чтобы было ясно, какая запись закрыта для каждой
   день 30‑дневного окна.
3. Импортируйте обновлённый Grafana‑дашборд (`dashboards/grafana/sorafs_fetch_observability.json`)
   в рабочей области подготовки/производства, согласно его ярлыку для записи, и убедитесь, что каждый
   Панель отображает выборки для тестируемых манифеста/региона.
4. Запускайте `scripts/telemetry/test_sorafs_fetch_alerts.sh` (или `promtool test rules …`)
   всегда при уходе `dashboards/alerts/sorafs_fetch_rules.yml`, чтобы задокументировать
   Соответствие маршрутизации оповещений-метрикам, экспортированным в Burn-in.
5. Архивируйте снимок дашборда, выведите тест alert‑ов и хвост журналов по запросам.
   `telemetry::sorafs.fetch.*` вместе с документами оркестратора, чтобы управление могло осуществляться.
   воспроизвести доказательства без обращения к live‑системам.

## 7. Развертывание чек‑листа

1. Пересоберите табло в CI с кандидатской конфигурацией и сохраните документы под VCS.
2. Запустите определенные устройства выборки в каждом процессе (лаборатория, постановка, канарейка, производство).
   и приложите реквизиты `--scoreboard-out` и `--json-out` к записи о развертывании.
3. Проверьте телеметрические приборные панели совместно с дежурным инженером, убедившись, что все
   Указанные метрики имеют живые выборки.
