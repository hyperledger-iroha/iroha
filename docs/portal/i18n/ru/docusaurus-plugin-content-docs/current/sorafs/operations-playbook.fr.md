## Шлюз Panne/disponabilité dégradée

**Обнаружение**

- Предупреждения: `SoraFSGatewayAvailabilityDrop`, `SoraFSGatewayLatencySlo`.
- Приборная панель: `dashboards/grafana/sorafs_gateway_overview.json`.

**Действия немедленные**

1. Подтвердите порт (fournisseur unique vs flotte) через панель taux de requêtes.
2. Выполните маршрутизацию Torii с помощью четырехпользовательского режима (si multi-fournisseur) и базового `sorafs_gateway_route_weights` в настройках конфигурации (`docs/source/sorafs_gateway_self_cert.md`).
3. Если все четыре специалиста задействованы, активируйте резервную «прямую выборку» для клиентов CLI/SDK (`docs/source/sorafs_node_client_protocol.md`).

**Сортировка**

- Проверьте использование токенов потока по взаимосвязи с `sorafs_gateway_stream_token_limit`.
- Проверьте шлюз журналов на наличие ошибок TLS или доступа.
- Выполните `scripts/telemetry/run_schema_diff.sh` для проверки того, что экспортированная схема шлюза соответствует существующей версии.

**Варианты исправления**

- Redémarrez уникальный шлюз процесса, затронутый; évitez de recycler tout le cluster sauf si plusieurs fournisseurs échouent.
- Временное увеличение лимита токенов потока на 10–15 %, если насыщение подтверждено.
- Relancez le self-cert (`scripts/sorafs_gateway_self_cert.sh`) после стабилизации.

**После инцидента**

- Редигес посмертно P1 с `docs/source/sorafs/postmortem_template.md`.
- Планируйте учения по устранению хаоса и устранению необходимости ручного вмешательства.

## Предварительное изображение (PoR/PoTR)

**Обнаружение**

- Предупреждения: `SoraFSProofFailureSpike`, `SoraFSPoTRDeadlineMiss`.
- Приборная панель: `dashboards/grafana/sorafs_proof_integrity.json`.
- Телеметрия: `torii_sorafs_proof_stream_events_total` и элементы `sorafs.fetch.error` с `provider_reason=corrupt_proof`.

**Действия немедленные**1. Рис. новые поступления манифестов в реестр манифестов (`docs/source/sorafs/manifest_pipeline.md`).
2. Уведомление о регулировании для приостановки воздействия воздействий на специалистов.

**Сортировка**

- Проверьте знатока файла проблем PoR, лицом к `sorafs_node_replication_backlog_total`.
- Подтвердите конвейер предварительной проверки (`crates/sorafs_node/src/potr.rs`) для недавних развертываний.
- Сравните версии встроенного программного обеспечения четырехнаборщиков с регистрацией операторов.

**Варианты исправления**

- Закрытие повторов PoR через `sorafs_cli proof stream` с более ранним манифестом.
- Если вы отдаете предпочтение слаженному манифесту, отмените работу учителя ансамбля в течение дня в регистре управления и заставьте обновить табло оркестратора.

**После инцидента**

- Рассмотрите сценарий тренировки хаоса PoR перед развертыванием prochain в производстве.
- Отправьте результаты по шаблону посмертного обследования и отправьте их в контрольный список квалификации специалистов.

## Задержка репликации/круассан дю отставание

**Обнаружение**

- Предупреждения: `SoraFSReplicationBacklogGrowing`, `SoraFSCapacityPressure`. Импортез
  `dashboards/alerts/sorafs_capacity_rules.yml` и выполнить
  `promtool test rules dashboards/alerts/tests/sorafs_capacity_rules.test.yml`
  Авангардное продвижение для того, чтобы Alertmanager отражал все ваши документы.
- Приборная панель: `dashboards/grafana/sorafs_capacity_health.json`.
- Метры: `sorafs_node_replication_backlog_total`, `sorafs_node_manifest_refresh_age_seconds`.

**Действия немедленные**

1. Проверьте порт отставания (fournisseur unique или flotte) и приостановите ненужные этапы репликации.
2. Если резервная копия изолирована, временно воздействуйте на новые команды альтернативно с помощью планировщика репликации.

**Сортировка**

- Проверьте оркестратор телеметрии для повторных попыток, которые позволяют выявить отставание.
- Подтвердите наличие запаса кабелей на достаточном уровне (`sorafs_node_capacity_utilisation_percent`).
- Пройдите проверку последних изменений конфигурации (пропустите в течение дня профиль фрагмента, периодичность проверки).

**Варианты исправления**

- Выполните `sorafs_cli` с опцией `--rebalance` для перераспределения контента.
- Горизонтальное масштабирование рабочих репликации для ударного воздействия.
- Отмените обновление манифестов для выравнивания окон TTL.

**После инцидента**

- Планируйте тренировку емкости для выполнения упражнений по насыщению.
- Ознакомьтесь с документацией SLA по репликации в `docs/source/sorafs_node_client_protocol.md`.

## Каденция упражнений хаоса

- **Триместриэль**: сочетание симуляции шлюза панорамы + оркестратора темпа повторных попыток.
- **Семестриэль** : инъекции PoR/PoTR для двух четырехместных тренажёров с восстановлением.
- **Выборочное меню**: сценарий задержки репликации с проявлениями промежуточной подготовки.
- Используйте тренировки в журнале Runbook Partage (`ops/drill-log.md`) через:

  ```bash
  scripts/telemetry/log_sorafs_drill.sh \
    --scenario "Gateway outage chaos drill" \
    --status pass \
    --ic "Alex Morgan" \
    --scribe "Priya Patel" \
    --notes "Failover to west cluster succeeded" \
    --log ops/drill-log.md \
    --link "docs/source/sorafs/postmortem_template.md"
  ```

- Проверьте журнал перед коммитами:

  ```bash
  scripts/telemetry/validate_drill_log.sh
  ```- Используйте `--status scheduled` для возврата сверл, `pass`/`fail` для завершения работы и `follow-up` для оставшихся действий.
- Замените пункт назначения с `--log` для пробных прогонов или автоматической проверки; Без этого сценарий продолжит работу в течение дня `ops/drill-log.md`.

## Шаблон посмертного заключения

Используйте `docs/source/sorafs/postmortem_template.md` для каждого инцидента P1/P2 и для ретроспектив учений хаоса. Шаблон содержит хронологию, количественную оценку воздействия, факторы, вносящие вклад, корректирующие действия и методы проверки правильности.
