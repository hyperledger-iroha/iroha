# План внедрения реестра контактов SoraFS (SF-4)

SF-4 вводит договор с реестром контактов и вспомогательными службами, которые находятся в Альмасенане
компрометация манифеста, получение политических решений и демонстрация API-интерфейсов Torii, шлюзы
y orquestadores. Этот документ включает в себя план проверки стоимости
конкретная реализация, подключение логики в цепочке, лос-сервисы дель-хоста, лос
приспособления и все необходимые оперативные материалы.

## Альканс

1. **Макина статуса реестра**: реестры, определенные для Norito для манифестов,
   псевдонимы, последующие коды, эпохи сохранения и метаданные правительства.
2. **Реализация контракта**: действия CRUD определяются для циклической жизни
   де булавки (`ReplicationOrder`, `Precommit`, `Completion`, выселение).
3. **Доступ к обслуживанию**: конечные точки gRPC/REST перераспределяются по используемому реестру.
   Torii и SDK, включая разбивку на страницы и подтверждение.
4. **Инструменты и приспособления**: помощники по CLI, векторы проверки и документация для обслуживания.
   манифесты, псевдонимы и конверты синхронизированного правительства.
5. **Телеметрия и операции**: метрики, оповещения и Runbook для обслуживания реестра.

## Модель данных

### Центральные регистры (Norito)

| Эструктура | Описание | Кампос |
|------------|-------------|--------|
| `PinRecordV1` | Канонический вход в манифест. | `manifest_cid`, `chunk_plan_digest`, `por_root`, `profile_handle`, `approved_at`, `retention_epoch`, `pin_policy`, `successor_of`, `governance_envelope_hash`. |
| `AliasBindingV1` | Псевдоним Mapea -> CID манифеста. | `alias`, `manifest_cid`, `bound_at`, `expiry_epoch`. |
| `ReplicationOrderV1` | Инструкции по тому, как поставщики должны зафиксировать манифест. | `order_id`, `manifest_cid`, `providers`, `redundancy`, `deadline`, `policy_hash`. |
| `ReplicationReceiptV1` | Ответ на вопрос поставщика. | `order_id`, `provider_id`, `status`, `timestamp`, `por_sample_digest`. |
| `ManifestPolicyV1` | Снимок государственной политики. | `min_replicas`, `max_retention_epochs`, `allowed_profiles`, `pin_fee_basis_points`. |

Ссылка на реализацию: версия `crates/sorafs_manifest/src/pin_registry.rs` для лос.
esquemas Norito в Rust и помощниках по проверке, которые обновляют эти реестры. Ла
проверка проверки инструментов манифеста (поиск в реестре блоков, контроль политики закрепления)
Для того, чтобы договориться, фасады Torii и CLI сравнивают неизменяемые идентификаторы.Тарес:
- Завершите выполнение заданий Norito и `crates/sorafs_manifest/src/pin_registry.rs`.
— Генерация кода (Rust + другие SDK) с использованием макросов Norito.
- Актуализировать документацию (`sorafs_architecture_rfc.md`) для всех этих списков.

## Реализация контракта

| Тарея | Ответственный(а) | Заметки |
|------|----------------|------|
| Внедрить резервирование реестра (sled/sqlite/off-chain) или по модулю смарт-контракта. | Основная инфраструктура/команда смарт-контрактов | Проверяйте детерминированное хеширование, evitar punto flotante. |
| Точки входа: `submit_manifest`, `approve_manifest`, `bind_alias`, `issue_replication_order`, `complete_replication`, `evict_manifest`. | Основная инфраструктура | Подтвердите план проверки `ManifestValidator`. Привязка псевдонима сейчас осуществляется через `RegisterPinManifest` (DTO Torii) с `bind_alias`, предназначенным для успешной реализации. |
| Transiciones de estado: самозванец (манифест A -> B), epocas de retencion, unicidad de alias. | Совет управления / Core Infra | Unicidad de alias, limites de retencion y verificaciones de aprobacion/retiro de predecesores viven en `crates/iroha_core/src/smartcontracts/isi/sorafs.rs`; обнаружение многопролетных последовательностей и контроль репликации в случае возобновления работы. |
| Параметры губернаторов: cargar `ManifestPolicyV1` из конфигурации/estado de gobernanza; разрешить актуализацию через события правительства. | Совет управления | Используйте CLI для реализации политики. |
| Отправка событий: отправка событий Norito для телеметрии (`ManifestApproved`, `ReplicationOrderIssued`, `AliasBound`). | Наблюдаемость | Определить порядок событий + ведение журнала. |

Пруэбас:
- Тесты единой точки входа в систему (positivo + rechazo).
- Тесты на навыки для последовательного развития (без циклов, монотонных эпох).
- Fuzz de validacion Generando проявляется в алеаториях (acotados).

## Сервисное обслуживание (Интеграция Torii/SDK)

| Компонент | Тарея | Ответственный(а) |
|-----------|-------|----------------|
| Сервис Torii | Exponer `/v1/sorafs/pin` (отправка), `/v1/sorafs/pin/{cid}` (поиск), `/v1/sorafs/aliases` (список/привязка), `/v1/sorafs/replication` (заказы/поступления). Проверяйте страницы + фильтр. | Сетевые TL/Core Infra |
| Атестасьон | Включить альтернативу/хэш реестра в ответах; совокупная структура подтверждения Norito, потребляемая SDK. | Основная инфраструктура |
| интерфейс командной строки | Расширитель `sorafs_manifest_stub` или новый CLI `sorafs_pin` с `pin submit`, `alias bind`, `order issue`, `registry export`. | Инструментальная рабочая группа |
| SDK | Генерация клиентских привязок (Rust/Go/TS) для конкретной задачи Norito; Агрегарные тесты интеграции. | Команды SDK |

Операции:
— Объединение кэша/ETag для конечных точек GET.
- Ограничение скорости проверки / аутентификация соответствует политике Torii.

## Светильники и CI- Директория приспособлений: `crates/iroha_core/tests/fixtures/sorafs_pin_registry/` для создания моментальных снимков фирм манифеста/псевдонима/заказа, повторно созданных для `cargo run -p iroha_core --example gen_pin_snapshot`.
- Шаг CI: `ci/check_sorafs_fixtures.sh` восстанавливает моментальный снимок и обнаруживает различия, сохраняя подключенные устройства CI.
- Тесты интеграции (`crates/iroha_core/tests/pin_registry.rs`) позволяют проверить поток дубликатов, защиту от апробации/сохранения псевдонимов, дескрипторы опреснения фрагментов, проверку содержимого реплик и защиту от последовательностей (punteros). desconocidos/preaprobados/retirados/autorreferencias); Ver casos `register_manifest_rejects_*` для подробной информации о кобертах.
- Унитарные тесты для проверки псевдонима, защиты от удержания и проверки преемника в `crates/iroha_core/src/smartcontracts/isi/sorafs.rs`; обнаружение многоступенчатого соединения после выхода на станцию ​​состояния.
- Золотой формат JSON для событий, используемых в конвейерах наблюдения.

## Телеметрия и наблюдение

Метрики (Prometheus):
- `torii_sorafs_registry_manifests_total{status="pending|approved|retired"}`
- `torii_sorafs_registry_aliases_total`
- `torii_sorafs_registry_orders_total{status="pending|completed|expired"}`
- `torii_sorafs_replication_sla_total{outcome="met|missed|pending"}`
- `torii_sorafs_replication_completion_latency_epochs{stat="avg|p95|max|count"}`
- `torii_sorafs_replication_deadline_slack_epochs{stat="avg|p95|max|count"}`
- Существующая телеметрия поставщиков (`torii_sorafs_capacity_*`, `torii_sorafs_fee_projection_nanos`) включена в список сквозных информационных панелей.

Журналы:
- Поток событий Norito создан для правительственных аудиторий (фирм?).

Оповещения:
- Ожидаемые условия репликации превышают соглашение об уровне обслуживания.
- Срок действия псевдонима для удаления тени.
- Нарушения сроков хранения (обнаружено отсутствие обновлений до истечения срока действия).

Панели мониторинга:
- В формате JSON Grafana `docs/source/grafana_sorafs_pin_registry.json` отображаются общие циклы отображения манифестов, настройки псевдонимов, насыщенность невыполненной работы, соотношение SLA, наложения задержки и резерва, а также действия по выполнению заказов для проверки по вызову.

## Runbook и документация

- Актуализируйте `docs/source/sorafs/migration_ledger.md` для включения актуализации состояния реестра.
- Руководство по эксплуатации: `docs/source/sorafs/runbooks/pin_registry_ops.md` (опубликовано) с доступом к метрикам, оповещениям, удалению, резервному копированию и восстановлению данных.
- Guia de gobernanza: описание параметров политики, рабочего процесса апробации, управления спорами.
- Страницы ссылок API для конечной точки (документы Docusaurus).

## Зависимость и секвенциация

1. Завершите выполнение плана проверки (интеграция ManifestValidator).
2. Завершить настройку Norito + настройки по умолчанию.
3. Внедрить контракт + обслуживание, подключить телеметрию.
4. Регенерируйте светильники, исправьте их интеграцию.
5. Актуализируйте документы/книги выполнения и отмечайте элементы дорожной карты как завершенные.

Каждый контрольный список SF-4 должен быть ссылаться на этот план, когда он будет продвигаться вперед.
Фашада REST сейчас находится между конечными точками списка с подтверждением:

- `GET /v1/sorafs/pin` y `GET /v1/sorafs/pin/{digest}` манифесты devuelven con
  привязки псевдонимов, порядок репликации и объект подтверждения, полученный
  хеш последнего блока.
- `GET /v1/sorafs/aliases` и `GET /v1/sorafs/replication` экспонируются в каталоге
  псевдоним активности и невыполненный заказ репликации с последовательным разбиением на страницы
  фильтры состояния.CLI включает эти сообщения (`iroha app sorafs pin list`, `pin show`, `alias list`,
`replication list`), чтобы операторы могли автоматизировать аудитории
реестр не содержит API-интерфейсов нижнего уровня.
