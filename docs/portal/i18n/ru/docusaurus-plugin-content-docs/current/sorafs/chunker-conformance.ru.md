:::note Канонический источник
:::

Этот документ фиксирует требования, которым должен соответствовать уровень поставок, чтобы
Остается совместимым с детерминированным профилем chunker SoraFS (SF1). Он также
документирует регенерацию рабочего процесса, политику подписей и шаги верификации, чтобы
Пользовательские приспособления в SDK выполняются синхронизированными.

## Канонический профиль

- Входной семя (шестнадцатеричное): `0000000000dec0ded`
- Целевой размер: 262144 байт (256 КиБ)
- Минимальный размер: 65536 байт (64 КиБ)
- Максимальный размер: 524288 байт (512 КиБ)
- Скользящий полином: `0x3DA3358B4DC173`
- Шестерня таблицы семян: `sorafs-v1-gear`
- Маска разрыва: `0x0000FFFF`

Эталонная продажа: `sorafs_chunker::chunk_bytes_with_digests_profile`.
Любое SIMD-ускорение должно выдавать определенные границы и дайджесты.

## Набор светильников

`cargo run --locked -p sorafs_chunker --bin export_vectors` регенерирует
приспособления и выпускает следующие файлы в `fixtures/sorafs_chunker/`:

- `sf1_profile_v1.{json,rs,ts,go}` — канонические границы чанков для
  потребители Rust, TypeScript и Go. Каждый документ объявляет канонический дескриптор как
  `sorafs.sf1@1.0.0`, затем `sorafs.sf1@1.0.0`). Порядок фиксируется
  `ensure_charter_compliance` и НЕ ДОЛЖЕН обновляться.
- `manifest_blake3.json` — BLAKE3-верифицированный манифест, покрывающий каждый файл фикстурами.
- `manifest_signatures.json` — подключение совета (Ed25519) верхнего дайджеста манифеста.
- `sf1_profile_v1_backpressure.json` и сырые тела в `fuzz/` —
  определенные сценарии потоковой передачи, используемые в тестах чанкера противодавления.

### Политика подписей

Регенерация светильников **должна** включать действительную подпись совета. Генератор
отклоняет незарегистрированный вывод, если явно не передан `--allow-unsigned` (предназначен
только для локальных экспериментов). Конверты подписей только для добавления и
дедулицируются по подписанту.

Чтобы добавить подпись совета:

```bash
cargo run --locked -p sorafs_chunker --bin export_vectors \
  --signing-key=<ed25519-private-key-hex> \
  --signature-out=fixtures/sorafs_chunker/manifest_signatures.json
```

## Верификация

Помощник CI `ci/check_sorafs_fixtures.sh` повторно запускает генератор с
`--locked`. Если светильники расходуются или отключаются, работа падает. Используйте
этот скрипт в ночных рабочих процессах и перед отправкой изменений приборов.

Шаги ручной проверки:

1. Запустите `cargo test -p sorafs_chunker`.
2. Выполните `ci/check_sorafs_fixtures.sh` локально.
3. Убедитесь, что `git status -- fixtures/sorafs_chunker` чист.

## Обновления Плейбука

При предложении нового профиля chunker или обновлении SF1:

См. также: [`docs/source/sorafs/chunker_profile_authoring.md`](./chunker-profile-authoring.md) для
требования к метаданным, шаблонам предложений и контрольным спискам проверки.1. Подготовьте `ChunkProfileUpgradeProposalV1` (см. RFC SF-1) с новыми параметрами.
2. Регенерируйте фикстуры через `export_vectors` и зафиксируйте новый дайджест-манифест.
3. Напишите манифест предписанным кворумом совета. Все приобретения должны быть
   добавлены в `manifest_signatures.json`.
4. Обновите фиксы SDK (Rust/Go/TS) и обеспечьте паритет между временем выполнения.
5. Регенерируйте ненужные корпуса при настройке параметров.
6. Обновите это руководство новой ручкой профиля, семенами и дайджестом.
7. Отредактируйте изменения вместе с обновленными и roadmap-обновлениями.

Изменения, которые затрагивают границы чанков или дайджестов без учета этого процесса,
недействительны и не должны быть смержены.
