# Дорожная карта путешествий SoraFS (SF-1)

Этот документ операционализирует рекомендации по привязке, зафиксированные в
`docs/source/sorafs_architecture_rfc.md`. Он разворачивает результаты СФ-1 в
готовые к выполнению транспортных средств, критерии гейтов и чек-листы, владельцы команды
Артефакты к публикации на базе SoraFS.

Дорожная карта намеренно определена: карта веха криптовалюты
документы, команды и этапы аттестации, чтобы произвести последующие пайплайны
определенные результаты, управление сохраняет аудируемый след.

## Обзор вех

| Веха | Окно | Основные цели | Должность быть доставлено | Владельцы |
|------|------|---------------|-------------------------|-----------|
| **M1 – Детерминированное применение** | Недели 7-12 | Принудить подписанные данные и подготовить доказательства псевдонимов, пока пайплайны включат флаги ожидания. | Ночная проверка, манифесты с подписью совета, размещение записей в реестре псевдонимов. | Хранение, управление, SDK |

Статус автомобиля отслеживается в `docs/source/sorafs/migration_ledger.md`. Все изменения
в этой дорожной карте ДОЛЖНЫ обновлять реестр, чтобы управлять и разрабатывать релизы.
остаются синхронизированными.

Переопределения разработчиков требуют отказа от управления в PR. | Инструментальная рабочая группа | Лог CI, ссылка на билет отказа (если применимо). |
| Флаги ожидания | М1 | Пайплайны вызывает `sorafs_manifest_stub` с явными ожиданиями для вывода результата: | Документы КИ | Обновленные скрипты со ссылкой на флаги ожидания (см. блок команды ниже). |

#### Каноническая команда публикации

```bash
cargo run -p sorafs_manifest --bin sorafs_manifest_stub -- docs/book \
  --manifest-out artifacts/docs/book/2025-11-01/docs.manifest \
  --manifest-signatures-out artifacts/docs/book/2025-11-01/docs.manifest_signatures.json \
  --car-out artifacts/docs/book/2025-11-01/docs.car \
  --chunk-fetch-plan-out artifacts/docs/book/2025-11-01/docs.fetch_plan.json \
  --car-digest=13fa919c67e55a2e95a13ff8b0c6b40b2e51d6ef505568990f3bc7754e6cc482 \
  --car-size=429391872 \
  --root-cid=f40101... \
  --dag-codec=0x71
```

Замените значения дайджеста, размера и CID на ожидаемые ссылки, опровергнутые в записи.
Миграционный реестр для документов.

### 3. Переход на псевдонимы и общение| Шаг | Веха | Описание | Владелец(и) | Выход |
|-----|------|----------|----------|-------|
| Доказательства псевдонимов в постановке | М1 | Зарегистрируйте утверждения псевдонимов в промежуточном реестре контактов и прикрепите доказательства Merkle к манифестам (`--alias`). | Управление, Документы | Пакет доказательств рядом с манифестом + книга комментариев с псевдонимом имени. |
| Обеспечение соблюдения доказательств | М2 | Шлюзы отклоняют манифесты без новых заголовков `Sora-Proof`; CI получает шаг `sorafs alias verify` для сбора доказательств. | Сеть | Патч конфигурации шлюза + вывод CI с успешной проверкой. |

### 4. Коммуникации и аудит

- **Дисциплина,** каждое изменение состояния (смещение прибора, отправка в реестр,
  активация псевдонима) необходимо внести датированную заметку в
  `docs/source/sorafs/migration_ledger.md`.
- **Протокол управления:** заседания совета, утверждающие изменения в реестре контактов или
  псевдоним Политик, должен ссылаться на эту дорожную карту и бухгалтерскую книгу.
- **Внешние коммуникации:** DevRel публикует обновления на каждом этапе (блог +
  выдержка из журнала изменений), включая определенные гарантии и псевдонимы таймлайнов.

## Зависимости и риски

| Зависимость | Оценка | Смягчение |
|------------|---------|-----------|
| Доступность контракта Реестр контактов | Блокирует выкатывание M2 вперед. | Подготовить контракт на М2 с повторными тестами; поддерживают резервный вариант конверта до последствий регрессий. |
| Ключи от совета | Требуются конверты с манифестами и разрешения реестра. | Церемония подписания описана в `docs/source/sorafs/signing_ceremony.md`; ротировать ключи с переводом и записью в бухгалтерскую книгу. |
| Каденция релизов SDK | Клиенты должны уважать доказательства псевдонимов в M3. | Синхронизировать Windows-релизы SDK с Milestone Gates; добавить контрольные списки миграции в шаблоны релизов. |

Остаточные риски и смягчение последствий отражены в `docs/source/sorafs_architecture_rfc.md`
и должен кросс-референситься при изменении.

## Управление изменениями

1. Предлагайте изменения через PR, обновляя этот файл *и**
   `docs/source/sorafs/migration_ledger.md`.
2. Ссылайтесь на протоколы управления и доказательства CI в описании PR.
3. После слияния уведомить хранилище + список рассылки DevRel с резюме и ожидаемыми.
   действиями операторов.

Следование этой процедуры гарантирует, что развертывание SoraFS останется детерминированным,
аудируемым и искренним между командами, участвующими в старте Nexus.
