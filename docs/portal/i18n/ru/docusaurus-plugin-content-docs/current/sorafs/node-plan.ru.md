SF-3 представляет первый исполняемый ящик `sorafs-node`, который превращает процесс Iroha/Torii в поставщика хранения SoraFS. Используйте этот план вместе с [гайдом по хранилищу узла](node-storage.md), [политикой допуска провайдеров](provider-admission-policy.md) и [дорожной картой маркетплейса емкости хранения](storage-capacity-marketplace.md) при выстраивании последовательности работ.

##Целевой объём (веха M1)

1. **Интеграционное хранилище фрагментов.** Обернуть `sorafs_car::ChunkStore` в персистентный бэкэнд, который хранит байты чанков, манифесты и PoR-деревья в заданной директории данных.
2. **Шлюз Endpoints.** Открытие Norito HTTP-эндпоинты для подачи PIN-кода, получения чанков, выборки PoR и хранения телеметрии внутри процесса Torii.
3. **Прокладка конфигурации.** Добавить структуру конфигурации `SoraFsStorage` (флаг включения, емкости, директории, лимит конкурентности), проводимую через `iroha_config`, `iroha_core` и `iroha_torii`.
4. **Квоты/планирование.** Примените лимиты диска/параллелизма оператора и поставьте команду по очереди с противодавлением.
5. **Телеметрия.** Эмитировать метрики/логи по успешным выводам, задержке выборки чанков, возможностей емкости и результатов выборки PoR.

### B. Персистентный магазин фрагментов

| Задача | Ответственный(е) | Примечания |
|------|------------------|------------|
| Создайте дисковый сервер вокруг `sorafs_car::ChunkStore` с индексом манифеста на диске (`sled`/`sqlite`). | Команда хранения | Определенный макет: `<data_dir>/<manifest_cid>/chunk_{idx}.bin`. |

### C. Шлюз Эндпоинты| Конечная точка | Поведение | Задачи |
|---------|-----------|--------|
| `POST /sorafs/pin` | Принимает `PinProposalV1`, валидирует манифесты, в свою очередь выдает прием, отвечает CID манифеста. | Валидировать профиль чанкера, применять квоты, стримить данные через хранилище чанкеров. |
| `GET /sorafs/chunks/{cid}` + запрос диапазона | Отдает байты чанка с заголовками `Content-Chunker`; соблюдает спецификацию дальности действия. | Используйте планировщик + бюджеты стрима (связаться с возможностями диапазона SF-2d). |
| `POST /sorafs/por/sample` | Запускает выборку PoR для манифеста и получает пакет доказательств. | Переиспользовать хранилище фрагментов выборки, есть полезные нагрузки JSON Norito. |
| `GET /sorafs/telemetry` | Сводки: емкость, успех PoR, счетчики ошибок fetch. | Предоставить данные для дашбордов/операторов. |

Рантайм-проводка направляет PoR-взаимодействие через `sorafs_node::por`: трекер фиксирует каждый `PorChallengeV1`, `PorProofV1` и `AuditVerdictV1`, чтобы метрики `CapacityMeter` отображали вердикты управления без отдельной логики. Torii.【crates/sorafs_node/src/scheduler.rs#L147】

Примечания по реализации:

- Используйте Axum стек Torii с полезными нагрузками `norito::json`.
- Добавить Norito схемы ответов (`PinResultV1`, `FetchErrorV1`, структуры телеметрии).

- ✅ `/v2/sorafs/por/ingestion/{manifest_digest_hex}` теперь показывает невыполненную работу, самую старую эпоху/крайний срок и последние временные метки успеха/неудачи по каждому провайдеру, за счет `sorafs_node::NodeHandle::por_ingestion_status`, а Torii фиксирует метрики `torii_sorafs_por_ingest_backlog`/`torii_sorafs_por_ingest_failures_total` для дашбордов.【crates/sorafs_node/src/lib.rs:510】【crates/iroha_torii/src/sorafs/api.rs:188 3】【crates/iroha_torii/src/routing.rs:7244】【crates/iroha_telemetry/src/metrics.rs:5390】

### D. Планировщик и приложение квот

| Задача | Детали |
|------|--------|
| Дисковая квота | Отслеживать байты на диске; отклонять новые контакты при превышении `max_capacity_bytes`. Подготовить хуки эвикции для будущего политика. |
| Конкурентность выборки | Глобальный семафор (`max_parallel_fetches`) плюс бюджеты для каждого провайдера из лимитов диапазона SF-2d. |
| Очередные булавки | Ограничить незавершенные задания по приему; Определите Norito статус эндпоинты для очереди. |
| Каденция PoR | Фоновый воркер, управляемый `por_sample_interval_secs`. |

### F. Стратегия тестирования1. **Юнит-тесты.** Персистентность chunk store, расчет квот, инварианты планировщика (см. `crates/sorafs_node/src/scheduler.rs`).
2. **Интеграционные тесты** (`crates/sorafs_node/tests`). Пин → выборка туда и обратно, восстановление после рестарта, отказ по квоте, проверка выборки PoR.
3. **Torii интеграционные тесты.** Запустить Torii с включенными хранилищами, прогнать HTTP-эндпоинты через `assert_cmd`.
4. **Дорожная карта хаоса.** Будущие буры моделируют исчерпание диска, снижение IO, удаление провайдеров.

## Зависимости

- Политика допуска SF-2b — подтверждает, что узлы проверяют входные конверты перед публикацией объявления.
- Маркетплейс емкости SF-2c — связать телеметрию с декларациями емкости.
- Расширение рекламы SF-2d — используйте возможности диапазона + бюджетные стримов по мере показа.

## Критерии заключения вехи

- `cargo run -p sorafs_node --example pin_fetch` работает на локальных приборах.
- Torii собирается с `--features sorafs-storage` и проходит интеграционные тесты.
- Документация ([гайд по хранилищам узла](node-storage.md)) обновлена ​​с конфигурацией по умолчанию + примерами CLI; Runbook для операторов доступен.
- Телеметрия видна на панели управления; предупреждайте о настроениях при насыщении емкостей и ошибках PoR.

## Документация и операционные результаты

- Обновить [справочник по хранилищу узла](node-storage.md) с дефолтами конфигурации, использованием CLI и шагами устранения неполадок.
