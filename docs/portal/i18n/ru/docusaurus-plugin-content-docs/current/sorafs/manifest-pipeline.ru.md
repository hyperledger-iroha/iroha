# Чанкинг SoraFS → Конвейерные манифесты

Этот материал дополняет краткое руководство и описывает полный пайплайн, который превращает сырные
байты в манифесте Norito, подходящие для реестра контактов SoraFS. Текст адаптирован из
[`docs/source/sorafs/manifest_pipeline.md`] (https://github.com/hyperledger-iroha/iroha/blob/master/docs/source/sorafs/manifest_pipeline.md);
приложения к настоящему документу по канонической спецификации и журналу изменений.

## 1. Детерминированный чанкинг

SoraFS использует профиль SF-1 (`sorafs.sf1@1.0.0`): роллинг-хэш, вдохновленный FastCDC, с
размер чанка 64 КиБ, целевым 256 КиБ, максимальным 512 КиБ и маской разрыва
`0x0000ffff`. Профиль зарегистрирован в `sorafs_manifest::chunker_registry`.

### Хелперы Rust

- `sorafs_car::CarBuildPlan::single_file` – Выдает смещения, длину и BLAKE3-дайджесты чанков
  при подготовке метаданных CAR.
- `sorafs_car::ChunkStore` – Стримит полезные нагрузки, сохраняет метаданные чанков и выводит дерево.
  выборки Доказательство возможности восстановления (PoR) 64 КиБ / 4 КиБ.
- `sorafs_chunker::chunk_bytes_with_digests` – Технический библиотечный помощник, конференции под обеими CLI.

### Инструменты CLI

```bash
cargo run -p sorafs_chunker --bin sorafs-chunk-dump -- ./payload.bin \
  > chunk-plan.json
```

JSON содержит упорядоченные смещения, длину и дайджесты чанков. Сохраняйте план при сборке
выборки манифестов или спецификаций для оркестратора.

### Свидетели PoR

`ChunkStore` содержит `--por-proof=<chunk>:<segment>:<leaf>` и `--por-sample=<count>`,
Чтобы аудиторы могли запросить определенные наборы свидетелей. Сочетайте эти флаги с
`--por-proof-out` или `--por-sample-out`, чтобы записать JSON.

## 2. Обернуть манифест

`ManifestBuilder` выбирают метаданные чанков с вложениями управления:

- Корневой CID (dag-cbor) и коммитменты CAR.
- Доказательства псевдонимов и претензий на возможности провайдеров.
- Подписи совета и метаданные опционные (например, идентификаторы сборки).

```bash
cargo run -p sorafs_manifest --bin sorafs-manifest-stub -- \
  ./payload.bin \
  --chunker-profile=sorafs.sf1@1.0.0 \
  --manifest-out=payload.manifest \
  --manifest-signatures-out=payload.manifest_signatures.json \
  --json-out=payload.report.json
```

Данные о важных выходных:

- `payload.manifest` – Norito-кодированные байты манифеста.
- `payload.report.json` – Сводка для людей/автоматизации, включая `chunk_fetch_specs`,
  `payload_digest_hex`, дайджесты CAR и метаданные псевдонимы.
- `payload.manifest_signatures.json` – Конверт, двойной BLAKE3-дайджест манифеста,
  SHA3-дайджест плана чанков и отсортированные загрузки Ed25519.

Используйте `--manifest-signatures-in`, чтобы проверить конверты от внешних подписантов перед
перезаписью, и `--chunker-profile-id` или `--chunker-profile=<handle>` для выбора фиксации
реестра.

## 3. Публикация и закрепление

1. **Отправка в управление** – Передайте дайджест манифеста и конверт подписки совета, чтобы
   штифт мог быть принят. Внешним аудиторам следует хранить SHA3-дневный план чанков рядом с
   дайджестом манифеста.
2. **Закрепление полезных данных** – скачайте архив CAR (и опциональный индекс CAR), указанный в
   манифест, в реестре контактов. Убедитесь, что манифест и CAR используют один и тот же корневой CID.
3. **Запись телеметрии** – Сохраните JSON-отчет, свидетелей PoR и любые метрики извлечение в
   релизных документов. Эти записи питают операторские дашборды и помогают воспроизвести.
   проблемы без загрузки больших полезных нагрузок.

##4. Моделирование выборки от нескольких провайдеров`пробег груза -p sorafs_car --bin sorafs_fetch -- --plan=payload.report.json \
  --provider=alpha=providers/alpha.bin --provider=beta=providers/beta.bin#4@3 \
  --output=payload.bin --json-out=fetch_report.json`

- `#<concurrency>` увеличивает параллелизм на провайдера (`#4` выше).
- `@<weight>` настраивает смещение планирования; по умолчанию 1.
- `--max-peers=<n>` ограничивает число провайдеров, запланированных на запуск, когда
  Обнаружение требует больше кандидатов, чем необходимо.
- `--expect-payload-digest` и `--expect-payload-len` защищают от тихой порчи данных.
- `--provider-advert=name=advert.to` теперь возможности провайдера перед использованием
  в симуляции.
- `--retry-budget=<n>` переопределяет число повторов на чанк (по умолчанию: 3), чтобы CI
  Быстрее выявляла регрессию при тестировании отказов.

`fetch_report.json` выводит агрегированные метрики (`chunk_retry_total`, `provider_failure_rate`,
и т. д. д.), подходящие для CI-ассертов и наблюдений.

## 5. Обновление реестра и управления

При предложении нового профиля chunker:

1. Подготовьте дескриптор в `sorafs_manifest::chunker_registry_data`.
2. Обновите `docs/source/sorafs/chunker_registry.md` и соответствующие уставы.
3. Перегенерируйте фикстуры (`export_vectors`) и зафиксируйте подписанные манифесты.
4. Откорректировать отчет о соответствии устава с подписями управления.

Автоматизации следует предпочитать канонические ручки (`namespace.name@semver`) и
обращаться к числовому идентификатору только при необходимости обратной совместимости.
