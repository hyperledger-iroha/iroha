## Назначение

Этот сборник задач выполняет SRE и дежурных инженеров через два последовательных процесса:

1. Выкатывать мультиисточниковый оркестратор контролируемыми волнами.
2. Занести в чёрный список или снизить приоритет проблемных провайдеров без дестабилизации текущих сессий.

Предполагается, что стек оркестрации, поставленный в рамках SF-6, уже + (`sorafs_orchestrator`, API шлюза чанков, экспортеры телеметрии).

> **См. Также:** [Runbook по эксплуатации оркестратора](./orchestrator-ops.md) подробно описаны процедуры по прогону (снятие табло, переключатели поэтапного выкатывания, отката). Используйте обе ссылки вместе во время живых изменений.

## 1. Предстартовая валидация

1. **Подтвердить входные данные управления.**
   - Все кандидаты-провайдеры должны публиковать конверты `ProviderAdvertV1` с разнообразными возможностями полезной нагрузки и бюджетными потоками. Проверяйте через `/v2/sorafs/providers` и сверяйте с ожидаемыми полями возможностей.
   - Снимки телеметрии с метриками латентности/сбоев должны быть не более 15 минут перед каждым канареечным-прогоном.
2. **Подготовить конфигурацию.**
   - Сохраните JSON-конфиг оркестратора в битостом дереве `iroha_config`:

     ```toml
     [torii.sorafs.orchestrator]
     config_path = "/etc/iroha/sorafs/orchestrator.json"
     ```

     Обновите JSON с лимитами под развертывание (`max_providers`, бюджеты ретраев). Используйте один и тот же файл для постановки/производства, чтобы различать выбросы.
3. **Прогнать канонические приспособления.**
   - Заполните переменные окружения манифеста/токена и выполните детерминированную выборку:

     ```bash
     sorafs_cli fetch \
       --plan fixtures/sorafs_manifest/ci_sample/payload.plan.json \
       --manifest-id "$CANARY_MANIFEST_ID" \
       --provider name=alpha,provider-id="$PROVIDER_ALPHA_ID",base-url=https://gw-alpha.example,stream-token="$PROVIDER_ALPHA_TOKEN" \
       --provider name=beta,provider-id="$PROVIDER_BETA_ID",base-url=https://gw-beta.example,stream-token="$PROVIDER_BETA_TOKEN" \
       --provider name=gamma,provider-id="$PROVIDER_GAMMA_ID",base-url=https://gw-gamma.example,stream-token="$PROVIDER_GAMMA_TOKEN" \
       --max-peers=3 \
       --retry-budget=4 \
       --scoreboard-out artifacts/canary.scoreboard.json \
       --json-out artifacts/canary.fetch.json
     ```

     Переменные окружения должны сохранять дайджест манифеста полезной нагрузки (шестнадцатеричный) и кодированные base64 потоковые токены для каждого провайдера, участвующего в Canary.
   - Сравните `artifacts/canary.scoreboard.json` с предыдущим релизом. Любой новый недействительный провайдер или сдвиг веса >10% требует ревью.
4. **Проверить, что телеметрия подключена.**
   - Откройте экспорт Grafana в `docs/examples/sorafs_fetch_dashboard.json`. Убедитесь, что метрики `sorafs_orchestrator_*` выполняются в стадии подготовки перед продолжением.

## 2. Экстренное занесение провайдеров в чёрный список

Следуйте этой процедуре, если провайдер отдает поврежденные чанки, устойчиво таймаутируется или не проходит проверку соответствия.1. **Зафиксировать доказательства.**
   - Экспортируйте последний fetch-свод (вывод `--json-out`). Запишите индексы сбойных чанков, псевдонимов провайдеров и дайджеста несовпадений.
   - Сохраните соответствующие фрагменты журналов из целевых объектов `telemetry::sorafs.fetch.*`.
2. **Применить немедленный переопределение.**
   - От провайдера, наложенного на снимок телеметрии, руководителя оркестратора (установите `penalty=true` или ограничьте `token_health` до `0`). Следующая сборка табло автоматически генерируется провайдером.
   - Для специальных дымовых тестов передайте `--deny-provider gw-alpha` в `sorafs_cli fetch`, чтобы отработать путь отказа без ожидания передачи телеметрии.
   - Переразвернуть обновленный пакет телеметрии/конфигурации в косой среде (постановка → канарейка → производство). Задокументируйте изменения в журнале происшествия.
3. **Проверить переопределение.**
   - Повторите принесение канонического приспособления. Убедитесь, что табло помечает провайдера как невалидного по причине `policy_denied`.
   - Проверьте `sorafs_orchestrator_provider_failures_total`, чтобы убедиться, что счетчик перестал расти из-за отклонения провайдера.
4. **Эскалировать долгие блокировки.**
   - Если провайдер будет заблокирован >24 часов, введите управляющий билет на ротацию или приостановите его рекламу. До составления списка запретов и обновления снимков телеметрии, чтобы провайдер не вернулся на табло.
5. **Протокол отката.**
   - Чтобы вернуть провайдера, удалите его из Deny-листа, переразверните и отключите новое снапшот табло. Приложите изменение к посмертному инциденту.

##3. План поэтапного выпуска

| Фаза | Охват | Требуемые сигналы | Критерии «Годен/Нет» |
|------|-------|-------------------|-------------------|
| **Лаборатория** | Выделенный интеграционный кластер | Ручная выборка CLI для фиксации полезных данных | Все чанки проходят, счетчики провайдерских сбоев остаются на 0, доля ретраев < 5%. |
| **Постановка** | Полный постановочный самолет управления | Приборная панель Grafana подключена; правила оповещений в режиме только предупреждение | `sorafs_orchestrator_active_fetches` возвращается к результату после каждого тестового прогона; нет алертов `warn/critical`. |
| **Канарейка** | ≤10% прод-трафика | Пейджер выключен, но телеметрия мониторится в мгновение ока | Доля ретраев < 10%, провайдерские наборы ограничены известными шумными пирами, гистограмма латентности соответствует базовому уровню ±20%. |
| **Общая доступность** | 100% внедрение | Правила пейджера активны | Ноль ошибок `NoHealthyProviders` за 24 часа, доля ретраев стабильна, панель SLA на дашборде зеленого цвета. |

Для каждой фазы:

1. Обновите оркестратор JSON с помощью приведенных `max_providers` и ретроспективных бюджетов.
2. Запустите `sorafs_cli fetch` или интеграционные тестовые SDK в стандартном приспособлении и репрезентативном манифесте среды.
3. Сохраните табло артефактов + сводку и приложите их к записи о релизе.
4. Проверьте дашборды телеметрии с дежурным инженером перед переходом к следующей фазе.

## 4. Наблюдаемость и крючки инцидентов- **Метрики:** Убедитесь, что Alertmanager мониторит `sorafs_orchestrator_fetch_failures_total{reason="no_healthy_providers"}` и `sorafs_orchestrator_retries_total`. Резкий всплеск обычно означает деградацию провайдера под настройку.
- **Логи:** Направляйте таргеты `telemetry::sorafs.fetch.*` в общий лог-агрегатор. Настройте сохраненные запросы для `event=complete status=failed`, чтобы ускорить триаж.
- **Табло:** Сохраняйте каждое табло артефактов в долговременном хранилище. JSON также служит доказательной трассой для проверки соответствия и поэтапных откатов.
- **Панели мониторинга:** Клонируйте канонический Grafana-дэшборд (`docs/examples/sorafs_fetch_dashboard.json`) в продакшн-папку с правилами оповещений из `docs/examples/sorafs_fetch_alerts.yaml`.

## 5. Коммуникации и документация

- Логируйте каждое изменение запрета/ускорения в операционном журнале изменений с отметкой времени, оператора, причины и инцидента.
- Уведомляйте команду SDK при изменении весов провайдеров или бюджетов ретраев, чтобы синхронизировать ожидания на стороне клиента.
- После завершения развертывания сводной версии GA обновите `status.md` и заархивируйте эту ссылку на runbook в релизных заметках.
