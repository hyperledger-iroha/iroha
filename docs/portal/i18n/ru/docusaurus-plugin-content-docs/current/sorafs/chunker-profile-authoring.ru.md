# Руководство по авторингу профиля chunker SoraFS

Это руководство, как предложить и опубликовать новые профили для SoraFS.
Оно дополняет архитектурный RFC (SF-1) и справочник реестра (SF-2a)
Эти требования к авторингу, шагам валидации и шаблонам предложений.
В качестве стандартного образца см.
`docs/source/sorafs/proposals/sorafs_sf1_profile_v1.json`
и соответствующий протокол пробного прогона в
`docs/source/sorafs/reports/sf1_determinism.md`.

## Обзор

Каждый профиль, попадающий в реестр, должен:

- объявлять определенные параметры CDC и настройки multihash, одинаковые для всех.
  архитектуры;
- по созданию воспроизводимых фикстур (JSON Rust/Go/TS + fuzz corpora + PoR-свидетель), которые
  последующие SDK могут проверяться без специализированного инструментария;
- включать метаданные, готовые для управления (пространство имен, имя, семвер), а также рекомендации
  совместимость с окнами и удобствами для путешествий; и
- представить детерминированную дифференцированную комплектацию на рассмотрение совета.

Следуйте контрольному списку ниже, чтобы подготовить предложение, полезное для этих правил.

## Снимок чарта реестрера

Перед подготовкой предложения убедиться, что оно соответствует уставному реестру,
который обеспечивает `sorafs_manifest::chunker_registry::ensure_charter_compliance()`:

- ID профиля — положительные целые числа, монотонно возрастающие без пропусков.
- Канонический дескриптор (`namespace.name@semver`) необходимо добавить в список псевдонимов и
- Ни один псевдоним не должен сравнивать с другим каноническим обращением и повторяться.
- Псевдонимы должны быть непустыми и без пробелов по краям.

Полезные помощники CLI:

```bash
# JSON список всех зарегистрированных дескрипторов (ids, handles, aliases, multihash)
cargo run -p sorafs_manifest --bin sorafs_manifest_chunk_store -- --list-profiles

# Эмитить метаданные для кандидата на профиль по умолчанию (канонический handle + aliases)
cargo run -p sorafs_manifest --bin sorafs_manifest_chunk_store -- \
  --promote-profile=sorafs.sf1@1.0.0 --json-out=-
```

Эти представители являются держателями предложений по соглашениям с уставным реестром и дают канонические соглашения.
метаданные для обсуждений управления.

## Требуемые метаданные| Поле | Описание | Пример (`sorafs.sf1@1.0.0`) |
|------|----------|------------------------------|
| `namespace` | Логическая группировка профилей. | `sorafs` |
| `name` | Читаемая человек метка. | `sf1` |
| `semver` | Строка семантической версии для выбора параметров. | `1.0.0` |
| `profile_id` | Монотонный числовой идентификатор, назначаемый после настройки профиля. Резервируйте следующий идентификатор, но не используйте дополнительные номера. | `1` |
| `profile.min_size` | Минимальная длина чанка в байтах. | `65536` |
| `profile.target_size` | Целевая длина чанка в байтах. | `262144` |
| `profile.max_size` | Максимальная длина чанка в байтах. | `524288` |
| `profile.break_mask` | Адаптивная маска для катания гашиша (шестигранник). | `0x0000ffff` |
| `profile.polynomial` | Константа зубчатого полинома (шестигранника). | `0x3da3358b4dc173` |
| `gear_seed` | Начальное значение для расчета 64 КиБ таблицы передач. | `sorafs-v1-gear` |
| `chunk_multihash.code` | Мультихэш-код для дайджеста по чанкам. | `0x1f` (BLAKE3-256) |
| `chunk_multihash.digest` | Дайджест канонического комплекта светильников. | `13fa...c482` |
| `fixtures_root` | Относительный каталог с регенерированными светильниками. | `fixtures/sorafs_chunker/sorafs.sf1@1.0.0/` |
| `por_seed` | Seed для детерминированной выборки PoR (`splitmix64`). | `0xfeedbeefcafebabe` (пример) |

Метаданные следует использовать как в предложениях документа, так и внутри сгенерированных.
приспособления, реестр, инструменты CLI и автоматизация управления.
ручных сверок. Если есть предложение, запустите chunk-store и манифест CLI с `--json-out=-`,
чтобы стримить вычисленные метаданные в заметки ревью.

### Точки взаимодействия CLI и реестра

- `sorafs_manifest_chunk_store --profile=<handle>` — повторно запускает метаданные чанка,
  дайджест манифеста и проверки PoR с предложенными параметрами.
- `sorafs_manifest_chunk_store --json-out=-` — стримит отчет chunk-store в stdout для
  военных сравнений.
- `sorafs_manifest_stub --chunker-profile=<handle>` — подтверждение, что манифесты и CAR
  Планы встраивают канонический дескриптор и псевдонимы.
- `sorafs_manifest_stub --plan=-` — подает сообщение `chunk_fetch_specs` для проверки.
  смещения/дайджесты после изменений.

Запишите вывод команды (дайджесты, корни PoR, хэши манифеста) в предложении, чтобы ревьюеры могли
воспроизвести их буквально.

## Чеклист детерминизма и валидации1. **Регенерировать светильники**
   ```bash
   cargo run --locked -p sorafs_chunker --bin export_vectors \
     --signature-out=fixtures/sorafs_chunker/manifest_signatures.json
   ```
2. **Запустить пакет паритета** — `cargo test -p sorafs_chunker` и межъязыковой дифференциал.
   (`crates/sorafs_chunker/tests/vectors.rs`) должны быть зелеными с новыми светильниками.
3. **Переиграть фузз/противодавление корпусов** — выполнил `cargo fuzz list` и потоковую обвязку
   (`fuzz/sorafs_chunker`) на регенерированных активах.
4. **Проверить свидетелей, подтверждающих возможность возврата** — запустите
   `sorafs_manifest_chunk_store --por-sample=<n>` с предлагаемым профилем и подтверждением,
   что корни совпадают с манифестом приспособления.
5. **Прогон CI** — выполняется `ci/check_sorafs_fixtures.sh` локально; скрипт должен
   пройти с новыми светильниками и существующим `manifest_signatures.json`.
6. **Подтверждение перекрестного выполнения** — убедитесь, что привязки Go/TS потребляют регенерированный
   JSON и выдаете личные границы чанков и дайджестов.

Документируйте команды и полученные дайджесты в предложении, чтобы Tooling WG могла просмотреть их без догадок.

### Подтверждение Манифест / PoR

После восстановления приборов запустите полный конвейер манифеста, чтобы убедиться, что
CAR метаданные и PoR доказательства аналогичны согласованным:

```bash
# Проверить метаданные чанка + PoR с новым профилем
cargo run -p sorafs_manifest --bin sorafs_manifest_chunk_store -- \
  --profile=sorafs.sf2@1.0.0 \
  --json-out=- --por-json-out=- fixtures/sorafs_chunker/input.bin

# Сгенерировать manifest + CAR и сохранить chunk fetch specs
cargo run -p sorafs_manifest --bin sorafs_manifest_stub -- \
  fixtures/sorafs_chunker/input.bin \
  --chunker-profile=sorafs.sf2@1.0.0 \
  --chunk-fetch-plan-out=chunk_plan.json \
  --manifest-out=sf2.manifest \
  --car-out=sf2.car \
  --json-out=sf2.report.json

# Повторно запустить с сохраненным планом fetch (защищает от устаревших offsets)
cargo run -p sorafs_manifest --bin sorafs_manifest_stub -- \
  fixtures/sorafs_chunker/input.bin \
  --chunker-profile=sorafs.sf2@1.0.0 \
  --plan=chunk_plan.json --json-out=-
```

Замените входной файл любым представительским корпусом, используемым в ваших устройствах.
(например, детерминированным потоком 1 ГиБ), и приложить полученные дайджесты к предложению.

## Рабочий процесс управления

1. **Отредактировать PR-предложения + светильники.** Включите сгенерированные активы, Norito
   предложение и обновления `chunker_registry_data.rs`.
2. **Review Tooling WG.** Ревьюеры повторно запускают проверку и подтверждение контрольного списка,
   что предложение соответствует правилам реестра (без идентификатора повторного использования,
   детерминизм достигнут).
3. **Конвертировать совет.** После одобрения члены совета подписывают дайджест предложений.
   (`blake3("sorafs-chunker-profile-v1" || canonical_bytes)`) и подключить подключение
   в конверт-профиле, хранящийся вместе со светильниками.
4. **Публикация реестра.** Объединить обновляемый реестр, документы и данные. По умолчанию CLI
   Остается в предыдущем профиле, пока управление не объявит миграцию готовой.
5. **Отслеживание депрекаций.** После приезда обновите реестр, отмените замену окон.

## Советы по авторингу

- Предпочитайте четные границы степенных двойок, чтобы свести к минимуму крайние случаи разделения на фрагменты.
- Не меняйте мультихэш-код без конкуренции с пользовательскими манифестами и шлюзами; добавляйте
  заметка о совместимости.
- Держите таблицы передач семян читаемыми, но глобальные достижения для упрощения аудита.
- Сохраните данные бенчмаркинга (например, сравнительную пропускную способность) в
  `docs/source/sorafs/reports/` для последующих ссылок.

Операционные ожидания во время развертывания см. в миграционном регистре
(`docs/source/sorafs/migration_ledger.md`). Правила выполнения условий соответствия см. в
`docs/source/sorafs/chunker_conformance.md`.
