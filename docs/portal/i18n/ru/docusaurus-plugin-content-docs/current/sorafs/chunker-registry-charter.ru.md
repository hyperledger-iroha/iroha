# Хартия управления реестром chunker SoraFS

> **Ратифицировано:** 2025-10-29 Панель инфраструктуры парламента Сора (см.
> `docs/source/sorafs/council_minutes_2025-10-29.md`). Любые требования
> формального голосования по управлению; команда специалистов должна считать этот документ
> нормативным, пока не будет утверждена новая хартия.

Эта Хартия определяет процесс и ролик для фрагмента реестра роста SoraFS.
Она дополняет [Руководство по авторингу профилей chunker](./chunker-profile-authoring.md), описывая, как новые
профили предоставляются, анализируются, ратифицируются и в итоге оформляются из обращения.

## Область

Хартия применяется к каждой записи в `sorafs_manifest::chunker_registry` и
к любому инструменту, который потребляет реестр (манифестный CLI, рекламный CLI поставщика,
SDK). Она фиксирует инварианты псевдонима и дескриптора, проверяемые
`chunker_registry::ensure_charter_compliance()`:

- ID профиля — положительные целые числа, монотонно возрастающие.
- Канонический дескриптор `namespace.name@semver` **должен** быть первой записью
- Строки псевдонима обрезаны, уникальны и не конфликтуют с традиционными обработками других записей.

## Роли

- **Автор(ы)** – готовят предложение, регенерируют светильники и собирают
  доказательства детерминизма.
- **Инструментальная рабочая группа (TWG)** – утверждает опубликованное предложение
  чеклистам и убеждается, что инварианты реестра соблюдены.
- **Управляющий совет (GC)** – рассмотреть отчет TWG, подписать конверт предложений
  и требует срочного издания/депрекации.
- **Команда хранения** – поддержка создания реестра и публикации.
  обновление документации.

## Жизненный цикл

1. **Подача предложения**
   - Автор запускает контрольный список валидаций от руководства по авторингу и представлениям.
     JSON `ChunkerProfileProposalV1` в
     `docs/source/sorafs/proposals/`.
   - Включает вывод CLI из:
     ```bash
     cargo run -p sorafs_manifest --bin sorafs_manifest_chunk_store -- --list-profiles
     cargo run -p sorafs_manifest --bin sorafs_manifest_chunk_store -- \
       --promote-profile=<handle> --json-out=-
     cargo run -p sorafs_manifest --bin sorafs_manifest_stub -- \
       --chunker-profile=<handle> --json-out=-
     ```
   - Отправляет PR, встроенные светильники, предложение, отчет о детерминировании и
     обновление реестра.

2. **Обзор инструмента (TWG)**
   - Повторяет чек-лист валидации (фикстуры, неточности, манифест конвейера/PoR).
   - Запускает `cargo test -p sorafs_car --chunker-registry` и убеждается, что
     `ensure_charter_compliance()` проходит с новой записью.
   - Проверяет, что поведение CLI (`--list-profiles`, `--promote-profile`, потоковая передача
     `--json-out=-`) Отметьте обновленные псевдоним и дескриптор.
   - Готовит краткий отчет с выводами и статусом «пройден/не пройден».

3. **Совет по одобрению (GC)**
   - Рассматривает отчет TWG и метаданные предложения.
   - Подписывает дайджест предложений (`blake3("sorafs-chunker-profile-v1" || bytes)`)
     и добавьте подключение в конверт, который хранится рядом со светильниками.
   - Фиксирует результаты голосования в протоколе управления.4. **Публикация**
   - Мержит PR, обновление:
     - `sorafs_manifest::chunker_registry_data`.
     - Документация (`chunker_registry.md`, руководства по авторингу/соответствию).
     - Светильники и отчеты о детерминизме.
   - Уведомляет операторов и команды SDK о новом профиле и планировании развертывания.

5. **Депрекация/Закат**
   - Предложения, заменяющие существующий профиль, должны включать окно прозрачной публикации.
     (грейс-периоды) и план обновления.
     в реестре и обновлении миграционного журнала.

6. **Экстренные изменения**
   - Удаление или исправление требует решения большинства.
   - TWG должна документировать шаги по снижению риска и обновлять журналы инцидентов.

## Ожидания от оснастки

- `sorafs_manifest_chunk_store` и `sorafs_manifest_stub` предоставляют:
  - `--list-profiles` для инспекционного реестра.
  - `--promote-profile=<handle>` для генерации канонического блока метаданных,
    Используемого при продвижении профиля.
  - `--json-out=-` для стриминга отчетов в стандартный вывод, результаты воспроизводимые
    логи ревью.
- `ensure_charter_compliance()` появляется при запуске релевантных бинарников
  (`manifest_chunk_store`, `provider_advert_stub`). CI-тесты должны провалиться, если
  новые записи нарушают хартию.

## Документирование

- Храните все отчеты о детерминировании `docs/source/sorafs/reports/`.
- Протоколы совета с решениями по настройке чанкера в
  `docs/source/sorafs/migration_ledger.md`.
- Обновляйте `roadmap.md` и `status.md` после каждого крупного изменения реестра.
