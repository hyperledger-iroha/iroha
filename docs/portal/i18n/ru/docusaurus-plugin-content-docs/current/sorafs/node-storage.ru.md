## Дизайн узла хранения SoraFS (Черновик)

Эта записка уточняет, как узел Iroha (Torii) может подключиться к слою.
наличие данных SoraFS и выделить часть локального диска для хранения и
обслуживание чанков. Она выполняет открытие-спецификацию
`sorafs_node_client_protocol.md` и работа по светильникам SF-1b, описывая архитектуру
хранилище, ресурсные ограничения и конфигурационную проводку, которые должны
появится в узле и коде шлюза. Практические операционные процедуры описаны в
[Runbook операций узла](./node-operations).

### Цели

- Разрешить любой валидатор или вспомогательный процесс Iroha опубликовать
  свободный диск как провайдер SoraFS без оплаты по книге обязанностей.
- Держать модуль хранения определенным и управляемым Norito: манифесты,
  планы чанков, корни Proof-of-Retrivability (PoR) и рекламных провайдеров — это
  источник истины.
- Применять операторские квоты, чтобы узел не исчерпал свои ресурсы из-за.
  слишком большой объем запроса pin или fetch.
- Отдавать здоровье/телеметрию (выборка PoR, задержка выборки чанков, давление на
  диск) обратно в управление и клиентов.

Ключевые модули:

- **Шлюз**: открывает Norito HTTP-эндпоинты для вывода предложений, выборки запросов.
  чанков, PoR отбор проб и телеметрии. Валидирует полезные нагрузки Norito и направляет
  запрос в хранилище фрагментов. Переиспользует существующий HTTP-стек Torii, чтобы не
  вводить нового демона.
- **Реестр контактов**: состояние контактов-манифестов в `iroha_data_model::sorafs` и
  `iroha_core`. При просмотре манифеста регистр хранит дайджест манифеста, дайджест
  план чанков, корень PoR и флаги возможностей провайдера.
- **Chunk Storage**: дисковая поставка `ChunkStore`, которая принимает подписанные
  манифесты, материализует планы чанков через `ChunkProfile::DEFAULT` и сохраняет
  чанки в детерминированном расположении. Каждый канал связан с содержимым отпечатков пальцев и
  PoR метаданными, поэтому выборку можно повторно валидировать без перечитывания
  всего файла.
- **Квота/Планировщик**: применяются лимиты операторов (макс. байты диска, макс. контакты).
  в очереди, макс. параллельная выборка, TTL чанков) и координирует ввод-вывод для решения задачи
  бухгалтерская книга не была вытеснена. Планировщик также отвечает за выдачу PoR-подтверждений и запросов.
  выборка с ограниченным процессором.

### Конфигурация

Добавьте новый раздел в `iroha_config`:

```toml
[sorafs.storage]
enabled = false
data_dir = "/var/lib/iroha/sorafs"
max_capacity_bytes = "100 GiB"
max_parallel_fetches = 32
max_pins = 10_000
por_sample_interval_secs = 600
alias = "tenant.alpha"            # опциональный человекочитаемый тег
adverts:
  stake_pointer = "stake.pool.v1:0x1234"
  availability = "hot"
  max_latency_ms = 500
  topics = ["sorafs.sf1.primary:global"]
```- `enabled`: переключить участие. Если false, шлюз получает 503 для хранения.
  эндпоинтов и узел не объявлены в открытии.
- `data_dir`: корневая директория для чанков, PoR деревьев и телеметрии.
  По умолчанию `<iroha.data_dir>/sorafs`.
- `max_capacity_bytes`: жесткий лимит на пин-данные. Фоновая задача отклоняет
  новые пины при передаче лимита.
- `max_parallel_fetches`: лимит конкурентности, накладываемый планировщик, чтобы
  балансировать сеть/IO-диск с помощью валидатора.
- `max_pins`: максимальное количество контактов манифеста, которые можно использовать для подключения узла.
  выселение/обратное давление.
- `por_sample_interval_secs`: каденция автоматических заданий выборки PoR. Каждый
  задание выбирает листья `N` (наказания по манифесту) и эмитит телеметрию.
  Управление может детерминированно масштабироваться `N`, задавая ключ
  `profile.sample_multiplier` (целое `1-4`). Значение может быть числом/строкой
  или объектом с переопределением по профилю, например `{"default":2,"sorafs.sf2@1.0.0":3}`.
- `adverts`: структура, используемая генератором, реклама для заполнения полей.
  `ProviderAdvertV1` (указатель ставки, подсказки по QoS, темы). Если опущено, беру узел
  значения по умолчанию из реестра управления.

Прокладка конфигурации:

- `[sorafs.storage]` определяется в `iroha_config` как `SorafsStorage` и загружается.
  конфигурационного файла узла.
- `iroha_core` и `iroha_torii` прокидывают конфиг хранилища в построитель шлюза и чанк
  магазин на старте.
- имеются переопределения dev/test (`SORAFS_STORAGE_*`, `SORAFS_STORAGE_PIN_*`), но
  в производстве следует использовать конфиг-файл.

### Жизненный узел цикла

1. **Запуск**:
   - При включенном хранилище создается узел хранилища фрагментов с заданной директорией.
     и емкостью. Это включает проверку/создание PoR баз манифестов и повторного воспроизведения.
     манифесты по прогреву кэшей.
   - Зарегистрировать маршруты шлюза SoraFS (Norito JSON POST/GET эндпоинты для PIN-кода,
     выборка, выборка PoR, телеметрия).
   - Запустить работника по отбору проб PoR и монитора квот.
2. **Обнаружение/Реклама**:
   - Сформировать `ProviderAdvertV1` на основе текущей емкости/здоровья, подписать
     ключом, одобренным советом и опубликовать через канал Discovery.
     значительными доступными.
3. **Рабочий процесс закрепления**:
   - Шлюз получает подписанный манифест (с планом чанков, корневым PoR и подписями).
     совет). Проверяет псевдоним списка (`sorafs.sf1@1.0.0` обязателен) и убеждается,
     что план чанков соответствует метаданному манифесту.
   - Проверяет квоты. При превышении емкости/пин-лимитов ответственность за политическую ошибку
     (структурированный Norito).
   - Стримит данные чанка в `ChunkStore`, проверяя дайджесты при приеме. Обновляет PoR
     деревья и хранит манифест метаданных в реестре.
4. **Извлечение рабочего процесса**:
   - Отдает запрос на диапазон чанков с диска. Планировщик документов `max_parallel_fetches`
     и возвращает `429` при насыщении.
   - Эмитит структурированную телеметрию (Norito JSON) с задержкой, обслуженными байтами и
     счетчиками ошибок для нижестоящих Диптихов.
5. **Выборка PoR**:
   - Работник выбирает манифесты пропорциональной весу (например, байтам хранения) и
     запускает детерминированную выборку через хранилище фрагментов PoR.
   - Сохран предоставляет результаты для управления аудитом и включает выводы в рекламе провайдера.
     / телем конечная точка.
6. **Выселение/квоты**:
   - При передаче оборудования узел по умолчанию отклоняет новые штифты. Опционально
     операторы могут настроить политику выселения (например, TTL, LRU) после
     согласование моделей управления; пока дизайн предусматривает строгие квоты и
     открепить операцию, инициируемую оператором.

### Интеграция деклараций емкости и планирования- Torii теперь ретранслирует обновления `CapacityDeclarationRecord` из `/v1/sorafs/capacity/declare`
  во встроенный `CapacityManager`, так что каждый узел строит в памяти представление своих
  зафиксированных чанкеров/линий аллокаций. Менеджер публикует снимки, доступные только для чтения, для телеметрии
  (`GET /v1/sorafs/capacity/state`) и применяются резервы для каждого профиля/полосы до принятия новых
  заказов.【crates/sorafs_node/src/capacity.rs:1】【crates/sorafs_node/src/lib.rs:60】
- Конечный пункт `/v1/sorafs/capacity/schedule` принимает выданный правительством `ReplicationOrderV1`
  полезные нагрузки. Когда заказ ориентирован на локального провайдера, менеджер впоследствии дублирует
  запланирований, валидирует емкость чанка/линии, резервирует слот и возвращает `ReplicationPlan`
  с описанием оставшейся емкости, чтобы оркестрация могла продолжить прием внутрь. Заказы для
  другие провайдеры подтверждаются ответом `ignored`, упрощая многооператорские рабочие процессы.【crates/iroha_torii/src/routing.rs:4845】
- Приемы завершения (например, после успешной загрузки) вызывают
  `POST /v1/sorafs/capacity/complete` для освобождения резерва через
  `CapacityManager::complete_order`. Ответ включает снимок `ReplicationRelease`
  (остаточные итоги, остатки чанка/дорожки), чтобы инструменты оркестровки могли работать
  поставить следующий заказ без голосования. Дальнейшая работа по подключению этого к трубопроводу
  хранилище фрагментов, как создается только логика приема.【crates/iroha_torii/src/routing.rs:4885】【crates/sorafs_node/src/capacity.rs:90】
- Встроенный `TelemetryAccumulator` можно обновлять через `NodeHandle::update_telemetry`,
  что позволяет фоновым работникам фиксировать образцы PoR/время безотказной работы и одновременно
  выводить кондуктивные полезные нагрузки `CapacityTelemetryV1` без изменений внутри
  планировщик частей.【crates/sorafs_node/src/lib.rs:142】【crates/sorafs_node/src/telemetry.rs:1】

### Интеграции и будущая работа

- **Управление**: прогресс `sorafs_pin_registry_tracker.md` телеметрии хранения.
  (Уровень успеха PoR, использование диска). Политики допускают требование минимального
  Эффективность или эффективность PoR при приеме рекламы.
- **Клиентские SDK**: раскрыта новая конфигурация хранилища (лимиты дисков, псевдоним), чтобы
  Инструменты управления позволяют загрузчику узлов программно.
- **Телеметрия**: интеграция с существующим стеком метрик (Prometheus /
  OpenTelemetry), чтобы метрики хранения появлялись на панелях наблюдения.
- **Безопасность**: запуск модуля хранения в отдельном пуле асинхронных задач с противодавлением.
  и настроить песочницу для чтения чанков через io_uring или ограниченные токийские пулы,
  чтобы злоумышленники не исчерпали ресурсы.

Этот дизайн сохраняет модуль хранения опциональных и детерминированных, позволяющих
Операторами нужные ручки для участия в предоставлении доступности данных SoraFS. Реализация
внести изменения в шлюз `iroha_config`, `iroha_core`, `iroha_torii` и Norito,
а также инструменты для провайдера рекламы.
