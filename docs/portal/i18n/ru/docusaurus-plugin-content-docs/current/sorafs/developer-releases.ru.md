# Процесс релиза

Бинарники SoraFS (`sorafs_cli`, `sorafs_fetch`, помощники) и крейты SDK
(`sorafs_car`, `sorafs_manifest`, `sorafs_chunker`) выпускаются вместе. Релизный
трубопровод, содержащий CLI и согласованные библиотеки, обеспечивает покрытие ворсом/тестом
и фиксирует принадлежности для последующих потребителей. Выполните контрольный список ниже для
тег каждого кандидата.

## 0. Подтвердить подпись по безопасности

Перед запуском технических выпускных ворот возьмите свежие материалы для проверки безопасности:

- скачать самый исходный меморандум SF-6 по безопасности ([reports/sf6-security-review](./reports/sf6-security-review.md))
  и зафиксируйте его хэш SHA256 в билете выпуска.
- Приложите ссылку на заявку на исправление (например, `governance/tickets/SF6-SR-2026.md`) и от рекомендаций.
  утвердить-ответственных из Рабочей группы по технике безопасности и инструментарию.
- проверьте, что контрольный список исправлений в мемо закрыт; незакрытые пункты блокируют релиз.
- Подготовьте загрузку журналов четности (`cargo test -p sorafs_car -- --nocapture sorafs_cli::proof_stream::bounded_channels`)
  вместе с манифестом пакета.
- Убедитесь, что команда подключения, с помощью которой вы планируете калибровку, включает и `--identity-token-provider`, и
  явный `--identity-token-audience=<aud>`, чтобы объем Fulcio был зафиксирован в релизных доказательствах.

Включите эти документы для ознакомления с руководством и публикации релиза.

## 1. Выполнить релиз/тестовый шлюз

Хелпер `ci/check_sorafs_cli_release.sh` запускает форматирование, Clippy и тесты
по CLI и ящикам SDK с локальной целевой директорией рабочей области (`.target`), чтобы избежать
вопросы правы при запуске внутри контейнеров CI.

```bash
CARGO_TARGET_DIR=.target ci/check_sorafs_cli_release.sh
```

Скрипт выполняет следующие проверки:

- `cargo fmt --all -- --check` (рабочая область)
- `cargo clippy --locked --all-targets` для `sorafs_car` (с функцией `cli`),
  `sorafs_manifest` и `sorafs_chunker`
- `cargo test --locked --all-targets` для этих же ящиков

Если какой-либо шаг выпадает, исправьте регрессию до разметки. Релизные сборки должны
продолжать непрерывность работы от основного; Не делайте выборочных исправлений в релиз-ветках. Ворота также
впоследствии наличие бесключевых флагов (`--identity-token-issuer`,
`--identity-token-audience`) там, где требуется; отсутствующие аргументы валят запуск.

## 2. Применить политику версий

Все ящики SoraFS CLI/SDK используют SemVer:

- `MAJOR`: Вводится с первым релизом 1.0. До 1.0 повышение минор `0.y`
  **означает критические изменения** на поверхности CLI или схемах Norito.
  за опциональной политикой, добавлением телеметрии).
- `PATCH`: исправления багов, релизы только для документации и обновления зависимостей,
  не меняющие наблюдательное поведение.

Держите `sorafs_car`, `sorafs_manifest` и `sorafs_chunker` на одной версии, чтобы
Нижестоящие пользователи SDK могли опираться на единую строку версии. При повышении версии:

1. Обновите поля `version =` в каждом `Cargo.toml`.
2. Перегенерируйте `Cargo.lock` через `cargo update -p <crate>@<new-version>` (рабочая область
   требует явных решений).
3. Снова запустите дверцу разблокировки, чтобы не осталось конфиденциальных документов.## 3. Подготовить примечания к выпуску

Каждый релиз должен публиковать список изменений уценки с акцентом на изменениях CLI, SDK.
и управление. Используйте шаблон `docs/examples/sorafs_release_notes.md` (скопируйте
его в каталог релизных документов и заполните раздел конкретикой).

Минимальный набор:

- **Основные моменты**: заголовки фич для потребителей CLI и SDK.
- **Шаги обновления**: команда TL;DR для обновления зависимостей груза и перезапуска.
  детерминированных светильников.
- **Проверка**: хэши или конверты, результаты команды и точная ревизия.
  `ci/check_sorafs_cli_release.sh`, которая была изготовлена.

Приложите заполненные примечания к выпуску к тегу (например, в теле релиза GitHub) и
Храните рядом с определенными сгенерированными документами.

## 4. Выполнить хуки освобождения

Запустите `scripts/release_sorafs_cli.sh`, чтобы сгенерировать пакет подписей и
Сводка проверки, которая отгружается с каждым релизом. Обертка по необходимости
собирает CLI, вызывает `sorafs_cli manifest sign` и сразу воспроизводит
`manifest verify-signature`, чтобы начать маркировку. Пример:

```bash
scripts/release_sorafs_cli.sh \
  --manifest artifacts/site.manifest.to \
  --chunk-plan artifacts/site.chunk_plan.json \
  --chunk-summary artifacts/site.car.json \
  --bundle-out artifacts/release/manifest.bundle.json \
  --signature-out artifacts/release/manifest.sig \
  --identity-token-provider=github-actions \
  --identity-token-audience=sorafs-release \
  --expect-token-hash "$(cat .release/token.hash)"
```

Подсказки:

- Отслеживайте релизные входные данные (полезная нагрузка, планы, сводки, ожидаемый хеш токена)
  в репозитории или конфигурации развертывания, чтобы обеспечить стабильное воспроизведение скрипта. КИ
  Комплект под `fixtures/sorafs_manifest/ci_sample/` показывает канонический макет.
- Строить автоматизацию CI на `.github/workflows/sorafs-cli-release.yml`; он работает
  Release Gate вызывает скрипт выше и архивирует пакеты/сигнатуры как документы рабочего процесса.
  Повторите тот же порядок команды (освободите ворота → подпишите → проверьте) в других CI,
  чтобы логи аудита совпадали со сгенерированными хэшами.
- Храните `manifest.bundle.json`, `manifest.sig`, `manifest.sign.summary.json` и
  `manifest.verify.summary.json` вместе — это пакет, на который ссылается уведомление управления.
- Если обновление обновляет канонические фикстуры, скопируйте обновленный манифест, план фрагментов и
  резюме в `fixtures/sorafs_manifest/ci_sample/` (и обновите
  `docs/examples/sorafs_ci_sample/manifest.template.json`) до маркировки. Нижестоящие операторы
  Решение от закоммиченных приспособлений для воспроизводимости выпуска комплекта.
- Зафиксировать выполнение проверки проверки ограниченного канала для `sorafs_cli proof stream` и
  приложите его к релизному пакету, чтобы показать, что гарантии потоковой передачи активны.
- Запишите память `--identity-token-audience`, использованный при подключении, в примечаниях к выпуску;
  управление сверяет аудиторию с политикой Фульчо перед одобрением публикации.

Используйте `scripts/sorafs_gateway_self_cert.sh`, если релиз включает шлюз развертывания.
Укажите тот же пакет манифеста, чтобы подтвердить, что аттестация соответствует сертификату кандидата:

```bash
scripts/sorafs_gateway_self_cert.sh --config docs/examples/sorafs_gateway_self_cert.conf \
  --manifest artifacts/site.manifest.to \
  --manifest-bundle artifacts/release/manifest.bundle.json
```

## 5. Тергирование и публикация

После прохождения проверок и выполнения хуков:1. Запустите `sorafs_cli --version` и `sorafs_fetch --version`, чтобы убедиться, что
   бинарники показывают новую версию.
2. Подготовьте релизный конфиг в `sorafs_release.toml` под контролем режима (предпочтительно)
   или в другом конфиг-файле, отслеживаемом вашим развертыванием в репозитории. Избегайте
   специальные условия окружения; передайте путь в CLI через `--config` (или аналог)
   чтобы освободить входные данные, которые были явными и воспроизводимыми.
3. Оформить подписанный тег (предпочтительно) или аннотированный тег:
   ```bash
   git tag -s sorafs-vX.Y.Z -m "SoraFS CLI & SDK vX.Y.Z"
   git push origin sorafs-vX.Y.Z
   ```
4. Загрузите документы (комплекты автомобилей, манифесты, сводки доказательств, примечания к выпуску,
   результаты аттестации) в реестре проекта, следуя контрольному списку управления
   [руководство по развертыванию](./developer-deployment.md). Если релиз создал новые светильники,
   отформатируйте их в общий репозиторий приспособлений или хранилище объектов, чтобы можно было автоматизировать аудит.
   Стандартный опубликованный пакет с контролем контроля.
5. Уведомите управление каналом ссылками на подписанный тег, примечания к выпуску, хэши.
   пакет/подписей манифест, архивированные `manifest.sign/verify` резюме и любые
   аттестационные конверты. Приложите URL CI job (или архив журналов), который выполнил
   `ci/check_sorafs_cli_release.sh` и `scripts/release_sorafs_cli.sh`. Обновите управление
   билет, чтобы аудиторы могли связаться с сертификатами; когда
   `.github/workflows/sorafs-cli-release.yml` публикует уведомления, связывается
   записанные хэши вместо специальных сводок.

- Убедитесь, что документация, указывающая на новую версию (краткие руководства, шаблоны CI),
  обновлено, либо подтвердите отсутствие изменений.
- Для составления дорожной карты запишите, если потребуется следующая работа (например, флаги миграции,
- Архивируйте логи вывода Release Gate для аудиторов - храните их рядом с подписанными
  документами.
