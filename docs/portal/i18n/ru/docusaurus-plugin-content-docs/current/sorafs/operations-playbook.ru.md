## Отключение шлюза/ухудшение доступности

**Обнаружение**

- Алерты: `SoraFSGatewayAvailabilityDrop`, `SoraFSGatewayLatencySlo`.
- Дашборд: `dashboards/grafana/sorafs_gateway_overview.json`.

**Немедленные действия**

1. Подтвердите масштаб (один провайдер или весь флот) на панели Request-Rate.
2. Переключите маршрутизацию Torii на здоровых провайдеров (если мультипровайдер), переключив `sorafs_gateway_route_weights` в ops-конфиге (`docs/source/sorafs_gateway_self_cert.md`).
3. Если подключены все провайдеры, резервная «прямая выборка» для клиентов CLI/SDK (`docs/source/sorafs_node_client_protocol.md`).

**Сортировка**

- Проверьте использование токена потока соответственно `sorafs_gateway_stream_token_limit`.
- Посмотрите логи шлюза на TLS или ошибки входа.
- Запустите `scripts/telemetry/run_schema_diff.sh`, чтобы убедиться, что схема экспортируемого шлюза соответствует ожидаемой версии.

**Варианты исправления**

- Перезапускайте только шлюз процесса; избегайте перезапуска всего кластера, если не упадет несколько провайдеров.
- Временно увеличьте лимит токена потока на 10–15%, если подтверждена насыщенность.
- Повторно выполнено самосертификат (`scripts/sorafs_gateway_self_cert.sh`) после возврата.

**После инцидента**

- Оформите постмортем P1 с помощью `docs/source/sorafs/postmortem_template.md`.
- Запланируйте следующий хаос-бур, если устранение основано на ручных действиях.

## Всплеск отказов доказательство (PoR/PoTR)

**Обнаружение**

- Алерты: `SoraFSProofFailureSpike`, `SoraFSPoTRDeadlineMiss`.
- Дашборд: `dashboards/grafana/sorafs_proof_integrity.json`.
- Телеметрия: `torii_sorafs_proof_stream_events_total` и события `sorafs.fetch.error` с `provider_reason=corrupt_proof`.

**Немедленные действия**

1. Заморозьте прием новых манифестов, пометив реестр манифестов (`docs/source/sorafs/manifest_pipeline.md`).
2. Уведомите Управление о приостановке стимулов для отдельных провайдеров.

**Сортировка**- Проверить вход в очередь запроса PoR относительно `sorafs_node_replication_backlog_total`.
- Подтвердите проверку трубопровода (`crates/sorafs_node/src/potr.rs`) для последних деплоев.
- Сравните версии прошивок провайдеров с реестром операторов.

**Варианты исправления**

- Запустите повторы PoR через `sorafs_cli proof stream` с последним манифестом.
- Если доказательства стабильно падают, выделите провайдера из активного набора через обновление реестра управления и необходимое обновление табло оркестратора.

**После инцидента**

- Запустить сценарий хаос-бурла PoR до следующего продакшен-деплоя.
- Зафиксируйте выводы в шаблоне постмортема и обновите контрольный список квалификации провайдеров.

## Задержка репликации / рост отставания

**Обнаружение**

- Алерты: `SoraFSReplicationBacklogGrowing`, `SoraFSCapacityPressure`. Импортировать
  `dashboards/alerts/sorafs_capacity_rules.yml` и выполнить
  `promtool test rules dashboards/alerts/tests/sorafs_capacity_rules.test.yml`
  до промоушена, чтобы Alertmanager отобразил документированные пороги.
- Дашборд: `dashboards/grafana/sorafs_capacity_health.json`.
- Метрики: `sorafs_node_replication_backlog_total`, `sorafs_node_manifest_refresh_age_seconds`.

**Немедленные действия**

1. Определите масштаб отставания (один провайдер или весь флот) и приостановите мелкие задачи репликации.
2. Если отставание локальное, временно переназначьте новые заказы на альтернативных провайдеров через планировщик репликации.

**Сортировка**

- Проверьте телеметрию оркестратора при повторных попытках, которые могут раздувать отставание.
- Убедитесь, что в целях хранения достаточный запас по высоте (`sorafs_node_capacity_utilisation_percent`).
- Проверьте последние изменения конфигурации (обновления профиля фрагмента, проверки каденции).

**Варианты исправления**

- Запустите `sorafs_cli` с опцией `--rebalance` для перераспределения контента.
- Масштабируйте рабочие репликации горизонтально для косого провайдера.
- Инициируйте обновление манифеста для спортивных окон TTL.

**После инцидента**

- Запланируйте мощную дрель, сосредоточив внимание на сбоях из-за насыщения провайдеров.
- Обновите документацию по репликации SLA в `docs/source/sorafs_node_client_protocol.md`.

## периодичность хаос-сверл

- **Ежеквартально**: совмещенная симуляция остановки шлюза + повторная попытка штурма оркестратора.
- **Два раза в год**: инъекция сбоев PoR/PoTR на двух провайдерах с восстановлением.
- **Ежемесячная выборочная проверка**: отсрочка репликации сценария с использованием промежуточных манифестов.
- Отслеживайте тренировки в общем журнале Runbook (`ops/drill-log.md`) через:

  ```bash
  scripts/telemetry/log_sorafs_drill.sh \
    --scenario "Gateway outage chaos drill" \
    --status pass \
    --ic "Alex Morgan" \
    --scribe "Priya Patel" \
    --notes "Failover to west cluster succeeded" \
    --log ops/drill-log.md \
    --link "docs/source/sorafs/postmortem_template.md"
  ```

- Проверьте регистрацию перед коммитами с помощью:

  ```bash
  scripts/telemetry/validate_drill_log.sh
  ```

- Используйте `--status scheduled` для последующих тренировок, `pass`/`fail` для завершенных запусков и `follow-up`, если остались открытые действия.
- Перезапишите назначение через `--log` для пробного прогона или автоматической проверки; без него график продолжать обновлять `ops/drill-log.md`.
