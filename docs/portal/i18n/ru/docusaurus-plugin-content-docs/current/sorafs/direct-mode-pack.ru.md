:::note Канонический источник
:::

Контуры SoraNet остаются транспортом по умолчанию для SoraFS, но пункт дорожной карты **SNNet-5a** требует регулируемого резервного варианта, чтобы операторы поддержки определили доступ к чтению, пока развертывание анонимности не завершится. Этот пакет фиксирует параметры CLI/SDK, конфигурацию профиля, проверки соответствия и чек-лист развертывания, необходимые для работы SoraFS в прямом режиме Torii/QUIC без обращения к частным транспортным средствам.

Резервный вариант применяется к промежуточным и регулируемым прод-средам, пока SNNet-5–SNNet-9 не пройдут шлюзы поддержки. Храните приведенные ниже материалы вместе с простыми материалами развертывания SoraFS, чтобы операторы могли переключаться между анонимными режимами и режимами управления в аппарате.

## 1. Флаги CLI и SDK

- `sorafs_cli fetch --transport-policy=direct-only ...` отключает реле планирования и использует транспортные средства Torii/QUIC. Справка CLI теперь включает `direct-only` как допустимое значение.
- SDK должен сохранять `OrchestratorConfig::with_transport_policy(TransportPolicy::DirectOnly)` каждый раз, когда они включают переключатель «прямого режима». Сгенерированные привязки в `iroha::ClientOptions` и `iroha_android` прокидывают тот же enum.
- Шлюз-жгут (`sorafs_fetch`, привязки Python) может разбирать переключатель напрямую только через общие JSON-помощники Norito, чтобы автоматизация получала такое же поведение.

Задокументируйте флаг в runbook'ах для партнеров и подключайте переключатели через `iroha_config`, а не через переменные окружения.

## 2. Шлюз Профили Политики

Используйте Norito JSON, чтобы сохранить детерминированную конфигурацию оркестратора. Пример профиля в `docs/examples/sorafs_direct_mode_policy.json` кодирует:

- `transport_policy: "direct_only"` — отклоняет провайдеров, которые рекламируют только транспортные реле SoraNet.
- `max_providers: 2` — ограничение направляет пиров наиболее надежными Torii/QUIC эндпойнтами. Настраивайте с учетом соответствия региональным требованиям.
- `telemetry_region: "regulated-eu"` — маркирует метрики, чтобы дашборды и аудиты включали резервные запуски.
- Консервативные бюджеты повторов (`retry_budget: 2`, `provider_failure_threshold: 3`) для того, чтобы не маскировать неправильные настроения шлюза.

Загружайте JSON через `sorafs_cli fetch --config` (автоматизация) или SDK-биндинги (`config_from_json`) для публикации политики операторам. Сохраняйте вывод табло (`persist_path`) для аудиторских наблюдений.Параметры основного режима на стороне шлюза описаны в `docs/examples/sorafs_gateway_direct_mode.toml`. Пример вывода `iroha app sorafs gateway direct-mode enable`, отключение проверок конверта/допуска, подключение значений по умолчанию для ограничения скорости и составление таблицы `direct_mode` хостами из плана и дайджеста значений манифеста. Замените заполнитель-значения на данные вашего плана развертывания перед фиксацией фрагмента в системе управления изменениями.

## 3. Наборы стандартов соответствия

Готовность прямого режима теперь включает покрытие как в оркестраторе, так и в CLI-крейтах:

- `direct_only_policy_rejects_soranet_only_providers` гарантирует, что `TransportPolicy::DirectOnly` быстро падает, когда каждый кандидат поддерживает только реле SoraNet.【crates/sorafs_orchestrator/src/lib.rs:7238】
- `direct_only_policy_prefers_direct_transports_when_available` гарантирует использование Torii/QUIC, когда они доступны, и замыкает реле SoraNet из сессии.【crates/sorafs_orchestrator/src/lib.rs:7285】
- `direct_mode_policy_example_is_valid` парсит `docs/examples/sorafs_direct_mode_policy.json`, чтобы документация непрерывной связи с утилитами-хелперами.【crates/sorafs_orchestrator/src/lib.rs:7509】【docs/examples/sorafs_direct_mode_policy.json:1】
- `fetch_command_respects_direct_transports` запускает `sorafs_cli fetch --transport-policy=direct-only` против мокнутого шлюза Torii, обеспечивая Smoke-тест для регулируемых сред, фиксирующих прямые перевозки.【crates/sorafs_car/tests/sorafs_cli.rs:2733】
- `scripts/sorafs_direct_mode_smoke.sh` оборачивает ту же команду JSON-политикой и сохраняет табло для развертывания автоматизации.

Запускайте сфокусированный набор тестов перед публикацией обновлений:

```bash
cargo test -p sorafs_orchestrator direct_only_policy
cargo test -p sorafs_car --features cli fetch_command_respects_direct_transports
```

Если сборка рабочей области падает из-за вышестоящих изменений, зафиксируйте блокирующую ошибку в `status.md` и повторите запуск после обновления в зависимости от.

##4. Автоматизированные дым-прогоны

Одна лишь проверка CLI не выявляет регрессии, специфичные для окружения (например, дрейф политики шлюза или несоответствие манифестов). Отдельный дым-скрипт находится в `scripts/sorafs_direct_mode_smoke.sh` и оборачивает `sorafs_cli fetch` политикой оркестратора прямого режима, сохранением табло и сбором сводки.

Пример использования:

```bash
./scripts/sorafs_direct_mode_smoke.sh \
  --config docs/examples/sorafs_direct_mode_smoke.conf \
  --provider name=gw-regulated,provider-id=001122...,base-url=https://gw.example/direct/,stream-token=BASE64
```- Скрипт уважает и CLI-флаги, и ключ=значение конфиг-файлов (см. `docs/examples/sorafs_direct_mode_smoke.conf`). Перед запуском заполните дайджест-манифест и записи рекламного провайдера с прод-значениями.
- `--policy` по умолчанию имеет значение `docs/examples/sorafs_direct_mode_policy.json`, но можно передать любой JSON-оркестратор, сгенерированный `sorafs_orchestrator::bindings::config_to_json`. CLI использует политику через `--orchestrator-config=PATH`, обеспечивающую воспроизводимые запуски без ручной подгонки флагов.
- Если `sorafs_cli` нет в `PATH`, помощник собирает его из крета `sorafs_orchestrator` (профиль выпуска), чтобы дым-прогоны проверяли поставляемую схему прямого режима.
- Выходные данные:
  - Собранный полезный груз (`--output`, по умолчанию `artifacts/sorafs_direct_mode/payload.bin`).
  - Сводная выборка (`--summary`, по умолчанию рядом с полезной нагрузкой), последовательно региональная телеметрия и отчеты провайдеров для развертывания доказательной базы.
  - Снимок табло, сохраненный по пути из политики JSON (например, `fetch_state/direct_mode_scoreboard.json`). Архивируйте его вместе с резюме в билетах изменений.
- Ворота принятия автоматизации: после завершения выборки скрипта возникает `cargo xtask sorafs-adoption-check`, с использованием сохраненных табло пути и сводки. Требуемый кворум по умолчанию приводит к провайдерам в командной строке; переопределите `--min-providers=<n>` при необходимости большего образца. Отчеты Adoption пишутся рядом со сводкой (`--adoption-report=<path>` позволяет задать место), скриптом по умолчанию, добавляющим `--require-direct-only` (в соответствии с резервным вариантом) и `--require-telemetry`, если вы передаете соответствующий флаг. Используйте `XTASK_SORAFS_ADOPTION_FLAGS` для передачи дополнительных аргументов xtask (например, `--allow-single-source` во время одобренного понижения версии, чтобы получить шлюз и терпел, а также принудительное откат). Пропускайте ворота усыновления с `--skip-adoption-check` только при локальной диагностике; Дорожная карта требует, чтобы каждый регулируемый запуск в прямом режиме принятия-отчета каждого пакета.

## 5. Чек-лист развертывания

1. **Заморозка конфигурации:** сохраните JSON-профиль прямого режима в репозитории `iroha_config` и запишите хэш в тикет изменений.
2. **Аудит шлюза:** убедитесь, что эндпойнты Torii обеспечивают TLS, возможности TLV и аудиторский журнал для переключения в прямой режим. Опубликуйте профиль политики шлюза для операторов.
3. **Согласование соответствия:** установите обновленное руководство с соблюдением требований/регуляторными ревьюерами и зафиксируйте одобрение на работу вне анонимного оверлея.
4. **Пробный прогон:** выполните набор тестов на соответствие и поэтапную выборку против доверенных Torii провайдеров. Архивируйте результаты табло и сводку CLI.
5. **Переключение в прод:** объявите изменения окна, переключите `transport_policy` на `direct_only` (если выбрали `soranet-first`) и следуйте за дашбордами прямого режима (задержка `sorafs_fetch`, счетчики сбоев провайдеров). Документируйте план отката к SoraNet-first после того, как SNNet-4/5/5a/5b/6a/7/8/12/13 перейдут в `roadmap.md:532`.
6. **Пост-изменение:** приложите табло, выборку сводки и результаты Диптихов к тикету изменений. Обновите `status.md` с датой применения силы и любыми аномалиями.Держите этот контрольный список рядом с Runbook `sorafs_node_ops`, чтобы операторы могли повторить процесс для живого переключения. Когда SNNet-5 перейдет в GA, выберите резервный вариант после подтверждения паритета в продовой телеметрии.

## 6. Требования к доказательствам и воротам усыновления

Захваты прямого режима по-прежнему должны обеспечивать внедрение SF-6c. Соберите табло, сводку, конверт манифеста и отчет о принятии для каждого запуска, чтобы `cargo xtask sorafs-adoption-check` мог проверить резервную позицию. Отсутствующие поля приводят к сбою ворот, поэтому фиксируйте ожидаемый метод в изменениях билетов.

- **Метаданные перевозки:** `scoreboard.json` должен проявляться `transport_policy="direct_only"` (и переключать `transport_policy_override=true`, когда вы форсируете понижение версии). Держите поля анонимной политики заполненными даже при наследовании по умолчанию, чтобы ревьюеры не выходили из поэтапного плана анонимности.
- **Счетчики провайдеров:** Сессии только для шлюза должны сохранять `provider_count=0` и заполнять `gateway_provider_count=<n>` числами Torii провайдеров. Не редактируйте JSON вручную: CLI/SDK уже вычисляет счетчики, шлюз принятия отклоняет захваты без раздельного учета.
- **Доказательства манифеста:** При участии шлюзов Torii передайте подписанный `--gateway-manifest-envelope <path>` (или эквивалент SDK), чтобы `gateway_manifest_provided` и `gateway_manifest_id`/`gateway_manifest_cid` были заменены `scoreboard.json`. Убедитесь, что `summary.json` содержит те же `manifest_id`/`manifest_cid`; проверка принятия провалится, если ни один файл не содержит пару.
- **Ожидания телеметрии:** Когда телеметрия сопровождается захватом, запустите ворота с `--require-telemetry`, чтобы принять подтверждение подтверждения отправителя метрики. Повторения с воздушным зазором могут пропускать флаг, но CI и изменения должны документировать отсутствие.

Пример:

```bash
cargo xtask sorafs-adoption-check \
  --scoreboard fetch_state/direct_mode_scoreboard.json \
  --summary fetch_state/direct_mode_summary.json \
  --allow-single-source \
  --require-direct-only \
  --json-out artifacts/sorafs_direct_mode/adoption_report.json \
  --require-telemetry
```

Приложите `adoption_report.json` вместе сосоставьте табло, аннотацию, конверт-манифест и пачку дым-логов. Эти документы повторяют требования по принятию задания в CI (`ci/check_sorafs_orchestrator_adoption.sh`) и сохраняют аудитируемость прямого понижения версии.
