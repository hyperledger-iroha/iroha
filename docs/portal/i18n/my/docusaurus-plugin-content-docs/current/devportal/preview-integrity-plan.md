ဤအစီအစဥ်သည် ထုတ်ဝေခြင်းမပြုမီ ပေါ်တယ်အကြိုကြည့်ရှုမှုတိုင်းကို အတည်ပြုနိုင်စေရန်အတွက် လိုအပ်သောကျန်အလုပ်များကို အကြမ်းဖျင်းဖော်ပြပါသည်။ ရည်ရွယ်ချက်မှာ သုံးသပ်သူများသည် CI တွင်တည်ဆောက်ထားသော လျှပ်တစ်ပြက်ရိုက်ချက်အတိအကျကို ဒေါင်းလုဒ်လုပ်ရန် အာမခံရန်ဖြစ်ပြီး၊ checksum manifest သည် မပြောင်းလဲနိုင်သော၊ နှင့် SoraFS မှတစ်ဆင့် အစမ်းကြည့်ရှုမှုကို Norito မက်တာဒေတာဖြင့် ရှာဖွေနိုင်စေရန်ဖြစ်သည်။

## ရည်ရွယ်ချက်များ

- **Deterministic builds-** `npm run build` သည် မျိုးပွားနိုင်သော output ကိုထုတ်ပေးပြီး `build/checksums.sha256` ကို အမြဲထုတ်လွှတ်ကြောင်း သေချာပါစေ။
- **စစ်ဆေးပြီးသော အစမ်းကြည့်ရှုမှုများ-** checksum manifest ဖြင့် ပေးပို့ရန် အကြိုကြည့်ရှုမှု အနုပညာတိုင်းကို လိုအပ်ပြီး စိစစ်မှု မအောင်မြင်ပါက ထုတ်ဝေမှုကို ငြင်းဆိုပါ။
- **Norito-ထုတ်ဝေသည့် မက်တာဒေတာ-** ဆက်၍ကြည့်ရန် ဖော်ပြချက်များ (မက်တာဒေတာ၊ ချက်ခ်ဆမ်းအချေအတင်၊ SoraFS CID) Norito JSON ဖြစ်သောကြောင့် အုပ်ချုပ်မှုကိရိယာသည် စာရင်းစစ်ထုတ်ဝေမှုများကို စစ်ဆေးနိုင်ပါသည်။
- **အော်ပရေတာ ကိရိယာတန်ဆာပလာ-** သုံးစွဲသူများသည် စက်တွင်းတွင် အသုံးပြုနိုင်သည့် တစ်ခုတည်းသော စစ်ဆေးမှု script ကို ပေးဆောင်ပါ (`./docs/portal/scripts/preview_verify.sh --build-dir build --descriptor <path> --archive <path>`); script သည် ယခု checksum + descriptor validation flow ကို အဆုံးမှ အဆုံးထိ ချုပ်ထားသည်။ စံအစမ်းကြည့်ရှုသည့်အမိန့် (`npm run serve`) သည် ယခု `docusaurus serve` မတိုင်မီတွင် ဤအကူအညီကို အလိုအလျောက်ခေါ်လာသောကြောင့် ဒေသတွင်းလျှပ်တစ်ပြက်ရိုက်ချက်များသည် checksum-gated ဖြစ်နေမည် (`npm run serve:verified` နှင့် တိကျပြတ်သားသောနာမည်တူများအဖြစ် သိမ်းဆည်းထားသည်)။

## အဆင့် 1 — CI ကျင့်သုံးမှု

1. `.github/workflows/docs-portal-preview.yml` ကို အပ်ဒိတ်လုပ်ပါ-
   - Docusaurus တည်ဆောက်ပြီးနောက် `node docs/portal/scripts/write-checksums.mjs` ကိုဖွင့်ပါ (ပြည်တွင်းတွင် တောင်းဆိုထားပြီးဖြစ်သည်)။
   - `cd build && sha256sum -c checksums.sha256` ကို အကောင်အထည်ဖော်ပြီး မတိုက်ဆိုင်မှုတွင် အလုပ်ပျက်ကွက်ပါ။
   - တည်ဆောက်မှုလမ်းညွှန်ကို `artifacts/preview-site.tar.gz` အဖြစ် ထုပ်ပိုးပါ၊ checksum မန်နီးဖက်စ်ကို မိတ္တူကူးပါ၊ `scripts/generate-preview-descriptor.mjs` ကိုခေါ်ဆိုကာ `scripts/sorafs-package-preview.sh` ကို JSON config ဖြင့် လုပ်ဆောင်ပါ (`docs/examples/sorafs_preview_publish.json` ကိုကြည့်ပါ) သို့မှသာ အလုပ်အသွားအလာသည် 000000080 ၏ metadata နှင့် a09 နှစ်ခုစလုံးကို သတ်မှတ်ပေးပါသည်။ အတွဲ။
   - တည်ငြိမ်သောဆိုက်၊ မက်တာဒေတာလက်ရာများ (`docs-portal-preview`၊ `docs-portal-preview-metadata`) နှင့် SoraFS အစုအဝေး (`docs-portal-preview-sorafs`) ကို အပ်လုဒ်လုပ်ပါ၊ သို့မှသာ မန်နီးဖက်စ်၊ CAR အကျဉ်းချုပ်နှင့် အစီအစဥ်ကို ပြန်လည်ကြည့်ရှုစစ်ဆေးနိုင်ပါသည်။
2. ဆွဲထုတ်တောင်းဆိုမှုများတွင် checksum စိစစ်ခြင်းရလဒ်ကို အကျဉ်းချုပ်ဖော်ပြသည့် CI တံဆိပ်မှတ်ချက်တစ်ခု ထည့်ပါ (✅ `docs-portal-preview.yml` GitHub Script မှတ်ချက်အဆင့်မှ လုပ်ဆောင်ခဲ့သည်)။
3. `docs/portal/README.md` (CI ကဏ္ဍ) တွင် အလုပ်အသွားအလာကို မှတ်တမ်းတင်ပြီး ထုတ်ဝေခြင်းစာရင်းရှိ အတည်ပြုခြင်းအဆင့်များသို့ လင့်ခ်ချိတ်ပါ။

## အတည်ပြုခြင်း Script

`docs/portal/scripts/preview_verify.sh` သည် လက်စွဲစာအုပ် `sha256sum` တောင်းခံမှုများမလိုအပ်ဘဲ ဒေါင်းလုဒ်လုပ်ထားသော အစမ်းကြည့်ပစ္စည်းများကို တရားဝင်စေပါသည်။ ဇာတ်ညွှန်းကို run ပြီး ဒေသတွင်း လျှပ်တစ်ပြက်ရိုက်ချက်များကို မျှဝေသည့်အခါ အဆင့်တစ်ဆင့်တွင် `docusaurus serve` (သို့မဟုတ် တိကျသော `npm run serve:verified` alias) ကိုသုံးပါ။ အတည်ပြုခြင်းဆိုင်ရာ ယုတ္တိဗေဒ-

1. သင့်လျော်သော SHA ကိရိယာ (`sha256sum` သို့မဟုတ် `shasum -a 256`) ကို `build/checksums.sha256` ကို လုပ်ဆောင်သည်။
2. အကြိုကြည့်ရှုဖော်ပြသူ၏ `checksums_manifest` အချေအတင်/ဖိုင်အမည်ကို ရွေးချယ်နိုင်ပြီး၊ ပေးသောအခါတွင်၊ အကြိုကြည့်ရှုသည့် မှတ်တမ်းမှတ်တမ်း/ဖိုင်အမည်ကို ရွေးချယ်နိုင်သည်။
3. တူညီမှုတစ်ခုခုကို တွေ့ရှိသောအခါ သုညမဟုတ်သော ထွက်သွားသောကြောင့် သုံးသပ်သူများသည် ကန့်လန့်ဖြတ်ထားသော အစမ်းကြည့်ရှုမှုများကို ပိတ်ဆို့နိုင်သည်။

ဥပမာအသုံးပြုမှု (CI artefacts များကို ထုတ်ယူပြီးနောက်)

```bash
./docs/portal/scripts/preview_verify.sh \
  --build-dir build \
  --descriptor artifacts/preview-descriptor.json \
  --archive artifacts/preview-site.tar.gz
```

CI နှင့် ထုတ်ဝေသည့် အင်ဂျင်နီယာများသည် အစမ်းကြည့်ရှုသည့်အတွဲကို ဒေါင်းလုဒ်လုပ်သည့်အခါတိုင်း သို့မဟုတ် ထုတ်ဝေခွင့်လက်မှတ်တွင် artefacts များကို ပူးတွဲတင်သည့်အခါတိုင်း script ကို ခေါ်သင့်သည်။

## အဆင့် 2 — SoraFS ထုတ်ဝေခြင်း။

1. အောက်ပါအလုပ်ဖြင့် အစမ်းကြည့်ရှုခြင်းလုပ်ငန်းကို တိုးချဲ့ပါ-
   - `sorafs_cli car pack` နှင့် `manifest submit` ကို အသုံးပြု၍ တည်ဆောက်ထားသော site ကို SoraFS တွင် အဆင့်မြှင့်တင်ခြင်းတံခါးပေါက်သို့ အပ်လုဒ်လုပ်ပါ။
   - ပြန်လာသော manifest digest နှင့် SoraFS CID ကို ဖမ်းယူသည်။
   - `{ commit, branch, checksum_manifest, cid }` ကို Norito JSON (`docs/portal/preview/preview_descriptor.json`) သို့ စီးရီးခွဲသည်။
2. သရုပ်ဖော်သူကို တည်ဆောက်ထားသည့် လက်ရာများနှင့်အတူ သိမ်းဆည်းပြီး ဆွဲထုတ်တောင်းဆိုချက် မှတ်ချက်တွင် CID ကို ဖော်ထုတ်ပါ။
3. အနာဂတ်ပြောင်းလဲမှုများသည် metadata schema တည်ငြိမ်နေစေရန် သေချာစေရန် `sorafs_cli` ကို ခြောက်သွေ့သောမုဒ်တွင် ကျင့်သုံးသည့် ပေါင်းစပ်စမ်းသပ်မှုများကို ပေါင်းထည့်ပါ။

## အဆင့် 3 — အုပ်ချုပ်မှုနှင့် စာရင်းစစ်

1. Norito (`PreviewDescriptorV1`) ကို `docs/portal/schemas/` အောက်တွင် ဖော်ပြသည့် ဖွဲ့စည်းပုံကို ဖော်ပြသည့် Norito schema (`PreviewDescriptorV1`) ကို ထုတ်ဝေပါ။
2. လိုအပ်ရန် DOCS-SORA ထုတ်ဝေခြင်းစစ်ဆေးရန်စာရင်းကို အပ်ဒိတ်လုပ်ပါ-
   - အပ်လုဒ်လုပ်ထားသော CID ကိုဆန့်ကျင်သည့် `sorafs_cli manifest verify` ကိုလုပ်ဆောင်သည်။
   - ထုတ်ပြန်ချက် PR ဖော်ပြချက်တွင် checksum manifest digest နှင့် CID ကို မှတ်တမ်းတင်ခြင်း။
3. လွှတ်တင်မဲများအတွင်း checksum manifest နှင့်ဆန့်ကျင်ဘက်ဖော်ပြချက်ကို အပြန်အလှန်စစ်ဆေးရန် အုပ်ချုပ်မှုအလိုအလျောက်စနစ်အား ချိတ်ဆက်ပါ။

| SoraFS အစမ်းကြည့်ရှုခြင်း ဖြန့်ချိခြင်း | Docs အခြေခံအဆောက်အဦ / သိုလှောင်မှုအဖွဲ့ | ရက်သတ္တပတ် 2 | အဆင့်သတ်မှတ်ထားသော အထောက်အထားများနှင့် Norito schema အပ်ဒိတ်များသို့ ဝင်ရောက်ခွင့် လိုအပ်သည်။ |

## မေးခွန်းများဖွင့်ပါ။

- မည်သည့် SoraFS ပတ်ဝန်းကျင်သည် အကြိုကြည့်ရှုခြင်းဆိုင်ရာ ပစ္စည်းများကို ကိုင်ဆောင်ထားသင့်သည် (staging vs. dedicated preview lane)
- မထုတ်ဝေမီ အကြိုကြည့်ရှုဖော်ပြသူတွင် လက်မှတ်နှစ်ခု (Ed25519 + ML-DSA) လိုအပ်ပါသလား။
- `sorafs_cli` ကိုအသုံးပြုနေချိန်တွင် CI အလုပ်အသွားအလာသည် သံစုံတီးဝိုင်းဖွဲ့စည်းမှုပုံစံ (`orchestrator_tuning.json`) ကို တွဲချိတ်သင့်ပါသလား။

`docs/portal/docs/reference/publishing-checklist.md` တွင် ဆုံးဖြတ်ချက်များကို ဖမ်းယူပြီး မသိနားမလည်မှုများကို ဖြေရှင်းပြီးသည်နှင့် ဤအစီအစဉ်ကို အပ်ဒိတ်လုပ်ပါ။
