##ရည်ရွယ်ချက်

ဤ runbook သည် အရေးကြီးသော အလုပ်အသွားအလာ နှစ်ခုအားဖြင့် SRE နှင့် ဖုန်းခေါ်ဆိုမှု အင်ဂျင်နီယာများကို လမ်းညွှန်ပေးသည်-

1. ထိန်းချုပ်ထားသော လှိုင်းများတွင် ရင်းမြစ်ပေါင်းစုံ တီးမှုတ်သူကို ဖြန့်ကျက်ပါ။
2. လက်ရှိဆက်ရှင်များကို မတည်မငြိမ်ဖြစ်စေခြင်းမရှိဘဲ အကျင့်ပျက် ဝန်ဆောင်မှုပေးသူများကို အမည်ပျက်စာရင်းသွင်းခြင်း သို့မဟုတ် ဦးစားပေးဆောင်ရွက်ခြင်း။

SF-6 အောက်တွင် ပေးပို့ထားသော orchestration stack ကို အသုံးပြုထားပြီး (`sorafs_orchestrator`၊ gateway chunk-range API၊ telemetry exporters) ဟု ယူဆပါသည်။

> **ကိုလည်းကြည့်ပါ-** [Orchestrator Operations Runbook](./orchestrator-ops.md) သည် လည်ပတ်မှုလုပ်ထုံးလုပ်နည်းများ (ရမှတ်ဘုတ်ဖမ်းခြင်း၊ အဆင့်သတ်မှတ်ထားသော ဖွင့်ပိတ်ခလုတ်များ၊ နောက်ပြန်ဆွဲခြင်း) တွင် အကျုံးဝင်သည်။ တိုက်ရိုက်ပြောင်းလဲမှုများအတွင်း ကိုးကားချက်နှစ်ခုလုံးကို အတူတကွသုံးပါ။

## 1. အကြိုပျံသန်းမှု အတည်ပြုခြင်း။

1. **အုပ်ချုပ်ရေးဆိုင်ရာ သွင်းအားစုများကို အတည်ပြုပါ။**
   - ကိုယ်စားလှယ်လောင်းဝန်ဆောင်မှုပေးသူအားလုံးသည် `ProviderAdvertV1` အကွာအဝေးပေးချေနိုင်စွမ်းနှင့် ထုတ်လွှင့်မှုဘတ်ဂျက်များပါရှိသော စာအိတ်များကို ထုတ်ဝေရပါမည်။ `/v1/sorafs/providers` မှတစ်ဆင့် မှန်ကန်ကြောင်း အတည်ပြုပြီး မျှော်လင့်ထားသည့် စွမ်းရည်နယ်ပယ်များနှင့် နှိုင်းယှဉ်ပါ။
   - ချိန်ညှိမှု/ပျက်ကွက်မှုနှုန်းများကို ထောက်ပံ့ပေးသည့် တယ်လီမီတာ လျှပ်တစ်ပြက်ရိုက်ချက်များသည် ကိန္နရီမပြေးမီ 15 မိနစ် သက်တမ်းရှိသင့်သည်။
2. ** အဆင့်သတ်မှတ်ချက်။**
   - အလွှာလိုက် `iroha_config` သစ်ပင်တွင် သံစုံတီးဝိုင်းဆရာ JSON config ကို ဆက်ထားပါ။

     ```toml
     [torii.sorafs.orchestrator]
     config_path = "/etc/iroha/sorafs/orchestrator.json"
     ```

     ထုတ်ပေးခြင်းဆိုင်ရာ သီးခြားကန့်သတ်ချက်များဖြင့် JSON ကို အပ်ဒိတ်လုပ်ပါ (`max_providers`၊ ထပ်စမ်းကြည့်ပါ ဘတ်ဂျက်များ)။ တူညီသောဖိုင်ကို အဆင့်သတ်မှတ်ခြင်း/ထုတ်လုပ်ခြင်းသို့ ကျွေးမွေးခြင်းဖြင့် ကွဲပြားမှုများ သေးငယ်နေမည်ဖြစ်သည်။
3. ** လေ့ကျင့်ခန်းလုပ်ထုံးလုပ်နည်းများ။**
   - မန်နီးဖက်စ်/တိုကင်ပတ်ဝန်းကျင် ကိန်းရှင်များကို ဖြည့်သွင်းပြီး အဆုံးအဖြတ်ရယူမှုကို လုပ်ဆောင်ပါ-

     ```bash
     sorafs_cli fetch \
       --plan fixtures/sorafs_manifest/ci_sample/payload.plan.json \
       --manifest-id "$CANARY_MANIFEST_ID" \
       --provider name=alpha,provider-id="$PROVIDER_ALPHA_ID",base-url=https://gw-alpha.example,stream-token="$PROVIDER_ALPHA_TOKEN" \
       --provider name=beta,provider-id="$PROVIDER_BETA_ID",base-url=https://gw-beta.example,stream-token="$PROVIDER_BETA_TOKEN" \
       --provider name=gamma,provider-id="$PROVIDER_GAMMA_ID",base-url=https://gw-gamma.example,stream-token="$PROVIDER_GAMMA_TOKEN" \
       --max-peers=3 \
       --retry-budget=4 \
       --scoreboard-out artifacts/canary.scoreboard.json \
       --json-out artifacts/canary.fetch.json
     ```

     ပတ်ဝန်းကျင် ကိန်းရှင်များတွင် ကိန္နရီတွင် ပါဝင်သော ဝန်ဆောင်မှုပေးသူတိုင်းအတွက် manifest payload digest (hex) နှင့် base64-encoded stream တိုကင်များ ပါဝင်သင့်သည်။
   - ယခင်ထုတ်လွှတ်မှုနှင့် `artifacts/canary.scoreboard.json` ကွဲပြားသည်။ အရည်အချင်းမပြည့်မီသော ပံ့ပိုးသူအသစ် သို့မဟုတ် ကိုယ်အလေးချိန် အပြောင်းအလဲ> 10% ကို ပြန်လည်သုံးသပ်ရန် လိုအပ်သည်။
4. **တယ်လီမီတာ ကြိုးတပ်ထားကြောင်း စစ်ဆေးပါ။**
   - Grafana ကို `docs/examples/sorafs_fetch_dashboard.json` တွင် ဖွင့်ပါ။ အဆင့်မမြှင့်မီ အဆင့်တွင် `sorafs_orchestrator_*` မက်ထရစ်များ ပြည့်နေကြောင်း သေချာပါစေ။

## 2. အရေးပေါ်ဝန်ဆောင်မှုပေးသူကို နာမည်ပျက်စာရင်းသွင်းခြင်း။

ဝန်ဆောင်မှုပေးသူတစ်ဦးသည် အကျင့်ပျက်ခြစားသောအပိုင်းများကို ဆောင်ရွက်ပေးခြင်း၊ ဆက်တိုက်အချိန်ကုန်ခြင်း သို့မဟုတ် လိုက်နာမှုစစ်ဆေးမှုများ ပျက်ကွက်သည့်အခါ ဤလုပ်ငန်းစဉ်ကို လိုက်နာပါ။

1. **အထောက်အထားများ ဖမ်းယူပါ။**
   - နောက်ဆုံးထုတ်ယူမှုအနှစ်ချုပ် (`--json-out` အထွက်) ကို ထုတ်ယူပါ။ ပျက်ကွက်နေသော အတုံးအခဲညွှန်းကိန်းများ၊ ဝန်ဆောင်မှုပေးသူအမည်တူများနှင့် မကိုက်ညီမှုများကို ချေဖျက်ပါ။
   - `telemetry::sorafs.fetch.*` ပစ်မှတ်များမှ သက်ဆိုင်ရာ မှတ်တမ်းကောက်နုတ်ချက်များကို သိမ်းဆည်းပါ။
2. **ချက်ချင်း override ကို အသုံးပြုပါ။**
   - သံစုံတီးဝိုင်းမှ ဖြန့်ဝေထားသည့် တယ်လီမီတာလျှပ်တစ်ပြက်တွင် အပြစ်ပေးသူကို အမှတ်အသားပြုပါ (`penalty=true` သို့မဟုတ် `token_health` ကို `0` သို့ ကုပ်ပါ)။ နောက်ရမှတ်ဘုတ်တည်ဆောက်မှုသည် ဝန်ဆောင်မှုပေးသူကို အလိုအလျောက်ဖယ်ထုတ်မည်ဖြစ်သည်။
   - ad-hoc မီးခိုးစမ်းသပ်မှုများအတွက်၊ `--deny-provider gw-alpha` ကို `sorafs_cli fetch` သို့ ကျော်ဖြတ်ပြီး တယ်လီမီတာ ပြန့်ပွားမှုကို စောင့်စရာမလိုဘဲ ကျရှုံးမှုလမ်းကြောင်းကို ကျင့်သုံးပါ။
   - မွမ်းမံထားသော telemetry/config အစုအဝေးကို ထိခိုက်သည့်ပတ်ဝန်းကျင်တွင် ပြန်လည်အသုံးပြုပါ (ဇာတ်ညွှန်း → ကိန္နရီ → ထုတ်လုပ်မှု)။ အဖြစ်အပျက်မှတ်တမ်းတွင် အပြောင်းအလဲကို မှတ်တမ်းတင်ပါ။
3. ** ထပ်လောင်းအတည်ပြုပါ။**
   - canonical fixture ကိုပြန်လည်လုပ်ဆောင်ပါ။ `policy_denied` အကြောင်းပြချက်ဖြင့် ဝန်ဆောင်မှုပေးသူကို အရည်အချင်းမပြည့်မီကြောင်း အမှတ်ပေးဇယားကို အတည်ပြုပါ။
   - ငြင်းဆိုထားသောဝန်ဆောင်မှုပေးသူအတွက် ကောင်တာသည် တိုးလာနေခြင်းကိုရပ်တန့်စေရန် `sorafs_orchestrator_provider_failures_total` ကိုစစ်ဆေးပါ။
4. **နှစ်ရှည်လများ တားမြစ်ပိတ်ပင်မှုများကို အရှိန်မြှင့်ပါ။**
   - ဝန်ဆောင်မှုပေးသူသည် 24 နာရီကြာ ဆက်လက်ပိတ်ဆို့နေပါက၊ ၎င်း၏ကြော်ငြာကို အလှည့်ကျ သို့မဟုတ် ဆိုင်းငံ့ရန် အုပ်ချုပ်မှုလက်မှတ်တစ်စောင်ကို မြှင့်တင်ပါ။ မဲမပြီးမချင်း၊ ငြင်းပယ်သည့်စာရင်းကို တစ်နေရာတည်းတွင် ထားရှိကာ တယ်လီမီတာရိုက်ချက်များကို ပြန်လည်ဆန်းသစ်စေသောကြောင့် ဝန်ဆောင်မှုပေးသူသည် အမှတ်စာရင်းကို ပြန်လည်ထည့်သွင်းခြင်းမပြုပါ။
5. ** ပြန်လှည့်ပရိုတိုကော။**
   - ဝန်ဆောင်မှုပေးသူကို ပြန်လည်ထည့်သွင်းရန်၊ ၎င်းကို ငြင်းဆိုစာရင်းမှ ဖယ်ရှားပါ၊ ပြန်လည်အသုံးပြုကာ အသစ်အသစ်သော ရမှတ်ဘုတ်လျှပ်တစ်ပြက်ဓာတ်ပုံကို ရိုက်ကူးပါ။ အပြောင်းအလဲကို အခင်းဖြစ်ပြီးနောက် အလောင်းပြန်စစ်ဆေးခြင်းတွင် ပူးတွဲပါ ။

## 3. အဆင့်မြှင့်တင်မှု အစီအစဉ်

| အဆင့် | နယ်ပယ် | လိုအပ်သောအချက်များ | Go/No-Go သတ်မှတ်ချက် |
|---------|-------|------------------|--------------------|
| **Lab** | အပ်နှံပေါင်းစပ်မှုအစုအဝေး | ခံစစ်ဝန်ထမ်းများ | အပိုင်းအားလုံး အောင်မြင်သည်၊ ဝန်ဆောင်မှုပေးသူ ပျက်ကွက်ကောင်တာများသည် 0 တွင်ရှိနေသည်၊ ထပ်စမ်းကြည့်နှုန်း < 5%။ |
| **Staging** | ထိန်းချုပ်မှုအပြည့်-လေယာဉ်အဆင့် | Grafana ဒက်ရှ်ဘုတ် ချိတ်ဆက်ထားသည်။ သတိပေးချက်-သီးသန့်မုဒ် | `sorafs_orchestrator_active_fetches` သည် စမ်းသပ်မှုတစ်ခုစီပြီးနောက် သုညသို့ ပြန်သွားသည်၊ `warn/critical` တပ်လှန့်ပစ်ခတ်မှုများမရှိပါ။ |
| **ကိန္နရီ** | ထုတ်လုပ်မှုအသွားအလာ ≤10% | ပေဂျာကို အသံပိတ်ထားသော်လည်း အချိန်နှင့်တပြေးညီ စောင့်ကြည့်နေသည့် တယ်လီမီတာ | ထပ်စမ်းကြည့်ပါ အချိုး < 10%, ဝန်ဆောင်မှုပေးသူ မအောင်မြင်မှုများသည် လူသိများသော ဆူညံသောရွယ်တူများနှင့် သီးခြားခွဲထားသည်၊ latency histogram သည် အခြေခံအဆင့် ±20% နှင့် ကိုက်ညီပါသည်။ |
| **အထွေထွေရရှိနိုင်မှု** | 100% ထုတ်လွှတ် | ပေဂျာစည်းမျဉ်းများ အသက်ဝင် | 24 နာရီအတွက် သုည `NoHealthyProviders` အမှားများ၊ ထပ်စမ်းကြည့်ပါ အချိုးတည်ငြိမ်သော၊ ဒက်ရှ်ဘုတ် SLA အကန့်များ အစိမ်းရောင်။ |

အဆင့်တစ်ခုစီအတွက်-

1. ရည်ရွယ်ထားသော `max_providers` ဖြင့် သံစုံတီးဝိုင်း JSON ကို အပ်ဒိတ်လုပ်ပြီး ဘတ်ဂျက်များကို ထပ်စမ်းကြည့်ပါ။
2. `sorafs_cli fetch` သို့မဟုတ် SDK ပေါင်းစပ်စမ်းသပ်မှုအစုံအလင်ကို canonical fixture နှင့် ပတ်ဝန်းကျင်မှ ထင်ရှားသော ကိုယ်စားလှယ်အဖြစ် လုပ်ဆောင်ပါ။
3. ရမှတ်ဘုတ် + အနှစ်ချုပ်လက်ရာများကို ဖမ်းယူပြီး ထုတ်ဝေမှုမှတ်တမ်းတွင် ပူးတွဲပါ ။
4. နောက်အဆင့်သို့မမြှင့်တင်မီ ဖုန်းခေါ်ဆိုမှုအင်ဂျင်နီယာနှင့်အတူ တယ်လီမီတာ ဒက်ရှ်ဘုတ်များကို ပြန်လည်စစ်ဆေးပါ။

## 4. Observability & Incident Hooks

- **Metrics-** Alertmanager မော်နီတာ `sorafs_orchestrator_fetch_failures_total{reason="no_healthy_providers"}` နှင့် `sorafs_orchestrator_retries_total` သေချာပါစေ။ ရုတ်တရက် ဆူးတက်ခြင်းသည် ပုံမှန်အားဖြင့် ဝန်ဆောင်မှုပေးသူတစ်ဦးသည် ဝန်အောက်ကျဆင်းနေသည်ဟု ဆိုလိုသည်။
- **မှတ်တမ်းများ-** `telemetry::sorafs.fetch.*` ပစ်မှတ်များကို မျှဝေထားသော မှတ်တမ်းစုစည်းမှုဆီသို့ လမ်းကြောင်းပေးပါ။ Triage ကိုအရှိန်မြှင့်ရန် `event=complete status=failed` အတွက်သိမ်းဆည်းထားသောရှာဖွေမှုများကိုတည်ဆောက်ပါ။
- **အမှတ်ပေးဇယားများ-** ရမှတ်ဘုတ်၏လက်ရာတိုင်းကို ရေရှည်သိုလှောင်မှုအဖြစ် ဆက်လက်လုပ်ဆောင်ပါ။ JSON သည် လိုက်နာမှုပြန်လည်သုံးသပ်ခြင်းနှင့် အဆင့်လိုက်ပြန်လှည့်ခြင်းများအတွက် အထောက်အထားလမ်းကြောင်းအဖြစ် နှစ်ဆတိုးလာသည်။
- **Dashboards-** Canonical Grafana board (`docs/examples/sorafs_fetch_dashboard.json`) ကို `docs/examples/sorafs_fetch_alerts.yaml` မှ သတိပေးချက်စည်းမျဉ်းများနှင့်အတူ ထုတ်လုပ်ရေးဖိုင်တွဲသို့ ပွားပါ။

## 5. Communication & Documentation

- အချိန်တံဆိပ်၊ အော်ပရေတာ၊ အကြောင်းပြချက်နှင့် ဆက်စပ်ဖြစ်ရပ်များနှင့်အတူ လုပ်ဆောင်ချက်ပြောင်းလဲမှုမှတ်တမ်းတွင် ငြင်းဆို/မြှင့်တင်ရန် ပြောင်းလဲမှုတိုင်းကို မှတ်တမ်းတင်ပါ။
- ဝန်ဆောင်မှုပေးသူအလေးများ သို့မဟုတ် သုံးစွဲသူဖက်မှမျှော်လင့်ချက်များကို ချိန်ညှိရန် ဘတ်ဂျက်များပြောင်းလဲသည့်အခါ SDK အဖွဲ့များကို အကြောင်းကြားပါ။
- GA ပြီးပါက၊ ထုတ်ဝေမှုအနှစ်ချုပ်ဖြင့် `status.md` ကို အပ်ဒိတ်လုပ်ကာ ဤထွက်ပေါ်စာအုပ်ကိုးကားချက်ကို ထုတ်ဝေမှုမှတ်စုများတွင် သိမ်းဆည်းပါ။
