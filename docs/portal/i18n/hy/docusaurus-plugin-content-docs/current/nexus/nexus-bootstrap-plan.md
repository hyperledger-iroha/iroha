# Sora Nexus Bootstrap & Observability Plan

## Նպատակներ
- Կառավարման ստեղներով, Torii API-ներով և կոնսենսուսի մոնիտորինգով կանգնեցրեք Sora Nexus բազային վավերացնող/դիտորդ ցանցը:
- Վավերացրեք հիմնական ծառայությունները (Torii, կոնսենսուս, համառություն) նախքան SoraFS/SoraNet piggyback տեղակայումները միացնելը:
- Ստեղծեք CI/CD աշխատանքային հոսքեր և դիտելիության վահանակներ/զգուշացումներ՝ ապահովելու ցանցի առողջությունը:

## Նախադրյալներ
- Կառավարման հիմնական նյութը (խորհրդի բազմանշանակ, կոմիտեի բանալիներ) հասանելի է HSM-ում կամ Vault-ում:
- Ելակետային ենթակառուցվածք (Kubernetes կլաստերներ կամ մերկ մետաղական հանգույցներ) առաջնային/երկրորդային շրջաններում:
- Թարմացված bootstrap կոնֆիգուրացիան (`configs/nexus/bootstrap/*.toml`), որն արտացոլում է կոնսենսուսի վերջին պարամետրերը:

## Ցանցային միջավայրեր
- Գործարկեք երկու Nexus միջավայրեր՝ հստակ ցանցային նախածանցներով.
- **Sora Nexus (mainnet)** – արտադրական ցանցի նախածանց `nexus`, հոսթինգ կանոնական կառավարում և SoraFS/SoraNet piggyback ծառայություններ (շղթայի ID `0x02F1` / UU000001):
- **Sora Testus (testnet)** – բեմադրող ցանցի նախածանց `testus`, հայելային ցանցի կոնֆիգուրացիան ինտեգրման փորձարկման և նախնական թողարկման վավերացման համար (շղթա UUID `809574f5-fee7-5e69-bfcf-52451e42d50f`):
- Պահպանեք առանձին ծագման ֆայլեր, կառավարման բանալիներ և ենթակառուցվածքների հետքեր յուրաքանչյուր միջավայրի համար: Testus-ը գործում է որպես SoraFS/SoraNet-ի բոլոր թողարկումների ապացուցման հիմք՝ նախքան Nexus-ի առաջխաղացումը:
- CI/CD խողովակաշարերը նախ պետք է տեղակայվեն Testus-ում, կատարեն ծխի ավտոմատ թեստեր և պահանջեն ձեռքով առաջխաղացում դեպի Nexus, երբ ստուգումները անցնեն:
- Հղման կազմաձևման փաթեթները գործում են `configs/soranexus/nexus/` (հիմնական ցանց) և `configs/soranexus/testus/` (փորձացանցի) ներքո, որոնցից յուրաքանչյուրը պարունակում է `config.toml`, `genesis.json` և Torii նմուշներ:

## Քայլ 1 – Կազմաձևման վերանայում
1. Աուդիտ գոյություն ունեցող փաստաթղթերը.
   - `docs/source/nexus/architecture.md` (համաձայնություն, Torii դասավորություն):
   - `docs/source/nexus/deployment_checklist.md` (ինֆրա պահանջներ):
   - `docs/source/nexus/governance_keys.md` (խնամակալության հիմնական ընթացակարգեր):
2. Վավերացրեք genesis ֆայլերը (`configs/nexus/genesis/*.json`)՝ համահունչ ընթացիկ վավերացնողների անվանացանկի և խաղադրույքների կշիռների հետ:
3. Հաստատեք ցանցի պարամետրերը.
   - Համաձայնության հանձնաժողովի չափը և քվորումը:
   - Արգելափակել միջակայքը / վերջնական շեմերը:
   - Torii սպասարկման նավահանգիստներ և TLS վկայագրեր:

## Քայլ 2 – Bootstrap Cluster-ի տեղակայում
1. Տրամադրման վավերացնող հանգույցներ.
   - Տեղադրեք `irohad` օրինակներ (վավերատորներ) մշտական ծավալներով:
   - Համոզվեք, որ ցանցի firewall-ի կանոնները թույլ են տալիս կոնսենսուս և Torii տրաֆիկ հանգույցների միջև:
2. Սկսեք Torii ծառայությունները (REST/WebSocket) յուրաքանչյուր վավերացնողի վրա TLS-ով:
3. Տեղադրեք դիտորդների հանգույցներ (միայն կարդալու համար) լրացուցիչ ճկունության համար:
4. Գործարկեք bootstrap scripts (`scripts/nexus_bootstrap.sh`)՝ գենեզը տարածելու, կոնսենսուս սկսելու և հանգույցները գրանցելու համար:
5. Կատարեք ծխի թեստեր.
   - Ներկայացրեք փորձնական գործարքներ Torii (`iroha_cli tx submit`) միջոցով:
   - Ստուգեք բլոկի արտադրությունը/վերջնականությունը հեռաչափության միջոցով:
   - Ստուգեք մատյանների կրկնօրինակումը վավերացնողների/դիտորդների միջև:

## Քայլ 3 – Կառավարում և հիմնական կառավարում
1. Բեռնման խորհրդի բազմաչափ կոնֆիգուրացիա; հաստատել կառավարման առաջարկները, որոնք կարող են ներկայացվել և վավերացվել:
2. Ապահով պահել կոնսենսուսի/կոմիտեի բանալիները; կարգավորել ավտոմատ կրկնօրինակումները՝ մուտքի գրանցման միջոցով:
3. Կարգավորեք շտապ բանալիների պտտման ընթացակարգերը (`docs/source/nexus/key_rotation.md`) և ստուգեք «runbook»-ը:

## Քայլ 4 – CI/CD ինտեգրում
1. Կազմաձևեք խողովակաշարերը.
   - Ստեղծեք և հրապարակեք վավերացնող/Torii պատկերներ (GitHub Actions կամ GitLab CI):
   - Ավտոմատացված կազմաձևման վավերացում (lint genesis, ստուգեք ստորագրությունները):
   - Տեղակայման խողովակաշարեր (Helm/Customize) բեմականացման և արտադրական կլաստերների համար:
2. Իրականացնել ծխի թեստեր CI-ում (շրջել ժամանակավոր կլաստերը, գործարկել կանոնական գործարքների փաթեթը):
3. Ավելացրեք հետադարձ սկրիպտներ ձախողված տեղակայումների և փաստաթղթերի աշխատատեղերի համար:

## Քայլ 5 – Դիտարկման հնարավորություն և ահազանգեր
1. Տեղադրեք մոնիտորինգի կույտ (Prometheus + Grafana + Alertmanager) յուրաքանչյուր տարածաշրջանի համար:
2. Հավաքեք հիմնական ցուցանիշները.
  - `nexus_consensus_height`, `nexus_finality_lag`, `torii_request_duration_seconds`, `validator_peer_count`:
   - Loki/ELK-ի միջոցով գրանցվում է Torii և կոնսենսուսային ծառայությունների համար:
3. Վահանակներ.
   - Կոնսենսուսի առողջություն (բլոկի բարձրություն, վերջնականություն, հասակակիցների կարգավիճակ):
   - Torii API-ի հետաձգման/սխալի մակարդակը:
   - Կառավարման գործարքներ և առաջարկի կարգավիճակներ:
4. Ահազանգեր.
   - Բլոկների արտադրության տաղավար (> 2 բլոկային ընդմիջումներով):
   - Գործընկերների թիվը նվազում է քվորումից:
   - Torii սխալի արագության բարձրացում:
   - Կառավարման առաջարկների հերթերի կուտակում:

## Քայլ 6 – Վավերացում և հանձնում
1. Գործարկել վերջից մինչև վերջ վավերացումը.
   - Ներկայացրեք կառավարման առաջարկ (օրինակ՝ պարամետրի փոփոխություն):
   - Այն մշակել խորհրդի հաստատման միջոցով՝ ապահովելու կառավարման խողովակաշարի աշխատանքը:
   - Գործարկել գրանցամատյանի վիճակի տարբերությունը՝ հետևողականություն ապահովելու համար:
2. Փաստաթղթերի հերթական գրքույկը կանչի համար (միջադեպի արձագանքում, ձախողում, մասշտաբավորում):
3. Հաղորդել պատրաստակամության մասին SoraFS/SoraNet թիմերին; հաստատել piggyback տեղակայումները կարող են մատնանշել Nexus հանգույցները:

## Իրականացման ստուգաթերթ
- [ ] Ծննդոց/կոնֆիգուրացիայի աուդիտ ավարտված է:
- [ ] Վավերացնող և դիտորդ հանգույցներ տեղակայված են առողջ համաձայնությամբ:
- [ ] Կառավարման բանալիները բեռնված են, առաջարկը փորձարկվել է:
- [ ] CI/CD խողովակաշարեր, որոնք աշխատում են (կառուցում + տեղակայում + ծխի փորձարկումներ):
- [ ] Դիտորդական վահանակները գործում են ահազանգով:
- [ ] Հանդոֆի փաստաթղթերը տրամադրվել են հոսանքով ներքևող թիմերին:
