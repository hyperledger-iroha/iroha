# SoraFS Պահպանման հզորության շուկա (SF-2c նախագիծ)

SF-2c ճանապարհային քարտեզի կետը ներկայացնում է կառավարվող շուկա, որտեղ պահեստավորում է
մատակարարները հայտարարում են պարտավորված հզորություն, ստանում են կրկնօրինակման պատվերներ և վաստակում վճարներ
առաքված հասանելիությանը համաչափ: Այս փաստաթուղթը ընդգրկում է առաքման արդյունքները
պահանջվում է առաջին թողարկման համար և դրանք բաժանում է գործող ուղիների:

## Նպատակներ

- Էքսպրես մատակարարի հզորության պարտավորություններ (ընդհանուր բայթեր, յուրաքանչյուր գծի սահմանաչափեր, ժամկետի ավարտ)
  ստուգելի ձևով, որը սպառվում է կառավարման, SoraNet տրանսպորտի և Torii-ի կողմից:
- Բաշխեք կապում մատակարարների միջև՝ ըստ հայտարարված հզորության, խաղադրույքի և
  քաղաքականության սահմանափակումները՝ պահպանելով դետերմինիստական վարքագիծը:
- Հաշվիչների պահեստավորման առաքում (կրկնօրինակման հաջողություն, գործարկման ժամանակ, ամբողջականության ապացույցներ) և
  արտահանման հեռաչափություն վճարների բաշխման համար:
- Ապահովեք չեղարկման և վիճարկման գործընթացները, որպեսզի անազնիվ մատակարարները լինեն
  պատժվել կամ հեռացվել:

#### Սխեմայի սահմանումներ (իրականացված)

- `CapacityDeclarationV1`-ը գրավում է յուրաքանչյուր մատակարարի համար նախատեսված հզորության պարտավորությունները, ներառյալ կանոնական բլոկների բռնակները, հնարավորությունների հղումները, կամընտիր գծերի կափարիչները, գնային ակնարկները, վավերականության պատուհանները և մետատվյալները: Վավերացումը ապահովում է ոչ զրոյական խաղադրույքը, կանոնական բռնակները, կրկնօրինակված անունները, յուրաքանչյուր երթուղու գլխարկները հայտարարված ընդհանուր գումարի սահմաններում և GiB-ի միապաղաղ հաշվառում:【crates/sorafs_manifest/src/capacity.rs:28】
- `ReplicationOrderV1`-ը կապում է մանիֆեստները կառավարման կողմից տրված հանձնարարականներին՝ ավելորդության թիրախներով, SLA շեմերով և յուրաքանչյուր հանձնարարության երաշխիքներով. վավերացնողները պարտադրում են կանոնական chunker-ի բռնակներ, եզակի մատակարարներ և վերջնաժամկետների սահմանափակումներ մինչև Torii-ը կամ գրանցամատյանը կլանել պատվերը:【crates/sorafs_manifest/src/capacity.rs:301】
- `CapacityTelemetryV1`-ն արտահայտում է դարաշրջանի նկարներ (հայտարարված ընդդեմ օգտագործված GiB-ի, կրկնօրինակման հաշվիչներ, ժամանակի/PoR տոկոսներ), որոնք սնուցում են վճարների բաշխումը: Սահմանների ստուգումները պահպանում են օգտագործումը հայտարարագրերի և տոկոսների սահմաններում 0 – 100%-ի սահմաններում:【crates/sorafs_manifest/src/capacity.rs:476】
- Համօգտագործվող օգնականները (`CapacityMetadataEntry`, `PricingScheduleV1`, գիծ/հանձնարարություն/SLA վավերացնողներ) ապահովում են դետերմինիստական բանալիների վավերացում և սխալի մասին հաղորդում, որ CI և ներքևի գործիքավորումը կարող են կրկին օգտագործել:【crates/sorafs_manifest/src/capacity.rs:230
- `PinProviderRegistry`-ն այժմ ցուցադրում է շղթայական նկարը `/v2/sorafs/capacity/state`-ի միջոցով՝ համատեղելով մատակարարի հայտարարագրերը և վճարների մատյանում գրանցումները դետերմինիստական Norito-ի հետևում JSON.【crates/iroha_torii/src/sorafs/registry.rs:17】【crates/iroha_torii/src/sorafs/api.rs:64】
- Վավերացման ծածկույթն իրականացնում է կանոնական բռնակի կիրառում, կրկնակի հայտնաբերում, յուրաքանչյուր գծի սահմաններ, կրկնօրինակման նշանակման պաշտպանիչներ և հեռաչափության տիրույթի ստուգումներ, որպեսզի ռեգրեսիաները անմիջապես հայտնվեն CI-ում:【crates/sorafs_manifest/src/capacity.rs:792】
- Օպերատորի գործիքավորում. `sorafs_manifest_stub capacity {declaration, telemetry, replication-order}`-ը փոխակերպում է մարդու համար ընթեռնելի բնութագրերը Norito կանոնական բեռների, base64 բլբերների և JSON ամփոփագրերի, որպեսզի օպերատորները կարողանան բեմադրել `/v2/sorafs/capacity/declare`, I18NI000000049X, վերարտադրման կարգի լոկալ հարմարանքներով, վավերացում։【crates/sorafs_car/src/bin/sorafs_manifest_stub/capacity.rs:1】 Հղման հարմարանքները գործում են `fixtures/sorafs_manifest/replication_order/`-ում (`order_v1.json`, I18NI000000052X, `order_v1.to`) և ստեղծվում են vi30.0000000052X-ում և ստեղծվում են vi30.

## GA պատրաստվածության ստուգաթերթ

Ճանապարհային քարտեզի կետ **SF-2c** դարպասների արտադրությունը ներդնում է հաշվապահական հաշվառման մեջ կոնկրետ ապացույցների հիման վրա,
վեճերի լուծում և ներբեռնում: Ընդունման չափանիշները պահպանելու համար օգտագործեք ստորև բերված արտեֆակտները
իրականացման հետ համաժամանակյա:

### Գիշերային հաշվառում և XOR հաշտեցում
- Արտահանեք հզորության վիճակի պատկերը և XOR մատյանը նույն պատուհանի համար, այնուհետև գործարկեք.
  ```bash
  python3 scripts/telemetry/capacity_reconcile.py \
    --snapshot artifacts/sorafs/capacity/state_$(date +%F).json \
    --ledger artifacts/sorafs/capacity/ledger_$(date +%F).ndjson \
    --label nightly-capacity \
    --json-out artifacts/sorafs/capacity/reconcile_$(date +%F).json \
    --prom-out "${SORAFS_CAPACITY_RECONCILE_TEXTFILE:-artifacts/sorafs/capacity/reconcile.prom}"
  ```
  Օգնողը դուրս է գալիս ոչ զրոյականից բացակայող/գերավճարված հաշվարկների կամ տույժերի դեպքում և թողարկում է Prometheus
  տեքստային ֆայլի ամփոփում.
- Զգուշացում `SoraFSCapacityReconciliationMismatch` (`dashboards/alerts/sorafs_capacity_rules.yml`-ում)
  հրդեհներ, երբ հաշտեցման չափիչները հաղորդում են բացթողումներ. վահանակները ապրում են տակ
  `dashboards/grafana/sorafs_capacity_penalties.json`.
- Արխիվացրեք JSON ամփոփագիրը և հեշերը `docs/examples/sorafs_capacity_marketplace_validation/` տակ
  կառավարման փաթեթների կողքին:

### Վեճ և կրճատող ապացույցներ
- Ներկայացրեք վեճերը `sorafs_manifest_stub capacity dispute`-ի միջոցով (թեստեր.
  `cargo test -p sorafs_car --test capacity_cli`), ուստի օգտակար բեռները մնում են կանոնական:

  կտրատում է դետերմինիստականորեն վերարտադրում:
- Հետևեք `docs/source/sorafs/dispute_revocation_runbook.md`-ին ապացույցների հավաքագրման և ընդլայնման համար;

### Մատակարարի միացման և ծխի ելքի թեստեր

1. Ավարտեք SF-2b (ընդունման քաղաքականություն) – շուկան հենվում է ստուգված մատակարարների վրա:
2. Իրականացրեք սխեման + ռեեստրի շերտ (այս փաստաթուղթը) մինչև Torii ինտեգրումը:
3. Ավարտեք հաշվառման խողովակաշարը նախքան վճարումները միացնելը:
4. Վերջնական քայլ. միացնել կառավարման կողմից վերահսկվող վճարների բաշխումը, երբ հաշվառման տվյալները ստուգվեն փուլային փուլում:
