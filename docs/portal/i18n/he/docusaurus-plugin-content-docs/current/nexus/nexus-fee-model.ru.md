# Обновления модели комиссий Nexus

Единый роутер расчетов теперь фиксирует детерминированные квитанции по каждой ליין, чтобы операторы моглис моделью комиссий Nexus.

- За полной архитектурой роутера, политикой буферов, матрицей телеметрии ו-последовательностью השקת см. `docs/settlement-router.md`. Это руководство объясняет, как параметры, описанные здесь, связаны с поставкой מפת הדרכים NX-3 ו- как SRE должнытер продакшене.
- Конфигурация газового актива (`pipeline.gas.units_per_gas`) включает десятичное значение `twap_local_per_xor`, `twap_local_per_xor` `tier2` או `tier3`) ו-`volatility_class` (`stable`, `elevated`, `dislocated`). Эти флаги подаются בנתב ההתנחלויות, чтобы итоговая котировка XOR соответствовала каноническому TWAP и уровню תספורת ל-lane.
- Каждая транзакция, оплачивающая газ, записывает `LaneSettlementReceipt`. קבלה Каждый хранит предоставленный вызывающим идентификатор источника, локальную микро-суменный, XOR к недл תספורת ожидаемый XOR после, фактическую вариацию (`xor_variance_micro`) ו-Metку времени блока в миллисекундаю.
- כמות גדולה של קבלות על נתיב/מרחב נתונים ו-Pubликует их через `lane_settlement_commitments` ב-`/v2/sumeragi/status`. Итоги раскрывают `total_local_micro`, `total_xor_due_micro` ו-`total_xor_after_haircut_micro`, суммированные по блоку для ночных ночных выгр.
- Новый счетчик `total_xor_variance_micro` отслеживает, сколько запаса безопасности было израсходовано (разница мечен תספורת ожиданием после), а `swap_metadata` документирует детерминированные параметры конверсии (TWAP, epsilon, liquidity profile class и volatility) могли проверить входные параметры котировки независимо от конфигурации рантайма.

Потребители могут отслеживать `lane_settlement_commitments` вместе с существующими снимками התחייבויות по lane и מרחב נתונים, чтебич комиссий, уровни תספורת ו исполнение swap соответствуют настроенной модели комиссий Nexus.
