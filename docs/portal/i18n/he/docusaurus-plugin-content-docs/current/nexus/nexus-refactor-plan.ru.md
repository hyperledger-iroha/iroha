# План рефакторинга Sora Nexus Ledger

מפת הדרכים של Sora Nexus לדג'ר ("Iroha 3"). Он отражает текущую структуру репозитория и регрессии, замеченные в учете genesis/WSV, консенсусе I180NT0г0, консенсусе I180NT00г0, смарт-контрактов, שאילתות תמונת מצב, כריכות מארח pointer-ABI ו-Codec Norito. Цель - прийти к согласованной, тестируемой архитектуре, не пытаясь посадить все исправления однемним монтим.

## 0. Направляющие принципы
- Сохранять детерминированное поведение на гетерогенном железе; использовать ускорение только через תכונת דגלים להצטרפות עם תקלות חוזרות.
- Norito - слой сериализации. Любые изменения מצב/סכימה должны включать Norito קידוד/פענוח רכיבי טיסות ודברים הלוך ושוב.
- Конфигурация проходит через `iroha_config` (משתמש -> בפועל -> ברירות מחדל). הפוך את הסביבה אד-הוק לחילופין из продакшн путей.
- פוליטיקה של ABI остается V1 и не подлежит обсуждению. מארחים должны детерминированно отклонять неизвестные סוגי מצביעים/שיחות.
- `cargo test --workspace` ובדיקות זהב (`ivm`, `norito`, `integration_tests`) остаются базовым гейтом для каждогош

## 1. Снимок топологии репозитория
- `crates/iroha_core`: Sumeragi שחקנים, WSV, טען בראשית, צינורות (שאילתה, שכבת על, נתיבי zk), דבק למארח חוזה חכם.
- `crates/iroha_data_model`: авторитетная схема данных и שאילתות.
- `crates/iroha`: לקוח API, используемый CLI, בדיקות, SDK.
- `crates/iroha_cli`: операторский CLI, сейчас зеркалит множество APIs из `iroha`.
- `crates/ivm`: Kotodama bytecode VM, נקודות כניסה לשילוב מארח pointer-ABI.
- `crates/norito`: רכיבי Codec בסידרה עם מתאמי JSON וקצה אחורי של AoS/NCB.
- `integration_tests`: קביעות кросс-компонентные, покрывающие genesis/bootstrap, Sumeragi, טריגרים, עימוד и т.д.
- Docs уже описывают цели Sora Nexus Ledger (`nexus.md`, `new_pipeline.md`, `ivm.md`), но реализареализа частично устарела относительно кода.

## 2. אופציונליות אופטימליות### שלב א' - יסודות וצפייה
1. **טלמטריית WSV + צילומי מצב**
   - Зафиксировать API של תמונת מצב קנונית ב-`state` (תכונה `WorldStateSnapshot`), שאילתות используемый, Sumeragi ו-CLI.
   - Использовать `scripts/iroha_state_dump.sh` לצילומי מצב של детерминированных через `iroha state dump --format norito`.
2. **דטרמיניזם בראשית/הגפיים**
   - Переписать ingest genesis так, чтобы он проходил через единый Norito צינור (`iroha_core::genesis`).
   - Добавить אינטגרציה/רגרסיה покрытие, которое воспроизводит genesis плюс первый блок ו подтверждает идентичные WSV roots arm м64ж8/8 (трекится в `integration_tests/tests/genesis_replay_determinism.rs`).
3. **מבחני קביעות בין ארגזים**
   - Расширить `integration_tests/tests/genesis_json.rs`, чтобы валидировать WSV, pipeline и ABI invariants в одном רתמה.
   - פיגום `cargo xtask check-shape`, סחף סכימת סכימת (трекится в DevEx צבר כלים; פריט פעולה ב-`scripts/xtask/README.md`).

### שלב ב' - WSV ושטח שאילתה
1. **עסקאות אחסון במדינה**
   - סיווג `state/storage_transactions.rs` לתקשורת ארגונית, מתחייבת לצורכי תקשורת ותקשורת תקשורת.
   - בדיקות יחידה теперь проверяют, что модификации assets/world/triggers откатываются при ошибках.
2. **מחזר מודל שאילתה**
   - Перенести עימוד/סמן логику в переиспользуемые компоненты под `crates/iroha_core/src/query/`. Согласовать Norito ייצוגים в `iroha_data_model`.
   - צור שאילתות של תמונת מצב לטריגרים, נכסים ותפקידים с детерминированным порядком (отслеживается через `crates/iroha_core/tests/snapshot_iterable.rs` для текипущя).
3. **עקביות של תמונת מצב**
   - Убедиться, что `iroha ledger query` CLI использует тот же נתיב תמונת מצב, что ו Sumeragi/מחזירים.
   - בדיקות רגרסיה של תמונת מצב של CLI находятся в `tests/cli/state_snapshot.rs` (מגודר תכונות עבור медленных прогонов).

### שלב ג' - צינור Sumeragi
1. **ניהול טופולוגיה ועידן**
   - מאפיינים `EpochRosterProvider` ב-trait с реализациями, основанными על צילומי מצב של WSV.
   - `WsvEpochRosterAdapter::from_peer_iter` предлагает простой constructor, удобный для mock в ספסלים/בדיקות.
2. **פישוט זרימת קונצנזוס**
   - Реорганизовать `crates/iroha_core/src/sumeragi/*` בעיצוב: `pacemaker`, `aggregation`, `availability`, `witness` וס опидид `consensus`.
   - בדוק הודעה אד-הוק שעוברת במעטפות טיפוסיות של Norito ובדוק בדיקות מאפיינים של שינוי תצוגה (טרקטיביות ב-Sumeragi צבר הודעות).
3. **שילוב נתיב/הוכחה**
   - Согласовать הוכחות נתיב עם התחייבויות DA и обеспечить единообразный RBC gating.
   - מקצה לקצה интеграционный тест `integration_tests/tests/extra_functional/seven_peer_consistency.rs` теперь проверяет путь с включенным RBC.### שלב ד' - חוזים חכמים ומארחי Pointer-ABI
1. **ביקורת גבולות מארח**
   - Консолидировать проверки מסוג מצביע (`ivm::pointer_abi`) ומתאמי מארח (`iroha_core::smartcontracts::ivm::host`).
   - Ожидания טבלת מצביעים и כריכות מניפסט מארח покрыты тестами `crates/iroha_core/tests/ivm_pointer_abi_tlv_types.rs` и `ivm_host_mapping.rs`, которые проверяют מיפוי TLV מוזהב.
2. **ארגז חול לביצוע טריגר**
   - הפעלת טריגרים, תקשורת הפעלה אוטומטית של `TriggerExecutor`, גז, אימות מצביע ורישום אירועים.
   - בדוק מבחני רגרסיה לטריגרים של שיחה/זמן с покрытием נתיבי כשל (трекится через `crates/iroha_core/tests/trigger_failure.rs`).
3. **CLI ויישור לקוח**
   - Обеспечить, чтобы CLI операции (`audit`, `gov`, `sumeragi`, `ivm`) использованки לקוח `iroha`, чтобы избежать סחף.
   - בדיקות תמונת מצב של CLI JSON находятся в `tests/cli/json_snapshot.rs`; держите их актуальными, чтобы вывод вывод core команд совпадал с הפניה קנונית JSON.

### שלב E - Norito הקשחת Codec
1. **רישום סכימות**
   - צור רישום סכימת Norito בתקן `crates/norito/src/schema/`, צור קידודים קנוניים לסוגי נתוני ליבה.
   - בדיקות doc Добавлены, проверяющие קידוד מטענים לדוגמה (`norito::schema::SamplePayload`).
2. **רענון מערכות הזהב**
   - אופטימיזציה של גופי זהב ב-`crates/norito/tests/*`, чтобы они соответствовали новой WSV schema после посадки рефакторинга.
   - `scripts/norito_regen.sh` детерминированно регенерирует Norito JSON goldens через helper `norito_regen_goldens`.
3. **IVM/Norito אינטגרציה**
   - Провалидировать сериализацию Kotodama מניפסט מקצה לקצה через Norito, гарантируя консистентность pointer ABI metadata.
   - `crates/ivm/tests/manifest_roundtrip.rs` удерживает Norito קידוד/פענוח паритет для מניפסטים.

## 3. Сквозные вопросы
- **אסטרטגיית בדיקות**: Каждый этап продвигает בדיקות יחידה -> בדיקות ארגז -> בדיקות אינטגרציה. Падающие тесты фиксируют текущие регрессии; новые тесты не дают им повториться.
- **תיעוד**: После посадки каждой фазы обновлять `status.md` ו- переносить незакрытые пункты в Sumeragi, завершенные задачи.
- **מדדי ביצועים**: Сохранять существующие ספסלים в `iroha_core`, `ivm` ו-`norito`; добавлять базовые измерения после рефакторинга, чтобы подтвердить отсутствие регрессий.
- **דגלי תכונה**: Оставлять מחליפים ברמת ארגזים только לקצה אחורי, требующих внешних כלים (`cuda`, `zk-verify-batch`). CPU SIMD пути всегда собираются и выбираются во время выполнения; обеспечить детерминированные נפילות סקלריות ל-неподдерживаемого железа.## 4. Ближайшие действия
- פיגום שלב א' (תכונת תמונת מצב + חיווט טלמטריה) - см. משימות ניתנות לפעולה в обновлениях מפת הדרכים.
- Недавний аудит дефектов по `sumeragi`, `state` ו-`ivm` יוצרו סמלים:
  - `sumeragi`: קצבאות קוד מת מונעות הוכחה לשידור צפייה-שינוי, מצב שידור חוזר של VRF ויצוא טלמטריה של EMA. Они остаются gated, пока не будут выполнены phase C по упрощению קונצנזוס и интеграции נתיב/הוכחה.
  - `state`: `Cell` ניקוי וניתוב טלמטריה פועל במסלול טלמטריה של שלב A של WSV, מכשירי SoA/יישום מקביל לאופטימיזציה של צינור צינורות של שלב C.
  - `ivm`: החלפת CUDA, אימות מעטפת וכיסוי Halo2/Metal маппятся на Phase D-מארח-גבול работу плюс сквозную тему GPU; ליבות остаются в отдельном צבר GPU до готовности.
- צור RFC חוצה צוות עם נקודת פתיחה עבור חתימה перед посадкой инвазивных изменений кода.

## 5. Открытые вопросы
- האם ה-RBC אופציונלי ל-P1, או אופציונלי ל-Nexus נתיבי ספר חשבונות? Требуется решение стейкхолдеров.
- האם יש קבוצות של DS Compositability ב-P1 או לאפשר הוכחות לנתיב?
- Какое каноническое место для параметров ML-DSA-87? Кандидат: новый ארגז `crates/fastpq_isi` (создание ожидается).
