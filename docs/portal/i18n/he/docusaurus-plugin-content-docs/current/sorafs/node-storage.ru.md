## Дизайн хранения узла SoraFS (Черновик)

Эта записка уточняет, как узел Iroha (Torii) может подключиться к слою
זמינות данных SoraFS и выделить часть локального диска для хранения и
обслуживания чанков. Она дополняет Discovery-спецификацию
`sorafs_node_client_protocol.md` и работу по גופי SF-1b, описывая архитектуру
сториджа, ресурсные ограничения и конфигурационную проводку, которые должны
появиться в узле и gateway-коде. Практические операционные процедуры описаны в
[Runbook операций узла](./node-operations).

### שי

- מצא את הציוד הרצוי או את החשבון של Iroha
  свободный диск как провайдер SoraFS ללא שוויון בפנקס החשבונות.
- Держать модуль хранения детерминированным и управляемым Norito: манифесты,
  планы чанков, корни הוכחת אחזור (PoR) и advert провайдеров — это
  источник истины.
- Применять операторские квоты, чтобы узел не исчерпал свои ресурсы из-за
  слишком большого количества запросов סיכה или להביא.
- Отдавать здоровье/телеметрию (דגימת PoR, חביון אחזור чанков, давление на
  диск) обратно в governance и клиентам.

Ключевые модули:

- **שער**: открывает Norito HTTP эндпоинты для предложений סיכה, запросов אחזור
  чанков, דגימת PoR и телеметрии. Валидирует Norito מטענים ו направляет
  запросы בחנות נתחים. Переиспользует существующий HTTP стек Torii, чтобы не
  вводить новый демон.
- **רישום פינים**: состояние pin манифестов в `iroha_data_model::sorafs` и
  `iroha_core`. При принятии манифеста регистр хранит digest манифеста, digest
  плана чанков, корень PoR и флаги возможностей провайдера.
- **אחסון נתחים**: дисковая реализация `ChunkStore`, которая принимает подписанные
  מנופים, חומרי גלם, חומרי גלם, חומרי גלם, רכיבים ותקשורים.
  פריסת чанки в детерминированном. Каждый чанк связан с טביעת אצבע контента и
  PoR метаданными, поэтому דגימה может повторно валидировать без перечитывания
  всего файла.
- **מכסה/מתזמן**: применяет лимиты оператора (макс. байты диска, макс. סיכות
  в очереди, макс. параллельные fetch, TTL чанков) и координирует IO, чтобы задачи
  ספר חשבונות не были вытеснены. מתזמן также отвечает за выдачу PoR proof и запросы
  דגימה с ограниченным CPU.

### Конфигурация

Добавьте новую секцию в `iroha_config`:

```toml
[sorafs.storage]
enabled = false
data_dir = "/var/lib/iroha/sorafs"
max_capacity_bytes = "100 GiB"
max_parallel_fetches = 32
max_pins = 10_000
por_sample_interval_secs = 600
alias = "tenant.alpha"            # опциональный человекочитаемый тег
adverts:
  stake_pointer = "stake.pool.v1:0x1234"
  availability = "hot"
  max_latency_ms = 500
  topics = ["sorafs.sf1.primary:global"]
```- `enabled`: переключатель участия. Когда false, gateway возвращает 503 לאחסון
  эндпоинтов и узел не объявляется בגילוי.
- `data_dir`: корневая директория для чанков, PoR деревьев ו- телеметрии אחזור.
  По умолчанию `<iroha.data_dir>/sorafs`.
- `max_capacity_bytes`: жесткий лимит на pin-данные. Фоновая задача отклоняет
  новые סיכות при достижении лимита.
- `max_parallel_fetches`: лимит конкурентности, накладываемый מתזמן, чтобы
  балансировать сеть/IO диска с нагрузкой валидатора.
- `max_pins`: סיכות מניפסט של максимум, которые принимает узел до применения
  פינוי/לחץ לאחור.
- `por_sample_interval_secs`: עבודות דגימת PoR. Каждый
  job выбирает `N` листьев (настраивается לכל מניפסט) и эмитит телеметрию.
  ממשל может детерминированно масштабировать `N`, задавая ключ
  `profile.sample_multiplier` (целое `1-4`). Значение может быть числом/строкой
  или объектом с override по профилю, например `{"default":2,"sorafs.sf2@1.0.0":3}`.
- `adverts`: структура, используемая генератором מודעה для заполнения полей
  `ProviderAdvertV1` (מצביע הימור, רמזים ל-QoS, נושאים). Если опущено, узел берет
  ברירת המחדל של מרשם הממשל.

Прокладка конфигурации:

- `[sorafs.storage]` אופרה ב-`iroha_config` как `SorafsStorage` и загружается
  из конфигурационного файла узла.
- `iroha_core` ו-`iroha_torii` מספקים תצורת אחסון ב-gateway Builder ו-chunk
  חנות при старте.
- עקיפות פיתוח/בדיקה של מערכת (`SORAFS_STORAGE_*`, `SORAFS_STORAGE_PIN_*`), לא
  в production следует использовать конфиг-файл.

### CLI утилиты

Пока HTTP поверхность Torii еще подключается, ארגז `sorafs_node` поставляет
тонкий CLI, чтобы операторы могли скриптовать מקדחות בליעה/ייצוא против
персистентного backend.【crates/sorafs_node/src/bin/sorafs-node.rs:1】

```bash
cargo run -p sorafs_node --bin sorafs-node ingest \
  --data-dir ./storage/sorafs \
  --manifest ./fixtures/manifest.to \
  --payload ./fixtures/payload.bin \
  --plan-json-out ./plan.json
```

- `ingest` ожидает Norito מניפסט מקודד `.to` כולל בתים של עומס מטען.
  Он восстанавливает план чанков из профиля чанкинга манифеста, проверяет паритет
  digest, сохраняет файлы чанков опционально эмитит JSON `chunk_fetch_specs`, чтобы
  במורד הזרם инструменты могли проверить פריסה.
- `export` מזהה מניפסט מניפסט ופרטי מניפסט/טעינה בדיסק
  (с опциональным plan JSON), чтобы fixtures оставались воспроизводимыми.

Обе команды печатают Norito סיכום JSON ב-stdout, что удобно для скриптов. CLI
покрыт интеграционным тестом, который подтверждает корректный מניפסטים הלוך ושוב
и מטענים до появления Torii API.【crates/sorafs_node/tests/cli.rs:1】> HTTP паритет
>
> שער Torii יכול לעזור לעוזרים לקריאה בלבד.
> `NodeHandle`:
>
> - `GET /v2/sorafs/storage/manifest/{manifest_id_hex}` — возвращает сохраненный
> מניפסט Norito (base64) вместе с digest/metadata.【crates/iroha_torii/src/sorafs/api.rs:1207】
> - `GET /v2/sorafs/storage/plan/{manifest_id_hex}` — возвращает детерминированный
> JSON план чанков (`chunk_fetch_specs`) עבור כלי עבודה במורד הזרם.【crates/iroha_torii/src/sorafs/api.rs:1259】
>
> Эти эндпоинты повторяют CLI вывод, поэтому пайплайны могут перейти от локальных
> скриптов к בדיקות HTTP без смены парсеров.【crates/iroha_torii/src/sorafs/api.rs:1207】【crates/iroha_torii/src/sorafs/api.rs:1259】

### Жизненный цикл узла

1. **אתחול**:
   - При включенном storage узел инициализирует chunk store с заданной директорией
     и емкостью. Это включает проверку/создание PoR базы манифестов и סיכה חוזרת
     манифестов для прогрева кэшей.
   - Зарегистрировать маршруты שער SoraFS (Norito JSON POST/GET эндпоинты ל-pin,
     להביא, דגימת PoR, телеметрии).
   - Запустить PoR sampling worker и монитор квот.
2. **גילוי / פרסומות**:
   - Сформировать `ProviderAdvertV1` основе текущей емкости/здоровья, подписать
     ключом, одобренным советом, и опубликовать через Discovery канал.
     оставались доступными.
3. **זרימת עבודה בהצמדה**:
   - Gateway получает подписанный מניפסט (с планом чанков, корнем PoR и подписями
     совета). Валидирует список כינוי (`sorafs.sf1@1.0.0` обязателен) и убеждается,
     что план чанков соответствует метаданным манифеста.
   - Проверяет квоты. קיבולת רווח/פין лимитов отвечает политикой ошибки
     (структурированный Norito).
   - Стримит chunk данные в `ChunkStore`, проверяя digests на ingest. Обновляет PoR
     деревья и хранит metadata манифеста в registry.
4. **זרימת עבודה של אחזור**:
   - Отдает запросы טווח чанков с диска. מתזמן применяет `max_parallel_fetches`
     и возвращает `429` при насыщении.
   - Эмитит структурированную телеметрию (Norito JSON) עם חביון, בתים שהוגשו и
     счетчиками ошибок для במורד הזרם мониторинга.
5. **דגימת PoR**:
   - Worker выбирает манифесты пропорционально весу (например, байтам хранения) и
     запускает детерминированный דגימה через PoR дерево chunk store.
   - Сохраняет результаты для ממשל аудита и включает сводки в ספק פרסומות
     / נקודת קצה של טל.
6. **פינוי / קבצים**:
   - При достижении емкости узел по умолчанию отклоняет новые סיכות. Опционально
     операторы смогут настроить политики פינוי (например, TTL, LRU) после
     согласования модели ממשל; пока дизайн предполагает строгие квоты и
     unpin операции, инициируемые оператором.

### Интеграция деклараций емкости и תזמון- Torii צור קשר עם `CapacityDeclarationRecord` או `/v2/sorafs/capacity/declare`
  во встроенный `CapacityManager`, так что каждый узел строит בזיכרון представление своих
  зафиксированных chunker/lane аллокаций. Менеджер публикует צילומי מצב לקריאה בלבד לטלמטרים
  (`GET /v2/sorafs/capacity/state`) и применяет резервы per-profile/per-lane до принятия новых
  заказов.【crates/sorafs_node/src/capacity.rs:1】【crates/sorafs_node/src/lib.rs:60】
- Эндпоинт `/v2/sorafs/capacity/schedule` принимает מינהל שהונפקו `ReplicationOrderV1`
  מטענים. Когда заказ нацелен на локального провайдера, менеджер проверяет дублирование
  расписаний, валидирует емкость chunker/lane, резервирует слот и возвращает `ReplicationPlan`
  с описанием оставшейся емкости, чтобы оркестрация могла продолжить בליעה. Заказы для
  других провайдеров подтверждаются ответом `ignored`, упрощая זרימות עבודה מרובות מפעילים.【crates/iroha_torii/src/routing.rs:484
- ווי השלמה (например, после успешной ingest) вызывают
  `POST /v2/sorafs/capacity/complete` עבור освобождения резерва через
  `CapacityManager::complete_order`. Ответ включает תמונת מצב `ReplicationRelease`
  (סכומים остаточные, остатки chunker/lane), чтобы כלי תזמור могло
  ставить следующий заказ без הסקרים. Дальнейшая работа подключит это к pipeline
  חנות נתחים, как только בליעה логика появится.【crates/iroha_torii/src/routing.rs:4885】【crates/sorafs_node/src/capacity.rs:90】
- Встроенный `TelemetryAccumulator` можно обновлять через `NodeHandle::update_telemetry`,
  что позволяет עובדי רקע фиксировать דגימות PoR/uptime и со временем
  выводить канонические `CapacityTelemetryV1` מטענים ללא תקלות
  частей מתזמן.【crates/sorafs_node/src/lib.rs:142】【crates/sorafs_node/src/telemetry.rs:1】

### Интеграции и будущая работа

- **ממשל**: расширить `sorafs_pin_registry_tracker.md` телеметрией хранения
  (שיעור הצלחה PoR, ניצול דיסק). Политики допуска могут требовать минимальную
  емкость или минимальный שיעור ההצלחה של PoR до принятия פרסומות.
- **ערכות פיתוח של לקוחות**: раскрыть новую תצורת אחסון (лимиты диска, כינוי), чтобы
  כלי ניהול могло bootstrapper узлы программно.
- **טלמטריה**: интегрировать с существующим ערימת מדדים (Prometheus /
  OpenTelemetry), чтобы метрики хранения были видимы в לוחות מחוונים של צפייה.
- **אבטחה**: запускать модуль хранения в отдельном מאגר משימות אסינכרון с לחץ אחורי
  и рассмотреть ארגז חול чтения чанков через io_uring או ограниченные בריכות טוקיו,
  чтобы злоумышленники не исчерпали ресурсы.

Этот дизайн сохраняет модуль хранения опциональным и детерминированным, давая
операторам нужные כפתורים для участия в слое доступности данных SoraFS. Реализация
потребует измений ב-`iroha_config`, `iroha_core`, `iroha_torii` ו-Norito שער,
а также כלי עבודה לפרסום ספק.
