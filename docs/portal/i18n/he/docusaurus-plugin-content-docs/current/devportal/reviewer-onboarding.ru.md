# תצוגה מקדימה של Онбординг ревьюеров

## תיאור

DOCS-SORA отслеживает поэтапный запуск портала разработчиков. Сборки с gate по checksum
(`npm run serve`) и усиленные потоки נסה את זה открывают следующий этап: онбординг проверенных
ревьюеров до широкого открытия תצוגה מקדימה ציבורית. Этот гид описывает, как собирать заявки,
проверять соответствие, выдавать доступ и безопасно завершать участие. См.
[תצוגה מקדימה של זרימת הזמנה](./preview-invite-flow.md) для планирования когорт, каденции приглашений
и экспорта телеметрии; шаги ниже фокусируются на действиях после выбора ревьюера.

- **В рамках:** ревьюеры, которым нужен доступ к מסמכי תצוגה מקדימה (`docs-preview.sora`,
  сборки GitHub Pages или бандлы SoraFS) עד GA.
- **אין אפשרות:** אופציות Torii או SoraFS (покрываются собственными onboarding-китами) и
  продакшн-развертывания портала (см.
  [`devportal/deploy-guide`](./deploy-guide.md)).

## Роли и требования

| Роль | Типичные цели | Требуемые артефакты | Примечания |
| --- | --- | --- | --- |
| מתחזק ליבה | Проверить новые гайды, запустить בדיקות עשן. | ידית GitHub, קשר ב-Matrix, подписанная CLA. | Обычно уже в команде GitHub `docs-preview`; все равно подайте заявку, чтобы доступ был аудируем. |
| סוקר שותף | בדוק את SDK-сниппеты או קונטנטים עבור управлению до публичного релиза. | דוא"ל Корпоративный, POC חוקי, תנאי תצוגה מקדימה של подписанные. | Должен подтвердить требования по телеметрии и обработке данных. |
| מתנדב קהילתי | Дать משוב по удобству использования гайдов. | ידית GitHub, предпочитаемый контакт, часовой пояс, согласие с CoC. | Держите когорты небольшими; приоритет ревьюерам, подписавшим הסכם תורם. |

Все типы ревьюеров должны:

1. Подтвердить политику допустимого использования preview-артефактов.
2. Прочитать приложения по אבטחה/צפיות
   ([`security-hardening`](./security-hardening.md),
   [`observability`](./observability.md),
   [`incident-runbooks`](./incident-runbooks.md)).
3. Согласиться запускать `docs/portal/scripts/preview_verify.sh` перед тем, как
   обслуживать любой локальный תמונת מצב.

## צריכת מספר

1. Попросите заявителя заполнить
   [`docs/examples/docs_preview_request_template.md`](../../../examples/docs_preview_request_template.md)
   פורמו (אולי ראה את הגיליון). Зафиксируйте как минимум: личность, способ связи,
   ידית GitHub, שרתים לשירותים ושירותי ביטחון.
2. Зафиксируйте заявку в трекере `docs-preview` (בעיית GitHub או תיקון управления)
   и назначьте מאשר.
3. Проверьте требования:
   - CLA / соглашение контрибьютора в наличии (או ссылка на партнерский контракт).
   - Подтверждение допустимого использования хранится в заявке.
   - Риск-оценка завершена (например, партнерские ревьюеры одобрены משפטי).
4. המאשר подписывает заявку и связывает בעיית מעקב с любой записью
   ניהול שינויים (דוגמה: `DOCS-SORA-Preview-####`).

## אספקה и инструменты

1. **Передать артефакты** — Предоставьте последний מתאר תצוגה מקדימה + ארכיון из
   זרימת עבודה של CI או פין SoraFS (פרטי `docs-portal-preview`). Напомните ревьюерам запустить:

   ```bash
   ./docs/portal/scripts/preview_verify.sh \
     --build-dir build \
     --descriptor artifacts/preview-descriptor.json \
     --archive artifacts/preview-site.tar.gz
   ```2. **Serve с Checksum** — Укажите команду с gate по Checksum:

   ```bash
   DOCS_RELEASE_TAG=preview-<stamp> npm run --prefix docs/portal serve
   ```

   Это переиспользует `scripts/serve-verified-preview.mjs`, чтобы неподтвержденный build
   не запускался случайно.

3. ** Дать доступ к GitHub (опционально)** — Если нужны неопубликованные ветки, добавьте
   ревьюеров в команду GitHub `docs-preview` ב-Perioд ревью и зафиксируйте изменение членства
   в заявке.

4. **Коммуникация каналов поддержки** — צור קשר כוננות (מטריקס/סלאק) וכן
   процедурой инцидентов из [`incident-runbooks`](./incident-runbooks.md).

5. **Телеметрия + משוב** — Напомните, что собирается анонимизированная аналитика
   (см. [`observability`](./observability.md)). צור משוב בצורת משוב או בעיה-שבלון,
   указанный в приглашении, и зафиксируйте событие через helper
   [`preview-feedback-log`](./preview-feedback-log), чтобы сводка волны оставалась актуальной.

## Чеклист ревьюера

Перед доступом к תצוגה מקדימה ревьюеры должны выполнить:

1. Проверить скачанные артефакты (`preview_verify.sh`).
2. Запустить портал через `npm run serve` (או `serve:verified`), чтобы guard checksum был активен.
3. Прочитать упомянутые выше אבטחה и צפיות заметки.
4. נסה את OAuth/נסה את זה באמצעות התחברות לקוד מכשיר (если применимо) и не переиспользовать
   ייצור токены.
.
   תצוגה מקדימה של их релизным тегом.

## Ответственности мейнтейнеров и offboard

| Фаза | Действия |
| --- | --- |
| בעיטה | Убедиться, что intake чеклист приложен к заявке, поделиться артефактами + инструкциями, добавить за18030X [`preview-feedback-log`](./preview-feedback-log), וסנכרון מסודר, חוץ מזה. |
| ניטור | Отслеживать תצוגה מקדימה телеметрию (необычный трафик נסה את זה, сбои probe) ו следовать инцидент-ранбуку при подозрениях. Логировать события `feedback-submitted`/`issue-opened` по мере поступления находок, чтобы метрики волниты бычы. |
| יציאה למטוס | Отозвать временный доступ GitHub или SoraFS, ראה `access-revoked`, архивировать заявку (תקציר + משוב действия), и обновить реестр ревьюеров. Попросить ревьюера удалить локальные сборки и приложить digest из [`docs/examples/docs_preview_feedback_digest.md`](../../../examples/docs_preview_feedback_digest.md). |

Используйте тот же процесс при ротации ревьюеров между волнами. Сохранение следов в репозитории
(בעיה + תקלות) помогает DOCS-SORA оставаться аудируемым и позволяет governance подтвердить,
что доступ к תצוגה מקדימה следовал документированным контролям.

## Шаблоны приглашений и трекинг- Начинайте каждое обращение с файла
  [`docs/examples/docs_preview_invite_template.md`](../../../examples/docs_preview_invite_template.md).
  Он фиксирует минимальный юридический язык, инструкции по checksum preview и ожидание,
  что ревьюеры признают политику допустимого использования.
- При редактировании шаблона замените плейсхолдеры `<preview_tag>`, `<request_ticket>` ו- каналы связи.
  Сохраните копию финального сообщения в intake тикете, чтобы ревьюеры, approver'ы аудиторы
  могли сослаться на точный текст.
- צור קשר עם גיליון אלקטרוני למעקב או בעיה עם חותמת זמן `invite_sent_at`
  и ожидаемой датой завершения, чтобы отчет
  [זרימת הזמנת תצוגה מקדימה](./preview-invite-flow.md) автоматически подхватил когорту.
