The roadmap calls for verifiable evidence whenever we invite reviewers into the
docs portal preview. This note explains how to capture that evidence with the
`preview-feedback-log` helper, how to share the resulting artefacts with
governance, and where to file digests once a wave closes.

## Why we log feedback

- **Compliance:** every invite must record who accessed the preview, when they
  acknowledged the acceptable-use policy, and when their access was revoked.
- **Telemetry correlation:** the log lets us correlate portal telemetry spikes
  or Try it traffic with specific invite cohorts.
- **Governance evidence:** each DOCS-SORA checkpoint needs a machine-readable
  summary showing invite counts, open reviewers, and the state of outstanding
  issues.

All evidence lives under `artifacts/docs_portal_preview/` so it can be attached
to the invite wave ticket or governance review without scraping CI logs.

## Live invite summary

`docs/portal/scripts/preview-feedback-summary.mjs` ingests the shared log and
writes `src/data/previewFeedbackSummary.json` during `npm run build`. The table
below renders that JSON so reviewers can see the current invite footprint
without running the CLI helper.

In addition to the invite totals, the summary now surfaces acknowledgement and
revocation counts, the derived response rate (feedback submissions divided by
total invites), and the average latency (with a hoverable p95) between an
invite and the reviewer’s acknowledgement or feedback submission. The new
**Completion %** column shows how many invites have been closed (acknowledged or
revoked), while **Ack %** highlights how many recipients explicitly confirmed
the acceptable-use steps. These derived percentages—plus the JSON-only
revocation/issue rates—feed the governance dashboards so reviewers can confirm
both participation levels and turnaround time without running the CLI helper.
The **Status** column highlights whether a wave is still
active (`active`), closed but within the 14-day monitoring window (`closed`), or
ready for archival (`archived`). The **Days since last event** metric shows how
long it has been since the final invite/feedback action so councils can spot
waves that need to be wrapped up or re-engage lingering recipients.

## Event taxonomy

| Event | When to record it | Notes |
| --- | --- | --- |
| `invite-sent` | Immediately after sending an invite email. | Append one entry per recipient so invite counts stay accurate. |
| `acknowledged` | When the reviewer confirms they read the security/observability docs and ran the checksum gate. | Optional but recommended for partner cohorts. |
| `feedback-submitted` | When a reviewer files a finding via the agreed tracker. | Include the issue/link in the `--notes` field. |
| `issue-opened` | When DOCS-SORA files a follow-up issue derived from feedback. | Helps governance correlate roadmap items to preview feedback. |
| `access-revoked` | Once the reviewer’s preview window ends or access is revoked early. | Required so the “open recipients” count stays accurate. |

If you need an additional event, add it to
`VALID_EVENTS` inside `docs/portal/scripts/preview-feedback-log.mjs` and update
this table in the same change.

## Logging workflow

The helper behind `npm run preview:log` writes a single JSON log that captures
all events. Each entry contains the wave label, recipient, event, timestamp, and
optional notes. Example:

```json
{
  "version": 1,
  "generated_at": "2026-02-04T18:20:15.321Z",
  "entries": [
    {
      "wave": "preview-20260204",
      "recipient": "alice@example.com",
      "event": "invite-sent",
      "timestamp": "2026-02-04T18:05:02.114Z",
      "notes": "seed reviewer"
    },
    {
      "wave": "preview-20260204",
      "recipient": "alice@example.com",
      "event": "feedback-submitted",
      "timestamp": "2026-02-05T09:14:43.009Z",
      "notes": "docs-preview/42"
    }
  ]
}
```

Commands:

```bash
# Append a new event (defaults to artifacts/docs_portal_preview/feedback_log.json)
npm run --prefix docs/portal preview:log -- \
  --wave preview-20260204 \
  --recipient alice@example.com \
  --event invite-sent \
  --notes "seed reviewer"

# Produce a text summary for the current repo
npm run --prefix docs/portal preview:log -- --summary

# Emit machine-readable summary for a specific wave
npm run --prefix docs/portal preview:log -- \
  --summary --summary-json --filter-wave preview-20260204 \
  > artifacts/docs_portal_preview/preview-20260204-summary.json

# Regenerate the portal summary data file
npm run --prefix docs/portal generate:preview-summary

# Generate a Markdown digest skeleton for a wave
npm run --prefix docs/portal preview:digest -- \
  --wave preview-20260204 \
  --invite-start 2026-02-04T00:00:00Z \
  --invite-end 2026-02-18T00:00:00Z \
  --dashboard "https://grafana.example/d/docs-preview?var-wave=preview-20260204" \
  --out artifacts/docs_portal_preview/preview-20260204-digest.md
```

Tips:

- Place the log under version control when working in a feature branch so other
  maintainers can review changes. For long-running waves store the log in the
  governance ticket and attach the final version to the release artefacts.
- `preview:log` accepts `--log <path>` when you need to log directly inside a
  SoraFS bundle or CI workspace.
- Always include a `--notes` hint (issue link, GitHub handle, cohort ID) so the
  summary is useful during the retrospective.
- When you need deterministic summary timestamps (for CI or docs snapshots),
  set `PREVIEW_FEEDBACK_NOW=<iso8601 timestamp>` before running the summary
  helper. This keeps the `generatedAt`, status, and day counters consistent
  across rebuilds.

## Summaries, digests, and archives

1. **Per-wave summary:** generate the JSON summary and attach it to the invite
   ticket together with the raw log. The summary enumerates total invites, open
   recipients, and feedback counts so governance can confirm we captured every
   reviewer. The portal build also publishes an aggregate summary at
   `src/data/previewFeedbackSummary.json` so stakeholders can see the current
   snapshot without running the CLI.
2. **Weekly digest:** run `npm run --prefix docs/portal preview:digest -- --wave <label>`
   to pre-populate the Markdown metrics, then tailor the placeholder sections
   using the template in
   [`docs/examples/docs_preview_feedback_digest.md`](../../../examples/docs_preview_feedback_digest.md)
   before sharing it with stakeholders. Include at least: wave label, open
   reviewers, notable findings, and next actions.
3. **Archival:** move closed-wave logs into
   `artifacts/docs_portal_preview/archive/<wave>/` and keep only the most recent
   active wave at the root. This keeps CI artefacts small while preserving the
   audit trail.
