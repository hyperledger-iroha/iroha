Күйі: басқаруды жүзеге асыру тапсырмаларын орындауға арналған жоба/эскиз. Іске асыру кезінде пішіндер өзгеруі мүмкін. Детерминизм және RBAC саясаты нормативтік шектеулер болып табылады; Torii `authority` және `private_key` берілгенде транзакцияларға қол қоя алады/жібере алады, әйтпесе клиенттер `/transaction`-ке құрастырады және жібере алады.

Шолу
- Барлық соңғы нүктелер JSON қайтарады. Транзакция жасайтын ағындар үшін жауаптар `tx_instructions` — бір немесе бірнеше нұсқау қаңқасының массивін қамтиды:
  - `wire_id`: нұсқау түріне арналған тізілім идентификаторы
  - `payload_hex`: Norito пайдалы жүктеме байты (он алтылық)
- Егер `authority` және `private_key` берілсе (немесе дауыс беру бюллетенінде `private_key`), Torii қол қойып, транзакцияға жібереді және `tx_instructions` қайтарады.
- Әйтпесе, клиенттер өздерінің өкілеттіктерін және chain_id идентификаторын пайдаланып SignedTransaction жинайды, содан кейін `/transaction` мекенжайына қол қояды және POST.
- SDK қамтуы:
- Python (`iroha_python`): `ToriiClient.get_governance_proposal_typed` қайтарады `GovernanceProposalResult` (күйді/түрді өрістерді қалыпқа келтіру), `ToriiClient.get_governance_referendum_typed` қайтарады `GovernanceReferendumResult`, `GovernanceReferendumResult`, I18NI0000000004014018 `ToriiClient.get_governance_locks_typed` `GovernanceLocksResult` қайтарады, `ToriiClient.get_governance_unlock_stats_typed` `GovernanceUnlockStats` қайтарады және `ToriiClient.list_governance_instances_typed` `GovernanceInstancesPage` қайтарады, мысалдар арқылы бізге терілген қол жеткізуді қамтамасыз етеді.
- Python жеңіл клиенті (`iroha_torii_client`): `ToriiClient.finalize_referendum` және `ToriiClient.enact_proposal` терілген `GovernanceInstructionDraft` бумаларын қайтарады (Torii Torii қаңқасын орау, I18NI50000 қолмен талдаудан аулақ болу), Ағындарды аяқтау/қолдану.
- JavaScript (`@iroha/iroha-js`): `ToriiClient` ұсыныстар, референдумдар, есептер, құлыптар, құлыпты ашу үшін терілген көмекшілерді көрсетеді, енді `listGovernanceInstances(namespace, options)` және кеңестің соңғы нүктелері (I18NI0000005900X, I18NI00000059000 `governancePersistCouncil`, `getGovernanceCouncilAudit`) сондықтан Node.js клиенттері `/v2/gov/instances/{ns}` беттерін белгілей алады және бар келісім-шарт данасы тізімімен қатар VRF қолдайтын жұмыс процестерін жүргізе алады.

Соңғы нүктелер

- POST `/v2/gov/proposals/deploy-contract`
  - Сұраныс (JSON):
    {
      "аттар кеңістігі": "қолданбалар",
      "contract_id": "my.contract.v1",
      "code_hash": "blake2b32:..." | "...64гекс",
      "abi_hash": "blake2b32:..." | "...64гекс",
      "abi_version": "1",
      "терезе": { "төменгі": 12345, "жоғарғы": 12400},
      "authority": "i105...?",
      "private_key": "...?"
    }
  - Жауап (JSON):
    { "жарайды": шын, "ұсыныс_идентификаторы": "...64hex", "tx_instructions": [{ "wire_id": "...", "пайдалы_алтылық": "..." }] }
  - Тексеру: түйіндер `abi_hash` берілген `abi_version` үшін канонизацияланады және сәйкессіздіктерді қабылдамайды. `abi_version = "v1"` үшін күтілетін мән `hex::encode(ivm::syscalls::compute_abi_hash(ivm::SyscallPolicy::AbiV1))`.

Contracts API (орналастыру)
- POST `/v2/contracts/deploy`
  - Сұраныс: { "authority": "i105...", "private_key": "...", "code_b64": "..." }
  - Мінез: `code_hash` IVM бағдарламасының денесінен және `abi_hash` `abi_version` тақырыбынан есептейді, содан кейін `RegisterSmartContractCode` (манифест) және I10040l жібереді `.to` байт) `authority` атынан.
  - Жауап: { "жарайды": шын, "код_хэш_хекс": "...", "аби_хэш_хекс": "..." }
  - Қатысты:
    - GET `/v2/contracts/code/{code_hash}` → сақталған манифестті қайтарады
    - `/v2/contracts/code-bytes/{code_hash}` алу → `{ code_b64 }` қайтарады
- POST `/v2/contracts/instance`
  - Сұрау: { "авторитет": "i105...", "жеке_кілт": "...", "аттар кеңістігі": "қолданбалар", "контракт_идентификаторы": "calc.v1", "code_b64": "..." }
  - Мінез-құлық: берілген байткодты орналастырады және `(namespace, contract_id)` салыстыруды `ActivateContractInstance` арқылы дереу іске қосады.
  - Жауап: { "жарайды": шын, "аттар кеңістігі": "қолданбалар", "контракт_идентификаторы": "calc.v1", "code_hash_hex": "...", "abi_hash_hex": "..." }

Бүркеншік ат қызметі
- POST `/v2/aliases/voprf/evaluate`
  - Сұраныс: { "blinded_element_hex": "..." }
  - Жауап: { "бағаланған_элемент_hex": "...128hex", "backend": "blake2b512-mock" }
    - `backend` бағалаушының орындалуын көрсетеді. Ағымдағы мән: `blake2b512-mock`.
  - Ескертпелер: `iroha.alias.voprf.mock.v1` домен бөлуімен Blake2b512 қолданбасын қолданатын детерминистік жалған бағалаушы. Өндірістік VOPRF құбыры Iroha арқылы өткізілгенге дейін сынақ құралдарын қолдануға арналған.
  - Қателер: дұрыс емес он алтылық енгізудегі HTTP `400`. Torii дешифратор қате туралы хабары бар Norito `ValidationFail::QueryFailed::Conversion` конвертін қайтарады.
- POST `/v2/aliases/resolve`
  - Сұрау: { "бүркеншік ат": "GB82 WEST 1234 5698 7654 32" }
  - Жауап: { "бүркеншік ат": "GB82WEST12345698765432", "account_id": "i105...", "индекс": 0, "көз": "iso_bridge" }
  - Ескертпелер: ISO көпірінің орындалу уақытының кезеңдерін қажет етеді (`[iso_bridge.account_aliases]` `iroha_config`). Torii іздеу алдында бос орын мен жоғарғы регистрді алып тастау арқылы бүркеншік аттарды қалыпқа келтіреді. Бүркеншік ат болмаған кезде 404 мәнін және ISO көпірінің орындалу уақыты өшірілгенде 503 мәнін қайтарады.
- POST `/v2/aliases/resolve_index`
  - Сұраныс: { "индекс": 0 }
  - Жауап: { "индекс": 0, "бүркеншік ат": "GB82WEST12345698765432", "account_id": "i105...", "көз": "iso_bridge" }
  - Ескертпелер: Бүркеншік ат индекстері конфигурация тәртібінен (0 негізінде) детерминирленген түрде тағайындалады. Клиенттер бүркеншік атпен растау оқиғалары үшін аудит жолдарын құру үшін жауаптарды желіден тыс кэштей алады.

Код өлшемінің қақпағы
- Теңшелетін параметр: `max_contract_code_bytes` (JSON u64)
  - Тізбектегі келісімшарт кодын сақтау үшін рұқсат етілген ең үлкен өлшемді (байтпен) басқарады.
  - Әдепкі: 16 МБ. `.to` кескін ұзындығы өзгермейтін бұзушылық қатесі бар шекті шектен асқанда түйіндер `RegisterSmartContractBytes` қабылдамайды.
  - Операторлар `SetParameter(Custom)` `id = "max_contract_code_bytes"` және сандық пайдалы жүктемені жіберу арқылы реттей алады.

- POST `/v2/gov/ballots/zk`
  - Сұраныс: { "authority": "i105...", "private_key": "...?", "chain_id": "...", "select_id": "e1", "proof_b64": "...", "public": {…} }
  - Жауап: { "жарайды": шын, "қабылданды": шын, "tx_instructions": [{…}] }
  - Ескертулер:
    - Сұлбаның жалпыға қолжетімді кірістері `owner`, `amount` және `duration_blocks` және дәлел конфигурацияланған VK-ге қарсы тексерілгенде, түйін I18NI0000010101010101010101010000000010101 бар басқару құлпын жасайды немесе кеңейтеді. Бағыт жасырын қалады (`unknown`); тек сома/аяқтау мерзімі жаңартылады. Қайта дауыс беру монотонды: сома мен жарамдылық мерзімі тек қана артады (түйін max(сома, алдыңғы сома) және max(жарамдылық мерзімі, алдыңғы.сома) қолданылады).
    - соманы қысқартуға немесе жарамдылық мерзімі аяқталуға әрекеттенетін ZK қайта дауыстары `BallotRejected` диагностикасымен сервер тарапынан қабылданбайды.
    - Келісімшартты орындау `SubmitBallot` кезекке қоймас бұрын `ZK_VOTE_VERIFY_BALLOT` шақыруы керек; хосттар бір реттік ысырманы қолданады.

- POST `/v2/gov/ballots/plain`
  - Сұраныс: { "авторитет": "i105...", "жеке_кілт": "...?", "тізбегі_идентификаторы": "...", "референдум_идентификаторы": "r1", "иеленуші": "i105...", "сома": "1000", "ұзақтығы_блоктар": 6000, "қабылдау|Name"
  - Жауап: { "жарайды": шын, "қабылданды": шын, "tx_instructions": [{…}] }
  - Ескертулер: қайта дауыс беру тек ұзартуға арналған — жаңа бюллетень бар құлыптың мөлшерін немесе жарамдылық мерзімін азайта алмайды. `owner` транзакция өкілеттігіне тең болуы керек. Ең аз ұзақтығы - `conviction_step_blocks`.

- POST `/v2/gov/finalize`
  - Сұраныс: { "referendum_id": "r1", "proposal_id": "…64hex", "authority": "i105…?", "private_key": "...?" }
  - Жауап: { "жарайды": ақиқат, "tx_instructions": [{ "wire_id": "...Референдумды аяқтау", "пайдалы_алтылық": "..." }] }
  - Тізбектік әсер (ағымдағы тірек): мақұлданған орналастыру ұсынысын енгізу күтілетін `abi_hash` бар `code_hash` арқылы кілттелген ең аз `ContractManifest` енгізеді және ұсынысты Күшті деп белгілейді. Басқа `abi_hash` бар `code_hash` үшін манифест әлдеқашан бар болса, қабылдау қабылданбайды.
  - Ескертулер:
    - ZK сайлауы үшін келісім-шарт жолдары `FinalizeElection` орындалмас бұрын `ZK_VOTE_VERIFY_TALLY` нөміріне қоңырау шалуы керек; хосттар бір реттік ысырманы қолданады. `FinalizeReferendum` сайлау саны аяқталғанша ZK референдумынан бас тартады.
    - `h_end` кезінде автоматты жабу тек Қарапайым референдум үшін мақұлданды/қабылданбады; ZK референдумы түпкілікті есеп берілгенге және `FinalizeReferendum` орындалғанға дейін жабық күйінде қалады.
    - Қатысушылардың қатысуын тексеру тек мақұлдау+қабылдамауды пайдаланады; қалыс қалу дауыс беруге есептелмейді.

- POST `/v2/gov/enact`
  - Сұрау: { "proposal_id": "...64hex", "preimage_hash": "…64hex?", "window": { "төменгі": 0, "жоғарғы": 0 }?, "авторитет": "i105…?", "private_key": "...?" }
  - Жауап: { "жарайды": ақиқат, "tx_instructions": [{ "wire_id": "...EnactReferendum", "пайдалы жүктеме_hex": "..." }] }
  - Ескертулер: Torii қол қойылған транзакцияны `authority`/`private_key` берілгенде жібереді; әйтпесе ол клиенттердің қол қоюы және жіберуі үшін қаңқаны қайтарады. Алдын ала сурет міндетті емес және қазіргі уақытта ақпараттық болып табылады.

- `/v2/gov/proposals/{id}` АЛУ
  - `{id}` жолы: ұсыныс идентификаторы он алтылық (64 таңба)
  - Жауап: { "табылды": bool, "ұсыныс": { … }? }

- `/v2/gov/locks/{rid}` АЛУ
  - `{rid}` жолы: референдум идентификаторы жолы
  - Жауап: { "табылды": bool, "referendum_id": "rid", "құлыптау": { … }? }

- `/v2/gov/council/current` АЛУ
  - Жауап: { "дәуір": N, "мүшелер": [{ "account_id": "..." }, …] }
  - Ескертулер: бар болған кезде тұрақты кеңесті қайтарады; әйтпесе конфигурацияланған үлес активі мен шекті мәндерді пайдаланып детерминирленген қалпына келтіруді алады (тізбекте тірі VRF дәлелдері сақталғанша VRF спецификациясын көрсетеді).- POST `/v2/gov/council/derive-vrf` (мүмкіндік: gov_vrf)
  - Сұраныс: { "committee_size": 21, "дәуір": 123? , "кандидаттар": [{ "account_id": "...", "variant": "Қалыпты|Шағын", "pk_b64": "...", "proof_b64": "..." }, …] }
  - Мінез-құлық: әрбір үміткердің VRF дәлелін `chain_id`, `epoch` және соңғы блок хэш-маякынан алынған канондық кіріске қарсы тексереді; тайбрейкерлермен шығыс байттары бойынша сұрыптау; жоғарғы `committee_size` мүшелерін қайтарады. Тұрмайды.
  - Жауап: { "дәуір": N, "мүшелер": [{ "аккаунт_идентификаторы": "…" } …], "жалпы_кандидаттар": M, "тексерілген": K }
  - Ескертулер: G1-де қалыпты = pk, G2-де дәлелдеу (96 байт). G2-де кіші = pk, G1-де дәлелдеу (48 байт). Енгізулер доменге бөлінген және `chain_id` қамтиды.

### Басқару әдепкілері (iroha_config `gov.*`)

Ешқандай тұрақты тізім болмаған кезде Torii пайдаланатын кеңес резерві `iroha_config` арқылы параметрленген:

```toml
[gov]
  vk_ballot.backend = "halo2/ipa"
  vk_ballot.name    = "ballot_v1"
  vk_tally.backend  = "halo2/ipa"
  vk_tally.name     = "tally_v1"
  plain_voting_enabled = false
  conviction_step_blocks = 100
  max_conviction = 6
  approval_q_num = 1
  approval_q_den = 2
  min_turnout = 0
  parliament_committee_size = 21
  parliament_term_blocks = 43200
  parliament_min_stake = 1
  parliament_eligibility_asset_id = "SORA#stake"
```

Эквивалентті ортаны қайта анықтау:

```
GOV_VK_BACKEND=halo2/ipa
GOV_VK_NAME=ballot_v1
GOV_PARLIAMENT_COMMITTEE_SIZE=21
GOV_PARLIAMENT_TERM_BLOCKS=43200
GOV_PARLIAMENT_MIN_STAKE=1
GOV_PARLIAMENT_ELIGIBILITY_ASSET_ID=SORA#stake
GOV_ALIAS_TEU_MINIMUM=0
GOV_ALIAS_FRONTIER_TELEMETRY=true
```

`parliament_committee_size` кеңес сақталмаған кезде қайтарылған резервтік мүшелердің санын шектейді, `parliament_term_blocks` тұқым шығару үшін пайдаланылатын дәуір ұзақтығын анықтайды (`epoch = floor(height / term_blocks)`), `parliament_min_stake` (ең аз және ең аз өлшем бірліктерін талап етеді) `parliament_eligibility_asset_id` үміткерлер жинағын құру кезінде қандай актив балансы сканерленетінін таңдайды.

Governance VK тексеруінің айналып өту жолы жоқ: бюллетеньді тексеру әрқашан кірістірілген байттары бар `Active` растау кілтін қажет етеді және орталар тексеруді өткізіп жіберу үшін тек сынақ қосқыштарына сенбеуі керек.

RBAC
- Тізбек бойынша орындау рұқсаттарды талап етеді:
  - Ұсыныстар: `CanProposeContractDeployment{ contract_id }`
  - Бюллетеньдер: `CanSubmitGovernanceBallot{ referendum_id }`
  - Қолданылуы: `CanEnactGovernance`
  - Кеңесті басқару (болашақ): `CanManageParliament`

Қорғалған аттар кеңістігі
- `gov_protected_namespaces` теңшелетін параметрі (JSON жолдар массиві) тізімделген аттар кеңістігіне орналастыруға рұқсат беруді қосады.
- Клиенттер қорғалған аттар кеңістігіне бағытталған орналастырулар үшін транзакция метадеректерінің кілттерін қамтуы керек:
  - `gov_namespace`: мақсатты аттар кеңістігі (мысалы, `"apps"`)
  - `gov_contract_id`: аттар кеңістігіндегі логикалық келісім-шарт идентификаторы
- `gov_manifest_approvers`: валидатор тіркелгісінің идентификаторларының қосымша JSON массиві. Жолақ манифесті біреуден көп кворум жариялағанда, қабылдау манифест кворумын қанағаттандыру үшін транзакция органынан және тізімде көрсетілген есептік жазбаларды талап етеді.
- Телеметрия `governance_manifest_admission_total{result}` арқылы біртұтас қабылдау есептегіштерін көрсетеді, осылайша операторлар сәтті қабылдауларды `missing_manifest`, `non_validator_authority`, `quorum_rejected`, `protected_namespace_rejected`, `protected_namespace_rejected`, I18NI0000156NIX, I1810000 жолдарынан ажырата алады.
- Телеметрия орындау жолын `governance_manifest_quorum_total{outcome}` (`satisfied` / `rejected` мәндері) арқылы көрсетеді, осылайша операторлар жетіспейтін бекітулерді тексере алады.
- Жолдар өздерінің манифестерінде жарияланған аттар кеңістігінің рұқсат тізімін күшейтеді. `gov_namespace` орнататын кез келген транзакция `gov_contract_id` қамтамасыз етуі керек және аттар кеңістігі манифесттің `protected_namespaces` жинағында пайда болуы керек. Бұл метадеректерсіз `RegisterSmartContractCode` жіберулер қорғау қосылған кезде қабылданбайды.
- `(namespace, contract_id, code_hash, abi_hash)` кортежі үшін Қабылданған басқару ұсынысының бар екеніне рұқсат беру; әйтпесе тексеру рұқсат етілмеген қатемен орындалмайды.

Орындау уақытын жаңарту ілмектері
- Жолақ манифесттері `hooks.runtime_upgrade` қақпасының орындалу уақытын жаңарту нұсқауларына (`ProposeRuntimeUpgrade`, `ActivateRuntimeUpgrade`, `CancelRuntimeUpgrade`) жариялауы мүмкін.
- Ілмек өрістері:
  - `allow` (bool, әдепкі `true`): `false` болғанда, барлық орындау уақытын жаңарту нұсқаулары қабылданбайды.
  - `require_metadata` (bool, әдепкі `false`): `metadata_key` арқылы көрсетілген транзакция метадеректерінің енгізуін талап етеді.
  - `metadata_key` (жол): ілмекпен орындалатын метадеректер атауы. Метадеректер қажет болғанда немесе рұқсат етілген тізім бар болса, әдепкі `gov_upgrade_id`.
  - `allowed_ids` (жолдар жиымы): метадеректер мәндерінің қосымша рұқсат етілген тізімі (қиюдан кейін). Берілген мән тізімде жоқ кезде бас тартады.
- Ілмек болған кезде, кезекті қабылдау транзакция кезекке кірмес бұрын метадеректер саясатын жүзеге асырады. Жетіспейтін метадеректер, бос мәндер немесе рұқсат етілген тізімнен тыс мәндер детерминирленген `NotPermitted` қатесін тудырады.
- Телеметрия `governance_manifest_hook_total{hook="runtime_upgrade", outcome="allowed|rejected"}` арқылы орындау нәтижелерін бақылайды.
- Ілгекті қанағаттандыратын транзакциялар манифест кворумы талап ететін кез келген валидатор мақұлдауларымен қатар `gov_upgrade_id=<value>` метадеректерін (немесе манифестте анықталған кілтті) қамтуы керек.

CastZkBallot тексеру жолы
- `CastZkBallot` берілген base64 дәлелін декодтайды және бос немесе дұрыс емес пайдалы жүктемелерден бас тартады (`BallotRejected` `invalid or empty proof`).
- `public_inputs_json` жеткізілсе, ол JSON нысаны болуы керек; объектілік емес пайдалы жүктемелер қабылданбайды.
- Хост референдумдағы бюллетеньді растайтын кілтті (`vk_ballot`) немесе басқару әдепкілерін шешеді және жазбаның `Active` болуын және кірістірілген байттардың болуын талап етеді.
- Сақталған тексеру кілті байттары `hash_vk` көмегімен қайта хэштелген; кез келген міндеттеме сәйкессіздігі бұрмаланған тізілім жазбаларынан (`verifying key commitment mismatch` бар `BallotRejected`) қорғау үшін тексеруден бұрын орындауды тоқтатады.
- Дәлелдеу байттары тіркелген серверге `zk::verify_backend` арқылы жіберіледі; жарамсыз транскрипттер `invalid proof` бар `BallotRejected` болып шығады және нұсқау айқын түрде орындалмайды.
- Дәлелдемелер сайлау бюллетеньіндегі міндеттеме мен жарамдылық түбірін жария ақпарат ретінде көрсетуі керек; түбір сайлаудың `eligible_root` сәйкес келуі керек, ал алынған нөлдік көрсеткіш кез келген берілген кеңеске сәйкес келуі керек.
- Сәтті дәлелдер `BallotAccepted` шығарады; қайталанатын жоққа шығарғыштар, ескірген жарамдылық түбірлері немесе құлыптау регрессиялары осы құжатта бұрын сипатталған бұрыннан бар қабылдамау себептерін тудырады.

## Валидатордың қате әрекеті және бірлескен консенсус

### Жұмыс процесі

Валидатор хаттаманы бұзған сайын консенсус Norito кодталған `Evidence` шығарады. Әрбір пайдалы жүктеме `EvidenceStore` жадына түседі және егер көрінбесе, WSV қолдайтын `consensus_evidence` картасына енгізіледі. `sumeragi.npos.reconfig.evidence_horizon_blocks` (әдепкі `7 200` блоктары) ескі жазбалар қабылданбайды, сондықтан мұрағат шектелген күйінде қалады, бірақ бас тарту операторлар үшін журналға жазылады. Көкжиек аясындағы дәлелдер бірлескен консенсус кезеңінің ережесіне (`mode_activation_height requires next_mode to be set in the same block`), белсендіру кідірісіне (`sumeragi.npos.reconfig.activation_lag_blocks`, әдепкі `1`) және кесу кідірісіне (`sumeragi.npos.reconfig.slashing_delay_blocks`, әдепкі I082) бағынады. олар қолданылғанға дейін айыппұлдар.

Танылған құқық бұзушылықтар `EvidenceKind` картасына бір-бірден; дискриминанттар тұрақты және деректер үлгісімен орындалады:

```rust
use iroha_data_model::block::consensus::EvidenceKind;

let offences = [
    EvidenceKind::DoublePrepare,
    EvidenceKind::DoubleCommit,
    EvidenceKind::InvalidQc,
    EvidenceKind::InvalidProposal,
    EvidenceKind::Censorship,
];

for (expected, kind) in offences.iter().enumerate() {
    assert_eq!(*kind as u16, expected as u16);
}
```

- **DoublePrepare/DoubleCommit** — валидатор бірдей `(phase,height,view,epoch)` кортежі үшін қайшы келетін хэштерге қол қойды.
- **InvalidQc** — агрегатор пішіні детерминирленген тексерулерден өтпеген (мысалы, бос қол қоюшының нүктелік суреті) орындалған сертификатты ғайбаттады.
- **InvalidProposal** — жетекші құрылымдық тексеруден өтпеген блокты ұсынды (мысалы, құлыпталған тізбек ережесін бұзады).
- **Цензура** — қол қойылған жіберу түбіртектері ешқашан ұсынылмаған/жасалмаған транзакцияны көрсетеді.

VRF жазалары `activation_lag_blocks` кейін автоматты түрде орындалады (құқық бұзушылар түрмеге жабылады). Басқару жазаны жоймаса, консенсусты кесу `slashing_delay_blocks` терезесінен кейін ғана қолданылады.

Операторлар мен құралдар пайдалы жүктемелерді мыналар арқылы тексеріп, қайта тарата алады:

- Torii: `GET /v2/sumeragi/evidence` және `GET /v2/sumeragi/evidence/count`.
- CLI: `iroha ops sumeragi evidence list`, `… count` және `… submit --evidence-hex <payload>`.

Басқару дәлел байттарын канондық дәлел ретінде қарастыруы керек:

1. **Пайдалы жүктемені ол ескірмей тұрып жинаңыз**. Биіктік/көрініс метадеректерімен қатар шикі Norito байттарын мұрағаттаңыз.
2. **Қажет болса, бас тартыңыз** `CancelConsensusEvidencePenalty` құжатын `slashing_delay_blocks` біткенге дейін дәлелдемелердің пайдалылығымен бірге жіберу; жазба `penalty_cancelled` және `penalty_cancelled_at_height` деп белгіленеді және қию қолданылмайды.
3. Пайдалы жүктемені референдумға немесе sudo нұсқауына (мысалы, `Unregister::peer`) енгізу арқылы **Айыппұлды кезеңге қойыңыз**. Орындау пайдалы жүктемені қайта растайды; дұрыс емес немесе ескірген дәлелдер детерминистикалық түрде қабылданбайды.
4. **Кейінгі топологияны жоспарлаңыз**, сондықтан бұзушы валидатор бірден қайта қосыла алмайды. Әдеттегі ағындар кезегі `SetParameter(Sumeragi::NextMode)` және `SetParameter(Sumeragi::ModeActivationHeight)` жаңартылған тізіммен.
5. **Аудит нәтижелері** `/v2/sumeragi/evidence` және `/v2/sumeragi/status` арқылы дәлелдемелердің жетілдірілгенін және басқарудың жоюды қамтамасыз ету үшін.

### Бірлескен консенсус реттілігі

Бірлескен консенсус шығыс валидатор жинағы жаңа жиын ұсынуды бастамас бұрын шекаралық блокты аяқтайтынына кепілдік береді. Орындау уақыты ережені жұптастырылған параметрлер арқылы жүзеге асырады:

- `SumeragiParameter::NextMode` және `SumeragiParameter::ModeActivationHeight` **бір блокта** жасалуы керек. `mode_activation_height` кем дегенде бір блоктық кешігуді қамтамасыз ететін жаңартуды тасымалдаған блок биіктігінен қатаң түрде үлкен болуы керек.
- `sumeragi.npos.reconfig.activation_lag_blocks` (әдепкі `1`) нөлдік кешігуді болдырмайтын конфигурация қорғаушысы:
- `sumeragi.npos.reconfig.slashing_delay_blocks` (әдепкі `259200`) консенсусты қысқартуды кейінге қалдырады, осылайша басқару жазаларды олар қолданылғанға дейін жоя алады.

```rust
use iroha_config::parameters::defaults::sumeragi::npos::RECONFIG_ACTIVATION_LAG_BLOCKS;
assert_eq!(RECONFIG_ACTIVATION_LAG_BLOCKS, 1);
```

- Орындау уақыты мен CLI `/v2/sumeragi/params` және `iroha --output-format text ops sumeragi params` арқылы кезеңдік параметрлерді көрсетеді, осылайша операторлар белсендіру биіктіктері мен валидатор тізімін растай алады.
- Басқаруды автоматтандыру әрқашан:
  1. Дәлелмен негізделген жою (немесе қалпына келтіру) туралы шешімді аяқтаңыз.
  2. `mode_activation_height = h_current + activation_lag_blocks` арқылы кейінгі қайта конфигурациялауды кезекке қойыңыз.
  3. `effective_consensus_mode` күтілетін биіктікте аударылғанша `/v2/sumeragi/status` мониторы.

Валидаторларды айналдыратын немесе қиғаш сызықты қолданатын кез келген сценарий **нөлдік кешігуді белсендіруге әрекеттенбеу керек немесе тапсыру параметрлерін өткізіп жіберуге болмайды; мұндай транзакциялар қабылданбайды және желіні алдыңғы режимде қалдырады.

## Телеметриялық беттер

- Prometheus экспортты басқару әрекетінің көрсеткіштері:
  - `governance_proposals_status{status}` (өлшеуіш) күй бойынша ұсыныс санын қадағалайды.
  - `governance_protected_namespace_total{outcome}` (есептеуіш) қорғалған аттар кеңістігін қабылдау орналастыруға рұқсат бергенде немесе қабылдамағанда өседі.
  - `governance_manifest_activations_total{event}` (есептегіш) манифест кірістірулерін (`event="manifest_inserted"`) және аттар кеңістігінің байланыстарын (`event="instance_bound"`) жазады.
- `/status` ұсыныс сандарын көрсететін, қорғалған аттар кеңістігінің жиынтықтары туралы есеп беретін және соңғы манифест белсендірулерінің тізімін көрсететін `governance` нысанын қамтиды (аттар кеңістігі, келісімшарт идентификаторы, код/ABI хэші, блок биіктігі, белсендіру уақыт белгісі). Операторлар осы өріске сауалнама жүргізе алады, акттердің жаңартылған манифесттерін және қорғалған аттар кеңістігі қақпаларының күшіне енуін растайды.
- Grafana үлгісі (`docs/source/grafana_governance_constraints.json`) және
  `telemetry.md` ішіндегі telemetry runbook дабылдарды кептелу үшін қалай қосу керектігін көрсетеді
  ұсыныстар, жоқ манифест белсендірулері немесе күтпеген қорғалған аттар кеңістігі
  орындау уақытын жаңарту кезінде қабылдамаулар.
