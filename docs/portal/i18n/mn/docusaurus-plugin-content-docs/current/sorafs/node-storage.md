## SoraFS Зангилаа хадгалах дизайн (Ноорог)

Энэ тэмдэглэл нь Iroha (Torii) зангилаа SoraFS датад хэрхэн нэвтрэх боломжтойг нарийвчлан тодорхойлдог.
availability давхарга болон хадгалах, үйлчлэхэд зориулж локал дискний зүсмэлийг зориулна
хэсгүүд. Энэ нь `sorafs_node_client_protocol.md` нээлтийн үзүүлэлт болон
SF-1b бэхэлгээ нь хадгалах талын архитектур, нөөцийг тодорхойлох замаар ажилладаг
удирдлага, зангилаа болон гарцанд буух ёстой тохиргооны сантехник
кодын замууд. Практик оператор өрөм нь амьдардаг
[Зангилааны үйлдлийн ном](./node-operations).

### Зорилго

- Аливаа баталгаажуулагч эсвэл туслах Iroha процесст нөөц дискийг ил гаргахыг зөвшөөрөх.
  SoraFS үйлчилгээ үзүүлэгч нь бүртгэлийн үндсэн үүрэг хариуцлагад нөлөөлөхгүй.
- Хадгалах модулийг тодорхойлогч, Norito-д суурилсан байлгах: манифест,
  бөөн төлөвлөгөөнүүд, Proof-of-Retrievability (PoR) үндэс, үйлчилгээ үзүүлэгчийн зарууд нь
  үнэний эх сурвалж.
- Операторын тодорхойлсон квотыг мөрдүүлснээр зангилаа өөрийн нөөцөө шавхаж чадахгүй
  хэт олон зүү эсвэл татах хүсэлтийг хүлээн авч байна.
- Гадаргуугийн эрүүл мэнд/телеметри (PoR дээж авах, бөөгнөрөл татах хоцролт, дискний даралт)
  засаглал болон үйлчлүүлэгчид рүү буцах.

### Өндөр түвшний архитектур

```
┌──────────────────────────────────────────────────────────────────────┐
│                         Iroha/Torii Node                             │
│                                                                      │
│  ┌──────────────┐      ┌────────────────────┐                        │
│  │  Torii APIs  │◀────▶│   SoraFS Gateway   │◀───────────────┐       │
│  └──────────────┘      │ (Norito endpoints) │                │       │
│                        └────────┬───────────┘                │       │
│                                 │                            │       │
│                        ┌────────▼────────┐                   │       │
│                        │  Pin Registry   │◀───── manifests   │       │
│                        │ (State / DB)    │                   │       │
│                        └────────┬────────┘                   │       │
│                                 │                            │       │
│                        ┌────────▼────────┐                   │       │
│                        │  Chunk Storage  │◀──── chunk plans  │       │
│                        │  (ChunkStore)   │                   │       │
│                        └────────┬────────┘                   │       │
│                                 │                            │       │
│                        ┌────────▼────────┐                   │       │
│                        │  Disk Quota/IO  │─Pin/serve chunks─▶│ Fetch │
│                        │  Scheduler      │                   │ Clients│
│                        └─────────────────┘                   │       │
│                                                                      │
└──────────────────────────────────────────────────────────────────────┘
```

Гол модулиуд:

- **Гарц**: зүү санал болгох, бөөгнөрөл татахад зориулж Norito HTTP төгсгөлийн цэгүүдийг гаргадаг
  хүсэлт, PoR дээж авах, телеметр. Энэ нь Norito ачааллыг баталгаажуулдаг ба
  бөөгнөрөл дэлгүүрт маршал хүсэлт илгээдэг. Одоо байгаа Torii HTTP стекийг дахин ашигладаг
  шинэ демоноос зайлсхийхийн тулд.
- **Pin Registry**: `iroha_data_model::sorafs`-д хянагдсан манифестийн зүү төлөв
  болон `iroha_core`. Манифестыг хүлээн авах үед бүртгэлд үүнийг тэмдэглэнэ
  манифест дижест, хэсэгчилсэн төлөвлөгөөний дижест, PoR үндэс, үйлчилгээ үзүүлэгчийн чадамжийн туг.
- **Бөөгийг хадгалах**: залгидаг дискээр дэмжигдсэн `ChunkStore` хэрэгжилт
  гарын үсэг зурсан манифестууд, `ChunkProfile::DEFAULT` ашиглан хэсэгчилсэн төлөвлөгөөг хэрэгжүүлэх, мөн
  детерминист зохион байгуулалтын дор хэсгүүдийг хадгалдаг. Хэсэг бүр нь а-тай холбоотой
  контентын хурууны хээ болон PoR мета өгөгдөлтэй тул түүвэрлэлт хийхгүйгээр дахин баталгаажуулах боломжтой
  файлыг бүхэлд нь дахин унших.
- **Квот/Хуваарьлагч**: операторын тохируулсан хязгаарыг хэрэгжүүлдэг (хамгийн их дискний байт,
  хамгийн их үлдэгдэл зүү, хамгийн их зэрэгцээ татах, хэсэг TTL) болон координат
  IO, ингэснээр зангилааны дэвтэрийн үүрэг өлсгөлөнд орохгүй. Төлөвлөгч нь мөн
  хязгаарлагдмал CPU бүхий PoR баталгаа болон дээж авах хүсэлтийг хангах үүрэгтэй.

### Тохиргоо

`iroha_config`-д шинэ хэсэг нэмнэ үү:

```toml
[sorafs.storage]
enabled = false
data_dir = "/var/lib/iroha/sorafs"
max_capacity_bytes = "100 GiB"
max_parallel_fetches = 32
max_pins = 10_000
por_sample_interval_secs = 600
alias = "tenant.alpha"            # optional human friendly tag
adverts:
  stake_pointer = "stake.pool.v1:0x1234"
  availability = "hot"
  max_latency_ms = 500
  topics = ["sorafs.sf1.primary:global"]
```

- `enabled`: оролцооны унтраалга. Худал бол гарц нь 503 гэсэн утгыг буцаана
  хадгалах төгсгөлийн цэгүүд ба зангилаа нь нээлтэд сурталчилдаггүй.
- `data_dir`: бөөн өгөгдөл, PoR мод, дуудах телеметрийн үндсэн лавлах.
  Өгөгдмөл нь `<iroha.data_dir>/sorafs`.
- `max_capacity_bytes`: хавчуулсан өгөгдлийн хатуу хязгаар. Суурь даалгавар
  хязгаарт хүрсэн үед шинэ зүүг татгалздаг.
- `max_parallel_fetches`: тэнцвэржүүлэхийн тулд хуваарь гаргагч мөрдсөн зэрэглэлийн хязгаар
  баталгаажуулагчийн ажлын ачааллын эсрэг зурвасын өргөн/дискний IO.
- `max_pins`: хэрэглэхээс өмнө зангилааны хүлээн авах хамгийн их тоо
  нүүлгэн шилжүүлэх/буцах дарамт.
- `por_sample_interval_secs`: PoR түүвэрлэлтийн автомат ажилд зориулсан каденс. Ажил бүр
  `N` дээжүүд (манифест бүрээр тохируулах боломжтой) гарч, телеметрийн үйл явдлуудыг ялгаруулдаг.
  Засаглал нь чадавхийн мета өгөгдлийг тохируулснаар `N`-ийг тодорхой хэмжээгээр өргөжүүлж болно.
  түлхүүр `profile.sample_multiplier` (бүхэл тоо `1-4`). Утга нь ганц байж болно
  тоо/мөр эсвэл профайл тус бүр дээр дарагдсан объект, жишээ нь.
  `{"default":2,"sorafs.sf2@1.0.0":3}`.
- `adverts`: үйлчилгээ үзүүлэгчийн бөглөхөд ашигладаг бүтэц
  `ProviderAdvertV1` талбарууд (гадасны заагч, QoS зөвлөмж, сэдвүүд). Хэрэв орхигдуулсан бол
  node нь засаглалын бүртгэлийн анхдагч тохиргоог ашигладаг.

Сантехникийн тохиргоо:

- `[sorafs.storage]` нь `iroha_config`-д `SorafsStorage` гэж тодорхойлогдсон бөгөөд
  зангилааны тохиргооны файлаас ачаалагдсан.
- `iroha_core` болон `iroha_torii` санах ойн тохиргоог гарц руу оруулна
  эхлүүлэх үед барилгачин болон бөөн дэлгүүр.
- Хөгжүүлэгч/туршилтын орчинг хүчингүй болгосон (`SORAFS_STORAGE_*`, `SORAFS_STORAGE_PIN_*`), гэхдээ
  үйлдвэрлэлийн байршуулалт нь тохиргооны файлд тулгуурлах ёстой.

### CLI хэрэгслүүд

Torii-ийн HTTP гадаргуу утсаар холбогдож байх үед `sorafs_node` хайрцаг нь
нимгэн CLI тул операторууд байнгын эсрэг дасгалуудыг залгих/экспортлох скрипт хийх боломжтой
backend.【crates/sorafs_node/src/bin/sorafs-node.rs:1】

```bash
cargo run -p sorafs_node --bin sorafs-node ingest \
  --data-dir ./storage/sorafs \
  --manifest ./fixtures/manifest.to \
  --payload ./fixtures/payload.bin \
  --plan-json-out ./plan.json
```

- `ingest` нь Norito кодлогдсон манифест `.to` файл болон тохирох ачааллыг хүлээж байна
  байт. Энэ нь манифестын хуваалтын профайлаас хэсэгчилсэн төлөвлөгөөг дахин бүтээдэг.
  digest parite-г мөрдүүлж, chunk файлуудыг хадгалах ба сонголтоор a
  `chunk_fetch_specs` JSON бөмбөрцөг тул доод талын хэрэгсэл нь эрүүл мэндийг шалгах боломжтой.
  зохион байгуулалт.
- `export` нь манифест ID-г хүлээн авч, хадгалсан манифест/ачаалыг дискэнд бичдэг
  (заавал биш JSON төлөвлөгөөтэй) тул бэхэлгээг орчин даяар давтах боломжтой хэвээр байна.

Хоёр тушаал хоёулаа Norito JSON хураангуйг stdout руу хэвлэхэд хялбар болгодог.
скриптүүд. CLI нь манифест болон баталгаажуулахын тулд интеграцийн тестээр хамрагдсан
ачааллыг Torii API-ууд буухаас өмнө цэвэрхэн эргүүлнэ.【crates/sorafs_node/tests/cli.rs:1】

### Зангилааны амьдралын мөчлөг

1. **Эхлүүлэх**:
   - Хадгалалт идэвхжсэн бол зангилаа нь бөөгнөрөл хадгалалтыг эхлүүлнэ
     тохируулсан лавлах болон багтаамж. Үүнд баталгаажуулах эсвэл үүсгэх зэрэг орно
     PoR манифест мэдээллийн сан болон бэхлэгдсэн манифестуудыг дулаан кэш рүү дахин тоглуулах.
   - SoraFS гарцын маршрутуудыг бүртгэх (pin-ийн Norito JSON POST/GET төгсгөлийн цэгүүд,
     татах, PoR дээж, телеметр).
   - PoR дээж авах ажилтан болон квотын хяналтыг суулгана.
2. **Нээлт / Зар сурталчилгаа**:
   - Одоогийн хүчин чадал/эрүүл мэнд, тэмдэг ашиглан `ProviderAdvertV1` баримт бичгийг үүсгэх
     тэдгээрийг зөвлөлөөс зөвшөөрсөн түлхүүрээр, мөн нээлтийн сувгаар нийтлэх.
     боломжтой.
3. **Зүү ажлын урсгал**:
   - Гарц нь гарын үсэг зурсан манифест хүлээн авдаг (бүлэг төлөвлөгөө, PoR үндэс, зөвлөл).
     гарын үсэг). Хорих нэрийн жагсаалтыг баталгаажуулах (`sorafs.sf1@1.0.0` шаардлагатай) болон
     Хэсэгчилсэн төлөвлөгөө нь манифест мета өгөгдөлтэй тохирч байгаа эсэхийг шалгаарай.
   - Квотоо шалгах. Хэрэв хүчин чадал/зүүний хязгаар хэтэрсэн бол a гэж хариулна уу
     бодлогын алдаа (Norito бүтэцтэй).
   - Хэсэгчилсэн өгөгдлийг `ChunkStore` руу дамжуулж, биднийг залгих үед боловсруулалтыг баталгаажуулна.
     PoR модыг шинэчилж, манифест мета өгөгдлийг бүртгэлд хадгална.
4. **Ажлын урсгалыг татах**:
   - Дискээс хэсэгчилсэн хүрээний хүсэлтийг үзүүлэх. Хуваарьлагч мөрддөг
     `max_parallel_fetches` ба ханасан үед `429` буцаана.
   - Хоцролт, үйлчилсэн байт болон бүтэцлэгдсэн телеметрийг (Norito JSON) гаргах
     доод урсгалын мониторингийн алдааны тоо.
5. **PoR дээж авах**:
   - Ажилчин жинд пропорциональ (жишээ нь, хадгалсан байт) болон манифестуудыг сонгоно
     бөөгнөрөл дэлгүүрийн PoR модыг ашиглан тодорхойлогч түүвэрлэлтийг явуулдаг.
   -Засаглалын аудитын үр дүнг хэвээр үлдээж, хураангуйг ханган нийлүүлэгчид оруулах
     зар сурталчилгаа / телеметрийн төгсгөлийн цэгүүд.
6. **Нүүлгэн шилжүүлэх / Квотын хэрэгжилт**:
   - Хүчин чадалд хүрсэн үед зангилаа нь анхдагчаар шинэ тээглүүрээс татгалздаг. Сонголтоор,
     операторууд нүүлгэн шилжүүлэх бодлогыг (жишээ нь, TTL-д суурилсан, LRU) нэг удаа тохируулж болно
     засаглалын загварыг зөвшөөрсөн; Одоогоор зураг төсөл нь хатуу квот гэж үздэг ба
     операторын эхлүүлсэн тогтоогоогүй үйлдлүүд.

### Чадавхийн тунхаглал ба хуваарь нэгтгэх- Torii одоо `/v2/sorafs/capacity/declare`-ээс `CapacityDeclarationRecord` шинэчлэлтүүдийг дамжуулж байна
  суулгагдсан `CapacityManager` руу оруулснаар зангилаа бүр өөрийнх нь санах ойд харагдах байдлыг бий болгодог.
  chunker болон эгнээний хуваарилалт хийсэн. Менежер зөвхөн унших боломжтой агшин зуурын агшинг ил гаргадаг
  телеметрийн хувьд (`GET /v2/sorafs/capacity/state`) болон профайл тус бүр эсвэл эгнээ тус бүрийг хэрэгжүүлдэг
  шинэ захиалга хүлээн авахаас өмнө захиалга авна.【crates/sorafs_node/src/capacity.rs:1】【crates/sorafs_node/src/lib.rs:60】
- `/v2/sorafs/capacity/schedule` төгсгөлийн цэг нь засаглалын гаргасан `ReplicationOrderV1`-г хүлээн зөвшөөрдөг
  ачаалал. Захиалга нь орон нутгийн үйлчилгээ үзүүлэгч рүү чиглэх үед менежер шалгадаг
  хуваарийг давхардуулах, chunker/ эгнээний багтаамжийг шалгах, зүсмэлийг нөөцлөх, мөн
  Үлдсэн хүчин чадлыг тодорхойлсон `ReplicationPlan`-г буцааснаар зохион байгуулах хэрэгслүүд
  залгих замаар үргэлжлүүлж болно. Бусад үйлчилгээ үзүүлэгчийн захиалгыг a-аар хүлээн зөвшөөрдөг
  Олон операторын ажлын урсгалыг хөнгөвчлөх `ignored` хариулт.【crates/iroha_torii/src/routing.rs:4845】
- Гүйцэтгэх дэгээнүүд (жишээ нь: залгих амжилттай болсны дараа үүснэ) цохив
  дамжуулан захиалга гаргах `POST /v2/sorafs/capacity/complete`
  `CapacityManager::complete_order`. Хариуд нь `ReplicationRelease` орно
  агшин зуурын зураг (үлдсэн нийлбэр, chunker/ эгнээний үлдэгдэл) нь зохион байгуулах хэрэгсэл нь
  дараагийн захиалгаа санал хураалгүйгээр дараалалд оруулна. Цаашдын ажил нь үүнийг хэсэг болгон холбох болно

### Интеграци ба ирээдүйн ажил

- **Засаглал**: `sorafs_pin_registry_tracker.md`-г хадгалах телеметрээр өргөтгөх
  (PoR амжилтын түвшин, дискний ашиглалт). Элсэлтийн бодлого нь доод тал нь шаардаж болно
  хүчин чадал эсвэл зар сурталчилгааг хүлээн авахаас өмнө хамгийн бага PoR амжилтын түвшин.
- **Клиент SDKs**: хадгалах сангийн шинэ тохиргоог (дискний хязгаар, нэр) харуулах
  удирдлагын хэрэгсэл нь зангилааг программтайгаар ачаалах боломжтой.
- **Телеметри**: одоо байгаа хэмжүүрийн стектэй нэгтгэх (Prometheus /
  OpenTelemetry) тул хадгалалтын хэмжүүрүүд ажиглалтын хяналтын самбарт харагдана.
- **Аюулгүй байдал**: хадгалах модулийг тусгай зориулалтын асинхрон даалгаврын сан дотор ажиллуулна
  буцаах даралт болон io_uring эсвэл tokio's-ээр дамжуулан хамгаалагдсан хязгаарлагдмал орчинд нэвтрэх хэсгийг уншина уу
  хорлонтой үйлчлүүлэгчид нөөцөө шавхахаас сэргийлэхийн тулд хязгаарлагдмал усан сангууд.

Энэ загвар нь хадгалах модулийг сонголттой байлгаж, өгөхдөө тодорхой болгодог
операторууд SoraFS өгөгдлийн хүртээмжид оролцох шаардлагатай товчлууруудыг
давхарга. Үүнийг хэрэгжүүлэхэд `iroha_config`, `iroha_core`,
`iroha_torii`, Norito гарц, үйлчилгээ үзүүлэгчийн сурталчилгааны хэрэгсэл.
