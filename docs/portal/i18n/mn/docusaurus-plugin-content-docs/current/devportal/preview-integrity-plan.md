Энэ төлөвлөгөөнд портал урьдчилан үзэх олдвор бүрийг хэвлэхээс өмнө шалгах боломжтой болгоход шаардагдах үлдсэн ажлуудыг тусгасан болно. Зорилго нь хянагчид CI-д суулгасан агшин зуурын агшин зургийг татаж авах, шалгах нийлбэрийн манифест өөрчлөгдөхгүй, Norito мета өгөгдөлтэй SoraFS-ээр дамжуулан урьдчилан харах боломжтой гэдгийг баталгаажуулах явдал юм.

## Зорилтууд

- **Тодорхойлолттой бүтээц:** `npm run build` нь давтагдах боломжтой гаралт гаргаж, үргэлж `build/checksums.sha256` ялгаруулдаг эсэхийг шалгаарай.
- **Баталгаажсан урьдчилж харах:** Урьдчилан үзэх олдвор бүрийг шалгах нийлбэрийн манифестийн хамт илгээхийг шаардаж, баталгаажуулалт амжилтгүй болсон үед нийтлэхээс татгалзах.
- **Norito-ийн нийтэлсэн мета өгөгдөл:** Урьдчилан харах тодорхойлогчдыг (мета өгөгдөл, хяналтын нийлбэр хураангуй, SoraFS CID) Norito JSON байдлаар хадгалах тул засаглалын хэрэгсэл нь хувилбаруудыг шалгах боломжтой.
- **Операторын хэрэглүүр:** Хэрэглэгчид дотооддоо ажиллуулах боломжтой нэг шатлалт баталгаажуулалтын скриптээр хангах (`./docs/portal/scripts/preview_verify.sh --build-dir build --descriptor <path> --archive <path>`); скрипт нь шалгах нийлбэр + тодорхойлогч баталгаажуулалтын урсгалыг төгсгөлөөс нь төгсгөлд нь ороож байна. Урьдчилан харах стандарт команд (`npm run serve`) одоо `docusaurus serve`-ээс өмнө энэ туслагчийг автоматаар дуудаж байгаа тул локал агшин агшинд шалгах нийлбэр хаалттай хэвээр байна (`npm run serve:verified` нь илэрхий өөр нэрээр хадгалагдана).

## 1-р үе шат — CI хэрэгжилт

1. `.github/workflows/docs-portal-preview.yml`-г шинэчлэх:
   - Docusaurus бүтээсний дараа `node docs/portal/scripts/write-checksums.mjs`-г ажиллуул (аль хэдийн дотооддоо дуудагдсан).
   - `cd build && sha256sum -c checksums.sha256`-г ажиллуулж, тохирохгүй байгаа тул ажил бүтэлгүйтэв.
   - Бүтээлийн лавлахыг `artifacts/preview-site.tar.gz` гэж багцалж, шалгах нийлбэрийн манифестийг хуулж, `scripts/generate-preview-descriptor.mjs` руу залгаж, `scripts/sorafs-package-preview.sh`-г JSON тохиргоогоор ажиллуулж (`docs/examples/sorafs_preview_publish.json`-г үзнэ үү) ажлын урсгал нь метадата I09008 болон I18NI00000030-г хоёуланг нь ялгаруулна. багц.
   - Статик сайт, мета өгөгдлийн олдворууд (`docs-portal-preview`, `docs-portal-preview-metadata`), SoraFS багцыг (`docs-portal-preview-sorafs`) байршуулснаар манифест, CAR хураангуй болон төлөвлөлтийг дахин шалгахгүйгээр дахин ажиллуулах боломжтой.
2. Татаж авах хүсэлтүүдэд шалгах нийлбэр баталгаажуулалтын үр дүнг нэгтгэсэн CI тэмдгийн тайлбарыг нэмнэ үү (✅ `docs-portal-preview.yml` GitHub Скрипт тайлбарын алхамаар хэрэгжсэн).
3. Ажлын явцыг `docs/portal/README.md` (CI хэсэг) дээр баримтжуулж, хэвлэн нийтлэх хяналтын хуудасны баталгаажуулах алхмуудыг холбоно уу.

## Баталгаажуулах скрипт

`docs/portal/scripts/preview_verify.sh` нь гар аргаар `sha256sum` дуудлагыг шаардахгүйгээр татаж авсан урьдчилан үзэх олдворуудыг баталгаажуулдаг. Скриптийг ажиллуулахын тулд `npm run serve` (эсвэл тодорхой `npm run serve:verified` нэр) ашиглан локал агшин зуурын агшин зургуудыг хуваалцахдаа `docusaurus serve`-г нэг алхамаар эхлүүлнэ үү. Баталгаажуулах логик:

1. `build/checksums.sha256`-ийн эсрэг тохирох SHA хэрэгслийг (`sha256sum` эсвэл `shasum -a 256`) ажиллуулдаг.
2. Сонголтоор урьдчилан харах тодорхойлогчийн `checksums_manifest` дижест/файлын нэр болон өгсөн тохиолдолд урьдчилан харах архивын тойм/файлын нэрийг харьцуулна.
3. Тохиромжгүй байдал илэрсэн үед тэгээс ялгаатай байх тул хянагчид хөндлөнгөөс урьдчилан харахыг хориглох боломжтой.

Хэрэглээний жишээ (CI олдворуудыг гаргаж авсны дараа):

```bash
./docs/portal/scripts/preview_verify.sh \
  --build-dir build \
  --descriptor artifacts/preview-descriptor.json \
  --archive artifacts/preview-site.tar.gz
```

CI болон хувилбарын инженерүүд урьдчилан үзэх багцыг татаж авах эсвэл олдворуудыг хувилбарын тасалбарт хавсаргах бүртээ скрипт рүү залгах ёстой.

## 2-р үе шат — SoraFS Нийтлэл

1. Урьдчилан харах ажлын урсгалыг дараах ажлуудаар өргөтгөх:
   - Баригдсан сайтыг `sorafs_cli car pack` болон `manifest submit` ашиглан SoraFS үе шатны гарц руу байршуулна.
   - Буцсан манифест дижест болон SoraFS CID-г авна.
   - `{ commit, branch, checksum_manifest, cid }`-г Norito JSON (`docs/portal/preview/preview_descriptor.json`) болгон цуваа болгодог.
2. Тодорхойлогчийг бүтээх олдворын хажууд хадгалж, CID-г татах хүсэлтийн тайлбарт ил гарга.
3. Ирээдүйн өөрчлөлтүүд мета өгөгдлийн схемийг тогтвортой байлгахын тулд `sorafs_cli`-ийг хуурай горимд ажиллуулдаг интеграцийн тестүүдийг нэмнэ үү.

## 3-р үе шат — Засаглал ба аудит

1. `docs/portal/schemas/` доор тодорхойлогч бүтцийг дүрсэлсэн Norito схемийг (`PreviewDescriptorV1`) нийтэл.
2. DOCS-SORA нийтлэх хяналтын хуудсыг шинэчлээрэй:
   - Байршуулсан CID-ийн эсрэг `sorafs_cli manifest verify` ажиллуулж байна.
   - Хувилбарын PR тайлбарт шалгах нийлбэр манифест дижест болон CID-г бүртгэх.
3. Санал хураалтын үеэр тодорхойлогчийг шалгах нийлбэрийн манифесттэй харьцуулан шалгахын тулд засаглалын автоматжуулалтыг утсаар холбоно.

## Нээлттэй асуултууд

- Аль SoraFS орчин нь урьдчилж харах олдворуудыг (үе шат ба тусгайлан үзэх эгнээ) хадгалах ёстой вэ?
- Нийтлэхээс өмнө урьдчилан харах тодорхойлогч дээр давхар гарын үсэг (Ed25519 + ML-DSA) хэрэгтэй юу?
- CI ажлын урсгал нь `sorafs_cli`-г ажиллуулж байх үед манифестуудыг давтагдах боломжтой байлгахын тулд найруулагчийн тохиргоог (`orchestrator_tuning.json`) бэхлэх ёстой юу?

`docs/portal/docs/reference/publishing-checklist.md`-д шийдвэр гаргаж, үл мэдэгдэх асуудлууд шийдэгдсэний дараа энэ төлөвлөгөөг шинэчил.
