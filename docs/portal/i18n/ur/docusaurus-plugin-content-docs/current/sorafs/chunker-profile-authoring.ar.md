# چنکر فائل کی تصنیف ڈائریکٹری SoraFS میں

یہ گائیڈ وضاحت کرتا ہے کہ کس طرح SoraFS کے لئے نئی چنکر فائلوں کی تجویز اور شائع کی جائے۔
یہ آرکیٹیکچرل آر ایف سی (SF-1) اور لاگ حوالہ (SF-2A) کی تکمیل کرتا ہے
واضح تصنیف کی ضروریات ، توثیق کے اقدامات ، اور تجویز ٹیمپلیٹس کے ساتھ۔
ایک تائید شدہ مثال کے لئے ، دیکھیں
`docs/source/sorafs/proposals/sorafs_sf1_profile_v1.json`
میں خشک کرنے والی افادیت کو ریکارڈ کریں
`docs/source/sorafs/reports/sf1_determinism.md`۔

## جائزہ

رجسٹری میں داخل ہونے والی ہر فائل کو درج ذیل کو حاصل کرنا ہوگا:

- فن تعمیرات میں ایک جیسی سی ڈی سی پیرامیٹرز اور ملٹی ہش کی ترتیبات کا اعلان کرنا ؛
- شپنگ ری پلے ایبل فکسچر (JSON رسٹ/گو/ٹی ایس + کارپورا فوز + پور گواہ) ایس ڈی کے نیچے بہا سکتے ہیں
  کسٹم ٹولنگ کے بغیر تصدیق شدہ ؛
- ہجرت کے رہنما خطوط اور تعمیل ونڈوز کے ساتھ گورننس کے لئے تیار ڈیٹا (نام کی جگہ ، نام ، سیمور) شامل کریں۔ اور
- بورڈ کے جائزے سے پہلے لازمی ڈف پیکیج پاس کریں۔

ان قواعد کو پورا کرنے والی تجویز تیار کرنے کے لئے نیچے دیئے گئے چیک لسٹ کی پیروی کریں۔

## رجسٹری چارٹر کا خلاصہ

اس تجویز کو تیار کرنے سے پہلے ، یقینی بنائیں کہ یہ ریکارڈ کے چارٹر سے مماثل ہے جس پر آپ نافذ کررہے ہیں
`sorafs_manifest::chunker_registry::ensure_charter_compliance()`:

- فائل IDs مثبت عدد ہیں جو بغیر کسی فرق کے اجارہ داری میں اضافہ کرتے ہیں۔
- منظور شدہ ہینڈل (`namespace.name@semver`) متبادلات کی فہرست میں ظاہر ہونا چاہئے
  اور ہونا ** پہلا **۔ اس کے بعد پرانے متبادل (جیسے `sorafs.sf1@1.0.0`)۔
- کوئی عرف کسی دوسرے منحصر ہینڈل سے متصادم نہیں ہوسکتا ہے یا اس کی نقل نہیں کی جاسکتی ہے۔
الیاس کو خالی اور خالی جگہوں سے چھوٹا ہونا چاہئے۔

مفید CLI مددگار:

```bash
# قائمة JSON بكل descripors المسجلة (ids, handles, aliases, multihash)
cargo run -p sorafs_manifest --bin sorafs_manifest_chunk_store -- --list-profiles

# إخراج بيانات ملف افتراضي مرشح (handle معتمد + aliases)
cargo run -p sorafs_manifest --bin sorafs_manifest_chunk_store -- \
  --promote-profile=sorafs.sf1@1.0.0 --json-out=-
```

یہ احکامات تجاویز کو رجسٹر کے چارٹر کے مطابق رکھتے ہیں اور منظور شدہ ڈیٹا فراہم کرتے ہیں
حکمرانی کے مباحثوں کے لئے ضروری ہے۔

## مطلوبہ ڈیٹا

| فیلڈ | تفصیل | مثال (`sorafs.sf1@1.0.0`) |
| ------- | ------- | -------------------------------- |
| `namespace` | متعلقہ فائلوں کی منطقی گروپ بندی۔ | `sorafs` |
| `name` | انسانی پڑھنے کے قابل لیبل۔ | `sf1` |
| `semver` | پیرامیٹر سیٹ کا ایک معنوی ورژن تار۔ | `1.0.0` |
| `profile_id` | جب فائل داخل کی جاتی ہے تو ایک مونوٹونک عددی شناخت کنندہ تفویض کیا جاتا ہے۔ اگلی ID محفوظ رکھیں اور موجودہ نمبروں کو دوبارہ استعمال نہ کریں۔ | `1` |
| `profile_aliases` | مذاکرات کے دوران مؤکلوں کو اختیاری اضافی ہینڈلز (میراثی نام ، مخففات) دکھائے گئے۔ منظور شدہ ہینڈل کو پہلے شامل کیا جانا چاہئے۔ | `["sorafs.sf1@1.0.0"]` |
| `profile.min_size` | بائٹس میں کم سے کم حصہ کی لمبائی۔ | `65536` |
| `profile.target_size` | بائٹس میں ہدف کا حصہ لمبائی۔ | `262144` |

| `profile.break_mask` | انکولی ماسک جو رولنگ ہیش (ہیکس) کے ذریعہ استعمال ہوتا ہے۔ | `0x0000ffff` |
| `profile.polynomial` | فکسڈ گیئر کثیر الجہتی (ہیکس)۔ | `0x3da3358b4dc173` |

| `chunk_multihash.code` | ہضم کرنے کے لئے ملٹی ہاش کوڈ فی حصہ۔ | `0x1f` (Blake3-256) |
| `chunk_multihash.digest` | منظور شدہ فکسچر پیکیج کا ڈائجسٹ۔ | `13fa...c482` |

| `por_seed` | ڈٹرمینسٹک پور نمونوں کے لئے بیج (`splitmix64`)۔ | `0xfeedbeefcafebabe` (مثال) |میٹا ڈیٹا کو پروپوزل دستاویز میں اور ریکارڈ دستیاب ہونے کے لئے تیار کردہ فکسچر کے اندر نظر آنا چاہئے
دستی مماثل کے بغیر اقدار کی تصدیق کے لئے سی ایل آئی کو ٹولنگ اور خودکار گورننس۔ جب شک ہو ، بھاگیں
CLIS کے لئے CHUNK اسٹور اور `--json-out=-` کے ساتھ ظاہر ہوتا ہے
جائزہ نوٹ

### سی ایل آئی اور رجسٹری ٹچ پوائنٹس

- `sorafs_manifest_chunk_store --profile=<handle>` - دوبارہ چلانے کا حصہ اور ہضم ڈیٹا
  تجویز کردہ پیرامیٹرز کے ساتھ ظاہر اور پور چیک کے لئے۔
- `sorafs_manifest_chunk_store --json-out=-`- STDOUT پر نشر شدہ CHUNK اسٹور کی رپورٹ
  خودکار موازنہ کے لئے۔
- `sorafs_manifest_stub --chunker-profile=<handle>` - ظاہر اور کار کے منصوبوں کی تصدیق کریں
  منظور شدہ ہینڈل اور تبدیلیاں شامل ہیں۔
- `sorafs_manifest_stub --plan=-` - چیک کرنے کے لئے پچھلا `chunk_fetch_specs` کو ریفڈ کریں
  تبدیلی کے بعد آفسیٹس/ہضم۔

تجویز میں کمانڈ آؤٹ پٹس (ہضم ، پور کی جڑیں ، ظاہر ہیش) ریکارڈ کریں تاکہ ...
جائزہ لینے والے اسے دوبارہ زبانی طور پر پیش کرتے ہیں۔

## عزم اور تصدیق کی فہرست کی فہرست

1. ** فکسچر کو دوبارہ تخلیق کریں **
   ```bash
   cargo run --locked -p sorafs_chunker --bin export_vectors \
     --signature-out=fixtures/sorafs_chunker/manifest_signatures.json
   ```
2.
   کراس لینگویج (`crates/sorafs_chunker/tests/vectors.rs`) سبز رنگ میں نئے فکسچر کے ساتھ۔
3.
   (`fuzz/sorafs_chunker`) دوبارہ پیدا ہونے والے اثاثوں پر۔
4
   `sorafs_manifest_chunk_store --por-sample=<n>` تجویز کردہ فائل کا استعمال کرتے ہوئے اور روٹس میچ کی تصدیق کی
   فکسچر کے ساتھ ظاہر ہوتا ہے۔
5. ** سی آئی کے لئے خشک رن ** - مقامی طور پر `ci/check_sorafs_fixtures.sh` چلائیں ؛ یہ لازمی طور پر کامیاب ہونا چاہئے
   نئے فکسچر اور موجودہ `manifest_signatures.json` کے ساتھ۔
6. ** کراس رنٹائم کی تصدیق کریں **-اس بات کو یقینی بنائیں کہ گو/ٹی ایس ہک دوبارہ پیدا ہونے والے JSON اور نتائج کو کھائے
   حصہ اور ہضم کی حدیں ایک جیسی ہیں۔

تجویز میں نتیجے میں کمانڈز اور ہضموں کو دستاویز کریں تاکہ ٹولنگ ڈبلیو جی بغیر کسی اندازہ کے انہیں دوبارہ چلاسکے۔

### مینی فیسٹ/پور کی تصدیق کریں

فکسچر کو دوبارہ تخلیق کرنے کے بعد ، پوری مینی فیسٹ پائپ لائن کو چلائیں تاکہ یہ یقینی بنایا جاسکے کہ کار اور پور ڈیٹا مستقل رہے:

```bash
# التحقق من بيانات chunk + PoR مع الملف الجديد
cargo run -p sorafs_manifest --bin sorafs_manifest_chunk_store -- \
  --profile=sorafs.sf2@1.0.0 \
  --json-out=- --por-json-out=- fixtures/sorafs_chunker/input.bin

# توليد manifest + CAR والتقاط chunk fetch specs
cargo run -p sorafs_manifest --bin sorafs_manifest_stub -- \
  fixtures/sorafs_chunker/input.bin \
  --chunker-profile=sorafs.sf2@1.0.0 \
  --chunk-fetch-plan-out=chunk_plan.json \
  --manifest-out=sf2.manifest \
  --car-out=sf2.car \
  --json-out=sf2.report.json

# إعادة التشغيل باستخدام خطة fetch المحفوظة (تمنع offsets القديمة)
cargo run -p sorafs_manifest --bin sorafs_manifest_stub -- \
  fixtures/sorafs_chunker/input.bin \
  --chunker-profile=sorafs.sf2@1.0.0 \
  --plan=chunk_plan.json --json-out=-
```

ان پٹ فائل کو اپنے فکسچر میں استعمال ہونے والے کسی بھی نمائندہ کارپس کے ساتھ تبدیل کریں
(جیسے سائز 1 گیب کا ایک لازمی سلسلہ) اور اس کے نتیجے میں ہضم کو تجویز میں جوڑیں۔

## تجویز ٹیمپلیٹ

تجاویز کو Norito ریکارڈ کے طور پر پیش کیا گیا ہے جس میں `ChunkerProfileProposalV1` قسم کے اندر موجود ہیں
`docs/source/sorafs/proposals/`۔ نیچے JSON ٹیمپلیٹ متوقع نظر کو ظاہر کرتا ہے
(ضرورت کے مطابق اقدار کو تبدیل کریں):


مارک ڈاون کی متعلقہ رپورٹ (`determinism_report`) فراہم کریں جو کمانڈ آؤٹ پٹ اور ڈائجسٹ کو ریکارڈ اور کسی بھی کے لئے ریکارڈ کرتا ہے
توثیق کے دوران انحرافات۔

## گورننس ورک فلو

1. ** پروپوزل + فکسچر کے ساتھ PR جمع کرانا۔
   `chunker_registry_data.rs`۔
2. ** ٹولنگ ڈبلیو جی جائزہ۔
   رجسٹری کے قواعد کے مطابق (شناخت کنندگان کا دوبارہ استعمال نہیں ، عزم حاصل کیا جاتا ہے)۔
3. ** کونسل کا لفافہ۔ ** منظوری کے بعد ، کونسل کے ممبران مجوزہ ڈائجسٹ پر دستخط کرتے ہیں
   (`blake3("sorafs-chunker-profile-v1" || canonical_bytes)`) اور ان کے دستخطوں میں شامل کریں
   فائل لفافہ فکسچر کے ساتھ محفوظ ہے۔
4. ** ریکارڈ شائع کریں۔ ** انضمام ریکارڈ ، دستاویزات اور فکسچر کو اپ ڈیٹ کرتا ہے۔ یہ ڈیفالٹ سی ایل آئی رہتا ہے
   پچھلی فائل پر جب تک کہ گورننس اعلان نہ کرے کہ امیگریشن تیار ہے۔
   آپریٹرز کو ہجرت لیجر کے ذریعے مطلع کیا جاتا ہے۔

## تشکیل کے نکات- کنارے کے معاملات میں chunking سلوک کو کم کرنے کے ل two دو حدود کے اختیارات کو بھی ترجیح دیں۔
- منشور اور گیٹ وے صارفین کے ساتھ ہم آہنگی کے بغیر ملٹی ہش کوڈ کو تبدیل کرنے سے گریز کریں۔ اس کے بعد ایک اتفاق نوٹ شامل کریں۔
- گیئر ٹیبل کے بیج پڑھنے کے قابل لیکن آسان آڈیٹنگ کے لئے عالمی سطح پر منفرد بنائیں۔
- کسی بھی کارکردگی کی پیمائش کے نمونے (جیسے تھروپپٹ موازنہ) کے اندر ذخیرہ کریں
  بعد میں حوالہ کے لئے `docs/source/sorafs/reports/`۔

رول آؤٹ کے دوران پلے بیک کی توقعات کے لئے ہجرت لیجر دیکھیں
(`docs/source/sorafs/migration_ledger.md`)۔ رن ٹائم ملاپ کے قواعد کے لئے دیکھیں
`docs/source/sorafs/chunker_conformance.md`۔
