每當我們邀請審稿人加入時，路線圖都要求提供可驗證的證據
文檔門戶預覽。本說明解釋瞭如何使用以下方法捕獲證據
`preview-feedback-log` 幫助程序，如何與以下人員共享生成的工件
治理，以及浪潮結束後在哪裡提交摘要。

## 為什麼我們記錄反饋

- **合規性：** 每個邀請都必須記錄誰訪問了預覽、何時訪問
  承認可接受的使用政策，以及他們的訪問權限何時被撤銷。
- **遙測關聯：** 日誌讓我們可以關聯門戶遙測峰值
  或嘗試特定邀請群體的流量。
- **治理證據：** 每個 DOCS-SORA 檢查點都需要一個機器可讀的
  顯示邀請計數、開放審稿人以及傑出狀態的摘要
  問題。

所有證據均位於 `artifacts/docs_portal_preview/` 下，因此可以附加
無需抓取 CI 日誌即可進行邀請波票或治理審查。

## 直播邀請總結

`docs/portal/scripts/preview-feedback-summary.mjs` 攝取共享日誌並
在 `npm run build` 期間寫入 `src/data/previewFeedbackSummary.json`。桌子
下面呈現該 JSON，以便審閱者可以看到當前的邀請足跡
無需運行 CLI 幫助程序。



除了邀請總數之外，摘要現在還顯示致謝和
撤銷計數，得出的響應率（反饋提交除以
總邀請數），以及每次邀請之間的平均延遲（使用可懸停的 p95）
邀請和審稿人的確認或反饋提交。新的
**完成%** 列顯示已關閉的邀請數量（已確認或已確認）
撤銷），而 **Ack %** 突出顯示有多少收件人明確確認
可接受的使用步驟。這些派生百分比 — 加上僅 JSON
撤銷/問題率——提供給治理儀表板，以便審閱者可以確認
無需運行 CLI 幫助程序即可實現參與級別和周轉時間。
**狀態**列突出顯示波浪是否靜止
活躍 (`active`)，關閉但在 14 天監控窗口內 (`closed`)，或
準備存檔 (`archived`)。 **自上次事件以來的天數**指標顯示瞭如何
距離最後一次邀請/反饋行動已經過去很長時間了，以便議會能夠發現
需要結束或重新吸引揮之不去的接收者的波浪。

## 事件分類

|活動 |什麼時候記錄|筆記|
| ---| ---| ---|
| `invite-sent` |發送邀請電子郵件後立即。 |為每個收件人附加一個條目，以便邀請計數保持準確。 |
| `acknowledged` |當審閱者確認他們閱讀了安全性/可觀察性文檔並運行校驗和門時。 |可選，但建議合作夥伴群體使用。 |
| `feedback-submitted` |當審閱者通過商定的跟踪器提交調查結果時。 |在 `--notes` 字段中包含問題/鏈接。 |
| `issue-opened` |當 DOCS-SORA 提交來自反饋的後續問題時。 |幫助治理將路線圖項目關聯起來以預覽反饋。 |
| `access-revoked` |一旦審閱者的預覽窗口結束或訪問權限被提前撤銷。 |必需的，以便“開放收件人”計數保持準確。 |

如果您需要其他活動，請將其添加到
`VALID_EVENTS` 內部 `docs/portal/scripts/preview-feedback-log.mjs` 並更新
這個表在同樣的變化。

## 日誌記錄工作流程

`npm run preview:log` 背後的幫助程序編寫一個捕獲的 JSON 日誌
所有事件。每個條目包含波標籤、收件人、事件、時間戳和
可選註釋。示例：

```json
{
  "version": 1,
  "generated_at": "2026-02-04T18:20:15.321Z",
  "entries": [
    {
      "wave": "preview-20260204",
      "recipient": "alice@example.com",
      "event": "invite-sent",
      "timestamp": "2026-02-04T18:05:02.114Z",
      "notes": "seed reviewer"
    },
    {
      "wave": "preview-20260204",
      "recipient": "alice@example.com",
      "event": "feedback-submitted",
      "timestamp": "2026-02-05T09:14:43.009Z",
      "notes": "docs-preview/42"
    }
  ]
}
```

命令：

```bash
# Append a new event (defaults to artifacts/docs_portal_preview/feedback_log.json)
npm run --prefix docs/portal preview:log -- \
  --wave preview-20260204 \
  --recipient alice@example.com \
  --event invite-sent \
  --notes "seed reviewer"

# Produce a text summary for the current repo
npm run --prefix docs/portal preview:log -- --summary

# Emit machine-readable summary for a specific wave
npm run --prefix docs/portal preview:log -- \
  --summary --summary-json --filter-wave preview-20260204 \
  > artifacts/docs_portal_preview/preview-20260204-summary.json

# Regenerate the portal summary data file
npm run --prefix docs/portal generate:preview-summary

# Generate a Markdown digest skeleton for a wave
npm run --prefix docs/portal preview:digest -- \
  --wave preview-20260204 \
  --invite-start 2026-02-04T00:00:00Z \
  --invite-end 2026-02-18T00:00:00Z \
  --dashboard "https://grafana.example/d/docs-preview?var-wave=preview-20260204" \
  --out artifacts/docs_portal_preview/preview-20260204-digest.md
```

溫馨提示：

- 在功能分支中工作時將日誌置於版本控制之下，以便其他
  維護者可以審查更改。對於長時間運行的波形，將日誌存儲在
  治理票證並將最終版本附加到發布工件中。
- 當您需要直接登錄時，`preview:log` 接受 `--log <path>`
  SoraFS 捆綁包或 CI 工作區。
- 始終包含 `--notes` 提示（問題鏈接、GitHub 句柄、群組 ID），以便
  總結在回顧期間很有用。
- 當您需要確定性摘要時間戳時（對於 CI 或文檔快照），
  在運行摘要之前設置 `PREVIEW_FEEDBACK_NOW=<iso8601 timestamp>`
  幫手。這使 `generatedAt`、狀態和日計數器保持一致
  跨越重建。

## 摘要、摘要和檔案1. **每波摘要：** 生成 JSON 摘要並將其附加到邀請中
   票證與原始日誌一起。摘要列舉了邀請總數、已開放
   收件人和反饋很重要，因此治理可以確認我們捕獲了每一個
   審稿人。門戶構建還在以下位置發布了匯總摘要：
   `src/data/previewFeedbackSummary.json`，以便利益相關者可以看到當前
   無需運行 CLI 即可生成快照。
2. **每週摘要：** 運行 `npm run --prefix docs/portal preview:digest -- --wave <label>`
   預先填充 Markdown 指標，然後定制佔位符部分
   使用中的模板
   [`docs/examples/docs_preview_feedback_digest.md`](../../../examples/docs_preview_feedback_digest.md)
   在與利益相關者分享之前。至少包括：波標籤、開放
   審稿人、值得注意的發現和下一步行動。
3. **歸檔：**將封閉波日誌移入
   `artifacts/docs_portal_preview/archive/<wave>/` 並僅保留最新的
   根部有源波。這使得 CI 偽影很小，同時保留了
   審計追踪。

## 鏈接到路線圖

- 每當您詢問時，請參考邀請和入職手冊中的此文檔
  審稿人提交反饋（參見
  [`preview-invite-flow`](./preview-invite-flow.md),
  [`reviewer-onboarding`](./reviewer-onboarding.md)，以及
  [`public-preview-invite`](./public-preview-invite.md))。
- 關閉波形時在 `status.md` 中提及生成的摘要，以便 DOCS-SORA
  路線圖項目（例如 DOCS-1b/2b/3c）可以引用工件。
- 上傳到 SoraFS 等外部時，將哈希摘要與摘要一起存儲
  審閱者可以驗證證據包。

將日誌記錄工作流程編碼在此處可以解除對路線圖要求的阻礙
“捕獲反饋/遙測以進行治理”並減少花費的時間
在每個里程碑重建邀請歷史。
