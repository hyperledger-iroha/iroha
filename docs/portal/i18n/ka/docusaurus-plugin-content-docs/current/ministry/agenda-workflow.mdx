## მიმოხილვა

საგზაო რუკის პუნქტი **MINFO-2** მოითხოვს განმსაზღვრელ გზას წინადადების ლინკიდან
საჯარო დახარისხების მტკიცებულებები, რათა ნებისმიერს შეეძლოს დაადასტუროს, როგორ მიაღწია საქმეს დღის წესრიგში
საბჭო. ეს გვერდი პირდაპირ აერთებს ადრე ჩამოსული CLI დამხმარეებს
დეველოპერის პორტალი, ასე რომ შემოთავაზებულებს, აუდიტორებს და პლატფორმის გუნდებს შეუძლიათ იგივე მიბაძონ
საკონტროლო სია მონო-რეპოს მართლწერის გარეშე.

სამუშაო პროცესი მოიცავს სამ განმეორებად ნაბიჯს:

1. დაადასტურეთ `AgendaProposalV1` დატვირთვა ლოკალურად და უარყოთ დუბლიკატი ჰეშები
   სანამ წარდგენა მიაღწევს მოდერაციის რიგს.
2. გამოაქვეყნეთ რეესტრის დუბლიკატი ჰიტები წინადადებასთან ერთად, რათა საბჭოს თანამშრომლებმა შეძლონ
   ადრეული შემთხვევების კვალი დეტერმინიზმის დაკარგვის გარეშე.
3. გაუშვით დახარისხების დამხმარე, აიღეთ Merkle-ის მტკიცებულების ნაკრები და მოაყარეთ ზედაპირი
   შერჩეული სია პორტალში, რათა მოქალაქეებმა შეძლონ გათამაშების ხელახლა თამაში.

ყველა დამხმარე ცხოვრობს `cargo xtask ministry-agenda`-ში; Norito სქემა არის
დოკუმენტირებულია `docs/source/ministry/agenda_council_proposal.md`-ით.

## 1. დაადასტურეთ წინადადებები ადგილობრივად

გამოიყენეთ CLI JSON დატვირთვის დასამაგრებლად და საჭირო მტკიცებულებების დასამტკიცებლად (ჰეშ
ოჯახები, დაიჯესტის სიგრძე, ენის ტეგი, გამომგზავნის კონტაქტი) არსებობს. The
ვალიდატორი აზიარებს კოდს Torii გადატანის გზასთან, ასე რომ, წინადადება გაივლის
აქაც გაივლის პორტალზე ატვირთვისას.

<CodeBlock language="bash">{`cargo xtask ministry-agenda დადასტურება \\
  --proposal docs/examples/ministry/agenda_proposal_example.json \\
  --registry docs/examples/ministry/agenda_duplicate_registry.json`}</CodeBlock>

- დაამატეთ `--allow-registry-conflicts` ისტორიული დატვირთვების ტრიაჟისას; ახალი
  წარდგენები სწრაფად უნდა ჩავარდეს, როდესაც თითის ანაბეჭდი უკვე არსებობს.
- მაგალითების დირექტორიაში გაგზავნილია საცნობარო წინადადება და რეესტრი, რათა SDK-ებმა შეძლონ
  მიიღეთ ფორმატი მყარი კოდირების პირადი დატვირთვის გარეშე.

## 2. გამოაქვეყნეთ დუბლიკატი რეესტრის კონტექსტი

ყველა მიღებული წინადადება მოიცავს რეესტრის დაიჯესტს, რომელიც ადრე იყო კონსულტირებული
ჩაყლაპავს. გადაიღეთ SoraFS CID ან Torii მანიფესტის გზა ამ რეესტრისთვის და დაამატეთ
ეს თქვენს წარდგენის მეტამონაცემებზე, რათა საბჭოს მიმომხილველებმა შეძლონ იგივე JSON-ის გადამოწმება.

```json title="Duplicate registry (excerpt)"
{
  "entries": [
    {
      "hash_family": "blake3-256",
      "hash_hex": "0d714bed4b7c63c23a2cf8ee9ce6c3cde1007907c427b4a0754e8ad31c91338d",
      "proposal_id": "AC-2025-014",
      "note": "Already handled in 2025-08 incident"
    }
  ]
}
```

როდესაც CLI იტყობინება შეჯახების შესახებ, პორტალი აჩვენებს დაკავშირებულ შეთავაზების ID-ებს
ოპერატორებს შეუძლიათ გაერთიანდნენ ან უარყონ ხელახალი გაგზავნა დეტერმინისტულად.

## 3. გაუშვით და გამოაქვეყნეთ დახარისხების შეჯამება

მას შემდეგ რაც დოკეტი მზად იქნება, გაუშვით დეტერმინისტული დახარისხების დამხმარე. ის მოიხმარს
სია JSON, თესლის და სამიზნე სლოტების რაოდენობა, შემდეგ გამოსცემს Merkle-proof მხარდაჭერილს
შეჯამება. JSON-ის შეჯამება არის ის, რასაც პორტალი იძლევა **საბჭოს გათამაშებაში**
პანელი.

<CodeBlock language="bash">{`cargo xtask ministry-agenda sortition \\
  -- ჩამონათვალი docs/examples/ministry/agenda_council_roster.json \\
  --სლოტები 3 \\
  --seed 0x0123456789abcdef0123456789abcdef0123456789abcdef0123456789abcdef \\
  --output artifacts/ministry/sortition/2026-03-12.json`}</CodeBlock>

შეინახეთ მიღებული JSON SoraFS-ზე (ან ხელმოწერილი დღის წესრიგის პაკეტთან ერთად) და
დააკავშირეთ CID წინადადების ტრეკერში. დეველოპერის პორტალი იგივეს მოიხმარს
JSON სტრუქტურა; ქვემოთ მოყვანილი მაგალითი ასახავს შეფუთულს
`agenda_sortition_summary_example.json`.

### გადაწყვეტილების ჟურნალი

როდესაც საბჭო წინადადებას უყრის კენჭს, დაურთოს გადაწყვეტილების მემორანდუმი და
დახარისხების შეჯამება CID მმართველობის მოქმედებისთვის (Torii `/v1/gov/*` ჩანაწერი). პორტალი
გვერდები აკავშირებს შენიშვნას, ხმების რაოდენობას და JSON-ს, რომელიც ზემოთ არის ნაჩვენები ყველა ქმედებაზე
შეიცავს **MINFO-2**-ის მიერ მოთხოვნილ მტკიცებულებების სრულ ჯაჭვს.
