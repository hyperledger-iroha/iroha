# მრავალწყაროების მისაღებად ორკესტრატორის სახელმძღვანელო

SoraFS მრავალ წყაროს მოტანის ორკესტრატორი დეტერმინისტული, პარალელურად მოძრაობს
ჩამოტვირთვები პროვაიდერის ნაკრებიდან, რომელიც გამოქვეყნებულია მმართველობით მხარდაჭერილ რეკლამებში. ეს
სახელმძღვანელო განმარტავს, თუ როგორ უნდა დააკონფიგურიროთ ორკესტრი, რა წარუმატებლობის სიგნალები უნდა ველოდოთ
გავრცელების დროს და რომელი ტელემეტრიული ნაკადები ავლენს ჯანმრთელობის ინდიკატორებს.

## 1. კონფიგურაციის მიმოხილვა

ორკესტრატორი აერთიანებს კონფიგურაციის სამ წყაროს:

| წყარო | დანიშნულება | შენიშვნები |
|--------|--------|-------|
| `OrchestratorConfig.scoreboard` | ახდენს პროვაიდერის წონების ნორმალიზებას, ამოწმებს ტელემეტრიის სიახლეს და აგრძელებს JSON ანგარიშების დაფას, რომელიც გამოიყენება აუდიტებისთვის. | მხარს უჭერს `crates/sorafs_car::scoreboard::ScoreboardConfig`. |
| `OrchestratorConfig.fetch` | იყენებს მუშაობის დროის ლიმიტებს (ბიუჯეტების ხელახლა ცდა, კონკურენტულობის საზღვრები, დადასტურების გადართვები). | რუქები `FetchOptions`-ზე `crates/sorafs_car::multi_fetch`-ში. |
| CLI / SDK პარამეტრები | შეზღუდეთ თანატოლების რაოდენობა, მიამაგრეთ ტელემეტრიის რეგიონები და ზედაპირის უარყოფის/გაძლიერების პოლიტიკა. | `sorafs_cli fetch` პირდაპირ ამხელს ამ დროშებს; SDK-ები აკავშირებენ მათ `OrchestratorConfig`-ის საშუალებით. |

JSON დამხმარეები `crates/sorafs_orchestrator::bindings`-ში ასრულებენ მთლიან სერიებს
კონფიგურაცია Norito JSON-ში, რაც მას პორტატულს ხდის SDK საკინძებში და
ავტომატიზაცია.

### 1.1 JSON კონფიგურაციის ნიმუში

```json
{
  "scoreboard": {
    "latency_cap_ms": 6000,
    "weight_scale": 12000,
    "telemetry_grace_secs": 900,
    "persist_path": "/var/lib/sorafs/scoreboards/latest.json"
  },
  "fetch": {
    "verify_lengths": true,
    "verify_digests": true,
    "retry_budget": 4,
    "provider_failure_threshold": 3,
    "global_parallel_limit": 8
  },
  "telemetry_region": "iad-prod",
  "max_providers": 6,
  "transport_policy": "soranet_first"
}
```

ფაილის შენარჩუნება ჩვეულებრივი `iroha_config` ფენით (`defaults/`, მომხმარებელი,
ფაქტობრივი) ასე რომ, დეტერმინისტული განლაგება მემკვიდრეობით იღებს იგივე საზღვრებს კვანძებში.
მხოლოდ პირდაპირი სარეზერვო პროფილისთვის, რომელიც შეესაბამება SNNet-5a გამოშვებას,
მიმართეთ `docs/examples/sorafs_direct_mode_policy.json`-ს და კომპანიონს
სახელმძღვანელო `docs/source/sorafs/direct_mode_pack.md`-ში.

### 1.2 შესაბამისობის უგულებელყოფა

SNNet-9 ორკესტრატორს ავრცელებს მმართველობაზე ორიენტირებულ შესაბამისობას. ახალი
`compliance` ობიექტი Norito JSON კონფიგურაციაში აღწერს ამოკვეთებს
რაც აიძულებს გამოტანის მილსადენს მხოლოდ პირდაპირ რეჟიმში:

```json
"compliance": {
  "operator_jurisdictions": ["US", "JP"],
  "jurisdiction_opt_outs": ["US"],
  "blinded_cid_opt_outs": [
    "C6B434E5F23ABD318F01FEDB834B34BD16B46E0CC44CD70536233A632DFA3828"
  ],
  "audit_contacts": ["mailto:compliance@example.org"]
}
```

- `operator_jurisdictions` აცხადებს ISO‑3166 alpha‑2 კოდებს, სადაც ეს
  ორკესტრის ინსტანცია მოქმედებს. დროს კოდები ნორმალიზდება დიდზე
  გარჩევა.
- `jurisdiction_opt_outs` ასახავს მმართველობის რეესტრს. როცა რომელიმე ოპერატორი
  იურისდიქცია ჩნდება სიაში, ორკესტრი აღასრულებს
  `transport_policy=direct-only` და გამოსცემს პოლიტიკის სარეზერვო მიზეზს
  `compliance_jurisdiction_opt_out`.
- `blinded_cid_opt_outs` ჩამოთვლის მანიფესტების დაიჯესტებს (დაბრმავებული CID, დაშიფრული როგორც
  დიდი თექვსმეტობითი). შესატყვისი დატვირთვა ასევე აიძულებს პირდაპირ დაგეგმვას და
  მოამზადეთ `compliance_blinded_cid_opt_out` სარეზერვო ტელემეტრიაში.
- `audit_contacts` ჩაწერს URI-ების მმართველობას, რომელსაც ოპერატორები აქვეყნებენ
  მათი GAR სათამაშო წიგნები.
- `attestations` ასახავს ხელმოწერილი შესაბამისობის პაკეტებს, რომლებიც მხარს უჭერენ პოლიტიკას.
  თითოეული ჩანაწერი განსაზღვრავს არასავალდებულო `jurisdiction` (ISO-3166 ალფა-2 კოდი), a
  `document_uri`, კანონიკური 64-სიმბოლოიანი `digest_hex`, გამოშვება
  დროის შტამპი `issued_at_ms` და სურვილისამებრ `expires_at_ms`. ეს არტეფაქტები
  შეიტანეთ ორკესტრატორის აუდიტის საკონტროლო სიაში, რათა მმართველობის ინსტრუმენტებმა შეძლონ ბმული
  გადაფარავს ხელმოწერილ დოკუმენტებს.

მიაწოდეთ შესაბამისობის ბლოკი ჩვეულებრივი კონფიგურაციის ფენის მეშვეობით ოპერატორებისთვის
მიიღეთ დეტერმინისტული გადაფარვები. ორკესტრატორი მიმართავს შესაბამისობას _ შემდეგ
მინიშნებები ჩაწერის რეჟიმში: მაშინაც კი, თუ SDK ითხოვს `upload-pq-only`, იურისდიქციის ან
მანიფესტური უარის თქმა კვლავ უბრუნდება მხოლოდ პირდაპირ ტრანსპორტს და სწრაფად ვერ ხერხდება, როდესაც არა
არსებობს შესაბამისი პროვაიდერები.

Canonical უარის თქმის კატალოგები ხელმისაწვდომია
`governance/compliance/soranet_opt_outs.json`; მმართველი საბჭო ავრცელებს
განახლებები მონიშნული რელიზების საშუალებით. სრული მაგალითის კონფიგურაცია (მათ შორის
ატესტაციები) ხელმისაწვდომია `docs/examples/sorafs_compliance_policy.json`-ში და
ოპერაციული პროცესი აღბეჭდილია
[GAR შესაბამისობის სათამაშო წიგნი] (../../../source/soranet/gar_compliance_playbook.md).

### 1.3 CLI & SDK ღილაკები

| დროშა / ველი | ეფექტი |
|--------------|--------|
| `--max-peers` / `OrchestratorConfig::with_max_providers` | ზღუდავს რამდენი პროვაიდერი გადარჩება შედეგების დაფის ფილტრს. დააყენეთ `None`-ზე, რათა გამოიყენოთ ყველა შესაბამისი პროვაიდერი. |
| `--retry-budget` / `FetchOptions::per_chunk_retry_limit` | Caps განმეორებითი ცდები თითო ნაწილზე. ლიმიტის გადაჭარბება ზრდის `MultiSourceError::ExhaustedRetries`. |
| `--telemetry-json` | შეაქვს შეყოვნება/შეცდომის სნეპშოტები შედეგების დაფის შემქმნელში. მოძველებული ტელემეტრია `telemetry_grace_secs`-ს მიღმა აღნიშნავს პროვაიდერებს, რომლებიც დაუშვებელია. |
| `--scoreboard-out` | შენარჩუნებულია გამოთვლილი შედეგების დაფა (დასაშვები + შეუსაბამო პროვაიდერები) ინსპექტირების შემდგომი შემოწმებისთვის. |
| `--scoreboard-now` | უგულებელყოფს ქულების დაფის დროის ნიშანს (Unix წამი), ასე რომ, მატჩების აღბეჭდვა დეტერმინისტული რჩება. |
| `--deny-provider` / ქულის პოლიტიკის კაკალი | დეტერმინისტულად გამორიცხეთ პროვაიდერები დაგეგმვისგან რეკლამის წაშლის გარეშე. სასარგებლოა სწრაფი რეაგირების შავ სიაში. |
| `--boost-provider=name:delta` | დაარეგულირეთ პროვაიდერისთვის შეწონილი მრგვალი კრედიტები, ხოლო მმართველობის წონა ხელუხლებლად დატოვეთ. |
| `--telemetry-region` / `OrchestratorConfig::with_telemetry_region` | ლეიბლები ასხივებდნენ მეტრიკას და სტრუქტურირებულ ჟურნალებს, რათა საინფორმაციო დაფებს შეეძლოთ გადაადგილება გეოგრაფიის ან გავრცელების ტალღის მიხედვით. |
| `--transport-policy` / `OrchestratorConfig::with_transport_policy` | ნაგულისხმევად არის `soranet-first` ახლა, როდესაც მრავალწყაროიანი ორკესტრატორი საბაზისოა. გამოიყენეთ `direct-only` რეიტინგის დაქვეითების ან შესაბამისობის დირექტივის შესრულებისას და შეინახეთ `soranet-strict` მხოლოდ PQ პილოტებისთვის; შესაბამისობის უგულებელყოფა კვლავ მოქმედებს როგორც მყარი ჭერი. |

SoraNet-first ახლა მიწოდების ნაგულისხმევია და უკან დაბრუნებაში უნდა იყოს მითითებული SNNet-ის შესაბამისი ბლოკერი. მას შემდეგ, რაც SNNet-4/5/5a/5b/6a/7/8/12/13 დაამთავრებს, მმართველობა დააყენებს საჭირო პოზას წინ (`soranet-strict`-ისკენ); მანამდე, მხოლოდ ინციდენტზე დაფუძნებულმა უგულებელყოფებმა უნდა მიენიჭოს `direct-only` პრიორიტეტი და ისინი უნდა ჩაიწეროს გაშვების ჟურნალში.

ყველა ზემოთ მოყვანილი დროშა იღებს `--` სტილის სინტაქსს როგორც `sorafs_cli fetch`-ში, ასევე
დეველოპერის წინაშე დგას `sorafs_fetch` ორობითი. SDK-ები ასახავს იგივე ვარიანტებს აკრეფის საშუალებით
მშენებლები.

### 1.4 დაცვის ქეში მენეჯმენტი

CLI ახლა მიდის SoraNet მცველის ამომრჩეველში, რათა ოპერატორებს შეეძლოთ შესვლის ჩამაგრება
რელეები განმსაზღვრელია SNNet-5 სატრანსპორტო სრულ გაშვებამდე. სამი
ახალი დროშები აკონტროლებენ სამუშაო პროცესს:

| დროშა | დანიშნულება |
|------|---------|
| `--guard-directory <PATH>` | მიუთითებს JSON ფაილზე, რომელიც აღწერს უახლეს სარელეო კონსენსუსს (ქვეჯგუფი ნაჩვენებია ქვემოთ). კატალოგის გადაცემა განაახლებს მცველის ქეშის მიღებამდე. |
| `--guard-cache <PATH>` | შენარჩუნებულია Norito-ში კოდირებული `GuardSet`. შემდგომი გაშვებები ხელახლა გამოიყენებს ქეშს მაშინაც კი, როცა ახალი დირექტორია არ არის მიწოდებული. |
| `--guard-target <COUNT>` / `--guard-retention-days <DAYS>` | არასავალდებულო უგულებელყოფა შესვლის მცველების რაოდენობის დასამაგრებლად (ნაგულისხმევი 3) და შეკავების ფანჯრის (ნაგულისხმევი 30 დღე). |
| `--guard-cache-key <HEX>` | არასავალდებულო 32-ბაიტიანი კლავიატურა, რომელიც გამოიყენება ქეშების დასაცავად Blake3 MAC-ით, რათა შესაძლებელი იყოს ფაილის გადამოწმება ხელახლა გამოყენებამდე. |

მცველის დირექტორიას დატვირთვა იყენებს კომპაქტურ სქემას:

`--guard-directory` დროშა ახლა მოელის Norito-ში დაშიფრულს
`GuardDirectorySnapshotV2` დატვირთვა. ბინარული სნეპშოტი შეიცავს:

- `version` — სქემის ვერსია (ამჟამად `2`).
- `directory_hash`, `published_at_unix`, `valid_after_unix`, `valid_until_unix` — კონსენსუსი
  მეტამონაცემები, რომლებიც უნდა შეესაბამებოდეს ყველა ჩაშენებულ სერტიფიკატს.
- `validation_phase` — სერტიფიკატის პოლიტიკის კარიბჭე (`1` = დაუშვას ერთი Ed25519 ხელმოწერა,
  `2` = უპირატესობას ანიჭებს ორმაგ ხელმოწერას, `3` = მოითხოვს ორმაგ ხელმოწერას).
- `issuers` — მმართველობის ემიტენტები `fingerprint`, `ed25519_public` და `mldsa65_public`.
  თითის ანაბეჭდები გამოითვლება როგორც
  `BLAKE3("soranet.src.v2.issuer" || ed25519 || u32(len(ml-dsa)) || ml-dsa)`.
- `relays` — SRCv2 პაკეტების სია (`RelayCertificateBundleV2::to_cbor()` გამომავალი). თითოეული შეკვრა
  ატარებს რელეს აღწერს, შესაძლებლობების დროშებს, ML-KEM პოლიტიკას და ორმაგ Ed25519/ML-DSA-65
  ხელმოწერები.

CLI ამოწმებს ყველა პაკეტს დეკლარირებული ემიტენტის გასაღებებთან მიმართებაში, სანამ დირექტორიას შეუერთდება

გამოიძახეთ CLI `--guard-directory`-ით უახლესი კონსენსუსის შერწყმის მიზნით
არსებული ქეში. სელექციონერი ინახავს მიმაგრებულ მცველებს, რომლებიც ჯერ კიდევ შიგნით არიან
შენახვის ფანჯარა და უფლებამოსილი დირექტორიაში; ახალი რელეების შეცვლა ვადაგასულია
ჩანაწერები. წარმატებული მიღების შემდეგ განახლებული ქეში იწერება უკან გზაზე
მიწოდებული `--guard-cache`-ის მეშვეობით, რაც შემდგომი სესიების განმსაზღვრელია. SDK-ები
შეუძლია იგივე ქცევის რეპროდუცირება დარეკვით
`GuardSelector::select(&RelayDirectory, existing_guard_set, now_unix_secs)` და
მიღებული `GuardSet` ხრახნიანი `SorafsGatewayFetchOptions`-მდე.

`ml_kem_public_hex` საშუალებას აძლევს სელექტორს, პრიორიტეტი მიანიჭოს PQ მცველებს
SNNet-5-ის გაშვება. სტადიის გადართვები (`anon-guard-pq`, `anon-majority-pq`,
`anon-strict-pq`) ახლა კლასიკური რელეების ავტომატურად ჩამორთმევა: როდესაც PQ მცველია
ხელმისაწვდომია სელექტორი ჩამოაგდებს ზედმეტ კლასიკურ ქინძისთავებს, ამიტომ შემდგომი სესიები ხელს უწყობს
ჰიბრიდული ხელის ჩამორთმევა. CLI/SDK შეჯამებები ასახავს შედეგად მიქსს
`anonymity_status`/`anonymity_reason`, `anonymity_effective_policy`,
`anonymity_pq_selected`,
`anonymity_classical_selected`, `anonymity_pq_ratio`,
`anonymity_classical_ratio` და კომპანიონი კანდიდატი/დეფიციტი/მიწოდების დელტა
ველები, რაც მკაფიოდ აქცევს ბრაუნაუტს და კლასიკურ ჩანაცვლებებს.

მცველთა დირექტორიაში ახლა შეიძლება ჩაშენდეს სრული SRCv2 პაკეტი
`certificate_base64`. ორკესტრატორი დეკოდირებს ყველა პაკეტს, ხელახლა ამოწმებს მას
Ed25519/ML-DSA ხელს აწერს და ინარჩუნებს გაანალიზებულ სერტიფიკატს გვერდით
დაცვის ქეში. როდესაც სერთიფიკატი არსებობს, ის ხდება კანონიკური წყარო
PQ გასაღებები, ხელის ჩამორთმევის კომპლექტის პრეფერენციები და წონა; ვადაგასული სერთიფიკატებია
გავრცელდება მიკროსქემის სასიცოცხლო ციკლის მენეჯმენტის მეშვეობით და ჩნდება ზედაპირზე
`telemetry::sorafs.guard` და `telemetry::sorafs.circuit`, რომლებიც ჩაწერენ
მოქმედების ფანჯარა, ხელის ჩამორთმევის ლუქსი და დაფიქსირდა თუ არა ორმაგი ხელმოწერები
თითოეული მცველი.

გამოიყენეთ CLI დამხმარეები გამომცემლებთან სნეპშოტების სინქრონიზაციისთვის:```bash
sorafs_cli guard-directory fetch \
  --url https://directory.soranet.dev/mainnet_snapshot.norito \
  --output ./state/guard_directory.norito \
  --expected-directory-hash <directory-hash-hex>

sorafs_cli guard-directory verify \
  --path ./state/guard_directory.norito \
  --expected-directory-hash <directory-hash-hex>
```

`fetch` ჩამოტვირთავს და ამოწმებს SRCv2 სნეპშოტს დისკზე ჩაწერამდე,
ხოლო `verify` იმეორებს ვალიდაციის მილსადენის არტეფაქტებს, რომლებიც მოპოვებულია სხვაგან
გუნდები, ასხივებენ JSON შეჯამებას, რომელიც ასახავს CLI/SDK მცველის ამომრჩევლის გამომავალს.

### 1.5 მიკროსქემის სასიცოცხლო ციკლის მენეჯერი

როდესაც უზრუნველყოფილია როგორც სარელეო დირექტორია, ასევე დაცვის ქეში, ორკესტრი
ააქტიურებს მიკროსქემის სასიცოცხლო ციკლის მენეჯერს SoraNet სქემების წინასწარ აშენებისა და განახლებისთვის
ყოველი მოტანის წინ. კონფიგურაცია მუშაობს `OrchestratorConfig`-ში
(`crates/sorafs_orchestrator/src/lib.rs:305`) ორი ახალი ველის მეშვეობით:

- `relay_directory`: ატარებს SNNet-3 დირექტორიას სნეპშოტს ისე, რომ შუა/გასასვლელი გადახტება
  შეიძლება შეირჩეს დეტერმინისტულად.
- `circuit_manager`: სურვილისამებრ კონფიგურაცია (ჩართულია ნაგულისხმევად) აკონტროლებს
  წრე TTL.

Norito JSON ახლა იღებს `circuit_manager` ბლოკს:

```json
"circuit_manager": {
  "enabled": true,
  "circuit_ttl_secs": 900
}
```

SDK-ები გადასცემს დირექტორიაში მონაცემებს
`SorafsGatewayFetchOptions::relay_directory`
(`crates/iroha/src/client.rs:320`) და CLI მას ავტომატურად აერთებს ნებისმიერ დროს
მოწოდებულია `--guard-directory` (`crates/iroha_cli/src/commands/sorafs.rs:365`).

მენეჯერი განაახლებს სქემებს, როდესაც იცვლება მცველი მეტამონაცემები (ბოლო წერტილი, PQ გასაღები,
ან ჩამაგრებული დროის შტამპი) ან TTL გადის. დამხმარე `refresh_circuits`
გამოძახებული ყოველი ჩამოტვირთვის წინ (`crates/sorafs_orchestrator/src/lib.rs:1346`)
ასხივებს `CircuitEvent` ჟურნალებს, რათა ოპერატორებმა შეძლონ სიცოცხლის ციკლის გადაწყვეტილებების კვალი. გაჟღენთილი
ტესტი `circuit_manager_latency_soak_remains_stable_across_rotations`
(`crates/sorafs_orchestrator/src/soranet.rs:1479`) აჩვენებს სტაბილურ შეყოვნებას
სამი მცველის ბრუნვის გასწვრივ; იხილეთ თანდართული ანგარიში
`docs/source/soranet/reports/circuit_stability.md:1`.

### 1.6 ადგილობრივი QUIC პროქსი

ორკესტრატორს სურვილისამებრ შეუძლია შექმნას ადგილობრივი QUIC პროქსი, ბრაუზერის გაფართოების მიზნით
და SDK ადაპტერებს არ სჭირდებათ სერთიფიკატების მართვა ან ქეშის გასაღებების დაცვა. The
პროქსი უკავშირდება loopback მისამართს, წყვეტს QUIC კავშირებს და აბრუნებს a
Norito მანიფესტი, რომელიც აღწერს სერთიფიკატს და არჩევით დამცავ ქეშის გასაღებს
კლიენტი. სატრანსპორტო ღონისძიებები, რომლებიც გამოქვეყნებულია პროქსის მიერ, დათვლილია მეშვეობით
`sorafs_orchestrator_transport_events_total`.

ჩართეთ პროქსი ახალი `local_proxy` ბლოკის მეშვეობით ორკესტრატორ JSON-ში:

```json
"local_proxy": {
  "bind_addr": "127.0.0.1:9443",
  "telemetry_label": "dev-proxy",
  "guard_cache_key_hex": "0123456789ABCDEF0123456789ABCDEF0123456789ABCDEF0123456789ABCDEF",
  "emit_browser_manifest": true,
  "proxy_mode": "bridge",
  "prewarm_circuits": true,
  "max_streams_per_circuit": 64,
  "circuit_ttl_hint_secs": 300,
  "norito_bridge": {
    "spool_dir": "./storage/streaming/soranet_routes",
    "extension": "norito"
  },
  "kaigi_bridge": {
    "spool_dir": "./storage/streaming/soranet_routes",
    "extension": "norito",
    "room_policy": "public"
  }
}
```

- `bind_addr` აკონტროლებს სად უსმენს პროქსი (გამოიყენეთ `0` პორტი მოთხოვნისთვის
  ეფემერული პორტი).
- `telemetry_label` ვრცელდება მეტრიკაში, რათა დაფებმა განასხვავონ
  მარიონეტები გამოტანის სესიებიდან.
- `guard_cache_key_hex` (არასავალდებულო) საშუალებას აძლევს პროქსის ზედაპირზე იგივე ღილაკით დამცავი
  ქეში, რომელსაც ეყრდნობა CLI/SDK-ები, ბრაუზერის გაფართოებების სინქრონიზაციას.
- `emit_browser_manifest` ცვლის, აბრუნებს თუ არა ხელის ჩამორთმევა მანიფესტს
  გაფართოებებს შეუძლიათ შეინახონ და დაადასტურონ.
- `proxy_mode` ირჩევს თუ არა პროქსი ხიდის მოძრაობას ადგილობრივად (`bridge`) ან
  მხოლოდ ასხივებს მეტამონაცემებს, რათა SDK-ებმა თავად გახსნან SoraNet სქემები
  (`metadata-only`). პროქსი ნაგულისხმევად არის `bridge`; დააყენეთ `metadata-only` როცა ა
  სამუშაო სადგურმა უნდა აჩვენოს მანიფესტი ნაკადების გადაცემის გარეშე.
- `prewarm_circuits`, `max_streams_per_circuit` და `circuit_ttl_hint_secs`
  მოამზადეთ დამატებითი მინიშნებები ბრაუზერისთვის, რათა მან შეძლოს პარალელური ნაკადების ბიუჯეტი და
  გაიგეთ რამდენად აგრესიულად იყენებს პროქსი სქემებს.
- `car_bridge` (სურვილისამებრ) მიუთითებს ადგილობრივ CAR არქივის ქეშზე. `extension`
  ველი აკონტროლებს სუფიქსს, რომელიც მიმაგრებულია, როდესაც ნაკადის სამიზნე გამოტოვებს `*.car`; კომპლექტი
  `allow_zst = true` წინასწარ შეკუმშული `*.car.zst` დატვირთვის პირდაპირ მოსამსახურებლად.
- `kaigi_bridge` (არასავალდებულო) ავლენს კაიგის გაფუჭებულ მარშრუტებს პროქსისკენ. The
  `room_policy` ველი რეკლამირებს, მუშაობს თუ არა ხიდი `public` ან
  `authenticated` რეჟიმი, რათა ბრაუზერის კლიენტებს შეეძლოთ წინასწარ აირჩიონ სწორი GAR ეტიკეტები.
- `sorafs_cli fetch` ამხელს `--local-proxy-mode=bridge|metadata-only` და
  `--local-proxy-norito-spool=PATH` უგულებელყოფს, რაც ოპერატორებს საშუალებას აძლევს გადართონ
  გაშვების რეჟიმი ან მიუთითეთ ალტერნატიულ კოჭებზე JSON პოლიტიკის შეცვლის გარეშე.
- `downgrade_remediation` (არასავალდებულო) აკონფიგურირებს ავტომატური დაქვეითების კაუჭს.
  როდესაც ჩართულია, ორკესტრატორი უყურებს რელე ტელემეტრიას დაქვეითების აფეთქებებისთვის
  და, `window_secs`-ში კონფიგურირებული `threshold`-ის შემდეგ, აიძულებს ადგილობრივ
  პროქსი `target_mode`-ში (ნაგულისხმევი `metadata-only`). მას შემდეგ, რაც შემცირება შეჩერდება
  პროქსი უბრუნდება `resume_mode`-ს `cooldown_secs`-ის შემდეგ. გამოიყენეთ `modes`
  მასივი ტრიგერის კონკრეტულ სარელეო როლებზე გადასატანად (ნაგულისხმევი შესვლის რელეებისთვის).

როდესაც პროქსი მუშაობს ხიდის რეჟიმში, ის ემსახურება ორ აპლიკაციის სერვისს:

- **`norito`** - კლიენტის ნაკადის სამიზნე გადაწყვეტილია
  `norito_bridge.spool_dir`. სამიზნეები გაწმენდილია (არ არის გადაკვეთა, არა აბსოლუტური
  ბილიკები), და როდესაც ფაილს აკლია გაფართოება, გამოიყენება კონფიგურირებული სუფიქსი
  სანამ დატვირთვა ბრაუზერში სიტყვასიტყვით გადაიცემა.
- **`car`** – ნაკადის სამიზნეები წყდება `car_bridge.cache_dir`-ში, მემკვიდრეობით
  კონფიგურირებული ნაგულისხმევი გაფართოება და უარყოთ შეკუმშული დატვირთვები, თუ
  `allow_zst` დაყენებულია. წარმატებული ხიდები ადრე პასუხობენ `STREAM_ACK_OK`-ით
  არქივის ბაიტების გადაცემა, რათა კლიენტებმა შეძლონ მილსადენის გადამოწმება.

ორივე შემთხვევაში პროქსი აწვდის ქეში-ტეგს HMAC (როცა დამცავი ქეში გასაღები იყო
იმყოფება ხელის ჩამორთმევის დროს) და აღრიცხავს `norito_*` / `car_*` ტელემეტრიის მიზეზს
კოდები ისე, რომ დაფებს შეუძლიათ განასხვავონ წარმატებები, დაკარგული ფაილები და გაწმენდა
წარუმატებლობა ერთი შეხედვით.

`Orchestrator::local_proxy().await` ავლენს გაშვებულ სახელურს, რათა აბონენტებმა შეძლონ
წაიკითხეთ სერტიფიკატი PEM, მიიღეთ ბრაუზერის მანიფესტი ან მოითხოვეთ მოხდენილი
გამორთვა აპლიკაციის გასვლისას.

როდესაც ჩართულია, პროქსი ახლა ემსახურება **მანიფესტის v2** ჩანაწერებს. გარდა არსებულისა
სერთიფიკატი და მცველი ქეში გასაღები, v2 ამატებს:

- `alpn` (`"sorafs-proxy/1"`) და `capabilities` მასივი, რათა კლიენტებმა დაადასტურონ
  ნაკადის პროტოკოლი მათ უნდა ისაუბრონ.
- `session_id` თითო ხელის ჩამორთმევისა და ქეშის მონიშვნის მარილი (`cache_tagging` ბლოკი)
  თითო სესიის დამცავი კავშირების და HMAC ტეგების გამომუშავება.
- მიკროსქემისა და მცველის შერჩევის მინიშნებები (`circuit`, `guard_selection`,
  `route_hints`) ასე რომ, ბრაუზერის ინტეგრაციამ შეიძლება გამოავლინოს უფრო მდიდარი ინტერფეისი ნაკადების დაწყებამდე
  გაიხსნა.
- `telemetry_v2` სინჯის აღების და კონფიდენციალურობის ღილაკებით ადგილობრივი ხელსაწყოებისთვის.
- თითოეული `STREAM_ACK_OK` მოიცავს `cache_tag_hex`. კლიენტები ასახავს მნიშვნელობას
  `x-sorafs-cache-tag` სათაური HTTP ან TCP მოთხოვნების გაცემისას ქეშირებული
  დაცვის არჩევანი რჩება დაშიფრული დასვენების დროს.

განაგრძეთ დაეყრდნოთ v1 ქვეჯგუფს.

## 2. წარუმატებლობის სემანტიკა

ორკესტრი ახორციელებს მკაცრ შესაძლებლობებს და ბიუჯეტის შემოწმებას სინგლის წინ
ბაიტი გადადის. წარუმატებლობა იყოფა სამ კატეგორიად:

1. **შესაბამისად წარუმატებლობა (ფრენის წინ).** პროვაიდერებს აკლიათ დიაპაზონის შესაძლებლობა,
   ვადაგასული რეკლამები, ან შემორჩენილი ტელემეტრია დარეგისტრირებულია ქულების დაფის არტეფაქტში და
   გამოტოვებული გრაფიკიდან. CLI შეჯამებები ავსებს `ineligible_providers`-ს
   მასივი მიზეზებით, რათა ოპერატორებმა შეძლონ მმართველობის დრიფტის შემოწმება გახეხვის გარეშე
   მორები.
2. **გაშვების დროის ამოწურვა.** თითოეული პროვაიდერი აკონტროლებს თანმიმდევრულ წარუმატებლობებს. ერთხელ,
   კონფიგურირებული `provider_failure_threshold` მიღწეულია, პროვაიდერი მონიშნულია
   `disabled` სესიის დარჩენილი პერიოდისთვის. თუ ყველა პროვაიდერი გადადის
   `disabled`, ორკესტრი ბრუნდება
   `MultiSourceError::NoHealthyProviders { last_error, chunk_index }`.
3. **დეტერმინისტული აბორტები.** მყარი ლიმიტების ზედაპირი, როგორც სტრუქტურირებული შეცდომები:
   - `MultiSourceError::NoCompatibleProviders` — მანიფესტი მოითხოვს ნაწილს
     გაფართოვება ან გასწორება, რომელსაც დანარჩენი პროვაიდერები ვერ აფასებენ.
   - `MultiSourceError::ExhaustedRetries` — თითო ცალი ხელახალი ცდის ბიუჯეტი იყო
     მოხმარებული.
   - `MultiSourceError::ObserverFailed` — ქვედა დინების დამკვირვებლები (სტრიმინგის კაკვები)
     უარყო დამოწმებული ნაწილი.

ყოველი შეცდომა ჩაშენებულია შეურაცხმყოფელი ნაწილის ინდექსს და, როდესაც ეს შესაძლებელია, საბოლოო
პროვაიდერის წარუმატებლობის მიზეზი. მოექეცით მათ, როგორც გათავისუფლების ბლოკატორებს - სცადეთ იგივე
შეყვანა განაახლებს წარუმატებლობას ფუძემდებლურ რეკლამამდე, ტელემეტრია ან
პროვაიდერის ჯანმრთელობის ცვლილებები.

### 2.1 შედეგების დაფის გამძლეობა

როდესაც `persist_path` არის კონფიგურირებული, ორკესტრი წერს საბოლოო დაფას
ყოველი გაშვების შემდეგ. JSON დოკუმენტი შეიცავს:

- `eligibility` (`eligible` ან `ineligible::<reason>`).
- `weight` (ნორმალიზებული წონა მინიჭებული ამ რბენისთვის).
- `provider` მეტამონაცემები (იდენტიფიკატორი, ბოლო წერტილები, თანმხლები ბიუჯეტი).

დაარქივეთ შედეგების დაფის სნეპშოტები გამოშვების არტეფაქტებთან ერთად, ასე რომ შავ სიაში მოხვდა და
გაშვების გადაწყვეტილებები რჩება აუდიტის ქვეშ.

2. **ხელახალი ცდის კოეფიციენტი** — აფრთხილებს, როდესაც `retry` მრიცხველები აჭარბებენ ისტორიულ საბაზისო ხაზებს.
3. **პროვაიდერის წარუმატებლობა** — იწვევს პეიჯერის გაფრთხილებებს, როდესაც რომელიმე პროვაიდერი კვეთს
   `session_failure > 0` 15 წუთში.

### 3.2 სტრუქტურირებული ჟურნალის მიზნები

ორკესტრი აქვეყნებს სტრუქტურირებულ მოვლენებს დეტერმინისტული მიზნებისთვის:

- `telemetry::sorafs.fetch.lifecycle` — `start` და `complete` სასიცოცხლო ციკლი
  მარკერები ნაჭრების დათვლით, განმეორებითი ცდებით და მთლიანი ხანგრძლივობით.
- `telemetry::sorafs.fetch.retry` — ხელახლა სცადეთ მოვლენები (`provider`, `reason`,
  `attempts`) ხელით ტრიაჟის შესანახად.
- `telemetry::sorafs.fetch.provider_failure` — პროვაიდერები გამორთულია იმის გამო
  განმეორებითი შეცდომები.
- `telemetry::sorafs.fetch.error` — ტერმინალის გაუმართაობა შეჯამებულია
  `reason` და არჩევითი პროვაიდერის მეტამონაცემები.

გადააგზავნეთ ეს ნაკადები არსებულ Norito ჟურნალის მილსადენში, რათა რეაგირება მოახდინოთ ინციდენტზე
აქვს სიმართლის ერთი წყარო. სასიცოცხლო ციკლის მოვლენები ავლენს PQ/კლასიკურ მიქსს
`anonymity_effective_policy`, `anonymity_pq_ratio`,
`anonymity_classical_ratio` და მათი კომპანიონი მრიცხველები,
რაც მარტივია მავთულის დაფების გაყვანის მეტრიკის გაფცქვნის გარეშე. დროს
GA გამოდის, ჩაამაგრეთ ჟურნალის დონე `info`-ზე სასიცოცხლო ციკლის/ხელახლა სცადეთ მოვლენებისთვის და დაეყრდნოთ
`warn` ტერმინალის შეცდომებისთვის.

### 3.3 JSON შეჯამებები

ორივე `sorafs_cli fetch` და Rust SDK აბრუნებენ სტრუქტურირებულ შეჯამებას, რომელიც შეიცავს:

- `provider_reports` წარმატების/წარუმატებლობის გათვალისწინებით და იყო თუ არა პროვაიდერი
  ინვალიდი.
- `chunk_receipts` დეტალურად, თუ რომელი პროვაიდერი დააკმაყოფილა თითოეული ნაწილი.
- `retry_stats` და `ineligible_providers` მასივები.

დაარქივეთ შემაჯამებელი ფაილი არასწორი მოქცევის პროვაიდერების გამართვისას — ქვითრების რუკა
პირდაპირ ზემოთ მოცემულ ჟურნალის მეტამონაცემებზე.

## 4. ოპერატიული ჩამონათვალი

1. **სცენის კონფიგურაცია CI-ში.** გაუშვით `sorafs_fetch` სამიზნით
   კონფიგურაცია, გაიარეთ `--scoreboard-out` დასაშვებობის ხედის გადასაღებად და
   განსხვავდება წინა გამოშვებისგან. ნებისმიერი მოულოდნელი შეუსაბამო პროვაიდერი ჩერდება
   დაწინაურება.
2. **ტელემეტრიის დადასტურება.** უზრუნველყოს განლაგების ექსპორტი `sorafs.fetch.*`
   მეტრიკა და სტრუქტურირებული ჟურნალები მომხმარებლებისთვის მრავალ წყაროს მოპოვების ჩართვამდე.
   მეტრიკის არარსებობა, როგორც წესი, მიუთითებს, რომ ორკესტრის ფასადი არ იყო
   მოწვეული.
3. **დოკუმენტის უგულებელყოფა.** საგანგებო `--deny-provider` ან
   `--boost-provider` პარამეტრები, ჩააბარეთ JSON (ან CLI გამოძახება) თქვენს
   ჟურნალის შეცვლა. უკან დაბრუნებამ უნდა დააბრუნოს უგულებელყოფა და დაიჭიროს ახალი ანგარიში
   სნეპშოტი.
4. ** ხელახლა გაუშვით კვამლის ტესტები. ** ხელახლა ცდის ბიუჯეტების ან პროვაიდერის ზღვრების შეცვლის შემდეგ,
   ხელახლა მოიყვანეთ კანონიკური მოწყობილობა (`fixtures/sorafs_manifest/ci_sample/`) და
   გადაამოწმეთ, რომ ქვითრები დეტერმინისტული რჩება.

ზემოთ მოყვანილი ნაბიჯების შესრულება ორკესტრის ქცევას ინარჩუნებს რეპროდუცირებულად
ეტაპობრივად ამუშავებს და უზრუნველყოფს ინციდენტის რეაგირებისთვის საჭირო ტელემეტრიას.

### 4.1 პოლიტიკის უგულებელყოფა

ოპერატორებს შეუძლიათ დაამაგრონ აქტიური ტრანსპორტის/ანონიმურობის ეტაპი რედაქტირების გარეშე
ბაზის კონფიგურაცია `policy_override.transport_policy` დაყენებით და
`policy_override.anonymity_policy` მათ `orchestrator` JSON-ში (ან მომწოდებელი
`--transport-policy-override=` / `--anonymity-policy-override=`
`sorafs_cli fetch`). როდესაც რომელიმე უგულებელყოფა არსებობს, ორკესტრი გამოტოვებს
ჩვეული ბრუნაუტის უკან დაბრუნება: თუ მოთხოვნილი PQ იარუსი ვერ დაკმაყოფილდება,
მიღება ვერ ხერხდება `no providers`-ით ჩუმად დაქვეითების ნაცვლად. უკან დაბრუნება
ნაგულისხმევი ქცევა ისეთივე მარტივია, როგორც გადაფარვის ველების გასუფთავება.

სტანდარტული `iroha_cli app sorafs fetch` ბრძანება ავლენს იგივე გადაფარვის დროშებს,
მათი გადაგზავნა კარიბჭის კლიენტზე, რათა მოხდეს ად-ჰოკ მოძიება და ავტომატიზაციის სკრიპტები
გააზიარეთ სცენის დამაგრების იდენტური ქცევა.

Cross-SDK მოწყობილობები მუშაობს `fixtures/sorafs_gateway/policy_override/` ქვეშ. The
CLI, Rust კლიენტი, JavaScript-ის საკინძები და Swift აღკაზმულობის გაშიფვრა
`override.json` მათ პარიტეტულ კომპლექტებში, ასე რომ, ნებისმიერი ცვლილება ზედმეტ დატვირთვებში
უნდა განაახლოს ეს მოწყობილობა და ხელახლა გაუშვას `cargo test -p iroha`, `npm test` და
`swift test` SDK-ების გასწორების მიზნით. ყოველთვის მიამაგრეთ რეგენერირებული მოწყობილობა
შეცვალეთ მიმოხილვა ისე, რომ ქვემოთ მომხმარებლებმა შეცვალონ კონტრაქტის უგულებელყოფა.

მმართველობა მოითხოვს Runbook-ის ჩანაწერს ყოველი უგულებელყოფისთვის. ჩაწერეთ მიზეზი,
მოსალოდნელი ხანგრძლივობა და დაბრუნების ტრიგერი თქვენს ცვლილებების ჟურნალში, აცნობეთ PQ-ს
დააწებეთ როტაციის არხი და დაურთოთ ხელმოწერილი დამტკიცება იმავე არტეფაქტს
ნაკრები, რომელიც ინახავს ანგარიშის დაფის სურათს. Overrides განკუთვნილია მოკლედ
გადაუდებელი შემთხვევები (მაგ., PQ მცველის დაღლილობა); გრძელვადიანი პოლიტიკის ცვლილებები უნდა წავიდეს
საკრებულოს ნორმალური კენჭისყრის გზით, ასე რომ, კვანძები გადადიან ახალ ნაგულისხმევზე.

SNNet-5 სატრანსპორტო გაშვება წარმოგიდგენთ მცველთა ახალ შერჩევას, მმართველობას

მიჰყევით საბოლოო მომხმარებლებისთვის მრავალ წყაროს მოპოვების ჩართვამდე, პლუს რეიტინგის შემცირებას
გზა უკან პირდაპირ რეჟიმში.

### 5.1 დეველოპერის წინასწარი ფრენა (CI / დადგმა)

1. **აღადგინეთ დაფები CI-ში.** გაუშვით `sorafs_cli fetch` (ან SDK
   ექვივალენტი) `fixtures/sorafs_manifest/ci_sample/` მანიფესტის წინააღმდეგ
   კანდიდატის კონფიგურაცია. განაგრძეთ ანგარიშის დაფა
   `--scoreboard-out=artifacts/sorafs/scoreboard.json` და ვამტკიცებ:
   - `anonymity_status=="met"` და `anonymity_pq_ratio` აკმაყოფილებს მიზანმიმართულს
     ეტაპი (`anon-guard-pq`, `anon-majority-pq`, ან `anon-strict-pq`).
   - დეტერმინისტული ქვითრები მაინც ემთხვევა ოქროს ნაკრებს
     საცავი.
2. **გადაამოწმეთ მანიფესტაციური მმართველობა.** შეამოწმეთ CLI / SDK რეზიუმე და დარწმუნდით
   ახლად გამოჩენილი `manifest_governance.council_signatures` მასივი შეიცავს
   მოსალოდნელია საბჭოს თითის ანაბეჭდები. ეს ადასტურებს კარიბჭის პასუხებს GAR-ზე
   კონვერტი და რომ `validate_manifest` მიიღო იგი.
3. ** სავარჯიშოების შესაბამისობა უგულებელყოფს. ** ჩატვირთეთ თითოეული იურისდიქციის პროფილი
   `docs/examples/sorafs_compliance_policy.json` და დაამტკიცე ორკესტრი
   გამოსცემს სწორ პოლიტიკას (`compliance_jurisdiction_opt_out` ან
   `compliance_blinded_cid_opt_out`). ჩაწერეთ მიღებული მარცხი, როდესაც არა
   ხელმისაწვდომია შესაბამისი ტრანსპორტი.
4. **დაქვეითების სიმულაცია.** გადააქციეთ `transport_policy` `direct-only`-ზე
   კონფიგურაცია ტესტირების პროცესშია და ხელახლა გაუშვით Fetch ორკესტრის უზრუნველსაყოფად
   უბრუნდება Torii/QUIC-ს SoraNet რელეების შეხების გარეშე. შეინახეთ ეს JSON
   ვარიანტი ვერსიის კონტროლის ქვეშ, ასე რომ მისი სწრაფი პოპულარიზაცია შესაძლებელია
   ინციდენტი.

### 5.2 ოპერატორის გაშვება (წარმოების ტალღები)1. **კონფიგურაციის დადგმა `iroha_config`-ით.** გამოაქვეყნეთ გამოყენებული ზუსტი JSON
   CI-ში, როგორც `actual` ფენის გადაფარვა. დაადასტურეთ ორკესტრის პოდი / ორობითი
   ჩაწერს ახალი კონფიგურაციის ჰეშს გაშვებისას.
2. **Prime guard caches.** განაახლეთ სარელეო დირექტორია `--guard-directory`-ის მეშვეობით
   და გააგრძელეთ Norito დამცავი ქეში `--guard-cache`-ით. შეამოწმეთ, რომ ქეში არის
   ხელმოწერილი (თუ `--guard-cache-key` არის კონფიგურირებული) და ინახება ვერსიის ქვეშ
   კონტროლის შეცვლა.
3. **ჩართეთ ტელემეტრიის დაფები.** მომხმარებელთა ტრაფიკის მომსახურებამდე, დარწმუნდით
   გარემო აქვეყნებს `sorafs.fetch.*`, `sorafs_orchestrator_policy_events_total` და
   პროქსის მეტრიკა (ლოკალური QUIC პროქსის გამოყენებისას). სიგნალიზაცია უნდა იყოს მიბმული
   `anonymity_brownout_effective` და შესაბამისობის სარეზერვო მრიცხველები.
4. **გაუშვით ცოცხალი კვამლის ტესტები.** მიიღეთ მმართველობის მიერ დამტკიცებული მანიფესტი თითოეულიდან
   პროვაიდერის კოჰორტა (PQ, კლასიკური და პირდაპირი) და დაადასტურეთ ქვითრები,
   CAR დაიჯესტები და საბჭოს ხელმოწერები ემთხვევა CI საწყისს.
5. **გააქტიურებასთან დაკავშირება.** განაახლეთ გაშვების ტრეკერი
   `scoreboard.json` არტეფაქტი, დამცავი ქეშის თითის ანაბეჭდი და ბმული
   ჟურნალები, რომლებიც აჩვენებენ მანიფესტის მმართველობის დადასტურებას პირველი წარმოების მისაღებად.

### 5.3 დაქვეითების / დაბრუნების პროცედურა

როდესაც ინციდენტები, PQ დეფიციტი ან მარეგულირებელი მოთხოვნები აიძულებს უკან დაბრუნებას, მიჰყევით
ეს დეტერმინისტული თანმიმდევრობა:

1. **შეცვალეთ სატრანსპორტო პოლიტიკა.** გამოიყენეთ `transport_policy=direct-only` (და, თუ
   დაუყოვნებლივ აჩერებს ახალი SoraNet მიკროსქემის მშენებლობას.
2. **Flush guard state.** წაშალეთ ან დაარქივეთ მცველის ქეში ფაილი მითითებულ
   `--guard-cache` ასე რომ, შემდგომი გაშვებები არ შეეცადოთ ხელახლა გამოიყენოთ დამაგრებული რელეები.
   გამოტოვეთ ეს ნაბიჯი მხოლოდ მაშინ, როდესაც დაგეგმილია სწრაფი ხელახალი ჩართვა და ქეში რჩება
   მოქმედებს.
3. **გათიშეთ ლოკალური პროქსი.** თუ ადგილობრივი QUIC პროქსი იყო `bridge` რეჟიმში,
   გადატვირთეთ ორკესტრი `proxy_mode="metadata-only"`-ით ან ამოიღეთ
   `local_proxy` მთლიანად ბლოკავს. დაასაბუთეთ პორტის გამოშვება სამუშაო სადგურისთვის და
   ბრაუზერის ინტეგრაცია უბრუნდება პირდაპირ Torii წვდომას.
4. ** მკაფიო შესაბამისობა უგულებელყოფს. ** დაურთეთ იურისდიქციული უარის თქმის ჩანაწერი (ან
   blinded-CID ჩანაწერი) შესაბამისობის პოლიტიკაზე დაზარალებული ტვირთამწეობით
   ავტომატიზაცია და დაფები ასახავს განზრახ პირდაპირი რეჟიმის მუშაობას.
5. **აუდიტორული მტკიცებულებების აღება.** გაუშვით ცვლილებების შემდგომი მიღება `--scoreboard-out`-ით
   და შეინახეთ CLI JSON რეზიუმე (მათ შორის `manifest_governance`) გვერდით
   ინციდენტის ბილეთი.
