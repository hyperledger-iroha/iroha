# ორკესტრატორის გავრცელების და ტუნინგის სახელმძღვანელო

ეს სახელმძღვანელო ეფუძნება [კონფიგურაციის მითითებას] (orchestrator-config.md) და
[multi-source rollout runbook] (multi-source-rollout.md). განმარტავს
როგორ მოვახდინოთ ორკესტრის მორგება თითოეული გამოშვების ფაზისთვის, როგორ განვახორციელოთ ინტერპრეტაცია
დაფის არტეფაქტები და ტელემეტრიის რომელი სიგნალები უნდა იყოს ადრე
მოძრაობის გაფართოება. გამოიყენეთ რეკომენდაციები თანმიმდევრულად CLI, SDK და
ავტომატიზაცია, ასე რომ, ყველა კვანძი მიჰყვება იმავე დეტერმინისტულ ჩამოტანის პოლიტიკას.

## 1. საბაზისო პარამეტრის ნაკრები

დაიწყეთ საერთო კონფიგურაციის შაბლონიდან და შეცვალეთ ღილაკების მცირე ნაკრები როგორც
გაშვება პროგრესირებს. ქვემოთ მოყვანილი ცხრილი ასახავს რეკომენდებულ მნიშვნელობებს
ყველაზე გავრცელებული ფაზები; ჩამოთვლილი მნიშვნელობები ბრუნდება ნაგულისხმევად
`OrchestratorConfig::default()` და `FetchOptions::default()`.

| ფაზა | `max_providers` | `fetch.per_chunk_retry_limit` | `fetch.provider_failure_threshold` | `scoreboard.latency_cap_ms` | `scoreboard.telemetry_grace_secs` | შენიშვნები |
|-------|----------------------------------------------|------------------------- ----------|----------------------------|---------------------------------|------|
| **ლაბორატორია / CI ** | `3` | `2` | `2` | `2500` | `300` | მჭიდრო შეყოვნების ქუდი და გრილი ფანჯრის ზედაპირის ხმაურიანი ტელემეტრია სწრაფად. შეინახეთ ხელახალი ცდები, რათა უფრო ადრე გამოაშკარავოთ არასწორი მანიფესტები. |
| **დადგმა** | `4` | `3` | `3` | `4000` | `600` | სარკეების წარმოება ნაგულისხმევია, ხოლო საძიებო თანატოლებს თავს ტოვებს. |
| **კანარი** | `6` | `3` | `3` | `5000` | `900` | შეესაბამება ნაგულისხმევს; დააყენეთ `telemetry_region`, რათა დაფებმა შეძლონ ტრიალი კანარის მოძრაობაზე. |
| ** ზოგადი ხელმისაწვდომობა ** | `None` (გამოიყენეთ ყველა დასაშვები) | `4` | `4` | `5000` | `900` | გაზარდეთ ხელახალი ცდისა და წარუმატებლობის ზღვრები გარდამავალი ხარვეზების ათვისებისთვის, სანამ აუდიტები განაგრძობენ დეტერმინიზმის აღსრულებას. |

- `scoreboard.weight_scale` რჩება ნაგულისხმევ `10_000`-ზე, თუ არ არის დაბლა
  სისტემა მოითხოვს სხვა მთელი რიცხვის გარჩევადობას. მასშტაბის გაზრდა არა
  პროვაიდერის შეკვეთის შეცვლა; ის მხოლოდ ასხივებს უფრო მჭიდრო საკრედიტო განაწილებას.
- ფაზებს შორის მიგრაციისას შეინარჩუნეთ JSON პაკეტი და გამოიყენეთ
  `--scoreboard-out` ასე რომ, აუდიტის ბილიკი აღრიცხავს ზუსტ პარამეტრებს.

## 2. საანგარიშო დაფის ჰიგიენა

შედეგების დაფა აერთიანებს მანიფესტის მოთხოვნებს, პროვაიდერის რეკლამებსა და ტელემეტრიას.
წინ გადახვევამდე:

1. **ტელემეტრიის სიახლის დადასტურება.** დარწმუნდით, რომ კადრები მითითებულია მიერ
   `--telemetry-json` იქნა გადაღებული კონფიგურირებული გრეის ფანჯრის ფარგლებში. ჩანაწერები
   უფრო ძველი ვიდრე კონფიგურირებული `telemetry_grace_secs` წარუმატებლობა
   `TelemetryStale { last_updated }`. მოექეცით ამას, როგორც რთულ გაჩერებას და განაახლეთ
   ტელემეტრიის ექსპორტი გაგრძელებამდე.
2. **შეამოწმეთ დასაშვებობის მიზეზები.** შენარჩუნებული არტეფაქტები მეშვეობით
   `--scoreboard-out=/var/lib/sorafs/scoreboards/preflight.json`. თითოეული ჩანაწერი
   ატარებს `eligibility` ბლოკს მარცხის ზუსტი მიზეზით. არ გადააჭარბოთ
   შესაძლებლობების შეუსაბამობა ან ვადაგასული რეკლამები; დააფიქსირეთ ზედა დინების დატვირთვა.
3. **გადახედეთ წონის დელტას.** შეადარეთ `normalised_weight` ველი
   წინა გამოშვება. წონის ცვლა >10% უნდა იყოს დაკავშირებული მიზანმიმართულ რეკლამასთან
   ან ტელემეტრია იცვლება და უნდა იყოს აღიარებული გაშვების ჟურნალში.
4. **არტეფაქტების დაარქივება.** დააკონფიგურირეთ `scoreboard.persist_path` ისე, რომ ყოველი გაშვება გამოსცემს
   საბოლოო ანგარიშის სურათი. მიამაგრეთ არტეფაქტი გამოშვების ჩანაწერს
   მანიფესტთან და ტელემეტრიის პაკეტთან ერთად.
5. ** ჩანაწერის პროვაიდერის მიქსის მტკიცებულება.** `scoreboard.json` მეტამონაცემები _და_
   შესაბამისი `summary.json` უნდა გამოაშკარავდეს `provider_count`,
   `gateway_provider_count` და მიღებული `provider_mix` იარლიყი ასე მიმომხილველები
   შეუძლია დაამტკიცოს, იყო თუ არა გაშვება `direct-only`, `gateway-only`, თუ `mixed`.
   კარიბჭე იჭერს, შესაბამისად, აცნობეთ `provider_count=0` plus
   `provider_mix="gateway-only"`, ხოლო შერეული გაშვებები საჭიროებს არანულოვან დათვლას
   ორივე წყარო. `cargo xtask sorafs-adoption-check` ახორციელებს ამ ველებს (და
   წარუმატებელია, როდესაც დათვლა/ეტიკეტები არ ეთანხმება), ასე რომ, ყოველთვის გაუშვით იგი
   `ci/check_sorafs_orchestrator_adoption.sh` ან თქვენი შეკვეთილი გადაღების სკრიპტი
   შექმენით `adoption_report.json` მტკიცებულების ნაკრები. როდესაც არის Torii კარიბჭეები
   ჩართული, შეინახეთ `gateway_manifest_id`/`gateway_manifest_cid` ანგარიშის დაფაზე
   მეტამონაცემები, ასე რომ მიღების კარიბჭეს შეუძლია მანიფესტის კონვერტის კორელაცია
   დატყვევებული პროვაიდერის ნაზავი.

ველის დეტალური განმარტებებისთვის იხ
`crates/sorafs_car/src/scoreboard.rs` და CLI შემაჯამებელი სტრუქტურა გამოვლენილი მიერ
`sorafs_cli fetch --json-out`.

## CLI & SDK დროშის მითითება

`sorafs_cli fetch` (იხ. `crates/sorafs_car/src/bin/sorafs_cli.rs`) და
`iroha_cli app sorafs fetch` შეფუთვა (`crates/iroha_cli/src/commands/sorafs.rs`)
იზიარებენ იგივე ორკესტრატორის კონფიგურაციის ზედაპირს. გამოიყენეთ შემდეგი დროშები, როდესაც
გამოქვეყნებული მტკიცებულებების აღება ან კანონიკური მოწყობილობების გამეორება:

გაზიარებული მრავალ წყაროს დროშის მითითება (შეინარჩუნეთ CLI დახმარება და დოკუმენტები სინქრონიზებული მხოლოდ ამ ფაილის რედაქტირებით):

- `--max-peers=<count>` ზღუდავს რამდენი უფლებამოსილი პროვაიდერი გადარჩება ქულების დაფის ფილტრს. დატოვეთ დაუყენებელი სტრიმინგისთვის ყველა უფლებამოსილი პროვაიდერისგან, დაყენებული `1`-ზე მხოლოდ მაშინ, როდესაც განზრახ ახორციელებთ ერთი წყაროს სარეზერვო ფუნქციას. ასახავს `maxPeers` სახელურს SDK-ებში (`SorafsGatewayFetchOptions.maxPeers`, `SorafsGatewayFetchOptions.max_peers`).
- `--retry-budget=<count>` გადადის თითო ნაწილზე ხელახალი ცდის ლიმიტზე, რომელიც აღსრულებულია `FetchOptions`-ით. გამოიყენეთ განლაგების ცხრილი ტიუნინგის სახელმძღვანელოში რეკომენდებული მნიშვნელობებისთვის; CLI გაშვებები, რომლებიც აგროვებენ მტკიცებულებებს, უნდა ემთხვეოდეს SDK ნაგულისხმევს, რათა შეინარჩუნონ პარიტეტი.
- `--telemetry-region=<label>` ტეგები `sorafs_orchestrator_*` Prometheus სერია (და OTLP რელეები) რეგიონის/env ეტიკეტით, რათა დაფებმა შეძლონ ლაბორატორიის, დადგმის, კანარის და GA ტრაფიკის გამოყოფა.
- `--telemetry-json=<path>` აწვდის სნეპშოტს, რომელიც მითითებულია დაფის მიერ. განაგრძეთ JSON ქულების დაფის გვერდით, რათა აუდიტორებმა შეძლონ გაშვების ხელახლა გაშვება (და ასე რომ, `cargo xtask sorafs-adoption-check --require-telemetry`-მა შეძლოს დაამტკიცოს, რომელმა OTLP ნაკადმა იკვებება გადაღება).
- `--local-proxy-*` (`--local-proxy-mode`, `--local-proxy-norito-spool`, `--local-proxy-kaigi-spool`, `--local-proxy-kaigi-policy`) ჩართავს ხიდის დამკვირვებლის კაუჭებს. როდესაც დაყენებულია, ორკესტრატორი აგზავნის ნაწილებს ადგილობრივი Norito/Kaigi პროქსის მეშვეობით, ასე რომ ბრაუზერის კლიენტები, დაცვის ქეშები და Kaigi ოთახები მიიღებენ იგივე ქვითრებს, რომლებსაც ასხივებენ Rust.
- `--scoreboard-out=<path>` (სურვილისამებრ დაწყვილებულია `--scoreboard-now=<unix_secs>`-თან) აგრძელებს აუდიტორებისთვის დასაშვებობის სნეპშოტს. ყოველთვის დააწყვილეთ შენარჩუნებული JSON ტელემეტრია და მანიფესტი არტეფაქტები, რომლებიც მითითებულია გამოშვების ბილეთში.
- `--deny-provider name=ALIAS` / `--boost-provider name=ALIAS:delta` მიმართავს დეტერმინისტულ კორექტირებას რეკლამის მეტამონაცემების თავზე. გამოიყენეთ ეს დროშები მხოლოდ რეპეტიციებისთვის; წარმოების დაქვეითება უნდა მოხდეს მმართველობის არტეფაქტების მეშვეობით, რათა ყველა კვანძმა გამოიყენოს იგივე პოლიტიკის პაკეტი.
- `--provider-metrics-out` / `--chunk-receipts-out` ინარჩუნებს თითო პროვაიდერს ჯანმრთელობის მაჩვენებლებს და ქვითრებს, რომლებიც მითითებულია გამოშვების საკონტროლო სიით; დაურთეთ ორივე არტეფაქტი შვილად აყვანის მტკიცებულებების წარდგენისას.

მაგალითი (გამოქვეყნებული მოწყობილობის გამოყენებით):

```bash
sorafs_cli fetch \
  --plan fixtures/sorafs_orchestrator/multi_peer_parity_v1/plan.json \
  --gateway-provider gw-alpha=... \
  --telemetry-source-label otlp::staging \
  --scoreboard-out artifacts/sorafs_orchestrator/latest/scoreboard.json \
  --json-out artifacts/sorafs_orchestrator/latest/summary.json \
  --provider-metrics-out artifacts/sorafs_orchestrator/latest/provider_metrics.json \
  --chunk-receipts-out artifacts/sorafs_orchestrator/latest/chunk_receipts.json

cargo xtask sorafs-adoption-check \
  --scoreboard artifacts/sorafs_orchestrator/latest/scoreboard.json \
  --summary artifacts/sorafs_orchestrator/latest/summary.json
```

SDK-ები მოიხმარენ იგივე კონფიგურაციას `SorafsGatewayFetchOptions`-ის მეშვეობით Rust-ში
კლიენტი (`crates/iroha/src/client.rs`), JS საკინძები
(`javascript/iroha_js/src/sorafs.js`) და Swift SDK
(`IrohaSwift/Sources/IrohaSwift/SorafsOptions.swift`). შეინახეთ ეს დამხმარეები
lock-step ერთად CLI ნაგულისხმევი, რათა ოპერატორებს შეუძლიათ დააკოპირონ პოლიტიკა მთელს ავტომატიზაციაში
შეკვეთილი თარგმანის ფენების გარეშე.

## 3. მოიძიეთ პოლიტიკის დარეგულირება

`FetchOptions` აკონტროლებს ხელახლა ცდის ქცევას, თანხვედრას და დადასტურებას. როცა
tuning:

- ** ხელახლა ცდები:** `per_chunk_retry_limit`-ის გაზრდა `4`-ზე მეტი ზრდის აღდგენას
  დრო, მაგრამ რისკავს პროვაიდერის ხარვეზების დაფარვას. უპირატესობა მიანიჭეთ `4` ჭერად შენარჩუნებას და
  ეყრდნობა პროვაიდერის როტაციას ზედაპირზე ცუდი შემსრულებლებისკენ.
- ** წარუმატებლობის ბარიერი:** `provider_failure_threshold` მართავს, როდესაც
  პროვაიდერი გამორთულია სესიის დარჩენილი პერიოდის განმავლობაში. გაასწორეთ ეს მნიშვნელობა
  ხელახალი ცდის პოლიტიკა: ხელახალი ცდის ბიუჯეტზე დაბალი ბარიერი აიძულებს ორკესტრატორს
  თანატოლის ამოღება ყველა განმეორებითი ცდის ამოწურვამდე.
- **კონკურენტულობა:** დატოვეთ `global_parallel_limit` დაუყენებელი (`None`), თუ
  კონკრეტული გარემო ვერ გაჯერებს რეკლამირებულ დიაპაზონებს. როდესაც დაყენებულია, დარწმუნდით
  მნიშვნელობა არის ≤ პროვაიდერის ნაკადის ბიუჯეტის ჯამი შიმშილის თავიდან ასაცილებლად.
- **ვერიფიკაციის გადამრთველები:** `verify_lengths` და `verify_digests` უნდა დარჩეს
  ჩართულია წარმოებაში. ისინი უზრუნველყოფენ დეტერმინიზმს, როდესაც შერეული პროვაიდერის ფლოტი
  თამაშში არიან; გამორთეთ ისინი მხოლოდ იზოლირებულ ბუნდოვან გარემოში.

## 4. ტრანსპორტი და ანონიმურობის დადგმა

გამოიყენეთ `rollout_phase`, `anonymity_policy` და `transport_policy` ველები
წარმოადგენს კონფიდენციალურობის პოზას:- ამჯობინეთ `rollout_phase="snnet-5"` და დაუშვით ნაგულისხმევი ანონიმურობის პოლიტიკა
  თვალყური ადევნეთ SNNet-5 ეტაპებს. უგულებელყოფა მხოლოდ `anonymity_policy_override`-ით
  როდესაც მმართველობა გასცემს ხელმოწერილ დირექტივას.
- შეინახეთ `transport_policy="soranet-first"` საწყისად, ხოლო SNNet-4/5/5a/5b/6a/7/8/12/13 არის 🈺
  (იხ. `roadmap.md`). გამოიყენეთ `transport_policy="direct-only"` მხოლოდ დოკუმენტირებული
  ამცირებს / შესაბამისობის წვრთნებს და დაელოდეთ PQ დაფარვის განხილვას მანამდე
  `transport_policy="soranet-strict"`-ის დაწინაურება — ეს დონე სწრაფად ჩავარდება, თუ
  დარჩა მხოლოდ კლასიკური რელეები.
- `write_mode="pq-only"` უნდა განხორციელდეს მხოლოდ მაშინ, როდესაც ჩაწერის ყველა გზა (SDK,
  ორკესტრი, მმართველობის ინსტრუმენტები) შეუძლია დააკმაყოფილოს PQ მოთხოვნები. დროს
  გავრცელება ინარჩუნებს `write_mode="allow-downgrade"`-ს, რათა გადაუდებელ რეაგირებას დაეყრდნოს
  პირდაპირ მარშრუტებზე, ხოლო ტელემეტრია მიუთითებს რეიტინგის შემცირებაზე.
- გვარდიის შერჩევა და წრედის დადგმა ეყრდნობა SoraNet დირექტორიას. მიწოდება
  ხელმოწერილი `relay_directory` სნეპშოტი და შეინარჩუნეთ `guard_set` ქეში, ასე რომ დაიცავით
  churn რჩება შეთანხმებული შენახვის ფანჯარაში. ქეშის თითის ანაბეჭდი დარეგისტრირებულია
  `sorafs_cli fetch`-ის მიერ არის გავრცელების მტკიცებულების ნაწილი.

## 5. შემცირების და შესაბამისობის კაკვები

ორკესტრის ორი ქვესისტემა ეხმარება პოლიტიკის აღსრულებას ხელით ჩარევის გარეშე:

- ** რემედიის დაქვეითება ** (`downgrade_remediation`): მონიტორები
  `handshake_downgrade_total` მოვლენები და კონფიგურირებული `threshold` შემდეგ არის
  გადააჭარბა `window_secs` ფარგლებში, აიძულებს ლოკალურ პროქსის `target_mode`-ში
  (ნაგულისხმევად მხოლოდ მეტამონაცემები). შეინახეთ ნაგულისხმევი (`threshold=3`, `window=300`,
  `cooldown=900`) თუ ინციდენტის მიმოხილვა არ აჩვენებს სხვა ნიმუშს. ნებისმიერი დოკუმენტი
  გადააჭარბეთ გაშვების ჟურნალს და უზრუნველყოთ დაფების თვალყურის დევნება
  `sorafs_proxy_downgrade_state`.
- ** შესაბამისობის პოლიტიკა ** (`compliance`): იურისდიქცია და მანიფესტაციები
  გადის მმართველობის მიერ მართული უარის თქმის სიების მეშვეობით. არასოდეს ინლაინ ad-hoc გადაფარვები
  კონფიგურაციის პაკეტში; ამის ნაცვლად, მოითხოვეთ ხელმოწერილი განახლება
  `governance/compliance/soranet_opt_outs.json` და ხელახლა განათავსეთ გენერირებული JSON.

ორივე სისტემისთვის შეინარჩუნეთ მიღებული კონფიგურაციის ნაკრები და შეიტანეთ იგი
გამოაქვეყნეთ მტკიცებულებები, რათა აუდიტორებმა შეძლონ თვალყური ადევნონ, თუ როგორ გამოიწვია ვარდნა.

## 6. ტელემეტრია და დაფები

გაშვების გაფართოებამდე, დაადასტურეთ, რომ შემდეგი სიგნალები პირდაპირ ეთერშია
სამიზნე გარემო:

- `sorafs_orchestrator_fetch_failures_total{reason="no_healthy_providers"}` —
  ნული უნდა იყოს კანარის დასრულების შემდეგ.
- `sorafs_orchestrator_retries_total` და
  `sorafs_orchestrator_retry_ratio` — უნდა სტაბილიზდეს 10%-ზე ქვემოთ ამ პერიოდის განმავლობაში
  კანარის და რჩება 5%-ზე ქვემოთ GA-ს შემდეგ.
- `sorafs_orchestrator_policy_events_total` — ადასტურებს, რომ მოსალოდნელია
  გაშვების ეტაპი აქტიურია (`stage` ლეიბლი) და აღრიცხავს ბრუნაუტებს `outcome`-ის საშუალებით.
- `sorafs_orchestrator_pq_candidate_ratio` /
  `sorafs_orchestrator_pq_deficit_ratio` — აკონტროლეთ PQ რელეს მიწოდება
  პოლიტიკის მოლოდინები.
- `telemetry::sorafs.fetch.*` ჟურნალის სამიზნეები — უნდა გადაეცეს გაზიარებულ ჟურნალში
  აგრეგატორი შენახული ძიებებით `status=failed`.

ჩატვირთეთ კანონიკური Grafana დაფა
`dashboards/grafana/sorafs_fetch_observability.json` (ექსპორტირებულია პორტალში
ქვეშ **SoraFS → Fetch დაკვირვებადობა**) ასე რომ რეგიონის/მანიფესტის სელექტორები,
პროვაიდერი ხელახლა სცადეთ სითბოს რუქა, ნაწილაკების შეყოვნების ჰისტოგრამები და შეჩერების მრიცხველები ემთხვევა
რას მიმოიხილავს SRE დამწვრობის დროს. შეიტანეთ Alertmanager წესები
`dashboards/alerts/sorafs_fetch_rules.yml` და დაადასტურეთ Prometheus სინტაქსი
`scripts/telemetry/test_sorafs_fetch_alerts.sh`-ით (დამხმარე ავტომატურად
მუშაობს `promtool test rules` ლოკალურად ან Docker-ში). გაფრთხილების ხელიდან გამორთვა მოითხოვს
იგივე მარშრუტიზაციის ბლოკი, რომელსაც სკრიპტი ბეჭდავს, რათა ოპერატორებმა შეძლონ მტკიცებულებების ჩამაგრება
გაშვების ბილეთი.

### ტელემეტრიის დამწვრობის სამუშაო პროცესი

საგზაო რუქის პუნქტი **SF-6e** საჭიროებს 30-დღიან ტელემეტრიის დამწვრობას, სანამ გადაატრიალებთ
მრავალ წყაროს ორკესტრატორი თავის GA ნაგულისხმევად. გამოიყენეთ საცავის სკრიპტები
გადაიღეთ რეპროდუცირებადი არტეფაქტის ნაკრები ყოველდღე ფანჯარაში:

1. გაუშვით `ci/check_sorafs_orchestrator_adoption.sh` დამწვრობის გარემოში
   სახელურების ნაკრები. მაგალითი:

   ```bash
   SORAFS_BURN_IN_LABEL=canary-week-1 \
   SORAFS_BURN_IN_REGION=us-east-1 \
   SORAFS_BURN_IN_MANIFEST=manifest-v4 \
   SORAFS_BURN_IN_DAY=7 \
   SORAFS_BURN_IN_WINDOW_DAYS=30 \
   ci/check_sorafs_orchestrator_adoption.sh
   ```

   დამხმარე იმეორებს `fixtures/sorafs_orchestrator/multi_peer_parity_v1`,
   წერს `scoreboard.json`, `summary.json`, `provider_metrics.json`,
   `chunk_receipts.json` და `adoption_report.json` ქვეშ
   `artifacts/sorafs_orchestrator/<timestamp>/` და ახორციელებს მინიმალურ რაოდენობას
   უფლებამოსილი პროვაიდერების მეშვეობით `cargo xtask sorafs-adoption-check`.
2. როდესაც დამწვრობის ცვლადები არსებობს, სკრიპტი ასევე ასხივებს
   `burn_in_note.json`, ლეიბლის აღება, დღის ინდექსი, მანიფესტის ID, ტელემეტრია
   წყარო და არტეფაქტი დაიჯესტები. მიამაგრეთ ეს JSON გაშვების ჟურნალს, ასეც არის
   აშკარაა, რომელი აღბეჭდვა კმაყოფილდება ყოველ დღე 30-დღიან ფანჯარაში.
3. განახლებული Grafana დაფის იმპორტი (`dashboards/grafana/sorafs_fetch_observability.json`)
   დადგმის/წარმოების სამუშაო სივრცეში, მონიშნეთ ჩამწერი ეტიკეტით და
   დაადასტურეთ, რომ ყველა პანელი აჩვენებს ნიმუშებს სატესტო მანიფესტის/რეგიონისთვის.
4. გაუშვით `scripts/telemetry/test_sorafs_fetch_alerts.sh` (ან `promtool test rules …`)
   როდესაც `dashboards/alerts/sorafs_fetch_rules.yml` შეიცვლება დოკუმენტურად
   გაფრთხილების მარშრუტი ემთხვევა ჩაწერის დროს ექსპორტირებულ მეტრებს.
5. დაარქივეთ მიღებული დაფის სნეპშოტი, გაფრთხილების ტესტის გამომავალი და ჟურნალის კუდი
   `telemetry::sorafs.fetch.*` ძიებებიდან ორკესტრატორთან ერთად
   არტეფაქტები, რათა მმართველობამ შეძლოს მტკიცებულებების გამეორება მეტრიკის ამოღების გარეშე
   ცოცხალი სისტემები.

## 7. გაშვების ჩამონათვალი

1. რეგენერაცია დაფები CI-ში კანდიდატის კონფიგურაციისა და დაჭერის გამოყენებით
   არტეფაქტები ვერსიის კონტროლის ქვეშ.
2. განახორციელეთ დეტერმინისტული მოწყობილობების მიღება თითოეულ გარემოში (ლაბორატორია, დადგმა,
   კანარის წარმოება) და მიამაგრეთ `--scoreboard-out` და `--json-out`
   არტეფაქტები გაშვების ჩანაწერში.
3. გადახედეთ ტელემეტრიის დაფებს მოწოდებულ ინჟინერთან ერთად, რათა უზრუნველყოთ ყველა მეტრიკა
   ზემოთ აქვს ცოცხალი ნიმუშები.
