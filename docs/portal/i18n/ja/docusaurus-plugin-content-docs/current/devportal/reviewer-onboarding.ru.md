# Онбординг ревьюеров プレビュー

## Обзор

DOCS-SORA は、жоэтапный запуск портала разработчиков です。チェックサムをチェックする
(`npm run serve`) и усиленные потоки 試してみる открывают следующий этап: онбординг проверенных
パブリック プレビューを公開します。 Этот гид описывает, как собирать заявки,
あなたの質問に答えてください。 См。
[プレビュー招待フロー](./preview-invite-flow.md) когорт, каденции приглазений
и экспорта телеметрии; ваги ниже фокусируются на действиях после выбора ревьюера.

- **В рамках:** ревьюеры、которым нужен доступ к プレビュー ドキュメント (`docs-preview.sora`、
  GitHub Pages と SoraFS) が GA です。
- **説明:** операторы Torii または SoraFS (покрываются собственными onboarding-китами)
  продаклазвертывания портала (см.
  [`devportal/deploy-guide`](./deploy-guide.md))。

## Роли и требования

| Роль | Типичные цели | Требуемые артефакты | Примечания |
| --- | --- | --- | --- |
|コアメンテナー |煙テストを実行します。 | GitHub ハンドル、マトリックス、CLA など。 | GitHub `docs-preview` を参照してください。あなたのことは、あなたが知っていることです。 |
|パートナーレビューアー | SDK が必要な場合は、SDK を使用してください。 |電子メール、法的POC、プレビュー規約。 | Должен подтвердить требования по телеметрии и обработке данных. |
|地域ボランティア |フィードバックを送信してください。 | GitHub ハンドル、предпочитаемый контакт、часовой пояс、согласие с CoC。 | Держите когорты небользими;貢献者契約に同意します。 |

Все типы ревьюеров должны:

1. プレビューを確認してください。
2. セキュリティ/可観測性の向上
   ([`security-hardening`](./security-hardening.md)、
   [`observability`](./observability.md)、
   [`incident-runbooks`](./incident-runbooks.md))。
3. Согласиться запускать `docs/portal/scripts/preview_verify.sh` перед тем, как
   живать любой локальный のスナップショット。

## 摂取量

1. Попросите заявителя заполнить
   [`docs/examples/docs_preview_request_template.md`](../../../examples/docs_preview_request_template.md)
   форму (или вставить ее в 問題)。 Зафиксируйте как минимум: личность, способ связи,
   GitHub ハンドル、セキュリティ - доков の機能。
2. Зафиксируйте заявку в трекере `docs-preview` (GitHub の問題 или тикет управления)
   承認者。
3. メッセージ:
   - CLA / соглазение контрибьютора в наличии (или ссылка на партнерский контракт)。
   - Подтверждение допустимого использования хранится в заявке.
   - Риск-оценка заверbolена (например、партнерские ревьюеры одобрены Legal)。
4. 承認者が問題を追跡するかどうかを確認する
   変更管理 (例: `DOCS-SORA-Preview-####`)。

## プロビジョニングと説明

1. **Передать артефакты** — プレビュー記述子とアーカイブの説明
   CI ワークフロー、ピン SoraFS (`docs-portal-preview`)。概要:

   ```bash
   ./docs/portal/scripts/preview_verify.sh \
     --build-dir build \
     --descriptor artifacts/preview-descriptor.json \
     --archive artifacts/preview-site.tar.gz
   ```2. **施行チェックサムを提供** — チェックサムをゲートする:

   ```bash
   DOCS_RELEASE_TAG=preview-<stamp> npm run --prefix docs/portal serve
   ```

   Это переиспользует `scripts/serve-verified-preview.mjs`, чтобы неподтвержденный build
   не запускался случайно.

3. **Дать доступ к GitHub (опционально)** — Если нужны неопубликованные ветки, добавьте
   GitHub `docs-preview` を使用して、GitHub `docs-preview` を使用してください。
   और देखें

4. **Коммуникация каналов поддержки** — オンコール контактом (Matrix/Slack) и
   [`incident-runbooks`](./incident-runbooks.md) を参照してください。

5. **Телеметрия + フィードバック** — Напомните, что собирается анонимизированная аналитика
   (最低。[`observability`](./observability.md))。フィードバックと問題点、
   ヘルパーをサポートします。
   [`preview-feedback-log`](./preview-feedback-log)、чтобы сводка волны оставалась актуальной。

## Чеклист ревьюера

プレビューの表示:

1. Проверить скачанные артефакты (`preview_verify.sh`)。
2. `npm run serve` (`serve:verified`) のガード チェックサムを確認します。
3. セキュリティと可観測性を両立させます。
4. OAuth/Try it を実行し、デバイス コード ログイン (если применимо) を実行します。
   生産記録。
5. Фиксировать находки в согласованном трекере (問題、общий документ или форма) и тегировать
   プレビューをご覧ください。

## Ответственности мейнтейнеров и offboarding

| Фаза | Действия |
| --- | --- |
|キックオフ | Убедиться, что 摂取量 чеклист приложен к заявке, поделиться артефактами + инструкциями, добавить запись `invite-sent` через [`preview-feedback-log`](./preview-feedback-log)、и запланировать промежуточный sync, если ревью длится более недели. |
|モニタリング | Отслеживать プレビュー телеметрию (необычный трафик 試してみましょう、プローブ) と следовать инцидент-ранбуку при подозрениях。 `feedback-submitted`/`issue-opened` を確認してください。 |
|オフボーディング | GitHub と SoraFS、`access-revoked`、要約フィードバック (概要フィードバック + открытые) を参照してください。 действия)、и обновить реестр ревьюеров。 [`docs/examples/docs_preview_feedback_digest.md`](../../../examples/docs_preview_feedback_digest.md) を参照してください。 |

Используйте тот же процесс при ротации ревьюеров между волнами. Сохранение следов в репозитории
(問題 + ваблоны) DOCS-SORA оставаться аудируемым и позволяет ガバナンス подтвердить,
プレビューを確認してください。
