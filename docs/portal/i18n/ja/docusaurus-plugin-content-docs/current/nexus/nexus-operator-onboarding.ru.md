# データスペース Sora Nexus のオンボーディング

データスペース Sora Nexus は、エンドツーエンドのデータを保持し、データスペースを保持します。デュアルトラック Runbook (`docs/source/release_dual_track_runbook.md`) と、標準のランブック (`docs/source/release_artifact_selection.md`) を参照してください。バンドル/イメージ、マニフェストと、レーン数を確認できます。

## Аудитория и предпосылки
- データスペース (レーンインデックス、データスペース ID/エイリアス、ルーティングポリシー) を取得します。
- リリース エンジニアリング (tarball、イメージ、マニフェスト、署名、公開キー) をリリースします。
- バリデータ/オブザーバーの検証 (Ed25519 идентичность узла; BLS) консенсусный ключ + PoP для バリデータ; плюс любые トグル конфиденциальных функций)。
- ソラ Nexus ピア、ブートストラップ グループを作成します。

## Шаг 1 - Подтвердить профиль релиза
1. エイリアスは、チェーン ID と呼ばれます。
2. `scripts/select_release_profile.py --network <alias>` (`--chain-id <id>`) をチェックアウトしてください。ヘルパーは `release/network_profiles.toml` とサポートしています。ソラ Nexus ответ должен быть `iroha3`。リリース エンジニアリングを担当します。
3. Зафиксируйте タグ версии из анонса релиза (например `iroha3-v3.2.0`);成果物とマニフェストのことです。

## Шаг 2 - Получить и проверить артефакты
1. バンドル `iroha3` (`<profile>-<version>-<os>.tar.zst`) とバンドル (`.sha256`、`.sig/.pub`、 `<profile>-<version>-manifest.json`、`<profile>-<version>-image.json`、これらはすべて同じです）。
2. 説明:
   ```bash
   sha256sum -c iroha3-<version>-linux.tar.zst.sha256
   openssl dgst -sha256 -verify iroha3-<version>-linux.tar.zst.pub \
       -signature iroha3-<version>-linux.tar.zst.sig \
       iroha3-<version>-linux.tar.zst
   ```
   Замените `openssl` на верификатор、одобренный организацией、еспользуете аппаратный KMS.
3. `PROFILE.toml` tarball と JSON マニフェストの例:
   - `profile = "iroha3"`
   - `version`、`commit`、`built_at` を確認してください。
   - OS/を使用してください。
4. コンテナー イメージ、ハッシュ/署名、`<profile>-<version>-<os>-image.tar`、イメージ ID、`<profile>-<version>-image.json` を指定します。

## Шаг 3 - Подготовка конфигурации из заблонов
1. バンドルと `config/` をダウンロードしてください。
2. Относитесь к файлам под `config/` как к подаблонам:
   - `public_key`/`private_key` は、Ed25519 クラスです。 Удалите приватные ключи с диска, если узел берет их из HSM; HSM コネクタに接続されています。
   - Настройте `trusted_peers`、`network.address`、および `torii.address` так、чтобы они отражали вази доступные интерфейсы иブートストラップピア。
   - `client.toml` とオペレーター側の Torii エンドポイント (TLS конфигурацию при необходимости) および учетными данными для операционного ツール。
3. チェーン ID とバンドル、ガバナンス - レーン ожидает единый канонический ガバナンス。
4. Планируйте запуск узла с флагом профиля ソラ: `irohad --sora --config <path>`。 Конфигурационный ローダー отклонит настройки SoraFS またはマルチレーン、если флаг отсутствует。## 手順 4 - データスペースとルーティングの手順
1. Отредактируйте `config/config.toml`, чтобы раздел `[nexus]` соответствовал каталогу data-space, выданному Nexus Council:
   - `lane_count` должен равняться общему числу レーン、включенных в текущей эпохе。
   - Каждая запись в `[[nexus.lane_catalog]]` и `[[nexus.dataspace_catalog]]` должна содержать уникальный `index`/`id` и別名。 Не удаляйте существующие глобальные записи;エイリアス、データスペースなどの情報が含まれます。
   - Убедитесь、что каждая запись dataspace включает `fault_tolerance (f)`;レーンリレー `3f+1` です。
2. Обновите `[[nexus.routing_policy.rules]]`, чтобы отразить выданную политику.レーン `1` とレーン `2` のガバナンスを管理します。 добавьте или измените правила так、чтобы трафик для вазего データスペース øел в правильный lane и alias。リリース エンジニアリングの機能を備えています。
3. `[nexus.da]`、`[nexus.da.audit]`、`[nexus.da.recovery]` を参照してください。 Ожидается, что операторы сохраняют значения, одобренные советом; изменяйте их только при ратификации обновленной политики。
4. Зафиксируйте финальную конфигурацию в операционном трекере.デュアルトラック Runbook требует прикрепить эффективный `config.toml` (с редактированием секретов) к тикету онбординга。

## Шаг 5 - Предполетная валидация
1. Запустите встроенный валидатор конфигурации перед подключением к сети:
   ```bash
   ./bin/irohad --sora --config config/config.toml --trace-config
   ```
   Команда выводит разреbolенную конфигурацию и падает рано, если записи каталога/марлога/марлогутизации неконсистентны илиジェネシスと設定が必要です。
2. Если вы деплоите контейнеры, выполните ту же команду внутри образа после загрузки `docker load -i <profile>-<version>-<os>-image.tar` (не забудьте) `--sora`)。
3. レーン/データスペースのプレースホルダーを表示します。 4 番目のプレースホルダー ID は、プレースホルダー ID です。
4. 煙の煙 сценарий (например, отправьте `FindNetworkStatus` запрос через `iroha_cli`, проверьте что テレメトリ エンドポイント) `nexus_lane_state_total`、ストリーミング キー (ストリーミング キー) を参照してください)。

## 手順 6 - カットオーバーとハンドオフ
1. Сохраните проверенный `manifest.json` および署名 артефакты в релизном тикете, чтобы аудиторы могли воспроизвестиぱん。
2. Nexus 操作、操作。結果:
   - Идентичность узла (ピア ID、ホスト名、Torii エンドポイント)。
   - レーン/データスペースとルーティングポリシーの組み合わせ。
   - Хэли проверенных бинарников/образов.
3. Скоординируйте финальный прием ピア (ゴシップの種と назначение レーン) с `@nexus-core`。 Не подключайтесь к сети, пока не получите одобрение; Sora Nexus требует детерминированной 占有レーンと обновленного 入場マニフェスト。
4. Runbook をオーバーライドし、タグ релиза、чтобы итерация стартовала をオーバーライドします。 Ѝтой ベースライン。

## Справочный чеклист
- [ ] Профиль релиза подтвержден как `iroha3`。
- [ ] バンドル/イメージのファイル。
- [ ] Ключи、адреса ピア、Torii エンドポイントが存在します。
- [ ] レーン/データスペースとルーティング ポリシー Nexus が必要です。
- [ ] Валидатор конфигурации (`irohad --sora --config ... --trace-config`) は、 проходит без предупреждений です。
- [ ] マニフェスト/署名は、操作を実行するためのマニフェスト/署名です。

問題は、Nexus と ожиданиях по телеметрии см です。 [Nexus 移行メモ](./nexus-transition-notes)。
