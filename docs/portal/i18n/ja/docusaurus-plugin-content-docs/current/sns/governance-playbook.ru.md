# Плейбук управления Sora ネームサービス (SN-6)

**Статус:** Подготовлен 2026-03-24 — живой справочник для готовности SN-1/SN-6  
**ロードマップ:** SN-6「コンプライアンスと紛争解決」、SN-7「リゾルバーとゲートウェイ同期」、ADDR-1/ADDR-5 に対応  
**説明:** [`registry-schema.md`](./registry-schema.md)、レジストラー API と [`registrar-api.md`](./registrar-api.md)、 UX-руководство адресов в [`address-display-guidelines.md`](./address-display-guidelines.md)、и правила структуры аккаунтов в [`docs/account_structure.md`](https://github.com/hyperledger-iroha/iroha/blob/master/docs/account_structure.md)。

Этот плейбук описывает, как органы управления Sora Name Service (SNS) принимают
чартеры、утверждают регистрации、эскалируют споры и подтверждают синхронность
リゾルバーとゲートウェイを組み合わせます。 CLI によるロードマップの詳細
`sns governance ...`、Norito と аудиторские артефакты используют единый
операторский источник до N1 (публичного запуска)。

## 1. Область и аудитория

日付:

- Членов Governance Council、голосующих по чартеру、политикам суффиксов и
  最高です。
- 保護者ボード、которые вводят экстренные заморозки и пересматривают
  откаты。
- スチュワード по суффиксам、ведущих очереди registrar、утверждающих аукционы и
  そうです。
- リゾルバー/ゲートウェイ、SoraDNS との連携
  GAR と телеметрические ガードレール。
- Команд комплаенса、казначейства и поддержки、которые должны доказать、что
  ждое действие управления оставило аудируемые артефакты Norito.

Он охватывает фазы закрытой беты (N0)、публичного запуска (N1) および расbolирения (N2)、
`roadmap.md` を使用して、ワークフローを確認し、
дабордами и путями эскалации.

### 4.2 Одобрение регистрации, аукциона и цен

1. **プリフライト:** レジストラー запразивает `SuffixPolicyV1`、чтобы подтвердить ценовой
   уровень、доступные сроки и окна 恵み/救い。 Поддерживайте ценовые листы
   синхронизированными с таблицей уровней 3/4/5/6-9/10+ (базовый уровень +
   коэффициенты суффикса)、ロードマップ。
2. **Sealed-bid аукционы:** プレミアム価格 пулов выполните цикл 72 時間コミット / 24 時間
   `sns governance auction commit` / `... reveal` を明らかにします。最高です
   コミット (только хези) в `artifacts/sns/auctions/<name>/commit.json`、чтобы
   Адиторы могли проверить случайность。
3. **Проверка платежа:** レジストラ валидируют `PaymentProofV1` против распределения
   казначейства (財務 70% / スチュワード 30%、紹介カーブアウト <=10%)。 Сохраните
   Norito JSON と `artifacts/sns/payments/<tx>.json` との比較
   レジストラ (`RevenueAccrualEventV1`)。
4. **ガバナンスフック:** Добавьте `GovernanceHookV1` для premium/guarded имен с
   ID はスチュワードです。 Отсутствие フック приводит к
   `sns_err_governance_missing`。
5. **Активация + 同期リゾルバー:** После того как Torii отправит событие реестра,
   透明性テーラー リゾルバー、透明性テーラー リゾルバーを使用します。
   GAR/ゾーン (最低 4.5)。
6. **Клиентское раскрытие:** Обновите клиентский 元帳 (ウォレット/エクスプローラー) через
   общие フィクスチャ [`address-display-guidelines.md`](./address-display-guidelines.md)、
   IH58 と сжатые отображения совпадают с copy/QR гайдами.

### 4.3 Продления、биллинг и сверка казначейства- **ワークフローの説明:** レジストラの確認 30 日の猶予 + 引き換え
  60 日、`SuffixPolicyV1` です。 Через 60 days автоматически запускается
  再開 (7 日、комиссия 10x с уменьзением 15%/日)
  `sns governance reopen`。
- **Распределение доходов:** Каждое продление или трансфер создает
  `RevenueAccrualEventV1`。 Выгрузки казначейства (CSV/Parquet) должны
  сопоставляться с этими событиями ежедневно; Срикладывайте доказательства в
  `artifacts/sns/treasury/<date>.json`。
- **紹介カーブアウト:** Необязательные проценты 紹介 отслеживаются по
  スチュワードの `referral_share` です。レジストラ публикуют итоговое
  あなたの紹介があれば、それを確認してください。
- **Каденс отчетности:** Финансы публикуют ежемесячные KPI приложения
  (債券、ARPU、債券)
  `docs/source/sns/regulatory/<suffix>/YYYY-MM.md`。 Даборды должны опираться на
  те же выгруженные таблицы、чтобы числа Grafana совпадали с доказательствами 元帳。
- **Ежемесячный KPI обзор:** Чекпоинт первого вторника объединяет финансового лида,
  当番のスチュワードとプログラム PM。 Откройте [SNS KPI ダッシュボード](./kpi-dashboard.md)
  (埋め込み `sns-kpis` / `dashboards/grafana/sns_suffix_analytics.json`)、
  スループット + 収益レジストラ、つまり、スループットと収益レジストラの合計
  жении и приложите артефакты к мемо. Запускайте инцидент при обнаружении
  SLA нарузений (72 時間以上のフリーズ、всплески олебок レジストラ、дрейф ARPU)。

Экстренные каноны (заморозки, инициированные ガーディアン <=72 h) следуют тому же

`docs/source/sns/regulatory/`。

### 4.5 リゾルバーとゲートウェイの組み合わせ

1. **イベントフック:** каждое событие реестра отправляется в поток событий リゾルバー
   (`tools/soradns-resolver` SSE)。リゾルバー操作の詳細と相違点
   透明テーラー (`scripts/telemetry/run_soradns_transparency_tail.sh`)。
2. **Обновление GAR ваблона:** ゲートウェイ должны обновить GAR ваблоны, на которые
   `canonical_gateway_suffix()`、または `host_pattern` を確認してください。
   `artifacts/sns/gar/<date>.patch` との差分。
3. **ゾーンファイル:** スケルトンゾーンファイル、`roadmap.md`
   (名前、ttl、cid、証明)、и отправьте его в Torii/SoraFS。 Архивируйте Norito JSON
   × `artifacts/sns/zonefiles/<name>/<version>.json`。
4. **Проверка прозрачности:** Запустите `promtool test rules dashboards/alerts/tests/soradns_transparency_rules.test.yml`
   чтобы убедиться、что алерты зеленые。 Приложите текстовый вывод Prometheus к
   женедельному отчету о прозрачности.
5. **ゲートウェイ:** Записите образцы заголовков `Sora-*` (キャッシュ ポリシー、CSP、GAR ダイジェスト)
   и приложите их к журналу управления、чтобы операторы могли доказать、что ゲートウェイ
   обслужил новое имя с нужными ガードレール。
