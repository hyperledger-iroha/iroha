#إصدار العملية

تُشحن ثنائيات SoraFS (`sorafs_cli`, `sorafs_fetch`, helpers) وحزم SDK
(`sorafs_car`, `sorafs_manifest`, `sorafs_chunker`) . تحافظ على خط واحد من الاصدار على
Land CLI والمكتبات، ويضمن كثافة الوبر/الاختبار، ويختار آرتيفاكتات للمستهلكين
اللاحقةين. نفّذ قائمة التحقق أدناه لكل مرشح.

## 0. تأكد من مراجعة الأمان

قبل تنفيذ بوابة الإصدار التقنية، تم إنشاء أحدث آرتيفاكتات مراجعة الأمان:

- نزيل أحدث تفاصيل مراجعة أمان SF-6 (([reports/sf6-security-review](./reports/sf6-security-review.md))
  وسجل تجزئة SHA256 الخاصة في تذكرة الإصدار.
- أرفق رابط تذكرة المعالجة المركزية (مثل `governance/tickets/SF6-SR-2026.md`) وسجّل
  موافقين على مجموعة عمل أدوات هندسة الأمن.
- تحقّق من إغلاق قائمة اللوحة الأم في المذكرة؛ النسخة غير المسومة لحظر النسخة.
- إعداد سجلات السجلات التكافؤ (`cargo test -p sorafs_car -- --nocapture sorafs_cli::proof_stream::bounded_channels`)
  حزمة المانيفست.
- تأكد أن أمر التوقيع الذي ستنفذه يشمل `--identity-token-provider` و
  `--identity-token-audience=<aud>` صريحًا حتى يُتصور نطاق Fulcio ضمن دليل النموذج.

بما أن هذه الآرتيفاكتات عند تأخر الإشعار ونشر الإصدار.

## 1. تنفيذ بوابة الإصدار/الاختبارات

يغغّل المساعد `ci/check_sorafs_cli_release.sh` لتحسين وClippy والاختبارات عبر
حزم CLI وSDK مع دليل الهدف المحلي داخل مساحة العمل (`.target`) تعارضات
الصلاحيات عند التنفيذ داخل كافيتريا CI.

```bash
CARGO_TARGET_DIR=.target ci/check_sorafs_cli_release.sh
```

ينفذ السكربت التحققات التالية:- `cargo fmt --all -- --check` (مساحة العمل)
- `cargo clippy --locked --all-targets` لحزمة `sorafs_car` (مع الهدف `cli`),
  و`sorafs_manifest` و`sorafs_chunker`
- `cargo test --locked --all-targets` الحزم نفسها

إذا فشل أيٌّ من المتابعة، أصلح السيطرة قبل وضع الوسم. يجب أن تكون بناءات
الإصدار المفضل بـ الرئيسي؛ لا تفعل ذلك، قم بعمل Cherry-Pick في تصميمات الأقسام.
تحقق البوابة أيضًا من توفير أعلام التوقيع دون مفاتيح (`--identity-token-issuer`,
`--identity-token-audience`) حيث يتطلب؛ لماذا السبب لفشل التشغيل.

## 2. تطبيق لتغيير

تستخدم جميع حزم CLI/SDK الخاصة بـ SoraFS نظام SemVer:

- `MAJOR`: المتطلبات لأول مرة الإصدار 1.0. قبل 1.0 يشير إلى رفع الثانوية العامة `0.y`
  **إلى المحاصيل الزراعية** في واجهة CLI أو مخططات Norito.
- `MINOR` : خدمات متاحة للخلف (أوامر/أعلام جديدة، لتغطية Norito جديدة خلف بناء
  اختيارية، إضافات تليمترية).
- `PATCH`: التعديلات، وإصدارات الوثائق فقط، وتحديثات تبعيات لا تغير السلوك
  الملاحظ.

الحفاظ الدائم على النسخة `sorafs_car` و`sorafs_manifest` و`sorafs_chunker` متطابقة كي
يعتمد مستهلكو SDK اللاحقون على إصدار السلسلة موحّدة. عند رفع التعديلات:

1. يستنزف `version =` في كل `Cargo.toml`.
2. إعادة توليد `Cargo.lock` عبر `cargo update -p <crate>@<new-version>` (تفترض
   مساحة العمل نسخة صريحة).
3. شغّل بوابة الإصدار مرة أخرى واستمر في عدم بقاء آرتيفاكتات قديمة.

## 3. نسخة إعداد التعليقاتيجب أن ينشر كل إصدارات سجل التغييرات Markdown يبرز التغيرات CLI وSDK وما يؤثر على
الانخراط. استخدم في موقع `docs/examples/sorafs_release_notes.md` (انسخه إلى
دليل آرتيفاكتات النموذج وانتشار الأقسام بتفاصيل الصابون).

الحد من المحتوى الجديد:

- **أبرز النقاط**: عناوين المستخدمين CLI وSDK.
- **التوافق**: نماذج كاسرة، ترقيةات مطلوبة، متطلبات الدنيا للبوابة/العقدة.
- **خطوات الترقية**: أوامر TL;DR لتحديث تبعيات البضائع وتشغيل التركيبات الحتمية.
- **التحقق**: تجزئات المخرجات أو الأظرف والإصدار الدقيق لـ
  `ci/check_sorafs_cli_release.sh` الذي تم تشغيله.

أرفق تعليقات النسخة المكتملة بالوسم (مثل نسخة نسخة GitHub) وتخزينها بجانب
الآرتيفاكتات المولدة حميمية.

## 4. تنفيذ نسخة خطافات

شغّل `scripts/release_sorafs_cli.sh` أصدر حزمة التوقيع وملخص التحقق الذي
يُشحن مع كل الإصدار. يقوم بتطبيق CLI عند الحاجة، ويستدعي
`sorafs_cli manifest sign`، ثم إعادة تشغيل `manifest verify-signature` فورًا
لتظهر التخفيضات قبل وضع الوسم. مثال:

```bash
scripts/release_sorafs_cli.sh \
  --manifest artifacts/site.manifest.to \
  --chunk-plan artifacts/site.chunk_plan.json \
  --chunk-summary artifacts/site.car.json \
  --bundle-out artifacts/release/manifest.bundle.json \
  --signature-out artifacts/release/manifest.sig \
  --identity-token-provider=github-actions \
  --identity-token-audience=sorafs-release \
  --expect-token-hash "$(cat .release/token.hash)"
```

نصائح:- تتبع واردات النسخة (التخصيص، الملخصات، التخصيص، الرمز) داخل
  مستودع أو إعداد النشر ليبقى السكربت قابلاً للإنتاج مرة أخرى. دائما حزمة
  تركيبات في `fixtures/sorafs_manifest/ci_sample/` الشخص المناسب.
- أسّس CI على `.github/workflows/sorafs-cli-release.yml`؛ فهي تغّل البوابة
  النسخة، وتستدعي السكربت أعلاه، وتؤرشف الحزم/التواقيع كآرتيفاكتات سير العمل.
  حافظ على الترتيب نفسه (بوابة الإصدار → التوقيع → التحقق) في نظام CI الآخر
  حتى يتم تطابق السجلات مع التجزئات الناشئة.
- تستخدم بالملفات `manifest.bundle.json` و`manifest.sig` و`manifest.sign.summary.json`
  و`manifest.verify.summary.json` مشهور — فهي تشكل مساهمتها في إشعار الـ تور.
- عندما يصحح إصدار التركيبات المعتمدة، انسخ المانيفست المتصاعدة الـ Chunk
  والملخصات إلى `fixtures/sorafs_manifest/ci_sample/` (وتحديث
  `docs/examples/sorafs_ci_sample/manifest.template.json`) قبل وضع الوسم.
  تعتمد إطلاقهم على اللاحقون على التركيبات الملتزمة من جديد إنتاج حزمة الإصدار.
- التقط سجل تشغيل فحص قنوات `sorafs_cli proof stream` ذات الحدود وأرفقه بحزمة
  نسخة لإثبات أن ضمانات إثبات التدفق لا تزال فعالة.
- سجّل القيمة `--identity-token-audience` الصغيرة المستخدمة أثناء التوقيع في الملاحظات
  الإصدار؛ ستحقق المزيد من الجماهير تجاه فولسيو قبل أن تحظى بالاحترام.

استخدم `scripts/sorafs_gateway_self_cert.sh` عندما يتم تخصيص نسخة الضغط أيضًا
للبوابة. ووجهه إلى حزمة المانيفيست بنفسه لإثبات أنه يتطابق مع الآرتيفاكت لـ:

```bash
scripts/sorafs_gateway_self_cert.sh --config docs/examples/sorafs_gateway_self_cert.conf \
  --manifest artifacts/site.manifest.to \
  --manifest-bundle artifacts/release/manifest.bundle.json
```

## 5. وضع الوسم والنشر

بعد النجاح واكتمال الخطافات:1. شغّل `sorafs_cli --version` و`sorafs_fetch --version` لتأكيد أن ثنائيات تعلن
   الإصدار الجديد.
2. تحضير نسخة الإعداد في ملف `sorafs_release.toml` مُلتزم في المستودع (متعمد) أو
   ملف إعداد آخر يتابعه مستودع النشر. تجنّب الاعتماد على تنوعات البيئة الطبيعية؛ مرّر
   المسارات إلى CLI عبر `--config` (أو ما بعده) حتى تكون مدخلات نسخة صريحة وقابلة للتخصيص
   إعادة الإنتاج.
3. أنشئ توقيعًا موقّعًا (متوقفًا) أو توقيعًا مُعلّقًا:
   ```bash
   git tag -s sorafs-vX.Y.Z -m "SoraFS CLI & SDK vX.Y.Z"
   git push origin sorafs-vX.Y.Z
   ```
4. رفع الآرتيفاكتات (حزمة CAR، المانيفستات، ملخصات الأدلة، ملفات الإصدار، مخرجات)
   الشهادات) إلى تسجيل المشروع لاختيار قائمة الاختيار في [دليل النشر](./developer-deployment.md).
   إذا كان الإصدار الجديد من التركيبات جديدًا، فسيتم رفعها إلى مستودع التركيبات أو مستودع الكائنات الفضائية
   كي براءات الاختراع ضد مقارنة التقرير المنشور مع التحكم بالمصدر.
5. أخطر قناة الويب الخاصة ببرواب الوسم المسجل، وملاحظات الإصدار، وتجزئات الحزمة المانيفست/
   التواقيع، وملخصات `manifest.sign/verify` المؤرشفة، واي أظرف الكريمات. إضافة رابط وظيفة CI
   (أو أرشيف تسجيل) التي شغّلت `ci/check_sorafs_cli_release.sh` و
   `scripts/release_sorafs_cli.sh`. تحديد تذكرة التسع ليتمكن من تأخير تأخيرها
   موافقات إلى الآرتيفاكتات؛ وعندما يرسل الوظيفة `.github/workflows/sorafs-cli-release.yml`
   إشعارات، ربط التجزئات المتنوعة بدل لصق ملخصات غير محددة.

## 6.ما بعد الإصدار- تأكد من تحديث الوثائق التي تشاهدها إلى الإصدار الجديد (البدء السريع، قوالب CI)
  أو شدد على عدم الحاجة إلى التغييرات.
- إضافة إلى خارطة الطريق إذا لزم الأمر عمل لاحق (مثل أعلام الرحيل أو إيقاف
  المانيفستات القديمة).
- أرشفة السجلات النهائية لبوابة الإصدار للمشرفين — واحفظها بجانب الآرتيفاكتات الموقع.

يتبع هذا المسار يُبقي CLI وحزم SDK وملحقات التشاركية في كل دورة الإصدار.
