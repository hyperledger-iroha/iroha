#تأهيل زائري المعاينة

## نظرة عامة

DOCS-SORA يتابع مؤتمرا مرحليا لبوابة المطورين. عمليات البناء المقيدة بالتحقق
(`npm run serve`) وتدفقات Try it المحصنة تفتح المرحلة التالية: تأهيل المتقدمين منتهي الصلاحية قبل ان ان
فتح المعاينة العامة على نطاق واسع. يشرح هذا الدليل كيفية جمع الطلبات، والتحقق من الأهلية،
تحسين الوصول، وانهااء المشاركة بشكل امن. ارجع الى
[معاينة تدفق الدعوة](./preview-invite-flow.md) لتخطيط الدفعات، تراجع الدعوات، وصادرات القياس
عن بعد؛ الخطوات ادناه لا يمكنك الوصول بعد اختيار المراجع.

- **النطاق:** المراجعين الذين وصلوا الى معاينة المستندات (`docs-preview.sora`,
  عمليات بناء GitHub Pages أو حزم SoraFS) قبل GA.
- **خارج النطاق:** مشغلو Torii او SoraFS (مغطيون بكتيبات onboarding بهم الخاصة) نشر
  البوابة الإنتاجية (انظر
  [`devportal/deploy-guide`](./deploy-guide.md)).

## الادوار والمتطلبات المسبقة

| الدور | الانتظام | الاثار المطلوبة | تعليقات |
| --- | --- | --- | --- |
| المشرف الأساسي | التحقق من تشغيل المعادلات الجديدة، اختبارات الدخان. | GitHub، جهة اتصال Matrix، CLA موقعة في الملف. | غالبًا ما يكون موجودًا بالفعل في فريق GitHub `docs-preview`؛ مع ذلك قدمت طلبا حتى يكون الوصول قابلا للدقيق. |
| مراجع شريك | التحقق من مقتطفات SDK او خصوصية ال تور قبل الاصدار العام. | بريد شركة، جهة اتصال، معاينة موقعة. | يجب الاقرار بمتطلبات القياس عن بعد + التعامل مع البيانات. |
| متطوع مجتمعي | تقديم أفكار قابلة للاستخدام على المعادلات. | GitHub، جهة اتصال مفضلة، منطقة عسكرية، تقبل CoC. | حافظ على الدفعات الصغيرة؛ إعطاء الاولوية للمراجعين الذين وقعوا هم المساهم. |

يجب على جميع أنواع المراجعين:

1. الاقرار بالسياسة المستخدمة لاثار المعاينة.
2. قراءة ملاحقة شعبية/الرصد
   ([`security-hardening`](./security-hardening.md)،
   [`observability`](./observability.md)،
   [`incident-runbooks`](./incident-runbooks.md)).
3. الموافقة على الموافقة على `docs/portal/scripts/preview_verify.sh` قبل تقديم اي
   لقطة محلية.

## سير عمل تناول

1. اطلب من مقدم الطلب نموذج الإنتاج
   [`docs/examples/docs_preview_request_template.md`](../../../examples/docs_preview_request_template.md)
   (اونسخة/لصقه في العدد). تم تصنيفها على أنها: جواز السفر، وسيلة الاتصال، مقبض GitHub، تصنيف المراجعة
   هيجية، وتكيد ان المستندات الامنية قراءتها.
2. سجل الطلب في تتبع `docs-preview` (اصدار GitHub او تذكرة تور) وعين معتمدا.
3. التحقق من المتطلبات المسبقة:
   - CLA / كونه مساهم في الملف (او مرجع عقد شريك).
   - اقرار الاستخدام متى محفوظ داخل الطلب.
   - تقييم مكتمل (مثال: المرشحين للحصول على الموافقة عليهم من الناحية القانونية).
4. أن يكون مؤهلاً عند الطلب ويتابع المشكلة متابعة باي إدخال لادارة التغيير
   (مثال: `DOCS-SORA-Preview-####`).

##تعلم والادوات

1. **مشاركة الاثار** — تقديم اخر واصف + ارشيف معاينة من سير العمل CI او pin SoraFS
   (اثر `docs-portal-preview`). ذكر المراجعين يسمح:

   ```bash
   ./docs/portal/scripts/preview_verify.sh \
     --build-dir build \
     --descriptor artifacts/preview-descriptor.json \
     --archive artifacts/preview-site.tar.gz
   ```

2. **التقديم مع فرض المجموع الاختباري** — وجه المراجعين الى الأمر المقيد بالتحقق:

   ```bash
   DOCS_RELEASE_TAG=preview-<stamp> npm run --prefix docs/portal serve
   ```

   هذا يعيد استخدام `scripts/serve-verified-preview.mjs` حتى لا يتم تشغيل البناء غير محقق بالخطأ.

3. **منح الوصول GitHub (اختياري)** — اذا اتصلون بالاتصالون الى عناوين غير منشورة، اضفهم الى فريق
   GitHub `docs-preview` خلل تعديل العضوية في الطلب.

4. **قنوات الدعم** — مشاركة جهة الاتصال المناوبة (Matrix/Slack) وإجراءاتها
   [`incident-runbooks`](./incident-runbooks.md).

5. **القياس عن بعد + دفتر الملاحظات** — ذكر المراجع في تحليلات التحليلات المجهولة التي يتم جمعها
   (انظر [`observability`](./observability.md)). نموذج دفتر الملاحظات او قالب العدد في التحرير
   وتأكد عبر المساعدة [`preview-feedback-log`](./preview-feedback-log) كي يبقى ملخص محدثا.

## قائمة المراجع

قبل الوصول إلى المعاينة، يجب على المراجعين أكمل التالي:

1. التحقق من الاثار التي تم تنزيلها (`preview_verify.sh`).
2. تشغيل الشعار عبر `npm run serve` (او `serve:verified`) وتفعيل المجموع الاختباري.
3. قراءة التعليقات للآراء والرصد المتعلقة بعلاه.
4. اختبار وحدة OAuth/جربها باستخدام رمز جهاز تسجيل الدخول (ان كان ذلك مطبقًا) يساهم في إعادة الاستخدام
   إنتاج العناصر.
5. تسجيل المفكرة في المتتبع المتفق عليه (الإصدار أو مستند مشترك أو نموذج) ووضع تحديد إصدار المعاينة.

##مسؤولات مينتينرز وانهااء المشاركة

| المرحلة | لوقت |
| --- | --- |
| ضربة البداية | تاكيد ان قائمة كمية المرفقة بالطلب، مشاركة الاثار + التعليمات، اضافة ادخال `invite-sent` عبر [`preview-feedback-log`](./preview-feedback-log)، وجدولة نوبات في الفترة فيما اذا دمج لاكثر من اسبوع. |
| الرصد | قياس المعاينة (حركة جربها غير معتادة، فشل مراقبة التحقيق) وتباع runbook اذا كان هناك شيء مريب. سجل احداث `feedback-submitted`/`issue-opened` عند الوصول للملاحظة حتى تستمر مقاييس دقيقة. |
| الخروج من الطائرة | الغاء الوصول المؤقت الى GitHub او SoraFS، تسجيل `access-revoked`، ارشفة الطلب (يتضمن مذكرة + مذكرة)، وتحديث سجل المراجعين. اطلب من المراجع حذف الملخصات الجماعية والملخصات الصادرة من [`docs/examples/docs_preview_feedback_digest.md`](../../../examples/docs_preview_feedback_digest.md). |

استخدم نفس الطريقة عند مراجعين بين الإرسال. ضمان الاثر داخل المستودع (الاصدار + القالب)
يساعد DOCS-SORA على البقاء قابلا للتدقيق للأوبرا بتاكيد ان يصل المعاينة لمتابعة الموثقة.

## الأحداث والتتبع

- ابدأ كل تواصل بملف
  [`docs/examples/docs_preview_invite_template.md`](../../../examples/docs_preview_invite_template.md).
  يلتقط الحد الادنى من اللغة القانونية لذلك المجموع الاختباري للمعاينة وتوقع ان يقر المراجعون بالسياسة
  تستخدم لفترة طويلة.
- عند تحرير موقع الويب، تم استبدال النائبة لـ `<preview_tag>` و`<request_ticket>` وقنوات الاتصال.
  احفظ نسخة من الرسالة النهائية داخل تذكرة الكمية حتى المصادرون والمعتمدون والمقررون من
  الرجوع إلى النص الدقيق الذي أرسلته.
- بعد تنفيذ الأحداث، حدث جدول التتبع او الإصدار بطابعة `invite_sent_at`، تم الانتهاء من النشرة الرائعة حتى بدأت
  تقرير [تدفق دعوة المعاينة](./preview-invite-flow.md) من التقاط الدفعة الصفراء.
