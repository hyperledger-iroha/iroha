## 概述

志愿者简报捕捉了信息部在提出黑名单或政策变更时审查的支持/反对/背景观点。每次提交都必须遵循 `iroha_data_model::ministry::VolunteerBriefV1`（版本 `1`）实现的规范模式，以便：

1. 治理仪表板可以确定性地比较事实表。
2. 利益冲突披露在发布前强制执行。
3. 偏离主题或不合时宜的提交内容将被抑制，但不会将其从审计跟踪中删除。

使用 `docs/source/ministry/volunteer_brief_template.md` 或 `cargo xtask ministry-transparency ingest` 帮助程序中的模板构建有效负载。 CLI 验证 Norito 结构、lints 审核标签，并发出一个摘要，显示有多少事实行、引用和披露被接受。

## 发布流程和 SLA

|舞台|窗口|详情 |
|--------|--------|---------|
|摄入和验证| T+0 至 6 小时 |有效负载通过志愿者表单或 CLI 上传。 `VolunteerBriefV1::validate` 强制实施架构（版本、RFC3339 时间戳、披露策略、摘要格式）。失败会立即被拒绝，并带有可操作的错误代码。 |
|翻译和审核 | T+6 至 24 小时 |审阅者应用审核标签（`needs-translation`、`duplicate`、`policy-escalation` 等）、编辑联系人并标记偏离主题的条目。标签存储在 Norito 有效负载中，因此下游自动化可以抑制或突出显示条目。 |
|门户刊物| T+24 至 48 小时 |有效的简报与相应的提案页面一起发布在**志愿者意见**下。离题条目保留在原始存档中，但从总体立场计数中排除。 |

当存在 `policy-escalation` 或 `astroturf` 标签时，会自动触发加急处理（≤12 小时）。运营商可以通过志愿者简报 Grafana 面板审核合规性，该面板跟踪 `submitted_at` 和门户发布之间的时间。

## 可交付成果

- **架构和文档：** `VolunteerBriefV1` 位于共享数据模型中，详细的现场指南和示例有效负载位于 `docs/source/ministry/volunteer_brief_template.md` 和 `docs/examples/ministry/volunteer_brief_template.json` 中。
- **验证工具：** `cargo xtask ministry-transparency volunteer-validate --input <file> [--json-output artifacts/ministry/volunteer-lint.json]` 运行门户构建所使用的相同 lint；当您想要暂存整个捆绑包以供发布时，`ingest --volunteer <file>` 仍然可用。
- **门户展示：** 每个提案页面都链接到解码的支持/反对/背景摘要，使公民意见可见，而无需下载原始 JSON。审核元数据与简报一起呈现，以便陪审员了解条目可能被延迟或排除的原因。

如果需要额外的自动化（例如批量导入 API 或扩展分析），请参阅路线图项目 **MINFO-3 — 志愿者兴趣小组基础设施**。
