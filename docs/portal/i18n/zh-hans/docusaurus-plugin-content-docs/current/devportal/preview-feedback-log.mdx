每当我们邀请审稿人加入时，路线图都要求提供可验证的证据
文档门户预览。本说明解释了如何使用以下方法捕获证据
`preview-feedback-log` 帮助程序，如何与以下人员共享生成的工件
治理，以及浪潮结束后在哪里提交摘要。

## 为什么我们记录反馈

- **合规性：** 每个邀请都必须记录谁访问了预览、何时访问
  承认可接受的使用政策，以及他们的访问权限何时被撤销。
- **遥测关联：** 日志让我们可以关联门户遥测峰值
  或尝试特定邀请群体的流量。
- **治理证据：** 每个 DOCS-SORA 检查点都需要一个机器可读的
  显示邀请计数、开放审稿人以及杰出状态的摘要
  问题。

所有证据均位于 `artifacts/docs_portal_preview/` 下，因此可以附加
无需抓取 CI 日志即可进行邀请波票或治理审查。

## 直播邀请总结

`docs/portal/scripts/preview-feedback-summary.mjs` 摄取共享日志并
在 `npm run build` 期间写入 `src/data/previewFeedbackSummary.json`。桌子
下面呈现该 JSON，以便审阅者可以看到当前的邀请足迹
无需运行 CLI 帮助程序。



除了邀请总数之外，摘要现在还显示致谢和
撤销计数，得出的响应率（反馈提交除以
总邀请数），以及每次邀请之间的平均延迟（使用可悬停的 p95）
邀请和审稿人的确认或反馈提交。新的
**完成%** 列显示已关闭的邀请数量（已确认或已确认）
撤销），而 **Ack %** 突出显示有多少收件人明确确认
可接受的使用步骤。这些派生百分比 — 加上仅 JSON
撤销/问题率——提供给治理仪表板，以便审阅者可以确认
无需运行 CLI 帮助程序即可实现参与级别和周转时间。
**状态**列突出显示波浪是否静止
活跃 (`active`)，关闭但在 14 天监控窗口内 (`closed`)，或
准备存档 (`archived`)。 **自上次事件以来的天数**指标显示了如何
距离最后一次邀请/反馈行动已经过去很长时间了，以便议会能够发现
需要结束或重新吸引挥之不去的接收者的波浪。

## 事件分类

|活动 |什么时候记录|笔记|
| ---| ---| ---|
| `invite-sent` |发送邀请电子邮件后立即。 |为每个收件人附加一个条目，以便邀请计数保持准确。 |
| `acknowledged` |当审阅者确认他们阅读了安全性/可观察性文档并运行校验和门时。 |可选，但建议合作伙伴群体使用。 |
| `feedback-submitted` |当审阅者通过商定的跟踪器提交调查结果时。 |在 `--notes` 字段中包含问题/链接。 |
| `issue-opened` |当 DOCS-SORA 提交来自反馈的后续问题时。 |帮助治理将路线图项目关联起来以预览反馈。 |
| `access-revoked` |一旦审阅者的预览窗口结束或访问权限被提前撤销。 |必需的，以便“开放收件人”计数保持准确。 |

如果您需要其他活动，请将其添加到
`VALID_EVENTS` 内部 `docs/portal/scripts/preview-feedback-log.mjs` 并更新
这个表在同样的变化。

## 日志记录工作流程

`npm run preview:log` 背后的帮助程序编写一个捕获的 JSON 日志
所有事件。每个条目包含波标签、收件人、事件、时间戳和
可选注释。示例：

```json
{
  "version": 1,
  "generated_at": "2026-02-04T18:20:15.321Z",
  "entries": [
    {
      "wave": "preview-20260204",
      "recipient": "alice@example.com",
      "event": "invite-sent",
      "timestamp": "2026-02-04T18:05:02.114Z",
      "notes": "seed reviewer"
    },
    {
      "wave": "preview-20260204",
      "recipient": "alice@example.com",
      "event": "feedback-submitted",
      "timestamp": "2026-02-05T09:14:43.009Z",
      "notes": "docs-preview/42"
    }
  ]
}
```

命令：

```bash
# Append a new event (defaults to artifacts/docs_portal_preview/feedback_log.json)
npm run --prefix docs/portal preview:log -- \
  --wave preview-20260204 \
  --recipient alice@example.com \
  --event invite-sent \
  --notes "seed reviewer"

# Produce a text summary for the current repo
npm run --prefix docs/portal preview:log -- --summary

# Emit machine-readable summary for a specific wave
npm run --prefix docs/portal preview:log -- \
  --summary --summary-json --filter-wave preview-20260204 \
  > artifacts/docs_portal_preview/preview-20260204-summary.json

# Regenerate the portal summary data file
npm run --prefix docs/portal generate:preview-summary

# Generate a Markdown digest skeleton for a wave
npm run --prefix docs/portal preview:digest -- \
  --wave preview-20260204 \
  --invite-start 2026-02-04T00:00:00Z \
  --invite-end 2026-02-18T00:00:00Z \
  --dashboard "https://grafana.example/d/docs-preview?var-wave=preview-20260204" \
  --out artifacts/docs_portal_preview/preview-20260204-digest.md
```

温馨提示：

- 在功能分支中工作时将日志置于版本控制之下，以便其他
  维护者可以审查更改。对于长时间运行的波形，将日志存储在
  治理票证并将最终版本附加到发布工件中。
- 当您需要直接登录时，`preview:log` 接受 `--log <path>`
  SoraFS 捆绑包或 CI 工作区。
- 始终包含 `--notes` 提示（问题链接、GitHub 句柄、群组 ID），以便
  总结在回顾期间很有用。
- 当您需要确定性摘要时间戳时（对于 CI 或文档快照），
  在运行摘要之前设置 `PREVIEW_FEEDBACK_NOW=<iso8601 timestamp>`
  帮手。这使 `generatedAt`、状态和日计数器保持一致
  跨越重建。

## 摘要、摘要和档案1. **每波摘要：** 生成 JSON 摘要并将其附加到邀请中
   票证与原始日志一起。摘要列举了邀请总数、已开放
   收件人和反馈很重要，因此治理可以确认我们捕获了每一个
   审稿人。门户构建还在以下位置发布了汇总摘要：
   `src/data/previewFeedbackSummary.json`，以便利益相关者可以看到当前
   无需运行 CLI 即可生成快照。
2. **每周摘要：** 运行 `npm run --prefix docs/portal preview:digest -- --wave <label>`
   预先填充 Markdown 指标，然后定制占位符部分
   使用中的模板
   [`docs/examples/docs_preview_feedback_digest.md`](../../../examples/docs_preview_feedback_digest.md)
   在与利益相关者分享之前。至少包括：波标签、开放
   审稿人、值得注意的发现和下一步行动。
3. **归档：**将封闭波日志移入
   `artifacts/docs_portal_preview/archive/<wave>/` 并仅保留最新的
   根部有源波。这使得 CI 伪影很小，同时保留了
   审计追踪。

- 每当您询问时，请参考邀请和入职手册中的此文档
  审稿人提交反馈（参见
  [`preview-invite-flow`](./preview-invite-flow.md),

  [`public-preview-invite`](./public-preview-invite.md))。
- 关闭波形时在 `status.md` 中提及生成的摘要，以便 DOCS-SORA
  路线图项目（例如 DOCS-1b/2b/3c）可以引用工件。
- 上传到 SoraFS 等外部时，将哈希摘要与摘要一起存储
  审阅者可以验证证据包。

将日志记录工作流程编码在此处可以解除对路线图要求的阻碍
“捕获反馈/遥测以进行治理”并减少花费的时间
在每个里程碑重建邀请历史。
