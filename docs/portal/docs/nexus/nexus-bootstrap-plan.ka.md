# Sora Nexus ჩამტვირთავი და დაკვირვების გეგმა

## მიზნები
- დააყენეთ Sora Nexus საბაზისო ვალიდატორი/დამკვირვებელი ქსელი მართვის გასაღებებით, Torii API-ებით და კონსენსუსის მონიტორინგით.
- გადაამოწმეთ ძირითადი სერვისები (Torii, კონსენსუსი, მდგრადობა) SoraFS/SoraNet piggyback განლაგების ჩართვამდე.
- ჩამოაყალიბეთ CI/CD სამუშაო ნაკადები და დაკვირვებადობის დაფები/გაფრთხილებები ქსელის სიჯანსაღის უზრუნველსაყოფად.

## წინაპირობები
- მმართველობის საკვანძო მასალა (საბჭოთა მრავალრიცხოვანი, კომიტეტის გასაღებები) ხელმისაწვდომია HSM-ში ან Vault-ში.
- საბაზისო ინფრასტრუქტურა (Kubernetes კლასტერები ან შიშველი ლითონის კვანძები) პირველადი/მეორადი რეგიონებში.
- განახლებული ჩატვირთვის კონფიგურაცია (`configs/nexus/bootstrap/*.toml`), რომელიც ასახავს უახლეს კონსენსუსის პარამეტრებს.

## ქსელის გარემო
- მართეთ ორი Nexus გარემო განსხვავებული ქსელის პრეფიქსებით:
- **Sora Nexus (mainnet)** – წარმოების ქსელის პრეფიქსი `nexus`, კანონიკური მმართველობის ჰოსტინგი და SoraFS/SoraNet piggyback სერვისები (ჯაჭვის ID `0x02F1` / UU000001).
- **Sora Taira (ტესტი ქსელი)** – დადგმული ქსელის პრეფიქსი `taira`, მაგისტრალური ქსელის კონფიგურაციის ასახვა ინტეგრაციის ტესტირებისთვის და წინასწარ გამოშვების ვალიდაციისთვის (ჯაჭვის UUID `809574f5-fee7-5e69-bfcf-52451e42d50f`).
- შეინახეთ გენეზისის ცალკეული ფაილები, მართვის გასაღებები და ინფრასტრუქტურის კვალი თითოეული გარემოსთვის. Taira მოქმედებს როგორც დამადასტურებელი საფუძველი SoraFS/SoraNet-ის ყველა გამოშვებისთვის Nexus-ზე დაწინაურებამდე.
- CI/CD მილსადენები ჯერ უნდა განლაგდეს Taira-ზე, შეასრულოს კვამლის ავტომატური ტესტები და მოითხოვოს ხელით დაწინაურება Nexus-ზე შემოწმების გავლის შემდეგ.
- საცნობარო კონფიგურაციის პაკეტები მუშაობს `configs/soranexus/nexus/` (mainnet) და `configs/soranexus/taira/` (ტესტი ქსელი), თითოეული შეიცავს `config.toml`, `genesis.json` და Torii ნიმუშებს.

## ნაბიჯი 1 - კონფიგურაციის მიმოხილვა
1. არსებული დოკუმენტაციის აუდიტი:
   - `docs/source/nexus/architecture.md` (კონსენსუსი, Torii განლაგება).
   - `docs/source/nexus/deployment_checklist.md` (ინფრა მოთხოვნები).
   - `docs/source/nexus/governance_keys.md` (საკვანძო პატიმრობის პროცედურები).
2. გენეზისის ფაილების ვალიდაცია (`configs/nexus/genesis/*.json`) შეესაბამება მიმდინარე ვალიდიატორის სიას და ფსონების წონებს.
3. დაადასტურეთ ქსელის პარამეტრები:
   - კონსენსუსის კომიტეტის ზომა და კვორუმი.
   - დაბლოკეთ ინტერვალის / საბოლოო ზღურბლები.
   - Torii სერვისის პორტები და TLS სერთიფიკატები.

## ნაბიჯი 2 – Bootstrap Cluster-ის განლაგება
1. უზრუნველყოფის დამადასტურებელი კვანძები:
   - განათავსეთ `irohad` ინსტანციები (დამოწმებები) მუდმივი მოცულობით.
   - დარწმუნდით, რომ ქსელის firewall-ის წესები იძლევა კონსენსუსს და Torii ტრაფიკს კვანძებს შორის.
2. დაიწყეთ Torii სერვისები (REST/WebSocket) თითოეულ ვალიდატორზე TLS-ით.
3. განათავსეთ დამკვირვებელი კვანძები (მხოლოდ წაკითხვადი) დამატებითი გამძლეობისთვის.
4. გაუშვით bootstrap სკრიპტები (`scripts/nexus_bootstrap.sh`) გენეზის გასავრცელებლად, კონსენსუსის დასაწყებად და კვანძების დასარეგისტრირებლად.
5. შეასრულეთ კვამლის ტესტები:
   - გაგზავნეთ სატესტო ტრანზაქციები Torii (`iroha_cli tx submit`) საშუალებით.
   - გადაამოწმეთ ბლოკის წარმოება/ფინალი ტელემეტრიის საშუალებით.
   - შეამოწმეთ წიგნის რეპლიკაცია ვალიდატორებს/დამკვირვებლებს შორის.

## ნაბიჯი 3 – მმართველობა და გასაღების მართვა
1. დატვირთვის საკრებულოს მრავალსიგიანი კონფიგურაცია; დაადასტურეთ, რომ მმართველობის წინადადებები შეიძლება წარდგენილი და რატიფიცირებული იყოს.
2. უსაფრთხოდ შეინახეთ კონსენსუსის/კომიტეტის გასაღებები; ავტომატური სარეზერვო ასლების კონფიგურაცია წვდომის აღრიცხვით.
3. დააყენეთ გადაუდებელი გასაღების როტაციის პროცედურები (`docs/source/nexus/key_rotation.md`) და გადაამოწმეთ runbook.

## ნაბიჯი 4 – CI/CD ინტეგრაცია
1. მილსადენების კონფიგურაცია:
   - შექმენით და გამოაქვეყნეთ ვალიდატორი/Torii სურათები (GitHub Actions ან GitLab CI).
   - ავტომატური კონფიგურაციის ვალიდაცია (lint genesis, შეამოწმეთ ხელმოწერები).
   - მილსადენების განლაგება (Helm/Customize) დადგმისა და წარმოების კლასტერებისთვის.
2. განახორციელეთ კვამლის ტესტები CI-ში (გადატრიალეთ ეფემერული კლასტერი, გაუშვით კანონიკური ტრანზაქციის კომპლექტი).
3. დაამატეთ უკან დაბრუნების სკრიპტები წარუმატებელი განლაგებისა და დოკუმენტების გაშვების წიგნებისთვის.

## ნაბიჯი 5 – დაკვირვება და გაფრთხილებები
1. განათავსეთ მონიტორინგის დასტა (Prometheus + Grafana + Alertmanager) თითო რეგიონში.
2. შეაგროვეთ ძირითადი მეტრიკა:
  - `nexus_consensus_height`, `nexus_finality_lag`, `torii_request_duration_seconds`, `validator_peer_count`.
   - ჟურნალი Loki/ELK-ის მეშვეობით Torii და კონსენსუსის სერვისებისთვის.
3. დაფები:
   - კონსენსუსის ჯანმრთელობა (ბლოკის სიმაღლე, საბოლოოობა, თანატოლების სტატუსი).
   - Torii API შეყოვნების/შეცდომის სიხშირე.
   - მმართველობითი ტრანზაქციები და წინადადებების სტატუსები.
4. გაფრთხილებები:
   - ბლოკის წარმოების სადგომი (>2 ბლოკის ინტერვალით).
   - თანატოლების რაოდენობა ქვორუმზე დაბლა.
   - Torii შეცდომის სიხშირის მწვერვალები.
   - მმართველობის წინადადებების რიგის ჩამორჩენა.

## ნაბიჯი 6 – დადასტურება და გადაცემა
1. გაუშვით ვალიდაცია ბოლომდე:
   - წარადგინეთ მმართველობის წინადადება (მაგ. პარამეტრის შეცვლა).
   - მისი დამუშავება საკრებულოს დამტკიცების გზით, რათა უზრუნველყოს მმართველობითი მილსადენის მუშაობა.
   - განახორციელეთ ledger state diff თანმიმდევრულობის უზრუნველსაყოფად.
2. დოკუმენტის რენტაბელური ჩანაწერი გამოძახებისთვის (ინციდენტის რეაგირება, მარცხი, მასშტაბირება).
3. მზადყოფნის კომუნიკაცია SoraFS/SoraNet გუნდებთან; დაადასტურეთ Piggyback-ის განლაგება შეიძლება მიუთითებდეს Nexus კვანძებზე.

## განხორციელების ჩამონათვალი
- [ ] Genesis/კონფიგურაციის აუდიტი დასრულდა.
- [ ] ვალიდატორი და დამკვირვებელი კვანძები განლაგებულია ჯანსაღი კონსენსუსით.
- [ ] მართვის გასაღებები ჩაიტვირთა, წინადადება ტესტირება.
- [ ] CI/CD მილსადენები გაშვებული (აშენება + განლაგება + კვამლის ტესტები).
- [ ] დაკვირვებადობის დაფები ცოცხალია გაფრთხილებით.
- [ ] Handoff დოკუმენტაცია მიეწოდება ქვედა დინების გუნდებს.
