Бұл жұмыс кітабы көп көзді алу оркестрін дайындау, шығару және пайдалану арқылы SRE нұсқаулығын береді. Ол әзірлеуші ​​нұсқаулығын кезеңді қосуды және бір қатардағы қара тізімді қоса алғанда, өндірісті шығару үшін реттелген процедуралармен толықтырады.

> **Сонымен қатар қараңыз:** [Multi-Source Rollout Runbook](./multi-source-rollout.md) флоттағы тарату толқындарына және төтенше жағдайда провайдердің бас тартуына бағытталған. Бұл құжатты күнделікті оркестрлік операциялар үшін пайдалану кезінде басқару/қойылымды үйлестіру үшін оған сілтеме жасаңыз.

## 1. Ұшу алдындағы бақылау тізімі

1. **Провайдер деректерін жинау**
   - Соңғы провайдер жарнамалары (`ProviderAdvertV1`) және мақсатты флотқа арналған телеметриялық сурет.
   - Сынақтағы манифесттен алынған пайдалы жүктеме жоспары (`plan.json`).
2. **Дерминирленген таблоны көрсетіңіз**

   ```bash
   sorafs_fetch \
     --plan fixtures/plan.json \
     --telemetry-json fixtures/telemetry.json \
     --provider alpha=fixtures/provider-alpha.bin \
     --provider beta=fixtures/provider-beta.bin \
     --provider gamma=fixtures/provider-gamma.bin \
     --scoreboard-out artifacts/scoreboard.json \
     --json-out artifacts/session.summary.json
   ```

   - `artifacts/scoreboard.json` әрбір өндіріс провайдерін `eligible` ретінде тізімдейтінін растаңыз.
   - JSON қорытындысын таблоның жанында мұрағаттау; өзгерту сұрауын растау кезінде аудиторлар бөлшекті қайталау есептегіштеріне сүйенеді.
3. **Арнамалармен құрғақ жұмыс** — Өндірістік пайдалы жүктемелерге қол тигізбес бұрын оркестрдің екілік нұсқасы күтілетін нұсқаға сәйкес келетініне көз жеткізу үшін `docs/examples/sorafs_ci_sample/` жүйесіндегі жалпыға ортақ құрылғыларға қарсы бірдей пәрменді орындаңыз.

## 2. Кезеңді шығару процедурасы

1. **Канарлы кезең (≤2 провайдер)**
   - Таблоны қайта құрыңыз және оркестрді шағын жиынтыққа қысу үшін `--max-peers=2` арқылы іске қосыңыз.
   - Монитор:
     - `sorafs_orchestrator_active_fetches`
     - `sorafs_orchestrator_fetch_failures_total{reason!="retry"}`
     - `sorafs_orchestrator_retries_total`
   - Толық манифестті алу үшін қайталау ставкалары 1%-дан төмен болған соң жалғастырыңыз және ешбір провайдер сәтсіздіктерді жинамайды.
2. **Рампа кезеңі (50% провайдерлер)**
   - `--max-peers` көбейтіңіз және жаңа телеметриялық суретпен қайта орындаңыз.
   - `--provider-metrics-out` және `--chunk-receipts-out` көмегімен әр жұмысты жалғастырыңыз. Артефактілерді ≥7 күн бойы сақтаңыз.
3. **Толық шығару**
   - `--max-peers` жойыңыз (немесе оны толық жарамды санға орнатыңыз).
   - Клиент орналастыруларында оркестрлік режимді қосыңыз: конфигурацияны басқару жүйесі арқылы тұрақты көрсеткіштер тақтасы мен конфигурация JSON таратыңыз.
   - `sorafs_orchestrator_fetch_duration_ms` p95/p99 көрсету үшін бақылау тақталарын жаңартыңыз және әр аймақтағы гистограммаларды қайталап көріңіз.

## 3. Қара тізімге енгізу және арттыру

Басқару жаңартуларын күтпей-ақ дұрыс емес провайдерлерді тексеру үшін CLI бағалау саясатын қайта анықтауды пайдаланыңыз.

```bash
sorafs_fetch \
  --plan fixtures/plan.json \
  --telemetry-json fixtures/telemetry.json \
  --provider alpha=fixtures/provider-alpha.bin \
  --provider beta=fixtures/provider-beta.bin \
  --provider gamma=fixtures/provider-gamma.bin \
  --deny-provider=beta \
  --boost-provider=gamma=5 \
  --json-out artifacts/override.summary.json
```

- `--deny-provider` тізімдегі бүркеншік атты ағымдағы сеанс үшін қараудан жояды.
- `--boost-provider=<alias>=<weight>` провайдердің жоспарлаушы салмағын арттырады. Мәндер нормаланған табло салмағына қосымша болып табылады және тек жергілікті жүгіріске қолданылады.
- Оқиға билетіндегі қайта анықтауды жазыңыз және негізгі мәселе шешілгеннен кейін иеленуші топ күйді салыстыра алатындай JSON шығыстарын тіркеңіз.

Тұрақты өзгертулер үшін бастапқы телеметрияны өзгертіңіз (құқық бұзушыны жазалады деп белгілеңіз) немесе CLI қайта анықтауларын өшірмес бұрын жаңартылған ағын бюджеттерімен жарнаманы жаңартыңыз.

## 4. Сәтсіздік триажы

Алу сәтсіз болғанда:

1. Қайта іске қосу алдында келесі артефактілерді түсіріңіз:
   - `scoreboard.json`
   - `session.summary.json`
   - `chunk_receipts.json`
   - `provider_metrics.json`
2. `session.summary.json` адам оқи алатын қате жолын тексеріңіз:
   - `no providers were supplied` → провайдердің жолдары мен жарнамаларын тексеріңіз.
   - `retry budget exhausted ...` → `--retry-budget` ұлғайту немесе тұрақсыз теңдестірулерді жою.
   - `no compatible providers available ...` → бұзушы провайдердің ауқым мүмкіндігі метадеректерін тексеру.
3. Провайдер атын `sorafs_orchestrator_provider_failures_total` параметрімен салыстырыңыз және метрика өссе, бақылау билетін жасаңыз.
4. Сәтсіздікті анықтау үшін `--scoreboard-json` және түсірілген телеметрия арқылы алуды офлайн режимінде қайталаңыз.

## 5. Кері қайтару

Оркестрдің шығарылымын қайтару үшін:

2. Табло бейтарап салмаққа оралуы үшін кез келген `--boost-provider` қайта анықтауды алып тастаңыз.
3. Ұшу кезінде қалдық алулардың жоқтығын растау үшін кем дегенде бір күн бойы оркестр көрсеткіштерін қырып алуды жалғастырыңыз.

Тәртіпті артефакті түсіруді және кезеңді шығаруды қамтамасыз ету көп көзді оркестрдің бақылау мүмкіндігі мен аудит талаптарын сақтай отырып, әр түрлі провайдерлер флоттарында қауіпсіз жұмыс істеуін қамтамасыз етеді.
