## SoraFS መስቀለኛ ማከማቻ ንድፍ (ረቂቅ)

ይህ ማስታወሻ የIroha (Torii) መስቀለኛ መንገድ ወደ SoraFS ውሂብ እንዴት እንደሚገባ ያጠራዋል።
የተገኝነት ንብርብር እና ለማከማቸት እና ለማገልገል የአካባቢያዊ ዲስክ ቁራጭን ይወስኑ
ቁርጥራጭ። የ `sorafs_node_client_protocol.md` ግኝት ዝርዝር እና ያሟላል።
የ SF-1b መግጠሚያ ሥራ የማከማቻ-ጎን አርክቴክቸርን, ሀብትን በመዘርዘር
በመስቀለኛ መንገድ እና በመግቢያው ላይ ማረፍ ያለባቸው መቆጣጠሪያዎች, እና የውቅር ቧንቧዎች
ኮድ መንገዶች. ተግባራዊ ኦፕሬተር ልምምዶች በ ውስጥ ይኖራሉ
[የመስቀለኛ መንገድ ስራዎች Runbook](./node-operations)።

## ግቦች

- ማንኛውም አረጋጋጭ ወይም ረዳት Iroha ሂደት መለዋወጫ ዲስክን እንዲያጋልጥ ፍቀድ
  የ SoraFS አቅራቢ ዋናውን የሂሳብ መዝገብ ኃላፊነቶችን ሳይነካ።
- የማጠራቀሚያ ሞጁሉን ቆራጥነት እና Norito-የሚመራውን ያቆዩት፡ ይገለጣል፣
  ቸንክ ዕቅዶች፣ የማረጋገጫ ማረጋገጫ (PoR) ሥሮች እና የአቅራቢ ማስታወቂያዎች እነዚህ ናቸው
  የእውነት ምንጭ.
- አንድ መስቀለኛ መንገድ የራሱን ሀብቶች ማሟጠጥ እንዳይችል በኦፕሬተር የተገለጹ ኮታዎችን ያስፈጽሙ
  በጣም ብዙ ፒን መቀበል ወይም ጥያቄዎችን ማምጣት።
- የገጽታ ጤና/ቴሌሜትሪ (PoR ናሙና፣ ቸንክ ማምጣት መዘግየት፣ የዲስክ ግፊት)
  ወደ አስተዳደር እና ደንበኞች መመለስ.

### ከፍተኛ-ደረጃ አርክቴክቸር

```
┌──────────────────────────────────────────────────────────────────────┐
│                         Iroha/Torii Node                             │
│                                                                      │
│  ┌──────────────┐      ┌────────────────────┐                        │
│  │  Torii APIs  │◀────▶│   SoraFS Gateway   │◀───────────────┐       │
│  └──────────────┘      │ (Norito endpoints) │                │       │
│                        └────────┬───────────┘                │       │
│                                 │                            │       │
│                        ┌────────▼────────┐                   │       │
│                        │  Pin Registry   │◀───── manifests   │       │
│                        │ (State / DB)    │                   │       │
│                        └────────┬────────┘                   │       │
│                                 │                            │       │
│                        ┌────────▼────────┐                   │       │
│                        │  Chunk Storage  │◀──── chunk plans  │       │
│                        │  (ChunkStore)   │                   │       │
│                        └────────┬────────┘                   │       │
│                                 │                            │       │
│                        ┌────────▼────────┐                   │       │
│                        │  Disk Quota/IO  │─Pin/serve chunks─▶│ Fetch │
│                        │  Scheduler      │                   │ Clients│
│                        └─────────────────┘                   │       │
│                                                                      │
└──────────────────────────────────────────────────────────────────────┘
```

ቁልፍ ሞጁሎች፡

- ** ጌትዌይ ***፡ ለፒን ፕሮፖዛል፣ ቸንክ ማምጣት የNorito HTTP የመጨረሻ ነጥቦችን ያጋልጣል።
  ጥያቄዎች፣ የPoR ናሙና እና ቴሌሜትሪ። የ Norito ክፍያ ጭነቶችን ያረጋግጣል
  የማርሻልስ ጥያቄዎች ወደ chunk መደብር. ያለውን የI18NT0000024X HTTP ቁልል ዳግም ይጠቀማል
  አዲስ ዴሞንን ለማስወገድ.
- ** የፒን መዝገብ ቤት ***: በ`iroha_data_model::sorafs` ውስጥ የሚከታተለው አንጸባራቂ ፒን ሁኔታ
  እና `iroha_core`. መግለጫው ተቀባይነት ሲያገኝ መዝገቡ ይመዘግባል
  አንጸባራቂ መፍጨት፣ chunk plan diest፣ PoR root እና የአቅራቢ አቅም ባንዲራዎች።
- ** ቸንክ ማከማቻ ***: በዲስክ የሚደገፍ `ChunkStore` ትግበራ
  የተፈረመ መግለጫዎች፣ `ChunkProfile::DEFAULT` ን በመጠቀም የተቆራረጡ እቅዶችን እውን ያደርጋል፣ እና
  በቆራጥነት አቀማመጥ ስር ያሉ ቁርጥራጮችን ይቀጥላል። እያንዳንዱ ቁራጭ ከሀ ጋር የተያያዘ ነው
  የይዘት አሻራ እና የPoR ሜታዳታ ስለዚህ ናሙና ያለሱ እንደገና ማረጋገጥ ይችላል።
  ሙሉውን ፋይል እንደገና ማንበብ.
- ** ኮታ/ መርሐግብር አዘጋጅ ***፡ በኦፕሬተር የተዋቀሩ ገደቦችን ያስፈጽማል (ከፍተኛው የዲስክ ባይት፣
  ከፍተኛው የላቀ ካስማዎች፣ ከፍተኛ ትይዩ ፈልሳፊዎች፣ ቸንክ ቲቲኤል) እና መጋጠሚያዎች
  IO ስለዚህ የመስቀለኛ መንገድ ደብተር ግዴታዎች አይራቡም። መርሐግብር አውጪውም እንዲሁ
  የPoR ማረጋገጫዎችን እና የናሙና ጥያቄዎችን ከተገደበ ሲፒዩ ጋር የማገልገል ኃላፊነት አለበት።

### ማዋቀር

አዲስ ክፍል ወደ I18NI0000038X ያክሉ፡

```toml
[sorafs.storage]
enabled = false
data_dir = "/var/lib/iroha/sorafs"
max_capacity_bytes = "100 GiB"
max_parallel_fetches = 32
max_pins = 10_000
por_sample_interval_secs = 600
alias = "tenant.alpha"            # optional human friendly tag
adverts:
  stake_pointer = "stake.pool.v1:0x1234"
  availability = "hot"
  max_latency_ms = 500
  topics = ["sorafs.sf1.primary:global"]
```

- `enabled`: ተሳትፎ መቀያየርን. ሲዋሽ መግቢያው 503 ለ
  የማከማቻ መጨረሻ ነጥቦች እና መስቀለኛ መንገድ በግኝት ውስጥ አያስተዋውቅም።
- `data_dir`፡ ለ chunk data፣ PoR ዛፎች፣ እና የቴሌሜትሪ ፈልሳፊ ስርወ ማውጫ።
  የ I18NI0000041X ነባሪዎች።
- `max_capacity_bytes`: ለተሰካው ቸንክ ውሂብ ከባድ ገደብ። የበስተጀርባ ተግባር
  ገደቡ ሲደርስ አዲስ ፒኖችን ውድቅ ያደርጋል።
- `max_parallel_fetches`፡ የሚዛናዊነትን ለመጠበቅ በጊዜ ሰሌዳው ተፈጻሚነት ያለው የኮንፈረንስ ካፕ
  የመተላለፊያ ይዘት / ዲስክ IO ከአረጋጋጭ የስራ ጫና ጋር።
- `max_pins`: ከመተግበሩ በፊት መስቀለኛ መንገዱ የሚቀበለው ከፍተኛው የማሳያ ፒኖች ብዛት
  ማስወጣት / የጀርባ ግፊት.
- `por_sample_interval_secs`: ለራስ-ሰር የPoR ናሙና ስራዎች ማስረጃ። እያንዳንዱ ሥራ
  ናሙናዎች `N` ቅጠሎች (በማሳያ የሚዋቀር) እና የቴሌሜትሪ ክስተቶችን ያስወጣሉ።
  የአቅም ሜታዳታ በማዘጋጀት አስተዳደር `N` ሊለካ ይችላል።
  ቁልፍ `profile.sample_multiplier` (ኢንቲጀር `1-4`)። እሴቱ ነጠላ ሊሆን ይችላል።
  ቁጥር/ሕብረቁምፊ ወይም በፕሮፋይል መሻሮች ያለው ነገር፣ ለምሳሌ
  `{"default":2,"sorafs.sf2@1.0.0":3}`.
- `adverts`: ለመሙላት አቅራቢው የማስታወቂያ ጀነሬተር የሚጠቀምበት መዋቅር
  `ProviderAdvertV1` መስኮች (ካስማ ጠቋሚ፣ QoS ፍንጮች፣ ርዕሶች)። ከተተወ
  መስቀለኛ መንገድ ከአስተዳደር መዝገብ ቤት ነባሪዎችን ይጠቀማል።

የቧንቧ ማዋቀር

- `[sorafs.storage]` በ I18NI0000054X እንደ I18NI0000055X ይገለጻል እና ነው
  ከኖድ ውቅር ፋይል ተጭኗል።
- `iroha_core` እና `iroha_torii` የማከማቻ ውቅሩን ወደ መግቢያው ውስጥ ይሰርዛሉ
  ጅምር ላይ ግንበኛ እና ቸንክ መደብር።
- Dev/የሙከራ env መሻር አለ (`SORAFS_STORAGE_*`፣ `SORAFS_STORAGE_PIN_*`)፣ ነገር ግን
  የምርት ማሰማራቶች በማዋቀር ፋይሉ ላይ መተማመን አለባቸው.

### CLI መገልገያዎች

የTorii የኤችቲቲፒ ወለል በሽቦ እየተሰራ ባለበት ወቅት፣ የ `sorafs_node` ሳጥን
ቀጭን CLI ስለዚህ ኦፕሬተሮች የመግቢያ/የመላክ ልምምድ በቋሚው ላይ መፃፍ ይችላሉ።
የኋላ።【crates/sorafs_node/src/bin/sorafs-node.rs:1】

```bash
cargo run -p sorafs_node --bin sorafs-node ingest \
  --data-dir ./storage/sorafs \
  --manifest ./fixtures/manifest.to \
  --payload ./fixtures/payload.bin \
  --plan-json-out ./plan.json
```

- `ingest` Norito-የተመሰጠረ የሰነድ መግለጫ `.to` ፋይል እና ተዛማጅ ጭነት ይጠብቃል
  ባይት. ከአንጸባራቂው መቆራረጥ ፕሮፋይል የችኩክ እቅዱን እንደገና ይገነባል፣
  የመዋሃድ እኩልነትን ያስፈጽማል፣ የተቆራረጡ ፋይሎችን ይቀጥላል እና እንደ አማራጭ ሀ
  `chunk_fetch_specs` JSON ብሎብ ስለዚህ የታችኛው ተፋሰስ መሣሪያ ጤነኛነት ማረጋገጥ ይችላል
  አቀማመጥ.
