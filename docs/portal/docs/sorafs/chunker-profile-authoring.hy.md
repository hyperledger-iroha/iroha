# SoraFS Chunker պրոֆիլի հեղինակային ուղեցույց

Այս ուղեցույցը բացատրում է, թե ինչպես առաջարկել և հրապարակել նոր chunker պրոֆիլներ SoraFS-ի համար:
Այն լրացնում է ճարտարապետության RFC (SF-1) և ռեեստրի հղումը (SF-2a)
հեղինակային կոնկրետ պահանջներով, վավերացման քայլերով և առաջարկի ձևանմուշներով:
Կանոնական օրինակի համար տե՛ս
`docs/source/sorafs/proposals/sorafs_sf1_profile_v1.json`
և ուղեկցող չոր վազում մուտք գործեք
`docs/source/sorafs/reports/sf1_determinism.md`.

## Տեսություն

Յուրաքանչյուր պրոֆիլ, որը մտնում է ռեեստր, պետք է.

- գովազդեք CDC-ի դետերմինիստական պարամետրերը և միանման բազմահաշի կարգավորումները
  ճարտարապետություններ;
- առաքել վերարտադրվող հարմարանքներ (Rust/Go/TS JSON + fuzz corpora + PoR վկաներ), որոնք
  ներքևի SDK-ները կարող են ստուգել առանց պատվիրված գործիքների.
- ներառել կառավարման համար պատրաստ մետատվյալներ (անունների տարածություն, անուն, սեմվեր) գումարած միգրացիա
- անցնել որոշիչ տարբերությունների փաթեթը նախքան խորհրդի վերանայումը:

Հետևեք ստորև բերված ստուգաթերթին՝ այդ կանոններին համապատասխանող առաջարկ պատրաստելու համար:

## Ռեեստրի կանոնադրության ակնարկ

Նախքան առաջարկի մշակումը հաստատեք, որ այն համապատասխանում է գրանցման կանոնադրությանը
կողմից `sorafs_manifest::chunker_registry::ensure_charter_compliance()`:

- Պրոֆիլների ID-ները դրական ամբողջ թվեր են, որոնք միապաղաղ աճում են առանց բացերի:
- Կանոնական բռնակը (`namespace.name@semver`) պետք է հայտնվի այլանունների ցանկում
  և **պետք է** լինի առաջին մուտքը:
- Ոչ մի կեղծանուն չի կարող բախվել մեկ այլ կանոնական բռնակի կամ հայտնվել մեկից ավելի անգամ:
- Փոխանունները պետք է լինեն ոչ դատարկ և կտրված լինեն բացատներից:

Հարմար CLI օգնականներ.

```bash
# JSON listing of all registered descriptors (ids, handles, aliases, multihash)
cargo run -p sorafs_manifest --bin sorafs_manifest_chunk_store -- --list-profiles

# Emit metadata for a candidate default profile (canonical handle + aliases)
cargo run -p sorafs_manifest --bin sorafs_manifest_chunk_store -- \
  --promote-profile=sorafs.sf1@1.0.0 --json-out=-
```

Այս հրամանները պահպանում են առաջարկները ռեեստրի կանոնադրությանը համապատասխան և ապահովում են
կանոնական մետատվյալներ, որոնք անհրաժեշտ են կառավարման քննարկումներում:

## Պահանջվող մետատվյալներ

| Դաշտային | Նկարագրություն | Օրինակ (`sorafs.sf1@1.0.0`) |
|-------|--------------------------------------------|
| `namespace` | Տրամաբանական խմբավորում հարակից պրոֆիլների համար: | `sorafs` |
| `name` | Մարդու համար ընթեռնելի պիտակ. | `sf1` |
| `semver` | Պարամետրերի հավաքածուի իմաստային տարբերակի տողը: | `1.0.0` |
| `profile_id` | Միապաղաղ թվային նույնացուցիչը նշանակվում է պրոֆիլի վայրէջքից հետո: Ամրագրեք հաջորդ ID-ն, բայց մի օգտագործեք առկա համարները: | `1` |
| `profile_aliases` | Ընտրովի լրացուցիչ բռնակներ, որոնք բացահայտվում են հաճախորդներին բանակցությունների ընթացքում: Միշտ ներառեք կանոնական բռնակը որպես առաջին մուտք: | `["sorafs.sf1@1.0.0"]` |
| `profile.min_size` | Հատվածի նվազագույն երկարությունը բայթերով: | `65536` |
| `profile.target_size` | Թիրախային հատվածի երկարությունը բայթերով: | `262144` |
| `profile.max_size` | Հատվածի առավելագույն երկարությունը բայթերով: | `524288` |
| `profile.break_mask` | Հարմարվողական դիմակ, որն օգտագործվում է շարժակազմի հեշի կողմից (վեցանկյուն): | `0x0000ffff` |
| `profile.polynomial` | Փոխանցման բազմանդամ հաստատուն (վեցանկյուն): | `0x3da3358b4dc173` |
| `gear_seed` | Սերմն օգտագործվում էր 64 KiB հանդերձանքի աղյուսակը ստանալու համար: | `sorafs-v1-gear` |
| `chunk_multihash.code` | Մուլտիհեշ կոդ՝ յուրաքանչյուր կտորի մարսողության համար: | `0x1f` (BLAKE3-256) |
| `chunk_multihash.digest` | Կանոնական հարմարանքների փաթեթի ամփոփում: | `13fa...c482` |
| `fixtures_root` | Վերականգնված հարմարանքները պարունակող հարաբերական գրացուցակ: | `fixtures/sorafs_chunker/sorafs.sf1@1.0.0/` |
| `por_seed` | Սերմ դետերմինիստական ​​PoR նմուշառման համար (`splitmix64`): | `0xfeedbeefcafebabe` (օրինակ) |

Մետատվյալները պետք է հայտնվեն ինչպես առաջարկի փաստաթղթում, այնպես էլ ստեղծված փաստաթղթում
հարմարանքներ, որպեսզի ռեեստրը, CLI գործիքավորումը և կառավարման ավտոմատացումը կարողանան հաստատել
արժեքներ՝ առանց ձեռքով խաչաձև հղումների: Երբ կասկածում եք, գործարկեք chunk-store-ը և
ցուցադրեք CLI-ները `--json-out=-`-ով` հաշվարկված մետատվյալները վերանայելու համար
նշումներ.

### CLI և ռեեստրի հպման կետեր

- `sorafs_manifest_chunk_store --profile=<handle>` – վերագործարկեք մետատվյալները,
  manifest digest, PoR-ը ստուգում է առաջարկվող պարամետրերով:
- `sorafs_manifest_chunk_store --json-out=-` – հեռարձակեք chunk-store հաշվետվությունը դեպի
  stdout ավտոմատ համեմատությունների համար:
- `sorafs_manifest_stub --chunker-profile=<handle>` – հաստատեք մանիֆեստները և մեքենան
  պլանները ներկառուցում են կանոնական բռնակը գումարած փոխանունները:
- `sorafs_manifest_stub --plan=-` – վերադարձրեք նախորդ `chunk_fetch_specs`-ը
  ներս՝ ստուգելու փոխհատուցումները/մարսումները՝ փոփոխություններից հետո:

Գրանցեք հրամանի ելքը (մարսումներ, PoR արմատներ, մանիֆեստային հեշեր) առաջարկի մեջ
այնպես որ գրախոսները կարող են դրանք բառացիորեն վերարտադրել:

## Դետերմինիզմ և վավերացման ստուգաթերթ

1. **Վերականգնել հարմարանքները **
   ```bash
   cargo run --locked -p sorafs_chunker --bin export_vectors \
     --signature-out=fixtures/sorafs_chunker/manifest_signatures.json
   ```
2. **Գործարկեք հավասարության փաթեթը** – `cargo test -p sorafs_chunker` և
   միջլեզու տարբերության զրահը (`crates/sorafs_chunker/tests/vectors.rs`) պետք է լինի
   կանաչ՝ նոր հարմարանքներով:
3. ** Կրկնել fuzz/back-pressure corpora** – գործարկել `cargo fuzz list` և
   հոսքային զրահ (`fuzz/sorafs_chunker`) վերականգնված ակտիվների դեմ:
4. **Ստուգեք առբերելիության ապացույցները** – վազել
   `sorafs_manifest_chunk_store --por-sample=<n>` օգտագործելով առաջարկվող պրոֆիլը և
   հաստատեք, որ արմատները համապատասխանում են ամրացման մանիֆեստին:
5. **CI dry run ** – կանչեք `ci/check_sorafs_fixtures.sh` տեղական; սցենարը
   պետք է հաջողվի նոր հարմարանքներով և գոյություն ունեցող `manifest_signatures.json`-ով:
6. **Cross-runtime հաստատում** – ապահովել Go/TS կապանքները սպառում են վերականգնվածը
   JSON և արտանետում են միանման մասերի սահմաններ և մարսողություններ:

Փաստաթղթավորեք հրամանները և արդյունքում ստացված ամփոփումները առաջարկության մեջ, որպեսզի Tooling WG-ն
կարող է դրանք նորից գործարկել առանց գուշակությունների:

### Մանիֆեստ / PoR հաստատում

Սարքերը վերականգնելուց հետո գործարկեք ամբողջական մանիֆեստի խողովակաշարը՝ CAR-ն ապահովելու համար
մետատվյալները և PoR ապացույցները մնում են հետևողական.

```bash
# Validate chunk metadata + PoR with the new profile
cargo run -p sorafs_manifest --bin sorafs_manifest_chunk_store -- \
  --profile=sorafs.sf2@1.0.0 \
  --json-out=- --por-json-out=- fixtures/sorafs_chunker/input.bin

# Generate manifest + CAR and capture chunk fetch specs
cargo run -p sorafs_manifest --bin sorafs_manifest_stub -- \
  fixtures/sorafs_chunker/input.bin \
  --chunker-profile=sorafs.sf2@1.0.0 \
  --chunk-fetch-plan-out=chunk_plan.json \
  --manifest-out=sf2.manifest \
  --car-out=sf2.car \
  --json-out=sf2.report.json

# Re-run using the saved fetch plan (guards against stale offsets)
cargo run -p sorafs_manifest --bin sorafs_manifest_stub -- \
  fixtures/sorafs_chunker/input.bin \
  --chunker-profile=sorafs.sf2@1.0.0 \
  --plan=chunk_plan.json --json-out=-
```

Փոխարինեք մուտքագրված ֆայլը ցանկացած ներկայացուցչական կորպուսով, որն օգտագործվում է ձեր հարմարանքների կողմից
(օրինակ՝ 1GiB դետերմինիստական հոսքը) և կցեք ստացված մարսողությունները
առաջարկություն.

## Առաջարկի ձևանմուշ

Առաջարկները ներկայացվում են որպես `ChunkerProfileProposalV1` Norito գրանցումներ
`docs/source/sorafs/proposals/`. Ստորև բերված JSON ձևանմուշը ցույց է տալիս սպասվածը
ձևը (փոխարինեք ձեր արժեքները ըստ անհրաժեշտության).


Տրամադրել համապատասխան Markdown հաշվետվություն (`determinism_report`), որը ներառում է
հրամանի ելք, կտորների բովանդակություն և վավերացման ընթացքում հանդիպող ցանկացած շեղում:

## Կառավարման աշխատանքային հոսք

1. **Ներկայացրեք PR առաջարկով + հարմարանքներով։** Ներառեք ստեղծված ակտիվները,
   Norito առաջարկը և `chunker_registry_data.rs`-ի թարմացումները:
2. **Tooling WG-ի վերանայում.** Վերանայողները վերագործարկում են վավերացման ստուգաթերթը և հաստատում
   առաջարկը համընկնում է ռեգիստրի կանոններին (ID-ի կրկնակի օգտագործում չկա, դետերմինիզմը բավարարված է):
3. **Խորհրդի ծրար.** Հաստատվելուց հետո ավագանու անդամները ստորագրում են առաջարկի ամփոփագիրը
   (`blake3("sorafs-chunker-profile-v1" || canonical_bytes)`) և կցել դրանց
   ստորագրություններ պրոֆիլի ծրարի վրա, որոնք պահվում են հարմարանքների կողքին:
4. **Ռեգեստրի հրապարակում.** Միավորել ռեեստրը, փաստաթղթերը և հարմարանքները: Այն
   լռելյայն CLI-ն մնում է նախորդ պրոֆիլում, քանի դեռ ղեկավարությունը չի հայտարարում
   միգրացիան պատրաստ է.
5. **Հեռացման հետագծում։** Միգրացիայի պատուհանից հետո թարմացրեք ռեեստրը
   մատյան.

## Հեղինակային խորհուրդներ

- Նախընտրեք նույնիսկ երկու հզորության սահմանները՝ եզրերի պատյանների բեկման պահվածքը նվազագույնի հասցնելու համար:
- Խուսափեք բազմահեշ կոդը փոխելուց՝ առանց մանիֆեստի և դարպասի համակարգման
- Պահպանեք հանդերձանքի սեղանի սերմերը մարդու համար ընթեռնելի, բայց ամբողջ աշխարհում եզակի՝ աուդիտը պարզեցնելու համար
  արահետներ.
- Պահպանեք չափորոշիչի ցանկացած արտեֆակտ (օրինակ՝ թողունակության համեմատությունները) տակ
  `docs/source/sorafs/reports/` հետագա հղման համար:

Տեղադրման ընթացքում գործառնական ակնկալիքների համար տես միգրացիայի մատյան
(`docs/source/sorafs/migration_ledger.md`): Գործարկման ժամանակի համապատասխանության կանոնները տե՛ս
`docs/source/sorafs/chunker_conformance.md`.
