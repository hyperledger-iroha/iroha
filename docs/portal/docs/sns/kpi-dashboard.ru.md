# Панель KPI Sora Name Service

Панель KPI дает стюардам, гардианам и регуляторам единое место для проверки сигналов принятия, ошибок и выручки перед ежемесячной каденцией аннекса (SN-8a). Определение Grafana хранится в репозитории по пути `dashboards/grafana/sns_suffix_analytics.json`, а портал отображает те же панели через встроенный iframe, чтобы опыт совпадал с внутренним экземпляром Grafana.

## Фильтры и источники данных

- **Фильтр суффикса** – ведет запросы `sns_registrar_status_total{suffix}`, чтобы `.sora`, `.nexus` и `.dao` можно было проверять отдельно.
- **Фильтр массовых релизов** – ограничивает метрики `sns_bulk_release_payment_*`, чтобы финансы могли сверять конкретный манифест регистратора.
- **Метрики** – берет из Torii (`sns_registrar_status_total`, `torii_request_duration_seconds`), CLI guardian (`guardian_freeze_active`), `sns_governance_activation_total` и метрик помощника массового онбординга.

## Панели

1. **Регистрации (последние 24h)** – количество успешных событий регистратора для выбранного суффикса.
2. **Активации управления (30d)** – инициативы устава/дополнения, зафиксированные CLI.
3. **Пропускная способность регистратора** – скорость успешных действий регистратора по суффиксу.
4. **Режимы ошибок регистратора** – 5-минутная скорость счетчиков `sns_registrar_status_total` с меткой ошибки.
5. **Окна заморозки guardian** – активные селекторы, где `guardian_freeze_active` сообщает об открытом тикете заморозки.
6. **Чистые платежные единицы по активам** – итоги, сообщенные `sns_bulk_release_payment_net_units` по каждому активу.
7. **Массовые запросы по суффиксу** – объемы манифестов по id суффикса.
8. **Чистые единицы на запрос** – расчет в стиле ARPU, полученный из метрик релиза.

## Ежемесячный чек-лист KPI

Финансовый руководитель проводит регулярный обзор в первый вторник каждого месяца:

1. Откройте страницу портала **Analytics → SNS KPI** (или дашборд Grafana `sns-kpis`).
2. Снимите экспорт PDF/CSV таблиц пропускной способности регистратора и выручки.
3. Сравните суффиксы на нарушения SLA (скачки ошибок, замороженные селекторы >72 h, дельты ARPU >10 %).
4. Запишите резюме и действия в соответствующем аннексе по пути `docs/source/sns/regulatory/<suffix>/YYYY-MM.md`.
5. Приложите экспортированные артефакты дашборда к коммиту аннекса и свяжите их в повестке совета.

Если обзор выявляет нарушения SLA, создайте инцидент PagerDuty для затронутого владельца (registrar duty manager, guardian on-call или steward program lead) и отслеживайте устранение в журнале аннекса.
