SNNet-17B შემოაქვს ფიქსირებული ტარიფის სატრანსპორტო ზოლები, რათა რელეები გადაადგილდეს ტრაფიკს 1024 B უჯრედებში, მიუხედავად
ტვირთამწეობის ზომის. ოპერატორები ირჩევენ სამი წინასწარ დაყენებიდან:

- **ბირთვი** - მონაცემთა ცენტრი ან პროფესიონალურად განთავსებული რელეები, რომლებსაც შეუძლიათ დაფარონ >=30 Mbps
  მოძრაობა.
- **სახლი** - საცხოვრებელი ან დაბალი ბმულის ოპერატორები, რომლებსაც ჯერ კიდევ სჭირდებათ ანონიმური მოზიდვა
  კონფიდენციალურობის კრიტიკული სქემები.
- **null** - SNNet-17A2 წინასწარ დაყენებული წინასწარი ტესტირება. ის ინარჩუნებს იგივე TLVs/კონვერტს, მაგრამ აჭიმავს
  ტკიპი და ჭერი დაბალი სიჩქარის დადგმისთვის.

## წინასწარ დაყენებული შეჯამება

| პროფილი | Tick ​​(ms) | უჯრედი (B) | ზოლის ქუდი | მოჩვენებითი იატაკი | თითო ზოლზე დატვირთვა (მბ/წმ) | ჭერის დატვირთვა (მბ/წმ) | ჭერის % uplink | რეკომენდირებული ბმულები (მბ/წმ) | მეზობლის ქუდი | ტრიგერის ავტომატური გამორთვა (%) |
|---------|----------|---------|---------|-----------|------------------------------------- --------|------------------------------------------------------------------------------------------
| ბირთვი | 5.0 | 1024 | 12 | 4 | 1.64 | 19.50 | 65 | 30.0 | 8 | 85 |
| მთავარი | 10.0 | 1024 | 4 | 2 | 0.82 | 4.00 | 40 | 10.0 | 2 | 70 |
| null | 20.0 | 1024 | 2 | 1 | 0.41 | 0.75 | 15 | 5.0 | 1 | 55 |

- ** ზოლის ქუდი ** - მაქსიმალური თანმხლები მუდმივი სიჩქარის მეზობლები. რელე ერთხელ უარყოფს დამატებით სქემებს
  თავსახური დაარტყა და მატულობს `soranet_handshake_capacity_reject_total`.
- ** მოჩვენებითი იატაკი ** - ზოლების მინიმალური რაოდენობა, რომლებიც ცოცხალი რჩება მოჩვენებითი ტრაფიკის დროსაც კი
  მოთხოვნა ნაკლებია.
- **ჭერის ტვირთამწეობა** - ამაღლების ბიუჯეტი, რომელიც ეძღვნება მუდმივი განაკვეთის ზოლებს ჭერის გამოყენების შემდეგ
  წილადი. ოპერატორებმა არასოდეს უნდა გადააჭარბონ ამ ბიუჯეტს, თუნდაც დამატებითი გამტარუნარიანობა იყოს ხელმისაწვდომი.
- **ავტომატური გამორთვა ტრიგერით** - მდგრადი გაჯერების პროცენტი (საშუალოდ წინასწარ დაყენებულზე), რაც იწვევს
  გაშვების დრო მოტყუებულ იატაკზე ჩამოსასვლელად. სიმძლავრე აღდგება აღდგენის ბარიერის შემდეგ
  (75% `core`-სთვის, 60% `home`-სთვის, 45% `null`-სთვის).

**მნიშვნელოვანია:** წინასწარ დაყენებული `null` განკუთვნილია მხოლოდ ინსცენირებისა და ფუნქციონალური ტესტირებისთვის; ის არ აკმაყოფილებს
საწარმოო სქემებისთვის საჭირო კონფიდენციალურობის გარანტიები.

## მონიშნეთ -> გამტარუნარიანობის ცხრილი

თითოეული დატვირთვის უჯრედი ატარებს 1,024 B, ასე რომ KiB/sec სვეტი უდრის გამოსხივებული უჯრედების რაოდენობას.
მეორე. გამოიყენეთ დამხმარე ცხრილის გასაგრძელებლად მორგებული ტკიპებით.

| Tick ​​(ms) | უჯრედები/წმ | დატვირთვის KiB/წმ | დატვირთვა მბ/წმ |
|-----------|-----------|---------------|--------------|
| 5.0 | 200.00 | 200.00 | 1.64 |
| 7.5 | 133.33 | 133.33 | 1.09 |
| 10.0 | 100.00 | 100.00 | 0.82 |
| 15.0 | 66.67 | 66.67 | 0.55 |
| 20.0 | 50.00 | 50.00 | 0.41 |

ფორმულა:

```
payload_mbps = (cell_bytes x 8 / 1_000_000) x (1000 / tick_ms)
```

CLI დამხმარე:

```bash
# Markdown table output for all presets plus default tick table
cargo xtask soranet-constant-rate-profile --tick-table --format markdown --json-out artifacts/soranet/constant_rate/report.json

# Restrict to a preset and emit JSON
cargo xtask soranet-constant-rate-profile --profile core --format json

# Custom tick series
cargo xtask soranet-constant-rate-profile --tick-table --tick-values 5,7.5,12,18 --format markdown
```

`--format markdown` ავრცელებს GitHub-ის სტილის ცხრილებს როგორც წინასწარ დაყენებული შეჯამებისთვის, ასევე არჩევითი მოტყუებისთვის
ფურცელი, ასე რომ თქვენ შეგიძლიათ ჩასვათ დეტერმინისტული გამომავალი პორტალში. დააწყვილეთ იგი `--json-out`-თან არქივისთვის
მიღებული მონაცემები მმართველობის მტკიცებულებებისთვის.

## კონფიგურაცია და უგულებელყოფა

`tools/soranet-relay` ასახავს წინასწარ დაყენებებს როგორც კონფიგურაციის ფაილებში, ასევე გაშვების დროის უგულებელყოფაში:

```bash
# Persisted in relay.json
"constant_rate_profile": "home"

# One-off override during rollout or maintenance
soranet-relay --config relay.json --constant-rate-profile core
```

კონფიგურაციის გასაღები იღებს `core`, `home`, ან `null` (ნაგულისხმევი `core`). CLI უგულებელყოფა სასარგებლოა
სავარჯიშოების დადგმა ან SOC მოთხოვნები, რომლებიც დროებით ამცირებს სამუშაო ციკლს კონფიგურაციის გადაწერის გარეშე.

## MTU დამცავი მოაჯირები

- დატვირთვის უჯრედები იყენებენ 1,024 B პლუს ~96 B Norito+ Noise ჩარჩოს და მინიმალური QUIC/UDP სათაურებს,
  თითოეული დატაგრამის შენარჩუნება IPv6 1,280 B მინიმალური MTU-ის ქვემოთ.
- როდესაც გვირაბები (WireGuard/IPsec) დაამატებენ დამატებით ინკაპსულაციას, თქვენ ** უნდა** შეამციროთ `padding.cell_size`
  ასე რომ, `cell_size + framing <= 1,280 B`. რელეს ვალიდატორი ახორციელებს ძალას
  `padding.cell_size <= 1,136 B` (1,280 B - 48 B UDP/IPv6 ზედნადები - 96 B ჩარჩო).
- `core` პროფილებმა უნდა დაამაგრონ >=4 მეზობელი, მაშინაც კი, როცა უმოქმედოა, ასე რომ მოჩვენებითი ზოლები ყოველთვის მოიცავს ქვეჯგუფს
  PQ მცველები. `home` პროფილებმა შეიძლება შეზღუდონ მუდმივი სიჩქარის სქემები საფულეებზე/აგრეგატორებზე, მაგრამ უნდა გამოიყენონ
  უკანა წნევა, როდესაც გაჯერება აღემატება 70%-ს სამი ტელემეტრიული ფანჯრისთვის.

## ტელემეტრია და გაფრთხილებები

რელეები ახორციელებს შემდეგი მეტრიკის ექსპორტს წინასწარ დაყენებულზე:

- `soranet_constant_rate_active_neighbors`
- `soranet_constant_rate_queue_depth`
- `soranet_constant_rate_saturation_percent`
- `soranet_constant_rate_dummy_lanes` / `soranet_constant_rate_dummy_ratio`
- `soranet_constant_rate_slot_rate_hz`
- `soranet_constant_rate_ceiling_hits_total`
- `soranet_constant_rate_degraded`

გაფრთხილება, როდესაც:

1. მოჩვენებითი თანაფარდობა რჩება წინასწარ დაყენებულ სართულზე ქვემოთ (`core >= 4/8`, `home >= 2/2`, `null >= 1/1`) მეტი ხნის განმავლობაში
   ორი ფანჯარა.
2. `soranet_constant_rate_ceiling_hits_total` იზრდება უფრო სწრაფად, ვიდრე ერთი დარტყმა ხუთ წუთში.
3. `soranet_constant_rate_degraded` გადადის `1`-ზე დაგეგმილი საბურღი გარეთ.

ჩაწერეთ წინასწარ დაყენებული ლეიბლი და მეზობლების სია ინციდენტების ანგარიშებში, რათა აუდიტორებმა დაამტკიცონ მუდმივი განაკვეთი
პოლიტიკა ემთხვეოდა საგზაო რუქის მოთხოვნებს.
