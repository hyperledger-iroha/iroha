ეს გეგმა ასახავს დარჩენილ სამუშაოს, რომელიც საჭიროა იმისათვის, რომ ყველა პორტალის წინასწარი გადახედვის არტეფაქტი გამოქვეყნებამდე დადასტურდეს. მიზანია გარანტირებული იყოს, რომ მიმომხილველებმა ჩამოტვირთონ CI-ში ჩაშენებული ზუსტი სნეპშოტი, რომ გამშვები ჯამის მანიფესტი უცვლელია და რომ წინასწარი გადახედვა შესაძლებელი იქნება SoraFS-ის მეშვეობით Norito მეტამონაცემებით.

## მიზნები

- **დეტერმინისტული ნაგებობები:** დარწმუნდით, რომ `npm run build` აწარმოებს გამეორებად გამომავალს და ყოველთვის გამოსცემს `build/checksums.sha256`.
- **დამოწმებული წინასწარი გადახედვები:** მოითხოვეთ ყველა გადახედვის არტეფაქტის გაგზავნა საკონტროლო ჯამის მანიფესტთან ერთად და უარი თქვით გამოქვეყნებაზე, როდესაც ვერიფიკაცია ვერ მოხერხდა.
- **Norito-გამოქვეყნებული მეტამონაცემები:** მუდმივი გადახედვის დესკრიპტორები (მეტამონაცემების ჩაბარება, გამშვები ჯამის დაიჯესტი, SoraFS CID) როგორც Norito JSON, რათა მმართველობითმა ინსტრუმენტებმა შეძლოს რელიზების აუდიტი.
- **ოპერატორის ინსტრუმენტები:** უზრუნველყოს ერთსაფეხურიანი გადამოწმების სკრიპტი, რომელიც მომხმარებლებს შეუძლიათ ადგილობრივად გაუშვან (`./docs/portal/scripts/preview_verify.sh --build-dir build --descriptor <path> --archive <path>`); სკრიპტი ახლა ახვევს საკონტროლო ჯამს + აღწერის ვალიდაციის ნაკადს ბოლოდან ბოლომდე. სტანდარტული წინასწარი გადახედვის ბრძანება (`npm run serve`) ახლა ავტომატურად გამოიძახებს ამ დამხმარეს `docusaurus serve`-მდე, ასე რომ, ლოკალური სნეპშოტები დარჩეს საკონტროლო ჯამით (`npm run serve:verified` ინახება როგორც აშკარა მეტსახელი).

## ფაზა 1 - CI აღსრულება

1. განაახლეთ `.github/workflows/docs-portal-preview.yml` შემდეგზე:
   - გაუშვით `node docs/portal/scripts/write-checksums.mjs` Docusaurus build-ის შემდეგ (უკვე გამოძახებული ადგილობრივად).
   - შეასრულეთ `cd build && sha256sum -c checksums.sha256` და დაასრულეთ სამუშაო შეუსაბამობის გამო.
   - ჩაალაგეთ build დირექტორია, როგორც `artifacts/preview-site.tar.gz`, დააკოპირეთ checksum manifest, დარეკეთ `scripts/generate-preview-descriptor.mjs` და შეასრულეთ `scripts/sorafs-package-preview.sh` JSON კონფიგურაციით (იხ. `docs/examples/sorafs_preview_publish.json`), რათა სამუშაო ნაკადმა გამოუშვას როგორც determinist0000, ასევე, NT00000X და NT80001 მეტამონაცემები. შეკვრა.
   - ატვირთეთ სტატიკური საიტი, მეტამონაცემების არტეფაქტები (`docs-portal-preview`, `docs-portal-preview-metadata`) და SoraFS ნაკრები (`docs-portal-preview-sorafs`), რათა მანიფესტი, CAR რეზიუმე და გეგმა ხელახლა შემოწმების გარეშე მოხდეს.
2. დაამატეთ CI ბეჯის კომენტარი, რომელიც აჯამებს საკონტროლო ჯამის დადასტურების შედეგს pull-ის მოთხოვნებში (✅ განხორციელდა `docs-portal-preview.yml` GitHub Script კომენტარის ნაბიჯის მეშვეობით).
3. დაასაბუთეთ სამუშაო პროცესი `docs/portal/README.md`-ში (CI განყოფილება) და დააკავშირეთ გადამოწმების ნაბიჯები გამოქვეყნების საკონტროლო სიაში.

## გადამოწმების სკრიპტი

`docs/portal/scripts/preview_verify.sh` ამოწმებს გადმოწერილი გადახედვის არტეფაქტებს ხელით `sha256sum` გამოძახების გარეშე. გამოიყენეთ `npm run serve` (ან მკაფიო `npm run serve:verified`) სკრიპტის გასაშვებად და `docusaurus serve`-ის გასაშვებად ერთი ნაბიჯით ადგილობრივი სნეპშოტების გაზიარებისას. გადამოწმების ლოგიკა:

1. აწარმოებს შესაბამის SHA ხელსაწყოს (`sha256sum` ან `shasum -a 256`) `build/checksums.sha256`-ის წინააღმდეგ.
2. სურვილისამებრ ადარებს გადახედვის დესკრიპტორის `checksums_manifest` დაიჯესტს/ფაილის სახელს და, მოწოდების შემთხვევაში, გადახედვისას არქივის დაიჯესტს/ფაილის სახელს.
3. გადის ნულოვანიდან, როდესაც აღმოჩენილია რაიმე შეუსაბამობა, რათა მიმომხილველებმა შეძლონ დაბლოკილი წინასწარი გადახედვები.

გამოყენების მაგალითი (CI არტეფაქტების მოპოვების შემდეგ):

```bash
./docs/portal/scripts/preview_verify.sh \
  --build-dir build \
  --descriptor artifacts/preview-descriptor.json \
  --archive artifacts/preview-site.tar.gz
```

CI და გამოშვების ინჟინრებმა უნდა დაურეკონ სკრიპტს, როცა ჩამოტვირთავენ გადახედვის პაკეტს ან დაურთოს არტეფაქტებს გამოშვების ბილეთს.

## ფაზა 2 — SoraFS გამოცემა

1. გააფართოვეთ გადახედვის სამუშაო პროცესი სამუშაოთი, რომელიც:
   - ატვირთავს აშენებულ საიტს SoraFS დადგმის კარიბჭეში `sorafs_cli car pack` და `manifest submit` გამოყენებით.
   - აღბეჭდავს დაბრუნებულ მანიფესტის დაიჯესტს და SoraFS CID-ს.
   - ანაწილებს `{ commit, branch, checksum_manifest, cid }`-ს Norito JSON-ში (`docs/portal/preview/preview_descriptor.json`).
2. შეინახეთ აღმწერი კონსტრუქციის არტეფაქტთან ერთად და გამოავლინეთ CID pull-მოთხოვნის კომენტარში.
3. დაამატეთ ინტეგრაციის ტესტები, რომლებიც ახორციელებენ `sorafs_cli`-ს მშრალი გაშვების რეჟიმში, რათა უზრუნველყოთ მომავალი ცვლილებები მეტამონაცემების სქემას სტაბილურად.

## ფაზა 3 — მმართველობა და აუდიტი

1. გამოაქვეყნეთ Norito სქემა (`PreviewDescriptorV1`), რომელიც აღწერს აღწერის სტრუქტურას `docs/portal/schemas/`-ში.
2. განაახლეთ DOCS-SORA გამოქვეყნების ჩამონათვალი, რათა მოითხოვოთ:
   - მუშაობს `sorafs_cli manifest verify` ატვირთული CID-ის წინააღმდეგ.
   - საკონტროლო ჯამის მანიფესტის დაიჯესტის და CID ჩაწერა გამოშვების PR აღწერაში.
3. შეაერთეთ მართვის ავტომატიზაცია, რათა გადაამოწმოთ აღმწერი საკონტროლო ჯამის მანიფესტთან გათავისუფლების ხმების დროს.

## ღია კითხვები

- რომელ SoraFS გარემოში უნდა იყოს გადახედული არტეფაქტები (დადგმა და გადახედვის გამოყოფილი ზოლი)?
- გვჭირდება თუ არა ორმაგი ხელმოწერა (Ed25519 + ML-DSA) წინასწარი გადახედვის დესკრიპტორზე გამოქვეყნებამდე?
- უნდა დაამაგროს თუ არა CI სამუშაო პროცესმა ორკესტრატორის კონფიგურაცია (`orchestrator_tuning.json`) `sorafs_cli`-ის გაშვებისას, რათა მანიფესტების გამეორება იყოს?

გადაიღეთ გადაწყვეტილებები `docs/portal/docs/reference/publishing-checklist.md`-ში და განაახლეთ ეს გეგმა, როგორც კი უცნობი იქნება.
