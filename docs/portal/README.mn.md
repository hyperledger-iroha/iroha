# SORA Nexus Developer Portal

This directory hosts the Docusaurus workspace for the interactive developer
portal. The portal aggregates Norito guides, SDK quickstarts, and the OpenAPI
reference generated by `cargo xtask openapi`, wrapping them in the SORA Nexus
branding used across the docs programme.

## Prerequisites

- Node.js 18.18 or newer (Docusaurus v3 baseline).
- Yarn 1.x or npm ≥ 9 for package management.
- Rust toolchain (used by the OpenAPI sync script).

## Bootstrap

```bash
cd docs/portal
npm install    # or yarn install
```

## Available scripts

| Command | Description |
|---------|-------------|

| `npm run tryit-proxy` | Launch the staging proxy that powers the “Try it” console (see below for configuration). |
| `npm run probe:tryit-proxy` | Issue a `/healthz` + sample request probe against the proxy (CI/monitoring helper). |

| `npm run sync-i18n` | Ensure Japanese, Hebrew, Spanish, Portuguese, French, Russian, Arabic, Urdu, Burmese, Georgian, Armenian, Azerbaijani, Kazakh, Bashkir, Amharic, Dzongkha, Uzbek, Mongolian, Traditional Chinese, and Simplified Chinese translation stubs exist under `i18n/`. |
| `npm run sync-norito-snippets` | Regenerate curated Kotodama example docs + downloadable snippets (also invoked automatically by the dev server plugin). |
| `npm run test:tryit-proxy` | Execute the proxy unit tests via Node’s test runner (`node --test`). |

The OpenAPI sync script requires that `cargo xtask openapi` is available from
the repository root; it emits a deterministic JSON file into `static/openapi/`
and now expects the Torii router to expose a live spec (use
`cargo xtask openapi --allow-stub` only for emergency placeholder output).

## Docs versioning & OpenAPI snapshots

- **Cutting a docs version:** run `npm run docs:version -- 2025-q3` (or any agreed tag).
  Commit the generated `versioned_docs/version-<label>`, `versioned_sidebars`,
  and `versions.json`. The navbar version dropdown automatically surfaces the
  new snapshot.
- **Syncing OpenAPI artifacts:** after cutting a version, refresh the canonical
  spec and manifest via `cargo xtask openapi --sign <path-to-ed25519-key>`, then
  capture a matching snapshot with
  `npm run sync-openapi -- --version=2025-q3 --mirror=current --latest`. The
  script writes `static/openapi/versions/2025-q3/torii.json`, mirrors the spec
  into `versions/current/torii.json`, updates `versions.json`, refreshes
  `/openapi/torii.json`, and clones the signed `manifest.json` into each version
  directory so historical specs carry the same provenance metadata. Supply any
  number of `--mirror=<label>` flags to copy the freshly generated spec into
  other historical snapshots.
- **CI expectations:** commits touching docs should include the version bump
  (when applicable) plus refreshed OpenAPI snapshots so the Swagger, RapiDoc,
  and Redoc panels can switch between historical specs without fetch errors.
- **Manifest enforcement:** the `sync-openapi` script now fails when the signed
  `manifest.json` is missing, malformed, or does not match the freshly generated
  spec so unsigned snapshots cannot be published by default. Re-run
  `cargo xtask openapi --sign <key>` to refresh the canonical manifest and
  re-run the sync so versioned snapshots pick up the signed metadata.
  Pass `--allow-unsigned` only for local previews (CI still runs
  `ci/check_openapi_spec.sh`, which regenerates the spec and verifies the
  manifest before allowing commits to merge).

### Try it proxy configuration

The “Try it” sandbox routes requests through `scripts/tryit-proxy.mjs`. Configure
the proxy with environment variables before launching it:

```bash
export TRYIT_PROXY_TARGET="https://torii.staging.sora"
export TRYIT_PROXY_ALLOWED_ORIGINS="http://localhost:3000"
export TRYIT_PROXY_BEARER="sora-dev-token"          # optional
export TRYIT_PROXY_PUBLIC_URL="http://localhost:8787"
npm run tryit-proxy
```

- `TRYIT_PROXY_LISTEN` (default `127.0.0.1:8787`) controls the bind address.
- `TRYIT_PROXY_RATE_LIMIT` / `TRYIT_PROXY_RATE_WINDOW_MS` configure the in-memory
  rate limiter (defaults to 60 requests per 60 seconds).
- Set `TRYIT_PROXY_ALLOW_CLIENT_AUTH=1` to forward the caller’s `Authorization`
  header instead of relying on the default bearer token.
- The proxy refuses to start if `static/openapi/torii.json` drifts from the
  signed manifest in `static/openapi/manifest.json`. Run
  `npm run sync-openapi -- --latest` to refresh the spec; export
  `TRYIT_PROXY_ALLOW_STALE_SPEC=1` only for emergency overrides (a warning is
  logged and the proxy starts anyway).
- Run `npm run probe:tryit-proxy` to smoke-test the staging proxy and wire the
  command into monitoring jobs; `npm run manage:tryit-proxy -- update` simplifies
  rotating Torii endpoints while keeping a `.env` backup for rollback.
- `TRYIT_PROXY_PROBE_METRICS_FILE` and `TRYIT_PROXY_PROBE_LABELS` let the probe
  emit `probe_success`/`probe_duration_seconds` in Prometheus textfile format so
  you can hook the health check into node_exporter or any batch collector.
- Prometheus scrapes expose `tryit_proxy_request_duration_ms_bucket`/`_count`/`_sum`
  histogram series; the docs portal Grafana dashboard consumes them for p95/p99 latency
  SLO tracking (`dashboards/grafana/docs_portal.json`).

### OAuth device-code login

When the portal is exposed outside trusted networks, configure OAuth device
authorization so reviewers never touch long-lived Torii tokens. Export the
variables below before running `npm run start` or `npm run build`:

| Variable | Notes |
|----------|-------|
| `DOCS_OAUTH_DEVICE_CODE_URL` / `DOCS_OAUTH_TOKEN_URL` | OAuth endpoints that implement the Device Authorization Grant. |
| `DOCS_OAUTH_CLIENT_ID` | Client ID registered for the docs preview. |
| `DOCS_OAUTH_SCOPE` / `DOCS_OAUTH_AUDIENCE` | Optional scope/audience strings to restrict the issued token. |
| `DOCS_OAUTH_POLL_INTERVAL_MS` | Minimum polling interval enforced by the portal widget (defaults to 5000 ms; lower values are rejected). |
| `DOCS_OAUTH_DEVICE_CODE_TTL_SECONDS` / `DOCS_OAUTH_TOKEN_TTL_SECONDS` | Fallback expiry windows used when the server omits `expires_in`. |
| `DOCS_OAUTH_ALLOW_INSECURE` | Set to `1` only for local development to bypass the build-time guard. Production builds fail if required variables are missing or TTLs fall outside the enforced ranges. |

Example:

```bash
export DOCS_OAUTH_DEVICE_CODE_URL="https://auth.dev.sora.example/oauth/device/code"
export DOCS_OAUTH_TOKEN_URL="https://auth.dev.sora.example/oauth/token"
export DOCS_OAUTH_CLIENT_ID="docs-preview"
export DOCS_OAUTH_SCOPE="torii openid offline_access"
```

After the build picks up these values, the Try it console renders a sign-in
panel that shows the device code, polls the token endpoint, populates the Bearer
field automatically, and clears the token when it expires. The portal refuses to
start when the OAuth configuration is incomplete or the token/device TTLs fall
outside the security budget; export `DOCS_OAUTH_ALLOW_INSECURE=1` only on
disposable local previews where anonymous access is acceptable. The proxy still
honours manual `X-TryIt-Auth` overrides, so you can remove the OAuth variables
whenever you prefer to paste ad-hoc tokens.

See [`docs/devportal/security-hardening.md`](docs/devportal/security-hardening.md) for
the full threat model and pen-test gates.

### Security headers

`docusaurus.config.js` now emits deterministic security headers—content security
policy, Trusted Types, Permissions-Policy, and Referrer-Policy—so static hosts
inherit the same guard rails as the dev server. The defaults only allow scripts
served from the portal origin and restrict cross-origin `connect-src` traffic to
the configured analytics endpoint and Try it proxy. Production builds reject
non-HTTPS analytics/try-it endpoints; for local previews against `http://`
targets, set `DOCS_SECURITY_ALLOW_INSECURE=1` explicitly so the guardrail
acknowledges the downgrade.

### Localization workflow

The portal ships with English as the source language and Japanese, Hebrew,
Spanish, Portuguese, French, Russian, Arabic, Urdu, Burmese, Georgian,
Armenian, Azerbaijani, Kazakh, Bashkir, Amharic, Dzongkha, Uzbek, Mongolian,
Traditional Chinese, and Simplified Chinese locales. Whenever new docs land,
run:

```bash
npm run sync-i18n
```

The script mirrors the repository-wide `scripts/sync_docs_i18n.py` behaviour by
generating stub translations under `i18n/<lang>/docusaurus-plugin-content-docs/current/...`.
Editors can then replace the placeholder text and update the front matter
metadata (`status`, `translation_last_reviewed`, etc.).

By default, production builds only include the locales listed in
`docs/i18n/published_locales.json` (currently English-only) so stub placeholders
are not shipped. Set `DOCS_I18N_INCLUDE_STUBS=1` when you need to preview
in-progress locales locally.

## CI & deployment

- `ci/check_docs_portal.sh` first runs `node scripts/check-sdk-recipes.mjs` to verify the SDK ledger recipes match their canonical source files, then executes `npm ci|install`, `npm run build`, and asserts key sections (homepage, SoraFS, Norito, publishing checklist) exist in the generated HTML.
- `ci/check_openapi_spec.sh` regenerates the Torii spec via `cargo xtask openapi`,
  compares it to `static/openapi/torii.json` and `versions/current/torii.json`,
  and verifies the checksum manifest so PRs fail when the tracked spec or hash
  is stale.
- `.github/workflows/check-docs.yml` runs the portal build on every PR to catch content regressions.
- `.github/workflows/docs-portal-preview.yml` builds the site for pull requests, writes `build/checksums.sha256`, validates it via `sha256sum -c`, packages the output as `artifacts/preview-site.tar.gz`, generates a descriptor via `scripts/generate-preview-descriptor.mjs`, and uploads both the static site and metadata artifacts so reviewers can audit the exact snapshot without re-running the build. The workflow also posts a pull-request comment summarising the manifest/archive digests (and SoraFS bundle, when available) so reviewers see the verification result without opening the Actions logs.
- `.github/workflows/docs-portal-deploy.yml` publishes the static build to GitHub Pages on pushes to `main`/`master`, providing a preview at the `github-pages` environment URL. The workflow runs a landing-page smoke check before deployment.
- `scripts/sorafs-package-preview.sh` converts the preview site into a deterministic SoraFS bundle (CAR, plan, manifest) and, when credentials are supplied via a JSON config (see `docs/examples/sorafs_preview_publish.json`), can submit the manifest to a staging Torii endpoint.
- `scripts/preview_verify.sh` performs the checksum + descriptor validation flow that reviewers previously executed manually. Point it at an extracted `build/` directory (and optional descriptor/archive paths) to confirm the preview artefacts are untampered.
- `scripts/sorafs-pin-release.sh` plus `.github/workflows/docs-portal-sorafs-pin.yml` automate the production SoraFS pipeline: build/test, CAR + manifest generation, Sigstore signing, verification, optional alias binding, and artifact uploads for governance review. Trigger the workflow via `workflow_dispatch` when promoting a release.
- `cargo xtask soradns-verify-gar --gar <path> --name <fqdn> [...]` validates Gateway Authorization Records before they are signed or shipped to ops. The helper ensures canonical/pretty hosts, manifest metadata, and telemetry labels match the deterministic policy and can emit a JSON summary for DG-3 evidence via `--json-out`.
- When a submission occurs, the pin helper also invokes `scripts/generate-dns-cutover-plan.mjs` to produce `artifacts/sorafs/portal.dns-cutover.json`. Set `DNS_CHANGE_TICKET`, `DNS_CUTOVER_WINDOW`, `DNS_HOSTNAME`, `DNS_ZONE`, and `DNS_OPS_CONTACT` (or pass the corresponding `--dns-*` flags) so the descriptor carries the metadata Ops needs for production DNS cutovers. When cache invalidation and rollback hooks are required, add `DNS_CACHE_PURGE_ENDPOINT`, `DNS_CACHE_PURGE_AUTH_ENV`, and `DNS_PREVIOUS_PLAN` (or the `--cache-purge-*` / `--previous-dns-plan` flags) so the descriptor records the purge API call and the prior descriptor for reversions. The descriptor now also documents the stapled `Sora-Route-Binding` (host, CID, header/binding paths, verification commands) so GAR promotion and fallback plan reviews reference the exact headers served at the edge.
- The same workflow can emit the SNS zonefile skeleton/resolver snippet via `scripts/sns_zonefile_skeleton.py`. Provide IPv4/IPv6/CNAME/SPKI/TXT metadata (either as env vars such as `DNS_ZONEFILE_IPV4` or CLI flags like `--dns-zonefile-ipv4`), pass the GAR digest with `DNS_GAR_DIGEST`, and the helper will write `artifacts/sns/zonefiles/<zone>/<hostname>.json` (plus the resolver snippet) automatically so SN-7 evidence lands alongside the cutover descriptor.
- The DNS descriptor now embeds a `gateway_binding` section that references the artefacts above (paths, content CID, proof status, and the literal header template) so DG-3 change approvals include the exact `Sora-Name/Sora-Proof/CSP/HSTS` bundle expected on the gateway.

Every `npm run build` invocation triggers the `postbuild` hook which generates
`build/checksums.sha256`. After building (or downloading the CI artefacts), run
`./docs/portal/scripts/preview_verify.sh --build-dir docs/portal/build` to
validate the manifest. Pass `--descriptor <path>`/`--archive <path>` when
verifying the metadata bundle from CI so the script can cross-check the recorded
digests and filenames.
- To share a local preview safely, run `npm run serve`; the command now wraps
  `scripts/serve-verified-preview.mjs`, which executes `preview_verify.sh`
  before launching `docusaurus serve` and aborts if the manifest check fails or
  is missing, keeping previews checksum-gated even outside CI. The
  `npm run serve:verified` alias remains available for explicit calls.
