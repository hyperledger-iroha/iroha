# Genesis тохиргоо

`genesis.json` файл нь Iroha сүлжээг эхлүүлэх үед хийгдэх эхний гүйлгээг тодорхойлдог. Файл нь эдгээр талбар бүхий JSON объект юм:

- `chain` – өвөрмөц хэлхээ танигч.
- `executor` (заавал биш) – гүйцэтгэгч байт код руу хүрэх зам (`.to`). Хэрэв байгаа бол,
  genesis нь Upgrade зааврыг эхний гүйлгээ болгон агуулдаг. орхигдуулсан бол,
  шинэчлэл хийгдээгүй бөгөөд суулгасан гүйцэтгэгчийг ашигладаг.
- `ivm_dir` – IVM байт кодын сангуудыг агуулсан лавлах. Хэрэв орхигдуулсан бол өгөгдмөл нь `"."`.
- `consensus_mode` – манифестэд сурталчилсан зөвшилцлийн горим. Шаардлагатай; нийтийн Sora Nexus мэдээллийн санд `"Npos"`, эсвэл бусад Iroha3 мэдээллийн орон зайд `"Permissioned"`/`"Npos"` ашиглана уу. Iroha2 нь анхдагчаар `"Permissioned"` байна.
- `transactions` – дараалсан гүйцэтгэсэн генезийн гүйлгээний жагсаалт. Бүртгэл бүр дараахь зүйлийг агуулж болно.
  - `parameters` – сүлжээний анхны параметрүүд.
  - `instructions` – бүтэцлэгдсэн Norito заавар (жишээ нь, `{ "Register": { "Domain": { "id": "wonderland" }}}`). Түүхий байт массивыг хүлээн авахгүй бөгөөд `SetParameter` зааврыг эндээс татгалзсан болно - параметрүүдийг `parameters` блокоор дамжуулан суулгаж, зааварчилгааг хэвийн болгох/гарын үсэг зурахыг зөвшөөрнө үү.
  - `ivm_triggers` – IVM байт кодын гүйцэтгэгдэх боломжтой триггерүүд.
  - `topology` – анхны үе тэнгийн топологи. Оруулсан зүйл бүр нь ижил id болон PoP-г хамтад нь хадгалдаг: `{ "peer": "<public_key>", "pop_hex": "<hex>" }`. `pop_hex` бичих явцад орхигдсон байж болох ч гарын үсэг зурахаас өмнө заавал байх ёстой.
- `crypto` – `iroha_config.crypto` (`default_hash`, `allowed_signing`, `allowed_curve_ids`, `sm2_distid_default`0, `allowed_signing`0, `iroha_config.crypto`) толин тусгал хийсэн криптографийн агшин зуурын зураг. `allowed_curve_ids` нь `crypto.curves.allowed_curve_ids`-ийг тусгадаг тул манифестууд нь кластер ямар хянагч муруйг хүлээн авахыг сурталчлах боломжтой. Багаж хэрэгсэл нь SM хослолуудыг хэрэгжүүлдэг: `sm2` жагсаалтын манифестууд нь мөн хэшийг `sm3-256` руу шилжүүлэх ёстой, харин `sm` функцгүйгээр эмхэтгэсэн бүтээлүүд нь `sm2`-г бүхэлд нь үгүйсгэдэг. Хэвийн байдал нь гарын үсэг зурсан генезид `crypto_manifest_meta` өөрчлөн тохируулсан параметрийг оруулдаг; Хэрэв тарьсан ачаалал нь сурталчилсан агшин зуурын зурагтай санал нийлэхгүй байвал зангилаа эхлүүлэхээс татгалздаг.

Жишээ (`kagami genesis generate default --consensus-mode npos` гаралт, зааврыг тайрсан):

```json
{
  "chain": "00000000-0000-0000-0000-000000000000",
  "ivm_dir": "defaults",
  "transactions": [
    {
      "parameters": { "sumeragi": { "block_time_ms": 2000 } },
      "instructions": [
        { "Register": { "Domain": { "id": "wonderland" } } }
      ],
      "ivm_triggers": [],
      "topology": [
        {
          "peer": "ed25519:...",
          "pop_hex": "ab12cd..."
        }
      ]
    }
  ],
  "consensus_mode": "Npos",
  "crypto": {
    "default_hash": "blake2b-256",
    "allowed_signing": ["ed25519", "secp256k1"],
    "allowed_curve_ids": [1],
    "sm2_distid_default": "1234567812345678",
    "sm_openssl_preview": false
  }
}
```

### `crypto` блокийг SM2/SM3-д суулгаарай

xtask туслагчийг ашиглан үндсэн бараа материал болон буулгахад бэлэн тохиргооны хэсгийг нэг алхамаар гаргана уу:

```bash
cargo xtask sm-operator-snippet \
  --distid CN12345678901234 \
  --json-out sm2-key.json \
  --snippet-out client-sm2.toml
```

`client-sm2.toml` одоо агуулж байна:

```toml
# Account key material
public_key = "sm2:8626530010..."
private_key = "A333F581EC034C1689B750A827E150240565B483DEB28294DDB2089AD925A569"
# public_key_pem = """\
-----BEGIN PUBLIC KEY-----
...
-----END PUBLIC KEY-----
"""
# private_key_pem = """\
-----BEGIN PRIVATE KEY-----
...
-----END PRIVATE KEY-----
"""

[crypto]
default_hash = "sm3-256"
allowed_signing = ["ed25519", "secp256k1", "sm2"]  # remove "sm2" to stay in verify-only mode
allowed_curve_ids = [1]               # add new curve ids (e.g., 15 for SM2) when controllers are allowed
sm2_distid_default = "CN12345678901234"
# enable_sm_openssl_preview = true  # optional: only when deploying the OpenSSL/Tongsuo path
```

`public_key`/`private_key` утгуудыг бүртгэл/үйлчлүүлэгчийн тохиргоонд хуулж, `genesis.json`-ийн `crypto` блокийг хэсэгчилсэн байдлаар шинэчилнэ үү. `"sm2"`-аас `allowed_signing` хүртэл, мөн зөв `allowed_curve_ids`-г оруулна). Kagami нь хэш/муруй тохиргоо болон гарын үсэг зурах жагсаалт зөрж байгаа манифестаас татгалзах болно.

> **Зөвлөгөө:** Гаралтыг шалгахыг хүссэн үедээ `--snippet-out -`-ээр хэсэгчилсэн хэсгийг stdout руу дамжуулаарай. `--json-out -`-г ашиглан stdout дээр гол бараа материалаа гарга.

Хэрэв та доод түвшний CLI командуудыг гараар жолоодохыг илүүд үзэж байгаа бол түүнтэй адилтгах урсгал нь:

```bash
# 1. Produce deterministic key material (writes JSON to disk)
cargo run -p iroha_cli --features sm -- \
  crypto sm2 keygen \
  --distid CN12345678901234 \
  --output sm2-key.json

# 2. Re-hydrate the snippet that can be pasted into client/config files
cargo run -p iroha_cli --features sm -- \
  crypto sm2 export \
  --private-key-hex "$(jq -r .private_key_hex sm2-key.json)" \
  --distid CN12345678901234 \
  --snippet-output client-sm2.toml \
  --emit-json --quiet
```

> **Зөвлөгөө:** `jq`-г гараар хуулах/буулгах алхамыг хадгалахын тулд дээр ашигласан. Хэрэв боломжгүй бол `sm2-key.json`-г нээж, `private_key_hex` талбарыг хуулж, `crypto sm2 export` руу шууд дамжуулаарай.

> **Шилжүүлэх гарын авлага:** Одоо байгаа сүлжээг SM2/SM3/SM4 болгон хөрвүүлэхдээ дараахыг дагана уу.
> [`docs/source/crypto/sm_config_migration.md`](source/crypto/sm_config_migration.md)
> давхраатай `iroha_config` хүчингүй болгох, манифест нөхөн сэргээх, буцаахад зориулагдсан
> төлөвлөлт.

## Үүсгэх ба баталгаажуулах

1. Загвар үүсгэх:
   ```bash
   cargo run -p iroha_kagami -- genesis generate \
     [--executor <path/to/executor.to>] \
     --consensus-mode npos \
     --ivm-dir <ivm/dir> \
     --genesis-public-key <PUBLIC_KEY> > genesis.json
   ```
`--consensus-mode` `parameters` блок руу Kagami үрийг зөвшилцөх параметрүүдийг хянадаг. Нийтийн Sora Nexus өгөгдлийн орон зайд `npos` шаардлагатай бөгөөд үе шаттай таслалтыг дэмждэггүй; бусад Iroha3 өгөгдлийн орон зай нь зөвшөөрөлтэй эсвэл NPoS ашиглаж болно. Iroha2 өгөгдмөл нь `permissioned` бөгөөд `--next-consensus-mode`/`--mode-activation-height`-ээр дамжуулан `npos` үе шаттай байж болно. `npos`-г сонгох үед Kagami нь NPoS коллекторын идэвхижүүлэлт, сонгуулийн бодлого, дахин тохируулгын цонхыг удирддаг `sumeragi_npos_parameters` ачааллыг суулгадаг; хэвийн болгох/гарын үсэг зурах нь тэдгээрийг гарын үсэг зурсан блок дахь `SetParameter` заавар болгон хувиргадаг.
2. Сонголтоор `genesis.json`-г засаад баталгаажуулаад гарын үсэг зурна уу:
   ```bash
   cargo run -p iroha_kagami -- genesis sign genesis.json \
     --public-key <PUBLIC_KEY> \
     --private-key <PRIVATE_KEY> \
     --out-file genesis.signed.nrt
   ```

   SM2/SM3/SM4-д бэлэн манифестуудыг гаргахын тулд `--default-hash sm3-256` дамжуулж, `--allowed-signing sm2` (нэмэлт алгоритмын хувьд `--allowed-signing`-г давтана уу). Хэрэв та анхдагч ялгах танигчийг хүчингүй болгох шаардлагатай бол `--sm2-distid-default <ID>`-г ашиглана уу.

   Та `irohad`-г зөвхөн `--genesis-manifest-json` (гарын үсэг зурсан генезийн блок байхгүй)-ээр эхлүүлэх үед зангилаа одоо манифестээс автоматаар ажиллах цагийн крипто тохиргоогоо суулгадаг; Хэрэв та мөн генезисын блок нийлүүлбэл манифест болон тохиргоо яг таарч байх ёстой.

- Баталгаажуулалтын тэмдэглэл:
  - Kagami `consensus_handshake_meta`, `confidential_registry_root`, `crypto_manifest_meta`-ийг `SetParameter` зааврын дагуу нормчлогдсон/гарын үсэгтэй блок дотор тарина. `irohad` нь тэдгээр ачааллаас зөвшилцсөн хурууны хээг дахин тооцоолж, гар барих мета өгөгдөл эсвэл крипто агшин зуурын зураг нь кодлогдсон параметрүүдтэй санал нийлэхгүй байвал эхлүүлэх боломжгүй болно. Манифест дээрх `instructions`-аас эдгээрийг хадгалаарай; тэдгээр нь автоматаар үүсгэгддэг.
- Нормчилсан блокыг шалгана уу:
  - `kagami genesis normalize genesis.json --format text`-г ажиллуулж эцсийн захиалсан гүйлгээг (үүнд оруулсан мета өгөгдлийг оруулаад) товчлуурын хослолгүйгээр харах боломжтой.
  - `--format json`-г ашиглан ялгах эсвэл шүүмжлэхэд тохиромжтой бүтэцлэгдсэн харагдацыг устгана уу.

`kagami genesis sign` нь JSON хүчинтэй эсэхийг шалгаж, зангилааны тохиргоонд `genesis.file`-ээр дамжуулан ашиглахад бэлэн Norito-кодлогдсон блок үүсгэдэг. Үүссэн `genesis.signed.nrt` нь аль хэдийн каноник утсан хэлбэртэй байна: хувилбарын байт, дараа нь ачааллын байршлыг тодорхойлсон Norito толгой. Энэ хүрээтэй гаралтыг үргэлж тараана. Гарын үсэг зурсан ачааллын хувьд `.nrt` дагаварыг илүүд үзэх; Хэрэв та генезисийн үед гүйцэтгэгчийг шинэчлэх шаардлагагүй бол `executor` талбарыг орхиж, `.to` файл оруулахыг алгасаж болно.

NPoS манифестэд (`--consensus-mode npos` эсвэл зөвхөн Iroha2-н үе шаттай зүсэлт) гарын үсэг зурах үед `kagami genesis sign` нь `sumeragi_npos_parameters` ачааллыг шаарддаг; `kagami genesis generate --consensus-mode npos` ашиглан үүсгэх эсвэл параметрийг гараар нэмнэ үү.
Анхдагчаар `kagami genesis sign` нь манифестын `consensus_mode`-г ашигладаг; үүнийг хүчингүй болгохын тулд `--consensus-mode`-г дамжуулна уу.

## Эхлэл юу хийж чадах вэ

Genesis нь дараах үйлдлүүдийг дэмждэг. Kagami нь тэдгээрийг сайн тодорхойлогдсон дарааллаар гүйлгээ болгон угсардаг тул үе тэнгийнхэн нь ижил дарааллыг тодорхойлогч байдлаар гүйцэтгэдэг.

- Параметрүүд: Sumeragi (блок/коммит хугацаа, зөрөх), Блок (хамгийн их txs), Гүйлгээ (хамгийн их заавар, байт кодын хэмжээ), Гүйцэтгэгч болон Ухаалаг гэрээний хязгаарлалт (түлш, санах ой, гүн) болон захиалгат параметрүүдийн анхны утгыг тохируулна. Kagami үрийн `Sumeragi::NextMode` болон `sumeragi_npos_parameters`-ийн даац (NPoS сонголт, дахин тохируулга) нь `parameters` блокоор дамждаг тул эхлүүлэх нь сүлжээн дэх төлөвөөс зөвшилцөх товчлууруудыг ашиглах боломжтой; гарын үсэг зурсан блок нь үүсгэсэн `SetParameter` зааврыг агуулдаг.
- Төрөлхийн заавар: Бүртгүүлэх/Бүртгэлээс хасах домэйн, данс, хөрөнгийн тодорхойлолт; Хөрөнгийг гаа/шатаах/шилжүүлэх; Домэйн болон хөрөнгийн тодорхойлолтын эзэмшлийг шилжүүлэх; Мета өгөгдлийг өөрчлөх; Зөвшөөрөл, үүрэг өгөх.
- IVM Триггер: IVM байт кодыг гүйцэтгэдэг триггерүүдийг бүртгэх (`ivm_triggers`-г үзнэ үү). Триггерүүдийн гүйцэтгэх файлууд `ivm_dir`-тэй харьцуулахад шийдэгддэг.
- Топологи: Аливаа гүйлгээний доторх `topology` массиваар (ихэвчлэн эхний эсвэл сүүлчийнх) үе тэнгийн багцыг өгнө. Оруулга бүр `{ "peer": "<public_key>", "pop_hex": "<hex>" }`; `pop_hex` бичих явцад орхигдсон байж болох ч гарын үсэг зурахаас өмнө заавал байх ёстой.
- Гүйцэтгэгчийг шинэчлэх (заавал биш): Хэрэв `executor` байгаа бол genesis нь нэг Upgrade зааврыг эхний гүйлгээ болгон оруулна; өөрөөр хэлбэл, генези нь параметрүүд/зааваруудаас шууд эхэлдэг.

### Гүйлгээний захиалга

Үзэл баримтлалын хувьд генезийн гүйлгээг дараах дарааллаар боловсруулдаг.

1) (Заавал биш) Гүйцэтгэгчийг шинэчлэх
2) `transactions` дахь гүйлгээ бүрийн хувьд:
   - Параметрийн шинэчлэлтүүд
   - Төрөлхийн заавар
   - IVM бүртгэлийг идэвхжүүлдэг
   - Топологийн бичилтүүд

Kagami болон зангилааны код нь энэ дарааллыг баталгаажуулдаг тул жишээ нь ижил гүйлгээний дараагийн зааварчилгааны өмнө параметрүүд хэрэгжинэ.

## Санал болгож буй ажлын урсгал

- Kagami бүхий загвараас эхэл:
  - Зөвхөн суурилагдсан ISI: `kagami genesis generate --ivm-dir <dir> --genesis-public-key <PK> --consensus-mode npos > genesis.json` (Sora Nexus нийтийн мэдээллийн зай; Iroha2 эсвэл хувийн Iroha3-д `--consensus-mode permissioned` ашиглана уу).
  - Захиалгат гүйцэтгэгчийг шинэчлэх (заавал биш): `--executor <path/to/executor.to>` нэмнэ үү
  - Зөвхөн Iroha2-д: NPoS-д ирээдүйн зүсэлт хийх бол `--next-consensus-mode npos --mode-activation-height <HEIGHT>` (одоогийн горимд `--consensus-mode permissioned`-г хадгална уу).
- `<PK>` нь `--features gost`-ээр бүтээгдсэн TC26 ГОСТ хувилбаруудыг багтаасан `iroha_crypto::Algorithm`-ээр хүлээн зөвшөөрөгдсөн аливаа multihash юм (жишээ нь `gost3410-2012-256-paramset-a:...`).
- Засварлах явцад баталгаажуулах: `kagami genesis validate genesis.json`
- Байршуулахаар гарын үсэг зурна уу: `kagami genesis sign genesis.json --public-key <PK> --private-key <SK> --out-file genesis.signed.nrt`
- Үе тэнгийнхний тохиргоог хийх: `genesis.file`-г гарын үсэг зурсан Norito файлд (жишээ нь, `genesis.signed.nrt`) болон `genesis.public_key`-г гарын үсэг зурахад ашигладаг `<PK>` болгон тохируулна уу.Тэмдэглэл:
- Kagami-ийн "өгөгдмөл" загвар нь жишээ домэйн болон бүртгэлийг бүртгэж, цөөн хэдэн хөрөнгө гаргаж, зөвхөн суулгасан ISI ашиглан хамгийн бага зөвшөөрөл олгодог - `.to` шаардлагагүй.
- Хэрэв та гүйцэтгэгчийн шинэчлэлийг оруулсан бол энэ нь анхны гүйлгээ байх ёстой. Kagami үүнийг үүсгэх/гарын үсэг зурах үед хэрэгжүүлдэг.
- Гарын үсэг зурахын өмнө хүчингүй `Name` утгууд (жишээ нь, хоосон зай) болон алдаатай зааварчилгааг барихын тулд `kagami genesis validate` ашиглана уу.

## Docker/Swarm-тай гүйж байна

Өгөгдсөн Docker Compose болон Swarm хэрэгсэл нь хоёр тохиолдлыг зохицуулдаг:

- Гүйцэтгэгчгүй: бичих команд нь дутуу/хоосон `executor` талбарыг арилгаж, файлд гарын үсэг зурна.
- Гүйцэтгэгчтэй: энэ нь контейнер доторх үнэмлэхүй зам руу харьцангуй гүйцэтгэгч замыг шийдэж, файлд гарын үсэг зурна.

Энэ нь урьдчилан бүтээгдсэн IVM загваргүй машинууд дээр хөгжүүлэлтийг хялбаршуулж, шаардлагатай үед гүйцэтгэгчийг шинэчлэх боломжийг олгодог.
