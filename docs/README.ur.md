# Iroha دستاویزات

جاپانی تعارف کے لیے ‎[`README.ja.md`](./README.ja.md)‎ دیکھیں۔

یہ ورک اسپیس ایک ہی کوڈ بیس سے دو ریلیز لائنیں فراہم کرتا ہے:
**Iroha 2** (خود میزبان نیٹ ورکس) اور **Iroha 3 / SORA Nexus** (واحد عالمی
Nexus لیجر)۔ دونوں میں ایک ہی Iroha Virtual Machine (IVM) اور Kotodama ٹول
چین استعمال ہوتی ہے، اس لیے کانٹریکٹس اور بائٹ کوڈ مختلف ڈپلائمنٹ ماحولوں
کے درمیان قابلِ انتقال رہتے ہیں۔ جب تک واضح طور پر اور کچھ نہ لکھا ہو، یہ
دستاویزات دونوں ریلیز لائنوں پر لاگو ہوتی ہیں۔

‎[Iroha کی مرکزی دستاویزات](https://docs.iroha.tech)‎ میں آپ کو درج ذیل مواد ملے گا:

- ‎[شروعاتی گائیڈ](https://docs.iroha.tech/get-started/)‎
- ‎[SDK ٹیوٹوریلز](https://docs.iroha.tech/guide/tutorials/)‎ برائے Rust، Python، JavaScript اور Java/Kotlin
- ‎[API ریفرنس](https://docs.iroha.tech/reference/torii-endpoints.html)‎

ہر ریلیز لائن کے لیے وائٹ پیپر اور وضاحتی دستاویزات:

- ‎[Iroha 2 وائٹ پیپر](./source/iroha_2_whitepaper.md) — خود میزبان نیٹ ورکس کی تفصیلی وضاحت۔
- ‎[Iroha 3 (SORA Nexus) وائٹ پیپر](./source/iroha_3_whitepaper.md) — Nexus کے ملٹی لین اور ڈیٹا اسپیس ڈیزائن کی وضاحت۔
- ‎[ڈیٹا ماڈل اور ISI وضاحت (نفاذ سے اخذ کردہ)](./source/data_model_and_isi_spec.md) — رویّے کی ریفرنس جو کوڈ سے نکالی گئی ہے۔
- ‎[ZK لفافے (Norito)](./source/zk_envelopes.md) — IPA/STARK پر مبنی نیٹو Norito لفافے اور ویریفائر کی توقعات۔

## لوکلائزیشن

جاپانی ‎(`*.ja.*`)‎، عبرانی ‎(`*.he.*`)‎، ہسپانوی ‎(`*.es.*`)‎، پرتگالی
‎(`*.pt.*`)‎، فرانسیسی ‎(`*.fr.*`)‎، روسی ‎(`*.ru.*`)‎، عربی ‎(`*.ar.*`)‎ اور
اردو ‎(`*.ur.*`)‎ میں دستاویزی stubs ہر انگریزی سورس فائل کے ساتھ موجود ہیں۔

کیسے شامل کی جائیں، ‎[`docs/i18n/README.md`](./i18n/README.md)‎ دیکھیں۔

## Norito (سیریلائزیشن کوڈیک)

Norito اس ورک اسپیس کا سیریلائزیشن کوڈیک ہے۔ ہم ‎`parity-scale-codec`‎
(SCALE) استعمال نہیں کرتے۔ جب دستاویزات یا بنچ مارکس میں SCALE کے ساتھ
موازنہ کیا جاتا ہے تو وہ صرف پس منظر فراہم کرنے کے لیے ہوتا ہے؛ پروڈکشن کے
تمام راستے Norito استعمال کرتے ہیں۔ ‎`norito::codec::{Encode, Decode}`‎ APIs
ایسے Norito پیلوڈ فراہم کرتے ہیں جو ہیڈر کے بغیر ("bare") ہوتے ہیں، تاکہ
hashing اور نیٹ ورک افادیت بہتر ہو — یعنی یہ Norito ہی رہتا ہے، SCALE نہیں۔

موجودہ حالت:

- ڈیٹرمنسٹک انکوڈنگ/ڈیکوڈنگ ایک فکسڈ ہیڈر کے ساتھ (magic ویلیو، ورژن، 16‑بائٹ
  اسکیمہ، کمپریشن، لمبائی، CRC64 اور فلیگز)۔
- CRC64-XZ چیک سم جس کے لیے رن ٹائم پر ایکسیلیریشن کا راستہ منتخب کیا جاتا ہے:
  - x86_64 پر PCLMULQDQ (کیری کے بغیر ضرب) کے ساتھ Barrett ریڈکشن، 32‑بائٹ بلاکس پر۔
  - aarch64 پر PMULL، اسی folding اسکیم کے ساتھ۔
  - slicing‑by‑8 اور مکمل بٹ‑بیسڈ راستے تاکہ پورٹیبلٹی برقرار رہے۔
- انکوڈڈ لمبائی کے ہنٹس جو derives اور بنیادی ٹائپس فراہم کرتے ہیں، میموری الاٹمنٹ کو کم کرنے میں مدد دیتے ہیں۔
- بڑا اسٹریم بفر (64 KiB) اور ڈیکوڈنگ کے دوران CRC کی تدریجی اپ ڈیٹ۔
- اختیاری zstd کمپریشن؛ GPU ایکسیلیریشن فیچر فلیگ کے ذریعہ کنٹرول ہوتی ہے اور ڈیٹرمنسٹک رہتی ہے۔
- اڈاپٹو راستہ انتخاب: ‎`norito::to_bytes_auto(&T)`‎ پیلوڈ سائز اور کیش کی گئی
  ہارڈ ویئر صلاحیتوں کی بنیاد پر بغیر کمپریشن، CPU‑بیسڈ zstd یا GPU‑آف لوڈڈ
  zstd میں سے انتخاب کرتا ہے۔ یہ انتخاب صرف کارکردگی اور ہیڈر کے
  ‎`compression`‎ بائٹ کو متاثر کرتا ہے؛ پیلوڈ کے معنی تبدیل نہیں ہوتے۔

مزید تفصیل، بنچ مارکس، اور مثالوں کے لیے ‎`crates/norito/README.md`‎ دیکھیں۔

نوٹ: کچھ سب سسٹم ڈاکومنٹس (مثلاً IVM ایکسیلیریشن اور ZK سرکٹس) ابھی ترقی کے
مرحلے میں ہیں۔ جہاں فنکشنلٹی مکمل نہیں ہوتی، وہاں فائل کھلے طور پر باقی کام
اور سمتِ سفر کی وضاحت کرتی ہے۔

اسٹیٹس اینڈ پوائنٹ انکوڈنگ سے متعلق نوٹس

- Torii کا `/status` اینڈ پوائنٹ ڈیفالٹ کے طور پر Norito پیلوڈ استعمال کرتا
  ہے جو ہیڈر کے بغیر ہوتا ہے، تاکہ پیغام مختصر رہے۔ کلائنٹس کو چاہیے کہ سب سے
  پہلے Norito کے ذریعے ڈیکوڈ کرنے کی کوشش کریں۔
- سرور ضرورت پڑنے پر JSON بھی واپس کر سکتا ہے؛ اس صورت میں اگر
  ‎`content-type`‎ ‎`application/json`‎ ہو تو کلائنٹس JSON پر فallback کر سکتے ہیں۔
- وائر فارمیٹ Norito ہے، SCALE نہیں۔ ہیڈر کے بغیر ("bare") ویریئنٹ کے لیے
  ‎`norito::codec::{Encode,Decode}`‎ APIs استعمال ہوتی ہیں۔
