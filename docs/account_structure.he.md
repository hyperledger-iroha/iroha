# מסמך RFC למבנה חשבונות

**סטטוס:** טיוטה  
**קהל יעד:** צוותי מודל הנתונים, Torii, Nexus, ארנקים וממשל  
**נושאים קשורים:** TBD

## תקציר

מסמך זה מציע לעדכן מן היסוד את שיטת הכתובות של חשבונות כדי להשיג את היעדים הבאים:

- כתובת **Iroha Bech32m (IH-B32)** עם סכום ביקורת, קריאה ידידותית לאדם, הקושרת מבחין־שרשרת (chain discriminant) לחתים ומספקת ייצוג טקסטואלי דטרמיניסטי ונוח לאינטגרציה.
- מזהי דומיין ייחודיים גלובלית מגובים ברשומה שניתן לשאול דרך Nexus למטרות ניתוב בין־שרשרתי.
- שכבות תאימות השומרות על פורמט `alias@domain` הקיים בזמן שהארנקים, ה-API והחוזים נעים אל הפורמט החדש.
- חשיפות `AccountAddress` לקודקים של IH58 (הפורמט המועדף לשיתוף), תצוגת סורה הדחוסה (`snx1…`, אפשרות שנייה ייעודית ל‑Sora) ו‑Hex קנוני, תוך נירמול קלטים ב‑Torii באמצעות `AccountAddress::parse_any`.

## מוטיבציה

הארנקים והכלים מחוץ לשרשרת נשענים כיום על מחרוזות `alias@domain` גולמיות. בכך יש שני חסרונות עיקריים:

1. **אין קשירת רשת.** במחרוזת לא מתקיים סכום ביקורת או קידומת שרשרת, כך שמשתמש יכול להדביק כתובת מרשת אחרת מבלי לקבל התראה מיידית. בסופו של דבר העסקה תידחה (אי התאמה בשרשרת) או גרוע מכך – תתקבל בחשבון שאינו מיועד אם קיים יעד זהה ברשת המקומית.
2. **התנגשות דומיינים.** דומיינים הם רק מרחבי שמות וניתנים לשימוש חוזר בכל שרשרת. פדרציה של שירותים (קסטודיאנים, גשרים, תהליכי חוצי-שרשראות) הופכת לשבירה מפני ש-`finance` בשרשרת A אינו קשור ל-`finance` בשרשרת B.

נחוצה לנו שיטת כתובות נגישה לבני אדם שמונעת טעויות העתקה/הדבקה ויוצרת התאמה דטרמיניסטית בין שם הדומיין לשרשרת המוסמכת.

## יעדים

- לעצב מעטפת Bech32m בהשראת IH58 עבור חשבונות Iroha תוך פרסום צורות טקסטואליות רשמיות של CAIP-10.
- לקודד את מבחין־השרשרת המוגדר ישירות בתוך כל כתובת ולהגדיר את תהליך הממשל/הרשומות שלו.
- לתאר כיצד להכניס רשומת דומיינים גלובלית מבלי לשבור פריסות קיימות ולפרט כללי נרמול והגנה מפני התחזות.
- לתעד ציפיות תאימות, שלבי מיגרציה ושאלות פתוחות.

## יעדים שאינם בטווח

- מימוש העברות נכסים חוצי-שרשראות. שכבת הניתוב רק מחזירה את השרשרת היעד.
- שינוי המבנה הפנימי של `AccountId` (נותר `DomainId + PublicKey`).
- סיום תהליך הממשל עבור הקצאת דומיינים גלובלית. ה-RFC מתמקד במודל הנתונים ובפרימיטיבים של התעבורה.

## רקע

### מזהה החשבון הנוכחי

```
AccountId {
    domain: DomainId,   // מעטפת מעל Name (מחרוזת דמוית ASCII)
    signatory: PublicKey // מחרוזת multihash (לדוגמה ed0120...)
}

הצגה / ניתוח: "<signatory multihash>@<domain name>"
```

`ChainId` נמצא מחוץ ל-`AccountId`. הצמתים בודקים את `ChainId` של העסקה מול הקונפיגורציה בזמן הקבלה (`AcceptTransactionFail::ChainIdMismatch`) ודוחים עסקאות זרות, אך מחרוזת החשבון עצמה אינה נושאת רמז לרשת.

### מזהי דומיין

`DomainId` עוטף `Name` (מחרוזת מנורמלת) ומוגבל לשרשרת המקומית. כל שרשרת רשאית לרשום באופן עצמאי דומיינים כגון `wonderland` או `finance`.

### הקשר Nexus

Nexus אחראי לתיאום בין רכיבים (נתיבים/מרחבי נתונים). כיום אין לו מושג על ניתוב דומיינים חוצי-שרשראות.

## העיצוב המוצע

### 1. מבחין־שרשרת דטרמיניסטי

הרחיבו את `iroha_config::parameters::actual::Common` כך:

```rust
pub struct Common {
    pub chain: ChainId,
    pub chain_discriminant: u16, // חדש, מתואם גלובלית
    // ... שדות קיימים
}
```

- **הגבלות:**
  - ייחודי לכל רשת פעילה; מנוהל דרך רשומה ציבורית חתומה עם טווחים שמורים מפורשים (לדוגמה `0x0000–0x0FFF` לטסט/דב, `0x1000–0x7FFF` לחלוקת קהילה, `0x8000–0xFFEF` מאושרי ממשל, `0xFFF0–0xFFFF` שמור).
  - בלתי ניתן לשינוי עבור שרשרת פעילה. החלפה דורשת hard fork ועדכון רשומה.
- **ממשל ורשומה:** סט ממשל רב-חתימה מתחזק רשומה חתומה בפורמט JSON (וכן עוגן IPFS) הממפה מבחינים לשמות ידידותיים ולמזהי CAIP-2. לקוחות מורידים ומטמינים את הרשומה כדי לאמת ולהציג מטא־נתוני שרשרת.
- **שימוש:** העבירו את המבחין דרך קבלת המצבים, Torii, ה-SDKים ו-API של ארנקים כדי שכל רכיב יוכל להטמיע או לאמת אותו. חשפו את המבחין כחלק מ-CAIP-2 (למשל `iroha:0x1234`).

### 2. קידוד כתובת Iroha Bech32m (IH-B32)

נכניס פורמט כתובת חדש מבוסס Bech32m בהשראת IH58, תוך שמירה על מטען קומפקטי הממוקד בזהות הקריפטוגרפית היציבה:

```
קידומת קריאה לאדם (HRP): "ih" (אותיות קטנות, קבוע)
מטען הנתונים (מילים של Bech32m, כולן באותיות קטנות):
    version        : 5 ביטים (התחל ב-0)
    chain_id       : 16 ביטים (chain_discriminant)
    signatory      : בתים של multihash (אורך משתנה, כולל את האלגוריתם)
סכום ביקורת: 6 תווים של Bech32m (נוצר אוטומטית)
```

- **גירסאות:** שמרו ערכים שאינם 0 לקידוד חשבונות עתידי (למשל מולטי-סיג, חשבונות מבוססי תוויות).
- **פונקציות עזר:** הוסיפו `AccountId::to_ih_b32(&Config)` ו-`AccountId::from_ih_b32(&Config, &str)` אל `iroha_data_model`.
- **התנהגות CLI/SDK:** ברירת המחדל להצגה וקלט היא IH-B32. כל ה-API הציבוריים מצפים למחרוזות IH-B32.
- **טיפול בשגיאות:** פענוח נכשל מיידית כאשר המבחין אינו תואם את ההגדרה המקומית. ארנקים יכולים לעצור משלוח חוצה-שרשראות מראש.
- **אורך:** עם דומיין מחוץ למטען, כתובת Ed25519 מקודדת לכ~54 תווים (שרשרת + multihash בגודל 34 בתים), ובכך משפרת קריאות לעומת טיוטת 80+ התווים הקודמת.
- **רישיות:** מחרוזות IH-B32 הן באותיות קטנות בלבד כנדרש ב-Bech32m; מטענים מעורבי-רישיות נדחים בזמן פענוח.

#### 2.1 אליאסים טקסטואליים לאינטגרציה

- **סגנון שרשרת:** ‎`ih:<chain-alias>:<alias@domain>` ליומנים ולהקלדת משתמש. הארנק חייב לנתח את הקידומת, לאמת את השרשרת המשולבת ולחסום אי-התאמות.
- **צורת CAIP-10:** ‎`iroha:<caip  -id>:<ih-b32-addr>` לאינטגרציות ניטרליות לשרשרת. כלים יכולים לגזור את רכיב CAIP-2 מרשומת המבחין.
- **עזרי מכונה:** פרסמו קודקים עבור Rust, TypeScript/JavaScript, Python ו-Kotlin המכסים IH-B32 ו-CAIP-10.

#### 2.2 אליאס IH58 דטרמיניסטי

- **מיפוי קידומת:** השתמשו ב-`chain_discriminant` כקידומת הרשת של IH58. הרשימה הסמכותית מתוחזקת במסמך [`address_prefix_registry.md`](source/references/address_prefix_registry.md) ובקובץ ה‑JSON הצמוד אליו, וה‑SDKs חייבים להישאר מסונכרנים כדי למנוע התנגשויות.
- **חומר חשבון:** חילצו את מטען המפתח הגולמי מה-multihash. אם האורך אינו 32 בתים בדיוק (למשל מפתח secp256k1 דחוס או סכמות עתידיות), הפיקו `AccountId32 = blake2b-256(multihash_bytes)` כדי להתאים לרוחב IH58. כך שומרים אליאסים דטרמיניסטיים בין אלגוריתמים.
- **קידוד:** הפעילו את תהליך סכום הביקורת התקני של IH58 עם הקידומת והמחרוזת `AccountId32` כדי לקבל מחרוזת Base58Check תואמת. פרסמו את לוגיקת המיפוי לצד ספריות הקודקים לשימוש חוזר על ידי המיישמים.

#### 2.3 וקטורי בדיקה נורמטיביים

- **Ed25519,‏ chain `0x1234`,‏ domain `wonderland`**  
  multihash קלט: `ed0120CE7FA46C9DCE7EA4B125E2E36BDB63EA33073E7590AC92816AE1E861B7048B03`  
  IH-B32: `ih1qzg6w6qfqeel6gmyaeel2fvf9ut3khkmragesw0n4jzkf9qt2u85xrdcy3vpsl337y0`  
  CAIP-10: `iroha:0x1234:ih1qzg6w6qfqeel6gmyaeel2fvf9ut3khkmragesw0n4jzkf9qt2u85xrdcy3vpsl337y0`  
  סגנון שרשרת: `ih:iroha-main:ed0120CE7FA46C9DCE7EA4B125E2E36BDB63EA33073E7590AC92816AE1E861B7048B03@wonderland`  
  אליאס IH58: קידומת `0x1234`, בתים של החשבון `blake2b-256(multihash)`.
- **secp256k1,‏ chain `0x0001`,‏ domain `treasury`**  
  multihash קלט: `e701210279BE667EF9DCBBAC55A06295CE870B07029BFCDB2DCE28D959F2815B16F81798`  
  IH-B32: `ih1qqqq7wqfpqfumuen7l8wthtz45p3ftn58pvrs9xlumvkuu2xet8egzkcklqtestc32u8`  
  CAIP-10: `iroha:0x0001:ih1qqqq7wqfpqfumuen7l8wthtz45p3ftn58pvrs9xlumvkuu2xet8egzkcklqtestc32u8`  
  סגנון שרשרת: `ih:iroha-dev:e701210279BE667EF9DCBBAC55A06295CE870B07029BFCDB2DCE28D959F2815B16F81798`  
  אליאס IH58: קידומת `0x0001`, בתים של החשבון `blake2b-256(multihash)`.
- **מקרה כשל:** פענוח הכתובת הראשונה בצומת שבו `chain_discriminant = 0x0001` חייב להניב `ChainMismatch` לפני שמנסים לנתב דומיין. שינוי תו כלשהו יפיל את סכום הביקורת של Bech32m.

### 3. דומיינים ייחודיים גלובלית ונרמול

לקבלת תיאור מלא של תהליך Norm v1 ראו את
`docs/source/references/address_norm_v1.md` (אנגלית).

הגדירו מחדש את `DomainId` כטופל מתויג:

```
DomainId {
    name: Name,
    authority: GlobalDomainAuthority, // enum חדש
}

enum GlobalDomainAuthority {
    LocalChain,                  // ברירת מחדל לשרשרת המקומית
    External { chain_discriminant: u16 },
}
```

`LocalChain` משמר את המודל הקיים שבו הדומיין מנוהל על ידי השרשרת המקומית. כאשר דומיין נרשם דרך הרשומה הגלובלית, נשמר מבחין־השרשרת של השרשרת הבעלים. התצוגה/הניתוח נשארים ללא שינוי לעת עתה, אך המבנה המורחב מאפשר החלטות ניתוב.

#### 3.1 נרמול והגנה מפני התחזות

- **מערך תווים:** ‎`Name` נשאר ASCII כברירת מחדל (אותיות, ספרות, `-`,‏ `_`). אם וכאשר תידרש בינאום, אימצו את **UTS-46** לרישום/חיפוש ואת **UTS-39** לזיהוי דמויי-כתיבה עם מדיניות סקריפטים ברורה.
- **רישיות:** אכפו אחסון באותיות קטנות; דחו קלט במעורב.
- **מגבלות אורך:** הגדירו אורך מרבי של תווית (למשל 63 תווים) ואורך כולל של דומיין (למשל 255 תווים) לציפיות דמויות DNS.
- **מיגרציה:** דומיינים קיימים נותרים תחת `LocalChain`; דומיינים גלובליים חדשים חייבים לעבור בדיקות נרמול והתחזות לפני פרסום.

### 4. רשומת דומיינים של Nexus וניתוב

- **סכמת הרשומה:** Nexus מתחזק מיפוי חתום `DomainName -> ChainRecord`, כאשר `ChainRecord` כולל את מבחין־השרשרת, מטא־נתונים אופציונליים (נקודות קצה RPC) והוכחת סמכות (למשל רב-חתימה של ממשל).
- **מנגנון סנכרון:**
  - שרשראות מגישות ל-Nexus תביעות דומיין חתומות (בזמן ג'נסיס או באמצעות הוראת ממשל).
  - Nexus מפרסם מניפסטים תקופתיים (JSON חתום ואפשרי שורש Merkle) דרך HTTPS ואחסון מוכוון תוכן (למשל IPFS). לקוחות מצמידים את המניפסט העדכני ומוודאים חתימות.
- **זרימת חיפוש:**
  - Torii מקבל עסקה המתייחסת ל-`DomainId`.
  - אם הדומיין אינו מוכר מקומית, Torii פונה למניפסט Nexus המטמון.
  - אם המניפסט מצביע על שרשרת זרה, Torii דוחה עם שגיאה דטרמיניסטית `ForeignDomain` ופרטי השרשרת המרוחקת.
  - אם הדומיין חסר במניפסט, Torii מחזיר `UnknownDomain`.
- **עוגני אמון וסבב מפתחות:** מפתחות הממשל חותמים על המניפסטים; רוטציה או ביטול מתפרסמים כערך מניפסט חדש. לקוחות אוכפים TTL למניפסט (למשל 24 שעות) ומסרבים להתייחס לנתונים שפג תוקפם.
- **מצבי כשל:** אם כשלו להביא מניפסט, Torii משתמש בנתונים המטמונים כל עוד ה-TTL תקף; לאחר פג התוקף הוא מחזיר `RegistryUnavailable` ומסרב לנתוב דומיינים זרים כדי למנוע מצב שגוי.

תמיכה אוטומטית: הפקודה `cargo xtask address-manifest verify --bundle <dir> --previous <dir>`
מריצה את כל בדיקות ה-runbook (סכומי בדיקה, סכמת רשומות וקישור `previous_digest`) כמתועד ב-
[address manifest runbook](source/runbooks/address_manifest_ops.md). צרפו את הפלט לטיקט השינוי כדי
להראות ששרשור ה-`sequence`/`previous_digest` אומת לפני הפצה.

### 5. ידידותיות לארנק ול-API

- **ברירת מחדל להצגה:** ארנקים מציגים את כתובת IH-B32 (קצרה ובעלת סכום ביקורת) לצד הדומיין שנשלף מהרשומה. הדומיין מודגש כמידע תיאורי שעשוי להשתנות, בעוד IH-B32 היא הכתובת היציבה.
- **מצב קשוח:** Torii,‏ SDKים וארנקים מספקים דגל תצורה שמחייב קלט IH-B32 מקצה לקצה כדי שקסטודיאנים יוכלו לבצע הרצת נוהלי ציות.
- **מניעת שגיאות:** ארנקים מנתחים קידומות IH-B32 ו-CAIP-10. אי-התאם שרשרת גורר כשל חמור עם הודעה אינפורמטיבית.
- **ספריות קודקים:** ספריות רשמיות ל-Rust,‏ TypeScript/JavaScript,‏ Python ו-Kotlin מספקות קידוד/פענוח IH-B32 והמרות CAIP-10 כדי למנוע פיצול יישומים.

#### הנחיות נגישות ושיתוף בטוח

- **זרימות שיתוף בטוחות:** מסכים שמציגים או מעתיקים כתובות צריכים להשתמש ב-IH-B32 כברירת מחדל ולספק כפתור "שיתוף" סמוך שמציג את המחרוזת המלאה ואת קוד ה-QR מאותו מטען כדי שמשתמשים יוכלו לאמת את סכום הביקורת בעין או בסריקה. כאשר לא ניתן להציג את כל הכתובת (למשל במסכים קטנים), שמרו את ההתחלה והסוף, הוסיפו אליפסות ברורות, והשאירו העתקה ללוח שמחזירה תמיד את הכתובת המלאה כדי למנוע חיתוכים בטעות.
- **הגנות IME:** טפסי כתובת צריכים לדחות שרידי קומפוזיציה של מקלדות IME. כפו קלט ASCII בלבד, הציגו אזהרה מיידית כאשר מזוהים תווים ברוחב מלא או קטקאנה, וספקו אזור הדבקה בטקסט פשוט שמסיר סימנים משולבים לפני האימות כדי שמשתמשים ביפנית/סינית לא יאבדו קלט כשהם משביתים זמנית את ה-IME.
- **תמיכת קוראי מסך:** הוסיפו תוויות סמויות (`aria-label`/`aria-describedby`) שמפרקות את ה-HRP ומחלקות את המטען לקבוצות של 4–8 תווים כך שכלי נגישות יקראו "ih מקף b שלוש שתיים" במקום מחרוזת אחת רציפה. הודיעו על הצלחת העתקה/שיתוף באמצעות אזורי live ברמת polite, ודאגו שלתצוגות QR יהיה טקסט חלופי תיאורי ("כתובת IH-B32 עבור <alias> בשרשרת 0x1234").
- **תצוגת Sora דחוסה בלבד:** תייגו תמיד את הפורמט `snx1…` כ"ל-Sora בלבד" ודרשו פעולה מפורשת לפני העתקה. כאשר מבחין-השרשרת אינו ערך Sora Nexus, SDKים וארנקים צריכים לסרב להציג את הפלט הדחוס ולהחזיר משתמשים ל-IH-B32 בעת העברות בין רשתות כדי למנוע ניתוב שגוי של כספים.

### 6. אסטרטגיית תאימות

1. **שלב 0 – הכנה**
   - הוסיפו פונקציות עזר לקידוד/פענוח ואת המבחין בקונפיגורציה מבלי לשנות את תצוגת ברירת המחדל. עדכנו SDKים במצב opt-in.
2. **שלב 1 – השקה רשמית**
   - Torii מקבל בקשות ב-IH-B32 ומשיב בצורתו הקנונית.
   - גלו מוני API כדי שמפעילים יוכלו לוודא אימוץ IH-B32.
3. **שלב 2 – אכיפה**
4. **שלב 3 – ברירת מחדל**
   - הסירו אליאסים מעבריים ודרשו IH-B32 בכל גבול.

במהלך כל השלבים, רישומי דומיין נשארים מקומיים אלא אם שרשרת בוחרת להצטרף לרשומה הגלובלית.

## רשימת בדיקות מימוש

- **מעטפת IH-B32:** HRP ‏`ih`,‏ גירסה `0`,‏ ערך‎ `chain_discriminant` ‎של 16 ביט ב-endian גדול, מטען multihash, ריפוד 5 ביטים לפי Bech32m ופלט באותיות קטנות.
- **חוזה רשומה:** JSON חתום (ואפשרית שורש Merkle) המפרסם `{discriminant, ih58_prefix, chain_alias, endpoints}` עם TTL של 24 שעות וסבב מפתחות.
- **מדיניות דומיינים:** `Name` ב-ASCII כיום; אם מפעילים ‎i18n, יישמו UTS-46 לנרמול ו-UTS-39 לזיהוי דמויי-תווים. אכפו אורך תווית מרבי (63) ואורך כולל (255).
- **עזרי טקסט:** ספקו קודקים IH-B32 ↔ CAIP-10 ↔ IH58 ב-Rust,‏ TypeScript/JavaScript,‏ Python ו-Kotlin עם וקטורי בדיקה משותפים.
- **בדיקות:** הוסיפו בדיקות יחידה לסיבובי קידוד/פענוח, כשלי שרשרת שגויה וחיפושי מניפסט; הוסיפו כיסוי אינטגרציה ב-Torii וב-SDKים לזרימות IH-B32 מקצה לקצה.

## חלופות שנשקלו

- **Base58Check (סגנון ביטקוין).** סכום ביקורת פשוט אך חלש יותר מ-Bech32m ואינו גמיש לגרסאות עתידיות.
- **הטמעת שם השרשרת במחרוזת הדומיין (למשל `finance@chain`).** שוברת תאימות ועדיין חסרה סכום ביקורת; קשה לאימות.
- **הישענות על ניתוב Nexus בלבד ללא שינוי כתובות.** המשתמשים ימשיכו להעתיק מחרוזות דו-משמעיות; אנו רוצים שהכתובת עצמה תישא הקשר.
- **אימוץ IH58 כלשונו.** מספק תקדים חזק ורשומה מתוחזקת, אך Base58Check פחות נוח ל-QR. ההצעה שומרת כתובות Bech32m קומפקטיות תוך פרסום מיפויים דטרמיניסטיים ל-IH58 ו-CAIP-10.

## שאלות פתוחות

- ודאו שמבחיני `u16` עם טווחים שמורים יספיקו לטווח הארוך; אחרת בחנו `u32` עם קידוד varint.
- השלימו את הליך הממשל הרב-חתימתי לעדכוני רשומה וכיצד מטפלים בביטולים/הקצאות שפג תוקפן.
- הגדירו את סכמת החתימה המדויקת של המניפסט (למשל רב-חתימה Ed25519) ואת אבטחת ההעברה (הצמדת HTTPS, פורמט hash של IPFS) להפצת Nexus.
- קבעו האם לתמוך באליאסי דומיין/הפניות במיגרציות וכיצד לחשוף אותם מבלי לפגוע בדטרמיניזם.
- הגדירו כיצד חוזי Kotodama/IVM ניגשים לעזרי IH-B32 (`to_address()`,‏ `parse_address()`) והאם אחסון על השרשרת יעדיף מחרוזות IH-B32 או CAIP-10.
- בחנו רישום שרשראות Iroha ברשומות חיצוניות (למשל רשם IH58, ספריית שמות CAIP) לשיפור התאמה במערכת האקולוגית.

### 7. פריסת מעטפת AccountAddress (ADDR-1a)

#### 7.1 בית הכותרת

כל מעטפת קנונית בנויה כ־`header · domain selector · controller`. בית הכותרת הראשון (בית אחד) מודיע
לפענחנים באילו כללים להשתמש:

```
bit index:   7        5 4      3 2      1 0
             ┌─────────┬────────┬────────┬────┐
payload bit: │version  │ class  │  norm  │ext │
             └─────────┴────────┴────────┴────┘
```

| ביטים | שדה | ערכים מותרים | שגיאה כאשר הערך אינו נתמך |
|-------|------|---------------|---------------------------|
| ‎7‑5  | `addr_version` | `0` (גרסת V1). ערכים `1‑7` שמורים. | ‎`AccountAddressError::InvalidHeaderVersion` |
| ‎4‑3  | `addr_class` | `0` = מפתח יחיד, ‎`1` = מולטי-סיג. | ‎`AccountAddressError::UnknownAddressClass` |
| ‎2‑1  | `norm_version` | ‎`1` (Norm v1). ערכים אחרים שמורים. | ‎`AccountAddressError::InvalidNormVersion` |
| ‎0    | `ext_flag` | חייב להיות ‎`0`. | ‎`AccountAddressError::UnexpectedExtensionFlag` |

#### 7.2 סלקטור הדומיין

הסלקטור מגיע מיד אחרי הכותרת וכולל תג + מטען:

```
domain selector
┌──────────┬──────────────────────────────────────────────┐
│ tag (u8) │ payload (מתאים לסוג הסלקטור)                │
└──────────┴──────────────────────────────────────────────┘
```

הדיג'סט נקבע לאחר נרמול UTS‑46/STD3/NFC:

- ‎`0x00` – דומיין ברירת המחדל. אין מטען.
- ‎`0x01` – סלקטור מקומי. המטען הוא דיג'סט בגודל 12 בתים (`blake2s_mac("SORA-LOCAL-K:v1", label)[0..12]`).
- ‎`0x02` – פריט רשומה גלובלית. המטען הוא `registry_id` של ‎4 בתים ב-endian גדול.

כל תג לא מוכר מפיל ‎`AccountAddressError::UnknownDomainTag`, וחוסר התאמה בין דומיין צפוי לבין הסלקטור
גורם ל־`AccountAddressError::DomainMismatch`.

#### 7.3 דוגמאות קנוניות

- **ברירת מחדל (tag ‎`0x00`)**  
  `0x0200000120641297079357229f295938a4b5a333de35069bf47b9d0704e45805713d13c201`
- **סלקטור מקומי ל־`treasury` (tag ‎`0x01`)**  
  `0x0201b18fe9c1abbac45b3e38fc5d0001203b77a042f1de02f6d5f418f36a20fd68c8329fe3bbfbecd26a2d72878cd827f8`
- **רשומה גלובלית עם `registry_id = 42` (tag ‎`0x02`)**  
  `0x02020000002a000120641297079357229f295938a4b5a333de35069bf47b9d0704e45805713d13c201`

בכל הדוגמאות: `0x02` הוא בית הכותרת (גרסה 0, מחלקה single-key, Norm v1, ללא הרחבות), לאחריו תג הסלקטור,
המטען הספציפי, ולבסוף התג `0x00` של בקר המפתח היחיד (`curve_id = 0x01`, אורך 32 בתים וערכי המפתח בפועל).
