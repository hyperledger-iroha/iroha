# Руководство по исполнению и эксплуатации AMX

**Статус:** Черновик (NX‑17)  
**Аудитория:** Команда протокола core, инженеры AMX/консенсуса, SRE/телеметрия,
команды SDK и Torii  
**Контекст:** Закрывает пункт roadmap’а «Документация (owner: Docs) —
обновить `docs/amx.md` диаграммами времени, каталогом ошибок, ожиданиями
операторов и рекомендациями для разработчиков по генерации/использованию
PVO».

## Краткое описание

Атомарные транзакции между пространствами данных (AMX) позволяют одной
транзакции затрагивать несколько data‑spaces (DS), сохраняя 1‑секундную
финальность слота, детерминированные коды ошибок и конфиденциальность
фрагментов в приватных DS. В этом документе описаны временная модель,
каноническая обработка ошибок, требования к доказательной базе для операторов
и ожидания к разработчикам относительно Proof Verification Objects (PVO),
чтобы результат roadmap’а оставался самодостаточным без постоянных ссылок на
дизайн‑док Nexus (`docs/source/nexus.md`).

Ключевые гарантии:

- Для каждой AMX‑транзакции заданы детерминированные бюджеты prepare/commit;
  превышение приводит к abort’у с документированными кодами, а не к
  зависанию lanes.
- DA‑выборки, не укладывающиеся в бюджет, переносят транзакцию на следующий
  слот и эмитят явную телеметрию (`missing-availability warning`) вместо тихого
  падения пропускной способности.
- PVO отделяют тяжёлые доказательства от 1‑секундного слота, позволяя
  клиентам/батчерам заранее регистрировать артефакты, которые Nexus‑host
  может быстро проверить внутри слота.

## Модель тайминга слота

### Временная шкала

```text
t=0ms           70ms             300ms              600ms       840ms    1000ms
│─────────┬───────────────┬───────────────────┬──────────────┬──────────┬────────│
│         │               │                   │              │          │        │
│  Mempool│Proof build + DA│Consensus PREP/COM │ IVM/AMX exec │Settlement│ Guard  │
│  ingest │sample (≤300ms) │(≤300ms)           │(≤250ms)      │(≤40ms)   │(≤40ms) │
```

- Бюджеты согласованы с глобальным планом ledger’а: mempool 70 мс,
  commit DA ≤300 мс, консенсус 300 мс, IVM/AMX 250 мс, settlement 40 мс,
  guard 40 мс.
- Транзакции, выходящие за DA‑окно, детерминированно переносятся в следующий
  слот; остальные нарушения приводят к кодам `AMX_TIMEOUT` или
  `SETTLEMENT_ROUTER_UNAVAILABLE`.
- guard‑срез поглощает экспорт телеметрии и финальный аудит так, чтобы слот
  закрывался за 1 с, даже если exporters немного отстают.

### Cross‑DS swim‑lane

```text
Client        DS A (public)        DS B (private)        Nexus Lane        Settlement
  │ submit tx │                     │                     │                 │
  │──────────▶│ prepare fragment    │                     │                 │
  │           │ proof + DA part     │ prepare fragment    │                 │
  │           │───────────────┬────▶│ proof + DA part     │                 │
  │           │               │     │─────────────┬──────▶│ Merge proofs    │
  │           │               │     │             │       │ verify PVO/DA   │
  │           │               │     │             │       │────────┬────────▶ apply
  │◀──────────│ result + code │◀────│ result + code │◀────│ outcome│          receipt
```

Каждый DS‑фрагмент должен завершить своё 30‑мс prepare‑окно до того, как
lane соберёт слот. Отсутствующие доказательства остаются в mempool’е до
следующего слота, а не блокируют peers.

### Чек‑лист по инструментированию

| Метрика / трейс | Источник | SLO / алерт | Примечания |
|-----------------|----------|-------------|------------|
| `iroha_slot_duration_ms` (гистограмма) / `iroha_slot_duration_ms_latest` (gauge) | `iroha_telemetry` | p95 ≤ 1000 мс | CI‑гейт описан в `ans3.md`. |
| `iroha_da_quorum_ratio` | `iroha_telemetry` (commit‑hook) | ≥0.95 за 30‑минутное окно | Получается из телеметрии по пересчёту DA; каждый блок обновляет gauge. |
| `iroha_amx_prepare_ms` | IVM‑host | p95 ≤ 30 мс на DS‑scope | Управляет abort’ами `AMX_TIMEOUT`. |
| `iroha_amx_commit_ms` | IVM‑host | p95 ≤ 40 мс на DS‑scope | Покрывает merge делт и исполнение триггеров. |
| `iroha_ivm_exec_ms` | IVM‑host | алерт при >250 мс на lane | Отражает окно исполнения IVM‑overlays. |
| `iroha_amx_abort_total{stage}` | executor | алерт при >0.05 abort’ов/слот или устойчивом росте по одной стадии | `stage`: `prepare`, `exec`, `commit`. |
| `iroha_amx_lock_conflicts_total` | планировщик AMX | алерт при >0.1 конфликтов/слот | Указывает на неточные R/W‑сеты. |

## Каталог ошибок AMX и ожидания к операторам

Типичные ошибки:

- `AMX_TIMEOUT` — фрагмент превысил свой бюджет (prepare/exec/commit).
- `AMX_LOCK_CONFLICT` — конфликт блокировки, обнаруженный планировщиком.
- `SETTLEMENT_ROUTER_UNAVAILABLE` — settlement‑router не смог маршрутизировать
  операцию внутри слота.
- `PVO_MISSING` — ожидался заранее зарегистрированный PVO, но он не найден.

Операторы должны:

- Следить за перечисленными метриками и поддерживать отдельные AMX‑dashboards
  в Grafana.
- Иметь runbook’и для реакции на всплески `AMX_TIMEOUT` и
  `AMX_LOCK_CONFLICT` (например, пересмотр hints по доступу или настройка
  бюджетов).
- Архивировать артефакты (логи, traces, PVO) для AMX‑инцидентов так, чтобы
  их можно было воспроизвести и исследовать offline.

## Рекомендации разработчикам: PVO

- Proof Verification Objects инкапсулируют тяжёлые доказательства (например,
  ZK‑доказательства для приватных фрагментов) в артефакт, который Nexus‑host
  может быстро проверить в рамках слота.
- Клиенты должны preregistrировать PVO вне критического пути слота и
  ссылаться на них в AMX‑транзакциях через стабильные идентификаторы.
- Дизайн PVO должен гарантировать:
  - Каноничное Norito‑инкапсулирование (версионированные поля, Norito‑header,
    checksum).
  - Чёткое разделение публичных/приватных данных, чтобы сохранять
    конфиденциальность.

Дополнительные детали дизайна и точные форматы PVO приведены в
`docs/source/nexus.md`.
