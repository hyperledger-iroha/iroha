## חבילת רגולציה של האיחוד האירופי (ETSI/GDPR)חבילת האיחוד האירופי קושרת את שלושת החפצים שלמעלה בתוספת התזכיר המשפטי:

- עדכן את `security_target.md` עם מזהה הגרסה, Torii hash מניפסט,
  ו-SBOM digest כך שמבקרים יכולים להתאים קבצים בינאריים להיקף המוצהר.
- שמור את סיכום ה-DPIA בהתאמה למדיניות הסריקה העדכנית של הטלמטריה
  צרף את קטע הבדל Norito המוזכר ב-`docs/source/sdk/android/telemetry_redaction.md`.
- הערך של אישור SBOM צריך לכלול: CycloneDX JSON hash, מקור
  חבילת hash, הצהרת cosign וכתובת האתר לעבודה של Buildkite שיצרה אותם.
- `legal_signoff_memo.md` חייב ללכוד את הייעוץ/תאריך, לרשום כל חפץ +
  SHA-256, תאר את כל בקרות הפיצוי, וקשר לשורת יומן הראיות
  בתוספת מזהה כרטיס PagerDuty שעקב אחר האישור.

## חבילת רגולציה ביפן (FISC/StrongBox)

הרגולטורים של יפן מצפים לצרור מקביל עם תיעוד דו לשוני:

- `fisc_controls_checklist.md` משקף את הגיליון האלקטרוני הרשמי; למלא את שניהם
  עמודות EN ו-JA והתייחסו לסעיף הספציפי של `sdk/android/security.md`
  או חבילת אישורי StrongBox שעונה על כל פקד.
- `strongbox_attestation.md` מסכם את הריצות האחרונות של
  `docs/source/sdk/android/readiness/android_strongbox_attestation_bundle.md`
  (לכל מכשיר JSON + Norito מעטפות). הטמע קישורים לחפצי אמנות בלתי ניתנים לשינוי
  תחת `artifacts/android/attestation/<device>/` ושימו לב לקצב הסיבוב.
- רשמו את תבנית המכתב המקדים הדו-לשוני הנשלחת עם הגשות בפנים
  `docs/source/compliance/android/jp/README.md` כדי שהתמיכה תוכל לעשות בו שימוש חוזר.
- עדכן את יומן ההוכחות בשורה אחת המפנה לרשימת הבדיקה, ה
  Hash של חבילת אישורים, וכל מזהה כרטיס שותף JP הקשור למשלוח.

## זרימת עבודה להגשה

1. **טיוטה** - הבעלים מכין את החפץ, מתעד את שם הקובץ המתוכנן
   הטבלה שלמעלה, ופותחת יחסי ציבור המכילים את ה-Markdown המעודכן בתוספת א
   סכום בדיקה של הקובץ המצורף החיצוני.
2. **סקירה** - הנדסת שחרור מאשרת ש-hash מוצא תואם לשלבים
   בינאריים; תאימות מאמתת את השפה הרגולטורית; תמיכה מבטיחה SLAs ו
   מדיניות טלמטריה זוכה להתייחסות נכונה.
3. **יציאה** - המאשרים מוסיפים את שמם ותאריכים שלהם לטבלה `Sign-off`
   למטה. יומן ההוכחות מתעדכן עם כתובת ה-PR ו-Buildkite.
4. **פרסום** - לאחר החתמת ממשל SRE, קשר את החפץ פנימה
   `status.md` ועדכן את ההפניות ל-Android Support Playbook.

## הזמנת מעבדת מכשירים ותוכנית מגירה

כדי להפחית את הסיכון של **זמינות מעבדת המכשיר** המופיעה במפת הדרכים:- עקוב אחר קיבולת שבועית ב-`docs/source/compliance/android/evidence_log.csv`
  (עמודה `device_lab_capacity_pct`). התראה על הנדסת שחרור אם זמינות
  יורד מתחת ל-70% במשך שבועיים רצופים.
- שמור מסלולים StrongBox/כלליים הבאים
  `docs/source/compliance/android/device_lab_reservation.md` לפני כל
  הקפאה, חזרות או תאימות לטאטא בקשות, אישורים וחפצי אמנות
  נלכדים בתור `_android-device-lab`. קשר את מזהי הכרטיסים שהתקבלו
  ביומן הראיות בעת הקלטת צילומי מצב.
- **בריכות נפילה:** פרצו תחילה לבריכה המשותפת של Pixel; אם עדיין רווי,
  תזמן הפעלת עשן של Firebase Test Lab לאימות CI.
- **מחזיק מעבדה חיצוני:** אחזק את המחזיק עם שותף StrongBox
  מעבדה כדי שנוכל לשמור חומרה במהלך הקפאת חלונות (הובלה של 7 ימים לכל הפחות).
- **הסלמה:** העלאת תקרית `AND6-device-lab` ב-PagerDuty כאשר הן
  בריכות ראשוניות ובריכות הגנה נופלות מתחת לקיבולת של 50%. מוביל מעבדת החומרה
  מתאם עם SRE כדי לתעדף מחדש מכשירים.
- **חבילות ראיות לכשל:** מאחסנים כל חזרה תחת
  `artifacts/android/device_lab_contingency/<YYYYMMDD>/` עם ההזמנה
  בקשה, ייצוא PagerDuty, מניפסט חומרה ותמלול שחזור. הפניה
  החבילה מ-`device_lab_contingency.md` והוסף את ה-SHA-256 ליומן ההוכחות
  כך ש-Legal יכול להוכיח שזרימת העבודה המקרית הופעלה.
- **תרגילים רבעוניים:** תרגיל את ספר ההפעלה
  `docs/source/compliance/android/device_lab_failover_runbook.md`, צרף את
  נתיב חבילה שהתקבל + מניפסט hash לכרטיס `_android-device-lab`, ו
  שיקוף את מזהה התרגיל גם ביומן המגירה וגם ביומן ההוכחות.

תיעוד כל הפעלה של תוכנית המגירה ב
`docs/source/compliance/android/device_lab_contingency.md` (כלול תאריך,
טריגר, פעולות ומעקבים).

## זרימת עבודה של מכשירים-מעבדה

כל הזמנה או חזרת תקלות חייבת לעקוב אחר
`device_lab_instrumentation.md` מדריך כך טלמטריה, תור ואישור
חפצי אמנות מתיישבים עם יומן ההזמנות:

1. **חפצי חריץ זרעים.** צור
   `artifacts/android/device_lab/<slot>/` עם תיקיות המשנה הסטנדרטיות והפעל
   `shasum` לאחר סגירת החריץ (ראה סעיף "פריסת חפץ" של הגרסה החדשה
   מדריך).
2. **הפעל פקודות מכשור.** בצע את לכידת הטלמטריה/תור,
   לעקוף עיכול, רתמת StrongBox וסריקת מוך/תלות בדיוק כמו
   מתועד כך שהפלטים משקפים את CI.
3. **הגשת ראיות.** עדכון
   `docs/source/compliance/android/evidence_log.csv` וכרטיס ההזמנה
   עם מזהה החריץ, נתיב מניפסט SHA-256 ולוח המחוונים/Buildkite המתאים
   קישורים.

צרף את תיקיית ה-Artefact ואת ה-hash מניפסט לחבילת השחרור של AND6 עבור
חלון ההקפאה המושפע. מבקרי ממשל ידחו רשימות ביקורת שכן
אל תציין מזהה חריץ בתוספת מדריך המכשור.

### עדות להזמנה ומוכנות לכשל

פריט מפת הדרכים "אישורי חפצים רגולטוריים ותקרית מעבדה" דורש יותר
מאשר מכשור. כל חבילת AND6 חייבת להתייחס גם לפרואקטיבית
זרימת עבודה של הזמנה וחזרה רבעונית לכשל:- **ספר משחקי הזמנה (`device_lab_reservation.md`).** עקוב אחר ההזמנה
  טבלה (זמני אספקה, בעלים, אורכי משבצות), ייצא את לוח השנה המשותף באמצעות
  `scripts/android_device_lab_export.py`, ורשומה `_android-device-lab`
  מזהי כרטיסים לצד צילומי מצב ב-`evidence_log.csv`. ספר המשחקים
  מפרט את סולם ההסלמה והטריגרים של מגירה; להעתיק את הפרטים האלה
  לכניסה לרשימת הבדיקה כאשר ההזמנות זזות או הקיבולת יורדת מתחת ל-
  יעד מפת דרכים של 80%.
- **פנקס תרגילי כשל (`device_lab_failover_runbook.md`).** בצע את
  חזרה רבעונית (לדמות הפסקה ← לקדם מסלולי נגיעה ← לעסוק
  פרץ Firebase + שותף StrongBox חיצוני) ואחסן את החפצים מתחת
  `artifacts/android/device_lab_contingency/<drill-id>/`. כל צרור חייב
  מכיל את המניפסט, ייצוא PagerDuty, קישורי הפעלה של Buildkite, פרץ Firebase
  דוח, ואישור שומר המצוין בספר ההפעלה. התייחס ל
  מזהה תרגיל, מניפסט SHA-256 וכרטיס מעקב גם ביומן ההוכחות וגם
  רשימת הבדיקה הזו.

יחד המסמכים הללו מוכיחים שתכנון קיבולת המכשיר, חזרות הפסקות,
וחבילות המכשור חולקות את אותו מסלול מבוקר שדרש ה-
מפת דרכים וסוקרים משפטיים.

## סקור את הקצב

- **רבעוני** - אמת שחפצי אמנות של האיחוד האירופי/JP מעודכנים; לרענן
  גיבוב יומן ראיות; להתאמן על לכידת מקור.
- **טרום שחרור** - הפעל את רשימת הבדיקה הזו במהלך כל חיתוך GA/LTS והצמד
  היומן שהושלם ל-RFC המהדורה.
- **אחרי תקרית** - אם תקרית חצי שנייה נוגעת בטלמטריה, חתימה או
  אישור, עדכן את בדלי החפץ הרלוונטיים עם הערות תיקון ו
  ללכוד את ההפניה ביומן הראיות.
