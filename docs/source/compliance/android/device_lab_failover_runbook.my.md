# Android Device Lab Failover Drill Runbook (AND6/AND7)

ဤအပြေးစာအုပ်သည် လုပ်ထုံးလုပ်နည်း၊ အထောက်အထားလိုအပ်ချက်များနှင့် အဆက်အသွယ်မက်ထရစ်ကို ဖမ်းယူထားသည်။
ရည်ညွှန်းထားသော **device-lab contingency plan** ကို ကျင့်သုံးသောအခါတွင် အသုံးပြုသည်။
`roadmap.md` (§“စည်းကမ်းထိန်းသိမ်းရေးဆိုင်ရာ အတည်ပြုချက်များနှင့် ဓာတ်ခွဲခန်းဆိုင်ရာ အရေးပေါ်အခြေအနေ”)။ ဖြည့်ပေးတယ်။
ကြိုတင်မှာကြားခြင်းလုပ်ငန်းအသွားအလာ (`device_lab_reservation.md`) နှင့် အဖြစ်အပျက်မှတ်တမ်း
(`device_lab_contingency.md`) ထို့ကြောင့် လိုက်နာမှုပြန်လည်သုံးသပ်သူများ၊ ဥပဒေအကြံပေးနှင့် SRE
ကျွန်ုပ်တို့သည် ကျရှုံးခြင်းအတွက် အဆင်သင့်ဖြစ်ခြင်းကို သက်သေပြနိုင်ပုံအတွက် အမှန်တရား၏ အရင်းအမြစ်တစ်ခုတည်းရှိသည်။

## ရည်ရွယ်ချက် & ကဒိန်

- Android StrongBox + ယေဘူယျစက်ပစ္စည်းအစုများသည် ပျက်ကွက်နိုင်ကြောင်း သရုပ်ပြပါ။
  ဘေးထွက် Pixel လမ်းကြောများ၊ မျှဝေထားသော ရေကူးကန်၊ Firebase Test Lab ဆက်တိုက်တန်းစီခြင်းနှင့်
  AND6/AND7 SLA များ မလွတ်ဘဲ ပြင်ပ StrongBox ထိန်းထိန်းသိမ်းသိမ်း။
- တရားဝင် ETSI/FISC တင်ပြမှုများတွင် ပူးတွဲပါနိုင်သည့် အထောက်အထားအစုအဝေးကို ထုတ်လုပ်ပါ။
  Feb လိုက်နာမှုပြန်လည်သုံးသပ်ခြင်းမတိုင်မီ။
- လေးပုံတစ်ပုံလျှင် အနည်းဆုံး တစ်ကြိမ် လုပ်ဆောင်နိုင်ပြီး ဓာတ်ခွဲခန်း ဟာ့ဒ်ဝဲစာရင်းကို ပြောင်းလဲသည့် အချိန်တိုင်း လုပ်ဆောင်ပါ။
  (စက်ပစ္စည်းအသစ်များ၊ အငြိမ်းစားယူခြင်း သို့မဟုတ် ပြုပြင်ထိန်းသိမ်းမှု 24 နာရီထက် ပိုကြာသည်)။

## လုပ်ထုံးလုပ်နည်း

### 1. ကြိုတင်ပြင်ဆင်မှု

1. `docs/source/sdk/android/android_strongbox_capture_status.md` တွင် အခြေခံလိုင်းစွမ်းရည်ကို အတည်ပြုပါ။
2. ပစ်မှတ် ISO ရက်သတ္တပတ်အတွက် ကြိုတင်မှာကြားထားသည့် ပြက္ခဒိန်ကို ထုတ်ယူပါ။
   `python3 scripts/android_device_lab_export.py --week <ISO week>`။
3. File `_android-device-lab` လက်မှတ်
   `AND6-DR-<YYYYMM>` နယ်ပယ် (“failover drill”)၊ စီစဉ်ထားသော အပေါက်များနှင့် ထိခိုက်မှု
   အလုပ်ဝန်များ (သက်သေခံချက်၊ CI မီးခိုး၊ တယ်လီမီတာ ပရမ်းပတာ)။
4. `device_lab_contingency.md` တွင် အရေးပေါ်အခြေအနေမှတ်တမ်းပုံစံကို အပ်ဒိတ်လုပ်ပါ။
   တူးသည့်ရက်စွဲအတွက် နေရာယူထားသောအတန်း။

### 2. ကျရှုံးမှုအခြေအနေများကို ပုံဖော်ပါ။

1. ဓာတ်ခွဲခန်းအတွင်း အဓိကလမ်းကြော (`pixel8pro-strongbox-a`) ကို ပိတ်ပါ သို့မဟုတ် ဖျက်သိမ်းပါ
   အချိန်ဇယားဆွဲပြီး ကြိုတင်စာရင်းသွင်းမှုအား "drill" အဖြစ် tag လုပ်ပါ။
2. PagerDuty (`AND6-device-lab` ဝန်ဆောင်မှု) တွင် ပုံသဏ္ဍန်ပြတ်တောက်မှုသတိပေးချက်ကို အစပျိုးပါ
   အထောက်အထားအစုအဝေးအတွက် အကြောင်းကြားချက်တင်ပို့မှုကို ဖမ်းယူပါ။
3. ပုံမှန်လမ်းသွားနေသော Buildkite အလုပ်များကို မှတ်သားပါ။
   တူး ID ဖြင့် (`android-strongbox-attestation`, `android-ci-e2e`)။

### 3. ရှုံးနိမ့်သူကို ကွပ်မျက်ခြင်း။1. နောက်ပြန်လှည့်လာသော Pixel7 လမ်းသွားကို အဓိက CI ပစ်မှတ်သို့ မြှင့်တင်ပြီး အချိန်ဇယားဆွဲပါ။
   ၎င်းနှင့် ပတ် သက်ပြီး စီစဉ်ထားသော အလုပ်များ။
2. အတွက် `firebase-burst` လမ်းကြောမှတစ်ဆင့် Firebase Test Lab burst suite ကို အစပျိုးပါ
   StrongBox လွှမ်းခြုံမှုကို မျှဝေနေစဉ် လက်လီ-ပိုက်ဆံအိတ် မီးခိုးစစ်ဆေးမှုများ
   လမ်းသွား။ စာရင်းစစ်အတွက် လက်မှတ်တွင် CLI တောင်းခံမှု (သို့မဟုတ် ကွန်ဆိုးလ်တင်ပို့ခြင်း) ကို ဖမ်းယူပါ။
   တန်းတူညီမျှမှု။
3. တိုတောင်းသောသက်သေပြချက်တစ်ခုအတွက် ပြင်ပ StrongBox ဓာတ်ခွဲခန်းထိန်းသိမ်းသူကို ချိတ်ဆက်ပါ။
   အောက်တွင်ဖော်ပြထားသည့်အတိုင်း ဆက်သွယ်ရန် အသိအမှတ်ပြုမှုမှတ်တမ်း။
4. Buildkite လည်ပတ် ID များ၊ Firebase အလုပ် URL များနှင့် ထိန်းသိမ်းသူမှတ်တမ်းများအားလုံးကို မှတ်တမ်းတင်ပါ
   `_android-device-lab` လက်မှတ်နှင့် အထောက်အထားအစုအဝေးကို ဖော်ပြသည်။

### 4. အတည်ပြုခြင်းနှင့် ပြန်လှည့်ခြင်း။

1. အခြေခံစာကြောင်းနှင့် သက်သေအထောက်အထား/CI အလုပ်ချိန်များကို နှိုင်းယှဉ်ပါ။ flag deltas > 10%
   Hardware Lab Lead ၊
2. ပင်မလမ်းကြောင်းကို ပြန်လည်ရယူပြီး စွမ်းရည်လျှပ်တစ်ပြက်ရိုက်ချက်နှင့် အဆင်သင့်ဖြစ်မှုကို အပ်ဒိတ်လုပ်ပါ။
   matrix သည် validation ပြီးသည်နှင့်။
3. နောက်ဆုံးအတန်းကို `device_lab_contingency.md` တွင် အစပျိုး၊ လုပ်ဆောင်ချက်များ၊
   နှင့်နောက်ဆက်တွဲ။
4. `docs/source/compliance/android/evidence_log.csv` ကို အပ်ဒိတ်လုပ်ပါ-
   အစုအဝေးလမ်းကြောင်း၊ SHA-256 မန်နီးဖက်စ်၊ Buildkite လည်ပတ်သော ID များ၊ PagerDuty ထုတ်ယူသည့် hash နှင့်
   ဝေဖန်သုံးသပ်သူ ဆိုင်းဘုတ်
