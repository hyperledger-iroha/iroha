# מדריך הפעלה לסוכני אוטומציה

דף זה מסכם את הגדרות המגן התפעוליות לכל סוכן אוטומציה העובד בתוך סביבת
Hyperledger Iroha. הוא משקף את ההנחיות הקנוניות ב‑`AGENTS.md` ואת ההפניות
ב‑roadmap כך ששינויים בבנייה, בתיעוד ובטלמטריה ייראו זהים בין אם נוצרו
בידי אדם ובין אם בידי תורם אוטומטי.

כל משימה צפויה לנחות עם קוד דטרמיניסטי לצד תיעוד, בדיקות וראיות תפעוליות
תואמות. התייחסו לסעיפים הבאים כאל מקור מהיר לפני נגיעה בפריטי `roadmap.md`
או מענה לשאלות התנהגות.

## פקודות התחלה מהירה

| פעולה | פקודה |
|--------|---------|
| בניית ה‑workspace | `cargo build --workspace` |
| הרצת כל הבדיקות | `cargo test --workspace` *(לרוב נמשך שעות רבות)* |
| הרצת clippy עם אזהרות כ‑deny‑by‑default | `cargo clippy --workspace --all-targets -- -D warnings` |
| עיצוב קוד Rust | `cargo fmt --all` *(edition 2024)* |
| בדיקת crate יחיד | `cargo test -p <crate>` |
| הרצת בדיקה בודדת | `cargo test -p <crate> <test_name> -- --nocapture` |
| בדיקות Swift SDK | מתוך `IrohaSwift/`, להריץ `swift test` |

## עקרונות זרימת עבודה

- קראו את נתיבי הקוד הרלוונטיים לפני מענה לשאלות או שינוי לוגיקה.
- פרקו פריטי roadmap גדולים למשימות ניתנות לביצוע; לעולם אל תדחו עבודה לחלוטין.
- הישארו במסגרת חברי ה‑workspace הקיים, השתמשו מחדש בקרייטים פנימיים, ו**אל** תשנו
  את `Cargo.lock` אלא אם ניתנה הוראה מפורשת.
- השתמשו ב‑feature flags וב‑capability toggles רק כאשר נדרש ע"י מאיצי חומרה;
  שמרו על נפילות דטרמיניסטיות זמינות בכל פלטפורמה.
- עדכנו תיעוד והפניות Markdown לצד כל שינוי פונקציונלי כדי שהתיעוד ישקף תמיד
  את ההתנהגות הנוכחית.
- הוסיפו לפחות בדיקת יחידה אחת לכל פונקציה חדשה או ששונתה. העדיפו מודולי
  `#[cfg(test)]` פנימיים או תיקיית `tests/` של הקרייט בהתאם להיקף.
- לאחר סיום עבודה, עדכנו את `status.md` עם תקציר קצר והפניות לקבצים רלוונטיים;
  השאירו את `roadmap.md` ממוקד בפריטים שעדיין דורשים עבודה.

## כללי יישום

### סריאליזציה ומודלי נתונים
- השתמשו בקודק Norito בכל מקום (בינארי דרך `norito::{Encode, Decode}`,
  JSON דרך `norito::json::*`). אל תוסיפו שימוש ישיר ב‑serde/`serde_json`.
- מטעני Norito חייבים לפרסם את פריסתן (בייט גרסה או דגלי header),
  ופורמטים חדשים דורשים עדכוני תיעוד תואמים (למשל, `norito.md`, `docs/source/da/*.md`).
- נתוני genesis, manifests ומטעני רשת צריכים להישאר דטרמיניסטיים כך ששני peers עם
  אותם קלטים יפיקו hashes זהים.

### תצורה והתנהגות זמן‑ריצה
- העדיפו knobs ב‑`crates/iroha_config` על פני משתני סביבה חדשים.
  העבירו ערכים במפורש דרך constructors או dependency injection.
- אל תאפשרו gating של syscalls או opcodes ב‑IVM — ABI v1 נשלח בכל בנייה.
- כאשר מוסיפים אפשרויות תצורה, עדכנו defaults, docs ותבניות קשורות
  (`peer.template.toml`, `docs/source/configuration*.md`, וכו').

### ABI, Syscalls וסוגי מצביעים
- התייחסו למדיניות ABI כבלתי מותנית. הוספה/הסרה של syscalls או סוגי מצביעים
  מחייבת עדכון:
  - `ivm::syscalls::abi_syscall_list` ו‑`crates/ivm/tests/abi_syscall_list_golden.rs`
  - `ivm::pointer_abi::PointerType` וכן בדיקות golden
  - `crates/ivm/tests/abi_hash_versions.rs` בכל פעם ש‑ABI hash משתנה
- syscalls לא מוכרים חייבים למפות ל‑`VMError::UnknownSyscall`, ומניפסטים חייבים
  לשמור על בדיקות admission לשוויון `abi_hash` חתום.

### האצת חומרה ודטרמיניזם
- פרימיטיבים קריפטוגרפיים חדשים או חישובים כבדים חייבים לכלול נתיבי האצה
  (METAL/NEON/SIMD/CUDA) תוך שמירה על fallback דטרמיניסטי.
- הימנעו מהפחתות מקביליות לא דטרמיניסטיות; העדיפות היא פלט זהה בכל peer
  גם כאשר החומרה שונה.
- שמרו על reproducibility של fixtures של Norito ו‑FASTPQ כדי ש‑SRE יוכלו לבקר
  טלמטריה לאורך הצי.

### תיעוד וראיות
- שיקוף כל שינוי תיעוד פומבי בפורטל (`docs/portal/...`) כאשר ישים, כדי שהאתר
  יישאר מסונכרן עם מקורות ה‑Markdown.
- כאשר מציגים זרימות חדשות, הוסיפו runbooks, הערות ממשל או checklists
  שמסבירים כיצד לבצע rehearsal, rollback ואיסוף ראיות.
- בעת תרגום חומר לאכדית, ספקו תרגום סמנטי בכתב יתדות ולא תעתיק פונטי.

### ציפיות בדיקה וכלים
- הריצו מקומית את מערכי הבדיקות הרלוונטיים (`cargo test`, `swift test`,
  harnesses של integration) ותעדו את הפקודות בסעיף Testing של ה‑PR.
- שמרו על סקריפטי CI guard (`ci/*.sh`) ודשבורדים מסונכרנים עם טלמטריה חדשה.
- עבור proc‑macros, שלבו בדיקות יחידה עם בדיקות UI של `trybuild` כדי לנעול אבחונים.

## רשימת בדיקה לשחרור

1. הקוד מתקמפל ו‑`cargo fmt` לא יצר דיפ.
2. התיעוד המעודכן (Markdown של ה‑workspace + מראות פורטל) מתאר את ההתנהגות,
   דגלי CLI או knobs תצורה החדשים.
3. הבדיקות מכסות כל נתיב קוד חדש ונכשלות דטרמיניסטית כאשר יש רגרסיה.
4. טלמטריה, דשבורדים והגדרות התראה מתייחסים למדדים או קודי שגיאה חדשים.
5. `status.md` כולל תקציר קצר עם הפניות לקבצים רלוונטיים וסעיף roadmap.

עמידה ברשימה זו משאירה את ביצוע ה‑roadmap ניתן לביקורת ומבטיחה שכל סוכן
תורם ראיות שאפשר לסמוך עליהן.
