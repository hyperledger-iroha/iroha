# Overview

SNNet-8 introduces a privacy-preserving telemetry surface for relay operators.
Before SNNet-5 can flip to default-on, governance requires published evidence
that the metrics pipeline:

1. Applies the documented differential privacy (DP) parameters.
2. Preserves contributor anonymity across the supported guard layouts.
3. Emits suppression signals when the minimum contributor threshold is not met.

This report tracks the artefacts necessary to satisfy that requirement.

# Differential Privacy Framework

The telemetry pipeline uses event-level differential privacy with a Laplace
mechanism applied to per-minute bucket counts. Each scenario enforces:

- **Sensitivity** (`Δf`) = 1.0, because every contributor can add or remove at
  most a single count per bucket once aggregation normalises relay events.
- **Epsilon** (`ε`) — calibrated per relay role to balance fidelity and
  anonymity.
- **Delta** (`δ`) — fixed to the governance-approved bound (`≤1e-5`) so the
  probability of privacy loss exceeding `ε` remains negligible.

Laplace noise is parameterised as `Lap(0, b)` where `b = Δf / ε`. The script
rounds scales and derived standard deviations to four decimal places so
artefacts remain deterministic across Python versions.

# Calibration Inputs

The harness (`scripts/telemetry/run_privacy_dp.py`) covers the three guard
cohorts required for SNNet-5: entry, middle, and exit relays. Contributor
counts are seeded from real deployment telemetry but rounded to envelope values
so calibration does not leak production statistics.

```text
Scenario seeds
--------------
entry-baseline      contributors = 18, ε = 1.20, δ = 5e-6
middle-congested    contributors = 12, ε = 0.95, δ = 1e-5
exit-low-sample     contributors =  9, ε = 0.85, δ = 1e-5
suppression_threshold = 12 contributors
```

The notebook ingests the deterministic summary JSON and recomputes the values
above to verify the JSON was produced by the expected harness revision.

# Calibration Results

| Scenario | Role | Contributors | ε | δ | Laplace scale (`b`) | Noise σ | Suppressed |
|----------|------|--------------|---|---|---------------------|---------|------------|
| entry-baseline | entry | 18 | 1.20 | 5×10⁻⁶ | 0.8333 | 1.1785 | No |
| middle-congested | middle | 12 | 0.95 | 1×10⁻⁵ | 1.0526 | 1.4886 | No |
| exit-low-sample | exit | 9 | 0.85 | 1×10⁻⁵ | 1.1765 | 1.6638 | **Yes** |

Key observations:

- The entry and middle scenarios remain below the governance epsilon ceiling of
  `ε ≤ 1.2`, keeping the Laplace scale < 1.1 while avoiding unnecessary
  suppression.
- The exit scenario intentionally falls below the contributor threshold. The
  pipeline suppresses the bucket entirely (`suppressed = true`), matching the
  product requirement that exit relays never expose low-sample lanes.
- The derived standard deviation aligns with `√2 · b`; the notebook asserts the
  relationship to guard against code regressions.

# Suppression Behaviour

`artifacts/soranet_privacy_dp/suppression_matrix.csv` records contributor counts
from 6–20 so governance can visualise the step function around the suppression
threshold. The orchestrator privacy collector performs the same comparison at
runtime; the Prometheus rule `soranet_privacy_rules.yml` watches the
`sorafs_privacy_suppressed_buckets_total` counter and raises a pager alert if
suppression persists for more than three consecutive minutes.

# Validation & Tooling

1. `python3 scripts/telemetry/run_privacy_dp.py`
   - Regenerates `summary.json` and `suppression_matrix.csv`.
   - Prints the output path so operators can archive the artefacts.
2. `scripts/telemetry/run_privacy_dp_notebook.sh`
   - Invokes the script above.
   - Executes the notebook headlessly via `papermill` (preferred) or
     `jupyter nbconvert --execute`.
   - Emits the rendered notebook into `artifacts/soranet_privacy_dp/`.
3. `notebooks/soranet_privacy_dp.ipynb`
   - Verifies scenario metadata hashes match `summary.json`.
   - Recomputes ε/δ bounds and Laplace scales.
   - Plots suppression probability vs. contributors, mirroring the CSV matrix.

CI jobs (`scripts/run_release_pipeline.py`) call the wrapper script so every
release bundle carries fresh DP artefacts. Local reproduction simply requires a
Python 3.10+ interpreter; no extra dependencies are needed.

# Governance Reporting

The weekly governance digest (`docs/source/status/soranet_privacy_dp_digest.md`)
summarises the latest ε bounds and any suppression anomalies discovered during
calibration runs. Copy the rendered notebook and the JSON summary into the
digest when preparing council packets so reviewers can diff parameter changes.

# Reproducibility Plan

1. **Synthetic workload generation** – produce deterministic workloads using the
   existing `PrivacyAggregator` harness. Each scenario should fix the random
   seed (Blake3) and export the per-bucket contributions as NDJSON so the same
   input can be replayed across observers.
2. **Collector replay** – feed the workloads through the orchestrator privacy
   collector (`ingest_privacy_payload`) to confirm Prio shares and relay events
   collapse into the same aggregated buckets.
3. **Notebook outputs** – publish the notebook under `notebooks/` with
   executable cells that:
   - compute ε/δ bounds for each workload;
   - chart suppression rates vs. contributor counts;
   - export tables to `artifacts/soranet_privacy_dp/`.
4. **CI hook** – `.github/workflows/release-pipeline.yml` invokes
   `scripts/telemetry/run_privacy_dp_notebook.sh` to execute the notebook
   non-interactively (prefers `papermill`, falls back to `jupyter nbconvert`)
   so promotions regenerate artefacts automatically.
5. **Governance digest** – append the calibration summary to
   `docs/source/status/soranet_privacy_dp_digest.md` for council review.

# Acceptance Criteria

- ε-bounds for the default bucket configuration (`bucket_secs = 60`,
  `min_contributors = 12`) are ≤ the value approved by SNNet governance.
- Suppression engages within the expected window when contributors fall below
  `min_contributors` for two consecutive buckets.
- Prometheus alerts (`soranet_privacy_rules.yml`) trip when suppression persists
  beyond the defined SLA.
- Artefacts are reproducible: re-running the notebook with the supplied seeds
  produces byte-for-byte identical outputs.
