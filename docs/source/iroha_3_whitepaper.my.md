#Iroha v3.0 (Nexus အစမ်းကြည့်)

ဤစာရွက်စာတမ်းသည် ရှေ့သွားကြည့်နိုင်သော Hyperledger Iroha v3 ဗိသုကာကို ဖမ်းယူထားပြီး လမ်းသွားများစွာကို အာရုံစိုက်သည်
ပိုက်လိုင်း၊ Nexus ဒေတာနေရာများနှင့် ပိုင်ဆိုင်မှုလဲလှယ်ရေးကိရိယာအစုံ (AXT)။ ၎င်းသည် Iroha v2 အဖြူရောင်စက္ကူဖြင့် ဖြည့်စွက်ထားသည်။
တက်ကြွစွာ ဖွံ့ဖြိုးတိုးတက်လာမည့် စွမ်းရည်များကို ဖော်ပြခြင်း။

---

## 1. ခြုံငုံသုံးသပ်ချက်

Iroha v3 သည် v2 ၏ အဆုံးအဖြတ်အခြေခံအုတ်မြစ်ကို အလျားလိုက် အတိုင်းအတာနှင့် ပိုမိုကြွယ်ဝသော ဒိုမိန်းဖြတ်ကျော်မှုဖြင့် တိုးချဲ့သည်
အလုပ်အသွားအလာများ။ **Nexus** ဟုအမည်ပေးထားသည့် ထုတ်ဝေမှုတွင် မိတ်ဆက်ပေးသည်-

- **SORA Nexus** ဟု ခေါ်သော တစ်ကမ္ဘာလုံး မျှဝေထားသော ကွန်ရက်တစ်ခု။ Iroha v3 သက်တူရွယ်တူများအားလုံး ဤ Universal တွင် ပါဝင်ပါသည်။
  သီးခြားခွဲဝေချထားမှုများကို လုပ်ဆောင်ခြင်းထက် စာရင်းဇယား။ အဖွဲ့အစည်းများသည် ၎င်းတို့၏ ကိုယ်ပိုင်ဒေတာနေရာများကို စာရင်းသွင်းခြင်းဖြင့် ပူးပေါင်းပါဝင်ခြင်း၊
  ဘုံလယ်ဂျာတွင် တွယ်ကပ်နေချိန်တွင် မူဝါဒနှင့် ကိုယ်ရေးကိုယ်တာအတွက် သီးခြားတည်ရှိနေပါသည်။
- မျှဝေထားသောကုဒ်ဘေ့စ်- တူညီသောသိုလှောင်မှုသည် Iroha v2 (ကိုယ်တိုင်လုပ်ဆောင်ထားသောကွန်ရက်များ) နှင့် Iroha v3 (SORA Nexus) နှစ်ခုလုံးကို တည်ဆောက်သည်။
  ဖွဲ့စည်းမှုပုံစံသည် ပစ်မှတ်မုဒ်ကို ရွေးချယ်ပေးသောကြောင့် အော်ပရေတာများသည် ဆော့ဖ်ဝဲလ်မပြောင်းဘဲ Nexus အင်္ဂါရပ်များကို လက်ခံကျင့်သုံးနိုင်သည်
  stacks Iroha Virtual Machine (IVM) သည် ထုတ်ဝေမှုနှစ်ခုစလုံးတွင် တူညီသောကြောင့် Kotodama စာချုပ်များနှင့် bytecode
  အနုပညာပစ္စည်းများသည် ကိုယ်တိုင်လက်ခံထားသောကွန်ရက်များနှင့် ကမ္ဘာလုံးဆိုင်ရာ Nexus လယ်ဂျာတွင် ချောမွေ့စွာလည်ပတ်သည်။
- လွတ်လပ်သောအလုပ်ဝန်များကိုအပြိုင်လုပ်ဆောင်ရန် Multi-lane block ထုတ်လုပ်ခြင်း။
- ကွင်းဆက်ကျောက်ဆူးများမှတစ်ဆင့် ကျန်ရှိနေချိန်တွင် လုပ်ဆောင်မှုပတ်ဝန်းကျင်များကို သီးခြားခွဲထုတ်သည့် ဒေတာနေရာလွတ်များ (DS)။
- အနုမြူဗုံး၊ အာကာသတန်ဖိုးလွှဲပြောင်းမှုများနှင့် စာချုပ်ထိန်းချုပ်ထားသော လဲလှယ်မှုများအတွက် ပိုင်ဆိုင်မှုလဲလှယ်ရေးကိရိယာအစုံ (AXT)။
- Reliable Broadcast Commit (RBC) လမ်းကြောများ၊ အဆုံးအဖြတ်သတ်မှတ်ထားသော နောက်ဆုံးရက်များနှင့် အထောက်အထားများမှတဆင့် ပိုမိုကောင်းမွန်သော ယုံကြည်စိတ်ချရမှု
  နမူနာဘတ်ဂျက်များ။

ဤအင်္ဂါရပ်များသည် တက်ကြွစွာ ဖွံ့ဖြိုးတိုးတက်မှုအောက်တွင် ရှိနေပါသည်။ API များနှင့် အပြင်အဆင်များသည် v3 ယေဘူယျမတိုင်မီတွင် ပြောင်းလဲလာနိုင်သည်။
ရရှိနိုင်မှုမှတ်တိုင်။ `nexus.md`၊ `nexus_transition_notes.md` နှင့် `new_pipeline.md` ကို ကိုးကားပါ
အင်ဂျင်နီယာအဆင့်အသေးစိတ်။

## 2. Multi-lane ဗိသုကာ

- **စီစဉ်သူ-** Nexus ဇယားကွက်များသည် data space identifiers များအပေါ် အခြေခံ၍ လမ်းကြောများအဖြစ် အလုပ်လုပ်သည် ။
  ပေါင်းစပ်နိုင်စွမ်းအုပ်စုများ။ လမ်းကြောင်းများသည် အတွင်းတွင် အဆုံးစွန်သော အဆုံးအဖြတ်ပေးသော အမိန့်အာမခံချက်များကို ထိန်းသိမ်းထားစဉ် အပြိုင်လုပ်ဆောင်သည်။
  လမ်းသွားတိုင်း။
- **လမ်းကြောအဖွဲ့များ-** ဆက်စပ်ဒေတာနေရာလွတ်များသည် `LaneGroupId` ကို မျှဝေကာ အလုပ်အသွားအလာများအတွက် ညှိနှိုင်းလုပ်ဆောင်မှုကို ဖွင့်ပေးသည်။
  များစွာသော အစိတ်အပိုင်းများ (ဥပမာ၊ CBDC DS နှင့် ၎င်း၏ ငွေပေးချေမှု dApp DS) ကို ပိုင်းခြားပါ။
- ** Deadlines :** အာမခံရန် လမ်းကြောတစ်ခုစီသည် သတ်မှတ်ထားသော နောက်ဆုံးရက်များ (ပိတ်ဆို့ခြင်း၊ အထောက်အထား၊ ဒေတာရရှိနိုင်မှု) ကို ခြေရာခံသည်
  တိုးတက်မှုနှင့် အရင်းအမြစ်အသုံးပြုမှုကို ကန့်သတ်ထားသည်။
- **Telemetry-** လမ်းကြောအဆင့် မက်ထရစ်များသည် ဖြတ်သန်းမှု၊ တန်းစီခြင်း၏ အနက်၊ နောက်ဆုံးရက်ကို ချိုးဖောက်မှုများ၊ လှိုင်းနှုန်းအသုံးပြုမှုကို ထင်ရှားစေသည်။
  ဒက်ရှ်ဘုတ်များကို အချိန်ဇယားဆွဲသူနှင့် လိုက်လျောညီထွေရှိစေရန် CI Script များက ဤကောင်တာများ ရှိနေကြောင်း အခိုင်အမာဆိုသည်။

## 3. ဒေတာနေရာလွတ်များ (Nexus)- **အထီးကျန်ခြင်း-** ဒေတာနေရာတစ်ခုစီသည် ၎င်း၏ကိုယ်ပိုင်ဆန္ဒလမ်းကြောင်း၊ ကမ္ဘာ့နိုင်ငံတော်အပိုင်းနှင့် Kura သိုလှောင်မှုကို ထိန်းသိမ်းထားသည်။ ဒီ
  ကျောက်ဆူးများမှတဆင့် ကမ္ဘာလုံးဆိုင်ရာ SORA Nexus ကို ထိန်းကျောင်းထားစဉ် သီးသန့်ဒိုမိန်းများကို ပံ့ပိုးပေးသည်။
- **ကျောက်ဆူးများ-** ပုံမှန် ကတိကဝတ်များသည် DS အခြေအနေကို အကျဉ်းချုပ်ဖော်ပြသည့် ကျောက်ဆူးများ ( Merkle အမြစ်များ၊ အထောက်အထားများ၊
  ကတိကဝတ်များ) နှင့် ၎င်းတို့ကို စာရင်းစစ်နိုင်စေရန် ဂလိုဘယ်လမ်းကြောင်းသို့ ထုတ်ဝေပါ။
- **လမ်းကြောအုပ်စုများနှင့် ပေါင်းစပ်နိုင်မှု-** ဒေတာနေရာများသည် အက်တမ် AXT ကိုခွင့်ပြုသည့် ပေါင်းစပ်နိုင်မှုအုပ်စုများကို ကြေညာနိုင်သည်
  အတည်ပြုပါဝင်သူများအကြား ငွေပေးငွေယူများ။ အုပ်ချုပ်မှုစနစ်သည် အဖွဲ့ဝင်အပြောင်းအလဲများနှင့် အသက်ဝင်သည့်အချိန်များကို ထိန်းချုပ်သည်။
- **Erasure-coded သိုလှောင်မှု-** Kura နှင့် WSV လျှပ်တစ်ပြက်များသည် ဒေတာကိုစကေးပြုလုပ်ရန်အတွက် ဖျက်ပစ်သောကုဒ်ဘောင်သတ်မှတ်ချက်များ `(k, m)` ကို လက်ခံကျင့်သုံးသည်
  အဆုံးအဖြတ်မပေးဘဲ ရရှိနိုင်မှု။ ပြန်လည်ရယူခြင်းလုပ်ရိုးလုပ်စဉ်များသည် ပျောက်ဆုံးနေသောအပိုင်းအစများကို အဆုံးအဖြတ်အတိုင်း ပြန်လည်ရယူသည်။

## 4. Asset Exchange Toolkit (AXT)

- **ဖော်ပြချက်နှင့် စည်းနှောင်ခြင်း-** ဖောက်သည်များသည် အဆုံးအဖြတ်ပေးသော AXT ဖော်ပြချက်များကို ဖန်တီးသည်။ `axt_binding` hash ကျောက်ဆူးများ
  စာအိတ်တစ်လုံးချင်းစီအတွက် ဖော်ပြချက်များအား ပြန်လည်ပြသခြင်းကို တားဆီးကာ အများဆန္ဒပါဝင်သူများမှ byte-for- မှန်ကန်ကြောင်း သေချာစေခြင်း၊
  byte Norito payloads
- **Syscalls-** IVM သည် `AXT_BEGIN`၊ `AXT_TOUCH` နှင့် `AXT_COMMIT` syscall များကို ဖော်ထုတ်ပေးပါသည်။ စာချုပ်များသည် ၎င်းတို့ကို ကြေငြာသည်။
  ဒေတာနေရာတစ်ခုစီတွင် အစုံဖတ်/ရေးခြင်းဖြင့် အိမ်ရှင်အား လမ်းကြောများတစ်လျှောက် အနုမြူဓာတ်ကို တွန်းအားပေးရန် ခွင့်ပြုသည်။
- **လက်ကိုင်များနှင့် ခေတ်များ-** ပိုက်ဆံအိတ်များသည် `(dataspace_id, epoch_id, sub_nonce)` နှင့် ချည်နှောင်ထားသော စွမ်းဆောင်ရည်လက်ကိုင်များကို ရရှိသည်။
  ကန့်သတ်ချက်များရှိနေသောအခါ တစ်ပြိုင်နက်တည်း ပဋိပက္ခကို အဆုံးအဖြတ်ပေးကာ စည်းမျဉ်းစည်းကမ်း `AxtTrap` ကုဒ်များကို ပြန်ပေးသည်
  ချိုးဖောက်ခဲ့သည်။
- ** မူဝါဒပြဋ္ဌာန်းခြင်း-** Core host များသည် ယခုအခါ WSV တွင် Space Directory မှဖော်ပြသော AXT မူဝါဒလျှပ်တစ်ပြက်ရိုက်ချက်များအား ရယူရရှိပါသည်၊
  manifest root၊ ပစ်မှတ်လမ်းကြော၊ activation-era၊ sub-nonce နှင့် သက်တမ်းကုန်စစ်ဆေးမှုများ (`current_slot >= expiry_slot`
  အနည်းအကျဉ်း စမ်းသပ်မှုတွင်ပင် ပျက်ကျသည်။ မူဝါဒများကို dataspace id ဖြင့် သော့ခတ်ထားပြီး လမ်းသွားကတ်တလောက်မှ တည်ဆောက်ထားသည်။
  လက်ကိုင်များသည် ၎င်းတို့၏ ထုတ်ပေးသည့်လမ်းကြောင်းမှ လွတ်ကင်းနိုင်သည် သို့မဟုတ် ဖောက်ပြန်သည့် သရုပ်ကို အသုံးပြု၍မရပါ။
  - ငြင်းပယ်ရခြင်းအကြောင်းရင်းများသည် အဆုံးအဖြတ်ဖြစ်သည်- အမည်မသိဒေတာနေရာလွတ်၊ ထင်ရှားသောအမြစ်မတူညီမှု၊ ပစ်မှတ်လမ်းကြောမတူညီမှု၊
    manifest activation အောက်ရှိ handle_era၊ မူဝါဒကြမ်းပြင်အောက်ရှိ sub_nonce၊ သက်တမ်းကုန်နေသော လက်ကိုင်၊ ပျောက်နေသော အထိအတွေ့
    ဒေတာနေရာလွတ်ကို ကိုင်တွယ်ရန် သို့မဟုတ် လိုအပ်သည့်အခါ အထောက်အထား ပျောက်ဆုံးနေပါသည်။
- **အထောက်အထားများနှင့် သတ်မှတ်ရက်များ-** အသက်ဝင်သောဝင်းဒိုး Δ အတွင်းတွင်၊ သက်သေအထောက်အထားများ စုဆောင်းခြင်း၊ ဒေတာရရှိနိုင်မှုနမူနာများ၊
  ထင်ရှားသည်။ သတ်မှတ်ရက်များ ပြည့်မီရန် ပျက်ကွက်ခြင်းသည် ဖောက်သည် ထပ်မံကြိုးစားခြင်းအတွက် လမ်းညွှန်ချက်ဖြင့် AXT အား အဆုံးအဖြတ်ပေးသည်။
- **အုပ်ချုပ်မှုပေါင်းစပ်ခြင်း-** မူဝါဒမော်ဂျူးများသည် AXT တွင် မည်သည့်ဒေတာနေရာများ ပါဝင်နိုင်သည်ကို သတ်မှတ်သည်၊ နှုန်းကန့်သတ်ချက်၊
  စာရင်းစစ်နှင့် လိုက်လျောညီထွေဖြစ်စေသော ကတိကဝတ်များ၊ ပျက်ပြယ်စေသော အရာများနှင့် ဖြစ်ရပ်မှတ်တမ်းများကို ဖမ်းယူဖော်ပြသည်။

## 5. Reliable Broadcast Commit (RBC) လမ်းကြောင်းများ- **လမ်းသွား-သတ်သတ်မှတ်မှတ် DA-** RBC လမ်းကြောများသည် မှန်လမ်းသွားအုပ်စုများဖြစ်ပြီး၊ လမ်းသွားများစွာသော ပိုက်လိုင်းတစ်ခုစီတွင် သီးသန့်ဒေတာရှိကြောင်း သေချာစေသည်
  ရရှိနိုင်မှုအာမခံချက်။
- **နမူနာဘတ်ဂျက်များ-** အထောက်အထားများကို အတည်ပြုရန်အတွက် အထောက်အထားများကို အတည်ပြုရန် သတ်မှတ်ထားသော နမူနာ နမူနာစည်းမျဉ်းများ (`q_in_slot_per_ds`) ကို လိုက်နာပါသည်။
  ဗဟိုညှိနှိုင်းမှုမရှိဘဲ သက်သေခံပစ္စည်းများ၊
- **Backpressure ထိုးထွင်းသိမြင်မှု-** Sumeragi pacemaker ဖြစ်ရပ်များသည် ရပ်နေသောလမ်းကြောင်းများကိုရှာဖွေဖော်ထုတ်ရန် RBC စာရင်းဇယားများနှင့် ဆက်စပ်နေသည်
  (`scripts/sumeragi_backpressure_log_scraper.py` ကိုကြည့်ပါ)။

## 6. လည်ပတ်မှုနှင့် ရွှေ့ပြောင်းခြင်း။

- **အကူးအပြောင်းအစီအစဉ်-** `nexus_transition_notes.md` သည် တစ်လမ်းသွား (Iroha v2) သို့ အဆင့်ဆင့်ပြောင်းရွေ့ခြင်းကို အကြမ်းဖျင်းဖော်ပြထားသည်
  တယ်လီမီတာ အဆင့်သတ်မှတ်ခြင်း၊ ပြင်ဆင်သတ်မှတ်ခြင်း နှင့် ဥပါဒ်မွမ်းမံခြင်းများ အပါအဝင် ဘက်စုံလမ်းသွား (Iroha v3)။
- **Universal network:** SORA Nexus ရွယ်တူများသည် ဘုံဥပါဒ်နှင့် အုပ်ချုပ်မှုအပိုင်းကို လုပ်ဆောင်သည်။ အော်ပရေတာအသစ်များ
  သီးခြားကွန်ရက်များကိုဖွင့်ခြင်းအစား ဒေတာနေရာလွတ် (DS) ဖန်တီးခြင်းနှင့် Nexus ဝင်ခွင့်မူဝါဒများကို ကျေနပ်စေသည်။
- **ဖွဲ့စည်းပုံ-** အသစ်ဖွဲ့စည်းမှု ခလုတ်များသည် လမ်းသွားဘတ်ဂျက်များ၊ သက်သေသတ်မှတ်ရက်များ၊ AXT ခွဲတမ်းများနှင့် ဒေတာ-အာကာသ မက်တာဒေတာကို အကျုံးဝင်ပါသည်။
  အော်ပရေတာများသည် Nexus မုဒ်သို့ ရွေးချယ်မပြီးမချင်း ပုံသေများသည် ရှေးရိုးဆန်သည်။
- **စမ်းသပ်ခြင်း-** ရွှေစမ်းသပ်မှုများသည် AXT ဖော်ပြချက်များ၊ လမ်းကြောဖော်ပြချက်များနှင့် syscall စာရင်းများကို ဖမ်းယူသည်။ ပေါင်းစပ်စစ်ဆေးမှုများ
  (`integration_tests/tests/repo.rs`, `crates/ivm/tests/axt_host_flow.rs`) လေ့ကျင့်ခန်း အဆုံးမှ အဆုံးသို့ စီးဆင်းသည်။
- **Tooling:** `kagami` သည် Nexus-aware genesis generation ကို ရရှိပြီး dashboard scripts များသည် lane throughput ကို validate၊
  အထောက်အထားဘတ်ဂျက်များနှင့် RBC ကျန်းမာရေး။

## 7. လမ်းပြမြေပုံ
