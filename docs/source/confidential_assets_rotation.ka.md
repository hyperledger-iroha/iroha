# კონფიდენციალური აქტივების როტაციის Runbook

ეს სახელმძღვანელო განმარტავს, თუ როგორ გეგმავთ ოპერატორებს და ახორციელებენ კონფიდენციალურ აქტივებს
როტაციები (პარამეტრების ნაკრები, გადამოწმების გასაღებები და პოლიტიკის გადასვლები) ხოლო
იმის უზრუნველყოფა, რომ საფულეები, Torii კლიენტები და მემპულის მცველები დარჩება განმსაზღვრელი.

## სასიცოცხლო ციკლი და სტატუსები

კონფიდენციალური პარამეტრების ნაკრები (`PoseidonParams`, `PedersenParams`, დამადასტურებელი გასაღებები)
გისოსი და დამხმარე, რომელიც გამოიყენება მოცემულ სიმაღლეზე ეფექტური სტატუსის გამოსატანად
`crates/iroha_core/src/state.rs:7540`–`7561`. გაშვების დროის დამხმარეები მოლოდინშია
გადასვლები სამიზნე სიმაღლის მიღწევისთანავე და ჟურნალის წარუმატებლობა მოგვიანებით
გადაცემები (`crates/iroha_core/src/state.rs:6725`–`6765`).

აქტივების პოლიტიკა ჩაშენებულია
`pending_transition { transition_id, new_mode, effective_height, conversion_window }`
ასე რომ, მმართველობას შეუძლია განახლებების დაგეგმვა
`ScheduleConfidentialPolicyTransition` და საჭიროების შემთხვევაში გააუქმეთ ისინი. იხ
`crates/iroha_data_model/src/asset/definition.rs:320` და Torii DTO სარკეები
(`crates/iroha_torii/src/routing.rs:1539`–`1580`).

## როტაციის სამუშაო პროცესი

1. ** ახალი პარამეტრების პაკეტების გამოქვეყნება. ** ოპერატორები წარადგენენ
   `PublishPedersenParams`/`PublishPoseidonParams` ინსტრუქციები (CLI
   `iroha app zk params publish ...`) ახალი გენერატორის ნაკრების დადგმა მეტამონაცემებით,
   აქტივაციის/გაუქმების ფანჯრები და სტატუსის მარკერები. შემსრულებელი უარყოფს
   ID-ების დუბლიკატი, არ მზარდი ვერსიები ან ცუდი სტატუსის გადასვლები
   `crates/iroha_core/src/smartcontracts/isi/world.rs:2499`–`2635` და
   რეესტრის ტესტები მოიცავს წარუმატებლობის რეჟიმებს (`crates/iroha_core/tests/confidential_params_registry.rs:93`–`226`).
2. **დარეგისტრირება/გადამოწმების გასაღების განახლებები.** `RegisterVerifyingKey` ახორციელებს backend-ს,
   ვალდებულება და მიკროსქემის/ვერსიის შეზღუდვები, სანამ გასაღები შევიდეს
   რეესტრი (`crates/iroha_core/src/smartcontracts/isi/world.rs:2067`–`2137`).
   გასაღების განახლება ავტომატურად აცილებს ძველ ჩანაწერს და წაშლის შიდა ბაიტებს,
   როგორც ახორციელებს `crates/iroha_core/tests/zk_vk_deprecate_marks_status.rs:1`.
3. ** აქტივების პოლიტიკის გადასვლების დაგეგმვა. ** მას შემდეგ, რაც ახალი პარამეტრის ID გამოცხადდება
   მმართველობა რეკავს `ScheduleConfidentialPolicyTransition` სასურველთან ერთად
   რეჟიმი, გარდამავალი ფანჯარა და აუდიტის ჰეში. შემსრულებელი უარს ამბობს კონფლიქტზე
   გადასვლები ან აქტივები გამორჩეული გამჭვირვალე მიწოდებით. ტესტები, როგორიცაა
   `crates/iroha_core/tests/confidential_policy_gates.rs:300`–`384` დაადასტურეთ, რომ
   შეწყვეტილი გადასვლები ნათელია `pending_transition`, ხოლო
   `confidential_policy_transition_reaches_shielded_only_on_schedule` ზე
   lines385–433 ადასტურებს, რომ დაგეგმილი განახლებები გადაბრუნდება `ShieldedOnly`-ზე ზუსტად
   ეფექტური სიმაღლე.
4. **პოლიტიკის განაცხადი და მემპულის დაცვა.** ბლოკის შემსრულებელი ასუფთავებს ყველა მოლოდინს
   გადადის თითოეული ბლოკის დასაწყისში (`apply_policy_if_due`) და ასხივებს
   ტელემეტრია, თუ გადასვლა ვერ მოხერხდა, ასე რომ ოპერატორებს შეუძლიათ ხელახლა დაგეგმონ. მიღების დროს
   მემპული უარს ამბობს ტრანზაქციებზე, რომელთა ეფექტური პოლიტიკა შეიცვლება შუა ბლოკში,
   გარდამავალ ფანჯარაში დეტერმინისტული ჩართვის უზრუნველყოფა
   (`docs/source/confidential_assets.md:60`).
