# Инструкция по развертыванию FASTPQ (этап 7–3)

В этом сборнике действий реализовано требование дорожной карты Stage7-3: каждое обновление автопарка
который обеспечивает выполнение FASTPQ на графическом процессоре, должен приложить воспроизводимый манифест теста,
парное свидетельство Grafana и документированное упражнение по откату. Он дополняет
`docs/source/fastpq_plan.md` (цели/архитектура) и
`docs/source/fastpq_migration_guide.md` (этапы обновления на уровне узла), сосредоточив внимание
в контрольном списке развертывания, ориентированном на оператора.

## Область применения и роли

– **Разработка релиза/SRE:** собственные тесты, подписание манифестов и
  экспорт информационной панели до утверждения развертывания.
– **Гильдия операций**: проводит поэтапное внедрение, записывает репетиции отката и сохраняет данные.
  пакет артефактов под `artifacts/fastpq_rollouts/<timestamp>/`.
– **Управление/соответствие**: проверяет, что каждое изменение сопровождается доказательствами.
  запрос до того, как для парка будет переключено значение FASTPQ по умолчанию.

## Контрольный список для сбора доказательств1. **Создавайте тесты графического процессора.**
   - Запустите каноническую рабочую нагрузку (20 000 логических строк, 32 768 дополненных строк) через
     `cargo run -p fastpq_prover --bin fastpq_metal_bench -- --rows 20000 --pretty`.
   - Оберните результат с помощью `scripts/fastpq/wrap_benchmark.py`, используя `--row-usage <decoded witness>`, чтобы пакет содержал доказательства гаджета вместе с телеметрией графического процессора. Передайте `--require-lde-mean-ms 950 --require-poseidon-mean-ms 1000 --sign-output`, чтобы оболочка быстро выходила из строя, если какой-либо ускоритель превышает целевой показатель или если телеметрия очереди/профиля Poseidon отсутствует, а также для создания отсоединенной подписи.
   — Повторите действия на хосте CUDA, чтобы манифест содержал оба семейства графических процессоров.
   - **Не** разбирайте `benchmarks.metal_dispatch_queue` или
     `benchmarks.zero_fill_hotspots` блокирует упакованный JSON. Ворота CI
     (`ci/check_fastpq_rollout.sh`) теперь читает эти поля и не работает при очереди
     запас падает ниже одного слота или когда какая-либо точка доступа LDE сообщает `mean_ms >
     0,40 мс, автоматически активируя защиту телеметрии Stage7.
2. **Создайте манифест.** Используйте `cargo xtask fastpq-bench-manifest …` в качестве
   показано в таблице. Сохраните `fastpq_bench_manifest.json` в пакете развертывания.
3. **Экспорт Grafana.**
   - Добавьте аннотацию к плате `FASTPQ Acceleration Overview` с помощью развертывающегося окна,
     ссылка на соответствующие идентификаторы панели Grafana.
   - Экспортируйте JSON панели мониторинга через API Grafana (команда выше) и включите
     раздел `annotations`, чтобы рецензенты могли сопоставить кривые внедрения с
     поэтапное внедрение.
4. **Снимки оповещений.** Скопируйте точные используемые правила оповещений (`dashboards/alerts/…`).
   путем выкатывания в бандл. Если правила Prometheus были переопределены, включите
   переопределение дифференциала.
5. **Prometheus/OTEL очистка.** Захват `fastpq_execution_mode_total{device_class="<matrix>"}` с каждого
   ведущий (до и после этапа) плюс стойка OTEL
   `fastpq.execution_mode_resolutions_total` и спаренный
   Записи журнала `telemetry::fastpq.execution_mode`. Эти артефакты доказывают, что
   Внедрение графических процессоров стабильно, и принудительные откаты ЦП по-прежнему выдают телеметрию.
6. **Архивировать телеметрию использования строк.** После декодирования ExecWitness запустите
   развертывания, поместите полученный JSON под `row_usage/` в комплекте. CI
   помощник (`ci/check_fastpq_row_usage.sh`) сравнивает эти снимки с
   канонические базовые линии, а `ci/check_fastpq_rollout.sh` теперь требует каждого
   пакет для отправки хотя бы одного файла `row_usage` для присоединения доказательств TF-5.
   к билету на выпуск.

## Поэтапное развертывание

Используйте три детерминированные фазы для каждого флота. Продвигайтесь только после выхода
критерии на каждом этапе удовлетворены и задокументированы в пакете доказательств.| Фаза | Область применения | Критерии выхода | Вложения |
|-------|-------|---------------|-------------|
| Пилот (P1) | 1 узел плоскости управления + 1 узел плоскости данных на регион | `fastpq_execution_mode_total{device_class="<matrix>", backend="metal"}` ≥90% в течение 48 часов, отсутствие инцидентов Alertmanager и прохождение тренировки по откату. | Пакет с обоих хостов (тестовые JSON-файлы, экспорт Grafana с пилотной аннотацией, журналы отката). |
| Рампа (P2) | ≥50% валидаторов плюс хотя бы одна архивная полоса на кластер | Выполнение графического процессора продолжалось в течение 5 дней, не более 1 скачка понижения версии > 10 минут, а счетчики Prometheus подтверждают аварийное оповещение в течение 60 секунд. | Обновлен экспорт Grafana, показывающий аннотацию рампы, различия очистки Prometheus, скриншот/журнал Alertmanager. |
| По умолчанию (P3) | Остальные узлы; FASTPQ отмечен по умолчанию в `iroha_config` | Подписанный манифест стенда + экспорт Grafana со ссылкой на окончательную кривую внедрения и документированное упражнение по откату, демонстрирующее переключение конфигурации. | Окончательный манифест, Grafana JSON, журнал отката, ссылка на билет для проверки изменений конфигурации. |

Документируйте каждый шаг продвижения в билете на внедрение и дайте прямую ссылку на
`grafana_fastpq_acceleration.json`, чтобы рецензенты могли сопоставить
сроки с доказательствами.

## Упражнения по откату

Каждый этап развертывания должен включать в себя репетицию отката:

1. Выберите по одному узлу в кластере и запишите текущие метрики:
   ```bash
   curl -s http://<host>:8180/metrics | rg 'fastpq_execution_mode_total{device_class'
   ```
2. Принудительно установите режим ЦП на 10 минут с помощью ручки настройки.
   (`zk.fastpq.execution_mode = "cpu"`) или переопределение среды:
   ```bash
   FASTPQ_GPU=cpu irohad --config <path> --genesis-manifest-json <path>
   ```
3. Подтвердите журнал перехода на более раннюю версию.
   (`telemetry::fastpq.execution_mode resolved="cpu" requested="gpu"`) и поцарапать
   конечную точку Prometheus снова, чтобы показать приращение счетчика.
4. Восстановите режим графического процессора и убедитесь, что `telemetry::fastpq.execution_mode` теперь сообщает
   `resolved="metal"` (или `resolved="cuda"/"opencl"` для неметаллических дорожек),
   подтвердите, что парсинг Prometheus содержит образцы как процессора, так и графического процессора в
   `fastpq_execution_mode_total{backend=…}` и запишите прошедшее время в
   обнаружение/очистка.
5. Храните стенограммы оболочки, метрики и подтверждения операторов как
   `rollback_drill.log` и `metrics_rollback.prom` в развертываемом пакете. Эти
   файлы должны иллюстрировать полный цикл перехода на более раннюю версию + восстановления, поскольку
   `ci/check_fastpq_rollout.sh` теперь завершается сбоем всякий раз, когда в журнале отсутствует графический процессор.
   Строка восстановления или снимок метрик не включают счетчики ЦП или ГП.

Эти журналы доказывают, что каждый кластер может постепенно деградировать и что команды SRE
знать, как детерминированно отступить, если драйверы графического процессора или ядра регрессируют.

## Резервный вариант смешанного режима (WP2-E.6)

Всякий раз, когда хосту требуется GPU FFT/LDE, но хеширование CPU Poseidon (согласно Stage7 <900 мс
требование), объедините следующие артефакты со стандартными журналами отката:1. **Разница в конфигурации**. Зарегистрируйте (или прикрепите) локальное переопределение хоста, которое устанавливает
   `zk.fastpq.poseidon_mode = "cpu"` (`FASTPQ_POSEIDON_MODE=cpu`) при выходе
   `zk.fastpq.execution_mode` нетронутый. Назовите патч
   `artifacts/fastpq_rollouts/<stamp>/<fleet>/<lane>/poseidon_fallback.patch`.
2. **Счетчик «Посейдона».**
   ```bash
   curl -s http://<host>:8180/metrics \
     | rg 'fastpq_poseidon_pipeline_total{.*device_class="<label>"' \
     > artifacts/fastpq_rollouts/<stamp>/<fleet>/<lane>/metrics_poseidon.prom
   ```
   Захват должен показывать, что `path="cpu_forced"` увеличивается синхронно с
   Счетчик GPU FFT/LDE для этого класса устройства. Сделайте вторую царапину после возвращения
   вернитесь в режим графического процессора, чтобы рецензенты могли видеть возобновление строки `path="gpu"`.

   Передайте полученный файл в `wrap_benchmark.py --poseidon-metrics …`, чтобы завернутый тест записал те же счетчики в свой раздел `poseidon_metrics`; это обеспечивает одинаковый рабочий процесс при развертывании Metal и CUDA и позволяет проверять резервные доказательства без открытия отдельных файлов очистки.
3. **Выдержка из журнала.** Скопируйте записи `telemetry::fastpq.poseidon`, подтверждающие
   резольвер переключен на процессор (`cpu_forced`) в
   `poseidon_fallback.log`, сохраняются временные метки, чтобы можно было просматривать временные шкалы Alertmanager.
   связано с изменением конфигурации.

CI обеспечивает проверку очереди/заполнения нулями сегодня; как только ворота смешанного режима приземлятся,
`ci/check_fastpq_rollout.sh` также будет настаивать на том, чтобы любой пакет, содержащий
`poseidon_fallback.patch` поставляет соответствующий снимок `metrics_poseidon.prom`.
Следование этому рабочему процессу обеспечивает возможность аудита резервной политики WP2-E.6 и ее привязку к
те же сборщики доказательств, которые использовались при развертывании по умолчанию.

  `.github/workflows/fastpq-rollout.yml`, поэтому отсутствующие артефакты быстро выходят из строя, прежде чем
  билет на выпуск может закрыться. Конвейер выпуска может архивировать проверенные пакеты.
  наряду с подписанными манифестами, передав
  `--fastpq-rollout-bundle artifacts/fastpq_rollouts/<stamp>/<fleet>/<lane>` для
  И18НИ00000136Х; помощник повторяет
  `ci/check_fastpq_rollout.sh` (если не установлен `--skip-fastpq-rollout-check`) и
  копирует дерево каталогов в `artifacts/releases/<version>/fastpq_rollouts/…`.
  В рамках этого шлюза сценарий обеспечивает глубину очереди Stage7 и заполнение нулями.
  бюджеты, прочитав `benchmarks.metal_dispatch_queue` и
  `benchmarks.zero_fill_hotspots` из каждого стендового JSON `metal`.

подписанный эталонный манифест.
