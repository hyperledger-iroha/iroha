# SM конфигурациясын тасымалдау

SM2/SM3/SM4 мүмкіндіктер жинағын шығару үшін компиляциядан көбірек қажет
`sm` мүмкіндік жалаушасы. Түйіндер қабаттардың артындағы функционалдылықты қамтамасыз етеді
`iroha_config` профильдері және генезис манифестінің сәйкестігін күту
әдепкі мәндер. Бұл жазбаны жылжыту кезінде ұсынылған жұмыс процесін түсіреді
бар желі «тек Ed25519-дан» «SM-қосылғанға» дейін.

## 1. Құрастыру профилін тексеріңіз

- `--features sm` көмегімен екілік файлдарды құрастырыңыз; кезде ғана `sm-ffi-openssl` қосыңыз
  OpenSSL/Tongsuo алдын ала қарау жолын қолдануды жоспарлаңыз. `sm`сыз құрастырылады
  мүмкіндік конфигурация қосулы болса да, қабылдау кезінде `sm2` қолтаңбаларын қабылдамайды.
  олар.
- CI `sm` артефактілерін және валидацияның барлық қадамдарын (`жүк) жариялайтынын растаңыз.
  test -p iroha_crypto --функциялар sm`, біріктіру қондырғылары, fuzz Suite) өту
  сіз орналастырғыңыз келетін нақты екілік файлдарда.

## 2. Қабат конфигурациясын қайта анықтау

`iroha_config` үш деңгейді қолданады: `defaults` → `user` → `actual`. SM жіберіңіз
операторлар валидаторларға тарататын `actual` профилінде қайта анықтайды және
`user` параметрін Ed25519 мекенжайында қалдырыңыз - әзірлеушінің әдепкі параметрлері өзгеріссіз қалады.

```toml
# defaults/actual/config.toml
[crypto]
enable_sm_openssl_preview = false         # flip to true only when the preview backend is rolled out
default_hash = "sm3-256"
allowed_signing = ["ed25519", "sm2"]      # keep sorted for deterministic manifests
sm2_distid_default = "CN12345678901234"   # organisation-specific distinguishing identifier
```

Сол блокты `defaults/genesis` манифестіне `kagami genesis арқылы көшіріңіз.
жасау …` (add `--рұқсат етілген-қол қою sm2 --әдепкі-хэш sm3-256` қажет болса
қайта анықтайды) сондықтан `parameters` блогы және енгізілген метадеректер
орындау уақытының конфигурациясы. Манифест пен конфигурация кезінде құрбылар бастаудан бас тартады
суреттер алшақтайды.

## 3. Жаратылыс Манифесттерін қалпына келтіріңіз

- Әрқайсысы үшін `kagami genesis generate --consensus-mode <mode>` іске қосыңыз
  ортасын орнатыңыз және TOML қайта анықтауларымен бірге жаңартылған JSON файлын орындаңыз.
- Манифестке қол қойыңыз (`kagami genesis sign …`) және `.nrt` пайдалы жүктемесін таратыңыз.
  Қол қойылмаған JSON манифестінен жүктелетін түйіндер орындалу уақыты криптосын шығарады
  тікелей файлдан конфигурациялау — бәрібір бірдей консистенцияға бағынады
  тексереді.

## 4. Трафик алдында тексеру

- Жаңа екілік файлдармен және конфигурациямен кезеңдік кластерді қамтамасыз етіңіз, содан кейін тексеріңіз:
  - `/status` теңдестілер қайта іске қосылғанда `crypto.sm_helpers_available = true` көрсетеді.
  - Torii қабылдау әлі де SM2 қолдарын қабылдамайды, ал `sm2` жоқ
    `allowed_signing` және тізім болған кезде аралас Ed25519/SM2 топтамаларын қабылдайды
    екі алгоритмді де қамтиды.
  - `iroha_cli tools crypto sm2 export …` негізгі материал жаңа арқылы егілген
    әдепкі мәндер.
- SM2 детерминирленген қолтаңбаларын қамтитын интеграциялық түтін сценарийлерін іске қосыңыз және
  Хост/VM сәйкестігін растау үшін SM3 хэштеу.

## 5. Қайтару жоспары- Қайтаруды құжаттаңыз: `sm2`-ті `allowed_signing` ішінен алып тастаңыз және қалпына келтіріңіз
  `default_hash = "blake2b-256"`. Өзгерісті бірдей `actual` арқылы басыңыз
  профиль құбыры, сондықтан әрбір валидатор монотонды түрде айналады.
- SM манифесттерін дискіде сақтау; сәйкес келмейтін конфигурация мен генезитті көретін құрдастар
  деректер бастаудан бас тартады, бұл ішінара кері қайтарудан қорғайды.
- OpenSSL/Tongsuo алдын ала қарауы қосылған болса, өшіру қадамдарын қосыңыз
  `crypto.enable_sm_openssl_preview` және ортақ нысандарды файлдан жою
  орындалу ортасы.
