# SM-ийн онцлогийг нэвтрүүлэх ба телеметрийн хяналтын хуудас

Энэхүү хяналтын хуудас нь SRE болон операторын багуудад SM (SM2/SM3/SM4) функцийг идэвхжүүлэхэд тусалдаг
аудит болон нийцлийн хаалгыг цэвэрлэсний дараа аюулгүйгээр тохируулна. Энэ баримт бичгийг дагаж мөрдөөрэй
`docs/source/crypto/sm_program.md` дахь тохиргооны товчлолын хажууд болон
`docs/source/crypto/sm_compliance_brief.md` дахь хууль эрх зүйн/экспортын удирдамж.

## 1. Нислэгийн өмнөх бэлэн байдал
- [ ] Ажлын талбарын хувилбарын тэмдэглэлийг `sm`-г зөвхөн баталгаажуулах эсвэл гарын үсэг зурж байгааг баталгаажуулах,
      нэвтрүүлэх үе шатаас хамаарна.
- [ ] флотыг багтаасан амлалтаас бүтээгдсэн хоёртын файлуудыг ажиллуулж байгаа эсэхийг шалгана уу
      SM телеметрийн тоолуур ба тохиргооны товчлуурууд. (Зорилтот хувилбар TBD; зам
      танилцуулах тасалбар дээр.)
- [ ] `scripts/sm_perf.sh --tolerance 0.25`-г үе шаттай зангилаа дээр ажиллуулах (зорилтод тус бүр
      архитектур) болон хураангуй гаралтыг архивлана. Скрипт одоо автоматаар сонгогддог
      хурдатгалын горимуудын харьцуулах зорилт болгон скаляр суурь
      (SM3 NEON ажиллах үед `--compare-tolerance` анхдагч нь 5.25 хүртэл);
      анхан шатны эсвэл харьцуулалт бол судалгаа хийх эсвэл нэвтрүүлэхийг хориглох
      хамгаалагч амжилтгүй боллоо. Linux/aarch64 Neoverse техник хангамж дээр зураг авахдаа дамжуулна уу
      `--baseline crates/iroha_crypto/benches/sm_perf_baseline_aarch64_unknown_linux_gnu_<mode>.json --write-baseline`
      экспортлогдсон `m3-pro-native` медиануудыг хостын зураг авалттай дарж бичих
      тээвэрлэхээс өмнө.
- [ ] `status.md` ба танилцуулах тасалбар нь дагаж мөрдөх мэдүүлгийг бүртгэнэ.
      шаардлагатай эрх мэдлийн хүрээнд үйл ажиллагаа явуулж буй аливаа зангилаа (дагаж мөрдөх товчийг үзнэ үү).
- [ ] Баталгаажуулагчид SM нэвтрэх түлхүүрүүдийг хадгалах бол KMS/HSM шинэчлэлтүүдийг бэлтгэ
      техник хангамжийн модулиуд.

## 2. Тохиргооны өөрчлөлтүүд
1. SM2 түлхүүрийн бүртгэл болон буулгахад бэлэн хэсгийг үүсгэхийн тулд xtask туслахыг ажиллуулна уу:
   ```bash
   cargo xtask sm-operator-snippet \
     --distid CN12345678901234 \
     --json-out sm2-key.json \
     --snippet-out client-sm2.toml
   ```
   Гаралтыг шалгах шаардлагатай үед `--snippet-out -` (мөн сонголтоор `--json-out -`) ашиглан stdout руу дамжуулаарай.
   Хэрэв та доод түвшний CLI командуудыг гараар жолоодохыг илүүд үзэж байгаа бол түүнтэй адилтгах урсгал нь:
   ```bash
   cargo run -p iroha_cli --features sm -- \
     crypto sm2 keygen \
     --distid CN12345678901234 \
     --output sm2-key.json

   cargo run -p iroha_cli --features sm -- \
     crypto sm2 export \
     --private-key-hex "$(jq -r .private_key_hex sm2-key.json)" \
     --distid CN12345678901234 \
     --snippet-output client-sm2.toml \
     --emit-json --quiet
   ```
   Хэрэв `jq` боломжгүй бол `sm2-key.json`-г нээж, `private_key_hex` утгыг хуулж, экспортын команд руу шууд дамжуулна.
2. Үүссэн хэсгийг зангилаа бүрийн тохиргоонд нэмнэ үү (утгуудыг
   зөвхөн баталгаажуулах үе шат; орчинд тохируулж, товчлууруудыг харуулсан шиг эрэмбэлэгдсэн байлгана уу):
```toml
[crypto]
default_hash = "sm3-256"
allowed_signing = ["ed25519", "sm2"]   # remove "sm2" to stay in verify-only mode
sm2_distid_default = "1234567812345678"
# enable_sm_openssl_preview = true  # optional: only when deploying the OpenSSL/Tongsuo path
```
3. Зангилааг дахин эхлүүлж, `crypto.sm_helpers_available` ба (хэрэв та урьдчилан харах арын хэсгийг идэвхжүүлсэн бол) `crypto.sm_openssl_preview_enabled` гадаргууг дараах байдлаар баталгаажуулна уу:
   - `/status` JSON (`"crypto":{"sm_helpers_available":true,"sm_openssl_preview_enabled":true,...}`).
   - Зангилаа тус бүрийн дүрсэлсэн `config.toml`.
4. Зөвшөөрөгдсөн жагсаалтад SM алгоритмуудыг нэмэхийн тулд манифест/генезийн оруулгуудыг шинэчилнэ үү.
   гарын үсэг зурах нь дараа нь танилцуулах үед идэвхжсэн. `--genesis-manifest-json` ашиглах үед
   Урьдчилан гарын үсэг зурсан генезис блокгүйгээр `irohad` одоо ажиллах цагийн криптог суулгаж байна
   манифестын `crypto` блокоос шууд агшин зуурын зураг—манифест байгаа эсэхийг шалгаарай.
   урагшлуулахаасаа өмнө өөрчлөлтийн төлөвлөгөөгөө шалгасан.## 3. Телеметр ба хяналт
- Prometheus төгсгөлийн цэгүүдийг хусаад дараах тоолуур/хэмжээ гарч ирэх эсэхийг шалгаарай:
  - `iroha_sm_syscall_total{kind="verify"}`
  - `iroha_sm_syscall_total{kind="hash"}`
  - `iroha_sm_syscall_total{kind="seal|open",mode="gcm|ccm"}`
  - `iroha_sm_openssl_preview` (0/1 хэмжигч нь урьдчилан харах сэлгэх төлөвийг мэдээлдэг)
  - `iroha_sm_syscall_failures_total{kind="verify|hash|seal|open",reason="..."}`
- SM2 гарын үсэг идэвхжсэний дараа дэгээгээр гарын үсэг зурах зам; тоологч нэмэх
  `iroha_sm_sign_total` ба `iroha_sm_sign_failures_total`.
- Grafana хяналтын самбар/сануулгыг үүсгэх:
  - Гэмтлийн тоолуур (цонх 5м).
  - SM syscall дамжуулах чадвар гэнэт буурсан.
  - Зангилаа хоорондын ялгаа (жишээ нь, таарахгүй идэвхжүүлэлт).

## 4. Дамжуулах алхамууд
| Үе шат | Үйлдлүүд | Тэмдэглэл |
|-------|---------|-------|
| Зөвхөн баталгаажуулах | `crypto.default_hash`-г `sm3-256` болгон шинэчилж, `allowed_signing`-г `sm2`гүйгээр үлдээж, баталгаажуулалтын тоолуурыг хяна. | Зорилго: зөвшилцлийн зөрүүг эрсдэлгүйгээр SM баталгаажуулах замыг хэрэгжүүлэх. |
| Холимог гарын үсэг зурах нисгэгч | Хязгаарлагдмал SM гарын үсэг зурахыг зөвшөөрөх (баталгаажуулагчдын дэд хэсэг); гарын үсэг зурах тоолуур болон хоцролтыг хянах. | Ed25519 руу буцах боломжтой хэвээр байгаа эсэхийг шалгах; Хэрэв телеметр таарахгүй байвал зогсоо. |
| GA гарын үсэг зурах | `allowed_signing`-г өргөтгөж `sm2`, манифест/SDK-г шинэчлэх, эцсийн runbook-ийг нийтлэх. | Хаалттай аудитын дүгнэлт, шинэчилсэн нийцлийн мэдүүлэг, тогтвортой телеметрийг шаарддаг. |

### Бэлэн байдлын үнэлгээ
- **Зөвхөн бэлэн байдлыг шалгана уу (SM-RR1).** Хуралдаан гаргах Eng, Crypto WG, Ops, and Legal. Шаардах:
  - `status.md` нь дагаж мөрдөх мэдүүлгийн статус + OpenSSL гарал үүслийг тэмдэглэж байна.
  - `docs/source/crypto/sm_program.md` / `sm_compliance_brief.md` / энэ хяналтын хуудсыг сүүлийн хувилбарын цонхонд шинэчилсэн.
  - `defaults/genesis` эсвэл хүрээлэн буй орчны өвөрмөц манифест нь `crypto.allowed_signing = ["ed25519","sm2"]` ба `crypto.default_hash = "sm3-256"` (эсвэл нэгдүгээр шатанд байгаа бол `sm2`-гүй зөвхөн баталгаажуулах хувилбарыг) харуулж байна.
  - `scripts/sm_openssl_smoke.sh` + `scripts/sm_interop_matrix.sh` бүртгэлийг танилцуулах тасалбарт хавсаргасан.
  - Телеметрийн хяналтын самбарыг (`iroha_sm_*`) тогтворжсон төлөвийн байдлыг шалгасан.
- **Нисгэгчийн бэлэн байдалд гарын үсэг зурах (SM-RR2).** Нэмэлт хаалга:
  - RustCrypto SM стекийн хаагдсан аудитын тайлан эсвэл Аюулгүй байдлын гарын үсэг бүхий хяналтыг нөхөх RFC.
  - Операторын ажлын дэвтэр (байгууламжийн онцлог) буцаах/буцах алхмуудаар шинэчлэгдсэн.
  - Туршилтын бүлэгт зориулсан Эхлэл манифест нь `allowed_signing = ["ed25519","sm2"]` багтдаг бөгөөд зөвшөөрөгдсөн жагсаалт нь зангилааны тохиргоо бүрд тусгагдсан байдаг.
  - Гарах/буцах төлөвлөгөөг баримтжуулсан (`allowed_signing`-г Ed25519 руу буцааж сольж, манифестуудыг сэргээх, хяналтын самбарыг дахин тохируулах).
- **GA бэлэн байдал (SM-RR3).** Эерэг туршилтын тайлан, баталгаажуулагчийн бүх эрх мэдлийн шинэчилсэн нийцлийн мэдүүлэг, гарын үсэг зурсан телеметрийн суурь мэдээлэл, Release Eng + Crypto WG + Ops/Legal гурвалаас гаргах тасалбарын зөвшөөрөл шаардлагатай.## 5. Сав баглаа боодол, нийцлийн хяналтын хуудас
- **OpenSSL/Tongsuo олдворуудыг багцлаарай.** OpenSSL/Tongsuo 3.0+ хуваалцсан сангуудыг (`libcrypto`/`libssl`) баталгаажуулагчийн багц бүрээр илгээх эсвэл системийн яг хамааралтай байдлыг баримтжуулна. Аудиторууд нийлүүлэгчийн бүтээцийг хянах боломжтой болохын тулд хувилбарыг бичиж, тугуудыг бүтээх, SHA256 хяналтын нийлбэрийг хувилбарын манифестэд бичнэ үү.
- **CI үед баталгаажуулна уу.** Зорилтот платформ бүр дээр савласан олдворуудын эсрэг `scripts/sm_openssl_smoke.sh`-ийг гүйцэтгэдэг CI алхамыг нэмнэ үү. Урьдчилан үзэх туг идэвхжсэн боловч үйлчилгээ үзүүлэгчийг эхлүүлэх боломжгүй (толгой дутуу, дэмжигдээгүй алгоритм гэх мэт) ажил бүтэлгүйтэх ёстой.
- **Дагаж мөрдөх тэмдэглэлийг нийтлэх.** Хувилбарын тэмдэглэлийг / `status.md`-ийг нийлүүлэгчийн багц хувилбар, экспортын хяналтын лавлагаа (GM/T, GB/T) болон SM алгоритмд шаардлагатай аливаа хууль эрх зүйн тусгай мэдүүлгийн хамт шинэчилнэ үү.
- **Операторын runbook-ийн шинэчлэлтүүд.** Шинэчлэлийн урсгалыг баримтжуулах: шинэ хуваалцсан объектуудыг үе шаттайгаар хийж, `crypto.enable_sm_openssl_preview = true` ашиглан үе тэнгийнхнээ дахин эхлүүлж, `/status` талбарыг баталгаажуулж, `iroha_sm_openssl_preview` хэмжигчийг `true` рүү шилжүүлж, төлөвлөгөөг дахин тохируулна уу. багц) хэрэв урьдчилан харах телеметр флот даяар хазайсан бол.
- **Нотлох баримтыг хадгалах.** OpenSSL/Tongsuo багцын бүтээх лог болон гарын үсэг зурсан гэрчилгээг баталгаажуулагч хувилбарын олдворуудын хамт архивлана, ингэснээр ирээдүйн аудитууд гарал үүслийн хэлхээг дахин гаргах боломжтой болно.

## 6. Ослын хариу арга хэмжээ
- **Баталгаажуулалтын алдааны огцом өсөлт:** SM-ийн дэмжлэггүйгээр бүтэц рүү буцах эсвэл `sm2`-г устгах
  `allowed_signing`-ээс (шаардлагатай бол `default_hash`-г буцаах) өмнөх рүү шилжих
  мөрдөн байцаалтын явцад суллах. Амжилтгүй ачаалал, харьцуулсан хэш болон зангилааны логуудыг авах.
- **Гүйцэтгэлийн регресс:** SM хэмжигдэхүүнийг Ed25519/SHA2 суурь үзүүлэлттэй харьцуул.
  Хэрэв ARM дотоод зам нь зөрүүг үүсгэж байвал `crypto.sm_intrinsics = "force-disable"`-г тохируулна уу.
  (хүлээгдэж буй хэрэгжилтийг сэлгэх) болон илэрцүүдийг мэдээлэх.
- **Телеметрийн цоорхой:** Хэрэв тоолуур байхгүй эсвэл шинэчлэгдээгүй бол асуудал гаргана уу
  Release Engineering-ийн эсрэг; цоорхой хүртэл илүү өргөн өнхрөлтийг үргэлжлүүлж болохгүй
  шийдэгдэж байна.

## 7. Хяналтын хуудасны загвар
- [ ] Тохиргоог үе шаттайгаар хийж, дахин эхлүүлсэн.
- [ ] Телеметрийн тоолуур харагдах ба хяналтын самбарыг тохируулсан.
- [ ] Дагаж мөрдөх/хууль ёсны алхмуудыг бүртгэсэн.
- [ ] Crypto WG / Release TL-ээс баталсан танилцуулах үе шат.
- [ ] Нэвтрүүлсний дараах хяналтыг дуусгаж, олдворуудыг баримтжуулсан.

Энэ хяналтын хуудсыг танилцуулах тасалбарт хөтөлж, `status.md`-г шинэчилнэ үү.
үе шат хоорондын флотын шилжилт.
