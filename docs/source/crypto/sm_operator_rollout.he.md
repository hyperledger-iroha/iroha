# רשימת בדיקות לפריסת תכונות SM וטלמטריה

רשימה זו מסייעת לצוותי התפעול וה־SRE להפעיל בבטחה את תכונות ה־SM (SM2/SM3/SM4) לאחר שהשלמתם את בקרות הבדיקה, הביקורת והציות. עבדו עם מסמך זה במקביל לתקציר התצורה ב־`docs/source/crypto/sm_program.md` ולמדריך הציות/ייצוא ב־`docs/source/crypto/sm_compliance_brief.md`.

## 1. היערכות מקדימה
- [ ] ודאו שהערות הגרסה של ה־workspace מציינות האם `sm` נמצא במצב אימות בלבד או במצב חתימה, בהתאם לשלב הפריסה.
- [ ] בדקו שהציוד מפעיל בינאריים שנבנו מקומיט שמכיל את מוני הטלמטריה וידיות התצורה של SM. (גרסת היעד תיקבע בטיקט הפריסה.)
- [ ] הפעילו `scripts/sm_perf.sh --tolerance 0.25` על צומת סטייג'ינג (עבור כל ארכיטקטורה יעד) וארכבו את סיכום הפלט. הסקריפט בוחר אוטומטית בבייסליין הסקלרי לצורכי השוואה במצבי האצה (`--compare-tolerance` מוגדר ל־5.25 כל עוד עבודת ה־SM3 NEON ננעלת); אם שמירה ראשית או משנית חורגת מהגבול – עצרו או חקרו לפני התקדמות. כאשר מודדים בחומרת Neoverse על לינוקס/aarch64, הפעילו `--baseline crates/iroha_crypto/benches/sm_perf_baseline_aarch64_unknown_linux_gnu_<mode>.json --write-baseline` כדי להחליף את המדדים שיוצאו מ־`m3-pro-native` במדידות של המארח לפני שילוח.
- [ ] ודאו ש־`status.md` והטיקט מתעדים את כל הגשות הציות הרלוונטיות לפריסות באזורים המחייבים זאת (ראו מדריך הציות).
- [ ] הכינו עדכוני KMS/HSM אם ולדטורים יאחסנו מפתחות חתימה של SM במודולים חומרתיים.

## 2. שינויי תצורה
1. הפעילו את xtask כדי להפיק במכה אחת את מאגר המפתחות וקטע התצורה:
   ```bash
   cargo xtask sm-operator-snippet \
     --distid CN12345678901234 \
     --json-out sm2-key.json \
     --snippet-out client-sm2.toml
   ```
   ניתן להזרים את הקטע ישירות ל־stdout בעזרת `--snippet-out -` (ואת ה־JSON בעזרת `--json-out -`) כאשר רק רוצים לעיין בתוצאה.
   אם בכל זאת מעדיפים להריץ את פקודות ה־CLI הנמוכות, השתמשו ברצף הבא:
   ```bash
   cargo run -p iroha_cli --features sm -- \
     crypto sm2 keygen \
     --distid CN12345678901234 \
     --output sm2-key.json

   cargo run -p iroha_cli --features sm -- \
     crypto sm2 export \
     --private-key-hex "$(jq -r .private_key_hex sm2-key.json)" \
     --distid CN12345678901234 \
     --snippet-output client-sm2.toml \
     --emit-json --quiet
   ```
   אם `jq` אינו זמין, פתחו את `sm2-key.json`, העתיקו את הערך `private_key_hex`, והעבירו אותו ישירות לפקודת ה־export.
2. הוסיפו את הקטע שהתקבל לקובץ התצורה של כל צומת (הערכים מדגימים שלב אימות בלבד; התאימו אותם לסביבה שלכם):
```toml
[crypto]
enable_sm = true
default_hash = "sm3-256"
allowed_signing = ["ed25519"]   # הוסיפו "sm2" בשלב הפיילוט/GA של חתימה
sm2_distid_default = "1234567812345678"
# enable_sm_openssl_preview = true  # אופציונלי: רק כאשר פורשים את נתיב OpenSSL/Tongsuo
```
3. אתחלו את הצומת ואמתו כי `crypto.sm_helpers_available` ו־`crypto.sm_openssl_preview_enabled` מופיעים כמצופה:
   - ב־`/status` JSON (`"crypto":{"sm_helpers_available":true,"sm_openssl_preview_enabled":true,...}`).
   - בקובץ `config.toml` המעודכן של הצומת.
4. עדכנו מניפסטים/קבצי Genesis כדי להוסיף אלגוריתמים של SM לרשימת ההרשאות כאשר חתימה מאופשרת בשלב מאוחר יותר. בהפעלה עם `--genesis-manifest-json` בלבד, `irohad` טוען את תצלום הצפנה הריצה ישירות ממקטע `crypto`; ודאו שהמניפסט נכלל בתוכנית השינויים לפני שמתקדמים.

## 3. טלמטריה וניטור
- גרדו את נקודות ה־Prometheus וודאו שהמונה/מדדים הבאים זמינים:
  - `iroha_sm_syscall_total{kind="verify"}`
  - `iroha_sm_syscall_total{kind="hash"}`
  - `iroha_sm_syscall_total{kind="seal|open",mode="gcm|ccm"}`
  - `iroha_sm_openssl_preview` (מדד 0/1 המציין את מצב תצוגת התצוגה המקדימה)
  - `iroha_sm_syscall_failures_total{kind="verify|hash|seal|open",reason="..."}`
- כאשר חתימת SM2 מאופשרת, הוסיפו מונים ל־`iroha_sm_sign_total` ול־`iroha_sm_sign_failures_total`.
- צרו לוחות Grafana/התראות עבור:
  - קפיצות במוני כשל (חלון של 5 דקות).
  - ירידות חדות בקצב הקריאות של SM.
  - פערי תצורה בין צמתים (למשל, הפעלה שאינה אחידה).

## 4. שלבי פריסה
| שלב | פעולות | הערות |
|------|---------|-------|
| אימות בלבד | עדכנו את `crypto.default_hash` ל־`sm3-256`, שמרו את `allowed_signing` על האלגוריתמים הקיימים, ונטרו את מוני האימות. | מטרה: להפעיל את נתיבי האימות של SM מבלי לסכן סטייה בקונצנזוס. |
| פיילוט חתימה חלקי | אפשרו חתימות SM לקבוצה מצומצמת של ולדטורים; נטרו מוני חתימה וזמני תגובה. | ודאו שיש אפשרות חזרה ל־Ed25519; עצרו אם הטלמטריה חושפת אי־התאימות. |
| חתימת GA | הרחיבו את `allowed_signing` כך שיכלול `"sm2"`, עדכנו מניפסטים/SDKs, ופרסמו ספר נהלים מעודכן. | דורש סגירת ממצאי ביקורת, רענון מסמכי ציות לכל אזורי הוולידטורים, וטלמטריה יציבה. |

### סקירות מוכנות
- **מוכנות אימות בלבד (SM-RR1).** מיפגש עם Release Eng, Crypto WG, Ops ו־Legal. דרושים:
  - עדכון `status.md` עם מצב הגשות הציות ופרובננס OpenSSL.
  - רענון המסמכים `docs/source/crypto/sm_program.md`, `sm_compliance_brief.md` והרשימה הנוכחית בחלון הגרסה האחרון.
  - מניפסט ברירת המחדל (`defaults/genesis` או ייעודי לסביבה) מציג `crypto.allowed_signing = ["ed25519", "sm2"]`, `crypto.default_hash = "sm3-256"` (או הגרסה ללא "sm2" בשלב האימות בלבד).
  - צירוף לוגים של `scripts/sm_openssl_smoke.sh` ו־`scripts/sm_interop_matrix.sh` לטיקט הפריסה.
  - סקירת לוח הטלמטריה (`iroha_sm_*`) כדי לוודא התנהגות יציבה.
- **מוכנות פיילוט חתימה (SM-RR2).** תנאים נוספים:
  - דו"ח ביקורת ל־RustCrypto SM נסגר או אושרה חלופת בקרה על־ידי צוות האבטחה.
  - ספרי נהלים ל־Operator (לפי אתר) מעודכנים עם שלבי נפילה/חזרה.
  - מניפסטים של גנזיס עבור קבוצת הפיילוט כוללים `allowed_signing = ["ed25519","sm2"]` והערך משוכפל בכל תצורת צומת.
  - תוכנית יציאה/גלגול לאחור מתועדת (הסרת `"sm2"` מ־`allowed_signing`, שחזור מניפסטים, איפוס דשבורדים).
- **מוכנות GA (SM-RR3).** מחייב דו"ח פיילוט חיובי, הגשות ציות מעודכנות לכל תחומי השיפוט, בייסליינים חתומים של טלמטריה, ואישור טיקט מה־Release Eng + Crypto WG + צוות Ops/Legal.

## 5. חבילות וציות
- **צרפו רכיבי OpenSSL/Tongsuo.** שלחו עם כל חבילת ולדטור גרסאות 3.0+ של `libcrypto`/`libssl` או תארו במדויק את התלות המערכתית. רשמו בגרסת השחרור את הגרסה, דגלי הבנייה ו־SHA256 לטובת שרשרת אספקה ניתנת לביקורת.
- **ודאו במסגרת CI.** הוסיפו משימת CI שמריצה `scripts/sm_openssl_smoke.sh` מול הרכיבים הארוזים בכל פלטפורמה נתמכת. המשימה חייבת להיכשל אם מופעל דגל התצוגה המקדימה וה־provider לא ניתן לאתחול (חסרים headers, אלגוריתם לא נתמך וכו').
- **פרסמו הערות ציות.** עדכנו את הערות הגרסה/`status.md` עם גרסת ה־provider הנלווית, הפניות לרגולציות הייצוא (GM/T, GB/T) וכל דרישה אזורית ל־SM.
- **עדכוני ספר נהלים.** תעדו את תהליך השדרוג: שלבו את קובצי ה־SO החדשים, אתחלו peers עם `crypto.enable_sm_openssl_preview = true`, תאשרו שהשדה ב־`/status` והמדד `iroha_sm_openssl_preview` מתעדכנים ל־`true`, ושמרו תוכנית נסיגה (כיבוי הדגל או החזרת החבילה) במקרה של חריגה בטלמטריה.
- **שמירת ראיות.** ארכבו לוגי בנייה ועדויות חתימה עבור חבילות OpenSSL/Tongsuo עם כל שחרור ולדטורים כדי שתוכלו לשחזר פרובננס בעתיד.

## 6. תגובה לתקלות
- **קפיצות בכשלי אימות:** חזרו לבנייה ללא תמיכת SM או הסירו את `"sm2"` מ־`allowed_signing` (והחזירו את `default_hash` בהתאם), וחזרו לשחרור הקודם בזמן חקירה. שמרו על מטענים כושלים, השוואות חשיש ולוגים.
- **ירידת ביצועים:** השוו מדדי SM לבייסלייני Ed25519/SHA2. אם נתיב אינטרינזי ב־ARM גורם לחריגה, הגדירו `crypto.sm_intrinsics = "force-disable"` ודווחו על הממצאים.
- **פערים בטלמטריה:** אם המונים חסרים או שאינם מתעדכנים, פתחו תקלה מול Release Engineering; אל תתרחבו בפריסה עד להשלמת התיקון.
