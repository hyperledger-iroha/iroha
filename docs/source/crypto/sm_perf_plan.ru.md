## Сбор данных о производительности SM и базовый план

Статус: Задрафтован — 18 мая 2025 г.  
Владельцы: Рабочая группа по производительности (руководитель), Infra Ops (планирование работы лаборатории), Гильдия контроля качества (контроль CI)  
Связанные задачи дорожной карты: SM-4c.1a/b, SM-5a.3b, FASTPQ Stage 7 захват данных между устройствами

### 1. Цели
1. Запишите медианы Neoverse в `sm_perf_baseline_aarch64_unknown_linux_gnu_{scalar,auto,neon_force}.json`. Текущие базовые показатели экспортируются из захвата `neoverse-proxy-macos` под `artifacts/sm_perf/2026-03-lab/neoverse-proxy-macos/` (метка ЦП `neoverse-proxy-macos`) с допуском сравнения SM3, расширенным до 0,70 для aarch64 macOS/Linux. Когда откроется время «голого железа», повторно запустите `scripts/sm_perf_capture_helper.sh --matrix --cpu-label neoverse-n2-b01 --output artifacts/sm_perf/<date>/neoverse-n2-b01` на хосте Neoverse и переместите агрегированные медианы в базовые показатели.  
2. Соберите соответствующие медианы x86_64, чтобы `ci/check_sm_perf.sh` мог защищать оба класса хостов.  
3. Опубликуйте повторяемую процедуру захвата (команды, расположение артефактов, рецензенты), чтобы будущие врата не зависели от племенных знаний.

### 2. Доступность оборудования
В текущей рабочей области доступны только хосты Apple Silicon (macOS Arm64). Захват `neoverse-proxy-macos` экспортируется как промежуточный базовый показатель Linux, но для захвата медиан Neoverse или x86_64 на «голом железе» по-прежнему требуется общее лабораторное оборудование, отслеживаемое под `INFRA-2751`, которое должно быть запущено рабочей группой по производительности после открытия окна лабораторной работы. Остальные окна захвата теперь зарезервированы и отслеживаются в дереве артефактов:

- Неполный комплект Neoverse N2 (стойка Токио B) забронирован на 12 марта 2026 г. Операторы будут повторно использовать команды из раздела 3 и сохранять артефакты под `artifacts/sm_perf/2026-03-lab/neoverse-b01/`.
- x86_64 Xeon (стойка D в Цюрихе) забронирован на 19 марта 2026 г. с отключенным SMT для снижения шума; артефакты будут находиться под `artifacts/sm_perf/2026-03-lab/xeon-d01/`.
- После того, как оба запуска завершатся, добавьте медианы в базовые JSON-файлы и включите шлюз CI в `ci/check_sm_perf.sh` (целевая дата переключения: 25 марта 2026 г.).

До этих дат локально можно обновлять только базовые версии macOS Arm64.### 3. Процедура захвата
1. **Синхронизация цепочек инструментов**  
   ```bash
   rustup override set $(cat rust-toolchain.toml)
   cargo fetch
   ```
2. **Создать матрицу захвата** (для каждого хоста)  
   ```bash
   scripts/sm_perf_capture_helper.sh --matrix \
     --output artifacts/sm_perf/2025-07-lab/${HOSTNAME}
   ```
   Теперь помощник записывает `capture_commands.sh` и `capture_plan.json` в целевой каталог. Сценарий устанавливает пути захвата `raw/*.json` для каждого режима, чтобы лаборанты могли детерминированно группировать прогоны.
3. **Запуск захвата**  
   Выполните каждую команду из `capture_commands.sh` (или запустите эквивалент вручную), гарантируя, что каждый режим создает структурированный большой двоичный объект JSON через `--capture-json`. Всегда указывайте метку хоста через `--cpu-label "<model/bin>"` (или `SM_PERF_CPU_LABEL=<label>`), чтобы метаданные захвата и последующие базовые показатели записывали точное оборудование, создавшее медианы. Помощник уже предоставляет соответствующий путь; для ручного запуска шаблон следующий:
   ```bash
   SM_PERF_CAPTURE_LABEL=auto \
   scripts/sm_perf.sh --mode auto \
     --cpu-label "neoverse-n2-lab-b01" \
     --capture-json artifacts/sm_perf/2025-07-lab/${HOSTNAME}/raw/auto.json
   ```
4. **Проверка результатов**  
   ```bash
   scripts/sm_perf_check \
     artifacts/sm_perf/2025-07-lab/${HOSTNAME}/raw/*.json
   ```
   Убедитесь, что разница между запусками не превышает ±3%. Если нет, перезапустите затронутый режим и отметьте повторную попытку в журнале.
5. **Продвигайте медианы**  
   Используйте `scripts/sm_perf_aggregate.py` для вычисления медиан и копирования их в базовые файлы JSON:
   ```bash
   scripts/sm_perf_aggregate.py \
     artifacts/sm_perf/2025-07-lab/${HOSTNAME}/raw/*.json \
     --output artifacts/sm_perf/2025-07-lab/${HOSTNAME}/aggregated.json
   ```
   Вспомогательные группы фиксируются `metadata.mode`, проверяя, что каждый набор имеет общий
   тот же тройной `{target_arch, target_os}` и выдает сводку JSON с одной записью
   за режим. Медианы, которые должны попадать в файлы базовых показателей, находятся под
   `modes.<mode>.benchmarks`, в то время как сопутствующие записи блока `statistics`
   полный список выборок, мин/макс, среднее значение и стандартное отклонение совокупности для рецензентов и CI.
   Как только агрегированный файл будет создан, вы сможете автоматически записать базовые JSON-файлы (с помощью
   стандартную карту допусков) через:
   ```bash
   scripts/sm_perf_promote_baseline.py \
     artifacts/sm_perf/2025-07-lab/${HOSTNAME}/aggregated.json \
     --out-dir crates/iroha_crypto/benches \
     --target-os unknown_linux_gnu \
     --overwrite
   ```
   Переопределите `--mode`, чтобы ограничить подмножество, или `--cpu-label`, чтобы закрепить
   записанное имя ЦП, если агрегированный источник его опускает.
   Как только оба хоста для каждой архитектуры завершат работу, обновите:
   - `sm_perf_baseline_aarch64_unknown_linux_gnu_{scalar,auto,neon_force}.json`
   - `sm_perf_baseline_x86_64_unknown_linux_gnu_{scalar,auto}.json` (новый)

   Файлы `aarch64_unknown_linux_gnu_*` теперь отражают `m3-pro-native`.
   захват (метка процессора и примечания к метаданным сохраняются), чтобы `scripts/sm_perf.sh` мог
   автоматическое обнаружение хостов aarch64-unknown-linux-gnu без ручных флагов. Когда
   Лабораторный запуск на «голом железе» завершен, перезапустите `scripts/sm_perf.sh --mode 
   --write-baseline crates/iroha_crypto/benches/sm_perf_baseline_aarch64_unknown_linux_gnu_.json`
   с новыми захватами, чтобы перезаписать промежуточные медианы и отметить реальные
   метка хоста.

   > Справка: данные Apple Silicon от июля 2025 г. (маркировка процессора `m3-pro-local`)
   > в архиве `artifacts/sm_perf/2025-07-lab/takemiyacStudio.lan/{raw,aggregated.json}`.
   > Отразите этот макет при публикации артефактов Neoverse/x86, чтобы рецензенты
   > может последовательно различать необработанные/агрегированные результаты.

### 6. Зависимости и связь
- Координировать изменения доступа к лаборатории через `infra-ops@iroha.tech`.  
- Рабочая группа по производительности ежедневно публикует обновления на канале `#perf-lab` во время записи.  
- QA Guild готовит сравнительный анализ (`scripts/sm_perf_compare.py`), чтобы рецензенты могли визуализировать различия.  
- После объединения базовых показателей обновите `roadmap.md` (SM-4c.1a/b, SM-5a.3b) и `status.md`, указав примечания о завершении захвата.

Благодаря этому плану работа по ускорению SM получает воспроизводимые медианы, гейтирование CI и отслеживаемый след доказательств, что соответствует пункту действия «резервировать лабораторные окна и собирать медианы».

### 7. CI Gate и локальный дым

— `ci/check_sm_perf.sh` — это каноническая точка входа CI. Он устанавливает `scripts/sm_perf.sh` для каждого режима в `SM_PERF_MODES` (по умолчанию `scalar auto neon-force`) и устанавливает `CARGO_NET_OFFLINE=true`, чтобы стенды детерминированно работали на образах CI.  
- `.github/workflows/sm-neon-check.yml` теперь вызывает гейт на бегуне macOS Arm64, поэтому каждый запрос на включение выполняет трио скалярного/автоматического/неонового усилия через тот же помощник, который используется локально; дополнительная линия Linux/Neoverse подключится, как только x86_64 захватит землю и базовые показатели прокси-сервера Neoverse будут обновлены при запуске с нуля.  
- Операторы могут локально переопределить список режимов: `SM_PERF_MODES="scalar" bash ci/check_sm_perf.sh` сокращает прогон до одного прохода для быстрого дымового теста, а дополнительные аргументы (например, `--tolerance 0.20`) пересылаются непосредственно в `scripts/sm_perf.sh`.  
- `make check-sm-perf` теперь оборачивает гейт для удобства разработчика; Задания CI могут напрямую вызывать сценарий, в то время как разработчики macOS используют цель make.  
- После получения базовых показателей Neoverse/x86_64 тот же сценарий подберет соответствующий JSON через логику автоматического обнаружения хоста, уже присутствующую в `scripts/sm_perf.sh`, поэтому в рабочих процессах не требуется никаких дополнительных подключений, кроме установки списка желаемых режимов для каждого пула хостов.

### 8. Помощник по ежеквартальному обновлению- Запустите `scripts/sm_perf_quarterly.sh --owner "<name>" --cpu-label "<label>" [--quarter YYYY-QN] [--output-root artifacts/sm_perf]`, чтобы создать каталог с четвертной отметкой, например `artifacts/sm_perf/2026-Q1/<label>/`. Помощник оборачивает `scripts/sm_perf_capture_helper.sh --matrix` и генерирует `capture_commands.sh`, `capture_plan.json` и `quarterly_plan.json` (метаданные владельца + квартала), поэтому операторы лаборатории могут планировать запуски без составления планов вручную.
- Выполните сгенерированный `capture_commands.sh` на целевом хосте, агрегируйте необработанные выходные данные с помощью `scripts/sm_perf_aggregate.py --output <dir>/aggregated.json` и переместите медианы в базовые JSON-файлы через `scripts/sm_perf_promote_baseline.py --out-dir crates/iroha_crypto/benches --overwrite`. Повторно запустите `ci/check_sm_perf.sh`, чтобы убедиться, что допуски остаются зелеными.
- При изменении аппаратного обеспечения или цепочек инструментов обновите допуски/примечания сравнения в `docs/source/crypto/sm_perf_baseline_comparison.md`, ужесточите допуски `ci/check_sm_perf.sh`, если новые медианы стабилизируются, и согласуйте любые пороговые значения информационной панели/предупреждений с новыми базовыми показателями, чтобы оперативные сигналы тревоги оставались значимыми.
- Зафиксируйте `quarterly_plan.json`, `capture_plan.json`, `capture_commands.sh` и агрегированный JSON вместе с базовыми обновлениями; прикрепляйте одни и те же артефакты к обновлениям статуса/дорожной карты для отслеживания.
