# SM хөтөлбөрийн архитектурын товч мэдээлэл

## Зорилго
Хятадын үндэсний криптографийг (SM2/SM3/SM4) Iroha v2 стек дээр нэвтрүүлэх техникийн төлөвлөгөө, ханган нийлүүлэлтийн сүлжээ, эрсдэлийн хил хязгаарыг тодорхойлохын зэрэгцээ тодорхой гүйцэтгэл, аудит хийх боломжтой байдлыг хадгалах.

## Хамрах хүрээ
- **Зөвшилцлийн чухал замууд:** `iroha_crypto`, `iroha`, `irohad`, IVM хост, Kotodama дотоод.
- **Үйлчлүүлэгчийн SDK болон багаж хэрэгсэл:** Rust CLI, Kagami, Python/JS/Swift SDK, генезисийн хэрэгслүүд.
- **Тохиргоо ба цуваа:** `iroha_config` товчлуурууд, Norito өгөгдлийн загварын шошго, манифест боловсруулах, multicodec шинэчлэлтүүд.
- **Туршилт ба дагаж мөрдөх:** Нэгж/өмч/хамтын ажиллагааны багц, Wycheproof бэхэлгээ, гүйцэтгэлийн профайл, экспорт/зохицуулалтын удирдамж. *(Төлөв байдал: RustCrypto-д тулгуурласан SM стекийг нэгтгэсэн; нэмэлт `sm_proptest` fuzz иж бүрдэл болон OpenSSL паритетийг өргөтгөсөн CI-д ашиглах боломжтой.)*

Хамрах хүрээнээс гадуур: PQ алгоритмууд, зөвшилцлийн зам дахь детерминистик бус хостын хурдатгал; wasm/`no_std` бүтээгдхүүн нь тэтгэвэрт гарсан.

- **Нэмэлт FFI:** OpenSSL 3.x үйлчилгээ үзүүлэгч API эсвэл Tongsuo баталгаажсан стек эсвэл техник хангамжийн хөдөлгүүр шаарддаг; consensus хоёртын хувилбарт онцлогтой, анхдагчаар идэвхгүй байна.### Үндсэн номын сангийн интеграцийн статус

Хэрэглээний шуурхай лавлагаа:
- **Гэрээ/туршилтын SM3 хэш:** `Sm3Digest::hash(b"...")` (Зэв) эсвэл Kotodama `sm::hash(input_blob)`. JSON 64 зургаан талт тэмдэгт хүлээж байна.
- **CLI-ээр дамжуулан SM4 AEAD:** `iroha tools crypto sm4 gcm-seal --key-hex <32 hex> --nonce-hex <24 hex> --plaintext-hex …` нь hex/base64 шифр текст+шошго хосыг гаргадаг. Тохирох `gcm-open` кодыг тайл.
- **Multicodec стрингүүд:** SM2 нийтийн түлхүүр/гарын үсэг нь `PublicKey::from_str`/`Signature::from_bytes`-ээр хүлээн зөвшөөрөгдсөн олон суурьт стрингээс/түлхүүр рүү задлан шинжилж, Norito манифест болон бүртгэлийн ID-г SM гарын үсэг зурсан хүмүүсийг авч явах боломжийг олгодог.

Өгөгдлийн загварын хэрэглэгчид SM4 түлхүүрүүд болон шошгуудыг түр зуурын блок гэж үзэх ёстой; Түүхий түлхүүрүүдийг гинж дээр хэзээ ч бүү хадгал. Аудит хийх шаардлагатай үед гэрээнд зөвхөн шифрлэгдсэн текст/шошгийн гаралт эсвэл үүсмэл мэдээлэл (жишээ нь, түлхүүрийн SM3) хадгалагдах ёстой.

SM-ийг мэддэг болон SM-ийн хөгжлийн бэрхшээлтэй үе тэнгийнхний хоорондын ялгаа.

--зөвшөөрөх-sm-гар барих-үл нийцэхгүй` and `--зөвшөөрөх-sm-openssl-preview-mismatch`, or the matching `--require-*`
хатуу хэрэгжилтийг сэргээх туг.#### OpenSSL/Tongsuo урьдчилан үзэх (`sm-ffi-openssl`)
- **Хамрах хүрээ.** OpenSSL-н ажиллах цагийн хүртээмжийг баталгаажуулж, OpenSSL-ээр дэмжигдсэн SM3 хэш, SM2 баталгаажуулалт болон SM4-GCM шифрлэлт/шифрийг тайлах боломжийг олгодог зөвхөн урьдчилан үзэх үйлчилгээ үзүүлэгчийн шифрийг (`OpenSslProvider`) бүтээдэг. Зөвшилцлийн хоёртын файлууд нь RustCrypto замыг үргэлжлүүлэн ашиглах ёстой; FFI backend нь захын баталгаажуулалт/гарын үсэг зурах нисгэгчдийг хатуу сонгон авдаг.
- **Урьдчилсан нөхцөлүүдийг бүрдүүлэх.** `cargo build -p iroha_crypto --features "sm sm-ffi-openssl"`-ээр эмхэтгэж, OpenSSL/Tongsuo 3.0+ (SM2/SM3/SM4 дэмжлэгтэй `libcrypto`)-ийн эсрэг хэрэгслийн гинжний холбоосыг баталгаажуулна уу. Статик холболт хийхийг зөвлөдөггүй; операторын удирддаг динамик сангуудыг илүүд үздэг.
- **Хөгжүүлэгчийн утааны тест.** `scripts/sm_openssl_smoke.sh`-г ажиллуулж `cargo check -p iroha_crypto --features "sm sm-ffi-openssl"`-г ажиллуулж, дараа нь `cargo test -p iroha_crypto --features "sm sm-ffi-openssl" --test sm_openssl_smoke -- --nocapture`; OpenSSL ≥3 хөгжүүлэлтийн гарчиг байхгүй (эсвэл `pkg-config` байхгүй) үед туслагч автоматаар алгасаж, утааны гаралтыг илрүүлдэг тул хөгжүүлэгчид SM2 баталгаажуулалт нь Rust хэрэгжүүлэлт рүү буцаж орсон эсэхийг харах боломжтой болно.
- **Зэврийн шат.** `openssl_sm` модуль нь одоо SM3 хэш, SM2 баталгаажуулалт (ZA prehash + SM2 ECDSA) болон SM4 GCM шифрлэлт/шифрлэлтийг OpenSSL-ээр дамжуулж, урьдчилан харах унтраалга болон буруу түлхүүр/nonce/tag уртыг хамарсан бүтэцтэй алдаатай; SM4 CCM нь нэмэлт FFI залгах хүртэл зөвхөн зэвэрсэн хэвээр байна.
- **Алгасах зан үйл.** OpenSSL ≥3.0 толгой хэсэг эсвэл номын сан байхгүй үед утааны тест алгасах баннер хэвлэдэг (`-- --nocapture`-ээр) гэхдээ CI нь хүрээлэн буй орчны цоорхойг жинхэнэ регрессээс ялгах боломжтой хэвээр байна.
- **Ажиллах үеийн хамгаалалтын хаалтууд.** OpenSSL урьдчилан харах нь анхдагчаар идэвхгүй байна; FFI замыг ашиглахыг оролдохоос өмнө тохиргоогоор (`crypto.enable_sm_openssl_preview` / `OpenSslProvider::set_preview_enabled(true)`) идэвхжүүлнэ үү. Үйлчилгээ үзүүлэгч төгсөх хүртэл үйлдвэрлэлийн кластеруудыг зөвхөн баталгаажуулах горимд байлга (`sm2`-г `allowed_signing`-ээс хас), RustCrypto-ийн тодорхойлогч нөөцөд найдаж, гарын үсэг зурах нисгэгчдийг тусгаарлагдсан орчинд хязгаарлаарай.
- **Баглаа боодлын хяналтын хуудас.** Байршуулах манифест дахь үйлчилгээ үзүүлэгчийн хувилбар, суулгах зам, бүрэн бүтэн байдлын хэшийг баримтжуулна уу. Операторууд зөвшөөрөгдсөн OpenSSL/Tongsuo бүтээцийг суулгаж, үйлдлийн системийн итгэмжлэгдсэн дэлгүүрт бүртгүүлэх (шаардлагатай бол) болон засвар үйлчилгээний цонхны ард шинэчлэлт хийх суулгацын скриптүүдийг өгөх ёстой.
- **Дараах алхмууд.** Ирээдүйн үе шатууд нь тодорхойлогч SM4 CCM FFI холболтууд, CI утааны ажил (`ci/check_sm_openssl_stub.sh`-г үзнэ үү) болон телеметрийг нэмнэ. `roadmap.md` дээрх SM-P3.1.x дагуу явцыг хянах.

## Тодорхойлогч бодлого
- SDK-н бүх SM2 гарын үсэг зурах замууд болон нэмэлт хост гарын үсэг зурахад RFC6979-аас үүсэлтэй nonnces-ийг хэрэгжүүлэх; Баталгаажуулагчид зөвхөн каноник r∥s кодчилолуудыг хүлээн зөвшөөрдөг.
- Хяналтын хавтгайн холбоо (урсгал) Ed25519 хэвээр байна; SM2 нь өгөгдлийн түвшний гарын үсгээр хязгаарлагдах бөгөөд хэрэв засаглал өргөжүүлэхийг зөвшөөрөөгүй бол.
- Intrinsics (ARM SM3/SM4) нь ажиллах үеийн онцлогийг илрүүлэх, програм хангамжийн нөөцийг сэргээх замаар тодорхойлогч баталгаажуулалт/хэш үйлдлээр хязгаарлагдсан.

## Norito & кодчилолын төлөвлөгөө
1. `iroha_data_model` дахь алгоритмын тоонуудыг `Sm2PublicKey`, `Sm2Signature`, `Sm3Digest`, `Sm4Key`-ээр өргөтгө.
2. DER-ийн тодорхой бус байдлаас зайлсхийхийн тулд SM2 гарын үсгийг том-эндиан тогтмол өргөнтэй `r∥s` массив (32+32 байт) болгон цуврал болгох; хувиргалтыг адаптеруудад зохицуулдаг. *(Гүйцэтгэсэн: `Sm2Signature` туслахуудад хэрэгжсэн; Norito/JSON хоёр талын аяллыг газар дээр нь суулгасан.)*
3. Олон формат ашиглаж байгаа бол олон кодлогчийн танигчийг (`sm3-256`, `sm2-pub`, `sm4-key`) бүртгэж, бэхэлгээ, баримт бичгийг шинэчилнэ үү. *(Явц: `sm2-pub` түр код `0x1306` одоо үүсмэл түлхүүрээр баталгаажсан; SM3/SM4 кодууд эцсийн даалгаврыг хүлээж байгаа, `sm_known_answers.toml`-ээр хянагддаг.)*
4. Хоёр талын аяллыг хамарсан Norito алтан тестийг шинэчилж, алдаатай кодчилол (богино/урт r эсвэл s, буруу муруй параметр) -ээс татгалзах.## Хост ба VM-ийг нэгтгэх төлөвлөгөө (SM-2)
1. Одоо байгаа ГОСТ хэш шимийг тусгал болгон хост талын `sm3_hash` системийн дуудлагыг хэрэгжүүлэх; `Sm3Digest::hash`-г дахин ашиглах ба детерминистик алдааны замыг ил гаргах. *(Газарласан: хост Blob TLV-г буцаана; `DefaultHost` хэрэгжилт болон `sm_syscalls.rs` регрессийг үзнэ үү.)*
2. VM системийн дуудлагын хүснэгтийг `sm2_verify`-ээр өргөтгөх ба энэ нь каноник r∥s гарын үсэг хүлээн авах, ялгах ID-г баталгаажуулах, бүтэлгүйтлийг детерминист буцаах код руу буулгах. *(Дууссан: хост + Kotodama үндсэн буцаах `1/0`; регрессийн иж бүрдэл нь одоо таслагдсан гарын үсэг, алдаатай нийтийн түлхүүрүүд, blob биш TLV-үүд болон UTF-8/хоосон/тохироогүй Hyperledger ачааллыг хамардаг.*)
3. `sm4_gcm_seal`/`sm4_gcm_open` (мөн сонголтоор CCM) системийн дуудлагыг тодорхой бус/шошготой хэмжээстэй (RFC 8998) өгнө үү. *(Дууссан: GCM нь тогтмол 12 байт nonces + 16 байт шошгуудыг ашигладаг; CCM нь `r14`-ээр хянагддаг {4,6,8,10,12,14,16} шошгоны урттай 7-13 байт байтыг дэмждэг; Kotodama зэрэг нь Kotodama ба илэрхий `sm::seal_ccm/open_ccm`.) Хөгжүүлэгчийн гарын авлагад дахин ашиглахгүй байх бодлогыг баримтжуулна.*
4. Эерэг болон сөрөг тохиолдлуудыг хамарсан Kotodama утсан гэрээ болон IVM интеграцийн тест (өөрчлөгдсөн шошго, алдаатай гарын үсэг, дэмжигдээгүй алгоритм). *(SM3/SM2/SM4-ийн хостын регрессийг `crates/ivm/tests/kotodama_sm_syscalls.rs` толин тусгах замаар хийсэн.)*
5. Системийн дуудлагын зөвшөөрөгдсөн жагсаалт, бодлого, ABI баримтуудыг (`crates/ivm/docs/syscalls.md`) шинэчилж, шинэ оруулгуудыг нэмсний дараа хэшлэгдсэн манифестуудыг сэргээнэ үү.

## Тохиргооны утас
- `crypto.allowed_signing`, `crypto.default_hash`, `crypto.sm2_distid_default`, нэмэлт `crypto.enable_sm_openssl_preview`-ийг `iroha_config` дээр нэмнэ. Дата-загварын сантехник нь крипто хайрцагт тусгагдсан эсэхийг шалгаарай (`iroha_data_model` нь `sm` → `iroha_crypto/sm`-г харуулж байна).
- Зөвшөөрөгдөх алгоритмуудыг тодорхойлох манифест/генезисийн файлуудыг элсэлтийн бодлогод тохируулна уу; удирдлагын хавтгай нь анхдагчаар Ed25519 хэвээр байна.### CLI & SDK ажил (SM-3)
1. **Torii CLI** (`crates/iroha_cli`): SM2 keygen/импорт/экспорт (мэдэгдэж байгаа), SM3 хэшлэх туслахууд болон SM4 AEAD шифрлэх/тайлах командуудыг нэмнэ үү. Интерактив сануулга болон баримт бичгийг шинэчилнэ үү.
2. **Genesis tooling** (`xtask`, `scripts/`): зөвшөөрөгдсөн гарын үсэг зурах алгоритмууд болон өгөгдмөл хэшүүдийг зарлах боломжийг манифестуудад зөвшөөрөх, хэрэв SM-г тохирох тохиргооны товчлуургүйгээр идэвхжүүлсэн бол хурдан бүтэлгүйтдэг. *(Дууссан: `RawGenesisTransaction` нь одоо `crypto` блоктой `default_hash`/`allowed_signing`/`sm2_distid_default`; `ManifestCrypto::validate` болон `ManifestCrypto::validate` болон reject-ийн тохиргоонууд I10220 defaults/genesis manifest нь хормын хувилбарыг сурталчилдаг.)*
3. **SDK гадаргуу**:
   - Зэв (`iroha_client`): SM2 гарын үсэг зурах/баталгаажуулах туслахууд, SM3 хэшинг, SM4 AEAD боодолуудыг тодорхойлогч өгөгдмөлтэй харуулах.
   - Python/JS/Swift: Rust API-г тусгах; Хэл хоорондын туршилтанд `sm_known_answers.toml` дахь шаталсан бэхэлгээг дахин ашиглах.
4. CLI/SDK хурдан эхлүүлэхэд SM-г идэвхжүүлэх операторын ажлын явцыг баримтжуулж, JSON/YAML тохиргоог шинэ алгоритмын шошготой эсэхийг баталгаажуулна уу.

#### CLI явц
- `cargo run -p iroha_cli --features sm -- crypto sm2 keygen --distid CN12345678901234` одоо `client.toml` (`public_key_config`, `private_key_hex`, `distid`)-ийн хамт SM2 түлхүүрийн хослолыг тодорхойлсон JSON ачааллыг ялгаруулж байна. Энэ команд нь `--seed-hex`-г тодорхойлогч үүсгэх ба хостуудын ашигладаг RFC 6979 деривацийг тусгадаг.
- `cargo xtask sm-operator-snippet --distid CN12345678901234` нь түлхүүр ген/экспортын урсгалыг ороож, ижил `sm2-key.json`/`client-sm2.toml` гаралтыг нэг алхамаар бичнэ. `--json-out <path|->` / `--snippet-out <path|->` ашиглан файлуудыг дахин чиглүүлэх эсвэл stdout руу дамжуулахын тулд автоматжуулалтын `jq` хамаарлыг арилгана.
- `iroha_cli tools crypto sm2 import --private-key-hex <hex> [--distid ...]` нь одоо байгаа материалаас ижил мета өгөгдлийг гаргаж авдаг тул операторууд элсэлтийн өмнө ялгах ID-г баталгаажуулах боломжтой.
- `iroha_cli tools crypto sm2 export --private-key-hex <hex> --emit-json` нь тохиргооны хэсгийг (`allowed_signing`/`sm2_distid_default` удирдамжийг оруулаад) хэвлэх ба сонголтоор скрипт хийхэд зориулж JSON түлхүүрийн бүртгэлийг дахин гаргадаг.
- `iroha_cli tools crypto sm3 hash --data <string>` нь дурын ачааллыг хэш болгодог; `--data-hex` / `--file` хоёртын оролтыг хамардаг бөгөөд команд нь манифест хэрэгсэлд зориулсан hex болон base64 дижестийг хоёуланг нь мэдээлдэг.
- `iroha_cli tools crypto sm4 gcm-seal --key-hex <KEY> --nonce-hex <NONCE> --plaintext-hex <PT>` (болон `gcm-open`) хостын SM4-GCM туслахуудыг ороож, `ciphertext_hex`/`tag_hex` буюу энгийн текстийн ачааллыг гадаргын гадаргуутай болгодог. `sm4 ccm-seal` / `sm4 ccm-open` нь нэг удаа урт (7–13 байт) болон шошгоны урт (4,6,8,10,12,14,16) баталгаажуулалт бүхий CCM-д ижил UX-ээр хангана; Хоёр тушаал хоёулаа диск рүү түүхий байтыг ялгаруулдаг.## Туршилтын стратеги

#### Архитектур хоорондын зураг авалтыг шалгах хуудас
- `scripts/sm_perf_capture_helper.sh` **зорилтот машин дээр** ажиллуул (x86_64 ажлын станц, Neoverse ARM сервер гэх мэт). `--cpu-label <host>`-ийг дамжуулж, зураг авалтыг дарж, (матриц горимд ажиллаж байх үед) лабораторийн хуваарьт зориулж үүсгэсэн төлөвлөгөө/командуудыг урьдчилан бөглөнө үү. Туслах нь горимд хамаарах командуудыг хэвлэдэг:
  1. Criterion багцыг зөв функцээр гүйцэтгэх, мөн
  2. `crates/iroha_crypto/benches/sm_perf_baseline_${arch}_${os}_${mode}.json` руу медиануудыг бич.
- Эхлээд скаляр суурь шугамыг аваад дараа нь `auto` (мөн AArch64 платформ дээр `neon-force`)-д зориулсан туслахыг дахин ажиллуулна уу. Шүүгчид JSON мета өгөгдлөөс хостын дэлгэрэнгүй мэдээллийг хянах боломжтой болохын тулд утга учиртай `SM_PERF_CPU_LABEL` ашигла.
- Ажиллуулах бүрийн дараа түүхий `target/criterion/sm_perf*` лавлахыг архивлаж, үүсгэсэн суурь үзүүлэлтүүдийн хамт PR-д оруулна. Хоёр дараалсан гүйлт тогтворжсон даруйд жишиг үзүүлэлт бүрийн хүлцлийг чангална уу (лавлагааны форматыг `sm_perf_baseline_aarch64_macos_*.json`-с үзнэ үү).
- Энэ хэсэгт медиан + хүлцэлийг тэмдэглэж, шинэ архитектурыг хамрах үед `status.md`/`roadmap.md`-г шинэчил. Линуксийн үндсэн үзүүлэлтүүдийг одоо `neoverse-proxy-macos` зураг авалтаас шалгасан (мета өгөгдөл нь aarch64-unknown-linux-gnu хаалга руу экспортлохыг тэмдэглэсэн); Neoverse/x86_64 нүцгэн металл хостууд дээр лабораторийн үүрүүд бэлэн болсон үед дагалдах хэрэгсэл болгон дахин ажиллуулаарай.

** Тодорхойлолт ба байршуулах заавар**
- Intrinsics нь ажиглагдаж болох үр дүнг хэзээ ч өөрчилдөггүй—`sm_accel` нь хурдасгасан зам боломжгүй үед `None`-г буцаадаг тул скаляр туслагч ажилладаг. Тиймээс зөвшилцлийн кодын замууд нь скаляр хэрэгжилт зөв байвал тодорхойлогддог.
- NEON замыг ашигласан эсэх талаар бизнесийн логикийг бүү **бүгээрэй. Хурдасгалыг зөвхөн төгс зөвлөмж гэж үзэж, статусыг зөвхөн телеметрээр (жишээ нь, `sm_intrinsics_enabled` хэмжигч) харуулна уу.
- SM кодыг хүрсний дараа `ci/check_sm_perf.sh` (эсвэл `make check-sm-perf`)-г үргэлж ажиллуулснаар Шалгуур үзүүлэлт нь үндсэн JSON бүрт суулгагдсан хүлцлийг ашиглан скаляр болон хурдасгасан замыг хоёуланг нь баталгаажуулдаг.
- Харьцуулах эсвэл дибаг хийхдээ эмхэтгэх хугацааны флагуудаас `crypto.sm_intrinsics` тохиргооны товчлуурыг илүүд үзэх; `sm-neon-force`-ээр дахин эмхэтгэх нь скаляр буцаалтыг бүхэлд нь идэвхгүй болгодог бол `force-enable` зүгээр л ажиллах цагийн илрүүлэлтийг түлхэж өгдөг.
- Сонгосон бодлогыг хувилбарын тэмдэглэлд баримтжуулна уу: үйлдвэрлэлийн бүтээцүүд нь `Auto` дээрх бодлогыг орхих ёстой бөгөөд баталгаажуулагч бүр техник хангамжийн чадавхийг бие даан олж илрүүлэхийн зэрэгцээ ижил хоёртын олдворуудыг хуваалцсан хэвээр байх болно.

- Одоогийн бүтээцүүд x86_64 дээр зөвхөн RustCrypto скаляр замыг л дамжуулдаг; `sm`-г идэвхжүүлсэн хэвээр байгаа хэдий ч SM-4c.1b-г буух хүртэл гадны AVX2/VAES цөмүүдийг **бүү** бүү хий. Ажиллах цагийн бодлого ARM-г тусгадаг: өгөгдмөл нь `Auto`, `crypto.sm_intrinsics`-г хүндэтгэх, мөн ижил телеметрийн хэмжигчийг гадаргуу дээр харуулна.
- Linux/x86_64 зураг авалтыг бичих шаардлагатай хэвээр байна; Тус техник хангамж дээр туслагчийг дахин ашиглаад `sm_perf_baseline_x86_64_unknown_linux_gnu_{mode}.json` руу медиануудыг Rosetta-н суурь болон дээрх хүлцлийн газрын зургийн хажууд буулгана.** Түгээмэл бэрхшээл**
1. **Виртуалжуулсан ARM тохиолдлууд:** Олон үүл нь NEON-г ил гаргадаг боловч `sm4_neon::is_supported()`-ийн шалгадаг SM4/AES өргөтгөлүүдийг нуудаг. Эдгээр орчинд скаляр замыг хүлээж, зохих үндсэн үзүүлэлтүүдийг авна.
2. **Хэсэгчилсэн хүчингүй болгох:** Ажиллуулах хооронд `crypto.sm_intrinsics`-ийн тогтвортой утгуудыг холих нь оновчтой уншилтад хүргэдэг. Туршилтын тасалбар дээр төлөвлөсөн хүчингүй болгохыг баримтжуулж, шинэ суурь үзүүлэлтүүдийг авахын өмнө тохиргоог дахин тохируулна уу.
3. **CI паритет:** Зарим macOS ажиллуулагч нь NEON идэвхтэй байх үед тоолуур дээр суурилсан төгс түүвэрлэлтийг зөвшөөрдөггүй. `scripts/sm_perf_capture_helper.sh` гаралтыг PR-д хавсаргасан байлгаснаар хянагч гүйгч тэдгээр тоолуурыг нуусан ч хурдасгасан замыг хэрэгжүүлсэн гэдгийг батлах боломжтой.
4. **Ирээдүйн ISA хувилбарууд (SVE/SVE2):** Одоогийн цөмүүд нь NEON эгнээний хэлбэртэй байна. SVE/SVE2 руу шилжүүлэхийн өмнө `sm_accel::NeonPolicy`-г тусгай хувилбараар сунгаж, бид CI, телеметр болон операторын товчлууруудыг зэрэгцүүлэн хадгалах боломжтой.

SM-5a/SM-4c.1-ийн дагуу хянасан үйлдлүүд нь CI нь шинэ архитектур бүрт паритет нотолгоог олж авах ба замын зураглал нь Neoverse/x86 суурь болон NEON-с скаляр хүлцэл нэгдэх хүртэл 🈺 хэвээр байх болно.

## Нийцлийн хавсралт (Улсын арилжааны криптограф)

- **Ангилал:** SM2/SM3/SM4 нь Хятадын *төрийн арилжааны криптографийн* горимын дагуу тээвэрлэгдэнэ (БНХАУ-ын Криптографийн тухай хууль, 3-р зүйл). Эдгээр алгоритмуудыг Iroha програм хангамжид хүргэх нь төслийг үндсэн/нийтлэг (төрийн нууц) давхаргад **байруулдаггүй, харин БНХАУ-ын байршуулалтад ашиглаж буй операторууд арилжааны крипто файл болон MLPS-ийн үүргийг дагаж мөрдөх ёстой.
- **Стандартуудын удам угсаа:** Олон нийтийн баримт бичгийг GM/T үзүүлэлтүүдийн албан ёсны GB/T хөрвүүлэлтүүдтэй нийцүүлнэ:

| Алгоритм | GB/T лавлагаа | GM/T гарал үүсэл | Тэмдэглэл |
|----------|----------------|-------------|-------|
| SM2 | GB/T32918 (бүх хэсгүүд) | GM/T0003 | ECC тоон гарын үсэг + түлхүүр солилцох; Iroha нь үндсэн зангилаа дахь баталгаажуулалт болон SDK-д тодорхойлогч гарын үсэг зурдаг. |
| SM3 | GB/T32905 | GM/T0004 | 256 битийн хэш; скаляр болон ARMv8 хурдасгасан замууд дээр детерминист хэш хийх. |
| SM4 | GB/T32907 | GM/T0002 | 128 битийн блок шифр; Iroha нь GCM/CCM-ийн туслагчаар хангадаг бөгөөд хэрэгжилтийн хооронд том-эндиан тэгш байдлыг баталгаажуулдаг. |- **Чадварын манифест:** Torii `/v1/node/capabilities` төгсгөлийн цэг нь дараах JSON хэлбэрийг сурталчилдаг тул операторууд болон хэрэгслүүд нь SM манифестийг программчлан ашиглах боломжтой:

```json
{
  "supported_abi_versions": [1],
  "default_compile_target": 1,
  "data_model_version": 1,
  "crypto": {
    "sm": {
      "enabled": true,
      "default_hash": "sm3-256",
      "allowed_signing": ["ed25519"],
      "sm2_distid_default": "1234567812345678",
      "openssl_preview": false,
      "acceleration": {
        "scalar": true,
        "neon_sm3": false,
        "neon_sm4": false,
        "policy": "auto"
      }
    }
  }
}
```

CLI дэд команд `iroha runtime capabilities` нь ижил ачааллыг орон нутагт гаргаж, нийцлийн нотолгоог цуглуулах JSON зарын хажууд нэг мөрт хураангуйг хэвлэдэг.

- **Баримт бичгийн нийлүүлэлт:** дээрх алгоритм/стандартуудыг тодорхойлсон хувилбарын тэмдэглэл болон SBOM-г нийтэлж, бүрэн дагаж мөрдөх товч мэдээллийг (`sm_chinese_crypto_law_brief.md`) хувилбарын олдворуудтай хамт хадгалснаар операторууд үүнийг мужийн захиргаанд хавсаргах боломжтой. мэдүүлэг.【docs/source/crypto/sm_chinese_crypto_law_brief.md:59】
- **Операторыг хүлээлгэн өгөх:** MLPS2.0/GB/T39786-2021 нь крипто хэрэглээний үнэлгээ, SM түлхүүрийн удирдлагын SOP болон ≥6 жилийн нотлох баримтыг хадгалах шаардлагатайг байршуулагчдад сануулах; тэдгээрийг дагаж мөрдөх товчлол дахь операторын шалгах хуудас руу зааж өгнө үү.【docs/source/crypto/sm_chinese_crypto_law_brief.md:43】【docs/source/crypto/sm_chinese_crypto_law_brief.md:74】

- **Үзэгчид:** Crypto WG-ийн үндсэн гишүүд, Хувилбарын инженерчлэл, Аюулгүй байдлын хяналтын зөвлөл, SDK хөтөлбөрийн удирдагчид.

- **Суваг:** Долоо хоног тутмын Crypto WG синхрончлолын хөтөлбөр + үйлдлийн зүйлийг нэгтгэн дүгнэж, түгжээг шинэчлэх, хараат байдалд оруулах зөвшөөрөл хүссэн имэйл (ноорог 2025-01-19-нд тараагдсан).
- **Эзэмшигч:** Crypto АХ-ын удирдагч (төлөөлөгчийг хүлээн зөвшөөрөх боломжтой).
