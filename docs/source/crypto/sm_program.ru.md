# Краткое описание архитектуры программы SM

## Цель
Определите технический план, состояние цепочки поставок и границы рисков для внедрения китайской национальной криптографии (SM2/SM3/SM4) в стеке Iroha v2, сохраняя при этом детерминированное выполнение и возможность аудита.

## Область применения
- **Критические для консенсуса пути:** `iroha_crypto`, `iroha`, `irohad`, IVM хост, встроенные функции Kotodama.
- **Клиентские SDK и инструменты:** Rust CLI, Kagami, Python/JS/Swift SDK, утилиты Genesis.
- **Конфигурация и сериализация:** ручки `iroha_config`, теги модели данных Norito, обработка манифестов, обновления мультикодеков.
- **Тестирование и соответствие требованиям:** наборы модулей/имущества/взаимодействия, средства защиты Wycheproof, профилирование производительности, руководство по экспорту и нормативным требованиям. *(Состояние: стек SM, поддерживаемый RustCrypto, объединен; для расширенной CI доступен дополнительный пакет фаззинга `sm_proptest` и программа контроля четности OpenSSL.)*

Вне области применения: алгоритмы PQ, недетерминированное ускорение хоста в консенсусных путях; Сборки wasm/`no_std` устарели.

## Выбранные библиотеки
- **Основной:** ящики RustCrypto (`sm2`, `sm3`, `sm4`) с включенной функцией `rfc6979` и SM3, привязанным к детерминированным одноразовым номерам.
- **Дополнительно FFI:** API-интерфейс поставщика OpenSSL 3.x или Tongsuo для развертываний, требующих сертифицированных стеков или аппаратных механизмов; функционально ограничен и отключен по умолчанию в согласованных двоичных файлах.### Статус интеграции базовой библиотеки

- `sm_accel` теперь обнаруживает AArch64 NEON во время выполнения и обрабатывает перехватчики SM3/SM4 через диспетчеризацию x86_64/RISC-V, учитывая при этом ручку конфигурации `crypto.sm_intrinsics` (`auto`/`force-enable`/`force-disable`). Когда векторные серверные части отсутствуют, диспетчер по-прежнему маршрутизирует скалярный путь RustCrypto, поэтому стенды и переключатели политик ведут себя согласованно на всех хостах.【crates/iroha_crypto/src/sm.rs:702】【crates/iroha_crypto/src/sm.rs:733】

Краткая справка по использованию:
- **Хеширование SM3 в контрактах/тестах:** `Sm3Digest::hash(b"...")` (Rust) или Kotodama `sm::hash(input_blob)`. JSON ожидает 64 шестнадцатеричных символа.
- **SM4 AEAD через CLI:** `iroha tools crypto sm4 gcm-seal --key-hex <32 hex> --nonce-hex <24 hex> --plaintext-hex …` дает пары шестнадцатеричный/base64 зашифрованный текст+тег. Расшифруйте с помощью соответствующего `gcm-open`.
- **Мультикодековые строки:** открытые ключи/подписи SM2 анализируются из/в многобазовую строку, принятую `PublicKey::from_str`/`Signature::from_bytes`, что позволяет манифестам Norito и идентификаторам учетных записей переносить подписантов SM.

Потребители модели данных должны рассматривать ключи и теги SM4 как временные объекты; никогда не сохраняйте необработанные ключи в цепочке. Контракты должны хранить только выходные данные зашифрованного текста/тегов или производные дайджесты (например, SM3 ключа), когда требуется аудит.

Сетевая политика теперь предоставляет `network.require_sm_handshake_match` и
`network.require_sm_openssl_preview_match` (оба по умолчанию — `true`). Сброс любого флага позволяет
смешанные развертывания, при которых наблюдатели, использующие только Ed25519, подключаются к валидаторам с поддержкой SM; несоответствия
зарегистрирован по адресу `WARN`, но узлы консенсуса должны сохранять настройки по умолчанию включенными, чтобы предотвратить случайное
расхождение между узлами, поддерживающими и не поддерживающими SM.
CLI отображает эти переключатели через обновление рукопожатия `iroha_cli app sorafs
--allow-sm-handshake-mismatch` and `--allow-sm-openssl-preview-mismatch`, or the matching `--require-*`
флаги для восстановления строгого соблюдения.#### Предварительная версия OpenSSL/Tongsuo (`sm-ffi-openssl`)
- **Область.** Создает прошивку поставщика, доступную только для предварительной версии (`OpenSslProvider`), которая проверяет доступность среды выполнения OpenSSL и обеспечивает хеширование SM3 на основе OpenSSL, проверку SM2 и шифрование/дешифрование SM4-GCM, сохраняя при этом согласие. Консенсусные двоичные файлы должны продолжать использовать путь RustCrypto; Серверная часть FFI строго поддерживает пилотные проекты проверки/подписания границ.
- **Предварительные требования для сборки.** Скомпилируйте с помощью `cargo build -p iroha_crypto --features "sm sm-ffi-openssl"` и убедитесь, что цепочка инструментов связана с OpenSSL/Tongsuo 3.0+ (`libcrypto` с поддержкой SM2/SM3/SM4). Статическое связывание не рекомендуется; предпочитают динамические библиотеки, управляемые оператором.
- **Дымовой тест разработчика.** Запустите `scripts/sm_openssl_smoke.sh`, чтобы выполнить `cargo check -p iroha_crypto --features "sm sm-ffi-openssl"`, а затем `cargo test -p iroha_crypto --features "sm sm-ffi-openssl" --test sm_openssl_smoke -- --nocapture`; помощник автоматически пропускает работу, когда заголовки разработки OpenSSL ≥3 недоступны (или отсутствует `pkg-config`) и отображает вывод дыма, чтобы разработчики могли видеть, была ли выполнена проверка SM2 или она вернулась к реализации Rust.
- **Rust scaffolding.** Модуль `openssl_sm` теперь маршрутизирует хеширование SM3, проверку SM2 (предварительный хэш ZA + SM2 ECDSA) и шифрование/дешифрование SM4 GCM через OpenSSL со структурированными ошибками, охватывающими переключение предварительного просмотра и недопустимые длины ключей/nonce/тегов; SM4 CCM остается чистым только на ржавчине до тех пор, пока не приземлятся дополнительные прокладки FFI.
- **Поведение при пропуске.** Когда заголовки или библиотеки OpenSSL ≥3.0 отсутствуют, дымовой тест печатает баннер пропуска (через `-- --nocapture`), но все равно завершается успешно, поэтому CI может отличить пробелы в среде от подлинных регрессий.
- **Ограждения времени выполнения.** Предварительный просмотр OpenSSL отключен по умолчанию; включите его через конфигурацию (`crypto.enable_sm_openssl_preview` / `OpenSslProvider::set_preview_enabled(true)`), прежде чем пытаться использовать путь FFI. Держите производственные кластеры в режиме только проверки (опустите `sm2` из `allowed_signing`) до тех пор, пока поставщик не завершит работу, положитесь на детерминированный резервный вариант RustCrypto и ограничьте пилотные проекты подписи изолированными средами.
- **Контрольный список упаковки.** Задокументируйте версию поставщика, путь установки и хэши целостности в манифестах развертывания. Операторы должны предоставить сценарии установки, которые устанавливают утвержденную сборку OpenSSL/Tongsuo, регистрируют ее в хранилище доверенных сертификатов ОС (при необходимости) и закрепляют обновления за окнами обслуживания.
- **Следующие шаги.** Будущие этапы добавят детерминированные привязки SM4 CCM FFI, дымовые задания CI (см. `ci/check_sm_openssl_stub.sh`) и телеметрию. Отслеживайте прогресс в SM-P3.1.x в `roadmap.md`.

следить за поэтапным процессом в

флот.

## Детерминированная политика
- Применять одноразовые номера, производные от RFC6979, для всех путей подписи SM2 в SDK и необязательной подписи хоста; верификаторы принимают только канонические кодировки r∥s.
- Связь в плоскости управления (потоковая передача) остается Ed25519; SM2 ограничен подписями уровня данных, если руководство не одобрит расширение.
- Внутренние функции (ARM SM3/SM4) ограничены детерминированными операциями проверки/хеширования с обнаружением функций во время выполнения и резервным программным обеспечением.

## Norito и план кодирования
1. Расширьте перечисления алгоритмов в `iroha_data_model` с помощью `Sm2PublicKey`, `Sm2Signature`, `Sm3Digest`, `Sm4Key`.
2. Сериализуйте подписи SM2 как массивы `r∥s` фиксированной ширины с обратным порядком байтов (32+32 байта), чтобы избежать неоднозначности DER; преобразования обрабатываются в адаптерах. *(Выполнено: реализовано в помощниках `Sm2Signature`; Norito/JSON туда и обратно.)*
3. #### прогресс CLI

`sm_accel` (см. `crates/iroha_crypto/src/sm.rs:739`) обеспечивает уровень диспетчеризации во время выполнения для помощников SM3/SM4, поддерживаемых NEON. Функция охраняется на трех уровнях:| Слой | Контроль | Заметки |
|-------|---------|-------|
| Время компиляции | `--features sm` (теперь `sm-neon` автоматически подтягивается к `aarch64`) или `sm-neon-force` (тесты/тесты) | Собирает модули NEON и связывает `sm3-neon`/`sm4-neon`. |
| Автоопределение среды выполнения | `sm4_neon::is_supported()` | Верно только для процессоров, которые предоставляют эквиваленты AES/PMULL (например, Apple M-серии, Neoverse V1/N2). Виртуальные машины, маскирующие NEON или FEAT_SM4, возвращаются к скалярному коду. |
| Переопределение оператора | `crypto.sm_intrinsics` (`auto`/`force-enable`/`force-disable`) | Отправка на основе конфигурации применяется при запуске; используйте `force-enable` только для профилирования в доверенных средах и отдавайте предпочтение `force-disable` при проверке скалярных резервных вариантов. |

**Детерминизм и руководство по развертыванию**
- Внутренние функции никогда не изменяют наблюдаемые результаты — `sm_accel` возвращает `None`, когда ускоренный путь недоступен, поэтому запускается скалярный помощник. Таким образом, согласованные пути кода остаются детерминированными, пока скалярная реализация правильна.
– **Не** проверяйте бизнес-логику, использовался ли путь NEON. Рассматривайте ускорение исключительно как подсказку о производительности и отображайте статус только с помощью телеметрии (например, датчика `sm_intrinsics_enabled`).
- Всегда запускайте `ci/check_sm_perf.sh` (или `make check-sm-perf`) после касания кода SM, чтобы система Criterion проверяла как скалярные, так и ускоренные пути, используя допуски, встроенные в каждый базовый JSON.
- При тестировании или отладке отдавайте предпочтение регулятору конфигурации `crypto.sm_intrinsics`, а не флагам времени компиляции; перекомпиляция с `sm-neon-force` полностью отключает скалярный резерв, тогда как `force-enable` просто подталкивает обнаружение во время выполнения.
- Задокументируйте выбранную политику в примечаниях к выпуску: в производственных сборках политика должна оставаться в `Auto`, позволяя каждому валидатору независимо обнаруживать возможности оборудования, сохраняя при этом одни и те же двоичные артефакты.
- Избегайте поставки двоичных файлов, в которых смешаны статически связанные встроенные функции поставщиков (например, сторонние библиотеки SM4), если они не соблюдают один и тот же поток отправки и тестирования — в противном случае регрессии производительности не будут обнаружены нашими базовыми инструментами.#### x86_64 Базовая версия Rosetta (Apple M3 Pro; снято 1 декабря 2025 г.)

#### x86_64 / другие цели, отличные от aarch64
- Текущие сборки по-прежнему поставляются только с детерминированным скалярным путем RustCrypto на x86_64; оставьте `sm` включенным, но **не** внедряйте внешние ядра AVX2/VAES до тех пор, пока не появится SM-4c.1b. Политика времени выполнения отражает ARM: по умолчанию используется `Auto`, учитывается `crypto.sm_intrinsics` и используются те же датчики телеметрии.
- Захваты Linux/x86_64 еще предстоит записать; повторно используйте помощник на этом оборудовании и поместите медианы в `sm_perf_baseline_x86_64_unknown_linux_gnu_{mode}.json` вместе с базовыми линиями Rosetta и картой допусков выше.**Распространённые ошибки**
1. **Виртуализированные экземпляры ARM:** Многие облака предоставляют NEON, но скрывают расширения SM4/AES, которые проверяет `sm4_neon::is_supported()`. Ожидайте скалярный путь в этих средах и соответствующим образом фиксируйте базовые показатели производительности.
2. **Частичное переопределение:** Смешение постоянных значений `crypto.sm_intrinsics` между прогонами приводит к несогласованным показаниям производительности. Задокументируйте предполагаемое переопределение в билете эксперимента и сбросьте конфигурацию перед записью новых базовых показателей.
3. **Четность CI:** Некоторые версии macOS не поддерживают выборку производительности на основе счетчика, когда NEON активен. Сохраняйте выходные данные `scripts/sm_perf_capture_helper.sh` прикрепленными к PR, чтобы проверяющие могли подтвердить, что ускоренный путь был использован, даже если участник скрывает эти счетчики.
4. **Будущие варианты ISA (SVE/SVE2):** Текущие ядра принимают форму полос NEON. Прежде чем портировать на SVE/SVE2, расширите `sm_accel::NeonPolicy` специальным вариантом, чтобы мы могли сохранить согласованность CI, телеметрии и ручек оператора.

Действия, отслеживаемые в соответствии с SM-5a/SM-4c.1, гарантируют, что CI собирает доказательства четности для каждой новой архитектуры, а дорожная карта остается на уровне 🈺 до тех пор, пока базовые показатели Neoverse/x86 и допуски NEON и скаляр не совпадут.

## Соответствие и нормативные примечания

### Стандарты и нормативные ссылки
- **GM/T 0002-2012** (SM4), **GM/T 0003-2012** + **GB/T 32918 серии** (SM2), **GM/T 0004-2012** + **GB/T 32905/32907** (SM3) и **RFC 8998** регулируют определения алгоритмов, тестовые векторы и Привязки KDF, которые используют наши фикстуры.【docs/source/crypto/sm_vectors.md#L79】
- Краткое описание соответствия в `docs/source/crypto/sm_compliance_brief.md` связывает эти стандарты с обязанностями по подаче/экспорту для инженерных, SRE и юридических групп; обновляйте это краткое описание при каждом обновлении каталога GM/T.

### Нормативный процесс материкового Китая
1. **Регистрация продукта (开发备案):** перед отправкой двоичных файлов с поддержкой SM из материкового Китая отправьте манифест артефакта, детерминированные этапы сборки и список зависимостей в провинциальную администрацию криптографии. Шаблоны документов и контрольный список соответствия находятся в `docs/source/crypto/sm_compliance_brief.md` и каталоге вложений (`sm_product_filing_template.md`, `sm_sales_usage_filing_template.md`, `sm_export_statement_template.md`).
2. **Отчеты о продажах/использовании (销售/使用备案):** Операторы, использующие локальные узлы с поддержкой SM, должны зарегистрировать область своего развертывания, состояние управления ключами и план телеметрии. При подаче прикрепите подписанные манифесты и снимки показателей `iroha_sm_*`.
3. **Аккредитованное тестирование.** Операторам критической инфраструктуры могут потребоваться сертифицированные лабораторные отчеты. Предоставьте воспроизводимые сценарии сборки, экспорт SBOM и артефакты Wycheproof/interop (см. ниже), чтобы последующие аудиторы могли воспроизводить векторы без изменения кода.
4. **Отслеживание статуса:** Запишите завершенные заявки в квитанцию ​​о выпуске и `status.md`; недостающие документы блокируют переход от пилотных проектов только для проверки к пилотным проектам с подпиской.### Положение об экспорте и распространении
- Считайте двоичные файлы с поддержкой SM контролируемыми объектами в соответствии с **Категорией 5 EAR США, Часть 2** и **Регламентом ЕС 2021/821, Приложение 1 (5D002)**. Публикация исходного кода по-прежнему соответствует критериям исключения из открытого исходного кода/ENC, но перераспространение в пункты назначения, на которые наложено эмбарго, по-прежнему требует юридической проверки.

## Приложение о соответствии (государственная коммерческая криптография)

- **Классификация:** SM2/SM3/SM4 поставляются в соответствии с *государственным режимом коммерческой криптографии* Китая (Закон КНР о криптографии, статья 3). Поставка этих алгоритмов в программное обеспечение Iroha **не** помещает проект на базовый/общий (государственная тайна) уровни, но операторы, использующие их в развертываниях в КНР, должны соблюдать обязательства по подаче заявок на коммерческую криптовалюту и обязательства MLPS.【docs/source/crypto/sm_chinese_crypto_law_brief.md:14】
– **Происхождение стандартов:** согласуйте общедоступную документацию с официальными преобразованиями GB/T спецификаций GM/T:
