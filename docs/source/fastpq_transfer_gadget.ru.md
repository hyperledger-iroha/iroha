% Дизайн гаджета передачи FastPQ

# Обзор

Текущий планировщик FASTPQ записывает каждую примитивную операцию, задействованную в инструкции `TransferAsset`, что означает, что каждая передача оплачивает арифметику баланса, хэш-раунды и обновления SMT отдельно. Чтобы уменьшить количество строк трассировки при каждой передаче, мы вводим специальный гаджет, который проверяет только минимальные арифметические проверки/проверки обязательств, в то время как хост продолжает выполнять канонический переход состояний.

- **Объем**: отдельные передачи и небольшие пакеты, отправляемые через существующую поверхность системных вызовов Kotodama/IVM `TransferAsset`.
- **Цель**: сократить объем столбцов FFT/LDE для больших объемов передачи за счет совместного использования справочных таблиц и объединения арифметических операций для каждой передачи в компактный блок ограничений.

# Архитектура

```
Kotodama builder → IVM syscall (transfer_v1 / transfer_v1_batch)
          │
          ├─ Host (unchanged business logic)
          └─ Transfer transcript (Norito-encoded)
                   │
                   └─ FASTPQ TransferGadget
                        ├─ Balance arithmetic block
                        ├─ Poseidon commitment check
                        ├─ Dual SMT path verifier
                        └─ Authority digest equality
```

## Формат транскрипта

Хост выдает `TransferTranscript` при каждом вызове системного вызова:

```rust
struct TransferTranscript {
    batch_hash: Hash,
    deltas: Vec<TransferDeltaTranscript>,
    authority_digest: Hash,
    poseidon_preimage_digest: Option<Hash>,
}

struct TransferDeltaTranscript {
    from_account: AccountId,
    to_account: AccountId,
    asset_definition: AssetDefinitionId,
    amount: Numeric,
    from_balance_before: Numeric,
    from_balance_after: Numeric,
    to_balance_before: Numeric,
    to_balance_after: Numeric,
    from_merkle_proof: Option<Vec<u8>>,
    to_merkle_proof: Option<Vec<u8>>,
}
```

- `batch_hash` связывает расшифровку с хэшем точки входа транзакции для защиты от повторного воспроизведения.
- `authority_digest` — это хеш хоста по отсортированным данным подписывающих лиц/кворума; гаджет проверяет равенство, но не повторяет проверку подписи. Конкретно хост Norito кодирует `AccountId` (в который уже встроен канонический контроллер мультиподписи) и хеширует `b"iroha:fastpq:v1:authority|" || encoded_account` с помощью Blake2b-256, сохраняя полученный `Hash`.
- `poseidon_preimage_digest` = Poseidon(account_from || account_to || актив || сумма || пакетный_хеш); гарантирует, что гаджет пересчитает тот же дайджест, что и хост. Байты прообраза создаются как `norito(from_account) || norito(to_account) || norito(asset_definition) || norito(amount) || batch_hash` с использованием простой кодировки Norito перед передачей их через общий помощник Poseidon2. Этот дайджест присутствует для транскриптов с одной дельтой и отсутствует для пакетов с несколькими дельтами.

## Макет гаджета

1. **Арифметический блок баланса**
   - Входы: `from_balance_before`, `amount`, `to_balance_before`.
   - Проверки:
     - `from_balance_before >= amount` (диапазонный гаджет с общим разложением СОК).
     - `from_balance_after = from_balance_before - amount`.
     - `to_balance_after = to_balance_before + amount`.
   - Упаковано в специальный элемент, поэтому все три уравнения занимают одну группу строк.2. **Блок «Обязательства Посейдона»**
   - Пересчитывает `poseidon_preimage_digest`, используя общую справочную таблицу Poseidon, уже используемую в других гаджетах. В следе нет патронов «Посейдон» для каждой передачи.

3. **Блок «Путь Меркла»**
   - Расширяет существующий гаджет Kaigi SMT режимом "парного обновления". Два листа (отправитель и получатель) используют один и тот же столбец для одноуровневых хэшей, что сокращает количество дублируемых строк.

4. **Проверка авторитетного дайджеста**
   — Простое ограничение равенства между дайджестом, предоставленным хостом, и значением-свидетелем. Подписи остаются в своем специальном гаджете.

5. **Пакетный цикл**
   - Программы вызывают `transfer_v1_batch_begin()` перед циклом сборщиков `transfer_asset` и `transfer_v1_batch_end()` после него. Пока область активна, хост буферизует каждую передачу и воспроизводит ее как один `TransferAssetBatch`, повторно используя контекст Poseidon/SMT один раз для каждого пакета. Каждая дополнительная дельта добавляет только арифметику и две проверки листьев. Декодер расшифровки теперь принимает пакеты с несколькими дельтами и отображает их как `TransferGadgetInput::deltas`, чтобы планировщик мог складывать свидетелей без повторного считывания Norito. Контракты, в которых уже есть полезная нагрузка Norito (например, CLI/SDK), могут полностью пропустить область действия, вызвав `transfer_v1_batch_apply(&NoritoBytes<TransferAssetBatch>)`, который передает хосту полностью закодированный пакет за один системный вызов.

Каждый аргумент кортежа должен иметь форму `(AccountId, AccountId, AssetDefinitionId, int)`; разовые передачи сохраняют существующий застройщик. |

Пример использования Kotodama:

```text
fn pay(a: AccountId, b: AccountId, asset: AssetDefinitionId, x: int) {
    transfer_batch((a, b, asset, x), (b, a, asset, 1));
}
```

`TransferAssetBatch` выполняет те же разрешения и арифметические проверки, что и отдельные вызовы `Transfer::asset_numeric`, но записывает все отклонения внутри одного `TransferTranscript`. Транскрипты с несколькими дельта исключают дайджест посейдона до тех пор, пока в последующем не появятся обязательства по каждой дельте. Компоновщик Kotodama теперь автоматически генерирует начальные и конечные системные вызовы, поэтому контракты могут развертывать пакетные передачи без ручного кодирования полезных данных Norito.

## Система регрессии с количеством строк

`fastpq_row_bench` (`crates/fastpq_prover/src/bin/fastpq_row_bench.rs:1`) синтезирует пакеты переходов FASTPQ с настраиваемым количеством селекторов и сообщает результирующую сводку `row_usage` (`total_rows`, количество каждого селектора, соотношение) вместе с дополненной длиной/log₂. Получите контрольные значения для потолка из 65536 рядов с помощью:

```bash
cargo run -p fastpq_prover --bin fastpq_row_bench -- \
  --transfer-rows 65536 \
  --mint-rows 256 \
  --burn-rows 128 \
  --pretty \
  --output fastpq_row_usage_max.json
```Создаваемый JSON отражает артефакты пакета FASTPQ, которые `iroha_cli audit witness` теперь выдает по умолчанию (передайте `--no-fastpq-batches` для их подавления), поэтому `scripts/fastpq/check_row_usage.py` и шлюз CI могут сравнивать синтетические прогоны с предыдущими снимками при проверке изменений планировщика.

# открытых вопросов

- **Ограничения домена**: текущий планировщик БПФ выдает панику, если трассировки превышают 2¹⁴ строк. TF-2 должен либо увеличить размер домена, либо задокументировать уменьшенный целевой показатель.
– **Пакеты с несколькими активами**: исходный гаджет предполагает одинаковый идентификатор актива для каждой дельты. Если нам нужны гетерогенные пакеты, мы должны гарантировать, что свидетель Poseidon каждый раз включает актив, чтобы предотвратить повторное воспроизведение между активами.
- **Повторное использование дайджеста полномочий**: в долгосрочной перспективе мы можем повторно использовать один и тот же дайджест для других разрешенных операций, чтобы избежать повторного расчета списков подписывающих лиц для каждого системного вызова.
