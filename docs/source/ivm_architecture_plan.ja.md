# IVM アーキテクチャ再設計計画

本ドキュメントは、Iroha Virtual Machine (IVM) を短期的に再構成し、レイヤー構造の明確化とセキュリティ／性能特性の維持を両立するためのマイルストーンをまとめたものです。責務の切り分け、ホスト連携の安全性向上、Kotodama 言語スタックの独立化準備に焦点を当てています。

## 目的

1. **レイヤードなランタイムファサードの導入** – VM コアを狭いトレイト背後に埋め込めるように明示的なランタイムインターフェイスを用意し、内部モジュールに触れることなく代替フロントエンドを進化させられる土台を整える。
2. **ホスト／システムコール境界の強化** – システムコールディスパッチを専用アダプタ経由に統一し、ホスト側コードが実行される前に ABI ポリシーとポインタ検証を必ず通す。
3. **言語／ツーリングの分離** – Kotodama 固有のコードを新しいクレートへ移し、`ivm` クレートにはバイトコード実行面のみを残す。
4. **設定の一元化** – アクセラレーションやフィーチャートグルを `iroha_config` から制御し、本番パスから環境変数ベースのフラグを排除する。

## フェーズ別内訳

### フェーズ1 – ランタイムファサード（進行中）
- ライフサイクル操作（`load_program`、`execute`、ホスト連携など）を記述する `VmEngine` トレイトを `runtime` モジュールに追加する。
- 既存の `IVM` 構造体にトレイトを実装し、コンクリート型ではなくインターフェイスへ依存する使い方（およびテスト）を可能にする。
- `lib.rs` からの直接的なモジュール再エクスポートを段階的に減らし、可能な箇所ではファサード経由で読み込むようにする。

**セキュリティ／性能への影響**: ファサードにより内部状態への直接アクセスが制限され、安全なエントリポイントのみが公開される。ホストとのやり取りの監査やガス／TLV 処理の検証が容易になる。

### フェーズ2 – システムコールディスパッチャ
- `IVMHost` をラップし、ABI ポリシーとポインタ検証を一元的に行う `SyscallDispatcher` コンポーネントを導入する。
- 既定ホストおよびモックホストをディスパッチャ経由に移行し、重複していた検証ロジックを削除する。
- ディスパッチャを差し替え可能にし、安全性チェックを迂回せずにホスト独自の計測を挿し込めるようにする。
- `SyscallDispatcher::shared(...)` ヘルパーを用意し、`Arc<Mutex<..>>` で共有されたホストをクローン済み VM からも安全に呼び出せるようにする。

**セキュリティ／性能への影響**: 中央集権的なゲートにより、`is_syscall_allowed` の呼び出し忘れを防ぎ、繰り返し呼ばれるシステムコールのポインタ検証を将来的にキャッシュできる余地を作る。

### フェーズ3 – Kotodama の切り出し
- Kotodama コンパイラを `crates/kotodama_lang` に切り出し（`crates/ivm/src/kotodama` から移行）。
- VM が利用する最小限のバイトコード API (`compile_to_ivm_bytecode` など) を提供する。
- リポジトリ全体への影響が行き渡る間は `ivm` に互換レイヤーを残す。

**セキュリティ／性能への影響**: 分割により VM コアの攻撃面積が縮小し、言語／ツール側の改良がインタプリタ回りのリグレッションに波及しにくくなる。

### フェーズ4 – 設定の統合
- GPU バックエンドの有効/無効といったアクセラレーション設定は `iroha_config` で定義しつつ、`IVM_DISABLE_CUDA` や `IVM_DISABLE_METAL` といった環境変数によるランタイムのフラグ切り替えは維持する。
- 新設したファサード越しに `RuntimeConfig` を公開し、ホストが決定論的なアクセラレーションポリシーを明示的に選択できるようにする。

**セキュリティ／性能への影響**: 環境変数トグルを全廃することでサイレントな設定ドリフトを防ぎ、デプロイ環境に依存しない決定論的挙動を保証する。

## 直近のアクション

- フェーズ1を完了させるため、ファサードトレイトを追加し、高レベルの呼び出し箇所がインターフェイスに依存するよう更新する。
- 公開されている再エクスポートを棚卸しし、ファサードと明示的に公開すべき API のみに限定する。
- システムコールディスパッチャの API を別モジュールで試作し、検証後に既定ホストへ適用する。

各フェーズの進捗は実装が進み次第 `status.md` で追跡します。
