# Грамматика и семантика языка Kotodama

Этот документ задает синтаксис языка Kotodama (лексический анализ и грамматику), правила типизации, детерминированную семантику и то, как программы понижаются в байткод IVM (`.to`) с соглашениями pointer‑ABI Norito. Исходные файлы Kotodama используют расширение `.ko`. Компилятор выпускает байткод IVM (`.to`) и может при необходимости вернуть манифест.

## Обзор и цели

- Детерминизм: программы должны давать одинаковый результат на любом железе; без плавающей точки и недетерминированных источников. Все взаимодействия с хостом происходят через syscalls с аргументами, кодированными Norito.
- Переносимость: цель — байткод Iroha Virtual Machine (IVM), а не физическая ISA. Кодировки в стиле RISC‑V, видимые в репозитории, — детали реализации декодера IVM и не должны менять наблюдаемое поведение.
- Аудитопригодность: компактная, явная семантика; четкое отображение синтаксиса в опкоды IVM и syscalls хоста.
- Ограниченность: циклы по неограниченным данным должны иметь явные границы. Итерация карт имеет строгие правила для гарантии детерминизма.

## Лексическая структура

Пробелы и комментарии
- Пробелы разделяют токены и в остальном незначимы.
- Строчные комментарии начинаются с `//` и идут до конца строки.
- Блочные комментарии `/* ... */` не вложенные.

Идентификаторы
- Начинаются с `[A-Za-z_]`, затем продолжаются `[A-Za-z0-9_]*`.
- Регистр значим; `_` — валидный идентификатор, но не рекомендуется.

Ключевые слова (зарезервированы)
- `seiyaku`, `hajimari`, `kotoage`, `kaizen`, `state`, `struct`, `fn`, `let`, `const`, `return`, `if`, `else`, `while`, `for`, `in`, `break`, `continue`, `true`, `false`, `permission`, `kotoba`.

Операторы и пунктуация
- Арифметика: `+ - * / %`
- Побитовые: `& | ^ ~`, сдвиги `<< >>`
- Сравнение: `== != < <= > >=`
- Логические: `&& || !`
- Присваивание: `= += -= *= /= %= &= |= ^= <<= >>=`
- Прочее: `: , ; . :: ->`
- Скобки: `() [] {}`

Литералы
- Целые: десятичный (`123`), hex (`0x2A`), двоичный (`0b1010`). Все целые в рантайме — знаковые 64‑битные; литералы без суффикса типизируются выводом или как `int` по умолчанию.
- Строки: в двойных кавычках с экранированием `\`; UTF‑8.
- Булевы: `true`, `false`.

## Типы и литералы

Скалярные типы
- `int`: 64‑битный двухкомплементный; арифметика оборачивается по модулю 2^64 для add/sub/mul; деление имеет определенные знаковые/беззнаковые варианты в IVM; компилятор выбирает корректную операцию.
- `bool`: логическое значение; понижается до `0`/`1`.
- `string`: неизменяемая UTF‑8 строка; при передаче в syscalls представлена как Norito TLV; внутри VM используются байтовые срезы и длина.
- `bytes`: сырой Norito payload; псевдоним типа `Blob` pointer‑ABI для входов хэша/крипто/доказательств и durable‑оверлеев.

Составные типы
- `struct Name { field: Type, ... }` пользовательские продукт‑типы. Конструкторы используют синтаксис вызова `Name(a, b, ...)` в выражениях. Доступ к полю `obj.field` поддерживается и понижается во внутренние позиционные поля в стиле кортежа. ABI устойчивого состояния в цепи кодируется Norito; компилятор генерирует оверлеи, отражающие порядок struct, а тесты (`crates/iroha_core/tests/kotodama_struct_overlay.rs`) удерживают layout фиксированным между релизами.
- `Map<K, V>`: детерминированная ассоциативная карта; семантика ограничивает итерацию и мутации во время итерации (см. ниже).
- `Tuple (T1, T2, ...)`: анонимный продукт‑тип с позиционными полями; используется для множественного возврата.

Специальные типы pointer‑ABI (для хоста)
- `AccountId`, `AssetDefinitionId`, `Name`, `Json`, `NftId`, `Blob` и подобные — не полноценные типы времени выполнения. Это конструкторы, выдающие типизированные неизменяемые указатели в регион INPUT (Norito TLV‑обертки) и могут использоваться только как аргументы syscalls или перемещаться между переменными без мутации.

Вывод типов
- Локальные `let` выводят тип из инициализатора. Параметры функций должны иметь явные типы. Возвращаемые типы могут выводиться, если нет неоднозначности.

## Объявления и модули

Элементы верхнего уровня
- Контракты: `seiyaku Name { ... }` содержат функции, состояние, структуры и метаданные.
- Несколько контрактов в файле разрешены, но не рекомендуется; один основной `seiyaku` используется как вход по умолчанию в манифестах.
- Объявления `struct` определяют пользовательские типы внутри контракта.

Видимость
- `kotoage fn` обозначает публичную точку входа; видимость влияет на права диспетчера, а не на генерацию кода.

## Контейнер контракта и метаданные

Синтаксис
```
seiyaku Name {
  meta {
    abi_version: 1,
    vector_length: 0,
    max_cycles: 0,
    features: ["zk", "simd"],
  }

  state int counter;

  hajimari() { counter = 0; }

  kotoage fn inc() { counter = counter + 1; }
}
```

Семантика
- `meta { ... }` переопределяет значения компилятора по умолчанию для заголовка IVM: `abi_version`, `vector_length` (0 означает не задано), `max_cycles` (0 означает дефолт компилятора), `features` включает биты фич (ZK‑трассировка, объявление вектора). Неподдерживаемые фичи игнорируются с предупреждением. Если `meta {}` отсутствует, компилятор эмитит `abi_version = 1` и использует дефолты опций для остальных полей заголовка.
- `features: ["zk", "simd"]` (алиасы: `"vector"`) явно запрашивает соответствующие биты заголовка. Неизвестные строки фич теперь вызывают ошибку парсера, а не игнорируются.
- `state` объявляет переменные контракта. Сейчас компилятор понижает их в эфемерное хранилище на запуск (выделяется при входе в функцию); durable‑оверлеи с хост‑бэкендом и отслеживание конфликтов остаются TODO. Для чтений/записей через хост используйте явные helpers `state_get/state_set/state_del` и map‑helpers `get_or_insert_default`; они проходят через Norito TLV и сохраняют стабильные имена/порядок полей для будущей персистентности.
- Идентификаторы `state` зарезервированы; затенение имени `state` в параметрах или `let` запрещено (`E_STATE_SHADOWED`).
- Значения state‑карт не являются первоклассными: используйте идентификатор состояния напрямую для операций карты и итерации. Привязка или передача state‑карт в пользовательские функции запрещена (`E_STATE_MAP_ALIAS`).
- Durable state‑карты сейчас поддерживают только ключи типов `int` и pointer‑ABI; другие типы ключей отклоняются при компиляции.
- Поля durable‑состояния должны быть `int`, `bool`, `Json`, `Blob`/`bytes` или pointer‑ABI типы (включая структуры/кортежи из этих полей); `string` для durable‑состояния не поддерживается.

## Функции и параметры

Синтаксис
- Объявление: `fn name(param1: Type, param2: Type, ...) -> Ret { ... }`
- Публичная: `kotoage fn name(...) { ... }`
- Инициализатор: `hajimari() { ... }` (вызывается при деплое рантаймом, а не самой VM).
- Хук обновления: `kaizen(args...) permission(Role) { ... }`.

Параметры и возвраты
- Аргументы передаются в регистрах `r10..r17` как значения или INPUT‑указатели (Norito TLV) по ABI; дополнительные аргументы проливаются на стек.
- Функции возвращают ноль или одно скалярное значение либо кортеж. Основной возврат — в `r10` для скаляра; кортежи материализуются на стеке/OUTPUT по соглашению.

## Операторы

- Привязки переменных: `let x = expr;`, `let mut x = expr;` (мутабельность — проверка компилятора; мутация во время выполнения разрешена только для локалов).
- Присваивание: `x = expr;` и составные формы `x += 1;` и т. п. Цели должны быть переменными или индексами карты; поля кортежей/struct — неизменяемые.
- Управление: `if (cond) { ... } else { ... }`, `while (cond) { ... }`, C‑style `for (init; cond; step) { ... }`.
  - Инициализаторы и шаги `for` должны быть простыми `let name = expr` или выражениями; сложный деструктуринг запрещен (`E0005`, `E0006`).
  - Область видимости `for`: привязки из init‑клаузы видимы внутри цикла и после него; привязки из тела или шага не выходят наружу.
- Равенство (`==`, `!=`) поддерживается для `int`, `bool`, `string`, pointer‑ABI скаляров (например, `AccountId`, `Name`, `Blob`/`bytes`, `Json`); кортежи, структуры и карты не сравнимы.
- Цикл по карте: `for (k, v) in map { ... }` (детерминированно; см. ниже).
- Управление потоком: `return expr;`, `break;`, `continue;`.
- Вызов: `name(args...);` или `call name(args...);` (обе формы принимаются; компилятор нормализует к вызовам).
- Проверки: `assert(cond);`, `assert_eq(a, b);` маппятся на IVM `ASSERT*` в не‑ZK сборках или на ZK‑ограничения в режиме ZK.

## Выражения

Приоритет (высокий → низкий)
1. Поле/индекс: `a.b`, `a[b]`
2. Унарные: `! ~ -`
3. Мультипликативные: `* / %`
4. Аддитивные: `+ -`
5. Сдвиги: `<< >>`
6. Отношения: `< <= > >=`
7. Равенство: `== !=`
8. Побитовые AND/XOR/OR: `& ^ |`
9. Логические AND/OR: `&& ||`
10. Тернарный: `cond ? a : b`

Вызовы и кортежи
- Вызовы используют позиционные аргументы: `f(a, b, c)`.
- Литерал кортежа: `(a, b, c)` и деструктуринг: `let (x, y) = pair;`.
- Деструктуринг кортежа требует совпадающей арности; несоответствия отклоняются.

Строки и bytes
- Строки — UTF‑8; функции, которым нужны сырые байты, принимают `Blob`‑указатели через конструкторы (см. Builtins).

## Builtins и конструкторы pointer-ABI

Конструкторы указателей (эмитят Norito TLV в INPUT и возвращают типизированный указатель)
- `account_id(string) -> AccountId*`
- `asset_definition(string) -> AssetDefinitionId*`
- `asset_id(string) -> AssetId*`
- `domain(string) | domain_id(string) -> DomainId*`
- `name(string) -> Name*`
- `json(string) -> Json*`
- `nft_id(string) -> NftId*`
- `blob(bytes|string) -> Blob*`
- `norito_bytes(bytes|string) -> NoritoBytes*`
- `dataspace_id(string|0xhex) -> DataSpaceId*`
- `axt_descriptor(string|0xhex) -> AxtDescriptor*`
- `asset_handle(string|0xhex) -> AssetHandle*`
- `proof_blob(string|0xhex) -> ProofBlob*`

Макросы прелюда дают более короткие алиасы и inline‑валидацию для этих конструкторов:
- `account!("alice@wonderland")`, `account_id!("alice@wonderland")`
- `asset_definition!("rose#wonderland")`, `asset_id!("rose#wonderland")`
- `domain!("wonderland")`, `domain_id!("wonderland")`
- `name!("example")`
- `json!("{\"hello\":\"world\"}")` или структурированные литералы типа `json!{ hello: "world" }`
- `nft_id!("dragon#demo")`, `blob!("bytes")`, `norito_bytes!("...")`

Макросы разворачиваются в перечисленные конструкторы и отвергают неверные литералы на этапе компиляции.

Статус реализации
- Реализовано: конструкторы выше принимают строковые литералы и понижаются в типизированные Norito TLV‑обертки в регионе INPUT. Они возвращают неизменяемые типизированные указатели для аргументов syscalls. Нелитеральные строковые выражения отвергаются; используйте `Blob`/`bytes` для динамических входов. `blob`/`norito_bytes` также принимают значения типа `bytes` в рантайме без макросов.
- Расширенные формы:
  - `json(Blob[NoritoBytes]) -> Json*` через syscall `JSON_DECODE`.
  - `name(Blob[NoritoBytes]) -> Name*` через syscall `NAME_DECODE`.
  - Декодирование указателей из Blob/NoritoBytes: любой конструктор указателя (включая AXT‑типы) принимает payload `Blob`/`NoritoBytes` и понижается в `POINTER_FROM_NORITO` с ожидаемым type id.
  - Прямой проход для pointer‑форм: `name(Name) -> Name*`, `blob(Blob) -> Blob*`, `norito_bytes(Blob) -> Blob*`.
  - Поддерживается sugar методов: `s.name()`, `s.json()`, `b.blob()`, `b.norito_bytes()`.

Builtins хоста/syscall (мапятся на SCALL; точные номера в ivm.md)
- `mint_asset(AccountId*, AssetDefinitionId*, int)`
- `burn_asset(AccountId*, AssetDefinitionId*, int)`
- `transfer_asset(AccountId*, AccountId*, AssetDefinitionId*, int)`

- `authority() -> AccountId*`
- `register_domain(DomainId*)`
- `unregister_domain(DomainId*)`

- `vrf_verify(Blob, Blob, Blob, int variant) -> Blob`
- `vrf_verify_batch(Blob) -> Blob`
- `axt_begin(AxtDescriptor*)`

- `axt_commit()`

Утилитарные builtins
- `info(string|int)`: эмитит структурированное событие/сообщение через OUTPUT.
- `hash(blob) -> Blob*`: возвращает Norito‑кодированный хэш как Blob.
- `build_submit_ballot_inline(election_id, ciphertext, nullifier32, backend, proof, vk) -> Blob*` и `build_unshield_inline(asset, to, amount, inputs32, backend, proof, vk) -> Blob*`: inline‑конструкторы ISI; все аргументы должны быть compile‑time литералами (строковые литералы или конструкторы указателей из литералов). `nullifier32` и `inputs32` должны быть ровно 32 байта (сырая строка или hex `0x`), а `amount` должен быть неотрицательным.
- `schema_info(Name*) -> Json* { "id": "<hex>", "version": N }`
- `pointer_to_norito(ptr) -> NoritoBytes*`: оборачивает существующий TLV pointer‑ABI как NoritoBytes для хранения или передачи.
- `isqrt(int) -> int`: целочисленный квадратный корень (`floor(sqrt(x))`) реализован как opcode IVM.
- `min(int, int) -> int`, `max(int, int) -> int`, `abs(int) -> int`, `div_ceil(int, int) -> int`, `gcd(int, int) -> int`, `mean(int, int) -> int` — слитые арифметические helper‑ы, поддерживаемые нативными IVM opcode (ceiling‑деление ловит деление на ноль).

Примечания
- Builtins — тонкие обертки; компилятор понижает их в перемещения регистров и `SCALL`.
- Конструкторы указателей чистые: VM гарантирует, что Norito TLV в INPUT неизменяем на время вызова.
 - Структуры с полями pointer‑ABI (например, `DomainId`, `AccountId`) можно использовать для удобной группировки syscall‑аргументов. Компилятор маппит `obj.field` в правильный регистр/значение без лишних аллокаций.

## Коллекции и карты

Тип: `Map<K, V>`
- Карты в памяти (heap‑аллоцированные через `Map::new()` или переданные параметрами) хранят одну пару ключ/значение; ключи и значения должны быть word‑размерными типами: `int`, `bool`, `string`, `Blob`, `bytes`, `Json` или pointer‑типы (например, `AccountId`, `Name`).
- Durable state‑карты (`state Map<...>`) используют Norito‑кодированные ключи/значения. Поддерживаемые ключи: `int` или pointer‑типы. Поддерживаемые значения: `int`, `bool`, `Json`, `Blob`/`bytes` или pointer‑типы.
- `Map::new()` аллоцирует и обнуляет единственную запись в памяти (ключ/значение = 0); для карт не `Map<int,int>` укажите явную аннотацию типа или тип возврата.
- State‑карты не являются первоклассными значениями: вы не можете их переназначать (например, `M = Map::new()`); обновляйте записи через индексацию (`M[key] = value`).
- Операции:
  - Индексация: `map[key]` получить/установить значение (установка через syscall хоста; см. маппинг runtime API).
  - Наличие: `contains(map, key) -> bool` (пониженный helper; может быть intrinsic syscall).
  - Итерация: `for (k, v) in map { ... }` с детерминированным порядком и правилами мутации.

Правила детерминированной итерации
- Множество итерации — снимок ключей на входе в цикл.
- Порядок строго возрастающий по байтовой лексикографической сортировке Norito‑кодированных ключей.
- Структурные изменения (вставка/удаление/очистка) итераторной карты во время цикла вызывают детерминированный trap `E_ITER_MUTATION`.
- Требуются границы: либо объявленный максимум (`@max_len`) у карты, либо явный атрибут `#[bounded(n)]`, либо явный предел через `.take(n)`/`.range(..)`; иначе компилятор эмитит `E_UNBOUNDED_ITERATION`.

Helpers для границ
- `#[bounded(n)]`: опциональный атрибут на выражении карты, например `for (k, v) in my_map #[bounded(2)] { ... }`.
- `.take(n)`: итерирует первые `n` записей с начала.
- `.range(start, end)`: итерирует записи в полуоткрытом интервале `[start, end)`. Семантика эквивалентна `start` и `n = end - start`.

Примечания по динамическим границам
- Литеральные границы: `n`, `start`, `end` как целочисленные литералы полностью поддерживаются и компилируются в фиксированное число итераций.
- Нелитеральные границы: когда фича `kotodama_dynamic_bounds` включена в crate `ivm`, компилятор принимает динамические выражения `n`, `start`, `end` и вставляет рантайм‑ассерты для безопасности (неотрицательность, `end >= start`). Понижение эмитит до K охраняемых итераций с `if (i < n)`, чтобы избежать лишних выполнений тела (K по умолчанию = 2). Вы можете настраивать K через `CompilerOptions { dynamic_iter_cap, .. }`.
- Запускайте `koto_lint` для проверки lint‑предупреждений Kotodama перед компиляцией; основной компилятор всегда продолжает lowering после парсинга и проверки типов.
- Коды ошибок задокументированы в [Kotodama Compiler Error Codes](./kotodama_error_codes.md); используйте `koto_compile --explain <code>` для быстрых пояснений.

## Ошибки и диагностика

Диагностика компиляции (примеры)
- `E_UNBOUNDED_ITERATION`: цикл по карте без границы.
- `E_MUT_DURING_ITER`: структурная мутация итераторной карты в теле цикла.
- `E_STATE_SHADOWED`: локальные привязки не могут затенять объявления `state`.
- `E_BREAK_OUTSIDE_LOOP`: `break` вне цикла.
- `E_CONTINUE_OUTSIDE_LOOP`: `continue` вне цикла.
- `E0005`: инициализатор for сложнее поддерживаемого.
- `E0006`: шаг for сложнее поддерживаемого.
- `E_BAD_POINTER_USE`: использование результата конструктора pointer‑ABI там, где требуется первоклассный тип.
- `E_UNRESOLVED_NAME`, `E_TYPE_MISMATCH`, `E_ARITY_MISMATCH`, `E_DUP_SYMBOL`.
- Инструменты: `koto_compile` запускает lint перед эмиссией байткода; `--no-lint` отключает, `--deny-lint-warnings` делает сборку ошибочной при lint‑выводе.

Ошибки VM во время выполнения (выборочно; полный список в ivm.md)
- `E_NORITO_INVALID`, `E_OOB`, `E_UNALIGNED`, `E_SCALL_UNKNOWN`, `E_ASSERT`, `E_ASSERT_EQ`, `E_ITER_MUTATION`.

Сообщения об ошибках
- Диагностика несет стабильные `msg_id`, которые сопоставляются с таблицами переводов `kotoba {}` при наличии.

## Отображение генерации кода в IVM

Pipeline
1. Lexer/Parser создают AST.
2. Семантический анализ разрешает имена, проверяет типы и заполняет таблицы символов.
3. Понижение IR в простую SSA‑подобную форму.
4. Аллокация регистров в IVM GPR (`r10+` для args/ret по соглашению); spill на стек.
5. Эмиссия байткода: смесь IVM‑native и RV‑совместимых кодировок при необходимости; заголовок метаданных с `abi_version`, features, длиной вектора и `max_cycles`.

Ключевые соответствия
- Арифметика и логика маппятся на IVM ALU ops.
- Ветвления и управление маппятся на условные переходы и jumps; компилятор использует сжатые формы при выгода.
- Память локалов проливается в стек VM; выравнивание обеспечивается.
- Builtins понижаются в перемещения регистров и `SCALL` с 8‑битным номером.
- Конструкторы указателей размещают Norito TLV в INPUT и выдают их адреса.
- Ассерты маппятся на `ASSERT`/`ASSERT_EQ`, которые ловят в не‑ZK исполнении и эмитят ограничения в ZK‑сборках.

Ограничения детерминизма
- Без FP; без недетерминированных syscalls.
- SIMD/GPU ускорение невидимо для байткода и должно быть бит‑идентичным; компилятор не эмитит аппаратно‑специфичных ops.

## ABI, заголовок и манифест

Поля заголовка IVM, задаваемые компилятором
- `version`: версия формата байткода IVM (major.minor).
- `abi_version`: версия таблицы syscalls и схемы pointer‑ABI.
- `feature_bits`: флаги фич (например, `ZK`, `VECTOR`).
- `vector_len`: логическая длина вектора (0 → не задано).
- `max_cycles`: предел допуска и подсказка для ZK padding.

Манифест (опциональный sidecar)
- `code_hash`, `abi_hash`, метаданные из блока `meta {}`, версия компилятора и подсказки сборки для воспроизводимости.

### Эмиссия манифеста

- API компилятора Kotodama может вернуть `ContractManifest` вместе с скомпилированным `.to` через `ivm::kotodama::compiler::Compiler::compile_source_with_manifest`.
- Поля:
  - `code_hash`: хэш байтов кода (исключая заголовок IVM и литералы), вычисленный компилятором для привязки артефакта.
  - `abi_hash`: стабильный дайджест разрешенной поверхности syscalls для `abi_version` программы (см. `ivm.md` и `ivm::syscalls::compute_abi_hash`).
- Опциональные `compiler_fingerprint` и `features_bitmap` зарезервированы для toolchains.
- `entrypoints`: упорядоченный список экспортированных entrypoints (публичные, `hajimari`, `kaizen`) с требуемыми строками `permission(...)` и best‑effort подсказками ключей чтения/записи компилятора, чтобы admission и schedulers могли оценивать ожидаемый доступ к WSV.
- Манифест предназначен для admission‑проверок и реестров; см. `docs/source/new_pipeline.md` для жизненного цикла.
