# IVM を主経路に据えた高速トランザクションパイプライン（AOT_SIMD 加速と並列化）

本ドキュメントは、IVM を主要な実行経路とし、組み込み ISI を統合しつつ、異種ハードウェア間でも完全な決定論を維持しながら全 CPU コアで検証と実行を並列化する、高スループットなトランザクションパイプラインを提案します。「Optimized Transaction Pipeline with IVM‑First Execution, AOT_SIMD Acceleration, and Parallelism」設計で示されたアイデアを、IVM・Kotodama・ISI・WSV・Norito・Sumeragi の各セマンティクスと調和させたものです。

IVM のレジスタ型バイトコードを活用し、トランザクションごとのアクセスセットに基づく競合認識スケジューラを導入します。提案フェーズのリーダーにも再検証を行うバリデータにも適した構成で、正直なノード間で観測される振る舞いを完全に一致させます。

---

## ゴールと非ゴール

ゴール
- 大半のトランザクション（スマートコントラクトおよび命令セマンティクス）で IVM をメインラインの実行経路に据えつつ、非 IVM の組み込み命令もサポートする。
- 競合認識型の並列実行により、トランザクションレベルで利用可能なコアをすべて活用し、ホットパスでは SIMD によるトランザクション内加速も任意で選択できるようにする。
- 決定論と安全性を維持する。同じ入力に対してすべてのノードが同じ出力を得られるようにし、競合によるレース依存の結果を生まない。メモリ使用量やレイテンシーの上限も予測可能に保つ。
- 事前デコードやキャッシュ、安定したスケジューリングによってブロックレイテンシーを短縮し、スループットを向上させる。

非ゴール
- JIT コンパイラを導入すること。実行は引き続きインタープリタ／ディスパッチ方式で行い、ベクトル化プリミティブはフィーチャーフラグで切り替える。
- コンセンサスセマンティクスを変更すること。提案するパイプラインは既存の Sumeragi propose/validate/commit フローにそのまま接続する。

## アーキテクチャ概要

主な拡張点
- 静的解析とアクセスセット: トランザクション（必要に応じてサブユニット）ごとに読み取り／書き込みキー集合を導出し、安全な並列実行を実現する。
- 決定論的な並列スケジューラ: アクセスセットから競合 DAG を構築し、安定したポリシーでワーカースレッドに準備完了タスクを割り当てる。
- IVM ファーストエグゼキュータ: 組み込み ISI と IVM 呼び出しを共通の ExecutionPlan 上で統一する。IVM バイトコードは事前にデコードし（IvmCache）、フィーチャーフラグで安定した加速を切り替える。
- トランザクション単位の二段階検証: ステートレス迅速経路（並列）と、スケジューラの管理する `StateBlock` オーバーレイ上でのステートフルシミュレーション。
- WSV オーバーレイ: ステートフル実行は WSV スナップショット上の書き込み分離オーバーレイを利用し、決定論的にマージしてグローバルな一貫性を維持する。

調和させる制約
- Kotodama は IVM バイトコード（`.to`）をターゲットとし、RISC‑V などスタンドアロン体系をターゲットにはしない。RISC‑V 風のエンコーディングが現れる場合も IVM 内部の実装詳細であり、観測可能な挙動を変えてはならない。
- 組み込み ISI と IVM プログラムは同じエグゼキュータ境界を通じて実行され、権限チェックやメータリング、イベントセマンティクスも同一でなければならない。
- AOT_SIMD や CUDA/Metal による加速はフィーチャーフラグで制御され、クロスハードウェアでもビット単位で同一の結果を出し、決定論的なフォールバック経路を常に提供する。
- Norito シリアライゼーションの互換性を保持し、すべての公開データ構造およびトランザクションで往復不変が成立するようにする。

現在実装されている状態（現行ノード）
- アクセスセット導出: 組み込み命令については静的に計算し、IVM については有効化されている場合に動的プリパスを尊重する。マニフェストは助言的なヒントを含められる。
- 決定論的スケジューラ: ウェーブごとに安定した順序を保ちつつ、競合のないレイヤーを構築する。並列 apply 経路は `pipeline.parallel_apply` で制御（フィーチャーとして完了、パリティテスト済み）。
- リカバリーサイドカー: Core は受理集合とブロックごとの安定した DAG フィンガープリントを永続化し、起動時スキャンで再計算して不一致を記録する。
- 警告イベント: DAG フィンガープリントが不一致の際には `PipelineEvent::Warning` を発行し、Torii が警告イベントをストリーミングしリカバリーサイドカーのエンドポイントを公開する。
- オーバーレイ制限／チャンク: `pipeline.overlay_max_*` や `pipeline.overlay_chunk_instructions` で強制し、境界ケースをカバーする否定テストを備える。

データフロー（リーダー／バリデータ共通）
1. 受理 → 2. ゴシップ → 3. 事前デコード + ステートレス検証 → 4. 振り分け:
   - 通常トランザクション → 次の提案まで mempool（保留集合）
   - マルチシグトランザクション → 閾値達成または期限切れまで MST プール
   - 無効／期限切れ → 診断情報とともに破棄
   5. 静的解析（準備完了トランザクションのアクセスセット導出）→ 6. 競合 DAG 構築 → 7. WSV スナップショット上の `StateBlock` オーバーレイで並列ステートフル実行 → 8. 決定論的マージ／コミット順序 → 9. 封止・署名・ブロードキャストまたは投票 → 10. 永続化 + コミット後イベント。

補助サービスレーン（パイプライン外）
- 証明生成レーン: 長時間実行・資源制限付きワーカーがコミット済み状態を参照する ZK などの暗号証明を生成し、結果は通常のトランザクションとして提出され、検証／スケジュール／包含の経路をたどる。
- クエリ処理レーン: 読み取り専用クエリは WSV スナップショット上で実行し、ページネーション／ストリーミングと厳格な時間・計算制限を備える。ブロックパイプラインを決してブロックしない。

ノード内での証明生成（ブリッジ等の高度フロー）
- オプションのノード内証明はフィーチャーフラグで有効化する。コンセンサスクリティカルなパイプラインとは完全に切り離されたバックグラウンドレーンで実行する。
- ジョブはコミット済み WSV スナップショットのみを消費し（高さとステートルートを公開入力に固定）、推測オーバーレイには依存しない。
- 出力は証明とメタデータを含む通常のトランザクション（またはハッシュ参照されたアーティファクト）として提出され、標準的な検証 → スケジュール → 包含の経路に従う。

検証状況
- ステートレスな ZK 事前検証や、フォークを伴わないトレース検証は実装済みで、設定／フィーチャーフラグで制御する。バックグラウンド証明生成や添付ファイルは依然としてパイプライン外で任意要素である。

受理（ZK 検証）
- フィーチャーフラグ: コンパイル時の `zk-preverify` とランタイムの `zk.halo2.enabled` が事前検証とトレース検証を制御する。ホスト側は `zk.halo2.max_k` や許可カーブで追加制御する。
- 決定論: 順序や提案選択には影響しない。事前検証失敗でもトランザクション順序を変えない。非フォーキング検証はパイプライン警告で報告する。
- 安全性: 証明とエンベロープはサイズや k（電路深度）を上限管理し、決定論的に検証する。アクセラレータはスカラ実装とビット単位で同一結果を出す必要がある。

コンセンサス統合
- リーダーは mempool から提案を構成し、バリデータは解析・スケジューリング・ステートフル検証を決定論的に再実行する。パイプラインは Sumeragi propose/validate/commit に接続され、コンセンサス規則を変更しない。

## ネットワーク時刻サービス（NTS）

目的
- ウォッチャーノードのトリガースケジューリングや Sumeragi タイマーで使用する堅牢でビザンチン耐性のある「ネットワーク時刻」推定を提供する。
- OS のクロックを改変せず、単調増加クロックをネットワーク時刻へ写像する `network_offset` を維持する。NEM Chapter 8 の時間同期を参照しつつ、刈り込み中央値・RTT フィルタ・スムージングで改善する。

設計
- ローカル基準: 単調増加クロック（`t_mono`）を使い、うるう秒や OS 調整の影響を避ける。`t_net = t_mono + network_offset` を定義する。
- ピアサンプリング: 数秒（既定 5 秒）ごとに上限数（既定 8）までのピアを選び、`(id, t1)` を含む ping を送信する。ピアは `(id, t2_recv, t3_send)` で応答する。
- NTP 風推定: 応答がローカル時刻 `t3_net` で到着した際、オフセット `θ = ((t1_net_recv − t2_recv) + (t3_send − t4_net_recv)) / 2` を計算し、ラウンドトリップ `δ` を求める。`δ` が外れ値や上限を超える場合は無視し、残りを固い中央値で集約する。
- 更新: EMA（指数移動平均）で `network_offset` を滑らかに更新する。`θ` が許容範囲内に収まっている限り、ステップ状の変化を避ける。
- セキュリティ: 署名付き NTS メッセージを採用し、中間者攻撃を防ぐ。タイムサーバの不正を検出した場合は Hijiri 評価と連携して重み付けを調整する。

API と統合
- CLI/API で現在の `network_offset`、サンプル数、`δ` の統計、外れ値除外理由を表示する。
- `iroha_config.pipeline.network_time` にサンプリング間隔、最大遅延、ハード上限などの設定を追加する。
- トリガースケジューラは `t_net` を使用し、オフセットの変動が大きすぎる場合は安全側に倒す（実行前に待機）。

## コンセンサスモード

- 閉集合 Sumeragi: ガバナンスで管理されるパーミッション型バリデータ集合。ノードの追加／削除はオンチェーン取引とガバナンスポリシーで行い、Sumeragi はこの固定集合でリーダーローテーション・提案・検証・コミットを実施する。コンソーシアムや規制対象ネットワーク向けに推奨。
- Nominated Proof‑of‑Stake（NPoS）Sumeragi（任意）: ネットワーク参加者がステークし、バリデータ候補をノミネートする。各エポック境界で鎖が決定論的にバリデータ集合を選出する（`epoch_length`, `max_validators`, `min_self_stake`, ノミネーション上限などのパラメータ）。バリデータ集合の選定機構のみが異なり、Sumeragi のコンセンサスプロトコルとファイナリティは変わらない。不正／オフラインのペナルティやアンボンディング期間はガバナンスが定義する。モード切り替えはチェーンパラメータ（例: `consensus.mode = { ClosedSet | NPoS }`）で行う。

## ノードロール

- **バリデータ:** Sumeragi（閉集合または NPoS 選出）に参加し、提案・投票・署名を行う。
- **同期専用オブザーバ:** バリデータではないフルノード。ブロックを継続的に同期し、WSV を維持し、クエリや証明を提供するが、提案や投票は行わない。読み取りスケーラビリティと監査可能性を向上させる。
- **ライトクライアント:** ヘッダーのみを保持し、バリデータ署名や Merkle 証明を検証する。トランザクション提出はできるが完全な状態は持たない。

パイプラインの重ね合わせ（N をコミットしながら N+1 を準備）
- コンセンサスがブロック N を最終化している間、ノードは高さ N−1 に固定されたスナップショット上で N+1 の検証準備を進められる。ステートレスチェックやアクセスセット導出、プリフェッチは安全であり、N がコミットするまで状態変更は行われない。これによりセマンティクスを変えずにレイテンシーを削減する。

投入：受理とゴシップ
- Torii の受理段階では、デコード前に厳格で設定可能な制限を適用する。最大ペイロードサイズ（圧縮前後）、ネスト／深さ上限、トランザクションごとの最大署名数、サポートする鍵／スキーム集合など。許容される最悪ケースのサイズは `MaxTxBytes`, `MaxDecompressedBytes`, `MaxSigCount` のような命名パラメータで文書化する。
- 過負荷時の挙動は明示する。受理キューが高水位を超えた場合、ノードは決定論的ポリシーで最優先度の低い、あるいは過大なトランザクションを切り捨て、`Rejected(Overloaded)` のような正規エラーとリトライ指示をクライアントに返す。入り口で黙ってドロップしてはならない。
- 再送／TTL チェックは受理時に実施する。高さベースの `expires_at_height` と `(chain_id, sender, seq)` を送信者ごとに厳密に増加させる。
- ゴシップ伝播はサイズ／時間上限（数ミリ秒またはサイズ N）でバッチ化し、決定論的コーデックで圧縮し、遅延を避けるため迅速にフラッシュする。重複排除キーは正準トランザクションハッシュ（圧縮前）。ラウンドごとにピアターゲットをシャッフルし、構造化検閲を緩和しつつ、各トランザクションのハッシュで完全性を保つ。
- テレメトリは受理／ゴシップのスロットリング状況（キュー深度、理由別の切り捨て件数、バッチサイズ／レイテンシー）を公開する。ポリシーはノード設定で再現可能に記載する。

mempool ポリシー
- ステートレスチェックを通過したトランザクションは、TTL・優先順位・サイズ／手数料ポリシーを伴って保留集合にキューイングされる。包含順序はリーダーの選択規則（ハッシュ、手数料、経過時間など）に対して決定論的であり、再現性を高めるため設定に公開する。

サイズ制御とエビクション
- mempool にはソフト／ハードの上限を設定する。ソフト上限を超えた際は決定論的順序（例: 最低手数料階層 → `expires_at_height` が最も古い → `tx_hash` の辞書順）で削除する。ヒステリシスを適用し、頻繁な入れ替わりを避ける。
- 期限切れ（高さベース TTL）や確定済みの競合書き込みで無効化されたトランザクションを定期的に削除する。効率的なスキャンのため、期限と優先度で O(log N) のインデックスを維持する。
- 発送元ごとのクォータを設け、アカウント／ドメイン単位で未処理トランザクション数に上限を設けてホットスポットの独占を防ぐ。優先度の低いトランザクションも飢餓しないよう逐次エイジングする。

可視性と診断
- `Admitted`、`Evicted{reason}`、`Expired` といった mempool イベントを発行する。運用者や SDK が存在確認・キュー位置・拒否理由を照会できるように `GET /mempool/status` と `GET /mempool/tx/{hash}` を提供する。

## IVM と組み込み ISI の統合

各トランザクションを ExecutionPlan で表現する。
- ノード: 1 個以上のステップからなり、以下のいずれかにマッピングされる。
  - IVM プログラムノード: 事前デコード済みのバイトコード断片とメタデータ（レジスタ利用状況、システムコール、宣言済みステートアクセスなど）。
  - 組み込みノード: 既知のアクセスセット導出とコストヒントを持つ純粋なホスト操作（例: パラメータ設定）。
- エッジ: トランザクション内部の順序制約であり、トランザクション内部を再順序化しない。安全かつ宣言済みであれば、トランザクション内の独立したサブセグメントを並列化する場合もあるが、基本はトランザクション間の並列化である。

大半の命令は IVM 呼び出しに解決される。組み込み命令も同じアクセスセット機構を利用することで、スケジューラが IVM ノードと相互にインターリーブできるようにする。

権限と不変条件
- 組み込み命令も IVM システムコールも、WSV を対象に権限チェックを行い（役割、所有権、ドメイン／アカウントポリシー）、資産保存などの不変条件を同じガード層で強制し、異なる挙動が生じることを避ける。
- ステートレス vs. ステートフル: 署名／フォーマットチェックや基本的な認証はステートレス段階で実施し、権限／役割チェックはステートフル段階で同じガード API を用いて行い、エンジン間の挙動を一致させる。
- 監査証跡: 拒否が発生した場合は、アカウント・命令 ID・権限 ID／理由コードといった最小限の構造化診断情報を記録し、秘匿ペイロードを漏らさない。

ISO 20022 との統合
- エンベロープメタデータ: Norito メタデータで正規化された ISO エンベロープフィールド（`UETR`（UUID）, `MsgId`, `CreDtTm`, `BizSvc`（マーケットプラクティス）, `SchemaHash`）を受付ける。
- 再実行／リプレイ防止: `UETR` と `MsgId` を `(chain_id, sender, seq)` に紐付け、高さベース TTL を強制し、重複や逆行を決定論的に拒否する。
- JDG へのルーティング: BIC/ClearingSystemId から `JurisdictionId` を導き、主権チェックが必要なトランザクションを JDG にルーティングして認証を得る。
- 検証: IBAN チェックサム、BIC フォーマット、ISO 4217 通貨／小数点、構造化された氏名／住所（NFC 正規化）を確認する。
- ステータスマッピング: パイプライン結果を pacs.002 の `ACTC/ACSP/ACSC`, `RJCT`, `PDNG`, `ACWC` にマッピングし、ISO コードセットの理由コードを付与する。
- 手数料／チャージ: 手数料記帳時に `ChargeBearer`（CRED/DEBT/SHAR/SLEV）をサポートする。JDG は個人情報を漏らさず規制／制裁スクリーニングの結果を保証する。
- IVM ヘルパー: `crates/ivm/docs/iso20022.md` に定義されたオペコード／ヘルパーを利用し、決定論的バジェット内でパースとシリアライズを行う。

ISO エンベロープ（Norito スキーマ例）
```
struct IsoEnvelope {
  uetr: bytes              // UUID (16 バイト)
  msg_id: bytes            // BAH MsgId（NFC）
  cre_dt_tm_ms: u64        // 作成時刻（エポックからのミリ秒）
  biz_svc: option<bytes>   // マーケットプラクティス（CBPR+/SEPA/HVPS+）
  schema_hash: [u8; 32]    // メッセージスキーマ／バージョンのハッシュ
  market_practice: option<bytes>
  payload: bytes           // 正規化 XML またはコンパクト形式の正準メッセージバイト列
}
```

pacs.008 → レジャーへのマッピング例
- ステートレス: エンベロープをパースし、`schema_hash` を検証し、`(uetr, msg_id)` の冪等性と高さ TTL を強制し、IBAN/BIC/通貨チェックや関係者フィールドの正規化を行う。
- JDG へルーティング: BIC/ClearingSystemId から導出した先に向け、JDG 認証（`JurisdictionAttestation`）を確認し、欠落していれば拒否する。
- ステートフル: 債務者／債権者アカウントを解決し、通貨を資産 ID にマッピングし、`ChargeBearer` に基づく手数料を適用し、Transfer を実行し、`uetr/msg_id/schema_hash` をキー／バリューに注記する。
- イベント: 結果を pacs.002 コードにマッピングし、拒否の場合は ISO 理由コードを添える。

## Kotodama スマートコントラクトのライフサイクル（IVM バイトコード）

フェーズ（規範的）
- 著者フェーズ: Kotodama ソース（`.ko`）を作成し、明示的なエントリポイントと Norito 型付けされたパラメータ／戻り値を宣言する。グローバルな可変状態を避け、可能であれば正準キー用ヘルパーで状態キーを宣言する。
- ビルド: IVM バイトコード（`.to`）へコンパイルし、固定化されたツールチェーンを使用する。以下を含むマニフェストを生成する。
  - `code_hash`（Blake2b‑32）、`abi_hash`、`compiler_fingerprint`（rustc/LLVM）、フィーチャービットマップ（例: `cuda=false`, `metal=false`）、エントリポイント別の助言的 `access_set_hints`（RW キー）、引数／戻り値用の `schema_hash`。CPU SIMD 能力は常に自動報告される。
  - 再現性: 同じ入力に対して CI ビルドがバイト単位で同一の `.to` とマニフェストを生成する必要がある。
- 配布: `ContractPackage { code_bytes, manifest, publisher_sig }` を Torii など経由または外部チャネルで配布する。ノードはマニフェストのハッシュと署名を検証し、`code_hash` でコンテンツアドレス化して保存する。
- コード登録: `Register<SmartContractCode>` が `code_hash → {manifest, (optional) code_bytes}` を WSV またはアーティファクトストアに保存する。大きなコードはコンテンツアドレス付き URL とハッシュでオフチェーン保管してもよい。
- デプロイ／有効化: `Register<SmartContractInstance>` が `contract_id`, `version`, `owner`, `domain`, `code_hash`, パラメータ、および任意の `state_namespace` を持つインスタンスを作成する。ドメインにデプロイできるのは権限を持つロールのみ。
- 呼び出し: `Call { contract_id, entrypoint, args }`。ステートレス段階で `code_hash` の存在、ABI／引数スキーマの一致、助言的 `access_set_hints` の存在を確認する。ステートフル段階で IVM を通じて実行し、`StateBlock` オーバーレイに書き込む。
- アップグレード: `UpgradeContractInstance` がロールアウトウィンドウ内で `version` を上げ、新しいマニフェストを適用する。旧バージョンは互換性ウィンドウ終了まで実行可。

監査要件
- マニフェストの JSON は RFC 8785（JCS）で正規化し、ドメインタグを適用する。
- `manifest` と `.to` はそれぞれ 128 KiB／1 MiB 未満であり、ハッシュは Blake2b‑256。
- 付随する `access_set_hints` は助言的情報であり、実行時に観測したアクセスセットと照合して警告を出す。

## Root/Sudo とランタイムアップグレードガバナンス

Root 命令
- `RootAdminSet`: Root マルチシグメンバーの集合と閾値。
- `RootMultisigPolicy`: `threshold`, `members`, `context_hash`, `expires_at_height`。
- `RootCall`: ルート権限での複合呼び出し（`call: InstructionBox`）。
- `RootSchedule`: 指定高さのトリガーとして Root 呼び出しを登録する。
- `RootCancel`: 既存の Root 呼び出しを `context_hash` で取り消す。

Root マルチシグ
- 閾値ベース。ガバナンスは Root セットを提案し、十分な署名を収集して `RootAdminSet` をコミットする。
- Root 呼び出しは `context_hash` でコンテキストを固定し、`expires_at_height` で期限を管理してリプレイを防止する。
- 承認／拒否のイベントを発行し、監査ログに `accounts`, `context_hash`, `call` の概要、`justification` を記録する。

ランタイムアップグレード
- `RuntimeUpgradeManifest`: `name`, `description`, `abi_version`（v1 固定）, `abi_hash`, `added_syscalls` / `added_pointer_types` は空, `start_height`, `end_height`。
- `RuntimeUpgradePlan`: 準備段階のアーティファクト展開を定義。
- `RuntimeUpgradeActivate`: `start_height` と `end_height` に基づきアクティベーションを管理する。アクティベーションはウィンドウ内で一度のみ有効。
- マニフェスト staged apply パス: `RuntimeUpgradeStageApply { manifest_hash, artifact_ref, reproducibility_proof }` が検証済みバイナリを Aleph（内容アドレス型アーティファクトストア）に格納する。ノードは受理時に (1) `manifest` とバイナリの `code_hash/abi_hash` 一致を検証し、(2) CI 由来の再現性レポートや署名を Aleph のメタデータとして保存し、(3) `RuntimeUpgradeActivate` 実行前に `artifact_ref` から再取得してハッシュ照合を再度行う。Aleph への書き込みは一意の `manifest_hash` ごとに一回のみ許可され、差し替えは新しい `manifest_hash` を伴うステージング申請で扱う。

Root Multi-sig → 合議制（例: 国会）
- Root セットによる準備 → 議会（Proposal/Review/Enactment ハウス）による投票で最終化 → `ParliamentEnactment { at_window }` で実行ウィンドウを設定。

## 静的解析とアクセスセット

目的
- 安全な並列実行のため、トランザクションごとの読み書きキー集合を導出する。
- 衝突検出と楽観的実行のロールバックを最小限に抑える。

入力
- 組み込み命令: スキーマとコードから静的に導出する。
- IVM: バイトコードを解析し、命令レベルの読み取り／書き込みキーを推測する。Kotodama コンパイラからの助言的 `access_set_hints` を併用する。
- トリガー: 生成される `Call` を解析し、対応するアクセスセットを得る。

確定度
- 確定的（静的）アクセスセットを優先する。動的に発生するキーは保守的に見積もり、検証段階で実際のアクセスと突き合わせて警告を発行する。
- 読み取り専用トランザクションは書き込み集合が空であることを証明する。書き込みセットが空でない場合は検証エラーとする。

実装メモ
- `AccessSet { reads: BTreeSet<Key>, writes: BTreeSet<Key> }`
- `Key` は正準化された `[Domain::Asset::Account::Detail]` パス。`AssetId`, `AccountId`, `TriggerId` の各ハッシュを用いる。
- IVM 実行中にアクセスが観測されたら動的セットを更新し、宣言と差異があれば `PipelineEvent::AccessSetMismatch` を警告として発行する。

## 決定論的並列スケジューラ

フェーズ
1. 準備完了トランザクションのアクセスセットを収集。
2. 各トランザクションをノードとする競合グラフ（DAG）を構築。読み／書きの交差がある場合はエッジを追加。
3. Kahn 法に類似した安定なトポロジカルソートで波（wave）を形成し、競合のないノード集合を並列実行する。
4. 各波で `StateBlock` オーバーレイを使用し、終了後に決定論的な順序でマージする。

安定性
- DAG 構築時にアクセスセットとトランザクションハッシュでソートし、同じ入力に対して常に同じ波構成を得る。
- ハッシュの衝突を避けるために `TxId`（`hash`, `admission_seq`）を用い、`admission_seq` は各ノードで決定論的に割り当てる。

フォールバック
- アクセスセット導出が不完全で競合が後から検出された場合、衝突したトランザクションのみ `StateBlock` をロールバックし、別波で再実行する。ただし同一ラウンド内で再試行回数上限を設定する。
- 上限を超えた場合はシリアル経路にフォールバックし、`PipelineEvent::FallbackSerial` を発行して運用者に知らせる。

スケジュール API
- `pipeline.parallel_apply` 設定で並列実行を制御する。`auto`（デフォルト）はアクセスセットの信頼度に応じて自動判定。
- `max_parallelism` で同時実行波数を制限し、CPU バインディングと NUMA 意識のスレッドプールを採用する。

### TTL とリプレイ防止（規範的）

- `expires_at_height` と `(chain_id, signer_id, seq)` を受理段階で検証する。
- `seq` は送信者ごとに厳密に増加する必要があり、再送は拒否する。欠落したシーケンスは「スキップ不可」として扱い、`Rejected(SequenceGap)` を返す。
- エポック切り替え時には `seq` の最小値（`epoch_start_seq`）をリセット可能だが、ガバナンスで公開されたウィンドウ内に限定する。
- マルチシグトランザクションは閾値に達すると `final_seq` にバインドされ、以降の重複提出は拒否する。

## 二段階検証（トランザクション単位）

1. **ステートレスフェーズ**（並列）
   - Norito デコード、署名検証、`chain_id` チェック、TTL/再送検証。
   - 署名数やペイロードサイズ、構造化制約（最大命令長）を確認する。
   - アクセスセットの宣言と静的検証を行い、助言的ヒントをキャッシュする。
   - ここで拒否されたトランザクションは診断情報付きで廃棄し、mempool へ格納しない。
2. **ステートフルフェーズ**（スケジューラ管理のオーバーレイ）
   - `StateBlock`（書き込み隔離）を作成し、IVM や組み込み命令を実行する。
   - 権限、ロール、資産量などのチェックを行い、拒否時は決定論的にロールバックする。
   - 成功したトランザクションはステート遷移を `StateUpdate` として記録する。

ロギング
- ステートレス段階とステートフル段階でそれぞれ `PipelineLog` を記録し、`tx_hash`, `phase`, `status`, `reason`（拒否時）を含める。
- 運用者は `GET /pipeline/logs/{height}` でブロック内ログを取得できる。

## IVM 実行エンジン（レジスタベース）

インタープリタディスパッチ
- Rust/LLVM がサポートする環境ではダイレクトスレッデッドインタープリタ（ジャンプテーブル）を採用し、サポートされない環境向けに移植性の高いフォールバックを用意する。
- 命令は事前にデコードしておき、実行時の 1 ステップごとのデコードオーバーヘッドを取り除く。

レジスタファイルとフレーム
- フレームごとにコンパクトなレジスタファイルを保持し、スタックスロットは積極的に再利用する。
- ホットループ内での動的アロケーションを避け、オペランドバッファは事前確保する。

ベクトル化・AOT_SIMD・アクセラレーション
- AOT_SIMD: ハッシュ／暗号／メモリアクセスなどのホットカーネルを、起動時に CPU 機能ビットに応じて選択される特化バリアントとして事前生成する。どのバリアントもビット単位で同一結果を返し、選択が結果に影響することはない。
- SIMD バックエンド（NEON/PMULL, CLMUL, AVX2/512）や任意の GPU オフロード（CUDA/Metal）はフィーチャーフラグで制御する。常に移植性の高いスカラ経路を正準経路として提供する。
- VM のワード単位命令（vadd/vxor など）は、モジュロ 2^N のラップアラウンドをレーン単位のセマンティクスとして固定し、バックエンド間でビット一致を保証する。

SIMD 命令の公開範囲
- ベクトル化は、公開されたベクトル命令が標準化されるまではスカラオペコードに対する実装詳細とみなす。ベクトル命令を公開する場合は Opcode Reference に完全なセマンティクスとガス価格を明記する必要がある。
  - 現状: VADD32/64, VAND/VXOR/VOR, VROT32 などの公開ベクトル命令が存在し、Opcode Reference にセマンティクスとガスを記載済み。

論理ベクトル長と並列マーカー（規範）
- `SETVL` は後続のベクトル命令で処理する論理要素数を設定する。ハードウェアの SIMD 幅を曝露せず、ハードウェアが変わっても結果は変化しない。
- ベクトル命令のガスは処理要素数に基づき算出し、CPU 幅には依存しない。端数（テイル）はスカラ経路で処理し、結果は完全に一致する。
- `PARBEGIN`/`PAREND` はコンパイラ用マーカーであり、ランタイムで並列実行セマンティクスを持たない。インタープリタでは no-op として扱われ、実際の並列化はブロックレベルのスケジューラのみが制御する。

GPU ターゲットとフラグ（有効化時）
- CUDA/PTX: 初期サポートは PTX ISA 7.8、`sm_80` と `sm_86`（Ampere）。検証後に `sm_90`（Hopper）を任意で追加する。
  - コンパイルフラグ: `--use_fast_math=false`, `--fmad=false`, `--prec-div=true`, `--prec-sqrt=true`, `--ftz=false`。整数カーネルのみを許可。
  - メモリ: 未定義動作は禁止。合流アクセスは推奨するが必須ではない。デバッグビルドでは境界チェックを有効化。
- Metal/MSL: Apple GPU Family 7（A14/M1）以降をターゲットとし、macOS 12+/iOS 15+ 上の MSL 2.4+ を要求する。整数パイプラインのみを使用し、コンセンサス経路に浮動小数点を許可しない。
- CI: 少なくとも 2 種類の GPU バックエンド（例: Apple AGX, Nvidia Ampere）と、スカラ／SIMD CPU 経路でゴールデンベクタを比較する。差異を検出した場合は GPU バックエンドを無効化する。

ホストシステムコール
- 副作用を持つシステムコール（状態の読み書きやイベント発行）は、明示的なケイパビリティ記述子を持つ vtable を経由する。メータリングと監査ログ（ドライランモード）を行い、アクセスセット監査に利用する。

互換性に関する注意
- Kotodama/IVM の ABI はマイナーバージョン間で安定しており、非推奨化はケイパビリティフラグ交渉によって進める。古いバイトコードは引き続き実行可能でなければならない。

規範的な VM セマンティクス
- コンセンサス経路では浮動小数点演算を使用しない。すべての IVM オペコードは整数／ビット演算で構成する。将来的に浮動小数点を導入する場合は IEEE モード・MXCSR・DAZ/FTZ・FMA 設定を固定し、適合テストを備える。
- 算術セマンティクス: 既定は mod 2^64 のラップアラウンド。飽和演算など別挙動のオペコードは Opcode Reference に明示する。
- エンディアンとレイアウト: 複数バイトのスカラはリトルエンディアン。ミスアラインドのロード／ストアは `MisalignedAccess` で決定論的にトラップし、未定義動作を許さない。範囲外や未初期化メモリアクセスは禁止。
- SIMD のチャンク処理方針: 固定 64 バイトブロック単位で処理し、テイルはスカラ経路にフォールバックする。ミスアライン時の扱いも仕様で規定する（必要に応じてバイト単位ロードなど）。

AOT アーティファクト
- 形式: ヘッダーに VM バージョン、コードハッシュ、コンパイラ指紋（Rust/LLVM）、`target-cpu`/`target-feature` ビットマップ、ビルドフラグを含める。本体にはコードとリロケーション情報を格納し、同一入力に対して再現可能であること。
- 検証: 読み込み時にインタープリタと比較する翻訳検証トレースを実行し、差異が発生した場合はアーティファクトを破棄してインタープリタにフォールバックする。

## ゼロ知識証明（生成と検証）

スコープ
- ノードはパイプライン内で証明を検証する。証明生成はチェーン外またはパイプライン外レーンで行う。トランザクションは証明と公開入力を運ぶか、オンチェーンのコミットメント／検証鍵を参照する。

証明生成（パイプライン外）
- 目的: ブリッジや高度なフローでは、コミット済み状態の性質を証明する必要がある。証明生成は計算コストが高く、マシンによって非決定論的になり得るため、必ずブロックパイプライン外で実行する。
- 入力: コミット済み WSV スナップショット、回路 ID／バージョン、公開入力、および状態やローカルシークレットから導出可能な必要証人。出力: 証明バイト列とメタデータ。
- 決定論: 生成には決定論を要求しない。検証のみがコンセンサスクリティカルである。ノードが異なる証明を生成しても、検証に成功したものだけが有効。

証明の受理
- トランザクションは `<ProofAttachment { circuit_id, version, proof_bytes, vk_hash, public_inputs_hash }>` を運ぶ。サイズや `k` の上限を設定し、`vk_hash` は登録済み検証鍵に一致しなければならない。
- `zk.halo2.enabled`（ランタイム）と `zk-preverify`（コンパイル時）で検証を制御する。ハードウェアアクセラレーションは常に deterministic fallback とペアで提供し、ビット単位で一致することを要求する。

検証手順（パイプライン内）
1. ステートレス: 証明サイズ・`k`・バージョン・登録済み VK をチェックし、公開入力を正規化してハッシュを計算する。
2. ステートフル: WSV から VK を引き当て、証明検証を実行する。検証が失敗した場合はトランザクションを決定論的に拒否し、診断イベントを出す。
3. オプション: 非フォーキングのトレース検証（再実行）をバックグラウンドで行い、`PipelineEvent::Warning` で進捗を報告する。

証明鍵とレジストリ
- `VerifyKeyRegistry`: `{ circuit_id, version, vk_hash, vk_bytes, allowed_domains }` を保持し、ガバナンスが更新する。
- 新しい検証鍵はチェーン上で登録し、`vk_hash` が一致しない証明は拒否する。

## ブリッジ設計（ライトクライアントおよび ZK）

目標
- Iroha ネットワークと外部チェーン間で、ポスト量子（PQ）対応の信頼最小化ブリッジを提供する。以下 2 パターンをサポートする。
  1. ICS-23/24 のメンバーシップ証明を用いたハッシュベースのライトクライアントブリッジ（ZK なし）。
  2. ZK を活用した透明なライトクライアント（zkBridge パターン）。証明をコンパクトに保つため定期的に再帰合成する。

推奨デフォルト
- IVM 内に組み込まれた透明 ZK ライトクライアント（STARK/FRI 系列）を展開する。PQ 対応を維持するためペアリングベースのラッパーは避け、証明生成／検証はすべて IVM パイプライン内で完結させる。

証明対象（規範）
- ヘッダー／ファイナリティ: `VerifyCommit(header, commit_certificate, validator_set_hash)`（i3.md の証明仕様）。
- 包含証明: `ICS23Verify(path, leaf, state_root)`（メッセージ／イベント／キー／値の包含）。
- ローリング同期: ブリッジは最新ヘッダーとコミット証明を追跡し、状態ルートとの整合性を確認する。

安全性と deterministic fallback
- 証明生成が失敗した場合でもネットワークが停止しないよう、ICS-23 ベースの非 ZK 経路を常に用意する。
- ブリッジトランザクションは `BridgeEvent` として記録され、検証結果は WSV に反映し、失敗時は確定的にロールバックする。

---

### IVM ZK ガジェットライブラリ（仕様）

- **Blake2b-32 スポンジガジェット**
  - ドメイン分離: すべての入力に固定ドメインタグ（例: `"iroha:hash:v1\x00"）` を付与する。上位仕様でタグを付けている場合は重複を避ける。
  - ラウンド／パディング: 標準の Blake2b パディングを実装し、`hash(bytes)` と `hash2(a, b)` ヘルパーを提供する。トランスクリプトは明示ラベル付きで同じスポンジを使用する。
  - API: `op HASH_LOAD ptr,len → h`, `op HASH_MERGE h1,h2 → h` を定義し、ホスト側フォールバックも用意する。
- **署名ガジェット**
  - Ed25519: 固定基数のスカラ乗算テーブルガジェット。検証 API は `ED25519_VERIFY(pubkey, msg, sig) → bool`。コンスタントタイムで実装する。
  - secp256k1: ウィンドウ化 MSM ガジェット。検証 API は `SECP256K1_VERIFY(pubkey, msg, sig) → bool`。定数時間のバケット加算を採用。
  - Dilithium（任意・`pqc` フィーチャー）: ML-DSA 検証を実装し、ラウンド構造とノイズサンプルは deterministic fallback と一致させる。
- **Merkle / ICS-23**
  - 固定分岐度のハッシュガジェットを提供し、ICS-23 のハッシュ順序とドメインタグを遵守する。
- **Poseidon / Rescue**
  - 可能な限り Poseidon をデフォルトハッシュとして使用する。ドメインタグとラウンド定数はレジストリで共有し、回路版とホスト版を同期させる。

---

### IVM ライトクライアント回路仕様（コミット証明 + ICS-23）

入力
- 公開入力: `header_bytes`, `validator_set_hash`, `signer_bitmap`, `state_root`, `ics23_path`, `leaf_bytes`。
- 非公開入力: `signatures[]`, `signer_pubkeys[]`（またはコミット済み集合参照）、`header_hash`（回路内で導出）。

制約（概要）
1. ヘッダーハッシュ: データモデルに従って `H(header_bytes)`（Blake2b-32）が `header_hash` に一致し、`header_bytes` から `state_root` と `validator_set_hash` を抽出する。
2. コミット証明: `signer_bitmap` で立っている各ビットについて、対応する署名と公開鍵が `header_hash` に対して正しいことを検証する。署名数が閾値に満たない場合は失敗する。
3. ICS-23: `ics23_path` に基づき `leaf_bytes` をマークル経路でハッシュし、得られたルートが公開された `state_root` に一致する。
4. 重み付き投票: バリデータ集合の投票重みを `validator_set_hash` から復元し、閾値（例: 2/3）を満たしていることを確認する。

出力
- 証明は `ZkLightClientProof { circuit_id, version, proof, public_inputs_hash }` としてエンコードされ、WSV で検証結果をキャッシュする。

---

### ビットコイン・オプティミスティックプログラム（パラメータと Taproot ツリー）

- ブリッジ専用の IVM プログラムで、Bitcoin Taproot スクリプトツリーに `OptimisticBridge` ロジックを配置する。
- パラメータ: 30 ブロックのチャレンジ期間、`n-of-m` サイン、`fee_rate_sat_per_vb`、`max_tx_weight`。
- Taproot ツリーには以下の枝を定義する。
  - `bridge_success`: Iroha 証明（ライトクライアント）を検証し資産を解放する。
  - `bridge_timeout`: チャレンジ期間後、Iroha 署名付きのロールバックを許可する。
  - `emergency_discharge`: ガバナンスが有効化する緊急ルートで、Multisig キーを用いて資産を安全に回収する。
- Iroha 側では `BridgeProgramState` に `btc_anchor_block_hash`, `pending_batches`, `challenge_deadline` を保持する。

### 初期サポートブリッジ（Day 1）

- **Iroha ↔ Iroha（サイドチェーン）**: IVM ライトクライアント証明を用いて相互運用。ヘッダー／証明をブリッジコントラクトに送信し、`ForwardedTransaction` を適用する。
- **Iroha ↔ SORA Nexus**: Kotodama コントラクト互換性を活用し、Nexus 側でライトクライアントを実行する。
- **Iroha ↔ Bitcoin**: 前述の OptimisticBridge プログラムと Taproot スクリプトを組み合わせ、BTC 保有資産を Iroha 上でラップする。
- 将来の候補: Cosmos/IBC 互換チェーン（ICS-23）、以太坊（zk-SNARK ベースのライトクライアント）など。

## データ可用性（任意構成）

- ブロック本体をオンチェーンに保持しつつ、`DataAvailabilitySidecar` として冗長チャンクを配布する。
- オプションで Reed-Solomon ベースの erasure coding を採用し、復元に必要なチャンク数を明示する。
- バリデータはサイドカーを検証して署名し、欠落チャンクはゴシップで再配布する。整合しない場合は DAG フィンガープリント警告を出す。
- 軽量クライアント向けに `DataAvailabilityProof` を提供し、チャンクのハッシュと位置を検証できるようにする。

---

## 高速同期とスナップショット配布

- 定期的に `SnapshotManifest { height, state_root, files[], hash }` を生成し、オブジェクトストレージまたは P2P で配布する。
- スナップショットは、最新ブロックとの差分を再生することで同期ノードが迅速に追いつけるようにする。
- チェックサムと署名を付与し、`snapshot.verify` API で検証を義務付ける。
- スナップショットの保持期間やローテーションは `pipeline.snapshot` 設定で制御する。

---

## マルチシグトランザクション（MST）

- `UnsignedTransaction` を部分署名束として扱い、`MstState` に `(tx_hash, collected_signers, expires_at_height)` を保持する。
- 署名は `SignatureSet` として Norito でエンコードし、送信者／ドメインごとに上限を設ける。
- 閾値に達すると最終トランザクションを生成し、通常のパイプラインに投入する。期限切れの束は `Expired` として通知し、原本を破棄する。
- イベント: `MstBundleCollected`, `MstBundleDropped`, `MstBundleCommitted` を発行する。

---

## クエリ処理（読み取り専用）

- 読み取り専用クエリは `QueryLane` で処理し、`workers`, `per_request_row_limit`, `per_request_time_limit` を設定で制御する。
- クエリはスナップショットに対して実行し、トランザクションパイプラインを妨げない。
- ストリーミング結果は Norito JSON を使用し、ページネーションやカーソルを標準化する。
- クエリ用の認可ポリシーを WSV で管理し、Hijiri 評価を組み合わせてレート制限を適用する。

---

## 並列フローにおけるトリガー

- トリガーで生成された内部トランザクションは、他のトランザクションと同じアクセスセット導出を受ける。
- `TriggerQueue` は優先度と期限でソートされ、競合がない限り次の波に含められる。
- 再入防止のため、トリガー ID と実行時刻を照合し、同一ブロック内での重複実行を禁止する。
- トリガー失敗時のリトライポリシー（最大試行回数、バックオフ）を設定可能にする。

---

## コミットとブロック順序

- 各波の `StateUpdate` は決定論的な順序（トランザクションハッシュと波番号）でマージし、WSV に適用する。
- ブロック内のトランザクション順序は、提案側での元の順序を保持する。ただし並列実行による内的順序変更が状態に影響しないよう保証する。
- コミット後に `BlockEvents { height, tx_count, warnings[] }` を発行し、サイドカーに DAG フィンガープリントを記録する。

---

## ブロックヘッダーとファイナリティ（ライトクライアント）

- ヘッダーには `height`, `prev_hash`, `state_root`, `commit_qc`, `validator_set_hash`, `timestamp_net`, `warnings_merkle_root` などを含める。
- ライトクライアントは `commit_qc`（署名集合）と `validator_set_hash` から閾値署名を検証する。
- `warnings_merkle_root` により、警告イベントの完全性を検証できるようにする。
- ファイナリティは commit certificate が揃った時点で発生し、ライトクライアントは確認ブロック数を設定で調整できる。

---

## データ可用性レイヤー（コンセンサスクリティカル）

- ブロックのフラグメントを `AvailabilityChunk { chunk_index, data, merkle_proof }` として管理する。
- バリデータは受け取ったチャンクを検証し、欠落チャンクがある場合は `RequestChunk` を送信する。
- 最終化前に全チャンクが揃っていることを確認し、欠落した場合はリーダーにリトライを要求する。
- 監査目的で `AvailabilityManifest` を保存し、外部からの再構築を支援する。

---

## ネットワーキングと受理最適化

- Torii は HTTP/2 と QUIC をサポートし、帯域の効率化と遅延削減を図る。
- 受理時に DoS 耐性を高めるため、IP/アカウント単位でのレート制限とバースト許容量を設定する。
- ゴシップは差分伝播を活用し、既知のトランザクションハッシュは再送しない。
- Telemetry で `gossip.queue_depth`, `ingress.overload_drops`, `quic.lost_packets` などを公開し、運用者が閾値を調整できるようにする。

---

## メモリ・キャッシュ・バックプレッシャー

- IVM キャッシュ（`IvmCache`）は LRU ポリシーで管理し、ヒット率とエビクション統計を公開する。
- `StateBlock` オーバーレイはプールで再利用し、過剰なアロケーションを避ける。上限を超えた場合は新規波の実行を待機させる。
- バックプレッシャーは `pipeline.backpressure.max_parallel_waves` や `overlay_pool_limit` で制御し、閾値超過時に運用者向け警告を発する。
- Telemetry に `pipeline.overlay_inflight`, `ivm_cache.hit_ratio`, `mempool.backpressure_events` を掲載する。

---

## 決定論と安全性メモ

- すべての乱数は決定論的 PRNG（Blake2b ベース）を使用し、シードはブロックハッシュとトランザクションハッシュから導出する。
- 並列実行によるレースが状態に影響しないよう、`StateBlock` とマージ順序を仕様で固定する。
- バリデータ間で異なる警告が発生した場合でも、警告ログは状態に影響せず、最終状態は一致する必要がある。

---

## 決定論仕様（規範）

- 乱数: `DeterministicRng(seed = H(block_hash || tx_hash))` を使用。`next()` は 64 ビット値を返し、消費順序は仕様で固定。
- タイムアウト: `pipeline` の各タイマーは `network_time` に基づく固定ウィンドウを使用し、`network_offset` の変動があっても決定論的。
- イベント順序: `PipelineEvent` はトランザクションハッシュとフェーズで安定ソートし、ログの順序を固定する。
- 例外処理: IVM トラップは `TrapReason` とともに記録し、同じ入力で常に同じトラップが発生する。

---

## メータリング仕様（規範）

- コストモデル
  - ステートレス: 署名検証、デコード、圧縮展開それぞれに定数コストを割り当てる。
  - ステートフル: 読み取り（`ReadKey`）と書き込み（`WriteKey`）に対して I/O 重みを設定し、重複アクセスはキャッシュヒット扱いで割引する。
  - IVM: オペコード毎のガス値を Opcode Reference に定義し、AOT/SIMD パスでも同じ値を使用する。
- メータリングの適用
  - トランザクションは `max_gas` と `payer` を宣言し、実行前に予算をチャージする。
  - オーバーラン時は `OutOfGas` として決定論的にロールバックし、支払済み分は没収する（ガバナンス設定可能）。
  - ガス単価はチェーン設定で定義し、ネットワーク全体で共有する。

---

## アップグレードとロールアウト計画（運用）

- ステージング: 新しい IVM コードやホストライブラリを各ノードに展開し、`RuntimeUpgradePlan` で署名する。
- アクティベーション: `start_height` で一度だけ機能を有効化し、`end_height` までに全ノードが適用できなかった場合は自動ロールバックする。
- フェールセーフ: アクティベーション中にエラーが発生した場合、`PipelineEvent::UpgradeAborted` を出し、前バージョンにフォールバックする。
- 文書化: 変更点とテスト結果を `docs/source/runtime_upgrades.*` や `status.md` に追記し、監査可能性を保つ。

## 脅威モデル（代表シナリオ）

- **Byzantine リーダー**: 無効な提案を送信しても、検証セットのステートレス／ステートフル再実行で拒否される。警告イベントが発行され、Hijiri 評価を減点する。
- **遅延攻撃**: ネットワーク時刻サービスとビュー変更により、遅延が一定しきい値を超えた場合にリーダー交代が実施される。
- **DoS（署名スパム）**: バッチ署名検証はミニバッチサイズと最大待機時間を設定し、過負荷時には受理段階で優先順位を下げる。
- **ZK 証明偽造**: 検証鍵をレジストリで管理し、証明サイズ／`k`／バージョンを検証する。不正証明は即時拒否され、警告を記録する。
- **トポロジ汚染**: ガバナンスによるピア追加／削除、Hijiri の信用スコア、監査ログにより検出し、悪性ピアを排除する。

## セキュリティと DoS 考慮

- 受理段階でのレート制限と署名検証バッチングを組み合わせ、攻撃面を縮小する。
- 並列スケジューラはアクセスセット検証に失敗したトランザクションを直ちに隔離し、他のトランザクションに影響させない。
- ZK や ISO 20022 パーサは入力サイズの上限を持ち、深い再帰を禁止する。
- GPU カーネルや AOT アーティファクトは読み込み時に検証し、不整合があれば自動的に無効化する。

## テスト戦略

- **ユニットテスト**: アクセスセット導出、StateBlock マージ、ISO 20022 パーサ、ライトクライアントハッシュ。
- **統合テスト**: 並列スケジューラの競合検出、フォールバックパス、トリガーフューズ、ブリッジフロー。
- **プロパティテスト**: 同じトランザクション集合を並列／シリアルで実行して一致性を検証する。
- **パフォーマンステスト**: 代表的なトランザクション混合で TPS とレイテンシーを測定し、レジレポートを取得する。
- **クロスプラットフォーム**: x86_64, ARM64, GPU バックエンドでゴールデンベクタを実行し、すべての結果が一致することを確認する。

## 適合テストチェックリスト

- IVM オペコード: すべてのオペコード（入力／出力／オーバーフロー）について x86-64 と ARM64 でゴールデンベクタを作成し、スカラ・SIMD・GPU 経路で一致を確認する。
- 算術セマンティクス: mod 2^N と飽和演算の振る舞いを境界条件でテストする。
- エンディアン／アラインメント: ミスアラインドアクセスやバイト順序、構造体レイアウトを検証し、未定義動作を起こさないことを確認する。
- SIMD チャンク処理: 64 バイトブロック処理とスカラテイルを検証し、スカラ経路と完全一致することを保証する。
- AOT 翻訳検証: 各 AOT カーネルでロード時にインタープリタとのトレース比較を行い、不一致があればキャッシュを拒否する。
- ツールチェーン再現性: 固定ツールチェーンで AOT アーティファクトを再ビルドし、バイト単位で同一になることを確認する。
- ISO 20022 テスト: `pacs.008`, `pacs.002`, `camt.053`, `camt.056`, `pacs.004`, `camt.029` などの正／負テストベクタを用意する。
- ブリッジ: ライトクライアント証明、Taproot ブリッジルート、失敗時のロールバックをテストする。

## 用語集

- **IVM**: `.to` バイトコードを実行する Iroha Virtual Machine。
- **Kotodama**: IVM をターゲットとする高水準スマートコントラクト言語。
- **ISI**: ホスト側で実行される組み込み命令。
- **WSV**: World State View。台帳状態とインデックスを保持する権威データ。
- **Norito**: オンワイヤデータに用いるシリアライゼーションコーデック。
- **AOT_SIMD**: SIMD カーネルの事前特化バリアント群。
- **ZK**: Zero-Knowledge 証明技術。検証はオンチェーンで行う。
- **VK**: 証明検証に使用する verifying key。
- **SRS**: 構造化参照文字列。特定の証明系で必要な公開パラメータ。

## Opcode リファレンス（規範的付録）

目的
- すべての IVM オペコードのセマンティクスとガス価格を一元的に定義し、適合テストとメータリングの基準とする。

慣習
- 既定の整数演算は mod 2^64。例外（飽和など）がある場合はオペコードごとに明記する。シフトは量を `0x3F` でマスクする。ゼロ除算は `AssertionFailed`（トラップ）。メモリはリトルエンディアンでアラインメントチェックを行い、ミスアライン時は `MisalignedAccess` を返す。

メモリアクセス
- `LOAD (0x03)`: `[rs1 + imm]` から `funct3` に応じて読み込む。
  - `0x0 = LB`（8bit 符号拡張）、`0x4 = LBU`（8bit ゼロ拡張）
  - `0x1 = LH`（16bit 符号拡張）、`0x5 = LHU`（16bit ゼロ拡張）
  - `0x2 = LWU`（32→64 ゼロ拡張）、`0x6 = LW`（32→64 符号拡張）
  - `0x3 = LD`（64bit）
  - ガス: 3
- `STORE (0x23)`: `rs2` を `[rs1 + imm]` に書き込む。`funct3` で幅を指定し、ミスアライン時はトラップ。
  - バイト書き込み: 2 ガス、ハーフワード: 3 ガス、ワード: 4 ガス、ダブル: 5 ガス。

算術・論理
- `ADD/SUB`: mod 2^64。ガス 1。
- `MUL`: 64bit 乗算、結果は下位 64bit。ガス 3。
- `DIV`/`REM`: ゼロ除算で `AssertionFailed`。ガス 5。
- `AND/OR/XOR`: ガス 1。
- `SHL/SHR/SAR`: シフト量は `0x3F` でマスク。論理右／算術右を区別する。

ベクトル命令（抜粋）
- `VADD32/64`: `SETVL` で設定された要素数だけ加算を行い、mod 2^N。ガスは `base + per_element`.
- `VAND/VXOR/VOR`: ビット演算。結果はスカラ経路と一致。
- `VROT32`: 32bit ローテート。オーバーヘッドを含むガスを定義。

制御フロー
- `JAL/JALR`: 相対／間接ジャンプ。レジスタに戻りアドレスを保存。ガス 2。
- `BRANCH` 系（BEQ/BNE/BLT/BGE など）: 条件評価後に PC を更新。ガス 2。
- `HALT`: フレームを終了し、戻り値をホストへ返す。

システムコール
- `SYSCALL` オペコードはシステムコール番号を取り、ホストの許可ポリシーに従って実行。ガスは syscall テーブルで定義する。

## 付録: 例示的ウォークスルー

1. リーダーは 2,000 件のトランザクションを受理し、1,800 件がステートレス検証を通過する。
2. アクセスセット解析で 1,500 件が十分に非競合と判定され、16 コアマシン向けに 6 波へ分割される。
3. IVM デコードの 70% はキャッシュヒットし、残りは一度だけデコードされる。
4. 第 1 波（300 件）が 16 ワーカーで並列実行され、オーバーレイを正準順でマージする。トリガーが 40 件の追跡タスクを第 2 波に追加する。
5. 6 波すべてが完了するとリーダーがブロックを封止・署名し、バリデータは同じスケジュールで再実行して決定論的に投票・コミットする。

結果として、高スループットと完全一致の最終状態／ブロック内容を正直なノード全体で達成する。

MVCC スナップショット（進化案）
- 将来的にはオーバーレイを MVCC スナップショットとして昇格し、コミットタイムスタンプを持たせる。
- 波はスナップショット上で実行し、コミットが新バージョンを生成する。
- オーバーレイをマージする際にバージョンチェンジログを生成し、監査を容易にする。

## ZK 資産（Halo2 シールドプール）

目的
- 発行者が許可した資産を秘匿形態に変換し、Halo2 回路を用いたシールドプール内でプライベートに送受信し、同額で透過トークンに戻す仕組みを提供する。総供給量を維持し、ダブルスペンド（ヌリファイア）を防ぎ、検証は決定論的かつ有界に保つ。

発行者ゲート（規範）
- AssetDefinition メタデータに `ZkConvertible=true` を設定し、発行者権限で管理する。任意パラメータとして `MinZkDeposit`, `MaxZkOutputs`, `ZkFeeBps` を用意する。

データ構造
- 資産ごとに `ZkPool { pool_id=AssetDefinitionId, root: [u8;32], height: u32, nullifiers: set<[u8;32]> }` を保持する。
- コミットメント: `C = Poseidon(asset_id || pk || value || rho)`。葉ノードは Poseidon(C) で、固定分岐度の Merkle 木に格納する。
- ヌリファイア: `N = Poseidon(sk || rho)`。チェーン全体で一意性を強制する。
- 出力ノート: `Note { commitment, amount, owner, rho }` を Norito でエンコードする。

操作フロー
1. **Deposit**: 透過資産をバーンし、シールドプールに同額ノートを発行する。公開入力には `asset_id`, `amount`, `commitment`, `merkle_path` を含む。
2. **Transfer**: 入力ノートのヌリファイアを提出し、Poseidon Merkle 証明と Halo2 証明を付与する。出力ノートを新たに生成する。
3. **Withdraw**: シールドノートを消費して透過資産に戻す。`withdraw_address` と `fee` を公開入力に含める。

検証
- すべての操作でヌリファイアの一意性をチェックし、既存ヌリファイアを検出した場合は拒否する。
- Merkle 木は固定高さと分岐度を持ち、`ZkPool` のルートを更新する。
- ガバナンスは `ZkVerifierRegistry` で利用可能な回路とバージョンを管理する。
