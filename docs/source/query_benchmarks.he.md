# מערך בנצ'מרקים לשאילתות

מסמך זה מתאר את המיקרו-בנוצים של שאילתות Iroha, כיצד הם נבנים וכיצד לפרש את התוצאות. הדגש הוא על הרצה מבוססת איטרטורים ועל תקורה בנתיב הלקוח, ללא תלות ברשת.

## מיקום הבנצים

- בנצי ליבה לשאילתות: `crates/iroha_core/benches/queries.rs`
- בנצי נתיב לקוח (מבצע מדומה): `crates/iroha_core/benches/queries_client.rs`

שניהם בנצי Criterion עם `fn main` ידני כדי לעמוד בלינטים של סביבת העבודה.

## מתודולוגיה

- הבנצים יוצרים `State` ו-`World` בזיכרון (ללא רשת) ומשתמשים ב-`ValidQuery::execute` לקבל איטרטורים של פריטים.
- כדי למנוע אופטימיזציה של תופעות לוואי, צורכים את התוצאות (`count()` או איסוף ואז מיון) ומשתמשים ב-`criterion::black_box` כדי להבטיח שהקוד לא יוסר.
- עבור תקורת נתיב הלקוח, המבצע המדומה (`MockExec`) מממש את `QueryExecutor` ומזין אצוות אל `QueryBuilder`/`QueryBuilderExt` ללא I/O רשת, כדי לבודד את עלות הבילדר ואת מסלול ההמרה הטיפוסי.

## מערכי נתונים

- חשבונות: `build_state_with_accounts(n)` יוצר דומיין יחיד ו-`n` חשבונות בבעלות סמכות בלבד.
- נכסים: `build_state_with_assets(n_accounts, assets_per_account)` מוסיף לנ"ל נכסים (אחד או שניים לכל חשבון).
- דומיינים: `build_state_with_domains(n)` יוצר `n` דומיינים נפרדים.
- הגדרות נכס: `build_state_with_asset_definitions(n)` יוצר דומיין, חשבון בעלים ו-`n` הגדרות נכס.

## מה נמדד

בנצי ליבה (`queries.rs`):
- FindAccounts
  - איטרציה וספירה (1k, ‏10k)
  - מיון לפי מזהה (10k)
  - סימולציית עימוד על וקטור מלא (גודל עמוד 100)
- FindAssets
  - איטרציה וספירה (~10k נכסים)
  - סינון לפי חשבון מסוים
  - סינון לפי סף כמות
- FindDomains
  - איטרציה וספירה (5k)
  - מיון לפי מזהה (10k)
- FindAssetDefinitions
  - איטרציה וספירה (10k)

בנצי נתיב לקוח (`queries_client.rs`):
- `QueryBuilder` + `execute_all()` עם `QueryExecutor` מדומה
  - 1k פריטים, ‏`fetch_size = 100`
  - 10k פריטים, ‏`fetch_size = 500`

הם מדמים את הנתיב שה-SDK של הלקוח מבצע ללא השפעות רשת.

## הרצה

בניית הבנצים:

```
cargo build -p iroha_core --benches
```

הרצת בנץ ספציפי:

```
# כל הבנצים של שאילתות ב-iroha_core
cargo bench -p iroha_core --bench queries

# בנצי נתיב הלקוח (מבצע מדומה)
cargo bench -p iroha_core --bench queries_client
```

הערה: Criterion כותב את התוצאות ל-`target/criterion`. לקבלת השוואות יציבות, נעצו את מצב ה-CPU והימנעו מתהליכים רועשים ברקע. הריצו מספר פעמים ובדקו את שונות הדוח.

## פרשנות התוצאות

- עלות האיטרטור גדלה עם מספר הפריטים; מרבית הבנצים מציגים התנהגות ליניארית. מיון מוסיף עלות `O(n log n)` והערכים רגישים לגודל המפתח ולמורכבות ההשוואה.
- סינון בצד האיטרטור מדגים תקרת עלות לוגיקה בצד השרת; בפועל יש להעדיף פילטרינג מוקדם כדי להקטין עומס downstream.
- סימולציית עימוד על וקטור מלא היא קירוב של חלוקה לקוח־צד. במערכות אמיתיות יש להסתמך על עימוד בצד השרת (`fetch_size`) כדי להימנע מטעינת כל הנתונים.
- בנצי נתיב הלקוח מבודדים את התקורה של בניית `QueryOutputBatchBoxTuple` והמרות טיפוסים.

## צעדים הבאים

- הוספת בנצים לנתיב השרת של מטפלי השאילתות (פרדיקטים חמים, מקשי מיון).
- הוספת בדיקות השלכה (projection) כשאופציה זו תחזור בצד השרת.
- הרחבה לטריגרים ובלוקים באמצעות בוני מצב סינתטיים.
