# Металл ба NEON хурдатгалын төлөвлөгөө (Swift & Rust)

Энэхүү баримт бичиг нь тодорхойлогч техник хангамжийг идэвхжүүлэх хуваалцсан төлөвлөгөөг тусгасан болно
хурдатгал (Metal GPU + NEON/Accelerate SIMD + StrongBox интеграци)
Rust ажлын талбар болон Swift SDK. Энэ нь замын зураглалыг дагаж мөрддөг зүйлсэд ханддаг
**Тоног төхөөрөмжийн хурдатгалын ажлын урсгал (macOS/iOS)**-ийн доор байгаа бөгөөд дамжуулалтыг өгдөг.
Rust IVM баг, Swift гүүр эзэмшигчдэд зориулсан олдвор, телеметрийн багаж хэрэгсэл.

> Сүүлд шинэчлэгдсэн: 2026-01-12  
> Эзэмшигчид: IVM Performance TL, Swift SDK Lead

## Зорилго

1. Apple-ийн техник хангамж дээрх Rust GPU цөмийг (Poseidon/BN254/CRC64) металлаар дамжуулан дахин ашиглах
   CPU-ийн замуудын эсрэг детерминист париттай тооцоолох шэйдерүүд.
2. Хурдасгах унтраалгауудыг (`AccelerationConfig`) төгсгөл хооронд нь нээгээрэй, ингэснээр Swift програмууд
   ABI/паритын баталгааг хадгалахын сацуу Metal/NEON/StrongBox-г сонгох боломжтой.
3. CI + хяналтын самбарыг гадаргуугийн паритет/бичмаркийн өгөгдөл болон далбааг тохируулах
   CPU ба GPU/SIMD замуудын хоорондох регресс.
4. Android (AND2) болон Swift хооронд StrongBox/scure-enclave хичээлүүдийг хуваалцаарай
   (IOS4) гарын үсэг зурах урсгалыг тодорхой уялдуулах.

**Шинэчлэлт (CRC64 + Stage‑1-ийн шинэчлэл):** CRC64 GPU туслахууд нь одоо 192KiB өгөгдмөл таслалтыг `norito::core::hardware_crc64` руу холбосон (`NORITO_GPU_CRC64_MIN_BYTES` эсвэл тодорхой туслагчийн Prometheus-ээр дамжуулан дарах) JSON Stage‑1 хасалтыг дахин харьцуулсан (`examples/stage1_cutover` → `benchmarks/norito_stage1/cutover.csv`), скаляр зүсэлтийг 4KiB-д байлгаж, Stage‑1 GPU-г 192KiB-д тохируулсан (`NORITO_STAGE1_GPU_MIN_BYTES` бага хэмжээний баримт бичгийг эхлүүлэхэд GPU-д ачаалал ихтэй байх болно).

## Интерфэйс ба API гэрээ### Зэв (`ivm::AccelerationConfig`)
- Одоо байгаа талбаруудыг (`enable_simd`, `enable_metal`, `enable_cuda`, `max_gpus`, босго) хадгал.
- Анхны хэрэглээний хоцролтоос зайлсхийхийн тулд тодорхой металл халаалт нэмнэ (Зэв №15875).
- Хяналтын самбарт статус/оношлогоо буцаах паритын API-г өгөх:
  - ж.нь. `ivm::vector::metal_status()` -> {идэвхжүүлсэн, тэнцүү, сүүлчийн_алдаа}.
- Гаралтын жишиг хэмжүүр (Merkle модны цаг хугацаа, CRC нэвтрүүлэх чадвар) дамжуулан
  `ci/xcode-swift-parity`-д зориулсан телеметрийн дэгээ.
- Метал хост одоо эмхэтгэсэн `fastpq.metallib`-г ачаалж, FFT/IFFT/LDE илгээдэг.
  болон Poseidon цөмүүд, мөн CPU-ийн хэрэгжилтэд буцаж унадаг
  metallib эсвэл төхөөрөмжийн дараалал боломжгүй байна.

### C FFI (`connect_norito_bridge`)
- Шинэ бүтэц `connect_norito_acceleration_config` (дууссан).
- Getter-ийн хамрах хүрээ нь одоо тохируулагчийг тусгах `connect_norito_get_acceleration_config` (зөвхөн тохиргоо) болон `connect_norito_get_acceleration_state` (тохиргоо + паритет) багтана.
- SPM/CocoaPods хэрэглэгчдэд зориулсан толгойн тайлбар дахь баримт бичгийн бүтцийн бүдүүвч.

### Свифт (`AccelerationSettings`)
- Өгөгдмөл: Металл идэвхжсэн, CUDA идэвхгүй, босго тэг (өвлөж авах).
- Сөрөг утгыг үл тоомсорлосон; `apply()`-г `IrohaSDK` автоматаар дуудсан.
- `AccelerationSettings.runtimeState()` одоо `connect_norito_get_acceleration_state` дээр гарч байна
  ачаалал (тохиргоо + Метал/CUDA паритын төлөв) тул Swift хяналтын самбар нь ижил телеметрийг ялгаруулдаг.
  Зэв (`supported/configured/available/parity`). Туслагч нь `nil`-г буцаана
  Туршилтыг зөөврийн байлгахын тулд гүүр байхгүй байна.
- `AccelerationBackendStatus.lastError`-аас идэвхгүй болгох/алдааны шалтгааныг хуулна
  `connect_norito_get_acceleration_state` ба стринг дууссаны дараа эх буферийг чөлөөлнө
  Мобайл паритын хяналтын самбарууд яагаад Металл/CUDA-г идэвхгүй болгосон талаар тайлбар хийх боломжтой болсон
  хост бүр.
- `AccelerationSettingsLoader` (`IrohaSwift/Sources/IrohaSwift/AccelerationSettingsLoader.swift`,
  `IrohaSwift/Tests/IrohaSwiftTests/AccelerationSettingsLoaderTests.swift`) дагуу туршилтууд одоо
  Norito демотой ижил тэргүүлэх дарааллаар операторын манифестийг шийддэг: honor
  `NORITO_ACCEL_CONFIG_PATH`, хайлтыг багцалсан `acceleration.{json,toml}` / `client.{json,toml}`,
  сонгосон эх сурвалжийг бүртгэж, өгөгдмөл рүү буцна уу. Аппликешнүүдэд захиалгат дуудагч хэрэггүй болсон
  Rust `iroha_config` гадаргууг толь.
- Товчлуур болон телеметрийн нэгдлийг харуулахын тулд жишээ програмууд болон README-г шинэчилнэ үү.

### Телеметр (Хяналтын самбар + Экспортлогч)
- Паритын хангамж (mobile_parity.json):
  - `acceleration.metal/neon/strongbox` -> {идэвхжүүлсэн, parity, perf_delta_pct}.
  - `perf_delta_pct` үндсэн CPU ба GPU харьцуулалтыг хүлээн авна уу.
  - `acceleration.metal.disable_reason` толь `AccelerationBackendStatus.lastError`
    Тиймээс Swift автоматжуулалт нь идэвхгүй GPU-г Rust-тэй адил үнэнч байдлаар тэмдэглэж чаддаг
    хяналтын самбарууд.
- CI тэжээл (mobile_ci.json):
  - `acceleration_bench.metal_vs_cpu_merkle_ms` -> {cpu, металл}
  - `acceleration_bench.neon_crc64_throughput_mb_s` -> Давхар.
- Экспортлогчид хэмжигдэхүүнийг Rust жишиг эсвэл CI гүйлтүүдээс (жишээ нь, run
  `ci/xcode-swift-parity`-ийн нэг хэсэг болох металл/CPU микро вандан).### Тохируулгын товчлуурууд ба өгөгдмөл (WP6-C)
- `AccelerationConfig` өгөгдмөл: macOS хувилбарууд дээрх `enable_metal = true`, CUDA функцийг хөрвүүлсэн үед `enable_cuda = true`, `max_gpus = None` (хязгааргүй). Swift `AccelerationSettings` боодол нь `connect_norito_set_acceleration_config`-ээр дамжуулан ижил өгөгдмөлүүдийг өвлөн авдаг.
- Norito Merkle heuristics (GPU vs CPU): `merkle_min_leaves_gpu = 8192` ≥8192 навчтай модны хувьд GPU хэшийг идэвхжүүлдэг; backend overrides (`merkle_min_leaves_metal`, `merkle_min_leaves_cuda`) нь тодорхой заагаагүй бол ижил босго дээр өгөгдмөл байна.
- CPU-ийн давуу эвристик (SHA2 ISA байгаа): AArch64 (ARMv8 SHA2) болон x86/x86_64 (SHA-NI) хоёуланд нь CPU-ийн замыг `prefer_cpu_sha2_max_leaves_* = 32_768` орхих хүртэл сонгосон хэвээр байна; түүнээс дээш GPU босго үйлчилнэ. Эдгээр утгыг `AccelerationConfig`-ээр тохируулах боломжтой бөгөөд зөвхөн жишиг нотолгоог ашиглан тохируулна.

## Туршилтын стратеги

1. **Нэгжийн паритет тест (Зэв)**: металл цөм нь CPU-ийн гаралттай таарч байгаа эсэхийг шалгаарай.
   детерминистик векторууд; `cargo test -p ivm --features metal` дор ажиллуулна.
   `crates/fastpq_prover/src/metal.rs` одоо зөвхөн macOS-д зориулсан паритет тестүүдийг илгээдэг.
   скаляр лавлагааны эсрэг FFT/IFFT/LDE болон Посейдон дасгал.
2. **Хурдан утааны бэхэлгээ**: CPU-ийн эсрэг металлыг ажиллуулахын тулд IOS6 туршилтын гүйгчийг сунгана
   эмулятор болон StrongBox төхөөрөмж дээр кодчилол (Merkle/CRC64); харьцуулах
   үр дүн болон бүртгэлийн паритын төлөв.
3. **CI**: түлхэхийн тулд `norito_bridge_ios.yml` (аль хэдийн `make swift-ci` руу залгасан)-г шинэчилнэ үү
   олдворуудын хурдатгалын хэмжүүр; Гүйлт нь Buildkite-г баталгаажуулж байгаа эсэхийг шалгаарай
   Морсын өөрчлөлтийг нийтлэхээс өмнө `ci/xcframework-smoke:<lane>:device_tag` мета өгөгдөл,
   болон паритет/бенчмарк дрифт дээр эгнээнд алдаа гаргана.
4. **Хяналтын самбар**: шинэ талбарууд одоо CLI гаралт дээр гарч байна. Экспортлогчдын үйлдвэрлэлийг хангах
   хянах самбарыг шууд эргүүлэхээс өмнө өгөгдөл.

## WP2-A Металл сүүдэрлэх төлөвлөгөө (Посейдон дамжуулах хоолой)

WP2-ын эхний үе шат нь Посейдон Металл цөмд зориулсан төлөвлөлтийн ажлыг хамардаг
Энэ нь CUDA хэрэгжилтийг тусгадаг. Төлөвлөгөө нь хүчин чармайлтыг цөмд хуваадаг.
хостын хуваарь, тогтмол үе шатыг хуваалцсанаар дараагийн ажил нь зөвхөн анхаарлаа төвлөрүүлэх боломжтой болно
хэрэгжилт, туршилт.

### Цөмийн хамрах хүрээ

1. `poseidon_permute`: `state_count` бие даасан улсуудыг өөрчилдөг. Утас бүр
   `STATE_CHUNK` (4 муж) эзэмшдэг ба бүх `TOTAL_ROUNDS` давталтуудыг ашиглан ажиллуулдаг.
   Диспетчерийн үед шаталсан threadgroup-shared дугуй тогтмолууд.
2. `poseidon_hash_columns`: сийрэг `PoseidonColumnSlice` каталогийг уншдаг ба
   Багана бүрийн Merkle-д ээлтэй хэшийг гүйцэтгэдэг (CPU-той таарч
   `PoseidonColumnBatch` зохион байгуулалт). Энэ нь ижил урсгалын бүлгийн тогтмол буферийг ашигладаг
   permute цөм шиг боловч `(states_per_lane * block_count)` дээр давтагдана
   гаралтуудыг гаргадаг тул цөм нь дарааллын илгээлтийг хорогдуулах боломжтой.
3. `poseidon_trace_fused`: ул мөрийн хүснэгтийн эх/навчны боловсруулалтыг тооцоолно
   нэг дамжлагад. Хайлуулсан цөм нь `PoseidonFusedArgs`-г ашигладаг тул хост
   зэргэлдээ бус бүс нутгууд болон `leaf_offset`/`parent_offset`-г дүрсэлж болно, мөн
   Энэ нь бүх дугуй/MDS хүснэгтүүдийг бусад цөмтэй хуваалцдаг.

### Тушаалын хуваарь ба хостын гэрээ- Цөмийн илгээлт бүр `MetalPipelines::command_queue`-ээр дамждаг
  дасан зохицох хуваарь гаргагч (зорилтот ~2 мс) болон дарааллын сэнсний хяналтыг хэрэгжүүлдэг
  `FASTPQ_METAL_QUEUE_FANOUT`-ээр дамжуулан ил гарсан ба
  `FASTPQ_METAL_COLUMN_THRESHOLD`. `with_metal_state`-д халаах зам
  Посейдоны бүх гурван цөмийг урд талд нь эмхэтгэдэг тул эхний диспетч нь тийм биш юм
  дамжуулах хоолой үүсгэх торгууль төлөх.
- Threadgroup хэмжээ нь одоо байгаа Металл FFT/LDE өгөгдмөлүүдийг тусгадаг: зорилт нь
  Бүлэг бүрт 256 сэдвээс бүрдэх хатуу хязгаар бүхий илгээлт бүрт 8,192 хэлхээс. The
  хост бага чадалтай төхөөрөмжүүдийн `states_per_lane` үржүүлэгчийг дараах байдлаар бууруулж болно.
  орчин руу залгах нь хүчингүй болсон (`FASTPQ_METAL_POSEIDON_STATES_PER_BATCH`
  WP2-B-д нэмэх) нь шэйдерийн логикийг өөрчлөхгүйгээр.
- Баганын үе шат нь FFT-д аль хэдийн ашиглагдаж байсан давхар буферт санг дагаж мөрддөг
  дамжуулах хоолой. Poseidon цөмүүд нь тухайн үе шатны буферт түүхий заагчийг хүлээн авдаг
  мөн санах ой-детерминизмыг хадгалдаг глобал овоолгын хуваарилалтад хэзээ ч бүү хүр
  CUDA хосттой нийцсэн.

### Хуваалцсан тогтмолууд

--д тайлбарласан `PoseidonSnapshot` манифест
  `docs/source/fastpq/poseidon_metal_shared_constants.md` одоо каноник болсон
  дугуй тогтмол ба MDS матрицын эх үүсвэр. Металл хоёулаа (`poseidon2.metal`)
  болон CUDA (`fastpq_cuda.cu`) цөмүүд нь манифест гарч ирэх бүрт дахин сэргээгдэх ёстой.
  өөрчлөлтүүд.
- WP2-B нь ажиллах үед манифест уншдаг жижиг хост дуудагч нэмэх болно
  SHA-256-г телеметрийн (`acceleration.poseidon_constants_sha`) ялгаруулдаг тул
  Паритын хяналтын самбар нь шэйдерийн тогтмолууд нийтлэгдсэнтэй таарч байгааг баталж чадна
  агшин зуурын зураг.
- Халаалтын үед бид `TOTAL_ROUNDS x STATE_WIDTH` тогтмолуудыг a руу хуулна
  `MTLBuffer` ба төхөөрөмж бүрт нэг удаа байршуулна уу. Дараа нь цөм бүр өгөгдлийг хуулдаг
  түүний хэсгийг боловсруулахын өмнө threadgroup санах ой руу оруулснаар детерминистикийг баталгаажуулна
  нислэгийн үед олон командын буфер ажиллаж байсан ч захиалга хийх.

### Баталгаажуулах дэгээ

- Нэгжийн туршилтууд (`cargo test -p fastpq_prover --features fastpq-gpu`) өсөх болно
  суулгагдсан шэйдер тогтмолуудыг хэш болгож, тэдгээрийг харьцуулах баталгаа
  GPU бэхэлгээний багцыг ажиллуулахын өмнө манифестын SHA.
- Одоо байгаа цөмийн статистикийг сэлгэх (`FASTPQ_METAL_TRACE_DISPATCH`,
  `FASTPQ_METAL_QUEUE_FANOUT`, дарааллын гүнийн телеметр) шаардлагатай нотлох баримт болно
  WP2-н гаралтын хувьд: туршилтын ажил бүр нь хуваарьлагч хэзээ ч зөрчдөггүй гэдгийг батлах ёстой
  тохируулагдсан fan-out ба хайлсан мөрийн цөм нь дарааллыг доор байлгадаг
  дасан зохицох цонх.
- Swift XCFramework утааны оосор болон Rust жишиг гүйгчид эхэлнэ
  `acceleration.poseidon.permute_p90_ms{cpu,metal}`-г экспортлож байгаа тул WP2-D диаграмм хийх боломжтой
  Телеметрийн шинэ хангамжийг дахин зохион бүтээхгүйгээр металл-процессорын дельта.
