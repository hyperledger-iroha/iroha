# အလိုအလျောက်စနစ်အေးဂျင့် အကောင်အထည်ဖော်မှုလမ်းညွှန်

ဤစာမျက်နှာသည် မည်သည့် အော်တိုမက်တစ် အေးဂျင့်အတွက် လုပ်ငန်းလည်ပတ်မှုဆိုင်ရာ အစောင့်အကြပ်များကို အကျဉ်းချုံးထားသည်။
Hyperledger Iroha အလုပ်ခွင်အတွင်းတွင် အလုပ်လုပ်သည်။ ၎င်းသည် canonical ကိုထင်ဟပ်စေသည်။
`AGENTS.md` လမ်းညွှန်ချက်နှင့် လမ်းပြမြေပုံ အကိုးအကားများကို တည်ဆောက်ခြင်း၊ မှတ်တမ်းပြုစုခြင်းနှင့်
တယ်လီမီတာ ပြောင်းလဲမှုအားလုံးသည် လူသားမှ ထုတ်လုပ်သည်ဖြစ်စေ တူညီသည်။
အလိုအလျောက်ပါဝင်ကူညီသူ။

အလုပ်တစ်ခုစီသည် အဆုံးအဖြတ်ကုဒ်များနှင့် ကိုက်ညီသော docs၊ စမ်းသပ်မှုများ၊
နှင့် လုပ်ငန်းဆောင်ရွက်မှုဆိုင်ရာ အထောက်အထားများ။ အောက်ဖော်ပြပါ ကဏ္ဍများကို အဆင်သင့် ရည်ညွှန်းချက်အဖြစ် သဘောထားပါ။
`roadmap.md` ပစ္စည်းများကို ထိခြင်း သို့မဟုတ် အပြုအမူမေးခွန်းများကို ပြန်ကြားခြင်း။

## Quickstart Commands များ

| အက် | အမိန့် |
|--------|---------|
| အလုပ်ခွင် | တည်ဆောက်ပါ။ `cargo build --workspace` |
| စမ်းသပ်မှုအစုံ | အပြည့်အစုံကို လုပ်ဆောင်ပါ။ `cargo test --workspace` *(ပုံမှန်အားဖြင့် နာရီများစွာကြာသည်)* |
| ငြင်းဆို-ပုံသေသတိပေးချက်များ | ဖြင့် clippy ကိုဖွင့်ပါ။ `cargo clippy --workspace --all-targets -- -D warnings` |
| ဖော်မတ်သံချေးကုဒ် | `cargo fmt --all` *(တည်းဖြတ် 2024)* |
| သေတ္တာတစ်လုံးကို စမ်းသပ်ပါ။ `cargo test -p <crate>` |
| စမ်းသပ်ခြင်း | `cargo test -p <crate> <test_name> -- --nocapture` |
| လျင်မြန်သော SDK စမ်းသပ်မှုများ | `IrohaSwift/` မှထွက်သည် `swift test` |

## အလုပ်အသွားအလာ အခြေခံအချက်များ

- မေးခွန်းများမဖြေမီ သို့မဟုတ် ယုတ္တိဗေဒပြောင်းလဲခြင်းမပြုမီ သက်ဆိုင်ရာကုဒ်လမ်းကြောင်းများကိုဖတ်ပါ။
- ကြီးမားသောလမ်းပြမြေပုံကို ခွဲခြမ်း၍ရနိုင်သော ကတိပြုချက်များအဖြစ်၊ အလုပ်ကို ဘယ်တော့မှ ပြတ်ပြတ်သားသား ငြင်းပယ်ပါ။
- ရှိပြီးသား လုပ်ငန်းခွင်တွင် အသင်းဝင်အဖြစ်နေပါ၊ အတွင်းသေတ္တာများကို ပြန်သုံးပါ၊ လုပ်ဆောင်ပါ။
  ** မဟုတ်ပါ*
- ဟာ့ဒ်ဝဲက သတ်မှတ်ပေးထားသည့် စွမ်းဆောင်ချက် အလံများနှင့် စွမ်းရည်ခလုတ်များကို အသုံးပြုပါ။
  အရှိန်မြှင့်စက်များ; ပလက်ဖောင်းတိုင်းတွင် ရရှိနိုင်သော အဆုံးအဖြတ်ပေးသော ဆုတ်ယုတ်မှုများကို ထိန်းသိမ်းထားပါ။
- လုပ်ဆောင်နိုင်သောပြောင်းလဲမှုများနှင့်အတူ စာရွက်စာတမ်းများနှင့် Markdown အကိုးအကားများကို အပ်ဒိတ်လုပ်ပါ။
  ထို့ကြောင့် docs သည် လက်ရှိအပြုအမူကို အမြဲဖော်ပြသည်။
- အသစ် သို့မဟုတ် ပြုပြင်ထားသော လုပ်ဆောင်ချက်တိုင်းအတွက် အနည်းဆုံး ယူနစ်စမ်းသပ်မှုတစ်ခုကို ထည့်ပါ။ အင်လိုင်းကို ဦးစားပေးပါ။
  နယ်ပယ်ပေါ်မူတည်၍ `#[cfg(test)]` မော်ဂျူးများ သို့မဟုတ် သေတ္တာ၏ `tests/` ဖိုင်တွဲ။
- အလုပ်ပြီးပါက `status.md` ကို အတိုချုပ်နှင့် အကိုးအကားဖြင့် အပ်ဒိတ်လုပ်ပါ။
  သက်ဆိုင်ရာဖိုင်များ၊ `roadmap.md` ကို အလုပ်လိုသေးသော အရာများကို အာရုံစိုက်ထားပါ။

## အကောင်အထည်ဖော်ရေး Guardrails

### အမှတ်စဉ်နှင့် ဒေတာမော်ဒယ်များ
- နေရာတိုင်းတွင် Norito ကိုသုံးပါ (`norito::{Encode, Decode}` မှတဆင့် binary၊
  `norito::json::*` မှတဆင့် JSON)။ တိုက်ရိုက် serde/`serde_json` အသုံးပြုမှုကို မထည့်ပါနှင့်။
- Norito payload များသည် ၎င်းတို့၏ အပြင်အဆင် (ဗားရှင်း byte သို့မဟုတ် header flags) ကို ကြော်ငြာရပါမည်။
  နှင့် ဖော်မတ်အသစ်များသည် သက်ဆိုင်ရာ စာရွက်စာတမ်း အပ်ဒိတ်များ လိုအပ်သည် (ဥပမာ၊
  `norito.md`၊ `docs/source/da/*.md`)။
- ကမ္ဘာဦးဒေတာ၊ သရုပ်ပြမှုများနှင့် ကွန်ရက်ချိတ်ဆက်မှုဆိုင်ရာ ပေးဆောင်မှုများသည် အဆုံးအဖြတ်အတိုင်း ရှိနေသင့်သည်။
  ထို့ကြောင့် တူညီသော သွင်းအားစုများရှိသော ရွယ်တူနှစ်ဦးသည် တူညီသော ဟက်ဂ်များကို ထုတ်လုပ်သည်။

### ဖွဲ့စည်းမှုပုံစံနှင့် Runtime အပြုအမူ
- ပတ်ဝန်းကျင် ကိန်းရှင်အသစ်များထက် `crates/iroha_config` တွင်နေထိုင်သည့် ခလုတ်များကို ဦးစားပေးပါ။
  တည်ဆောက်သူများ သို့မဟုတ် မှီခိုမှုထိုးသွင်းခြင်းမှတစ်ဆင့် ကြိုးတန်ဖိုးများကို အတိအလင်းဖော်ပြသည်။
- IVM syscalls သို့မဟုတ် opcode အမူအကျင့်—ABI v1 သည် နေရာတိုင်းတွင် မည်သည့်အခါမျှ ဂိတ်မပေါက်ပါနှင့်။
- config ရွေးချယ်စရာအသစ်များကို ပေါင်းထည့်သောအခါ၊ ပုံသေများ၊ docs နှင့် ဆက်စပ်မှုမှန်သမျှကို အပ်ဒိတ်လုပ်ပါ။
  နမူနာပုံစံများ (`peer.template.toml`၊ `docs/source/configuration*.md` စသည်ဖြင့်)။### ABI၊ Syscalls နှင့် Pointer အမျိုးအစားများ
- ABI မူဝါဒကို ခြွင်းချက်မရှိအဖြစ် သဘောထားပါ။ syscalls သို့မဟုတ် pointer အမျိုးအစားများကို ထည့်ခြင်း/ဖယ်ရှားခြင်း။
  အပ်ဒိတ်လုပ်ရန် လိုအပ်သည်-
  - `ivm::syscalls::abi_syscall_list` နှင့် `crates/ivm/tests/abi_syscall_list_golden.rs`
  - `ivm::pointer_abi::PointerType` နှင့် ရွှေစမ်းသပ်မှုများ
  - ABI hash ပြောင်းလဲသည့်အခါတိုင်း `crates/ivm/tests/abi_hash_versions.rs`
- အမည်မသိ syscalls များသည် `VMError::UnknownSyscall` သို့ မြေပုံရှိရမည်၊ နှင့် manifest များ လိုအပ်သည်
  ဝင်ခွင့်စစ်ဆေးမှုများတွင် လက်မှတ်ရေးထိုးထားသော `abi_hash` တန်းတူညီမျှမှုစစ်ဆေးမှုများကို ဆက်လက်ထိန်းသိမ်းထားပါ။

### Hardware Acceleration & Determinism
- လျှို့ဝှက်သင်္ကေတအသစ်များ သို့မဟုတ် လေးလံသောသင်္ချာများသည် ဟာ့ဒ်ဝဲကို အရှိန်မြှင့်၍ ပို့ဆောင်ရပါမည်။
  အဆုံးစွန်သော ဆုတ်ယုတ်မှုများကို ထိန်းသိမ်းထားစဉ် (METAL/NEON/SIMD/CUDA) လမ်းကြောင်းများ။
- အဆုံးအဖြတ်မရှိသော အပြိုင်လျှော့ချမှုများကို ရှောင်ကြဉ်ပါ။ ဦးစားပေးသည် အပေါ်ထပ်တူညီသော အထွက်များဖြစ်သည်။
  ဟာ့ဒ်ဝဲလ် ကွဲပြားနေရင်တောင်မှ သက်တူရွယ်တူအားလုံး။
- Norito နှင့် FASTPQ ပစ္စည်းများကို မျိုးပွားနိုင်စေရန် SRE သည် ရေယာဉ်စုတစ်ခုလုံးကို စာရင်းစစ်နိုင်သည်
  telemetry ။

### စာရွက်စာတမ်းနှင့်အထောက်အထား
- ပေါ်တယ် (`docs/portal/...`) တွင် အများသူငှာ ရင်ဆိုင်နေရသော doc ပြောင်းလဲမှု တစ်ခုခုကို မှန်ကြည့်သည့်အခါ၊
  သက်ဆိုင်သောကြောင့် Docs ဆိုက်သည် Markdown ရင်းမြစ်များနှင့်အတူ လက်ရှိတည်ရှိနေပါသည်။
- အလုပ်အသွားအလာအသစ်များကို မိတ်ဆက်သောအခါ၊ ရှေ့ပြေးစာအုပ်များ၊ အုပ်ချုပ်မှုမှတ်စုများ သို့မဟုတ် ထည့်ပါ။
  အစမ်းလေ့ကျင့်နည်း၊ ပြန်လှည့်ခြင်းနှင့် အထောက်အထားများ ဖမ်းယူနည်းတို့ကို ရှင်းပြသည့် စစ်ဆေးစာရင်းများ။
- အကြောင်းအရာကို Akkadian သို့ ဘာသာပြန်သောအခါ၊ ရေးသားထားသော အဓိပ္ပါယ်ဖော်ချက်များကို ပေးဆောင်ပါ။
  အသံထွက်အသုံးအနှုန်းများထက် ကျူနီဖောင်းတွင်။

### စမ်းသပ်ခြင်းနှင့် တန်ဆာပလာ မျှော်မှန်းချက်များ
- သက်ဆိုင်ရာ စမ်းသပ်မှုအစုံများကို စက်တွင်းတွင် လုပ်ဆောင်ပါ (`cargo test`၊ `swift test`၊
  ပေါင်းစည်းခြင်းကြိုးများ) နှင့် PR စမ်းသပ်မှုအပိုင်းရှိ အမိန့်များကို မှတ်တမ်းတင်ပါ။
- CI guard scripts (`ci/*.sh`) နှင့် ဒက်ရှ်ဘုတ်များကို telemetry အသစ်ဖြင့် ထပ်တူပြုထားပါ။
- proc-macro အတွက်၊ ရောဂါရှာဖွေမှုများကို လော့ခ်ချရန် `trybuild` UI စမ်းသပ်မှုများနှင့် ယူနစ်စမ်းသပ်မှုများကို တွဲချိတ်ပါ။

## တင်ပို့ရန် အဆင်သင့်စာရင်း

1. Code compiles နှင့် `cargo fmt` သည် ကွဲပြားမှုမရှိပါ။
2. Updated docs (workspace Markdown plus portal mirrors) အသစ်ကို ဖော်ပြပါ။
   အပြုအမူ၊ CLI အလံအသစ်များ သို့မဟုတ် ပြင်ဆင်သည့်ခလုတ်များ။
3. စမ်းသပ်မှုများသည် ကုဒ်လမ်းကြောင်းအသစ်တိုင်းကို အကျုံးဝင်ပြီး ဆုတ်ယုတ်သွားသည့်အခါတွင် အဆုံးအဖြတ်ကျရှုံးသည်။
