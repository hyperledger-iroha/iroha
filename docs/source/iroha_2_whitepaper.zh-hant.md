# Iroha v2.0

Hyperledger Iroha v2 是一個確定性、拜占庭容錯的分佈式賬本，強調
模塊化架構、強大的默認設置和易於使用的 API。該平台作為一組 Rust 板條箱發貨
可以嵌入到定制部署中或一起使用來操作生產區塊鍊網絡。

---

## 1. 概述

Iroha 2 延續了 Iroha 1 引入的設計理念：提供精選的集合
開箱即用的功能，使運營商無需編寫大量自定義內容即可建立網絡
代碼。 v2版本將執行環境、共識管道和數據模型整合為一個
單一有凝聚力的工作空間。

v2 系列專為想要運營自己的許可或聯盟的組織而設計
區塊鏈。每個部署都運行自己的共識網絡，保持獨立治理，並且可以定制
配置、創世數據和升級節奏，無需依賴第三方。共享工作空間
允許多個獨立網絡針對完全相同的代碼庫構建，同時選擇功能和
與其用例相匹配的策略。

Iroha 2 和 SORA Nexus (Iroha 3) 都運行相同的 Iroha 虛擬機 (IVM)。開發者可以編寫 Kotodama
合約一次並將其部署在自託管網絡或全局 Nexus 分類賬上，無需重新編譯或
分叉執行環境。
### 1.1 與 Hyperledger 生態系統的關係

Iroha 組件旨在與其他 Hyperledger 項目互操作。共識、數據模型和
序列化包可以在復合堆棧中或與 Fabric、Sawtooth 和 Besu 部署一起重複使用。
通用工具（例如 Norito 編解碼器和治理清單）有助於保持接口在整個系統中的一致
生態系統，同時允許 Iroha 提供固執己見的默認實現。

### 1.2 客戶端庫和 SDK

為了確保一流的移動和網絡體驗，該項目發布了維護的 SDK：

- `IrohaSwift` 適用於 iOS 和 macOS 客戶端，在確定性回退後集成 Metal/NEON 加速。
- `iroha_js` 適用於 JavaScript 和 TypeScript 應用程序，包括 Kaigi 構建器和 Norito 幫助器。
- `iroha_python` 用於 Python 集成，具有 HTTP、WebSocket 和遙測支持。
- `iroha_cli` 用於終端驅動的管理和腳本編寫。

語言和平台。

### 1.3 設計原則- **確定性第一：** 每個節點執行相同的代碼路徑並在給定相同的情況下產生相同的結果
  輸入。 SIMD/CUDA/NEON 路徑是功能門控的，並回退到確定性標量實現。
- **可組合模塊：** 網絡、共識、執行、遙測和存儲均位於專用的模塊中
  板條箱，因此嵌入器可以採用子集，而無需攜帶整個堆棧。
- **顯式配置：** 行為旋鈕通過 `iroha_config` 呈現；環境切換是
  僅限於開發人員的便利。
- **安全默認值：** 規範編解碼器、嚴格的指針 ABI 實施和版本化清單使
  跨網絡升級可預測。

## 2. 平台架構

### 2.1 節點組成

Iroha 節點運行多個協作服務：

- **Torii (`iroha_torii`)** 公開用於事務、查詢、流事件和
  遙測（`/v1/...` 端點）。
- **核心 (`iroha_core`)** 協調驗證、共識、執行、治理和狀態管理。
- **Sumeragi (`iroha_core::sumeragi`)** 通過視圖更改實現 NPoS 就緒共識管道，
  可靠的廣播數據可用性和提交證書。請參閱
  詳情請參閱[Sumeragi共識指南](./sumeragi.md)。
- **Kura (`iroha_core::kura`)** 在磁盤上保留規範塊、恢復 sidecar 和見證元數據。
- **世界狀態視圖 (`iroha_core::state`)** 存儲用於驗證的權威內存快照
  和查詢。
- **Iroha 虛擬機 (`ivm`)** 執行 Kotodama 字節碼 (`.to`) 並強制執行指針 ABI 策略。
- **Norito (`crates/norito`)** 為每種在線類型提供確定性二進制和 JSON 序列化。
- **遙測 (`iroha_telemetry`)** 導出 Prometheus 指標、結構化日誌記錄和流事件。
- **P2P (`iroha_p2p`)** 管理對等點之間的八卦、拓撲和安全連接。

### 2.2 組網和拓撲

Iroha 對等點維護從提交狀態派生的有序拓撲。每輪共識都會選出一位領導者，
驗證集、代理尾部和集 B 驗證器。交易使用 Norito 編碼消息進行八卦
在領導者將它們捆綁成提案之前。可靠的廣播保證了阻塞和支持
證據到達所有誠實的同行，即使在網絡混亂的情況下也確保數據可用性。視圖更改旋轉
當錯過最後期限時的領導力，提交證書確保每個提交的塊都帶有
所有對等方使用的規範簽名集。

### 2.3 密碼學

`iroha_crypto` crate 支持密鑰管理、散列和簽名驗證：- Ed25519 是默認的驗證器密鑰方案。
- 可選後端包括 Secp256k1、TC26 GOST、BLS（用於聚合證明）和 ML-DSA 幫助程序。
- 流媒體通道將 Ed25519 身份與基於 Kyber 的 HPKE 配對，以保護 Norito 流媒體會話。
- 所有散列例程都使用確定性實現（SHA-2、SHA-3、Blake2、Poseidon2）和工作區
  審核記錄在 `docs/source/crypto/dependency_audits.md` 中。

### 2.4 流媒體和應用程序橋

- **Norito 流式傳輸（`iroha_core::streaming`、`norito::streaming`）** 提供確定性加密媒體
  以及帶有會話快照、HPKE 密鑰輪換和遙測掛鉤的數據通道。 Kaigi 會議和
  機密證據傳輸使用此通道。
- **連接橋 (`connect_norito_bridge`)** 公開為平台 SDK 提供支持的 C ABI 表面
  （Swift、Kotlin/Android），同時在底層重用 Rust 客戶端。
- **ISO 20022 橋 (`iroha_torii::iso20022_bridge`)** 將受監管的支付消息轉換為 Norito
  交易，實現與財務工作流程的互操作性，而無需繞過共識或驗證。
- 所有網橋都保留確定性 Norito 有效負載，以便下游系統可以驗證狀態轉換。

## 3.數據模型

`iroha_data_model` 包定義了所有賬本對象、指令、查詢和事件。亮點：

- **域名、賬戶和資產**使用規範的 I105 賬戶 ID（首選）； `name@dataspace` / `name@domain.dataspace` 仍然是路由
  明確提供時的別名。元數據是確定性的（`Metadata` 地圖）。數字資產支持定點
  運營； NFT 攜帶任意結構化元數據。
- **角色和權限** 使用直接映射到執行程序檢查的 Norito 枚舉令牌。
- **觸發器**（基於時間、基於塊或謂詞驅動）通過鏈上發出確定性交易
  執行人。
- **事件**通過 Torii 流並鏡像提交的狀態轉換，包括機密流和
  治理行動。
- **交易、區塊和清單**採用 Norito 編碼（`SignedTransaction`、`SignedBlockWire`）
  顯式版本標頭，確保可前向擴展的解碼。
- **定制**通過執行器數據模型進行：操作員可以註冊定制指令，
  權限和參數，同時保留確定性。
- **存儲庫 (`RepoInstruction`)** 允許捆綁確定性升級計劃（執行程序、清單和
  資產），因此可以在治理批准的情況下在鏈上管理多步部署。
- **共識工件**——例如提交證書和見證人列表——駐留在數據模型中並且
  通過黃金測試來保證 `iroha_core`、Torii 和 SDK 之間的兼容性。
- **機密註冊表和事件**捕獲受保護的資產描述符、驗證者密鑰、承諾、
  無效符和事件有效負載 (`ConfidentialEvent::{Shielded,Transferred,Unshielded}`)，因此機密流
  在不洩露明文數據的情況下保持可審計性。

## 4. 交易生命週期1. **准入：** Torii 解碼 Norito 有效負載，檢查簽名、TTL 和大小限制，然後將
   本地交易。
2. **Gossip：** 事務在拓撲中傳播；對等點通過哈希和重複接納進行重複數據刪除
   檢查。
3. **選擇：** 當前領導者從待處理集中拉取交易並執行無狀態驗證。
4. **有狀態模擬：** 候選事務在瞬態 `StateBlock` 內執行，調用 IVM 或
   內置說明。衝突或違反規則的行為會被確定性地丟棄。
5. **觸發實現：** 本輪到期的預定觸發將轉換為內部交易
   並使用相同的管道進行驗證。
6. **提案密封：** 當達到區塊限製或超時到期時，領導者會發出 Norito 編碼的
   `BlockCreated` 消息。
7. **驗證：** 驗證集中的節點重新運行無狀態/有狀態檢查。成功同行簽名
   `BlockSigned` 消息並將其轉發到確定性收集器集。
8. **提交：**收集器一旦收集到規範簽名集，就會組裝提交證書，
   廣播 `BlockCommitted`，並在本地完成該塊。
9. **應用：** 所有節點在 Kura 中記錄區塊，應用狀態更新，發出遙測/事件，清除
   從內存池提交交易，並輪換拓撲角色。

恢復路徑使用確定性廣播來重新傳輸丟失的塊，並查看更改輪換領導力
當最後期限到期時。 Sidecar 和遙測技術提供診斷見解，而不會改變共識結果。

## 5. 智能合約和執行

智能合約在 Iroha 虛擬機 (IVM) 上運行：

- **Kotodama** 將高級 `.ko` 源編譯為確定性 `.to` 字節碼。
- **指針 ABI 強制執行**確保合約通過經過驗證的指針類型與主機內存交互。
  系統調用表面在 `ivm/docs/syscalls.md` 中描述； ABI 列表經過哈希處理和版本控制。
- **系統調用和主機**涵蓋賬本狀態訪問、觸發器調度、機密原語、Kaigi 媒體
  流動性和確定性隨機性。
- **內置執行器**繼續支持資產、賬戶、權限的Iroha特殊指令（ISI）
  和治理運作。自定義執行器可以擴展指令集，同時遵循 Norito 模式。
- **機密功能**——包括屏蔽傳輸和驗證者註冊表——通過執行器公開
  指令並由具有波塞冬承諾的主機進行驗證。

## 6. 存儲和持久化- **Kura 塊存儲** 將每個最終塊寫入為帶有 Norito 標頭的 `SignedBlockWire` 有效負載，保持
  規範標頭、交易、提交證書和見證數據在一起。
- **世界狀態視圖** 將權威狀態保留在內存中以進行快速查詢。確定性快照和
  pipeline sidecar (`pipeline/sidecars.norito` + `pipeline/sidecars.index`) 支持恢復和審計。
- **狀態分層**允許對大型部署進行熱/冷分區，同時保持確定性
  驗證。
- **同步和重放**使用相同的驗證規則將提交的塊加載回狀態。確定性
  廣播確保對等方可以從鄰居處恢復丟失的數據，而無需依賴可信存儲。

## 7. 治理和經濟

- 鏈上參數（`SetParameter`）控制共識計時器、內存池限制、遙測旋鈕、費用帶、
  和功能標誌。 `kagami` 生成的 Genesis 清單安裝初始配置。
- **Kaigi** 指令管理協作會話（創建/加入/離開/結束）並提供 Norito 流媒體
  會議用例的遙測。
- **Hijiri** 提供確定性的同行和賬戶聲譽，與共識、准入相結合
  政策和費用乘數（第 16 題定點數學）。證據清單、檢查點和聲譽
  註冊表在鏈上提交，觀察者資料管理收據來源。
- **NPoS 模式**（啟用後）使用 VRF 支持的選舉窗口和權益加權委員會，同時保留
  確定性配置默認值。
- **機密註冊表**管理零知識驗證者密鑰、證明生命週期和承諾
  屏蔽流。

## 8. 客戶端體驗和工具

- **Torii API** 提供 REST 和 WebSocket 接口，用於事務、查詢、事件流、遙測和
  治理端點。 JSON 投影源自 Norito 模式。
- **CLI 工具**（`iroha_cli`、`iroha_monitor`）涵蓋管理、實時對等儀表板和管道
  檢查。
- **Genesis 工具** (`kagami`) 生成 Norito 編碼的清單、驗證器密鑰材料和配置
  模板。
- **SD​​​​K**（Swift、JS/TS、Python）提供對指令、查詢、觸發器和遙測的慣用訪問。
- `scripts/` 內的 **腳本和 CI 掛鉤** 自動執行儀表板驗證、編解碼器重新生成和煙霧
  測試。

## 9. 性能、彈性和路線圖- 當前管道的目標是**20,000 tps**，在有利的網絡下具有**2-3秒**的區塊時間
  條件，由批量簽名驗證和確定性調度支持。
- **遙測**公開 Prometheus 共識計時器、內存池佔用率、區塊傳播健康狀況的指標，
  Kaigi 使用情況和 Hijiri 聲譽更新。
- **彈性功能**包括確定性數據可用性、恢復邊車、拓撲旋轉和
  可配置的查看/更改閾值。
- 未來路線圖里程碑（參見 `roadmap.md`）繼續在 Nexus 數據空間上開展工作，增強保密性
  工具和更廣泛的硬件加速，同時保留確定性的輸出。

## 10. 操作和部署

- **工件：** Dockerfiles、Nix flake 和 `cargo` 工作流程支持可重現的構建。 `kagami` 發出
  創世清單、驗證器密鑰以及許可部署和 NPoS 部署的示例配置。
- **自託管網絡：**運營商管理自己的對等集、准入規則和升級節奏。的
  工作區支持多個獨立的 Iroha 2 網絡共存，無需協調，僅共享
  上游代碼。
- **配置生命週期：** `iroha_config` 解析用戶→實際→默認層，確保每個旋鈕都是
  顯式且受版本控制。運行時更改通過 `SetParameter` 指令流動。
- **可觀察性：** `iroha_telemetry` 導出已檢查的 Prometheus 指標、結構化日誌和儀表板數據
  通過 CI 腳本（`ci/check_swift_dashboards.sh`、`scripts/render_swift_dashboards.sh`、
  `scripts/check_swift_dashboard_data.py`）。流媒體、共識和 Hijiri 事件可通過
  WebSocket 和 `scripts/sumeragi_backpressure_log_scraper.py` 將起搏器背壓與
  遙測以排除故障。
- **測試：** `cargo test --workspace`、集成測試 (`integration_tests/`)、語言 SDK 套件和
  Norito 黃金裝置保護確定性。指針 ABI、系統調用列表和治理清單
  專門的黃金測試。
- **恢復：** Kura sidecar、確定性重播和廣播同步允許節點從磁盤恢復狀態
  或同齡人。 Hijiri 檢查點和治理清單提供了可審計的合規性快照。
