# Sora Nexus План приема данных о доступности

_Составлено: 20 февраля 2026 г. – Владелец: Рабочая группа по базовому протоколу / Группа хранения данных / DA WG_

Рабочий поток DA-2 расширяет Torii с помощью API приема больших двоичных объектов, который генерирует Norito.
метаданные и начальная репликация SoraFS. В этом документе отражены предлагаемые
схема, поверхность API и поток проверки, поэтому реализация может продолжаться без
блокировка выдающихся симуляций (последующие действия DA-1). Все форматы полезной нагрузки ДОЛЖНЫ
используйте кодеки Norito; резервные версии serde/JSON не допускаются.

## Цели

- Принимать большие капли (сегменты Тайкай, коляски, артефакты управления)
  детерминированно над Torii.
- Создание канонических манифестов Norito, описывающих большой двоичный объект, параметры кодека,
  профиль стирания и политика хранения.
- Сохранять метаданные фрагментов в горячем хранилище SoraFS и ставить в очередь задания репликации.
- Публикация намерений контактов и тегов политики в реестре и управлении SoraFS.
  наблюдатели.
- Предоставляйте квитанции о приеме, чтобы клиенты могли получить детерминированные доказательства публикации.

## Поверхность API (Torii)

```
POST /v2/da/ingest
Content-Type: application/norito+v1
```

Полезная нагрузка представляет собой `DaIngestRequest` в кодировке Norito. Ответы используют
`application/norito+v1` и верните `DaIngestReceipt`.

| Ответ | Значение |
| --- | --- |
| 202 Принято | Blob поставлен в очередь на фрагментирование/репликацию; чек вернулся. |
| 400 неверный запрос | Нарушение схемы/размера (см. проверки проверки). |
| 401 Несанкционированный | Отсутствует/неверный токен API. |
| 409 Конфликт | Дубликат `client_blob_id` с несовпадающими метаданными. |
| 413 Полезная нагрузка слишком велика | Превышено настроенное ограничение длины большого двоичного объекта. |
| 429 Слишком много запросов | Достигнут лимит ставки. |
| 500 Внутренняя ошибка | Неожиданный сбой (зарегистрированный + оповещение). |

```
GET /v2/da/proof_policies
Accept: application/json | application/x-norito
```

Возвращает версию `DaProofPolicyBundle`, полученную из текущего каталога дорожек.
Пакет рекламирует `version` (в настоящее время `1`), `policy_hash` (хэш
упорядоченный список политик), а также записи `policies`, содержащие `lane_id`, `dataspace_id`,
`alias` и принудительное `proof_scheme` (сегодня `merkle_sha256`; полосы KZG
отклонено путем приема до тех пор, пока не будут доступны обязательства KZG). Заголовок блока сейчас
фиксирует пакет через `da_proof_policies_hash`, чтобы клиенты могли закрепить
активная политика, установленная при проверке обязательств или доказательств DA. Получить эту конечную точку
прежде чем собирать доказательства, чтобы убедиться, что они соответствуют политике полосы и текущим
хэш пакета. Конечные точки списка обязательств/доказательства содержат один и тот же пакет, поэтому SDK
не требуется дополнительный обход для привязки доказательства к активному набору политик.

```
GET /v2/da/proof_policy_snapshot
Accept: application/json | application/x-norito
```

Возвращает `DaProofPolicyBundle`, содержащий упорядоченный список политик плюс
`policy_hash`, чтобы SDK могли закрепить версию, использованную при создании блока.
хеш вычисляется по массиву политики, закодированному Norito, и изменяется всякий раз, когда
`proof_scheme` полосы обновлен, что позволяет клиентам обнаруживать дрейф между
кэшированные доказательства и конфигурация цепочки.

## Предлагаемая схема Norito

```rust
/// Top-level ingest request.
pub struct DaIngestRequest {
    pub client_blob_id: BlobDigest,      // submitter-chosen identifier
    pub lane_id: LaneId,                 // target Nexus lane
    pub epoch: u64,                      // epoch blob belongs to
    pub sequence: u64,                   // monotonic sequence per (lane, epoch)
    pub blob_class: BlobClass,           // TaikaiSegment, GovernanceArtifact, etc.
    pub codec: BlobCodec,                // e.g. "cmaf", "pdf", "norito-batch"
    pub erasure_profile: ErasureProfile, // parity configuration
    pub retention_policy: RetentionPolicy,
    pub chunk_size: u32,                 // bytes (must align with profile)
    pub total_size: u64,
    pub compression: Compression,        // Identity, gzip, deflate, or zstd
    pub norito_manifest: Option<Vec<u8>>, // optional pre-built manifest
    pub payload: Vec<u8>,                 // raw blob data (<= configured limit)
    pub metadata: ExtraMetadata,          // optional key/value metadata map
    pub submitter: PublicKey,             // signing key of caller
    pub signature: Signature,             // canonical signature over request
}

pub enum BlobClass {
    TaikaiSegment,
    NexusLaneSidecar,
    GovernanceArtifact,
    Custom(u16),
}

pub struct ErasureProfile {
    pub data_shards: u16,
    pub parity_shards: u16,
    pub chunk_alignment: u16, // chunks per availability slice
    pub fec_scheme: FecScheme,
}

pub struct RetentionPolicy {
    pub hot_retention_secs: u64,
    pub cold_retention_secs: u64,
    pub required_replicas: u16,
    pub storage_class: StorageClass,
    pub governance_tag: GovernanceTag,
}

pub struct ExtraMetadata {
    pub items: Vec<MetadataEntry>,
}

pub struct MetadataEntry {
    pub key: String,
    pub value: Vec<u8>,
    pub visibility: MetadataVisibility, // public vs governance-only
}

pub enum MetadataVisibility {
    Public,
    GovernanceOnly,
}

pub struct DaIngestReceipt {
    pub client_blob_id: BlobDigest,
    pub lane_id: LaneId,
    pub epoch: u64,
    pub blob_hash: BlobDigest,          // BLAKE3 of raw payload
    pub chunk_root: BlobDigest,         // Merkle root after chunking
    pub manifest_hash: BlobDigest,      // Norito manifest hash
    pub storage_ticket: StorageTicketId,
    pub pdp_commitment: Option<Vec<u8>>,     // Norito-encoded PDP bytes
    #[norito(default)]
    pub stripe_layout: DaStripeLayout,   // total_stripes, shards_per_stripe, row_parity_stripes
    pub queued_at_unix: u64,
    #[norito(default)]
    pub rent_quote: DaRentQuote,        // XOR rent + incentives derived from policy
    pub operator_signature: Signature,
}
```> Примечание по реализации: канонические представления Rust для этих полезных нагрузок теперь живут под
> `iroha_data_model::da::types`, с оболочками запроса/получения в `iroha_data_model::da::ingest`
> и структура манифеста в `iroha_data_model::da::manifest`.

Поле `compression` сообщает, как вызывающая сторона подготовила полезную нагрузку. Torii принимает
`identity`, `gzip`, `deflate` и `zstd`, прозрачно распаковывающие байты перед
хеширование, разбиение на фрагменты и проверка дополнительных манифестов.

### Контрольный список проверки

1. Убедитесь, что заголовок запроса Norito соответствует `DaIngestRequest`.
2. Ошибка, если `total_size` отличается от канонической (распакованной) длины полезной нагрузки или превышает настроенный макс.
3. Обеспечьте выравнивание `chunk_size` (степень двойки, = 2.
5. `retention_policy.required_replica_count` должен соответствовать базовым показателям управления.
6. Проверка подписи по каноническому хешу (за исключением поля подписи).
7. Отклонять дубликат `client_blob_id`, если хэш полезной нагрузки + метаданные не идентичны.
8. При предоставлении `norito_manifest` проверьте схему + совпадения хэша пересчитываются.
   проявиться после фрагментации; в противном случае узел генерирует манифест и сохраняет его.
9. Примените настроенную политику репликации: Torii переписывает отправленные данные.
   `RetentionPolicy` с `torii.da_ingest.replication_policy` (см.
   `replication_policy.md`) и отклоняет предварительно созданные манифесты, сохранение которых
   метаданные не соответствуют принудительному профилю.

### Блокирование и процесс репликации1. Разбейте полезную нагрузку на `chunk_size`, вычислите BLAKE3 для каждого фрагмента + корень Меркла.
2. Создайте Norito `DaManifestV1` (новая структура), фиксирующая обязательства фрагмента (role/group_id),
   макет стирания (подсчет четности строк и столбцов плюс `ipa_commitment`), политика хранения,
   и метаданные.
3. Поставьте байты канонического манифеста в очередь под `config.da_ingest.manifest_store_dir`.
   (Torii записывает файлы `manifest.encoded` с ключами по полосе/эпохе/последовательности/тикету/отпечатку пальца), поэтому SoraFS
   оркестровка может принять их и связать билет хранилища с постоянными данными.
4. Опубликуйте намерения закрепления через `sorafs_car::PinIntent` с тегом управления + политикой.
5. Вызовите событие Norito `DaIngestPublished` для уведомления наблюдателей (легких клиентов,
   управление, аналитика).
6. Верните `DaIngestReceipt` (подписанный сервисным ключом DA Torii) и добавьте
   Заголовок ответа `Sora-PDP-Commitment`, содержащий кодировку base64 Norito.
   производного обязательства, чтобы SDK могли немедленно сохранить начальное значение выборки.
   Квитанция теперь включает `rent_quote` (`DaRentQuote`) и `stripe_layout`.
   чтобы отправители могли раскрыть обязательства XOR, резервную долю, бонусные ожидания PDP/PoTR,
   и размеры двумерной матрицы стирания вместе с метаданными билетов на хранение перед выделением средств.
7. Необязательные метаданные реестра:
   - `da.registry.alias` — общедоступная, незашифрованная строка псевдонима UTF-8 для заполнения записи реестра PIN-кода.
   - `da.registry.owner` — общедоступная незашифрованная строка `AccountId` для записи владельца реестра.
   Torii копирует их в сгенерированный `DaPinIntent`, чтобы последующая обработка выводов могла связывать псевдонимы.
   и владельцы без повторного анализа карты необработанных метаданных; некорректные или пустые значения отклоняются во время
   проверка приема.

  хэш манифеста + идентификатор билета.
- Обновление конвейеров наблюдения для отслеживания задержки приема, фрагментации пропускной способности,
  отставание в репликации и количество неудач.
- Ответы Torii `/status` теперь включают массив `taikai_ingest`, который отображает последние
  задержка между кодером и приемом, дрейф фронта и счетчики ошибок на (кластер, поток), что позволяет использовать DA-9
  информационные панели для получения снимков состояния непосредственно с узлов без очистки Prometheus.

### Поток потоковой передачи больших объемов полезной нагрузкиКлиенты, которым необходимо принять ресурсы, превышающие настроенный лимит одного запроса, инициируют
сеанс потоковой передачи, вызвав `POST /v2/da/ingest/chunk/start`. Torii отвечает
`ChunkSessionId` (BLAKE3 — получено из запрошенных метаданных большого двоичного объекта) и согласованный размер фрагмента.
Каждый последующий запрос `DaIngestChunk` содержит:

- `client_blob_id` — идентичен финальному `DaIngestRequest`.
- `chunk_session_id` — привязывает слайсы к работающему сеансу.
- `chunk_index` и `offset` — обеспечивают детерминированный порядок.
- `payload` — до согласованного размера чанка.
- `payload_hash` — BLAKE3-хеш среза, чтобы Torii мог проверяться без буферизации всего большого двоичного объекта.
- `is_last` — указывает на терминальный срез.

Torii сохраняет проверенные фрагменты под `config.da_ingest.manifest_store_dir/chunks/<session>/` и
записывает прогресс внутри кэша воспроизведения для соблюдения идемпотентности. Когда приземлится последний кусочек, Torii
повторно собирает полезную нагрузку на диске (потоковая передача через каталог фрагментов, чтобы избежать скачков памяти),
вычисляет канонический манифест/квитанцию точно так же, как при однократной загрузке, и, наконец, отвечает на
`POST /v2/da/ingest` путем использования промежуточного артефакта. Неудачные сеансы могут быть прерваны явно или
собираются мусором после `config.da_ingest.replay_cache_ttl`. Эта конструкция сохраняет сетевой формат
Совместим с Norito, позволяет избежать возобновляемых протоколов, специфичных для клиента, и повторно использует существующий конвейер манифеста.

## Артефакты проверки клиента Rust

SDK, в которые встроен клиент Rust, больше не требуют использования CLI для
создать канонический пакет PoR JSON. `Client` предоставляет двух помощников:

- `build_da_proof_artifact` возвращает точную структуру, сгенерированную
  `iroha app da prove --json-out`, включая предоставленные аннотации манифеста и полезной нагрузки.
  через [`DaProofArtifactMetadata`].【crates/iroha/src/client.rs:3638】
- `write_da_proof_artifact` оборачивает сборщик и сохраняет артефакт на диск.
  (по умолчанию довольно JSON + завершающая новая строка), чтобы автоматизация могла прикрепить файл
  к выпускам или пакетам доказательств управления.【crates/iroha/src/client.rs:3653】

### Пример

```rust

    da::{DaProofArtifactMetadata, DaProofConfig},
    Client,
};

    "artifacts/da/manifest.norito",
    "artifacts/da/payload.car",
);

// Build the JSON artefact in-memory.
let artifact = client.build_da_proof_artifact(
    &manifest,
    &payload,
    &DaProofConfig::default(),
    &metadata,
)?;

// Persist it next to other DA artefacts.
client.write_da_proof_artifact(
    &manifest,
    &payload,
    &DaProofConfig::default(),
    &metadata,
    "artifacts/da/proof_summary.json",

)?;

### Размещение + ремонт жгута

`cargo run -p sorafs_car --bin da_reconstruct -- --manifest <path> --chunks-dir <dir>` сейчас
вычисляет хэш размещения по `(index, role, stripe/column, offsets)` и выполняет сначала строку, затем
ремонт столбца RS(16) перед восстановлением полезной нагрузки:

- Размещение по умолчанию равно `total_stripes`/`shards_per_stripe`, если оно присутствует, и возвращается к чанку.
- Отсутствующие/поврежденные фрагменты сначала перестраиваются с контролем четности строк; оставшиеся зазоры устраняются с помощью
  полосовая (столбцовая) четность. Восстановленные фрагменты записываются обратно в каталог фрагментов, а JSON
  сводка фиксирует хэш размещения плюс счетчики восстановления строк/столбцов.
- Если четность строки+столбца не может удовлетворить недостающий набор, жгут быстро выходит из строя с невосстановимым
  индексы, чтобы аудиторы могли отмечать непоправимые проявления.
