# Обзор конфигурации

Эта страница суммирует верхнеуровневые разделы файла конфигурации Iroha (TOML) и то, чем каждый управляет. Для конкретного шаблона см. `docs/source/references/peer.template.toml`.

Сначала значения по умолчанию: конфигурационные значения подобраны для типичных развёртываний Iroha. В большинстве случаев можно запускать без изменений. Меняйте настройки только при наличии конкретной операционной необходимости (например, нестандартные порты, строгие лимиты, известные сетевые ограничения).

- Разделы, специфичные для Sora (`torii.sorafs`, `sorafs.*`, многополосные `nexus.*`), требуют запуска демона с `irohad --sora …` (или экспортом `IROHA_SORA_PROFILE=1`). Без флага загрузчик отклоняет конфигурации, включающие эти функции.
- При `nexus.enabled = false` (режим Iroha 2) конфигурации остаются однополосными: `/status` и `/v1/sumeragi/status` скрывают секции lane/dataspace, `/metrics` полностью исключает метки lane/dataspace, а хвосты статуса для lane возвращают `StatusSegmentNotFound`, поэтому SDK должны ветвиться по флагу перед отправкой параметров lane. Парсинг конфигурации отклоняет overrides lane/dataspace/routing в этом режиме — удаляйте секции `nexus.*`, если не включаете Nexus/Sora через `nexus.enabled=true` или `--sora`.
- `[common]`: Идентичность цепочки (`chain`), пара ключей узла, доверенные пиры, неявная метка домена аккаунта (`default_account_domain_label`) и дискриминант цепочки IH58 (`chain_discriminant`). Переопределяйте их, когда домен genesis или префикс сети отличаются от встроенных значений, чтобы адреса IH58/компрессированные оставались каноничными.
- Допуск валидаторов: `trusted_peers` должен содержать ключи валидаторов BLS‑Normal, а PoP задаются в `[[trusted_peers_pop]] { public_key, pop_hex }`. Карта PoP должна покрывать весь ростер; отсутствующие или неверные PoP отклоняются при разборе конфигурации. Парсинг конфигурации теперь отклоняет устаревшее отображение `trusted_peers_bls` и любые неблс‑ключи валидаторов. Транспортная идентичность по‑прежнему использует `public_key`/`private_key` для P2P/Torii, тогда как подписи консенсуса используют только ключ BLS + PoP.
- Разделение идентичности: пара `public_key`/`private_key` защищает транспорт (P2P, Torii) и может оставаться Ed25519/SECP/etc. независимо от `allowed_signing`. Консенсус использует ключ BLS‑Normal из `trusted_peers` с его PoP. Допуск транзакций — единственное место, где применяются `crypto.allowed_signing`/`allowed_curve_ids`; если хотите отправлять BLS‑подписанные транзакции, явно добавьте `bls_normal` в `allowed_signing` и соответствующий id кривой в `crypto.curves.allowed_curve_ids`. Консенсус продолжит работать с BLS‑валидаторами даже если `allowed_signing` оставлен на дефолтных Ed25519/secp.
- `[genesis]`: расположение подписанного файла genesis и публичный ключ для его проверки. Ссылаемый манифест содержит семейства параметров вроде `transaction.max_signatures`, `transaction.max_instructions`, `transaction.ivm_bytecode_size`, `transaction.max_tx_bytes`, `transaction.max_decompressed_bytes`; обновляйте манифест при ужесточении лимитов допуска. Узлы, стартующие с пустого хранилища и без локального `genesis.file`, могут получить genesis у доверенных пиров, при необходимости закрепив `expected_hash`, задав `bootstrap_allowlist` (по умолчанию `trusted_peers`) и ограничив передачу `bootstrap_max_bytes` и `bootstrap_response_throttle_ms`. Попытки bootstrap уважают `bootstrap_request_timeout_ms`, `bootstrap_retry_interval_ms`, `bootstrap_max_attempts`, `bootstrap_enabled`; отвечающие всегда проверяют совпадения chain‑id/публичного ключа/хэша и отклоняют слишком большие payload или неразрешённые пиры.
- `[network]`: P2P‑адреса, периоды/размеры gossip, обновление DNS, ограниченные ёмкости очередей и опции транспорта (TLS/QUIC при включении).
- `[ivm]`: настройки runtime VM. `ivm.memory_budget_profile` (по умолчанию: `compute.default_resource_profile`) выбирает профиль вычислительных ресурсов, который задаёт `max_stack_bytes` как потолок стека гостя.
- `[tiered_state]`: настройки WSV hot/cold (`enabled`, `hot_retained_keys`, `hot_retained_bytes`, `hot_retained_grace_snapshots`, `cold_store_root`, `da_store_root`, `max_snapshots`, `max_cold_bytes`). Если `cold_store_root` не задан, снапшоты пишутся в `da_store_root`; смена любой из корневых директорий сбрасывает метаданные tiering.
- `[ivm.banner]` (по умолчанию: `show = true`, `beep = true`): управляет стартовым баннером Norito/IVM и опциональным джинглом. Флаг `beep` учитывается только если бинарь собран с feature `beep`. Это заменяет устаревший env‑шим `IROHA_BEEP` — отключайте баннер/бип через конфиг, а не через env.
- `[concurrency]`: размер пулов потоков и бюджеты стека для worker‑ов scheduler/prover и гостя IVM. `gas_to_stack_multiplier` (по умолчанию `4`) ограничивает стек гостя как `min(guest_stack_bytes, gas_limit * multiplier, stack_budget)` с минимумом 64 KiB и максимумом 1 GiB; `stack_budget` берётся из профиля `ivm.memory_budget_profile`. Запросы вне окна клэмпятся, логируются и экспортируются через `/status.stack` и метрики `iroha_ivm_stack_*` (включая счётчики pool‑fallback/budget‑hit). Уменьшайте множитель для ограничения рекурсии на малом gas, увеличивайте (в пределах лимита стека гостя) для более глубоких фреймов Kotodama.
- `network.peer_gossip_period_ms` (по умолчанию: `250`ms) также задаёт cadence обновления топологии. Env‑оверрайд `IROHA_P2P_TOPOLOGY_UPDATE_MS` больше не используется; настраивайте поле конфигурации. Периоды gossip (`block_gossip_period_ms`, `transaction_gossip_period_ms`, `peer_gossip_period_ms`) и `idle_timeout_ms` ограничены минимумом 100 ms, чтобы избежать tight‑loops.
- `network.trust_gossip` (по умолчанию: `true`) объявляет поддержку TrustGossip‑кадров. Значение `false` отключает возможность на этапе рукопожатия, отбрасывая TrustGossip‑сообщения при сохранении gossip адресов пиров. Отбрасывания помечаются `p2p_trust_gossip_skipped_total{direction,reason}` для выявления opt‑out или отказавших peers.
- Параметры доверия в `[network]` (`trust_decay_half_life_ms`, `trust_penalty_bad_gossip`, `trust_penalty_unknown_peer`, `trust_min_score`) применяются в permissioned overlays: trust‑кадры, ссылающиеся на пиров вне активной топологии, получают штраф unknown‑peer, увеличивают `p2p_trust_penalties_total{reason="unknown_peer"}`, а отправители ниже `trust_min_score` игнорируются до восстановления после decay. Публичные/NPoS overlays пропускают штраф unknown‑peer, чтобы trust gossip оставался открытым.
- Настройки gossip транзакций: `network.transaction_gossip_drop_unknown_dataspace` отбрасывает батчи для dataspaces, отсутствующих в каталоге lane; `network.transaction_gossip_restricted_target_cap` ограничивает restricted‑fanout (по умолчанию полная commit‑топология); `network.transaction_gossip_public_target_cap` ограничивает fanout публичной lane (по умолчанию 16, `null` для broadcast); `network.transaction_gossip_public_target_reshuffle_ms` и `network.transaction_gossip_restricted_target_reshuffle_ms` задают частоту reshuffle целей по плоскостям (по умолчанию `transaction_gossip_period_ms`, clamp ≥100 ms); `network.transaction_gossip_resend_ticks` задаёт, сколько периодов gossip проходит перед повторной отправкой тех же транзакций (чем больше, тем меньше дубликатов); `network.transaction_gossip_restricted_fallback` (`drop` | `public_overlay`) определяет, падает ли restricted gossip к online‑пирам при пустой commit‑топологии; `network.transaction_gossip_restricted_public_payload` (`refuse` | `forward`, по умолчанию `refuse`) сейчас запрещает утечки restricted payload при наличии только public overlay (forward заблокирован до появления шифрованного транспорта). Телеметрия отображает политику drop/forward через gauge и записи `/status.tx_gossip` (caps, targets/peers, fallback‑surface, reason/outcome), чтобы операторы могли реагировать на блокировки/перенаправления.
- `network.soranet_handshake.pow`: конфигурация challenge для допуска SoraNet. Поля: `required`, `difficulty`, `max_future_skew_secs`, `min_ticket_ttl_secs`, `ticket_ttl_secs`, `revocation_store_capacity`, `revocation_max_ttl_secs` и `revocation_store_path`. Revocation store задаёт, сколько подписанных билетов хранится на диске и как долго они валидны для replay‑детекции.
- `network.soranet_handshake.pow.puzzle`: переключатели Argon2 puzzle. Установите `enabled = true`, чтобы требовать memory‑hard tickets, и настройте `memory_kib` (>=4096), `time_cost` и `lanes` (1–16). При включении relay всё ещё уважает `difficulty`, `max_future_skew_secs` и `min_ticket_ttl_secs`; отключение блока возвращает классический hashcash‑PoW.
- `network.soranet_handshake.kem_suite`: опциональный текстовый override (`"mlkem512"`, `"mlkem768"` или `"mlkem1024"`; также принимаются `kyber*`‑алиасы) для профиля ML‑KEM в SoraNet‑handshake. Если задан, заменяет `kem_id` и валидирует длины Kyber‑ключей для выбранного профиля.
- `[sumeragi]`: роль консенсуса (validator/observer) и настройки коллектора (K, redundant‑send r), а также slack‑опции по времени. Отслеживание DA включается через `sumeragi.da.enabled`; evidence доступности (availability evidence или quorum RBC `READY`) носит advisory‑характер и не блокирует commit. Отсутствующий локальный payload подтягивается через RBC‑доставку или BlockCreated/блок‑синк. Backpressure по очередям настраивается через `sumeragi.advanced.queues.votes`, `sumeragi.advanced.queues.block_payload`, `sumeragi.advanced.queues.rbc_chunks`, `sumeragi.advanced.queues.blocks`, `sumeragi.advanced.queues.control`. `sumeragi.mode_flip.enabled` (по умолчанию `true`) — kill‑switch для live‑переключений permissioned↔NPoS: при `false` узел объявляет staged‑mode через статус/телеметрию, но отказывается переключаться до восстановления флага. Ошибки персистенции backoff‑ятся по `sumeragi.persistence.kura_retry_interval_ms`/`sumeragi.persistence.kura_retry_max_attempts`, чтобы блоки оставались pending до успеха или исчерпания retries; `sumeragi.persistence.commit_inflight_timeout_ms` прерывает зависшие commit‑job, чтобы предложения не замирали навсегда. Получение недостающих блоков предпочитает commit certificate‑подписантов в течение `sumeragi.recovery.missing_block_signer_fallback_attempts` попыток (дефолт 1; 0 — отключить предпочтение) перед fallback на полную commit‑топологию. Дрейф хэша состава отображается через `sumeragi.gating.membership_mismatch_alert_threshold` (количество последовательных рассогласований до алерта) и `sumeragi.gating.membership_mismatch_fail_closed` (отбрасывать сообщения консенсуса от пиров, которые все еще расходятся). Интервал предложений pacemaker выводится из `SumeragiParameters.block_time_ms (scaled by pacing_factor_bps)`, `sumeragi.advanced.npos.timeouts.propose_ms`, `sumeragi.advanced.pacemaker.rtt_floor_multiplier` и ограничивается `sumeragi.advanced.pacemaker.max_backoff_ms`; увеличивайте RTT‑floor или block‑time, чтобы замедлить предложения на высоких задержках. Поля `sumeragi.advanced.npos.timeouts.*_ms` и `sumeragi.npos.vrf.*`, если не заданы, выводятся из `SumeragiParameters.block_time_ms (scaled by pacing_factor_bps)` и `sumeragi.npos.epoch_length_blocks` по дефолтным соотношениям. Стэши RBC в ожидании ограничиваются между сессиями параметром `sumeragi.advanced.rbc.pending_session_limit`. RBC READY/DELIVER quorum всегда следует commit‑топологии (`Topology::min_votes_for_commit()`).
- `[torii]`: настройки публичного API‑сервера (адрес, лимиты размера запросов, параметры query‑store).
- `torii.data_dir` (по умолчанию: `./storage/torii`): базовый каталог для персистентности Torii (attachments, реестр webhooks, DA‑кэши). Заменяет устаревший env‑шим `IROHA_TORII_DATA_DIR`; тесты могут временно переопределять через `data_dir::OverrideGuard`.
- `torii.events_buffer_capacity` (по умолчанию: `10000`): размер broadcast‑канала для `/v1/events/sse` и постановки webhooks в очередь. Уменьшайте значение, чтобы ограничить память/backpressure при отстающих подписчиках.
- `torii.ws_message_timeout_ms` (по умолчанию: `10000`): timeout сообщений WebSocket (чтение/запись) для стримов событий/блоков Torii; увеличьте для медленных клиентов или высоких задержек.
- `torii.app_api.*`: параметры пагинации/давления по умолчанию для JSON‑эндпоинтов. `default_list_limit` задаёт `limit` при отсутствии, `max_list_limit` и `max_fetch_size` ограничивают размеры страницы/выборки, `rate_limit_cost_per_row` задаёт стоимость строки под лимитером app API.
- `torii.webhook.*`: параметры доставки/backoff для webhook‑worker. `queue_capacity` ограничивает ожидающие доставки на диске (по умолчанию `10000`), `max_attempts` ограничивает retries (по умолчанию `12`), `backoff_initial_ms`/`backoff_max_ms` задают экспоненциальное окно повторов (по умолчанию `1000`/`60000`), а `{connect,write,read}_timeout_ms` задают HTTP‑timeouts попыток доставки.
- `torii.push.*` (feature stub): переключатели push‑bridge и лимиты. `enabled` включает мост; `rate_per_minute`/`burst` применяют token‑bucket к исходящим запросам, `max_topics_per_device` ограничивает темы на токен, а `connect_timeout_ms`/`request_timeout_ms` ограничивают попытки HTTP к FCM/APNS. Укажите `fcm_api_key` или APNS `apns_endpoint` + `apns_auth_token` для активации соответствующего провайдера.
- `torii.da_ingest.*`: реплей‑кэш и конфигурация spool манифестов для DA‑пайплайна.
  - `telemetry_cluster_label` (опционально): задаёт label `cluster` для метрик Taikai ingest/viewer, чтобы dashboards различали окружения (например, `region-a` vs `staging`).
    Установите `governance_metadata_key_hex` как 32‑байтовый ключ ChaCha20‑Poly1305, когда Torii должен запечатывать governance‑метаданные; парно задайте `governance_metadata_key_label` (короткий идентификатор вроде `"primary"`), чтобы зашифрованные записи объявляли, какой ключ создал ciphertext. Torii автоматически шифрует governance‑метаданные в открытом виде и отклоняет манифесты с отсутствующими или несовпадающими labels в pre‑encrypted payloads. Вложенная таблица `replication_policy` задаёт `RetentionPolicy` для класса blob (см. `docs/source/da/replication_policy.md`). Переопределяйте `default_retention`, чтобы изменить базовые окна hot/cold, и добавляйте записи в `replication_policy.overrides` (например, `class = "taikai_segment"`) для ужесточения отдельных классов; Torii переписывает манифесты, приведя их к этим значениям, и отклоняет несоответствующие envelopes.
  - `debug_match_filters` (по умолчанию: `false`) включает детерминированные debug‑trace фильтров, ранее закрытые env‑переключателем `TORII_DEBUG_MATCH`. Держите выключенным в проде.
- `[torii.iso_bridge.reference_data]`: пути к ISO‑референс‑набором (`isin_crosswalk_path`, `bic_lei_path`, `mic_directory_path`), cadence ingest (`refresh_interval_secs`) и опциональный `cache_dir` для хранения копий плюс метаданных происхождения (версия/источник/контрольная сумма) каждого снапшота.
- `[torii.soranet_privacy_ingest]`: guards для `/v1/soranet/privacy/{event,share}`. `enabled` должен быть `true` для экспонирования endpoints; `require_token` + `tokens` требуют заголовок `X-SoraNet-Privacy-Token` (или `X-API-Token`), `allow_cidrs` ограничивает отправителей по CIDR (пустой список по умолчанию запрещает), `rate_per_sec`/`burst` применяют token‑bucket limiter. Отказы возвращают `401/403/429` с подсказками retry и помеченными телеметрическими счётчиками, чтобы операторы могли отследить злоупотребления; см. `docs/source/sorafs_authz_runbook.md` для чек‑листа rollout.
- `[torii.sorafs]`: параметры кэша SoraFS discovery на стороне Torii. Значения копируются из канонических `[sorafs.*]` при парсинге, поэтому задавайте корневые секции; torii‑view нужен для runtime‑снапшотов/телеметрии.
- `[sorafs.discovery]`: канонический toggle discovery и allow‑list возможностей. `discovery_enabled` по умолчанию `false` (профили Sora вроде `tests/fixtures/base.toml` включают явно), `known_capabilities` поставляется как `["torii_gateway", "chunk_range_fetch", "vendor_reserved"]`, а опциональная `[sorafs.discovery.admission]` указывает на каталог governance‑envelopes (`envelopes_dir`).
- `[sorafs.storage]`: конфигурация встроенного SoraFS‑worker. Ключевые поля: `enabled`, `data_dir` (корень хранения), `max_capacity_bytes`, `max_parallel_fetches`, `max_pins`, `por_sample_interval_secs`, опциональный `alias`, и (если нужно сохранять settlements на диск) `governance_dag_dir`, указывающий каталог файловой системы для материализации governance‑артефактов. Таблица `[sorafs.storage.adverts]` раскрывает advert‑метаданные (`stake_pointer`, `availability`, `max_latency_ms`, `topics`); runtime‑снапшоты отражают ту же структуру под `torii.sorafs_storage`.
- `[sorafs.storage.pin]`: аутентификация/лимиты для `/v1/sorafs/storage/pin`. `require_token=true` требует bearer‑токен (через `Authorization: Bearer ...` или `X-SoraFS-Pin-Token`), `tokens` задаёт allow‑list, `allow_cidrs` ограничивает клиентов по CIDR, `rate_limit` применяет token‑bucket с опциональным баном. Torii отклоняет pin‑попытки до обращения к хранилищу при нарушении guard, возвращая `401/403/429` с `Retry-After` на throttled‑ответах.
- `[sorafs.repair]`: параметры планировщика ремонта. Ключи: `enabled` (по умолчанию `false`), опциональный `state_dir` (по умолчанию `<sorafs.storage.data_dir>/repair` если не задан), `claim_ttl_secs` (по умолчанию `900`), `heartbeat_interval_secs` (по умолчанию `60`), `max_attempts` (по умолчанию `3`), `worker_concurrency` (по умолчанию `4`), `backoff_initial_secs` (по умолчанию `5`), `backoff_max_secs` (по умолчанию `60`), и `default_slash_penalty_nano` (по умолчанию `1000000000`).
- `[sorafs.gc]`: параметры планировщика GC. Ключи: `enabled` (по умолчанию `false`), опциональный `state_dir` (по умолчанию `<sorafs.storage.data_dir>/gc` если не задан), `interval_secs` (по умолчанию `900`), `max_deletions_per_run` (по умолчанию `500`), `retention_grace_secs` (по умолчанию `86400`), и `pre_admission_sweep` (по умолчанию `true`).

  **SM‑роллаут чеклист.** Лейеринг `iroha_config` (`defaults` → `user` → `actual`) позволяет поэтапные rollout:
  1. **Проверка сборки.** Соберите бинарь с `--features sm` (и `sm-ffi-openssl` только при необходимости preview‑backend). Без feature‑флага узел отвергает `sm2` в `allowed_signing`.
  2. **Распространение конфигурации.**
     - Обновите `crypto.allowed_signing`, добавив `sm2` (список должен быть лексикографически отсортирован), и задайте `crypto.default_hash = "sm3-256"`.
     - Переопределите `crypto.sm2_distid_default` отличительным идентификатором, требуемым политикой (например, значением ISO/TC 68 для вашей организации).
  3. **Согласование genesis.** Перегенерируйте манифесты genesis через `kagami genesis generate --consensus-mode <mode>` (используйте `--allowed-signing sm2 --default-hash sm3-256` при необходимости) так, чтобы `parameters`‑блоки и injected‑метаданные отражали snapshot конфигурации; зафиксируйте их в VCS вместе с обновлённым TOML.
  4. **Smoke‑тесты телеметрии и допуска.** Перед включением трафика запустите `cargo test -p iroha_crypto --features sm` и SM‑интеграционную матрицу, затем проверьте, что `/status` показывает `crypto.sm_helpers_available = true` и что Torii отклоняет SM2‑подписи при отсутствии `sm2` в `allowed_signing`.
  5. **План отката.** Документируйте возврат в режим Ed25519‑only (удалить `sm2` из `allowed_signing`, восстановить `default_hash = "blake2b-256"`) и подтвердите, что хосты детерминированно сбрасывают транзакции SM в полёте.
