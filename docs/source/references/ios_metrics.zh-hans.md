# Swift Dashboard Metrics (Parity & CI)

This note describes the JSON schema and tooling used by the Swift SDK dashboards.
It serves as the contract for the telemetry exporters that will feed the parity
and CI views referenced in the roadmap and `status.md`. Read this alongside
`docs/source/swift_xcframework_device_matrix.md` (lane matrix), the parity triage
runbook, and the CI operations guide to understand how `device_tag` metadata and SLA
thresholds propagate through Buildkite, dashboards, and on-call workflows.

## Feeds

- `dashboards/data/mobile_parity.sample.json`
- `dashboards/data/mobile_ci.sample.json`
- `dashboards/data/swift_schema.sample.json`
- `dashboards/data/swift_salt_status.sample.json`
- `dashboards/data/mobile_pipeline_metadata.sample.json`

Validate the shared pipeline metadata feed with
`python3 scripts/check_swift_pipeline_metadata.py dashboards/data/mobile_pipeline_metadata.sample.json`
before publishing new snapshots.

The sample files live in the repository to document the expected shape. Real
feeds should be generated by telemetry exporters and dropped in the same
directory (the repository ignores non-`.sample.json` files).

## `mobile_parity` schema

| Field | Type | Notes |
|-------|------|-------|
| `generated_at` | string (ISO 8601) | Timestamp when the snapshot was produced. |
| `fixtures` | object | Outstanding Norito parity diffs and oldest age. |
| `pipeline` | object | `/v2/pipeline` integration status (last run, failures). |
| `pipeline.metadata` | object (optional) | Shared job/test timing data (`job_name`, `duration_seconds`, `tests[]`) emitted via `MOBILE_PARITY_PIPELINE_METADATA`; `metadata_source` records the path/URL for audit trails. |
| `regen_sla` | object | Fixture regeneration SLA streak. |
| `alerts` | array | Optional free-form alert entries. |
| `acceleration` | object (optional) | Metadata for Metal/NEON/StrongBox parity & perf (`enabled`, `parity`, `perf_delta_pct`). |
| `telemetry` | object (optional) | Redaction state summary (`salt_epoch`, rotation age, `overrides_open`, `device_profile_alignment`, `schema_version`, policy gate data), plus `notes`. |

## `mobile_ci` schema

| Field | Type | Notes |
|-------|------|-------|
| `generated_at` | string (ISO 8601) | Timestamp when the snapshot was produced. |
| `buildkite` | object | Lane success rates, queue depth, runtimes. |
| `devices` | object | Pass/fail counts for emulator vs StrongBox pools. |
| `alert_state` | object | Consecutive failure counters and open incidents. |
| `acceleration_bench` | object (optional) | Benchmark metrics comparing Metal vs CPU or NEON throughput. |

### `buildkite.lanes[]`

| Field | Type | Notes |
|-------|------|-------|
| `name` | string | Lane identifier (`ci/xcode-swift-parity`, `ci/xcframework-smoke:iphone-sim`, etc.). |
| `success_rate_14_runs` | number | Fraction (0–1) of green runs in the last 14 executions. |
| `last_failure` | string (ISO 8601) or `null` | Timestamp of the most recent failure. |
| `flake_count` | integer | Count of known flakes impacting the lane (manual input). |
| `mttr_hours` | number | Mean time to recovery in hours. |
| `device_tag` | string (optional) | Short tag (`iphone-sim`, `strongbox`, `mac-fallback`, …) emitted by the harness for dashboards. |

## `connect.error` event schema

Swift's `ConnectError` wrapper (see `docs/source/connect_error_taxonomy.md`) exposes
`telemetryAttributes(fatal:httpStatus:)`, which exporters forward to OTEL as
structured events. Each record should include:

| Attribute | Type | Notes |
|-----------|------|-------|
| `category` | string | One of `transport`, `codec`, `authorization`, `timeout`, `queueOverflow`, `internal`. |
| `code` | string | Stable identifier (`client.closed`, `network.timeout`, etc.). |
| `fatal` | string (`"true"`/`"false"`) | Whether the error terminated the session. |
| `http_status` | string (optional) | HTTP/WebSocket status associated with the error. |
| `underlying` | string (optional) | Debug-only textual form of the wrapped error. |
| `sid` | string (optional) | Session identifier when safe to log (hashed GUID). |
| `queue_depth` | integer (optional) | FIFO depth when the error fired (queue overflow drill-down). |

Emit these events at most once per failure path (e.g., when surfacing the
error to application callbacks) to avoid noisy dashboards. Downstream Grafana
panels join the event feed with queue depth and reconnection metrics to
highlight trend regressions.

## Tooling

- Render both dashboards: `make swift-dashboards` (invokes schema validator + Swift renderers)
  and enforces SLA thresholds (14-day Norito diff limit, 48h regen SLA, `ci/xcode-swift-parity`
  ≥95% success rate). Override thresholds with `SWIFT_PARITY_MAX_OLDEST_HOURS`,
  `SWIFT_PARITY_MAX_REGEN_HOURS`, `SWIFT_CI_PARITY_MIN_SUCCESS`, and
  `SWIFT_CI_MAX_CONSEC_FAIL` environment variables if needed.
  Hardware-acceleration gates confirm that Metal/NEON paths stay enabled and pass parity,
  and that the CI feed proves deterministic speedups: configure these checks with
  `SWIFT_PARITY_ACCEL_REQUIRE` (`backend=value`, space-separated), `SWIFT_PARITY_ACCEL_REQUIRE_ENABLED`
  (backend list), `SWIFT_CI_METAL_MIN_SPEEDUP` (ratio of CPU/Metal time), and
  `SWIFT_CI_NEON_MIN_THROUGHPUT` (MB/s floor for the NEON CRC64 benchmark).
- The same target honours optional `SWIFT_PARITY_FEED` / `SWIFT_CI_FEED` environment variables,
  letting CI jobs point at live telemetry feeds (instead of the checked-in samples) while
  reusing the schema validation + rendering pipeline.
- CI entry point: `ci/check_swift_dashboards.sh`.
- Manual render: `scripts/render_swift_dashboards.sh [/path/to/parity.json [/path/to/ci.json]]`.
- Schema validator: `scripts/check_swift_dashboard_data.py <files…>`.
- Telemetry enrichment: `scripts/swift_enrich_parity_feed.py --input parity.json --salt-epoch …`
  injects salt/override/profile metadata (fed via env vars in `ci/swift_status_export.sh`) so the parity
  feed carries the redaction block described above.
- Schema diff policy status: telemetry blocks may also include `schema_policy_violation_count`
  and `schema_policy_violations` (strings surfaced directly from `telemetry-schema-diff`) so dashboards
  can flag unreviewed redaction differences alongside the allowlist checks.
- Telemetry collector: `scripts/swift_collect_redaction_status.py --salt-config dashboards/data/swift_salt_status.sample.json --overrides-store artifacts/swift_telemetry_overrides.json`
  produces the telemetry JSON block automatically from salt/override sources. CI can point the `SWIFT_TELEMETRY_*`
  environment variables at live files and let the collector feed the enrichment step.
  maintains the local override ledger (`artifacts/swift_telemetry_overrides.json`)
  so support engineers can track masked actor roles, reasons, and expirations that surface in the telemetry block.
- Connect telemetry readiness: exporters should set `connect_metrics_present` in the telemetry block to reflect whether
  Connect OTLP counters (queue depth/resume attempts) were reported; the digest treats `false`/missing as “dark” so
  SRE can spot gaps quickly.
- Status export: `python3 scripts/swift_status_export.py [--parity <file> | --parity-url <url>] [--ci <file> | --ci-url <url>]`
  emits Markdown (`--format markdown`) or JSON for weekly digests.
- Feed provisioning details live in `docs/source/telemetry/swift_status_feeds.md`.
- JSON Schema: `docs/source/references/ios_metrics.schema.json` (validated via `python3 -m jsonschema …` in `make swift-dashboards`).
- Device coverage reference: `docs/source/swift_xcframework_device_matrix.md` defines the Buildkite lane matrix feeding the `devices` section.

### Prometheus export (`swift_parity_success_total`)

- `swift_status_export.py` can emit Prometheus textfile metrics by supplying
  `--metrics-path /path/to/swift_status.prom`. The exporter writes `swift_parity_success_total`
  and `swift_parity_failure_total` counters backed by a JSON state file (defaults to
  `artifacts/swift_status_metrics_state.json`, override with `--metrics-state …`).
- `swift_parity_success_total` increments whenever the latest parity snapshot has
  zero outstanding diffs, no regen breach, and no failing `/v2/pipeline` tests.
- The textfile also exposes gauges for the current status flag (`swift_parity_status`),
  outstanding diff count, oldest diff age, and hours since the last successful regen so
  textfile collectors and on-call dashboards can alert on stale fixtures.
- When the telemetry block is present the exporter also emits:
  - `swift_telemetry_overrides_open` – gauge for the number of active redaction overrides in the snapshot.
  - `swift_telemetry_salt_rotation_age_hours` – gauge recording the salt rotation age in hours.
  - `swift_telemetry_snapshot_present` – gauge that flips to 0 when the telemetry block is missing entirely.
  - `swift_telemetry_connect_metrics_present` – gauge indicating whether Connect telemetry counters were reported
    (`0` highlights “dark” exporters and should page the Connect telemetry owners).

## Exporter guidelines

1. Always emit valid UTF‑8 JSON matching the schema above.
2. Omit optional blocks (`acceleration`, `acceleration_bench`) when data is unavailable.
3. Keep timestamps in UTC (ISO 8601 with `Z` suffix).
4. Include benchmark numbers as raw doubles (e.g., milliseconds or MB/s) so the
   Swift renderers can derive human-readable strings.
5. Upload feeds to a secure location (S3, build artifact, etc.) and sync them to
   `dashboards/data/` during reporting jobs. Remember that the repository
   ignores non-sample files, so exporters must manage retention.
6. When running under CI, emit Buildkite metadata `ci/xcframework-smoke:<lane>:device_tag`
   alongside the JSON so downstream tooling can correlate telemetry with the physical
   device lanes without parsing destination strings.
