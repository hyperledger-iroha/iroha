# Реестр кривых учетных записей

Адреса учетных записей кодируют свои контроллеры как помеченный payload,
который начинается с 8‑битного идентификатора кривой. Валидаторы, SDK и
инструменты опираются на общий реестр, чтобы идентификаторы кривых оставались
стабильными между релизами и обеспечивали детерминированное декодирование между
реализациями.

Таблица ниже — нормативная ссылка для каждого назначенного `curve_id`.
Машиночитаемая копия публикуется рядом с этим документом в
[`address_curve_registry.json`](address_curve_registry.json); автоматизированные
инструменты ДОЛЖНЫ использовать JSON‑версию и фиксировать её поле `version` при
генерации фикстур.

## Зарегистрированные кривые

| ID (`curve_id`) | Алгоритм | Гейт функциональности | Статус | Кодирование публичного ключа | Примечания |
|-----------------|----------|-----------------------|--------|------------------------------|------------|
| `0x01` (1) | `ed25519` | — | Продакшн | Сжатый ключ Ed25519 (32 байта) | Каноническая кривая для V1. Все сборки SDK ДОЛЖНЫ поддерживать этот идентификатор. |
| `0x02` (2) | `ml-dsa` | — | Продакшн (конфигурационно ограничен) | Публичный ключ Dilithium3 (1952 байта) | Доступен во всех сборках. Включайте в `crypto.allowed_signing` + `crypto.curves.allowed_curve_ids` перед выпуском payload контроллеров. |
| `0x03` (3) | `bls_normal` | `bls` | Продакшн (фича‑гейт) | Сжатый публичный ключ G1 (48 байт) | Требуется для валидаторов консенсуса. Admission допускает BLS‑контроллеры даже если `allowed_signing`/`allowed_curve_ids` их не перечисляют. |
| `0x04` (4) | `secp256k1` | — | Продакшн | SEC1‑сжатый ключ (33 байта) | Детерминированный ECDSA поверх SHA‑256; подписи используют канонический формат `r∥s` на 64 байта. |
| `0x05` (5) | `bls_small` | `bls` | Продакшн (фича‑гейт) | Сжатый публичный ключ G2 (96 байт) | Профиль BLS с компактной подписью (подписи меньше, ключи больше). |
| `0x0A` (10) | `gost3410-2012-256-paramset-a` | `gost` | Зарезервировано | Точка TC26 param set A (64 байта, little‑endian) | Активируется с фичей `gost` после одобрения управления. |
| `0x0B` (11) | `gost3410-2012-256-paramset-b` | `gost` | Зарезервировано | Точка TC26 param set B (64 байта, little‑endian) | Соответствует набору параметров TC26 B; заблокировано фичей `gost`. |
| `0x0C` (12) | `gost3410-2012-256-paramset-c` | `gost` | Зарезервировано | Точка TC26 param set C (64 байта, little‑endian) | Зарезервировано для будущего одобрения управления. |
| `0x0D` (13) | `gost3410-2012-512-paramset-a` | `gost` | Зарезервировано | Точка TC26 param set A (128 байт, little‑endian) | Зарезервировано до появления спроса на GOST 512‑битные кривые. |
| `0x0E` (14) | `gost3410-2012-512-paramset-b` | `gost` | Зарезервировано | Точка TC26 param set B (128 байт, little‑endian) | Зарезервировано до появления спроса на GOST 512‑битные кривые. |
| `0x0F` (15) | `sm2` | `sm` | Зарезервировано | Длина DistID (u16 BE) + байты DistID + несжатый ключ SM2 SEC1 (65 байт) | Станет доступен, когда фича `sm` выйдет из preview. |

### Рекомендации по использованию

- **Fail closed:** Кодеры ДОЛЖНЫ отклонять неподдерживаемые алгоритмы с
  `ERR_UNSUPPORTED_ALGORITHM`. Декодеры ДОЛЖНЫ выдавать `ERR_UNKNOWN_CURVE` для
  любого идентификатора, отсутствующего в реестре.
- **Фича‑гейт:** BLS/GOST/SM2 остаются за указанными фичами сборки. Операторы
  должны включить соответствующие записи в `iroha_config.crypto.allowed_signing`
  и активировать фичи сборки перед выпуском адресов с этими кривыми.
- **Исключения admission:** BLS‑контроллеры разрешены для валидаторов консенсуса
  даже если `allowed_signing`/`allowed_curve_ids` их не перечисляют.
- **Паритет config + manifest:** Используйте `iroha_config.crypto.allowed_curve_ids`
  (и соответствующий `ManifestCrypto.allowed_curve_ids`) для публикации
  поддерживаемых идентификаторов; admission теперь применяет этот список вместе
  с `allowed_signing`.
- **Детерминированное кодирование:** Публичные ключи кодируются строго так, как
  их возвращает реализация подписи (сжатые байты Ed25519, публичные ключи ML‑DSA,
  сжатые точки BLS и т. д.). SDK должны выдавать ошибки валидации до отправки
  некорректных payload.
- **Паритет манифестов:** Genesis‑манифесты и манифесты контроллеров ДОЛЖНЫ
  использовать одни и те же идентификаторы, чтобы admission мог отклонить
  контроллеры сверх возможностей кластера.

## Анонс битовой маски возможностей

`GET /v1/node/capabilities` теперь публикует список `allowed_curve_ids` и
упакованный массив `allowed_curve_bitmap` в `crypto.curves`. Bitmap — это
little‑endian массив 64‑битных lane (до четырёх значений, покрывающих диапазон
`u8` 0–255). Установленный бит `i` означает, что идентификатор кривой `i`
разрешён политикой admission кластера.

- Пример: `{ allowed_curve_ids: [1, 15] }` ⇒ `allowed_curve_bitmap: [32770]`,
  потому что `(1 << 1) | (1 << 15) = 32770`.
- Кривые выше `63` устанавливают биты в последующих lane. Хвостовые нулевые
  lane опускаются для компактности, поэтому конфигурация, которая также
  включает `curve_id = 130`, выдаст `allowed_curve_bitmap = [32768, 0, 4]`
  (установлены биты 15 и 130).

Используйте bitmap для дашбордов и health‑check: одно тестирование бита даёт
ответ без сканирования всего массива, а инструменты, которым нужны упорядоченные
идентификаторы, могут продолжать использовать `allowed_curve_ids`. Публикация
обоих представлений удовлетворяет требование **ADDR-3** по детерминированным
битовым маскам возможностей для операторов и SDK.

## Контрольный список валидации

Каждый компонент, принимающий контроллеры (Torii, admission, кодеры SDK,
offline‑инструменты), должен применять одинаковые детерминированные проверки
перед принятием payload. Следующие шаги обязательны:

1. **Определить политику кластера:** Извлеките ведущий байт `curve_id` из payload
   учётной записи и отклоните контроллер, если идентификатор отсутствует в
   `iroha_config.crypto.allowed_curve_ids` (и зеркальном
   `ManifestCrypto.allowed_curve_ids`). BLS‑контроллеры — исключение: если они
   включены в сборке, admission разрешает их независимо от allowlist, чтобы
   ключи валидаторов консенсуса продолжали работать. Это предотвращает приём
   preview‑кривых без явного включения операторами.
2. **Проверить длину кодирования:** Сравните длину payload с каноническим
   размером алгоритма до попытки распаковки/декодирования ключа. Отклоняйте
   значения, не проходящие проверку, чтобы отсечь некорректные данные раньше.
3. **Алгоритм‑специфическое декодирование:** Используйте те же канонические
   декодеры, что и `iroha_crypto` (`ed25519_dalek`, `pqcrypto_mldsa`,
   `w3f_bls`/`blstrs`, `sm2`, хелперы TC26 и т. д.), чтобы все реализации
   разделяли одинаковое поведение проверки подгрупп/точек.
4. **Проверить размеры подписей:** Admission и SDK должны проверять длины
   подписи ниже и отклонять payload с усечённой или слишком длинной подписью до
   запуска верификатора.

| Алгоритм | `curve_id` | Байты публичного ключа | Байты подписи | Критические проверки |
|----------|------------|------------------------|--------------|----------------------|
| `ed25519` | `0x01` | 32 | 64 | Отклонять неканонические сжатые точки, очищать кофактор (без точек малых порядков) и проверять `s < L` при валидации подписей. |
| `ml-dsa` (Dilithium3) | `0x02` | 1952 | 3309 | Отклонять payload длиной не ровно 1952 байта до декодирования; парсить ключ Dilithium3 и проверять подписи pqcrypto‑dilithium с каноническими длинами. |
| `bls_normal` | `0x03` | 48 | 96 | Принимать только канонические сжатые публичные ключи G1 и сжатые подписи G2; отклонять точки идентичности и неканонические кодировки. |
| `secp256k1` | `0x04` | 33 | 64 | Принимать только SEC1‑сжатые точки; распаковать и отвергнуть неканонические/некорректные точки, проверять подписи в каноническом формате `r∥s` (нормализация low‑`s` выполняется подписантом). |
| `bls_small` | `0x05` | 96 | 48 | Принимать только канонические сжатые ключи G2 и сжатые подписи G1; отклонять точки идентичности и неканонические кодировки. |
| `gost3410-2012-256-paramset-a` | `0x0A` | 64 | 64 | Интерпретировать payload как координаты `(x||y)` little‑endian, проверять, что каждая координата `< p`, отвергать точку идентичности и требовать канонические 32‑байтовые limb `r`/`s` при верификации. |
| `gost3410-2012-256-paramset-b` | `0x0B` | 64 | 64 | Та же проверка, что для набора A, но с доменными параметрами TC26 B. |
| `gost3410-2012-256-paramset-c` | `0x0C` | 64 | 64 | Та же проверка, что для набора A, но с доменными параметрами TC26 C. |
| `gost3410-2012-512-paramset-a` | `0x0D` | 128 | 128 | Интерпретировать `(x||y)` как 64‑байтовые limbs, проверять `< p`, отвергать точку идентичности и требовать 64‑байтовые `r`/`s` для подписей. |
| `gost3410-2012-512-paramset-b` | `0x0E` | 128 | 128 | Та же проверка, что для набора A, но с доменными параметрами TC26 B на 512 бит. |
| `sm2` | `0x0F` | 2 + distid + 65 | 64 | Декодировать длину distid (u16 BE), проверить байты DistID, распарсить несжатую точку SEC1, применить правила подгруппы GM/T 0003, применить настроенный DistID и требовать канонические limb `(r, s)` по SM2. |

Каждая строка соответствует объекту `validation` внутри
[`address_curve_registry.json`](address_curve_registry.json). Инструменты,
потребляющие JSON‑экспорт, могут полагаться на поля `public_key_bytes`,
`signature_bytes` и `checks`, чтобы автоматизировать те же шаги валидации;
переменные по длине кодировки (например SM2) задают `public_key_bytes` как null
и описывают правило длины в `checks`.

## Запрос нового идентификатора кривой

1. Подготовьте спецификацию алгоритма (кодирование, валидация, обработка ошибок)
   и получите одобрение управления для rollout.
2. Отправьте pull request, обновив этот документ и
   `address_curve_registry.json`. Новые идентификаторы должны быть уникальны и
   попадать в диапазон `0x01..=0xFE`.
3. Обновите SDK, фикстуры Norito и операторскую документацию новым
   идентификатором до развертывания в production‑сетях.
4. Согласуйте с командами безопасности и наблюдаемости, чтобы телеметрия,
   runbook и политики admission отражали новый алгоритм.
