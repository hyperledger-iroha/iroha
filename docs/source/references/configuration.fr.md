# Vue d’ensemble de la configuration

Cette page résume les sections de niveau supérieur du fichier de configuration Iroha (TOML) et ce que chacune contrôle. Pour un modèle concret, voir `docs/source/references/peer.template.toml`.

Les valeurs par défaut d’abord : les valeurs de configuration sont conçues pour des déploiements Iroha typiques. Dans la plupart des cas, vous pouvez exécuter avec les valeurs par défaut inchangées. N’ajustez les paramètres que si vous avez un besoin opérationnel spécifique (p. ex. ports personnalisés, limites de débit strictes, ou contraintes réseau connues).

- Les sections spécifiques à Sora (`torii.sorafs`, `sorafs.*`, multi‑lane `nexus.*`) exigent de démarrer le daemon avec `irohad --sora …` (ou d’exporter `IROHA_SORA_PROFILE=1`). Sans le drapeau, le chargeur rejette les configurations qui activent ces fonctionnalités.
- Lorsque `nexus.enabled = false` (mode Iroha 2), les configs restent mono‑lane : `/status` et `/v1/sumeragi/status` masquent les sections lane/dataspace, `/metrics` saute complètement les labels lane/dataspace, et les tails de statut à portée de lane renvoient `StatusSegmentNotFound`, donc les SDK doivent brancher sur le drapeau avant d’envoyer des paramètres de lane. Le parsing de config rejette les overrides lane/dataspace/routing dans ce mode — supprimez les sections `nexus.*` sauf si vous activez Nexus/Sora avec `nexus.enabled=true` ou `--sora`.
- `[common]` : Identité de chaîne (`chain`), paire de clés du nœud, pairs de confiance, l’étiquette implicite de domaine de compte (`default_account_domain_label`), et le discriminant de chaîne IH58 (`chain_discriminant`). Remplacez‑les lorsque votre domaine genesis ou préfixe réseau diffère des valeurs par défaut intégrées afin que les adresses IH58/compressées restent canoniques.
- Admission des validateurs : `trusted_peers` doit lister les clés de validateurs BLS‑Normal, et les PoP se trouvent sous `[[trusted_peers_pop]] { public_key, pop_hex }`. Le parsing de config rejette désormais le mappage hérité `trusted_peers_bls` et toute clé de validateur non‑BLS. L’identité de transport utilise toujours `public_key`/`private_key` pour P2P/Torii, tandis que les signatures de consensus utilisent uniquement la clé BLS + PoP.
- Séparation d’identité : la paire `public_key`/`private_key` sécurise le transport (P2P, Torii) et peut rester Ed25519/SECP/etc. indépendamment de `allowed_signing`. Le consensus utilise la clé BLS‑Normal de `trusted_peers` avec sa PoP. L’admission des transactions est le seul endroit où `crypto.allowed_signing`/`allowed_curve_ids` sont appliqués ; si vous voulez soumettre des transactions signées BLS, ajoutez explicitement `bls_normal` à `allowed_signing` et l’identifiant de courbe correspondant à `crypto.curves.allowed_curve_ids`. Le consensus continue de fonctionner avec des validateurs BLS même si `allowed_signing` reste aux valeurs par défaut Ed25519/secp.
- `[genesis]` : Emplacement du fichier genesis signé et clé publique utilisée pour le vérifier. Le manifeste référencé contient des familles de paramètres telles que `transaction.max_signatures`, `transaction.max_instructions`, `transaction.ivm_bytecode_size`, `transaction.max_tx_bytes` et `transaction.max_decompressed_bytes` ; mettez à jour le manifeste lorsque vous resserrez les limites d’admission. Les nœuds démarrant avec un stockage vide et sans `genesis.file` local peuvent récupérer la genesis depuis des pairs de confiance en fixant éventuellement `expected_hash`, en fournissant un `bootstrap_allowlist` (par défaut `trusted_peers`), et en bornant le transfert via `bootstrap_max_bytes` et `bootstrap_response_throttle_ms`. Les tentatives de bootstrap respectent `bootstrap_request_timeout_ms`, `bootstrap_retry_interval_ms`, `bootstrap_max_attempts` et `bootstrap_enabled` ; les répondants appliquent toujours les correspondances chain‑id/clé publique/hash et rejettent les charges trop volumineuses ou les pairs hors allowlist.
- `[network]` : Adresses P2P, périodes/tailles de gossip, rafraîchissement DNS, capacités de files bornées, et options de transport (TLS/QUIC lorsqu’activées).
- `[ivm]` : Paramètres du runtime VM. `ivm.memory_budget_profile` (défaut : `compute.default_resource_profile`) sélectionne le profil de ressources de calcul qui fournit `max_stack_bytes` comme plafond du budget de pile invité.
- `[ivm.banner]` (défauts : `show = true`, `beep = true`) : Contrôle la bannière de démarrage Norito/IVM et le jingle optionnel. Le drapeau `beep` n’est respecté que lorsque le binaire est construit avec la fonctionnalité `beep`. Cela remplace le shim d’environnement `IROHA_BEEP` hérité — désactivez la bannière ou le bip via la config plutôt que par des overrides d’environnement.
- `[concurrency]` : Dimensionnement des pools de threads ainsi que budgets de pile pour les workers scheduler/prover et l’invité IVM. `gas_to_stack_multiplier` (défaut `4`) borne l’espace de pile invité comme `min(guest_stack_bytes, gas_limit * multiplier, stack_budget)` avec un plancher de 64 KiB et un plafond de 1 GiB ; `stack_budget` provient du profil `ivm.memory_budget_profile`. Les requêtes hors de cette fenêtre sont clampées, journalisées et exportées via `/status.stack` et les métriques `iroha_ivm_stack_*` (y compris les compteurs pool‑fallback/budget‑hit). Baissez le multiplicateur pour plafonner la récursivité sur les routes à faible gas ou augmentez‑le (dans la limite de pile invité) pour admettre des frames Kotodama plus profondes.
- `network.peer_gossip_period_ms` (défaut : `250`ms) gouverne également la cadence de rafraîchissement de la topologie. L’override d’environnement `IROHA_P2P_TOPOLOGY_UPDATE_MS` est désormais ignoré ; ajustez le champ de configuration à la place. Les périodes de gossip (`block_gossip_period_ms`, `transaction_gossip_period_ms`, `peer_gossip_period_ms`) et `idle_timeout_ms` sont clampées à un minimum de 100 ms pour éviter des boucles trop serrées.
- `network.trust_gossip` (défaut : `true`) annonce le support des trames TrustGossip. Le régler à `false` négocie la capacité hors du handshake, en supprimant les messages TrustGossip tout en conservant le gossip d’adresses de pairs. Les abandons sont tagués via `p2p_trust_gossip_skipped_total{direction,reason}` pour que les opérateurs repèrent les opt‑outs ou les pairs qui refusent la capacité.
- Les paramètres de scoring de confiance sous `[network]` (`trust_decay_half_life_ms`, `trust_penalty_bad_gossip`, `trust_penalty_unknown_peer`, `trust_min_score`) s’appliquent aux overlays permissionnés : les trames de confiance qui référencent des pairs hors de la topologie active subissent la pénalité unknown‑peer, incrémentent `p2p_trust_penalties_total{reason="unknown_peer"}`, et les expéditeurs en dessous de `trust_min_score` sont ignorés jusqu’à ce que la décroissance se rétablisse. Les overlays publics/NPoS ignorent la pénalité unknown‑peer afin que le trust gossip reste ouvert.
- Réglage du gossip des transactions : `network.transaction_gossip_drop_unknown_dataspace` abandonne les lots pour les dataspaces absents du catalogue de lanes ; `network.transaction_gossip_restricted_target_cap` plafonne le fanout restreint (par défaut la topologie complète de commit) ; `network.transaction_gossip_public_target_cap` plafonne le fanout de la lane publique (par défaut 16, mettre `null` pour broadcast) ; `network.transaction_gossip_public_target_reshuffle_ms` et `network.transaction_gossip_restricted_target_reshuffle_ms` contrôlent la fréquence de remélange des cibles par plan (défaut `transaction_gossip_period_ms`, clampé à ≥100 ms) ; `network.transaction_gossip_restricted_fallback` (`drop` | `public_overlay`) définit si le gossip restreint bascule vers les pairs en ligne lorsque la topologie de commit est vide ; et `network.transaction_gossip_restricted_public_payload` (`refuse` | `forward`, défaut `refuse`) refuse pour l’instant de divulguer des payloads restreints lorsque seul l’overlay public est disponible (le forward est bloqué jusqu’à l’arrivée du transport chiffré). La télémétrie expose la politique drop/forward via des jauges et des entrées `/status.tx_gossip` (caps, targets/peers, surface de fallback, raison/résultat) afin que les opérateurs puissent alerter sur des lots bloqués ou reroutés.
- `network.soranet_handshake.pow` : configuration du challenge pour l’admission SoraNet. Champs : `required`, `difficulty`, `max_future_skew_secs`, `min_ticket_ttl_secs`, `ticket_ttl_secs`, `revocation_store_capacity`, `revocation_max_ttl_secs` et `revocation_store_path`. Le revocation store contrôle combien de tickets signés sont conservés sur disque et leur durée de validité pour la détection de replay.
- `network.soranet_handshake.pow.puzzle` : bascules du puzzle Argon2. Mettre `enabled = true` pour exiger des tickets mémoire‑dure et régler `memory_kib` (>=4096), `time_cost`, et `lanes` (1–16). Lorsqu’il est activé, le relay honore toujours `difficulty`, `max_future_skew_secs`, et `min_ticket_ttl_secs` ; désactiver le bloc revient au PoW hashcash classique.
- `network.soranet_handshake.kem_suite` : override textuel optionnel (`"mlkem512"`, `"mlkem768"`, ou `"mlkem1024"`; les alias `kyber*` sont acceptés) pour le profil ML‑KEM utilisé durant le handshake SoraNet. Lorsqu’il est défini, il supplante `kem_id` et valide les longueurs de clé Kyber configurées pour la suite sélectionnée.
- `[sumeragi]` : rôle de consensus (validator/observer) et paramètres de collecteur (K, redundant‑send r), plus options de slack temporel. Le gating DA utilise `da_enabled` pour différer le commit jusqu’à observation d’une preuve de disponibilité (un `availability evidence` ou un quorum RBC `READY`) ; le commit n’attend pas les livraisons RBC locales. La backpressure par file est configurée avec `msg_channel_cap_votes`, `msg_channel_cap_block_payload`, `msg_channel_cap_rbc_chunks`, `msg_channel_cap_blocks`, et `control_msg_channel_cap` ; l’ancien `msg_channel_cap` n’agit comme fallback que lorsque les caps par file sont omis. `mode_flip_enabled` (défaut `true`) sert d’interrupteur de sécurité pour les bascules permissioned↔NPoS en direct : lorsqu’il est `false`, le nœud annonce le mode préparé via status/telemetry mais refuse de basculer tant que le drapeau n’est pas restauré. Les échecs de persistance font un backoff selon `kura_store_retry_interval_ms`/`kura_store_retry_max_attempts` afin que les blocs restent en attente jusqu’à ce que Kura accepte ou que le budget de retry soit épuisé ; `commit_inflight_timeout_ms` annule les jobs de commit bloqués pour éviter que les propositions ne se figent. La récupération des blocs manquants préfère les signataires commit certificate pour `missing_block_signer_fallback_attempts` tentatives (défaut 1 ; mettre 0 pour éviter la préférence) avant de revenir à la topologie complète de commit. La dérive du hachage d'appartenance est signalée via `membership_mismatch_alert_threshold` (désaccords consécutifs avant alerte) et `membership_mismatch_fail_closed` (rejeter les messages de consensus des pairs toujours en désaccord). L’intervalle de proposition du pacemaker est dérivé de `npos.block_time_ms`, `npos.timeouts.propose`, `pacemaker_rtt_floor_multiplier`, et plafonné par `pacemaker_max_backoff_ms` ; augmentez le plancher RTT ou le block time pour ralentir les propositions sur des liens à forte latence. Le quorum RBC READY/DELIVER suit toujours la topologie de commit (`Topology::min_votes_for_commit()`).
- `[torii]` : Paramètres du serveur API public tels que l’adresse d’écoute, les plafonds de taille de requêtes, et les paramètres du store de requêtes.
- `torii.data_dir` (défaut : `./storage/torii`) : répertoire de base pour la persistance Torii (attachments, registre des webhooks, caches DA). Remplace le shim d’environnement hérité `IROHA_TORII_DATA_DIR` ; les tests peuvent temporairement override via `data_dir::OverrideGuard`.
- `torii.events_buffer_capacity` (défaut : `10000`) : taille du canal broadcast utilisé pour `/v1/events/sse` et la mise en file des webhooks. Réduisez‑la pour borner mémoire/backpressure lorsque les abonnés sont en retard.
- `torii.app_api.*` : valeurs de pagination/backpressure par défaut pour les endpoints JSON de convenance. `default_list_limit` initialise `limit` lorsqu’il est omis, `max_list_limit` et `max_fetch_size` bornent les tailles de page/fetch, et `rate_limit_cost_per_row` contrôle le coût par ligne demandée sous le limiteur de taux de l’app API.
- `torii.webhook.*` : réglage de livraison/backoff du worker webhook. `queue_capacity` borne les livraisons en attente sur disque (défaut : `10000`), `max_attempts` borne les retries (défaut : `12`), `backoff_initial_ms`/`backoff_max_ms` définissent la fenêtre de retry exponentiel (défaut : `1000`/`60000`), et `{connect,write,read}_timeout_ms` configurent les timeouts HTTP des tentatives de livraison.
- `torii.push.*` (feature stub) : bascules du pont de push‑notifications et limites de taux. `enabled` active le pont ; `rate_per_minute`/`burst` appliquent un token bucket aux requêtes sortantes, `max_topics_per_device` borne les abonnements par token, et `connect_timeout_ms`/`request_timeout_ms` bornent les tentatives HTTP vers FCM/APNS. Fournissez `fcm_api_key` ou l’APNS `apns_endpoint` + `apns_auth_token` pour activer le provider correspondant.
- `torii.da_ingest.*` : cache de replay et configuration du spool de manifestes pour le pipeline DA.
  - `telemetry_cluster_label` (optionnel) : définit le label `cluster` émis sur les métriques Taikai ingest/viewer afin que les dashboards distinguent les environnements (p. ex., `region-a` vs `staging`).
    Définissez `governance_metadata_key_hex` sur une clé ChaCha20‑Poly1305 de 32 octets lorsque Torii doit sceller des métadonnées governance‑only ; associez‑la à `governance_metadata_key_label` (un identifiant court tel que `"primary"`) pour que les entrées chiffrées indiquent quelle clé a produit le ciphertext. Torii chiffre automatiquement les métadonnées governance en clair et refuse les manifestes dont les payloads pré‑chiffrés portent des labels manquants ou non concordants. La table imbriquée `replication_policy` définit la `RetentionPolicy` appliquée par classe de blob (voir `docs/source/da/replication_policy.md`). Remplacez `default_retention` pour changer les fenêtres hot/cold de base et ajoutez des entrées sous `replication_policy.overrides` (p. ex., `class = "taikai_segment"`) pour durcir des classes spécifiques ; Torii réécrit les manifestes fournis par l’appelant pour correspondre à ces valeurs et rejette les enveloppes non conformes.
  - `debug_match_filters` (défaut : `false`) active des traces déterministes de debug de filtres auparavant derrière le toggle d’environnement `TORII_DEBUG_MATCH`. Laissez‑le désactivé en production.
- `[torii.iso_bridge.reference_data]` : chemins vers les datasets ISO de référence (`isin_crosswalk_path`, `bic_lei_path`, `mic_directory_path`), la cadence d’ingestion (`refresh_interval_secs`), et un `cache_dir` optionnel pour persister des copies plus des métadonnées de provenance (version/source/checksum) de chaque snapshot chargé.
- `torii.strict_addresses` (défaut : `true`) : lorsqu’il est activé (par défaut), Torii rejette les adresses `alias@domain` héritées / littéraux de clé publique bruts sur tous les endpoints app‑facing et renvoie `ERR_STRICT_ADDRESS_REQUIRED`. Ne mettez ceci à `false` que sur des clusters dev/test en cas de diagnostic, puis repassez à `true` une fois l’adoption canonique confirmée par la télémétrie.
- `[torii.soranet_privacy_ingest]` : gardes pour `/v1/soranet/privacy/{event,share}`. `enabled` doit être `true` pour exposer les endpoints ; `require_token` + `tokens` imposent l’en‑tête `X-SoraNet-Privacy-Token` (ou `X-API-Token`), `allow_cidrs` restreint les émetteurs par CIDR (liste vide = deny par défaut), et `rate_per_sec`/`burst` appliquent un limiteur token‑bucket. Les rejets renvoient `401/403/429` avec des indices de retry et des compteurs télémétriques étiquetés afin que les opérateurs détectent les abus ; voir `docs/source/sorafs_authz_runbook.md` pour la checklist de déploiement.
- `[torii.sorafs]` : contrôles de cache de découverte SoraFS exposés côté Torii. Les valeurs sont copiées depuis les sections `[sorafs.*]` canoniques lors du parsing de config, donc configurez les sections racine pour l’intention opérateur ; la vue Torii existe pour les snapshots runtime/miroirs télémétrie.
- `[sorafs.discovery]` : toggle canonique de découverte et allow‑list de capacités. `discovery_enabled` est `false` par défaut (les profils Sora comme `tests/fixtures/base.toml` l’activent explicitement), `known_capabilities` est livré avec `["torii_gateway", "chunk_range_fetch", "vendor_reserved"]`, et la table optionnelle `[sorafs.discovery.admission]` pointe vers le répertoire des enveloppes de gouvernance (`envelopes_dir`).
- `[sorafs.storage]` : configuration du worker SoraFS embarqué. Champs clés : `enabled`, `data_dir` (racine de stockage), `max_capacity_bytes`, `max_parallel_fetches`, `max_pins`, `por_sample_interval_secs`, `alias` optionnel, et (si vous voulez écrire les settlements sur disque) `governance_dag_dir` pointant vers le répertoire où matérialiser les artefacts de gouvernance. La table `[sorafs.storage.adverts]` expose les métadonnées d’advert (`stake_pointer`, `availability`, `max_latency_ms`, `topics`) ; les snapshots runtime reflètent la même structure sous `torii.sorafs_storage`.
- `[sorafs.storage.pin]` : authentification/limites de débit pour `/v1/sorafs/storage/pin`. `require_token=true` impose un token bearer (via `Authorization: Bearer ...` ou `X-SoraFS-Pin-Token`), `tokens` fournit l’allow‑list, `allow_cidrs` restreint les clients par CIDR, et `rate_limit` applique un token‑bucket avec ban optionnel. Torii rejette les tentatives de pin avant de toucher le stockage lorsque ces gardes échouent, renvoyant `401/403/429` avec `Retry-After` sur les réponses throttled.
- `[sorafs.quota]` : fenêtres de quotas du plan de contrôle (déclaration de capacité, pin de stockage, télémétrie, télémétrie des deals, dispute, et soumission PoR). La vue Torii sous `torii.sorafs_quota` hérite ces valeurs telles quelles.
- `[sorafs.alias_cache]` : politique de cache alias‑proof (`positive_ttl`, `refresh_window`, `hard_expiry`, `negative_ttl`, `revocation_ttl`, `rotation_max_age`). Les défauts imposent 10 min / 2 min / 15 min / 60 s / 5 min / 6 h ; ne durcissez qu’en diminuant les valeurs. Le snapshot runtime reflète ces valeurs sous `torii.sorafs_alias_cache`.
- `sorafs.rollout_phase` : sélecteur de déploiement PQ haut niveau (`canary`, `ramp`, `default`). Les clients héritent du stade d’anonymat correspondant à moins que `sorafs.anonymity_policy` ne soit explicitement défini.
- `sorafs.anonymity_policy` : drapeau de déploiement PQ par étapes SoraNet (`anon-guard-pq`, `anon-majority-pq`, `anon-strict-pq`). Défaut : Stage A (`anon-guard-pq`) ; passez en Stage B/C lorsque la couverture PQ est suffisante. N’overridez que lorsqu’un composant doit diverger de la phase.
- `[nts]` : cadence d’échantillonnage et lissage du Network Time Service. `sample_interval_ms` est clampé à >=100 ms ; consultatif seulement, le consensus n’en dépend pas.
- `[kura]` : chemin de stockage des blocs et fenêtre mémoire ; politique d’initialisation pour snapshots/replay. `block_sync_roster_retention` borne le nombre d’instantanés de roster de commit certifiés conservés sur disque pour la récupération block‑sync (défaut 7200), et `roster_sidecar_retention` limite le nombre de sidecars de roster conservés dans `pipeline/roster_sidecars.*` après compaction (défaut 512).
- `[logger]` : niveau et format des logs.
- `[queue]` : limites et TTL de la file de transactions.
- `[nexus]` : catalogue de routage multi‑lane, registre de manifestes, et modules de gouvernance. Les entrées de lane acceptent `visibility`, `lane_type`, `governance`, `settlement`, `proof_scheme` (`merkle_sha256` par défaut ; les lanes `kzg_bls12_381` reçoivent des engagements KZG déterministes dérivés à l’ingest), et un `metadata = { key = "value" }` libre. Le bloc `registry` contrôle `manifest_directory`, `cache_directory`, et `poll_interval_ms` (défaut 60 s). Le bloc `governance` fournit un `default_module` et un mapping `modules`, où chaque module expose `module_type` plus des overrides `params` clé/valeur pour les hôtes downstream.
  - La clé de métadonnées de lane `da_manifest_policy` ajuste l’application des manifestes DA par lane : `strict` (défaut) bloque les manifestes manquants/illisibles et les mismatches de hash, tandis que `audit`/`audit_only`/`warn` autorisent des manifestes manquants avec des avertissements (les mismatches de hash restent fatals).
  - `nexus.staking.public_validator_mode` / `nexus.staking.restricted_validator_mode` sélectionnent le cycle de vie des validateurs par lane : `stake_elected` (défaut pour lanes publiques) garde la voie staking/NPoS activée, tandis que `admin_managed` (défaut pour lanes restreintes) bypass le staking et attend l’admission via les flux governance/admin. Les lanes stake‑elected exigent un peer enregistré avec une clé de consensus active présente dans la topologie de commit avant admission et s’activent aux frontières d’époque ; les lanes admin‑managed maintiennent le roster genesis afin que les déploiements permissionnés progressent sans staking.
  - `[nexus.axt]` : garde‑fous de timing/expiration. `slot_length_ms` doit être dans `1..=600_000` (défaut `1`) ; les hôtes dérivent `current_slot = block.creation_time_ms / slot_length_ms`. `max_clock_skew_ms` est par défaut `0`, plafonné à `60_000`, et ne doit pas dépasser `slot_length_ms` ; les checks d’expiration pour proofs/handles incluent cette marge. `proof_cache_ttl_slots` (défaut `1`, validé `1..=64`) borne combien de slots les proofs acceptées/rejetées restent dans le cache de l’hôte pour réutilisation/rejet replay. `replay_retention_slots` (défaut `128`, validé `1..=4_096`) contrôle la durée de conservation des journaux d’usage/replay des handles à travers pairs/redémarrages ; ne l’augmentez que si les handles restent valides longtemps et que vous avez besoin d’une protection replay couvrant ces fenêtres. Voir `crates/iroha_config/tests/fixtures/nexus_axt_full.toml` pour un exemple 1s/250ms/TTL=8.
- `[governance]` : clés de vérification, paramètres de conviction, seuils d’approbation, seuil de participation, et taille/éligibilité du conseil.
  - `debug_tie_break` / `debug_tx_eval` (défaut : `false`) activent des traces d’évaluation d’overlay forensique pour le debug. Elles remplacent les shims d’environnement hérités `IROHA_DEBUG_TIE_BREAK`/`IROHA_DEBUG_TX_EVAL` ; laissez‑les désactivées en production.
- `[norito]` : réglages de sérialisation Norito (tailles/niveaux de compression et toggles de layout adaptatif). Déterministe par conception.
- `[streaming]` : matériel de clés du plan de contrôle pour le streaming Norito (HPKE + signatures de contrôle).
  - `identity_public_key` / `identity_private_key` (optionnels) : chaînes multihash Ed25519 qui remplacent la paire de clés principale du nœud pour les signatures de streaming — utile lorsque la clé de validateur est GOST ou un autre algo non‑Ed25519.
  - `kyber_public_key` / `kyber_secret_key` (optionnels) : clés Kyber‑768 hexadécimales activant HPKE ; doivent être fournies ensemble.
  - `session_store_dir` (défaut : `./storage/streaming`) : répertoire des snapshots de session chiffrés.
  - `feature_bits` (défaut : `0b11`) : masque de capacités annoncé lors de la négociation QUIC. Le bit 0 active les feedback hints, le bit 1 signale le support du fournisseur privacy‑overlay, et les bits supérieurs sont réservés. Les viewers demandant des bits non supportés sont rejetés.
  - `soranet` (défaut : activé) : pont automatique des routes privacy de manifestes vers SoraNet quand les entrées de manifeste omettent des métadonnées explicites.
    - `enabled` (défaut : `true`) : active la logique d’auto‑provisionnement ; désactivez pour laisser les manifestes existants inchangés.
    - `exit_multiaddr` (défaut : `/dns/torii/udp/9443/quic`) : multi‑adresse de sortie relayée vers Torii une fois le circuit sorti de SoraNet.

- `[crypto]` : valeurs par défaut cryptographiques exposées via `iroha_config`.
  - `sm_intrinsics` (défaut : `auto`) : politique de dispatch des intrinsics SM (`auto`, `force-enable`, ou `force-disable`). Laisser `auto` en production ; utiliser `force-disable` pour valider les fallbacks scalaires ou `force-enable` pour des profils contrôlés.

Les builds compilés sans la fonctionnalité `sm` rejetteront `sm2` dans cette liste.     - `allowed_curve_ids` (défaut : dérivé de `allowed_signing`, typiquement `[1]` pour Ed25519) : identifiants de courbe (voir [`address_curve_registry.md`](../references/address_curve_registry.md)) autorisés dans les contrôleurs de comptes. Remplacez cette liste pour préparer des déploiements ML‑DSA/GOST/SM même si les binaires supportent déjà les algorithmes ; les manifestes genesis reflètent ce champ via `crypto.allowed_curve_ids`.
    - `sm2_distid_default` (défaut : `1234567812345678`) : identifiant distinctif SM2 de repli appliqué lorsque les signataires omettent leur propre `distid`.     - *Preview (feature `sm-ffi-openssl`)* : construire Iroha avec `--features "sm sm-ffi-openssl"` expose le provider OpenSSL/Tongsuo optionnel contrôlé par `crypto.enable_sm_openssl_preview`. Les opérateurs doivent livrer OpenSSL/Tongsuo 3.0+ avec les binaires et laisser le drapeau désactivé jusqu’à l’arrivée du provider audité.

  **Checklist de déploiement SM.** Le layering `iroha_config` (`defaults` → `user` → `actual`) permet des déploiements par étapes :
  1. **Validation de build.** Compiler les binaires avec `--features sm` (et `sm-ffi-openssl` uniquement si le backend preview est requis). Sans le feature flag, le nœud rejette `sm2` dans `allowed_signing`.
  2. **Propagation de configuration.**
     - Mettre à jour `crypto.allowed_signing` pour inclure `sm2` (liste triée lexicographiquement) et définir `crypto.default_hash = "sm3-256"`.
     - Remplacer `crypto.sm2_distid_default` par l’identifiant distinctif exigé par votre politique (p. ex., la valeur ISO/TC 68 pour votre organisation).
  3. **Alignement genesis.** Régénérer les manifestes genesis via `kagami genesis generate --consensus-mode <mode>` (utilisez `--allowed-signing sm2 --default-hash sm3-256` si vous avez besoin d’overrides) afin que les blocs `parameters` et les métadonnées injectées reflètent l’instantané de config ; validez‑les dans le contrôle de version avec le TOML mis à jour.
  4. **Tests de fumée telemetry/admission.** Avant d’activer le trafic, exécuter `cargo test -p iroha_crypto --features sm` et les scripts de la matrice d’intégration SM, puis vérifier que `/status` expose `crypto.sm_helpers_available = true` et que Torii rejette les signatures SM2 lorsque `sm2` est absent de `allowed_signing`.
  5. **Plan de rollback.** Documenter comment revenir au mode Ed25519‑only (retirer `sm2` de `allowed_signing`, restaurer `default_hash = "blake2b-256"`) et confirmer que les hôtes abandonnent les transactions SM en vol de manière déterministe.
