# Configuration Overview

This page summarizes the top-level sections of the Iroha configuration file (TOML) and what each controls. For a concrete template, see `docs/source/references/peer.template.toml`.

Defaults first: configuration values are curated for typical Iroha blockchain deployments. In most cases, you can run with the defaults unchanged. Only adjust settings when you have a specific operational need (e.g., custom ports, strict rate limits, or known network constraints).

- Sora-specific sections (`torii.sorafs`, `sorafs.*`, multi-lane `nexus.*`) require starting the daemon with `irohad --sora …` (or exporting `IROHA_SORA_PROFILE=1`). Without the flag the loader rejects configurations that enable those features.
- When `nexus.enabled = false` (Iroha 2 mode), configs stay single-lane: `/status` and `/v1/sumeragi/status` hide lane/dataspace sections, `/metrics` skips lane/dataspace labels entirely, and lane-scoped status tails return `StatusSegmentNotFound` so SDKs must branch on the flag before sending lane parameters. Config parsing rejects lane/dataspace/routing overrides in this mode—strip `nexus.*` sections unless you enable Nexus/Sora with `nexus.enabled=true` or `--sora`.
- `[common]`: Chain identity (`chain`), node key pair, trusted peers, the implicit account-domain label (`default_account_domain_label`), and the IH58 chain discriminant (`chain_discriminant`). Override these when your genesis domain or network prefix differs from the built-in defaults so IH58/compressed addresses remain canonical.
- Identity separation: the `public_key`/`private_key` pair secures transport (P2P, Torii) and is allowed to stay Ed25519/SECP/etc. regardless of `allowed_signing`. Consensus uses the BLS-Normal key from `trusted_peers` with its PoP. Transaction admission is the only place `crypto.allowed_signing`/`allowed_curve_ids` are enforced; if you want to submit BLS-signed transactions, explicitly add `bls_normal` to `allowed_signing` and the matching curve id to `crypto.curves.allowed_curve_ids`. Consensus will still function with BLS validators even when `allowed_signing` is left at the Ed25519/secp defaults.
- `[genesis]`: Location of the signed genesis file and the public key used to verify it. The referenced manifest carries parameter families such as `transaction.max_signatures`, `transaction.max_instructions`, `transaction.ivm_bytecode_size`, `transaction.max_tx_bytes`, and `transaction.max_decompressed_bytes`; update the manifest when tightening admission limits. Nodes that start with empty storage and no local `genesis.file` can fetch genesis from trusted peers by optionally pinning `expected_hash`, supplying a `bootstrap_allowlist` (defaults to `trusted_peers`), and bounding the transfer via `bootstrap_max_bytes` and `bootstrap_response_throttle_ms`. Bootstrap attempts honour `bootstrap_request_timeout_ms`, `bootstrap_retry_interval_ms`, `bootstrap_max_attempts`, and `bootstrap_enabled`; responders always enforce chain-id/public-key/hash matches and reject oversized payloads or non-allowlisted peers.
- `[network]`: P2P addresses, gossip periods/sizes, DNS refresh, bounded queue capacities, and transport options (TLS/QUIC when enabled).
- `[ivm]`: Runtime VM settings. `ivm.memory_budget_profile` (default: `compute.default_resource_profile`) selects which compute resource profile supplies `max_stack_bytes` as the guest stack budget cap.
- `[concurrency]`: Thread-pool sizing plus stack budgets for scheduler/prover workers and the IVM guest. `gas_to_stack_multiplier` (default `4`) bounds guest stack space as `min(guest_stack_bytes, gas_limit * multiplier, stack_budget)` with a 64 KiB floor and 1 GiB ceiling; `stack_budget` comes from the `ivm.memory_budget_profile` selection. Requests outside that window are clamped, logged, and exported via `/status.stack` and the `iroha_ivm_stack_*` metrics (including pool-fallback/budget-hit counters). Lower the multiplier to cap recursion on low-gas routes or raise it (within the guest stack cap) to admit deeper Kotodama frames.
- `network.peer_gossip_period_ms` (default: `250`ms) also governs the topology refresh cadence. The `IROHA_P2P_TOPOLOGY_UPDATE_MS` env override is now ignored; adjust the config field instead. Gossip periods (`block_gossip_period_ms`, `transaction_gossip_period_ms`, `peer_gossip_period_ms`) and `idle_timeout_ms` are clamped to a minimum of 100ms to avoid tight loops.
- `network.trust_gossip` (default: `true`) advertises support for TrustGossip frames. Setting it to `false` negotiates the capability off at handshake time, dropping TrustGossip messages while leaving peer-address gossip intact. Drops are tagged via `p2p_trust_gossip_skipped_total{direction,reason}` so operators can spot opt-outs or peers that refused the capability.
- Trust scoring knobs under `[network]` (`trust_decay_half_life_ms`, `trust_penalty_bad_gossip`, `trust_penalty_unknown_peer`, `trust_min_score`) apply in permissioned overlays: trust frames that reference peers outside the active topology incur the unknown-peer penalty, increment `p2p_trust_penalties_total{reason="unknown_peer"}`, and senders below `trust_min_score` are ignored until decay recovers. Public/NPoS overlays skip the unknown-peer penalty so trust gossip remains open.
- Transaction gossip tuning: `network.transaction_gossip_drop_unknown_dataspace` drops batches for dataspaces missing from the lane catalog; `network.transaction_gossip_restricted_target_cap` caps restricted fanout (defaults to the full commit topology); `network.transaction_gossip_public_target_cap` caps public-lane fanout (default 16, set `null` for broadcast); `network.transaction_gossip_public_target_reshuffle_ms` and `network.transaction_gossip_restricted_target_reshuffle_ms` control how often target selection reshuffles per plane (defaults to `transaction_gossip_period_ms`, clamped to ≥100ms); `network.transaction_gossip_restricted_fallback` (`drop` | `public_overlay`) controls whether restricted gossip falls back to online peers when the commit topology is empty; and `network.transaction_gossip_restricted_public_payload` (`refuse` | `forward`, default `refuse`) currently refuses to leak restricted payloads when only the public overlay is available (forwarding is blocked until encrypted transport lands). Telemetry surfaces the drop/forward policy via gauges and `/status.tx_gossip` entries (caps, targets/peers, fallback surface, reason/outcome) so operators can alert on blocked or rerouted batches.
- `network.soranet_handshake.pow`: Challenge configuration for SoraNet admission. Fields: `required`, `difficulty`, `max_future_skew_secs`, `min_ticket_ttl_secs`, `ticket_ttl_secs`, `revocation_store_capacity`, `revocation_max_ttl_secs`, and `revocation_store_path`. The revocation store controls how many signed tickets are retained on disk and how long they remain valid for replay detection.
- `network.soranet_handshake.pow.puzzle`: Argon2 puzzle gate toggles. Set `enabled = true` to require memory-hard tickets and tune `memory_kib` (>=4096), `time_cost`, and `lanes` (1–16). When enabled the relay still honors `difficulty`, `max_future_skew_secs`, and `min_ticket_ttl_secs`; disabling the block reverts to classical hashcash PoW.
- `network.soranet_handshake.kem_suite`: Optional textual override (`"mlkem512"`, `"mlkem768"`, or `"mlkem1024"`; `kyber*` aliases are also accepted) for the ML-KEM profile used during the SoraNet handshake. When set it supersedes `kem_id` and validates the configured Kyber key lengths for the selected suite.
- `[sumeragi]`: Consensus role (validator/observer) and collector settings (K, redundant-send r), plus timing slack options. When `da_enabled=true`, availability evidence (availability votes or an RBC `READY` quorum) is tracked for DA commitments, but consensus does **not** wait on it. Availability deadlines are scaled by `da_quorum_timeout_multiplier` and `da_availability_timeout_multiplier`, with `da_availability_timeout_floor_ms` enforcing a minimum window for logging/rebroadcast heuristics. Per-queue backpressure is configured with `msg_channel_cap_votes`, `msg_channel_cap_block_payload`, `msg_channel_cap_rbc_chunks`, `msg_channel_cap_blocks`, and `control_msg_channel_cap`; legacy `msg_channel_cap` acts as a fallback only when per-queue caps are omitted. `mode_flip_enabled` (default `true`) acts as an operator kill switch for live permissioned↔NPoS cutovers: when `false` the node advertises the staged mode via status/telemetry but refuses to flip until the flag is restored. Persistence failures back off according to `kura_store_retry_interval_ms`/`kura_store_retry_max_attempts` so blocks stay pending until Kura accepts them or the retry budget is exhausted; `commit_inflight_timeout_ms` aborts stuck commit jobs so proposals cannot stall indefinitely. Missing-block fetch prefers certificate signers for `missing_block_signer_fallback_attempts` attempts (default 1; set 0 to skip signer preference) before falling back to the full commit topology. Membership hash drift is surfaced through `membership_mismatch_alert_threshold` (consecutive mismatches before alerting) and `membership_mismatch_fail_closed` (drop consensus messages from still-mismatching peers). The pacemaker proposal interval is derived from `npos.block_time_ms`, `npos.timeouts.propose`, `pacemaker_rtt_floor_multiplier`, and capped by `pacemaker_max_backoff_ms`; raise the RTT floor or block time to slow proposals on high-latency links. RBC READY/DELIVER quorum always follows the commit topology (`Topology::min_votes_for_commit()`).
- `[torii]`: Public API server settings such as listen address, request size caps, and query store parameters.
- `[torii.operator_auth]`: WebAuthn/mTLS gate for operator endpoints. `enabled` turns on operator authentication; `require_mtls` enforces ingress mTLS (expects `X-Forwarded-Client-Cert` from the edge proxy). `token_fallback` controls bootstrap token usage (`disabled|bootstrap|always`), `token_source` selects `operator|api|both`, and `tokens` configures the operator-token allowlist. `rate_per_minute`/`burst` apply to auth attempts, while `lockout.{failures,window_secs,duration_secs}` controls temporary lockouts. `webauthn` is required when `enabled = true` and configures `rp_id`, `rp_name`, `origins`, `user_id`, `user_name`, `user_display_name`, `challenge_ttl_secs`, `session_ttl_secs`, `require_user_verification`, and `allowed_algorithms`. Credentials persist under `<torii.data_dir>/operator_auth/operator_webauthn.json`. When `token_fallback = "bootstrap"`, tokens are accepted only until the first credential is enrolled; after that operators must log in and send `x-iroha-operator-session`.
- `torii.events_buffer_capacity` (default: `10000`): Broadcast channel size used for `/v1/events/sse` and webhook enqueuing. Lower the value to bound memory/backpressure when subscribers lag.
- `torii.ws_message_timeout_ms` (default: `10000`): WebSocket message timeout (read/write) for Torii event/block streams; increase it for slow clients or high-latency links.
- `torii.app_api.*`: Pagination/backpressure defaults for JSON convenience endpoints. `default_list_limit` seeds `limit` when omitted, `max_list_limit` and `max_fetch_size` bound page/fetch sizes, and `rate_limit_cost_per_row` controls how much each requested row costs under the app API rate limiter.
- `torii.webhook.*`: Delivery/backoff tuning for the webhook worker. `queue_capacity` caps on-disk pending deliveries (default: `10000`), `max_attempts` bounds retries (default: `12`), `backoff_initial_ms`/`backoff_max_ms` set the exponential retry window (defaults: `1000`/`60000`), and `{connect,write,read}_timeout_ms` configure HTTP timeouts for delivery attempts.
- `torii.push.*` (feature stub): Push-notification bridge toggles and rate limits. `enabled` gates the bridge; `rate_per_minute`/`burst` apply a token bucket to outbound requests, `max_topics_per_device` bounds topic subscriptions per token, and `connect_timeout_ms`/`request_timeout_ms` bound HTTP attempts to FCM/APNS. Provide either `fcm_api_key` or APNS `apns_endpoint` + `apns_auth_token` to enable the respective provider.
- `torii.da_ingest.*`: Replay cache and manifest spool configuration for the DA pipeline.
  - `telemetry_cluster_label` (optional): Sets the `cluster` label emitted on Taikai ingest/viewer metrics so dashboards can distinguish environments (e.g., `region-a` vs `staging`).
    Set `governance_metadata_key_hex` to a 32-byte ChaCha20-Poly1305 key when you need
    Torii to seal governance-only metadata; pair it with `governance_metadata_key_label`
    (a short identifier such as `"primary"`) so encrypted entries advertise which key
    produced the ciphertext. Torii encrypts cleartext governance metadata automatically
    and refuses manifests whose pre-encrypted payloads carry missing or mismatched labels.
    The nested `replication_policy` table defines the enforced `RetentionPolicy` per
    blob class (see `docs/source/da/replication_policy.md`). Override `default_retention`
    to change the baseline hot/cold windows and add entries under
    `replication_policy.overrides` (e.g., `class = "taikai_segment"`) to tighten
    specific classes; Torii rewrites caller-supplied manifests to match these values
    and rejects mismatched envelopes.
  - `debug_match_filters` (default: `false`) enables deterministic filter-debug traces formerly gated by the `TORII_DEBUG_MATCH` environment toggle. Keep this off in production.
- `[torii.iso_bridge.reference_data]`: Paths to ISO reference datasets (`isin_crosswalk_path`, `bic_lei_path`, `mic_directory_path`), the ingest cadence (`refresh_interval_secs`), and optional `cache_dir` for persisting copies plus provenance metadata (version/source/checksum) of each loaded snapshot.
- `[torii.soranet_privacy_ingest]`: Guards for `/v1/soranet/privacy/{event,share}`. `enabled` must be `true` to expose the endpoints; `require_token` + `tokens` enforce the `X-SoraNet-Privacy-Token` (or `X-API-Token`) header, `allow_cidrs` restricts submitters by CIDR (empty list denies by default), and `rate_per_sec`/`burst` apply a token-bucket limiter. Rejects surface `401/403/429` with retry hints and labelled telemetry counters so operators can alert on abuse; see `docs/source/sorafs_authz_runbook.md` for the rollout checklist.
- `[torii.sorafs]`: SoraFS discovery cache controls surfaced on the Torii side. Values are copied from the canonical `[sorafs.*]` blocks during config parsing so set the root sections for operator intent; the Torii view exists for runtime snapshots/telemetry mirrors.
- `[sorafs.discovery]`: Canonical discovery toggle and capability allow-list. `discovery_enabled` defaults to `false` (Sora profiles such as `tests/fixtures/base.toml` enable it explicitly), `known_capabilities` ships with `["torii_gateway", "chunk_range_fetch", "vendor_reserved"]`, and the optional `[sorafs.discovery.admission]` table points to the governance envelope directory (`envelopes_dir`).
- `[sorafs.storage]`: Embedded SoraFS worker configuration. Key fields: `enabled`, `data_dir` (storage root), `max_capacity_bytes`, `max_parallel_fetches`, `max_pins`, `por_sample_interval_secs`, optional `alias`, and (when you want settlements written to disk) `governance_dag_dir` pointing at the filesystem directory where governance artefacts should be materialised. The `[sorafs.storage.adverts]` table exposes advert metadata (`stake_pointer`, `availability`, `max_latency_ms`, `topics`); runtime snapshots mirror the same structure under `torii.sorafs_storage`.
- `[sorafs.storage.pin]`: Authentication/rate limits for `/v1/sorafs/storage/pin`. `require_token=true` enforces a bearer token (either via `Authorization: Bearer ...` or `X-SoraFS-Pin-Token`), `tokens` supplies the allow-list, `allow_cidrs` restricts clients by CIDR, and `rate_limit` applies a token-bucket with optional ban. Torii rejects pin attempts before touching storage when any of these guards fail, returning `401/403/429` with `Retry-After` on throttled responses.
- `[sorafs.quota]`: Control-plane quota windows (capacity declaration, storage pin, telemetry, deal telemetry, dispute, and PoR submission). The Torii view under `torii.sorafs_quota` inherits these values verbatim.
- `[sorafs.alias_cache]`: Alias-proof cache policy (`positive_ttl`, `refresh_window`, `hard_expiry`, `negative_ttl`, `revocation_ttl`, `rotation_max_age`). Defaults enforce 10 min / 2 min / 15 min / 60 s / 5 min / 6 h windows; tighten by lowering values only. The runtime snapshot mirrors them under `torii.sorafs_alias_cache`.
- `sorafs.rollout_phase`: High-level PQ rollout selector (`canary`, `ramp`, `default`). Clients inherit the corresponding anonymity stage unless `sorafs.anonymity_policy` is set explicitly.
- `sorafs.anonymity_policy`: Staged SoraNet PQ rollout flag (`anon-guard-pq`, `anon-majority-pq`, `anon-strict-pq`). Defaults to Stage A (`anon-guard-pq`); ratchet forward to Stage B/C when PQ coverage is sufficient. Override only when a component must diverge from the phase.
- `[nts]`: Network Time Service sampling cadence, smoothing, and health thresholds. `sample_interval_ms` is clamped to >=100ms. Health is evaluated using `min_samples`, `max_offset_ms`, and `max_confidence_ms` (0 disables the max checks); `enforcement_mode` controls admission behavior for time-sensitive instructions (`warn` logs and allows, `reject` blocks). Advisory only; consensus does not depend on NTS.
- `[kura]`: Block storage path and memory window; initialization policy for snapshots/replay. `block_sync_roster_retention` bounds how many certified commit-roster snapshots are kept on disk for block-sync recovery (default 7200), and `roster_sidecar_retention` caps how many roster sidecars remain in `pipeline/roster_sidecars.*` after compaction (default 512).
- `[logger]`: Log level and format.
- `[queue]`: Transaction queue limits and TTL.
- `[nexus]`: Multi-lane routing catalog, manifest registry, and governance modules. Lane entries accept optional `visibility`, `lane_type`, `governance`, `settlement`, `proof_scheme` (`merkle_sha256` default; `kzg_bls12_381` lanes receive deterministic KZG commitments derived at ingest), and free-form `metadata = { key = "value" }`. The `registry` block controls `manifest_directory`, `cache_directory`, and `poll_interval_ms` (default 60s). The `governance` block provides a `default_module` and `modules` mapping, where each module exposes `module_type` plus `params` key/value overrides for downstream hosts.
  - Lane metadata key `da_manifest_policy` tunes DA manifest enforcement per lane: `strict` (default) blocks missing/unreadable manifests and hash mismatches, while `audit`/`audit_only`/`warn` allow missing manifests to proceed with warnings (hash mismatches stay fatal).
  - `nexus.staking.public_validator_mode` / `nexus.staking.restricted_validator_mode` select the validator lifecycle per lane: `stake_elected` (default for public lanes) keeps the staking/NPoS path enabled, while `admin_managed` (default for restricted lanes) bypasses staking and expects peer admission via governance/admin flows. Stake-elected lanes require a registered peer with a live consensus key present in the commit topology before admission and activate at epoch boundaries; admin-managed lanes keep the genesis peer roster alive so permissioned deployments can progress without staking.
  - `[nexus.axt]`: Timing/expiry guardrails. `slot_length_ms` must be within `1..=600_000` (default `1`); hosts derive `current_slot = block.creation_time_ms / slot_length_ms`. `max_clock_skew_ms` defaults to `0`, is capped at `60_000`, and must not exceed `slot_length_ms`; expiry checks for proofs/handles include this allowance. `proof_cache_ttl_slots` (default `1`, validated `1..=64`) bounds how many slots accepted/rejected proofs stay in the host cache for reuse/replay rejection. `replay_retention_slots` (default `128`, validated `1..=4_096`) controls how long handle-usage/replay ledgers are retained across peers/restarts; raise it only when handles remain valid for many slots and you need replay protection to span those windows. See `crates/iroha_config/tests/fixtures/nexus_axt_full.toml` for a 1s/250ms/TTL=8 sample.
  - `[nexus.lane_relay_emergency]`: Guards the emergency validator override path used by `SetLaneRelayEmergencyValidators`. `enabled` defaults to `false`; when enabled, the authority must be a multisig account meeting `multisig_threshold`/`multisig_members` minimums (defaults 3-of-5). Overrides are ignored while disabled.
- `[governance]`: Verifying keys, conviction parameters, approval thresholds, turnout floor, and council sizing/eligibility.
  - `jdg_signature_schemes` (default: `["simple_threshold"]`): Allowed JDG attestation signature schemes. Supported values: `simple_threshold`, `bls_normal_aggregate`.
  - `[governance.runtime_upgrade_provenance]`: Runtime upgrade provenance enforcement. Configure `mode` (`optional|required`), `require_sbom`, `require_slsa`, `signature_threshold`, and `trusted_signers` to enforce SBOM/SLSA/signature policy on runtime upgrade manifests.
- `[governance.sorafs_pin_policy]`: Governance-enforced SoraFS manifest policy. Fields: `min_replicas_floor` (default 1), optional `max_replicas_ceiling` (default 5), optional `max_retention_epoch`, and `allowed_storage_classes` (e.g., `["hot", "cold"]`; omit to allow any tier).
- `[governance.sorafs_pricing]`: Launch pricing schedule covering storage/egress rates, collateral
  multiplier, and credit settlement parameters. By default the schedule matches the values in
  `sorafs_pricing.md`; updates should be rolled out with the `SetPricingSchedule` instruction so all
  peers replay the same changes.
- `[governance.sorafs_telemetry]`: Capacity telemetry authz/replay guardrails. `require_submitter` and
  `require_nonce` default to `true`; `submitters` defines the global allow-list, while
  `per_provider_submitters` overrides the list for specific providers (overrides take precedence over
  the global list). `max_window_gap_secs` and `reject_zero_capacity` enforce window hygiene. Configure
  per-provider overrides whenever the submitting account differs from the global defaults; the authz
  runbook in `docs/source/sorafs_authz_runbook.md` provides operator steps and config examples.
- `[snapshot]`: Periodic snapshotting controls and on-disk location.
- `telemetry_enabled`: Master kill switch (default: true). When false the daemon skips telemetry workers, hides Torii telemetry routes, and bypasses metric collection regardless of profile.
- `telemetry_profile`: Capability bundle (default: `operator`). `operator` exposes `/status`, `extended` also enables Prometheus `/metrics`, `developer` unlocks the Sumeragi JSON/SSE endpoints, and `full` combines both; `telemetry_enabled = false` forces the effective profile to `disabled`.
- `[telemetry_redaction]`: Telemetry field redaction policy. `mode` (`strict|allowlist|disabled`) defaults to `strict`; `operator`/`extended`/`full` profiles require `strict` and a build that includes `log-obfuscation`. `allowlist` is a list of normalized field names permitted to bypass keyword redaction and must match the approved policy documented in `docs/source/telemetry.md`.
- `[telemetry_integrity]`: Tamper-evident export settings. `enabled` (default: `true`) attaches a hash chain to websocket and dev-telemetry payloads. `signing_key_hex` is an optional 32-byte hex key for keyed Blake3 signatures, `signing_key_id` labels key-rotation events, and `state_dir` persists the chain across restarts (each sink writes a `telemetry_integrity_<sink>.json` state file).
- Build-time instrumentation: ISI counters (`isi{kind="total|success"}`) and timing histograms (`isi_times`) emitted by `#[metrics]` require building with `iroha_core` `expensive-telemetry` (or the `irohad` `expensive-telemetry` feature, which implies `telemetry`). Without that feature the macro is a no-op and ISI metrics are not emitted.
- `[telemetry]`: Optional destination and retry policies for metrics/events. `telegram_metrics_period_ms` is clamped to >=100ms when set.
- `[dev_telemetry]`: Developer-only structured log sink (JSON file).
- `[pipeline]`: Execution pipeline: parallel overlay construction/apply, worker count, overlay bounds, signature batch limits, and Kotodama admission guardrails.
  - `ivm_max_cycles_upper_bound` caps the `max_cycles` header value accepted from Kotodama bytecode. Set to `0` to disable the admission guard (still clamped at execution time by the VM).
  - `ivm_max_decoded_instructions` limits the number of decoded VM instructions per contract. `0` disables the check.
  - `ivm_max_decoded_bytes` limits the decoded byte length after instruction expansion. `0` disables the check.
- `[norito]`: Serialization tuning for Norito (compression sizes/levels and adaptive layout toggles). Deterministic by design.
- `[streaming]`: Control-plane key material for Norito streaming (HPKE + control signatures).
  - `identity_public_key` / `identity_private_key` (optional): Ed25519 multihash strings that override the node’s main key pair for streaming signatures—useful when the validator key is GOST or another non-Ed25519 algorithm.
  - `kyber_public_key` / `kyber_secret_key` (optional): Hex-encoded Kyber-768 keys enabling HPKE; must be provided together.
  - `session_store_dir` (default: `./storage/streaming`): Directory for encrypted session snapshots.
  - `feature_bits` (default: `0b11`): Capability mask advertised during QUIC capability negotiation. Bit 0 enables feedback hints, bit 1 flags privacy-overlay provider support, and higher bits are reserved. Viewers demanding unsupported bits are rejected.
  - `soranet` (default: enabled): Automatically bridges manifest privacy routes onto SoraNet when manifest entries omit explicit metadata.
    - `enabled` (default: `true`): Gate the auto-provisioning logic; disable to keep existing manifests untouched.
    - `exit_multiaddr` (default: `/dns/torii/udp/9443/quic`): Exit relay multi-address forwarded to Torii once the circuit leaves SoraNet.
    - `padding_budget_ms` (default: `25`): Low-latency padding allowance in milliseconds applied to the circuit; `null` disables extra padding.
    - `access_kind` (default: `authenticated`): Exit posture enforced when bridging (`authenticated` or `read-only`).
    - `channel_salt` (default: `iroha.soranet.channel.seed.v1`): Domain string hashed with the stream/route identifiers to derive blinded channel IDs.
  - `sync` (default: observe-only): Audio/video sync enforcement policy for NSC-28b.
    - `enabled` (default: `false`): Flip to `true` once the validator should start evaluating drift reports.
    - `observe_only` (default: `true`): When `true`, violations are logged and exported via telemetry but segments are still accepted; set to `false` to hard-fail.
    - `min_window_ms` (default: `5000`): Minimum diagnostic window required before enforcement.
    - `ewma_threshold_ms` (default: `10`): Sustained EWMA drift threshold (milliseconds) that triggers rejections.
    - `hard_cap_ms` (default: `12`): Maximum per-sample drift (milliseconds) before immediate failure.
- `[relay]` (tools/soranet-relay): Reference daemon for SoraNet entry/exit relays.
  - `mode` (`"Entry" | "Middle" | "Exit"`): Selects the privacy role, telemetry labels, and default padding rules.
  - `listen` / `admin_listen`: QUIC listener and optional admin socket (metrics + control).
  - `padding`: Stream-padding profile. `cell_size` (bytes) and `max_idle_millis`
    (interval) describe the per-circuit heartbeat. `global_rate_limit_bytes_per_sec`
    (default: `0`, disabled) enables the token-bucket throttle that caps aggregate
    padding bandwidth across every circuit, and `burst_bytes` bounds the shared
    bucket capacity (defaults to `max(cell_size, rate)` when omitted) so relays can
    absorb short spikes without violating the steady-state budget.
- `exit_routing.norito_stream`: Optional bridge to Torii’s Norito streaming adapter. When enabled the relay scans Torii’s filesystem spool (`exit-<relay-id>/norito-stream/*.norito`) and forwards active routes over WebSocket.
- `exit_routing.kaigi_stream`: Optional bridge to Kaigi conferencing hubs. When enabled the relay loads routes from `exit-<relay-id>/kaigi-stream/*.norito`, dials the configured `hub_ws_url`, and derives blinded room identifiers from the channel metadata. Tuning knobs include the connection timeout (`connect_timeout_millis`), public/authenticated GAR categories, filesystem spool override, and `route_refresh_secs` interval.
    - `torii_ws_url`: `ws://` or `wss://` endpoint exposed by Torii for Norito streaming traffic.
    - `connect_timeout_millis` (default: `5000`): Relays abort connection attempts that exceed this bound.
    - `padding_target_millis` (default: `35`): Idle padding cadence applied when the spool entry omits `padding_budget_ms`. Set to `0` to disable background padding.

  **SM rollout checklist.** The `iroha_config` layering (`defaults` → `user` → `actual`) allows staged rollouts:
  1. **Build validation.** Compile binaries with `--features sm` (and `sm-ffi-openssl` only if the preview backend is required). Without the feature flag the node rejects `sm2` in `allowed_signing`.
  2. **Configuration propagation.**
     - Update `crypto.allowed_signing` to include `sm2` (keep the list lexicographically sorted) and set `crypto.default_hash = "sm3-256"`.
     - Override `crypto.sm2_distid_default` with the distinguishing identifier mandated by your policy (e.g., the ISO/TC 68 value for your organisation).
  3. **Genesis alignment.** Regenerate genesis manifests via `kagami genesis generate --consensus-mode <mode>` (use `--allowed-signing sm2 --default-hash sm3-256` if you need overrides) so the `parameters` blocks and injected metadata mirror the config snapshot; commit them to version control alongside the updated TOML.
  4. **Telemetry & admission smoke tests.** Before turning on traffic, run `cargo test -p iroha_crypto --features sm` and the SM integration matrix scripts, then verify `/status` exposes `crypto.sm_helpers_available = true` and that Torii rejects SM2 signatures when `sm2` is absent from `allowed_signing`.
  5. **Rollback plan.** Document how to revert to Ed25519-only mode (remove `sm2` from `allowed_signing`, restore `default_hash = "blake2b-256"`) and confirm hosts drop in-flight SM transactions deterministically.

Notes
- Acceleration is automatic by default. Backends run golden self-tests on first use and disable themselves at runtime if a mismatch is detected; the VM falls back to scalar/SIMD and remains bit-identical across hardware.
- Prefer editing `peer.template.toml`-style files and extending them in deployment. Runtime behavior is configured via `iroha_config`; environment variable overrides exist for developer tooling but should not be relied on for operator workflows.
