# Репо үйл ажиллагаа ба нотлох гарын авлага (замын зураг F1)

Репо хөтөлбөр нь хоёр талт болон гурван талын санхүүжилтийг тодорхойлогч байдлаар нээж өгдөг
Norito заавар, CLI/SDK туслахууд болон ISO 20022 паритет. Энэ тэмдэглэлийг барьж байна
замын зураглалын чухал үе шатыг хангахад шаардлагатай үйл ажиллагааны гэрээ **F1 — репо
амьдралын мөчлөгийн баримтжуулалт ба багаж хэрэгсэл**. Энэ нь ажлын урсгалд чиглэсэн нэмэлт юм
[`repo_runbook.md`](./repo_runbook.md) дараах байдлаар илэрхийлнэ.

- CLI/SDK/хэрэглэх хугацаандаа амьдралын мөчлөгийн гадаргуу (`crates/iroha_cli/src/main.rs:3821`,
  `python/iroha_python/iroha_python_rs/src/lib.rs:2216`,
  `crates/iroha_core/src/smartcontracts/isi/repo.rs:1`);
- тодорхойлогч нотлох баримт/нотолгоо барих (`integration_tests/tests/repo.rs:1`);
- гурван талын асран хамгаалах, барьцаа хөрөнгө орлуулах зан үйл; болон
- засаглалын хүлээлт (хос хяналт, аудитын зам, буцаах тоглоомын ном).

## 1. Хамрах хүрээ & Хүлээн авах шалгуур

Замын газрын зургийн F1 зүйл нь дөрвөн сэдвээр хаалттай хэвээр байна; энэ баримт бичигт
шаардлагатай олдворууд болон тэдгээрийг хангасан код/тестийн холбоосууд:

| Шаардлага | Нотлох баримт |
|-------------|----------|
| Репо → урвуу репо → орлуулах |-г хамарсан детерминист тооцооны баталгаа `integration_tests/tests/repo.rs` нь төгсгөл хоорондын урсгал, давхардсан ID хамгаалалт, ахиуц түвшний шалгалт, барьцаа солих амжилт/бүтэлгүйтлийн тохиолдлуудыг авч үздэг. The suite runs as part of `cargo test --workspace`. `crates/iroha_core/src/smartcontracts/isi/repo.rs` (`repo_deterministic_lifecycle_proof_matches_fixture`) агшин зуурын эхлэл → захын зай → орлуулах хүрээн дэх тодорхойлогч амьдралын мөчлөгийн дижест төхөөрөмж нь аудиторууд каноник ачааллыг ялгах боломжтой. |
| Гурван намын мэдээлэл | Ажиллах цаг нь кастодианы мэддэг урсгалуудыг хэрэгжүүлдэг: `RepoAgreement::custodian` + `RepoAccountRole::Custodian` үйл явдлууд (`crates/iroha_data_model/src/repo.rs:74`, `crates/iroha_data_model/src/events/data/events.rs:742`). |
| Барьцаа хөрөнгө орлуулах туршилт | Урвуу хөлийн инвариантууд нь дутуу барьцаалсан орлуулалтыг үгүйсгэдэг (`crates/iroha_core/src/smartcontracts/isi/repo.rs:417`) ба интеграцийн туршилтууд нь орлуулалтын эргэлтийн дараа (`integration_tests/tests/repo.rs:261`) дэвтэр зөв арилдаг болохыг баталж байна. |
| Маржин дуудлагын хэмнэл ба оролцогчийн хэрэгжилт | `integration_tests/tests/repo.rs::repo_margin_call_enforces_cadence_and_participant_rules` нь `RepoMarginCallIsi` дасгал хийж, хэмнэлд тохируулсан хуваарь гаргах, дутуу дуудлагаас татгалзах, зөвхөн оролцогчдод зориулсан зөвшөөрөл зэргийг нотлон харуулж байна. |
| Засаглалаар батлагдсан runbooks | Энэхүү гарын авлага болон `repo_runbook.md` нь аудитад зориулсан CLI/SDK журам, залилан мэхлэх/буцаах алхмууд болон нотлох баримт цуглуулах зааварчилгааг өгдөг. |

## 2. Амьдралын мөчлөгийн гадаргуу

### 2.1 CLI & Norito барилгачид

- `iroha app repo initiate|unwind|margin|margin-call` боолт `RepoIsi`,
  `ReverseRepoIsi`, `RepoMarginCallIsi`
  (`crates/iroha_cli/src/main.rs:3821`). Дэд команд бүр `--input` / дэмждэг.
  `--output`, ингэснээр ширээ нь зааварчилгааны ачааллыг давхар баталгаажуулах боломжтой
  өргөн мэдүүлэх. Кастодианы чиглүүлэлт нь `--custodian`-ээр илэрхийлэгддэг.
- `repo query list|get` нь `FindRepoAgreements`-г ашиглан гэрээний агшин зуурын зургийг авах боломжтой.
  нотлох баримтын багцад зориулж JSON олдвор руу дахин чиглүүлэх.
- `crates/iroha_cli/tests/cli_smoke.rs:2637`-ийн дагуу CLI утааны сорил нь баталгаатай
  Аудиторуудын хувьд файлыг ялгаруулах зам нь тогтвортой хэвээр байна.

### 2.2 SDK болон автоматжуулалтын дэгээ- Python холболтууд нь `RepoAgreementRecord`, `RepoCashLeg`,
  `RepoCollateralLeg`, болон тав тухтай барилгачид
  (`python/iroha_python/iroha_python_rs/src/lib.rs:2216`) тиймээс автоматжуулалт боломжтой
  гүйлгээг цуглуулж, `next_margin_check_after`-г дотооддоо үнэлэх.
- JS/Swift туслахууд ижил Norito байршлыг ашиглан дахин ашигладаг.
  `javascript/iroha_js/src/instructionBuilders.js` ба
  Санамж бичигт зориулсан `IrohaSwift/Sources/IrohaSwift/ConfidentialEncryptedPayload.swift`
  харьцах; SDK нь репо удирдлагын товчлууруудыг холбохдоо энэ баримт бичигт хандах ёстой.

### 2.3 Бүртгэлийн үйл явдал ба телеметр

Амьдралын мөчлөгийн үйлдэл бүр нь агуулсан `AccountEvent::Repo(...)` бичлэгүүдийг ялгаруулдаг
`RepoAccountEvent::{Initiated,Settled,MarginCalled}` ачааллын хамрах хүрээг хамарсан
оролцогчийн үүрэг (`crates/iroha_data_model/src/events/data/events.rs:742`). Түлх
эдгээр үйл явдлуудыг өөрийн SIEM/логийн агрегатор руу оруулж, хөндлөнгийн хяналт шалгалтын бүртгэлийг олж авна
ширээний үйлдэл, маржин дуудлага, кастодианы мэдэгдлийн хувьд.

### 2.4 Тохиргооны тархалт ба баталгаажуулалт

Зангилаанууд нь `[settlement.repo]` мөрийн репо засаглалын товчлууруудыг залгидаг.
`iroha_config` (`crates/iroha_config/src/parameters/user.rs:4071`). Үүнийг эмчил
засаглалын нотолгооны гэрээний нэг хэсэг болгон хэсэгчлэн - хувилбарын хяналт дээр үе шаттай
репо багцын хажууд шилжүүлж, өөрчлөлтийг өөрийнхөөрөө оруулахаас өмнө хэш хийнэ
автоматжуулалт эсвэл ConfigMap. Хамгийн бага профайл дараах байдлаар харагдаж байна.

```toml
[settlement.repo]
default_haircut_bps = 1500
margin_frequency_secs = 86400
eligible_collateral = ["bond#wonderland", "note#wonderland"]

[settlement.repo.collateral_substitution_matrix]
"bond#wonderland" = ["note#wonderland", "bill#wonderland"]
```

Үйл ажиллагааны хяналтын хуудас:

1. Дээрх хэсэгчилсэн хэсгийг (эсвэл өөрийн үйлдвэрлэлийн хувилбарыг) тохиргооны репо руу оруулна уу
   Энэ нь `irohad`-ийг тэжээж, удирдлагын багц дотор өөрийн SHA-256-г бичнэ.
   Шүүгчид таны байрлуулахаар төлөвлөж буй байтыг ялгаж чадна.
2. Өөрчлөлтийг флот даяар шилжүүлээрэй (системийн нэгж, Kubernetes ConfigMap гэх мэт).
   болон зангилаа бүрийг дахин эхлүүлнэ үү. Нэвтрүүлсний дараа шууд Torii-г аваарай
   гарал үүслийн тохиргооны агшин зураг:

   ```bash
   curl -sS "${TORII_URL}/v2/configuration" \
     -H "Authorization: Bearer ${TOKEN}" | jq .
   ```

   `ToriiClient.get_configuration()` нь Python SDK дээр мөн адил боломжтой
   автоматжуулалтад бичигдсэн нотлох баримт хэрэгтэй үед зорилго.【python/iroha_python/src/iroha_python/client.py:5791】
3. Ажиллах цаг нь хүссэн хэмнэл/үсний засалтыг одоо хэрэгжүүлж байгааг асууж баталгаажуулна уу.
   `FindRepoAgreements` (эсвэл `iroha app repo margin --agreement-id ...`) болон
   суулгагдсан `RepoGovernance` утгуудыг шалгаж байна. JSON хариултуудыг хадгалах
   `artifacts/finance/repo/<agreement>/agreements_after.json` дор; эдгээр үнэт зүйлс
   нь `[settlement.repo]`-ээс гаралтай тул хоёрдогч гэрч болдог.
   Torii-ийн `/v2/configuration` агшин зуурын зураг хангалтгүй байна.
4. TOML хэсэг болон Torii/CLI хормын хувилбаруудыг хоёуланг нь хадгална уу.
   засаглалын хүсэлт гаргахаас өмнө нотлох баримтын багц. Аудиторууд чадвартай байх ёстой
   хэсэгчилсэн хэсгийг дахин тоглуулж, түүний хэшийг баталгаажуулж, ажиллах цагийн харагдацтай холбоно уу.

Энэхүү ажлын урсгал нь репо ширээ нь түр орчмын хувьсагчдаас хэзээ ч найддаггүй
тохиргооны зам нь тодорхойгүй хэвээр байх бөгөөд засаглалын тасалбар бүрийг агуулна
Замын зураглал F1 дээр хүлээгдэж буй ижил `iroha_config` нотолгооны багц.

### 2.5 Тодорхойлох баталгааны бэхэлгээ

`repo_deterministic_lifecycle_proof_matches_fixture` нэгжийн туршилт (харна уу
`crates/iroha_core/src/smartcontracts/isi/repo.rs`) нь үе шат бүрийг цуваа болгодог
репо амьдралын мөчлөгийг Norito JSON хүрээ болгон тохируулж, үүнийг каноник бэхэлгээтэй харьцуулна.
`crates/iroha_core/tests/fixtures/repo_lifecycle_proof.json`, мөн хэш хийдэг
багц (бэхэлгээний тоймыг дагаж мөрддөг
`crates/iroha_core/tests/fixtures/repo_lifecycle_proof.digest`). Үүнийг дотооддоо дараахаар ажиллуулна уу:

```bash
cargo test -p iroha_core \
  -- --exact smartcontracts::isi::repo::tests::repo_deterministic_lifecycle_proof_matches_fixture
```Энэ туршилтыг одоо анхдагч `cargo test -p iroha_core` багцын нэг хэсэг болгон ажиллуулж байгаа тул CI
агшин зуурын зургийг автоматаар хамгаална. Репо семантик эсвэл бэхэлгээ өөрчлөгдөх бүрт,
JSON-г аль алиныг нь сэргээж, дараах байдлаар боловсруулаарай:

```bash
scripts/regen_repo_proof_fixture.sh
```

Туслах нь бэхлэгдсэн `rust-toolchain.toml` сувгийг ашиглаж, бэхэлгээг дахин бичдэг.
`crates/iroha_core/tests/fixtures/`-ийн дагуу, мөн детерминист бэхэлгээг дахин ажиллуулна
Тиймээс шалгасан агшин зуурын зураг/дижест нь ажиллах үеийн төлөвтэй синхрончлогддог
аудиторууд дахин тоглох болно.

### 2.4 Torii API гадаргуу

- `GET /v2/repo/agreements` нь нэмэлт хуудаслалт, шүүлтүүр бүхий идэвхтэй гэрээг буцаана
  (`filter={...}`), эрэмбэлэх, хаяг форматлах параметрүүд. Use this for quick audits or
  түүхий JSON ачаалал хангалттай байх үед хяналтын самбар.
- `POST /v2/repo/agreements/query` нь бүтэцлэгдсэн асуулгын дугтуйг (хуудаслалт, эрэмбэлэх,
  `FilterExpr`, `fetch_size`) тул доод урсгалын үйлчилгээнүүд дэвтэрт тодорхой байдлаар хуудаслах боломжтой.
- JavaScript SDK нь одоо `listRepoAgreements`, `queryRepoAgreements` болон давтагчийг харуулж байна.
  туслахууд нь browser/Node.js хэрэгсэл нь Rust/Python-той ижил төрлийн DTO-г хүлээн авдаг.

### 2.4 Анхдагч тохиргоо

Зангилаанууд `[settlement.repo]` руу уншина
эхлүүлэх үед `iroha_config::parameters::actual::Repo`; аливаа репо заавар
Параметрийг тэг дээр үлдээх нь өмнөх өгөгдмөлүүдийн эсрэг хэвийн байна
гинжин хэлхээнд бүртгэгдсэн байна.【crates/iroha_core/src/smartcontracts/isi/repo.rs:40】 Энэ нь
засаглал нь SDK бүрийг хөндөхгүйгээр суурь бодлогыг нэмэгдүүлэх (эсвэл бууруулах) боломжийг олгодог
бодлогын өөрчлөлтийг бүрэн баримтжуулсан тохиолдолд дуудлагын сайт.

- `default_haircut_bps` – `RepoGovernance::haircut_bps()` үед дахин үс засах
  тэгтэй тэнцүү. Ажиллах хугацаа нь үүнийг хадгалахын тулд хатуу 10000bps таазанд хавчуулдаг
  тохиргоонууд нь эрүүл.【crates/iroha_core/src/smartcontracts/isi/repo.rs:44】
- `margin_frequency_secs` – `RepoMarginCallIsi`-ийн каденс. Хүсэлтүүдийг тэглэсэн
  энэ утгыг өвлөж авдаг тул хэмнэлтийг богиносгосноор ширээг илүү их захын зайд оруулахад хүргэдэг
  ихэвчлэн анхдагчаар.【crates/iroha_core/src/smartcontracts/isi/repo.rs:49】
- `eligible_collateral` – `AssetDefinitionId`-ийн нэмэлт зөвшөөрлийн жагсаалт. Хэзээ
  жагсаалт хоосон биш `RepoIsi` багцаас гадуур аливаа барьцаанаас татгалзаж, урьдчилан сэргийлэх
  шалгагдаагүй бондын санамсаргүй холболт.【crates/iroha_core/src/smartcontracts/isi/repo.rs:57】
- `collateral_substitution_matrix` – эх барьцааны газрын зураг →
  зөвшөөрөгдсөн орлуулагчид. `ReverseRepoIsi` нь орлуулалтыг зөвхөн
  матриц нь түлхүүр болон түүний орлуулах бичигдсэн тодорхойлолтыг агуулдаг
  утгын массив; эс бөгөөс тайвширч бүтэлгүйтэж, засаглал баталсан гэдгийг нотлох
  шат.【crates/iroha_core/src/smartcontracts/isi/repo.rs:74】

Эдгээр товчлуурууд нь зангилааны тохиргоонд `[settlement.repo]` дор байрладаг бөгөөд
`iroha_config::parameters::user::Repo`-ээр задлан шинжилсэн тул тэдгээрийг барьж авах хэрэгтэй
засаглалын нотлох баримтын багц бүр.【crates/iroha_config/src/parameters/user.rs:3956】

```toml
[settlement.repo]
default_haircut_bps = 1750
margin_frequency_secs = 43200
eligible_collateral = ["bond#wonderland", "note#wonderland"]

[settlement.repo.collateral_substitution_matrix]
"bond#wonderland" = ["note#wonderland", "bill#wonderland"]
```

**Өөрчлөлтийн удирдлагын хяналтын хуудас**1. Санал болгож буй TOML хэсгийг (орлуулах матрицын дельтануудыг оруулаад), хэш
   үүнийг SHA-256-тай холбож, хэсэгчилсэн болон хэшийг хоёуланг нь засаглалд хавсаргана
   тасалбарыг ашигласнаар тоймчид байтыг үгчлэн хуулбарлах боломжтой.
2. Санал/ард нийтийн санал асуулга доторх хэсэгчилсэн хэсгийг лавлана уу (жишээ нь
   засаглалын CLI дээрх `--notes` талбар) болон шаардлагатай зөвшөөрлийг цуглуул.
   F1-ийн хувьд. Гарын үсэг зурсан зөвшөөрлийн багцыг хэсэгчилсэн байдлаар хадгална уу.
3. Өөрчлөлтийг флот даяар шилжүүлээрэй: `[settlement.repo]`-г шинэчилж, тус бүрийг дахин эхлүүлнэ үү.
   зангилаа, дараа нь `GET /v2/configuration` агшин зуурын зургийг авах (эсвэл
   `ToriiClient.getConfiguration`) үе тэнгийнх нь хэрэглэж буй утгыг нотлох.
4. `integration_tests/tests/repo.rs` plus-г дахин ажиллуулна уу
   `repo_deterministic_lifecycle_proof_matches_fixture` ба дараа нь бүртгэлүүдийг хадгална
   config diff руу оруулснаар аудиторууд шинэ өгөгдмөлүүд хадгалагдаж байгааг харах боломжтой
   детерминизм.

Матрицын оруулгагүйгээр ажиллах хугацаа нь хөрөнгийг өөрчилдөг орлуулалтаас татгалздаг
ерөнхий `eligible_collateral` жагсаалт зөвшөөрсөн ч гэсэн тодорхойлолт; хийх
репо нотлох баримтын хамт агшин зуурын тохиргоог хийснээр аудиторууд яг таг хуулбарлах боломжтой
репо захиалах үед хэрэгжүүлсэн бодлого.

### 2.5 Configuration evidence & drift detection

Norito/`iroha_config` сантехник нь одоо шийдвэрлэсэн репо бодлогыг ил болгож байна.

## 6. Залилан, буцаах журам

- **Хос удирдлага:** `--output` ашиглан зааварчилгааг байнга гаргаж,
  Хамтран гарын үсэг зурахад зориулсан JSON. Процессын түвшинд нэг намын мэдүүлгээс татгалзах
  ажиллуулах цаг нь санаачлагчийн эрх мэдлийг хэрэгжүүлдэг ч гэсэн.
- **Өөрчлөгдсөн бүртгэл:** `RepoAccountEvent` урсгалыг өөрийн системд тусгана уу
  SIEM нь хуурамч зааварчилгааг илрүүлэх боломжтой (үе тэнгийн гарын үсэг байхгүй).
- **Буцах:** Хэрэв репо хугацаанаас нь өмнө задлах шаардлагатай бол `repo unwind` бичнэ үү.
  ижил гэрээний id-тай бөгөөд `--notes` талбарыг өөрийн үйл явдалд хавсаргана уу
  GAR-аар батлагдсан буцаах тоглуулах номыг лавлаж буй трекер.
- **Луйврын өсөлт:** Хэрэв зөвшөөрөлгүй репо гарч ирвэл зөрчлийг экспортлоорой
  `RepoAccountEvent` ачааллыг хүлээн авч, удирдлагын бодлогоор дамжуулан дансуудыг царцаах, мөн
  репо засаглалын SOP дагуу зөвлөлд мэдэгдэх.

2. **Дахин шалгах.** Дахин ажиллуулах
   `sorafs reserve ledger --quote <...> --json-out ...` ба гаралтыг хэвийн болгох
   `scripts/telemetry/reserve_ledger_digest.py`-ээр дамжуулан. Хажууд нь хоолыг байрлуул
   аудиторууд XOR-ийн дүнг репотой харьцуулах боломжтой байхын тулд хөрөнгийн агшин зуурын зураг
   CLI-г дахин тоглуулахгүйгээр дэвтэр проекц.
3. **Эвлэрүүлэн зуучлалын тэмдэглэлийг нийтлэх.** Дельтануудыг нэгтгэн бичнэ үү
   `artifacts/finance/repo/<period>/treasury_reconciliation.md` лавлагаагаар:
   хөрөнгийн агшин зуурын хэш, дэвтэр дайжест хэш, хамрагдсан гэрээнүүд.
   Санхүүгийн засаглалын хянагчаас авсан тэмдэглэлийг холбоно уу, ингэснээр хянагчид баталгаажуулна
   репо гаргахыг батлахаас өмнө төрийн сангийн хамгаалалт.

### 7.2 Өрөмдлөг, буцаах давталтын нотлох баримт

Хүлээн авах шалгуур нь үе шаттай буцаалт болон ослын дасгалуудыг шаарддаг. Бүр
Өрөмдлөг эсвэл эмх замбараагүй байдлын сургуулилт нь дараахь олдворуудыг цуглуулах ёстой.

1. `repo_runbook.md` Хэсэг 4-5, ослын командлагч гарын үсэг зурсан шалгах хуудас болон
   санхүүгийн зөвлөл.
2. Бэлтгэл сургуулилтын CLI/SDK бүртгэл (`repo initiate|margin-call|unwind`) дээр нэмэх нь
   шинэчлэгдсэн амьдралын мөчлөгийн нотлох хормын хувилбар болон нотлох баримтын манифест (§§2.7–3.2) хадгалагдсан
   `artifacts/finance/repo/drills/<timestamp>/` доор.
3. Тарилгын дохиог харуулсан дохиоллын менежер эсвэл пейжерийн хуулбар
   хүлээн зөвшөөрөх зам. Бичлэгийг өрөмдлөгийн олдворуудын хажууд хаях ба
   ашиглаж байх үед Alertmanager чимээгүй байдлын ID-г оруулна уу.
4. `ops/drill-log.md` оруулга нь GAR id, манифест хэш болон өрөмдлөгийг заана.
   багц зам нь ирээдүйн аудитууд чатын бүртгэлийг хусалгүйгээр сургуулилтыг хянах боломжтой.

### 7.3 Засаглалын хяналт ба эрүүл ахуй

- Энэ `repo_runbook.md` баримт бичиг болон санхүүгийн засаглалын хянагчийг дотор нь хадгална уу.
  CLI/SDK эсвэл ажиллах үеийн төлөв өөрчлөгдөх бүрт түгжигдэх; тоймчид хүлээж байна
  үнэн зөв байхын тулд хүлээн авах хүснэгт.
- Бүрэн нотлох баримтыг хавсаргана уу (`agreements.json`, үе шаттай заавар, SSE
  хуулбар, тохиргооны агшин зуурын зураг, эвлэрэл, өрмийн олдворууд болон тест
  бүртгэлүүд) улирал тутам хянагч руу илгээнэ.
- Зохицуулахдаа `docs/source/finance/settlement_iso_mapping.md` лавлагаа
  ISO гүүрний операторуудтай тул систем хоорондын эвлэрэл нь нийцсэн хэвээр байна.

Энэхүү гарын авлагыг дагаснаар операторууд замын зураглалын F1 хүлээн авах мөрийг хангана:
детерминистик нотолгоонууд баригдсан, гурван намын болон орлуулах урсгалууд байдаг
баримтжуулсан, засаглалын журам (хос хяналт + ослын бүртгэл) байна
мод дотор кодлогдсон.
