# Repo-ի գործառնությունների և ապացույցների ուղեցույց (ճանապարհային քարտեզ F1)

Ռեպո ծրագիրը բացում է երկկողմ և եռակողմ ֆինանսավորումը վճռականությամբ
Norito հրահանգներ, CLI/SDK օգնականներ և ISO 20022 հավասարություն: Այս նշումը գրավում է
գործառնական պայմանագիրը, որն անհրաժեշտ է ճանապարհային քարտեզի կարևորագույն կետը բավարարելու համար **F1 — ռեպո
lifecycle documentation & tooling**. Այն լրացնում է աշխատանքային հոսքին ուղղված աշխատանքը
[`repo_runbook.md`](./repo_runbook.md)՝ արտահայտելով.

- կյանքի ցիկլի մակերեսները CLI/SDK-ների/աշխատանքի ընթացքում (`crates/iroha_cli/src/main.rs:3821`,
  `python/iroha_python/iroha_python_rs/src/lib.rs:2216`,
  `crates/iroha_core/src/smartcontracts/isi/repo.rs:1`);
- դետերմինիստական ​​ապացույցների/ապացույցների հավաքագրում (`integration_tests/tests/repo.rs:1`);
- եռակողմ խնամակալության և գրավի փոխարինման վարքագիծ; և
- կառավարման ակնկալիքներ (երկակի հսկողություն, աուդիտի հետքեր, հետադարձ խաղագրքեր):

## 1. Շրջանակ և ընդունման չափանիշներ

Ճանապարհային քարտեզի F1 կետը պահպանվում է չորս թեմաներով. այս փաստաթուղթը թվարկում է
պահանջվող արտեֆակտներ և կոդի/թեստերի հղումներ, որոնք արդեն բավարարում են դրանք.

| Պահանջը | Ապացույցներ |
|-------------|----------|
| Դետերմինիստական ​​հաշվարկային ապացույցներ, որոնք ծածկում են ռեպո → հակադարձ ռեպո → փոխարինում | `integration_tests/tests/repo.rs`-ը ֆիքսում է ծայրից ծայր հոսքեր, նույնականացման կրկնօրինակ պաշտպանիչներ, մարժան արագության ստուգումներ և գրավի փոխարինման հաջող/ձախողման դեպքեր: Հավաքածուն աշխատում է որպես `cargo test --workspace`: `crates/iroha_core/src/smartcontracts/isi/repo.rs` (`repo_deterministic_lifecycle_proof_matches_fixture`) ակնթարթային նկարների մեկնարկի → լուսանցք → փոխարինող շրջանակներ, որպեսզի աուդիտորները կարողանան տարբերել կանոնական ծանրաբեռնվածությունները, դետերմինիստական ​​կյանքի ցիկլի մարսողությունը: |
| Եռակողմ լուսաբանում | Runtime-ն ապահովում է պահառուին տեղեկացված հոսքերը՝ `RepoAgreement::custodian` + `RepoAccountRole::Custodian` իրադարձություններ (`crates/iroha_data_model/src/repo.rs:74`, `crates/iroha_data_model/src/events/data/events.rs:742`): |
| Գրավի փոխարինման թեստեր | Հակադարձ ոտքի ինվարիանտները մերժում են ոչ գրավադրված փոխարինումները (`crates/iroha_core/src/smartcontracts/isi/repo.rs:417`), իսկ ինտեգրման թեստերը հաստատում են, որ հաշվապահական հաշվառումը ճիշտ է մաքրվում փոխարինման հետադարձ ճանապարհից հետո (`integration_tests/tests/repo.rs:261`): |
| Մարգին-զանգերի արագություն և մասնակցի կիրարկում | `integration_tests/tests/repo.rs::repo_margin_call_enforces_cadence_and_participant_rules` վարժություններ `RepoMarginCallIsi`, որոնք ապացուցում են կադենսային պլանավորումը, վաղաժամ զանգերի մերժումը և միայն մասնակցի թույլտվությունը: |
| Կառավարության կողմից հաստատված ուղեգրեր | Այս ուղեցույցը և `repo_runbook.md`-ը տրամադրում են CLI/SDK ընթացակարգեր, խարդախության/վերադարձի քայլեր և ապացույցներ հավաքելու հրահանգներ աուդիտի համար: |

## 2. Կյանքի ցիկլի մակերեսներ

### 2.1 CLI & Norito շինարարներ

- `iroha app repo initiate|unwind|margin|margin-call` փաթաթան `RepoIsi`,
  `ReverseRepoIsi` և `RepoMarginCallIsi`
  (`crates/iroha_cli/src/main.rs:3821`): Յուրաքանչյուր ենթահրաման աջակցում է `--input` /
  `--output`, որպեսզի աշխատասեղանները կարողանան նախապես դասավորել հրահանգների օգտակար բեռները՝ կրկնակի հաստատման համար
  ներկայացում. Պահառուի երթուղին արտահայտվում է `--custodian`-ի միջոցով:
- `repo query list|get`-ն օգտագործում է `FindRepoAgreements`-ը համաձայնագրերը պատկերելու համար և կարող է
  վերահղվել JSON արտեֆակտների մեջ՝ ապացույցների փաթեթների համար:
- CLI ծխի թեստերը `crates/iroha_cli/tests/cli_smoke.rs:2637`-ի ներքո ապահովում են
  emit-to-file ուղին կայուն է մնում աուդիտորների համար:

### 2.2 SDK և ավտոմատացման կեռիկներ- Python կապերը բացահայտում են `RepoAgreementRecord`, `RepoCashLeg`,
  `RepoCollateralLeg` և հարմարավետության շինարարներ
  (`python/iroha_python/iroha_python_rs/src/lib.rs:2216`), այնպես որ ավտոմատացումը կարող է
  հավաքել գործարքները և գնահատել `next_margin_check_after` տեղական մակարդակում:
- JS/Swift օգնականները նորից օգտագործում են նույն Norito դասավորությունները՝ միջոցով
  `javascript/iroha_js/src/instructionBuilders.js` և
  `IrohaSwift/Sources/IrohaSwift/ConfidentialEncryptedPayload.swift` հուշագրի համար
  բեռնաթափում; SDK-ները պետք է հղում կատարեն այս փաստաթղթին, երբ ռեպո կառավարման կոճակները կապում են:

### 2.3 Լեջերի իրադարձություններ և հեռաչափություն

Յուրաքանչյուր կյանքի ցիկլի գործողություն արձակում է `AccountEvent::Repo(...)` գրառումներ, որոնք պարունակում են
`RepoAccountEvent::{Initiated,Settled,MarginCalled}` ծանրաբեռնվածության շրջանակը դեպի
մասնակցի դերը (`crates/iroha_data_model/src/events/data/events.rs:742`): Հրել
այդ իրադարձությունները ձեր SIEM/log ագրեգատորում, որպեսզի ստանաք կեղծումներից ակնհայտ աուդիտի մատյան
գրասեղանի գործողությունների, մարժա զանգերի և պահառուի ծանուցումների համար:

### 2.4 Կազմաձևման տարածում և ստուգում

Հանգույցները կլանում են ռեպո կառավարման կոճակները `[settlement.repo]` տողից
`iroha_config` (`crates/iroha_config/src/parameters/user.rs:4071`): Վերաբերվեք դրան
հատվածը որպես կառավարման ապացույցների պայմանագրի մաս՝ այն բեմադրել տարբերակի վերահսկման մեջ
ռեպո փաթեթի կողքին և հաշշիր այն նախքան փոփոխությունը ձեր միջոցով մղելը
ավտոմատացում կամ ConfigMap: Նվազագույն պրոֆիլն ունի հետևյալ տեսքը.

```toml
[settlement.repo]
default_haircut_bps = 1500
margin_frequency_secs = 86400
eligible_collateral = ["4fEiy2n5VMFVfi6BzDJge519zAzg", "7dk8Pj8Bqo6XUqch4K2sF8MCM1zd"]

[settlement.repo.collateral_substitution_matrix]
"4fEiy2n5VMFVfi6BzDJge519zAzg" = ["7dk8Pj8Bqo6XUqch4K2sF8MCM1zd", "6zK1LDcJ3FvkpfoZQ8kHUaW6sA7F"]
```

Գործառնական ստուգաթերթ.

1. Վերևի հատվածը (կամ ձեր արտադրության տարբերակը) մուտքագրեք կազմաձևման ռեպո
   որը կերակրում է `irohad`-ին և գրանցում է իր SHA-256-ը կառավարման փաթեթի ներսում
   վերանայողները կարող են տարբերել բայթերը, որոնք դուք նախատեսում եք տեղադրել:
2. Փոխեք փոփոխությունը ամբողջ նավատորմի վրա (համակարգային միավոր, Kubernetes ConfigMap և այլն)
   և վերագործարկեք յուրաքանչյուր հանգույց: Թողարկումից անմիջապես հետո գրավեք Torii-ը
   ծագման կոնֆիգուրացիայի նկար՝

   ```bash
   curl -sS "${TORII_URL}/v1/configuration" \
     -H "Authorization: Bearer ${TOKEN}" | jq .
   ```

   Նույնի համար `ToriiClient.get_configuration()` հասանելի է Python SDK-ում
   Նպատակը, երբ ավտոմատացման համար անհրաժեշտ է մուտքագրված ապացույց:【python/iroha_python/src/iroha_python/client.py:5791】
3. Ապացուցեք, որ գործարկման ժամանակն այժմ պարտադրում է պահանջվող արագությունը/սանրվածքը՝ հարցումով
   `FindRepoAgreements` (կամ `iroha app repo margin --agreement-id ...`) և
   ներկառուցված `RepoGovernance` արժեքների ստուգում: Պահպանեք JSON պատասխանները
   `artifacts/finance/repo/<agreement>/agreements_after.json`-ի տակ; այդ արժեքները
   բխում են `[settlement.repo]`-ից, ուստի նրանք հանդես են գալիս որպես երկրորդական վկա, երբ
   Torii-ի `/v1/configuration` նկարը բավարար չէ:
4. Պահեք երկու արտեֆակտները՝ TOML հատվածը և Torii/CLI նկարները,
   ապացույցների փաթեթ մինչև կառավարման հարցում ներկայացնելը: Աուդիտորները պետք է կարողանան
   վերարտադրեք հատվածը, ստուգեք դրա հեշը և կապեք այն գործարկման ժամանակի տեսքի հետ:

This workflow ensures repo desks never rely on ad-hoc environment variables, the
կոնֆիգուրացիայի ուղին մնում է որոշիչ, և կառավարման յուրաքանչյուր տոմս կրում է այն
`iroha_config` ապացույցների նույն հավաքածուն սպասվում է F1 ճանապարհային քարտեզում:

### 2.5 Դետերմինիստական ապացույցի ամրագոտի

`repo_deterministic_lifecycle_proof_matches_fixture` միավորի թեստը (տես
`crates/iroha_core/src/smartcontracts/isi/repo.rs`) սերիականացնում է ծրագրի յուրաքանչյուր փուլը
ռեպո կյանքի ցիկլը Norito JSON շրջանակում, այն համեմատում է կանոնական սարքի հետ՝
`crates/iroha_core/tests/fixtures/repo_lifecycle_proof.json` և հեշեր է անում
փաթեթ (հարմարանքների ամփոփումը հետևվում է
`crates/iroha_core/tests/fixtures/repo_lifecycle_proof.digest`): Գործարկեք այն տեղական միջոցով՝

```bash
cargo test -p iroha_core \
  -- --exact smartcontracts::isi::repo::tests::repo_deterministic_lifecycle_proof_matches_fixture
```Այս թեստն այժմ աշխատում է որպես լռելյայն `cargo test -p iroha_core` փաթեթի մաս, ուստի CI
ինքնաբերաբար պահպանում է լուսանկարը: Ամեն անգամ, երբ փոխվում են ռեպո իմաստաբանությունը կամ հարմարանքները,

### 2.4 Torii API մակերեսներ

- `GET /v1/repo/agreements`-ը վերադարձնում է ակտիվ համաձայնագրերը կամընտիր էջադրմամբ, զտմամբ
  (`filter={...}`), տեսակավորման և հասցեի ձևաչափման պարամետրեր: Օգտագործեք սա արագ աուդիտի համար կամ

- JavaScript SDK-ն այժմ բացահայտում է `listRepoAgreements`, `queryRepoAgreements` և կրկնիչ
  օգնականներ, որպեսզի բրաուզերի/Node.js գործիքավորումը ստանա նույն մուտքագրված DTO-ները, ինչ Rust/Python-ը:

### 2.4 Կազմաձևման լռելյայն

Հանգույցները կարդում են `[settlement.repo]`
`iroha_config::parameters::actual::Repo` գործարկման ժամանակ; ցանկացած ռեպո հրահանգ
որը թողնում է պարամետրը զրոյական մակարդակում, նորմալացվում է դրա առաջ եղած կանխադրվածների համեմատ
գրանցված է շղթայի վրա:【crates/iroha_core/src/smartcontracts/isi/repo.rs:40】 Սա
թույլ է տալիս կառավարմանը բարձրացնել (կամ իջեցնել) ելակետային քաղաքականությունը՝ առանց դիպչելու յուրաքանչյուր SDK-ին
զանգի կայք՝ պայմանով, որ քաղաքականության փոփոխությունն ամբողջությամբ փաստաթղթավորված է:

- `default_haircut_bps` – հետադարձ սանրվածք, երբ `RepoGovernance::haircut_bps()`
  հավասար է զրոյի: Աշխատաժամանակը ամրացնում է այն 10000bps կոշտ առաստաղին՝ պահելու համար

- `margin_frequency_secs` – կադանս `RepoMarginCallIsi`-ի համար: Զրոյացված հարցումներ
  ժառանգել այս արժեքը, ուստի արագության կրճատումը ստիպում է աշխատասեղաններին ավելի շատ լուսանցքներ սահմանել
  հաճախ լռելյայն:【crates/iroha_core/src/smartcontracts/isi/repo.rs:49】
- `eligible_collateral` – `AssetDefinitionId`-ների կամընտիր թույլատրելի ցուցակ: Երբ որ
  ցուցակը դատարկ չէ `RepoIsi`-ը մերժում է հավաքածուից դուրս ցանկացած գրավ՝ կանխելով
  չստուգված պարտատոմսերի պատահական մուտքագրում:【crates/iroha_core/src/smartcontracts/isi/repo.rs:57】
- `collateral_substitution_matrix` – բնօրինակ գրավի քարտեզ →
  թույլատրված փոխարինիչներ. `ReverseRepoIsi`-ն ընդունում է փոխարինումը միայն այն դեպքում, երբ
  մատրիցը պարունակում է գրանցված սահմանումը որպես բանալի և փոխարինում դրա մեջ
  արժեքային զանգված; հակառակ դեպքում լիցքաթափումը ձախողվում է՝ ապացուցելով, որ իշխանությունը հավանություն է տվել
  սանդուղք.【crates/iroha_core/src/smartcontracts/isi/repo.rs:74】

վերլուծվել է `iroha_config::parameters::user::Repo`-ի միջոցով, ուստի դրանք պետք է գրավվեն

**Փոփոխությունների կառավարման ստուգաթերթ**1. Բեմադրել առաջարկվող TOML հատվածը (ներառյալ փոխարինող մատրիցային դելտաները), հեշ
   այն SHA-256-ով և կցեք և՛ հատվածը, և՛ հեշը կառավարմանը
   տոմս, որպեսզի գրախոսները կարողանան բառացիորեն վերարտադրել բայթերը:
2. Հղում առաջարկի/հանրաքվեի ներսում գտնվող հատվածին (օրինակ՝ միջոցով
   `--notes` դաշտը կառավարման CLI-ի վրա) և հավաքեք պահանջվող հաստատումները
   F1-ի համար. Պահեք ստորագրված հաստատման փաթեթը՝ կցված հատվածով:
3. Փոխեք փոփոխությունը ամբողջ նավատորմի վրա. թարմացրեք `[settlement.repo]`, վերագործարկեք յուրաքանչյուրը
   հանգույց, այնուհետև լուսանկարեք `GET /v1/configuration` լուսանկարը (կամ
   `ToriiClient.getConfiguration`)՝ ապացուցելով կիրառական արժեքները մեկ գործընկերոջ համար:
4. Կրկին գործարկել `integration_tests/tests/repo.rs` plus
   `repo_deterministic_lifecycle_proof_matches_fixture` և պահեք տեղեկամատյանները հաջորդիվ
   config diff-ին, որպեսզի աուդիտորները կարողանան տեսնել, որ նոր կանխադրվածները պահպանվում են
   դետերմինիզմ.

Առանց մատրիցային մուտքագրման, գործարկման ժամանակը մերժում է փոխարինումները, որոնք փոխում են ակտիվը
սահմանումը, նույնիսկ եթե ընդհանուր `eligible_collateral` ցանկը դա թույլ է տալիս. պարտավորվել
config snapshots հետ ռեպո ապացույցների, որպեսզի աուդիտորները կարողանան վերարտադրել ճշգրիտ
քաղաքականությունը, որն իրականացվում է ռեպո ամրագրման ժամանակ:

### 2.5 Կազմաձևման ապացույցներ և դրեյֆի հայտնաբերում

Norito/`iroha_config` սանտեխնիկան այժմ բացահայտում է լուծված ռեպո քաղաքականությունը՝
`iroha_config::parameters::actual::Repo`, ուստի կառավարման փաթեթները պետք է ապացուցեն
կիրառվող արժեքներ մեկ գործընկերոջ համար, ոչ միայն առաջարկվող TOML-ը: Վերցրեք լուծվածը
կոնֆիգուրացիան և դրա ամփոփումը յուրաքանչյուր թողարկումից հետո.

1. Վերցրեք կոնֆիգուրացիան յուրաքանչյուր գործընկերից (`GET /v1/configuration` կամ
   `ToriiClient.getConfiguration`) և մեկուսացնել ռեպո տողը.

   ```bash
   curl -s http://<torii-host>/v1/configuration \
     | jq -cS '.settlement.repo' \
     > artifacts/finance/repo/<agreement-id>/config/repo_config_actual.json
   ```

2. Հասեք կանոնական JSON-ը և գրանցեք այն ապացույցների մանիֆեստում: Երբ որ
   նավատորմը առողջ է, հեշը պետք է համապատասխանի հասակակիցներին, քանի որ `actual`
   համատեղում է կանխադրվածները փուլային `[settlement.repo]` հատվածի հետ.

   ```bash
   shasum -a 256 artifacts/finance/repo/<agreement-id>/config/repo_config_actual.json
   ```

3. Կցեք JSON + հեշը կառավարման փաթեթին և արտացոլեք մուտքագրումը
   մանիֆեստը վերբեռնվել է կառավարման DAG-ում: Եթե որևէ գործընկեր զեկուցում է տարբերվողների մասին
   մարսել, դադարեցնել տարածումը և հաշտեցնել կազմաձևման/ վիճակի դրեյֆը նախկինում
   ընթացքը։

### 2.6 Կառավարման հաստատումներ և ապացույցների փաթեթ

Ճանապարհային քարտեզ F1 փակվում է միայն այն ժամանակ, երբ ռեպո գրասեղանները դետերմինիստական փաթեթ են տալիս
կառավարման DAG, այնպես որ յուրաքանչյուր փոփոխություն (նոր սանրվածք, պահառուի քաղաքականություն կամ գրավ
մատրիցա) պետք է ուղարկի նույն արտեֆակտները նախքան քվեարկությունը նշանակելը:【docs/source/governance_playbook.md:1】

** Ընդունման փաթեթ **1. **Հետևման ձևանմուշ** – պատճենել
   `docs/examples/finance/repo_governance_packet_template.md` ձեր ապացույցների մեջ
   գրացուցակ (օրինակ
   `artifacts/finance/repo/<agreement-id>/packet.md`) և լրացրեք մետատվյալները
   արգելափակել նախքան արտեֆակտների հեշավորումը սկսելը: Կաղապարը պահպանում է կառավարումը
   խորհրդի արագությունը դետերմինիստական է՝ թվարկելով ֆայլերի ուղիները, SHA-256 ամփոփումները և
   վերանայողների երախտագիտությունները մեկ տեղում:
2. **Հրահանգների օգտակար բեռներ** – բեմադրել մեկնարկը, լիցքաթափել և սահմանազանգել
   հրահանգներ `iroha app repo ... --output`-ով, այնպես որ կրկնակի հսկողությամբ հաստատողները վերանայում են
   բայթ-նույնական ծանրաբեռնվածություն: Հաշեք յուրաքանչյուր ֆայլ և պահեք այն տակ
   `artifacts/finance/repo/<agreement-id>/` գրասեղանի ապացույցների փաթեթի կողքին
   նշված է այս գրառման մեկ այլ տեղ:【crates/iroha_cli/src/main.rs:3821】
3. **Կազմաձևման տարբերություն** – ներառում է ճշգրիտ `[settlement.repo]` TOML հատվածը
   (կանխադրված գումարած փոխարինման մատրիցա) և դրա SHA-256-ը: Սա ապացուցում է, թե որն է

### 2.8 Տոկոսների հաշվեգրում և մարման ժամկետների կառավարում**Դատական տոկոսադրույքի մաթ.** `RepoIsi` և `ReverseRepoIsi` ստացվում են կանխիկ գումարը
հանգստանալու ժամանակ պարտք է ACT/360 օգնականից
`compute_accrued_interest()`【crates/iroha_core/src/smartcontracts/isi/repo.rs:100】
և `expected_cash_settlement()` ներսում գտնվող պահակը, որը մերժում է մարման ոտքերը
որոնք վերադարձնում են *հիմնական + տոկոս**։【crates/iroha_core/src/smartcontracts/isi/repo.rs:132】
Օգնականը նորմալացնում է `rate_bps`-ը չորս տասնորդական կոտորակի, բազմապատկելով այն
`elapsed_ms / (360 * 24h)` օգտագործելով 18 տասնորդական տեղ, և վերջապես կլորացվում է դեպի
`NumericSpec`-ի կողմից հայտարարված սանդղակը: Կառավարման փաթեթը պահպանելու համար
վերարտադրելի, գրավել չորս արժեքները, որոնք կերակրում են օգնականին.

1. `cash_leg.quantity` (հիմնական),
2. `rate_bps`,
3. `initiated_timestamp_ms`, եւ
4. Լիցքաթափման ժամանակացույցը, որը դուք մտադիր եք օգտագործել (պլանավորված GL գրառումների համար սա է
   սովորաբար `maturity_timestamp_ms`, բայց արտակարգ արձակումները գրանցում են իրականը
   `ReverseRepoIsi::settlement_timestamp_ms`):

Պահպանեք tuple-ը փուլային արձակման հրահանգի կողքին և կցեք կարճ ապացույց
հատված, ինչպիսին է.

```python
from decimal import Decimal
ACT_360_YEAR_MS = 24 * 60 * 60 * 1000 * 360

principal = Decimal("1000")
rate_bps = Decimal("1500")  # 150 bps
elapsed_ms = Decimal(maturity_ms - initiated_ms)
interest = principal * (rate_bps / Decimal(10_000)) * (elapsed_ms / Decimal(ACT_360_YEAR_MS))
expected_cash = principal + interest.quantize(Decimal("0.01"))
```

Կլորացված `expected_cash`-ը պետք է համապատասխանի հակառակ կողմում կոդավորված `quantity`-ին
ռեպո հրահանգ. Պահպանեք սցենարի ելքը (կամ հաշվիչի աշխատաթերթը):
`artifacts/finance/repo/<agreement>/interest.json`, որպեսզի աուդիտորները կարողանան վերահաշվարկել
պատկեր առանց մեկնաբանելու ձեր առևտրային աղյուսակը: Ինտեգրման հավաքակազմ
արդեն իսկ պարտադրում է նույն ինվարիանտը
(`repo_roundtrip_transfers_balances_and_clears_agreement`), բայց օպերատիվ ապացույցներ
պետք է նշի ճշգրիտ արժեքները, որոնք կվերացվեն:【integration_tests/tests/repo.rs:1】

**Մարգինա և հաշվեգրման աստիճանը:** Յուրաքանչյուր համաձայնագիր բացահայտում է կադենս օգնողներին

3. Համեմատեք `.settlement.repo` բլոկը յուրաքանչյուր լուսանկարում բեմադրվածի հետ
   `[settlement.repo]` TOML հատված; գրանցել ցանկացած դրեյֆ և կրկնել
   `repo query get --agreement-id <id> --pretty`, այնպես որ ապացույցների փաթեթը ցույց է տալիս
   ինչպես գործարկման ժամանակի կոնֆիգուրացիան, այնպես էլ նորմալացված `RepoGovernance` արժեքները
   պահվում է համաձայնագրով։【crates/iroha_cli/src/main.rs:3821】
4. Կցեք պատկերի ֆայլերը և ամփոփ ցուցիչը ապացույցների մանիֆեստին (տես
   §3.2), այնպես որ կառավարման գրառումը հաստատված փոփոխությունը կապում է իրական գործընկերոջ հետ
   կոնֆիգուրացիայի բայթեր: Կառավարման ձևանմուշը թարմացվել է՝ սա ներառելու համար
   աղյուսակ, այնպես որ յուրաքանչյուր ապագա ռեպո փաթեթ ունի նույն ապացույցը:

Այս նկարահանումները փակում են `iroha_config` փաստաթղթերի բացը
Ճանապարհային քարտեզում. վերանայողներն այժմ կարող են տարբերել բեմադրված TOML-ը բայթերից
գործընկերների հաշվետվությունները, և աուդիտորները կարող են կրկնել համեմատությունը, երբ տեղի է ունենում ռեպո փոփոխություն
հետաքննության տակ։

## 3. Դետերմինիստական ​​ապացույցների աշխատանքային հոսք1. **Գրանցեք հրահանգի ծագումը**
   - Ստեղծեք ռեպո/թուլացրեք օգտակար բեռը `iroha app repo ... --output`-ի միջոցով:
   - Պահեք `InstructionBox` JSON-ը տակ
     `artifacts/finance/repo/<agreement-id>/initiation.json`.
2. **Գրավել մատյանային վիճակը**
   - Նախկինում գործարկեք `iroha app repo query list --pretty > artifacts/.../agreements.json`
     իսկ հաշվարկից հետո՝ ապացուցելու մնացորդների մաքրվածությունը:
   - Ընտրովիորեն հարցրեք `FindAssets` `iroha json` կամ SDK օգնականների միջոցով արխիվացնելու համար
     ակտիվների մնացորդները, որոնք շոշափվել են ռեպո մասում:
3. **Միջոցառումների շարունակական հոսքեր**
   - Բաժանորդագրվեք `AccountEvent::Repo`-ին Torii SSE-ի միջոցով կամ քաշեք և կցեք
     թողարկեց JSON-ը ապացույցների գրացուցակում: Սա բավարարում է կեղծիքը
     անտառահատումների դրույթ, քանի որ իրադարձությունները ստորագրված են դիտարկված հասակակիցների կողմից
     յուրաքանչյուր փոփոխություն:
4. **Գործարկել դետերմինիստական թեստեր**
   - CI-ն արդեն աշխատում է `integration_tests/tests/repo.rs`; ձեռքով ստորագրման համար,
     գործարկեք `cargo test -p integration_tests repo::` և արխիվացրեք log plus-ը
     `target/debug/deps/repo-*` JUnit ելք:
5. **Սերիականացնել կառավարումը և կազմաձևումը**
   - Մուտք գործեք (կամ կցեք) ժամանակաշրջանի համար օգտագործված `[settlement.repo]` կոնֆիգուրը,
     ներառյալ սանրվածքը/իրավասուների ցուցակները: Սա թույլ է տալիս աուդիտի կրկնությունները համընկնել
     Գործարկման ժամանակով նորմալացված կառավարումը գրանցված է `RepoAgreement`-ում:

### 3.1 Ապացույցների փաթեթի դասավորություն

Պահպանեք այս բաժնում նշված յուրաքանչյուր արտեֆակտ մեկ համաձայնագրի ներքո
գրացուցակը, որպեսզի կառավարումը կարողանա արխիվացնել կամ հաշել մեկ ծառ: Առաջարկվող դասավորությունը հետևյալն է.

```
artifacts/finance/repo/<agreement-id>/
├── agreements_before.json
├── agreements_after.json
├── initiation.json
├── unwind.json
├── margin/
│   └── 2026-04-30.json
├── events/
│   └── repo-events.ndjson
├── config/
│   ├── settlement_repo.toml
│   └── peers/
│       ├── peer01.json
│       └── peer02.json
├── repo_proof_snapshot.json
├── repo_proof_digest.txt
└── tests/
    └── repo_lifecycle.log
```

- `agreements_before/after.json`-ը գրավում է `repo query list` արդյունքը, որպեսզի աուդիտորները կարողանան
  ապացուցել, որ գրանցամատյանը մաքրել է պայմանագիրը.
- `initiation.json`, `unwind.json` և `margin/*.json` ճշգրիտ Norito են
  `iroha app repo ... --output`-ով բեմադրված օգտակար բեռներ:
- `events/repo-events.ndjson`-ը կրկնում է `AccountEvent::Repo(*)` հոսքը, մինչդեռ
  `tests/repo_lifecycle.log`-ը պահպանում է `cargo test` ապացույցը:
- `repo_proof_snapshot.json`-ը և `repo_proof_digest.txt`-ը գալիս են լուսանկարից
  թարմացնել ընթացակարգը §2.7-ում և թույլ տալ վերանայողներին վերահաշվարկել կյանքի ցիկլի հեշը
  առանց զրահը նորից գործարկելու:
- `config/settlement_repo.toml` պարունակում է `[settlement.repo]` հատվածը
  (սանրվածքներ, փոխարինման մատրիցա), որն ակտիվ էր ռեպո-ի կատարման ժամանակ:
- `config/peers/*.json`-ը լուսանկարում է `/v1/configuration` նկարներ յուրաքանչյուր հասակակիցի համար,
  փակելով օղակը փուլային TOML-ի և գործարկման ժամանակի արժեքների գործընկերների հաշվետվության միջև
  ավելի քան Torii:

### 3.2 Հեշ մանիֆեստի ստեղծում

Յուրաքանչյուր փաթեթին կցեք դետերմինիստական մանիֆեստ, որպեսզի վերանայողները կարողանան ստուգել հեշերը
առանց արխիվը բացելու: Օգնականը `scripts/repo_evidence_manifest.py`-ում
քայլում է համաձայնագրի գրացուցակով, գրանցում `size`, `sha256`, `blake2b` և վերջինը
փոփոխել է յուրաքանչյուր ֆայլի ժամանակի դրոշմը և գրում է JSON ամփոփագիր.

```bash
python3 scripts/repo_evidence_manifest.py \
  --root artifacts/finance/repo/wonderland-2026q1 \
  --agreement-id 7mxD1tKRyv32je4kZwcWa9wa33bX \
  --output artifacts/finance/repo/wonderland-2026q1/manifest.json \
  --exclude 'scratch/*'
```

Գեներատորը դասավորում է ուղիները բառարանագրորեն, բաց է թողնում ելքային ֆայլը, երբ այն ապրում է
նույն գրացուցակի ներսում և թողարկում է ընդհանուր գումարներ, որոնք կառավարումը կարող է ուղղակիորեն պատճենել
փոփոխության տոմսի մեջ: Երբ `--output`-ը բաց է թողնվում, մանիֆեստը տպվում է
stdout, որը հարմար է աշխատասեղանի վերանայումների ժամանակ արագ տարբերությունների համար:
Օգտագործեք `--exclude <glob>`՝ քերծվածքային նյութը բաց թողնելու համար (օրինակ՝ `--exclude 'scratch/*' --exclude '*.tmp'`)
առանց ֆայլերը փաթեթից դուրս տեղափոխելու; գլոբային օրինաչափությունը միշտ կիրառվում է
`--root`-ի համեմատ ուղին:Մանիֆեստի օրինակ (կտրված հակիրճության համար).

### 3.3 Կառավարման փոփոխության գրանցամատյան և հետադարձ զորավարժություններ

Ֆինանսական խորհուրդն ակնկալում է ռեպո-ի յուրաքանչյուր հարցում, սանրվածքի ուղղում կամ փոխարինում
մատրիցայի փոփոխություն՝ հասնելու համար վերարտադրվող կառավարման փաթեթով, որը կարող է կապված լինել
անմիջապես հանրաքվեի արձանագրությունից։【docs/source/governance_playbook.md:1】

1. **Կառուցեք կառավարման փաթեթը**
   - Պատճենեք պայմանագրի համար նախատեսված ապացույցների փաթեթը
     `artifacts/finance/repo/<agreement-id>/governance/`.
   - Ավելացնել `gar.json` (խորհրդի հաստատման գրառում), `referendum.md` (ով հաստատեց
     և որ հեշերը նրանք վերանայեցին), և `rollback_playbook.md`
     ամփոփելով հակադարձման ընթացակարգը `repo_runbook.md`-ից
     §§4–5.【docs/source/finance/repo_runbook.md:1】
   - Վերցրեք դետերմինիստական մանիֆեստի հեշը §3.2 դյույմից
     `hashes.txt`, որպեսզի վերանայողները կարողանան հաստատել այն բեռները, որոնք տեսնում են Torii համընկնումով
     բեմադրված բայթերը:
2. **Հղում հանրաքվեի փաթեթին**
   - Երբ աշխատում է `iroha app governance referendum submit` (կամ համարժեք SDK
     օգնական) ներառել մանիֆեստի հեշը `hashes.txt`-ից `--notes`-ում
     ծանրաբեռնվածություն, որպեսզի GAR-ը հետ մատնանշի դեպի անփոփոխ փաթեթը:
   - Ներկայացրեք նույն հեշը կառավարման հետախույզի կամ տոմսերի համակարգի ներսում
     աուդիտի հետքերը չեն հիմնվում սքրինշոթերի վահանակների վրա:
3. **Փաստաթղթային վարժանքներ և հետադարձումներ**
   - Հանրաքվեի անցումից հետո թարմացրեք `ops/drill-log.md` ռեպո-ով
     համաձայնագրի id, տեղակայված կազմաձևման հեշ, GAR id և օպերատորի կոնտակտ, որպեսզի
     եռամսյակային վարժանքների գրառումը ներառում է ֆինանսական գործողություններ:【ops/drill-log.md:1】
   - Եթե հետադարձ փորվածք է կատարվում, ամրացրեք ստորագրվածը
     `rollback_playbook.md` և CLI ելքը `iroha app repo unwind`-ից մինչև
     `governance/drills/<timestamp>.log` և տեղեկացնել խորհրդին՝ օգտագործելով նույնը
     քայլեր, որոնք նկարագրված են կառավարման խաղագրքում:

Օրինակ դասավորությունը:

```
artifacts/finance/repo/<agreement-id>/governance/
├── gar.json
├── hashes.txt
├── referendum.md
├── rollback_playbook.md
└── drills/
    └── 2026-05-12T09-00Z.log
```

GAR-ի, հանրաքվեի և փորված արտեֆակտների պահպանում կյանքի ցիկլի հետ մեկտեղ
ապացույցները երաշխավորում են, որ ռեպո յուրաքանչյուր փոփոխություն բավարարում է F1-ի կառավարման ճանապարհային քարտեզը
բար, առանց պահանջելու պատվերով տոմսերի ուղղագրություն հետագայում:

### 3.4 Կյանքի ցիկլի կառավարման ստուգաթերթ

Ճանապարհային քարտեզը **F1** կոչ է անում կառավարման ծածկույթ մեկնարկի, հաշվեգրման/մարժայի և
եռակողմ հանգստություն. Ստորև բերված աղյուսակը համախմբում է հաստատումները, որոշիչ
արտեֆակտներ և փորձարկման տեղեկանքներ կյանքի ցիկլի յուրաքանչյուր քայլի համար, որպեսզի ֆինանսական սեղանները կարողանան մեջբերել a

** Գրասեղանի ժամանակացույց.**1. Պատճենեք ընդունման ձևանմուշը, լրացրեք մետատվյալների բլոկը (համաձայնագրի ID, GAR տոմս,
   պահառու, կոնֆիգուրացիայի հեշ) և ստեղծել ապացույցների գրացուցակը:
2. Բեմադրեք յուրաքանչյուր հրահանգ (`initiate`, `margin-call`, `unwind`, փոխարինում)
   `--output` ռեժիմ, հեշեք JSON-ը և գրանցեք հաստատումները յուրաքանչյուր հեշի կողքին:
3. Թողարկեք կյանքի ցիկլի ապացուցված նկարը և դրսևորեք այն բեմադրելուց անմիջապես հետո
   Կառավարման վերանայողները կարող են վերահաշվարկել ամփոփումը նույն ռեպո հարմարանքներով:
4. Հայելի `RepoAccountEvent::*` SSE ծանրաբեռնվածություն տուժած հաշիվների համար և իջեցրեք
   արտահանված NDJSON-ը `artifacts/finance/repo/<agreement-id>/events.ndjson`-ում
   նախքան փաթեթը ներկայացնելը:
5. Քվեարկությունն անցնելուց հետո թարմացրեք `hashes.txt`-ը GAR նույնացուցիչով,
   կոնֆիգուրացիայի հեշը և ցուցիչ ստուգման գումարը, որպեսզի խորհուրդը կարողանա հետևել տարածմանը
   առանց տեղական սցենարների վերագործարկման:

### 3.5 Կառավարման փաթեթի արագ մեկնարկ

Ճանապարհային քարտեզի F1 վերանայողները խնդրեցին ներկայացնել հակիրճ ստուգաթերթ, որին կարող են հղում կատարել
ապացույցների փաթեթի հավաքում. Հետևեք ստորև նշված հաջորդականությանը, երբ ռեպո խնդրեք
կամ քաղաքականության փոփոխությունը գնում է դեպի կառավարում.

1. **Արտահանեք կյանքի ցիկլի ապացուցված արտեֆակտները։**
   ```bash
   mkdir -p artifacts/finance/repo/<slug>
   REPO_PROOF_SNAPSHOT_OUT=artifacts/finance/repo/<slug>/repo_proof_snapshot.json \
   REPO_PROOF_DIGEST_OUT=artifacts/finance/repo/<slug>/repo_proof_digest.txt \
   cargo test -p iroha_core \
     -- --exact smartcontracts::isi::repo::tests::repo_deterministic_lifecycle_proof_matches_fixture
   ```
   Արտահանված JSON + digest-ը արտացոլում է մուտքագրված սարքերի տակ
   `crates/iroha_core/tests/fixtures/`, այնպես որ վերանայողները կարող են վերահաշվարկել կյանքի ցիկլը
   շրջանակ՝ առանց ամբողջ հավաքակազմը նորից գործարկելու (տես §2.7): Կարող եք նաև զանգահարել
   `scripts/regen_repo_proof_fixture.sh --bundle-dir artifacts/finance/repo/<slug>`
   նույն ֆայլերը մեկ քայլով թարմացնելու և պատճենելու համար:
2. **Բեմադրեք և հեշեք յուրաքանչյուր հրահանգը։** Ստեղծեք մեկնարկ/մարժա/թուլացում
   օգտակար բեռներ `iroha app repo ... --output`-ով: Գրավեք SHA-256-ը յուրաքանչյուր ֆայլի համար
   (պահել `hashes/` տակ) ուստի `docs/examples/finance/repo_governance_packet_template.md`
   կարող է վկայակոչել նույն բայթերը, որոնք վերանայված գրասեղանները:
3. **Պահպանեք մատյան/կոնֆիգուրացիայի նկարները։** Արտահանեք `repo query list` ելքը առաջ/հետո
   կարգավորում, թափեք `[settlement.repo]` TOML բլոկը, որը կկիրառվի, և
   արտացոլել համապատասխան `AccountEvent::Repo(*)` SSE հոսքը
   `artifacts/finance/repo/<slug>/events/repo-events.ndjson`. ԳԱՐ-ից հետո
   անցումներ, նկարահանեք `/v1/configuration` նկարներ յուրաքանչյուր գործընկերոջ համար (§2.9) և պահեք դրանք
   `config/peers/`-ի ներքո, ուստի կառավարման փաթեթն ապացուցում է, որ ներդրումը հաջողվել է:
4. **Ստեղծեք ապացույցների մանիֆեստը:**
   ```bash
   python3 scripts/repo_evidence_manifest.py \
     --root artifacts/finance/repo/<slug> \
     --agreement-id <repo-id> \
     --output artifacts/finance/repo/<slug>/manifest.json
   ```
   Ներառեք մանիֆեստի հեշը կառավարման տոմսի կամ GAR րոպեների ներսում
   աուդիտորները կարող են տարբերակել փաթեթը՝ առանց չմշակված փաթեթը ներբեռնելու (տես §3.2):
5. **Հավաքեք փաթեթը։** Պատճենեք ձևանմուշը
   `docs/examples/finance/repo_governance_packet_template.md`, լրացրեք մետատվյալները,
   կցեք ապացույցի նկարը/դիզեստը, մանիֆեստը, կազմաձևման հեշը, SSE արտահանումը և փորձարկումը
   տեղեկամատյանները, ապա մեջբերեք SHA-256 մանիֆեստը հանրաքվեի `--notes` դաշտում:
   Պահեք ավարտված Markdown-ը արտեֆակտների կողքին, որպեսզի հետադարձները ժառանգեն
   ճշգրիտ ապացույցներ, որոնք դուք ուղարկել եք հաստատման համար:

Վերոնշյալ քայլերն իրականացնելը ռեպո հարցումը կատարելուց անմիջապես հետո նշանակում է
կառավարման փաթեթը պատրաստ է խորհրդի գումարմանն պես՝ խուսափելով վերջին րոպեից
փորձում է վերստեղծել հեշեր կամ իրադարձությունների հոսքեր:

## 4. Եռակողմ խնամակալություն և գրավի փոխարինում- **Պահառուներ.** Անցնելով `--custodian <account>` երթուղիների գրավը դեպի
  պահառու պահոց; Runtime-ը պարտադրում է հաշվի գոյությունը և թողարկում դերերի պիտակավորում
  իրադարձություններ, որպեսզի պահառուները կարողանան հաշտեցնել (`RepoAccountRole::Custodian`): Պետությունը
  մեքենան մերժում է այն համաձայնագրերը, որոնց պահառուն համապատասխանում է կողմերից որևէ մեկին:
- **Գրավի փոխարինում.** Լիցքաթափվող ոտքը կարող է այլ գրավ տալ
  քանակ/սերիա փոխարինման ընթացքում այնքան ժամանակ, քանի դեռ այն **ոչ պակաս** է
  գրավադրված գումարը *և* փոխարինման մատրիցը թույլ է տալիս զույգին. `ReverseRepoIsi`
  կատարում է երկու պայմանները
  (`crates/iroha_core/src/smartcontracts/isi/repo.rs:414`–`437`): Ինտեգրումը
  թեստային հավաքածուն իրականացնում է և՛ մերժման ուղին, և՛ հաջող փոխարինումը
  շրջագայություն (`integration_tests/tests/repo.rs:261`–`359`), մինչդեռ ռեպո միավորը
  թեստերը ներառում են նոր մատրիցային քաղաքականությունը:
- **ISO 20022 քարտեզագրում.** ISO ծրարներ կառուցելիս կամ արտաքին համադրման ժամանակ
  համակարգեր, նորից օգտագործեք դաշտի քարտեզագրումը, որը փաստաթղթավորված է
  `docs/source/finance/settlement_iso_mapping.md` (`colr.007`, `sese.023`,
  `sese.025`), այնպես որ Norito օգտակար բեռը և ISO հաստատումները մնան համաժամեցված:

## 5. Գործառնական ստուգաթերթեր

### Ամենօրյա նախաբաց

1. Արտահանեք `iroha app repo query list`-ի միջոցով սահմանված չմարված համաձայնագիրը:
2. Համեմատեք գանձապետական գույքագրման հետ և ապահովեք իրավասու գրավի կազմաձևը
   համապատասխանում է նախատեսված գրքին:
3. Բեմադրեք գալիք ռեպոները/թուլացրեք `--output`-ով և հավաքեք կրկնակի հաստատումներ:

### Ներօրյա մոնիտորինգ

1. Բաժանորդագրվեք `AccountEvent::Repo`-ին նախաձեռնողի/համաձայնագրի/պահառուի համար
   հաշիվներ; զգոն, երբ անսպասելի նախաձեռնություններ են տեղի ունենում:
2. Օգտագործեք `iroha app repo margin --agreement-id ID` (կամ
   `RepoAgreementRecord::next_margin_check_after`) ամենժամյա արագությունը հայտնաբերելու համար
   դրեյֆ; ձգան `repo margin-call` երբ `is_due = true`.
3. Մուտքագրեք բոլոր լուսանցքային զանգերը օպերատորի սկզբնատառերով և կցեք CLI JSON ելքը
   ապացույցների տեղեկատու.

### Օրվա վերջ + հետհաշվարկ

1. Կրկին գործարկեք `repo query list`-ը և հաստատեք, որ չլուծված համաձայնությունները հանվել են:
2. Արխիվացրեք `RepoAccountEvent::Settled` օգտակար բեռները և ստուգեք կանխիկ գումարը/գրավը
   մնացորդներ `FindAssets`-ի միջոցով:
3. Գրեք փորված գրառում `ops/drill-log.md`-ում, երբ ռեպո վարժանքները կամ միջադեպերի փորձարկումները
   վազել; կրկին օգտագործել `scripts/telemetry/log_sorafs_drill.sh` կոնվենցիաները ժամանակային դրոշմանիշների համար:

## 6. Խարդախության և վերադարձի ընթացակարգեր

- **Երկակի հսկողություն.** Միշտ ստեղծեք հրահանգներ `--output`-ով և պահեք
  JSON for co-signing. Մերժել միակողմանի ներկայացումները գործընթացի մակարդակով
  չնայած գործարկման ժամանակը պարտադրում է նախաձեռնողի լիազորությունները:
- **Ակնհայտ գրանցում.** Հայելի `RepoAccountEvent` հոսքը
  SIEM, այնպես որ ցանկացած կեղծ հրահանգ կարող է հայտնաբերվել (բացակայում են գործընկերների ստորագրությունները):
- **Վերադարձ.** Եթե ռեպո պետք է ժամանակից շուտ արձակվի, ներկայացրեք `repo unwind`
  նույն համաձայնագրի ID-ով և կցեք `--notes` դաշտը ձեր միջադեպում
  որոնիչ, որը վկայակոչում է GAR-ի կողմից հաստատված հետ վերադարձի գրքույկը:
- **Խարդախության սրացում.** Եթե չարտոնված ռեպո հայտնվեն, արտահանեք վիրավորանքը
  `RepoAccountEvent` ծանրաբեռնվածություն, սառեցրեք հաշիվները կառավարման քաղաքականության միջոցով և
  տեղեկացնել խորհրդին ռեպո կառավարման SOP-ի վերաբերյալ:

## 7. Զեկուցում և հետևում

### 7.1 Գանձապետարանի հաշտեցում և մատյանային ապացույցներՃանապարհային քարտեզ **F1** և գլոբալ բնակավայրերի պաշտպանիչ ուղին (ճանապարհային քարտեզ.md#L1975-L1978)
պահանջում են, որ յուրաքանչյուր ռեպո վերանայում ներառի գանձապետական դետերմինիստական ապացույցներ: Արտադրել ա
եռամսյակային փաթեթ յուրաքանչյուր գրքի համար՝ հետևելով ստորև ներկայացված ստուգաթերթին:

1. **Պատկերի մնացորդներ։** Օգտագործեք `FindAssets` հարցումը, որը գործում է
   `iroha ledger asset list` (`crates/iroha_cli/src/main_shared.rs`) կամ
   `iroha_python` օգնական՝ արտահանելու XOR մնացորդները `<katakana-i105-account-id>`-ի համար,
   `<katakana-i105-account-id>` և յուրաքանչյուր գրասեղանի հաշիվ, որը ներգրավված է վերանայման մեջ: Խանութ
   որ JSON տակ
   `artifacts/finance/repo/<period>/treasury_assets.json` և ձայնագրեք git-ը
   commit/toolchain ուղեկցող `README.md`-ում:
2. **Խաչակի ստուգում մատյանների կանխատեսումները։** Կրկին գործարկել
   `sorafs reserve ledger --quote <...> --json-out ...` և նորմալացնել ելքը
   `scripts/telemetry/reserve_ledger_digest.py`-ի միջոցով: Դիջեստը տեղադրեք կողքին
   ակտիվների նկարը, որպեսզի աուդիտորները կարողանան տարբերել XOR-ի գումարները ռեպո-ից
   մատյանային պրոյեկցիա՝ առանց CLI-ի վերարտադրման:
3. **Հրապարակեք հաշտեցման նշումը։** Ամփոփեք դելտաները
   `artifacts/finance/repo/<period>/treasury_reconciliation.md`՝ հղում կատարելով.
   ակտիվների լուսապատկերի հեշը, գրանցամատյանների ամփոփագիրը և ծածկված համաձայնագրերը:
   Կցեք գրառումը ֆինանսական կառավարման հետագծից, որպեսզի վերանայողները կարողանան հաստատել
   գանձապետական ծածկույթը մինչև ռեպո թողարկումը հաստատելը:

### 7.2 Հորատեք և հետադարձ փորձի ապացույցներ

Ընդունման չափանիշները պահանջում են նաև փուլային հետադարձումներ և միջադեպերի վարժանքներ: Ամեն
փորված կամ քաոսի փորձը պետք է հավաքի հետևյալ արտեֆակտները.

1. `repo_runbook.md` Բաժիններ 4–5 ստուգաթերթ, ստորագրված միջադեպի հրամանատարի և
   ֆինանսների խորհուրդ.
2. CLI/SDK տեղեկամատյանները փորձի համար (`repo initiate|margin-call|unwind`) գումարած
   թարմացված կյանքի ցիկլի ապացույցի լուսանկարը և ապացույցների մանիֆեստը (§§2.7–3.2) պահվում են
   `artifacts/finance/repo/drills/<timestamp>/`-ի ներքո:
3. Alertmanager կամ փեյջերի վերծանումներ, որոնք ցույց են տալիս ներարկված ազդանշանները և
   ճանաչման հետք. Գցեք սղագրությունը փորված արտեֆակտների կողքին և
   ներառել Alertmanager լռության ID-ն, երբ այն օգտագործվում էր:
4. `ops/drill-log.md` գրառում, որը հղում է կատարում GAR id-ին, մանիֆեստի հեշին և փորվածին
   փաթեթավորեք ուղին, որպեսզի ապագա աուդիտները կարողանան հետևել փորձերին՝ առանց չաթի մատյանները քերելու:

### 7.3 Կառավարման հետագծում և փաստաթղթերի հիգիենա

- Պահեք այս փաստաթուղթը, `repo_runbook.md`-ը և ֆինանսական կառավարման հետագծիչը
  lockstep ամեն անգամ, երբ փոխվում է CLI/SDK կամ գործարկման ժամանակի վարքագիծը. վերանայողները ակնկալում են
  Ընդունման աղյուսակը ճշգրիտ մնալու համար:
- Կցեք ապացույցների ամբողջական փաթեթը (`agreements.json`, փուլային հրահանգներ, SSE
  տառադարձումներ, կազմաձևման պատկեր, հաշտեցումներ, փորված արտեֆակտներ և փորձարկում
  տեղեկամատյաններ) հետևողին յուրաքանչյուր եռամսյակային վերանայման համար:
