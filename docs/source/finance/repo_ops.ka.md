## 2. სასიცოცხლო ციკლის ზედაპირები

### 2.1 CLI & Norito მშენებლები

- `iroha app repo initiate|unwind|margin|margin-call` შეფუთვა `RepoIsi`,
  `ReverseRepoIsi` და `RepoMarginCallIsi`
  (`crates/iroha_cli/src/main.rs:3821`). თითოეული ქვებრძანება მხარს უჭერს `--input` /
  `--output`, რათა მერხებმა შეძლონ ინსტრუქციის დატვირთვა ორმაგი დამტკიცებისთვის მანამდე
  წარდგენა. მეურვის მარშრუტი გამოხატულია `--custodian`-ით.
- `repo query list|get` იყენებს `FindRepoAgreements`-ს ხელშეკრულებების გადასაღებად და შეუძლია
  გადამისამართდეს JSON არტეფაქტებში მტკიცებულების პაკეტებისთვის.
- CLI კვამლის ტესტები `crates/iroha_cli/tests/cli_smoke.rs:2637`-ის ქვეშ უზრუნველყოფს
  Emit-to-File გზა აუდიტორებისთვის სტაბილური რჩება.

### 2.2 SDK და ავტომატიზაციის კაკვები- პითონის შეკვრა ამჟღავნებს `RepoAgreementRecord`, `RepoCashLeg`,
  `RepoCollateralLeg` და კომფორტის მშენებლები
  (`python/iroha_python/iroha_python_rs/src/lib.rs:2216`) ასე რომ ავტომატიზაციას შეუძლია
  შეაგროვეთ ტრანზაქციები და შეაფასეთ `next_margin_check_after` ადგილობრივად.
- JS/Swift დამხმარეები ხელახლა იყენებენ იგივე Norito განლაგებას
  `javascript/iroha_js/src/instructionBuilders.js` და
  `IrohaSwift/Sources/IrohaSwift/ConfidentialEncryptedPayload.swift` შენიშვნისთვის
  დამუშავება; SDK-ებმა უნდა მიმართონ ამ დოკუმენტს რეპოს მართვის ღილაკების გადაცემისას.

### 2.3 ლეჯერის მოვლენები და ტელემეტრია

ყველა სასიცოცხლო ციკლის მოქმედება ასხივებს `AccountEvent::Repo(...)` ჩანაწერებს, რომლებიც შეიცავს
`RepoAccountEvent::{Initiated,Settled,MarginCalled}` ტვირთამწეობა სკუპირებული
მონაწილის როლი (`crates/iroha_data_model/src/events/data/events.rs:742`). ბიძგი
ეს მოვლენები თქვენს SIEM/log-ის აგრეგატორში, რათა მიიღოთ შეცდომის აშკარა აუდიტის ჟურნალი
სამაგიდო მოქმედებებისთვის, ზღვრული ზარებისთვის და მეურვის შეტყობინებებისთვის.

### 2.4 კონფიგურაციის გავრცელება და დადასტურება

კვანძები შთანთქავს რეპო მართვის ღილაკებს `[settlement.repo]` სტროფიდან
`iroha_config` (`crates/iroha_config/src/parameters/user.rs:4071`). მკურნალობა რომ
ფრაგმენტი, როგორც მმართველობის მტკიცებულების კონტრაქტის ნაწილი - დააინსტალირეთ იგი ვერსიის კონტროლში
რეპო პაკეტის გვერდით და გახეხეთ, სანამ ცვლილებას თქვენს მეშვეობით შეასრულებთ
ავტომატიზაცია ან ConfigMap. მინიმალური პროფილი ასე გამოიყურება:

```toml
[settlement.repo]
default_haircut_bps = 1500
margin_frequency_secs = 86400
eligible_collateral = ["4fEiy2n5VMFVfi6BzDJge519zAzg", "7dk8Pj8Bqo6XUqch4K2sF8MCM1zd"]

[settlement.repo.collateral_substitution_matrix]
"4fEiy2n5VMFVfi6BzDJge519zAzg" = ["7dk8Pj8Bqo6XUqch4K2sF8MCM1zd", "6zK1LDcJ3FvkpfoZQ8kHUaW6sA7F"]
```

ოპერატიული საკონტროლო სია:

1. ჩადეთ ზემოთ მოცემული ფრაგმენტი (ან თქვენი წარმოების ვარიანტი) კონფიგურაციის რეპოში
   რომელიც კვებავს `irohad`-ს და ჩაწერს მის SHA-256-ს მართვის პაკეტში.
   მიმომხილველებს შეუძლიათ განასხვავონ ბაიტები, რომელთა გამოყენებასაც აპირებთ.
2. გადაიტანეთ ცვლილება ფლოტზე (სისტემური ერთეული, Kubernetes ConfigMap და ა.შ.)
   და გადატვირთეთ თითოეული კვანძი. გაშვებისთანავე გადაიღეთ Torii
   კონფიგურაციის სნეპშოტი წარმოშობისთვის:

   ```bash
   curl -sS "${TORII_URL}/v1/configuration" \
     -H "Authorization: Bearer ${TOKEN}" | jq .
   ```

   `ToriiClient.get_configuration()` ხელმისაწვდომია პითონის SDK-ში იმავესთვის
   მიზანი, როდესაც ავტომატიზაციას სჭირდება აკრეფილი მტკიცებულება.【python/iroha_python/src/iroha_python/client.py:5791】
3. დაამტკიცეთ, რომ გაშვების დრო ახლა ახორციელებს მოთხოვნილ კადენციას/თმის შეჭრას შეკითხვით
   `FindRepoAgreements` (ან `iroha app repo margin --agreement-id ...`) და
   ჩაშენებული `RepoGovernance` მნიშვნელობების შემოწმება. შეინახეთ JSON პასუხები
   `artifacts/finance/repo/<agreement>/agreements_after.json` ქვეშ; იმ ღირებულებებს
   მიღებულია `[settlement.repo]`-დან, ამიტომ ისინი მოქმედებენ როგორც მეორადი მოწმე, როცა
   Torii-ის `/v1/configuration` სნეპშოტი არასაკმარისია.
4. შეინახეთ ორივე არტეფაქტი — TOML ფრაგმენტი და Torii/CLI სნეპშოტი —
   მტკიცებულებათა ნაკრები მმართველობითი მოთხოვნის წარდგენამდე. აუდიტორებს უნდა შეეძლოთ
   სნიპეტის ხელახლა დაკვრა, მისი ჰეშის გადამოწმება და მისი დაკავშირება გაშვების ხედთან.

ეს სამუშაო პროცესი უზრუნველყოფს სარეპო მერხებს არასოდეს დაეყრდნოს ad-hoc გარემოს ცვლადებს
კონფიგურაციის გზა განმსაზღვრელი რჩება და მმართველობის ყველა ბილეთი ატარებს ამას
`iroha_config` მტკიცებულებების იგივე ნაკრები მოსალოდნელია საგზაო რუკა F1-ში.

### 2.5 დეტერმინისტული მტკიცებულების აღკაზმულობა

რეპოს სიცოცხლის ციკლი Norito JSON ჩარჩოში, ადარებს მას კანონიკურ მოწყობილობას:
`crates/iroha_core/tests/fixtures/repo_lifecycle_proof.json` და ჰეშირებს
პაკეტი (ფიქსირების დაიჯესტი თვალყური ადევნეთ
`crates/iroha_core/tests/fixtures/repo_lifecycle_proof.digest`). ლოკალურად გაუშვით:

```bash
cargo test -p iroha_core \
  -- --exact smartcontracts::isi::repo::tests::repo_deterministic_lifecycle_proof_matches_fixture
```ეს ტესტი ახლა მუშაობს როგორც ნაგულისხმევი `cargo test -p iroha_core` კომპლექტის ნაწილი, ამიტომ CI
ავტომატურად იცავს კადრს. როდესაც იცვლება რეპოს სემანტიკა ან მოწყობილობები,

დამხმარე იყენებს დამაგრებულ `rust-toolchain.toml` არხს, ხელახლა წერს მოწყობილობებს
`crates/iroha_core/tests/fixtures/` ქვეშ და ხელახლა აწარმოებს დეტერმინისტულ აღკაზმულობას
ასე რომ, შემოწმებული სნეპშოტი/დაიჯესტი სინქრონიზებული დარჩება გაშვების დროის ქცევასთან
აუდიტორები ხელახლა ითამაშებენ.

### 2.4 Torii API ზედაპირები

- `GET /v1/repo/agreements` აბრუნებს აქტიურ შეთანხმებებს სურვილისამებრ პაგინაცია, ფილტრაცია
  (`filter={...}`), დახარისხება და მისამართის ფორმატირების პარამეტრები. გამოიყენეთ ეს სწრაფი აუდიტისთვის ან
  დაფები, როდესაც JSON-ის დაუმუშავებელი დატვირთვები საკმარისია.
- `POST /v1/repo/agreements/query` იღებს შეკითხვის სტრუქტურირებულ კონვერტს (გვერდები, დახარისხება,
  `FilterExpr`, `fetch_size`) ასე რომ, ქვედა დინებაში სერვისებს შეუძლიათ განმსაზღვრელი გვერდის ავლა წიგნში.
- JavaScript SDK ახლა ამჟღავნებს `listRepoAgreements`, `queryRepoAgreements` და იტერატორს
  helpers so browser/Node.js tooling receives the same typed DTOs as Rust/Python.

   `--notes` ველი მმართველობის CLI-ზე) და შეაგროვეთ საჭირო დამტკიცებები

3. გადაიტანეთ ცვლილება ფლოტში: განაახლეთ `[settlement.repo]`, გადატვირთეთ თითოეული

   `ToriiClient.getConfiguration`) გამოყენებული მნიშვნელობების დამადასტურებელი თანატოლებზე.
4. ხელახლა გაუშვით `integration_tests/tests/repo.rs` plus
   `repo_deterministic_lifecycle_proof_matches_fixture` და შეინახეთ ჟურნალები შემდეგში
   კონფიგურაციის განსხვავებას, რათა აუდიტორებმა დაინახონ, რომ ახალი ნაგულისხმევი პარამეტრები შენარჩუნებულია

მატრიცის ჩანაწერის გარეშე გაშვების დრო უარყოფს ჩანაცვლებებს, რომლებიც ცვლის აქტივს

### 2.5 კონფიგურაციის მტკიცებულება და დრიფტის გამოვლენა

Norito/`iroha_config` სანტექნიკა ახლა ასახავს გადაწყვეტილ რეპო პოლიტიკას
`iroha_config::parameters::actual::Repo`, ამიტომ მმართველობის პაკეტებმა უნდა დაამტკიცონ
გამოყენებული მნიშვნელობები თანატოლებზე - არა მხოლოდ შემოთავაზებული TOML. გადაჭრის დაჭერა
კონფიგურაცია და მისი დაიჯესტი ყოველი გაშვების შემდეგ:

1. მიიღეთ კონფიგურაცია თითოეული თანატოლისგან (`GET /v1/configuration` ან
   `ToriiClient.getConfiguration`) და გამოყავით რეპო სტროფი:

   ```bash
   curl -s http://<torii-host>/v1/configuration \
     | jq -cS '.settlement.repo' \
     > artifacts/finance/repo/<agreement-id>/config/repo_config_actual.json
   ```

2. გახეხეთ კანონიკური JSON და ჩაწერეთ იგი მტკიცებულების მანიფესტში. როცა
   ფლოტი ჯანმრთელია, ჰეში უნდა ემთხვეოდეს თანატოლებს, რადგან `actual`
   აერთიანებს ნაგულისხმევს ეტაპობრივ `[settlement.repo]` სნიპეტთან:

   ```bash
   shasum -a 256 artifacts/finance/repo/<agreement-id>/config/repo_config_actual.json
   ```

3. მიამაგრეთ JSON + ჰეში მართვის პაკეტს და აკრიფეთ ჩანაწერი
   მანიფესტი ატვირთულია მმართველობის DAG-ში. თუ რომელიმე თანატოლი იტყობინება განსხვავებული
   დაიჯესტი, შეაჩერე გაშვება და შეაჯერე კონფიგურაციის/მდგომარეობის დრიფტი მანამდე
   მიმდინარეობს.

საგზაო რუქის F1 მიმომხილველებმა მოითხოვეს მოკლე ჩამონათვალი, რომლებზეც მათ შეუძლიათ მითითება ამ დროს

   ```bash
   mkdir -p artifacts/finance/repo/<slug>

   ჩარჩო მთელი კომპლექტის ხელახალი გაშვების გარეშე (იხ. §2.7). ასევე შეგიძლიათ დარეკოთ

   განაახლეთ და დააკოპირეთ იგივე ფაილები ერთი ნაბიჯით.

   `config/peers/`-ის ქვეშ, ასე რომ მართვის პაკეტი ადასტურებს, რომ გაშვება წარმატებით დასრულდა.

   აუდიტორებს შეუძლიათ განასხვავონ პაკეტი ნედლეულის პაკეტის ჩამოტვირთვის გარეშე (იხ. §3.2).

   ზუსტი მტკიცებულება, რომელიც თქვენ გაგზავნეთ დასამტკიცებლად.

ზემოაღნიშნული ნაბიჯების გაშვება რეპოს მოთხოვნის დადგმისთანავე ნიშნავს

## 5. ოპერატიული სიები

### ყოველდღიური წინასწარი გახსნა

1. ექსპორტი გაუწიეთ `iroha app repo query list`-ის მეშვეობით.
2. შეადარეთ სახაზინო ინვენტარს და უზრუნველყოთ შესაბამისი უზრუნველყოფის კონფიგურაცია
   ემთხვევა დაგეგმილ წიგნს.
3. მოაწყეთ მომავალი რეპო/განტვირთვა `--output`-ით და შეაგროვეთ ორმაგი მოწონება.

### შიდადღიური მონიტორინგი

1. გამოიწერეთ `AccountEvent::Repo` ინიციატორის/კონტრაგენტის/მეურვისთვის
   ანგარიშები; გაფრთხილება, როდესაც ხდება მოულოდნელი ინიციაციები.
2. გამოიყენეთ `iroha app repo margin --agreement-id ID` (ან
   `RepoAgreementRecord::next_margin_check_after`) საათობრივად კადენციის გამოსავლენად
   დრიფტი; გამომწვევი `repo margin-call` როდესაც `is_due = true`.
3. ჩაწერეთ ყველა ზღვრული ზარი ოპერატორის ინიციალებით და მიამაგრეთ CLI JSON გამომავალი
   მტკიცებულებების დირექტორია.

### დღის დასასრული + განსახლების შემდგომი

1. ხელახლა გაუშვით `repo query list` და დაადასტურეთ, რომ გაუქმებული შეთანხმებები წაშლილია.
2. დაარქივეთ `RepoAccountEvent::Settled` ტვირთამწეობა და გადაამოწმეთ ნაღდი ფული/გირაო
   ნაშთები `FindAssets`-ის საშუალებით.
3. შეიტანეთ საბურღი ჩანაწერი `ops/drill-log.md`-ში რეპო წვრთნების ან ინციდენტის ტესტების დროს
   გაშვება; ხელახლა გამოიყენეთ `scripts/telemetry/log_sorafs_drill.sh` კონვენციები დროის ანაბეჭდებისთვის.

## 6. თაღლითობისა და უკან დაბრუნების პროცედურები

- **Dual control:** Always generate instructions with `--output` and store the
  JSON თანა-ხელმოწერისთვის. უარყოთ ერთი მხარის წარდგენა პროცესის დონეზე
  მიუხედავად იმისა, რომ გაშვების დრო ახორციელებს ინიციატორის უფლებამოსილებას.
- **შეცდომის აშკარა აღრიცხვა:** ასახეთ `RepoAccountEvent` ნაკადი თქვენს
  SIEM, ასე რომ, ნებისმიერი ყალბი ინსტრუქცია იქნება შესამჩნევი (გამოტოვებული თანატოლების ხელმოწერები).
- **გაბრუნება:** თუ რეპო ნაადრევად უნდა დაიშალოს, გაგზავნეთ `repo unwind`
  იგივე შეთანხმებით ID და მიამაგრეთ `--notes` ველი თქვენს ინციდენტში
  ტრეკერი, რომელიც მიუთითებს GAR-ის მიერ დამტკიცებულ დაბრუნების წიგნზე.
- **თაღლითობის ესკალაცია:** თუ არაავტორიზებული რეპო გამოჩნდება, შეურაცხყოფის ექსპორტი
  `RepoAccountEvent` იტვირთება, გაყინავს ანგარიშებს მმართველობის პოლიტიკის მეშვეობით და
  აცნობოს საბჭოს რეპო მმართველობის SOP-ის შესახებ.
