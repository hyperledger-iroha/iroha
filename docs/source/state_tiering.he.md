# תכנית שכבות למצב העולם

מסמך זה מרכז את הצעדים הראשונים לפיצול World State View (WSV) לשכבות *חמה* ו-*קרה*, כדי שמאמתים יוכלו להגביל שימוש בזיכרון בלי לוותר על אימות טרנזקציות מהיר.

## מוטיבציה

- WSV בזיכרון שומר כל מפתח באופן רציף, כך שרשתות ותיקות צוברות עשרות ג׳יגה-בייטים ומכבידות על חומרת המאמתים.
- מפעילים זקוקים לגבולות משאבים צפויים תוך שמירה על נתונים נגישים ומהירים לשאילתות ולאימות.

## יישום נוכחי

- `TieredStateBackend` (ראו `crates/iroha_core/src/state/tiered.rs`) סורק את ה-WSV לאחר כל בלוק שנחתם, מדרג מפתחות לפי המוטציה האחרונה ושופך רשומות שמעבר לקיבולת החמה המוגדרת אל שכבת קירור בדיסק. המטענים הקרים נכתבים בקידוד Norito לצד מניפסט סנאפשוט (`manifest.json`) המכיל hash מפתחות, טביעות ערכים ונתיב spill יחסי.
- התצורה נמצאת תחת `iroha_config.parameters.tiered_state` (`enabled`, ‏`hot_retained_keys`, ‏`cold_store_root`, ‏`max_snapshots`). הצומת מיישם את הכפתורים בהפעלה באמצעות `State::set_tiered_backend`, והבנייה ברירת המחדל משאירה את הפיצ'ר כבוי עד שהמפעיל מפעיל אותו מפורשות.
- `StateBlock::commit` רושם סנאפשוט חדש תחת שורש הקירור המוגדר (ומגזם ספריות ישנות בהתאם ל-`max_snapshots`) תוך החזקת נעילת כתיבה על מצב העולם, כדי להבטיח מניפסטים דטרמיניסטיים בין פירים.

## צעדים הבאים

1. **העשרת מדדים:** לחשוף יחס פגיעות חם/קר, ספירת spill וזמני סנאפשוט כדי לכייל את השכבה החמה לפני הפעלה ברירת מחדל.
2. **אחסון פלגבילי:** לאפשר החלפת שכבת הקירור בין sled, ‏LMDB או הזרמת offload בהתאם לדרישות עמידות.
3. **ממשק API:** להוסיף ממשקי ניהול/טלמטריה לבדיקה בזמן אמת ולהציג את פרמטרי `tiered_state` דרך CLI/REST לשליטה דינמית.
4. **אימות:** להרחיב בדיקות אינטגרציה להרצה עם שכבה חמה מוגבלת ולהבטיח שאימות טרנזקציות מושך נתונים קרים לפי הצורך ללא סטיות.

צעדים אלו שומרים על יישום קטן ובדיקה פשוטה תוך התקדמות ל-WSV בעל גבולות זיכרון שממשיך לספק את הבטחות האימות הקיימות.
