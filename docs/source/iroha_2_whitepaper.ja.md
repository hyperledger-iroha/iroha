# Iroha v2.0

Hyperledger Iroha v2 は、決定論的でビザンチン耐性を備えた分散型台帳です。モジュール型アーキテクチャ、堅牢なデフォルト設定、そして扱いやすい API を重視して設計されています。プラットフォームは複数の Rust クレートとして提供され、個別ソリューションに組み込むことも、統合して本番用のブロックチェーンネットワークを運用することもできます。

---

## 1. 概要

Iroha 2 は Iroha 1 から続く設計思想を継承し、大量のカスタムコードを書かなくてもネットワークを立ち上げられるよう、厳選された機能群を最初から同梱します。v2 リリースでは、実行環境・コンセンサスパイプライン・データモデルを単一の統合ワークスペースに集約しました。

v2 系は、独自のパーミッションドまたはコンソーシアム型ブロックチェーンを運営したい組織向けに設計されています。各デプロイメントは独自のコンセンサスネットワークを稼働させ、独立したガバナンスを維持し、第三者に依存することなく設定・ジェネシスデータ・アップグレード周期を調整できます。共有ワークスペースにより、複数の独立したネットワークが同一のコードベース上で、自組織に適した機能とポリシーを選択しながら開発できます。

Iroha 2 と SORA Nexus（Iroha 3）は同じ Iroha Virtual Machine（IVM）を実行します。開発者は Kotodama コントラクトを 1 度作成すれば、自身が運営するネットワークにもグローバルな Nexus レジャーにも、実行環境を再コンパイルしたりフォークしたりすることなくデプロイできます。

### 1.1 Hyperledger エコシステムとの関係

Iroha のコンポーネントは、他の Hyperledger プロジェクトと相互運用できるよう設計されています。コンセンサス、データモデル、シリアライゼーションの各クレートは、複合スタックや Fabric、Sawtooth、Besu との連携で再利用できます。Norito コーデックやガバナンスマニフェストといった共通ツール群により、エコシステム全体でインターフェースの一貫性を保ちつつ、Iroha は意図的に意見のあるデフォルト実装を提供します。

### 1.2 クライアントライブラリと SDK

モバイルおよび Web での一級の体験を保証するため、プロジェクトは以下の保守済み SDK を提供しています。

- `IrohaSwift`（iOS/macOS 向けクライアント）。Metal/NEON アクセラレーションを決定論的フォールバックの背後で統合。
- `iroha_js`（JavaScript/TypeScript 向け）。Kaigi ビルダーや Norito ヘルパーを含む。
- `iroha_python`（Python 連携向け）。HTTP・WebSocket・テレメトリ対応。
- `iroha_cli`（ターミナルベースの管理・スクリプティング向け）。

すべての SDK はワークスペースが生成する Norito スキーマを利用し、言語やプラットフォームを跨いでもペイロード互換性を維持します。

### 1.3 設計原則

- **決定論を最優先:** 各ノードは同じコードパスを実行し、同じ入力に対して同じ結果を得ます。SIMD/CUDA/NEON パスはフィーチャーフラグで制御され、常に決定論的なスカラ実装にフォールバックできます。
- **コンポーザブルなモジュール:** ネットワーク、コンセンサス、実行、テレメトリ、ストレージはそれぞれ専用クレートとして分離され、利用者はスタック全体を抱え込む必要なく必要な部分を採用できます。
- **明示的な設定:** すべての挙動を `iroha_config` で制御し、環境変数による切り替えは開発時の利便性に限定します。
- **安全なデフォルト:** 正準コーデック、厳密なポインター ABI の強制、バージョン管理されたマニフェストにより、ネットワークを跨いだアップグレードを予測可能に保ちます。

## 2. プラットフォームアーキテクチャ

### 2.1 ノード構成

Iroha ノードは互いに連携する複数のサービスで構成されます。

- **Torii（`iroha_torii`）:** 取引・クエリ・イベントストリーム・テレメトリ向けの HTTP/WebSocket API（`/v1/...` エンドポイント）を公開します。
- **Core（`iroha_core`）:** 検証、コンセンサス、実行、ガバナンス、状態管理を統括します。
- **Sumeragi（`iroha_core::sumeragi`）:** ビュー変更、信頼できるブロードキャストによるデータ可用性、commit certificate を備えた NPoS 対応コンセンサスパイプラインを実装します。詳細は [Sumeragi コンセンサスガイド](./sumeragi.md) を参照してください。
- **Kura（`iroha_core::kura`）:** 正準ブロック、リカバリーサイドカー、証跡メタデータをディスクに永続化します。
- **World State View（`iroha_core::state`）:** 検証やクエリで使用する権威あるインメモリスナップショットを保持します。
- **Iroha Virtual Machine（`ivm`）:** Kotodama バイトコード（`.to`）を実行し、ポインター ABI ポリシーを強制します。
- **Norito（`crates/norito`）:** すべてのオンワイヤ型に対して決定論的なバイナリ／JSON シリアライゼーションを提供します。
- **Telemetry（`iroha_telemetry`）:** Prometheus 指標、構造化ログ、ストリーミングイベントを公開します。
- **P2P（`iroha_p2p`）:** ゴシップ、トポロジ、ピア間の安全な接続を管理します。

### 2.2 ネットワーキングとトポロジ

Iroha のピアは、確定済み状態から導出された順序付きトポロジを維持します。各コンセンサスラウンドではリーダー、検証セット、プロキシテール、オブザーバーピアが選出されます。取引はリーダーが提案にまとめる前に、Norito でエンコードされたメッセージとしてゴシップされます。信頼できるブロードキャストにより、ブロックと裏付けとなる証拠はネットワーク状況が揺らいでも正直なピア全員に届き、データ可用性が確保されます。デッドラインを超過するとビュー変更がリーダーをローテーションし、クォーラム証明により各コミットメントが網羅的に署名されていることが保証されます。

### 2.3 データモデル

Iroha のデータモデルは、アカウント・資産・ドメインといった一次オブジェクトと、それらに作用する命令（Instruction Set Interface: ISI）で構成されます。すべての構造体は Norito コーデックでシリアライズされ、トランザクションやブロック、メタデータはバージョンを明示した正準レイアウトで運搬されます。動的なトリガー、マニフェスト、ガバナンス提案も同じスキーマに従い、ポインター ABI ポリシーを破る入力は受理にも実行にも到達できません。

### 2.4 トリガーとスケジューラ

トリガーはブロック高・時刻・イベント条件でスケジュールされ、Core が監視するタイムホイールに登録されます。該当ラウンドで期限を迎えたトリガーは内部トランザクションに変換され、通常の取引と同じ検証・実行パイプラインを経ます。再入防止や冪等性はトリガーハンドラの責務として文書化されています。

## 3. コンセンサスとネットワークパイプライン

### 3.1 Sumeragi パイプラインの段階

Sumeragi は提案（propose）、検証（validate）、コミット（commit）の 3 段階で構成されます。リーダー（提案者）は mempool から取引を選択し、Norito でエンコードされた `BlockCreated` メッセージとして提案を生成します。検証セットのピアは同じ検証手順を再実行し、成功した場合は `BlockSigned` を発行します。プロキシテールは十分な署名を収集して commit certificate を組み立て、`BlockCommitted` をブロードキャストしてブロックを確定させます。観測者ピアはトポロジと状態を同期させ、証跡やクエリを提供します。

### 3.2 ビュー変更とフェイルオーバー

リーダーがデッドライン内に提案できない場合、ビュー変更が発動し、トポロジが次のリーダーにローテーションします。信頼できるブロードキャストにより、欠落したブロックやサイドカーは正直なピアから再送されます。フェイルオーバーに伴うイベントはテレメトリ経由で公開され、運用者がボトルネックや構成ミスを素早く把握できます。

### 3.3 トポロジの管理

トポロジはジェネシスおよびガバナンス命令で明示的に管理されます。`AddPeer` や `RemovePeer` 命令は Norito マニフェストで署名され、HL ネットワーク全体にレプリケートされます。NPoS モードでは、VRF を用いた選挙ウィンドウとステークに基づく委員会選定が追加されますが、基本的な構成ノブは決定論的に保持されます。

## 4. トランザクションライフサイクル

1. **受理:** Torii が Norito ペイロードをデコードし、署名・TTL・サイズ制限を確認したうえで、取引をローカルにキューイングします。
2. **ゴシップ:** 取引はトポロジ全体に伝播し、各ピアはハッシュで重複排除しつつ同じ受理チェックを繰り返します。
3. **選定:** 現在のリーダーが保留集合から取引を取得し、ステートレス検証を実行します。
4. **状態シミュレーション:** 候補取引は一時的な `StateBlock` 内で実行され、IVM または組み込み命令を呼び出します。競合や規則違反は決定論的に破棄されます。
5. **トリガーの具現化:** ラウンド内で期限を迎えたスケジュールトリガーは内部取引として変換され、同じパイプラインで検証されます。
6. **提案の封止:** ブロック上限に達するかタイムアウトに到達すると、リーダーは Norito でエンコードされた `BlockCreated` メッセージを発行します。
7. **検証:** 検証セットのピアがステートレス／ステートフルチェックを再実行します。成功したピアは `BlockSigned` を署名してプロキシテールへ転送します。
8. **コミット:** プロキシテールは正準な署名集合を集めて commit certificate を組み立て、`BlockCommitted` をブロードキャストしローカルでもブロックを確定させます。
9. **適用:** すべてのピアがブロックを Kura に記録し、状態更新を適用し、テレメトリ／イベントを発行し、確定済み取引を mempool から削除し、トポロジロールを更新します。

リカバリ経路では、決定論的ブロードキャストにより欠落ブロックを再送し、ビュー変更でリーダーをローテーションします。サイドカーとテレメトリは、コンセンサス結果を変えずに診断情報を提供します。

## 5. スマートコントラクトと実行

スマートコントラクトは Iroha Virtual Machine（IVM）上で実行されます。

- **Kotodama:** 高レベルの `.ko` ソースを決定論的な `.to` バイトコードへコンパイルします。
- **ポインター ABI の強制:** コントラクトがホストメモリとやり取りする際は検証済みポインター型を通す必要があります。システムコールの詳細は `ivm/docs/syscalls.md` に記載され、ABI の一覧はハッシュ化されバージョン管理されています。
- **システムコールとホスト:** 台帳状態アクセス、トリガースケジューリング、秘匿プリミティブ、Kaigi メディアフロー、決定論的ランダムネスなどを提供します。
- **組み込みエグゼキュータ:** 資産・アカウント・許可・ガバナンス操作向けの Iroha 特殊命令（ISI）を引き続きサポートします。Norito スキーマを順守する限り、カスタムエグゼキュータで命令セットを拡張できます。
- **秘匿機能:** 秘匿送金や検証者レジストリなどはエグゼキュータ命令経由で公開され、ホストが Poseidon コミットメントで検証します。

## 6. ストレージと永続化

- **Kura ブロックストア:** 各確定ブロックを Norito ヘッダー付きの `SignedBlockWire` ペイロードとして書き込み、正準ヘッダー・トランザクション・commit certificate・証跡データを一体として保存します。
- **World State View:** 権威ある状態をメモリ上に保持し、高速なクエリを可能にします。決定論的スナップショットとパイプラインサイドカー（`pipeline/block_<height>.json`）は復旧や監査を支援します。
- **状態階層化:** 大規模デプロイメント向けにホット／コールド分割を可能にしつつ、決定論的検証を維持します。
- **同期と再生:** 確定ブロックを同じ検証規則で再適用し、状態を復元します。決定論的ブロードキャストにより、信頼済みストレージに依存せずピアから欠落データを取得できます。

## 7. ガバナンスと経済

- オンチェーンパラメータ（`SetParameter`）でコンセンサスタイマー、mempool 制限、テレメトリ設定、手数料帯、フィーチャーフラグを制御します。`kagami` が生成するジェネシスマニフェストは初期設定を導入します。
- **Kaigi:** コラボレーションセッションの作成／参加／離脱／終了を管理し、Norito ストリーミングテレメトリを会議ユースケース向けに提供します。
- **Hijiri:** 決定論的なピア／アカウント評価を提供し、コンセンサス、受理ポリシー、手数料倍率（Q16 固定小数点）と統合します。証跡マニフェスト、チェックポイント、評価レジストリはオンチェーンでコミットされ、オブザーバープロファイルが証跡の真正性を管理します。
- **NPoS モード:** 有効化時は VRF に基づく選挙ウィンドウとステーク加重委員会を提供しますが、決定論的な設定デフォルトは維持されます。
- **秘匿レジストリ:** ゼロ知識検証者キー、証明ライフサイクル、秘匿フロー向けコミットメントを管理します。

## 8. クライアント体験とツール群

- **Torii API:** 取引、クエリ、イベントストリーム、テレメトリ、ガバナンスエンドポイントに対する REST / WebSocket インターフェースを提供します。JSON の射影は Norito スキーマから導出されます。
- **CLI ツール:** `iroha_cli` や `iroha_monitor` が管理、ライブピアダッシュボード、パイプライン検査をカバーします。
- **ジェネシスツール:** `kagami` が Norito エンコードされたマニフェスト、バリデータ鍵、設定テンプレートを生成します。
- **SDK:** Swift、JS/TS、Python の各 SDK が命令・クエリ・トリガー・テレメトリへの自然なアクセスを提供します。
- **スクリプトと CI フック:** `scripts/` 以下にダッシュボード検証、コーデック再生成、スモークテストを自動化するスクリプトが揃っています。

## 9. パフォーマンス・レジリエンス・ロードマップ

- 現行パイプラインは、整ったネットワーク条件下で **20,000 tps** / **2〜3 秒** ブロックタイムを目標とし、バッチ署名検証と決定論的スケジューリングで裏付けます。
- **テレメトリ:** コンセンサスタイマー、mempool 占有率、ブロック伝播ヘルス、Kaigi 利用状況、Hijiri 評価更新を Prometheus メトリクスで公開します。
- **レジリエンス:** 決定論的データ可用性、リカバリーサイドカー、トポロジローテーション、設定可能なビュー変更閾値を備えます。
- 今後のロードマップ（`roadmap.md` を参照）は、決定論的出力を維持しながら Nexus データスペース、秘匿ツール群の拡張、より広範なハードウェアアクセラレーションの強化を進めます。

## 10. 運用とデプロイ

- **アーティファクト:** Dockerfile、Nix flake、`cargo` ワークフローにより再現可能なビルドを提供します。`kagami` はパーミッションドおよび NPoS デプロイメント向けにジェネシスマニフェスト、バリデータ鍵、サンプル設定を出力します。
- **セルフホスト型ネットワーク:** オペレーターは自組織のピア集合、受理ルール、アップグレード周期を管理します。ワークスペースは多数の独立した Iroha 2 ネットワークが協調なしで共存し、共有するのは上流コードだけです。
- **設定ライフサイクル:** `iroha_config` は user → actual → defaults のレイヤーで解決し、すべてのノブを明示的かつバージョン管理された状態に保ちます。ランタイム変更は `SetParameter` 命令を通じて適用されます。
- **オブザーバビリティ:** `iroha_telemetry` は Prometheus メトリクス、構造化ログ、ダッシュボードデータを公開し、CI スクリプト（`ci/check_swift_dashboards.sh`, `scripts/render_swift_dashboards.sh`, `scripts/check_swift_dashboard_data.py`）で監視します。ストリーミング、コンセンサス、Hijiri の各イベントは WebSocket 経由で提供され、`scripts/sumeragi_backpressure_log_scraper.py` はペースメーカーのバックプレッシャーとテレメトリを相関させてトラブルシューティングを支援します。
- **テスト:** `cargo test --workspace`、`integration_tests/` の統合テスト、言語 SDK のスイート、Norito のゴールデンフィクスチャが決定論を守ります。ポインター ABI、システムコール一覧、ガバナンスマニフェストには専用のゴールデンテストがあります。
- **リカバリ:** Kura サイドカー、決定論的リプレイ、ブロードキャスト同期により、ノードはディスクやピアから状態を復旧できます。Hijiri チェックポイントとガバナンスマニフェストはコンプライアンス向けの監査可能なスナップショットを提供します。
