# လျှို့ဝှက်ပိုင်ဆိုင်မှုများနှင့် ZK လွှဲပြောင်းခြင်းဒီဇိုင်း

## စေ့ဆော်မှု
- ပွင့်လင်းမြင်သာသောလည်ပတ်စီးဆင်းမှုကိုမပြောင်းလဲဘဲ domains များသည် ငွေပေးငွေယူကိုယ်ရေးကိုယ်တာအား ထိန်းသိမ်းထားနိုင်သောကြောင့် ရွေးချယ်မှုအကာအရံရှိသော ပိုင်ဆိုင်မှုစီးဆင်းမှုများကို ပေးပို့ပါ။
- စာရင်းစစ်များနှင့် အော်ပရေတာများအား ဆားကစ်များနှင့် cryptographic ကန့်သတ်ချက်များအတွက် lifecycle ထိန်းချုပ်မှုများ (အသက်သွင်းခြင်း၊ လည်ပတ်ခြင်း၊ ရုတ်သိမ်းခြင်း) ကို ပေးပါ။

## ခြိမ်းခြောက်မှုပုံစံ
- စိစစ်သူများသည် ရိုးသားသော်လည်း စူးစမ်းလိုသည်- ၎င်းတို့သည် အများဆန္ဒကို သစ္စာရှိရှိ လုပ်ဆောင်သော်လည်း လယ်ဂျာ/ပြည်နယ်ကို စစ်ဆေးရန် ကြိုးစားကြသည်။
- ကွန်ရက်လေ့လာသူများသည် ဒေတာပိတ်ဆို့ခြင်းနှင့် အတင်းအဖျင်းအပေးအယူများကို ကြည့်ရှုခြင်း၊ ပုဂ္ဂလိက အတင်းအဖျင်း ချန်နယ်များဟု ယူဆခြင်း မရှိပါ။
- နယ်ပယ်ပြင်ပ- လယ်ဂျာလမ်းကြောင်းခွဲခြမ်းစိတ်ဖြာမှု၊ ကွမ်တမ်ရန်ဘက်များ (PQ လမ်းပြမြေပုံအောက်တွင် သီးခြားခြေရာခံထားသည်)၊ လယ်ဂျာရရှိနိုင်မှုတိုက်ခိုက်မှုများ။

## ဒီဇိုင်းအကျဉ်းချုပ်
- ပိုင်ဆိုင်မှုများသည် ရှိပြီးသား ပွင့်လင်းမြင်သာသော လက်ကျန်များအပြင် *အကာအရံရှိသော ရေကူးကန်* ကို ကြေညာနိုင်သည်။ အကာအကွယ်ပေးထားသော လည်ပတ်မှုကို ကုဒ်ဝှက်ထားသော ကတိကဝတ်များမှတစ်ဆင့် ကိုယ်စားပြုသည်။
- မှတ်စုများတွင် `(asset_id, amount, recipient_view_key, blinding, rho)` ဖြင့်-
  - ကတိကဝတ်- `Comm = Pedersen(params_id || asset_id || amount || recipient_view_key || blinding)`။
  - Nullifier- `Null = Poseidon(domain_sep || nk || rho || asset_id || chain_id)`၊ မှတ်စုမှာယူခြင်းမရှိဘဲ။
  - ကုဒ်ဝှက်ထားသော payload- `enc_payload = AEAD_XChaCha20Poly1305(ephemeral_shared_key, note_plaintext)`။
- ငွေပေးငွေယူ သယ်ယူပို့ဆောင်ရေး Norito-ကုဒ်လုပ်ထားသော `ConfidentialTransfer` ပါ၀င်သော payloads
  - အများသူငှာ သွင်းအားစုများ- Merkle ကျောက်ဆူး၊ ပျက်ပြယ်စေသော အရာများ၊ ကတိကဝတ်အသစ်များ၊ ပိုင်ဆိုင်မှု ID၊ circuit ဗားရှင်း။
  - လက်ခံသူများနှင့် ရွေးချယ်နိုင်သော စာရင်းစစ်များအတွက် ကုဒ်ဝှက်ထားသော payloads။
  - တန်ဖိုးထိန်းသိမ်းမှု၊ ပိုင်ဆိုင်မှုနှင့် ခွင့်ပြုချက်တို့ကို သက်သေအထောက်အထား သုည-အသိပညာ။
- သော့များနှင့် ပါရာမီတာအစုံများကို အကောင့်ဖွင့်ခြင်းပြတင်းပေါက်များဖြင့် ထိန်းချုပ်ထားသည်။ node များသည် အမည်မသိ ရည်ညွှန်းသော သို့မဟုတ် ရုပ်သိမ်းထားသော အထောက်အထားများကို တရားဝင်အတည်ပြုရန် ငြင်းဆိုထားသည်။
- အများသဘောတူသော ခေါင်းစီးများသည် တက်ကြွသောလျှို့ဝှက်အင်္ဂါရပ်ကို ချေဖျက်ရန် ကတိပြုထားသောကြောင့် မှတ်ပုံတင်ခြင်းနှင့် ကန့်သတ်ချက်အခြေအနေနှင့် ကိုက်ညီသည့်အခါမှသာ ဘလောက်များကို လက်ခံပါသည်။
- အထောက်အထားတည်ဆောက်မှုတွင် ယုံကြည်စိတ်ချရသော စနစ်ထည့်သွင်းခြင်းမရှိဘဲ Halo2 (Plonkish) စတန်းကို အသုံးပြုသည်။ Groth16 သို့မဟုတ် အခြား SNARK မျိုးကွဲများကို v1 တွင် ရည်ရွယ်ချက်ရှိရှိ ပံ့ပိုးမထားပါ။

### Deterministic Fixtures

လျှို့ဝှက်မှတ်စုစာအိတ်များကို ယခု `fixtures/confidential/encrypted_payload_v1.json` တွင် Canonical fixture ဖြင့် ပို့ပေးပါသည်။ ဒေတာအတွဲသည် အပြုသဘောဆောင်သော v1 စာအိတ်တစ်ခုနှင့် အနုတ်ပုံသဏ္ဍာန်မမှန်သောနမူနာများကို ဖမ်းယူထားသောကြောင့် SDK များသည် ခွဲခြမ်းစိတ်ဖြာမှုကွာဟမှုကို အခိုင်အမာအတည်ပြုနိုင်သည်။ Rust data-model tests (`crates/iroha_data_model/tests/confidential_encrypted_payload_vectors.rs`) နှင့် Swift suite (`IrohaSwift/Tests/IrohaSwiftTests/ConfidentialEncryptedPayloadTests.swift`) တို့သည် Norito encoding၊ error surfaces နှင့် regression coverage တို့သည် fixture ကို တိုက်ရိုက်တင်ပေးပြီး Norito encoding၊ error surfaces နှင့် regression coverage တို့သည် ကုဒ်ဒက်များပြောင်းလဲလာသည်နှင့်အမျှ ဆက်နေမည်ဟုအာမခံပါသည်။

Swift SDK များသည် စိတ်ကြိုက် JSON ကော်မပါဘဲ အကာအရံညွှန်ကြားချက်များကို ထုတ်လွှတ်နိုင်ပါပြီ- a ကို တည်ဆောက်ပါ။
`ShieldRequest` 32-byte မှတ်စုကတိကဝတ်၊ ကုဒ်ဝှက်ထားသော payload နှင့် debit မက်တာဒေတာ၊
ထို့နောက် `IrohaSDK.submit(shield:keypair:)` (သို့မဟုတ် `submitAndWait`) သို့ ခေါ်ဆို၍ လက်မှတ်ရေးထိုးပြီး ထပ်ဆင့်ပေးပို့ပါ
`/v1/pipeline/transactions` ကျော် အရောင်းအဝယ်။ အကူအညီပေးသူက ကတိကဝတ်အရှည်များကို သက်သေပြသည်၊
`ConfidentialEncryptedPayload` ကို Norito ကုဒ်ဒါသို့ တွဲပေးကာ `zk::Shield` ကို အလင်းပြန်သည်
အောက်တွင်ဖော်ပြထားသော အပြင်အဆင်သည် ပိုက်ဆံအိတ်များကို Rust ဖြင့် လော့ခ်ချသည့်အဆင့်တွင် ရှိနေစေပါသည်။## အများသဘောတူ ကတိကဝတ်များ နှင့် စွမ်းဆောင်ရည် သတ်မှတ်ခြင်း
- ပိတ်ဆို့ခေါင်းစီးများကို `conf_features = { vk_set_hash, poseidon_params_id, pedersen_params_id, conf_rules_version }` ဖော်ထုတ်ပါ။ digest သည် အများသဘောတူထားသော hash တွင်ပါဝင်ပြီး block လက်ခံမှုအတွက် local registry view နှင့် တူညီရပါမည်။
- အုပ်ချုပ်ရေးသည် အနာဂတ် `activation_height` ဖြင့် `next_conf_features` ပရိုဂရမ်ဖြင့် အဆင့်မြှင့်တင်နိုင်သည် ။ ထိုအမြင့်အထိ၊ လုပ်ကွက်ထုတ်လုပ်သူများသည် ယခင်အနှစ်ချုပ်ကို ဆက်လက်ထုတ်လွှတ်ရမည်ဖြစ်သည်။
- Validator nodes များသည် `confidential.enabled = true` နှင့် `assume_valid = false` ဖြင့် လုပ်ဆောင်ရပါမည်။ အခြေအနေတစ်ခုခုပျက်သွားပါက သို့မဟုတ် ဒေသဆိုင်ရာ `conf_features` ကွဲပြားသွားပါက စတင်စစ်ဆေးခြင်းများသည် တရားဝင်သတ်မှတ်ခြင်းတွင် ပါဝင်ရန် ငြင်းဆိုထားသည်။
- ယခု P2P လက်ဆွဲခြင်း မက်တာဒေတာတွင် `{ enabled, assume_valid, conf_features }` ပါဝင်ပါသည်။ ရွယ်တူများ ကြော်ငြာခြင်းကို ပံ့ပိုးမထားသော အင်္ဂါရပ်များကို `HandshakeConfidentialMismatch` ဖြင့် ပယ်ချပြီး အများဆန္ဒအလှည့်ကို ဘယ်တော့မှ မထည့်ပါ။
- တရားဝင်မဟုတ်သော အကဲခတ်သူများသည် `assume_valid = true` ကို သတ်မှတ်နိုင်သည်။ ၎င်းတို့သည် လျှို့ဝှက်မြစ်ဝကျွန်းပေါ်ဒေသများကို မျက်စိစုံမှိတ် အသုံးချသော်လည်း အများသဘောတူသော လုံခြုံမှုကို မလွှမ်းမိုးပါ။## ပိုင်ဆိုင်မှုမူဝါဒများ
- ပိုင်ဆိုင်မှု အဓိပ္ပါယ်ဖွင့်ဆိုချက်တစ်ခုစီတွင် ဖန်တီးသူမှ သတ်မှတ်ထားသော `AssetConfidentialPolicy` ပါရှိသည်-
  - `TransparentOnly`- မူရင်းမုဒ်။ ပွင့်လင်းမြင်သာသော ညွှန်ကြားချက်များ (`MintAsset`၊ `TransferAsset` စသည်ဖြင့်) ကိုသာ ခွင့်ပြုထားပြီး အကာအရံလုပ်ဆောင်မှုများကို ပယ်ချပါသည်။
  - `ShieldedOnly`- ထုတ်ပေးခြင်းနှင့် လွှဲပြောင်းခြင်းများအားလုံးသည် လျှို့ဝှက်ညွှန်ကြားချက်များကို အသုံးပြုရပါမည်။ `RevealConfidential` ကို တားမြစ်ထားသောကြောင့် လက်ကျန်များကို လူသိရှင်ကြား ပေါ်မလာတော့ပါ။
  - `Convertible`- ကိုင်ဆောင်သူများသည် အောက်ဖော်ပြပါ အဖွင့်/အပိတ် လမ်းညွှန်ချက်များကို အသုံးပြု၍ ပွင့်လင်းမြင်သာမှုနှင့် အကာအရံရှိသော ကိုယ်စားပြုမှုများအကြား တန်ဖိုးကို ရွှေ့နိုင်သည်။
- ဘဏ္ဍာငွေများ သောင်တင်နေခြင်းကို ကာကွယ်ရန် မူဝါဒများသည် ကန့်သတ်ထားသော FSM ကို လိုက်နာသည်-
  - `TransparentOnly → Convertible` (အကာအရံရှိသော ရေကန်ကို ချက်ချင်းဖွင့်ခြင်း)။
  - `TransparentOnly → ShieldedOnly` (ဆိုင်းငံ့ထားသောအကူးအပြောင်းနှင့် ပြောင်းလဲခြင်းဝင်းဒိုး လိုအပ်သည်)။
  - `Convertible → ShieldedOnly` (အနည်းဆုံး နှောင့်နှေးမှုကို ကန့်သတ်ထားသည်)။
  - `ShieldedOnly → Convertible` (ရွှေ့ပြောင်းနေထိုင်မှုအစီအစဉ် လိုအပ်သောကြောင့် အကာအရံမှတ်စုများ ဆက်လက်သုံးစွဲရန်)။
  - `ShieldedOnly → TransparentOnly` သည် အကာအရံမရှိသော ရေကန်သည် ဗလာဖြစ်နေသည် သို့မဟုတ် အုပ်ချုပ်ရေးမှ ထူးခြားသောမှတ်စုများကို အကာအရံမရှိသော ရွှေ့ပြောင်းခြင်းအား ကုဒ်နံပါတ်ဝှက်ထားခြင်းကို ခွင့်မပြုပါ။
- အုပ်ချုပ်မှုလမ်းညွှန်ချက်များသည် `pending_transition { new_mode, effective_height, previous_mode, transition_id, conversion_window }` ကို `ScheduleConfidentialPolicyTransition` ISI မှတစ်ဆင့် သတ်မှတ်ပြီး `CancelConfidentialPolicyTransition` ဖြင့် စီစဉ်ထားသော ပြောင်းလဲမှုများကို ပယ်ဖျက်နိုင်သည်။ Mempool validation သည် အကူးအပြောင်းအမြင့်ကို ကျော်တက်သွားခြင်းမရှိကြောင်း သေချာစေပြီး မူဝါဒစစ်ဆေးမှုတစ်ခုသည် အလယ်အလတ်တုံးကို ပြောင်းလဲမည်ဆိုပါက ပါဝင်မှုမှာ အဆုံးအဖြတ်မအောင်မြင်ပါ။
- ပိတ်ဆို့ခြင်းအသစ်တစ်ခုဖွင့်သောအခါ ဆိုင်းငံ့ထားသောအကူးအပြောင်းများကို အလိုအလျောက်အသုံးပြုသည်- ဘလောက်အမြင့်သည် ပြောင်းလဲခြင်းဝင်းဒိုး (`ShieldedOnly` အဆင့်မြှင့်တင်မှုများအတွက်) သို့မဟုတ် ပရိုဂရမ်ပြုလုပ်ထားသော `effective_height` သို့ရောက်သည်နှင့်၊ runtime အပ်ဒိတ်များ `AssetConfidentialPolicy` သည် `ShieldedOnly` ကို ပြန်လည်စတင်ပြီး metadata၊ ထည့်သွင်းမှုနှင့် ရှင်းရှင်းလင်းလင်းဖြစ်သည်။ `ShieldedOnly` အသွင်ကူးပြောင်းမှု ရင့်လာသောအခါတွင် ဖောက်ထွင်းမြင်ရသော ထောက်ပံ့မှုသည် ကျန်ရှိနေပါက၊ runtime သည် အပြောင်းအလဲကို ပျက်ပြယ်စေပြီး ယခင်မုဒ်ကို နဂိုအတိုင်းထားကာ သတိပေးချက်တစ်ခုကို မှတ်တမ်းတင်ထားသည်။
- Config knobs `policy_transition_delay_blocks` နှင့် `policy_transition_window_blocks` သည် ပိုက်ဆံအိတ်များကို switch ပတ်ပတ်လည်တွင် မှတ်စုများအဖြစ်ပြောင်းခွင့်ပြုရန် အနိမ့်ဆုံးသတိပေးချက်နှင့် ကျေးဇူးတော်ကာလများကို ပြဌာန်းထားသည်။
- `pending_transition.transition_id` သည် စာရင်းစစ်လက်ကိုင်အဖြစ် နှစ်ဆတိုးသည်။ အော်ပရေတာများသည် အဖွင့်/အပိတ်အစီရင်ခံချက်များကို ဆက်စပ်ဆောင်ရွက်နိုင်စေရန် အကူးအပြောင်းများကို အပြီးသတ်သည့်အခါ သို့မဟုတ် ပယ်ဖျက်သည့်အခါ အုပ်ချုပ်မှုစနစ်သည် ၎င်းကိုကိုးကားရပါမည်။
- `policy_transition_window_blocks` သည် 720 (≈12 နာရီ 60 s ပိတ်ဆို့ချိန်) သို့ ပုံသေဖြစ်သည်။ တိုတောင်းသောသတိပေးချက်ကို ကြိုးပမ်းသော Nodes များက ကုပ်ကုပ်အုပ်ချုပ်မှုတောင်းဆိုမှုများ။
- ကမ္ဘာဦးကျမ်းတွင် ဖော်ပြပြီး CLI သည် လက်ရှိနှင့် ဆိုင်းငံ့ထားသောမူဝါဒများကို ပေါ်လွင်စေသည်။ လျှို့ဝှက်ညွှန်ကြားချက်တစ်ခုစီကို ခွင့်ပြုကြောင်း အတည်ပြုရန် ဝင်ခွင့်ဆိုင်ရာ လော့ဂျစ်သည် အကောင်အထည်ဖော်ချိန်၌ မူဝါဒကို ဖတ်သည်။
- Milestone M0 လမ်းကြောင်းအတိုင်း အဆင့်မြှင့်ထားသော အဆင့်မြှင့်တင်မှု အစီအစဉ်အတွက် အောက်ဖော်ပြပါ “ရွှေ့ပြောင်းခြင်းဆိုင်ရာ စီစစ်ခြင်း” ကို ကြည့်ပါ။

#### Torii မှတစ်ဆင့် အကူးအပြောင်းများကို စောင့်ကြည့်ခြင်း။ပိုက်ဆံအိတ်များနှင့် စာရင်းစစ်များသည် `GET /v1/confidential/assets/{definition_id}/transitions` ကို စစ်ဆေးရန် စစ်တမ်းကောက်ယူသည်။
အသက်ဝင်သော `AssetConfidentialPolicy`။ JSON payload တွင် canonical အမြဲပါဝင်သည်။
ပိုင်ဆိုင်မှု id၊ နောက်ဆုံးတွေ့ရှိထားသော ဘလောက်အမြင့်၊ မူဝါဒ၏ `current_mode`၊ ဖြစ်သည့် မုဒ်
ထိုအမြင့်တွင် ထိရောက်မှု (ပြောင်းလဲခြင်း windows ယာယီ `Convertible`) နှင့်
မျှော်လင့်ထားသော `vk_set_hash`/Poseidon/Pedersen ပါရာမီတာ ခွဲခြားသတ်မှတ်မှုများ။ Swift SDK သုံးစွဲသူများ ခေါ်ဆိုနိုင်ပါသည်။
`ToriiClient.getConfidentialAssetPolicy` မပါဘဲ ရိုက်ထည့်ထားသည့် DTO များကဲ့သို့ တူညီသောဒေတာကို လက်ခံရရှိရန်
လက်ရေးကုဒ်ဖြင့်ရေးသည်။ အုပ်ချုပ်မှုအကူးအပြောင်းကို ဆိုင်းငံ့ထားသည့်အခါ တုံ့ပြန်ချက်သည်လည်း အကျုံးဝင်သည်-

- `transition_id` — `ScheduleConfidentialPolicyTransition` မှ ပြန်ပေးထားသော စာရင်းစစ်လက်ကိုင်။
- `previous_mode`/`new_mode`။
- `effective_height`။
- `conversion_window` နှင့် ဆင်းသက်လာသော `window_open_height` (ပိုက်ဆံအိတ်ရှိရမည့် ပိတ်ဆို့ခြင်း
  ShieldedOnly ဖြတ်တောက်ခြင်းများအတွက် စတင်ပြောင်းလဲခြင်း)။

ဥပမာ တုံ့ပြန်ချက်-

```json
{
  "asset_id": "62Fk4FPcMuLvW5QjDGNF2a4jAmjM",
  "block_height": 4217,
  "current_mode": "Convertible",
  "effective_mode": "Convertible",
  "vk_set_hash": "8D7A4B0A95AB1C33F04944F5D332F9A829CEB10FB0D0797E2D25AEFBAAF1155D",
  "poseidon_params_id": 7,
  "pedersen_params_id": 11,
  "pending_transition": {
    "transition_id": "BF2C6F9A4E9DF389B6F7E5E6B5487B39AE00D2A4B7C0FBF2C9FEF6D0A961C8ED",
    "previous_mode": "Convertible",
    "new_mode": "ShieldedOnly",
    "effective_height": 5000,
    "conversion_window": 720,
    "window_open_height": 4280
  }
}
```

`404` တုံ့ပြန်မှုသည် ကိုက်ညီသော ပိုင်ဆိုင်မှု အဓိပ္ပါယ်ဖွင့်ဆိုချက် မရှိသည်ကို ညွှန်ပြပါသည်။ အကူးအပြောင်းမရှိတဲ့အခါ
စီစဉ်ထားသော `pending_transition` အကွက်သည် `null` ဖြစ်သည်။

### ရွှေ့ပြောင်းခြင်း အစီအစဉ်

2. **အကူးအပြောင်းကို အဆင့်သတ်မှတ်ပါ-** `policy_transition_delay_blocks` ကို လေးစားသော `effective_height` ဖြင့် `effective_height` ကို တင်ပြပါ။ `ShieldedOnly` သို့ပြောင်းသောအခါ၊ ပြောင်းလဲခြင်းဝင်းဒိုး (`window ≥ policy_transition_window_blocks`) ကို သတ်မှတ်ပါ။
3. **အော်ပရေတာလမ်းညွှန်ချက်ကို ထုတ်ဝေပါ-** ပြန်ပေးထားသော `transition_id` ကို မှတ်တမ်းတင်ပြီး အဖွင့်/အပိတ် ပြေးစာအုပ်ကို ဖြန့်ဝေပါ။ Wallets များနှင့် စာရင်းစစ်များသည် `/v1/confidential/assets/{id}/transitions` တွင် စာရင်းသွင်းပြီး ဖွင့်ထားသော window အမြင့်ကို လေ့လာပါ။
4. **Window enforcement-** ဝင်းဒိုးဖွင့်သောအခါ၊ runtime သည် မူဝါဒကို `Convertible` သို့ပြောင်းပြီး၊ `PolicyTransitionWindowOpened { transition_id }` ကို ထုတ်လွှတ်ကာ ကွဲလွဲနေသော အုပ်ချုပ်မှုတောင်းဆိုချက်များကို စတင်ပယ်ချပါသည်။
5. ** အပြီးသတ်ခြင်း သို့မဟုတ် ဖျက်ပစ်ခြင်း-** `effective_height` တွင်၊ runtime သည် အကူးအပြောင်းအတွက် လိုအပ်ချက်များကို စစ်ဆေးပေးသည် (ဖောက်ထွင်းမြင်ရသည့် ထောက်ပံ့မှု သုည၊ အရေးပေါ်ငွေထုတ်ခြင်း မရှိခြင်း စသည်)။ အောင်မြင်မှု မူဝါဒကို တောင်းဆိုထားသည့် မုဒ်သို့ ပြောင်းသည်။ ချို့ယွင်းချက်သည် `PolicyTransitionPrerequisiteFailed` ကို ထုတ်လွှတ်သည်၊ ဆိုင်းငံ့ထားသော အကူးအပြောင်းကို ရှင်းလင်းစေပြီး မူဝါဒကို မပြောင်းလဲစေပါ။
6. **Schema အဆင့်မြှင့်တင်မှုများ-** အောင်မြင်သောအကူးအပြောင်းတစ်ခုပြီးနောက်၊ အုပ်ချုပ်မှုစနစ်သည် ပိုင်ဆိုင်မှုအစီအစဉ်ဗားရှင်း (ဥပမာ၊ `asset_definition.v2`) ကို အဖုအထစ်ဖြစ်စေပြီး CLI ကိရိယာသည် စီးရီးလိုက်ဖော်ပြသည့်အခါ `confidential_policy` လိုအပ်သည်။ Genesis မှ အဆင့်မြှင့်ထားသော docs သည် validators ကို ပြန်လည်စတင်ခြင်းမပြုမီ မူဝါဒဆက်တင်များနှင့် မှတ်ပုံတင်လက်ဗွေများကို ထည့်သွင်းရန် အော်ပရေတာများအား ညွှန်ကြားထားသည်။

လျှို့ဝှက်ချက်ဖြင့် စတင်သည့် ကွန်ရက်အသစ်များသည် လိုချင်သောမူဝါဒကို ဥပါဒ်တွင် တိုက်ရိုက်ကုဒ်လုပ်ထားသည်။ လွှင့်တင်ပြီးနောက် မုဒ်များကို ပြောင်းလဲသည့်အခါတွင် ၎င်းတို့သည် အထက်ဖော်ပြပါ စစ်ဆေးစာရင်းကို လိုက်နာဆဲဖြစ်သောကြောင့် ပြောင်းလဲခြင်းဝင်းဒိုးများသည် အဆုံးအဖြတ်အတိုင်း ဆက်လက်တည်ရှိပြီး ပိုက်ဆံအိတ်များကို ချိန်ညှိရန် အချိန်ရှိသည်။

### Norito ပေါ်လွင်သောဗားရှင်းပြုလုပ်ခြင်းနှင့် အသက်သွင်းခြင်း။- ကမ္ဘာဦးကျမ်းများတွင် စိတ်ကြိုက် `confidential_registry_root` သော့အတွက် `SetParameter` ပါဝင်ရပါမည်။ payload သည် Norito JSON သည် `ConfidentialRegistryMeta { vk_set_hash: Option<String> }` နှင့် ကိုက်ညီသည်- 18NI000001370 I10 နှင့်ညီမျှသော 32-byte hex string (`0x…`) I180 နှင့်ညီမျှသော `0x…`) ကို ထုတ်ပေးသောအခါ (`0x…`) ကို ဖယ်ပါ မန်နီးဖက်စ်တွင် စိစစ်ရေးမှူးညွှန်ကြားချက်များ ပေးပို့ထားသည်။ ကန့်သတ်ချက်များ ပျောက်ဆုံးနေပါက သို့မဟုတ် hash များသည် ကုဒ်လုပ်ထားသော registry တွင် ရေးသားထားသည်ကို သဘောမတူပါက Node များ စတင်ရန် ငြင်းဆိုသည်။
- on-wire `ConfidentialFeatureDigest::conf_rules_version` သည် manifest layout ဗားရှင်းကို ထည့်သွင်းထားသည်။ v1 ကွန်ရက်များအတွက် ၎င်းသည် `Some(1)` ရှိနေမည်ဖြစ်ပြီး `iroha_config::parameters::defaults::confidential::RULES_VERSION` နှင့် ညီမျှရပါမည်။ စည်းမျဥ်းများ တိုးလာသောအခါ၊ ကိန်းသေများကို ဆောင့်ကန်ပါ၊ သရုပ်ကို ပြန်လည်ထုတ်ပါ၊ သော့ခတ်အဆင့်တွင် binaries ကို ထုတ်ပါ။ ဗားရှင်းများ ရောနှောခြင်းသည် `ConfidentialFeatureDigestMismatch` ဖြင့် ပိတ်ဆို့ခြင်းများကို ပယ်ချရန် အတည်ပြုပေးသည်။
- အသက်သွင်းခြင်းသည် အစုအစည်း မှတ်ပုံတင်အပ်ဒိတ်များ၊ ပါရာမီတာ ဘဝသံသရာပြောင်းလဲမှုများနှင့် မူဝါဒအကူးအပြောင်းများကို ဖော်ပြသင့်သည်-
  1. အော့ဖ်လိုင်းအခြေအနေမြင်ကွင်းတွင် စီစဉ်ထားသော မှတ်ပုံတင်ထားသော ဗီဇပြောင်းလဲမှုများ (`Publish*`၊ `Set*Lifecycle`) ကို အသုံးပြုပြီး `compute_confidential_feature_digest` ဖြင့် တွက်ချက်ပြီး အသက်သွင်းပြီးနောက် အချေအတင်ကို တွက်ချက်ပါ။
  2. တွက်ချက်ထားသော hash ကို အသုံးပြု၍ `SetParameter::custom(confidential_registry_root, {"vk_set_hash": "0x…"})` ကို ထုတ်လွှတ်သောကြောင့် နောက်ကျနေသော ရွယ်တူများသည် အလယ်အလတ် စာရင်းသွင်းခြင်း ညွှန်ကြားချက်များကို လွတ်သွားသော်လည်း မှန်ကန်သော အချေအတင်ကို ပြန်လည်ရယူနိုင်ပါသည်။
  3. `ScheduleConfidentialPolicyTransition` ညွှန်ကြားချက်များကို ဖြည့်စွက်ပါ။ ညွှန်ကြားချက်တစ်ခုစီသည် အုပ်ချုပ်ရေးမှထုတ်ပေးသော `transition_id` ကိုကိုးကားရပါမည်။ မေ့သွားပါက ၎င်းကို runtime မှ ပယ်ချမည် ဖြစ်သည်။
  4. ထင်ရှားသောဘိုက်များ၊ SHA-256 လက်ဗွေနှင့် စတင်လုပ်ဆောင်မှုအစီအစဉ်တွင် အသုံးပြုသည့် အချေအတင်များကို ဆက်လက်ထားရှိပါ။ အပိုင်းများကို ရှောင်ရှားရန် အော်ပရေတာများသည် သရုပ်ပြမှုကို အသက်ဝင်စေရန် မဲပေးခြင်းမပြုမီ ရှေးဟောင်းပစ္စည်း သုံးခုလုံးကို စစ်ဆေးအတည်ပြုပါသည်။
- ဖြန့်ချိမှုများသည် ရွှေ့ဆိုင်းထားသောဖြတ်တောက်မှုတစ်ခု လိုအပ်သည့်အခါ၊ အဖော်စိတ်ကြိုက်ကန့်သတ်ဘောင်တစ်ခုတွင် ပစ်မှတ်အမြင့်ကို မှတ်တမ်းတင်ပါ (ဥပမာ `custom.confidential_upgrade_activation_height`)။ ၎င်းသည် စာရင်းစစ်များအား Norito-ကုဒ်လုပ်ထားသော အထောက်အထားတစ်ခုအား အတည်ပြုပေးသည့် အထောက်အထားပြတင်းပေါက်အား ချေဖျက်မှုမစတင်မီ သတိပေးချက်ဝင်းဒိုးအား ဂုဏ်ပြုပါသည်။## Verifier & Parameter Lifecycle

- လက်ရှိတွင် `proving_system` ကို `Halo2` သို့ ပြင်ဆင်ထားသည့် `ZkVerifierEntry { vk_id, circuit_id, version, proving_system, curve, public_inputs_schema_hash, vk_hash, vk_len, max_proof_bytes, gas_schedule_id, activation_height, deprecation_height, withdraw_height, status, metadata_uri_cid, vk_bytes_cid }` လယ်ဂျာစတိုးဆိုင်များ။
- `(circuit_id, version)` အတွဲများသည် တစ်ကမ္ဘာလုံးတွင် ထူးခြားပါသည်။ registry သည် circuit metadata ဖြင့်ရှာဖွေမှုအတွက် ဒုတိယအညွှန်းကို ထိန်းသိမ်းထားသည်။ ထပ်နေသောအတွဲကို ဝင်ခွင့်စာရင်းသွင်းရန် ကြိုးပမ်းမှုကို ငြင်းပယ်သည်။
- `circuit_id` သည် အလွတ်မဟုတ်ရမည်ဖြစ်ပြီး `public_inputs_schema_hash` ကို ပေးရမည် (ပုံမှန်အားဖြင့် အတည်ပြုသူ၏ canonical public-input encoding ၏ Blake2b-32 hash တစ်ခု)။ ဝင်ခွင့်သည် ဤနယ်ပယ်များကို ချန်လှပ်ထားသည့် မှတ်တမ်းများကို ငြင်းပယ်သည်။
- အုပ်ချုပ်မှု ညွှန်ကြားချက်များ ပါဝင်သည်။
  - မက်တာဒေတာဖြင့်သာ `Proposed` ထည့်သွင်းရန် `PUBLISH`။
  - `ACTIVATE { vk_id, activation_height }` သည် ခေတ်နယ်နိမိတ်တစ်ခုတွင် ဝင်ခွင့်စတင်ခြင်းကို အချိန်ဇယားဆွဲရန်။
  အထောက်အထားများသည် ဝင်ရောက်မှုကို ကိုးကားနိုင်သည့် နောက်ဆုံးအမြင့်ကို မှတ်သားရန် `DEPRECATE { vk_id, deprecation_height }`။
  - အရေးပေါ်ပိတ်ရန်အတွက် `WITHDRAW { vk_id, withdraw_height }`၊ ကန့်သတ်ထားသော ပိုင်ဆိုင်မှုများသည် အသစ်များ စတင်မဖွင့်မချင်း အမြင့်ကို ရုပ်သိမ်းပြီးနောက် လျှို့ဝှက်အသုံးစရိတ်ကို ရပ်တန့်စေပါသည်။
- Genesis သည် `confidential_registry_root` စိတ်ကြိုက်ဘောင်တစ်ခု အလိုအလျောက်ထုတ်လွှတ်ကြောင်း ထင်ရှားစေပြီး `vk_set_hash` သည် တက်ကြွသောထည့်သွင်းမှုများနှင့် ကိုက်ညီပါသည်။ node သည် အများသဘောတူဆန္ဒမပါဝင်မီ ဤအချက်များကို ဒေသဆိုင်ရာ registry state နှင့် ကိုက်ညီမှုရှိမရှိ စစ်ဆေးပေးပါသည်။
- အတည်ပြုသူအား မှတ်ပုံတင်ခြင်း သို့မဟုတ် အပ်ဒိတ်လုပ်ခြင်း `gas_schedule_id` လိုအပ်သည်။ `Active` သည် `(circuit_id, version)` အညွှန်းတွင် ပါရှိပြီး Halo2 အထောက်အထားများသည် `OpenVerifyEnvelope` ဖြစ်သည့် `OpenVerifyEnvelope`၊ `(circuit_id, version)` နှင့် `transparent_supply == 0` တို့ကို ပံ့ပိုးပေးသည် မှတ်ပုံတင်မှတ်တမ်း။

### သက်သေသော့များ
- သက်သေပြသော့များသည် လယ်ဂျာမဟုတ်သော်လည်း အကြောင်းအရာ-လိပ်စာသတ်မှတ်မှုစနစ်များ (`pk_cid`၊ `pk_hash`၊ `pk_len`) တို့နှင့်အတူ ထုတ်ဝေထားသော အတည်ပြုသူ မက်တာဒေတာများဖြင့် ရည်ညွှန်းပါသည်။
- Wallet SDKs များသည် PK ဒေတာကို ရယူပြီး၊ ဟက်ကာများကို စစ်ဆေးကာ စက်တွင်းရှိ ကက်ရှ်များ။

### Pedersen & Poseidon ကန့်သတ်ချက်များ
- သီးခြားမှတ်ပုံတင်ခြင်း (`PedersenParams`၊ `PoseidonParams`) မှန်မှန်ကန်ကန်အတည်ပြုမှုဘဝစက်ဝန်းထိန်းချုပ်မှုများ၊ တစ်ခုစီတွင် `params_id`၊ ဂျင်နရေတာ/ကိန်းသေများ၏ ကန့်လန့်ကာများ၊ အသက်သွင်းခြင်း၊ ခွဲထုတ်ခြင်းနှင့် အမြင့်များကို ရုပ်သိမ်းခြင်း။

## Deterministic Ordering & Nullifiers
- ပိုင်ဆိုင်မှုတစ်ခုစီသည် `next_leaf_index` နှင့်အတူ `CommitmentTree` ကို ထိန်းသိမ်းထားသည်။ ပိတ်ဆို့ခြင်းများသည် အဆုံးအဖြတ်အစီအစဥ်အတိုင်း ကတိကဝတ်များကို ပေါင်းထည့်သည်- ဘလောက်အစီအစဥ်တွင် အရောင်းအဝယ်များကို ထပ်လောင်းပါ။ အရောင်းအ၀ယ်တစ်ခုစီတွင် အမှတ်စဉ် `output_idx` ကို အတက်အဆင်းပြုလုပ်ခြင်းဖြင့် အကာအရံထုတ်ပေးမှုများကို ထပ်တလဲလဲပြုလုပ်သည်။
- `note_position` သည် သစ်ပင် အော့ဖ်ဆက်များမှ ဆင်းသက်လာသော်လည်း nullifier ၏ အစိတ်အပိုင်း **မဟုတ်ပါ**။ ၎င်းသည် သက်သေသက်သေများအတွင်း အသင်းဝင်လမ်းကြောင်းများကိုသာ ကျွေးမွေးပါသည်။
- reorgs အောက်တွင် Nullifier တည်ငြိမ်မှုကို PRF ဒီဇိုင်းဖြင့် အာမခံပါသည်။ PRF ထည့်သွင်းမှုသည် `{ nk, note_preimage_hash, asset_id, chain_id, params_id }` နှင့် `max_anchor_age_blocks` ကန့်သတ်ထားသော သမိုင်းဆိုင်ရာ Merkle အမြစ်များကို ကိုးကားသော ကျောက်ဆူးများ။## Ledger Flow
1. **MintConfidential { asset_id၊ ပမာဏ၊ လက်ခံသူ_အရိပ်အမြွက်}**
   - ပိုင်ဆိုင်မှုမူဝါဒ `Convertible` သို့မဟုတ် `ShieldedOnly` လိုအပ်သည်။ ဝင်ခွင့်စစ်ဆေးမှုများသည် ပိုင်ဆိုင်မှုအာဏာပိုင်၊ လက်ရှိ `params_id`၊ နမူနာ `rho`၊ ကတိကဝတ်များကို ထုတ်လွှတ်သည်၊ Merkle သစ်ပင်ကို အပ်ဒိတ်လုပ်ပါသည်။
   - ကတိကဝတ်အသစ်၊ Merkle root delta နှင့် စာရင်းစစ်လမ်းကြောင်းများအတွက် ငွေပေးငွေယူခေါ်ဆိုမှု hash ဖြင့် `ConfidentialEvent::Shielded` ကို ထုတ်လွှတ်သည်။
2. **TransferConfidential { asset_id၊ အထောက်အထား၊ circuit_id၊ ဗားရှင်း၊ nullifiers၊ new_commitments၊ enc_payloads၊ anchor_root၊ မှတ်စုတို }**
   - VM syscall သည် registry entry ကိုအသုံးပြု၍ အထောက်အထားကိုစစ်ဆေးသည်။ host သည် အသုံးမပြုသော nullifiers များကို သေချာစေသည်၊ ကတိကဝတ်များကို အဆုံးအဖြတ်အတိုင်း ပေါင်းထည့်ထားသည်၊ anchor သည် မကြာသေးပါ။
   - Ledger သည် `NullifierSet` သွင်းမှုများကို မှတ်တမ်းတင်သည်၊ လက်ခံသူများ/စာရင်းစစ်များအတွက် ကုဒ်ကုဒ်လုပ်ထားသော payloads များကို သိမ်းဆည်းကာ `ConfidentialEvent::Transferred` သည် nullifiers အကျဉ်းချုပ်များ၊ မှာယူထားသော outputs၊ proof hash နှင့် Merkle roots များကို ထုတ်လွှတ်ပါသည်။
3. **RevealConfidential { asset_id၊ အထောက်အထား၊ circuit_id၊ ဗားရှင်း၊ nullifier၊ ပမာဏ၊ recipient_account၊ anchor_root }**
   - `Convertible` ပိုင်ဆိုင်မှုများအတွက်သာ ရရှိနိုင်ပါသည်။ အထောက်အထားသည် မှတ်စုတန်ဖိုးကို ထုတ်ဖော်ပြသသည့်ပမာဏနှင့် ညီမျှသည်၊ လယ်ဂျာခရက်ဒစ်များ ပွင့်လင်းသော လက်ကျန်ငွေကို သက်သေပြပြီး သုံးစွဲထားသော nullifier ကို အမှတ်အသားပြုခြင်းဖြင့် အကာအရံများကို လောင်ကျွမ်းစေပါသည်။
   - `ConfidentialEvent::Unshielded` ကို အများသူငှာ ပမာဏ၊ စားသုံးထားသော ပျက်ပြယ်စေသော၊ အထောက်အထား အထောက်အထားများနှင့် ငွေပေးငွေယူ ခေါ်ဆိုမှု hash တို့ဖြင့် ထုတ်လွှတ်ပါသည်။

## ဒေတာမော်ဒယ် ထပ်လောင်းမှုများ
- ဖွင့်ထားသောအလံ၊ `assume_valid`၊ ဓာတ်ငွေ့/ကန့်သတ်ခလုတ်များ၊ ကျောက်ဆူးဝင်းဒိုး၊ အတည်ပြုစနစ်နောက်ကွယ်ပါရှိသော `ConfidentialConfig` (ပြင်ဆင်မှုအပိုင်းအသစ်)။
- `ConfidentialNote`၊ `ConfidentialTransfer`၊ နှင့် `ConfidentialMint` Norito တိကျပြတ်သားသောဗားရှင်း byte (`CONFIDENTIAL_ASSET_V1 = 0x01`) နှင့် schemas။
- `ConfidentialEncryptedPayload` သည် XChaCha20-Poly1305 အပြင်အဆင်အတွက် `version = CONFIDENTIAL_ENCRYPTED_PAYLOAD_V1` ဖြင့် `version = CONFIDENTIAL_ENCRYPTED_PAYLOAD_V1` ဖြင့် AEAD မှတ်စုဘိုက်များကို ခြုံငုံထားသည်။
- Canonical key-derivation vector များသည် `docs/source/confidential_key_vectors.json` တွင်နေထိုင်သည်; CLI နှင့် Torii အဆုံးမှတ် နှစ်ခုစလုံးသည် ဤပွဲများနှင့် ဆန့်ကျင်ဘက်သို့ ပြန်သွားသည်။ အသုံးစရိတ်/ပျက်ပြယ်သော/ကြည့်ရှုခြင်းလှေကားအတွက် Wallet-facing ဆင်းသက်လာမှုများကို `fixtures/confidential/keyset_derivation_v1.json` တွင် ထုတ်ဝေပြီး ဘာသာစကား စုံညီမှုကို အာမခံရန်အတွက် Rust + Swift SDK စမ်းသပ်မှုများဖြင့် ကျင့်သုံးသည်။
- `asset::AssetDefinition` သည် `confidential_policy: AssetConfidentialPolicy { mode, vk_set_hash, poseidon_params_id, pedersen_params_id, pending_transition }` ရရှိသည်။
- `ZkAssetState` သည် `(backend, name, commitment)` ကို လွှဲပြောင်းခြင်း/အကာအကွယ်မရှိ စစ်ဆေးခြင်းအတွက် စည်းနှောင်ခြင်းကို ဆက်လက်လုပ်ဆောင်သည်။ စာရင်းသွင်းကတိကဝတ်များနှင့် မကိုက်ညီသည့် ကိုးကားထားသော သို့မဟုတ် အတွင်းမှစစ်ဆေးသည့်သော့အား အထောက်အထားများ ပယ်ချပြီး အခြေအနေမပြောင်းမီ ဖြေရှင်းထားသော နောက်ခံကီးနှင့် လွှဲပြောင်းခြင်း/အကာအကွယ်မရှိသော အထောက်အထားများကို စစ်ဆေးသည်။
- `CommitmentTree` (နယ်ခြားစစ်ဆေးရေးဂိတ်များပါရှိသည့် ပစ္စည်းတစ်ခုလျှင်)၊ `NullifierSet`၊ `(chain_id, asset_id, nullifier)`၊ `ZkVerifierEntry`၊ `PedersenParams`၊ `PoseidonParams` တွင် စတိုးဆိုင်
- Mempool သည် စောစီးစွာ ထပ်နေသည့်ရှာဖွေတွေ့ရှိမှုနှင့် ကျောက်ဆူးအသက်စစ်ဆေးမှုများအတွက် ယာယီ `NullifierIndex` နှင့် `AnchorIndex` တည်ဆောက်မှုများကို ထိန်းသိမ်းထားသည်။
- Norito schema အပ်ဒိတ်များတွင် အများသူငှာ သွင်းအားစုများအတွက် canonical ordering ပါဝင်သည်။ အသွားအပြန် စမ်းသပ်မှုများသည် ကုဒ်နံပါတ်သတ်မှတ်မှုကို သေချာစေသည်။
- ကုဒ်ဝှက်ထားသော payload အသွားအပြန်ခရီးများကို ယူနစ်စစ်ဆေးမှုများ (`crates/iroha_data_model/src/confidential.rs`) နှင့် အထက်ရှိပိုက်ဆံအိတ်သော့မှဆင်းသက်လာသော vector များသည် စာရင်းစစ်များအတွက် AEAD စာအိတ်၏ဆင်းသက်လာမှုများကို ကျောက်ချထားသည်။ `norito.md` သည် စာအိတ်အတွက် on-wire header ကို မှတ်တမ်းတင်ထားသည်။## IVM ပေါင်းစပ်ခြင်းနှင့် Syscall
- `VERIFY_CONFIDENTIAL_PROOF` syscall လက်ခံခြင်းကို မိတ်ဆက်ပေးပါ-
  - `circuit_id`, `version`, `scheme`, `public_inputs`, `proof` နှင့် ရလဒ် `ConfidentialStateDelta { asset_id, nullifiers, commitments, enc_payloads }`။
  - Syscall သည် မှတ်ပုံတင်ခြင်းမှ အတည်ပြုသည့် မက်တာဒေတာကို တင်ဆောင်သည်၊ အရွယ်အစား/အချိန်ကန့်သတ်ချက်များကို ပြဋ္ဌာန်းပေးသည်၊ အဆုံးအဖြတ်ပေးသည့်ဓာတ်ငွေ့ကို ကောက်ခံပြီး အထောက်အထားအောင်မြင်ပါက မြစ်ဝကျွန်းပေါ်ဒေသကိုသာ သက်ရောက်မှုရှိသည်။
- Host သည် Merkle root လျှပ်တစ်ပြက်ရိုက်ချက်များနှင့် nullifier အခြေအနေကို ပြန်လည်ရယူရန်အတွက် ဖတ်ရန်-သီးသန့် `ConfidentialLedger` လက္ခဏာကို ဖော်ထုတ်ပေးပါသည်။ Kotodama စာကြည့်တိုက်သည် သက်သေခံ စည်းဝေးပွဲ အထောက် အကူများနှင့် schema တရားဝင်မှုကို ပံ့ပိုးပေးပါသည်။
- Pointer-ABI docs များကို အထောက်အထားကြားခံပုံစံနှင့် registry လက်ကိုင်များကို ရှင်းလင်းရန် အပ်ဒိတ်လုပ်ထားသည်။

## Node Capability Negotiation
- `feature_bits.confidential` ကို `ConfidentialFeatureDigest { vk_set_hash, poseidon_params_id, pedersen_params_id, conf_rules_version }` နှင့်အတူ Handshake ကြော်ငြာသည်။ Validator ပါဝင်မှုသည် `confidential.enabled=true`၊ `assume_valid=false`၊ တူညီသော verifier backend identifiers နှင့် ကိုက်ညီသော digests လိုအပ်ပါသည်။ `HandshakeConfidentialMismatch` နှင့် လက်ဆွဲခြင်း မကိုက်ညီပါ။
- Config သည် `assume_valid` ကို လေ့လာသူ node များအတွက်သာ ပံ့ပိုးပေးသည်- ပိတ်ထားသည့်အခါ၊ လျှို့ဝှက်ညွှန်ကြားချက်များကို ကြုံတွေ့ရပါက ထိတ်လန့်ခြင်းမရှိဘဲ အဆုံးအဖြတ်ပေးသော `UnsupportedInstruction` ကို ထုတ်ပေးပါသည်။ ဖွင့်ထားသောအခါတွင် လေ့လာသူများသည် အထောက်အထားများကို မစစ်ဆေးဘဲ ကြေညာထားသော မြစ်ဝကျွန်းပေါ်ဒေသများကို အသုံးပြုသည်။
- ဒေသတွင်းလုပ်ဆောင်နိုင်စွမ်းကိုပိတ်ထားပါက Mempool သည် လျှို့ဝှက်ငွေပေးချေမှုများကို ငြင်းပယ်သည်။ အတင်းအဖျင်း စစ်ထုတ်မှုများသည် အရွယ်အစား ကန့်သတ်ချက်များအတွင်း မျက်ကန်းမသိသော အတည်ပြုသူ ID များကို အရွယ်အစား ကန့်သတ်ချက်များအတွင်း လိုက်လျောညီထွေဖြစ်စေသော စွမ်းဆောင်ရည်မရှိဘဲ တူညီသောစွမ်းရည်မရှိဘဲ ရွယ်တူများထံ ပေးပို့ခြင်းကို ရှောင်ကြဉ်ပါသည်။

### Pruning & Nullifier Retention Policy ကို ဖော်ပြပါ။

လျှို့ဝှက်စာရင်းဇယားများသည် ဆန်းသစ်မှုနှင့် မှတ်သားထားရန် လုံလောက်သောမှတ်တမ်းကို ထိန်းသိမ်းထားရပါမည်။
အုပ်ချုပ်မှုစနစ်ဖြင့် မောင်းနှင်သော စာရင်းစစ်များကို ပြန်လည်ဖွင့်ပါ။ ပုံသေမူဝါဒဖြင့် ပြဋ္ဌာန်းထားသည်။
`ConfidentialLedger` မှာ-

- **Nullifier retention-** သုံးစွဲထားသော nullifier များကို *အနည်းဆုံး* `730` ရက် (24 ရက်) ထားရှိပါ။
  လများ) အမြင့်ကိုသုံးပြီးနောက်၊ သို့မဟုတ် ပိုကြာပါက ထိန်းညှိပေးသော window ကို။
  အော်ပရေတာများသည် `confidential.retention.nullifier_days` မှတစ်ဆင့် ဝင်းဒိုးကို တိုးချဲ့နိုင်သည်။
  ထိန်းသိမ်းထားသောဝင်းဒိုးထက်ငယ်သော Nullifiers များသည် Torii မှတစ်ဆင့် စုံစမ်းနိုင်ဆဲဖြစ်ရမည်၊
  စာရင်းစစ်များသည် နှစ်ဆသုံးစွဲခြင်းမရှိကြောင်း သက်သေပြနိုင်သည်။
- **Reveal pruning:** ဖောက်ထွင်းမြင်ရသော (`RevealConfidential`) သုတ်သင်ခြင်း
  ပိတ်ဆို့ခြင်း အပြီးသတ်ပြီးနောက် ချက်ခြင်း ဆက်စပ်မှတ်စု ကတိကဝတ်များ ရှိသော်လည်း၊
  စားသုံးထားသော nullifier သည် အထက်ဖော်ပြပါ ထိန်းသိမ်းမှုစည်းမျဉ်းအတိုင်း ဆက်လက်တည်ရှိနေပါသည်။ ဆက်စပ်ဖော်ပြသည်။
  ဖြစ်ရပ်များ (`ConfidentialEvent::Unshielded`) အများသူငှာ ပမာဏကို မှတ်တမ်းတင်ပါ၊ လက်ခံသူ၊
  နှင့် အထောက်အထား hash ဖြစ်သောကြောင့် သမိုင်းဝင်ဖော်ပြမှုများကို ပြန်လည်တည်ဆောက်ရာတွင် ဖြတ်တောက်ရန်မလိုအပ်ပါ။
  စာဝှက်စာသား
- ** Frontier စစ်ဆေးရေးဂိတ်များ-** ကတိကဝတ်နယ်ခြားများသည် လှိမ့်စစ်ဆေးရေးဂိတ်များကို ထိန်းသိမ်းထားသည်။
  ပိုကြီးသော `max_anchor_age_blocks` နှင့် retention window ကို ဖုံးအုပ်ထားသည်။ ဆုံမှတ်များ
  ကြားကာလအတွင်း nullifiers များအားလုံး သက်တမ်းကုန်ပြီးမှသာ အဟောင်းစစ်ဆေးရေးဂိတ်များကို ကျစ်လစ်စေပါသည်။
- ** Stale digest remedation :** `HandshakeConfidentialMismatch` သည် ကြောင့်ဖြစ်လျှင်
  ပျံ့လွင့်မှုကို ချေဖျက်ရန်၊ အော်ပရေတာများသည် (၁) ထို nullifier retention windows ကို အတည်ပြုသင့်သည်။
  (၂) `iroha_cli app confidential verify-ledger` သို့ အစုအဝေးကို ညှိပါ။
  သိမ်းထားသော nullifier set ကိုဆန့်ကျင်သော digest ကို ပြန်လည်ထုတ်ပေးပြီး (3) ၎င်းကို ပြန်လည်အသုံးချပါ။
  ပြန်လည်ဆန်းသစ်ထားသော မန်နီးဖက်စ်။ အချိန်မတန်မီ ဖြတ်တောက်ထားသော မည်သည့် nullifiers များမှ ပြန်လည်ရယူရပါမည်။
  ကွန်ရက်သို့မဝင်မီ အအေးခန်းသိုလှောင်မှု။လုပ်ငန်းလည်ပတ်မှုစာရင်းစာအုပ်တွင် စက်တွင်းမှ အစားထိုးမှုများကို မှတ်တမ်းတင်ပါ။ အုပ်ချုပ်မှုမူဝါဒများ တိုးချဲ့ဆောင်ရွက်ခြင်း
ထိန်းသိမ်းဝင်းဒိုးသည် node configuration နှင့် archival storage အစီအစဉ်များကို အပ်ဒိတ်လုပ်ရပါမည်။
သော့ခတ်ခြင်း။

### နှင်ထုတ်ခြင်းနှင့် ပြန်လည်ရယူခြင်း စီးဆင်းမှု

1. ဖုန်းခေါ်ဆိုစဉ်အတွင်း `IrohaNetwork` သည် ကြော်ငြာထားသော စွမ်းရည်များကို နှိုင်းယှဉ်ပါသည်။ မည်သည့် မကိုက်ညီမှုမဆို `HandshakeConfidentialMismatch` ကို တိုးစေသည်။ ချိတ်ဆက်မှုကို ပိတ်ထားပြီး ရွယ်တူချင်းသည် `Ready` သို့ ရာထူးတိုးခြင်းမပြုဘဲ ရှာဖွေတွေ့ရှိမှုတန်းစီတွင် ရှိနေပါသည်။
2. ပျက်ကွက်မှုကို ကွန်ရက်ဝန်ဆောင်မှုမှတ်တမ်း (အဝေးမှချေဖျက်ခြင်းနှင့် နောက်ကွယ်တွင် အပါအဝင်) မှတစ်ဆင့် ပေါ်လာပြီး Sumeragi သည် အဆိုပြုချက် သို့မဟုတ် မဲပေးခြင်းအတွက် သက်တူရွယ်တူကို မည်သည့်အခါမျှ အချိန်ဇယားဆွဲမည်မဟုတ်ပါ။
3. အော်ပရေတာများသည် အတည်ပြုမှုစာရင်းသွင်းမှုများနှင့် ကန့်သတ်ချက်အစုံများ (`vk_set_hash`, `pedersen_params_id`, `poseidon_params_id`) သို့မဟုတ် `next_conf_features` ကို သဘောတူညီထားသော `next_conf_features` ဖြင့် အဆင့်သတ်မှတ်ခြင်းဖြင့် ပြန်လည်ပြင်ဆင်သည်။ အချေအတင် ကိုက်ညီသည်နှင့် နောက်လက်ဆွဲခြင်းသည် အလိုအလျောက် အောင်မြင်ပါသည်။
4. အရွယ်တူရွယ်တူတစ်ဦးသည် ဘလောက်တစ်ခုကို ထုတ်လွှင့်ရန် စီမံနိုင်သည် (ဥပမာ၊ မှတ်တမ်းဟောင်းကို ပြန်လည်ပြသခြင်းမှတစ်ဆင့်)၊ အတည်ပြုသူများသည် ၎င်းအား `BlockRejectionReason::ConfidentialFeatureDigestMismatch` ဖြင့် အဆုံးအဖြတ်ပေးကာ ကွန်ရက်တစ်ခုလုံးတွင် စာရင်းဇယားအခြေအနေကို လိုက်လျောညီထွေဖြစ်စေရန် အဆုံးအဖြတ်ပေးသည်။

### ပြန်ဖွင့်သည်- အန္တရာယ်ကင်းသော လက်ဆွဲနှုတ်ဆက်ခြင်း

1. အပြင်ထွက်ကြိုးပမ်းမှုတိုင်းသည် လတ်ဆတ်သော Noise/X25519 သော့ချက်ပစ္စည်းကို ခွဲဝေပေးသည်။ (`handshake_signature_payload`) တွင် ရေးထိုးထားသည့် လက်ဆွဲဝန်ထုပ်ဝန်ပိုးသည် ဒေသတွင်းနှင့် အဝေးထိန်း ပေါ်ပင် အများသူငှာသော့များ၊ Norito-ကုဒ်ဖြင့် ကြော်ငြာထားသော ဆော့ဖ်ဝဲလိပ်စာနှင့်—`handshake_chain_id`—ကွင်းဆက် identifier တို့ကို ပေါင်းစပ်ထားသည်။ မက်ဆေ့ဂျ်သည် node မှမထွက်ခွာမီ AEAD-ကုဒ်ဝှက်ထားသည်။
2. တုံ့ပြန်သူသည် လုပ်ဖော်ကိုင်ဖက်/ဒေသခံသော့အမှာစာဖြင့် ပေးဆောင်မှုကို ပြန်လည်တွက်ချက်ပြီး `HandshakeHelloV1` တွင် ထည့်သွင်းထားသော Ed25519 လက်မှတ်ကို အတည်ပြုသည်။ ပေါ်ပင်သော့များနှင့် ကြော်ငြာထားသည့်လိပ်စာ နှစ်ခုစလုံးသည် လက်မှတ်ဒိုမိန်း၏ တစ်စိတ်တစ်ပိုင်းဖြစ်သောကြောင့်၊ ဖမ်းယူထားသော မက်ဆေ့ချ်ကို အခြားမျိုးတူများနှင့် ပြန်လည်ပြသခြင်း သို့မဟုတ် ဖောက်ပြန်နေသည့် ချိတ်ဆက်မှုအား ပြန်လည်ရယူခြင်းသည် အဆုံးအဖြတ်အရ စိစစ်မှု မအောင်မြင်ပါ။
3. လျှို့ဝှက်လုပ်ဆောင်နိုင်မှု အလံများနှင့် `ConfidentialFeatureDigest` သည် `HandshakeConfidentialMeta` အတွင်းသို့ လည်ပတ်သည်။ လက်ခံသူသည် tuple `{ enabled, assume_valid, verifier_backend, digest }` ကို ၎င်း၏စက်တွင်းသတ်မှတ်ထားသော `ConfidentialHandshakeCaps` နှင့် နှိုင်းယှဉ်သည်။ သယ်ယူပို့ဆောင်ရေး `Ready` သို့ ကူးပြောင်းခြင်းမပြုမီ `HandshakeConfidentialMismatch` နှင့် ကိုက်ညီမှုမရှိပါက စောစောထွက်ပါမည်။
4. အော်ပရေတာများသည် (`compute_confidential_feature_digest`) မှတဆင့် digest ကို ပြန်လည်တွက်ချက်ပြီး ပြန်လည်ချိတ်ဆက်ခြင်းမပြုမီ အပ်ဒိတ်စာရင်းသွင်းခြင်း/မူဝါဒများဖြင့် ဆုံမှတ်များကို ပြန်လည်စတင်ရပါမည်။ သက်တူရွယ်တူများသည် အထောက်အထားဟောင်းများကို ကြော်ငြာရာတွင် လက်ဆွဲနှုတ်ဆက်ခြင်းမှ ဆက်လက်ပျက်ကွက်သွားကာ ပျက်ယွင်းနေသောအခြေအနေသို့ ပြန်လည်ထည့်သွင်းခြင်းမှ တားဆီးပေးပါသည်။
5. လက်ဆွဲနှုတ်ဆက်ခြင်း အောင်မြင်မှုများနှင့် ကျရှုံးမှုများသည် စံ `iroha_p2p::peer` ကောင်တာများ (`handshake_failure_count`၊ အမှားအယွင်းများကို ခွဲခြားသတ်မှတ်ပေးသူများ) ကို အပ်ဒိတ်လုပ်ပြီး အဝေးထိန်းမျိုးတူ ID ဖြင့် တဂ်လုပ်ထားသော ဖွဲ့စည်းတည်ဆောက်ပုံမှတ်တမ်းများကို ထုတ်လွှတ်ပြီး လက်ဗွေကို ချေဖျက်ပါ။ ထုတ်လွှင့်နေစဉ်အတွင်း ပြန်လည်ပြသရန် ကြိုးပမ်းမှုများ သို့မဟုတ် ပုံစံမှားယွင်းမှုများကို ဖမ်းမိရန် ဤညွှန်ကိန်းများကို စောင့်ကြည့်ပါ။## သော့စီမံခန့်ခွဲမှုနှင့် ပေးဆောင်မှုများ
- အကောင့်တစ်ခုချင်းသော့ဆင်းသက်လာမှု အထက်တန်းအဆင့်-
  - `sk_spend` → `nk` ( nullifier key), `ivk` (အဝင်ကြည့်ရှုသောကီး), `ovk` (အထွက်ကြည့်ရှုသော့), `fvk`။
- ကုဒ်ဝှက်ထားသော မှတ်စု ပေးဆောင်မှုများသည် ECDH မှရရှိသော မျှဝေသောသော့များဖြင့် AEAD ကို အသုံးပြုသည်။ ရွေးချယ်နိုင်သော စာရင်းစစ်ကြည့်ရှုမှုသော့များကို ပိုင်ဆိုင်မှုမူဝါဒအလိုက် အထွက်များနှင့် တွဲထားနိုင်သည်။
- CLI ထပ်တိုးမှုများ- `confidential create-keys`၊ `confidential send`၊ `confidential export-view-key`၊ စာရင်းစစ်မှတ်စုစာဝှက်ခြင်းအတွက် စာရင်းစစ်ကိရိယာနှင့် `iroha app zk envelope` အကူအညီပေးသူ Prometheus

## ဓာတ်ငွေ့၊ ကန့်သတ်ချက်များနှင့် DoS ထိန်းချုပ်မှုများ
- သတ်မှတ်ဓာတ်ငွေ့အချိန်ဇယား-
  - Halo2 (Plonkish) : အခြေခံ `250_000` ဓာတ်ငွေ့ + `2_000` အများသူငှာ ထည့်သွင်းမှုအလိုက် ဓာတ်ငွေ့။
  - `5` ဓါတ်ငွေ့တစ်ဘိုက်၊ အပေါင်း ပျက်ပြယ်သွားသော (`300`) နှင့် ကတိကဝတ် (`500`) ကျသင့်ငွေ။
  - အော်ပရေတာများသည် node configuration (`confidential.gas.{proof_base, per_public_input, per_proof_byte, per_nullifier, per_commitment}`); စတင်ချိန်တွင် သို့မဟုတ် config အလွှာ hot-reloads နှင့် cluster တစ်လျှောက် အဆုံးအဖြတ်အတိုင်း အသုံးချသည့်အခါတွင် အပြောင်းအလဲများ ပျံ့နှံ့သွားပါသည်။
- Hard limits (configurable defaults):
- `max_proof_size_bytes = 262_144`။
- `max_nullifiers_per_tx = 8`, `max_commitments_per_tx = 8`, `max_confidential_ops_per_block = 256`။
- `verify_timeout_ms = 750`, `max_anchor_age_blocks = 10_000`။ `verify_timeout_ms` ထက်ကျော်လွန်သော အထောက်အထားများသည် ညွှန်ကြားချက်ကို အဆုံးအဖြတ်ဖြင့် ပျက်ပြယ်သွားသည် (အုပ်ချုပ်မှုမဲများသည် `proof verification exceeded timeout`၊ `VerifyProof` သည် အမှားတစ်ခု ပြန်ဖြစ်သွားသည်)။
- ထပ်လောင်းခွဲတမ်းများသည် အသက်ရှင်သန်မှုကို သေချာစေသည်- `max_proof_bytes_block`, `max_verify_calls_per_tx`, `max_verify_calls_per_block`, နှင့် `max_public_inputs` ဘောင်းဘီဘလောက်တည်ဆောက်သူများ၊ `reorg_depth_bound` (≥ `max_anchor_age_blocks`) သည် နယ်ခြားစစ်ဆေးရေးဂိတ် ထိန်းသိမ်းမှုကို ထိန်းချုပ်သည်။
- ယခု Runtime execution သည် ဤငွေလွှဲနှုန်းတစ်ခု သို့မဟုတ် တစ်တုံးကန့်သတ်ချက်ထက်ကျော်လွန်သော ငွေပေးငွေယူများကို ပယ်ချပြီး အဆုံးအဖြတ်ပေးသော `InvalidParameter` အမှားများကို ထုတ်လွှတ်ကာ လယ်ဂျာအခြေအနေကို မပြောင်းလဲပါ။
- အရင်းအမြစ်အသုံးပြုမှုကို ကန့်သတ်ထားရန် အတည်ပြုသူအား မခေါ်ဆိုမီ အထောက်အထားအလျားနှင့် ကျောက်ဆူးခေတ် `vk_id` ဖြင့် လျှို့ဝှက်ငွေပေးငွေယူများကို Mempool မှ ကြိုတင်စစ်ထုတ်ပါသည်။
- အချိန်ကုန်သွားခြင်း သို့မဟုတ် ကန့်သတ်ချိုးဖောက်မှုအပေါ် အတည်ပြုခြင်းအား အဆုံးအဖြတ်ပြုခြင်း ရပ်တန့်ခြင်း တိကျပြတ်သားသော အမှားများဖြင့် ငွေပေးငွေယူများ ပျက်ကွက်သည်။ SIMD နောက်ခံများသည် ရွေးချယ်နိုင်သော်လည်း ဓာတ်ငွေ့စာရင်းအင်းအား ပြောင်းလဲခြင်းမပြုပါ။

### Calibration Baselines & Acceptance Gates
- **အကိုးအကားပလပ်ဖောင်းများ။** Calibration သည် အောက်ဖော်ပြပါ ဟာ့ဒ်ဝဲပရိုဖိုင်သုံးခုကို ဖုံးအုပ်ထားရပါမည်။ ပရိုဖိုင်အားလုံးကို ဖမ်းယူရန် ပျက်ကွက်သည့် လုပ်ဆောင်မှုများကို ပြန်လည်သုံးသပ်နေစဉ်အတွင်း ပယ်ချပါသည်။| ကိုယ်ရေးအကျဉ်း | ဗိသုကာ | CPU/Instance | Compiler အလံများ | ရည်ရွယ်ချက် |
  | ---| ---| ---| ---| ---|
  | `baseline-simd-neutral` | `x86_64` | AMD EPYC 7B12 (32c) သို့မဟုတ် Intel Xeon Gold 6430 (24c) | `RUSTFLAGS="-C target-feature=-avx,-avx2,-fma"` | vector ပင်ကိုယ်စိတ်မပါဘဲ ကြမ်းပြင်တန်ဖိုးများကို ထူထောင်ပါ။ ကုန်ကျစရိတ်ဇယားများကို ညှိရန်အသုံးပြုသည်။ |
  | `baseline-avx2` | `x86_64` | Intel Xeon Gold 6430 (24c) | ပုံသေထုတ်လွှတ်မှု | AVX2 လမ်းကြောင်းကို အတည်ပြုသည်။ SIMD အရှိန်မြှင့်မှုများသည် ကြားနေဓာတ်ငွေ့များ၏ ခံနိုင်ရည်အောက်တွင် ရှိနေကြောင်း စစ်ဆေးသည်။ |
  | `baseline-neon` | `aarch64` | AWS Graviton3 (c7g.4xlarge) | ပုံသေထုတ်လွှတ်မှု | NEON backend သည် အဆုံးအဖြတ်ရှိပြီး x86 အချိန်ဇယားများနှင့် ကိုက်ညီကြောင်း သေချာစေသည်။ |

- **Benchmark ကြိုးကြိုး။** ဓာတ်ငွေ့ ချိန်ညှိခြင်း အစီရင်ခံစာများအားလုံးကို အောက်ပါတို့နှင့်အတူ ထုတ်လုပ်ရပါမည်-
  - `CRITERION_HOME=target/criterion cargo bench -p iroha_core isi_gas_calibration -- --sample-size 200 --warm-up-time 5 --save-baseline <profile-label>`
  - အဆုံးအဖြတ်ခံစစ်မှူးကို အတည်ပြုရန် `cargo test -p iroha_core bench_repro -- --ignored`။
  - VM opcode ကုန်ကျစရိတ် ပြောင်းလဲသည့်အခါတိုင်း `CRITERION_HOME=target/criterion cargo bench -p ivm gas_calibration -- --sample-size 200 --warm-up-time 5 --save-baseline <profile-label>`။

- **ကျပန်းပြင်ဆင်ထားသည်။** `IROHA_CONF_GAS_SEED=conf-gas-seed-2026Q1` ကို ခုံတန်းလျားများမပြေးဆွဲမီတွင် `iroha_test_samples::gen_account_in` သည် အဆုံးအဖြတ်ပေးသော `KeyPair::from_seed` လမ်းကြောင်းသို့ ပြောင်းသည်။ သံကြိုးသည် `IROHA_CONF_GAS_SEED_ACTIVE=…` တစ်ကြိမ် ရိုက်နှိပ်သည်။ variable ပျောက်နေပါက ပြန်လည်စစ်ဆေးရန် မအောင်မြင်ပါ။ auxiliary randomness ကိုမိတ်ဆက်ပေးသောအခါတွင် calibration utilities အသစ်တိုင်းသည် ဤ env var ကို ဂုဏ်ပြုပေးရပါမည်။

- ** ရလဒ်ကို ဖမ်းယူပါ။**
  - ပရိုဖိုင်တစ်ခုစီအတွက် စံသတ်မှတ်ချက်အကျဉ်းချုပ်များ (`target/criterion/**/raw.csv`) ကို ထုတ်ဝေပြီးသော အနုပညာပစ္စည်းများသို့ အပ်လုဒ်လုပ်ပါ။
  - `docs/source/confidential_assets_calibration.md` တွင် အသုံးပြုထားသော git commit နှင့် compiler ဗားရှင်းနှင့်အတူ - ဆင်းသက်လာသောမက်ထရစ်များ (`ns/op`, `gas/op`, `ns/gas`) ကို သိမ်းဆည်းပါ။
  - ပရိုဖိုင်တစ်ခုလျှင် နောက်ဆုံးအခြေခံလိုင်းနှစ်ခုကို ထိန်းသိမ်းပါ။ နောက်ဆုံးအစီရင်ခံစာကို အတည်ပြုပြီးသည်နှင့် လျှပ်တစ်ပြက်အဟောင်းများကို ဖျက်ပါ။

- **လက်ခံမှု သည်းခံနိုင်မှု။**
  - `baseline-simd-neutral` နှင့် `baseline-avx2` အကြား ဂတ်စ်မြစ်ဝကျွန်းပေါ်ဒေသ ≤ ±1.5% ရှိနေရမည်။
  - `baseline-simd-neutral` နှင့် `baseline-neon` အကြား ဂတ်စ်မြစ်ဝကျွန်းပေါ်ဒေသ ≤ ±2.0% ရှိနေရမည်။
  - ဤသတ်မှတ်ချက်များထက်ကျော်လွန်၍ ချိန်ညှိခြင်းအဆိုပြုချက်များသည် အချိန်ဇယား ချိန်ညှိမှုများ သို့မဟုတ် ကွာဟမှုနှင့် လျော့ပါးသွားမှုကို ရှင်းပြသည့် RFC တစ်ခုခု လိုအပ်သည်။

- **စစ်ဆေးရန်စာရင်း။** တင်သွင်းသူများသည် အောက်ပါတို့အတွက် တာဝန်ရှိပါသည်။
  - စံကိုက်ချိန်ညှိမှုမှတ်တမ်းတွင် `uname -a`၊ `/proc/cpuinfo` ကောက်နှုတ်ချက် (မော်ဒယ်၊ အဆင့်တက်ခြင်း) နှင့် `rustc -Vv` ပါဝင်သည်။
  - `IROHA_CONF_GAS_SEED` သည် ခုံတန်းလျားအထွက်တွင် ပဲ့တင်ထပ်နေကြောင်း အတည်ပြုခြင်း (ခုံတန်းလျားများသည် တက်ကြွသောမျိုးစေ့များကို ထုတ်ယူသည်)။
  - နှလုံးခုန်နှုန်းထိန်းစက်နှင့် လျှို့ဝှက်စစ်ဆေးသည့်စနစ် လုပ်ဆောင်ချက် အလံများ ကြေးမုံပြင်ထုတ်လုပ်မှု (`--features confidential,telemetry`) ကို Telemetry ဖြင့် ခုံတန်းလျားများ လည်ပတ်နေချိန်တွင် သေချာစေခြင်း။

  - `GET /confidential/zk_registry`
  - `GET /confidential/params`## စမ်းသပ်ခြင်းဗျူဟာ
- အဆုံးအဖြတ်ပေးခြင်း- လုပ်ကွက်များအတွင်း ကျပန်းငွေပေးငွေယူ မွှေနှောက်ခြင်းသည် တူညီသော Merkle အမြစ်များနှင့် nullifier အစုံများကို ထုတ်ပေးသည်။
- Reorg ခံနိုင်ရည်အား- ကျောက်ဆူးများဖြင့် ပိတ်ဆို့ထားသော reorg အများအပြားကို အတုယူပါ။ nullifiers များသည် တည်ငြိမ်နေပြီး ဟောင်းနွမ်းသော ကျောက်ဆူးများကို ငြင်းပယ်ထားသည်။
- ဓာတ်ငွေ့မျိုးကွဲများ- SIMD အရှိန်မြှင့်ခြင်းနှင့်အတူ node များတစ်လျှောက် တူညီသောဓာတ်ငွေ့အသုံးပြုမှုကို စစ်ဆေးပါ။
- နယ်နိမိတ်စမ်းသပ်ခြင်း- အရွယ်အစား/ဓာတ်ငွေ့မျက်နှာကျက်ရှိ အထောက်အထားများ၊ အများဆုံးဝင်/ထွက်အရေအတွက်၊ အချိန်ကုန်ဆုံးမှု ပြဋ္ဌာန်းခြင်း။
- ဘဝသံသရာ- အတည်ပြုစနစ်နှင့် ကန့်သတ်ချက်ဖွင့်ခြင်း/ကန့်ကွက်ခြင်း၊ အလှည့်ကျစမ်းသပ်မှုများအတွက် အုပ်ချုပ်မှုဆိုင်ရာ လုပ်ဆောင်ချက်များ။
- မူဝါဒ FSM- ခွင့်ပြု/မထားသော အကူးအပြောင်းများ၊ ဆိုင်းငံ့ထားသော အကူးအပြောင်းနှောင့်နှေးမှုများ၊ ထိရောက်သော အမြင့်များအနီးရှိ mempool ငြင်းပယ်ခြင်း။
- Registry အရေးပေါ်အခြေအနေများ- အရေးပေါ်ငွေထုတ်ခြင်းသည် `withdraw_height` တွင် ထိခိုက်သည့်ပိုင်ဆိုင်မှုများကို ရပ်တန့်စေပြီး အထောက်အထားများပြီးနောက် ငြင်းပယ်သည်။
- စွမ်းဆောင်ရည်ဂိတ်ပေါက်- မကိုက်ညီသော `conf_features` နှင့် ကိုက်ညီမှုမရှိသော အထောက်အထားများ `assume_valid=true` ပါသော လေ့လာသူများသည် အများဆန္ဒကို မထိခိုက်စေဘဲ ဆက်လက်လုပ်ဆောင်ပါ။
- State equivalence- validator/full/observer node များသည် canonical chain တွင် ထပ်တူထပ်မျှသော state root များကို ထုတ်လုပ်သည်။
- အနုတ်လက္ခဏာပြခြင်း- ပုံသဏ္ဍာန်မမှန်သော အထောက်အထားများ၊ ကြီးမားသောဝန်ထုပ်ဝန်ပိုးများနှင့် nullifier တိုက်မိမှုများသည် အဆုံးအဖြတ်အရ ငြင်းပယ်ပါသည်။

## ထူးချွန်သောအလုပ်
- Benchmark Halo2 ကန့်သတ်ဘောင်များ (ဆားကစ်အရွယ်အစား၊ ရှာဖွေမှုနည်းဗျူဟာ) နှင့် ရလဒ်များကို ချိန်ညှိခြင်းဖွင့်စာအုပ်တွင် မှတ်တမ်းတင်ထားသောကြောင့် ဓာတ်ငွေ့/အချိန်လွန်ပုံသေများကို လာမည့် `confidential_assets_calibration.md` ပြန်လည်ဆန်းသစ်မှုနှင့်အတူ မွမ်းမံပြင်ဆင်နိုင်ပါသည်။
- စာရင်းစစ်၏ထုတ်ဖော်မှုမူဝါဒများနှင့် ဆက်စပ်ရွေးချယ်မှု-ကြည့်ရှုခြင်း API များကို အပြီးသတ်လုပ်ဆောင်ပြီး အုပ်ချုပ်မှုမူကြမ်းကို လက်မှတ်ရေးထိုးပြီးသည်နှင့် အတည်ပြုထားသော အလုပ်အသွားအလာကို Torii သို့ ချိတ်ဆက်ပါ။
- SDK အကောင်အထည်ဖေါ်သူများအတွက် စာအိတ်ဖော်မတ်ကို မှတ်တမ်းတင်ခြင်း - လက်ခံသူများစွာ၏ ထွက်ပေါက်များနှင့် အစုလိုက်အစီအစဥ်များကို ကာမိစေရန် သက်သေခံ ကုဒ်ဝှက်ခြင်းအစီအစဉ်ကို တိုးချဲ့ပါ။
- ဆားကစ်များ၊ မှတ်ပုံတင်ခြင်းများ၊ နှင့် ကန့်သတ်လည်ပတ်ခြင်းဆိုင်ရာ လုပ်ထုံးလုပ်နည်းများကို ပြင်ပလုံခြုံရေးပြန်လည်သုံးသပ်မှုအား ကော်မရှင်ဖွဲ့ပြီး အတွင်းစစ်ဆေးမှုအစီရင်ခံစာများဘေးတွင် တွေ့ရှိချက်များကို သိမ်းဆည်းပါ။
- စာရင်းစစ်သုံးစွဲမှုပြန်လည်ပေါင်းစည်းရေး APIs များကိုသတ်မှတ်ပြီး မြင်ကွင်း-သော့နယ်ပယ်လမ်းညွှန်ချက်ကို ထုတ်ဝေခြင်းဖြင့် ပိုက်ဆံအိတ်ရောင်းချသူများသည် တူညီသောသက်သေအထောက်အထားကို အကောင်အထည်ဖော်နိုင်စေရန်။## အကောင်အထည်ဖော်မှု Phasing
1. ** Phase M0 — Stop-Ship Hardening**
   - ✅ Nullifier ဆင်းသက်လာမှုသည် ယခု Poseidon PRF ဒီဇိုင်း (`nk`, `rho`, `asset_id`, `chain_id`) တို့ကို လယ်ဂျာမွမ်းမံမှုများတွင် ပြဋ္ဌာန်းထားသော တိကျသေချာသော ကတိကဝတ်ပြုသည့်အမိန့်ကို လိုက်နာပါသည်။
   - ✅ Execution သည် သက်သေအရွယ်အစားထုပ်များနှင့် အရောင်းအ၀ယ်/တစ်ကွက်လျှင် လျှို့ဝှက်ခွဲတမ်းများကို ပြဋ္ဌာန်းထားပြီး၊ ဘတ်ဂျက်ပိုငွေလွှဲမှုများကို အဆုံးအဖြတ်ပေးသောအမှားများဖြင့် ပယ်ချပါသည်။
   - ✅ P2P လက်ဆွဲနှုတ်ဆက်ခြင်းသည် `ConfidentialFeatureDigest` (backend digest + registry fingerprints) ကိုကြော်ငြာပြီး `HandshakeConfidentialMismatch` မှတစ်ဆင့် တိကျစွာမကိုက်ညီပါ။
   - ✅ လျှို့ဝှက်လုပ်ဆောင်မှုလမ်းကြောင်းများတွင် ထိတ်လန့်ကြောက်ရွံ့မှုများကို ဖယ်ရှားပြီး လိုက်ဖက်ညီသောစွမ်းရည်မရှိဘဲ node များအတွက် အခန်းကဏ္ဍကို ထည့်သွင်းပါ။
   - ⚪ နယ်ခြားစစ်ဆေးရေးဂိတ်များအတွက် အချိန်ကုန်သွားသော ဘတ်ဂျက်များကို တွန်းအားပေးရန်နှင့် နယ်မြေစစ်ဆေးရေးဂိတ်များအတွက် နက်ရှိုင်းသောအပိုင်းများကို ကန့်သတ်ပါ။
     - ✅ အတည်ပြုခြင်း အချိန်ကုန်သွားသော ဘတ်ဂျက်များကို ပြဌာန်းထားသည်။ ယခုအခါ `verify_timeout_ms` ထက်ကျော်လွန်သော အထောက်အထားများသည် အဆုံးအဖြတ်အရ မအောင်မြင်ပါ။
     - ✅ နယ်ခြားစစ်ဆေးရေးဂိတ်များသည် ယခု `reorg_depth_bound` ကို လေးစားလိုက်နာပြီး အဆုံးအဖြတ်ပေးသော လျှပ်တစ်ပြက်များကို ထိန်းသိမ်းထားစဉ် ပြင်ဆင်ထားသော ပြတင်းပေါက်ထက် ဟောင်းနေသည့် စစ်ဆေးရေးဂိတ်များကို ဖြတ်တောက်လိုက်ပါသည်။
   - mint/transfer/reveal ညွှန်ကြားချက်များအတွက် `AssetConfidentialPolicy`၊ မူဝါဒ FSM နှင့် ဥပဒေစိုးမိုးရေးဂိတ်များကို မိတ်ဆက်ပေးပါ။
   - ဘလောက်ခေါင်းစီးများတွင် `conf_features` ကို သွင်းပြီး registry/parameter များ ကွဲပြားသွားသောအခါ validator ပါဝင်မှုကို ငြင်းဆိုပါ။
2. ** Phase M1 — မှတ်ပုံတင်ခြင်းနှင့် ကန့်သတ်ချက်များ**
   - Land `ZkVerifierEntry`၊ `PedersenParams`၊ နှင့် `PoseidonParams` သည် အုပ်ချုပ်မှုဆိုင်ရာ ops၊ genesis anchoring, and cache management တို့ပါရှိသော မှတ်ပုံတင်မှုများ။
   - မှတ်ပုံတင်ခြင်းရှာဖွေမှုများ၊ ဓာတ်ငွေ့အချိန်ဇယား IDs၊ schema hashing နှင့် size စစ်ဆေးမှုများလိုအပ်ရန် Wire syscall ။
   - Ship encrypt လုပ်ထားသော payload format v1၊ wallet key ဆင်းသက်လာသော vector များနှင့် လျှို့ဝှက်သော့စီမံခန့်ခွဲမှုအတွက် CLI ပံ့ပိုးမှု။
3. **အဆင့် M2 — ဓာတ်ငွေ့နှင့် စွမ်းဆောင်ရည်**
   - သတ်မှတ်ပြဋ္ဌာန်းထားသော ဓာတ်ငွေ့အချိန်ဇယား၊ တစ်တုံးတစ်တုံးလျှင် ကောင်တာများနှင့် တယ်လီမီတာဖြင့် စံကိုက်ကြိုးများကို အကောင်အထည်ဖော်ပါ (latency၊ အထောက်အထားအရွယ်အစား၊ mempool ငြင်းပယ်မှုများ) ကို အကောင်အထည်ဖော်ပါ။
   - ပိုင်ဆိုင်မှုလုပ်ငန်းအများအပြားအတွက် ခိုင်မာစေမည့် CommitmentTree စစ်ဆေးရေးဂိတ်များ၊ LRU တင်ခြင်းနှင့် nullifier အညွှန်းများ။
4. **Phase M3 — လှည့်ခြင်းနှင့် ပိုက်ဆံအိတ်တူးခြင်း**
   - multi-parameter နှင့် multi-version proof လက်ခံမှုကို ဖွင့်ပါ။ အသွင်ကူးပြောင်းရေးဆိုင်ရာ စာအုပ်များနှင့်အတူ အုပ်ချုပ်မှု-မောင်းနှင်သော အသက်သွင်းခြင်း/ကန့်ကွက်ခြင်းကို ပံ့ပိုးပါ။
   - ပိုက်ဆံအိတ် SDK/CLI ရွှေ့ပြောင်းခြင်းစီးဆင်းမှုများ၊ စာရင်းစစ်စကင်န်ဖတ်ခြင်းလုပ်ငန်းအသွားအလာများနှင့် သုံးစွဲမှုပြန်လည်သင့်မြတ်ရေးကိရိယာတို့ကို ပေးပို့ပါ။
5. ** Phase M4 — စာရင်းစစ်နှင့် အလုပ်များ**
   - စာရင်းစစ်သော့လုပ်ငန်းအသွားအလာများ၊ ရွေးချယ်ထားသောထုတ်ဖော် APIs နှင့် လုပ်ငန်းလည်ပတ်မှုဆိုင်ရာစာအုပ်များကို ပံ့ပိုးပေးပါ။
   - ပြင်ပ ကုဒ်ဝှက်ခြင်း/လုံခြုံရေး ပြန်လည်သုံးသပ်ခြင်းကို အချိန်ဇယားဆွဲပြီး `status.md` တွင် တွေ့ရှိချက်များကို ထုတ်ဝေပါ။

အဆင့်တစ်ခုစီသည် blockchain ကွန်ရက်အတွက် အဆုံးအဖြတ်ပေးသည့် အာမခံချက်များကို ထိန်းသိမ်းထားရန် လမ်းပြမြေပုံမှတ်တိုင်များနှင့် ဆက်စပ်စစ်ဆေးမှုများကို အပ်ဒိတ်လုပ်သည်။

### SDK နှင့် Fixture Coverage (Phase M1)

ကုဒ်ဝှက်ထားသော payload v1 သည် ယခုအခါတွင် canonical fixtures များနှင့်အတူ တင်ပို့နေသောကြောင့် SDK တိုင်းသည် ၎င်းကိုထုတ်လုပ်သည်။
တူညီသော Norito စာအိတ်များနှင့် ငွေပေးငွေယူ hashe များ။ ရွှေထည်ပစ္စည်းများ နေထိုင်သည်။
`fixtures/confidential/wallet_flows_v1.json` တို့ကို တိုက်ရိုက်ကျင့်သုံးသည်။
Rust နှင့် Swift suites (`crates/iroha_data_model/tests/confidential_wallet_fixtures.rs`၊
`IrohaSwift/Tests/IrohaSwiftTests/ConfidentialWalletFixturesTests.swift`):

```bash
# Rust parity (verifies the signed hex + hash for every case)
cargo test -p iroha_data_model confidential_wallet_fixtures

# Swift parity (builds the same envelopes via TxBuilder/NativeBridge)
cd IrohaSwift && swift test --filter ConfidentialWalletFixturesTests
```ပွဲတိုင်းတွင် case identifier၊ signed transaction hex နှင့် မျှော်လင့်ထားသည်များကို မှတ်တမ်းတင်ပါသည်။
hash Swift ကုဒ်ပြောင်းကိရိယာသည် အမှုကိစ္စကို မထုတ်လုပ်နိုင်သေးသောအခါ—`zk-transfer-basic` သည်
`ZkTransfer` တည်ဆောက်သူမှ တံခါးပိတ်ထားဆဲဖြစ်သည်—စမ်းသပ်မှုအစုံသည် `XCTSkip` ကို ထုတ်လွှတ်သောကြောင့်၊
လမ်းပြမြေပုံသည် စီးဆင်းနေသည့်လမ်းကြောင်းများကို ရှင်းရှင်းလင်းလင်း ချည်နှောင်ထားရန် လိုအပ်သေးသည်။ ခံစစ်မှူးကို အဆင့်မြှင့်တင်ခြင်း။
ဖော်မတ်ဗားရှင်းကို ဆောင့်မတိုက်ဘဲ SDK များကို သိမ်းဆည်းထားခြင်းဖြင့် အတွဲနှစ်ခုစလုံး ပျက်သွားပါမည်။
သော့ခတ်မှုအဆင့်တွင် သံချေးတက်ခြင်း ရည်ညွှန်းခြင်း အကောင်အထည်ဖော်ခြင်း။

#### လျင်မြန်သော တည်ဆောက်သူများ
`TxBuilder` သည် တစ်ခုချင်းစီအတွက် asynchronous နှင့် callback-based helpers များကို ဖော်ထုတ်ပေးသည်
လျှို့ဝှက်တောင်းဆိုချက် (`IrohaSwift/Sources/IrohaSwift/TxBuilder.swift:1183`)။
တည်ဆောက်သူများသည် `connect_norito_bridge` တင်ပို့မှုများကို အားကိုးသည်။
(`crates/connect_norito_bridge/src/lib.rs:3337`၊
`IrohaSwift/Sources/IrohaSwift/NativeBridge.swift:1014`) ဒါကြောင့်ထုတ်ပေးပါတယ်။
payloads များသည် Rust host encoders byte-for-byte နှင့် ကိုက်ညီပါသည်။ ဥပမာ-

```swift
let account = AccountId.make(publicKey: keypair.publicKey, domain: "wonderland")
let request = RegisterZkAssetRequest(
    chainId: chainId,
    authority: account,
    assetDefinitionId: "62Fk4FPcMuLvW5QjDGNF2a4jAmjM",
    zkParameters: myZkParams,
    ttlMs: 60_000
)
let envelope = try TxBuilder(client: client)
    .buildRegisterZkAsset(request: request, keypair: keypair)
try await TxBuilder(client: client)
    .submit(registerZkAsset: request, keypair: keypair)
```

အကာအရံ/အကာအကွယ်ကို တူညီသောပုံစံအတိုင်း လုပ်ဆောင်ပါ (`submit(shield:)`၊
`submit(unshield:)`) နှင့် Swift fixture tests များသည် builders များဖြင့် ပြန်လည်လုပ်ဆောင်သည် ။
ထုတ်ပေးထားသော ငွေပေးငွေယူ hashes များ ကျန်ရှိနေသေးကြောင်း အာမခံရန် အဆုံးအဖြတ်ပေးသော အဓိကပစ္စည်း
`wallet_flows_v1.json` တွင် သိမ်းဆည်းထားသည့်အရာများနှင့် ညီမျှသည်။

#### JavaScript တည်ဆောက်သူများ
JavaScript SDK သည် တင်ပို့သော ငွေပေးငွေယူ အကူအညီများမှတစ်ဆင့် အလားတူစီးဆင်းမှုများကို ထင်ဟပ်စေသည်။
`javascript/iroha_js/src/transaction.js` မှ။ အစရှိတဲ့ ဆောက်လုပ်ရေးသမားတွေ၊
`buildRegisterZkAssetTransaction` နှင့် `buildRegisterZkAssetInstruction`
(`javascript/iroha_js/src/instructionBuilders.js:1832`) အတည်ပြုခြင်းကီးကို ပုံမှန်ဖြစ်စေသည်။
Rust host မှ လက်ခံနိုင်သည့် Norito payloads များကို identifier များနှင့် ထုတ်လွှတ်သည်
အဒက်တာများ။ ဥပမာ-

```js
import {
  buildRegisterZkAssetTransaction,
  signTransaction,
  ToriiClient,
} from "@hyperledger/iroha";

const unsigned = buildRegisterZkAssetTransaction({
  registration: {
    authority: "<i105-account-id>",
    assetDefinitionId: "62Fk4FPcMuLvW5QjDGNF2a4jAmjM",
    zkParameters: {
      commit_params: "vk_shield",
      reveal_params: "vk_unshield",
    },
    metadata: { displayName: "Rose (Shielded)" },
  },
  chainId: "00000000-0000-0000-0000-000000000000",
});
const signed = signTransaction(unsigned, myKeypair);
await new ToriiClient({ baseUrl: "https://torii" }).submitTransaction(signed);
```

အကာအရံများ၊ လွှဲပြောင်းခြင်းနှင့် အကာအကွယ်မဲ့ တည်ဆောက်သူများသည် JS ကို ပေးသည့် တူညီသောပုံစံအတိုင်း လိုက်နာကြသည်။
Swift နှင့် Rust ကဲ့သို့ တူညီသော ergonomics ကိုခေါ်ဆိုသူများ။ အောက်တွင် စမ်းသပ်မှုများ
`javascript/iroha_js/test/transactionBuilder.test.js` သည် normalization ကို ဖုံးအုပ်ထားသည်။
အထက်ဖော်ပြပါ ပစ္စည်းများသည် လက်မှတ်ထိုးထားသော ငွေပေးငွေယူဘိုက်များကို တသမတ်တည်းဖြစ်စေသော်လည်း ယုတ္တိဗေဒအရ၊

### Telemetry & Monitoring (Phase M2)

Phase M2 သည် ယခုအခါ Prometheus နှင့် Grafana မှတစ်ဆင့် CommitmentTree ကျန်းမာရေးကို တိုက်ရိုက်တင်ပို့သည်-

- `iroha_confidential_tree_commitments`၊ `iroha_confidential_tree_depth`၊ `iroha_confidential_root_history_entries` နှင့် `iroha_confidential_frontier_checkpoints` တို့သည် ပိုင်ဆိုင်မှုအလိုက် တိုက်ရိုက် Merkle နယ်နိမိတ်ကို ဖော်ထုတ်စေပြီး `iroha_confidential_root_evictions_total` / `iroha_confidential_frontier_evictions_total` ဖြင့် Lcount အရေအတွက်ကို အင်အားသုံးရန် `zk.root_history_cap` နှင့် checkpoint depth window ။
- `iroha_confidential_frontier_last_checkpoint_height` နှင့် `iroha_confidential_frontier_last_checkpoint_commitments` သည် လတ်တလော နယ်ခြားစစ်ဆေးရေးဂိတ်၏ အမြင့် + ကတိကဝတ်ရေတွက်မှုကို ထုတ်ဝေလိုက်သောကြောင့် reorg လေ့ကျင့်ခန်းများနှင့် rollbacks များသည် စစ်ဆေးရေးဂိတ်များ တိုးတက်လာပြီး မျှော်မှန်းထားသော payload ပမာဏကို ထိန်းသိမ်းထားကြောင်း သက်သေပြနိုင်သည်။
- Grafana ဘုတ် (`dashboards/grafana/confidential_assets.json`) တွင် အတိမ်အနက် စီးရီး၊ ဖယ်ရှားမှုနှုန်း အကန့်များနှင့် ရှိပြီးသား အတည်ပြု ကက်ရှ်ဝစ်ဂျက်များ ပါဝင်သောကြောင့် အော်ပရေတာများသည် စစ်ဆေးရေးဂိတ်များ လည်ပတ်နေချိန်တွင်ပင် CommitmentTree အတိမ်အနက်သည် ဘယ်တော့မှ ပြိုမကျကြောင်း သက်သေပြနိုင်သည်။
- သတိပေးချက် `ConfidentialTreeDepthZero` (`dashboards/alerts/confidential_assets_rules.yml` တွင်) ခရီးစဉ်များကို ကတိကဝတ်ပြုထားသည်ကို တွေ့ရှိသော်လည်း အစီရင်ခံထားသည့် အတိမ်အနက်သည် ငါးမိနစ်အတွင်း သုညတွင် ရှိနေသည်။

Grafana ကို ဝါယာမဆက်မီ စက်တွင်းရှိ မက်ထရစ်များကို သင်စစ်ဆေးနိုင်သည်-

```bash
curl -s http://127.0.0.1:8180/metrics \
  | rg 'iroha_confidential_(tree_(commitments|depth)|root_history_entries|frontier_(checkpoints|last_checkpoint_height|last_checkpoint_commitments)|root_evictions_total|frontier_evictions_total){asset_id="4cuvDVPuLBKJyN6dPbRQhmLh68sU"}'
```

မှတ်တမ်းများကို ချုံ့လိုက်သည့်အခါမှသာ ဖယ်ရှားမှုကောင်တာများ တိုးလာသော်လည်း ကတိကဝတ်အသစ်များဖြင့် အနက်တိုးလာသည်ကို အတည်ပြုရန် ၎င်းကို `rg 'iroha_confidential_tree_depth'` နှင့် တွဲပါ။ ဤတန်ဖိုးများသည် သင်အုပ်ချုပ်မှုဆိုင်ရာ အထောက်အထားအစုအဝေးတွင် ပူးတွဲပါရှိသော Grafana ဒက်ရှ်ဘုတ်ထုတ်ယူမှုနှင့် ကိုက်ညီရပါမည်။

#### ဂက်စ်အချိန်ဇယား telemetry & သတိပေးချက်များPhase M2 သည် ထုတ်လွှင့်မှုအား အတည်ပြုခြင်းမပြုမီတွင် အတည်ပြုသူတိုင်းသည် တူညီသောအတည်ပြုချက်ကုန်ကျစရိတ်ကို မျှဝေကြောင်း သက်သေပြနိုင်စေရန် တယ်လီမက်ထရီ ပိုက်လိုင်းသို့ ပြုပြင်နိုင်သော ဓာတ်ငွေ့မြှောက်ကိန်းများကို တယ်လီမက်ထရီ ပိုက်လိုင်းသို့ ချိတ်ဆက်ပေးသည်-

- `iroha_confidential_gas_base_verify` မှန်များ `confidential.gas.proof_base` (မူရင်း `250_000`)။
- `iroha_confidential_gas_per_public_input`, `iroha_confidential_gas_per_proof_byte`, `iroha_confidential_gas_per_nullifier`, နှင့် `iroha_confidential_gas_per_commitment` တို့သည် `ConfidentialConfig` တွင် ၎င်းတို့၏ သက်ဆိုင်ရာ ခလုတ်များကို ရောင်ပြန်ဟပ်သည်။ စတင်ချိန်တွင် နှင့် config hot-reloads များ အချိန်တိုင်းတွင် တန်ဖိုးများ အပ်ဒိတ်လုပ်ခြင်း၊ `irohad` (`crates/irohad/src/main.rs:1591,1642`) သည် `Telemetry::set_confidential_gas_schedule` မှတစ်ဆင့် လက်ရှိအချိန်ဇယားကို တွန်းပို့သည်။

အဖုများသည် သက်တူရွယ်တူများနှင့် တူညီကြောင်း အတည်ပြုရန် CommitmentTree မက်ထရစ်များနှင့်အတူ တိုင်းထွာများကို ခြစ်ပါ။

```bash
# compare active multipliers across validators
for host in validator-a validator-b validator-c; do
  curl -s "http://$host:8180/metrics" \
    | rg 'iroha_confidential_gas_(base_verify|per_public_input|per_proof_byte|per_nullifier|per_commitment)'
done
```

Grafana ဒက်ရှ်ဘုတ် `confidential_assets.json` ယခုတွင် တိုင်းတာမှုငါးခုနှင့် ကွဲပြားမှုကို မီးမောင်းထိုးပြသည့် “ဓာတ်ငွေ့အချိန်ဇယား” အကန့်တစ်ခု ပါဝင်သည်။ `dashboards/alerts/confidential_assets_rules.yml` ကာဗာရှိ သတိပေးချက်စည်းမျဉ်းများ-
- `ConfidentialGasMismatch`- 3 မိနစ်ထက်ပို၍ ကွဲပြားသွားသည့်အခါ ခြစ်ရန်ပစ်မှတ်များနှင့် စာမျက်နှာများအားလုံးရှိ မြှောက်ကိန်းတစ်ခုစီ၏ အမြင့်ဆုံး/မိနစ်ကို စစ်ဆေးပြီး `confidential.gas` ကို hot-reload သို့မဟုတ် ပြန်လည်အသုံးချရန် အော်ပရေတာများအား လှုံ့ဆော်ပေးသည်။
- `ConfidentialGasTelemetryMissing`- Prometheus သည် ပျောက်ဆုံးနေသော ခြစ်ပစ်မှတ် သို့မဟုတ် ပိတ်ထားသော တယ်လီမီတာကို ညွှန်ပြသည့် မြှောက်ကိန်းငါးခုမှ 5 မိနစ်ကြာ မခြစ်နိုင်သည့်အခါ သတိပေးသည်။

ဖုန်းခေါ်ဆိုမှုစုံစမ်းစစ်ဆေးမှုများအတွက် အောက်ဖော်ပြပါ PromQL ကို အသုံးဝင်အောင်ထားပါ။

```promql
# ensure every multiplier matches across validators (uses the same projection as the alert)
(max without(instance, job) (iroha_confidential_gas_per_public_input)
  - min without(instance, job) (iroha_confidential_gas_per_public_input)) == 0
```

သွေဖည်မှုသည် ထိန်းချုပ်ထားသော config ဖြန့်ချိမှုများ၏ပြင်ပတွင် သုညရှိနေသင့်သည်။ ဓာတ်ငွေ့ဇယားကိုပြောင်းသည့်အခါ၊ မခြစ်မီ/ပြီးနောက် ဖမ်းယူ၊ ပြောင်းလဲမှုတောင်းဆိုချက်တွင် ပူးတွဲပါနှင့် `docs/source/confidential_assets_calibration.md` ကို အမြှောက်အသစ်များဖြင့် အပ်ဒိတ်လုပ်ခြင်းဖြင့် အုပ်ချုပ်မှုပြန်လည်သုံးသပ်သူများသည် တယ်လီမီတာအထောက်အထားများကို ချိန်ညှိမှုအစီရင်ခံစာနှင့် ချိတ်ဆက်နိုင်သည်။
