# ლითონის და ნეონის აჩქარების გეგმა (Swift & Rust)

ეს დოკუმენტი ასახავს საერთო გეგმას დეტერმინისტული ტექნიკის გასააქტიურებლად
აჩქარება (მეტალის GPU + NEON / Accelerate SIMD + StrongBox ინტეგრაცია) მასშტაბით
Rust სამუშაო სივრცე და Swift SDK. ის მიმართავს საგზაო რუქის მოკვლევას
**Hardware Acceleration Workstream-ის ქვეშ (macOS/iOS)** და უზრუნველყოფს ხელმისაწვდომობას
არტეფაქტი Rust IVM გუნდისთვის, Swift ხიდის მფლობელებისთვის და ტელემეტრიის ხელსაწყოებისთვის.

> ბოლო განახლება: 2026-01-12  
> მფლობელები: IVM Performance TL, Swift SDK წამყვანი

## გოლები

1. ხელახლა გამოიყენეთ Rust GPU ბირთვები (Poseidon/BN254/CRC64) Apple-ის აპარატურაზე Metal-ის საშუალებით
   გამოთვალეთ ჩრდილები დეტერმინისტული პარიტეტით CPU ბილიკებთან მიმართებაში.
2. გაამჟღავნეთ აჩქარების გადამრთველები (`AccelerationConfig`) თავიდან ბოლომდე, რათა Swift აპები
   შეუძლია მონაწილეობა მიიღოს Metal/NEON/StrongBox-ში ABI/პარიტეტის გარანტიების შენარჩუნებისას.
3. ინსტრუმენტის CI + dashboards ზედაპირზე პარიტეტის/საორიენტაციო მონაცემები და დროშა
   რეგრესია CPU-ს წინააღმდეგ GPU/SIMD ბილიკებზე.
4. გააზიარეთ StrongBox/უსაფრთხო ანკლავის გაკვეთილები Android (AND2) და Swift-ს შორის
   (IOS4) ხელმოწერის ნაკადების დეტერმინისტულად გასწორების მიზნით.

**განახლება (CRC64 + Stage‑1 განახლება):** CRC64 GPU დამხმარეები ახლა ჩართულია `norito::core::hardware_crc64`-ში 192KiB ნაგულისხმევი წყვეტით (გადალახვა `NORITO_GPU_CRC64_MIN_BYTES`-ით ან აშკარა დამხმარე ბილიკით Prometheus-ით, ხოლო Norito-ით გადაინაცვლებს). JSON Stage-1 ამონაჭრები ხელახლა შეფასდა (`examples/stage1_cutover` → `benchmarks/norito_stage1/cutover.csv`), შეინარჩუნა სკალარული ამოჭრა 4KiB-ზე და Stage‑1 GPU ნაგულისხმევი გასწორება 192KiB-მდე (Prometheus დატვირთავს GPU-ს ანაზღაურების და მცირე ზომის დოკუმენტების გაშვებას6. ხარჯები.

## ინტერფეისები და API კონტრაქტები### ჟანგი (`ivm::AccelerationConfig`)
- შეინახეთ არსებული ველები (`enable_simd`, `enable_metal`, `enable_cuda`, `max_gpus`, ზღურბლები).
- დაამატეთ აშკარა ლითონის დათბობა, რათა თავიდან აიცილოთ პირველი გამოყენების შეყოვნება (Rust #15875).
- უზრუნველყოთ პარიტეტული API-ების სტატუსის/დიაგნოსტიკის დამაბრუნებელი დაფები:
  - მაგ. `ivm::vector::metal_status()` -> {ჩართულია, პარიტეტი, ბოლო_შეცდომა}.
- გამომავალი ბენჩმარკინგის მეტრიკა (მერკლის ხის ვადები, CRC გამტარუნარიანობა) მეშვეობით
  ტელემეტრიული კაკვები `ci/xcode-swift-parity`-ისთვის.
- ლითონის ჰოსტი ახლა იტვირთება კომპილირებული `fastpq.metallib`, აგზავნის FFT/IFFT/LDE
  და Poseidon kernels და უბრუნდება CPU-ს განხორციელებას, როდესაც ეს
  metallib ან მოწყობილობის რიგი მიუწვდომელია.

### C FFI (`connect_norito_bridge`)
- ახალი სტრუქტურა `connect_norito_acceleration_config` (დასრულებულია).
- Getter-ის დაფარვა ახლა მოიცავს `connect_norito_get_acceleration_config` (მხოლოდ კონფიგურაცია) და `connect_norito_get_acceleration_state` (კონფიგურაცია + პარიტეტი) სეტერის ასახვის მიზნით.
- დოკუმენტის სტრუქტურის განლაგება სათაურის კომენტარებში SPM/CocoaPods მომხმარებლებისთვის.

### Swift (`AccelerationSettings`)
- ნაგულისხმევი: მეტალი ჩართულია, CUDA გამორთულია, ზღურბლები ნული (მემკვიდრეობა).
- უარყოფითი ღირებულებების იგნორირება; `apply()` გამოძახებულია ავტომატურად `IrohaSDK`-ის მიერ.
- `AccelerationSettings.runtimeState()` ახლა ჩნდება `connect_norito_get_acceleration_state`
  ტვირთამწეობა (კონფიგურაცია + ლითონის/CUDA პარიტეტის სტატუსი), ასე რომ Swift-ის დაფები ასხივებენ იგივე ტელემეტრიას
  როგორც Rust (`supported/configured/available/parity`). დამხმარე აბრუნებს `nil`-ს, როდესაც
  ხიდი არ არის, რომ ტესტები პორტატული იყოს.
- `AccelerationBackendStatus.lastError` კოპირებს გამორთვის/შეცდომის მიზეზს
  `connect_norito_get_acceleration_state` და ათავისუფლებს ბუფერს, როგორც კი სტრიქონი იქნება
  მატერიალიზებულია ისე, რომ მობილური პარიტეტის დაფებს შეეძლოთ ანოტაცია, თუ რატომ იყო გამორთული Metal/CUDA
  თითოეულ მასპინძელს.
- `AccelerationSettingsLoader` (`IrohaSwift/Sources/IrohaSwift/AccelerationSettingsLoader.swift`,
  ტესტები ქვეშ `IrohaSwift/Tests/IrohaSwiftTests/AccelerationSettingsLoaderTests.swift`) ახლა
  წყვეტს ოპერატორის მანიფესტებს იგივე პრიორიტეტული თანმიმდევრობით, როგორც Norito დემო: პატივი
  `NORITO_ACCEL_CONFIG_PATH`, ძიება შეფუთული `acceleration.{json,toml}` / `client.{json,toml}`,
  შედით არჩეული წყარო და დაუბრუნდით ნაგულისხმევს. აპებს აღარ სჭირდებათ შეკვეთილი ჩამტვირთავი
  ასახავს Rust `iroha_config` ზედაპირს.
- განაახლეთ ნიმუშის აპლიკაციები და README გადართვისა და ტელემეტრიის ინტეგრაციის საჩვენებლად.

### ტელემეტრია (დაფები + ექსპორტიორები)
- პარიტეტული არხი (mobile_parity.json):
  - `acceleration.metal/neon/strongbox` -> {ჩართულია, პარიტეტი, perf_delta_pct}.
  - მიიღეთ `perf_delta_pct` საბაზისო CPU და GPU შედარება.
  - `acceleration.metal.disable_reason` სარკეები `AccelerationBackendStatus.lastError`
    ასე რომ, Swift ავტომატიზაციას შეუძლია გამორთული GPU-ების დროშის მონიშვნა ისეთივე ერთგულებით, როგორც Rust
    დაფები.
- CI feed (mobile_ci.json):
  - `acceleration_bench.metal_vs_cpu_merkle_ms` -> {cpu, ლითონი}
  - `acceleration_bench.neon_crc64_throughput_mb_s` -> ორმაგი.
- ექსპორტიორებმა უნდა მოიპოვონ მეტრიკა Rust საორიენტაციო ნიშნებიდან ან CI გაშვებებიდან (მაგ. გაშვება
  ლითონის/CPU მიკროსკამი, როგორც `ci/xcode-swift-parity`-ის ნაწილი).### კონფიგურაციის ღილაკები და ნაგულისხმევი (WP6-C)
- `AccelerationConfig` ნაგულისხმევი: `enable_metal = true` macOS build-ებზე, `enable_cuda = true` როდესაც CUDA ფუნქცია შედგენილია, `max_gpus = None` (კედლის გარეშე). Swift `AccelerationSettings` შეფუთვა იღებს იგივე ნაგულისხმევს `connect_norito_set_acceleration_config`-ის მეშვეობით.
- Norito Merkle-ის ევრისტიკა (GPU vs CPU): `merkle_min_leaves_gpu = 8192` საშუალებას აძლევს GPU-ის ჰეშინგს ხეებისთვის ≥8192 ფოთლის მქონე; backend-ის უგულებელყოფა (`merkle_min_leaves_metal`, `merkle_min_leaves_cuda`) ნაგულისხმევად იგივე ზღურბლზე, თუ ცალსახად არ არის დაყენებული.
- CPU უპირატესობის ევრისტიკა (SHA2 ISA დღემდე): AArch64-ზე (ARMv8 SHA2) და x86/x86_64 (SHA-NI) CPU-ის გზა რჩება სასურველი `prefer_cpu_sha2_max_leaves_* = 32_768`-მდე; ამაზე ზემოთ მოქმედებს GPU ბარიერი. ამ მნიშვნელობების კონფიგურირება შესაძლებელია `AccelerationConfig`-ის საშუალებით და უნდა დარეგულირდეს მხოლოდ საორიენტაციო მტკიცებულებებით.

   დეტერმინისტული ვექტორები; მუშაობს `cargo test -p ivm --features metal` ქვეშ.
   `crates/fastpq_prover/src/metal.rs` ახლა აგზავნის მხოლოდ macOS-ის პარიტეტის ტესტებს, რომლებიც
   ვარჯიში FFT/IFFT/LDE და პოსეიდონი სკალარული მითითების საწინააღმდეგოდ.
2. **სწრაფი კვამლის აღკაზმულობა**: გააფართოვეთ IOS6 ტესტირება CPU–ს წინააღმდეგ მეტალის შესასრულებლად
   კოდირება (Merkle/CRC64) როგორც ემულატორებზე, ასევე StrongBox მოწყობილობებზე; შეადარეთ
   შედეგები და ჟურნალის პარიტეტის სტატუსი.
3. **CI**: განაახლეთ `norito_bridge_ios.yml` (უკვე იძახებს `make swift-ci`) დააყენებს
   არტეფაქტების აჩქარების მეტრიკა; დარწმუნდით, რომ გაშვება ადასტურებს Buildkite-ს
   `ci/xcframework-smoke:<lane>:device_tag` მეტამონაცემები აღკაზმულობის ცვლილებების გამოქვეყნებამდე,
   და ავარიდე ზოლს პარიტეტის/სასაზღვრო დრიფტის დროს.
4. **Dashboards**: ახალი ველები ახლა გამოისახება CLI გამომავალში. დარწმუნდით, რომ ექსპორტიორებმა აწარმოონ
   მონაცემები საინფორმაციო დაფის პირდაპირ გადატრიალებამდე.

## WP2-A Metal Shader Plan (Poseidon Pipelines)

პირველი WP2 ეტაპები მოიცავს Poseidon Metal-ის ბირთვების დაგეგმვის სამუშაოებს
რომელიც ასახავს CUDA-ს განხორციელებას. გეგმა ანაწილებს ძალისხმევას ბირთვებად,
მასპინძლის დაგეგმვა და გაზიარებული მუდმივი დადგმა, ასე რომ, მოგვიანებით მუშაობა შეიძლება მხოლოდ ფოკუსირებული იყოს
განხორციელება და ტესტირება.

### ბირთვის ფარგლები

1. `poseidon_permute`: ცვლის `state_count` დამოუკიდებელ სახელმწიფოებს. თითოეული ძაფი
   ფლობს `STATE_CHUNK` (4 მდგომარეობა) და აწარმოებს ყველა `TOTAL_ROUNDS` გამეორებას გამოყენებით
   ძაფთა ჯგუფის მიერ გაზიარებული მრგვალი მუდმივები დადგმული დისპეტჩერიზაციის დროს.
2. `poseidon_hash_columns`: კითხულობს იშვიათ `PoseidonColumnSlice` კატალოგს და
   ასრულებს Merkle-ის მეგობრულ ჰეშინგს ყველა სვეტისთვის (CPU-ის შესაბამისი
   `PoseidonColumnBatch` განლაგება). ის იყენებს იგივე ხრახნიანი ჯგუფის მუდმივ ბუფერს
   როგორც permute kernel, მაგრამ მარყუჟის `(states_per_lane * block_count)`
   გამოდის ისე, რომ ბირთვმა შეძლოს რიგის წარდგენის ამორტიზაცია.
3. `poseidon_trace_fused`: ითვლის მშობლის/ფოთლის დაჯესტს კვალის ცხრილისთვის
   ერთი პასით. შერწყმული ბირთვი მოიხმარს `PoseidonFusedArgs`-ს, ასე რომ, მასპინძელი
   შეუძლია აღწეროს არა მომიჯნავე რეგიონები და `leaf_offset`/`parent_offset` და
   ის იზიარებს ყველა მრგვალ/MDS ცხრილებს სხვა ბირთვებთან.

### ბრძანების დაგეგმვა და მასპინძელი კონტრაქტები- ყველა ბირთვის გაგზავნა გადის `MetalPipelines::command_queue`-ის მეშვეობით, რომელიც
  ახორციელებს ადაპტირებულ განრიგს (სამიზნე ~2 ms) და რიგის გამაძლიერებელი კონტროლი
  ექსპოზიცია `FASTPQ_METAL_QUEUE_FANOUT`-ით და
  `FASTPQ_METAL_COLUMN_THRESHOLD`. გახურების გზა `with_metal_state`-ში
  აგროვებს სამივე პოსეიდონის ბირთვს წინ, ასე რომ პირველი გაგზავნა არ ხდება
  მილსადენის შექმნის ჯარიმის გადახდა.

  8,192 თემა თითო წარდგენაზე, ჯგუფში 256 ძაფების მყარი ქუდით. The
  მასპინძელმა შეიძლება შეამციროს `states_per_lane` მულტიპლიკატორი დაბალი სიმძლავრის მოწყობილობებისთვის

  მილსადენები. პოსეიდონის ბირთვები იღებენ ნედლეულ მაჩვენებლებს ამ დადგმის ბუფერში
