# Iroha v3.0（Nexus プレビュー）

本ドキュメントは、Iroha v2 ホワイトペーパーを補完しつつ、マルチレーンパイプライン、Nexus データスペース、Asset Exchange Toolkit（AXT）に焦点を当てた次世代アーキテクチャをまとめたものです。ここに記載された機能は開発中であり、一般提供までに API やレイアウトが変更される可能性があります。

---

## 1. 概要

Iroha v3 は v2 で確立した決定論的基盤を拡張し、水平スケーラビリティとドメイン横断ワークフローを強化します。リリースコードネーム **Nexus** の主な要素は以下のとおりです。

- **SORA Nexus** と呼ばれる単一のグローバルネットワーク。Iroha v3 のすべてのピアはこの普遍台帳に参加し、各組織は独自ポリシーとプライバシーを維持したまま、独立したデータスペースを登録して共通台帳へアンカーします。
- 共通コードベース：同じリポジトリから Iroha v2（自営ネットワーク）と Iroha v3（SORA Nexus）がビルドされ、設定でモードを切り替えます。Iroha Virtual Machine（IVM）は v2 / v3 共通であり、Kotodama コントラクトやバイトコードアーティファクトは自営ネットワークと Nexus 台帳の双方でそのまま動作します。
- 独立したワークロードを並列処理するマルチレーン型のブロック生成。
- 実行環境を分離しつつオンチェーンアンカーで結合できるデータスペース（DS）。
- データスペース間のアトミックな価値移転とコントラクト制御スワップを可能にする Asset Exchange Toolkit（AXT）。
- 信頼できるブロードキャストコミット（RBC）レーン、決定論的デッドライン、サンプリング予算による信頼性向上。

詳細なエンジニアリング仕様については `nexus.md`、`nexus_transition_notes.md`、`new_pipeline.md` を参照してください。

## 2. マルチレーンアーキテクチャ

- **スケジューラ:** Nexus スケジューラはデータスペース識別子とコンポーザビリティグループに基づいてワークロードをレーンに分割します。各レーンは内部の順序保証を保ちながら並列実行されます。
- **レーングループ:** 関連するデータスペースは共通の `LaneGroupId` を共有し、複数コンポーネントにまたがるワークフロー（例：CBDC データスペースと決済 dApp データスペース）を同期的に処理できます。
- **デッドライン:** 各レーンはブロック、証明、データ可用性などの決定論的デッドラインを追跡し、進捗とリソース使用を制御します。
- **テレメトリ:** レーンレベルのスループット、キュー深度、デッドライン違反、帯域使用量を公開し、ダッシュボードスクリプトで指標の存在をチェックします。

## 3. データスペース（Nexus）

- **分離:** 各データスペースは独自のコンセンサスレーン、World State セグメント、Kura ストレージを保持し、プライバシードメインを確保します。同時に、定期的なアンカーによってグローバルな SORA Nexus 台帳との整合性を保ちます。
- **アンカー:** コミット時にアンカーアーティファクトを生成し、Merkle ルートや証明、コミットメントをまとめてグローバルレーンへ公開します。
- **コンポーザビリティ:** データスペースは相互運用を許可するコンポーザビリティグループを宣言でき、ガバナンスによって参加者変更やアクティベーションエポックを管理します。
- **消失訂正ストレージ:** Kura や WSV スナップショットにはエラージャコーディング `(k, m)` を採用し、決定論を損なわずにデータ可用性をスケールさせます。復旧処理は欠落フラグメントを決定論的に再構築します。

## 4. Asset Exchange Toolkit（AXT）

- **ディスクリプタとバインディング:** クライアントは決定論的な AXT ディスクリプタを構築します。`axt_binding` ハッシュがディスクリプタを個々のエンベロープに結び付け、リプレイを防ぎ、ノードが Norito ペイロードをバイト単位で検証できるようにします。
- **システムコール:** IVM は `AXT_BEGIN`、`AXT_TOUCH`、`AXT_COMMIT` を提供します。コントラクトはデータスペース別の読書き集合を宣言し、ホストはレーンを跨ぐアトミシティを強制します。
- **ハンドルとエポック:** ウォレットは `(dataspace_id, epoch_id, sub_nonce)` にバインドされたケイパビリティハンドルを取得します。同時利用が発生すると決定論的に競合し、規範的な `AxtTrap` コードを返します。
- **証明とデッドライン:** 有効ウィンドウ Δ の間、バリデータは証明、データ可用性サンプル、マニフェストを収集します。期限未達の場合、AXT は決定論的に中断され、クライアントのリトライ手順が提示されます。
- **ガバナンス統合:** ポリシーモジュールが参加可能なデータスペース、ハンドルのレート制限、監査向けマニフェスト（コミットメント、ヌリファイア、イベントログ）を管理します。

## 5. 信頼できるブロードキャストコミット（RBC）レーン

- **レーン固有のデータ可用性:** RBC レーンはレーングループを反映し、マルチレーンパイプラインごとに専用のデータ可用性保証を提供します。
- **サンプリング予算:** バリデータは決定論的なサンプリング規則（`q_in_slot_per_ds`）に従い、証明や証跡を検証します。
- **バックプレッシャー分析:** Sumeragi のペースメーカーイベントと RBC 統計を相関させ、停滞したレーンを診断します（`scripts/sumeragi_backpressure_log_scraper.py` を参照）。

## 6. 運用と移行

- **移行計画:** `nexus_transition_notes.md` は、単一レーン（Iroha v2）からマルチレーン（Iroha v3）への段階的移行を説明し、テレメトリ、設定ゲート、ジェネシス更新をガイドします。
- **ユニバーサルネットワーク:** SORA Nexus のピアは共通ジェネシスとガバナンススタックで稼働します。新規オペレータは独立ネットワークを立ち上げる代わりに、データスペースを作成し、Nexus のアドミッションポリシーを満たして参加します。
- **設定:** レーン予算、証明デッドライン、AXT クォータ、データスペースメタデータなど新しい設定ノブを追加します。デフォルトは保守的に設定され、オペレータが Nexus モードへ明示的に移行するまで v2 と同程度の挙動を保ちます。
- **テスト:** ゴールデンテストで AXT ディスクリプタ、レーンマニフェスト、システムコール一覧を管理します。統合テスト（`integration_tests/tests/repo.rs`、`crates/ivm/tests/axt_host_flow.rs`）でエンドツーエンドフローを検証します。
- **ツール:** `kagami` は Nexus 対応のジェネシス生成をサポートし、ダッシュボードスクリプトはレーンスループット、証明予算、RBC の健全性を検証します。

## 7. ロードマップ

- **フェーズ 1:** 単一ドメインでのマルチレーン実行とローカル AXT サポートを有効化し、監査機能を整備。
- **フェーズ 2:** 許可付きクロスドメイン AXT のためのコンポーザビリティグループを稼働させ、テレメトリカバレッジを拡張。
- **フェーズ 3:** Nexus データスペース連携、消失訂正ストレージ、高度な証明共有を本格展開。

各フェーズのステータスは `roadmap.md` と `status.md` に掲載されます。Nexus 設計に合致するコントリビューションは、v3 で定義された決定論的実行とガバナンスポリシーに従ってください。
