# Kaigi нууцлал ба реле дизайн

Энэхүү баримт бичиг нь 0-мэдлэгийг нэвтрүүлсэн хувийн нууцад төвлөрсөн хувьслыг тусгасан болно.
оролцоо нотлох баримтууд болон сонгины маягийн буухиа золиослолгүйгээр детерминизм эсвэл
дэвтэрт аудит хийх чадвар.

# Тойм

Дизайн нь гурван давхаргыг хамардаг:

- **Жагсаалтын нууцлал** – хостын зөвшөөрөл болон төлбөр тооцоог тогтвортой байлгахын зэрэгцээ гинжин хэлхээнд оролцогчийн хувийн мэдээллийг нуух.
- **Ашиглалтын тунгалаг байдал** – сегмент тус бүрийн дэлгэрэнгүй мэдээллийг нийтэд ил тод зарлахгүйгээр тоолууртай хэрэглээг бүртгэх боломжийг хостуудад олгоно.
- **Overlay relays** – зөөвөрлөх пакетуудыг олон хоп үе тэнгийнхэнээр дамжуулан чиглүүлдэг тул сүлжээний ажиглагчид аль оролцогчид холбогдож байгааг мэдэж чадахгүй.

Бүх нэмэлтүүд нь Norito хэвээр үлдэж, ABI хувилбар 1-ийн дагуу ажилладаг бөгөөд нэг төрлийн бус техник хангамжид тодорхой хэмжээгээр ажиллах ёстой.

#Зорилгууд

1. Бүртгэлийн дэвтэр нь дансны түүхий ID-г хэзээ ч ил гаргахгүйн тулд 0-мэдлэгийн нотолгоог ашиглан оролцогчдыг хүлээн авах/нүүлгэн шилжүүлэх.
2. Нягтлан бодох бүртгэлийн хүчтэй баталгааг хадгалах: нэгдэх, гарах, ашиглалтын үйл явдал бүр тодорхой нийцэж байх ёстой.
3. Хяналтын/өгөгдлийн сувгуудын сонгины маршрутыг тодорхойлсон, гинжин хэлхээнд аудит хийх боломжтой нэмэлт реле манифестуудыг өгөх.
4. Хувийн нууцлалыг шаарддаггүй байршуулалтад зориулж нөөцийг (бүрэн ил тод жагсаалт) ажиллуулах.

# аюулын загварын хураангуй

- **Өрсөлдөгчид:** Сүлжээний ажиглагчид (ISP), сониуч баталгаажуулагч, хорлонтой реле операторууд, хагас шударга хостууд.
- **Хамгаалагдсан хөрөнгө:** Оролцогчийн хувийн мэдээлэл, оролцох хугацаа, сегмент бүрийн хэрэглээ/тооцооны дэлгэрэнгүй мэдээлэл, сүлжээний чиглүүлэлтийн мета өгөгдөл.
- **Таамаглал:** Хөтлөгчид жинхэнэ оролцогчийг сүлжээнээс гадуур сурсан хэвээр байна; дэвтэрийн нөхдүүд нотлох баримтыг тодорхой хэмжээгээр баталгаажуулдаг; давхар реле нь найдвартай биш боловч хурдны хувьд хязгаарлагдмал; HPKE болон SNARK командууд аль хэдийн кодын санд байдаг.

# Өгөгдлийн загварын өөрчлөлт

Бүх төрлийн `iroha_data_model::kaigi`-д амьдардаг.

```rust
/// Commitment to a participant identity (Poseidon hash of account + domain salt).
pub struct KaigiParticipantCommitment {
    pub commitment: FixedBinary<32>,
    pub alias_tag: Option<String>,
}

/// Nullifier unique to each join action, prevents double-use of proofs.
pub struct KaigiParticipantNullifier {
    pub digest: FixedBinary<32>,
    pub issued_at_ms: u64,
}

/// Relay path description used by clients to set up onion routing.
pub struct KaigiRelayManifest {
    pub hops: Vec<KaigiRelayHop>,
    pub expiry_ms: u64,
}

pub struct KaigiRelayHop {
    pub relay_id: AccountId,
    pub hpke_public_key: FixedBinary<32>,
    pub weight: u8,
}
```

`KaigiRecord` дараах талбаруудыг олно.

- `roster_commitments: Vec<KaigiParticipantCommitment>` – нууцлалын горимыг идэвхжүүлсний дараа ил гарсан `participants` жагсаалтыг орлуулна. Сонгодог байршуулалт нь шилжих явцад хоёуланг нь хүн амтай байлгах боломжтой.
- `nullifier_log: Vec<KaigiParticipantNullifier>` – зөвхөн хавсаргах боломжтой, мета өгөгдлийг хязгаартай байлгахын тулд өнхрөх цонхоор таглана.
- `room_policy: KaigiRoomPolicy` – сессийн үзэгчийн нотолгооны байрлалыг сонгоно (`Public` өрөөнүүд зөвхөн уншигдах релейг тусгадаг; `Authenticated` өрөөнүүд пакетуудыг дамжуулахын өмнө үзэгчийн тасалбар шаарддаг).
- `relay_manifest: Option<KaigiRelayManifest>` – Norito-ээр кодлогдсон бүтэцлэгдсэн манифест, ингэснээр үсрэлт, HPKE товчлуурууд болон жин нь JSON жийргэвчгүйгээр каноник хэвээр үлдэнэ.
- `privacy_mode: KaigiPrivacyMode` enum (доороос үзнэ үү).

```rust
pub enum KaigiPrivacyMode {
    Transparent,
    ZkRosterV1,
}
```

`NewKaigi` нь тохирох нэмэлт талбаруудыг хүлээн авдаг тул хостууд үүсгэх үед нууцлалыг сонгох боломжтой.


- Талбарууд нь каноник кодчилолыг хэрэгжүүлэхийн тулд `#[norito(with = "...")]` туслахуудыг ашигладаг (бүхэл тоонуудын хувьд бага-эндиан, хопуудыг байрлалаар нь эрэмбэлсэн).
- `KaigiRecord::from_new` шинэ векторуудыг хоосон болгож, өгсөн реле манифестийг хуулна.

# Зааврын гадаргуугийн өөрчлөлт

## Демо хурдан эхлүүлэх туслах

Түр зуурын үзүүлэн ба харилцан ажиллах чадварыг шалгахын тулд CLI одоо ил болгож байна
`iroha kaigi quickstart`. Энэ нь:- `--domain`/`--host`-ээр дарагдаагүй тохиолдолд CLI тохиргоог (домэйн `wonderland` + данс) дахин ашигладаг.
- `--call-name`-г орхигдуулсан үед цагийн тэмдэгт суурилсан дуудлагын нэрийг үүсгэж, `CreateKaigi`-г идэвхтэй Torii төгсгөлийн цэгийн эсрэг илгээнэ.
- Сонголтоор хосттой (`--auto-join-host`) автоматаар холбогддог тул үзэгчид шууд холбогдох боломжтой.
- Torii URL, дуудлагын таниулбар, нууцлал/өрөөний бодлого, хуулбарлахад бэлэн нэгдэх команд, дамарны замыг шалгагчид хянах ёстой (жишээ нь, `storage/streaming/soranet_routes/exit-<relay-id>/kaigi-stream/*.norito`) агуулсан JSON хураангуйг гаргадаг. Блобыг хадгалахын тулд `--summary-out path/to/file.json`-г ашиглана уу.

Энэ туслах нь ажиллаж байгаа `irohad --sora` зангилааны хэрэгцээг орлохгүй**: нууцлалын чиглүүлэлтүүд, дамар файлууд болон реле манифестууд дэвтэрт тулгуурласан хэвээр байна. Гадны үдэшлэгт зориулсан түр өрөөгүүдийг эргүүлэх үед энэ нь зүгээр л зуухны хавтанг засдаг.

### Нэг тушаалын демо скрипт

Илүү хурдан замд туслах скрипт байдаг: `scripts/kaigi_demo.sh`.
Энэ нь танд дараах үйлдлийг гүйцэтгэдэг.

1. Багцлагдсан `defaults/nexus/genesis.json` дээр `target/kaigi-demo/genesis.nrt` дээр гарын үсэг зурна.
2. `irohad --sora`-г гарын үсэг зурсан блоктой ажиллуулж (`target/kaigi-demo/irohad.log` дор бүртгэлүүд) Torii-г `http://127.0.0.1:8080/status`-г харуулахыг хүлээнэ.
3. `iroha kaigi quickstart --auto-join-host --summary-out target/kaigi-demo/kaigi_summary.json` ажиллуулна.
4. JSON хураангуйд хүрэх замыг болон дамарын лавлах (`storage/streaming/soranet_routes/exit-<relay-id>/kaigi-stream/`) хэвлэх ба ингэснээр та үүнийг гадны шалгагчтай хуваалцах боломжтой.

Хүрээлэн буй орчны хувьсагчид:

- `TORII_URL` — санал асуулга явуулахын тулд Torii төгсгөлийн цэгийг дарна (өгөгдмөл `http://127.0.0.1:8080`).
- `RUN_DIR` — ажлын лавлахыг хүчингүй болгох (анхдагч `target/kaigi-demo`).

`Ctrl+C` товчийг дарж үзүүлэнг зогсоо; скрипт дэх урхи `irohad` автоматаар дуусгавар болно. Дамрын файлууд болон хураангуй нь дискэн дээр үлдэх тул процесс дууссаны дараа олдворуудыг шилжүүлэх боломжтой.

## `CreateKaigi`

- `privacy_mode`-г хостын зөвшөөрлийн эсрэг баталгаажуулна.
- Хэрэв `relay_manifest` нийлүүлэгдсэн бол ≥3 хоп, тэгээс ялгаатай жин, HPKE түлхүүр байгаа эсэх, өвөрмөц байдлыг мөрдүүлэхийн тулд гинжин манифестыг шалгах боломжтой хэвээр үлдээнэ үү.
- SDK/CLI (`public` vs `authenticated`)-аас `room_policy` оролтыг баталгаажуулж, үүнийг SoraNet провизицид шилжүүлснээр релений кэш нь зөв GAR ангиллыг (`stream.kaigi.public`010000) харуулах болно. Хостууд үүнийг `iroha kaigi create --room-policy …`, JS SDK-н `roomPolicy` талбараар эсвэл Swift үйлчлүүлэгчид илгээхээсээ өмнө Norito ачааллыг угсарч байх үед `room_policy` тохиргоогоор дамжуулдаг.
- Хоосон амлалт / хүчингүй болгох бүртгэлүүдийг хадгалдаг.

## `JoinKaigi`

Параметрүүд:

- `proof: ZkProof` (Norito байт боодол) – Грот16 нотолгоо нь залгагч `(account_id, domain_salt)`-ийг мэддэг бөгөөд Посейдон хэш нь нийлүүлсэн `commitment`-тэй тэнцүү байна.
- `commitment: FixedBinary<32>`
- `nullifier: FixedBinary<32>`
- `relay_hint: Option<KaigiRelayHop>` – дараагийн хопод оролцогч тус бүрийг заавал дарах боломжтой.

Гүйцэтгэлийн үе шатууд:

1. Хэрэв `record.privacy_mode == Transparent` бол одоогийн төлөв рүү буцна уу.
2. Groth16 нотолгоог `KAIGI_ROSTER_V1` хэлхээний бүртгэлийн бичилттэй тулгаж шалгана уу.
3. `record.nullifier_log` дээр `nullifier` гарч ирээгүй эсэхийг шалгаарай.
4. Амлалт / хүчингүй болгох бичилтүүдийг хавсаргах; хэрэв `relay_hint` нийлүүлэгдсэн бол энэ оролцогчийн релений манифест харагдацыг нөхнө үү (зөвхөн санах ойн сессийн төлөвт хадгалагдана, сүлжээнд биш).## `LeaveKaigi`

Ил тод горим нь одоогийн логиктой таарч байна.

Хувийн горим нь дараахь зүйлийг шаарддаг.

1. Дуудлага хийгч `record.roster_commitments` дахь амлалтаа мэддэг болохыг нотлох баримт.
2. Нэг удаагийн чөлөөг нотлох хүчингүй болсон шинэчлэлт.
3. Амлалт / хүчингүй болгох бичилтүүдийг арилгах. Аудит нь бүтцийн алдагдалаас зайлсхийхийн тулд булшны чулууг тогтмол хадгалах цонхонд зориулж хадгалдаг.

## `RecordKaigiUsage`

Ачаалах ачааллыг нэмэгдүүлнэ:

- `usage_commitment: FixedBinary<32>` – түүхий ашиглалтын багц (үргэлжлэх хугацаа, хий, сегментийн ID)-ийн амлалт.
- Дельта нь дэвтэрээс гадуур өгсөн шифрлэгдсэн бүртгэлтэй таарч байгааг баталгаажуулах нэмэлт ZK баталгаа.

Хостууд ил тод нийт дүнг илгээх боломжтой хэвээр байна; нууцлалын горим нь зөвхөн амлалтын талбарыг заавал байлгах болно.

# Onion Routing Overlay

## Реле бүртгэл- HPKE түлхүүр материал болон зурвасын өргөнийг багтаасан `kaigi_relay::<relay_id>` домэйны мета өгөгдлийн оруулгуудаар өөрийгөө бүртгэдэг.
- `RegisterKaigiRelay` заавар нь домэйны мета өгөгдлийн тодорхойлогчийг хэвээр үлдээж, `KaigiRelayRegistered` хураангуйг (HPKE хурууны хээ болон зурвасын өргөний ангилалтай) ялгаруулж, түлхүүрүүдийг тодорхой байдлаар эргүүлэхийн тулд дахин дуудаж болно.
- Засаглал нь домэйн мета өгөгдлөөр (`kaigi_relay_allowlist`) зөвшөөрөгдсөн жагсаалтуудыг хянадаг ба бүртгэл/манифест шинэчлэлтүүд нь шинэ замыг хүлээн зөвшөөрөхөөс өмнө гишүүнчлэлийг хэрэгжүүлдэг.

## Ил тод бүтээл

- Хостууд боломжтой релеүүдээс олон хоп зам (хамгийн бага урт 3) байгуулдаг. Манифест нь давхаргын дугтуйг шифрлэхэд шаардагдах AccountIds болон HPKE нийтийн түлхүүрүүдийн дарааллыг кодлодог.
- Сүлжээнд хадгалагдсан `relay_manifest` нь хоп тодорхойлогч, дуусах хугацаа (Norito кодлогдсон `KaigiRelayManifest`) агуулдаг; Бодит түр зуурын түлхүүрүүд болон сесс бүрийн офсетуудыг HPKE ашиглан бүртгэлээс гадуур солилцдог.

## Дохио ба медиа

- SDP/ICE солилцоо Кайги мета өгөгдлөөр үргэлжилдэг боловч нэг хопдоо шифрлэгдсэн байдаг. Баталгаажуулагчид зөвхөн HPKE шифрлэгдсэн текст болон толгойн индексүүдийг хардаг.
- Медиа пакетууд нь битүүмжилсэн ачаалалтай QUIC ашиглан релейгээр дамждаг. Хоп бүр дараагийн хоп хаягийг сурахын тулд нэг давхаргын шифрийг тайлдаг; Эцсийн хүлээн авагч нь бүх давхаргыг арилгасны дараа медиа урсгалыг авдаг.

## Ажилгүйдэл

- Үйлчлүүлэгчид домайны мета өгөгдөлд (`kaigi_relay_feedback::<relay_id>`) гарын үсэг зурсан санал хүсэлтийг хэвээр үлдээж, `KaigiRelayHealthUpdated` цацаж, засаглал/хостуудад одоогийн бэлэн байдлын талаар дүгнэлт хийх боломжийг олгодог `ReportKaigiRelayHealth` заавраар дамжуулан үйлчлүүлэгчид релений эрүүл мэндийг хянадаг. Реле амжилтгүй болоход хост шинэчилсэн манифест гаргаж, `KaigiRelayManifestUpdated` үйл явдлыг бүртгэдэг (доороос харна уу).
- Хостууд `SetKaigiRelayManifest` зааварчилгаагаар дэвтэр дээрх манифест өөрчлөлтүүдийг хэрэгжүүлдэг бөгөөд энэ нь хадгалагдсан замыг солих эсвэл бүрмөсөн арилгадаг. Клиринг нь `hop_count = 0`-ийн хураангуйг гаргадаг тул операторууд шууд чиглүүлэлт рүү буцах шилжилтийг ажиглах боломжтой.
- Prometheus хэмжигдэхүүнүүд (`kaigi_relay_registered_total`, `kaigi_relay_registration_bandwidth_class`, `kaigi_relay_manifest_updates_total`, `kaigi_relay_manifest_hop_count`, `kaigi_relay_health_reports_total`, `kaigi_relay_health_reports_total`, Prometheus, Norito `kaigi_relay_failover_hop_count`) одоо операторын хяналтын самбарт зориулсан релений гацалт, эрүүл мэндийн байдал, бүтэлгүйтлийн хэмнэл зэрэг болно.

# Үйл явдал

`DomainEvent` хувилбаруудыг сунгах:

- `KaigiRosterSummary` – нэрээ нууцалсан тоо болон одоогийн жагсаалтаар ялгарсан
  жагсаалт өөрчлөгдөх бүрд root (root нь ил тод горимд `None`).
- `KaigiRelayRegistered` – реле бүртгэлийг үүсгэх эсвэл шинэчлэх бүрт ялгардаг.
- `KaigiRelayManifestUpdated` – реле манифест өөрчлөгдөх үед ялгардаг.
- `KaigiRelayHealthUpdated` – хостууд `ReportKaigiRelayHealth`-ээр дамжуулан релений эрүүл мэндийн тайлан ирүүлэх үед ялгардаг.
- `KaigiUsageSummary` – ашиглалтын сегмент бүрийн дараа ялгардаг бөгөөд зөвхөн нийт дүнг харуулна.

Үйл явдлуудыг Norito-ээр цувуулж, зөвхөн амлалтын хэш болон тоонуудыг ил гаргадаг.CLI хэрэгсэл (`iroha kaigi …`) нь ISI бүрийг боож өгдөг тул операторууд сешнүүдийг бүртгэх,
жагсаалтын шинэчлэлтийг илгээх, релений эрүүл мэндийг мэдээлэх, гар урлалын гүйлгээгүйгээр ашиглалтын бүртгэл хийх.
Дамжуулсан JSON/hex файлуудаас реле манифест болон нууцлалын баталгааг ачаална
CLI-ийн ердийн илгээх зам нь скриптийн гэрээг хийхэд хялбар болгодог
тайзны орчинд элсүүлэх.

# Хийн нягтлан бодох бүртгэл

- `crates/iroha_core/src/gas.rs` дахь шинэ тогтмолууд:
  - `BASE_KAIGI_JOIN_ZK`, `BASE_KAIGI_LEAVE_ZK`, `BASE_KAIGI_USAGE_ZK`
    Halo2 баталгаажуулах хугацаатай харьцуулахад тохируулсан (бүртгэлийн хувьд ≈1.6ms
    нэгдэх/навчлах, Apple M2 Ultra дээр ашиглахад ≈1.2ms). Нэмэлт хураамж үргэлжилсээр байна
    `PER_KAIGI_PROOF_BYTE`-ээр дамжуулан нотлох байт хэмжээтэй масштаб.
- `RecordKaigiUsage` нь амлалтын хэмжээ болон нотлох баримтын баталгаажуулалтад үндэслэн нэмэлт төлбөр төлдөг.
- Шалгалт тохируулгын бэхэлгээ нь нууц хөрөнгийн дэд бүтцийг тогтмол үрээр дахин ашиглана.

# Туршилтын стратеги

- `KaigiParticipantCommitment`, `KaigiRelayManifest`-ийн Norito кодчилол/декод тайлж буй нэгжийн туршилт.
- Каноник дарааллыг баталгаажуулдаг JSON харагдах алтан тестүүд.
- Мини сүлжээг бий болгох интеграцийн туршилтууд (харна уу
  Одоогийн хамрах хүрээний хувьд `crates/iroha_core/tests/kaigi_privacy.rs`):
  - Хуурамч нотолгоо ашиглан хувийн нэгдэх/хэрэглэх циклүүд (`kaigi_privacy_mocks` онцлог).
  - Мета өгөгдлийн үйл явдлуудаар дамжсан манифест шинэчлэлтүүдийг дамжуулах.
- Хостын буруу тохиргоог (жишээ нь, нууцлалын горимд байхгүй реле манифест) хамарсан UI тестийг туршиж үзээрэй.
- Хязгаарлагдмал орчинд нэгж/интеграцийн туршилтыг ажиллуулах үед (жишээ нь, Codex
  хамгаалагдсан хязгаарлагдмал орчин), Norito холболтыг тойрч гарахын тулд `NORITO_SKIP_BINDINGS_SYNC=1` экспортлох
  синхрончлолын шалгалтыг `crates/norito/build.rs` хэрэгжүүлсэн.

# Нээлттэй асуултууд

- Merkle модны тогтвортой байдлын стратегийг тодорхойл: гинжин хэлхээнд ба гинжин хэлхээнээс гадуур (одоогийн налуу: гинжин хэлхээний үндэс амлалт бүхий гинжнээс гадуурх мод). *(KPG-201-д мөрдсөн.)*
- Релей манифест нь олон зам (нэмэлт замуудыг зэрэг) дэмжих ёстой эсэхийг тодорхойлох. *(KPG-202-д мөрдсөн.)*
- Нэр хүндийг унагахын тулд засаглалыг тодорхой болгох - бидэнд багасгах хэрэгтэй юу эсвэл зүгээр л зөөлөн хориг хэрэгтэй юу? *(KPG-203-д мөрдсөн.)*

`KaigiPrivacyMode::ZkRosterV1`-г үйлдвэрлэлд идэвхжүүлэхийн өмнө эдгээр зүйлийг шийдэх хэрэгтэй.
