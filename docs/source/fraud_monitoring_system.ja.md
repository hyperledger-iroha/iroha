# 不正監視システム

本ドキュメントは、コア台帳に付随する共有不正監視機能の参照設計をまとめたものです。目的は、決済事業者（PSP）が各トランザクションに対して高品質なリスクシグナルを得つつ、カストディやプライバシー、ポリシー判断を決済エンジン外の指定オペレータが制御できるようにすることです。

## 目標と成功指標

- 決済エンジンを通過するすべての支払いについて、リアルタイムな不正リスク評価を提供（95 パーセンタイル <120ms、中央値 <40ms）。
- 中央サービスが個人情報（PII）を扱わず、仮名化された識別子と行動テレメトリのみを取得することでユーザープライバシーを保護。
- PSP がそれぞれ自律性を保ちつつ共有インテリジェンスを照会できるマルチ PSP 環境をサポート。
- 監視対象の攻撃パターンに対し、教師あり・教師なしモデルを継続的に適応させながら、台帳上の決定性は保持。
- 規制当局や第三者レビュアーのために、ウォレットや相手方の秘匿性を損なうことなく監査可能な意思決定履歴を提供。

## スコープ

- **対象**: トランザクションリスクスコアリング、行動分析、PSP 間相関、異常検知アラート、ガバナンスフック、PSP 向け統合 API。
- **対象外**: 直接的な執行（PSP の責務）、制裁スクリーニング（既存コンプライアンスパイプラインが扱う）、本人確認（エイリアス管理がカバー）。

## 機能要件

1. **トランザクションスコアリング API**: PSP が決済エンジンへ送信する前に呼び出す同期 API。リスクスコア、カテゴリ判定、根拠となる特徴量を返す。
2. **イベント取り込み**: 決済結果、ウォレットライフサイクルイベント、デバイスフィンガープリント、PSP レベルの不正フィードバックを継続的に受け取るストリーム。
3. **モデルライフサイクル管理**: オフライン学習、シャドーデプロイ、段階的ロールアウト、ロールバックを備えたバージョン管理。すべての特徴量に対して決定的なフォールバックヒューリスティックを用意。
4. **フィードバックループ**: PSP が確定した不正事案、誤検知（false positive）、是正メモを送信可能。フィードバックを特徴量と突合し、分析に反映。
5. **プライバシーコントロール**: 保管・送信するデータはすべてエイリアスベース。生のアイデンティティ情報を含むリクエストは拒否しログ化。
6. **ガバナンス報告**: 集計メトリクス（PSP ごとの検知件数、類型、レスポンスレイテンシ）の定期エクスポートと、認可された監査人向けの随時調査 API。
7. **レジリエンス**: 最低 2 拠点でのアクティブ／アクティブ構成、自動キュードレイン＆リプレイ。サービス劣化時、PSP はローカルルールへフォールバックしつつ台帳処理を止めない。

## 非機能要件

- **決定性と一貫性**: リスクスコアは PSP の意思決定支援に留まり、台帳の実行を変更しない。台帳コミットはノード間で決定的。
- **スケーラビリティ**: 水平スケールとメッセージパーティションにより、1 秒あたり 1 万件以上のリスク評価に耐える。
- **可観測性**: `fraud.scoring_latency_ms`, `fraud.risk_score_distribution`, `fraud.api_error_rate`, `fraud.model_version_active` などのメトリクスと、各スコアリング呼び出しの構造化ログを公開。
- **セキュリティ**: PSP と中央サービス間の双方向 TLS、応答エンベロープ署名用ハードウェアセキュリティモジュール、改ざん検知可能な監査ログ。
- **コンプライアンス**: AML/CFT 要件に準拠し、保持期間を設定可能。証拠保存ワークフローと統合。

## アーキテクチャ概要

1. **API ゲートウェイ層**
   - 認証済み HTTP/JSON API でスコアリングとフィードバックを受け付け。
   - Norito コーデックによるスキーマ検証と、PSP ID ごとのレート制限を実施。

2. **特徴量集約サービス**
   - 過去の集計データ（トランザクション速度、地理的パターン、デバイス利用状況）を時系列特徴量ストアから結合。
   - 分析ウィンドウは分／時／日単位など設定可能で、決定的な集計関数を使用。

3. **リスクエンジン**
   - 稼働中のモデルパイプライン（勾配ブースティング木、異常検知器、ルール群）を実行。
   - モデルスコアが利用不可な場合に備えて決定的なフォールバックルールセットを保持。
   - スコア、リスク帯、寄与特徴量、モデルバージョンを含む `FraudAssessment` エンベロープを出力。

## スコアリングモデルとヒューリスティック

- **スコアスケールと帯域**: スコアは 0–1000 に正規化。帯域は `0–249`（低）、`250–549`（中）、`550–749`（高）、`750+`（クリティカル）。各帯域は PSP 向け推奨アクション（自動承認、追加認証、レビュー待ち、自動拒否）に対応。ただし実施判断は PSP が行う。
- **モデルアンサンブル**:
  - 勾配ブースティング木: 金額、エイリアス／デバイス速度、マーチャントカテゴリ、認証強度、PSP 信頼レベル、ウォレットグラフ特徴量などの構造化特徴量を入力。
  - オートエンコーダを用いた異常検知や、異常スコアによるアラート。
  - ルールベースのヒューリスティック（過剰再利用の検知など）を組み合わせてディフェンスインデプスを実現。

## アナリティクスとモデルプロモーション

- **異常検知**: テナント（PSP）単位の意思決定率の逸脱を監視するストリーミングジョブを維持。アラートをガバナンスダッシュボードに連携し、四半期レビュー用にサマリを保存。
- **グラフ分析**: リレーショナルデータのエクスポートに対して夜間バッチでグラフトラバーサルを実行し、共謀クラスタを特定。`GovernanceExport` 経由で証拠リンクとともにポータルへ公開。
- **フィードバック取り込み**: 手動審査結果やチャージバックレポートを整理し、特徴量差分へ変換して学習データセットに反映。取り込み状況メトリクスを公開して停滞を検出。
- **モデルプロモーションパイプライン**: 候補モデルの評価（オフライン指標、カナリースコアリング、ロールバック準備）を自動化。プロモーション時には署名付き `FraudAssessment` サンプルセットを発行し、`GovernanceExport` の `model_version` を更新。

## 監査ワークフロー

1. 最新の `GovernanceExport` をスナップショットし、`policy_digest` がリスクチーム提供のマニフェストと一致するか検証。
2. サンプリング期間におけるルール集計が台帳側の意思決定総数と一致するか確認。
3. 異常検知およびグラフ分析のレポートから未解決事項をレビューし、エスカレーションと担当者を記録。
4. レビューチェックリストへ署名・保管。Norito でエンコードした成果物をガバナンスポータルに保存し再現性を確保。

## フォールバックプレイブック

- **リスクエンジン障害**: 60 秒超でリスクエンジンが利用不可となった場合、ゲートウェイはレビュー専用モードへ切り替え、すべてのリクエストに `AssessmentDecision::Review` を返し、オペレータへアラート。
- **テレメトリ欠損**: メトリクスやトレースが 5 分間欠落した場合、自動モデルプロモーションを停止しオンコールエンジニアへ通知。
- **モデル退行**: デプロイ後のフィードバックで不正被害が増加している場合、前の署名済みモデルバンドルにロールバックし、是正アクションをロードマップへ追記。

## データ共有契約

- 管轄地域ごとに保持期間・暗号化・インシデント通知 SLA をまとめた付録を維持。`FraudAssessment` エクスポートを受け取る前にパートナーが署名する必要がある。
- 各インテグレーションにおいてデータ最小化の実践（アカウント ID のハッシュ化、カード番号のトランケーションなど）を文書化。
- 規制要件が変更された場合、または年次で合意を更新。

## レッドチーム演習

- 四半期ごとに演習を実施。次回は **2026-01-15** に予定されており、特徴量ポイズニング、リプレイ増幅、署名偽造の試みを含む。
- 発見事項は不正脅威モデルに記録し、`roadmap.md` の「Fraud & Telemetry Governance Loop」ワークストリームへタスクを追加。

## API スキーマ

ゲートウェイは `crates/iroha_data_model::fraud` に定義された Norito 型と 1 対 1 に対応する JSON エンベロープを公開しています。

- **Risk intake** – `POST /v1/fraud/query` は `RiskQuery` スキーマを受理:
  - `query_id` (`[u8; 32]`, hex エンコード)
  - `subject` (`AccountId`, `domainless encoded literal; canonical I105 only (non-canonical I105 literals rejected)`)
  - `operation` (`RiskOperation` に対応するタグ付き enum。JSON の `type` は変種名と一致)
  - `related_asset` (`AssetId`, 任意)
  - `features`（`FeatureInput` からマッピングされた `{ key: String, value_hash: hex32 }` の配列）
  - `issued_at_ms` (`u64`)
  - `context` (`RiskContext`。`tenant_id`, 任意の `session_id`, 任意の `reason`)
- **Risk decision** – `POST /v1/fraud/assessment` は `FraudAssessment` ペイロードを受理（ガバナンスエクスポートでも使用）:
  - `query_id`, `engine_id`, `risk_score_bps`, `confidence_bps`, `decision` (`AssessmentDecision` enum), `rule_outcomes`（`{ rule_id, score_delta_bps, rationale? }` の配列）
  - `generated_at_ms`
  - `signature`（任意。Norito エンコード済みアセスメントを Base64 でラップ）
- **Governance export** – `GET /v1/fraud/governance/export` は `GovernanceExport` 構造体を返し、活性パラメータ、最新の施行、モデルバージョン、ポリシーダイジェスト、`DecisionAggregate` ヒストグラムを束ねる（`governance` フィーチャが有効な場合）。

`crates/iroha_data_model/src/fraud/types.rs` のラウンドトリップテストにより、これらスキーマが Norito コーデックとバイナリ互換であることが保証され、`integration_tests/tests/fraud_monitoring_requires_assessment_bands.rs` では intake／decision パイプライン全体のエンドツーエンドが検証されています。

## PSP SDK 参照資料

以下は PSP 向け連携例を追跡する言語別スタブです。

- **Rust** – `integration_tests/tests/fraud_monitoring_requires_assessment_bands.rs` はワークスペース内の `iroha` クライアントを使って `RiskQuery` メタデータを生成し、受付の成功／失敗を検証。
- **TypeScript** – `docs/source/governance_api.md` が PSP デモダッシュボードで使用する軽量 Torii ゲートウェイの REST サーフェスを記述。スモークドリル用のスクリプトクライアントは `scripts/ci/schedule_fraud_scoring.sh` に配置。
- **Swift & Kotlin** – 既存 SDK（`IrohaSwift` および `crates/iroha_cli/docs/multisig.md` で触れられるもの）が `fraud_assessment_*` フィールドを付与する Torii メタデータフックを提供。PSP 固有ヘルパーは `status.md` の「Fraud & Telemetry Governance Loop」マイルストーンで追跡され、これら SDK のトランザクションビルダを再利用。

これらの参照資料はマイクロサービスゲートウェイと同期され、PSP 実装者が常に最新のスキーマと各言語向けサンプルコードを参照できるよう保守されます。
