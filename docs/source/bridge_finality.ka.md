# ხიდის საბოლოო მტკიცებულება

ეს დოკუმენტი აღწერს Iroha-ის ხიდის საბოლოო მტკიცებულების საწყის ზედაპირს.
მიზანია გარე ჯაჭვებს ან მსუბუქ კლიენტებს დაადასტურონ, რომ Iroha ბლოკი
დასრულებულია ჯაჭვის გარეთ გამოთვლის ან სანდო რელეების გარეშე.

## დადასტურების ფორმატი

`BridgeFinalityProof` (Norito/JSON) შეიცავს:

- `height`: ბლოკის სიმაღლე.
- `chain_id`: Iroha ჯაჭვის იდენტიფიკატორი ჯაჭვის ჯვარედინი გამეორების თავიდან ასაცილებლად.
- `block_header`: კანონიკური `BlockHeader`.
- `block_hash`: ჰედერის ჰეში (კლიენტები ხელახლა გამოთვლიან დასადასტურებლად).
- `commit_certificate`: ვალიდატორის ნაკრები + ხელმოწერები, რომლებმაც დაასრულეს ბლოკი.
- `validator_set_pops`: საკუთრების დამადასტურებელი ბაიტები, რომლებიც შეესაბამება ვალიდიატორის კომპლექტს
  შეკვეთა (აუცილებელია BLS აგრეგატის შემოწმებისთვის).

მტკიცებულება თვითმყოფადია; არ არის საჭირო გარეგანი მანიფესტაციები ან გაუმჭვირვალე ბლოგები.
შენახვა: Torii ემსახურება საბოლოო მტკიცებულებებს ბოლო ვალდებულების სერტიფიკატის ფანჯრისთვის
(ესაზღვრება კონფიგურირებული ისტორიის ქუდით; ნაგულისხმევია 512 ჩანაწერი მეშვეობით
`sumeragi.commit_cert_history_cap` / `SUMERAGI_COMMIT_CERT_HISTORY_CAP`). კლიენტები
უნდა ქეში ან წამყვანი მტკიცებულებები, თუ მათ სჭირდებათ უფრო გრძელი ჰორიზონტები.
კანონიკური ტოპი არის `(block_header, block_hash, commit_certificate)`:
ჰედერის ჰეში უნდა ემთხვეოდეს commit სერტიფიკატის შიგნით არსებულ ჰეშს და
ჯაჭვის id აკავშირებს მტკიცებულებას ერთ დავთართან. სერვერები უარყოფენ და შედიან ა
`CommitCertificateHashMismatch` როდესაც სერტიფიკატი მიუთითებს სხვა ბლოკზე
ჰაში.

## ვალდებულებების ნაკრები

`BridgeFinalityBundle` (Norito/JSON) აფართოებს ძირითად მტკიცებულებას აშკარად
ვალდებულება და გამართლება:

- `commitment`: `{ chain_id, authority_set { id, validator_set, validator_set_hash, validator_set_hash_version }, block_height, block_hash, mmr_root?, mmr_leaf_index?, mmr_peaks?, next_authority_set? }`
- `justification`: ხელმოწერები უფლებამოსილი ორგანოსგან, რომელიც დადგენილია ვალდებულების შესახებ
  payload (ხელახლა იყენებს ვალდებულების სერტიფიკატის ხელმოწერებს).
- `block_header`, `commit_certificate`: იგივე, რაც ძირითადი მტკიცებულება.

ამჟამინდელი ჩანაცვლების ადგილი: `mmr_root`/`mmr_peaks` მიღებულია ხელახალი გამოთვლით
ბლოკ-ჰეშ MMR მეხსიერებაში; ჩართვის მტკიცებულებები ჯერ არ დაბრუნებულა. კლიენტებს შეუძლიათ
დღესაც გადაამოწმეთ იგივე ჰეში ვალდებულებების დატვირთვის მეშვეობით.

MMR მწვერვალები დალაგებულია მარცხნიდან მარჯვნივ. ხელახლა გამოთვალეთ `mmr_root` მწვერვალების შეფუთვით
მარჯვნიდან მარცხნივ: `root = H(p_n, H(p_{n-1}, ... H(p_1, p_0)))`.

API: `GET /v1/bridge/finality/bundle/{height}` (Norito/JSON).

დადასტურება არის ძირითადი მტკიცებულების ანალოგი: ხელახლა გამოთვალეთ `block_hash`
სათაური, გადაამოწმეთ ვალდებულების სერტიფიკატის ხელმოწერები და შეამოწმეთ ვალდებულება
ველები ემთხვევა სერტიფიკატს და ბლოკის ჰეშს. პაკეტი ამატებს ვალდებულებას/
დასაბუთების შეფუთვა ხიდის პროტოკოლებისთვის, რომლებიც უპირატესობას ანიჭებენ განცალკევებას.

## გადამოწმების ნაბიჯები1. ხელახლა გამოთვალეთ `block_hash` `block_header`-დან; უარყოფა შეუსაბამობაზე.
2. შეამოწმეთ `commit_certificate.block_hash` ემთხვევა ხელახლა გამოთვლილ `block_hash`-ს;
   უარყოს შეუსაბამო სათაური/დასრულება სერტიფიკატის წყვილები.
3. შეამოწმეთ `chain_id` ემთხვევა მოსალოდნელ Iroha ჯაჭვს.
4. ხელახლა გამოთვალეთ `validator_set_hash` `commit_certificate.validator_set`-დან და
   შეამოწმეთ, რომ ემთხვევა ჩაწერილ ჰეშს/ვერსიას.
5. დარწმუნდით, რომ `validator_set_pops` სიგრძე ემთხვევა ვალიდატორის კომპლექტს და შეამოწმეთ
   თითოეული PoP მისი BLS საჯარო გასაღების წინააღმდეგ.
6. გადაამოწმეთ ხელმოწერები commit სერტიფიკატში ჰედერის ჰეშის გამოყენებით
   მითითებულ ვალიდატორის საჯარო გასაღებები და ინდექსები; კვორუმის აღსრულება
   (`2f+1` როდესაც `n>3`, სხვა შემთხვევაში `n`) და უარყოთ დუბლიკატი/დიაპაზონის გარეთ ინდექსები.
7. სურვილისამებრ დაუკავშირდით სანდო საკონტროლო პუნქტს ვალიდატორის ნაკრების ჰეშის შედარებით
   დამაგრებულ მნიშვნელობამდე (სუსტი სუბიექტურობის წამყვანი).
8. სურვილისამებრ მიამაგრეთ მოსალოდნელი ეპოქის წამყვანს, რათა დაამტკიცოთ ძველი/ახალი
   ეპოქები უარყოფილია მანამ, სანამ წამყვანმა განზრახ არ შემობრუნდება.

`BridgeFinalityVerifier` (`iroha_data_model::bridge`-ში) იყენებს ამ შემოწმებებს,
ჯაჭვის ID/სიმაღლის დრიფტის უარყოფა, ვალიდატორის ნაკრები ჰეშის/ვერსიის შეუსაბამობა, გამოტოვებული
ან არასწორი PoP-ები, დუბლიკატი/სასაზღვრო ხელმომწერები, არასწორი ხელმოწერები და
მოულოდნელი ეპოქები კვორუმის დათვლამდე, რათა მსუბუქ კლიენტებს შეეძლოთ ერთის ხელახლა გამოყენება
შემმოწმებელი.

## მითითების დამადასტურებელი

`BridgeFinalityVerifier` იღებს მოსალოდნელ `chain_id` პლუს არჩევით სანდოს
Validator-set და epoch anchors. ის ახორციელებს ჰედერს/ბლოკ-ჰაში/
commit-certificate tuple, ამოწმებს ვალიდატორის ნაკრები ჰეშის/ვერსიას, ამოწმებს
ხელს აწერს/ქვორუმს რეკლამირებული ვალიდატორების სიაში და აკონტროლებს უახლესს
სიმაღლე უარყოს შემორჩენილი/გამოტოვებული მტკიცებულებები. როდესაც წამყვანები მიეწოდება, ის უარყოფს
გამეორება ეპოქებში/რსტერებში გამოკვეთილი `UnexpectedEpoch`/
`UnexpectedValidatorSet` შეცდომები; წამყვანების გარეშე იღებს პირველ მტკიცებულებებს
ვალიდიატორის ნაკრები ჰეში და ეპოქა, სანამ გააგრძელებთ დუბლიკატის/გარეშე
დიაპაზონი/არასაკმარისი ხელმოწერები დეტერმინისტული შეცდომებით.

## API ზედაპირი

- `GET /v1/bridge/finality/{height}` - აბრუნებს `BridgeFinalityProof`-ს
  მოთხოვნილი ბლოკის სიმაღლე. კონტენტზე მოლაპარაკება `Accept`-ის საშუალებით მხარს უჭერს Norito ან
  JSON.
- `GET /v1/bridge/finality/bundle/{height}` – აბრუნებს `BridgeFinalityBundle`
  (ვალდებულება + დასაბუთება + სათაური/სერთიფიკატი) მოთხოვნილ სიმაღლეზე.

## შენიშვნები და შემდგომი დაკვირვებები

- მტკიცებულებები ამჟამად მიღებულია შენახული ვალდებულების სერთიფიკატებიდან. შემოსაზღვრული
  ისტორია მიჰყვება ვალდებულების სერტიფიკატის შენახვის ფანჯარას; კლიენტებმა უნდა ქეში
  წამყვანი მტკიცებულებები, თუ მათ სჭირდებათ უფრო გრძელი ჰორიზონტები. ფანჯრის მიღმა დაბრუნების მოთხოვნა
  `CommitCertificateNotFound(height)`; დააფიქსირე შეცდომა და დაბრუნდი ან
  დამაგრებული საგუშაგო.
- ხელახალი ან გაყალბებული მტკიცებულება შეუსაბამო `block_hash`-ით (header vs.
  მოწმობა) უარყოფილია `CommitCertificateHashMismatch`-ით; კლიენტებმა უნდა
  შეასრულეთ იგივე ორმხრივი შემოწმება ხელმოწერის დადასტურებამდე და გააუქმეთ
  შეუსაბამო დატვირთვები.
- მომავალმა მუშაობამ შეიძლება დაამატოს MMR/ავტორიტეტის მიერ დაყენებული ვალდებულებების ჯაჭვები მტკიცებულების ზომის შესამცირებლად
  ვალდებულების მოწმობა უფრო მდიდარი ვალდებულების კონვერტებში.
