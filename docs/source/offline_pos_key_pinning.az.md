# Offline POS Key Pinning & Rotation (OA12 Draft)

The OA12 track introduces hardened POS clients that **pin backend keys** and rehearse
rotation before regulators sign off on bearer deployments. This runbook captures the
current tooling so Android/iOS POS teams can prepare while the final SDK plumbing lands.

## 1. Generate signed manifests (operators)

1. Prepare a manifest spec describing the backend roots, validity window, and optional
   rotation hint. The existing helper accepts JSON; see
   `scripts/offline_pos_provision/spec.example.json` for a working template.
2. Run:
   ```bash
   cargo xtask offline-pos-provision \
     --spec scripts/offline_pos_provision/spec.example.json \
     --output artifacts/offline_pos_provision
   ```
   The command emits `manifest.json`, `manifest.norito`, and a `summary.json` entry for every
   `OfflinePosProvisionManifest`, plus optional `revocation` bundles when the spec supplies
   deny-lists.
3. Archive the generated `pos_provision_fixtures.manifest.json` alongside the manifests. The POS
   clients will hash the Norito bytes and compare against this manifest before pinning a key.

## 2. Verify manifests on POS devices

Devices (or the merchant staging app) should import the manifest via an offline channel,
then verify:

1. The operator account matches the expected controller (enforced via `operator` field).
2. `published_at_ms` / `valid_{from,until}_ms` fall inside the treasury-approved window.
3. The operator signature over `OfflinePosProvisionManifestPayload` matches the account’s
   public key. The Norito payload bytes are now embedded in `manifest.json` under the
   `payload_base64` field, allowing POS apps to decode the canonical bytes and verify the
   signature without needing to reimplement the codec.
4. Each `OfflinePosBackendRoot` entry’s `public_key` is pinned locally until the manifest
   expires or the `rotation_hint_ms` reminder triggers.

## 3. Rotation drills & revocation bundles

- Operators can stage rotations by publishing the next manifest (`sequence` incremented) and
  distributing it alongside the current one. POS clients should alert when `rotation_hint_ms`
  approaches and require the new manifest before `valid_until_ms` lapses.
- Revocation bundles generated via `revocation_bundles` in the spec are distributed the same
  way. POS devices should verify `OfflineVerdictRevocationBundlePayload`, apply the deny-list,
  and log the bundle id for compliance.

### Automation cheatsheet

- `scripts/offline_pos_provision/rotation_drill.sh --spec <spec.json> --output artifacts/offline_pos_demo --notes "merchants rotation drill"`
  wraps `cargo xtask offline-pos-provision` and appends a timestamped entry to
  `<output>/rotation_drill.log`. Drop this log in the regulator packet to prove drills were
  rehearsed on schedule.
- `cargo xtask offline-pos-verify --bundle artifacts/offline_pos_demo/revocations.json`
  replays the `OfflineVerdictRevocationBundle` signatures emitted by the helper so POS teams can
  cross-check deny-lists before importing them onto devices.

## 4. Sample client readiness (OA12.b)

- The Android retail wallet sample (`examples/android/retail-wallet`) now ships `pos_manifest.json`,
  a manifest loader, and a manifest status card that enforces dual-signature coverage, surfaces
  rotation hints, and writes audit entries via `pos_security_audit.log`. See
  `SecurityPolicyLoader`, `PosManifestLoader`, and `PosAuditLogger` for reference.
- The Swift Norito demo (`examples/ios/NoritoDemo`) mirrors the same manifest fixture under
  `Sources/Resources/pos_manifest.json` and renders the warnings/backend-root inventory so iOS POS
  teams can rehearse the same drills before the final SDK plumbing arrives.
- Both samples reuse the helper artefacts generated by `cargo xtask offline-pos-provision`,
  keeping OA12 rehearsals reproducible across platforms. Attach the manifest screenshots +
  `pos_security_audit.log` when regulators request rotation evidence.

## 4. Android SDK / POS integration notes

- Android reference POS clients should import the Norito bytes from `manifest.norito`, pin the
  keys via `OfflinePosBackendRoot`, and surface rotation countdowns to operators.
- When OA12 wiring lands in the SDK, these manifests will be referenced directly; until then,
  the JSON artifacts + this runbook provide enough structure for regulators to audit key pinning
  rehearsals.

Refer back to this document whenever OA12 stories mention “key pinning” or “rotation drills” to
ensure the manifests, proofs, and revocation bundles stay in sync.
