% FastPQ Transfer гаджетінің дизайны

# Шолу

Ағымдағы FASTPQ жоспарлаушысы `TransferAsset` нұсқаулығына қатысты әрбір қарапайым операцияны жазады, яғни әрбір аударым баланс арифметикасы, хэш раундтары және SMT жаңартулары үшін бөлек төлейді. Тасымалдаудағы жолдарды азайту үшін хост канондық күйге көшуді орындауды жалғастырған кезде тек минималды арифметикалық/міндеттеме тексерулерін тексеретін арнайы гаджетті енгіземіз.

- **Қолдану аясы**: бар Kotodama/IVM `TransferAsset` жүйе беті арқылы шығарылатын бір реттік тасымалдаулар мен шағын партиялар.
- **Мақсат**: іздеу кестелерін ортақ пайдалану және әр тасымалдау арифметикасын ықшам шектеу блогына жию арқылы үлкен көлемді тасымалдаулар үшін FFT/LDE бағанының ізін кесіңіз.

# Сәулет

```
Kotodama builder → IVM syscall (transfer_v1 / transfer_v1_batch)
          │
          ├─ Host (unchanged business logic)
          └─ Transfer transcript (Norito-encoded)
                   │
                   └─ FASTPQ TransferGadget
                        ├─ Balance arithmetic block
                        ├─ Poseidon commitment check
                        ├─ Dual SMT path verifier
                        └─ Authority digest equality
```

## Транскрипт пішімі

Хост жүйені шақыру үшін `TransferTranscript` шығарады:

```rust
struct TransferTranscript {
    batch_hash: Hash,
    deltas: Vec<TransferDeltaTranscript>,
    authority_digest: Hash,
    poseidon_preimage_digest: Option<Hash>,
}

struct TransferDeltaTranscript {
    from_account: AccountId,
    to_account: AccountId,
    asset_definition: AssetDefinitionId,
    amount: Numeric,
    from_balance_before: Numeric,
    from_balance_after: Numeric,
    to_balance_before: Numeric,
    to_balance_after: Numeric,
    from_merkle_proof: Option<Vec<u8>>,
    to_merkle_proof: Option<Vec<u8>>,
}
```

- `batch_hash` қайта ойнатудан қорғау үшін транскриптті транзакцияның кіру нүктесі хэшіне байланыстырады.
- `authority_digest` – сұрыпталған қол қоюшылар/кворум деректері бойынша хост хэші; гаджет теңдікті тексереді, бірақ қолтаңбаны тексеруді қайта жасамайды. Нақтырақ айтқанда Norito хосты `AccountId` (канондық мультисиг контроллерін ендіреді) кодтайды және `b"iroha:fastpq:v1:authority|" || encoded_account` хэштерін Blake2b-256 көмегімен жасайды, нәтижесінде алынған Kotodama.
- `poseidon_preimage_digest` = Poseidon( || шоттан_шоттан || активке || сома || пакеттік_хэш); гаджеттің хост сияқты дайджестті қайта есептеуін қамтамасыз етеді. Алдын ала кескін байттары `norito(from_account) || norito(to_account) || norito(asset_definition) || norito(amount) || batch_hash` ретінде оларды ортақ Poseidon2 көмекшісі арқылы өткізбес бұрын жалаң Norito кодтау арқылы құрастырылады. Бұл дайджест бір дельталық транскрипттер үшін бар және көп дельталық топтамалар үшін қабылданбайды.

Барлық өрістер Norito арқылы серияланады, сондықтан бар детерминизм кепілдіктері сақталады.
`from_path` және `to_path` екеуі де Norito блоктары ретінде шығарылады.
`TransferMerkleProofV1` схемасы: `{ version: 1, path_bits: Vec<u8>, siblings: Vec<Hash> }`.
Болашақ нұсқалар сұлбаны кеңейте алады, ал провер нұсқа тегін бекітеді
декодтау алдында. `TransitionBatch` метадеректері Norito кодталған транскрипцияны енгізеді
`transfer_transcripts` пернесінің астындағы вектор, дәлелдеуші куәгерді декодтауы үшін
диапазоннан тыс сұрауларды орындамай. Жалпы енгізулер (`dsid`, `slot`, түбірлер,
`perm_root`, `tx_set_hash`) `FastpqTransitionBatch.public_inputs` ішінде тасымалданады,
енгізу хэш/транскрипт есебін есепке алу үшін метадеректер қалдыру. Хост сантехникасына дейін
lands, провер синтетикалық жолмен кілт/баланс жұптарынан дәлелдер шығарады, сондықтан жолдар
транскрипт қосымша өрістерді өткізіп жіберсе де, әрқашан детерминирленген SMT жолын қамтиды.

## Гаджет орналасуы

1. **Баланстың арифметикалық блогы**
   - Кірістер: `from_balance_before`, `amount`, `to_balance_before`.
   - Тексерулер:
     - `from_balance_before >= amount` (ортақ RNS декомпозициясы бар ауқым гаджеті).
     - `from_balance_after = from_balance_before - amount`.
     - `to_balance_after = to_balance_before + amount`.
   - Барлық үш теңдеу бір жол тобын тұтынатындай етіп реттелетін қақпаға салынған.2. **Посейдон міндеттеме блогы**
   - `poseidon_preimage_digest` басқа гаджеттерде бұрыннан қолданылған ортақ Poseidon іздеу кестесін пайдаланып қайта есептейді. Посейдон раундтары транзакцияда жоқ.

3. **Merkle Path Block**
   - Қолданыстағы Kaigi SMT гаджетін «жұптастырылған жаңарту» режимімен кеңейтеді. Екі жапырақ (жіберуші, қабылдаушы) қайталанатын жолдарды азайта отырып, бауырлас хэштер үшін бір бағанды ​​ортақ пайдаланады.

4. **Authority Digest Check**
   - хост ұсынған дайджест пен куәгер мәні арасындағы қарапайым теңдік шектеуі. Қолтаңбалар өздерінің арнайы гаджетінде қалады.

5. **Пакеттік цикл**
   - Бағдарламалар `transfer_asset` құрастырушылар циклінің алдында `transfer_v1_batch_begin()` және одан кейін `transfer_v1_batch_end()` шақырады. Аумақ белсенді болған кезде хост әрбір тасымалдауды буферлейді және оларды бір `TransferAssetBatch` ретінде қайта ойнатады, Poseidon/SMT мәтінмәнін әр пакетте бір рет қайта пайдаланады. Әрбір қосымша дельта тек арифметикалық және екі жапырақ тексеруін қосады. Транскрипт декодері қазір көп дельталық топтамаларды қабылдайды және оларды `TransferGadgetInput::deltas` ретінде көрсетеді, осылайша жоспарлаушы Norito қайта оқымай-ақ куәгерлерді бүктей алады. Norito пайдалы жүктемесі бар келісім-шарттар (мысалы, CLI/SDK) хостқа бір жүйеде толық кодталған топтаманы беретін `transfer_v1_batch_apply(&NoritoBytes<TransferAssetBatch>)` телефонына қоңырау шалу арқылы ауқымды толығымен өткізіп жіберуі мүмкін.

Kotodama пайдалану мысалы:

```text
fn pay(a: AccountId, b: AccountId, asset: AssetDefinitionId, x: int) {
    transfer_batch((a, b, asset, x), (b, a, asset, 1));
}
```

`TransferAssetBatch` жеке `Transfer::asset_numeric` шақырулары сияқты бірдей рұқсат пен арифметикалық тексерулерді орындайды, бірақ жалғыз `TransferTranscript` ішіндегі барлық дельталарды жазады. Көп дельталық транскрипттер посейдон дайджестін әр дельта бойынша міндеттемелер кейінге қалдырмайынша жоққа шығарады. Kotodama құрастырушы енді автоматты түрде бастау/аяқтау жүйелік қоңырауларын шығарады, сондықтан келісім-шарттар Norito пайдалы жүктемелерін қолмен кодтаусыз пакеттік тасымалдауларды қолдана алады.

# Ашық сұрақтар

- **Домен шектеулері**: ағымдағы FFT жоспарлаушы 2¹⁴ жолдан кейінгі іздер үшін дүрбелең. TF-2 домен өлшемін ұлғайтуы керек немесе азайтылған эталондық мақсатты құжаттауы керек.
- **Көп активті топтамалар**: бастапқы гаджет әр delta үшін бірдей актив идентификаторын қабылдайды. Егер бізге гетерогенді топтамалар қажет болса, активтердің қайталануын болдырмау үшін Poseidon куәгерінің активті әр уақытта қосуына көз жеткізуіміз керек.
- **Өкімет дайджестін қайта пайдалану**: әр жүйеге қол қоюшылар тізімдерін қайта есептеуді болдырмау үшін ұзақ мерзімді біз сол дайджестті басқа рұқсат етілген операциялар үшін қайта пайдалана аламыз.


Бұл құжат жобалау шешімдерін қадағалайды; межелер жерге түскен кезде оны жол картасының жазбаларымен синхрондаңыз.
