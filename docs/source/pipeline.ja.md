# Iroha v2 トランザクション処理パイプライン

このドキュメントは、トランザクションが Torii への送信から最終コミット、コミット後の副作用までどのように流れるかを、ノード内部とネットワーク全体の観点で説明します。アーキテクチャ上のステップ、関与するコンポーネント、およびトランザクション種別ごとの流れに焦点を当てています。バッククォートで示した名称は、このリポジトリ内のクレートや主要モジュールを指します。

## コンポーネントと役割

- `iroha_torii`（Torii API）: HTTP / WebSocket インターフェース。クライアントからのトランザクション送信、クエリ実行、イベント・テレメトリ購読を受け付けます。
- `iroha_core`（Core）: 合意、検証、実行、状態遷移を統括します。
  - `sumeragi`: 合意、リーダー／プロポーザ選定、ブロック伝播、コミットプロトコル。
  - `state`（World/State, StateBlock）: 検証・適用に用いる権威ある状態とトランザクションビュー。
  - `kura`: 追記型のブロックストア。永続化とブロック取得を担当します。
  - `smartcontracts::ivm`: IVM 連携とエグゼキュータ管理。
  - `query::store::LiveQueryStore`: 接続クライアントへのライブクエリ・イベント配信。
- `iroha_data_model`: 命令／クエリ型、Norito シリアライズ、パーミッション、トリガー、スキーマ定義。
- `ivm`（Iroha Virtual Machine）: エグゼキュータおよび Kotodama などが生成したバイトコード（`.to`）を決定論的に実行します。SIMD/CUDA/NEON はフィーチャで制御され、結果を変えてはなりません。
- `norito`: トランザクション、ブロック、クエリ等の決定論的ワイヤフォーマットを提供するコンパクトなコーデック。
- `iroha_p2p`: P2P ネットワーク、ゴシップ、トポロジ管理。

**ランタイムターゲット**: Iroha Core、Torii、および関連クレートは Rust 標準ライブラリ（`std`）向けのみをサポートします。v2 では WASM / no-std は非対応です。

## トランザクション種別

すべてのトランザクションは署名済みリクエストであり、状態を変更する命令（Instruction）を含みます。

- **外部ユーザートランザクション**: アカウント権限による署名。複数の ISI を含む場合があります。
- **トリガー起動トランザクション**: スケジュール／条件付きトリガーが生成する内部トランザクション。
- **管理系トランザクション**: パラメータ更新（`SetParameter`）、ロール／パーミッション付与など。
- **エグゼキュータアップグレード**: 新しい IVM バイトコードを導入するトランザクション。合意されたタイミングで有効化されます。
- **ジェネシス**: ブロック 0 に相当する特別な初期状態定義（ゴシップされません）。

いずれも提案ブロックに含まれた後は同一の検証・合意パイプラインを通ります。

## 単一ノードでの高レベルライフサイクル

### 1. 受け付け（Torii）
- クライアントが署名済みトランザクションを `iroha_torii` に送信。
- Torii が Norito デコード、署名、チェーン ID、TTL、サイズ等の基本チェックを実施。
- 合格したトランザクションは Core のメモリプール（pending set）へ送られ、受信 ACK を返します（確定通知ではありません）。

### 2. ゴシップ（P2P）
- 受信ノードが現在のトポロジに従い `iroha_p2p` 経由で隣接ピアに転送。
- 各ピアはハッシュで重複排除し、同じ admission チェック後に自身の pending set に追加します。

### 3. リーダー／プロポーザ選定（Sumeragi）
- `sumeragi` がトポロジ、ビュー／高さに基づきリーダーと検証委員を決定。
- 役割はラウンドごとに回転し、公平性と耐障害性を確保します。

### 4. 提案組み立て（リーダー）
- リーダーはメモリプールからポリシー（FIFO、サイズ、TTL など）に従ってトランザクションを取得。
- 候補ごとに**ステートレス検証**（署名、形式、TTL 等）を実行。失敗したものは破棄または隔離。
- `StateBlock` 上で**ステートフル検証**を実施し、オンチェーンエグゼキュータで仮適用します。
- 両方を通過したものだけを提案に含め、ブロックサイズ・命令数・ガス等の制限に達すると停止。

### 5. トリガー評価（リーダー）
- ラウンド到来済みの時間トリガーを内部トランザクションとして取り込みます。
- データ駆動トリガーはシミュレーション結果に応じて現在の提案に含めるか、決定論性を保つため次の高さへ延期されます。
- トリガー起点のトランザクションも同じステートレス／ステートフル検証を受けます。

### 6. ブロック構築（リーダー）
- リーダーは提案をシールし、ブロックヘッダー・ボディを作成。メタデータ（高さ、ハッシュ、ビュー、QC インジケータ等）を設定し、提案ハッシュを計算します。
- 生成されたブロックは `sumeragi` を介して検証者／コレクターへ配布されます。

### 7. 投票と QC
- バリデータは受信した提案に対し再度ステートレス／ステートフル検証を実施し、可用性投票（Availability）、Prevote、Precommit を指定コレクターに送信。
- コレクターは投票を集約し、QC（Quorum Certificate）を生成してブロードキャスト。
- `First-QC-wins` により、同じ `(height, hash)` の Precommit QC は最初に届いた正当なものが採用されます。

### 8. コミットとストレージ
- 2 段 QC パイプラインの条件を満たすとブロックがコミット可能になります。
- `kura` がブロックを永続化し、`state` が適用を確定。
- commit テレメトリとイベントが配信され、メモリプールからコミット済みトランザクションを削除。

### 9. コミット後の副作用
- トリガーの再スケジュール、イベント配信、テレメトリ更新が行われます。
- オンチェーン設定の変更（パラメータ、エグゼキュータ等）は定義済みポイントで有効化されます。

## 検証の二段構え

- **ステートレス検証**: 署名数、フォーマット、チェーン ID、TTL、命令サイズなどをチェック。失敗したトランザクションは即時拒否。
- **ステートフル検証**: `StateBlock` で実行し、権限、残高、ドメイン制約、トリガー条件などを確認。Kotodama/IVM が実際の指示セットをシミュレートします。
- 非決定論的な挙動は検証側で検出され、提案は棄却。`pipeline` の警告イベントとして観測できます。

## トランザクション種別ごとのまとめ

- **ユーザー取引**: Torii 受付 → ゴシップ → リーダー選定 → 二段検証 → 提案 → QC → コミット → イベント／テレメトリ → メモリプールから削除。
- **トリガー起動**: 提案時にトリガーが Materialize → 同じ検証手順 → ユーザー取引と共にコミット。
- **管理取引**: ユーザー取引と同じパイプライン。パラメータ変更などの効果は決定論的な境界で適用されます。
- **エグゼキュータアップグレード**: 通常の検証後にコミットされ、指定タイミングで `ivm` のキャッシュやエグゼキュータが更新されます。

## オペレーター／統合向け注意事項

- Torii の ACK は最終化ではありません。イベント購読やコミット済みトランザクション照会で確定状態を確認してください。
- 大規模／複雑なトランザクションはブロック制限により複数ラウンドに分散され得ます。命令バッチは合理的なサイズに保つことが推奨されます。
- ノード間で実行パスに影響するフィーチャセット（SIMD/CUDA 等）を揃えるか、無効化して決定論的挙動を維持してください。
- Recovery sidecar や警告イベントは観測指標であり、コンセンサスには影響しませんが、オペレーターの調査を促す信号です。

## リカバリ／警告イベント

- Core はリカバリサイドカーを `kura` ディレクトリ下の `pipeline/sidecars.norito`（`pipeline/sidecars.index`）に集約して記録し、トランザクション毎の読み書き集合や DAG フィンガープリントを保持します。
- 起動時に直近 16 高さを再計算し、差異があれば警告ログと `PipelineEventBox::Warning { kind: "dag_fingerprint_mismatch" }` を発火します。
- 提案検証中にサイドカーと再計算結果が一致しない場合も警告を発行しつつ再計算したスケジュールを採用します（フォークは発生しません）。
- Torii の `GET /v1/pipeline/recovery/:height` でサイドカーを取得できます（存在しない場合は 404）。

## 実行ウィットネスイベント

- `ExecVote` を組み立てる際、`PipelineEventBox::Witness` を発行し、ブロックメタデータ（ハッシュ、高さ、ビュー、エポック）と実行ウィットネス（reads/writes）を公開します。
- Torii のストリーミング・JSON アダプタは read/write 件数を要約し、完全なウィットネスが必要なクライアントは Norito ストリームをデコードしてください。
- `PipelineEventFilterBox::Witness` に `for_block_hash`、`for_height`、`for_view` などのフィルタが用意され、特定ブロックの追跡が容易です。
- このイベントは既存のウィットネス永続化（`PipelineEvent::Merge` と Kura サイドカー）を補完し、SBV-AM 再計算の監査をリアルタイムで支援します。

## 署名バッチ処理（決定論的）

- Core は署名方式ごとにマイクロバッチ検証を行い、バイセクションで不正署名を特定します。ブロック当たりの作業量を削減しつつ決定論を維持します。
- 主な設定（パイプライン節。実運用では `iroha_config` を使用）:
  - `signature_batch_max_ed25519`
  - `signature_batch_max_secp256k1`
- `signature_batch_max_pqc`（ML-DSA / Dilithium3）
- `signature_batch_max_bls`（デフォルト 4）
- 従来の `signature_batch_max` は個別設定が 0 のとき Ed25519 に適用。
- BLS サポートは Core のコンパイルタイムフィーチャ `--features bls` で有効化します。
  - BLS-Normal のバッチ検証を使う場合、各トランザクションのメタデータに `bls_pop` キーで PoP（hex 文字列）を含めてください。BLS-Small の場合は `bls_pop_small`（hex）が必要です。PoP がないトランザクションは個別検証のままです。
  - バリデータノードは `signature_batch_max_bls > 0` を必須とし、0 の場合はピア登録が拒否されます。
