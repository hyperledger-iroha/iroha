# SoraFS Developer Documentation Plan

> **Status (Feb 2026):** The initial developer-facing bundle now lives under
> `docs/source/sorafs/developer/`. This plan remains as a reference for future
> expansions (additional SDK languages, automated snippet extraction, and portal
> wiring).

## Proposed Structure

- `docs/source/sorafs/developer/overview.md` — high-level intro, quickstart.
- `docs/source/sorafs/developer/cli.md` — CLI commands, examples.
- `docs/source/sorafs/developer/sdk/` (language-specific guides: Rust, TS, Go).
- `docs/source/sorafs/developer/ci.md` — CI integration steps (link to templates).
- `docs/source/sorafs/developer/examples/` — end-to-end sample projects.

## Quickstart Outline

1. Install CLI (`cargo install`, Homebrew, etc.).
2. Build manifest from sample spec.
3. Pack CAR and submit manifest via Torii.
4. Fetch chunks using orchestrator CLI.
5. Verify proofs and view telemetry.

## Example Projects

- `examples/rust/simple_fetch` — Rust orchestrator usage.
- `examples/ts/frontend` — TypeScript client hooking into chunk-range endpoints.
- `examples/go/backend` — Go service verifying proofs.

## CI Recipes

- GitHub Actions + Sigstore keyless (link to CI templates).
- GitLab pipeline with proof verification stage.
- Optional chunk-range smoketest job (link to chunk_range_smoketest.md).

## Release Notes & Changelog

- Document release process and schema changes.
- Provide migration guides for CLI/SDK versions.

## Documentation Tooling Integration

- **Primary surface:** integrate the developer guides into the Docusaurus-based docs portal (`docs/portal/`).
  - Each new page under `docs/source/sorafs/developer/` is rendered into the portal via the existing
    `scripts/sync_docs_portal.py` pipeline.
  - Sidebar configuration (`docs/portal/sidebars/sorafs.js`) gains a new “Developer Guides” section with
    entries for overview, CLI, language SDKs, CI integration, and examples.
  - Add `.. toctree::` entries in `docs/source/index.rst` linking to the developer overview so that PDF
    exports include the new material.
- **Autogenerated API docs:** language-specific SDK pages embed generated docs:
  - Rust: `cargo doc --document-private-items` output is converted via `mdbook` fragments and included with
    `.. include::`.
  - TypeScript: `typedoc` JSON feed is transformed into Markdown placed under
    `docs/source/sorafs/developer/sdk/typescript-api.md`.
  - Go: `godocdown` extracts comments into `docs/source/sorafs/developer/sdk/go-api.md`.
  CI updates (`scripts/docs_build.sh`) ensure the portal and Sphinx pipelines ingest these artifacts on every
  release candidate build.

## Code Snippet Pipeline

- Add a `docs/snippets/` directory containing language-tagged files (e.g., `rust/fetch.rs`,
  `typescript/manifest.ts`, `go/verify.go`).
- Introduce a `cargo xtask docs-snippets` command that:
  1. Extracts canonical examples from CLI/SDK repositories (by running unit tests or doc tests where
     available).
  2. Normalizes formatting via `rustfmt`, `prettier`, and `gofmt`.
  3. Regenerates Markdown partials (`docs/source/sorafs/developer/snippets/*.md`) that can be included in the
     guides using `mdinclude`/`include-code-block`.
- For the CLI quickstart, embed commands using a `{{#include}}` pattern so updates to the CLI help output
  propagate automatically.
- Snippet verification runs in CI (`docs_snippets` job) to guarantee the examples compile or lint successfully
  before documentation is published.

## DevRel Coordination

- Schedule a monthly sync between the SoraFS engineering leads and DevRel/content team to review upcoming
  features, video scripts, and blog posts.
- Produce a launch checklist:
  1. Developer guide outline reviewed and approved (product + DevRel).
  2. Video walkthrough scripts drafted and peer-reviewed.
  3. Demo repositories (from the `examples/` directory) deployed to public GitHub with README badges pointing
     back to the docs.
- Track assets in a shared Notion/issue board with statuses (`draft`, `recording`, `editing`, `published`).
  Completion of these tasks is a gating condition for marking SF-6 as “docs ready”.
