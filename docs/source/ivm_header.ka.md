# IVM ბაიტიკოდის სათაური


მაგია
- 4 ბაიტი: ASCII `IVM\0` ოფსეტური 0.

განლაგება (მიმდინარე)
- ოფსეტები და ზომები (სულ 17 ბაიტი):
  - 0..4: ჯადოსნური `IVM\0`
  - 4: `version_major: u8`
  - 5: `version_minor: u8`
  - 6: `mode: u8` (ფუნქციის ბიტები; იხილეთ ქვემოთ)
  - 7: `vector_length: u8`
  - 8..16: `max_cycles: u64` (ცოტა-ენდიანი)
  - 16: `abi_version: u8`

რეჟიმის ბიტები
- `ZK = 0x01`, `VECTOR = 0x02`, `HTM = 0x04` (დაჯავშნილი/ფუნქციური კარიბჭე).

ველები (მნიშვნელობა)
- `abi_version`: syscall ცხრილისა და პოინტერ-ABI სქემის ვერსია.
- `mode`: ფუნქციური ბიტები ZK ტრასინგისთვის/VECTOR/HTM.
- `vector_length`: ლოგიკური ვექტორის სიგრძე ვექტორული ოპერაციებისთვის (0 → დაუყენებელი).
- `max_cycles`: შესრულების padding შეკრული გამოიყენება ZK რეჟიმში და დაშვება.

შენიშვნები
- ენდიანურობა და განლაგება განისაზღვრება განხორციელებით და უკავშირდება `version`-ს. მავთულის ზემოთ განლაგება ასახავს მიმდინარე იმპლემენტაციას `crates/ivm_abi/src/metadata.rs`-ში.
- მინიმალურ მკითხველს შეუძლია დაეყრდნოს ამ განლაგებას მიმდინარე არტეფაქტებისთვის და უნდა გაუმკლავდეს მომავალ ცვლილებებს `version` კარიბჭის მეშვეობით.
- აპარატურის აჩქარება (SIMD/Metal/CUDA) ნებადართულია თითო ჰოსტზე. გაშვების დრო კითხულობს `AccelerationConfig` მნიშვნელობებს `iroha_config`-დან: `enable_simd` აიძულებს სკალარულ რეზერვებს, როდესაც false, ხოლო `enable_metal` და `enable_cuda` მათი შესაბამისი backends გამოიყენება მაშინაც კი, როდესაც ისინი გამოიყენება იმ შემთხვევაშიც კი, როდესაც ისინი შედგენილია ღობეზე. `ivm::set_acceleration_config` VM შექმნამდე.
- მობილური SDK-ები (Android/Swift) ზედაპირზე იგივე ღილაკებით; `IrohaSwift.AccelerationSettings`
  იძახებს `connect_norito_set_acceleration_config`-ს, რათა macOS/iOS-ის ბილდებმა შეძლონ მონაწილეობა მიიღონ Metal /
  NEON დეტერმინისტული ჩანაცვლების შენარჩუნებისას.
- ოპერატორებს ასევე შეუძლიათ აიძულონ გამორთონ კონკრეტული სარეზერვო სისტემა დიაგნოსტიკისთვის `IVM_DISABLE_METAL=1` ან `IVM_DISABLE_CUDA=1` ექსპორტით. ეს გარემოს უგულებელყოფა უპირატესობას ანიჭებს კონფიგურაციას და ინარჩუნებს VM-ს CPU-ს დეტერმინისტულ გზაზე.

გამძლე სახელმწიფო დამხმარეები და ABI ზედაპირი
- მდგრადი მდგომარეობის დამხმარე სისტემები (0x50–0x5A: STATE_{GET,SET,DEL}, ENCODE/DECODE_INT, BUILD_PATH_* და JSON/SCHEMA დაშიფვრა/გაშიფვრა) არის V1 ABI-ს ნაწილი და შედის `abi_hash`-ში.
- CoreHost მავთულები STATE_{GET,SET,DEL} WSV-ის მხარდაჭერით გრძელვადიანი ჭკვიანი კონტრაქტის მდგომარეობამდე; dev/ტესტი ჰოსტებს შეუძლიათ გამოიყენონ გადაფარვები ან ადგილობრივი მდგრადობა, მაგრამ უნდა შეინარჩუნონ იგივე დაკვირვებადი ქცევა.

ვალიდაცია
- კვანძის დაშვება იღებს მხოლოდ `version_major = 1` და `version_minor = 0` სათაურებს.
- `mode` უნდა შეიცავდეს მხოლოდ ცნობილ ბიტებს: `ZK`, `VECTOR`, `HTM` (უცნობი ბიტები უარყოფილია).
- `vector_length` საკონსულტაციოა და შეიძლება არ იყოს ნულოვანი, მაშინაც კი, თუ `VECTOR` ბიტი არ არის დაყენებული; დაშვება ახორციელებს მხოლოდ ზედა ზღვარს.
- მხარდაჭერილი `abi_version` მნიშვნელობები: პირველი გამოშვება იღებს მხოლოდ `1` (V1); სხვა ღირებულებები უარყოფილია მიღებისას.

- მცირე განახლებებმა შეიძლება დაამატოს ინსტრუქციები `feature_bits` და დაჯავშნილი opcode სივრცის მიღმა; ძირითად განახლებებს შეუძლია შეცვალოს კოდირება ან წაშალოს/გადაიყენოს მხოლოდ პროტოკოლის განახლებასთან ერთად.
- Syscall დიაპაზონი სტაბილურია; უცნობი აქტიური `abi_version` იძლევა `E_SCALL_UNKNOWN`.
- გაზის განრიგი მიბმულია `version`-თან და მოითხოვს ოქროს ვექტორებს ცვლილებაზე.

არტეფაქტების შემოწმება
- გამოიყენეთ `ivm_tool inspect <file.to>` სათაურის ველების სტაბილური ხედვისთვის.
- განვითარებისთვის, მაგალითები/შეიცავს პატარა Makefile სამიზნეს `examples-inspect`, რომელიც ამოწმებს ჩაშენებულ არტეფაქტებს.

მაგალითი (ჟანგი): მინიმალური მაგია + ზომის შემოწმება

```rust
use std::fs::File;
use std::io::{Read};

fn is_ivm_artifact(path: &std::path::Path) -> std::io::Result<bool> {
    let mut f = File::open(path)?;
    let mut magic = [0u8; 4];
    if f.read(&mut magic)? != 4 { return Ok(false); }
    if &magic != b"IVM\0" { return Ok(false); }
    let meta = std::fs::metadata(path)?;
    Ok(meta.len() >= 64)
}
```
