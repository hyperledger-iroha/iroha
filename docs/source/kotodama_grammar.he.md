# דקדוק וסמנטיקה של שפת Kotodama

מסמך זה מגדיר את תחביר Kotodama (לכסינג ודקדוק), את כללי הטיפוס, את הסמנטיקה הדטרמיניסטית, ואת הדרך שבה תכניות מורדות לבייטקוד של IVM (`.to`) בהתאם למוסכמות ה-pointer‑ABI של Norito. קובצי מקור של Kotodama משתמשים בסיומת `.ko`. הקומפיילר מפיק בייטקוד של IVM (`.to`) ויכול להחזיר גם מניפסט אופציונלי.

תוכן העניינים
- סקירה ומטרות
- מבנה לקסיקלי
- טיפוסים וליטרלים
- הצהרות ומודולים
- קונטיינר חוזה ומטא־דאטה
- פונקציות ופרמטרים
- פקודות
- ביטויים
- בילט-אינים וקונסטרוקטורי pointer‑ABI
- אוספים ומפות
- איטרציה דטרמיניסטית ומגבלות
- שגיאות ודיאגנוסטיקה
- מיפוי לקוד IVM
- ABI, כותרת ומניפסט
- מפת דרכיםים

## סקירה ומטרות

- **דטרמיניזם** – תכניות חייבות להניב תוצאה זהה על פני חומרות שונות; אין שימוש בנקודות צפות או מקורות לא דטרמיניסטיים. כל אינטראקציה עם ההוסט נעשית דרך קריאות מערכת עם ארגומנטים המקודדים ב-Norito.
- **ניידות** – היעד הוא בייטקוד של Iroha Virtual Machine (IVM) ולא ISA פיזי. קידודים דמויי RISC‑V שמופיעים בריפו הם פרטי מימוש של הדקודר ואסור שישנו התנהגות נצפת.
- **יכולת ביקורת** – סמנטיקה מצומצמת וברורה; יש מיפוי חד בין תחביר לקודי אופקודים של IVM ולקריאות מערכת של ההוסט.
- **גבולות** – לולאות על נתונים חסרי גבול מחייבות ציון גבול מפורש. לאיטרציה על Map יש חוקים קשיחים כדי לשמר דטרמיניזם.

## מבנה לקסיקלי

### רווחים והערות
- רווחים מפרידים בין טוקנים ואינם משפיעים מעבר לכך.
- הערות שורה מתחילות ב-`//` וממשיכות עד סוף השורה.
- הערות בלוק `/* ... */` אינן ניתנות לקינון.

### מזהים
- התו הראשון: `[A-Za-z_]`, ולאחריו `[A-Za-z0-9_]*`.
- רגיש לאותיות גדולות/קטנות; `_` כשם בפני עצמו מותר אך אינו מומלץ.

### מילות שמורות
`seiyaku`, ‏`hajimari`, ‏`kotoage`, ‏`kaizen`, ‏`state`, ‏`struct`, ‏`fn`, ‏`let`, ‏`const`, ‏`return`, ‏`if`, ‏`else`, ‏`while`, ‏`for`, ‏`in`, ‏`break`, ‏`continue`, ‏`true`, ‏`false`, ‏`permission`, ‏`kotoba`.

### אופרטורים וסימני פיסוק
- אריתמטיקה: `+ - * / %`
- בינאריים: `& | ^ ~` והזחות `<< >>`
- השוואה: `== != < <= > >=`
- לוגיקה: `&& || !`
- השמה: `= += -= *= /= %= &= |= ^= <<= >>=`
- נוספים: `: , ; . :: ->`
- סוגריים: `() [] {}`

### ליטרלים
- מספרים: עשרוני (`123`), הקסדצימלי (`0x2A`), בינרי (`0b1010`). בזמן ריצה כולם 64bit חתומים. ללא סיומת – הטיפוס ינובע, ברירת המחדל `int`.
- מחרוזות: תחומות בגרשיים כפולים עם רצף `\`; מקודדות UTF‑8.
- בוליאנים: ‎`true`, ‏`false`.

## טיפוסים וליטרלים

### טיפוסים סקלריים
- `int`: משלים ל-2 על 64 ביט; חיבור/חיסור/כפל עוטפים מודולו ‎2^64; חלוקה משתמשת באופ קיים ב-IVM (חתום או לא חתום) על פי החלטת הקומפיילר.
- `bool`: ערך לוגי; נכתב ל-`0`/`1` בבייטקוד.
- `string`: מחרוזת UTF‑8 בלתי ניתנת לשינוי; כארגומנט לקריאות מערכת – TLV של Norito; בתוך ה-VM מנוהל כפרוסת בתים ואורך.

### טיפוסים מורכבים
- `struct Name { field: Type, ... }`: טיפוסי מוצר מוגדרים ע"י המשתמש. בנאי בסינטקס של קריאה `Name(a, b, ...)`. גישת שדה `obj.field` נתמכת ונמכת לפוזיציות בטאפל פנימי. ה-ABI של מצב Durable מקובע לאנקוד Norito, והאוברליי של Iroha Core (בדיקת `kotodama_struct_overlay`) נועל את סדר השדות והפריסה.
- `Map<K, V>`: מפה אסוציאטיבית דטרמיניסטית; קיימות מגבלות איטרציה ושינוי (ראו בהמשך).
- `Tuple (T1, T2, ...)`: מוצר אנונימי לפלט של פונקציות עם מספר ערכים.

### טיפוסי pointer‑ABI מיוחדים (פונים להוסט)
`AccountId`, ‏`AssetDefinitionId`, ‏`Name`, ‏`Json`, ‏`NftId`, ‏`Blob` וכדומה אינם טיפוסי ריצה מדרגה ראשונה. הם קונסטרוקטורים שמחזירים מצביעים טיפוסיים ובלתי ניתנים לשינוי אל מעטפות TLV באזור INPUT. מותר להשתמש בהם רק כארגומנט לקריאות מערכת או להעבירם בין משתנים בלי שינוי.

### הנחת טיפוסים
- `let` מקומי ינבע מהמאפס. לפרמטרי פונקציה נדרשת הצמדה מפורשת. טיפוס החזרה ינבע אם אינו עמום.

## הצהרות ומודולים

### ישויות ברמת העל
- חוזים: ‎`seiyaku Name { ... }` כוללים פונקציות, מצב, מבנים ומטא־דאטה.
- ניתן להגדיר כמה חוזים בקובץ, אך מומלץ חוזה עיקרי יחיד; המניפסט בוחר `seiyaku` ראשי כברירת מחדל.
- הצהרות `struct` מגדירות טיפוסים מותאמים בתוך החוזה.

### חשיפה
- `kotoage fn` מציין נקודת כניסה ציבורית; החשיפה משפיעה על הרשאות הדיספצ'ר, לא על הקוד הגנרטורי.

## קונטיינר חוזה ומטא־דאטה

### תחביר
```kotodama
seiyaku Name {
  meta {
    abi_version: 1,
    vector_length: 0,
    max_cycles: 0,
    features: ["zk", "simd"],
  }

  state int counter;

  hajimari() { counter = 0; }

  kotoage fn inc() { counter = counter + 1; }
}
```

### סמנטיקה
- `meta { ... }` מחליף ערכי ברירת מחדל של כותרת IVM: `abi_version`, ‏`vector_length` (0 = לא הוגדר), ‏`max_cycles` (0 = ברירת המחדל של הקומפיילר), ו-`features` שמדליקים ביטי כותרת (עקיבה ZK, הכרזת וקטורים). פיצ'רים לא נתמכים מתעלמים עם אזהרה. אם אין בלוק `meta {}`, הקומפיילר מוציא `abi_version = 1` ומשאיר את שאר השדות בערכי ברירת המחדל של האפשרויות.
- `state` מצהיר על משתנים Durable שהרנטיים שומרים; ה-ABI שלהם הוא Norito ושומר על סדר ושמות שדות. שינויים בפונקציות מניבים ISI דרך קריאות מערכת; קוד ה-VM אינו משנה WSV ישירות.

## הצהרות טריגרים

הצהרות טריגרים מצרפות מטא-דאטה של תזמון למניפסטים של נקודות כניסה ונרשמות אוטומטית בעת
הפעלת מופע חוזה (ונמחקות בעת השבתה). הן מפוענחות בתוך בלוק `seiyaku`.

### תחביר
```
register_trigger wake {
  call run;
  on time pre_commit;
  repeats 2;
  metadata { tag: "alpha"; count: 1; enabled: true; }
}
```

### הערות
- `call` חייב להפנות ל-`kotoage fn` ציבורית באותו חוזה; `namespace::entrypoint` אופציונלי
  נרשם במניפסט אך קריאות חוצות-חוזים נדחות כרגע (מקומי בלבד).
- מסננים נתמכים: `time pre_commit` ו-`time schedule(start_ms, period_ms?)`, וכן
  `execute trigger <name>` עבור טריגרים לפי קריאה, `data any` עבור אירועי נתונים ומסנני
  pipeline (`pipeline transaction`, `pipeline block`, `pipeline merge`, `pipeline witness`).
- ערכי metadata חייבים להיות ליטרלים JSON (`string`, `number`, `bool`, `null`) או `json!(...)`.
- מפתחות metadata שמוזרקים ע"י הרנטיים: `contract_namespace`, `contract_id`,
  `contract_entrypoint`, `contract_code_hash`, `contract_trigger_id`.

## פונקציות ופרמטרים

### תחביר
- הצהרה: ‎`fn name(param1: Type, ...) -> Ret { ... }`
- ציבורית: ‎`kotoage fn name(...) { ... }`
- מאתחל: ‎`hajimari() { ... }` – נקרא ע"י הרנטיים בזמן פריסה ולא ע"י ה-VM עצמו.
- וו שדרוג: ‎`kaizen(args...) permission(Role) { ... }`.

### פרמטרים והחזרות
- הארגומנטים מועברים ברגיסטרים ‎`r10..r22` כערכים או מצביעי INPUT (TLV). עודפים נשפכים לסטאק.
- פונקציות מחזירות 0 או 1 ערך סקלרי או טאפל. ערך סקלרי חוזר ב-`r10`; טאפלות נכתבות לסטאק/OUTPUT לפי מוסכמה.

## פקודות

- הצמדת משתנים: ‎`let x = expr;`, ‏`let mut x = expr;` (ה-mut נשמר ברמת הקומפילציה; בזמן ריצה ניתן לעדכן מקומיים).
- השמה: ‎`x = expr;` וצורות מורחבות כמו ‎`x += 1;`.
- שליטה: ‎`if (cond) { ... } else { ... }`, ‏`while (cond) { ... }`, לולאת ‎`for (init; cond; step) { ... }` בסגנון C.
- לולאת Map: ‎`for (k, v) in map { ... }` – דטרמיניסטית (ראו להלן).
- זרימה: ‎`return expr;`, ‏`break;`, ‏`continue;`.
- קריאה: ‎`name(args...);` או ‎`call name(args...);` (הקומפיילר מנרמל).
- טענות: ‎`assert(cond, msg_id?);`, ‏`assert_eq(a, b, msg_id?);` – מתמפות ל-`ASSERT*` ב-build שאינו ZK או לאילוצים במצב ZK.

## ביטויים

### סדר קדימויות (מהגבוה לנמוך)
1. גישה/אינדקס: ‎`a.b`, ‏`a[b]`
2. יונרי: ‎`! ~ -`
3. כפל/חלוקה/מודולו: ‎`* / %`
4. חיבור/חיסור: ‎`+ -`
5. הזחות: ‎`<< >>`
6. יחסים: ‎`< <= > >=`
7. שוויון: ‎`== !=`
8. AND/XOR/OR ביטי: ‎`& ^ |`
9. AND/OR לוגי: ‎`&& ||`
10. תנאי משולש (שמור): ‎`cond ? a : b` (טרם ממומש)

### קריאות וטאפלות
- קריאות הן פוזיציוניות: ‎`f(a, b, c)`.
- ליטרל טאפל: ‎`(a, b, c)`; פירוק: ‎`let (x, y) = pair;`.

### מחרוזות ובתים
- מחרוזות הן UTF‑8. פונקציות שדורשות בתים גולמיים יקבלו `Blob` דרך קונסטרוקטור (ראו בילט-אינים).

## בילט-אינים וקונסטרוקטורי pointer‑ABI

### קונסטרוקטורים (יוצרים TLV ב-INPUT ומחזירים מצביע טיפוסי)
- ‎`account_id(string) -> AccountId*`
- ‎`asset_definition(string) -> AssetDefinitionId*`
- ‎`asset_id(string) -> AssetId*`
- ‎`domain(string)` / ‎`domain_id(string) -> DomainId*`
- ‎`name(string) -> Name*`
- ‎`json(string) -> Json*`
- ‎`nft_id(string) -> NftId*`
- ‎`blob(bytes|string) -> Blob*`
- ‎`norito_bytes(bytes|string) -> NoritoBytes*`

ה-prelude כולל גם מקרואים שסוגרים על אותם קונסטרוקטורים עם בדיקות קומפייל־טיים:
- ‎`account!("ih58...")` / ‎`account_id!("ih58...")`
- ‎`asset_definition!("rose#wonderland")` / ‎`asset_id!("rose#wonderland")`
- ‎`domain!("wonderland")` / ‎`domain_id!("wonderland")`
- ‎`name!("example")`
- ‎`json!("{\"hello\":\"world\"}")` וכן ‎`json!{ hello: "world" }`
- ‎`nft_id!("dragon$demo")`, ‏`blob!("bytes")`, ‏`norito_bytes!("...")`

המקרואים מתרחבים לבילט-אינים לעיל ומונעים ליטרלים שגויים כבר בזמן הקומפילציה.

### סטטוס מימוש
- המימוש הנוכחי מקבל ליטרלי מחרוזת ומניח TLV טיפוסי באזור INPUT. מתקבל מצביע בלתי ניתן לשינוי שאפשר להעביר לקריאות מערכת.
- הרחבות:
  - ‎`json(Blob[NoritoBytes]) -> Json*` דרך syscall ‏`JSON_DECODE`.
  - ‎`name(Blob[NoritoBytes]) -> Name*` דרך syscall ‏`NAME_DECODE`.
  - מעבר ישיר לטיפוסים: ‎`name(Name) -> Name*`, ‏`blob(Blob) -> Blob*`, ‏`norito_bytes(Blob) -> Blob*`.
  - סוכריות מתודה: ‎`s.name()`, ‏`s.json()`, ‏`b.blob()`, ‏`b.norito_bytes()`.

### בילט-אינים של ההוסט/קריאות מערכת (ממפים ל-SCALL; המספרים ב-`ivm.md`)
- ‎`mint_asset(AccountId*, AssetDefinitionId*, numeric)`
- ‎`burn_asset(AccountId*, AssetDefinitionId*, numeric)`
- ‎`transfer_asset(AccountId*, AccountId*, AssetDefinitionId*, numeric)`
- ‎`set_account_detail(AccountId*, Name*, Json*)`
- ‎`nft_mint_asset(NftId*, AccountId*)`
- ‎`nft_transfer_asset(AccountId*, NftId*, AccountId*)`
- ‎`nft_set_metadata(NftId*, Json*)`
- ‎`nft_burn_asset(NftId*)`
- ‎`authority() -> AccountId*`
- ‎`register_domain(DomainId*)`
- ‎`unregister_domain(DomainId*)`
- ‎`transfer_domain(AccountId*, DomainId*, AccountId*)`
- ‎`contains(Map<K,V>, K) -> bool`

### בילט-אינים שירותיים
- ‎`info(string|int|tuple)`: מפיק אירוע/מסר מובנה דרך OUTPUT.
- ‎`hash(blob) -> Blob*`: מחזיר TLV של Norito המכיל hash.
- `build_submit_ballot_inline(election_id, ciphertext, nullifier32, backend, proof, vk)` ו-`build_unshield_inline(asset, to, amount, inputs32, backend, proof, vk)`: בוני ISI אינליין. כל הארגומנטים חייבים להיות ליטרלים בזמן קומפילציה (ליטרל מחרוזת או קונסטרוקטור מצביע שמבוסס על ליטרל). `nullifier32` ו-`inputs32` חייבים להיות בדיוק 32 בתים (מחרוזת גולמית או hex עם `0x`), ו-`amount` חייב להיות לא שלילי. הערך המוחזר הוא `Blob*` שמצביע ל-NoritoBytes.

### הערות
- בילט-אינים הם שכבות דקות; הקומפיילר מנמיך להעברות רגיסטר וקריאת `SCALL`.
- קונסטרוקטורים הם פונקציות טהורות: ה-VM מבטיח שה-TLV ב-INPUT בלתי משתנה לאורך הקריאה.
- מבנים עם שדות pointer‑ABI (למשל `DomainId`, ‏`AccountId`) מאפשרים קיבוץ ארגומנטים. הקומפיילר ממפה `obj.field` לרגיסטרים מתאימים ללא הקצאות נוספות.

## אוספים ומפות

### טיפוס `Map<K, V>`
- טיפוסי מפתח וערך חייבים להיות ניתנים לסריאליזציה ל-TLV של Norito. מפתחות נתמכים: `int`, ‏`string`, טאפלות ומבנים מטיפוסים נתמכים. ערכים: כל טיפוס נתמך.
- פעולות:
  - אינדוקס: ‎`map[key]` לקבלה/הצבה (הצבה מתבצעת דרך קריאת מערכת).
  - בדיקת קיום: ‎`contains(map, key) -> bool` (אינטרינזי/קריאת מערכת).
  - איטרציה: ‎`for (k, v) in map { ... }` עם סדר דטרמיניסטי וכללי שינוי.

### כללי איטרציה דטרמיניסטית
- קבוצת האיטרציה היא צילום מצב המפתחות עם כניסה ללולאה.
- הסדר הוא לקסיקוגרפי עולה של המפתחות כפי שהם מקודדים ב-Norito.
- שינוי מבני (הוספה/הסרה/ניקוי) של המפה במהלך לולאה גורר `E_ITER_MUTATION` דטרמיניסטי.
- נדרש גבול: או הצהרת `@max_len` למפה, שימוש במאפיין `#[bounded(n)]`, או שימוש מפורש ב-`.take(n)` / `.range(..)`. אחרת הקומפיילר מעלה `E_UNBOUNDED_ITERATION`.

### מסייעי גבולות
- ‎`#[bounded(n)]`: לדוגמה `for (k, v) in map #[bounded(2)] { ... }`.
- ‎`.take(n)`: איטרציה על `n` הערכים הראשונים.
- ‎`.range(start, end)`: איטרציה בטווח `[start, end)` (חצי פתוח); שקול להגדרה של `start` ו-`n = end - start`.

### גבולות דינמיים
- גבולות ליטרליים: ‎`n`, ‏`start`, ‏`end` כקבועים מספריים → איטרציה במספר קבוע של צעדים.
- גבולות לא ליטרליים: עם הפעלת הפיצ'ר `kotodama_dynamic_bounds` בקרייט `ivm`, הקומפיילר מקבל ביטויים דינמיים ומוסיף בדיקות ריצה (אי-שליליות ו-`end >= start`). ההורדה מפיקה עד K חזרות מוגנות עם תנאי ‎`if (i < n)` (ברירת מחדל K=2). K ניתן לכיוון דרך ‎`CompilerOptions { dynamic_iter_cap, .. }`.
- מומלץ להריץ ‎`koto_lint` לקבלת אזהרות לפני קומפילציה; הקומפיילר ממשיך להורדה לאחר פרסינג וטייפינג.
- קודי שגיאה מתועדים ב-[Kotodama Compiler Error Codes](./kotodama_error_codes.md); הפקודה ‎`koto_compile --explain <code>` מספקת הסברים.

## שגיאות ודיאגנוסטיקה

### דוגמאות לשגיאות קומפילציה
- ‎`E_UNBOUNDED_ITERATION`: לולאת Map בלי גבול.
- ‎`E_MUT_DURING_ITER`: שינוי מבני של מפה בזמן איטרציה.
- ‎`E_BAD_POINTER_USE`: שימוש לא נכון בתוצאת קונסטרוקטור pointer‑ABI.
- ‎`E_UNRESOLVED_NAME`, ‏`E_TYPE_MISMATCH`, ‏`E_ARITY_MISMATCH`, ‏`E_DUP_SYMBOL`.

### שגיאות זמן ריצה ב-VM (מבחר; הרשימה המלאה ב-`ivm.md`)
- ‎`E_NORITO_INVALID`, ‏`E_OOB`, ‏`E_UNALIGNED`, ‏`E_SCALL_UNKNOWN`, ‏`E_ASSERT`, ‏`E_ASSERT_EQ`, ‏`E_ITER_MUTATION`.

### מסרי שגיאה
- לכל דיאגנוסטיקה מוצמד `msg_id` יציב שניתן לרמות בטבלאות `kotoba {}` אם קיימות.

## מיפוי לקוד IVM

### פייפליין
1. לקסר/פרסר בונה AST.
2. ניתוח סמנטי פותר שמות, בודק טיפוסים וממלא טבלאות סמלים.
3. הורדה ל-IR בסגנון SSA פשוט.
4. הקצאת רגיסטרים ל-GPR של IVM (`r10+`) וספילים לסטאק.
5. פליטת בייטקוד: שילוב אופקודים מקוריים וגרסאות תואמות RV; יצירת כותרת עם `abi_version`, פיצ'רים, אורך וקטור ו-`max_cycles`.

### הדגשים
- אריתמטיקה ולוגיקה → אופקודי ALU של IVM.
- הסתעפויות → תנאים וקפיצות; שימוש בגרסאות מקודדות-דחוסות כשמשתלם.
- זיכרון למשתנים מקומיים → סטאק, עם שמירת יישור.
- בילט-אינים → העברת רגיסטרים וקריאת `SCALL`.
- קונסטרוקטורי מצביע → כתיבת TLV ל-INPUT והחזרת הכתובת.
- טענות → `ASSERT` / `ASSERT_EQ`; ב-Non-ZK גורם ל-trap, ב-ZK יוצר אילוצים.

### אילוצי דטרמיניזם
- אין FP ואין קריאות מערכת לא דטרמיניסטיות.
- האצת SIMD/GPU אינה חשופה בבייטקוד וחייבת להיות זהה ביטית; הקומפיילר אינו מפיק הוראות תלויות חומרה.

## ABI, כותרת ומניפסט

### שדות כותרת שהקומפיילר מגדיר
- `version`: גרסת פורמט הבייטקוד (major.minor).
- `abi_version`: גרסת טבלת הקריאות וה-pointer‑ABI.
- `feature_bits`: דגלי פיצ'ר (`ZK`, ‏`VECTOR` וכו').
- `vector_len`: אורך וקטור לוגי (0 = לא הוגדר).
- `max_cycles`: חסם קבלה ורמז לריפוד ZK.

### מניפסט אופציונלי
- כולל `code_hash`, ‏`abi_hash`, מידע מבלוק `meta {}`, גרסת קומפיילר ורמזי בנייה לצורך שיחזור.

## מפת דרכים

- **KD-231 (אפריל 2026):** הוספת ניתוח טווח סטטי לגבולות איטרציה כך שהקומפיילר מספק סטי גישה מדויקים למתזמן.
- **KD-235 (מאי 2026):** הוספת טיפוס `bytes` מדרגה ראשונה (נפרד מ-`string`) לשיפור בהירות ה-pointer ABI והולכת טיעונים.
- **KD-242 (יוני 2026):** הרחבת הבילט-אינים (hash קריפטוגרפי, בדיקת חתימה) מאחורי דגלי פיצ'ר תוך שמירה על fallback דטרמיניסטי.
- **KD-247 (יוני 2026):** ייצוב מזהי השגיאה (`msg_id`) וסנכרון עם טבלאות `kotoba {}` עבור לוקליזציה ודיאגנוסטיקה תואמת.

### פליטת מניפסט

- API הקומפיילר מאפשר ‎`ivm::kotodama::compiler::Compiler::compile_source_with_manifest` כדי להחזיר `ContractManifest` יחד עם `.to`.
- שדות:
  - `code_hash`: hash של קוד נטול כותרת וליטרלים, עבור קשירת הארטיפקט.
  - `abi_hash`: תקציר יציב של פני הקריאות עבור `abi_version` (ראו `ivm.md` ו-`ivm::syscalls::compute_abi_hash`).
  - `compiler_fingerprint` ו-`features_bitmap` שמורים לכלי build.
  - המניפסט מיועד לבדיקה בזמן קבלה ולרגיסטרים; ראו `docs/source/new_pipeline.md` למחזור חיים.
