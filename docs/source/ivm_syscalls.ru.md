# IVM ABI системного вызова

В этом документе определяются номера системных вызовов IVM, соглашения о вызовах указателя-ABI, зарезервированные диапазоны номеров и каноническая таблица системных вызовов, связанных с контрактом, используемых при понижении Kotodama. Он дополняет `ivm.md` (архитектура) и `kotodama_grammar.md` (язык).

Управление версиями
- Набор распознаваемых системных вызовов зависит от поля заголовка байт-кода `abi_version`. Первый выпуск принимает только `abi_version = 1`; другие значения отклоняются при приеме. Неизвестные номера для активного `abi_version` детерминированно перехватываются с помощью `E_SCALL_UNKNOWN`.
- Обновления среды выполнения сохраняют `abi_version = 1` и не расширяют поверхности системных вызовов или указателей-ABI.
- Затраты на газ системных вызовов являются частью версионного расписания газа, привязанного к версии заголовка байт-кода. См. `ivm.md` (политика в отношении газа).

Диапазоны нумерации
- `0x00..=0x1F`: ядро/утилита виртуальной машины (помощники отладки/выхода доступны под `CoreHost`; остальные помощники разработки предназначены только для макетного хоста).
- `0x20..=0x5F`: мост Iroha ядра ISI (стабильный в ABI v1).
- `0x60..=0x7F`: расширение ISI, ограниченное функциями протокола (все еще является частью ABI v1, если оно включено).
- `0x80..=0xFF`: помощники хоста/криптографии и зарезервированные слоты; принимаются только номера, присутствующие в белом списке ABI v1.

Надежные помощники (ABI v1)
- Вспомогательные системные вызовы устойчивого состояния (0x50–0x5A: STATE_{GET,SET,DEL}, ENCODE/DECODE_INT, BUILD_PATH_*, кодирование/декодирование JSON/SCHEMA) являются частью V1 ABI и включены в вычисления `abi_hash`.
- CoreHost связывает STATE_{GET,SET,DEL} с устойчивым состоянием смарт-контракта, поддерживаемым WSV; Хосты dev/test могут сохраняться локально, но должны сохранять идентичную семантику системных вызовов.

Соглашение о вызовах Pointer‑ABI (системные вызовы смарт-контракта)
- Аргументы помещаются в регистры `r10+` как необработанные значения `u64` или как указатели в области INPUT на неизменяемые конверты TLV Norito (например, `AccountId`, `AssetDefinitionId`, `Name`, `Json`, `NftId`).
— Скалярные возвращаемые значения — это `u64`, возвращенные с хоста. Результаты указателя записываются хостом в `r10`.

Газовое правоприменение
- CoreHost взимает дополнительную плату за системные вызовы ISI, используя собственное расписание ISI; Пакетные переводы FASTPQ взимаются за каждую запись.
- Системные вызовы ZK_VERIFY повторно используют расписание конфиденциального проверочного газа (базовый + контрольный размер).
- SMARTCONTRACT_EXECUTE_QUERY начисляет базовую плату + за элемент + за байт; сортировка умножает стоимость единицы товара, а несортированные смещения добавляют штраф за единицу товара.

Примечания
- Все аргументы указателя ссылаются на конверты TLV Norito в области INPUT и проверяются при первом разыменовании (`E_NORITO_INVALID` в случае ошибки).
- Все мутации применяются через стандартный исполнитель Iroha (через `CoreHost`), а не непосредственно виртуальной машиной.
- Точные газовые константы (`G_*`) определяются активным газовым графиком; см. `ivm.md`.

Ошибки
- `E_SCALL_UNKNOWN`: номер системного вызова не распознан для активного `abi_version`.
- Ошибки проверки ввода распространяются как ловушки VM (например, `E_NORITO_INVALID` для неверных TLV).

## Примеры TLV администратора/роли (фиктивный хост)

В этом разделе описаны формы TLV и минимальные полезные данные JSON, принимаемые макетным узлом WSV для системных вызовов в стиле администратора, используемых в тестах. Все аргументы указателя следуют за указателем-ABI (конверты TLV Norito помещены в INPUT). Производственные хосты могут использовать более сложные схемы; эти примеры направлены на разъяснение типов и основных форм.- REGISTER_PEER / UNREGISTER_PEER
  - Аргументы: `r10=&Json`
  – Пример JSON: `{ "peer": "peer-id-or-info" }`.
  - Примечание CoreHost: `REGISTER_PEER` ожидает объект JSON `RegisterPeerWithPop` с байтами `peer` + `pop` (необязательно `activation_at`, `expiry_at`, `hsm`); `UNREGISTER_PEER` принимает строку идентификатора узла или `{ "peer": "..." }`.

- CREATE_TRIGGER/REMOVE_TRIGGER/SET_TRIGGER_ENABLED
  - CREATE_TRIGGER:
    - Аргументы: `r10=&Json`
    - Минимальный JSON: `{ "name": "t1" }` (дополнительные поля игнорируются макетом)
  - УДАЛИТЬ_ТРИГГЕР:
    - Args: `r10=&Name` (имя триггера)
  - SET_TRIGGER_ENABLED:
    - Аргументы: `r10=&Name`, `r11=enabled:u64` (0 = отключено, ненулевое = включено)
  - Примечание CoreHost: `CREATE_TRIGGER` ожидает полную спецификацию триггера (строка base64 Norito `Trigger` или
    `{ "id": "<trigger_id>", "action": ... }` с `action` в качестве строки base64 Norito `Action` или
    объект JSON), а `SET_TRIGGER_ENABLED` переключает ключ метаданных триггера `__enabled` (отсутствует
    по умолчанию включено).

- Роли: CREATE_ROLE / DELETE_ROLE / GRANT_ROLE / REVOKE_ROLE.
  - CREATE_ROLE:
    - Args: `r10=&Name` (имя роли), `r11=&Json` (набор разрешений)
    - JSON принимает ключ `"perms"` или `"permissions"`, каждый из которых представляет собой строковый массив имен разрешений.
    - Примеры:
      - `{ "perms": [ "mint_asset:rose#wonder" ] }`
      - `{ "permissions": [ "read_assets:i105...", "transfer_asset:rose#wonder" ] }`
    - Поддерживаемые префиксы имен разрешений в макете:
      - И18НИ00000153Х, И18НИ00000154Х, И18НИ00000155Х
      - `read_assets:<account_id>`
      - `mint_asset:<asset_definition_id>`
      - `burn_asset:<asset_definition_id>`
      - `transfer_asset:<asset_definition_id>`
  - DELETE_ROLE:
    - Аргументы: `r10=&Name`
    - Ошибка, если какой-либо учетной записи все еще назначена эта роль.
  - GRANT_ROLE / REVOKE_ROLE:
    - Аргументы: `r10=&AccountId` (субъект), `r11=&Name` (имя роли)
  - Примечание CoreHost: разрешение JSON может быть полным объектом `Permission` (`{ "name": "...", "payload": ... }`) или строкой (полезная нагрузка по умолчанию — `null`); `GRANT_PERMISSION`/`REVOKE_PERMISSION` принимают `&Name` или `&Json(Permission)`.

- Отменить регистрацию операций (домен/учетная запись/актив): инварианты (макет)
  - UNREGISTER_DOMAIN (`r10=&DomainId`) завершается сбоем, если в домене существуют учетные записи или определения активов.
  - UNREGISTER_ACCOUNT (`r10=&AccountId`) завершается с ошибкой, если учетная запись имеет ненулевой баланс или владеет NFT.
  - UNREGISTER_ASSET (`r10=&AssetDefinitionId`) завершается с ошибкой, если для актива существуют какие-либо остатки.

Примечания
— Эти примеры отражают макет хоста WSV, использованный в тестах; хосты реальных узлов могут предоставлять более сложные схемы администрирования или требовать дополнительной проверки. Правила указателя-ABI по-прежнему применяются: TLV должны быть в INPUT, версия = 1, идентификаторы типов должны совпадать, а хэши полезной нагрузки должны быть проверены.
