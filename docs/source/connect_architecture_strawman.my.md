# Session Architecture Strawman (Swift / Android / JS) ကို ချိတ်ဆက်ပါ

ဤ strawman အဆိုပြုချက်သည် Nexus Connect workflows အတွက် မျှဝေထားသော ဒီဇိုင်းကို အကြမ်းဖျင်းဖော်ပြသည်
Swift၊ Android နှင့် JavaScript SDKs တစ်လျှောက်။ ပံ့ပိုးကူညီရန် ရည်ရွယ်ပါသည်။
Feb 2026 ဖြတ်ကျော်-SDK အလုပ်ရုံဆွေးနွေးပွဲနှင့် အကောင်အထည်မဖော်မီ ပွင့်လင်းသောမေးခွန်းများကို ဖမ်းယူပါ။

> နောက်ဆုံးမွမ်းမံမှု- 2026-01-29  
> ရေးသားသူများ- Swift SDK Lead၊ Android Networking TL၊ JS Lead  
> အခြေအနေ- ကောင်စီပြန်လည်သုံးသပ်မှုအတွက်မူကြမ်း (ခြိမ်းခြောက်မှုပုံစံ + ဒေတာထိန်းသိမ်းထားမှုကို ထည့်သွင်းပြီး 2026-03-12)

## ပန်းတိုင်

1. ပိုက်ဆံအိတ်ကို ညှိပါ ↔ ချိတ်ဆက်မှု bootstrapping အပါအဝင်၊
   အတည်ပြုချက်များ၊ လက်မှတ်ရေးထိုးတောင်းဆိုမှုများ၊ နှင့် ဖြိုဖျက်ခြင်း။
2. အားလုံးမျှဝေထားသည့် Norito စာအိတ်ပုံစံ (open/approve/sign/control) ကို သတ်မှတ်ပါ
   SDKs နှင့် `connect_norito_bridge` နှင့် တန်းတူမှုရှိစေရန်။
3. သယ်ယူပို့ဆောင်ရေး (WebSocket/WebRTC)၊ ကုဒ်ဝှက်ခြင်းကြားတွင် တာဝန်ခွဲဝေခြင်း၊
   (Norito ဘောင်များချိတ်ဆက်ပါ + သော့လဲလှယ်ခြင်း) နှင့် အပလီကေးရှင်းအလွှာများ (SDK မျက်နှာစာများ)။
4. အပါအဝင် ဒက်စတော့/မိုဘိုင်းပလပ်ဖောင်းများတစ်လျှောက် အဆုံးအဖြတ်ပေးသည့် အပြုအမူကို သေချာပါစေ။
   အော့ဖ်လိုင်းကြားခံစနစ်နှင့် ပြန်လည်ချိတ်ဆက်မှု။

## Session Lifecycle (အဆင့်မြင့်)

```
┌────────────┐      ┌─────────────┐      ┌────────────┐
│  dApp SDK  │←────→│  Connect WS │←────→│ Wallet SDK │
└────────────┘      └─────────────┘      └────────────┘
      │                    │                    │
      │ 1. open (app→wallet) frame (metadata, permissions, chain_id)
      │────────────────────────────────────────>│
      │                    │                    │
      │                    │ 2. route frame     │
      │                    │────────────────────│
      │                    │                    │
      │                    │     3. approve frame (wallet pk, account,
      │                    │        permissions, proof/attest)
      │<────────────────────────────────────────│
      │                    │                    │
      │ 4. sign request    │                    │
      │────────────────────────────────────────>│
      │                    │                    │
      │                    │ 5. sign result     │
      │                    │────────────────────│
      │                    │                    │
      │ 6. control frames for reject/close, error propagation, heartbeats.
```

## စာအိတ်/Norito Schema

SDK အားလုံးသည် `connect_norito_bridge` တွင် သတ်မှတ်ထားသော canonical Norito schema ကို အသုံးပြုရပါမည်-

- `EnvelopeV1` (ဖွင့် / အတည်ပြု / လက်မှတ်ထိုး / ထိန်းချုပ်)
- `ConnectFrameV1` (ciphertext frames နှင့် AEAD payload)
- ထိန်းချုပ်ကုဒ်များ
  - `open_ext` (မက်တာဒေတာ၊ ခွင့်ပြုချက်များ)
  - `approve_ext` (အကောင့်၊ ခွင့်ပြုချက်များ၊ အထောက်အထားများ၊ လက်မှတ်)
  - `reject`၊ `close`၊ `ping/pong`၊ `error`

Swift သည် ယခင်က နေရာယူထားသည့် JSON ကုဒ်နံပါတ် (`ConnectCodec.swift`) ကို တင်ပို့ခဲ့သည်။ ဧပြီလ 2026 မှ SDK
Norito တံတားကို အမြဲတမ်းအသုံးပြုပြီး XCFramework ပျောက်ဆုံးသွားသောအခါ ပိတ်သွားသော်လည်း ဤ strawman
တံတားပေါင်းစည်းရေးကို ဦးတည်စေသည့် လုပ်ပိုင်ခွင့်ကို ဖမ်းဆုပ်ထားဆဲဖြစ်သည်။

| လုပ်ဆောင်ချက် | ဖော်ပြချက် | အဆင့်အတန်း |
|----------|-------------|--------|
| `connect_norito_encode_control_open_ext` | dApp ဘောင် | ဖွင့်ပါ။ တံတား|၌ အကောင်အထည်ဖော်သည်။
| `connect_norito_encode_control_approve_ext` | ပိုက်ဆံအိတ်ခွင့်ပြုချက် | ဟိုဟာ |
| `connect_norito_encode_envelope_sign_request_tx/raw` | လက်မှတ်ရေးထိုးတောင်းဆိုမှုများ | ဟိုဟာ |
| `connect_norito_encode_envelope_sign_result_ok/err` | ရလဒ်များ | ဟိုဟာ |
| `connect_norito_decode_*` | ပိုက်ဆံအိတ်/dApps အတွက် ပိုင်းခြားစိတ်ဖြာခြင်း | ဟိုဟာ |

### လိုအပ်သောအလုပ်

- Swift- နေရာကိုင်ဆောင်သူ `ConnectCodec` JSON အထောက်အကူများကို တံတားခေါ်ဆိုမှုများနှင့် မျက်နှာပြင်များဖြင့် အစားထိုးပါ။
  မျှဝေထားသော Norito အမျိုးအစားများကို အသုံးပြု၍ ရိုက်ထည့်ထားသော ထုပ်ပိုးမှုများ (`ConnectFrame`၊ `ConnectEnvelope`)။ ✅ (ဧပြီ 2026)
- Android/JS- တူညီသော ထုပ်ပိုးမှုများ ရှိနေကြောင်း သေချာပါစေ။ အမှားကုဒ်များနှင့် မက်တာဒေတာကီးများကို ချိန်ညှိပါ။
- မျှဝေထားသည်- တစ်သမတ်တည်းသော့ဆင်းသက်လာခြင်းဖြင့် စာရွက်စာတမ်း ကုဒ်ဝှက်ခြင်း (X25519 သော့လဲလှယ်မှု၊ AEAD)
  Norito spec အလိုက်၊ Rust တံတားကို အသုံးပြု၍ နမူနာ ပေါင်းစပ်စမ်းသပ်မှုများကို ပေးဆောင်ပါ။

## ပို့ဆောင်ရေးစာချုပ်- အဓိကသယ်ယူပို့ဆောင်ရေး- WebSocket (`/v2/connect/ws?sid=<session_id>`)။
- ရွေးချယ်နိုင်သောအနာဂတ်- WebRTC (TBD) - ကနဦး strawman အတွက် နယ်ပယ်မပြည့်မီ။
- ပြန်လည်ချိတ်ဆက်နည်းဗျူဟာ- တုန်လှုပ်ချောက်ချားမှုအပြည့်ဖြင့် exponential back-off (အခြေခံ 5s၊ အများဆုံး 60s); Swift၊ Android နှင့် JS တို့တွင် မျှဝေထားသည့် ကိန်းသေများကို မျှဝေထားသောကြောင့် ထပ်မံကြိုးစားမှုများသည် ခန့်မှန်းနိုင်မည်ဖြစ်သည်။
- Ping/pong cadence- ပြန်လည်ချိတ်ဆက်ခြင်းမပြုမီ လွတ်သွားသော pong သုံးခုအတွက် သည်းခံနိုင်မှုနှင့်အတူ အသက် 30 နှစ်များအတွင်း နှလုံးခုန်ခြင်း၊ JS သည် ဘရောက်ဆာ အဟန့်တားခြင်း စည်းမျဉ်းများကို ကျေနပ်စေရန် အနည်းဆုံး ကြားကာလ 15s အထိ ကန့်သတ်ထားသည်။
- Push hooks- Android wallet SDK သည် နှိုးဆော်မှုများအတွက် စိတ်ကြိုက် FCM ပေါင်းစပ်မှုကို ဖော်ထုတ်ပေးသည်၊ JS သည် မဲရုံအခြေခံ (ဘရောက်ဆာတွန်းအားပေးခွင့်ပြုချက်များအတွက် မှတ်တမ်းတင်ထားသော ကန့်သတ်ချက်များ)။
- SDK တာဝန်များ-
  - ပင်း/ပေါင်နှလုံးခုန်သံများကို ထိန်းသိမ်းပါ (မိုဘိုင်းပေါ်ရှိ ဘက်ထရီအားကုန်ခြင်းမှ ရှောင်ကြဉ်ပါ)။
  - အော့ဖ်လိုင်းတွင် အထွက်ဘောင်များ (ကန့်သတ်ထားသောတန်းစီ၊ dApp အတွက် ဆက်လက်ရှိနေသည်)။
- အစီအစဉ်ထုတ်လွှင့်မှု API (Swift Combine `AsyncStream`၊ Android Flow၊ JS async iter) ကို ပံ့ပိုးပါ။
- Surface ချိတ်များကို ပြန်လည်ချိတ်ဆက်ပြီး လူကိုယ်တိုင် ပြန်လည်စာရင်းသွင်းခွင့်ပြုပါ။
- Telemetry ပြန်လည်တုံ့ပြန်ခြင်း- စက်ရှင်အဆင့် ကောင်တာများကိုသာ ထုတ်လွှတ်သည် (`sid` hash၊ ဦးတည်ချက်၊
  အစီအစဥ်ပြတင်းပေါက်၊ တန်းစီခြင်းအတိမ်အနက်) Connect telemetry တွင် မှတ်တမ်းတင်ထားသော ဆားများဖြင့်
  လမ်းညွှန် ခေါင်းစီး/ကီးများသည် မှတ်တမ်းများ သို့မဟုတ် အမှားရှာပြင်စာကြောင်းများတွင် ဘယ်သောအခါမှ မပေါ်ရပါ။

## ကုဒ်ဝှက်ခြင်းနှင့် သော့စီမံခန့်ခွဲမှု

### ကဏ္ဍသတ်မှတ်ခြင်းများနှင့် ဆားများ

- `sid` သည် `BLAKE2b-256("iroha-connect|sid|" || chain_id || app_ephemeral_pk || nonce16)` မှဆင်းသက်လာသော 32-byte အမှတ်အသားတစ်ခုဖြစ်သည်။  
  DApps သည် `/v2/connect/session` ကိုမခေါ်ဆိုမီ ၎င်းကိုတွက်ချက်သည်။ ပိုက်ဆံအိတ်များသည် `approve` ဖရိန်များတွင် သံယောင်လိုက်နေသောကြောင့် နှစ်ဖက်စလုံးသည် ဂျာနယ်များနှင့် တယ်လီမီတာကို သော့ခတ်နိုင်မည်ဖြစ်သည်။
- တူညီသောဆားသည် သော့ဆင်းသက်လာမှုအဆင့်တိုင်းကို ကျွေးမွေးပေးသောကြောင့် SDKs များသည် host platform မှ ရိတ်သိမ်းထားသော entropy ပေါ်တွင် ဘယ်တော့မှ အားမကိုးပါ။

### ပေါ်ပင်သော့ ကိုင်တွယ်ခြင်း။

- စက်ရှင်တိုင်းသည် လတ်ဆတ်သော X25519 သော့ချက်ပစ္စည်းကို အသုံးပြုသည်။  
  Swift သည် ၎င်းကို `ConnectCrypto` မှတစ်ဆင့် Keychain/Secure Enclave တွင် သိမ်းဆည်းထားကာ၊ Android ပိုက်ဆံအိတ်များသည် StrongBox (TEE ကျောထောက်နောက်ခံပြုသော့ဆိုင်များသို့ ပြန်လည်ရောက်ရှိသွားသည်)၊ Android ပိုက်ဆံအိတ်များကို မူရင်းအတိုင်း သိမ်းဆည်းထားကာ JS သည် လုံခြုံသောအကြောင်းအရာ WebCrypto ဥပမာ သို့မဟုတ် မူရင်း `iroha_js_host` ပလပ်အင် လိုအပ်ပါသည်။
- ဖွင့်ထားသောဘောင်များတွင် dApp ပေါ်ပင်အများပြည်သူသော့အပြင် စိတ်ကြိုက်အတည်ပြုချက်အတွဲတစ်ခုပါဝင်သည်။ Wallet အတည်ပြုချက်များသည် ပိုက်ဆံအိတ် အများသူငှာသော့နှင့် လိုက်လျောညီထွေဖြစ်စေသော စီးဆင်းမှုများအတွက် လိုအပ်သော ဟာ့ဒ်ဝဲအတည်ပြုချက်မှန်သမျှကို ပြန်ပေးသည်။
- အသိအမှတ်ပြု ဝန်ဆောင်ခများသည် လက်ခံထားသော အစီအစဉ်အတိုင်း လိုက်နာသည်-  
  `attestation { platform, evidence_b64, statement_hash }`။  
  ဘရောက်ဆာများသည် ပိတ်ဆို့ခြင်းကို ချန်လှပ်ထားနိုင်သည်။ ဟာ့ဒ်ဝဲကျောထောက်နောက်ခံသော့များကို အသုံးပြုသည့်အခါတိုင်း မူရင်းပိုက်ဆံအိတ်များတွင် ၎င်းကို ပါရှိသည်။

### လမ်းညွှန်သော့များနှင့် AEAD

- မျှဝေထားသောလျှို့ဝှက်ချက်များကို HKDF-SHA256 (Rust တံတားအကူများမှတဆင့်) နှင့် ဒိုမိန်း-ခြားထားသော အချက်အလက်ကြိုးများဖြင့် တိုးချဲ့သည်-
  - `iroha-connect|k_app` → အက်ပ် → ပိုက်ဆံအိတ် အသွားအလာ။
  - `iroha-connect|k_wallet` → ပိုက်ဆံအိတ် → အက်ပ်လမ်းကြောင်း။
- AEAD သည် v1 စာအိတ်အတွက် ChaCha20-Poly1305 (`connect_norito_bridge` သည် ပလက်ဖောင်းတိုင်းတွင် အထောက်အကူများကို ဖော်ထုတ်သည်)။  
  ဆက်စပ်ဒေတာသည် `("connect:v1", sid, dir, seq_le, kind=ciphertext)` နှင့် ညီမျှသောကြောင့် ခေါင်းစီးများပေါ်တွင် ဆော့ကစားနေသည်ကို တွေ့ရှိရသည်။
- Nonce များကို 64-bit sequence ကောင်တာ (`nonce[0..4]=0`, `nonce[4..12]=seq_le`) မှ ဆင်းသက်လာသည်။ မျှဝေထားသော ကူညီသူစမ်းသပ်မှုများသည် BigInt/UInt ပြောင်းလဲမှုများကို SDKs တစ်လျှောက်တွင် တူညီစွာလုပ်ဆောင်ကြောင်း သေချာစေသည်။

### လှည့်ခြင်းနှင့် ပြန်လည်ရယူရန် လက်ဆွဲနှုတ်ဆက်ခြင်း။- လှည့်ခြင်းသည် ရွေးချယ်နိုင်သော်လည်း ပရိုတိုကောကို သတ်မှတ်ထားသည်- dApps သည် အတွဲလိုက်ကောင်တာများအနီးသို့ ထုပ်ပိုးလိုက်သောအခါ၊ ပိုက်ဆံအိတ်များသည် အများသူငှာသော့အသစ်အပြင် လက်မှတ်ရေးထိုးထားသော အသိအမှတ်ပြုချက်တစ်ခုဖြင့် တုံ့ပြန်ကြပြီး နှစ်ဖက်စလုံးသည် စက်ရှင်ကိုပိတ်စရာမလိုဘဲ ဦးတည်ချက်သော့အသစ်များကို ချက်ချင်းရရှိကြသည်။
- Wallet-side သော့ပျောက်ဆုံးမှုသည် `resume` ထိန်းချုပ်မှုဖြင့် တူညီသောလက်ဆွဲခြင်းကို အစပျိုးစေသောကြောင့် dApps သည် အငြိမ်းစားသော့ကိုပစ်မှတ်ထားသည့် ကက်ရှ်စာဝှက်စာသားကို ဖယ်ရှားရန် သိနိုင်သည်။

သမိုင်းဝင် CryptoKit အမှားများအတွက် `docs/connect_swift_ios.md` ကိုကြည့်ပါ၊ Kotlin နှင့် JS သည် `docs/connect_kotlin_ws*.md` အောက်တွင် ကိုက်ညီသော အကိုးအကားများရှိသည်။

## ခွင့်ပြုချက်များနှင့် အထောက်အထားများ

- ခွင့်ပြုချက်ဖော်ပြချက်များသည် တံတားမှထုတ်သော မျှဝေထားသော Norito တည်ဆောက်မှုမှတစ်ဆင့် အသွားအပြန်ရှိရမည်။  
  အကွက်များ-
  - `methods` — ကြိယာ (`sign_transaction`၊ `sign_raw`၊ `submit_proof`၊ …)။  
  - `events` — စာရင်းသွင်းမှုများကို dApp နှင့် ပူးတွဲခွင့်ပြုထားသည်။  
  - `resources` — ရွေးချယ်နိုင်သော အကောင့်/ပိုင်ဆိုင်မှု စစ်ထုတ်မှုများကြောင့် ပိုက်ဆံအိတ်များကို ဝင်ရောက်ကြည့်ရှုနိုင်မည်ဖြစ်သည်။  
  - `constraints` — ကွင်းဆက် ID၊ TTL သို့မဟုတ် လက်မှတ်မထိုးမီ ပိုက်ဆံအိတ်တွင် ပြဋ္ဌာန်းထားသည့် စိတ်ကြိုက်မူဝါဒ ခလုတ်များ။
- ခွင့်ပြုချက်များနှင့်အတူ လိုက်လျောညီထွေရှိသော မက်တာဒေတာစီးနင်းမှုများ
  - ရွေးချယ်နိုင်သော `attachments[]` တွင် Norito ပူးတွဲပါဖိုင်အကိုးအကားများ (KYC အစုအဝေးများ၊ စည်းကမ်းထိန်းသိမ်းရေးဖြတ်ပိုင်းများ) ပါရှိသည်။  
  - `compliance_manifest_id` သည် တောင်းဆိုချက်အား ယခင်က အတည်ပြုထားသော မန်နီးဖက်စ်တစ်ခုနှင့် ချိတ်ဆက်ပေးသောကြောင့် အော်ပရေတာများသည် သက်သေအထောက်အထားကို စစ်ဆေးနိုင်သည်။
- Wallet တုံ့ပြန်မှုများသည် သဘောတူထားသောကုဒ်များကို အသုံးပြုသည်-
  - `user_declined`, `permissions_mismatch`, `compliance_failed`, `internal_error`။  
  တစ်ခုစီတွင် UI အရိပ်အမြွက်များအတွက် `localized_message` နှင့် စက်ဖတ်နိုင်သော `reason_code` တို့ကို ဆောင်ထားနိုင်သည်။
- အတည်ပြုချက်ဘောင်များတွင် ရွေးချယ်ထားသော အကောင့်/ထိန်းချုပ်ကိရိယာ၊ ခွင့်ပြုချက်ပဲ့တင်သံ၊ အထောက်အထားအစုအဝေး (ZK အထောက်အထား သို့မဟုတ် သက်သေအထောက်အထား) နှင့် မူဝါဒခလုတ်များ (ဥပမာ၊ `offline_queue_enabled`) ပါဝင်သည်။  
  ငြင်းဆိုမှုများသည် `proof` အချည်းနှီးဖြစ်သော `proof` နှင့် တူညီသော schema ကို ထင်ဟပ်နေသော်လည်း စာရင်းစစ်ရန်အတွက် `sid` ကို မှတ်တမ်းတင်ထားဆဲဖြစ်သည်။

## SDK မျက်နှာစာများ

| SDK | အဆိုပြုထားသော API | မှတ်စုများ |
|-----|-----------------|-------|
| Swift | `ConnectClient`, `ConnectSession`, `ConnectRequest`, `ConnectApproval` | နေရာကိုင်ဆောင်သူများကို ရိုက်ထည့်ထားသော ထုပ်ပိုးမှုများ + async streams များဖြင့် အစားထိုးပါ။ |
| Android | Kotlin coroutines + frames အတွက် အလုံပိတ်အတန်းများ | သယ်ဆောင်ရလွယ်ကူစေရန် Swift ဖွဲ့စည်းပုံနှင့် ချိန်ညှိပါ။ |
| JS | Async iterators + frame အမျိုးအစားများအတွက် TypeScript enums | အစုအဝေးအတွက် အဆင်ပြေသော SDK (ဘရောက်ဆာ/node) ပေးပါ။ |

### အဖြစ်များသော အပြုအမူများ- `ConnectSession` သည် ဘဝသံသရာ-
  1. WebSocket ကိုတည်ဆောက်ပါ၊ လက်ဆွဲနှုတ်ဆက်ပါ။
  2. ဖရိမ်များကို ဖွင့်/အတည်ပြု လဲလှယ်ပါ။
  3. လက်မှတ်ရေးထိုးတောင်းဆိုမှုများ/တုံ့ပြန်မှုများကို ကိုင်တွယ်ပါ။
  4. အပလီကေးရှင်းအလွှာသို့ ဖြစ်ရပ်များကို ထုတ်လွှတ်ပါ။
- အဆင့်မြင့် အထောက် အကူ များ ပေး ပါ ။
  - `requestSignature(tx, metadata)`
  - `approveSession(account, permissions)`
  - `reject(reason)`
  - `cancelRequest(hash)` - ပိုက်ဆံအိတ်မှ အသိအမှတ်ပြုသည့် ထိန်းချုပ်ဘောင်တစ်ခုကို ထုတ်လွှတ်သည်။
- ကိုင်တွယ်မှု အမှား- Norito အမှားကုဒ်များကို SDK သီးသန့်အမှားများသို့ မြေပုံဆွဲပါ။ ပါဝင်ပါတယ်။
  မျှဝေထားသော taxonomy (`Transport`, `Codec`, `Authorization`, `Timeout`, `QueueOverflow`, I180NI000) ကိုအသုံးပြု၍ UI အတွက် ဒိုမိန်း-သီးသန့်ကုဒ်များ။ Swift ၏ အခြေခံအချက်များကို အကောင်အထည်ဖော်ခြင်း + တယ်လီမီတာလမ်းညွှန်သည် [`connect_error_taxonomy.md`](connect_error_taxonomy.md) တွင်တည်ရှိပြီး Android/JS တူညီမှုအတွက် ရည်ညွှန်းချက်ဖြစ်သည်။
- တန်းစီခြင်းအတိမ်အနက်အတွက် တယ်လီမီတာချိတ်များ၊ အရေအတွက်များကို ပြန်လည်ချိတ်ဆက်ရန်နှင့် latency တောင်းဆိုရန် (`connect.queue_depth`၊ `connect.reconnects_total`၊ `connect.latency_ms`)။
 
## Sequence Numbers & Flow Control

- ဦးတည်ချက်တစ်ခုစီသည် စက်ရှင်ဖွင့်သည့်အခါ သုညမှစတင်သည့် သီးသန့် 64-bit `sequence` ကောင်တာကို သိမ်းဆည်းထားသည်။ မျှဝေအသုံးပြုသူ အမျိုးအစားများသည် ကုပ်နံပါတ်များ တိုးမြင့်လာပြီး ကောင်တာမချုပ်မီတွင် `ConnectError.sequenceOverflow` + သော့လှည့်ခြင်း လက်ဆွဲနှုတ်ဆက်ခြင်းကို အစပျိုးပေးသည်။
- Nonces နှင့် ဆက်စပ်ဒေတာများသည် အတွဲနံပါတ်ကို ရည်ညွှန်းသောကြောင့် ထပ်တူများကို payloads ခွဲခြမ်းစိတ်ဖြာခြင်းမရှိဘဲ ပယ်ချနိုင်ပါသည်။ SDKs များသည် `{sid, dir, seq, payload_hash}` ကို ပြန်လည်ချိတ်ဆက်မှုများတစ်လျှောက် ကူးယူဖော်ပြမှုကို အဆုံးအဖြတ်ပေးရန်အတွက် ၎င်းတို့၏ဂျာနယ်များတွင် သိမ်းဆည်းရပါမည်။
- Wallets များသည် logical window (`FlowControl` control frames) မှတဆင့် back-pressure ကို ကြော်ငြာသည်။ ဝင်းဒိုးတိုကင်တစ်ခုရနိုင်မှသာ DApps သည် ခေတ္တရပ်သည်။ ပိုက်လိုင်းများကို ကန့်သတ်ထားရန် ciphertext ကို လုပ်ဆောင်ပြီးနောက် ပိုက်ဆံအိတ်များသည် တိုကင်အသစ်များကို ထုတ်လွှတ်သည်။
- ပြန်လည်စတင်ရန် ညှိနှိုင်းမှုသည် ရှင်းလင်းပြတ်သားသည်- နှစ်ဖက်စလုံးမှ `Control::Resume { seq_app_max, seq_wallet_max, queue_depths }` ကို ပြန်လည်ချိတ်ဆက်ပြီးနောက် နှစ်ဖက်စလုံးမှ ဒေတာမည်မျှပြန်လည်ပေးပို့ခဲ့ကြောင်းနှင့် ဂျာနယ်များတွင် ကွာဟချက်ရှိမရှိကို လေ့လာသူများမှ စစ်ဆေးနိုင်မည်ဖြစ်သည်။
- ပဋိပက္ခများ (ဥပမာ၊ တူညီသော `(sid, dir, seq)` နှင့် မတူညီသော hash နှစ်ခု) သည် `ConnectError.Internal` သို့ တိုးလာပြီး အသံတိတ်ကွဲပြားမှုကို ရှောင်ရှားရန် `sid` အသစ်ကို တွန်းအားပေးပါ။

## ခြိမ်းခြောက်မှုပုံစံနှင့် ဒေတာထိန်းသိမ်းမှု ချိန်ညှိမှု- ** မျက်နှာပြင်များကို ထည့်သွင်းစဉ်းစားသည်-** WebSocket သယ်ယူပို့ဆောင်ရေး၊ Norito တံတားကုဒ်/ကုဒ်၊
  ဂျာနယ်စွဲမြဲမှု၊ တယ်လီမီတာ တင်ပို့ရောင်းချသူများ၊ နှင့် အက်ပ်မျက်နှာစာပြန်ခေါ်မှုများ။
- **မူလရည်မှန်းချက်များ-** စက်ရှင်လျှို့ဝှက်ချက်များကို ကာကွယ်ရန် (X25519 သော့များ၊ ဆင်းသက်လာသော AEAD သော့များ၊
  nonce/sequence ကောင်တာများ) မှတ်တမ်းများ/တယ်လီမီတာ ပေါက်ကြားမှုများမှ ပြန်လည်ဖွင့်ခြင်းကို တားဆီးပေးပြီး
  တိုက်ခိုက်မှုများကို အဆင့်နှိမ့်ချရန်နှင့် ဂျာနယ်များနှင့် ကွဲလွဲနေသော အစီရင်ခံစာများကို ထိန်းသိမ်းထားရန်။
- ** လျော့ပါးသက်သာစေရန် ပြင်ဆင်ထားသည်-**
  - ဂျာနယ်များသည် စာဝှက်စာသားကိုသာ သယ်ဆောင်သည်။ သိမ်းဆည်းထားသည့် မက်တာဒေတာကို ဟက်ရှ်များ၊ အရှည် ကန့်သတ်ထားသည်။
    အကွက်များ၊ အချိန်တံဆိပ်တုံးများနှင့် နံပါတ်စဉ်များ။
  - Telemetry payloads သည် မည်သည့် header/payload အကြောင်းအရာကိုမဆို ပြန်လည်တုံ့ပြန်ပြီး သာလျှင် ပါဝင်သည်။
    `sid` နှင့် အစုလိုက်ကောင်တာများ၏ ဆားနယ်ထားသော ဟက်ကာများ။ ပြန်လည်ဖြေရှင်းချက်စာရင်းကို မျှဝေခဲ့သည်။
    စာရင်းစစ်ဆိုင်ရာ တန်းတူညီမျှမှုအတွက် SDKs များအကြား။
  - စက်ရှင်မှတ်တမ်းများကို မူရင်းအတိုင်း 7 ရက်အကြာတွင် လှည့်ပတ်ထားသည်။ ပိုက်ဆံအိတ်တွေ ပြီးရော
    `connectLogRetentionDays` ခလုတ် (SDK ပုံသေ 7) နှင့် အပြုအမူကို မှတ်တမ်းတင်ပါ။
    ထို့ကြောင့် စည်းမျဥ်းချထားမှုများသည် ပိုမိုတင်းကျပ်သော ပြတင်းပေါက်များကို ချိတ်နိုင်သည်။
  - Bridge API ကို အလွဲသုံးစားလုပ်ခြင်း (နှောင်ကြိုးများ ပျောက်နေခြင်း၊ ယိုယွင်းနေသော စာဝှက်စာသား၊ မမှန်ကန်သော အတွဲများ)
    ကုန်ကြမ်း payload သို့မဟုတ် သော့များကို ပဲ့တင်ထပ်ခြင်းမရှိဘဲ ရိုက်ထည့်ထားသော အမှားများကို ပြန်ပေးသည်။

သုံးသပ်ချက်မှ စောင့်ဆိုင်းနေသော မေးခွန်းများကို `docs/source/sdk/swift/connect_workshop.md` တွင် ခြေရာခံပါသည်။
လွှတ်တော်အစည်းအဝေးများတွင် ဖြေရှင်းဆောင်ရွက်သွားမည်ဖြစ်သည်။ ပိတ်ပြီးသည်နှင့် strawman ဖြစ်လိမ့်မည်။
မူကြမ်းမှ လက်ခံရန် တိုးမြှင့်ထားသည်။

## အော့ဖ်လိုင်းကြားခံစနစ်နှင့် ပြန်လည်ချိတ်ဆက်မှုများ

### ဂျာနယ်တိုက် စာချုပ်

SDK တိုင်းသည် စက်ရှင်တစ်ခုစီတွင် နောက်ဆက်တွဲ-သီးသန့်ဂျာနယ်ကို ထိန်းသိမ်းထားသောကြောင့် dApp နှင့် ပိုက်ဆံအိတ်
အော့ဖ်လိုင်းတွင် ဘောင်များကို တန်းစီနိုင်သည်၊ ဒေတာဆုံးရှုံးမှုမရှိဘဲ ပြန်လည်စတင်နိုင်ပြီး အထောက်အထားများ ပေးနိုင်သည်။
telemetry အတွက် စာချုပ်တွင် Norito တံတားအမျိုးအစားများကို ထင်ဟပ်စေသောကြောင့် တူညီသောဘိုက်ဖြစ်သည်။
ကိုယ်စားပြုမှုသည် မိုဘိုင်း/JS အစုအဝေးများတစ်လျှောက် ရှင်သန်နေပါသည်။- ဂျာနယ်များသည် hashed session identifier (`sha256(sid)`) အောက်တွင် ဂျာနယ်နှစ်ခုကို ထုတ်လုပ်သည်
  စက်ရှင်တစ်ခုလျှင် ဖိုင်များ- `app_to_wallet.queue` နှင့် `wallet_to_app.queue`။ Swift ကိုအသုံးပြုသည်။
  sandboxed file wrapper တစ်ခု၊ Android သည် `Room`/`FileChannel` မှတစ်ဆင့် ဖိုင်များကို သိမ်းဆည်းထားသည်။
  JS သည် IndexedDB သို့ စာရေးသည်။ ဖော်မတ်အားလုံးသည် ဒွိစုံနှင့် endian-တည်ငြိမ်သည်။
- မှတ်တမ်းတစ်ခုစီသည် `ConnectJournalRecordV1` အဖြစ် စီစဥ်ထားသည်။
  - `direction: u8` (`0 = app→wallet`၊ `1 = wallet→app`)
  - `sequence: u64`
  - `payload_hash: [u8; 32]` (ciphertext + ခေါင်းစီးများ၏ Blake3)
  - `ciphertext_len: u32`
  - `received_at_ms: u64`
  - `expires_at_ms: u64`
  - `ciphertext: [u8; ciphertext_len]` (အတိအကျ Norito ဖရိန် AEAD-ထုပ်ပြီးသား)
- ဂျာနယ်များသည် စာဝှက်စာသား စကားလုံးများကို သိမ်းဆည်းထားသည်။ ကျွန်ုပ်တို့သည် payload ကို ဘယ်တော့မှ ပြန်စာဝှက်မထားပါ။ AEAD
  ခေါင်းစီးများသည် ဦးတည်ချက်သော့များကို စစ်မှန်ကြောင်း အထောက်အထားပြပြီးဖြစ်သောကြောင့် စွဲမြဲမှုအား လျော့နည်းစေသည်။
  နောက်ဆက်တွဲမှတ်တမ်းကို fsycing လုပ်ခြင်း။
- Memory ရှိ `ConnectQueueState` struct သည် ဖိုင် metadata ကို ထင်ဟပ်စေသည် (အတိမ်အနက်၊
  အသုံးပြုထားသော bytes၊ အဟောင်းဆုံး/အသစ်ဆုံး seq)။ ၎င်းသည် telemetry တင်ပို့သူများနှင့် အစာကျွေးသည်။
  `FlowControl` အကူအညီပေးသူ။
- ပုံမှန်အားဖြင့် 32 frames / 1MiB တွင် ဂျာနယ်ထုပ်၊ ဦးထုပ်ကို နှိပ်လိုက်တာနဲ့ နှင်ထုတ်တယ်။
  ရှေးအကျဆုံး ထည့်သွင်းမှုများ (`reason=overflow`)။ `ConnectFeatureConfig.max_queue_len`
  ဖြန့်ကျက်မှုတစ်ခုစီတွင် ဤပုံသေများကို အစားထိုးသည်။
- ဂျာနယ်များသည် 24 နာရီ (`expires_at_ms`) အတွက် ဒေတာကို ထိန်းသိမ်းထားသည်။ နောက်ခံ GC သည် ဟောင်းနွမ်းခြင်းကို ဖယ်ရှားပေးသည်။
  အပိုင်းများကို စိတ်အားထက်သန်စွာ ခွဲထားသောကြောင့် on-disk footprint သည် အကန့်အသတ်ရှိနေပါသည်။
- ပျက်စီးမှုဘေးကင်းရေး- နောက်ဆက်တွဲ၊ fsync နှင့် memory mirror _before_ အသိပေးခြင်းတို့ကို အပ်ဒိတ်လုပ်ပါ။
  ခေါ်ဆိုသူ။ စတင်ချိန်တွင်၊ SDKs များသည် လမ်းညွှန်ကိုစကင်န်ဖတ်ပြီး၊ မှတ်တမ်းစစ်ဆေးမှုများကို အတည်ပြုပါ
  `ConnectQueueState` ကို ပြန်လည်တည်ဆောက်ပါ။ အကျင့်ပျက်ခြစားမှုက ရာဇ၀တ်မှုကို ဖြစ်စေတယ်။
  ကျော်သွားသည်၊ တယ်လီမီတာဖြင့် အလံပြထားပြီး ပံ့ပိုးမှုအမှိုက်ပုံများအတွက် သီးသန့်ခွဲထားနိုင်သည်။
- ciphertext သည် Norito ကိုယ်ရေးကိုယ်တာစာအိတ်ကို ကျေနပ်နေပြီဖြစ်သောကြောင့် တစ်ခုတည်းသော၊
  မှတ်တမ်းတင်ထားသော နောက်ထပ် metadata သည် hashed session id ဖြစ်သည်။ အပိုအလိုရှိသောအက်ပ်များ
  ကိုယ်ရေးကိုယ်တာသည် ဂျာနယ်များကိုသိမ်းဆည်းသည့် `telemetry_opt_in = false` သို့ ရွေးချယ်နိုင်သည်။
  တန်းစီကျကျ တင်ပို့မှုများကို ပြန်လည်ပြင်ဆင်ပြီး `sid` မှတ်တမ်းများတွင် ခွဲဝေခြင်းကို ပိတ်သည်။
- SDKs သည် `ConnectQueueObserver` ကို ဖော်ထုတ်ထားသောကြောင့် wallets/dApps များသည် တန်းစီခြင်း၏ အနက်ကို စစ်ဆေးနိုင်သည်၊
  မြောင်းများနှင့် GC ရလဒ်များ၊ ဤချိတ်သည် ခွဲခြမ်းစိတ်ဖြာမှုမှတ်တမ်းများမပါဘဲ အခြေအနေ UI များကို ကျွေးမွေးသည်။

### ပြန်ဖွင့်ပြီး ဝေါဟာရပြန်စပါ။

1. ပြန်လည်ချိတ်ဆက်သည့်အခါ၊ SDK များသည် `{seq_app_max၊ `Control::Resume` ကို ထုတ်လွှတ်သည်
   seq_wallet_max၊ queued_app၊ queued_wallet၊ journal_hash}`။ hash သည် the
   Blake3 သည် နောက်ဆက်တွဲ-သာ ဂျာနယ်၏ အဆုံးအဖြတ်ဖြစ်သောကြောင့် မလိုက်ဖက်သောရွယ်တူများသည် ပျံ့လွင့်မှုကို သိရှိနိုင်သည်။
2. လက်ခံရရှိသည့်ရွယ်တူသည် ကိုယ်ရေးအကျဉ်းကို ၎င်း၏အခြေအနေ၊ တောင်းဆိုချက်များနှင့် နှိုင်းယှဉ်သည်။
   ကွက်လပ်များရှိသည့်အခါ ပြန်လည်ထုတ်လွှင့်ခြင်းနှင့် ပြန်လည်ပြသသောဘောင်များကို အသိအမှတ်ပြုသည်။
   `Control::ResumeAck`။
3. ပြန်ဖွင့်ထားသောဘောင်များသည် ထည့်သွင်းမှုအစီအစဥ်ကို အမြဲလေးစားသည် (`sequence` ထို့နောက် စာရေးချိန်)။
   Wallet SDK များသည် `FlowControl` တိုကင်များထုတ်ပေးခြင်းဖြင့် back-pressure ကို အသုံးပြုရမည် (လည်း
   ဂျာနယ်တွင်ဖော်ပြထားသည်) ထို့ကြောင့် dApps သည် အော့ဖ်လိုင်းဖြစ်နေချိန်တွင် တန်းစီခြင်းကို လွှမ်းမိုးနိုင်မည်မဟုတ်ပေ။
4. ဂျာနယ်များသည် ciphertext verbatim ကို သိမ်းဆည်းထားသောကြောင့် ပြန်လည်ဖွင့်ခြင်းသည် မှတ်တမ်းတင်ထားသော ဘိုက်များကို ရိုးရှင်းစွာ စုပ်ယူပါသည်။
   သယ်ယူပို့ ဆောင်မှုနှင့် ကုဒ်ဒါဖြင့် ပြန်ပို့သည်။ တစ်ခါ SDK ပြန်လည်ကုဒ်သွင်းခြင်းကို ခွင့်မပြုပါ။

### ပြန်လည်ချိတ်ဆက်မှု စီးဆင်းခြင်း။1. ပို့ဆောင်မှုသည် WebSocket ကို ပြန်လည်တည်ထောင်ပြီး ping ကြားကာလအသစ်ကို ညှိနှိုင်းပါ။
2. dApp သည် ပိုက်ဆံအိတ်မှ နောက်ပြန်ဖိအားကို လေးစားကာ စီတန်းထားသောဘောင်များကို အစီအစဉ်တကျ ပြန်ဖွင့်သည်။
   (`ConnectSession.nextControlFrame()` သည် `FlowControl` တိုကင်များ)။
3. Wallet သည် buffed ရလဒ်များကို ကုဒ်ကုဒ်လုပ်သည်၊ sequence monotonicity ကို စစ်ဆေးပေးပြီး၊
   စောင့်ဆိုင်းနေသော အတည်ပြုချက်များ/ရလဒ်များကို ပြန်လည်ပြသသည်။
4. နှစ်ဖက်စလုံးသည် `resume` အကျဉ်းချုပ် `seq_app_max`၊ `seq_wallet_max`၊
   နှင့် telemetry အတွက် အတိမ်အနက် တန်းစီသည်။
5. မိတ္တူဘောင်များ (`sequence` + `payload_hash`) နှင့် ကိုက်ညီသော ဘောင်များကို အသိအမှတ်ပြုပြီး ဖြုတ်ချလိုက်ပါ။ ပဋိပက္ခများသည် `ConnectError.Internal` ကို မြှင့်တင်ပြီး အတင်းအကြပ် သတ်မှတ်ချိန် ပြန်လည်စတင်ရန် အစပျိုးသည်။

### ပျက်ကွက်မုဒ်များ

- စက်ရှင်ကို ပျက်သည်ဟု ယူဆပါက (`offline_timeout_ms`၊ ပုံသေ ၅ မိနစ်)၊
  ကြားခံဘောင်များကို ဖယ်ရှားပြီး SDK သည် `ConnectError.sessionExpired` ကို မြှင့်တင်သည်။
- ဂျာနယ် အကျင့်ပျက်ခြစားမှုတွင်၊ SDK များသည် Norito ကုဒ်ကို ပြုပြင်ရန် တစ်ခုတည်းကြိုးစားပါ။ on
  မအောင်မြင်ပါက ဂျာနယ်ကို ပစ်ချလိုက်ပြီး `connect.queue_repair_failed` တယ်လီမီတာကို ထုတ်လွှတ်သည်။
- တစ်ဆက်တည်း မကိုက်ညီမှုသည် `ConnectError.replayDetected` ကို အစပျိုးစေပြီး အသစ်အသစ်သော စွမ်းအားကို ဖြစ်ပေါ်စေသည်။
  လက်ဆွဲနှုတ်ဆက်ခြင်း (`sid` အသစ်ဖြင့် ပြန်လည်စတင်ခြင်း)။

  ရောဂါရှာဖွေရေး) ပရမ်းပတာစစ်ဆေးမှုများအတွင်း၊ ဤ command သည် state transitions ကို print ထုတ်သည်၊
  ရေစာမှတ်တမ်းနှင့် အထောက်အထားများ ပြန်လည်စတင်ရန် အုပ်ချုပ်မှုပြန်လည်သုံးသပ်ခြင်းအပေါ် မမူတည်ပါ။

### အထောက်အထားအစုအဝေး အလုပ်အသွားအလာ

ပံ့ပိုးကူညီမှုနှင့် လိုက်လျောညီထွေမှုအဖွဲ့များသည် စာရင်းစစ်သည့်အခါတွင် အဆုံးအဖြတ်ပေးသည့် အထောက်အထားများအပေါ် အားကိုးသည်။
အော့ဖ်လိုင်း အပြုအမူ။ ထို့ကြောင့် SDK တစ်ခုစီသည် တူညီသော အဆင့်သုံးဆင့်တင်ပို့မှုကို လုပ်ဆောင်သည်-

1. `exportJournalBundle(..)` သည် `{app_to_wallet,wallet_to_app}.queue` အပေါင်း a ရေးသည်
   build hash၊ အင်္ဂါရပ်အလံများနှင့် disk watermarks များကိုဖော်ပြသောဖော်ပြချက်။
2. `exportQueueMetrics(..)` သည် နောက်ဆုံး 1000 တယ်လီမီတာနမူနာများကို ထုတ်လွှတ်သောကြောင့် ဒက်ရှ်ဘုတ်များ
   အော့ဖ်လိုင်းပြန်လည်တည်ဆောက်နိုင်သည်။ နမူနာများတွင် hashed session id ပါဝင်သည်။
   အသုံးပြုသူရွေးချယ်ခဲ့သည်။
3. CLI အကူအညီပေးသူက ပို့ကုန်နှစ်ခုလုံးကို ဇစ်တိုက်ပြီး Norito မက်တာဒေတာဖိုင်ကို ပူးတွဲပါရှိသည်။
   (`ConnectQueueEvidenceV1`) ထို့ကြောင့် Torii သည် အစုအဝေးကို SoraFS တွင် သိမ်းဆည်းနိုင်သည်။

အတည်ပြုခြင်းပျက်ကွက်သောအစုအဝေးများကို `connect.evidence_invalid` ဖြင့် ပယ်ချပါသည်။
SDK အဖွဲ့သည် တင်ပို့သူကို မျိုးပွားပြီး ဖာထေးနိုင်စေရန် တယ်လီမီတာဖြင့် တိုင်းတာသည်။

## Telemetry & Diagnostics- မျှဝေထားသော OpenTelemetry တင်ပို့သူများမှတစ်ဆင့် Norito JSON ဖြစ်ရပ်များကို ထုတ်လွှတ်ပါ။ မဖြစ်မနေ တိုင်းတာမှုများ-
  - `connect.queue_depth{direction}` (gauge) ကို `ConnectQueueState` မှ ကျွေးမွေးပါသည်။
  - ဒစ်ခ်ကျောထောက်နောက်ခံပြုထားသောခြေရာအတွက် `connect.queue_bytes{direction}` (gauge)။
  - `overflow|ttl|repair` အတွက် `connect.queue_dropped_total{reason}` (ကောင်တာ)။
  - `connect.offline_flush_total{direction}` (ကောင်တာ) တန်းစီသည့်အခါ အတိုးများ
    သယ်ယူပို့ဆောင်ရေးမပါဘဲမြောင်း; ကျရှုံးမှုများ တိုးလာခြင်း `connect.offline_flush_failed`။
  - `connect.replay_success_total`/`connect.replay_error_total`။
  - `connect.resume_latency_ms` histogram (ပြန်လည်ချိတ်ဆက်ခြင်းနှင့် တည်ငြိမ်ခြင်းကြား အချိန်
    ပြည်နယ်) အပေါင်း `connect.resume_attempts_total`။
  - `connect.session_duration_ms` histogram (ပြီးဆုံးသည့်စက်ရှင်တစ်ခုလျှင်)။
  - `connect.error` `code`၊ `fatal`၊ `telemetry_profile` ဖြင့် ဖွဲ့စည်းထားသော ဖြစ်ရပ်များ။
- ပို့ကုန်များသည် `{platform, sdk_version, feature_hash}` တံဆိပ်များ ပူးတွဲပါရှိရမည်။
  ဒက်ရှ်ဘုတ်များကို SDK တည်ဆောက်ခြင်းဖြင့် ခွဲနိုင်သည်။ hashed `sid` သည် စိတ်ကြိုက်ရွေးချယ်နိုင်ပြီး အားလုံးအတွက်ဖြစ်သည်။
  တယ်လီမီတာ ရွေးချယ်မှု မှန်သောအခါ ထုတ်လွှတ်သည်။
- SDK အဆင့်ချိတ်များသည် တူညီသောဖြစ်ရပ်များကို ပေါ်လွင်စေသောကြောင့် အပလီကေးရှင်းများက ပိုမိုအသေးစိတ်အချက်အလက်များကို ထုတ်ယူနိုင်သည်-
  - Swift: `ConnectSession.addObserver(_:) -> ConnectEvent`။
  - Android: `Flow<ConnectEvent>`။
  - JS- async iterator သို့မဟုတ် callback။
- CI gating- လျင်မြန်သောအလုပ်များ `make swift-ci`၊ Android သည် `./gradlew sdkConnectCi` ကိုအသုံးပြုသည်၊
  JS သည် `npm run test:connect` ကို run သောကြောင့် telemetry/dashboards များသည် အစိမ်းရောင်ရှိနေသည်
  ပေါင်းစည်းခြင်း Connect အပြောင်းအလဲများ။
- ဖွဲ့စည်းတည်ဆောက်ထားသော မှတ်တမ်းများတွင် ဖျက်ထားသော `sid`၊ `seq`၊ `queue_depth` နှင့် `sid_epoch` တို့ ပါဝင်သည်
  တန်ဖိုးများသည် အော်ပရေတာများသည် ဖောက်သည်ပြဿနာများကို ဆက်စပ်ပေးနိုင်သောကြောင့်ဖြစ်သည်။ ပြုပြင်မှု ပျက်ကွက်သော ဂျာနယ်များ ထွက်ပေါ်လာသည်။
  `connect.queue_repair_failed{reason}` ဖြစ်ရပ်များ နှင့် ရွေးချယ်နိုင်သော ပျက်စီးမှု အမှိုက်ပုံလမ်းကြောင်း။

### ကြေးနန်းချိတ်များနှင့် အုပ်ချုပ်မှုဆိုင်ရာ အထောက်အထားများ

- `connect.queue_state` သည် လမ်းပြမြေပုံအန္တရာယ်ညွှန်ပြချက်အဖြစ် နှစ်ဆတိုးလာသည်။ ဒက်ရှ်ဘုတ်အဖွဲ့
  `{platform, sdk_version}` ဖြင့် အုပ်ချုပ်မှုနမူနာယူနိုင်စေရန် အချိန်-နိုင်ငံတော်သို့ တင်ဆက်ပါ
  အဆင့်လိုက် ထုတ်လွှင့်မှုများကို အတည်ပြုခြင်းမပြုမီ လစဉ်လေ့ကျင့်မှု အထောက်အထား။
- `connect.queue_watermark` နှင့် `connect.queue_bytes` သည် Connect အန္တရာယ်ရမှတ်ကို ဖြည့်သွင်းသည်
  (`risk.connect.offline_buffer`) သည် SRE ထက်ပိုသည့်အခါ အလိုအလျောက် စာမျက်နှာများကို ပေးသည်။
  စက်ရှင်များ၏ 5% သည် `Throttled` တွင် > 10 မိနစ်ကြာသည်။
- တင်ပို့သူများသည် `feature_hash` ကို ပွဲတိုင်းတွင် ပူးတွဲပါရှိသောကြောင့် စာရင်းစစ်ကိရိယာကို အတည်ပြုနိုင်သည်
  Norito codec + အော့ဖ်လိုင်းအစီအစဉ်သည် ပြန်လည်သုံးသပ်ထားသော တည်ဆောက်မှုနှင့် ကိုက်ညီကြောင်း။ SDK CI မအောင်မြင်ပါ။
  Telemetry သည် အမည်မသိ hash တစ်ခုကို သတင်းပို့သောအခါ မြန်ဆန်သည်။
- strawman သည် ခြိမ်းခြောက်မှုပုံစံ နောက်ဆက်တွဲတစ်ခု လိုအပ်နေသေးသည်။ မက်ထရစ်များထက်ကျော်လွန်သောအခါ
  မူဝါဒသတ်မှတ်ချက်များ၊ SDK များသည် `connect.policy_violation` ဖြစ်ရပ်များကို အကျဉ်းချုပ်ထုတ်လွှတ်သည်
  စော်ကားသော Sid (hashed)၊ ပြည်နယ်နှင့် ဖြေရှင်းထားသော လုပ်ဆောင်မှု (`drain|purge|quarantine`)။
- တူညီသော SoraFS အမည်စကွက်များတွင် `exportQueueMetrics` မှတစ်ဆင့် ဖမ်းယူထားသော အထောက်အထား
  Connect runbook သည် အနုပညာလက်ရာများဖြစ်သောကြောင့် ကောင်စီမှ သုံးသပ်သူများသည် လေ့ကျင့်မှုတိုင်းကို ခြေရာခံနိုင်သည်။
  အတွင်းမှတ်တမ်းများ မတောင်းဆိုဘဲ သီးခြား telemetry နမူနာများသို့ ပြန်သွားပါ။

## Frame Ownership & Responsibilities| Frame / Control | ပိုင်ရှင် | တစ်ဆက်တည်း ဒိုမိန်း | ဂျာနယ်ဆက်ရှိနေသလား? | Telemetry အညွှန်းများ | မှတ်စုများ |
|--------------------|-------|-----------------|----------------------------------------------------|--------|
| `Control::Open` | dApp | `seq_app` | ✅ (`app_to_wallet`)| `event=open` | မက်တာဒေတာ + ခွင့်ပြုချက်ဘစ်မြေပုံကို သယ်ဆောင်သည်။ ညွှန်ကြားချက်များမပြမီ နောက်ဆုံးဖွင့်ထားသော ပိုက်ဆံအိတ်များကို ပြန်ဖွင့်ပါ။ |
| `Control::Approve` | ပိုက်ဆံအိတ် | `seq_wallet` | ✅ (`wallet_to_app`)| `event=approve` | အကောင့်၊ အထောက်အထားများ၊ လက်မှတ်များ ပါဝင်သည်။ မက်တာဒေတာဗားရှင်း တိုးမြှင့်မှုများကို ဤနေရာတွင် မှတ်တမ်းတင်ထားသည်။ |
| `Control::Reject` | ပိုက်ဆံအိတ် | `seq_wallet` | ✅ | `event=reject`, `reason` | စိတ်ကြိုက်ရွေးချယ်နိုင်သော မက်ဆေ့ဂျ်၊ dApp သည် ဆိုင်းငံ့ထားသော ဆိုင်းဘုတ်တောင်းဆိုမှုများကို ရုပ်သိမ်းလိုက်သည်။ |
| `Control::Close` (init) | dApp | `seq_app` | ✅ | `event=close`, `initiator=app` | Wallet သည် ၎င်း၏ကိုယ်ပိုင် `Close` ဖြင့် အသိအမှတ်ပြုသည်။ |
| `Control::Close` (ack) | ပိုက်ဆံအိတ် | `seq_wallet` | ✅ | `event=close`, `initiator=wallet` | မျက်ရည်ကျကြောင်းအတည်ပြု; ဘောင်ကို နှစ်ဖက်စလုံးက ဆက်လက်လုပ်ဆောင်သည်နှင့် GC သည် ဂျာနယ်များကို ဖယ်ရှားသည်။ |
| `SignRequest` | dApp | `seq_app` | ✅ | `event=sign_request`, `payload_hash` | ပြန်လည်ပြသရန် ပဋိပက္ခရှာဖွေခြင်းအတွက် Payload hash ကို မှတ်တမ်းတင်ထားသည်။ |
| `SignResult` | ပိုက်ဆံအိတ် | `seq_wallet` | ✅ | `event=sign_result`, `status=ok|err` | လက်မှတ်ထိုးထားသော ဘိုက်များ၏ BLAKE3 hash ပါ၀င်သည် ။ ကျရှုံးမှုများသည် `ConnectError.Signing` ကိုတိုးစေသည်။ |
| `Control::Error` | ပိုက်ဆံအိတ် (အများစု) / dApp (သယ်ယူပို့ဆောင်ရေး) | ပိုင်ရှင်နှင့် ကိုက်ညီသော ဒိုမိန်း | ✅ | `event=error`, `code` | ဆိုးရွားသော အမှားများသည် သတ်မှတ်ချိန်ကို ပြန်လည်စတင်ရန် တွန်းအားပေးသည်။ တယ်လီမီတာအမှတ် `fatal=true`။ |
| `Control::RotateKeys` | ပိုက်ဆံအိတ် | `seq_wallet` | ✅ | `event=rotate_keys`, `reason` | ဦးတည်ချက်သော့အသစ်များကို ကြေညာသည်။ dApp သည် `RotateKeysAck` ဖြင့် စာပြန်သည် (အက်ပ်ဘက်ခြမ်းတွင် ဂျာနယ်ထုတ်ထားသည်)။ |
| `Control::Resume` / `ResumeAck` | နှစ်ခုလုံး | ဒေသတွင်း ဒိုမိန်း သီးသန့် | ✅ | `event=resume`, `direction=app|wallet` | တန်းစီခြင်းအတိမ်အနက် + seq အခြေအနေ အကျဉ်းချုပ်; hashed journal digest aids diagnosis ။ |

- Directional cipher keys များသည် အခန်းကဏ္ဍအလိုက် အချိုးညီစွာ ရှိနေသည် (`app→wallet`, `wallet→app`)။
  ပိုက်ဆံအိတ်လှည့်ခြင်း အဆိုပြုချက်များကို `Control::RotateKeys` နှင့် dApps မှတစ်ဆင့် ကြော်ငြာထားသည်
  `Control::RotateKeysAck` ကို ထုတ်လွှတ်ခြင်းဖြင့် အသိအမှတ်ပြုပါ။ frame နှစ်ခုလုံးသည် disk ကိုထိရပါမည်။
  ကွက်လပ်များကို ရှောင်ရှားရန် သော့လဲလှယ်ခြင်းမပြုမီ။
- မက်တာဒေတာ ပူးတွဲပါဖိုင် (အိုင်ကွန်များ၊ ဒေသန္တရပြုထားသော အမည်များ၊ လိုက်နာမှု အထောက်အထားများ) ကို လက်မှတ်ရေးထိုးထားသည်။
  ပိုက်ဆံအိတ်ကို dApp ဖြင့် သိမ်းဆည်းထားသည်။ အပ်ဒိတ်များတွင် လတ်ဆတ်သော အတည်ပြုချက်ဘောင်တစ်ခု လိုအပ်ပါသည်။
  `metadata_version` တိုးလာသည်။
- အထက်ဖော်ပြပါ ပိုင်ဆိုင်မှု matrix ကို SDK docs မှ ကိုးကားထားသောကြောင့် CLI/web/automation
  ဖောက်သည်များသည် တူညီသော စာချုပ်နှင့် စာရွက်စာတမ်းဆိုင်ရာ ပုံသေများကို လိုက်နာကြသည်။

## မေးခွန်းများဖွင့်ပါ။

1. **Session ရှာဖွေတွေ့ရှိမှု**- WalletConnect ကဲ့သို့ QR ကုဒ်များ/ လက်ဆွဲနှုတ်ဆက်ခြင်းမျိုး လိုအပ်ပါသလား။ (အနာဂတ်အလုပ်။)
2. **Multisig**- လက်မှတ်ပေါင်းများစွာ အတည်ပြုချက်များကို မည်သို့ကိုယ်စားပြုသနည်း။ (လက်မှတ်များစွာကို ပံ့ပိုးရန်အတွက် ရလဒ်ကို တိုးချဲ့ပါ။)
3. **လိုက်နာမှု**- ထိန်းညှိစီးဆင်းမှုများ (လမ်းပြမြေပုံတစ်ခု) အတွက် မည်သည့်နယ်ပယ်များ မဖြစ်မနေလိုအပ်သနည်း။ (လိုက်နာရမည့် အဖွဲ့၏ လမ်းညွှန်ချက်ကို စောင့်မျှော်နေပါသည်။)
4. **SDK ထုပ်ပိုးခြင်း**- ကျွန်ုပ်တို့သည် မျှဝေထားသောကုဒ် (ဥပမာ၊ Norito Connect codecs) ကို ပလပ်ဖောင်းပုံစံသေတ္တာထဲသို့ ထည့်သင့်သလား။ (TBD။)

- SDK အလိုက် အကောင်အထည်ဖော်မှုအပိုင်းကို အချိန်ဇယားဆွဲပါ (Swift IOS7၊ Android AND7၊ JS Connect မှတ်တိုင်များ)။
