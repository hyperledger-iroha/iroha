* 概要

本書は、Iroha v2 のプレリリース候補を円滑に公開するための推奨手順と、v2.0.0 を一般公開したあとや後続のセマンティックバージョンのパッケージを配布するときに手順をどのように調整すべきかをまとめたものです。

* タイムベースのリリースサイクル

Iroha にはプロジェクトを区切る限定的なスコープや機能ロードマップがなく、開発の途中で新しいアイデアが提案されることも多いため、機能完了を基準にしたリリースではなく時間ベースのリリースサイクルを採用しています。この方法により、SDK との変更同期が小さな単位で進むため移行が滑らかになり、アイデアも徐々に擦り合わせることができます。コア開発チームはアジャイルな状態を維持でき、状況の変化に柔軟に対応できます。

ただし、開発ペースが落ちて機能追加や修正よりもリファクタや CI 改善が中心になる場合には、この cadence のデメリットが顕在化する可能性があります。それでも可能な限り継続性を重視し、タイムベースのリリースを維持することが推奨されます。

* リリースウィンドウ

リリースウィンドウとは、特定のリリースを告知してからリリース手順を完了するまでの期間を指します。延期されたテストなどは含みません。

一般的なリリースウィンドウは 1〜2 週間程度です（特に難易度の高いリリースでは長くなる場合があります）。

* 内部リリース

特定のビルドに主要 SDK のサポートが欠けていたり、重大な不具合が存在して短期間で修正できない場合、テクニカルリードは [[*Release process][プレリリース同期ミーティング]] の場でそのリリースを「内部向け」と宣言できます。主要なステークホルダーが、直接必要としない機能の実装負荷を軽減するためにスナップショットの公開を求めることもあります。

* リリーススケジュール

スケジュールの決定はテクニカルリードに委ねられますが、経験則として毎月 28 日を含む週にリリースし、月次リリースを維持することが推奨されます。

多くの要望が寄せられている機能を短期間で実装できる見込みがある場合、リリースを延期することができます。見積もりが外れて実装に時間がかかるようなら、リリースをさらに遅らせるのではなく、その機能を次回リリースに回すべきです（いわゆる「ストライクワン」ポリシー）。

以前に延期した分を取り戻す、あるいは重要な機能の提供を早めるためにリリースを前倒しすることも可能です。

他プロジェクトの締切に合わせてリリース時期を動かすことは推奨されませんが、最終判断はテクニカルリードに委ねられます。

* 開発

Iroha レジャーへの変更は =main= ブランチにコミットします。すべてのコミットには署名（sign-off）が必要です（法的責任の観点）。マージされるコミットはすべて CI が「All green」であることが前提です。未解消の要件を残したままマージするかどうかはテクニカルリードが判断し、その結果生じる問題について責任を負います。

開発は [[../../CONTRIBUTING.md]] に従って進めます。Soramitsu の従業員は、個人フォークではなく upstream の専用ブランチで作業することも認められています（Hyperledger が推奨する標準手法）。コミットメッセージは冗長にしすぎないでください。複数の課題にまたがる修正は、時間的な制約など特別な事情がない限り分割することが望ましいとされます。コミットの内容に合わせてメッセージを調整する裁量はテクニカルリードに与えられており、社員はそれに口頭または文書で同意する必要があります。執筆者が他のメールアドレス（GitHub 提供のアドレスなど）を使用していて本人のものだと確認できる場合、テクニカルリードは DCO チェックを手動で通すことができます。

* リリースプロセス

リリースウィンドウが開く 1 週間前に、プロジェクトマネージャーがプレリリース同期ミーティングを開きます。テクニカルリードは主要な API 破壊的変更について参加者に説明し、必要な対処を促します。その準備として、テクニカルリードは =stable= ブランチと =main= ブランチの差分を確認し、 =schema.json= の変更やその他 API 破壊の兆候をチェックすることが推奨されます。

この時点以降、重大な API 破壊的変更は推奨されません。既にプルリクエストが存在する場合を除き、そのような変更は次回リリースへ延期し、明示的にマークしておきます。コアチームと SDK チームの両方がリリースに合意したら、Telegram の =i2::lib= チャンネルでリリースウィンドウの開始を告知します。

同時に、Iroha v2 の OSS コミュニティにも変更内容を共有してください。Telegram チャンネル、Hyperledger Discord チャンネルを通じて告知し、規模の大きい変更については [[*RFC process][RFC プロセス]] を開始します。

リリースウィンドウを開く発表のあと、継続的デリバリ（CD）システムを通じて =nightly= Docker コンテナを公開します。続けて =main= ブランチから =stable=（および [[lts_selection.org]] に従って =iroha2-lts=）へのプルリクエストを作成し、説明にリリースチェックリストを記載します。チェック項目は達成次第チェックマークを付けます。

重大な不具合が見つかった場合はチャットで報告し、コアチームが対応します。このためにコアチームは当番制でエンジニアを配置し、問い合わせに対応できるようにします。同じ変更に複数のエンジニアが関与している場合は、その変更を実装した人が優先的に回答します。

リリースウィンドウが閉じ（チェックリストが完了し）たら、Nix、=AppImage=、=.deb=、=.rpm= など標準的なパッケージを生成して各配布場所にアップロードします。最後に正式な GitHub リリースを作成し、リンクを関連チャットに共有します。

* リリースチェックリスト

以下は関連ブランチに対するプルリクエストの説明に含めることが推奨される内容です。

- [ ] 現在の =main= がすべての CI チェックを通過している。
- [ ] 最新の =nightly= ビルドが公開され、主要 SDK 開発者にリンクを共有した。
- [ ] SDK のコード生成（必要な場合）が成功した。
- [ ] SDK のテストがすべて成功した。
- [ ] SDK エンジニアがリリースに承認を与えた。
- [ ] 各ターゲットネットワーク向けに =scripts/select_release_profile.py= を実行し、Sora Nexus が Iroha 3 バンドルを取得する一方で他ネットワークは Iroha 2 を使うようにした（チェックリストに =artifacts/network_profiles.json= を添付）。
- [ ] =scripts/generate_release_manifest.py= で生成した =artifacts/release_manifest.json= をアーカイブし、tarball/AppImage を署名・アップロードする前の正準ハッシュリストとして保管した。
- [ ] テストの不安定性の記録を行った（リリース後に実施してもよい）。
- [ ] ワークスペース全体のベンチマーク結果を記録した。
- [ ] すべてのクレートが個別にコンパイルできることを確認した。
- [ ] クレートの全フィーチャー組み合わせがコンパイルできることを確認した。
- [ ] ワークスペース内のクレートのバージョンを更新した。
- [ ] コミュニティマネージャーがチェンジログを作成した。

* バックポート方針

安定版リリースに対してバックポートすることは避けてください。問題が緊急であれば、次回リリースの日程を前倒しして対処するのが望ましいです。それでも問題による損失が大きい場合は、適切なセマンティックバージョンを付けた緊急リリースを行う必要があります。

* RFC プロセス

Iroha v2 の重要な変更や機能拡張は、アーキテクチャ変更プロセスである「Request for Comments (RFC)」を経る必要があります。手順は次のとおりです。

1. 変更を提案する人は、=docs/source/rfc/= 配下にファイルを追加するプルリクエストを =main= ブランチ向けに作成します。ファイル形式は任意（Markdown, Org, reStructuredText など）ですが、変更内容を詳細に説明してください。

   代替案の比較まで必須ではありませんが、可能であれば盛り込むことが推奨されます。記載内容は最新情報に基づいて調査したものであるべきです。

   文書はドラフトとして公開してもかまいません。レビュー中に寄せられたコメントに対応しつつ、必要に応じて内容を拡充できます。

2. 文書は最低 2 週間のレビュー期間を設けます。その間、主要なステークホルダーやテクニカルチーム、コミュニティが内容を精査し、提案・議論・質問を行います。

   期間を延長することはできますが、1 週間未満に短縮することはできません。ステークホルダーが情報を理解し、変更提案を行う十分な時間が確保できないためです。ドラフトとして公開した場合は、著者が文書が完成したと判断した時点から期間を数えます。

3. オプションですが、コミュニティや主要ステークホルダー向けの説明会を、コメント対応が完了したあとに開催できます。

   文書に主要ステークホルダーへ悪影響を及ぼす可能性のある内容や、複数の実行計画が含まれている場合には *説明会を必須* とします。

   説明会では追加の質疑応答を行えます。コメントが意思決定に関係する（たとえばステークホルダーが実行計画の重大な欠陥を指摘した）場合は、著者が文書内で長期的に追跡すべきコメントとして残すよう依頼できます。

4. RFC ミーティングのあと、実行計画に変更が生じた場合（実装上の変更など）は、その内容を RFC 文書にも反映させなければなりません。

* 脚注
