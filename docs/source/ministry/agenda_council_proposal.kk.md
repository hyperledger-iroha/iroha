# Күн тәртібі кеңесінің ұсыныс схемасы (MINFO-2a)

Жол картасының анықтамасы: **MINFO-2a — Ұсыныс форматының валидаторы.**

Күн тәртібіндегі кеңестің жұмыс процесі азаматтар ұсынған қара тізім мен саясатты өзгерту пакеттерін жасайды
ұсыныстарды басқару комиссиялары қарау алдында. Бұл құжат анықтайды
пайдалы жүктеменің канондық схемасы, дәлелдерге қойылатын талаптар және қайталануды анықтау ережелері
жаңа валидатор (`cargo xtask ministry-agenda validate`) тұтынған, сондықтан
Ұсынушылар JSON жіберілімдерін порталға жүктеп салмас бұрын жергілікті түрде сыза алады.

## Жүктемеге шолу

(`iroha_data_model::ministry::AgendaProposalV1`). Өрістер JSON ретінде кодталған кезде
CLI/портал беттері арқылы жіберу.

### Мақсатты жазбалар (`targets[]`)

Әрбір мақсат ұсыныспен сілтеме жасалған хэштер отбасы дайджестін білдіреді.

| Өріс | Сипаттама | Валидация |
|-------|-------------|------------|
| `label` | Тексеруші контекстіне ыңғайлы атау. | Бос емес. |
| `hash_family` | Хэш идентификаторы (`blake3-256`, `sha256`, т.б.). | ASCII әріптері/цифрлары/`-_.`, ≤48 таңба. |
| `hash_hex` | Кіші он алты әріппен кодталған дайджест. | ≥16 байт (32 он алтылық таңба) және жарамды он алтылық болуы керек. |
| `reason` | Неліктен дайджест әрекет ету керектігінің қысқаша сипаттамасы. | Бос емес. |

Валидатор бірдей ішіндегі қайталанатын `hash_family:hash_hex` жұптарын қабылдамайды.
бірдей саусақ ізі бұрыннан бар болса, ұсыныс пен есептер қайшы келеді
қайталанатын тізілім (төменде қараңыз).

### Дәлелдемелер тіркемелері (`evidence[]`)

Тексерушілер қолдау көрсететін мәтінмәнді ала алатын дәлел жазбалары құжаты.| Өріс | |түрі Ескертпелер |
|-------|------|-------|
| `kind` | enum (`url`, `torii-case`, `sorafs-cid`, `attachment`) | Дайджест талаптарын анықтайды. |
| `uri` | жол | HTTP(S) URL мекенжайы, Torii іс идентификаторы немесе SoraFS URI. |
| `digest_blake3_hex` | жол | `sorafs-cid` және `attachment` түрлері үшін қажет; басқалар үшін міндетті емес. |
| `description` | жол | Рецензенттер үшін қосымша еркін форматтағы мәтін. |

### Тізілімнің көшірмелері

Операторлар көшірмелерді болдырмау үшін бар саусақ іздерінің тізілімін жүргізе алады
жағдайлар. Валидатор келесі пішіндегі JSON файлын қабылдайды:

```json
{
  "entries": [
    {
      "hash_family": "blake3-256",
      "hash_hex": "0d714bed4b7c63c23a2cf8ee9ce6c3cde1007907c427b4a0754e8ad31c91338d",
      "proposal_id": "AC-2025-014",
      "note": "Already handled in 2025-08 incident"
    }
  ]
}
```

Ұсыныс мақсаты жазбаға сәйкес келгенде, валидатор егер ол болмаса, тоқтатады
`--allow-registry-conflicts` көрсетілген (ескертулер әлі де шығарылады).
үшін [`cargo xtask ministry-agenda impact`](impact_assessment_tooling.md) пайдаланыңыз.
телнұсқаға сілтеме жасайтын референдумға дайын түйіндемені жасаңыз
тізілім және саясат суреттері.

## CLI пайдалану

Бір ұсынысты сызыңыз және оны қайталанатын тізілімге қарсы тексеріңіз:

```bash
cargo xtask ministry-agenda validate \
  --proposal docs/examples/ministry/agenda_proposal_example.json \
  --registry docs/examples/ministry/agenda_duplicate_registry.json
```

Қайталанатын соққыларды ескертуге дейін төмендету үшін `--allow-registry-conflicts` өтіңіз.
тарихи аудиттер жүргізу.

CLI бірдей Norito схемасына және жеткізілген тексеру көмекшілеріне сүйенеді.
`iroha_data_model`, сондықтан SDK/порталдар `AgendaProposalV1::validate` қайта пайдалана алады
дәйекті мінез-құлық әдісі.

## Сұрыптау CLI (MINFO-2b)

Жол картасының анықтамасы: **MINFO-2b — Көп ұяшықты сұрыптау және аудит журналы.**

Күн тәртібі кеңесінің тізімі енді азаматтар үшін детерминирленген сұрыптау арқылы басқарылады
әрбір ұтыс ойынын өз бетінше тексере алады. Жаңа пәрменді пайдаланыңыз:

```bash
cargo xtask ministry-agenda sortition \
  --roster docs/examples/ministry/agenda_council_roster.json \
  --slots 3 \
  --seed 0123456789abcdef0123456789abcdef0123456789abcdef0123456789abcdef \
  --out artifacts/ministry/agenda_sortition_2026Q1.json
```

- `--roster` — әрбір жарамды мүшені сипаттайтын JSON файлы:

  ```json
  {
    "format_version": 1,
    "members": [
      {
        "member_id": "citizen:ada",
        "weight": 2,
        "role": "citizen",
        "organization": "Artemis Cooperative"
      },
      {
        "member_id": "citizen:erin",
        "weight": 1,
        "role": "citizen",
        "eligible": false
      }
    ]
  }
  ```

  Мысал файлы мекендейді
  `docs/examples/ministry/agenda_council_roster.json`. Қосымша өрістер (рөл,
  ұйым, контакт, метадеректер) Merkle парағында түсіріледі, сондықтан аудиторлар
  ұтыс ойынын қамтамасыз еткен тізімді дәлелдей алады.

- `--slots` — толтырылатын кеңес орындарының саны.
- `--seed` — файлда жазылған 32 байт BLAKE3 тұқымы (64 кіші әріп алтылық таңба)
  ұтыс ойынының басқару хаттамасы.
- `--out` — қосымша шығыс жолы. Өткізілмесе, JSON қорытындысы басып шығарылады
  stdout.

### Шығару қорытындысы

Пәрмен `SortitionSummary` JSON блогын шығарады. Үлгі шығысы мына жерде сақталады
`docs/examples/ministry/agenda_sortition_summary_example.json`. Негізгі өрістер:

| Өріс | Сипаттама |
|-------|-------------|
| `algorithm` | Сұрыптау белгісі (`agenda-sortition-blake3-v1`). |
| `roster_digest` | Тізім файлының BLAKE3 + SHA-256 дайджесттері (аудиттердің бір мүше тізімі бойынша жұмыс істейтінін растау үшін пайдаланылады). |
| `seed_hex` / `slots` | Аудиторлар ұтыс ойынын қайта шығаруы үшін CLI кірістерін қайталаңыз. |
| `merkle_root_hex` | Тізім Merkle ағашының түбірі (`hash_node`/`hash_leaf` көмекшілері `xtask/src/ministry_agenda.rs`). |
| `selected[]` | Әрбір ұяшыққа арналған жазбалар, соның ішінде канондық мүше метадеректері, жарамды индекс, бастапқы тізім индексі, детерминирленген сызба энтропиясы, жапырақ хэші және Merkle дәлелі бауырлары. |

### Ұтыс ойынын тексеру1. `roster_path` сілтеме жасаған тізімді алып, оның BLAKE3/SHA-256-ны тексеріңіз
   дайджесттер қорытындыға сәйкес келеді.
2. CLI-ді бірдей тұқым/слоттар/тізіммен қайта іске қосыңыз; алынған `selected[].member_id`
   реті жарияланған түйіндемеге сәйкес келуі керек.
3. Арнайы мүше үшін, серияланған JSON мүшесін пайдаланып Merkle жапырағын есептеңіз
   (`norito::json::to_vec(&sortition_member)`) және әрбір дәлелдеу хэшінде бүктеңіз. Финал
   дайджест `merkle_root_hex` тең болуы керек. Мысал жиынындағы көмекші көрсетеді
   `eligible_index`, `leaf_hash_hex` және `merkle_proof[]` біріктіру жолы.

Бұл артефактілер тексерілетін кездейсоқтық үшін MINFO-2b талабын қанағаттандырады,
k-of-m таңдау және тізбектегі API қосылғанша тек тексеру журналдарын қосу.

## Тексеру қатесінің анықтамасы

`AgendaProposalV1::validate` `AgendaProposalValidationError` нұсқаларын шығарады
пайдалы жүк линиядан шықса. Төмендегі кестеде ең көп тарағандар келтірілген
қателер, сондықтан порталды тексерушілер CLI шығысын әрекет етуші нұсқаулыққа аудара алады.| Қате | Мағынасы | Түзету |
|-------|---------|-------------|
| `UnsupportedVersion { expected, found }` | `version` пайдалы жүктемесі валидатор қолдау көрсететін схемасынан ерекшеленеді. | Нұсқа `expected` сәйкес келуі үшін соңғы схемалар бумасын пайдаланып JSON файлын қайта жасаңыз. |
| `MissingProposalId` / `InvalidProposalIdFormat { value }` | `proposal_id` бос немесе `AC-YYYY-###` пішінінде жоқ. | Қайта жіберу алдында құжатталған пішімге сәйкес бірегей идентификаторды толтырыңыз. |
| `MissingSubmissionTimestamp` | `submitted_at_unix_ms` нөлге тең немесе жоқ. | Unix миллисекундтарында жіберу уақыт белгісін жазыңыз. |
| `InvalidLanguageTag { value }` | `language` жарамды BCP‑47 тегі емес. | `en`, `ja-JP` сияқты стандартты тегті немесе BCP‑47 арқылы танылған басқа тілді пайдаланыңыз. |
| `MissingSummaryField { field }` | `summary.title`, `.motivation` немесе `.expected_impact` біреуі бос. | Көрсетілген жиынтық өріс үшін бос емес мәтінді беріңіз. |
| `MissingSubmitterField { field }` | `submitter.name` немесе `submitter.contact` жоқ. | Тексерушілер ұсыныс берушімен байланыса алуы үшін жетіспейтін жіберушінің метадеректерін жеткізіңіз. |
| `InvalidTag { value }` | `tags[]` жазбасы рұқсат етілген тізімде жоқ. | Тегті құжатталған мәндердің біріне алып тастаңыз немесе атын өзгертіңіз (`csam`, `malware`, т.б.). |
| `MissingTargets` | `targets[]` массиві бос. | Кемінде бір мақсатты хэш тобының жазбасын қамтамасыз етіңіз. |
| `MissingTargetLabel { index }` / `MissingTargetReason { index }` | Мақсатты жазбада `label` немесе `reason` өрістері жоқ. | Қайта жіберу алдында индекстелген жазба үшін қажетті өрісті толтырыңыз. |
| `InvalidHashFamily { index, value }` | Қолдау көрсетілмейді `hash_family` белгісі. | Хэш тегінің ASCII алфавиттік-сандық және `-_`-мен шектелуі. |
| `InvalidHashHex { index, value }` / `TargetDigestTooShort { index }` | Дайджест жарамсыз он алтылық немесе 16 байттан қысқа. | Индекстелген мақсат үшін кіші әріппен он алтылық дайджестті (≥32 он алтылық таңба) қамтамасыз етіңіз. |
| `DuplicateTarget { index, fingerprint }` | Мақсатты дайджест бұрынғы жазбаның немесе тізілімнің саусақ ізін қайталайды. | Көшірмелерді жойыңыз немесе растайтын дәлелдерді бір мақсатқа біріктіріңіз. |
| `MissingEvidence` | Ешқандай дәлелдеме қосымшалары берілмеді. | Репродукциялық материалға сілтеме жасайтын кем дегенде бір дәлелдеме жазбасын тіркеңіз. |
| `MissingEvidenceUri { index }` | Дәлелдеу жазбасында `uri` өрісі жоқ. | Индекстелген дәлел жазбасы үшін алынатын URI немесе іс идентификаторын қамтамасыз етіңіз. |
| `MissingEvidenceDigest { index }` / `InvalidEvidenceDigest { index, value }` | Дайджестті қажет ететін дәлел жазбасы (SoraFS CID немесе тіркеме) жоқ немесе жарамсыз `digest_blake3_hex`. | Индекстелген жазба үшін 64 таңбадан тұратын кіші әріпті BLAKE3 дайджестін жеткізіңіз. |

## Мысалдар

- `docs/examples/ministry/agenda_proposal_example.json` — канондық,
  екі дәлелдеме тіркемесі бар lint-clean ұсыныс пайдалы жүктемесі.
- `docs/examples/ministry/agenda_duplicate_registry.json` — іске қосу тізілімі
  жалғыз BLAKE3 саусақ ізі және негіздемесі бар.

Портал құралдарын біріктіру немесе CI жазу кезінде бұл файлдарды үлгілер ретінде қайта пайдаланыңыз
автоматтандырылған жіберулерді тексереді.
