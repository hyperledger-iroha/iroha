საგზაო რუკის პუნქტი **MINFO-5 — პოლიტიკის ჟიურის ხმის მიცემის ხელსაწყოები** საჭიროებს პორტატულ ფორმატს
ნაფიც მსაჯულთა დეტერმინისტული შერჩევისთვის პლუს დალუქული ჩადენა → ბიულეტენების გამოვლენა.  The
`iroha_data_model::ministry::jury` მოდული ახლა აგზავნის სამ Norito დატვირთვას, რომლებიც
მოიცავს ხმის მიცემის მთელ პროცესს:

1. **`PolicyJurySortitionV1`** – ჩაწერს გათამაშების მეტამონაცემებს (წინადადების ID,
   მრგვალი ID, პიროვნების დამადასტურებელი კადრის დაიჯესტი, შემთხვევითობის შუქურა),
   კომიტეტის ზომა, შერჩეული ნაფიც მსაჯულთა სია და ავტომატიზაციისთვის გამოყენებული ლოდინის სია
   მარცხი.  თითოეული ძირითადი სლოტი შეიძლება შეიცავდეს `PolicyJuryFailoverPlan`
   მოლოდინის რანგის მითითებით, ის უნდა გაიზარდოს მისი მადლის შემდეგ
   პერიოდი გადის.  სტრუქტურა განზრახ დეტერმინისტულია, ამიტომ აუდიტორები
   შეუძლია გათამაშების ხელახლა დაკვრა და მანიფესტის რეგენერაცია იმავე POP-დან
   სნეპშოტი + შუქურა.
2. **`PolicyJuryBallotCommitV1`** – კენჭისყრის წინ დაწერილი დალუქული ვალდებულება
   გამოვლენილია.  ის ინახავს რაუნდის/წინადადების/ჟურორის იდენტიფიკატორებს
   ნაფიც მსაჯულთა id + ხმის არჩევა + nonce tuple, დაჭერა
   დროის შტამპი და კენჭისყრის რეჟიმი (`plaintext` ან `zk-envelope`, როდესაც
   `zk-ballot` ფუნქცია აქტიურია).  `PolicyJuryBallotCommitV1::verify_reveal`
   უზრუნველყოფს შენახული დაიჯესტის შესაბამისობას გამოვლენის დატვირთვასთან.
3. **`PolicyJuryBallotRevealV1`** – საჯარო გამოვლენის ობიექტის შემცველი
   ხმის არჩევანი, ჩადენის დროს გამოყენებული ნონსი და არჩევითი ZK მტკიცებულება URI-ები.
   გამოვლენას სჭირდება მინიმუმ 16 ბაიტი, რათა მმართველობამ შეძლოს ამის მკურნალობა
   ვალდებულება სავალდებულოა მაშინაც კი, როდესაც ნაფიც მსაჯულთა წევრები მოქმედებენ დაუცველ არხებზე.

`PolicyJurySortitionV1::validate` დამხმარე ახორციელებს კომიტეტის ზომას,
დუბლიკატის აღმოჩენა (არ შეიძლება გამოჩნდეს ნაფიც მსაჯულში როგორც კომიტეტში, ასევე
Waitlist), შეკვეთილი ლოდინის სიის რიგები და მოქმედი წარუმატებლობის მითითებები.  ბიულეტენი
ვალიდაციის რუტინები ამაღლებს `PolicyJuryBallotError`-ს, როდესაც წინადადება ან მრგვალი ID-ები
დრიფტი, როდესაც ნაფიც მსაჯულები ცდილობენ გამოავლინონ არასწორი ნონსით, ან როცა ა
`zk-envelope` ვალდებულება ვერ უზრუნველყოფს შესატყვისი მტკიცებულების მითითებებს თავისში
გამოავლინოს.

### კლიენტებთან ინტეგრაცია

- მმართველობის ინსტრუმენტები უნდა შენარჩუნდეს დახარისხების მანიფესტში და შეიცავდეს მას
  პოლიტიკის პაკეტები, რათა დამკვირვებლებმა შეძლონ POP სნეპშოტის დაიჯესტის ხელახლა გამოთვლა და
  დაადასტურეთ, რომ შემთხვევითობის შუქურა პლუს კანდიდატის ნაკრები იგივეს იწვევს
  ნაფიც მსაჯულთა დავალებები.
- ნაფიც მსაჯულთა კლიენტები ჩაწერენ `PolicyJuryBallotCommitV1`-ს მაშინვე
  მათი ხმის მიცემის ნონსის წარმოქმნას.  მიღებული ვალდებულების ბაიტი შეიძლება იყოს
  გაგზავნილია Torii-ზე, როგორც საბაზისო64 მნიშვნელობა ან ჩართულია პირდაპირ Norito-ში
  მოვლენები.
- გამოვლენის ფაზაში ნაფიცი მსაჯულები ასხივებენ `PolicyJuryBallotRevealV1`.  ოპერატორები
  მიაწოდეთ დატვირთვა `PolicyJuryBallotCommitV1::verify_reveal`-მდე
  კენჭისყრის მიღება, იმის უზრუნველყოფა, რომ გამოვლენა არ იყო გაცვლილი ან გაყალბებული.
- როდესაც ჩართულია `zk-ballot` ფუნქცია, ნაფიც მსაჯულებს შეუძლიათ დაურთონ დეტერმინისტული
  მტკიცებულების URI-ები (მაგ., `sorafs://proofs/pj-2026-02/juror-5`) ასე ქვემოთ
  აუდიტორებს შეუძლიათ მიიღონ ნულოვანი ცოდნის მქონე მოწმეების ნაკრები, რომელიც მითითებულია
  ვალდებულება.სამივე სტრუქტურა წარმოშობს `Encode`, `Decode` და `IntoSchema`, რაც იმას ნიშნავს, რომ ისინი
ხელმისაწვდომია ISI ნაკადებისთვის, CLI ინსტრუმენტებისთვის, SDK-ებისთვის და მართვის REST API-სთვის.
იხილეთ `crates/iroha_data_model/src/ministry/jury.rs` კანონიკური Rust-ისთვის
განმარტებები და დამხმარე მეთოდები.

### CLI მხარდაჭერა დახარისხების მანიფესტებისთვის

საგზაო რუკის პუნქტი **MINFO-5** ასევე მოითხოვდა რეპროდუცირებადი ინსტრუმენტების გამოყენებას, რათა მმართველობამ შეძლოს
ყოველი რეფერენდუმის პაკეტის გამოქვეყნებამდე გაგზავნეთ გადამოწმებადი პოლიტიკის ჟიურის სია.
სამუშაო სივრცე ახლა აჩვენებს `cargo xtask ministry-jury sortition` ბრძანებას:

```bash
cargo xtask ministry-jury sortition \
  --roster docs/examples/ministry/policy_jury_roster_example.json \
  --proposal AC-2026-042 \
  --round PJ-2026-02 \
  --beacon 22b1e48d47123f5c9e3f0cc0c8e34aa3c5f9c49a2cbb70559d3cb0ddc1a6ef01 \
  --committee-size 3 \
  --waitlist-size 2 \
  --drawn-at 2026-01-15T09:00:00Z \
  --waitlist-ttl-hours 72 \
  --out artifacts/ministry/policy_jury_sortition.json
```

- `--roster` იღებს დეტერმინისტულ PoP სიას (JSON მაგალითი:
  `docs/examples/ministry/policy_jury_roster_example.json`).  თითოეული ჩანაწერი
  აცხადებს `juror_id`, `pop_identity`, წონა და სურვილისამებრ
  `grace_period_secs`.  შეუსაბამო ჩანაწერები ავტომატურად იფილტრება.
- `--beacon` ახდენს 32-ბაიტიანი შემთხვევითობის შუქურას, რომელიც აღბეჭდილია მმართველობაში
  წუთები.  CLI აერთებს შუქურს პირდაპირ ChaCha20 RNG-ში, ასე რომ აუდიტორებს
  შეუძლია გათამაშების გამეორება ბაიტი-ბაიტი.
- `--committee-size`, `--waitlist-size` და `--waitlist-ttl-hours` აკონტროლებენ
  მჯდომარე ნაფიც მსაჯულთა რაოდენობა, ჩავარდნილი ბუფერი და ვადის გასვლის ვადის შტამპი გამოყენებული
  ლოდინის სიის ჩანაწერებზე.  როდესაც სლოტისთვის არსებობს failover რანგი, ბრძანება
  ჩაწერს `PolicyJuryFailoverPlan`, რომელიც მიუთითებს შესაბამის ლოდინის სიაზე.
- `--drawn-at` ჩაწერს კედლის საათის დროის ნიშანს დახარისხებისთვის; ხელსაწყო
  გარდაქმნის მას Unix მილიწამებში მანიფესტისთვის.

გენერირებული manifest არის სრულად დადასტურებული `PolicyJurySortitionV1` დატვირთვა.
დიდი განლაგება, როგორც წესი, ინახავს გამომავალს `artifacts/ministry/`-ის ქვეშ, ასე რომ
შეიძლება შეფუთული იყოს პირდაპირ რეფერენდუმის პაკეტებში განხილვის პანელთან ერთად
შეჯამება.  საილუსტრაციო გამომავალი ხელმისაწვდომია
`docs/examples/ministry/policy_jury_sortition_example.json` ასე რომ SDK გუნდებს შეუძლიათ
განახორციელეთ მათი Norito დეკოდერები მთლიანი გათამაშების ადგილობრივად გამეორების გარეშე.

### ბიულეტენის ჩადება/გამოვლენა დამხმარეები

ნაფიც მსაჯულ კლიენტებს ესაჭიროებათ დეტერმინისტული ინსტრუმენტები ჩადენისთვის → გამოავლინონ ნაკადი ასევე.
იგივე `cargo xtask ministry-jury` ბრძანება ახლა ავლენს შემდეგ დამხმარეებს:

```bash
cargo xtask ministry-jury ballot commit \
  --proposal AC-2026-042 \
  --round PJ-2026-02 \
  --juror citizen:ada \
  --choice approve \
  --nonce-hex aabbccddeeff00112233445566778899aabbccddeeff00112233445566778899 \
  --committed-at 2026-02-01T13:00:00Z \
  --out artifacts/ministry/policy_jury_commit_ada.json \
  --reveal-out artifacts/ministry/policy_jury_reveal_ada.json

cargo xtask ministry-jury ballot verify \
  --commit artifacts/ministry/policy_jury_commit_ada.json \
  --reveal artifacts/ministry/policy_jury_reveal_ada.json
```

- `ballot commit` ასხივებს `PolicyJuryBallotCommitV1` JSON დატვირთვას.  როცა
  `--out` გამოტოვებულია ბრძანება, რომელიც ბეჭდავს stdout-ის ვალდებულებას.  თუ
  მოწოდებულია `--reveal-out` ინსტრუმენტი ასევე წერს შესატყვისს
  `PolicyJuryBallotRevealV1`, მოწოდებულის ხელახლა გამოყენება და გამოყენება
  სურვილისამებრ `--revealed-at` დროის შტამპი (ნაგულისხმევი `--committed-at` ან
  მიმდინარე დრო).
- `--nonce-hex` იღებს ნებისმიერ ლუწ სიგრძის ექვს სტრიქონს ≥16 ბაიტი.  როდესაც გამოტოვებულია
  დამხმარე აგენერირებს 32 ბაიტიან ნონსს `OsRng`-ის გამოყენებით, რაც აადვილებს სკრიპტს
  ნაფიც მსაჯულთა სამუშაო ნაკადები ხდება შემთხვევითი სანტექნიკის გარეშე.
- `--choice` არ არის რეგისტრირებული და იღებს `approve`, `reject`, ან `abstain`.

`ballot verify` ამოწმებს ვალდებულების/გამოვლენის წყვილს მეშვეობით
`PolicyJuryBallotCommitV1::verify_reveal`, გარანტიას, რომ მრგვალი id,
წინადადების id, ნაფიცი მსაჯულის პირადობის მოწმობა, nonce და ხმის არჩევანი ყველა სწორდება გამოვლენამდე
დაშვებულია Torii-ზე.  დამხმარე გადის არა-ნულოვანი სტატუსით ვალიდაციისას
ვერ მოხერხდება, რაც უსაფრთხოს ხდის CI ან ადგილობრივ ნაფიც მსაჯულთა პორტალებში დაკავშირებას.
