# Emergency Canon & TTL Policy (MINFO-6a)

Roadmap reference: **MINFO-6a — Emergency canon & TTL policy**.

This document defines the denylist tiering rules, TTL enforcement, and governance obligations that now ship in Torii and the CLI. Operators must follow these rules before publishing new entries or invoking emergency canons.

## Tier Definitions

| Tier | Default TTL | Review Window | Requirements |
|------|-------------|---------------|--------------|
| Standard | 180 days (`torii.sorafs_gateway.denylist.standard_ttl`) | n/a | `issued_at` must be provided. `expires_at` may be omitted; Torii defaults to `issued_at + standard_ttl` and rejects longer windows. |
| Emergency | 30 days (`torii.sorafs_gateway.denylist.emergency_ttl`) | 7 days (`torii.sorafs_gateway.denylist.emergency_review_window`) | Requires a non-empty `emergency_canon` label referencing the pre-approved canon (e.g., `csam-hotline`). `issued_at` + `expires_at` must land inside the 30‑day window, and review evidence must cite the autogenerated deadline (`issued_at + review_window`). |
| Permanent | No expiry | n/a | Reserved for supermajority governance decisions. Entries must cite a non-empty `governance_reference` (vote id, manifesto hash, etc.). `expires_at` is rejected. |

The defaults remain configurable through `torii.sorafs_gateway.denylist.*` and `iroha_cli` mirrors the limits to catch invalid entries before Torii reloads a file.

## Workflow

1. **Prepare metadata:** include `policy_tier`, `issued_at`, `expires_at` (when applicable), and `emergency_canon`/`governance_reference` inside each JSON entry (`docs/source/sorafs_gateway_denylist_sample.json`).
2. **Validate locally:** run `iroha app sorafs gateway lint-denylist --path <denylist.json>` so the CLI enforces tier-specific TTLs and required fields before the file is committed or pinned.
3. **Publish evidence:** attach the canon id or governance reference cited in the entry to the GAR case bundle (agenda packet, referendum minutes, etc.) so auditors can trace the decision.
4. **Review emergency entries:** emergency canons auto-expire within 30 days. Operators must complete post-facto review inside the 7-day window and record the outcome in the ministry tracker/SoraFS evidence store.
5. **Reload Torii:** once validated, deploy the denylist path via `torii.sorafs_gateway.denylist.path` and restart/reload Torii; the runtime enforces the same limits before admitting entries.

## Tooling & References

- Runtime policy enforcement lives in `sorafs::gateway::denylist` (`crates/iroha_torii/src/sorafs/gateway/denylist.rs`) and the loader now applies tier metadata when parsing `torii.sorafs_gateway.denylist.*` inputs.
- CLI validation mirrors the runtime semantics inside `GatewayDenylistRecord::validate` (`crates/iroha_cli/src/commands/sorafs.rs`). The linter fails when TTLs exceed the configured window or when mandatory canon/governance references are missing.
- Config knobs are defined under `torii.sorafs_gateway.denylist` (`crates/iroha_config/src/parameters/{defaults,actual.rs,user.rs}`) so operators can tweak TTLs/review deadlines if governance approves different bounds.
- The public sample denylist (`docs/source/sorafs_gateway_denylist_sample.json`) now illustrates all three tiers and should be used as the canonical template for new entries.

These guardrails satisfy roadmap item **MINFO-6a** by codifying the emergency canon list, preventing unbounded TTLs, and forcing explicit governance evidence for permanent blocks.

## Registry Automation & Evidence Exports

Emergency canon approvals must produce a deterministic registry snapshot and a
diff bundle before Torii loads the denylist. The tooling under
`xtask/src/sorafs.rs` plus the CI harness `ci/check_sorafs_gateway_denylist.sh`
cover the entire workflow.

### Canonical bundle generation

1. Stage raw entries (typically the file reviewed by governance) in a working
   directory.
2. Canonicalise and seal the JSON via:
   ```bash
   cargo xtask sorafs-gateway denylist pack \
     --input path/to/denylist.json \
     --out artifacts/ministry/denylist_registry/$(date +%Y%m%dT%H%M%SZ) \
     --label ministry-emergency \
     --force
   ```
   The command emits a signed-friendly `.json` bundle, the Norito `.to`
   envelope, and the Merkle-root text file expected by governance reviewers.
   Store the directory under `artifacts/ministry/denylist_registry/` (or your
   chosen evidence bucket) so `scripts/ministry/transparency_release.py` can
   pick it up later with `--artifact denylist_bundle=<path>`.
3. Keep the generated `checksums.sha256` alongside the bundle before pushing it
   to SoraFS/GAR. CI’s `ci/check_sorafs_gateway_denylist.sh` exercises the same
   `pack` helper against the sample denylist to guarantee the tooling works on
   every release.

### Diff + audit bundle

1. Compare the new bundle against the previous production snapshot using the
   xtask diff helper:
   ```bash
   cargo xtask sorafs-gateway denylist diff \
     --old artifacts/ministry/denylist_registry/2026-05-01/denylist_old.json \
     --new artifacts/ministry/denylist_registry/2026-05-14/denylist_new.json \
     --report-json artifacts/ministry/denylist_registry/2026-05-14/denylist_diff.json
   ```
   The JSON report lists all additions/removals and mirrors the evidence
   structure consumed by `MinistryDenylistChangeV1` (referenced by
   `docs/source/sorafs_gateway_self_cert.md` and the compliance plan).
2. Attach `denylist_diff.json` to every canon request (it proves how many
   entries were touched, which tier changed, and what evidence hash maps to the
   canonical bundle).
3. When diffs are generated automatically (CI or release pipelines), export the
   `denylist_diff.json` path via `--artifact denylist_diff=<path>` so the
   transparency manifest records it alongside sanitized metrics. The same CI
   helper accepts `--evidence-out <path>` which runs the CLI summary step and
   copies the resulting JSON to the requested location for later publishing.

### Publication & transparency

1. Drop the pack + diff artefacts into the quarterly transparency directory
   (`artifacts/ministry/transparency/<YYYY-Q>/denylist/`). The transparency
   release helper can then include them:
   ```bash
   scripts/ministry/transparency_release.py \
     --quarter 2026-Q3 \
     --output-dir artifacts/ministry/transparency/2026-Q3 \
     --sanitized artifacts/ministry/transparency/2026-Q3/sanitized_metrics.json \
     --dp-report artifacts/ministry/transparency/2026-Q3/dp_report.json \
     --artifact denylist_bundle=artifacts/ministry/denylist_registry/2026-05-14/denylist_new.json \
     --artifact denylist_diff=artifacts/ministry/denylist_registry/2026-05-14/denylist_diff.json
   ```
2. Reference the generated bundle/diff in the quarterly report
   (`docs/source/ministry/reports/<YYYY-Q>.md`) and attach the same paths to the
   GAR vote packet so auditors can replay the evidence trail without access to
   internal CI. `ci/check_sorafs_gateway_denylist.sh --evidence-out \
   artifacts/ministry/denylist_registry/<STAMP>/denylist_evidence.json` now
   performs the pack/diff/evidence dry-run (calling `iroha_cli app sorafs gateway
   evidence` under the hood) so automation can persist the summary alongside the
   canonical bundles.
3. After publication, anchor the governance payload via
   `cargo xtask ministry-transparency anchor` (called automatically by
   `transparency_release.py` when `--governance-dir` is provided) so the
   denylist registry digest appears in the same DAG tree as the transparency
   release.

Following this process closes the “registry automation and evidence export”
gap called out in `roadmap.md:450` and ensures that every emergency canon
decision arrives with reproducible artefacts, JSON diffs, and transparency log
entries.

### TTL & Canon Evidence Helper

After the bundle/diff pair is produced, run the CLI evidence helper to capture
the TTL summaries and emergency-review deadlines that governance requires:

```bash
iroha app sorafs gateway evidence \
  --denylist artifacts/ministry/denylist_registry/2026-05-14/denylist.json \
  --out artifacts/ministry/denylist_registry/2026-05-14/denylist_evidence.json \
  --label csam-canon-2026-05
```

The command hashes the source JSON, validates every entry, and emits a compact
summary containing:

- Total entries per `kind` and per policy tier with the earliest/latest
  timestamps observed.
- An `emergency_reviews[]` list enumerating each emergency canon with its
  descriptor, effective expiry, max allowed TTL, and the computed
  `review_due_by` deadline.

Attach `denylist_evidence.json` alongside the packed bundle/diff so auditors can
confirm TTL compliance without re-running the CLI. CI jobs that already generate
bundles can invoke the helper and publish the evidence artefact (for example by
calling `ci/check_sorafs_gateway_denylist.sh --evidence-out <path>`), ensuring
every canon request lands with a consistent summary.

### Merkle registry evidence

The Merkle registry introduced in MINFO-6 requires operators to publish the
root and per-entry proofs alongside the TTL summary. Immediately after running
the evidence helper, capture the Merkle artefacts:

```bash
iroha app sorafs gateway merkle snapshot \
  --denylist artifacts/ministry/denylist_registry/2026-05-14/denylist.json \
  --json-out artifacts/ministry/denylist_registry/2026-05-14/denylist_merkle_snapshot.json \
  --norito-out artifacts/ministry/denylist_registry/2026-05-14/denylist_merkle_snapshot.to

iroha app sorafs gateway merkle proof \
  --denylist artifacts/ministry/denylist_registry/2026-05-14/denylist.json \
  --index 12 \
  --json-out artifacts/ministry/denylist_registry/2026-05-14/denylist_merkle_proof_entry_12.json \
  --norito-out artifacts/ministry/denylist_registry/2026-05-14/denylist_merkle_proof_entry_12.to
```

The snapshot JSON records the BLAKE3 Merkle root, leaf count, and every
descriptor/hash pair so GAR votes can reference the exact tree that was hashed.
Supplying `--norito-out` stores a `.to` artefact alongside the JSON, letting
gateways ingest the registry entries directly via Norito without scraping
stdout. `merkle proof` emits the direction bits and sibling hashes for any
zero-based entry index, making it simple to attach an inclusion proof for each
emergency canon cited in the GAR memo—the optional Norito copy keeps the proof
ready for on-ledger distribution. Store both the JSON and Norito artefacts next
to the TTL summary and diff bundle so transparency releases and governance
anchors reference the same root.
