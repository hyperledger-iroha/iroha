# Օրակարգային Խորհրդի Առաջարկների Սխեման (MINFO-2a)

Ճանապարհային քարտեզի տեղեկանք՝ **MINFO-2a — Առաջարկի ձևաչափի վավերացնող։**

Օրակարգային խորհրդի աշխատանքային հոսքը ներառում է քաղաքացիների կողմից ներկայացված սև ցուցակը և քաղաքականության փոփոխությունները
առաջարկները, մինչ դրանք կքննարկվեն կառավարման խորհուրդները: Այս փաստաթուղթը սահմանում է
կանոնական ծանրաբեռնվածության սխեման, ապացույցների պահանջները և կրկնօրինակման հայտնաբերման կանոնները
սպառվում է նոր վավերացնողի կողմից (`cargo xtask ministry-agenda validate`) այսպես
Առաջարկողները կարող են տեղայնացնել JSON-ի ներկայացումները՝ նախքան դրանք պորտալ վերբեռնելը:

## Օգտակար բեռի ակնարկ

Օրակարգային առաջարկներն օգտագործում են `AgendaProposalV1` Norito սխեման
(`iroha_data_model::ministry::AgendaProposalV1`): Դաշտերը կոդավորված են որպես JSON, երբ
ներկայացնելով CLI/պորտալի մակերեսների միջոցով:

| Դաշտային | Տեսակ | Պահանջներ |
|-------|------|--------------|
| `version` | `1` (u16) | Պետք է հավասար լինի `AGENDA_PROPOSAL_VERSION_V1`-ին: |
| `proposal_id` | տող (`AC-YYYY-###`) | Կայուն նույնացուցիչ; ուժի մեջ է մտել վավերացման ընթացքում: |
| `submitted_at_unix_ms` | u64 | Միլվայրկյաններ Unix դարաշրջանից: |
| `language` | լարային | BCP‑47 թեգ (`"en"`, `"ja-JP"` և այլն): |
| `action` | enum (`add-to-denylist`, `remove-from-denylist`, `amend-policy`) | Նախարարության խնդրանքով. |
| `summary.title` | լարային | Առաջարկվում է ≤256 նիշ: |
| `summary.motivation` | լարային | Ինչու է պահանջվում գործողությունը: |
| `summary.expected_impact` | լարային | Արդյունքները, եթե գործողությունն ընդունվի: |
| `tags[]` | փոքրատառ տողեր | Ընտրովի տրաժի պիտակներ: Թույլատրելի արժեքներ՝ `csam`, `malware`, `fraud`, `harassment`, `impersonation`, `policy-escalation`, I18030000 `spam`. |
| `targets[]` | օբյեկտներ | Հեշ ընտանիքի մեկ կամ մի քանի գրառում (տես ստորև): |
| `evidence[]` | օբյեկտներ | Մեկ կամ մի քանի ապացույցների հավելվածներ (տես ստորև): |
| `submitter.name` | լարային | Ցուցադրվող անուն կամ կազմակերպություն: |
| `submitter.contact` | լարային | փոստ, Matrix բռնակ կամ հեռախոս; վերամշակված հանրային վահանակներից: |
| `submitter.organization` | տող (ըստ ցանկության) | Տեսանելի է գրախոսի միջերեսում: |
| `submitter.pgp_fingerprint` | տող (ըստ ցանկության) | 40 վեցանկյուն մեծատառ մատնահետք: |
| `duplicates[]` | լարեր | Նախկինում ներկայացված առաջարկի ID-ների կամընտիր հղումներ: |

### Թիրախային գրառումներ (`targets[]`)

Յուրաքանչյուր թիրախ ներկայացնում է հաշ ընտանիքի ամփոփում, որը նշված է առաջարկով:

| Դաշտային | Նկարագրություն | Վավերացում |
|-------|-------------|------------|
| `label` | Ընկերական անուն գրախոսի համատեքստի համար: | Ոչ դատարկ. |
| `hash_family` | Հեշի նույնացուցիչ (`blake3-256`, `sha256` և այլն): | ASCII տառեր/նիշեր/`-_.`, ≤48 նիշ: |
| `hash_hex` | Digest կոդավորված է փոքրատառ վեցանկյունով: | ≥16 բայթ (32 վեցանկյուն նիշ) և պետք է լինի վավեր վեցանկյուն: |
| `reason` | Կարճ նկարագրություն այն մասին, թե ինչու պետք է գործել մարսողությունը: | Ոչ դատարկ. |

Վավերացնողը մերժում է կրկնօրինակ `hash_family:hash_hex` զույգերը նույն ներսում
առաջարկություն և հաղորդում հակասությունների մասին, երբ նույն մատնահետքն արդեն գոյություն ունի
կրկնօրինակ գրանցամատյան (տես ստորև):

### Ապացույցների հավելվածներ (`evidence[]`)

Ապացույցների գրառումների փաստաթուղթ, որտեղ գրախոսները կարող են ստանալ օժանդակ համատեքստ:| Դաշտային | Տեսակ | Ծանոթագրություններ |
|-------|------|-------|
| `kind` | enum (`url`, `torii-case`, `sorafs-cid`, `attachment`) | Որոշում է մարսողության պահանջները: |
| `uri` | լարային | HTTP(S) URL, Torii գործի ID կամ SoraFS URI: |
| `digest_blake3_hex` | լարային | Պահանջվում է `sorafs-cid` և `attachment` տեսակների համար; ընտրովի մյուսների համար: |
| `description` | լարային | Ընտրովի ազատ ձևի տեքստ գրախոսողների համար: |

### Կրկնօրինակ ռեեստր

Օպերատորները կարող են պահպանել գոյություն ունեցող մատնահետքերի ռեեստր՝ կրկնօրինակումը կանխելու համար
դեպքեր. Վավերացնողն ընդունում է JSON ֆայլ՝ ձևավորված հետևյալ կերպ.

```json
{
  "entries": [
    {
      "hash_family": "blake3-256",
      "hash_hex": "0d714bed4b7c63c23a2cf8ee9ce6c3cde1007907c427b4a0754e8ad31c91338d",
      "proposal_id": "AC-2025-014",
      "note": "Already handled in 2025-08 incident"
    }
  ]
}
```

Երբ առաջարկի թիրախը համընկնում է մուտքի հետ, վավերացնողը դադարում է, եթե
Նշված է `--allow-registry-conflicts` (նախազգուշացումները դեռ ուղարկվում են):
Օգտագործեք [`cargo xtask ministry-agenda impact`](impact_assessment_tooling.md)՝
ստեղծել հանրաքվեի համար պատրաստ ամփոփագիր, որը խաչաձև հղում է կատարում կրկնօրինակին
ռեեստրի և քաղաքականության ակնարկներ:

## CLI-ի օգտագործում

Տեղադրեք մեկ առաջարկ և ստուգեք այն կրկնօրինակ գրանցամատյանում.

```bash
cargo xtask ministry-agenda validate \
  --proposal docs/examples/ministry/agenda_proposal_example.json \
  --registry docs/examples/ministry/agenda_duplicate_registry.json
```

Անցեք `--allow-registry-conflicts`՝ կրկնօրինակ այցելությունները նախազգուշացումների իջեցնելու համար, երբ
պատմական ստուգումների իրականացում։

CLI-ն հիմնվում է նույն Norito սխեմայի և վավերացման օգնականների վրա, որոնք առաքվել են այստեղ
`iroha_data_model`, այնպես որ SDK-ները/պորտալները կարող են կրկին օգտագործել `AgendaProposalV1::validate`-ը
հետևողական վարքագծի մեթոդ.

## Տեսակավորում CLI (MINFO-2b)

Ճանապարհային քարտեզի տեղեկանք՝ **MINFO-2b — Բազմակողմանի տեսակավորում և աուդիտի մատյան։**

Օրակարգային խորհրդի ցուցակն այժմ կառավարվում է քաղաքացիների որոշիչ տեսակավորման միջոցով
կարող է ինքնուրույն ստուգել յուրաքանչյուր խաղարկություն: Օգտագործեք նոր հրամանը.

```bash
cargo xtask ministry-agenda sortition \
  --roster docs/examples/ministry/agenda_council_roster.json \
  --slots 3 \
  --seed 0123456789abcdef0123456789abcdef0123456789abcdef0123456789abcdef \
  --out artifacts/ministry/agenda_sortition_2026Q1.json
```

- `--roster` — JSON ֆայլ, որը նկարագրում է յուրաքանչյուր իրավասու անդամ.

  ```json
  {
    "format_version": 1,
    "members": [
      {
        "member_id": "citizen:ada",
        "weight": 2,
        "role": "citizen",
        "organization": "Artemis Cooperative"
      },
      {
        "member_id": "citizen:erin",
        "weight": 1,
        "role": "citizen",
        "eligible": false
      }
    ]
  }
  ```

  Օրինակի ֆայլը ապրում է
  `docs/examples/ministry/agenda_council_roster.json`. Ընտրովի դաշտեր (դեր,
  կազմակերպություն, կոնտակտ, մետատվյալներ) ներառված են Մերկլի տերևում, որպեսզի աուդիտորները
  կարող է ապացուցել այն ցուցակը, որը կերակրեց վիճակահանությանը:

- `--slots` — ավագանու մանդատների քանակը լրացնելու համար:
- `--seed` — 32 բայթ BLAKE3 սերմ (64 փոքրատառ վեցանկյուն նիշ) գրանցված է
  վիճակահանության համար կառավարման րոպե.
- `--out` — կամընտիր ելքային ուղի: Երբ բաց թողնված է, JSON ամփոփագիրը տպվում է այստեղ
  stdout.

### Արդյունքների ամփոփում

Հրամանն արտանետում է `SortitionSummary` JSON բշտիկ: Նմուշի ելքը պահվում է
`docs/examples/ministry/agenda_sortition_summary_example.json`. Հիմնական դաշտերը.

| Դաշտային | Նկարագրություն |
|-------|-------------|
| `algorithm` | Տեսակավորման պիտակ (`agenda-sortition-blake3-v1`): |
| `roster_digest` | BLAKE3 + SHA-256 ամփոփում է ցուցակի ֆայլը (օգտագործվում է նույն անդամների ցուցակի վրա աուդիտների գործողությունը հաստատելու համար): |
| `seed_hex` / `slots` | Կրկնել CLI-ի մուտքերը, որպեսզի աուդիտորները կարողանան վերարտադրել խաղարկությունը: |
| `merkle_root_hex` | Merkle ծառի անվանացանկի արմատը (`hash_node`/`hash_leaf` օգնականներ `xtask/src/ministry_agenda.rs`-ում): |
| `selected[]` | Գրառումներ յուրաքանչյուր տեղամասի համար, ներառյալ կանոնական անդամի մետատվյալները, իրավասու ինդեքսը, սկզբնական ցուցակի ինդեքսը, որոշիչ խաղարկության էնտրոպիան, տերևների հաշը և Merkle-ի ապացույցները: |

### Վիճակահանության ստուգում1. Վերցրեք `roster_path`-ի կողմից նշված ցուցակը և հաստատեք դրա BLAKE3/SHA-256-ը
   մարսողությունները համապատասխանում են ամփոփմանը:
2. Կրկին գործարկել CLI-ն նույն սերմով/սլոթերով/ցուցակով; ստացված `selected[].member_id`
   պատվերը պետք է համապատասխանի հրապարակված ամփոփմանը:
3. Կոնկրետ անդամի համար հաշվարկեք Մերկլի տերևը՝ օգտագործելով սերիականացված JSON անդամը
   (`norito::json::to_vec(&sortition_member)`) և ծալեք յուրաքանչյուր ապացույց հեշի մեջ: Եզրափակիչ
   digest-ը պետք է հավասար լինի `merkle_root_hex`-ին: Օրինակի ամփոփման օգնականը ցույց է տալիս
   ինչպես համատեղել `eligible_index`, `leaf_hash_hex` և `merkle_proof[]`:

## Վավերացման սխալի հղում

`AgendaProposalV1::validate`-ն արտանետում է `AgendaProposalValidationError` տարբերակներ
երբ ծանրաբեռնվածությունը չի փչում: Ստորև բերված աղյուսակը ամփոփում է ամենատարածվածները
սխալներ, որպեսզի պորտալի վերանայողները կարողանան թարգմանել CLI ելքը գործող ուղեցույցի:| Սխալ | Իմաստը | Վերականգնում |
|-------|---------|-------------|
| `UnsupportedVersion { expected, found }` | Payload `version`-ը տարբերվում է վավերացնողի աջակցվող սխեմայից: | Վերականգնեք JSON-ը՝ օգտագործելով սխեմայի վերջին փաթեթը, որպեսզի տարբերակը համապատասխանի `expected`-ին: |
| `MissingProposalId` / `InvalidProposalIdFormat { value }` | `proposal_id` դատարկ է, թե ոչ `AC-YYYY-###` ձևով: | Նախքան նորից ներկայացնելը, լրացրեք եզակի նույնացուցիչը, որը հետևում է փաստաթղթավորված ձևաչափին: |
| `MissingSubmissionTimestamp` | `submitted_at_unix_ms`-ը զրո է կամ բացակայում է: | Գրանցեք ներկայացման ժամանակացույցը Unix միլիվայրկյաններով: |
| `InvalidLanguageTag { value }` | `language`-ը վավեր BCP‑47 թեգ չէ: | Օգտագործեք ստանդարտ պիտակ, ինչպիսիք են `en`, `ja-JP` կամ BCP‑47-ի կողմից ճանաչված այլ տեղայնացում: |
| `MissingSummaryField { field }` | `summary.title`, `.motivation` կամ `.expected_impact`-ից մեկը դատարկ է: | Նշված ամփոփ դաշտի համար տրամադրեք ոչ դատարկ տեքստ: |
| `MissingSubmitterField { field }` | `submitter.name` կամ `submitter.contact` բացակայում է: | Տրամադրեք բացակայող ներկայացնողի մետատվյալները, որպեսզի վերանայողները կարողանան կապվել առաջարկողի հետ: |
| `InvalidTag { value }` | `tags[]` գրառումը թույլտվության ցանկում չէ: | Հեռացրեք կամ վերանվանեք պիտակը փաստաթղթավորված արժեքներից մեկի (`csam`, `malware` և այլն): |
| `MissingTargets` | `targets[]` զանգվածը դատարկ է: | Տրամադրեք առնվազն մեկ թիրախային հեշ ընտանիքի մուտք: |
| `MissingTargetLabel { index }` / `MissingTargetReason { index }` | Թիրախային մուտքագրում բացակայում են `label` կամ `reason` դաշտերը: | Լրացրեք ինդեքսավորված մուտքի համար անհրաժեշտ դաշտը, նախքան նորից ուղարկելը: |
| `InvalidHashFamily { index, value }` | Չաջակցվող `hash_family` պիտակ: | Սահմանափակել հեշ ազգանունները ASCII այբբենական թվերով գումարած `-_`: |
| `InvalidHashHex { index, value }` / `TargetDigestTooShort { index }` | Digest-ը վավեր վեցանկյուն չէ կամ 16 բայթից կարճ է: | Տրամադրեք փոքրատառ վեցանկյուն նիշ (≥32 վեցանկյուն նիշ) ինդեքսավորված թիրախի համար: |
| `DuplicateTarget { index, fingerprint }` | Target digest-ը կրկնօրինակում է ավելի վաղ մուտքի կամ ռեեստրի մատնահետքը: | Հեռացրեք կրկնօրինակները կամ միացրեք աջակցող ապացույցները մեկ թիրախի մեջ: |
| `MissingEvidence` | Ապացույցների հավելվածներ չեն տրամադրվել: | Կցեք վերարտադրման նյութին կապող առնվազն մեկ ապացույց: |
| `MissingEvidenceUri { index }` | Ապացույցների մուտքագրում բացակայում է `uri` դաշտը: | Տրամադրեք առբերվող URI կամ գործի նույնացուցիչը ինդեքսավորված ապացույցների մուտքագրման համար: |
| `MissingEvidenceDigest { index }` / `InvalidEvidenceDigest { index, value }` | Ապացույցների մուտքագրումը, որը պահանջում է ամփոփում (SoraFS CID կամ հավելված), բացակայում է կամ ունի անվավեր `digest_blake3_hex`: | Տրամադրեք 64 նիշանոց փոքրատառ BLAKE3 բովանդակություն ինդեքսավորված մուտքի համար: |
