# Env → Отслеживание миграции конфигурации

Этот трекер суммирует обнаруженные переключатели переменных среды, связанные с производством.
по `docs/source/agents/env_var_inventory.{json,md}` и предполагаемой миграции
путь к `iroha_config` (или явная область действия только для разработки и тестирования).


Примечание. `ci/check_env_config_surface.sh` теперь завершается сбоем при новом **производственном** окружении.
прокладки появляются относительно `AGENTS_BASE_REF`, если только `ENV_CONFIG_GUARD_ALLOW=1` не является
набор; задокументируйте здесь намеренные дополнения, прежде чем использовать переопределение.

## Завершенные миграции- **IVM Отказ от ABI** — Удален `IVM_ALLOW_NON_V1_ABI`; компилятор теперь отклоняет
  ABI, отличные от v1, безусловно, с модульным тестом, защищающим путь ошибки.
- **IVM отладочная оболочка баннера env** — удален отказ от env `IVM_SUPPRESS_BANNER`;
  подавление баннеров остается доступным через программный установщик.
- **IVM кэш/размер** — Поточный кэш/проверка/калибровка графического процессора
  И18НИ00000014Х (И18НИ00000015Х,
  `accel.max_gpus`) и удалены прокладки среды выполнения. Хозяева сейчас звонят
  `ivm::ivm_cache::configure_limits` и `ivm::zk::set_prover_threads`, тесты используют
  `CacheLimitsGuard` вместо переопределений env.
- **Корень очереди подключения** — добавлен `connect.queue.root` (по умолчанию:
  `~/.iroha/connect`) в конфигурацию клиента и пропустил ее через CLI и
  JS-диагностика. Помощники JS разрешают конфигурацию (или явный `rootDir`) и
  учитывайте только `IROHA_CONNECT_QUEUE_ROOT` в разработке/тестировании через `allowEnvOverride`;
  шаблоны документируют ручку, поэтому операторам больше не нужны переопределения окружения.
- **Izanami согласие на участие в сети** — добавлен явный флаг `allow_net` CLI/config для
  инструмент хаоса Izanami; для запуска теперь требуются `allow_net=true`/`--allow-net` и
- **Звуковой сигнал баннера IVM** — заменена оболочка env `IROHA_BEEP` на управляемую конфигурацией.
  `ivm.banner.{show,beep}` переключает (по умолчанию: true/true). Баннер запуска/звуковой сигнал
  проводка теперь считывает конфигурацию только в продакшене; dev/test сборки по-прежнему в чести
  переопределение env для ручного переключения.
- **Переопределение спула DA (только тесты)** — Теперь переопределение `IROHA_DA_SPOOL_DIR`
  огороженный позади помощников `cfg(test)`; производственный код всегда создает спул
  путь из конфигурации.
- **Внутренние функции шифрования** — Заменено `IROHA_DISABLE_SM_INTRINSICS` /
  `IROHA_ENABLE_SM_INTRINSICS` с управляемой конфигурацией
  Политика `crypto.sm_intrinsics` (`auto`/`force-enable`/`force-disable`) и
  удалил защиту `IROHA_SM_OPENSSL_PREVIEW`. Хозяева применяют политику на
  запуск, стенды/тесты могут быть включены через `CRYPTO_SM_INTRINSICS` и OpenSSL.
  предварительный просмотр теперь учитывает только флаг конфигурации.
  Izanami уже требует `--allow-net`/persisted config, и тесты теперь полагаются на
  эта ручка, а не переключатели окружающего окружения.
- **Настройка графического процессора FastPQ** — Добавлен `fastpq.metal.{max_in_flight,threadgroup_width,metal_trace,metal_debug_enum,metal_debug_fused}`.
  ручки конфигурации (по умолчанию: `None`/`None`/`false`/`false`/`false`) и пропустите их через анализ CLI.
  Прокладки `FASTPQ_METAL_*` / `FASTPQ_DEBUG_*` теперь действуют как запасные варианты для разработки/тестирования и
  игнорируются после загрузки конфигурации (даже если в конфигурации они не установлены); документы/опись были
  обновлено, чтобы отметить миграцию.【crates/irohad/src/main.rs:2609】【crates/iroha_core/src/fastpq/lane.rs:109】【crates/fastpq_prover/src/overrides.rs:11】
  (`IVM_DECODE_TRACE`, `IVM_DEBUG_WSV`, `IVM_DEBUG_COMPACT`, `IVM_DEBUG_INVALID`,
  И18НИ00000053Х, И18НИ00000054Х, И18НИ00000055Х,
  И18НИ00000056Х, И18НИ00000057Х, И18НИ00000058Х,
  `IVM_DISABLE_METAL`, `IVM_DISABLE_CUDA`) теперь защищены отладочными/тестовыми сборками через общий
  helper, поэтому производственные двоичные файлы игнорируют их, сохраняя при этом ручки для локальной диагностики. Конв.
  инвентарь был перегенерирован, чтобы отразить область только для разработки и тестирования.- **Обновления приборов FASTPQ** — `FASTPQ_UPDATE_FIXTURES` теперь отображается только при интеграции FASTPQ.
  тесты; производственные источники больше не читают переключатель окружения, а инвентарь отражает только тестовые данные.
  сфера применения.
- **Обновление инвентаря + определение области** — Инструмент инвентаризации env теперь помечает файлы `build.rs` как
  объем сборки и отслеживание модулей `#[cfg(test)]`/интеграции, чтобы переключалось только для тестирования (например,
  `IROHA_TEST_*`, `IROHA_RUN_IGNORED`) и флаги сборки CUDA отображаются за пределами счетчика продукции.
  Инвентарь был обновлен 7 декабря 2025 г. (518 ссылок / 144 переменных), чтобы сохранить зеленый цвет защиты различий env-config.
- **Защитник освобождения прокладки окружения топологии P2P** — `IROHA_P2P_TOPOLOGY_UPDATE_MS` теперь запускает детерминированный
  ошибка запуска в сборках выпуска (предупреждение только при отладке/тестировании), поэтому производственные узлы полагаются исключительно на
  `network.peer_gossip_period_ms`. Инвентарь окружения был регенерирован, чтобы отразить охрану и
  обновленный классификатор теперь охватывает переключатели, защищенные `cfg!`, как отладочные/тестовые.

## Высокоприоритетные миграции (производственные пути)

- _Нет (инвентарь обновлен с помощью обнаружения cfg!/debug; защита env-config становится зеленой после усиления P2P-прокладки)._

## Только для разработки/тестирования переключается на ограждение

— Текущая проверка (7 декабря 2025 г.): флаги CUDA только для сборки (`IVM_CUDA_*`) имеют область действия `build`, а
  переключатели жгута (`IROHA_TEST_*`, `IROHA_RUN_IGNORED`, `IROHA_SKIP_BIND_CHECKS`) теперь регистрируются как
  `test`/`debug` в инвентаре (включая защитные шайбы `cfg!`). Никакого дополнительного ограждения не требуется;
  оставляйте будущие дополнения позади `cfg(test)`/помощников, предназначенных только для скамейки, с маркерами TODO, когда прокладки являются временными.

## Конфигурации времени сборки (оставьте как есть)

- Грузовые/функциональные контейнеры (`CARGO_*`, `OUT_DIR`, `DOCS_RS`, `PROFILE`, `CUDA_HOME`,
  `CUDA_PATH`, `JSONSTAGE1_CUDA_ARCH`, `FASTPQ_SKIP_GPU_BUILD` и т. д.) остаются
  Проблемы со сценарием сборки и выходят за рамки миграции конфигурации во время выполнения.

## Следующие действия

1) Запустите `make check-env-config-surface` после обновлений конфигурации, чтобы поймать новые прокладки рабочей среды.
   заранее и назначьте владельцев подсистем/ETA.  
2) Обновляйте инвентарь (`make check-env-config-surface`) после каждой проверки, чтобы
   трекер остается согласованным с новыми ограждениями, а дифференциал защиты env-config остается бесшумным.
