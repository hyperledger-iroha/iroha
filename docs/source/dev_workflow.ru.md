# АГЕНТЫ Рабочий процесс разработки

В этом модуле Runbook объединены ограничения участников из дорожной карты АГЕНТОВ, поэтому
новые патчи следуют тем же воротам по умолчанию.

## Цели быстрого старта

- Запустите `make dev-workflow` (обертку `scripts/dev_workflow.sh`), чтобы выполнить:
  1. И18НИ00000004Х
  2. И18НИ00000005Х
  3. И18НИ00000006Х
  4. И18НИ00000007Х
  5. `swift test` из `IrohaSwift/`
- `cargo test --workspace` работает долго (часто несколько часов). Для быстрых итераций
  используйте `scripts/dev_workflow.sh --skip-tests` или `--skip-swift`, затем запустите полную версию
  последовательность перед отправкой.
- Если `cargo test --workspace` зависает из-за блокировки каталога сборки, перезапустите его с помощью
  `scripts/dev_workflow.sh --target-dir target/codex-tests` (или установить
  `CARGO_TARGET_DIR` на изолированный путь), чтобы избежать конфликтов.
- Все этапы погрузки используют `--locked` для соблюдения политики хранения хранилища.
  `Cargo.lock` нетронутый. Предпочитаете расширять существующие ящики, а не добавлять
  новые участники рабочей области; получить одобрение, прежде чем вводить новый ящик.

## Ограждения- `make check-agents-guardrails` (или `ci/check_agents_guardrails.sh`) завершается сбоем, если
  ветка изменяет `Cargo.lock`, вводит новых членов рабочей области или добавляет новые
  зависимости. Скрипт сравнивает рабочее дерево и `HEAD` с
  `origin/main` по умолчанию; установите `AGENTS_BASE_REF=<ref>`, чтобы переопределить базу.
- `make check-dependency-discipline` (или `ci/check_dependency_discipline.sh`)
  сравнивает зависимости `Cargo.toml` с базовыми и завершается сбоем в новых крейтах; набор
  `DEPENDENCY_DISCIPLINE_ALLOW=<dep1,dep2>` для подтверждения преднамеренного
  дополнения.
- `make check-missing-docs` (или `ci/check_missing_docs_guard.sh`) блокирует новые
  Записи `#[allow(missing_docs)]`, флаги коснулись ящиков (ближайший `Cargo.toml`)
  чьи `src/lib.rs`/`src/main.rs` не имеют документов уровня `//!` и отклоняют новые
  общедоступные элементы без документов `///` относительно базовой ссылки; набор
  `MISSING_DOCS_GUARD_ALLOW=1` только с одобрения рецензента. Охранник также
  проверяет свежесть `docs/source/agents/missing_docs_inventory.{json,md}`;
  регенерируйте с помощью `python3 scripts/inventory_missing_docs.py`.
- `make check-tests-guard` (или `ci/check_tests_guard.sh`) помечает ящики, чьи
  измененным функциям Rust не хватает доказательств модульного тестирования. Карты охраны изменили линии
  к функциям, проходит, если тесты ящика изменились в diff, и в противном случае сканирует
  существующие тестовые файлы для сопоставления вызовов функций с уже существующим покрытием
  считает; ящики без каких-либо соответствующих тестов потерпят неудачу. Установить `TEST_GUARD_ALLOW=1`
  только тогда, когда изменения действительно нейтральны для тестирования и рецензент согласен.
- `make check-docs-tests-metrics` (или `ci/check_docs_tests_metrics_guard.sh`)
  обеспечивает соблюдение политики дорожной карты, согласно которой основные этапы перемещаются вместе с документацией,
  тесты и метрики/панели мониторинга. Когда `roadmap.md` изменяется относительно
  `AGENTS_BASE_REF`, охранник ожидает как минимум одно изменение документа, одно изменение теста,
  и одно изменение показателей/телеметрии/панели мониторинга. Установить `DOC_TEST_METRIC_GUARD_ALLOW=1`
  только с одобрения рецензента.
- `make check-todo-guard` (или `ci/check_todo_guard.sh`) завершается сбоем, когда маркеры TODO
  исчезнут без сопутствующих изменений документации/тестов. Добавить или обновить покрытие
  при разрешении TODO или установите `TODO_GUARD_ALLOW=1` для преднамеренного удаления.
- `make check-std-only` (или `ci/check_std_only.sh`) блокирует `no_std`/`wasm32`
  cfgs, поэтому рабочая область останется только `std`. Установите `STD_ONLY_GUARD_ALLOW=1` только для
  санкционированные эксперименты CI.
- `make check-status-sync` (или `ci/check_status_sync.sh`) сохраняет дорожную карту открытой.
  раздел свободен от завершенных элементов и требует `roadmap.md`/`status.md` для
  меняйтесь вместе, чтобы план/статус оставались согласованными; набор
  `STATUS_SYNC_ALLOW_UNPAIRED=1` только для редких исправлений опечаток только для статуса после
  закрепление `AGENTS_BASE_REF`.
- `make check-proc-macro-ui` (или `ci/check_proc_macro_ui.sh`) запускает пробную сборку.
  Наборы пользовательского интерфейса для ящиков Derivate/proc-macro. Запустите его, коснувшись proc-macros, чтобы
  поддерживать стабильность диагностики `.stderr` и ловить панические регрессии пользовательского интерфейса; набор
  `PROC_MACRO_UI_CRATES="crate1 crate2"`, чтобы сосредоточиться на конкретных ящиках.
- `make check-env-config-surface` (или `ci/check_env_config_surface.sh`) перестраивает
  инвентарь env-toggle (`docs/source/agents/env_var_inventory.{json,md}`),
  сбой, если он устарел, **и** сбой, когда появляются новые прокладки рабочей среды
  относительно `AGENTS_BASE_REF` (определяется автоматически; задается явно при необходимости).
  Обновите трекер после добавления/удаления поисков окружения через
  `python3 scripts/inventory_env_toggles.py --json docs/source/agents/env_var_inventory.json --md docs/source/agents/env_var_inventory.md`;
  используйте `ENV_CONFIG_GUARD_ALLOW=1` только после документирования намеренных ручек envв трекере миграции.
- `make check-serde-guard` (или `ci/check_serde_guard.sh`) регенерирует серде
  инвентаризация использования (`docs/source/norito_json_inventory.{json,md}`) во временную
  местоположение, завершается сбоем, если зафиксированный инвентарь устарел, и отклоняет любые новые
  производство `serde`/`serde_json` соответствует `AGENTS_BASE_REF`. Установить
  `SERDE_GUARD_ALLOW=1` только для экспериментов CI после подачи плана миграции.
- `make guards` применяет политику сериализации Norito: он запрещает новые
  Использование `serde`/`serde_json`, специальные помощники AoS и внешние зависимости SCALE.
  скамейки Norito (`scripts/deny_serde_json.sh`,
  И18НИ00000083Х, И18НИ00000084Х,
  `scripts/check_no_scale.sh`).
- **Политика пользовательского интерфейса proc-macro:** каждый контейнер proc-macro должен поставлять `trybuild`.
  жгут проводов (`tests/ui.rs` с метками «прошел/не прошел») за `trybuild-tests`
  функция. Поместите образцы счастливого пути под `tests/ui/pass`, случаи отклонения — под
  `tests/ui/fail` с зафиксированными выходами `.stderr` и продолжайте диагностику.
  не паникующий и стабильный. Обновите светильники с помощью
  `TRYBUILD=overwrite cargo test -p <crate> -F trybuild-tests` (опционально с
  `CARGO_TARGET_DIR=target-codex`, чтобы не засорять существующие сборки) и
  не полагайтесь на сборки покрытия (ожидаются охранники `cfg(not(coverage))`).
  Для макросов, которые не создают двоичную точку входа, предпочитайте
  `// compile-flags: --crate-type lib` в фикстурах, чтобы сосредоточить внимание на ошибках. Добавить
  новые отрицательные случаи при изменении диагноза.
- CI запускает сценарии ограждения через `.github/workflows/agents-guardrails.yml`.
  поэтому запросы на включение быстро завершаются неудачей, когда политики нарушаются.
- Пример git-хука (`hooks/pre-commit.sample`) запускает Guardrail, Dependency,
  сценарии отсутствующих документов, std-only, env-config и status-sync, чтобы участники могли
  выявляйте нарушения политики до CI. Сохраняйте хлебные крошки TODO для любых намеренных
  последующие действия вместо того, чтобы молча откладывать большие изменения.
