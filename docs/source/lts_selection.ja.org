* 概要

エンジニアリング資源を守るため、リリース後サポートの範囲を絞り込む必要があります。Iroha の複数バージョンを並行で稼働させると、稼働しているバージョンの数だけサポート作業を複製することになりかねません。

すべてのリリースを等しくサポートしようとするのではなく、想定利用ケースのおよそ 80% を 20% の労力で満たせるよう折り合いを付け、最新の安定版だけを支えるアプローチとの間でバランスを取ります。

具体的には、常に最新の =stable= リリースをサポート対象にします。多くの場合、このリリースはテストやドキュメントが十分ではなく、SDK によるサポートも十分ではありません。さらに API や ABI が頻繁に変化するため、互換性の的が動いてしまいます。そのため Iroha では月次リリースのリズムを維持し、そのうち年に 1 回のリリースを長期的なサポートと開発の対象に選びます。

* LTS の選定基準

以下は、特定のバージョンを LTS 候補とする際に推奨される検討事項の一覧です（網羅的ではありません）。

1. リリースには重大な /既知/ の問題が含まれていないことが期待できるべきです。

   もう少し具体的には、基本的なユースケースを十分にカバーしている必要があります。例えば (Pre-)RC-6[fn:1] は旧コンセンサスエンジンを含んだ最後のリリースでした。アクター方式のコンセンサスは安定時で最大 10 TPS しか出せなかったため、(Pre-)RC-6 は実際には使われていても LTS とは認定されませんでした。

   RC-9 には新しいコンセンサスで Iroha ノードを運用するための機能が揃っており、設定構造の変更や =p2p= サブシステムの修正が含まれていました。Iroha を基盤にしたプロジェクトのニーズを満たせる内容だったため、テストが十分とは言えなかったにもかかわらず LTS としてラベル付けされました。

   RC8、RC12、RC13、RC17 はこの基準に照らして失格となりました。

2. 大規模な機能追加が API 互換性の破壊を伴う場合、「従来のやり方」が最も安定しているリリースを LTS 候補として優先すべきです。

   例えば RC12 と RC13 は、まだハードコードされたパーミッションバリデータを使っていた最後のバージョンでした。これらが LTS 候補とされなかったのは、既知の重大な問題、とりわけ安定性の低さやカバレッジ不足、そして RC9 の後で LTS を出すとパーミッションバリデータの硬直性を解消するためのリソースが分散してしまうことが理由です。

   一方で RC14 と RC15 は機能の段階的移行の途上にありました。そのため後者のバージョンは、最適化が進み機能カバレッジも高まっていることから、以前のバージョンからの直接的なアップグレードとして扱えます。

3. リリースは特定のユースケースに対して機能が完全でなければなりません。

   これは 1 の条件とは異なります。RC10 や RC13 には重大な問題はありませんでしたが、主要なステークホルダーが必要とする機能を十分に提供できていませんでした。

4. リリースはすべての SDK と互換でなければなりません。

   Iroha Python のサポートは主に LTS 候補に限定されており、キャッチ 22 の状況を生みますが、これは例外です。その他の主要 SDK、すなわち Kotlin と TypeScript は LTS 候補と互換である必要があります。候補と互換にできない SDK がある場合、そのリリースは失格です。

5. リリースは十分にテストされていなければなりません。

   これは条件 1 を補足するもので、既知機能の不足とテストカバレッジの低さが組み合わさると、重大な問題が存在しないとは言い切れないことを意味します。

   RC6 以降 RC17 までのすべてのリリースに概ね当てはまります。開発チームはインテグレーションテスト群の再構成方法を知らされていましたが、そのプロセスが完了したのは RC17 で =pytest= フレームワークが導入されてからでした。

   インテグレーションテストに合格することは必要条件ですが十分条件ではありません。LTS 候補として認めるには少なくとも次のテストを満たす必要があります。

   1. ユニットテストとインテグレーションテスト。長時間・不安定なものも含めすべて実行しなければなりません。異常に不安定なテストがある場合は問題の兆候であり、そのリリースは失格にすべきです（[[../../flakyness.org]] 参照）。

   2. =pytest= テスト。形式的にはインテグレーションテストに分類されます。

   3. 長期稼働テスト。候補は最低 2 週間の継続テストを経ていなければなりません。LTS に採用された場合は少なくとも 4 週間実施する必要があります。

   4. 負荷テスト。LTS 候補は QA チームが定めた運用・開発上の要件を負荷下で満たす必要があります（エクストラファンクショナルとも呼ばれ、誤ってノンファンクショナルと呼ばれることもあります）。適切なタイミングでブロックを生成できない場合は即失格です。その他の失格条件は QA チームの裁量に委ねられます。

   5. 少なくとも 1 つのプロジェクトで =development= コンテナとして 2 週間以上デプロイされ、重大な問題が発生しないこと。RC9 はこのテストを通過できず、本書で述べる更新された基準では失格となっていたでしょう。

6. リリースは十分にドキュメント化されていなければなりません。

   ドキュメントのない新機能が多数含まれてはいけません。ドキュメントカバレッジの不足は、候補リリースのウィンドウ内（最長でもその期間）に解消できる必要があります。できない場合は自動的に失格です。

通常のリリース手続き（[[release_procedure.org]]）に加えて、LTS では次の追加ステップを踏む必要があります。

通常リリースに /加えて/、=stable= リリースのレビューを担当したのと同じメンバーでレビューされる、タイトル =[release]: <semver> LTS=（<semver> は開発ブランチから切り出したコードのセマンティックバージョン）の別 PR を用意します。通常のチェックリストは適用せず、最大でも 2 人のレビューで、すべての変更が正しい順序で適用され、未ドキュメントの問題が潜り込んでいないことを確認します。

LTS リリースが確定すると、CI は Alpine Linux と Ubuntu Linux 向けに glibc をプラットフォーム標準のバージョンでリンクした Docker コンテナを生成します。また、サポート対象のパッケージをすべて生成し、それぞれのリポジトリに公開しなければなりません。例えば、プルリクエストがマージされた後に正しいパッケージを生成するよう AUR パッケージを更新する必要があります。GitHub のリリースページには LTS と題した個別のリリースを用意し、単一ファイルの =AppImage= を配置します。これらのリリースは、安定版とは互換性がないと明示した別パッケージとして提供する必要があります。

* バックポート方針

一般に、LTS へのバックポートは推奨されません。許容される理由は次の通りです。
1. Iroha API に結び付かない追加（汎用ツールやドキュメントなど）。
2. 重大なセキュリティ脆弱性の修正。
3. 重大なバグの修正。
4. 現行 LTS の主要ステークホルダー／利用者が口頭でバックポートが望ましいと認めた場合に限り、利用を妨げる重大な設計上の欠陥の修正。

バックポートはリリースと同じチャネルで告知しなければなりません。必要な変更は、修正が取り込まれた元のリリースと、バックポート先の双方で backported とラベル付けする必要があります。

告知には次の説明を含めてください。
1. なぜバックポートを選択したのか（上記理由のどれに該当するか）。
2. バックポートが与える影響。
3. 機能テスト、インテグレーションテスト、負荷／長期稼働テストの現状。

* 適用範囲

Iroha の最初の機能完成版（v2.0.0）が発表された時点で、LTS／Stable のリリースポリシーは見直されるべきです。積極的な開発が終われば、コア開発チームは分岐したバージョンのサポートにより多くの時間を割けるようになるはずです。

* 脚注

[fn:1] 命名規則は Iroha のリリース計画が立ち上がるずっと前に固まっていました。そのため、Iroha のどのバージョンもリリース候補（release candidate）と呼ぶことには含意があります。pre release candidate と呼ぶのも厳密には不適切ですが、既存の用語を誤用するのではなく、新しい語を作って微妙なニュアンスの違いを持たせたに過ぎない点で幾分ましです。実際のところ、これらは *開発スナップショット* と呼ぶべきものでした。ただし既存の SDK との整合性を大きく損なわないよう、妥協としてこの呼び名を採用しています。
