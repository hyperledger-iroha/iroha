# Kaigi Գաղտնիություն և ռելեի ձևավորում

Այս փաստաթուղթը ներկայացնում է գաղտնիության վրա հիմնված էվոլյուցիան, որը ներկայացնում է զրոյական գիտելիքներ
մասնակցության ապացույցներ և սոխի ոճի ռելեներ՝ առանց դետերմինիզմի զոհաբերության կամ
հաշվապահական հաշվառման ստուգելիություն:

# Ընդհանուր ակնարկ

Դիզայնը ներառում է երեք շերտ.

- **Գաղտնիության անվանացանկ ** – թաքցրեք մասնակիցների ինքնությունը շղթայի վրա՝ միաժամանակ պահպանելով հյուրընկալողի թույլտվությունները և վճարումները:
- **Օգտագործման անթափանցիկություն** – թույլ տվեք հյուրընկալողներին գրանցել չափված օգտագործումը` առանց յուրաքանչյուր հատվածի մանրամասները հրապարակայնորեն բացահայտելու:
- **Վերածածկման ռելեներ** – երթուղիներ փոխադրող փաթեթները բազմաբնույթ ալիքների միջոցով, որպեսզի ցանցի դիտորդները չկարողանան իմանալ, թե որ մասնակիցներն են հաղորդակցվում:

Բոլոր լրացումները մնում են Norito-նախ, գործարկվում են ABI տարբերակի 1-ի ներքո և պետք է կատարվեն որոշիչ կերպով տարասեռ ապարատում:

#Գոլեր

1. Ընդունել/վտարել մասնակիցներին՝ օգտագործելով զրոյական գիտելիքների ապացույցներ, որպեսզի գրանցամատյանը երբեք չբացահայտի չմշակված հաշվի ID-ները:
2. Պահպանեք հաշվապահական հաշվառման ամուր երաշխիքներ. յուրաքանչյուր միացում, հեռանալ և օգտագործման իրադարձություն դեռևս պետք է դետերմինիստականորեն համապատասխանի:
3. Տրամադրել կամընտիր ռելեի մանիֆեստներ, որոնք նկարագրում են սոխի երթուղիները հսկողության/տվյալների ալիքների համար և կարող են ստուգվել ցանցում:
4. Պահպանեք հետադարձ (լիովին թափանցիկ ցուցակը) գործառնական այն տեղակայումների համար, որոնք գաղտնիություն չեն պահանջում:

# սպառնալիքների մոդելի ամփոփում

- **Հակառակորդներ.** Ցանցի դիտորդներ (ISP), հետաքրքրասեր վավերացնողներ, վնասակար ռելե օպերատորներ և կիսաազնիվ հյուրընկալողներ:
- **Պաշտպանված ակտիվներ. ** Մասնակիցի ինքնությունը, մասնակցության ժամկետը, յուրաքանչյուր հատվածի օգտագործման/վճարման մանրամասները և ցանցի երթուղղման մետատվյալները:
- **Ենթադրություններ.** Հյուրընկալները դեռ սովորում են իրական մասնակցին անջատված շղթայով; հաշվառման գործընկերները հաստատում են ապացույցները դետերմինիստորեն. ծածկույթի ռելեները անվստահելի են, բայց արագությամբ սահմանափակ են. HPKE և SNARK պրիմիտիվներն արդեն կան կոդի բազայում:

# Տվյալների մոդելի փոփոխություններ

Բոլոր տեսակները ապրում են `iroha_data_model::kaigi`-ում:

```rust
/// Commitment to a participant identity (Poseidon hash of account + domain salt).
pub struct KaigiParticipantCommitment {
    pub commitment: FixedBinary<32>,
    pub alias_tag: Option<String>,
}

/// Nullifier unique to each join action, prevents double-use of proofs.
pub struct KaigiParticipantNullifier {
    pub digest: FixedBinary<32>,
    pub issued_at_ms: u64,
}

/// Relay path description used by clients to set up onion routing.
pub struct KaigiRelayManifest {
    pub hops: Vec<KaigiRelayHop>,
    pub expiry_ms: u64,
}

pub struct KaigiRelayHop {
    pub relay_id: AccountId,
    pub hpke_public_key: FixedBinary<32>,
    pub weight: u8,
}
```

`KaigiRecord`-ը ստանում է հետևյալ դաշտերը.

- `roster_commitments: Vec<KaigiParticipantCommitment>` – փոխարինում է բացված `participants` ցանկը, երբ գաղտնիության ռեժիմը միացված է: Դասական տեղակայումները կարող են երկուսն էլ բնակեցված պահել միգրացիայի ժամանակ:
- `nullifier_log: Vec<KaigiParticipantNullifier>` – միայն խստորեն կցվում է, որը ծածկված է պտտվող պատուհանով` մետատվյալները սահմանափակված պահելու համար:
- `room_policy: KaigiRoomPolicy` – ընտրում է դիտողի նույնականացման դիրքորոշումը նիստի համար (`Public` սենյակները արտացոլում են միայն կարդալու ռելեներ; `Authenticated` սենյակները պահանջում են դիտողի տոմսեր, նախքան ելքը ուղարկում է փաթեթները):
- `relay_manifest: Option<KaigiRelayManifest>` – կառուցվածքային մանիֆեստ, որը կոդավորված է Norito-ով, որպեսզի հոփերը, HPKE ստեղները և կշիռները կանոնական մնան առանց JSON շղթաների:
- `privacy_mode: KaigiPrivacyMode` թվով (տես ստորև):

```rust
pub enum KaigiPrivacyMode {
    Transparent,
    ZkRosterV1,
}
```

`NewKaigi`-ը ստանում է համապատասխան ընտրովի դաշտեր, որպեսզի հյուրընկալողները կարողանան ընտրել գաղտնիությունը ստեղծման պահին:


- Դաշտերը օգտագործում են `#[norito(with = "...")]` օգնականներ՝ կանոնական կոդավորումը կիրառելու համար (փոքր էնդիան ամբողջ թվերի համար, դասավորված հոլովակներ ըստ դիրքի):
- `KaigiRecord::from_new`-ը դատարկում է նոր վեկտորները և պատճենում տրամադրված ռելեի մանիֆեստը:

# Հրահանգի մակերեսային փոփոխություններ

## Դեմո արագ մեկնարկի օգնական

Ժամանակավոր ցուցադրությունների և փոխգործունակության թեստերի համար CLI-ն այժմ բացահայտում է
`iroha kaigi quickstart`. Այն:- Կրկին օգտագործում է CLI-ի կազմաձևը (տիրույթ `wonderland` + հաշիվ), եթե այն չվերագրվի `--domain`/`--host`-ի միջոցով:
- Ստեղծում է ժամանակի դրոշմակնիքի վրա հիմնված զանգի անուն, երբ `--call-name`-ը բաց է թողնվում և ներկայացնում է `CreateKaigi`՝ ընդդեմ ակտիվ Torii վերջնակետի:
- Ընտրովի ավտոմատ կերպով միանում է հյուրընկալողին (`--auto-join-host`), որպեսզի դիտողները կարողանան անմիջապես միանալ:
- Թողարկում է JSON ամփոփագիր, որը պարունակում է Torii URL, զանգերի նույնացուցիչներ, գաղտնիության/սենյակի քաղաքականություն, պատրաստի պատճենման միացման հրաման, և կծիկի ուղու փորձարկողները պետք է վերահսկեն (օրինակ՝ `storage/streaming/soranet_routes/exit-<relay-id>/kaigi-stream/*.norito`): Օգտագործեք `--summary-out path/to/file.json`՝ բլբերը պահպանելու համար:

Այս օգնականը **չի** փոխարինում գործող `irohad --sora` հանգույցի անհրաժեշտությունը. գաղտնիության երթուղիները, փաթաթվող ֆայլերը և ռելեի մանիֆեստները մնում են մատյանում: Արտաքին խնջույքների համար ժամանակավոր սենյակները պտտելիս այն պարզապես կտրում է կաթսայատանը:

### Մեկ հրամանով ցուցադրական սցենար

Նույնիսկ ավելի արագ ճանապարհի համար կա ուղեկից սցենար՝ `scripts/kaigi_demo.sh`:
Այն ձեզ համար կատարում է հետևյալը.

1. Ստորագրում է փաթեթավորված `defaults/nexus/genesis.json`-ը `target/kaigi-demo/genesis.nrt`-ի մեջ:
2. Գործարկում է `irohad --sora`-ը ստորագրված բլոկով (գրանցվում է `target/kaigi-demo/irohad.log` տակ) և սպասում, որ Torii-ը բացահայտի `http://127.0.0.1:8080/status`-ը:
3. Աշխատում է `iroha kaigi quickstart --auto-join-host --summary-out target/kaigi-demo/kaigi_summary.json`:
4. Տպում է JSON ամփոփագրի ուղին, գումարած spool գրացուցակը (`storage/streaming/soranet_routes/exit-<relay-id>/kaigi-stream/`), որպեսզի կարողանաք այն համօգտագործել արտաքին փորձարկողների հետ:

Շրջակա միջավայրի փոփոխականներ.

- `TORII_URL` — վերացնել Torii վերջնակետը հարցման համար (կանխադրված `http://127.0.0.1:8080`):
- `RUN_DIR` — վերացնել աշխատանքային գրացուցակը (կանխադրված `target/kaigi-demo`):

Դադարեցրեք ցուցադրությունը՝ սեղմելով `Ctrl+C`; սցենարի թակարդը ինքնաբերաբար ավարտվում է `irohad`: Պտուտակային ֆայլերը և ամփոփագիրը մնում են սկավառակի վրա, այնպես որ դուք կարող եք հանձնել արտեֆակտները գործընթացի ավարտից հետո:

## `CreateKaigi`

- Վավերացնում է `privacy_mode`-ը հյուրընկալողի թույլտվությունների դեմ:
- Եթե մատակարարվում է `relay_manifest`, կիրառեք ≥3 ցատկ, ոչ զրոյական կշիռներ, HPKE բանալիների առկայությունը և եզակիությունը, որպեսզի շղթայական դրսևորումները մնան ստուգման ենթակա:
- Վավերացրեք `room_policy` մուտքագրումը SDKs/CLI-ից (`public` vs `authenticated`) և տարածեք այն SoraNet-ի տրամադրման մեջ, որպեսզի ռելեի քեշերը բացահայտեն GAR-ի ճիշտ կատեգորիաները (Prometheuss): Հոսթները փոխանցում են սա `iroha kaigi create --room-policy …`-ի, JS SDK-ի `roomPolicy` դաշտի կամ `room_policy`-ի միջոցով, երբ Swift-ի հաճախորդները հավաքում են Norito օգտակար բեռը մինչև ներկայացումը:
- Պահպանում է դատարկ պարտավորությունների/չեղյալ տեղեկամատյանները:

## `JoinKaigi`

Պարամետրեր:

- `proof: ZkProof` (Norito բայթերի փաթաթում) – Groth16 ապացույցը, որը հաստատում է, որ զանգահարողը գիտի `(account_id, domain_salt)`, որի Poseidon հեշը հավասար է մատակարարված `commitment`-ին:
- `commitment: FixedBinary<32>`
- `nullifier: FixedBinary<32>`
- `relay_hint: Option<KaigiRelayHop>` – կամընտիր մեկ մասնակցի հաղթահարում հաջորդ թռիչքի համար:

Կատարման քայլեր.

1. Եթե `record.privacy_mode == Transparent`, հետադարձ ընթացիկ վարքագծին:
2. Ստուգեք Groth16-ի ապացույցը շղթայի ռեեստրի `KAIGI_ROSTER_V1` մուտքի դեմ:
3. Համոզվեք, որ `nullifier`-ը չի հայտնվել `record.nullifier_log`-ում:
4. Կցել պարտավորությունների/չեղյալ համարվող գրառումները; եթե `relay_hint`-ը մատակարարված է, կարկատեք ռելեի մանիֆեստի տեսքը այս մասնակցի համար (պահվում է միայն հիշողության աշխատաշրջանի վիճակում, ոչ թե շղթայի վրա):## `LeaveKaigi`

Թափանցիկ ռեժիմը համապատասխանում է ընթացիկ տրամաբանությանը:

Մասնավոր ռեժիմը պահանջում է.

1. Ապացույց, որ զանգահարողը գիտի պարտավորություն `record.roster_commitments`-ում:
2. Չեղարկողի թարմացում, որն ապացուցում է մեկանգամյա օգտագործման արձակուրդը:
3. Հեռացրեք պարտավորությունների/չեղյալ համարվող գրառումները: Աուդիտը պահպանում է տապանաքարերը ֆիքսված պահվող պատուհանների համար՝ կառուցվածքային արտահոսքից խուսափելու համար:

## `RecordKaigiUsage`

Ընդլայնում է ծանրաբեռնվածությունը հետևյալով.

- `usage_commitment: FixedBinary<32>` – հավատարմություն հումքի օգտագործման կրկնակի (տեւողություն, գազ, հատվածի ID):
- Ընտրովի ZK ապացույց, որը հաստատում է, որ դելտան համընկնում է գաղտնագրված տեղեկամատյանների հետ, որոնք տրամադրվել են մատյանից դուրս:

Հյուրընկալները դեռ կարող են ներկայացնել թափանցիկ գումարներ. Գաղտնիության ռեժիմը միայն պարտավորությունների դաշտը պարտադիր է դարձնում:

# Ստուգում և սխեմաներ

- `iroha_core::smartcontracts::isi::kaigi::privacy`-ն այժմ կատարում է ամբողջական ցուցակը
  լռելյայն ստուգում: Այն լուծում է `zk.kaigi_roster_join_vk` (միանում է) և
  `zk.kaigi_roster_leave_vk` (հեռանում է) կոնֆիգուրացիայից,
  փնտրում է համապատասխան `VerifyingKeyRef`-ը WSV-ում (ապահովելով գրառումը
  `Active`, հետնամասի/շղթայի նույնացուցիչները համընկնում են, և պարտավորությունները համընկնում են), գանձումներ
  բայթերի հաշվառում և ուղարկում կազմաձևված ZK հետին պլան:
- `kaigi_privacy_mocks` հատկանիշը պահպանում է դետերմինիստական կոճղային ստուգիչը
  միավորի/ինտեգրման թեստերը և սահմանափակված CI-ի աշխատանքները կարող են աշխատել առանց Halo2 հետին պլանի:
  Արտադրական կառուցվածքները պետք է անջատված պահեն գործառույթը՝ իրական ապացույցներ կիրառելու համար:
- Արկղը թողարկում է կոմպիլյացիայի ժամանակի սխալ, եթե `kaigi_privacy_mocks`-ը միացված է
  ոչ թեստային, ոչ `debug_assertions` կառուցվածք, որը կանխում է երկուականների պատահական թողարկումը
  կոճղով առաքումից:
- Օպերատորները պետք է (1) գրանցեն կառավարման համակարգի միջոցով սահմանված ցուցակի ստուգիչ, և
  (2) սահմանել `zk.kaigi_roster_join_vk`, `zk.kaigi_roster_leave_vk` և
  `zk.kaigi_usage_vk` `iroha_config`-ում, որպեսզի հյուրընկալողները կարողանան լուծել դրանք գործարկման ժամանակ:
  Քանի դեռ ստեղները չեն, գաղտնիությունը միանում է, հեռանում է, և օգտագործման զանգերը ձախողվում են
  դետերմինիստորեն.
- `crates/kaigi_zk`-ն այժմ առաքում է Halo2 սխեմաներ՝ ցուցակի միացման/հեռացման և օգտագործման համար
  պարտավորություններ բազմակի օգտագործման կոմպրեսորների հետ մեկտեղ (`commitment`, `nullifier`,
  `usage`): Ցուցակի սխեմաները բացահայտում են Մերկլի արմատը (չորս փոքր-էնդիան
  64-բիթանոց վերջույթներ) որպես լրացուցիչ հանրային մուտքեր, որպեսզի հյուրընկալողը կարողանա ստուգել ապացույցը
  պահպանված ցուցակի արմատի դեմ նախքան ստուգումը: Օգտագործման պարտավորություններն են
  պարտադրված `KaigiUsageCommitmentCircuit`-ով, որը կապում է «(տեւողությունը, գազ,
  սեգմենտ)` մատյանների հեշին:
- `Join` շղթայի մուտքեր՝ `(commitment, nullifier, domain_salt)` և մասնավոր
  `(account_id)`. Հանրային մուտքերը ներառում են `commitment`, `nullifier` և
  Մերկլի արմատի չորս վերջույթներ ցուցակի պարտավորությունների ծառի համար (անվանացանկ
  մնում է շղթայից դուրս, բայց արմատը կապված է արտագրության մեջ):
- Դետերմինիզմ. մենք ամրագրում ենք Պոսեյդոնի պարամետրերը, շղթայի տարբերակները և ինդեքսները
  գրանցամատյան. Ցանկացած փոփոխություն համընկնում է `KaigiPrivacyMode`-ից մինչև `ZkRosterV2`
  թեստեր/ոսկե ֆայլեր.

# Սոխի երթուղային ծածկույթ

## Ռելեի գրանցում- Ռելեներ ինքնագրանցվում են որպես տիրույթի մետատվյալների մուտքագրումներ `kaigi_relay::<relay_id>`, ներառյալ HPKE հիմնական նյութը և թողունակության դասը:
- `RegisterKaigiRelay` հրահանգը պահպանում է նկարագրիչը տիրույթի մետատվյալներում, թողարկում է `KaigiRelayRegistered` ամփոփագիր (HPKE մատնահետքով և թողունակության դասով) և կարող է կրկին կանչվել՝ ստեղները դետերմինիստականորեն պտտելու համար:
- Կառավարումը վերահսկում է թույլտվությունների ցուցակները տիրույթի մետատվյալների միջոցով (`kaigi_relay_allowlist`), իսկ ռելեի գրանցման/մանիֆեստի թարմացումները պարտադրում են անդամակցությունը նախքան նոր ուղիներ ընդունելը:

## Մանիֆեստի ստեղծում

- Հոսթները կառուցում են բազմահոփ ուղիներ (նվազագույն երկարությունը 3) հասանելի ռելեներից: Մանիֆեստը կոդավորում է AccountIds-ի և HPKE հանրային բանալիների հաջորդականությունը, որոնք անհրաժեշտ են շերտավոր ծրարը գաղտնագրելու համար:
- `relay_manifest`-ը, որը պահվում է շղթայում, պարունակում է հոփ նկարագրիչներ և ժամկետանց (Norito-կոդավորված `KaigiRelayManifest`); իրական ժամանակավոր ստեղները և յուրաքանչյուր նստաշրջանի օֆսեթները փոխանակվում են մատյանից դուրս՝ օգտագործելով HPKE:

## Ազդանշանային և լրատվամիջոցներ

- SDP/ICE փոխանակումը շարունակվում է Kaigi-ի մետատվյալների միջոցով, բայց կոդավորված է մեկ հոփով: Վավերացնողները տեսնում են միայն HPKE ծածկագրված տեքստը գումարած վերնագրի ինդեքսները:
- Մեդիա փաթեթները շարժվում են ռելեներով՝ օգտագործելով QUIC՝ կնքված ծանրաբեռնվածությամբ: Յուրաքանչյուր հոփ ապակոդավորում է մեկ շերտ՝ հաջորդ հոփ հասցեն սովորելու համար; վերջնական ստացողը ստանում է մեդիա հոսքը բոլոր շերտերը հեռացնելուց հետո:

## Failover

- Հաճախորդները վերահսկում են ռելեի առողջությունը `ReportKaigiRelayHealth` հրահանգի միջոցով, որը պահպանում է ստորագրված հետադարձ կապը տիրույթի մետատվյալներում (`kaigi_relay_feedback::<relay_id>`), հեռարձակում է `KaigiRelayHealthUpdated` և թույլ է տալիս կառավարման/հոսթավարներին հիմնավորել ընթացիկ հասանելիությունը: Երբ ռելեը ձախողվում է, հոսթը թողարկում է թարմացված մանիֆեստ և գրանցում `KaigiRelayManifestUpdated` իրադարձություն (տես ստորև):
- Հոսթները կիրառում են մանիֆեստի փոփոխությունները գրանցամատյանում `SetKaigiRelayManifest` հրահանգի միջոցով, որը փոխարինում է պահված ուղին կամ ամբողջությամբ մաքրում այն: Մաքրումը թողարկում է ամփոփում `hop_count = 0`-ով, որպեսզի օպերատորները կարողանան հետևել անցումը դեպի ուղիղ երթուղի:
- Prometheus չափումներ (`kaigi_relay_registered_total`, `kaigi_relay_registration_bandwidth_class`, `kaigi_relay_manifest_updates_total`, `kaigi_relay_manifest_hop_count`, `kaigi_relay_health_reports_total`, `kaigi_relay_health_reports_total`, `kaigi_relay_registration_bandwidth_class`, `kaigi_relay_manifest_updates_total`, `kaigi_relay_manifest_hop_count`, `kaigi_relay_health_reports_total`, `kaigi_relay_health_reports_total`, Norito `kaigi_relay_failover_hop_count`) այժմ մակերևութային ռելեի անջատում, առողջական վիճակ և ձախողման արագություն օպերատորների վահանակների համար:

# Իրադարձություններ

Ընդլայնել `DomainEvent` տարբերակները.

- `KaigiRosterSummary` – թողարկվել է անանուն թվերով և ընթացիկ ցուցակով
  արմատ, երբ ցուցակը փոխվում է (արմատը `None` է թափանցիկ ռեժիմում):
- `KaigiRelayRegistered` – թողարկվում է ամեն անգամ, երբ ռելեի գրանցումը ստեղծվում կամ թարմացվում է:
- `KaigiRelayManifestUpdated` – արտանետվում է, երբ փոխվում է ռելեի մանիֆեստը:
- `KaigiRelayHealthUpdated` – թողարկվում է, երբ հյուրընկալողները ներկայացնում են ռելեի առողջության հաշվետվություն `ReportKaigiRelayHealth`-ի միջոցով:
- `KaigiUsageSummary` – արտանետվում է յուրաքանչյուր օգտագործման հատվածից հետո՝ բացահայտելով միայն ընդհանուր գումարները:

Իրադարձությունները սերիականացվում են Norito-ով` բացահայտելով միայն պարտավորությունների հեշերը և հաշվումները:CLI գործիքավորումը (`iroha kaigi …`) փաթաթում է յուրաքանչյուր ISI, որպեսզի օպերատորները կարողանան գրանցել նիստերը,
ներկայացրեք ցուցակի թարմացումները, զեկուցեք ռելեի առողջության մասին և գրանցեք օգտագործումը առանց ձեռքի գործառնությունների:
Ռելեի մանիֆեստները և գաղտնիության ապացույցները բեռնվում են JSON/hex ֆայլերից, որոնք փոխանցվել են
CLI-ի սովորական ներկայացման ուղին, ինչը պարզ է դարձնում սցենարի պայմանագիրը
ընդունելություն բեմական միջավայրում.

# Գազի հաշվառում

- Նոր հաստատուններ `crates/iroha_core/src/gas.rs`-ում.
  - `BASE_KAIGI_JOIN_ZK`, `BASE_KAIGI_LEAVE_ZK` և `BASE_KAIGI_USAGE_ZK`
    չափորոշված Halo2-ի ստուգման ժամանակացույցի համեմատ (≈1.6ms ցուցակի համար
    միանում/հեռանում է, ≈1.2ms՝ Apple M2 Ultra-ում օգտագործման համար): Հավելավճարները շարունակվում են

- `RecordKaigiUsage`-ը պարտավորվում է վճարել լրացուցիչ վճար՝ հիմնված պարտավորության չափի և ապացույցների հաստատման վրա:
- Կալիբրացիոն զրահը կվերօգտագործի գաղտնի ակտիվների ենթակառուցվածքը ֆիքսված սերմերով:

# Փորձարկման ռազմավարություն

- Միավորի թեստեր, որոնք հաստատում են Norito կոդավորումը/վերծանումը `KaigiParticipantCommitment`, `KaigiRelayManifest`-ի համար:
- Ոսկե թեստեր JSON դիտման համար, որոնք ապահովում են կանոնական պատվեր:
- Ինտեգրման թեստեր, որոնք պտտվում են մինի ցանցի հետ (տես
  `crates/iroha_core/tests/kaigi_privacy.rs` ընթացիկ ծածկույթի համար):
  - Մասնավոր միանալու/հեռանալու ցիկլեր՝ օգտագործելով կեղծ ապացույցներ (հատկանիշի դրոշակ `kaigi_privacy_mocks`):
  - Ռելե մանիֆեստի թարմացումները, որոնք տարածվում են մետատվյալների իրադարձությունների միջոցով:
- Փորձեք կառուցել UI թեստեր, որոնք ընդգրկում են հյուրընկալողի սխալ կազմաձևումը (օրինակ՝ գաղտնիության ռեժիմում բացակայում է ռելեի մանիֆեստը):
- Երբ աշխատում են միավորի/ինտեգրման թեստերը սահմանափակ միջավայրերում (օրինակ՝ Codex
  Sandbox), արտահանել `NORITO_SKIP_BINDINGS_SYNC=1`՝ շրջանցելու Norito կապը
  համաժամացման ստուգում, որն իրականացվում է `crates/norito/build.rs`-ի կողմից:

# Միգրացիոն ծրագիր

1. ✅ Տվյալների մոդելային հավելումներ ուղարկել `KaigiPrivacyMode::Transparent` լռելյայն:
2. ✅ Լարերի երկուղի ստուգում. արտադրությունն անջատում է `kaigi_privacy_mocks`,

   դեռևս միացնել հատկանիշը դետերմինիստական կոճղերի համար:
