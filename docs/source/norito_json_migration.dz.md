# Norito JSON Migration Guide

This guide captures the end-to-end migration of the workspace from Serde/`serde_json`
to Norito-native serialization. It complements the automated inventory generated by
`scripts/inventory_serde_usage.py` and describes the Norito API surface that replaces
Serde derives, visitors, JSON DOM helpers, and config loaders.

> **Status (Dec 6, 2026):** Norito-only serialization is complete across production
> crates with guardrails enforced in CI. Builder-level JSON roundtrips now cover
> account/asset-definition/NFT registration surfaces, closing the migration checklist.

## Why switch to Norito end-to-end?

- Determinism: Norito’s binary and JSON codecs share the same canonicalization rules
  that underpin on-chain hashing and replay protection.
- Feature parity: Norito derives now cover the patterns previously handled through
  `serde::{Serialize, Deserialize}`, including map visitor hooks, enum tagging, and
  raw-value passthrough.
- Simpler dependency graph: removing Serde eliminates duplicate derive machinery and
  an entire class of feature guards.

## Serde Use-Case Checklist

| Serde use case | Norito replacement | Reference |
| -------------- | ------------------ | --------- |
| Struct derives / trait bounds | `NoritoSerialize` + `NoritoDeserialize` derives and manual impls | "Struct + enum serialization" |
| Enums + tagging | `#[norito(tag = ...)]`, `#[norito(rename = ...)]`, `#[norito(other)]` | "Enum tagging" |
| Map / collection access | `json::MapVisitor`, `json::SeqVisitor`, `json::CoerceKey`, derives with `#[norito(with = ...)]` | "Visitor + builder hooks", "Map key handling" |
| Optional fields & defaults | `#[norito(default)]`, `#[norito(skip_serializing_if = Option::is_none)]`, `MapVisitor::read_optional` | "Optional fields and defaults" |
| Config parsing & IO | `norito_config::TomlValue`, `norito::json::from_reader`, `to_writer` | "Config loading and Norito-backed TOML" |
| Snapshot / binary payloads | `norito::core::{to_bytes, from_bytes}`, JSON pretty printers | "Snapshot + binary IO" |
| Visitor hooks & custom parsing | `JsonSerialize`/`JsonDeserialize` traits, `json::Visitor` dispatch helpers | "Visitor + builder hooks" |

### Visitor + builder hooks

- `norito::json::MapVisitor` / `SeqVisitor`: typed equivalents of Serde’s
  `MapAccess`/`SeqAccess` with built-in helpers (`missing_field`,
  `duplicate_field`, `unknown_field`).
- `norito::json::RawValue`: owned raw JSON buffer that replaces
  `serde_json::value::RawValue` for deferring parsing.
- `norito::json::CoerceKey`: helper for canonical map key coercion when porting
  Serde’s custom key serialization.
- `norito::json::Visitor` + `visit_value` expose typed hooks (`visit_bool`,
  `visit_i64`, `visit_map`, `visit_seq`, …) so bespoke parsers can mirror
  Serde’s visitor dispatch without touching the DOM.
- Procedural derives `#[derive(JsonSerialize, JsonDeserialize)]` (named structs)
  generate deterministic writers + `MapVisitor`-based parsers, providing a
  drop-in replacement for the common Serde derive pattern.

### Enum tagging

- Use `#[norito(tag = "kind", content = "payload")]` for adjacently tagged
  enums. The current `JsonSerialize`/`JsonDeserialize` derives require both
  keys; support for tag-only and untagged layouts will land as part of the
  ongoing Norito JSON parity work.
- `#[serde(other)]` maps to `#[norito(other)]`, preserving the “catch-all”
  semantics for unions.
- `#[norito(rename = "...")]` works on individual variants to mirror Serde’s
  renaming controls.

### Optional fields and defaults

- `Option<T>` continues to round-trip via `NoritoSerialize`/`NoritoDeserialize`.
  Pair `#[norito(skip_serializing_if = Option::is_none)]` with
  `#[norito(default)]` or `#[norito(default = "fn")]` to mirror Serde’s
  skip-if-none and default initialisers.
- For manual implementations, `json::MapVisitor::read_optional` (for keys) and
  `visit_optional` helpers on `json::Visitor` remove the need for custom
  `Option` juggling inside visitors.
- `json::Value::is_null()` and `Value::take_if_null()` ease bridging from DOM
  pipelines that previously switched on `serde_json::Value::Null`.

> **Status (Nov 7, 2025):** The `JsonSerialize`/`JsonDeserialize` derives now
> honour `#[norito(default = "path")]` and
> `#[norito(skip_serializing_if = "path")]` on named-struct fields, bringing
> parity with the manual JSON implementations used during the migration.

### Map key handling

- The derives understand `#[norito(with = "module::map_key")]` to plug custom
  string ↔ key conversions in the same fashion as `#[serde(with = ...)]`.
- Manual visitors can lean on `json::MapVisitor::read_string_key` and
  `read_identifier` for enums or newtypes that previously used Serde’s
  `visit_map` + `next_key` / `next_value` dance.
- `json::CoerceKey` supports canonical map key coercion (e.g., numbers →
  strings, case folding) and replaces Serde helpers such as `deserialize_with`
  on map fields.

## JSON DOM and streaming helpers

- DOM access: `norito::json::Value`, `Map`, `Number`, `json!` macro.
- Round-trip conversions: `norito::json::value::{to_value, from_value, to_raw_value}`.
- Parser/writer access: `norito::json::{Parser, Reader, TapeWalker}` plus
  iterator adapters (`Tokens`).
- New readers: `from_reader<R: Read, T: JsonDeserialize>` for buffered input,
  `from_slice`, and `from_io_buf` for small stack buffers.
- Writers: `to_writer<W: Write, T: JsonSerialize>` mirrors Serde’s streaming APIs.

## Config loading and Norito-backed TOML

Config crates migrate from Serde to Norito in three layers:

1. Parse TOML with `norito_config::TomlValue` (thin wrapper around `toml_edit`
   without Serde dependencies).
2. Apply defaults/env overrides using `iroha_config_base::read` just as before,
   but builder methods now emit/consume `NoritoDeserialize` values.
3. Transform TOML → JSON via `norito_config::toml::to_json_value` when a JSON
   intermediate is needed (e.g., to reuse JSON visitors).

### Mapping Serde helpers to Norito config APIs

| Serde helper | Norito alternative |
| ------------- | ------------------ |
| `serde_with::DeserializeFromStr` | `norito::json::JsonDeserialize` + `FromStr` wrapper via `json::MapVisitor::read_string` |
| `serde_with::SerializeDisplay` | Implement `JsonSerialize` by delegating to `Display` or `JsonSerialize::json_serialize` on the formatted string. |
| `#[serde(deny_unknown_fields)]` | Use `json::MapVisitor::reject_unknown_fields()` inside manual impls or enable `deny_unknown` on the Norito derive (mirrors Serde). |
| `#[serde(default = "fn")]` | `#[norito(default = "fn")]` provides the same behaviour. |
| `serde_json::from_reader` | `norito::json::from_reader` (reads via the Norito parser, no Serde). |

### Snapshot + binary IO

- All binary snapshot paths call `norito::core::{to_bytes, from_bytes}` with the
  derived `NoritoSerialize`/`NoritoDeserialize` implementations.
- JSON snapshots in tests use `norito::json::{to_json_pretty, from_json}`.

## Error mapping

- Replace `serde_json::Error` with `norito::json::Error` and use `?` directly.
- For config and CLI surfaces, implement `From<norito::json::Error>` on local
  error types and retain existing `eyre`/`thiserror` conversions.

## Migration workflow

1. Run the inventory script to identify remaining Serde usage in the crate you
   are touching. Focus on production modules first (`config` → `core` → `cli`).
2. Swap derives/imports to `Norito*` traits, ensuring tests compile with Norito
   as the only serialization dependency.
3. Replace JSON access helpers (`serde_json::Value`, macros, visitors) with the
   corresponding Norito modules.
4. Add Norito round-trip tests (`norito::to_bytes`/`from_bytes`,
   `norito::json::to_json`/`from_json`) for each new or migrated type.
5. Update crate docs/in-source comments to state that Norito is the canonical
   serialization backend.

Refer to `status.md` for the per-crate migration checklist and current status.

## Crate-by-crate playbook

### Config stack (`iroha_config`, `iroha_config_base`, `iroha_config_cli`)

- Replace `serde::{Serialize, Deserialize}` derives with
  `NoritoSerialize`/`NoritoDeserialize` or manual `JsonSerialize`/
  `JsonDeserialize` impls when TOML → JSON coercion requires visitors.
- Parse configuration sources through `norito_config::load_layers` (user →
  actual → defaults) and thread the resulting Norito structs to dependants via
  existing builder APIs.
- Convert `serde_with` adapters to `#[norito(with = ...)]` or explicit visitor
  helpers (`json::MapVisitor::read_string_key`).
- Surface Norito parsing errors as `iroha_config_base::Error` via
  `From<norito::json::Error>` conversions so CLI/services emit deterministic
  diagnostics.
- Add regression tests in `crates/iroha_config/tests` that load representative
  user configs with Norito and assert parity with historical Serde snapshots.

> **Status (Nov 7, 2025):** `iroha_config` has dropped its `serde` and
> `serde_with` dependencies; configuration defaults, visitors, and emitters now
> rely exclusively on Norito traits.

### Core runtime (`iroha_core`, `iroha_data_model`, `iroha_executor`, `ivm`)

- Update state/snapshot IO to call `norito::core::{to_bytes, from_bytes}` and
  delete Serde bridges (for example `serde_json::to_vec`).
- Ensure ABI and network message types derive `NoritoSerialize`/
  `NoritoDeserialize`; hand-write impls for pointer-sensitive structs where
  canonical ordering must stay explicit.
- Rewrite bespoke visitors (ISI triggers, ledger history, admission manifests)
  using `json::MapVisitor` / `json::Visitor` so validation logic stays
  deterministic.
- Regenerate golden snapshots under Norito and compare byte payloads with
  archived Serde output; document any intentional deltas and add Norito-hosted
  goldens alongside.
- Extend integration tests to exercise Norito-only serialization paths,
  especially admission manifests, governance ballots, and telemetry payloads.

### CLI & tooling (`iroha_cli`, `scripts/`, `iroha_kagami`, `iroha_swarm`)

- Swap command DTOs to Norito derives and route JSON writers through
  `norito::json::to_writer` / `to_json_pretty`.
- Replace `serde_json::json!` usages in scripts with `norito::json::json!` and
  parse inputs via `norito::json::from_reader` for streaming scenarios.
- Reuse the shared Norito config loader from `iroha_config_base` wherever the
  CLI accepted Serde-based configuration blobs.
- Update snapshot exporters/importers to rely on Norito binary codecs and add
  round-trip tests in `crates/iroha_cli/tests` to guard regressions.
