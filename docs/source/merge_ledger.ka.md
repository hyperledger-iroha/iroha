# შერწყმა ლეჯერის დიზაინი - ზოლის საბოლოო და გლობალური შემცირება

ეს ჩანაწერი ასრულებს Milestone 5-ის შერწყმის წიგნის დიზაინს. ის განმარტავს
არა ცარიელი ბლოკის პოლიტიკა, ჯვარედინი QC შერწყმის სემანტიკა და საბოლოო სამუშაო პროცესი
რომელიც აკავშირებს ხაზის დონეზე შესრულებას გლობალურ მსოფლიო სახელმწიფოს ვალდებულებასთან.

დიზაინი აფართოებს Nexus არქიტექტურას, რომელიც აღწერილია `nexus.md`-ში. ტერმინები, როგორიცაა
"ზოლის ბლოკი", "ზოლი QC", "შერწყმის მინიშნება" და "შეერთების წიგნი" მემკვიდრეობით იღებენ მათ
განმარტება ამ დოკუმენტიდან; ეს შენიშვნა ყურადღებას ამახვილებს ქცევის წესებზე და
განხორციელების ინსტრუქცია, რომელიც უნდა განხორციელდეს გაშვების, შენახვისა და WSV-ით
ფენები.

## 1. არა ცარიელი ბლოკის პოლიტიკა

**წესი (MUST):** ზოლის შემოთავაზების მომწოდებელი გასცემს ბლოკს მხოლოდ მაშინ, როდესაც ბლოკი შეიცავს at
მინიმუმ ერთი შესრულებული ტრანზაქციის ფრაგმენტი, დროზე დაფუძნებული ტრიგერი ან დეტერმინისტული
არტეფაქტის განახლება (მაგ., DA არტეფაქტის შეკრება). ცარიელი ბლოკები აკრძალულია.

** შედეგები: **

- სლოტის შენარჩუნება-ცოცხალი: როდესაც არცერთი ტრანზაქცია არ აკმაყოფილებს მის განმსაზღვრელ ჩაბარების ფანჯარას,
ზოლი არ ასხივებს ბლოკს და უბრალოდ მიიწევს შემდეგ სლოტზე. შერწყმის წიგნი
რჩება წინა წვერზე ამ ზოლისთვის.
- ტრიგერის ჯგუფური შეკრება: ფონური ტრიგერები, რომლებიც არ წარმოქმნიან მდგომარეობის გადასვლას (მაგ.
cron, რომელიც კიდევ ერთხელ ადასტურებს ინვარიანტებს) ითვლება ცარიელი და უნდა გამოტოვოთ ან
ბლოკის წარმოებამდე სხვა სამუშაოებთან ერთად შეფუთული.
- ტელემეტრია: `pipeline_detached_merged` და შემდგომი მეტრიკის მკურნალობა გამოტოვებულია
სლოტები მკაფიოდ — ოპერატორებს შეუძლიათ განასხვავონ „არ მუშაობს“ „მილსადენის გაჩერებისგან“.
- გამეორება: ბლოკის საცავი არ აყენებს სინთეზურ ცარიელ ჩანაცვლებებს. კურა
გამეორების მარყუჟი უბრალოდ აკვირდება იმავე მშობლის ჰეშს თანმიმდევრული სლოტებისთვის, თუ არა
ბლოკი გამოიცა.

**კანონიკური შემოწმება:** ბლოკის შეთავაზებისა და ვალიდაციის დროს, `ValidBlock::commit`
ამტკიცებს, რომ ასოცირებული `StateBlock` ატარებს მინიმუმ ერთ ჩადებულ გადაფარვას
(დელტა, არტეფაქტი, ტრიგერი). ეს შეესაბამება `StateBlock::is_empty` მცველს
რაც უკვე უზრუნველყოფს no-op ჩაწერის ამოღებას. აღსრულება ხდება ადრე
ხელმოწერების მოთხოვნილებაა, ასე რომ კომიტეტები არასოდეს აძლევენ ხმას ცარიელ ტვირთებზე.

## 2. Cross-Lane QC Merge Semantics

მისი კომიტეტის მიერ დასრულებული თითოეული ზოლის ბლოკი `B_i` აწარმოებს:

- `lane_state_root_i`: Poseidon2-SMT ვალდებულება თითო DS სახელმწიფო ფესვების შეხებაზე
ბლოკში.
- `merge_hint_root_i`: მოძრავი კანდიდატი შერწყმის წიგნისთვის (`tag =
"iroha:merge:candidate:v1\0"`).
- `lane_qc_i`: გაერთიანებული ხელმოწერები ზოლის კომიტეტისგან
  execution-vote preimage (ბლოკის ჰეში, `parent_state_root`,
  `post_state_root`, სიმაღლე/ხედვა/ეპოქა, chain_id და რეჟიმის ტეგი).

კვანძების შერწყმა აგროვებს `{(B_i, lane_qc_i, merge_hint_root_i)}` უახლეს რჩევებს
ყველა ზოლი `i ∈ [0, K)`.

** შერწყმის ჩანაწერი (უნდა):**

```
MergeLedgerEntry {
    epoch_id: u64,
    lane_tips: [Hash32; K],
    merge_hint_root: [Hash32; K],
    global_state_root: Hash32,
    merge_qc: QuorumCertificate,
}
```- `lane_tips[i]` არის ზოლის ბლოკის ჰეში, შერწყმის შესვლის ბეჭდები ზოლისთვის
  `i`. თუ ზოლმა არ გამოუშვა ბლოკი წინა შერწყმის შემდეგ, ეს მნიშვნელობა არის
  გაიმეორა.
- `merge_hint_root[i]` არის `merge_hint_root` შესაბამისი ზოლიდან
  ბლოკი. ის მეორდება, როდესაც `lane_tips[i]` მეორდება.
- `global_state_root` უდრის `ReduceMergeHints(merge_hint_root[0..K-1])`, a
  Poseidon2 fold დომენის გამოყოფის ტეგით
  `"iroha:merge:reduce:v1\0"`. შემცირება არის განმსაზღვრელი და MUST
  იგივე მნიშვნელობის რეკონსტრუქცია თანატოლებში.
- `merge_qc` არის BFT კვორუმის სერტიფიკატი შერწყმის კომიტეტისგან.
  სერიული ჩანაწერი.

** QC Payload შერწყმა (უნდა):**

შერწყმის კომიტეტის წევრები ხელს აწერენ დეტერმინისტულ დაიჯესტს:

```
merge_qc_digest = blake2b32(
    "iroha:merge:qc:v1\0" ||
    chain_id ||
    norito(MergeLedgerSignPayload {
        view,
        epoch_id,
        lane_tips,
        merge_hint_roots,
        global_state_root,
    })
)
```

- `view` არის გაერთიანების კომიტეტის ხედი, რომელიც მიღებულია ზოლის რჩევებიდან (მაქს.
  `view_change_index` ზოლის სათაურების გასწვრივ დალუქული შესასვლელით).
- `chain_id` არის კონფიგურირებული ჯაჭვის იდენტიფიკატორის სტრიქონი (UTF-8 ბაიტი).
- დატვირთვა იყენებს Norito დაშიფვრას ზემოთ ნაჩვენები ველის თანმიმდევრობით.

შედეგად დაიჯესტი ინახება `merge_qc.message_digest`-ში და წარმოადგენს შეტყობინებას
დამოწმებული BLS ხელმოწერებით.

** QC Construction-ის შერწყმა (უნდა):**

- შერწყმის კომიტეტის სია არის ამჟამინდელი commit-topology ვალიდიატორის ნაკრები.
- საჭირო კვორუმი = `commit_quorum_from_len(roster_len)`.
- `merge_qc.signers_bitmap` შიფრავს მონაწილე ვალიდატორის ინდექსებს (LSB-პირველი)
  ვალდებულება-ტოპოლოგიის მიხედვით.
- `merge_qc.aggregate_signature` არის BLS-ნორმალური აგრეგატი მონელებისთვის
  ზემოთ.

** ვალიდაცია (უნდა):**

1. გადაამოწმეთ თითოეული `lane_qc_i` `lane_tips[i]`-თან და დაადასტურეთ ბლოკის სათაურები
   მოიცავს შესატყვისი `merge_hint_root_i`.
2. დარწმუნდით, რომ არ არის `lane_qc_i` წერტილი `Invalid` ან შეუსრულებელ ბლოკზე. The
   ზემოაღნიშნული არა ცარიელი პოლიტიკა უზრუნველყოფს, რომ სათაური შეიცავს სახელმწიფოს გადაფარვას.
3. ხელახლა გამოთვალეთ `ReduceMergeHints` და შეადარეთ `global_state_root`-ს.
4. ხელახლა გამოთვალეთ შერწყმის QC დაიჯესტი და გადაამოწმეთ ხელმომწერის ბიტმაპი, კვორუმის ზღვარი,
   და ერთობლივი ხელმოწერა commit-topology როსტერთან მიმართებაში.

**დაკვირვებადობა:** შერწყმის კვანძები ასხივებენ Prometheus მრიცხველებს ამისთვის
`merge_entry_lane_repeats_total{i}` ხაზების ხაზგასასმელად, რომლებზეც გამოტოვებულია სლოტები
ოპერატიული ხილვადობა.

## 3. საბოლოო სამუშაო პროცესი

### 3.1 ზოლის დონის დასრულება

1. ტრანზაქციები დაგეგმილია თითო ზოლზე დეტერმინისტულ სლოტებში.
2. შემსრულებელი იყენებს გადაფარვებს `StateBlock`-ში, აწარმოებს დელტას და
არტეფაქტები.
3. დადასტურების შემდეგ ზოლის კომიტეტი ხელს აწერს აღსრულება-კენჭისყრის პრეიმაჟს, რომ
   აკავშირებს ბლოკის ჰეშს, მდგომარეობის ფესვებს და სიმაღლეს/ხედს/ეპოქას. ტუპლი
   `(block_hash, lane_qc_i, merge_hint_root_i)` ითვლება შესახვევად.
4. მსუბუქი კლიენტებმა შეიძლება მიიჩნიონ ზოლის წვერი, როგორც საბოლოო DS-შეზღუდული მტკიცებულებებისთვის, მაგრამ
უნდა ჩაიწეროს ასოცირებული `merge_hint_root` შერწყმის წიგნის შესაჯერებლად
მოგვიანებით.ზოლის კომიტეტები არის თითო მონაცემთა სივრცეში და არ ცვლის გლობალურ ვალდებულებას
ტოპოლოგია. კომიტეტის ზომა ფიქსირდება `3f+1`, საიდანაც `f` მოდის
მონაცემთა სივრცის კატალოგი (`fault_tolerance`). ვალიდატორის ფონდი არის მონაცემთა სივრცე
ვალიდიატორები (ზოლის მართვა ვლინდება ადმინისტრატორის მიერ მართული ზოლებისთვის ან საჯარო ზოლებისთვის
ფსონების ჩანაწერები ფსონების მიერ არჩეული ბილიკებისთვის). კომიტეტის წევრობა არის
განმსაზღვრელი სინჯის აღება ეპოქაში ერთხელ VRF ეპოქის თესლის გამოყენებით შეკრული
`dataspace_id` და `lane_id`. თუ აუზი `3f+1`-ზე პატარაა, ზოლის დასრულება
ჩერდება კვორუმის აღდგენამდე (გადაუდებელი აღდგენა განიხილება ცალკე).

### 3.2 Merge-Ledger Finality

1. შერწყმის კომიტეტი აგროვებს უახლეს რჩევებს, ამოწმებს თითოეულ `lane_qc_i`-ს და
აშენებს `MergeLedgerEntry`-ს, როგორც ზემოთ არის განსაზღვრული.
2. დეტერმინისტული შემცირების შემოწმების შემდეგ შერწყმის კომიტეტი ხელს აწერს
ჩანაწერი (`merge_qc`).
3. კვანძები აერთებენ ჩანაწერს შერწყმის წიგნის ჟურნალში და აგრძელებენ მას გვერდით
ზოლის ბლოკის მითითებები.
4. `global_state_root` ხდება მსოფლიო სახელმწიფოს ავტორიტეტული ვალდებულება
ეპოქა/სლოტი. სრული კვანძები განაახლებს WSV გამშვები პუნქტის მეტამონაცემებს ამის ასახვის მიზნით
ღირებულება; დეტერმინისტული გამეორება უნდა აწარმოებდეს იგივე შემცირებას.

### 3.3 WSV და შენახვის ინტეგრაცია

- `State::commit_merge_entry` ჩაწერს თითო ზოლის მდგომარეობის ფესვებს და
  საბოლოო `global_state_root`, ხიდის ზოლის შესრულება გლობალურ საკონტროლო ჯამთან.

  გამეორებას შეუძლია აღადგინოს როგორც ხაზის დონის, ასევე გლობალური საბოლოო თანმიმდევრობა.

  ჩანაცვლების შერწყმის ჩანაწერები იქმნება მანამ, სანამ მინიმუმ ერთი ხაზი არ გამოიმუშავებს ახალს
  ბლოკი.
- API ზედაპირები (Torii, ტელემეტრია) ასახავს ორივე ზოლის მინიშნებებს და უახლეს შერწყმას
  ჩანაწერი, რათა ოპერატორებმა და კლიენტებმა შეძლონ თითო ზოლის და გლობალური ხედების შეჯერება.
