# Genesis კონფიგურაცია

`genesis.json` ფაილი განსაზღვრავს პირველ ტრანზაქციებს, რომლებიც აწარმოებს Iroha ქსელის დაწყებისას. ფაილი არის JSON ობიექტი შემდეგი ველებით:

- `chain` - უნიკალური ჯაჭვის იდენტიფიკატორი.
- `executor` (სურვილისამებრ) – გზა შემსრულებელი ბაიტეკოდისკენ (`.to`). თუ არსებობს,
  genesis მოიცავს განახლების ინსტრუქციას, როგორც პირველ ტრანზაქციას. თუ გამოტოვებულია,
  განახლება არ არის შესრულებული და გამოყენებულია ჩაშენებული შემსრულებელი.
- `ivm_dir` – დირექტორია, რომელიც შეიცავს IVM ბაიტეკოდების ბიბლიოთეკებს. ნაგულისხმევად არის `"."`, თუ ​​გამოტოვებულია.
- `consensus_mode` – კონსენსუსის რეჟიმი რეკლამირებული მანიფესტში. საჭირო; გამოიყენეთ `"Npos"` Sora Nexus საჯარო მონაცემთა სივრცისთვის, ან `"Permissioned"`/`"Npos"` სხვა Iroha3 მონაცემთა სივრცისთვის. Iroha2 ნაგულისხმევად არის `"Permissioned"`.
- `transactions` – გენეზისის ტრანზაქციების სია, რომლებიც შესრულებულია თანმიმდევრობით. თითოეული ჩანაწერი შეიძლება შეიცავდეს:
  - `parameters` – საწყისი ქსელის პარამეტრები.
  - `instructions` – სტრუქტურირებული Norito ინსტრუქციები (მაგ., `{ "Register": { "Domain": { "id": "wonderland" }}}`). ბაიტის ნედლეული მასივები არ მიიღება და `SetParameter` ინსტრუქციები უარყოფილია აქ - დათესეთ პარამეტრები `parameters` ბლოკის მეშვეობით და ნორმალიზება/ხელმოწერა ნება მიეცით ინსტრუქციების ინექცია.
  - `ivm_triggers` – ტრიგერები IVM ბაიტიკოდის შემსრულებლებით.
  - `topology` – საწყისი თანატოლების ტოპოლოგია. თითოეული ჩანაწერი ინახავს თანატოლების ID-ს და PoP-ს ერთად: `{ "peer": "<public_key>", "pop_hex": "<hex>" }`. `pop_hex` შეიძლება გამოტოვდეს შედგენისას, მაგრამ უნდა იყოს წარმოდგენილი ხელმოწერამდე.
- `crypto` – კრიპტოგრაფიული სურათი, რომელიც ასახულია `iroha_config.crypto`-დან (`default_hash`, `allowed_signing`, `allowed_curve_ids`, I18NI00000000620X). `allowed_curve_ids` სარკეები `crypto.curves.allowed_curve_ids`, რათა მანიფესტებმა შეძლოს რეკლამირება, რომელი კონტროლერის მრუდი მიიღებს კლასტერს. ინსტრუმენტები ახორციელებს SM კომბინაციებს: ცხადყოფს, რომ სიაში `sm2` ასევე უნდა შეცვალოს ჰეში `sm3-256`-ზე, ხოლო `sm` ფუნქციის გარეშე შედგენილმა ბილდებმა მთლიანად უარი თქვას `sm2`-ზე. ნორმალიზაცია შემოაქვს `crypto_manifest_meta` მორგებულ პარამეტრს ხელმოწერილ გენეზისში; კვანძები უარს ამბობენ დაწყებაზე, თუ ინექციური დატვირთვა არ ეთანხმება რეკლამირებულ სურათს.

მაგალითი (`kagami genesis generate default --consensus-mode npos` გამომავალი, ინსტრუქციები ამოჭრილია):

```json
{
  "chain": "00000000-0000-0000-0000-000000000000",
  "ivm_dir": "defaults",
  "transactions": [
    {
      "parameters": { "sumeragi": { "block_time_ms": 2000 } },
      "instructions": [
        { "Register": { "Domain": { "id": "wonderland" } } }
      ],
      "ivm_triggers": [],
      "topology": [
        {
          "peer": "ed25519:...",
          "pop_hex": "ab12cd..."
        }
      ]
    }
  ],
  "consensus_mode": "Npos",
  "crypto": {
    "default_hash": "blake2b-256",
    "allowed_signing": ["ed25519", "secp256k1"],
    "allowed_curve_ids": [1],
    "sm2_distid_default": "1234567812345678",
    "sm_openssl_preview": false
  }
}
```

### დათესეთ `crypto` ბლოკი SM2/SM3-ისთვის

გამოიყენეთ xtask დამხმარე გასაღების ინვენტარის და კონფიგურაციის მზა ჩასმის ფრაგმენტის შესაქმნელად ერთ ნაბიჯში:

```bash
cargo xtask sm-operator-snippet \
  --distid CN12345678901234 \
  --json-out sm2-key.json \
  --snippet-out client-sm2.toml
```

`client-sm2.toml` ახლა შეიცავს:

```toml
# Account key material
public_key = "sm2:8626530010..."
private_key = "A333F581EC034C1689B750A827E150240565B483DEB28294DDB2089AD925A569"
# public_key_pem = """\
-----BEGIN PUBLIC KEY-----
...
-----END PUBLIC KEY-----
"""
# private_key_pem = """\
-----BEGIN PRIVATE KEY-----
...
-----END PRIVATE KEY-----
"""

[crypto]
default_hash = "sm3-256"
allowed_signing = ["ed25519", "secp256k1", "sm2"]  # remove "sm2" to stay in verify-only mode
allowed_curve_ids = [1]               # add new curve ids (e.g., 15 for SM2) when controllers are allowed
sm2_distid_default = "CN12345678901234"
# enable_sm_openssl_preview = true  # optional: only when deploying the OpenSSL/Tongsuo path
```

დააკოპირეთ `public_key`/`private_key` მნიშვნელობები ანგარიშის/კლიენტის კონფიგურაციაში და განაახლეთ `genesis.json`-ის `crypto` ბლოკი ისე, რომ იგი ემთხვეოდეს სნიპეტს (მაგალითად, დააყენეთ I18NI0000000-ზე I18NI00000000 `"sm2"`-დან `allowed_signing`-მდე და მოიცავს მარჯვენა `allowed_curve_ids`-ს). Kagami უარს იტყვის მანიფესტებზე, სადაც ჰეშის/მრუდის პარამეტრები და ხელმოწერის სია არათანმიმდევრულია.

> **მინიშნება:** გადაიტანეთ სნიპეტი stdout-ისთვის `--snippet-out -`-ით, როდესაც უბრალოდ გსურთ გამომავალი ვერსიის შემოწმება. გამოიყენეთ `--json-out -` გასაღების ინვენტარის გამოსაცემად stdout-ზეც.

თუ გსურთ ქვედა დონის CLI ბრძანებების ხელით მართვა, ექვივალენტური ნაკადი არის:

```bash
# 1. Produce deterministic key material (writes JSON to disk)
cargo run -p iroha_cli --features sm -- \
  crypto sm2 keygen \
  --distid CN12345678901234 \
  --output sm2-key.json

# 2. Re-hydrate the snippet that can be pasted into client/config files
cargo run -p iroha_cli --features sm -- \
  crypto sm2 export \
  --private-key-hex "$(jq -r .private_key_hex sm2-key.json)" \
  --distid CN12345678901234 \
  --snippet-output client-sm2.toml \
  --emit-json --quiet
```

> **მინიშნება:** `jq` გამოიყენება ზემოთ, ხელით კოპირების/ჩასმის ნაბიჯის შესანახად. თუ ის მიუწვდომელია, გახსენით `sm2-key.json`, დააკოპირეთ `private_key_hex` ველი და გადასვით პირდაპირ `crypto sm2 export`-ზე.

> **მიგრაციის გზამკვლევი:** არსებული ქსელის SM2/SM3/SM4-ზე კონვერტაციისას მიჰყევით
> [`docs/source/crypto/sm_config_migration.md`](source/crypto/sm_config_migration.md)
> ფენიანი `iroha_config` უგულებელყოფისთვის, მანიფესტის რეგენერაციისა და უკან დაბრუნებისთვის
> დაგეგმვა.

## გენერირება და დადასტურება

1. შექმენით შაბლონი:
   ```bash
   cargo run -p iroha_kagami -- genesis generate \
     [--executor <path/to/executor.to>] \
     --consensus-mode npos \
     --ivm-dir <ivm/dir> \
     --genesis-public-key <PUBLIC_KEY> > genesis.json
   ```
`--consensus-mode` აკონტროლებს რომელი კონსენსუსის პარამეტრებს Kagami ხვდება `parameters` ბლოკში. საჯარო Sora Nexus მონაცემთა სივრცე მოითხოვს `npos` და არ უჭერს მხარს ეტაპობრივ ამოჭრას; სხვა Iroha3 მონაცემთა სივრცეში შეიძლება გამოიყენონ ნებართვა ან NPoS. Iroha2 ნაგულისხმევად არის `permissioned` და შეუძლია `npos` დადგმა `--next-consensus-mode`/`--mode-activation-height`-ით. როდესაც არჩეულია `npos`, Kagami ამუშავებს `sumeragi_npos_parameters` დატვირთვას, რომელიც ამოძრავებს NPoS კოლექციონერის fan-out, საარჩევნო პოლიტიკას და რეკონფიგურაციის ფანჯრებს; ნორმალიზაცია/ხელმოწერა აქცევს მათ `SetParameter` ინსტრუქციებად ხელმოწერილ ბლოკში.
2. სურვილისამებრ შეცვალეთ `genesis.json`, შემდეგ გადაამოწმეთ და მოაწერეთ ხელი მას:
   ```bash
   cargo run -p iroha_kagami -- genesis sign genesis.json \
     --public-key <PUBLIC_KEY> \
     --private-key <PRIVATE_KEY> \
     --out-file genesis.signed.nrt
   ```

   SM2/SM3/SM4-მზად მანიფესტების გამოსაცემად, გაიარეთ `--default-hash sm3-256` და ჩართეთ `--allowed-signing sm2` (გაიმეორეთ `--allowed-signing` დამატებითი ალგორითმებისთვის). გამოიყენეთ `--sm2-distid-default <ID>`, თუ ​​დაგჭირდებათ ნაგულისხმევი განმასხვავებელი იდენტიფიკატორის უგულებელყოფა.

   როდესაც იწყებთ `irohad`-ს მხოლოდ `--genesis-manifest-json`-ით (ხელმოწერილი გენეზის ბლოკის გარეშე), კვანძი ახლა ასახავს მის გაშვების კრიპტო კონფიგურაციას მანიფესტიდან ავტომატურად; თუ თქვენ ასევე მიაწოდებთ გენეზის ბლოკს, მანიფესტი და კონფიგურაცია მაინც ზუსტად უნდა ემთხვეოდეს.

- დადასტურების შენიშვნები:
  - Kagami შემოაქვს `consensus_handshake_meta`, `confidential_registry_root` და `crypto_manifest_meta`, როგორც `SetParameter` ინსტრუქციები ნორმალიზებულ/ხელმოწერილ ბლოკში. `irohad` ხელახლა გამოთვლის კონსენსუსის თითის ანაბეჭდს ამ დატვირთვებიდან და ვერ დაიწყებს გაშვებას, თუ ხელის ჩამორთმევის მეტამონაცემები ან კრიპტო სნეპშოტი არ ეთანხმება დაშიფრულ პარამეტრებს. შეინახეთ ისინი `instructions`-დან manifest-ში; ისინი იქმნება ავტომატურად.
- შეამოწმეთ ნორმალიზებული ბლოკი:
  - გაუშვით `kagami genesis normalize genesis.json --format text` საბოლოო შეკვეთილი ტრანზაქციების სანახავად (ინექციური მეტამონაცემების ჩათვლით) გასაღების წყვილის გარეშე.
  - გამოიყენეთ `--format json` სტრუქტურირებული ხედის გადასატანად, რომელიც შესაფერისია განსხვავებისთვის ან მიმოხილვებისთვის.

`kagami genesis sign` ამოწმებს, რომ JSON მართებულია და აწარმოებს Norito-ში დაშიფრულ ბლოკს, რომელიც მზად არის გამოსაყენებლად `genesis.file`-ის მეშვეობით კვანძის კონფიგურაციაში. შედეგად მიღებული `genesis.signed.nrt` უკვე კანონიკური მავთულის ფორმაშია: ვერსიის ბაიტი, რომელსაც მოჰყვება Norito სათაური, რომელიც აღწერს დატვირთვის განლაგებას. ყოველთვის გაანაწილეთ ეს ჩარჩოში გამომავალი. უპირატესობა მიანიჭეთ `.nrt` სუფიქსს ხელმოწერილი დატვირთვისთვის; თუ არ გჭირდებათ აღმასრულებელის განახლება გენეზისში, შეგიძლიათ გამოტოვოთ `executor` ველი და გამოტოვოთ `.to` ფაილის მიწოდება.

NPoS მანიფესტების ხელმოწერისას (`--consensus-mode npos` ან მხოლოდ Iroha2-ის ეტაპობრივი ამოჭრა), `kagami genesis sign` მოითხოვს `sumeragi_npos_parameters` დატვირთვას; შექმენით ის `kagami genesis generate --consensus-mode npos`-ით ან დაამატეთ პარამეტრი ხელით.
ნაგულისხმევად, `kagami genesis sign` იყენებს manifest-ის `consensus_mode`-ს; გაიარეთ `--consensus-mode` მის გასაუქმებლად.

## რა შეუძლია Genesis-ს

Genesis მხარს უჭერს შემდეგ ოპერაციებს. Kagami აგროვებს მათ ტრანზაქციებში კარგად განსაზღვრული თანმიმდევრობით, რათა თანატოლებმა დეტერმინისტულად შეასრულონ იგივე თანმიმდევრობა.

- პარამეტრები: დააყენეთ საწყისი მნიშვნელობები Sumeragi-სთვის (ბლოკის/დასრულების დრო, დრიფტი), ბლოკი (მაქს txs), ტრანსაქცია (მაქს ინსტრუქციები, ბაიტიკოდის ზომა), შემსრულებელი და ჭკვიანი კონტრაქტის ლიმიტები (საწვავი, მეხსიერება, სიღრმე) და მორგებული პარამეტრები. Kagami seeds `Sumeragi::NextMode` და `sumeragi_npos_parameters` დატვირთვა (NPoS არჩევნები, ხელახალი კონფიგურაცია) `parameters` ბლოკის მეშვეობით, რათა გაშვებამ შეძლოს კონსენსუსის ღილაკების გამოყენება ჯაჭვის მდგომარეობიდან; ხელმოწერილი ბლოკი ატარებს გენერირებულ `SetParameter` ინსტრუქციებს.
- მშობლიური ინსტრუქციები: დომენის, ანგარიშის, აქტივების განსაზღვრის რეგისტრაცია/გაუქმება; ზარაფხანა/დაწვა/აქტივების გადაცემა; დომენის და აქტივების განსაზღვრის საკუთრების გადაცემა; მეტამონაცემების შეცვლა; მიანიჭეთ ნებართვები და როლები.
- IVM ტრიგერები: დაარეგისტრირეთ ტრიგერები, რომლებიც ასრულებენ IVM ბაიტიკოდს (იხ. `ivm_triggers`). ტრიგერების შესრულებადი ფაილები წყდება `ivm_dir`-თან შედარებით.
- ტოპოლოგია: მიაწოდეთ თანატოლების საწყისი ნაკრები `topology` მასივის მეშვეობით ნებისმიერი ტრანზაქციის შიგნით (ჩვეულებრივ პირველი ან ბოლო). თითოეული ჩანაწერი არის `{ "peer": "<public_key>", "pop_hex": "<hex>" }`; `pop_hex` შეიძლება გამოტოვდეს შედგენისას, მაგრამ უნდა იყოს წარმოდგენილი ხელმოწერამდე.
- Executor Upgrade (არასავალდებულო): თუ არსებობს `executor`, genesis ჩასვამს ერთი განახლების ინსტრუქციას, როგორც პირველ ტრანზაქციას; წინააღმდეგ შემთხვევაში, გენეზისი იწყება პირდაპირ პარამეტრებით/ინსტრუქციებით.

### ტრანზაქციის შეკვეთა

კონცეპტუალურად, გენეზის ტრანზაქციები მუშავდება ამ თანმიმდევრობით:

1) (სურვილისამებრ) შემსრულებლის განახლება
2) `transactions`-ში თითოეული ტრანზაქციისთვის:
   - პარამეტრის განახლებები
   - მშობლიური ინსტრუქციები
   - IVM ტრიგერების რეგისტრაცია
   - ტოპოლოგიის ჩანაწერები

Kagami და კვანძის კოდი უზრუნველყოფს ამ დალაგებას ისე, რომ, მაგალითად, პარამეტრი მოქმედებდეს იმავე ტრანზაქციის შემდგომ ინსტრუქციებამდე.

## რეკომენდებული სამუშაო პროცესი

- დაიწყეთ შაბლონიდან Kagami:
  - მხოლოდ ჩაშენებული ISI: `kagami genesis generate --ivm-dir <dir> --genesis-public-key <PK> --consensus-mode npos > genesis.json` (Sora Nexus საჯარო მონაცემთა სივრცე; გამოიყენეთ `--consensus-mode permissioned` Iroha2-სთვის ან კერძო Iroha3-ისთვის).
  - მორგებული შემსრულებლის განახლებით (არასავალდებულო): დაამატეთ `--executor <path/to/executor.to>`
  - მხოლოდ Iroha2: NPoS-ზე მომავალი გადასასვლელად, გაიარეთ `--next-consensus-mode npos --mode-activation-height <HEIGHT>` (შეინარჩუნეთ `--consensus-mode permissioned` მიმდინარე რეჟიმისთვის).
- `<PK>` არის ნებისმიერი მულტიჰეში, რომელიც აღიარებულია `iroha_crypto::Algorithm`-ის მიერ, მათ შორის TC26 GOST ვარიანტები, როდესაც Kagami აგებულია `--features gost`-ით (მაგალითად, `gost3410-2012-256-paramset-a:...`).
- დადასტურება რედაქტირებისას: `kagami genesis validate genesis.json`
- განლაგების ნიშანი: `kagami genesis sign genesis.json --public-key <PK> --private-key <SK> --out-file genesis.signed.nrt`
- დააკონფიგურირეთ თანატოლები: დააყენეთ `genesis.file` ხელმოწერილი Norito ფაილზე (მაგ., `genesis.signed.nrt`) და `genesis.public_key` იმავე `<PK>`-ზე, რომელიც გამოიყენება ხელმოწერისთვის.შენიშვნები:
- Kagami-ის „ნაგულისხმევი“ შაბლონი არეგისტრირებს დომენის და ანგარიშების ნიმუშს, ჭრის რამდენიმე აქტივს და ანიჭებს მინიმალურ ნებართვებს მხოლოდ ჩაშენებული ISI-ების გამოყენებით – არ არის საჭირო `.to`.
- თუ თქვენ შეიცავს შემსრულებლის განახლებას, ეს უნდა იყოს პირველი ტრანზაქცია. Kagami ახორციელებს ამას გენერირების/ხელმოწერის დროს.
- გამოიყენეთ `kagami genesis validate`, რათა დაიჭიროთ `Name` არასწორი მნიშვნელობები (მაგ., უფსკრული) და არასწორი ინსტრუქციები ხელმოწერამდე.

## სირბილი Docker/Swarm-ით

მოწოდებული Docker Compose და Swarm ხელსაწყოები ამუშავებს ორივე შემთხვევას:

- შემსრულებელის გარეშე: compose ბრძანება აშორებს გამოტოვებულ/ცარიელ `executor` ველს და ხელს აწერს ფაილს.
- შემსრულებლით: ის წყვეტს შემსრულებლის ფარდობით გზას კონტეინერის შიგნით აბსოლუტურ გზამდე და ხელს აწერს ფაილს.

ეს ინარჩუნებს მარტივ განვითარებას მანქანებზე წინასწარ ჩაშენებული IVM ნიმუშების გარეშე, ხოლო საჭიროების შემთხვევაში განახლების საშუალებას იძლევა.
