# Iroha2 IVM ISO 20022 Compliance: Opcode Set and VM Architecture Design

## Introduction and Goals

Hyperledger Iroha 2 is a permissioned blockchain platform known for its role-based access control and rich asset/account model. To integrate with mainstream finance, Iroha-based systems like Cambodia’s Bakong have adopted ISO 20022 messaging for interoperability. ISO 20022 is a global financial messaging standard covering payments, securities, trade services, foreign exchange (FX), and card transactions – over 700 message types in total. Achieving **full ISO 20022 compliance** in Iroha 2 requires a custom virtual machine (IVM) instruction set that can handle the structure and semantics of all these message types, while respecting Iroha’s architectural principles (domains, permissions, assets, triggers). The key design goals and constraints are:

## Opcode Set Design for ISO 20022 in IVM

To support ISO 20022 within the Iroha Virtual Machine, we introduce a set of low-level **opcodes** (machine instructions) that extend IVM’s base instruction set. These opcodes are grouped by function for clarity. At a high level, we have:

* **Financial Messaging Opcodes:** Instructions to create, manipulate, parse, and serialize ISO 20022 message structures. These allow the VM to construct messages (or read incoming ones) and access nested fields, effectively mapping message data to Iroha’s internal model.
* **Data Encoding/Decoding Opcodes:** Utility instructions for data format conversions required by financial messages – e.g. translating between IVM internal data types and ISO 20022 field formats (dates, numerical formats, identifiers, etc.). This category also covers generic parsing of structured data (like XML or possibly ASN.1/JSON if needed) at a low level.
* **Permission and Security Opcodes:** Instructions enforcing Iroha’s permission model and other security checks during message processing. These opcodes ensure that any state-changing operations (like posting a payment or security transfer) comply with the role-based access control rules and that message handling can be gated by proper authorizations.

In addition to these new categories, the IVM continues to support standard operations (arithmetic, logic, control flow, and **ledger state opcodes** for asset/account manipulation). Many of Iroha’s native “Special Instructions” (such as asset transfers, account creation, etc.) remain available. The new ISO 20022 opcodes complement these, enabling the VM to bridge between external ISO 20022 messages and internal ledger instructions.

Below, we detail each opcode category with tables and explanations:

### Financial Messaging Opcodes

These opcodes are dedicated to handling ISO 20022 message objects and their content. The IVM treats an ISO message as a structured data object in memory, which can be manipulated via these instructions. The design assumes a **stack-based VM** (like many blockchain VMs), where opcodes pop operands from a stack and push results, although we describe the behavior abstractly. Table 1 lists the key financial messaging opcodes:

**Table 1: Financial Messaging Opcodes**

| **Opcode**             | **Description** |
| ---------------------- | --------------- |
| `MSG_CREATE type`      | Create a new ISO 20022 message object of a given **type** on the stack. The `type` operand might be an identifier (string or enum) like `"pacs.008.001.08"` for a customer credit transfer, or `"camt.053.001.04"` for an account statement. Allocates an internal structure with all defined fields. |
| `MSG_CLONE`            | Copy a message object (e.g. to fork or to create a template). |
| `MSG_SET field`        | Set the value of a specified **field** in the current message object. |
| `MSG_GET field`        | Retrieve a field’s value from a message object. |
| `MSG_ADD field`        | Append a new occurrence of a repeating field/segment. |
| `MSG_REMOVE field`     | Remove a field or substructure from the message. |
| `MSG_CLEAR`            | Clear all fields of the message object (reset to default/empty). |
| `MSG_PARSE format`     | Parse a raw message from a given format into an internal message object. |
| `MSG_SERIALIZE format` | Serialize a message object to a raw format (e.g. XML string). |
| `MSG_VALIDATE`         | Validate the message object against ISO 20022 schema rules and business rules. |
| `MSG_SIGN`             | Apply a digital signature or MAC to the message. |
| `MSG_VERIFY_SIG`       | Verify a digital signature on an incoming message. |
| `MSG_SEND channel`     | Validate then serialize the current message and dispatch it through a channel. |

### Data Encoding and Decoding Opcodes

ISO 20022 messages carry a variety of data: numeric amounts, dates, currency codes, identifiers (like IBANs, BICs), code lists, and free-text fields. The IVM provides low-level opcodes to encode or decode these data types to and from internal representations. These **encoding/decoding opcodes** serve two purposes: (1) help parse and format fields when converting ISO messages to internal data, and (2) enforce data format rules (for example, IBAN structure or date format) as part of message validation.
The prototype includes a minimal, constant-time Base64 encoder/decoder with streaming support to allow binary blobs to be attached to messages via `ENCODE_STR`/`DECODE_STR` without pulling in heavy external dependencies.
