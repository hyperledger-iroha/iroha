# Iroha Virtual Machine (IVM) – Updated Architecture Specification

## Introduction & Goals

The Iroha Virtual Machine (IVM) is a 64-bit register-based VM designed for **deterministic smart contract execution** on the Hyperledger Iroha platform, now expanded with features for **parallel execution** and **zero-knowledge (ZK) proof support**. This specification incorporates feedback to improve hardware utilization, concurrency, and proof efficiency. The VM’s design emphasizes consistent results across nodes, even as it exploits modern CPUs for performance. Key goals of this update include:

Compatibility Note
- Kotodama compiles to Iroha Virtual Machine (IVM) bytecode (`.to`). It does not target “risc5”/RISC‑V or any external ISA. Any mention of RISC‑V (e.g., vector length control like `vsetvl`) is an analogy to aid understanding, not a statement of conformance. IVM’s instruction encodings and semantics are IVM‑specific and normative for this project.

Note
- This document is a living specification. Some sections below are still forward-looking, but the current tree already ships deterministic Metal/CUDA helper surfaces with parity/fallback gates, and production Halo2 verifiers are exercised via real IPA proofs (`crates/ivm/tests/zk_open_envelope_determinism.rs`) so determinism and failure handling are covered. Optional backends (Metal, CUDA) are strictly best-effort: builds remain deterministic with scalar/SIMD fallbacks, and tests guard that disabling those features yields correct (non-accelerated) behaviour.
- Acceleration toggles (developer aids, not production feature gates): `IVM_DISABLE_METAL=1` forces scalar/SIMD on macOS builds; `IVM_FORCE_METAL_ENUM=1` enumerates all devices instead of the default. For CUDA, `IVM_DISABLE_CUDA=1` forces CPU paths, `IVM_MAX_GPUS` caps device count, and `IVM_CUDA_NVCC`/`IVM_CUDA_GENCODE` allow PTX build overrides. These toggles never change observable results—only whether the optional backend is attempted.

* **Deterministic Parallelism:** Leveraging multi-core processors for higher throughput **without sacrificing determinism**. The VM can execute independent operations in parallel but will always produce the same final state as a sequential execution, ensuring all network nodes remain in sync.
* **Zero-Knowledge Proof Efficiency:** Enhancing the VM’s optional ZK mode to enable succinct proofs of execution for privacy-preserving contracts. Techniques like *register tagging* and *state Merkle trees* are used to reduce proof sizes by an estimated 20–30%, facilitating scalable private transactions.
* **Hardware-Agnostic Acceleration:** Aligning the VM’s crypto and vector instruction set with concepts from **modern ISAs** (e.g., RISC‑V Vector Extension, ARM SVE) for scalable SIMD operations — as analogies only. The VM can automatically utilize available hardware features (wide vector units, cryptographic instruction sets, hardware transactional memory, etc.) to speed up execution, while providing software fallbacks for portability.
* **Privacy-Oriented Use Cases:** Explicit support for confidential asset transfers and verifiable secret voting. New cryptographic opcodes and proof-friendly features make it easier to implement use cases like **anonymous asset transactions** and **private voting in elections** directly on Iroha, with strong confidentiality guarantees.

## Key Features (Summary)

* **Large Register File with Tags:** 256 general-purpose 64-bit integer registers (`r0`–`r255`) in the current implementation, where `r0` is read-only zero. This expanded register set (up from 32 in earlier drafts) reduces memory traffic and enables efficient parallel and ZK operations. Each register carries a 1-bit *privacy tag* in ZK mode indicating whether it holds secret data, enabling the VM to enforce constraints on confidential information flows. Tagged registers (inspired by ZK-sync’s VM) allow the proving system to differentiate public vs. private values, helping reduce circuit complexity for public data. The instruction encoding leaves headroom for future increases in the architected register count without changing observable semantics.

* **Scalable Vector Extension:** The design calls for a vector/SIMD facility that adapts to the host’s SIMD width. The current implementation already detects the widest available vector width at runtime and falls back to 32‑bit scalar operations when no SIMD is present.

* **Mixed Instruction Encoding:** A compact **16-bit** instruction format for common operations and a full **32-bit** format for others. To accommodate register indices for the full 256-register file (with room to grow) without bloating code size, frequently used instructions that only need a small register range or immediate value use the 16-bit encoding (e.g. operations on `r0`–`r31` or small constants). Less common or more complex instructions use the 32-bit form, which provides wider fields for operands and larger immediates. All instructions are aligned on 16-bit boundaries. This scheme preserves code density for typical cases while still allowing a large register file and rich opcode space.

* **Memory Model:** **Byte-addressable, segmented memory** with enforced alignment and bounds checking. Memory is divided into distinct regions (code, stack, heap, etc.) with defined access permissions. The heap may grow at runtime via a dedicated syscall, extending its limit up to a safe maximum. *Unaligned accesses are uniformly forbidden* in all execution modes – any access to a multi-byte value must be naturally aligned, or the VM will trap (abort execution). This rule simplifies memory access logic and avoids ambiguity between normal and ZK modes. (In earlier designs, unaligned loads/stores were considered for non-ZK mode at the cost of extra interpreter overhead; this update opts for a single strict rule to keep developer experience consistent and proofs simpler.) Out-of-bounds or unauthorized memory accesses also trigger an immediate trap. These measures ensure memory safety and ease of verification in ZK circuits.

* **Deterministic Parallel Execution:** **Multi-core concurrency** is supported at both the instruction and transaction level, with a focus on determinism. The VM can identify independent instructions within a single contract and execute them in parallel (ILP – *Instruction-Level Parallelism*), and it can run multiple smart contract transactions in parallel if their state accesses do not conflict (TLP – *Transaction-Level Parallelism*). A fixed scheduling algorithm and ordered commit mechanism guarantee that parallel execution has the same effect as some well-defined sequential order. In other words, all nodes will apply transactions and instructions in a logically identical sequence, preventing any nondeterministic interleaving of reads/writes. If conflicts or dependencies are detected, tasks are serialized in a predetermined manner. This approach yields significant speedup (empirically **3–5× throughput** on multi-core machines in preliminary tests) while maintaining network consensus on outcomes.

* **Hybrid STM/HTM Concurrency Control:** To implement parallel execution safely, the VM uses a hybrid of **Software Transactional Memory (STM)** and **Hardware Transactional Memory (HTM)**. Whenever possible, the interpreter will utilize HTM support on the host CPU (e.g. Intel TSX instructions) to execute groups of operations or transactions speculatively in parallel, relying on hardware to detect conflicts. If the hardware transaction aborts (due to conflicts or capacity limits), or if HTM is not available on the platform, the VM gracefully falls back to a software-managed transaction (STM) or fine-grained locking. This best-effort use of HTM accelerates low-contention workloads, while STM ensures progress and correctness in all cases. The design prioritizes consistency: the **commit order of transactions is pre-established** (e.g. by transaction index or timestamp) so that even with retries and aborts, the final committed state is deterministic and identical across all nodes. Developers do not need to explicitly handle these concurrency details – it’s managed by the VM runtime.

* **Gas Accounting:** Each instruction has an associated **gas cost** which is charged to the execution budget. The VM tracks gas usage precisely, and if the predefined gas limit is exhausted, execution is halted to prevent denial-of-service. The gas costs for parallel operations are designed to be **additive** and reflect the true total work done (including any aborted retry overhead in concurrency). This discourages abuse of parallel instructions and ensures that even if the interpreter speculatively executes operations, the gas consumption remains deterministic and predictable. Gas for cryptographic and ZK-related operations accounts for their higher computational cost (and proving cost, if in ZK mode).

* **Zero-Knowledge Mode Enhancements:** A special execution mode can be enabled (per contract or per transaction) to facilitate generating a **zero-knowledge proof** of the execution trace. In ZK mode, the VM imposes additional constraints and uses alternate implementations of some operations to make the whole execution arithmetization-friendly:

  * **Deterministic & Constant-Time:** All sources of nondeterminism (e.g. random seeds, timestamps) are fixed or made part of the circuit, and operations run in constant time (no data-dependent timing or memory access patterns) to ease circuit generation.
  * **Field Arithmetic:** Optionally, arithmetic can be performed over a prime field (such as the base field of the Halo2 proving system) to simplify representation in the circuit. Standard 64-bit arithmetic is mapped into field operations with appropriate range checks when needed.
  * **Execution Trace Commitments:** The VM uses **Merkle trees** to commit to state during execution. Notably, the register file and possibly memory pages are each represented by Merkle roots that update as the program runs. Rather than including every register value at every step in the proof, the proving system can include just the root and a short proof for any accessed element, significantly reducing proof data size. This technique, combined with tagged registers to distinguish public constants from private witnesses, yields more compact proofs (estimated 20–30% smaller) by avoiding redundancy in representing values that are public or untouched in the circuit.
  * **ZK-Friendly Instructions:** The instruction set includes operations specifically for use in circuits, such as efficient range checks, cryptographic hash primitives (e.g. Poseidon or SHA-256) optimized for circuits, and an `ASSERT` opcode for enforcing conditions (e.g. equality of a register to a public value) that must hold true in the proven execution. These facilitate writing contracts that will be proven, such as mixers, confidential asset transfers, or voting, by providing building blocks that correspond directly to common ZK proof constraints.

* **Privacy and Anonymity Primitives:** Building on the ZK features, the VM provides built-in support for **confidential assets and private voting**. Example opcodes include zero-knowledge proof verification (to verify a proof within a transaction), cryptographic accumulators or Merkle proof verification (for checking inclusion of a commitment in a state tree), and asset shielding/unshielding instructions (to move assets between public and private pools). These primitives make it easier to implement use cases like **private asset transfers** (where transaction amounts or asset types are hidden on-chain, but correctness is proven) and **private voting in elections** directly on Iroha, with strong confidentiality guarantees.

* **Cross-Platform Compatibility & Extensibility:** The IVM is designed to run efficiently on a wide range of hardware and to be forward-compatible with new extensions:

  * **Dynamic Feature Detection:** The runtime automatically detects available hardware features and optimizes execution accordingly. Developers and deployers do **not** need to manually toggle VM features for specific CPUs. For example, if vector instructions are supported by the host, the VM will execute vector opcodes using the widest available registers; if HTM is present, it will be used for parallel commits. On hardware lacking these, the VM seamlessly falls back to portable techniques (looping scalar operations for vectors, software locks for transactions, etc.). This ensures one build of the VM can adapt to many environments, from cloud servers to IoT devices.
  * **Optional Extensions Without Fragmentation:** All major features (vector SIMD, cryptographic instructions, parallel execution, ZK mode) are integrated into this specification as optional **extensions** that do not break baseline functionality. The base VM can run a simple contract on a basic CPU, while the same VM image will automatically utilize vectors or HTM if present. We avoid requiring custom builds per hardware. For extremely resource-constrained deployments, it is *possible* to compile a subset of the VM (excluding, say, the vector/crypto opcodes to reduce binary size), but this is not needed in general. The modular design allows adding new instructions or extensions in future without affecting existing opcodes or semantics, ensuring longevity of the platform.

## Register Architecture

**General Registers:** The VM provides 256 general-purpose integer registers in the current implementation, each 64 bits wide. These are labeled `r0` through `r255`. By convention, `r0` is a read-only zero register (always contains 0; writes to `r0` are no-ops), which is useful for common operations like comparisons or clearing values without needing a separate constant. The enlarged register file (up from 32 in the previous version) enables the VM to keep more values in fast storage, reducing the need to spill to memory. This is particularly beneficial for cryptographic and batch-processing workloads, as well as for organizing data when exploiting parallel execution (each parallel task or loop iteration can use its own subset of registers). It also aligns well with zero-knowledge proof requirements – a larger register file means more variables can be represented as registers that are committed in a Merkle tree, rather than having to be moved in and out of memory, which simplifies the proving process. The encoding keeps enough bits reserved to allow future extensions of the register file without changing existing programs.

**Register Tags:** In ZK mode, each register has an associated 1-bit *tag* indicating whether it holds a **private (secret)** value or a **public** value. The VM uses these tags at runtime to enforce that certain operations only mix compatible types (e.g. a secret and a public value might require an explicit operation to mix, or might be disallowed to ensure one cannot inadvertently leak a secret via a public output). For example, an instruction might only accept two operands if their privacy tags match, or otherwise it throws a runtime error (or in ZK mode, aborts proof generation). This enforcement can prevent mistakes in smart contracts that could compromise privacy. In terms of implementation, tag checks are designed to be lightweight (often a single conditional check in the interpreter). In proving, these tags help the circuit logic: public values can be treated as constants, saving on constraints, whereas secret values are treated as witnesses that need arithmetic constraints. We will continuously benchmark this tagging approach against other zkVMs (like ZKsync’s mechanism) to ensure the overhead remains low both in runtime and circuit size. If needed, future versions might optimize how tags are represented (for instance, packing multiple tags into one field or using static analysis to eliminate some checks), but the functional model is that every register’s confidentiality is tracked throughout execution.

**Special-Purpose Registers:** Aside from the general registers, the architecture includes a small number of special registers or register-like state:

* **Program Counter (PC):** Holds the address of the next instruction. Not directly accessible to the program (manipulated implicitly by control-flow instructions).
* **Condition Flags:** Instead of a fixed flags register, the IVM uses either specific opcodes for conditional moves and comparisons or designated registers to store boolean results. (The design avoids implicit status flags to simplify analysis and proof modeling. For example, a compare instruction may write `1` or `0` into a register rather than setting a hidden flag.)
* **Vector Registers:** The VM’s vector extension does **not** use a separate fixed register file as in some architectures; instead, vector operations reuse the general registers or a part of them. In effect, a vector instruction treats a group of consecutive general registers as a vector, or uses one register to reference a vector in memory, depending on the operation. This choice was made to keep the architecture simple and unify resource management. For instance, a vector-add instruction could be defined to take `rX` as destination and `rY` and `rZ` as sources, but actually operate on wider values than 64-bit depending on a vector length setting. **Scalable vector length** is handled via a special control register or instruction that sets the active vector length (in elements or bytes) for subsequent vector ops, conceptually similar to RISC‑V’s `vsetvl` (analogy only; IVM is not RISC‑V compatible). The active vector length will be chosen based on hardware capability or a program request, and could be as large as the entire vector that fits in available SIMD registers or as small as needed for the data size. The programmer (or code generator) typically writes vectorized code in a length-agnostic way, and the VM ensures it processes the full data in chunks if the hardware vector length is smaller than the data size.

**Operand Encoding:** With 256 general-purpose registers architected today, instruction formats provide enough bits to specify any register operand (and reserve space for possible future growth). In the full 32-bit instruction format, this supports instructions with up to three explicit register operands (and a few bits left for opcode or modifiers). In the 16-bit compact format, using full-width register indices is not possible; instead, the compact instructions either implicitly operate on specific registers (e.g. certain operations always use `r0` or a small range of regs) or use a reduced register index size (such as 4 or 5 bits, allowing access to, say, registers `r0`–`r15` or `r31` in those encodings). This means that most performance-critical or common instructions can use a small subset of registers and benefit from the 2-byte encoding, whereas instructions needing larger register indices will use the 4-byte encoding. This two-tier encoding strategy ensures that the increase in register count does not overly inflate typical bytecode size. As an example, an ALU operation adding two registers might have a 16-bit form that only encodes the lower 16 registers, and a 32-bit form for arbitrary registers. Assemblers and compilers for IVM will choose the smallest encoding possible automatically.

## Instruction Set and Operation Semantics

The IVM instruction set covers general arithmetic, logic, control flow, memory access, cryptographic operations, and special ZK-oriented opcodes. This update introduces no fundamental changes to most scalar instruction semantics, but expands the set with new categories and ensures all operations are deterministic and (when in ZK mode) either arithmetization-friendly or forbidden. Below we outline major instruction categories and any notable new instructions:

* **Arithmetic and Logic:** Standard operations (ADD, SUB, MUL, DIV, AND, OR, XOR, NOT, shifts, etc.) on 64-bit integers. These behave as pure 64-bit two’s complement arithmetic, wrapping on overflow (in deterministic mode) unless specified otherwise. In ZK mode, overflow behaviors may be constrained or cause a failure if they would exceed field ranges (to maintain consistency with field arithmetic). We include conditional moves and comparisons (e.g., CMOV, CMP) that produce boolean results rather than relying on implicit flags.

* **Memory Access:** LOAD and STORE instructions for reading and writing memory. Addresses are 64-bit and interpreted relative to specific segments (e.g., stack or heap pointers). The instructions ensure the address is aligned to the data size (e.g., a 64-bit load must have an 8-byte aligned address) – misaligned access triggers a trap. In ZK mode, memory accesses must also be deterministic and index calculations are typically done in such a way that they can be translated into circuits (e.g., using fixed offsets or simple linear expressions to avoid data-dependent pointer chasing in secret). The VM may include specialized opcodes for copying memory (MEMCPY) or zeroing memory, which can be optimized by the interpreter.

* **Control Flow:** Unconditional jumps, conditional branches (based on a register value or comparison result), function calls and returns. The control flow is strict and deterministic; there are no asynchronous events or interrupts. All control flow changes are part of the instruction stream. In ZK mode, secret-dependent conditional branches (i.e., branching on a private value) are disallowed or handled via special means (such as executing both paths and masking the results, or by an `ASSERT` to enforce a condition instead of branching on it) to avoid ambiguity in the execution trace. The updated spec adds that contracts intended for ZK mode should avoid data-dependent loops or branches on secret data, or use provided patterns to unroll or enforce conditions, since these are difficult to represent in ZK circuits.

* **Cryptographic Instructions:** The VM includes built-in instructions for cryptographic primitives commonly needed in smart contracts and ZK applications. Examples are:

  * Hash functions: e.g. `SHA256BLOCK` (compress one 512-bit block with SHA-256), `BLAKE2s` or a Poseidon hash instruction for hashing field elements.
  * Signature primitives: e.g. an opcode to verify an Ed25519 signature or ECDSA signature given public key and message (this would consume a lot of gas but provides a native way to verify within the VM).
  * Encryption: e.g. `AESENC` for a round of AES encryption (if needed for certain applications, though full symmetric encryption can also be done by contracts).
  * The current Rust implementation supports `SHA256BLOCK`, `SHA3BLOCK` (deterministic minimal form), `POSEIDON2`, `POSEIDON6`, `ECADD`, `ECMUL_VAR`, `PUBKGEN`, `VALCOM`, `AESENC`, `AESDEC`, `BLAKE2S` (deterministic minimal form), `ED25519VERIFY`, `ED25519BATCHVERIFY`, and `PAIRING`. Opcodes such as `PARBEGIN` and `PAREND` are reserved and not yet implemented; `SETVL` is supported for logical vector-length selection.

  These instructions are typically implemented to take advantage of hardware accelerators when available (for instance, `SHA256BLOCK` might use Intel SHA extensions if present, or a software routine if not). They are considered optional in the sense that not all contracts will use them, but they are part of the standard VM arsenal to support privacy and authentication use cases. Gas costs for these are set high relative to basic operations, reflecting their complexity. Note that while these are provided, higher-level protocols (like a full shielded transfer circuit) will use a combination of these primitives and ZK-oriented logic.

* **Vector and SIMD Instructions:** Under the scalable vector extension, a set of vector operations is available to accelerate data-parallel tasks:

  * Basic integer vector ops: e.g. `VADD64 vr, va, vb` which adds two vectors of 64-bit integers element-wise (the length of the vectors is determined by the current vector length setting and hardware capabilities). Similar ops exist for subtraction, bitwise AND/OR/XOR, shifts, etc., possibly also for 32-bit and 16-bit element sizes.
  * Reduction and scatter/gather ops: e.g. summing all elements of a vector, or gathering elements from memory given a vector of addresses (with certain restrictions to maintain determinism).
  * Cryptographic uses: small-block parallel operations like processing multiple hash blocks at once, or applying a cipher round to multiple blocks in parallel.

  All vector instructions are deterministic; they operate on a well-defined number of elements (the program can set a maximum elements count, and the VM will execute in chunks if needed to cover all elements on hardware with shorter vectors). For example, if a vector operation is meant to process 128 elements and the CPU can only process 32 at a time, the VM will perform 4 iterations behind the scenes – the effect is identical to having a single logical 128-element vector processed. This guarantees that a vector computation yields the same result everywhere, with only performance varying. We advise developers that performance scaling may not be linear on all platforms (e.g., a high-end server with 512-bit vectors might run a certain vectorized loop much faster than a mobile device with 128-bit NEON vectors, even though both will produce correct results). The reference implementation ensures fairness by always completing the full operation; it does not, for instance, use different algorithms on different hardware – just different degrees of parallelism.

* **Parallel Execution Primitives:** To facilitate the new parallel execution engine, a few new pseudo-instructions or bytecodes are defined. These include:

  * Markers or hints for independent instruction groups (the VM’s compiler or static analyzer might insert these). For example, a no-op instruction that indicates the start or end of a group of instructions that have no dependencies on each other, which the scheduler could run in parallel. (At runtime, the interpreter or JIT may ignore hints if it dynamically determines dependencies, but they can aid static scheduling.)
  * Transaction demarcation opcodes (for future use): If not already inherent in the execution model, there could be an opcode to start a new transaction context or to label a transaction’s read/write set for conflict detection. In the current design, conflict detection is handled outside of the VM at the transaction executor level (since read/write sets can be derived from VM state accesses), so no explicit instruction is needed in the bytecode. But we mention it for completeness of design – the system as a whole knows each contract call’s reads and writes.
  * Atomic operations: In some concurrency models, one might include atomic memory ops or synchronization primitives. However, for IVM’s deterministic parallelism, we do *not* expose locks or atomic instructions to the smart contract code. All synchronization is handled under-the-hood by the VM scheduler. Smart contracts logically run in isolation or in a predetermined sequence, so exposing atomics could actually break determinism (and would be hard to model in ZK). Thus, the VM does not provide typical threading primitives to contracts.

## Deterministic Parallel Execution Model

Parallel execution in IVM is achieved by the runtime **without any changes to smart contract code** – it is an implementation strategy that remains invisible to the contract logic (which is written as if it runs sequentially). Here we outline how the VM can execute things in parallel and still produce a deterministic result.

**Instruction-Level Parallelism (ILP):** Within a single transaction (smart contract execution), many instructions can be analyzed for dependencies. For instance, independent arithmetic operations or data loads that don’t depend on each other’s results can potentially be executed simultaneously. The IVM interpreter (or JIT compiler) can perform a data-flow analysis to build a *dependency graph* of instructions. Nodes in this graph are instructions and edges represent data dependencies (register or memory). Using this graph, the VM identifies sets of instructions that have no incoming edges from each other (i.e. can run in parallel). These can be dispatched to different threads or processed in parallel on multiple functional units. The **scheduler** uses a fixed algorithm to decide the order in which independent instructions are issued and results are committed, such that if there is any ambiguity, it defaults to the original program order. This means even if two independent instructions (`A` and `B`) run in parallel, the effect is as if one occurred right before the other in the original sequence order (the difference is only in timing, which is not observable in the VM since we eliminate timing side-effects). If the code has branches or loops, the dependency graph is built per basic block or through speculative execution of both paths (ensuring no divergence in ZK mode). The end result is that ILP can speed up execution on multi-core or pipelined hardware, but the **logical outcome (register and memory state)** is identical to sequential execution. The VM’s parallel engine takes care to handle corner cases like instruction-induced traps or gas exhaustion in a deterministic way (e.g., if two parallel instructions both could trap, the one that appears first in program order will take precedence in causing an abort).

**Transaction-Level Parallelism (TLP):** At the level of the blockchain, multiple transactions in the same block (or pool of pending transactions) can be executed concurrently if they do not conflict. Conflict typically means that two transactions access the same state variable or asset, with at least one writing to it. IVM’s environment (the node software) utilizes known read/write sets for each transaction to decide which transactions can be grouped. For example, if transaction X and Y operate on disjoint sets of accounts or tokens, they can be executed in parallel. Our design extends the transaction executor to handle this: it may start multiple contract interpreter instances in parallel threads. Each instance runs a transaction in isolation (on a snapshot or copy of state or with buffered writes). If any data conflicts are detected (by monitoring memory addresses or keys accessed), one of the transactions will be designated to abort/rollback and retry after the other completes. We use a deterministic conflict resolution strategy – e.g., transactions are ordered by their position in the block, so a later transaction that conflicts with an earlier one will always be the one that rolls back. This ensures all nodes resolve conflicts in the same way. Once all parallel groups finish, their results are merged in the predefined order to produce the final state for the block. If the underlying hardware supports HTM, the VM could execute each transaction group inside a hardware transaction, letting the CPU detect any conflicting memory writes; if a conflict abort occurs, the software knows to fall back to the sequential resolution as described. On hardware without HTM, a software lock or validation mechanism (STM) is used to achieve the same effect. In either case, the **commit order is deterministic** and matches the block order, so the combined effect is as if each transaction ran one-by-one in that order (except faster in the non-conflicting case).

**Hybrid HTM/STM Approach:** We leverage **Hardware Transactional Memory** to optimistically run parallel tasks and detect conflicts with minimal overhead. For instance, if two transactions are running in parallel, each can be wrapped in a hardware transaction (`XBEGIN`/`XEND` on x86). If they truly don’t conflict, both hardware transactions will commit successfully, making the parallel execution effective. If a conflict occurs (both write to the same memory cache line, etc.), at least one transaction’s HTM will abort. On abort, our runtime will fall back to a Software Transactional Memory approach: typically, this means acquiring a lock or otherwise preventing concurrency and re-executing the aborted transaction (or both transactions sequentially) to ensure completion. Because of the deterministic priority rule (the one with higher priority continues, the other retries later), the outcome remains the same on all nodes. We are aware that HTM is a best-effort mechanism — it can also abort for reasons unrelated to logical conflicts (like cache capacity limits, interrupts, or unsupported instructions in the transaction). These spurious aborts could vary across processor models, which is a potential source of nondeterministic timing. However, our design treats an HTM abort simply as a hint to fall back; the exact moment or number of retries doesn’t affect the final state, only the performance. To maintain fairness and predictability, we will **benchmark high-conflict scenarios** and possibly introduce thresholds (e.g., if a certain transaction aborts more than N times under HTM, the system might switch to STM for that batch entirely) to avoid performance outliers. In essence, HTM is used to speed up conflict-free cases, and in worst-case it gracefully degrades to pure STM. This hybrid method has been shown to potentially improve throughput significantly while still guaranteeing the same logical result as a sequential execution.

**Ordered Commit and Isolation:** A crucial part of parallel execution is how and when results become visible. In IVM, each parallel task (be it an instruction or a whole transaction) operates on either separate copies of state or within an isolated transaction context. When tasks complete, their side effects (register updates, memory writes, emitted logs, etc.) are collected. The VM then **commits these side effects in a predetermined global order**. For ILP within one transaction, this commit order is simply the original program order of the instructions. For TLP among transactions, it could be the sorted order of transaction IDs or their order in the block. Commit involves merging changes into the main state one task at a time according to that order, which may involve checking that the state hasn’t changed in ways that violate assumptions (for safety, though if conflicts were handled, it should be fine). By deferring the visible effects until commit, we ensure that even if tasks ran truly concurrently, the outside observer (and the state) sees them as if one-by-one. This model makes reasoning about state straightforward and is friendly to generating proofs (we could even generate a proof per transaction and merge them knowing the final state is as if sequential). It also means that if a task fails (traps or aborts) midway, it can be cleanly rolled back without partial side effects visible, since nothing was committed yet – the VM can then handle the trap or retry logic appropriately.

**Determinism and Replay:** The parallel execution engine is constructed so that if you log the sequence of commits or the schedule decisions, they will be identical every run given the same initial state and transactions. For additional assurance, one could imagine running the same block in a single-threaded mode (for testing or verification) and comparing state with the parallel run – they will match exactly. This determinism is vital for a blockchain environment: it prevents consensus issues and makes the system amenable to auditing and ZK proving. In ZK mode specifically, parallelism is largely a performance consideration (since the proof ultimately has to show a valid sequential execution trace). The VM might decide not to parallelize certain things in ZK mode because it doesn’t change the proof outcome, but any parallelism that does occur must yield the same trace as a sequential one (and indeed we could have the prover generate the trace sequentially while the actual node executed in parallel for speed). In summary, determinism is non-negotiable, and all parallel scheduling is a behind-the-scenes optimization that does not alter the logical order of operations.

## Zero-Knowledge Proof Integration

When ZK mode is enabled, the IVM becomes a **zkVM**, producing cryptographic proofs of contract execution that can be verified on-chain or by third parties. This section details how the design choices help in producing efficient proofs:

**Halo2 Proving System:** The IVM targets the Halo 2 proving system (a variant of PLONK) to generate and verify proofs. Halo 2 is a zero-knowledge proof system with no trusted setup and supports recursive proof composition (useful for rollups or proving multiple transactions succinctly). In practical terms, the VM’s arithmetic and boolean logic operations are mapped to equivalent constraints in a Halo2 circuit. For example, a 64-bit addition in the VM would be represented as a field addition plus constraints to ensure the result fits in 64 bits (overflow handling). The choice of Halo2 influences some design aspects: field operations modulo a large prime (~$2^{254}$) are naturally efficient, whereas bit operations or large integers require additional constraints. The VM spec thus tries to make typical operations align with field-friendly computations when in ZK mode (for instance, by recommending using mod-$2^{64}$ arithmetic only where necessary, and otherwise using field ops).

**Trace Commitment via Merkle Trees:** One of the challenges in ZK is that the entire execution trace (all intermediate states of registers and memory after each instruction) might need to be somehow communicated or committed to, in order to prove that the final output is correct. Rather than explicitly including every register value in the proof (which would be huge for long executions), IVM uses Merkle trees to compress this information:

* **Register State Tree:** At any point, the 256 general-purpose registers can be seen as leaves of a Merkle tree. The VM can maintain a Merkle root representing the current register state. Each instruction that writes to a register will update the corresponding leaf and thus change the root. In the proof, the prover will provide the Merkle path for any register value that is read or written, showing that it was correctly updated. The verifier only needs to track the root, which is much smaller than all registers. By the end of execution, the proof will convince the verifier that there exists some sequence of states (authenticated by these Merkle roots) that led to the final state, without enumerating them all. This drastically reduces the data size in the proof, at the cost of additional Merkle path checks (which are relatively cheap in circuits, especially if using efficient hash gadgets).
* **Memory State Tree:** Similarly, memory (or at least certain memory regions like persistent storage or contract memory) can be committed via a Merkle tree. Not every memory access needs to be proved if we commit to the state at certain points (like pre- and post-execution). However, for full transparency, the VM can also prove each memory read/write by showing inclusion in a memory Merkle tree. The design leans towards using such authenticated data structures for any state that crosses the VM boundary, ensuring that a proof can be succinct (only roots are in the public proof input) while still guaranteeing consistency.

**Public vs Private Inputs/Outputs:** The register tagging mechanism helps separate public inputs (those known to the verifier or all blockchain observers) from private inputs (known only to the party generating the proof). In practice, public inputs could include things like transaction plaintext data that’s meant to be public, or block data like timestamps if used. Private inputs would be secret witness data (e.g., a secret key or a hidden amount). The VM can be initialized with certain registers tagged as containing public data and others as private. The instructions that consume or produce data will propagate tags, and by the end, we know which parts of the state are public outputs vs. private. The proof will reveal public outputs (so they can be used on-chain, e.g., “transfer 50 tokens to account X” might be public who the recipient is and that *some* amount was transferred, while keeping the amount secret, in which case maybe the recipient is public output and amount is private output committed to). The spec ensures that if something is supposed to remain private, it never inadvertently becomes a public output without an explicit operation (like a reveal or commit instruction). This model aligns with other zkVMs and provides clarity for developers about which data stays hidden.

**ZK Mode Restrictions:** To maintain soundness and efficiency, not all VM features are available or advisable in ZK mode:

* Self-modifying code or dynamic code loading is disallowed in ZK mode (and generally not allowed in IVM at all) because it complicates static circuit generation.
* Floating point or nondeterministic operations (like random number generation) are not allowed unless replaced by deterministic equivalents (e.g., a random oracle modeled via a hash function).
* The parallel execution model in ZK mode is conceptually sequential in the proof. The actual node might execute in parallel for speed, but the proof will still go instruction-by-instruction. We ensure that any parallel optimization does not violate the ability to linearly enumerate the execution trace for the prover.
* Certain cryptographic instructions may not be used in ZK mode if they cannot be efficiently arithmetized (for instance, a complicated algorithm that isn’t circuit-friendly). In such cases, alternative approaches or gadgets are provided. However, primitives like hash functions have known circuits, so they are supported albeit with high gas costs reflecting the proving effort.

**Use Cases Enabled:** With these ZK enhancements, IVM can directly support advanced use cases:

* *Private Asset Transfers:* A contract can implement a shielded token, where balances are stored in commitments (in memory or as UTXO-style records) and transfers are done by proving knowledge of secrets opening previous commitments and creating new ones. The VM provides the cryptographic hash instructions to manipulate commitments and the ZK mode to prove correctness of the transfer (no double-spend, sum balances, etc.) without revealing amounts or identities.
* *Private Voting:* Using homomorphic tally or reveal of votes with ZK proofs, a voting smart contract can accept encrypted votes. Each vote’s validity (e.g., voter is eligible, vote is well-formed) can be proven by a zkSNARK that runs within the VM context. The tally can then be revealed either by another proof or at least by ensuring each vote was counted correctly. The VM’s features like Merkle trees (to commit to voter lists or vote records) and on-chain proof verification instructions allow these complex processes to be implemented mostly at the smart-contract level.
* *General zkApps:* Developers can write general programs that execute in the VM and produce a proof, similar to how one would use Arbitrum Nitro or StarkNet, but here within Iroha. The advantage of integrating it at the VM level is that the same code can run either normally or in ZK mode with minimal changes, and the platform’s security (gas, determinism) covers both.

## Cross-Platform Support and Fallbacks

One of the strengths of IVM’s design is **hardware abstraction**: smart contracts are written once and can run on any compliant node regardless of the underlying hardware differences. To achieve this, the implementation uses a combination of runtime feature detection and portable defaults:

* **Auto-Selection of Optimizations:** Upon initialization, an IVM instance queries the host environment for supported CPU features (using CPUID on x86 or equivalent on ARM, etc.). It detects things like supported instruction sets (AVX2, AVX-512, SVE length, AES-NI, SHA extensions, HTM support, number of cores). Based on this, the VM enables or tunes certain internal behaviors. For example, if wide vectors are present, it will process more elements per vector instruction; if only narrow SIMD is present, it might loop more. If Intel TSX or ARM transaction instructions are present, it will attempt hardware transactions for parallel sections; if not, it will use locks or other software methods. This process is entirely automatic – developers do not need to compile different versions or toggle configuration flags for different machines. The **goal is for the same bytecode** to run optimally on a high-end server and correctly (just possibly slower) on a low-end device.
* **Metal GPU Acceleration on macOS:** The VM automatically checks for an available Apple Metal device at runtime and offloads all vector operations to it when present. If no compatible GPU is found, or on non-macOS platforms, the software fallback is used transparently. This provides an extra performance boost on Apple hardware without affecting determinism.
* **CUDA GPU Acceleration:** When compiled with the optional `cuda` feature and identical NVIDIA GPUs are present, hashing and vector instructions run on deterministic CUDA kernels. Nodes may disable this at runtime via `IVM_DISABLE_CUDA` or limit devices with `IVM_MAX_GPUS`. Results remain bit-for-bit identical to CPU execution.
* **Performance Portability:** While performance will vary, we ensure no catastrophic performance cliffs. The interpreter and JIT (if used) are optimized to use baseline algorithms that are efficient even without special hardware. For instance, cryptographic opcodes have highly optimized pure-Rust implementations (possibly using algorithmic optimizations or even GPU offloading in future) so that even if CPU lacks AES-NI, the AES operation will run in a reasonable time in software. Similarly, if the vector extension is not available, the operations simply execute in a scalar loop form. This means the **VM’s correctness and basic efficiency do not depend on any advanced hardware**, they only get better with it.
* **Consistency Across Nodes:** In a distributed network, different nodes might have different CPUs. It’s crucial that this does not lead to divergent behavior. Because the VM’s logic is deterministic and does not depend on actual wall-clock time or true concurrency, even if one node executes slower and another faster, they will come to the same final state given the same inputs. The faster node might finish computing a block sooner, but it cannot commit results until consensus allows, so any speed advantage just means more idle time or ability to handle higher load, not a divergence in results. We avoid any construct where a hardware feature could produce a different numerical result; for example, floating-point math (which can yield slightly different results on different architectures) is avoided entirely or standardized through integer or fixed-point alternatives.
* **Documentation of Differences:** We will document which instructions or patterns benefit from specific hardware so developers and auditors are aware. For example, a note might say: "This vector instruction will execute 4× faster on a 256-bit SIMD machine compared to a 64-bit only machine." However, developers do not need to write different code; it’s more for understanding performance. If certain hardware features are particularly rare or optional (like HTM), the documentation might also explain how the VM behaves if those are absent, to assure that it only impacts throughput, not correctness or finality.
* **Extensibility:** The modular design allows new instructions or features to be added as extensions. For instance, if a new cryptographic algorithm becomes popular (say Poseidon hash, or a post-quantum signature), a future version of the spec can introduce new opcodes for those. The register-tagging and Merkle commitment approach is also extensible to new forms of state (e.g., if we wanted to also commit to the call stack or logs in the ZK proof, we could do so in a future update). Critical extensions go through governance and require network-wide activation so contracts only rely on features every validating node supports.

## Reference Implementation Notes (Rust)

The reference implementation of IVM is written in Rust, chosen for its safety and performance. Below we outline how some of the spec features are realized in the Rust implementation, highlighting any important details for developers and contributors working on the VM codebase:

* **Interpreter Structure:** The core of the interpreter is an execution loop that fetches, decodes, and executes instructions. In the updated implementation, this loop can dispatch work to worker threads when parallel execution is possible. We use Rust’s standard library and community crates to manage concurrency (e.g., threads via `std::thread` or a threadpool like Rayon for ILP, and async tasks or scoped threads for TLP at the transaction scheduler level). The interpreter ensures thread safety using synchronization primitives (`Arc`, `Mutex`, etc.) when accessing shared state. Key data structures include a thread-safe work queue for ready-to-run instruction groups and a mechanism (like channels or condition variables) for threads to signal completion and readiness to commit.

* **Dependency Graph & Scheduling:** In Rust, we implement the dependency analysis either as a static preprocessing pass or on the fly. A simple approach uses an adjacency list representing which registers or memory locations each instruction reads and writes. We then perform a topological sort that groups independent instructions. This is done per basic block to avoid complexity across branches. The scheduler might be a custom piece of code or leverage Rayon’s ability to spawn tasks and join them deterministically. We have to be careful to impose an order on commits – one strategy is to assign a sequence number to each instruction equal to its original program position, and have each thread log its results along with that number. A coordinator thread (or the main thread) then waits for threads and applies results in increasing sequence order. Rust’s ownership model helps here: each worker thread gets a clone of the state or exclusive access to certain registers such that data races are avoided by design.

* **Transactional Memory in Rust:** For HTM on x86, there is no direct stable Rust API, but we can use `core::arch::x86_64::_xbegin()` and `_xend()` intrinsics (nightly or via a crate) to interact with Intel TSX. We’ll wrap those in a safe abstraction that attempts a hardware transaction for a critical section of code. For example, when executing a set of transactions in parallel, we could do:

  ```rust
  if htm_supported {
      if _xbegin() == _XBEGIN_STARTED {
          // perform transaction operations
          let res = execute_transaction(tx);
          // if no conflict/abort:
          _xend();
          res
      } else {
          // abort path
          execute_transaction_with_locks(tx)
      }
  } else {
      execute_transaction_with_locks(tx)
  }
  ```

  This is a simplified sketch – in practice, if an abort happens, we might get an error code to distinguish a conflict vs. other abort reason. We will implement exponential backoff or give up after a few tries to avoid live-lock. The STM fallback (`execute_transaction_with_locks`) would use something like acquiring a global lock or versioning system to run the transaction exclusively. All of this is encapsulated so that the higher-level scheduler just calls an `execute(tx)` which internally decides to use HTM or STM. Ensuring determinism, as discussed, comes from always resolving conflicts in the same order (the code that detects conflict will choose the victim based on transaction ordering and not on thread race conditions – e.g., using a sorted list of running transactions and always aborting the one with the larger index first).

* **SIMD in Rust:** We take advantage of Rust’s powerful support for SIMD via crates like `packed_simd` or stable `std::arch` intrinsics. For each vector opcode, we will have multiple implementations selected at runtime:

  * A scalar fallback (e.g., a simple loop in Rust that processes one element at a time) which is used if no SIMD is available.
  * Possibly a medium-width vector path (128-bit) and a wide path (256 or 512-bit), depending on the target. We can detect at runtime if AVX2 or AVX-512 is available using `is_x86_feature_detected!` macro, and similarly for Neon on ARM. Once detected, we can dispatch to the appropriate function pointer or branch.
  * Alternatively, we may use Rust’s portable packed_simd which can abstract over some differences, but often specific intrinsics yield better performance.
  * For example, the `VADD64` instruction’s implementation might look at a global config like `VECTOR_WIDTH` (set at startup) which could be 1 (scalar), 2 (128 bits means two 64-bit lanes), 4 (256 bits), or 8 (512 bits). Then it will call an optimized routine that adds in chunks of that many lanes.
  * We ensure that the results are the same regardless of the width setting. We test vector operations heavily to make sure that splitting work into chunks yields identical outcomes to doing it all at once (which it should, by design).
  * If an operation is larger than the hardware supports (say the contract wants to process 1000 elements and we only have 8 lanes hardware), our loop will iterate (1000/8 = 125 iterations, plus a remainder if not divisible). Each iteration uses SIMD and moves the pointer. This is all invisible to the contract.

* **Cryptographic Ops Implementation:** For cryptographic instructions, we plan to integrate well-vetted libraries. For instance:

  * `SHA256BLOCK` can use Rust’s `sha2` crate for portability, and if the CPU has SHA-NI, we could use the `_mm_sha256rnds2_epu32` intrinsics internally for speed. We will hide this behind a feature detection as well.
  * Signature verification might use the `ed25519-dalek` or `ring` crate, which are optimized in pure Rust or assembly.
  * Where applicable, we might use GPU acceleration (OpenCL/CUDA via Rust frameworks) for heavy ZK-proof generation tasks in future, but currently, our scope is CPU.
  * All cryptographic code will be constant-time (no data-dependent branches) to avoid any timing nondeterminism.

* **Memory and Safety:** Rust’s safety guarantees help ensure we don’t have buffer overflows or invalid memory access in the VM implementation. We represent the VM memory as a vector/array of bytes with bounds, and all accesses are bounds-checked (the performance cost is acceptable given gas accounting is usually dominant). For alignment, each load/store will have an assertion or check like `assert!(addr % size == 0)` in debug mode, and in release it might be an if that triggers a trap on failure. Because unaligned is not allowed, we can also confidently use pointer aliasing for speed when needed (knowing alignment is correct).

  * We also maintain separate segments as distinct slices or offsets, so that an access in one segment cannot accidentally read data from another.
  * The Rust implementation will likely have an enum or struct for different segments and methods to translate a segment+offset to an index in the main memory vector, performing the necessary bound checks.
  * Common failure paths (like misaligned accesses or permission errors) are marked with branch prediction hints so the CPU can fast‑path the typical case where checks succeed.

* **Gas Metering:** We incorporate gas metering by decrementing a gas counter on each instruction. In Rust, this is just a numeric decrement and compare. We must be careful to account for multi-step operations: if an instruction internally does a lot of work (like a vector loop or a big cryptographic operation), we either charge up front for the worst case or charge incrementally. Typically, we will charge up front based on input sizes. For instance, a vector add on N elements might cost `N * base_cost` gas. If N is huge and we chunk it, we ensure that if it runs out of gas mid-way, we handle that gracefully (perhaps by checking gas in each chunk iteration and aborting if exhausted). This way, gas usage remains deterministic and tied to actual work.

* **Testing and Benchmarks:** The Rust implementation will be accompanied by extensive tests:

  * **Functional tests** to ensure the VM produces correct results for all instructions, including edge cases (max integer values, misalignment traps, etc.), and that the semantics in normal vs ZK mode align where expected.
  * **Determinism tests** where we run the same workload with different numbers of threads, or with HTM enabled vs disabled, and confirm the final state is identical and any logged sequence of commits is the same.
  * **Performance benchmarks** for parallel execution (comparing single-thread vs multi-thread speedup on various core counts), for cryptographic operations (comparing against native libraries to ensure we are in line), and for ZK proof-related features (measuring the size of the trace or number of constraints for sample contracts). We specifically will simulate *high-conflict scenarios* to tune the hybrid HTM/STM logic – e.g., 10 transactions all contending on one key, to see that the system gracefully falls back to sequential and doesn’t thrash with repeated aborts. These tests help us choose parameters like how many HTM retries to attempt before giving up.
  * **Compatibility tests** on different hardware: we will run the VM on machines with and without TSX, with different vector widths, etc., to ensure our feature detection and fallback logic works as expected. The output of computations should never differ, and performance should scale in the anticipated way.

* **Rust Crate Design:** The IVM might be packaged as a Rust crate (`iroha-vm` for example). Its structure could have modules like `decoder` (for instruction decoding), `executor` (for the main loop and parallel scheduler), `crypto` (for cryptographic ops), `zk` (for ZK mode specifics like circuit generation hooks or integration with Halo2 API), and so on. We keep the code modular for maintainability. Unsafe Rust will be kept to a minimum (perhaps only in parts interfacing with HTM intrinsics or doing manual vector operations for performance). Any unsafe code will be thoroughly reviewed and isolated.

* **Rust Implementation of ZK Mode:** As of now, generating an actual proof (Halo2 circuit) from a VM execution is a separate concern from just running the VM. Eventually, we aim to have an **execution trace collector** in the Rust implementation that can log all relevant operations and state changes during ZK mode execution, which can then be fed into a circuit generator. We might integrate with existing ZK VMs or circuits (for example, leveraging the circuit design from DarkFi or an adaptation of ZK-sync’s circuit). The immediate spec, however, focuses on the *constraints* the VM must obey to be provable, rather than the full proving implementation. So the Rust VM when in ZK mode might run with a special flag that causes it to record hash digests of states, or interact with a proving backend. This aspect will evolve as we integrate proof generation, but it does not affect the external behavior of the VM except for performance and the additional data (proof) output.

In summary, the Rust implementation will realize the spec by combining modern system programming techniques with careful adherence to the determinism and security requirements of a blockchain VM. Developers working on the VM should keep in mind both the low-level efficiency (using hardware optimally) and the high-level guarantees (safety, determinism, consistency with proofs). We will continue to refine the implementation guided by real-world testing, and any insights (for example, from benchmarking IVM against other VMs in the space) will feed back into future spec improvements.

#### CUDA Poseidon/BN254 Kernel ABI (WP1-A)

This subsection pins down the ABI between the Rust runtime (`GpuManager`) and the first CUDA kernels so follow-on work (WP1-B–WP1-D) can focus purely on implementation. The scope covers Poseidon2/Poseidon6 permutations and BN254 field add/sub/mul.

##### Shared layout and launch rules

- Field elements use the canonical BN254 representation: four little-endian `u64` limbs matching `bn254_vec::FieldElem` (`[u64; 4]`). Kernels must treat every limb as an unsigned integer and perform reductions modulo `0x30644e72e131a029b85045b68181585d2833e84879b9709143e1f593f0000001`.
- Batches are stored in **array-of-struct** form. A Poseidon2 state occupies 3 elements (12 limbs), Poseidon6 uses 6 elements (24 limbs), and BN254 helpers consume one element (4 limbs). Kernels receive both `state_stride_words` and `batch_len` so padding/alignment tricks stay deterministic across hosts.
- Round constants and MDS matrices are flattened in row-major order (round-major for constants, row-major for MDS) and uploaded to device memory for each launch. The kernel parameters receive raw pointers so the CUDA code never needs to re-derive constants.
- The runtime launches up to 8 GPUs. `task_id = (call_depth << 32) | instruction_index` seeded via the interpreter is hashed and reduced to `[0, device_count)` so every node picks the same GPU for a given instruction stream.
- All kernels return a `KernelStatus` struct in device memory (`status_ptr`). `status_ptr->code == 0` indicates success; any non-zero value disables the CUDA backend at runtime and falls back to SIMD/scalar paths.

##### Poseidon kernels

```
extern "C" __global__ void poseidon2_permute_kernel(
    uint64_t* states,
    uint32_t state_stride_words,
    uint32_t batch_len,
    uint32_t domain_tag,
    const uint64_t* round_constants,
    const uint64_t* mds_matrix,
    uint32_t full_rounds,
    uint32_t partial_rounds,
    KernelStatus* status_ptr);

extern "C" __global__ void poseidon6_permute_kernel(
    uint64_t* states,
    uint32_t state_stride_words,
    uint32_t batch_len,
    uint32_t domain_tag,
    const uint64_t* round_constants,
    const uint64_t* mds_matrix,
    uint32_t full_rounds,
    uint32_t partial_rounds,
    KernelStatus* status_ptr);
```

- Inputs: `states` points to the first limb of the batch. The domain tag matches the Rust opcode (`PoseidonDomain::Fast`, `::Wide`, etc.) and is XORed into the capacity cell before the first round so kernels do not need to branch on opcodes.
- Each thread handles `state_stride_words` limbs. `state_stride_words` defaults to 12 for Poseidon2 and 24 for Poseidon6, but the parameter allows future packed layouts without changing kernel logic.
- Round counts are fixed to the Halo2 parameters already used in `poseidon_primitives` (`full_rounds = 8`, `partial_rounds = 56`), so `total_rounds = 72`. Kernels must execute all rounds sequentially to preserve determinism even if batching could permit early exits.
- MDS multiplication order follows the CPU implementation: re-materialise the full state before writing it back. Kernels must avoid in-place updates that would change intermediate state ordering.
- A per-launch `KernelStatus` is provided so kernels can report deterministic failure codes (e.g., invalid round count). Hosts disable the CUDA backend on the first non-zero status.

Poseidon constant tables are flattened in the host runtime before launch. The exact layout is:

| Constant | Dimensions | Source of truth | Notes |
|----------|------------|-----------------|-------|
| `POSEIDON2_RC` | `[72][3][4]` little-endian limbs | `poseidon2_round_constants_words()` in `crates/ivm/src/poseidon.rs` | First index = round; second = lane (`rate[0]`, `rate[1]`, `capacity`); third = limb. |
| `POSEIDON2_MDS` | `[3][3][4]` | `poseidon2_mds_words()` | Row-major matrix multiplied after each round. |
| `POSEIDON6_RC` | `[72][6][4]` | `poseidon6_round_constants_words()` | Six-lane state (rate 5 + capacity 1). |
| `POSEIDON6_MDS` | `[6][6][4]` | `poseidon6_mds_words()` | Same derivation as Poseidon2 but with a 6×6 matrix. |

The host copies these tables verbatim into device memory for each launch. Kernels must never mutate the tables and must read them with deterministic indexing (`round_idx * t + lane_idx` with `t ∈ {3,6}`).

To amortise host-side preparation work, the Rust accelerator caches the flattened constants in `OnceLock`s and exposes batched helpers (`ivm::poseidon2_many`, `ivm::poseidon6_many`) that feed the CUDA backend when available. Language bindings surface their own batch adapters (`iroha_python.gpu.poseidon2_cuda_many`, `poseidon6_cuda_many`, `CudaAccelerators.poseidon2Batch`, `poseidon6Batch`) so downstream SDKs can hash large manifests with a single launch while retaining deterministic CPU fallbacks.

##### BN254 field kernels

```
extern "C" __global__ void bn254_add_kernel(
    const uint64_t* lhs,
    const uint64_t* rhs,
    uint64_t* out,
    uint32_t elem_count,
    uint32_t stride_words);

extern "C" __global__ void bn254_sub_kernel(
    const uint64_t* lhs,
    const uint64_t* rhs,
    uint64_t* out,
    uint32_t elem_count,
    uint32_t stride_words);

extern "C" __global__ void bn254_mul_kernel(
    const uint64_t* lhs,
    const uint64_t* rhs,
    uint64_t* out,
    uint32_t elem_count,
    uint32_t stride_words);
```

- `elem_count` counts field elements, not limbs. Kernels iterate over `elem_count`, load four limbs from each operand respecting `stride_words`, and write four limbs back to `out`.
- Addition/subtraction mirrors the scalar routines in `bn254_vec`: perform limb-wise operation with carry/borrow tracking, then conditionally add/subtract the modulus once to bring the result into `[0, MODULUS)`.
- Multiplication performs a schoolbook `wide_mul` (8 limbs), then calls the same constant-time reduction as `bn254_vec::reduce_wide`. The reduction constants (`MODULUS`, `MODULUS_INV`, `R^2 mod p`) will be uploaded to `__constant__` arrays named `BN254_MOD`, `BN254_MOD_INV`, `BN254_R2`.
- Kernels must operate exclusively on integer types and keep intermediate results in 128-bit lanes (CUDA `unsigned __int128` or the emulation helpers already used in CPU code) so the arithmetic matches the Rust implementation bit-for-bit.

The host wrapper (`crate::cuda::bn254_*_cuda`) owns stream submission, zero-copy staging buffers, and parity tests. Kernel authors only have to satisfy the ABI above; the runtime handles batching, retries, and disabling the backend on first failure. As of Jan 2026 these kernels live in `crates/ivm/cuda/bn254.cu` and are wired into the BN254 helpers via the shared `bn254_launch_kernel` path (`crates/ivm/src/cuda.rs`), allowing scalar/SIMD fallbacks and CUDA execution to share identical integer arithmetic.

## Conclusion

This updated specification for the Iroha VM introduces robust enhancements to make the VM future-proof, high-performance, and suitable for privacy-preserving applications. By addressing prior critiques – such as ensuring automatic hardware feature utilization, targeting specific ZK use cases (private assets and voting), and maintaining a unified developer experience – the design strikes a balance between cutting-edge capability and pragmatic simplicity. The core principles remain **determinism**, **safety**, and **portability**, which are crucial in a blockchain context.

Moving forward, the implementation in Rust will serve as a reference to validate these design decisions. We anticipate that this updated IVM will enable the Hyperledger Iroha community to execute complex smart contracts with greater speed and to verify their execution with cryptographic certainty, unlocking new possibilities like confidential finance and transparent yet private governance.

All changes in this spec are made with deterministic upgrades in mind: new features activate through governance and require node upgrades before contracts can rely on them.

Finally, ongoing feedback and testing will be important. We will rigorously test the concurrency model under adversarial conditions and measure zero-knowledge proof performance in real scenarios (e.g., benchmark a private token transfer workflow) to fine-tune parameters. Any significant findings will be published and may lead to minor revisions. The aim is that IVM remains a **reliable, efficient, and innovative VM** for the blockchain’s next generation of applications.
