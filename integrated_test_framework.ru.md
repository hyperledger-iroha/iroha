# Фреймворк интеграционных тестов для Hyperledger Iroha (сеть из 7 узлов)

## Введение

Hyperledger Iroha v2 предоставляет богатый набор специальных инструкций Iroha (ISI) для управления доменами, активами, правами и пирами. Этот документ определяет фреймворк интеграционных тестов для сети из 7 пиров, чтобы проверять корректность, консенсус и межпировую согласованность в нормальных и отказных условиях (до 2 неисправных пиров).

Эти рекомендации отражают недавнюю миграцию схемы генезиса на много-транзакционные блоки.

Примечание по API: в этой кодовой базе Torii — это HTTP/WebSocket API (Axum). Тесты должны использовать HTTP‑эндпоинты (обычно порты 8080+). Никакой дополнительный RPC‑сервис не слушает на 50051.

## Цели

- Автоматизированная оркестрация 7 пиров: запуск и управление пирами программно для быстрого CI (основной вариант) или через Docker Compose (опционально).
- Конфигурация генезиса: запуск с общего Norito‑генезиса с детерминированными аккаунтами/ключами и необходимыми правами.
- Проверка ISI: систематически покрывать ISI через транзакции и проверять результирующее состояние.
- Межпировая согласованность: опрашивать несколько пиров после каждого шага, чтобы гарантировать идентичное состояние реестра.
- Проверки устойчивости к сбоям: при 1–2 офлайн или разделённых пирах остальные продолжают работу; вернувшиеся пиры догоняют без дивергенции.

## Режимы оркестрации

- Rust‑каркас (рекомендуется): `crates/iroha_test_network` локально запускает процессы `irohad`, выделяет порты, пишет конфиги на запуск и Norito‑генезис `.nrt`, следит за готовностью и высотами блоков и предоставляет утилиты остановки/перезапуска.
- Docker Compose (опционально): `crates/iroha_swarm` генерирует файл Compose для N пиров. Используйте, когда нужна контейнерная сеть или внешняя оркестрация.

## Настройка тестовой сети

**Блок генезиса:**

- Источник: `defaults/genesis.json`, который теперь группирует инструкции в массив `transactions`. Тесты могут добавлять инструкции через `NetworkBuilder::with_genesis_instruction` и начинать новую транзакцию через `.next_genesis_transaction()`. Получившийся блок сериализуется в Norito `.nrt`.
- Топология: хранится в первой транзакции (`transactions[0].topology`) и включает все 7 пиров (публичный ключ + адрес), чтобы каждый пир знал сеть с самого начала.
- Аккаунты/права: предпочитайте стандартные аккаунты из `crates/iroha_test_samples` (`ALICE_ID`, `BOB_ID`, `SAMPLE_GENESIS_ACCOUNT_KEYPAIR`) с явными грантами для сценариев тестирования, например `CanManagePeers`, `CanManageRoles`, `CanMintAssetWithDefinition`.
- Стратегия инъекции: в каркасе генезис обычно подаётся в один пир (“genesis submitter”); остальные пиры догоняют через синхронизацию блоков. В Compose укажите всем пирам один и тот же путь к генезису.

**Сеть и порты:**

- Torii HTTP API: `API_ADDRESS` (Axum). Для Compose пробросьте `8080`–`8086` для 7 пиров на хост. Хранис сам выделяет loopback‑порты.
- P2P: внутренний peer‑to‑peer адрес задаётся в `trusted_peers` и распространяется через gossip. Хранис автоматически выставляет `trusted_peers` на каждый запуск.

**Хранение данных:**

- Kura: хранилище блоков Iroha v2 (контейнер RocksDB/Postgres не нужен). Конфигурация через `[kura]` (например, `store_dir`). Отключите снапшоты для тестов через `[snapshot]`.

**Поток каркаса:**

1) Собрать сеть: `NetworkBuilder::new().with_peers(7)`; при необходимости `.with_pipeline_time(...)` и `.with_config_layer(...)` для overrides; выбрать топливо IVM через `IvmFuelConfig`.
2) Запустить пиры: `.start()` или `.start_blocking()`; пишет слои конфигурации, выставляет `trusted_peers`, инъецирует генезис для одного пира и ждёт готовности.
3) Готовность: `Network::ensure_blocks(height)` или `once_blocks_sync(...)` гарантирует, что высоты непустых блоков достигнут ожиданий на всех пирах. Альтернатива — опрос `Client::get_status()`.

**Поток Docker Compose (опционально):**

1) Сгенерировать compose: используйте `iroha_swarm::Swarm`, чтобы получить файл compose для N пиров. Пробросьте API‑порты на хост и установите переменные окружения (CHAIN, keys, TRUSTED_PEERS, GENESIS).
2) Запуск: `docker compose up`.
3) Готовность: опрашивайте HTTP‑эндпоинты Torii, пока статус не станет здоровым и высота блока не будет >= 1.

## Реализация тестового каркаса (Rust)

**Клиент и транзакции:**

- Используйте `iroha::client::Client` (HTTP/WebSocket) для отправки транзакций и запросов в Torii.
- Собирайте транзакции из последовательностей ISI `InstructionBox` или IVM‑байткода; подписывайте детерминированными `KeyPair` из `iroha_test_samples`.
- Полезные вызовы: `submit_blocking`, `submit_all_blocking`, `query(...).execute()/execute_all()`, `get_status()`, потоки блоков и событий через WebSocket.

**Межпировая согласованность:**

- После каждой операции опрашивайте каждый работающий пир (`Network::peers()` -> `peer.client()`) и сравнивайте результаты (например, балансы, определения, список пиров). Это обеспечивает строгие проверки согласованности, а не только проверку одного пира.

**Инъекция сбоев (без контейнеров):**

- Используйте утилиты relay/proxy в `integration_tests/tests/extra_functional/unstable_network.rs`, чтобы переписать `trusted_peers` на TCP‑прокси и выборочно приостанавливать связи. Это позволяет организовывать разделения, потери и целевые повторные подключения.

**Требования IVM:**

- Некоторые тесты требуют заранее собранных IVM‑примеров; убедитесь, что существует `crates/ivm/target/prebuilt/build_config.toml`. При отсутствии тесты могут быть пропущены (текущие интеграционные тесты это проверяют).

## Сценарий сети из 7 узлов

1) Запустить сеть из 7 пиров (каркас или Compose) и дождаться коммита генезиса на всех пирах.
2) Выполнить набор ISI:
   - Регистрация доменов/аккаунтов/активов; выдача/отзыв прав; чеканка/сжигание/перевод активов; установка/удаление ключ‑значение; регистрация триггеров; обновление исполнительного окружения.
3) После каждого логического шага выполнить межпировые запросы для проверки идентичного состояния.
4) Остановить 1–2 пира, продолжить отправку транзакций с оставшихся 5 пиров; убедиться в прогрессе и согласованности между работающими пирами.
5) Перезапустить остановленные пиры и проверить догон и равенство межпирового состояния.

## Использование в CI

- Build: `cargo build --workspace`
- Предварительная сборка IVM‑примеров (по необходимости)
- Test: `cargo test --workspace`
- Дополнительный строгий lint (опционально): `cargo clippy --workspace --all-targets -- -D warnings`

## Заключение

Этот фреймворк использует репозиторный Rust‑каркас (`iroha_test_network`) и HTTP‑клиент (`iroha::client::Client`) для проверки сети Iroha v2 из 7 пиров. Он делает упор на межпировую согласованность, реалистичные сценарии сбоев и воспроизводимый запуск/останов, подходящий для CI. Docker Compose через `iroha_swarm` доступен, когда предпочтительна контейнеризация.

## Explorer

- Любой блокчейн‑эксплорер, поддерживающий Torii HTTP/WebSocket, может подключаться к каждому пиру независимо. Каждый пир предоставляет Torii‑эндпоинт (host:port) для статуса, блоков, запросов и событий.
- С Rust‑каркасом: после сборки сети можно получить URL всех пиров:

  ```rust
  use iroha_test_network::NetworkBuilder;

  let network = NetworkBuilder::new().with_peers(7).build();
  let urls = network.torii_urls();
  // напр. ["http://127.0.0.1:8080", ..., "http://127.0.0.1:8086"]
  ```

  Также доступны хелперы на уровне пира: `peer.api_address()` и `peer.torii_url()`.

- С Docker Compose (`iroha_swarm`): сгенерируйте compose для 7 пиров и пробросьте `8080..8086` на хост; укажите эти адреса в эксплорере. Если эксплорер поддерживает несколько эндпоинтов — настройте все 7; иначе запускайте по одному экземпляру на пир.
